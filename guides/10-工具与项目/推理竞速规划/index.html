❓ 为什么技术点不能孤立展示？

通俗类比：学习LLM推理技术就像品尝一道菜，不能只看食材（单个技术），还要理解烹饪手法（技术组合）和用餐体验（整体效果）。
常见误区：只盯着一个技术优化（如只调KV缓存）可能忽略与其他模块（如批处理、采样）的交互，导致整体效果反而变差。
深入探讨：例如，在优化KV缓存时，如果不对批处理策略同步调整，可能导致缓存命中率下降，实际吞吐量提升不足5%，而协同优化可实现30%+提升。

第 1 层：算法设计 — 决定"算什么"

改变模型的计算方式，减少不必要的计算量

MHA → MQA → GQA → MLA
注意力头的演进：从多头到分组到潜空间，KV Cache 越来越小
Speculative Decoding
小模型猜 + 大模型验，并行验证多个 token
Early Exit
简单 token 在浅层就输出，不用跑完整个模型
MoE（混合专家）
每次只激活部分专家，计算量按比例减少
第 2 层：计算优化 — 决定"怎么算"

优化单次计算的效率

FlashAttention (v1→v2→v3)
IO 感知的注意力计算，减少 HBM 访问
FlashDecoding
Decode 阶段的并行化，解决长序列瓶颈
量化（FP8/INT8/INT4）
降低精度换速度，GPTQ/AWQ/SmoothQuant
Kernel Fusion
合并多个操作为一个 kernel，减少启动开销
第 3 层：内存管理 — 决定"怎么存"

优化 KV Cache 和权重的存储方式

PagedAttention
KV Cache 分页管理，消除内存碎片
Prefix Caching
相同前缀的请求共享 KV Cache
RadixAttention
用 Radix Tree 管理 KV Cache，更灵活的前缀共享
KV Cache 量化
KV Cache 也量化，进一步压缩内存
第 4 层：调度策略 — 决定"什么先算"

决定多个请求如何排队和并行

Static Batching
固定批次，等最长的算完（浪费）
Continuous Batching
迭代级调度，完成的请求立即替换
Chunked Prefill
长 prompt 分块处理，避免阻塞 decode
Priority Scheduling
按优先级调度，SLA 感知
第 5 层：系统框架 — "组装"所有优化

将上述技术组合成完整的推理引擎

vLLM
PagedAttention + Continuous Batching + Prefix Caching
SGLang
RadixAttention + Continuous Batching + Constrained Decoding
TensorRT-LLM
FlashAttention + FP8 + In-flight Batching + Graph Optimization
llama.cpp
GGUF 量化 + CPU/NPU 适配 + 边缘部署

🔗 技术演进链（不是孤立的，是有因果关系的）

通俗类比：技术迭代像菜品改良：顾客抱怨上菜慢（延迟高）→ 厨师优化备菜流程（批处理）→ 又发现菜品质量下降（精度损失）→ 调整火候（量化校准）。
常见误区：误认为新技术完全替代旧技术（如Mamba替代Transformer），实际上多数是互补融合（如混合架构），需理解演进逻辑而非简单替换。
深入探讨：例如：Attention机制（2017）→ KV缓存（优化推理）→ PagedAttention（解决内存碎片）→ FlashAttention（优化计算）。每一步都解决上一步的遗留问题。

四个可视化模式

模式 1：技术树（Tech Tree）

像游戏科技树一样展示技术演进关系

解锁机制：理解前置技术才能解锁后续技术

模式 2：赛道竞速（Racing）

多条赛道对比不同技术组合的性能

交互：

  • 勾选/取消技术，看性能变化（像改装赛车）
  • 调整参数（batch_size、seq_len），看不同场景
  • 动画展示 token 流动过程
模式 3：流水线透视（Pipeline Inspector）

深入单个请求的处理过程，看每一步在做什么

展示：

  • Prefill 和 Decode 的计算特点差异
  • GPU 利用率随时间变化
  • KV Cache 的增长过程
  • Speculative Decoding 的猜-验过程
模式 4：性能仪表盘（Dashboard）

展示不同配置下的真实 benchmark 数据
