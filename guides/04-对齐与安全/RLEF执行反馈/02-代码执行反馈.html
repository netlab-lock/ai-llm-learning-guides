1. CodeRL (Le et al. 2022)

1. CodeRL (Le et al. 2022)的核心组件与工作原理

CodeRL 是首个用代码执行结果训练 LLM 的方法。

CodeRL 训练流程: Actor (LLM) 生成代码 │ ▼ ┌─────────────────────────────────┐ │ Critic Model │ │ 预测代码的功能正确性 │ │ 输入: 代码 + 测试用例 │ │ 输出: 正确概率 │ └───────────────┬─────────────────┘ │ ▼ ┌─────────────────────────────────┐ │ 执行验证 │ │ │ │ 编译代码: │ │ 编译错误 → 奖励 = -1.0 │ │ 编译成功 → 继续 │ │ │ │ 运行测试: │ │ 运行时错误 → 奖励 = -0.5 │ │ 测试失败 → 奖励 = -0.1 │ │ 测试通过 → 奖励 = +1.0 │ └───────────────┬─────────────────┘ │ ▼ PPO 更新 Actor (奖励 = 执行结果)
通俗类比： RLHF训练就像训练导盲犬——第一步(SFT)教它基本技能，第二步(奖励模型)让它学习人类对"好行为"的偏好，第三步(PPO)通过反复实践强化正确行为。奖励模型就是那个"打分的训练师"。
常见误区： RLHF最容易出现"奖励黑客"(Reward Hacking)——模型学会了讨好评分器而非真正变好。比如模型发现加"让我为您详细解答"这种套话能提高奖励分，就疯狂使用。对策是定期更新奖励模型和KL约束。
深入探讨： DPO的核心推导从RLHF的最优策略出发。RLHF目标：max_π E[r(x,y)] - β·KL(π||π_ref)。其闭式解为π*(y|x) = π_ref(y|x)·exp(r(x,y)/β)/Z(x)。反解奖励：r(x,y) = β·log(π*(y|x)/π_ref(y|x)) + β·log(Z(x))。代入Bradley-Terry偏好模型P(y_w≻y_l|x) = σ(r(y_w)-r(y_l))，Z(x)被消去，得到DPO损失：L = -E[log σ(β·(log π(y_w)/π_ref(y_w) - log π(y_l)/π_ref(y_l)))]。

2. StepCoder (Dou et al. 2024)

2. StepCoder (Dou et al. 2024)的核心组件与工作原理

StepCoder 将长代码生成分解为多步，每步用执行结果反馈。

StepCoder 分步训练: 问题: "实现一个排序算法" Step 1: 生成函数签名 def sort(arr): ... 测试: 函数可调用 → 奖励 = +0.3 Step 2: 生成核心逻辑 def sort(arr): for i in range(len(arr)): for j in range(i+1, len(arr)): if arr[i] > arr[j]: arr[i], arr[j] = arr[j], arr[i] 测试: 部分测试通过 → 奖励 = +0.6 Step 3: 完善边界处理 def sort(arr): if not arr: return arr ... 测试: 全部通过 → 奖励 = +1.0 课程学习: 从简单任务到复杂任务
通俗类比： DPO就像直接从"对比照"学化妆——不需要一个评分系统(奖励模型)，只需要看到"这个妆比那个好"的对比，就能学到怎么化。数学上证明了DPO和RLHF的最优解等价，但DPO少了一步训练奖励模型的开销。

# Docker 沙箱执行
import docker

def run_in_sandbox(code, timeout=10):
    """在 Docker 沙箱中执行代码"""
    client = docker.from_env()

    # 创建受限容器
    container = client.containers.run(
        "python:3.11-slim",
        f"python -c '{code}'",
        detach=True,
        mem_limit="256m",      # 内存限制
        cpu_period=100000,
        cpu_quota=50000,        # CPU 限制
        network_disabled=True,  # 禁用网络
        read_only=True,         # 只读文件系统
    )

    # 等待执行完成或超时
    try:
        result = container.wait(timeout=timeout)
        logs = container.logs().decode()
        return {"exit_code": result["StatusCode"], "output": logs}
    except:
        container.kill()
        return {"exit_code": -1, "output": "Timeout"}
    finally:
        container.remove()

  1. 核心原理: 分级代码执行奖励函数设计：基于HumanEval的测试用例，将奖励分为多个层级，而非简单的0/1二值信号。(1)编译/解析奖励（+0.1）——代码能成功被Python解释器解析，无语法错误；(2)类型检查奖励（+0.1）——通过基本的类型检查（如mypy静态分析）；(3)部分测试通过奖励（+0.2×通过率）——按测试用例通过比例给分，通过50%的测试给0.1分；(4)全部测试通过奖励（+0.5）——所有测试用例通过，这是核心奖励；(5)效率奖励（+0.1）——执行时间在合理范围内（如<1秒），避免暴力解法。这种分级设计的好处：缓解奖励稀疏问题（大部分代码至少能获得编译奖励）、引导模型逐步改进（从语法正确到逻辑正确到高效）、提供更丰富的梯度信号。
    关键要点: ①分级奖励缓解稀疏奖励问题 ②编译通过是最基本的奖励信号 ③部分通过率提供连续梯度 ④效率奖励引导优化意识 ⑤总奖励权重可调，需根据任务特性微调
    实际案例: 在HumanEval基准上，使用分级奖励训练的模型比使用二值奖励的模型pass@1提升约8-12%，因为分级奖励提供了更丰富的优化信号。
    关联知识: 与执行反馈训练总览（01）的奖励设计原则、RLVR（06）的可验证奖励机制直接相关。
  2. 核心原理: Docker沙箱执行环境实现要点：(1)基础镜像——使用精简的Python官方镜像（python:3.11-slim），仅包含运行时依赖，减少攻击面；(2)资源限制——通过Docker的--memory（256MB）、--cpus（0.5）、--pids-limit（100）限制资源，防止OOM和CPU滥用；(3)网络隔离——--network=none完全禁止网络访问，防止代码外联；(4)文件系统隔离——--read-only挂载只读根文件系统，仅/tmp可写，防止代码篡改宿主机文件；(5)用户隔离——以非root用户（--user=1000:1000）运行，限制系统调用权限；(6)超时控制——使用signal.alarm或subprocess的timeout参数，执行超过10秒强制终止；(7)输出捕获——重定向stdout/stderr，限制输出大小（max 1MB）防止输出爆炸攻击。实现示例：subprocess.run(["docker", "run", "--rm", "--memory=256m", "--cpus=0.5", "--network=none", "--read-only", "--user=1000:1000", "python:3.11-slim", "python", "/code/solution.py"], timeout=10)。
    关键要点: ①网络隔离是安全沙箱的第一要务 ②资源限制防止拒绝服务攻击 ③只读文件系统+非root用户是双重保护 ④超时控制防止无限循环 ⑤生产环境建议使用gVisor替代标准Docker获得更强隔离
    实际案例: 某代码竞赛平台使用上述Docker沙箱方案，在10万次代码执行中成功阻止了所有逃逸尝试（包括fork炸弹、磁盘填充、网络外联等攻击）。
    关联知识: 与执行反馈训练总览（01）的沙箱架构、工业部署安全（08）的容器安全策略直接相关。

多语言沙箱执行与安全隔离工程实践

在大规模代码执行反馈训练中，构建一个支持多编程语言的安全沙箱执行环境是工程上的核心挑战。不同于简单的 Python 脚本执行，生产级的 RLEF 系统需要支持 Python、Java、C++、JavaScript、Go 等 10+ 种编程语言的并发执行，并保证每种语言的执行环境完全隔离。技术方案采用"微容器"架构：每个代码执行请求在独立的 gVisor 沙箱容器中运行，gVisor 通过在用户空间实现 Linux 内核系统调用来提供比标准 Docker 更强的安全隔离——它拦截并审计每一个系统调用，阻止文件系统写入（仅允许 /tmp 目录）、网络访问（完全禁止）、以及危险的进程操作（禁止 fork、execve 的嵌套调用）。资源限制方面，每个容器分配 256MB 内存、0.5 CPU 核心、10MB 磁盘空间，执行超时设为 10 秒（编译型语言如 C++/Java 额外给予 30 秒编译时间）。并发管理采用"沙箱池"预热策略——预先创建 1000 个空闲容器，新请求直接从池中获取，执行完毕后容器不销毁而是重置状态后归还池中，这将容器创建延迟从平均 800ms 降低到 5ms 以内。日志和审计方面，每次执行记录完整的输入代码、编译输出、运行输出、退出码、执行时间和资源消耗，形成可追溯的执行审计链，用于后续的奖励计算和质量分析。

自动测试生成与覆盖率提升策略

代码执行反馈的一个根本性局限是"测试覆盖率悖论"——通过所有已知测试用例并不意味着代码完全正确。研究表明，在 HumanEval 基准上，即使测试通过率 100%，仍有 5-10% 的代码存在隐藏 bug（如边界条件处理不当、整数溢出、浮点精度问题）。为了缓解这一问题，前沿的 RLEF 系统引入了"LLM 辅助测试生成"策略：使用一个独立的测试生成模型（通常是同一基座模型的微调版本）为每个编程问题自动生成额外的测试用例。生成流程分为三步：(1) 边界条件分析——让 LLM 分析问题描述，识别可能的边界情况（如空输入、极大值、负数、重复元素等），生成对应的测试输入；(2) 随机测试生成——使用基于属性的测试框架（类似 Hypothesis）生成大量随机输入，验证代码在随机数据上的鲁棒性；(3) 差分测试——如果存在多个正确代码样本，比较它们在相同输入上的输出差异，输出不一致的输入即为潜在的 bug 检测用例。实验表明，将自动生成的测试用例加入执行验证后，代码质量评估的准确率从 90% 提升至 96%，误判率（将错误代码判为正确）从 8% 降低到 3%。这一改进对于 RLEF 训练的最终效果影响显著——更准确的奖励信号意味着更强的模型优化方向。

# 多语言沙箱执行管理器
import asyncio
from typing import Dict, List
from dataclasses import dataclass

@dataclass
class ExecutionResult:
    language: str
    exit_code: int
    stdout: str
    stderr: str
    execution_time_ms: float
    memory_used_mb: float

class MultiLangSandbox:
    """多语言安全沙箱执行管理器"""

    LANGUAGE_CONFIGS = {
        "python": {
            "image": "python:3.11-slim",
            "run_cmd": "python -c '{code}'",
            "timeout": 10,
            "compile": False,
        },
        "cpp": {
            "image": "gcc:13",
            "compile_cmd": "g++ -O2 -o /tmp/sol /tmp/sol.cpp",
            "run_cmd": "/tmp/sol",
            "timeout_compile": 30,
            "timeout_run": 10,
            "compile": True,
        },
        "java": {
            "image": "openjdk:21-slim",
            "compile_cmd": "javac /tmp/Solution.java",
            "run_cmd": "java -cp /tmp Solution",
            "timeout_compile": 30,
            "timeout_run": 10,
            "compile": True,
        },
    }

    def __init__(self, pool_size: int = 200):
        self.pool = asyncio.Queue(maxsize=pool_size)
        self.pool_size = pool_size

    async def execute(
        self, code: str, language: str, test_cases: List[str]
    ) -> List[ExecutionResult]:
        """在沙箱中执行代码并运行测试用例"""
        config = self.LANGUAGE_CONFIGS[language]
        results = []

        for test_input in test_cases:
            container = await self._get_container(config["image"])
            try:
                if config.get("compile"):
                    # 编译型语言: 先编译再运行
                    compile_result = await container.exec(
                        config["compile_cmd"],
                        timeout=config["timeout_compile"]
                    )
                    if compile_result.exit_code != 0:
                        results.append(ExecutionResult(
                            language=language, exit_code=-1,
                            stdout="", stderr=compile_result.stderr,
                            execution_time_ms=0, memory_used_mb=0
                        ))
                        continue

                # 运行代码
                result = await container.exec(
                    config["run_cmd"].format(code=code),
                    stdin=test_input,
                    timeout=config.get("timeout", config.get("timeout_run", 10))
                )
                results.append(ExecutionResult(
                    language=language,
                    exit_code=result.exit_code,
                    stdout=result.stdout,
                    stderr=result.stderr,
                    execution_time_ms=result.duration_ms,
                    memory_used_mb=result.memory_mb,
                ))
            finally:
                await self._return_container(container)

        return results

    async def _get_container(self, image: str):
        """从池中获取容器或创建新容器"""
        try:
            return self.pool.get_nowait()
        except asyncio.QueueEmpty:
            return await self._create_container(image)

    async def _return_container(self, container):
        """重置容器状态并归还到池中"""
        await container.cleanup()
        try:
            self.pool.put_nowait(container)
        except asyncio.QueueFull:
            await container.destroy()

4. 代码执行反馈的核心技术挑战与工程实践

沙箱安全与执行隔离

代码执行反馈系统面临的首要技术挑战是安全的代码执行环境构建。LLM生成的代码可能包含恶意操作——例如文件系统遍历、网络外联、无限循环或内存炸弹。因此，所有代码必须在严格隔离的沙箱环境中执行。业界主流的方案包括三类：基于容器的隔离（Docker/gVisor）——每次执行启动一个轻量级容器，通过namespace和seccomp限制系统调用，安全性高但冷启动延迟约200-500ms；基于虚拟机的隔离（Firecracker/microVM）——使用轻量级虚拟机提供硬件级隔离，安全性最高但启动延迟约100-300ms；基于WebAssembly的隔离（Wasmtime/Wasmer）——通过WASM字节码提供内存安全的沙箱，启动延迟极低（<10ms）但语言支持有限。在生产环境中，通常采用容器池化（Container Pooling）技术预先启动一批容器实例，通过队列管理避免频繁的创建/销毁开销，将平均延迟降低到50ms以内。同时，需要设置严格的资源限制：CPU时间（通常5-30秒）、内存上限（256MB-1GB）、磁盘空间（100MB-1GB）、网络访问（默认禁止）和进程数量（限制为10-50个）。

5. 从执行反馈到强化学习的完整链路

奖励信号设计与信用分配

将代码执行结果转化为有效的强化学习奖励信号是一个精巧的工程问题。最简单的方案是二元奖励——测试全部通过给+1，否则给0。但这种稀疏奖励在长代码生成场景中面临严重的信用分配问题：一个100行的代码可能因为第37行的一个off-by-one错误导致所有测试失败，模型难以将失败归因到具体的错误行。解决方案包括：逐步奖励（Step-wise Reward）——将代码生成分解为多个步骤，每步完成后执行当前代码并给予部分奖励，如StepCoder和CoderEval采用的方案；行级反馈（Line-level Feedback）——通过代码覆盖率分析和差异比较，定位到具体的错误行并在奖励中编码位置信息；编译器反馈（Compiler Feedback）——将编译器的错误和警告信息作为额外的上下文输入，帮助模型理解错误原因。研究表明，结合编译器反馈的RL训练在HumanEval基准上比纯二元奖励提升约8-12个百分点，证明了细粒度反馈信号的重要性。
