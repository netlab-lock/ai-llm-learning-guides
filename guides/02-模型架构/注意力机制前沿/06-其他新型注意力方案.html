🔧 06 · 其他新型注意力方案

除了 Differential Attention 和线性注意力，还有多种针对特定场景优化的注意力方案。本章介绍这些方案的原理和应用。

1. Sliding Window Attention

注意力机制的前沿研究与工程优化

注意力机制的工程优化是推理加速的核心。FlashAttention系列(A1/A2/A3)通过IO-aware的tiling策略，将注意力计算的HBM访问次数从O(n²)降到O(n²d²/M)，其中M是SRAM大小。在A100 GPU上(SRAM 20MB)，这使得4K序列的注意力计算速度提升3倍，128K序列提升8倍。FlashAttention-3进一步利用Hopper架构的异步特性(TMA+WGMMA)，实现了接近理论峰值的性能。

GQA(Grouped Query Attention)是另一个重要的工程优化。标准MHA中每个头有独立的K和V，导致KV Cache随头数线性增长。GQA将K/V头数减少到Q头数的1/g(通常g=8)，KV Cache减少到1/8。LLaMA-2/3使用GQA(g=8)，使得70B模型的KV Cache从每token 2.5MB降到0.3MB，显著提升了batch_size和吞吐量。

线性注意力的最新进展值得关注。RWKV系列将线性注意力与RNN的递推形式结合，实现了O(1)的推理复杂度——每步只需常量计算，不像标准注意力需要重新计算所有历史。Mamba系列(State Space Model)进一步将递推状态压缩到固定大小，在超长序列(100K+)上展现出比标准Transformer更好的效率-质量权衡。

通俗类比：滑动窗口像"用放大镜看书"——只能看到放大镜范围内的文字，但可移动放大镜。多层叠加像多次移动，最终看到整页。
常见误区：滑动窗口不是"免费午餐"——复杂度从O(n²)降到O(wn)，但远距离token需多层传播。对需要精确远距离检索的任务可能不够。
深入探讨：Mistral的实现使用"滚动缓冲区"——KV Cache是固定大小循环数组，新token覆盖最旧token。KV Cache大小恒定，与序列长度无关。

1.1 原理

Sliding Window Attention 限制每个 token 只关注固定窗口内的 token：

Sliding Window Attention

Attention(Q_i, K, V) = softmax(Q_i · K_{[i-w:i]}^T / √d) · V_{[i-w:i]}

其中 w = 窗口大小

1.5 Longformer 的 Dilated Sliding Window

Longformer 使用了带间隔的滑动窗口：

  • 局部窗口：每个 token 关注附近的 token
  • 间隔注意力：每隔几个位置关注一个 token
  • 全局 token：某些特殊 token 关注所有位置

2. Grouped Query Attention (GQA)

# FlashAttention核心原理(伪代码)
import torch

def flash_attention(Q, K, V, block_size=256):
    """分块计算注意力，避免存储N×N矩阵"""
    N, d = Q.shape
    O = torch.zeros_like(Q)
    l = torch.zeros(N)  # log-sum-exp的分母
    m = torch.full((N,), float('-inf'))  # 行最大值

    # 将K/V分块
    for j in range(0, N, block_size):
        Kj = K[j:j+block_size]
        Vj = V[j:j+block_size]

        # 计算当前块的注意力分数
        S = Q @ Kj.T / (d ** 0.5)  # (N, block_size)

        # 在线softmax更新
        m_new = torch.max(m, S.max(dim=-1).values)
        exp_old = torch.exp(m - m_new)
        exp_new = torch.exp(S - m_new.unsqueeze(-1))

        l_new = exp_old * l + exp_new.sum(dim=-1)
        O = (exp_old.unsqueeze(-1) * O + exp_new @ Vj) / l_new.unsqueeze(-1)

        m = m_new
        l = l_new

    return O  # 与标准注意力数学等价，但内存O(N)而非O(N²)

# 核心算法实现示例
import torch
import torch.nn as nn

class TransformerBlock(nn.Module):
    """Transformer基础模块"""
    def __init__(self, d_model=4096, n_heads=32, d_ff=11008):
        super().__init__()
        self.attn = nn.MultiheadAttention(d_model, n_heads, batch_first=True)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.SiLU(),
            nn.Linear(d_ff, d_model)
        )
        self.norm1 = nn.RMSNorm(d_model)
        self.norm2 = nn.RMSNorm(d_model)
    
    def forward(self, x, mask=None):
        # Pre-norm架构
        h = self.norm1(x)
        h, _ = self.attn(h, h, h, attn_mask=mask)
        x = x + h  # 残差连接
        x = x + self.ffn(self.norm2(x))
        return x
通俗类比：GQA像"共享笔记"——标准MHA每学生（Q头）都有自己的笔记（KV头），GQA让几人共享一份。信息略有损失，但节省大量存储。
常见误区：GQA不是"降级版MHA"。实验表明GQA(32:8)性能损失仅0.5%，但KV Cache减少4倍。极具性价比的优化。
# Sliding Window Attention 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

练习：计算GQA(32:8)的KV Cache大小。d_k=128, n=4096时，GQA比MHA节省多少显存？
✅ 参考答案

核心原理：MHA的KV Cache：每层每token存储K和V各32头×128维=8192元素。FP16下每token每层=8192×2B×2(K+V)=32KB。n=4096时每层=4096×32KB=128MB。GQA(32:8)的KV Cache：32个Q头共享8个KV头，每层每token存储K和V各8头×128维=2048元素。FP16下每token每层=2048×2B×2=8KB。n=4096时每层=4096×8KB=32MB。节省=(128-32)/128=75%。GQA的KV Cache仅为MHA的1/4。
关键要点：①GQA(32:8)的KV Cache是MHA的25%；②节省的显存可用于更大的batch size或更长的序列；③GQA的性能损失通常<1%（在大多数基准上）。
实际案例：Llama-3-70B使用GQA(8:1)，KV Cache从每层256MB降至32MB，节省87.5%。
关联知识：与MQA(Multi-Query Attention)对比——MQA是GQA的极端情况(所有头共享1个KV头)。

3. Multi-Latent Attention (MLA)

通俗类比：MLA像"压缩文件"——标准MHA存完整KV（原始文件），MLA只存压缩潜变量c_t（zip文件），需时再解压。压缩比5-10倍，质量损失约1%。
常见误区：MLA的"压缩"是有损的——从潜变量恢复的K/V与原始值不完全相同。但实践中损失通常可忽略。
深入探讨：MLA需特殊处理RoPE——RoPE需在Key上应用，但MLA的Key从潜变量恢复。DeepSeek的方案是额外添加小型RoPE分量。

3.1 DeepSeek-V2/V3 的 MLA

MLA (Multi-Latent Attention) 是 DeepSeek 提出的一种 KV Cache 压缩方案：

MLA 公式

标准 MHA: Cache = [K₁...K_h, V₁...V_h] ← 存储 h 个头的 KV

MLA: 压缩为低维潜变量:
c_t = W_c · x_t ← 压缩到低维 (d << h×d_k)
K_t = W_k · c_t ← 从潜变量恢复 Key
V_t = W_v · c_t ← 从潜变量恢复 Value

Cache 只需存储 c_t (维度 d)，而非完整的 K, V

3.3 MLA 与 RoPE 的兼容性

MLA 需要特殊处理位置编码：

  • 问题：RoPE 需要在 Key 上应用，但 MLA 的 Key 是从潜变量恢复的
  • 解决方案：额外添加一个小型 RoPE 分量
  • 效果：保持位置编码能力，同时享受 MLA 的压缩优势

4. Cross-Layer Attention

通俗类比：跨层注意力像"共享会议室"——标准Transformer每层有独立会议室（KV Cache），跨层让相邻层共享，节省空间但可能需排队。
常见误区：跨层共享KV会导致性能下降——相邻层KV通常相似，但跨太多层（如全局共享）会导致信息损失。最优策略是相邻2层共享。
