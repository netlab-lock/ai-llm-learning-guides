💾 05 · 长上下文训练的显存优化

长上下文训练的显存瓶颈不仅来自模型参数，更来自注意力矩阵和激活值。序列长度从 4K 增加到 128K，显存需求增加 1024 倍（O(n²)）。

1. 长上下文训练的显存瓶颈

通俗类比：GPU显存就像你书桌的面积。桌子越大（显存多），能同时摊开的书（模型参数）和草稿纸（KV Cache）就越多。但桌子总是不够大，你得学会整理和归档（内存管理）。
常见误区：最大的错误是认为显存不足就直接升级硬件。在购买前，应先分析瓶颈：是模型参数太大，还是上下文太长导致KV Cache爆炸？错误诊断会导致金钱和时间的巨大浪费。
深入探讨：以训练一个7B参数的模型为例，FP16下仅模型参数就占14GB显存。当序列长度为4096时，单层KV Cache约占0.5GB。一个32层模型，仅KV Cache就需要约16GB。这还不包括优化器状态、梯度和激活值，总显存需求轻松超过80GB。
显存构成饼图 (示意):
[ ] 参数 (14GB)
[=] KV Cache (16GB)
[.] 梯度/优化器 (14GB)
[o] 激活值 (巨大且动态)

KV Cache~16 GB (随序列长度线性增长)高 (核心瓶颈)

激活值动态，可达数十GB高 (可用激活重计算)

1.1 显存消耗分析

长上下文训练的显存消耗

模型参数: P × sizeof(dtype) (与序列长度无关)
梯度: P × sizeof(dtype) (与序列长度无关)
优化器状态: 2P × sizeof(dtype) (Adam)

注意力矩阵: n² × H × sizeof(dtype) ← O(n²) 增长!
KV 存储: 2 × n × d × L × sizeof(dtype) ← O(n) 增长
激活值: n × d × L × sizeof(dtype) ← O(n) 增长

例: 7B 模型, FP16, batch=1
n=4K: 注意力矩阵 ≈ 0.5GB
n=128K: 注意力矩阵 ≈ 512GB ← 超出单 GPU 显存!

1.2 为什么注意力矩阵是瓶颈

在反向传播时，需要存储注意力矩阵的中间结果（用于计算梯度）。这就是为什么长上下文训练比推理困难得多——推理不需要存储这些中间结果。

2. Sequence Parallelism (SP)

通俗类比：并行策略就像分工做菜。张量并行(TP)是一道菜由多个厨师同时切不同食材；流水线并行(PP)是不同厨师负责洗、切、炒等不同工序。序列并行(SP)则是把长食谱（序列）分给不同厨师，各自负责一部分。
常见误区：最易犯的错误是在非Transformer层（如嵌入层、最终LayerNorm）也强行进行序列切分。这些层的计算通常不可分割，错误的切分会导致张量形状错误和数据不一致。
深入探讨：SP的核心思想是将序列维度（通常是第1维）进行切分。对于多头注意力，每个SP组只需处理一部分KV序列，然后通过AllGather通信汇总。它常与Tensor Parallelism结合，使每块GPU的计算负载和显存占用更均衡。例如，在Megatron-LM中，SP可将每块GPU的峰值激活显存降低近8倍。
序列并行 (SP) 示意图:
GPU0: [token_0 ... token_N-1]
GPU1: [token_N ... token_2N-1]
  ↓ AllGather 通信
GPU0/1: 完整的Key/Value序列 (用于注意力计算)
并行策略切分维度主要解决的问题主要通信开销

2.1 基本思想

Sequence Parallelism 将序列维度分割到多个 GPU 上，每个 GPU 只处理序列的一部分。

2.2 SP 的挑战

  • 注意力通信：Attention 层需要所有 token 的信息，必须跨 GPU 通信
  • 因果注意力：在因果注意力中，前面的 token 不需要看到后面的 token，可以优化通信
  • 负载均衡：不同位置的 token 数量可能不均（如 padding）

3. Ring Attention

通俗类比：Transformer的注意力计算像一群人开大会。传统方式是每个人( token)都和所有其他人直接对话，通信量巨大。Ring Attention则像把大家分成小组，围成一个圆圈(Ring)传递信息，依次完成讨论，大大减少了同时通信的压力。
常见误区：Ring Attention并非万能。当序列长度并不远超单卡显存容量时，其引入的额外通信延迟和计算复杂度可能得不偿失。它最适合用于处理超长序列(如数十万token)的训练场景。
深入探讨：Ring Attention通过将Q、K、V张量按序列维度分块，并安排到一个环形拓扑的GPU组中。在计算注意力时，K、V块在环上循环传递，每块GPU只计算本地Q块与当前接收的K、V块的注意力分数，并进行增量更新。通信与计算可以重叠，从而隐藏大部分通信时间。
Ring Attention 通信示意图:
  [GPU0: Q0] ←-- K0, V0 -- [GPU3: Q3]
     ↑                     ↓
  [GPU1: Q1] -- K1, V1 --→ [GPU2: Q2]
 (初始K,V块分布)  (循环传递并计算)
特性传统注意力 (单卡)Ring Attention

3.1 原理

Ring Attention (Liu et al. 2023) 是一种分布式注意力算法，将注意力计算分布在多个 GPU 上，通过环形通信实现。

3.2 Ring Attention 的关键优化

  • 计算-通信重叠：在接收下一块 K,V 的同时计算当前块的注意力
  • 因果优化：对于因果注意力，后面的 GPU 不需要等待前面的 K,V
  • 显存效率：每个 GPU 只存储 n/P 的 K,V 块

3.3 Ring Attention 的复杂度

Ring Attention 复杂度 (P 个 GPU):

计算量: O(n²d/P) ← 线性并行
通信量: O(n²d/P) ← 与计算量重叠
显存: O(n²/P) ← 每 GPU 存储 1/P 的注意力矩阵

4. Ulysses (DeepSpeed)

通俗类比：想象一个需要完成的超长报告。传统方式是所有人都看完整份材料再讨论，效率低。Ulysses方法则是将报告按章节分给不同小组（GPU），每个小组只研究自己那部分，并与其他小组交换核心观点（All-to-All通信），最后整合出完整见解。
常见误区：Ulysses的All-to-All通信在跨节点、高延迟网络上可能成为性能瓶颈。在配置时，需要仔细平衡序列并行度（SP group size）和节点内/节点间的拓扑结构，避免通信成为主要耗时部分。
深入探讨：DeepSpeed的Ulysses序列并行采用All-to-All通信原语。在计算注意力前，各GPU将自己持有的序列分块（包含Q, K, V）重新分发，使得每块GPU最终获得一个完整的、但更短的序列（所有头的全部注意力成分）。这要求通信量为 O(seq_len * hidden_dim)，但能实现更均衡的计算负载。
Ulysses (DeepSpeed) All-to-All 通信:
GPU0持有: [seq_chunk_0] -> 所有GPU的部分序列头
GPU1持有: [seq_chunk_1] -> 所有GPU的部分序列头
  ... 执行 All-to-All ...
GPU0得到: [head_0_full_seq, head_1_full_seq, ...]

实现复杂度较高相对简洁 (依赖库)

4.1 原理

Ulysses (DeepSpeed) 使用 All-to-All 通信实现序列并行。与 Ring Attention 的环形通信不同，Ulysses 在每个注意力层前后各做一次 All-to-All。

5. 激活重计算 (Activation Checkpointing)

通俗类比：这就像做数学题时不把所有中间步骤都写在草稿纸上，而是只记下几个关键步骤。当需要回头检查时（反向传播），再根据关键步骤快速重新推导出需要的中间结果。用少量计算时间换取了大量草稿纸（显存）空间。
常见误区：最大的误用是“过度检查点”，即在每个小操作后都设置检查点。这会导致在反向传播时重新计算过多，计算开销剧增，训练速度可能反而下降。通常只在Transformer层之间或每个残差块之间设置。
深入探讨：激活重计算（梯度检查点）通过丢弃前向传播中的部分激活值来节省显存，在反向传播需要时重新计算。以Transformer层为例，不重计算时，需要存储该层所有注意力分数和中间FFN结果。重计算后，只需存储该层的输入激活，在反向传播时重新运行该层的前向计算，显存节省比例可达70%-80%，计算时间增加约30%。
激活重计算 vs. 标准计算:
前向传播:
标准: [in] -> Layer -> [out, act]  (保存act)
重计算: [in] -> Layer -> [out]     (丢弃act)

反向传播:
标准: 从 [out] 和 [act] 计算梯度
重计算: 从 [in] 重新计算得到 [act]，再计算梯度
方法前向显存占用 (单层)反向计算时间增加适用场景
标准前向高 (存储所有激活)无显存充足，追求最快训练速度
全重计算最低 (仅输入/输出)高 (约100%)显存极度紧张
选择性重计算 (常用)中等 (检查点层输入)中等 (约30-50%)在显存和速度间平衡

5.1 基本原理

激活重计算通过丢弃前向传播的中间结果，在反向传播时重新计算来节省显存。

5.2 长上下文训练的特殊考虑

  • 注意力矩阵：是最大的显存消耗者，必须重计算
  • 选择性重计算：只重计算注意力矩阵，保留其他激活值
  • 与 SP 结合：SP 减少了每 GPU 的激活值大小，与激活重计算互补
激活重计算的显存-计算权衡:

无重计算: 显存 O(n²), 计算 1x
全重重算: 显存 O(n), 计算 ~1.33x (额外 33% 计算)
选择性重算: 显存 O(n), 计算 ~1.15x (只重算注意力)

6. 梯度累积策略

通俗类比：想买一个昂贵的物品（大Batch），但一次付不起全款（显存不够）。梯度累积就像分期付款：每次只支付一小部分（小Mini-Batch），连续支付几次（累积步数）后，才完成一次真正的购买（参数更新），且效果等同于一次付清。
常见误区：梯度累积会降低参数更新频率，可能影响模型收敛的动态特性。如果累积步数过大，相当于使用了更大的等效学习率，可能需要调整学习率调度策略。在实现时，务必确保在每个累积循环的最后才执行梯度清零和参数更新。
深入探讨：梯度累积通过多次前向-反向传播累积梯度，再用累积的平均梯度更新参数。设目标等效全局批次大小为B，单卡最大批次为b，累积步数为A，则B = b * A * GPU数。其显存占用主要由小Mini-Batch（b）决定。例如，使用b=1，A=4，8卡训练，等效全局批次大小为32，但每张卡的显存占用只相当于训练批次为1的模型。
梯度累积循环 (累积步数A=3):
Step1: Forward (b) -> Backward -> 梯度累加 (不清零，不更新)
Step2: Forward (b) -> Backward -> 梯度累加 (不清零，不更新)
Step3: Forward (b) -> Backward -> 梯度累加 (不清零，不更新)
Step4: 梯度平均 -> 参数更新 -> 梯度清零

每卡批次 (micro_batch)单次前向的数据量**直接影响显存** (主要瓶颈之一)影响单步计算时间

6.1 长上下文的梯度累积

由于显存限制，长上下文训练通常使用很小的批大小（1-4）。梯度累积可以模拟更大的批大小：

有效批大小 = 微批大小 × 梯度累积步数 × 数据并行度

例: 微批=2, 累积=16, DP=64 → 有效批 = 2048

6.2 长上下文的特殊考虑

  • 累积步数：长上下文需要更多累积步数（因为微批更小）
  • 梯度同步频率：可以减少同步频率以降低通信开销
  • 梯度裁剪：长序列的梯度方差大，需要更严格的梯度裁剪

7. Flash Attention 对长上下文训练的影响

通俗类比：传统注意力像在嘈杂的广场上，每个人都向所有人大声喊话，信息混乱且记录困难（需要巨大空间存储）。Flash Attention则像一位高效主持人，将人们分组讨论，并利用速记本（SRAM）在小组间快速传递和汇总核心观点，速度快且记录本（HBM）占用小。
常见误区：Flash Attention的主要优势是计算效率和显存优化，但它并不改变注意力计算的数学结果。一个常见误区是认为使用了Flash Attention就能无限扩展序列长度。实际上，它依然受限于GPU的总显存，只是将显存占用从O(N^2)降到了O(N)。
深入探讨：Flash Attention的核心是IO感知算法，通过分块（Tiling）计算，将Q, K, V矩阵分割成适合GPU SRAM的小块。它避免在HBM中实例化庞大的N×N注意力矩阵，而是在线计算softmax并输出结果。这使得训练时的显存占用与序列长度N成线性关系（O(N)），而非平方关系（O(N^2)）。对于4096长度的序列，可节省高达20倍的显存。
Flash Attention 分块计算示意:
1. 将 Q, K, V 分块加载到 SRAM
2. 在 SRAM 中计算: Q块 * K块^T -> 得到部分注意力分数
3. 在线更新 softmax 和输出结果块
4. 重复直到所有 K, V 块处理完毕

显存复杂度 (单层)O(N^2)O(N)

7.1 Flash Attention 的显存优化

Flash Attention 通过分块计算和重计算，避免存储完整的 n×n 注意力矩阵：

  • 显存：从 O(n²) 降到 O(n)
  • 计算：利用 GPU 的内存层次结构，减少 HBM 访问
  • 精度：使用在线 softmax，数值精度与标准注意力相同

7.2 Flash Attention + SP 的组合

Flash Attention 与 Sequence Parallelism 是互补的：

  • Flash Attention 减少单 GPU 的显存消耗
  • SP 将序列分布到多个 GPU
  • 两者结合可以训练超长序列（1M+）

8. 各方案的显存节省与通信开销对比

通俗类比：选择显存优化方案就像装修预算有限时分配资金。激活重计算是“花时间换空间”，梯度累积是“分批次付款”，序列并行是“多人分摊房租”，Flash Attention是“升级收纳工具”。每种方案都有不同的“成本”（通信开销、计算速度）和“收益”（显存节省）。
常见误区：没有一种方案是“银弹”。最大的错误是盲目叠加所有优化技术。这可能导致通信开销激增、计算流水线效率低下，训练速度反而比优化前更慢。必须根据硬件条件（显存、网络带宽）、模型规模和序列长度进行针对性选择和调优。
深入探讨：以训练一个7B模型、序列长度8192为例：仅靠激活重计算可节省约40%显存，计算开销增加30%。引入Flash Attention (IO感知)后，可将注意力模块的显存需求降低90%以上，计算速度提升2-4倍。当仍需更长序列时，再结合Ring Attention或Ulysses进行序列并行，但会引入额外的AllGather或All-to-All通信，其时间开销取决于网络带宽。
