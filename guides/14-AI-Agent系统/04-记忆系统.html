短期记忆、长期记忆、工作记忆——让Agent拥有"记忆"的关键技术

1. 背景与动机

记忆是将LLM从"无状态文本生成器"转变为"有状态Agent"的关键。根据2026年Du et al.的综述《Memory for Autonomous LLM Agents》，记忆系统的设计直接决定了Agent的适应性、持续性和个性化能力。

3. RAG作为记忆增强

RAG（Retrieval-Augmented Generation）是实现长期记忆最常用的技术。核心思想：将记忆存储在外部数据库中，需要时检索相关信息注入上下文。

3.1 RAG的关键：分块策略 (Chunking)

RAG的效果很大程度上取决于如何将长文档切分为小块。分块太大，检索精度低；分块太小，丢失上下文。

分块策略方法适用场景典型大小
固定大小按字符/token数切分通用文本500-1000 tokens
句子级按句子边界切分短文档、FAQ1-3句
段落级按段落边界切分文章、报告1段
语义级用Embedding相似度判断切分点主题变化明显的文档动态
递归分割先按大结构分，再递归细分结构化文档（代码、Markdown）多级
💡 分块最佳实践：(1) 块之间保留10-20%的重叠，避免上下文断裂 (2) 为每个块附加元数据（来源、标题、时间）便于过滤 (3) 对于代码，按函数/类切分而非按行 (4) 典型生产配置：chunk_size=800, chunk_overlap=200。

3.3 Chunking的数学分析

Chunk大小的选择是一个精度-召回率权衡：

检索精度 ∝ 1/chunk_size (越小越精确)
上下文完整性 ∝ chunk_size (越大越完整)
最优chunk_size = argmax [α·precision(s) + (1-α)·completeness(s)]

实际经验数据：

chunk_size检索精度上下文完整性推荐场景
200 tokens最高差（信息碎片化）FAQ、短问答
500 tokens高中等技术文档、知识库
800 tokens中高好通用推荐
1500 tokens中等很好长文档、报告
3000+ tokens低最好不推荐（检索精度太低）
# LangChain递归分块实现
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,           # 每块最大800字符
    chunk_overlap=200,        # 重叠200字符
    separators=["\n\n", "\n", "。", "！", "？", ".", " "],  # 优先按段落→句子→词分割
    length_function=len,
)

chunks = splitter.split_text(document)
# 结果: 每块<=800字符，相邻块有200字符重叠

3.2 相似度计算的数学基础

RAG检索的核心是计算query和文档的相似度。三种主流方法：

cosine_sim(A, B) = (A · B) / (|A| × |B|) = Σ(aᵢ×bᵢ) / (√Σaᵢ² × √Σbᵢ²)

Cosine相似度：衡量两个向量的方向一致性，值域[-1, 1]。对向量长度不敏感——两段意思相同但长短不同的文本，cosine相似度仍然很高。这是RAG最常用的相似度指标。

BM25(D, Q) = Σ IDF(qᵢ) · [f(qᵢ,D) · (k₁+1)] / [f(qᵢ,D) + k₁ · (1 - b + b·|D|/avgdl)]

BM25公式详解：

  • f(qᵢ,D)：词qᵢ在文档D中的词频（TF）
  • IDF(qᵢ)：逆文档频率 = log[(N-n(qᵢ)+0.5)/(n(qᵢ)+0.5)+1]，罕见词权重高
  • k₁（通常1.2-2.0）：词频饱和参数，越大越信任高频词
  • b（通常0.75）：文档长度归一化参数，越长文档惩罚越大
  • avgdl：语料库平均文档长度
💡 面试回答："BM25是TF-IDF的改进版。三个关键改进：(1) 词频饱和——f(q,D)越大收益递减，由k₁控制 (2) 文档长度归一化——长文档天然词频高，需要惩罚 (3) IDF用BM25变体而非log(N/n)。BM25对精确关键词匹配效果好，但不理解语义——'开心'和'高兴'对BM25是完全不同的词。"
RRF_score(d) = Σ 1/(k + rank_i(d)) 其中 k=60

RRF（Reciprocal Rank Fusion）：将Dense和Sparse检索的排名融合。k=60是经验值——如果一个文档在Dense排第1、Sparse排第10，RRF=1/61+1/70=0.0307。RRF的优势是不需要归一化不同检索器的分数，直接用排名。

4. 记忆管理策略

4.1 记忆编码

如何将Agent的经历转化为可存储的记忆？

  • 直接存储：将对话/行动原文存入向量DB（简单但冗余）
  • 摘要存储：用LLM生成摘要后再存储（压缩但可能丢失细节）
  • 结构化存储：提取实体和关系，存入知识图谱（精确但需要额外处理）
  • 混合存储：原文+摘要+结构化信息同时存储（最全面但复杂）

4.2 记忆检索

检索方式原理优势劣势
语义检索向量相似度（cosine）理解语义，支持模糊匹配可能漏掉关键词匹配
关键词检索BM25/TF-IDF精确匹配，速度快不理解同义词
混合检索语义+关键词融合兼顾两者优势实现复杂度高
时间衰减越近的记忆权重越高符合"遗忘曲线"可能忽略重要的旧记忆

4.3 记忆遗忘的数学模型

Agent的记忆管理可以借鉴认知科学的遗忘曲线（Ebbinghaus Forgetting Curve）：

R(t) = e^(-t/S) 其中 R=保留概率, t=时间, S=记忆强度

直觉：记忆强度S越大，遗忘越慢。每次成功检索到该记忆，S增加（类似"复习"效果）。

在Agent中的实现：

importance(t) = base_importance × e^(-λt) + access_boost × Σ e^(-λ(t-tᵢ))

其中：

  • base_importance：记忆创建时LLM评估的基础重要性（1-10分）
  • λ：衰减速率（可调参数，越大遗忘越快）
  • access_boost：每次被检索到的加分
  • tᵢ：第i次被检索的时间
# 记忆重要性计算实现
import math
import time

class MemoryManager:
    def __init__(self, decay_rate=0.01, access_boost=0.5):
        self.decay_rate = decay_rate
        self.access_boost = access_boost

    def compute_importance(self, memory):
        t = time.time() - memory.created_at
        base = memory.base_importance * math.exp(-self.decay_rate * t)
        access = sum(
            self.access_boost * math.exp(-self.decay_rate * (time.time() - t_i))
            for t_i in memory.access_times
        )
        return base + access

    def should_forget(self, memory, threshold=0.5):
        return self.compute_importance(memory) < threshold

    def periodic_cleanup(self, memories):
        return [m for m in memories if not self.should_forget(m)]

4.4 记忆冲突解决

当新信息与旧信息矛盾时，需要冲突解决策略：

策略方法适用场景
时间优先新信息覆盖旧信息时效性信息（价格、状态）
来源优先可信来源覆盖不可信来源多源信息（官方 vs 论坛）
频率优先出现次数多的信息覆盖少数共识性信息
保留两者两条记忆都保留，标注矛盾需要人工判断的场景
🔬 记忆研究的前沿：2026年Du et al.的综述《Memory for Autonomous LLM Agents》提出了Agent记忆的三维分类法：(1) 时间维度——短期(会话内)、中期(跨会话)、长期(永久) (2) 表示维度——文本、向量、知识图谱、结构化数据 (3) 控制维度——被动存储(全量保存)、主动管理(选择性保存)、自适应(根据重要性动态调整)。最先进的系统（如Generative Agents）同时使用三个维度的组合。

5. Grounding：事实锚定与幻觉防控

Grounding（事实锚定）是Agent记忆系统的核心挑战——如何确保Agent的回答基于真实数据而非LLM的"想象"？

┌─────────────────────────────────────────────────────────────────┐ │ Grounding 机制 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 无Grounding (幻觉风险高): │ │ 用户: "我们上次会议讨论了什么?" │ │ Agent: [凭空编造] "讨论了Q3预算和招聘计划" ✗ │ │ │ │ 有Grounding (基于真实记忆): │ │ 用户: "我们上次会议讨论了什么?" │ │ Agent: [检索记忆] → 找到会议记录 │ │ → "6月20日会议讨论了: (1)产品发布日期 (2)用户反馈处理" ✓ │ │ │ │ Grounding的三层防线: │ │ 1. 检索层: 从记忆库检索最相关的证据 │ │ 2. 引用层: 回答必须引用具体来源 │ │ 3. 验证层: 用另一个LLM检查回答是否有证据支持 │ │ │ └─────────────────────────────────────────────────────────────────┘
💡 Grounding实践：在Agent的system prompt中加入："只基于检索到的信息回答。如果信息不足，明确说'我没有找到相关信息'，不要编造。" 这条简单规则可以减少60-80%的幻觉。更高级的方案：使用Guardrails AI的FactChecker验证器自动检查回答是否有证据支持。

# 使用LangChain构建带记忆的Agent
from langchain.memory import ConversationSummaryBufferMemory
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

# 1. 短期记忆: 对话摘要缓冲
memory = ConversationSummaryBufferMemory(
    llm=ChatOpenAI(model="gpt-4"),
    max_token_limit=2000,  # 超过2000 tokens时自动摘要
    return_messages=True
)

# 2. 长期记忆: 向量数据库
embeddings = OpenAIEmbeddings()
vectorstore = Chroma(embedding_function=embeddings, persist_directory="./memory")
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# 3. 存储新记忆
vectorstore.add_texts(
    texts=["用户喜欢Python，偏好简洁风格"],
    metadatas=[{"type": "preference", "timestamp": "2026-06-24"}]
)

# 4. 检索相关记忆
relevant = retriever.invoke("用户喜欢什么编程语言?")
💡 实践建议：开始时用最简单的方案——直接将对话存入Chroma。只有当简单方案出现问题（如检索不准、存储太大）时，才考虑摘要、重要性评分等高级策略。
⚠️ 常见陷阱：记忆系统的最大挑战不是技术实现，而是记忆质量。存入低质量、冗余、过时的记忆会导致检索结果差，进而影响Agent的决策。定期清理和更新记忆比增加更多记忆更重要。

💡 参考答案与详解

解题思路: 构建Agent记忆系统需要理解三层记忆架构：短期记忆（对话历史）、工作记忆（任务状态）、长期记忆（向量DB）。练习中演示了如何用LangChain实现短期对话记忆和基于Chroma的长期语义记忆。

# 完整可运行的记忆系统示例
from langchain.memory import ConversationSummaryBufferMemory
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

# ── Part 1: 短期记忆 (对话摘要缓冲) ──
memory = ConversationSummaryBufferMemory(
    llm=ChatOpenAI(model="gpt-4o-mini"),
    max_token_limit=200,        # 超过200 tokens自动触发摘要
    return_messages=True
)

# 添加多轮对话，观察摘要触发
conversations = [
    ("我叫张三", "你好张三!"),
    ("我喜欢Python和AI", "很好的兴趣!Python和AI是当今最热门的技术领域。"),
    ("我在北京工作", "了解，北京是中国AI产业的中心之一。"),
    ("我之前说了什么?", ""),  # 触发记忆检索
]
for inp, out in conversations:
    if out:
        memory.save_context({"input": inp}, {"output": out})

# 查看记忆状态 (会看到对话被压缩为摘要)
vars = memory.load_memory_variables({})
print("短期记忆内容:", vars["history"])

# ── Part 2: 长期记忆 (向量数据库) ──
embeddings = OpenAIEmbeddings()
vectorstore = Chroma(embedding_function=embeddings, persist_directory="./my_memory")

# 存储多条记忆 (带元数据)
memories = [
    ("张三喜欢Python，讨厌Java", {"type": "preference", "importance": 8}),
    ("张三在北京中关村工作", {"type": "location", "importance": 7}),
    ("张三的项目使用FastAPI和React", {"type": "project", "importance": 9}),
    ("张三每周五有团队会议", {"type": "schedule", "importance": 6}),
]
for text, meta in memories:
    vectorstore.add_texts(texts=[text], metadatas=[meta])

# 语义检索测试
queries = ["张三用什么技术栈?", "张三喜欢什么?", "张三在哪上班?"]
for q in queries:
    results = vectorstore.similarity_search(q, k=2)
    print(f"\n查询: {q}")
    for r in results:
        print(f"  → {r.page_content} (类型: {r.metadata['type']})")

# 预期结果:
# "技术栈" → 命中 project 记忆 (FastAPI+React)
# "喜欢"   → 命中 preference 记忆 (Python)
# "在哪"   → 命中 location 记忆 (北京中关村)

关键知识点: (1) 短期记忆用ConversationSummaryBufferMemory实现，超长对话自动摘要压缩 (2) 长期记忆用向量数据库(Chroma)存储，支持语义检索 (3) 元数据(type/importance)支持后续的过滤和重要性排序 (4) 生产中应实现遗忘机制：基于时间衰减和访问频率自动清理低价值记忆 (5) chunk_size=200适合FAQ场景，通用场景推荐800。
