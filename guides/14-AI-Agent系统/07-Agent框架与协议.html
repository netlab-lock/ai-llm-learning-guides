⚙️ 模块07：Agent框架与协议

LangChain、CrewAI、AutoGPT、MetaGPT、AutoGen——主流框架深度对比

1. 背景与动机

2023年以来，Agent框架如雨后春笋般涌现。选择合适的框架是Agent开发的第一个关键决策。本模块将深入分析5个主流框架的架构设计、适用场景和优劣势。

3. LangChain / LangGraph

LangChain是最早也是最流行的LLM应用框架（2022年10月发布），LangGraph是其Agent编排引擎（2024年）。

3.1 核心架构

组件说明
LangChain Core基础组件：LLM接口、Prompt模板、Output Parser、Tool定义
LangGraph状态图编排：定义Agent的控制流为有向图，支持条件分支、循环、并行
LangSmith可观测性平台：追踪Agent的每一步决策、调试、评估
LangServe部署工具：将LangChain应用部署为REST API

3.2 LangChain vs LangGraph

维度LangChainLangGraph
抽象级别高（Chain/Agent封装）低（状态图/节点/边）
控制流预定义（Sequential/Conditional）完全自定义（任意图结构）
状态管理隐式（通过Memory）显式（StateGraph状态）
持久化无内建内建checkpoint（可恢复）
人机协作需要手动实现内建interrupt_before/after
适用场景简单链式调用复杂Agent流程
💡 选型建议：2026年的最佳实践是：直接用LangGraph，跳过LangChain的高层封装。LangGraph更底层、更灵活，且与LangChain生态完全兼容。LangChain的Chain/Agent抽象在复杂场景下反而成为束缚。

3.3 LangGraph的设计哲学

# LangGraph: 将Agent建模为状态图
from langgraph.graph import StateGraph, MessagesState

# 定义节点（Agent的各个步骤）
def researcher(state):
    # 研究逻辑
    return {"messages": [result]}

def writer(state):
    # 写作逻辑
    return {"messages": [result]}

# 构建状态图
graph = StateGraph(MessagesState)
graph.add_node("researcher", researcher)
graph.add_node("writer", writer)
graph.add_edge("researcher", "writer")
graph.set_entry_point("researcher")

# 编译并执行
app = graph.compile()
result = app.invoke({"messages": ["写一篇AI Agent的博客"]})
💡 适用场景：LangGraph适合需要精细控制Agent流程的场景。如果你需要自定义复杂的控制流（条件分支、重试逻辑、并行执行），LangGraph是最佳选择。

4. CrewAI

CrewAI（2023年）是一个角色驱动的多Agent框架，将Agent建模为有角色、目标和背景的"团队成员"。

4.1 核心概念

概念说明类比
Agent有角色、目标、工具的独立实体团队成员
TaskAgent需要完成的具体任务工作任务单
CrewAgent和Task的组合，执行工作流项目团队
Process任务执行策略（顺序/层级）项目管理方式

4.2 CrewAI核心实现

# CrewAI完整示例: 自动研究+写作团队
from crewai import Agent, Task, Crew, Process

# 定义Agent (角色+目标+背景+工具)
researcher = Agent(
    role="高级研究分析师",
    goal="搜索并分析{topic}的最新技术进展",
    backstory="你是一位有10年经验的技术研究员，擅长从大量文献中提取关键信息。",
    tools=[search_tool, arxiv_tool],
    llm="gpt-4o",
    verbose=True
)

writer = Agent(
    role="技术博客写手",
    goal="将研究结果写成易懂的技术文章",
    backstory="你是一位擅长将复杂技术用简单语言解释的写手。",
    llm="gpt-4o"
)

# 定义Task (描述+期望输出+负责Agent)
research_task = Task(
    description="搜索{topic}的最新进展，整理关键发现",
    expected_output="一份包含5-10个关键发现的结构化报告",
    agent=researcher
)

write_task = Task(
    description="基于研究结果撰写技术博客",
    expected_output="一篇1500字的技术博客，包含代码示例",
    agent=writer,
    context=[research_task]  # 依赖研究任务的结果
)

# 创建Crew并执行
crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, write_task],
    process=Process.sequential,  # 顺序执行
    verbose=True
)
result = crew.kickoff(inputs={{"topic": "Agentic RAG"}})

4.3 CrewAI vs LangGraph

维度CrewAILangGraph
抽象级别高（角色+任务）低（状态图+节点）
控制流预定义（顺序/层级）完全自定义
学习曲线10分钟上手需要理解状态图概念
灵活性有限（受Process约束）无限（任意图结构）
适合场景角色明确的团队任务复杂自定义流程
💡 适用场景：CrewAI适合角色分工明确的多Agent场景。如果你的任务可以自然地分解为不同角色的工作（如PM+开发+测试），CrewAI的抽象非常直观。

5. AutoGPT

AutoGPT（2023年3月）是第一个广泛传播的自主Agent，目标是"给定一个目标，自主完成所有步骤"。

5.1 AutoGPT Platform (2025)

2025年AutoGPT进化为可视化Agent构建平台，提供：

  • 可视化编排器：拖拽式构建Agent工作流
  • 块系统：预构建的功能块（LLM、工具、条件判断）
  • 市场：社区分享的Agent模板
  • 监控面板：实时查看Agent执行状态

5.2 AutoGPT的演进

版本时间特点
AutoGPT v12023.03命令行Agent，目标驱动，自主循环
AutoGPT v22024引入插件系统，支持更多工具
AutoGPT Platform2025可视化构建器，块系统，Agent市场
💡 适用场景：AutoGPT Platform适合非开发者或快速原型。可视化界面降低了Agent开发的门槛。

6. MetaGPT

MetaGPT（2023年8月）将软件工程的标准化流程引入多Agent系统，模拟了一个完整的软件开发团队。

6.1 核心设计

┌─────────────────────────────────────────────────────────────────┐ │ MetaGPT 工作流程 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 产品经理 ──→ 架构师 ──→ 项目经理 ──→ 工程师 ──→ QA │ │ │ │ │ │ │ │ │ 需求文档 系统设计 任务拆分 代码实现 测试报告 │ │ │ │ │ │ │ │ │ └──────────┴──────────┴──────────┴──────────┘ │ │ 共享的结构化文档 │ │ │ │ 关键创新: SOP (Standard Operating Procedure) │ │ 每个角色遵循标准流程，输出结构化文档 │ │ 下游角色基于上游文档工作，减少幻觉 │ │ │ └─────────────────────────────────────────────────────────────────┘
💡 适用场景：MetaGPT最适合软件开发场景。它的SOP机制确保了输出质量，但灵活性不如LangGraph。

7. AutoGen (Microsoft)

AutoGen（2023年10月，Microsoft）是一个对话驱动的多Agent框架，核心思想是通过Agent之间的对话来完成任务。

7.1 核心概念

概念说明
ConversableAgent能参与对话的Agent，可以发送和接收消息
GroupChat多个Agent的群聊，由GroupChatManager管理
HumanProxy代表人类参与对话的Agent
代码执行内建的代码执行环境（Docker沙箱）

7.2 AutoGen的独特优势

  • 代码执行内建：AutoGen内建Docker沙箱，Agent可以安全地执行生成的代码
  • 人类代理：HumanProxy类让人类自然地加入Agent对话
  • 群聊管理：GroupChatManager自动管理多Agent对话的发言顺序
  • 对话缓存：支持对话序列化和恢复，适合长时间运行的任务
💡 适用场景：AutoGen适合需要人机协作的场景。它的对话式交互非常自然，支持人类在关键节点介入。

🔬 框架演进史：Agent框架经历了三代演进。第一代(2022-2023): LangChain独占鳌头，提供了LLM应用的基础组件（Prompt、Chain、Agent），但架构过于庞大、抽象层次混乱。第二代(2023-2024): 专业化分工——LangGraph专注编排、CrewAI专注多Agent、AutoGen专注对话。第三代(2025-2026): 标准化与互操作——MCP统一工具接口、A2A统一Agent通信，框架不再是封闭生态。关键教训：不要把框架当"信仰"——它们是工具，不是宗教。选择最适合当前需求的，保持可迁移性。

9. 选型决策树

┌─────────────────────────────────────────────────────────────────┐ │ Agent框架选型决策树 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 你的需求是什么? │ │ │ │ │ ├─→ 需要精细控制Agent流程? │ │ │ └─→ 是 → LangGraph │ │ │ │ │ ├─→ 角色分工明确的多Agent? │ │ │ └─→ 是 → CrewAI │ │ │ │ │ ├─→ 软件开发场景? │ │ │ └─→ 是 → MetaGPT │ │ │ │ │ ├─→ 需要人机协作? │ │ │ └─→ 是 → AutoGen │ │ │ │ │ ├─→ 快速原型/非开发者? │ │ │ └─→ 是 → AutoGPT Platform │ │ │ │ │ └─→ 不确定? │ │ └─→ 从CrewAI开始 (最易上手) │ │ 或LangGraph (最灵活) │ │ │ └─────────────────────────────────────────────────────────────────┘
⚠️ 选型警告：没有"最好"的框架，只有"最合适"的框架。不要被Star数或热度误导——LangGraph虽然最灵活但也最难学，CrewAI虽然最易用但也最受限。根据你的具体需求和团队能力选择。

💡 参考答案与详解

解题思路: 本练习展示CrewAI的最小可用示例。与模块06的练习相比，这个更简洁——两个Agent、两个Task、顺序执行。重点理解Agent的role/goal定义如何影响LLM的行为。

# 完整可运行: CrewAI研究团队
# pip install crewai crewai-tools

from crewai import Agent, Task, Crew, Process

# 定义分析师Agent (专注于数据收集和分析)
analyst = Agent(
    role="市场数据分析师",
    goal="收集和分析AI Agent市场的最新数据和趋势",
    backstory="""你是一位专注于AI行业的市场分析师，拥有丰富的
    数据收集和分析经验。你善于从技术博客、论文和产品发布中
    提取关键市场信号。""",
    llm="gpt-4o",
    verbose=True
)

# 定义报告撰写者Agent (专注于信息整合和表达)
reporter = Agent(
    role="报告撰写专家",
    goal="将分析结果写成清晰、结构化的市场分析报告",
    backstory="""你是一位商业报告撰写专家，善于将复杂的数据
    转化为管理层可理解的洞察。你的报告逻辑清晰、结论明确。""",
    llm="gpt-4o",
    verbose=True
)

# 定义任务链
t1 = Task(
    description="""分析AI Agent市场趋势:
    1. 主要玩家(OpenAI/Anthropic/Google/开源)的产品和策略
    2. 关键技术趋势(MCP/A2A/多模态Agent)
    3. 市场规模和增长预测""",
    expected_output="包含数据支撑的市场分析要点列表",
    agent=analyst
)

t2 = Task(
    description="基于分析结果撰写一份500字的市场分析报告摘要",
    expected_output="结构化报告，含市场概况、趋势分析、投资建议",
    agent=reporter,
    context=[t1]  # 自动获取analyst的分析结果
)

# 创建团队并执行
crew = Crew(
    agents=[analyst, reporter],
    tasks=[t1, t2],
    process=Process.sequential,
    verbose=True
)

result = crew.kickoff()
print(result)

关键知识点: (1) CrewAI的Agent角色定义(role/goal/backstory)直接决定了LLM的system prompt质量——越详细，Agent表现越好 (2) context=[t1]让reporter自动获取analyst的输出，无需手动传递 (3) 与LangGraph相比，CrewAI的优势是上手快(10分钟)、劣势是灵活性低(受Process约束) (4) 生产环境建议加上max_rpm限流和max_iter防无限循环。

小结
