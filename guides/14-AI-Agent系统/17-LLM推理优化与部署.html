2. 模型量化 面试高频

量化是用更低精度的数值表示模型权重，减少内存和计算需求。

2.1 量化方法对比

方法精度内存节省质量损失特点
FP3232bit基准无训练精度
FP16/BF1616bit2x极小推理标准精度
GPTQ4bit8x小训练后量化，需要校准数据
AWQ4bit8x更小保护重要权重通道
GGUF2-8bit4-16x因精度而异llama.cpp格式，CPU友好
INT88bit4x很小SmoothQuant等

2.2 量化公式

INT4量化: q = round(x / scale) scale = (max - min) / 15
反量化: x ≈ q × scale 误差 = |x - x̂|
💡 面试回答："量化的核心是用更少的bit表示权重。GPTQ通过逐层量化+误差补偿保持精度。AWQ发现只有1%的'显著权重'对精度影响大，保护这些通道后4bit量化几乎无损。GGUF是llama.cpp的格式，支持CPU推理，适合边缘部署。"

2.3 量化对Agent的影响

场景推荐量化原因
GPU服务器部署AWQ 4bit精度最高，GPU加速好

4. 推测解码 (Speculative Decoding)

用一个小模型(draft)快速生成多个候选token，再用大模型(target)一次性验证。

┌─────────────────────────────────────────────────────────────────┐ │ 推测解码流程 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 传统自回归: T1 → T2 → T3 → T4 → T5 (5次大模型调用) │ │ │ │ 推测解码: │ │ 1. 小模型快速生成: T1' T2' T3' T4' T5' (5次小模型调用, 快) │ │ 2. 大模型一次性验证: [T1'✓ T2'✓ T3'✗ T4'? T5'?] │ │ 3. 接受前2个, 从T3开始重新生成 │ │ │ │ 效果: 大模型调用次数从5次降到2-3次 │ │ 加速比: 1.5-3倍 (取决于draft模型的"猜中率") │ │ │ │ 关键: draft模型要足够快+足够准 │ │ 常见组合: 7B(draft) + 70B(target) │ │ │ └─────────────────────────────────────────────────────────────────┘

5. 连续批处理 (Continuous Batching)

传统批处理等所有请求完成才处理下一批。连续批处理在请求完成时立即插入新请求。

传统批处理吞吐: requests/time = batch_size / max_latency
连续批处理吞吐: requests/time ≈ batch_size / avg_latency (显著更高)

✅ 参考答案与详解
  1. 核心原理: Decode阶段每步只生成1个新token，计算量很小（一次矩阵向量乘法），但需要读取整个KV Cache（与所有历史token的K、V相关）。对于70B模型、128K上下文，KV Cache可达数十GB，每步都需要从HBM（高带宽显存）读取这些数据。由于计算量小而数据读取量大，GPU的计算单元大部分时间在等待数据到达——这就是"内存带宽瓶颈"。具体来说，70B模型decode的计算强度(FLOPs/Byte)约为2-3，而A100的算力/带宽比约为2048/1555≈1.3，计算强度远低于算力/带宽比，说明瓶颈在带宽。对Agent架构设计的影响：(1)增大batch size可以提高计算利用率（多个请求共享模型权重读取）→ 需要连续批处理 (2)减少KV Cache大小可以直接提升速度 → 需要GQA/MQA减少KV头数 (3)decode阶段无法通过增加算力加速 → 需要模型压缩或推测解码 (4)多轮对话中历史KV Cache的管理直接影响延迟 → 需要KV Cache复用和压缩策略。
    关键要点: (1) Decode计算量小但数据读取量大 (2) GPU算力利用率低 (3) 连续批处理提高利用率 (4) KV Cache大小直接影响延迟
    实际案例: 一个70B Agent处理128K上下文的多轮对话，decode延迟约500ms/token。通过GQA将KV头数从64减到8，KV Cache缩小8倍，decode延迟降到约80ms/token，提升6倍。
  2. 核心原理: GPTQ(Generative Pre-trained Quantization)：逐层量化，用Hessian矩阵的逆做误差补偿，4bit量化时精度损失较小。核心思想是量化一个权重时，根据它对其他权重的影响来补偿误差。AWQ(Activation-aware Weight Quantization)：发现只有约1%的"显著权重"(salient weights)对精度影响大，量化前先将这些通道乘以缩放因子保护起来，再量化其余99%的通道。AWQ精度通常优于GPTQ。GGUF(GPT-Generated Unified Format)：llama.cpp的格式，支持2-8bit多种精度，核心特点是CPU友好——针对x86/ARM的缓存层次结构优化，支持混合精度（重要层用高精度，其余用低精度）。选择Function Calling Agent的量化方案：首选AWQ 4bit。理由：(1) Function Calling对精度敏感——JSON格式生成需要精确的token概率，低精度会导致格式错误 (2) AWQ保护显著权重的策略对保持格式生成能力最有效 (3) GPTQ虽然精度也不错，但AWQ在4bit下通常更好 (4) GGUF适合CPU部署，但Function Calling Agent通常需要GPU以保证低延迟 (5) 不建议低于4bit，因为2-3bit会严重破坏JSON生成能力。
    关键要点: (1) GPTQ用Hessian补偿 (2) AWQ保护显著权重 (3) GGUF是CPU格式 (4) Function Calling首选AWQ 4bit (5) 不低于4bit
    实际案例: 在Llama-3-70B的Function Calling任务中，FP16的JSON格式正确率为98.2%，AWQ 4bit为97.1%，GPTQ 4bit为95.8%，GGUF Q3为89.3%。AWQ 4bit在精度和效率间取得最佳平衡。
  3. 核心原理: 影响猜中率的因素：(1)任务类型——结构化输出(代码、JSON)猜中率低(30-50%)，自然语言猜中率高(70-90%) (2)模型相似度——draft和target模型越相似(同系列不同大小)，猜中率越高 (3)温度参数——低温度(确定性高)时猜中率高，高温度(多样性)时猜中率低 (4)上下文复杂度——简单问答猜中率高，复杂推理猜中率低。Agent场景下的适用性差异：工具调用参数生成(JSON格式)猜中率低，因为JSON的精确格式(引号、括号、逗号)很难被小模型准确预测，推测解码加速比可能只有1.1-1.3倍。而自然语言回答(如Agent的思考过程Thought)猜中率高，加速比可达1.5-2.5倍。因此，混合策略更优：对Thought部分用推测解码，对Action部分用普通解码。或者使用"自适应推测解码"——检测当前生成的是结构化还是非结构化内容，动态开关推测解码。
    关键要点: (1) 结构化输出猜中率低 (2) 模型相似度影响猜中率 (3) 工具调用参数不适合推测解码 (4) 自然语言部分适合推测解码 (5) 混合策略最优
    实际案例: 在CodeLlama-70B + CodeLlama-7B的组合中，自然语言注释的猜中率85%(加速2.1x)，Python代码的猜中率55%(加速1.4x)，JSON参数的猜中率仅35%(加速1.1x)。
  4. 核心原理: PagedAttention的实现原理：(1)将KV Cache分割为固定大小的block(如16个token一组)，类似操作系统的内存页 (2)每个序列维护一个block table，记录逻辑页到物理页的映射 (3)按需分配物理block——只有当序列长度增长时才分配新block (4)物理block不需要连续，通过block table寻址 (5)不同序列可以共享相同的物理block(如共享system prompt的KV Cache)。相比传统方式（为每个序列预分配最大长度的连续内存），PagedAttention消除了内部碎片(预分配但未使用的空间)和外部碎片(不同序列间的无法利用的空闲空间)。对Agent系统特别重要的原因：(1)Agent的多轮对话导致序列长度动态变化大——传统方式按最大长度预分配，浪费严重 (2)Agent的system prompt(工具描述+角色设定)通常很长(2000-5000 tokens)且跨请求共享——PagedAttention的共享机制让这些KV Cache只需计算一次 (3)Agent的batch中不同请求的对话轮数差异大——短对话不需要为长对话预留空间 (4)Agent的工具返回结果长度不可预测——按需分配避免了结果过长导致的OOM。
    关键要点: (1) 分页管理消除内存碎片 (2) 按需分配避免预分配浪费 (3) 共享机制适合固定前缀 (4) Agent序列长度动态变化大 (5) System prompt共享节省计算
    实际案例: 一个Agent服务同时处理100个请求，每个请求的system prompt为3000 tokens。传统方式需要为每个请求预分配128K空间(约50GB)，PagedAttention只需为system prompt分配共享block(约120MB) + 每个请求实际使用的空间(约8GB)，内存节省约80%。

▶显示答案
参考答案与详解

解题思路: 推理性能分析器需要基于Roofline模型估算延迟。Prefill是计算密集(与input_tokens×参数成正比)，Decode是内存密集(每步读取整个KV Cache)。KV Cache大小=2×batch×seq_len×n_layers×n_kv_heads×head_dim×bytes_per_element。推测解码加速比=1/((1-α)+α×draft_ratio)，α为猜中率，draft_ratio为小模型与大模型的延迟比。

class InferenceProfiler:
    """LLM推理性能分析器"""
    
    def __init__(self, model_config):
        """
        model_config示例:
        {
            "params": 7e9,        # 参数量
            "layers": 32,          # 层数
            "hidden_dim": 4096,    # 隐藏维度
            "num_heads": 32,       # 注意力头数
            "num_kv_heads": 8,     # KV头数(GQA)
            "head_dim": 128,       # 每头维度
            "max_seq_len": 128000, # 最大上下文
            "gpu_flops": 312e12,   # GPU算力(FLOPS) A100
            "gpu_bw": 1555e9,      # GPU带宽(bytes/s) A100
        }
        """
        self.cfg = model_config
        # 量化对应的bytes per element
        self.quant_bytes = {
            "fp32": 4, "fp16": 2, "bf16": 2,
            "int8": 1, "int4": 0.5, "int3": 0.375,
        }
    
    def compute_kv_cache_size(self, batch_size, seq_len, use_gqa=True):
        """
        计算KV Cache大小(bytes)
        KV Cache = 2 × batch × seq_len × layers × kv_heads × head_dim × bytes
        2是因为有K和V两个矩阵
        """
        n_kv_heads = self.cfg["num_kv_heads"] if use_gqa else self.cfg["num_heads"]
        bytes_per_elem = self.quant_bytes.get("fp16", 2)  # KV Cache通常用FP16
        
        kv_size = (2 * batch_size * seq_len * self.cfg["layers"] 
                   * n_kv_heads * self.cfg["head_dim"] * bytes_per_elem)
        return kv_size
    
    def estimate_latency(self, input_tokens, output_tokens, quant="fp16"):
        """
        估算推理延迟(ms)
        Prefill: 计算密集 ≈ 2 × params × input_tokens / gpu_flops
        Decode: 内存密集 ≈ (2 × params × output_tokens) / gpu_bw + kv_cache读取
        """
        params = self.cfg["params"]
        flops = self.cfg["gpu_flops"]
        bw = self.cfg["gpu_bw"]
        bytes_per_param = self.quant_bytes.get(quant, 2)
        
        # Prefill延迟 (计算密集)
        # 每个token需要约2×params次FLOPs(矩阵乘法)
        prefill_flops = 2 * params * input_tokens
        prefill_time_s = prefill_flops / flops
        # 量化加速: int4比fp16快约2倍(假设)
        quant_speedup = {  "fp32": 1, "fp16": 1, "bf16": 1, "int8": 1.5, "int4": 2, "int3": 2.5}
        prefill_time_s /= quant_speedup.get(quant, 1)
        
        # Decode延迟 (内存密集)
        # 每步需要读取整个模型权重 + KV Cache
        model_read_time = (params * bytes_per_param) / bw
        # KV Cache读取时间(随已生成长度增长)
        avg_kv_size = self.compute_kv_cache_size(1, input_tokens + output_tokens // 2)
        kv_read_time = avg_kv_size / bw
        decode_per_step = model_read_time + kv_read_time
        decode_time_s = decode_per_step * output_tokens
        
        return {
            "prefill_ms": round(prefill_time_s * 1000, 2),
            "decode_per_step_ms": round(decode_per_step * 1000, 2),
            "decode_total_ms": round(decode_time_s * 1000, 2),
            "total_ms": round((prefill_time_s + decode_time_s) * 1000, 2),
            "quant": quant
        }
    
    def speculative_decoding_speedup(self, draft_accept_rate, draft_ratio=0.3):
        """
        计算推测解码加速比
        draft_accept_rate: 小模型猜中率(0~1)
        draft_ratio: 小模型与大模型的延迟比(小模型快多少)
        
        公式推导:
        - 传统: 每步需要1次大模型调用
        - 推测: 每轮生成k个token需要1次小模型(k步) + 1次大模型(验证)
        - 猜中α×k个token, 浪费(1-α)×k步小模型计算
        - 加速比 = k / (k×draft_ratio + 1) 当α=1时
        - 更精确: 1 / ((1-α) + α×draft_ratio + 1/k) ≈ 1/((1-α) + α×draft_ratio)
        """
        alpha = draft_accept_rate
        # 简化公式: 每轮尝试生成k个token
        k = 5  # 典型draft步数
        
        # 每轮实际接受的token数
        expected_accept = alpha * k
        # 每轮时间 = k步小模型 + 1步大模型验证
        time_per_round = k * draft_ratio + 1
        # 传统方式: expected_accept步 × 1大模型时间
        traditional_time = expected_accept * 1
        # 加速比
        speedup = traditional_time / time_per_round if time_per_round > 0 else 1
        
        return {
            "speedup": round(speedup, 2),
            "accept_rate": alpha,
            "tokens_per_round": round(expected_accept, 1),
            "efficiency": round(expected_accept / k * 100, 1)  # 计算效率%
        }
    
    def memory_breakdown(self, batch_size, seq_len, quant="fp16"):
        """分析内存使用构成"""
        bytes_per_param = self.quant_bytes.get(quant, 2)
        
        # 模型权重
        model_memory = self.cfg["params"] * bytes_per_param
        
        # KV Cache
        kv_memory = self.compute_kv_cache_size(batch_size, seq_len)
        
        # 激活内存(粗略估算)
        activation_memory = (batch_size * seq_len * self.cfg["hidden_dim"] 
                            * 4 * 2)  # 4是bf16, 2是前后向
        
        total = model_memory + kv_memory + activation_memory
        
        return {
            "model_weights_gb": round(model_memory / 1e9, 2),
            "kv_cache_gb": round(kv_memory / 1e9, 2),
            "activation_gb": round(activation_memory / 1e9, 2),
            "total_gb": round(total / 1e9, 2),
            "kv_cache_percent": round(kv_memory / total * 100, 1),
            "quant": quant
        }
    
    def compare_quantization(self, input_tokens=2000, output_tokens=500):
        """对比不同量化方案"""
        results = {}
        for quant in ["fp16", "int8", "int4", "int3"]:
            latency = self.estimate_latency(input_tokens, output_tokens, quant)
            memory = self.memory_breakdown(1, input_tokens + output_tokens, quant)
            results[quant] = {
                "total_latency_ms": latency["total_ms"],
                "model_memory_gb": memory["model_weights_gb"],
                "total_memory_gb": memory["total_gb"]
            }
        return results
