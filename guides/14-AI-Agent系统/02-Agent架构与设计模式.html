🏗️ 模块02：Agent架构与设计模式

从感知-推理-行动循环到ReAct、Plan-Execute、Reflexion——掌握Agent的核心设计范式

1. 背景与动机：为什么需要架构模式？

一个Agent系统的核心问题是：如何将LLM的文本生成能力转化为可靠的自主行动能力？这不是简单地把LLM包装一下就能解决的——需要精心设计的架构来处理：

  • 推理与行动的耦合：LLM生成的文本如何转化为可执行的动作？
  • 错误恢复：行动失败后如何调整策略？
  • 长程规划：如何将复杂目标分解为可管理的子任务？
  • 自我改进：如何从过去的成功和失败中学习？

2022年以来，研究者提出了多种Agent架构模式。本模块将系统讲解最重要的5种模式。

2. 核心原理：Agent的基本控制循环

2.1 感知-推理-行动 (Perception-Reasoning-Action) 循环

所有Agent架构的基础都是一个控制循环：

┌─────────────────────────────────────────────────────────────────┐ │ Agent 基本控制循环 (PRA Loop) │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ ┌──────────┐ │ │ │ 目标设定 │ "帮我写一个爬虫程序" │ │ └────┬─────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ 推理 (Reasoning) │ │ │ │ LLM分析当前状态，决定下一步行动 │ │ │ │ 输入: 目标 + 当前状态 + 历史行动 + 环境反馈 │ │ │ │ 输出: 行动决策 (如: "调用search工具查找requests库") │ │ │ └────────┬─────────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ 行动 (Action) │ │ │ │ 执行具体的工具调用/代码生成/文件操作 │ │ │ │ 例: execute_python("import requests; ...") │ │ │ └────────┬─────────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────────────────────────────────────┐ │ │ │ 感知 (Perception) │ │ │ │ 获取行动结果和环境变化 │ │ │ │ 例: "requests库已安装，版本2.31.0" │ │ │ └────────┬─────────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────┐ │ │ │ 目标完成? │──否──→ 返回"推理"步骤 │ │ └────┬─────┘ │ │ │是 │ │ ▼ │ │ ┌──────────┐ │ │ │ 返回结果 │ │ │ └──────────┘ │ │ │ └─────────────────────────────────────────────────────────────────┘

2.2 形式化定义

Agent Loop: s₀ →[reason] a₀ →[act] s₁ →[observe] s₁' →[reason] a₁ → ... → goal

其中 sᵢ 是状态，aᵢ 是行动，sᵢ' 是观察到的新状态。每一步都经过LLM的推理决策。

2.3 POMDP视角下的Agent

2026年《Agentic AI: Architectures, Taxonomies, and Evaluation》论文提出用POMDP（部分可观测马尔可夫决策过程）来形式化Agent架构：

Agent POMDP = (S, A, T, R, Ω, O, γ)

其中：

  • S (State)：环境的真实状态，Agent无法直接观测
  • A (Action)：Agent的行动空间（工具调用、代码生成等）
  • T (Transition)：状态转移函数 T(s'|s,a)
  • R (Reward)：奖励函数，衡量任务完成度
  • Ω (Observation)：Agent能观测到的信息（工具返回、环境反馈）
  • O (Observation function)：O(o|s',a)，真实状态到观测的映射
  • γ (Discount)：折扣因子，权衡短期vs长期收益
💡 为什么POMDP重要：POMDP框架揭示了Agent的核心挑战——部分可观测性。Agent永远无法完全了解环境的真实状态（比如不知道API是否可靠、用户的真实意图是什么），必须通过推理和探索来减少不确定性。这解释了为什么Reflexion和LATS等"探索性"架构比简单的ReAct更有效。

3. ReAct：推理+行动统一框架

ReAct（Reasoning + Acting）是2022年由Yao et al.提出的里程碑式架构，首次将推理和行动统一在一个框架中。论文：《ReAct: Synergizing Reasoning and Acting in Language Models》

3.1 核心思想

ReAct的关键洞察是：让LLM在每一步都同时进行思考（Thought）和行动（Action），然后根据观察（Observation）继续推理。这形成了 Thought → Action → Observation 的三元组循环。

ReAct 架构示意图 — "苹果公司CEO的出生地的人口是多少?"
Thought 1
需要先找到CEO是谁
→
Action 1
search("苹果公司CEO")
→
Observation 1
CEO是Tim Cook，出生于阿拉巴马州
↓
Thought 2
需要查Mobile的人口
→
Action 2
search("Mobile Alabama population")
→
Observation 2
人口约 187,041
↓
Thought 3
已找到答案
→
Finish
约 187,041 人

💡 关键: 每一步都有显式的推理过程(Thought)，使决策可追踪、可调试

3.2 ReAct的优势

优势说明
可解释性Thought步骤提供了清晰的推理链，便于调试和审计
减少幻觉每步行动都有观察反馈，错误可以及时纠正
灵活性可以根据观察动态调整推理方向
通用性适用于问答、决策、代码生成等多种任务

3.3 ReAct的局限

  • 单步决策：每一步只考虑当前最优，缺乏全局规划
  • 上下文窗口消耗：Thought+Action+Observation快速填满上下文
  • 无法回溯：一旦做出错误行动，无法"撤销"
💡 实践建议：ReAct是大多数Agent系统的默认架构。如果你不知道用什么模式，从ReAct开始。LangChain的AgentExecutor和大多数Agent框架都默认实现了ReAct模式。

4. Plan-and-Execute：先规划后执行

Plan-and-Execute模式将Agent的工作分为两个明确阶段：先制定完整计划，然后逐步执行。这解决了ReAct"走一步看一步"的局限。

4.2 与ReAct的对比

维度ReActPlan-and-Execute
规划方式隐式（每步推理）显式（预先规划）
全局视野弱（只看当前步）强（有完整计划）
错误恢复在下一步纠正重规划器调整计划
适用场景简单任务、单步操作复杂任务、多步流程
Token消耗中等较高（规划+执行+重规划）
实现复杂度低中等
🔬 实现细节：LangChain的LangGraph框架原生支持Plan-and-Execute模式。核心是将Agent建模为状态图（StateGraph），其中Planner和Executor是不同的节点，通过条件边（conditional edges）连接。Re-planner节点在每个子任务执行后评估是否需要调整计划。

5. Reflexion：自我反思机制

Reflexion（Shinn et al., 2023）引入了语言反馈的自我改进机制。核心思想是：Agent在完成任务后，回顾自己的行动轨迹，总结经验教训，并将这些反思存储在记忆中供未来使用。

5.1 Reflexion循环

┌─────────────────────────────────────────────────────────────────┐ │ Reflexion 架构 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ Actor │────→│ Evaluator │────→│ Reflector │ │ │ │ (执行者) │ │ (评估器) │ │ (反思器) │ │ │ └──────────┘ └──────────┘ └──────────┘ │ │ ↑ │ │ │ │ │ │ │ │ ┌──────────┐ │ │ │ └─────────│ Memory │←───────────────┘ │ │ │ (记忆库) │ │ │ └──────────┘ │ │ │ │ 流程: │ │ 1. Actor执行任务，生成行动轨迹 τ │ │ 2. Evaluator评估结果 (成功/失败 + 分数) │ │ 3. 如果失败: Reflector分析τ，生成反思文本 r │ │ 4. 将r存入Memory，下次执行时作为上下文 │ │ 5. 重复执行，利用反思改进 │ │ │ └─────────────────────────────────────────────────────────────────┘

5.2 反思的内容

Reflexion的反思不是简单的"我错了"，而是具体的、可操作的分析：

# 错误的反思（无用）:
"我的回答是错误的。"

# 正确的反思（有用）:
"我在第3步使用了requests.get()但没有设置超时参数，
导致程序挂起。下次应该使用 requests.get(url, timeout=10)。
此外，我应该先检查URL是否可访问再发送请求。"

6. LATS：语言Agent树搜索

LATS（Language Agent Tree Search, Zhou et al., 2023）将蒙特卡洛树搜索 (MCTS) 引入Agent架构，通过树状搜索探索多条行动路径。

6.1 核心思想

┌─────────────────────────────────────────────────────────────────┐ │ LATS 树搜索示意图 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ [初始状态] │ │ / | \ │ │ / | \ │ │ [行动A] [行动B] [行动C] │ │ / \ | | │ │ / \ | | │ │ [A1] [A2] [B1] [C1] │ │ / \ | | | │ │ / \ | | | │ │ [A1a] [A1b] [A2a] [B1a] [C1a] │ │ ✓ ✗ ✓ ✗ ✓ │ │ │ │ ✓ = 评估为正向（继续探索） │ │ ✗ = 评估为负向（剪枝） │ │ │ │ 关键: 用LLM作为"价值函数"评估每个节点的潜力 │ │ 比ReAct的优势: 可以"回溯"到更好的分支 │ │ │ └─────────────────────────────────────────────────────────────────┘

6.2 LATS vs 其他模式

特性ReActReflexionLATS
搜索策略贪心（单路径）试错+反思树搜索（多路径）
回溯能力无有限（下轮重试）强（树内回溯）
计算成本低中高（多分支探索）
适用场景简单任务可评估的任务复杂决策/推理

7. 其他重要架构模式

7.1 Tool-augmented LLM

最简单的Agent架构——LLM + 工具调用。不需要复杂的循环，LLM根据用户请求直接选择并调用工具。

# Tool-augmented LLM 伪代码
response = llm.chat(
    messages=[{"role": "user", "content": query}],
    tools=available_tools  # 工具列表
)
if response.tool_calls:
    result = execute_tool(response.tool_calls[0])
    final = llm.chat(messages + [tool_result(result)])

7.2 Generative Agent (Stanford)

2023年Stanford的"Generative Agents"论文提出了一种面向社会模拟的Agent架构，包含三个核心组件：

  • 记忆流 (Memory Stream)：记录Agent的所有经历
  • 反思 (Reflection)：定期从记忆中提取高级洞察
  • 规划 (Planning)：基于记忆和反思制定行动日程

7.3 Cognitive Architecture (认知架构)

受认知科学启发的Agent架构，将Agent分为多个认知模块：

💡 参考答案与详解

解题思路: ReAct模式的核心是 Thought→Action→Observation 三元组循环。使用LangChain的 create_react_agent 需要：定义工具、创建prompt模板、组装Agent并用AgentExecutor运行。

# 完整可运行的ReAct Agent示例
from langchain.agents import AgentExecutor, create_react_agent
from langchain_openai import ChatOpenAI
from langchain.tools import Tool
from langchain_core.prompts import PromptTemplate
import requests

# 1. 定义工具函数
def search_weather(city: str) -> str:
    """模拟天气搜索工具"""
    weather_data = {"北京": "晴，28°C，湿度45%", "上海": "多云，26°C，湿度60%"}
    return weather_data.get(city, f"{city}天气数据暂未收录")

def calculate(expression: str) -> str:
    """安全的数学计算工具"""
    try:
        allowed = set("0123456789+-*/.(). ")
        if all(c in allowed for c in expression):
            return str(eval(expression))
        return "只支持基本数学运算"
    except Exception as e:
        return f"计算错误: {e}"

# 2. 创建工具列表
tools = [
    Tool(name="WeatherSearch", func=search_weather, description="查询城市天气，输入城市名"),
    Tool(name="Calculator", func=calculate, description="计算数学表达式，输入数学表达式字符串"),
]

# 3. 定义ReAct Prompt (标准格式)
react_prompt = PromptTemplate.from_template("""
Answer the following questions as best you can. You have access to the following tools:

{tools}

Use the following format:

Question: the input question
Thought: reasoning about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question

Begin!

Question: {input}
Thought:{agent_scratchpad}""")

# 4. 创建并运行Agent
llm = ChatOpenAI(model="gpt-4o", temperature=0)
agent = create_react_agent(llm, tools, react_prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True, max_iterations=5)

# 5. 测试
result = executor.invoke({"input": "今天北京天气如何?"})
print(result["output"])
# 输出示例: "今天北京天气晴朗，气温28°C，湿度45%。"

关键知识点: (1) ReAct的prompt必须包含Thought/Action/Observation的格式约束 (2) AgentExecutor负责实际执行工具调用并将结果回传给LLM (3) max_iterations防止无限循环 (4) 实际生产中推荐用LangGraph替代AgentExecutor，因为LangGraph支持状态持久化、人机协作和更灵活的控制流。

⚠️ 选型建议：不要过度工程化。80%的Agent场景用ReAct就够了。只有当你发现Agent频繁犯同样的错误（需要Reflexion）或需要全局规划（需要Plan-and-Execute）时，才考虑更复杂的架构。

小结

概念一句话
PRA循环所有Agent的基础：感知→推理→行动的持续循环
ReAct每步同时思考(Thought)和行动(Action)，可解释性强
Plan-and-Execute先制定完整计划再执行，支持动态重规划
Reflexion通过语言反思从失败中学习，持续改进
LATS将MCTS引入Agent，通过树搜索探索多条路径
选型原则简单任务用ReAct，复杂任务用Plan-Execute，需要学习用Reflexion
