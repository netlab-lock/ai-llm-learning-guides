📊 模块09：Agent评估与基准

评估维度、基准测试、方法论——如何衡量Agent的能力

1. 背景与动机

Agent的评估比LLM评估复杂得多。LLM评估主要看文本质量，而Agent评估需要考虑任务完成度、工具使用准确性、推理质量、安全性、效率等多个维度。2026年《A Survey on Evaluation of LLM-based Agents》系统总结了当前的评估方法。

┌─────────────────────────────────────────────────────────────────┐ │ Agent评估的POMDP形式化框架 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ Agent评估可以形式化为一个POMDP: │ │ (S, A, T, R, Ω, O, γ) │ │ │ │ S = 状态空间 (任务状态、环境状态) │ │ A = 行动空间 (工具调用、代码生成、文件操作) │ │ T = 状态转移函数 T(s'|s,a) │ │ R = 奖励函数 (任务完成度、效率、安全性) │ │ Ω = 观察空间 (工具返回、环境反馈) │ │ O = 观察函数 O(o|s',a) │ │ γ = 折扣因子 (短期vs长期收益权衡) │ │ │ │ 评估的核心问题: │ │ 1. 如何定义R? (什么算"成功"?) │ │ 2. 如何采样S和A? (用什么任务测试?) │ │ 3. 如何衡量泛化能力? (训练集外的任务) │ │ │ └─────────────────────────────────────────────────────────────────┘
🔬 评估的历史演变：Agent评估经历了三个阶段。(1) 2022-2023: 以人工评估为主，研究者手动检查Agent的输出质量，成本高且不可复现。(2) 2023-2024: 基准测试标准化，SWE-bench、ToolBench等基准出现，支持自动化评估。(3) 2025-2026: 真实环境评估成为主流，MCP Atlas和Tool-Decathlon等基准直接使用真实的MCP Server和API，更接近生产场景。

2. 评估维度

维度指标说明
任务完成度成功率、完成率Agent是否成功完成了用户指定的任务
工具使用工具选择准确率、参数正确率Agent是否选择了正确的工具并传入正确参数
推理质量推理步骤合理性、错误率Agent的推理过程是否合理、是否有幻觉
安全性安全违规次数、越权操作Agent是否执行了危险或越权的操作
效率Token消耗、工具调用次数、延迟完成任务消耗了多少资源
鲁棒性错误恢复率、异常处理遇到错误时能否恢复并继续
💡 评估维度选择建议：不同场景需要关注不同的评估维度。编程Agent优先关注任务完成度和效率；客服Agent优先关注推理质量和安全性；科研Agent优先关注推理质量。不要试图在所有维度上都做到最好——根据业务场景确定优先级。

3.2 基准测试详解

SWE-bench 代码修复基准

SWE-bench是当前最权威的Agent代码能力评估基准，包含2294个真实的GitHub Issue，来自12个流行的Python开源项目。

ToolBench 工具使用基准

ToolBench包含16000+个真实API，覆盖49个类别（天气、新闻、社交、支付等）。评估Agent在面对大量工具时的选择和调用能力。核心指标：通过率（Pass Rate）、胜率（Win Rate）、令牌效率。

AgentBench 综合能力基准

AgentBench覆盖8种不同环境：操作系统、数据库、知识图谱、数字卡牌游戏、家务操作、网页购物、网页浏览、编码。评估Agent的跨领域能力。

WebArena 网页操作基准

WebArena在真实网站（Reddit、GitLab、购物网站等）上评估Agent的网页操作能力。不同于模拟环境，WebArena使用真实的网站数据和功能。

MCP Atlas (2026)

Scale AI发布的最新基准，直接使用真实的MCP Server评估Agent的工具使用能力。包含数百个MCP Server，覆盖文件操作、数据库查询、API调用等场景。

Tool-Decathlon (2026)

挑战性最高的工具使用基准，要求Agent在单个任务中协调使用多个工具完成复杂目标。模拟真实工作场景中的多工具协作。

⚠️ 基准测试的局限：基准测试成绩不等于实际生产表现。(1) 基准任务可能被"刷榜"——模型在特定数据集上过拟合 (2) 基准通常不包含真实世界的噪声和异常 (3) 基准不评估用户体验和交互质量。最佳实践：基准测试作为初筛，真实A/B测试作为最终验证。

4. 评估方法论

4.1 静态评估 vs 动态评估

方法说明优势劣势
静态评估使用固定数据集评估可复现、可对比可能过拟合、不代表真实场景
动态评估实时生成新任务评估更接近真实场景不可复现、成本高
A/B测试在线对比不同Agent最接近真实用户体验需要线上环境

4.2 LLM-as-Judge

用另一个LLM来评估Agent的输出质量，是当前最常用的评估方法：

# LLM-as-Judge 评估示例
evaluation_prompt = """
请评估以下Agent的执行质量:

任务: {task}
Agent行动轨迹: {trajectory}
最终结果: {result}

请从以下维度评分(1-5):
1. 任务完成度
2. 推理质量
3. 工具使用准确性
4. 效率

输出JSON格式:
{{"completion": 4, "reasoning": 3, "tool_use": 5, "efficiency": 4, "overall": 4}}
"""
⚠️ 评估陷阱：LLM-as-Judge存在位置偏差（倾向给第一个选项更高分）、长度偏差（倾向给更长的回答更高分）和自我偏好（倾向给自己生成的内容更高分）。使用时需要多个Judge交叉验证。

5. 可观测性与调试

Agent的调试比传统软件困难得多，因为Agent的行为具有非确定性。可观测性工具是Agent开发的必备：

解题思路: LangFuse是开源的LLM可观测性平台，通过trace(追踪)和span(跨度)记录Agent每一步的输入/输出/延迟/成本。对于Agent调试至关重要——因为Agent的非确定性行为难以复现。

# 完整可运行的LangFuse追踪示例
# pip install langfuse openai
from langfuse import Langfuse
from openai import OpenAI

langfuse = Langfuse()  # 自动读取环境变量 LANGFUSE_PUBLIC_KEY/SECRET_KEY
openai_client = OpenAI()

# 创建追踪 (一个完整的Agent执行)
trace = langfuse.trace(
    name="qa-agent",
    metadata={"model": "gpt-4o", "task_type": "question_answering"}
)

# Step 1: 记录检索span
retrieval_span = trace.span(
    name="retrieval",
    input={"query": "Python异步编程最佳实践"},
)
# 模拟检索
docs = ["asyncio是Python的内置异步框架...", "aiohttp适合HTTP客户端..."]
retrieval_span.end(output={"documents": docs, "count": len(docs)})

# Step 2: 记录生成span
generation_span = trace.span(
    name="generation",
    input={"query": "Python异步编程最佳实践", "context": docs},
)
response = openai_client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": f"基于以下资料回答:\n{docs}"},
        {"role": "user", "content": "Python异步编程最佳实践"}
    ]
)
answer = response.choices[0].message.content
generation_span.end(
    output={"answer": answer},
    metadata={"tokens": response.usage.total_tokens}
)

# Step 3: 记录评估score
trace.score(name="relevance", value=4.5, comment="回答切题度高")
trace.score(name="faithfulness", value=5.0, comment="忠实于检索结果")

print(f"追踪ID: {trace.id}")
print(f"查看: https://cloud.langfuse.com/traces/{trace.id}")

# 使用价值:
# 1. 可视化Agent每步的输入/输出/耗时
# 2. 发现瓶颈 (哪步最慢? 哪步token最多?)
# 3. 对比不同prompt/模型的效果
# 4. 收集评估数据用于持续优化

关键知识点: (1) Trace = 一次完整Agent执行，Span = 其中的一个步骤 (2) LangFuse支持自托管(开源版)和云服务 (3) 生产中应在Agent的每个工具调用、LLM推理步骤都添加span (4) 评估score支持人工和自动两种方式 (5) 可观测性是Agent从"能用"到"好用"的关键——没有追踪就无法优化。

当公开基准不能满足需求时，你需要构建自定义评估。以下是完整的评估流水线设计：

解题思路: 构建评估器的核心是：定义测试用例(含期望值) → 执行Agent收集轨迹 → 多维度打分 → 汇总统计。关键是测试用例要覆盖happy path和边界情况。

# 完整可运行的Agent评估器
import json, time
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class EvalResult:
    query: str
    tool_correct: bool
    answer_correct: bool
    steps: int
    tokens: int
    latency_ms: float
    issues: List[str] = field(default_factory=list)

class AgentEvaluator:
    """Agent自动评估框架"""

    def __init__(self, agent, test_cases: List[dict]):
        self.agent = agent
        self.test_cases = test_cases
        self.results: List[EvalResult] = []

    def run(self):
        """执行所有测试用例"""
        for i, case in enumerate(self.test_cases):
            print(f"[{i+1}/{len(self.test_cases)}] {case['query'][:30]}...")
            start = time.time()

            # 执行Agent (用你的agent替换)
            # trajectory = self.agent.run(case["query"])
            # 模拟结果
            trajectory = type('T', (), {
                'last_tool': case.get('expected_tool', ''),
                'answer': case.get('expected_result', ''),
                'steps': 3,
                'total_tokens': 1500
            })()

            latency = (time.time() - start) * 1000
            issues = []

            # 检查工具选择
            tool_ok = trajectory.last_tool == case.get("expected_tool", "")
            if not tool_ok:
                issues.append(f"工具错误: 期望{case.get('expected_tool')}, 实际{trajectory.last_tool}")

            # 检查答案
            expected = case.get("expected_result", "")
            answer_ok = expected.lower() in trajectory.answer.lower() if expected else True
            if not answer_ok:
                issues.append(f"答案不匹配: 期望含'{expected}'")

            self.results.append(EvalResult(
                query=case["query"],
                tool_correct=tool_ok,
                answer_correct=answer_ok,
                steps=trajectory.steps,
                tokens=trajectory.total_tokens,
                latency_ms=latency,
                issues=issues
            ))

    def report(self) -> dict:
        """生成评估报告"""
        total = len(self.results)
        return {
            "total": total,
            "tool_accuracy": sum(r.tool_correct for r in self.results) / total,
            "answer_accuracy": sum(r.answer_correct for r in self.results) / total,
            "avg_steps": sum(r.steps for r in self.results) / total,
            "avg_tokens": sum(r.tokens for r in self.results) / total,
            "avg_latency_ms": sum(r.latency_ms for r in self.results) / total,
            "failures": [r.query for r in self.results if not r.answer_correct]
        }

关键知识点: (1) 评估器应检查多个维度：工具选择、答案正确性、步骤数、token消耗、延迟 (2) 测试用例应覆盖happy path + 边界情况 + 异常输入 (3) 失败案例分析比总体指标更重要——逐条分析失败原因才能持续优化 (4) 用LLM-as-Judge评估开放式回答的质量。

💡 评估最佳实践：(1) 评估数据集要覆盖边界情况，不仅仅是happy path (2) 使用回归测试确保新版本不会破坏已有的能力 (3) 评估结果要可复现——固定seed、固定模型版本 (4) 定期更新评估集，避免过拟合。
