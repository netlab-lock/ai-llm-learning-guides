🔧 模块05：工具使用与函数调用

Function Calling、MCP协议、工具编排——让Agent拥有"手"的能力

1. 背景与动机

LLM本身只能生成文本。要让Agent真正"做事"（搜索网页、调用API、操作数据库、写文件），就需要工具使用能力。2023年Toolformer论文的发表标志着LLM工具使用从"手工工程"走向"自动化"。

┌─────────────────────────────────────────────────────────────────┐ │ 工具使用的核心价值 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 纯LLM: │ │ "今天北京天气如何?" → "我无法获取实时天气信息" ✗ │ │ │ │ 工具增强LLM: │ │ "今天北京天气如何?" │ │ → 调用 weather_api(city="Beijing") │ │ → 获取结果 "晴，28°C，湿度45%" │ │ → 生成回答 "今天北京天气晴朗，气温28°C..." ✓ │ │ │ │ 工具扩展了LLM的能力边界: │ │ • 实时信息 (搜索、天气、股价) │ │ • 精确计算 (计算器、代码执行) │ │ • 外部操作 (发邮件、创建文件、调用API) │ │ • 专业能力 (数据库查询、图像生成、语音合成) │ │ │ └─────────────────────────────────────────────────────────────────┘

2. Function Calling 机制

Function Calling是OpenAI在2023年6月引入的功能，让LLM能够结构化地调用外部函数。这是当前Agent工具使用的主流技术。

2.2 工具定义的JSON Schema

Function Calling的核心是JSON Schema——它告诉LLM每个工具接受什么参数、参数类型是什么、哪些是必填的。

// 标准的工具定义JSON Schema
{{
  "type": "function",
  "function": {{
    "name": "search_database",
    "description": "在数据库中搜索产品信息。适用于精确查询。",
    "parameters": {{
      "type": "object",
      "properties": {{
        "query": {{
          "type": "string",
          "description": "搜索关键词"
        }},
        "category": {{
          "type": "string",
          "enum": ["electronics", "clothing", "food"],
          "description": "产品类别过滤"
        }},
        "max_results": {{
          "type": "integer",
          "default": 10,
          "description": "最大返回数量"
        }}
      }},
      "required": ["query"]
    }}
  }}
}}
💡 Schema设计要点：(1) description要详细——LLM靠它理解工具用途 (2) 用enum限制参数取值范围 (3) 提供default值减少LLM的决策负担 (4) 必填参数越少越好——降低参数遗漏的风险。

2.3 并行函数调用

现代LLM支持并行函数调用——一次请求中调用多个工具：

# 用户: "北京和上海的天气分别如何?"

# LLM返回两个并行的tool_calls:
tool_calls = [
    {"function": {"name": "get_weather", "arguments": '{"city":"北京"}'}},
    {"function": {"name": "get_weather", "arguments": '{"city":"上海"}'}}
]

3. MCP协议：Model Context Protocol

MCP（Model Context Protocol）是Anthropic于2024年11月发布的开放协议，旨在标准化LLM与工具/数据源之间的连接方式。它被称为"Agent的USB-C接口"。

3.1 MCP的核心设计

┌─────────────────────────────────────────────────────────────────┐ │ MCP 协议架构 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 传统方式 (N个模型 × M个工具 = N×M个集成): │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │GPT-4 │ │Claude│ │Llama │ │ │ └──┬───┘ └──┬───┘ └──┬───┘ │ │ │ │ │ │ │ ┌──┴──┬──┬──┴──┬──┬──┴──┬──┬──────┐ │ │ │搜索 │DB│邮件 │DB│文件 │DB│ ... │ │ │ └─────┘ └─────┘ └─────┘ └──────┘ │ │ 需要 N×M = 9个集成代码 │ │ │ │ MCP方式 (N+M个集成): │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │GPT-4 │ │Claude│ │Llama │ │ │ └──┬───┘ └──┬───┘ └──┬───┘ │ │ │ │ │ │ │ └─────────┼─────────┘ │ │ │ MCP协议 │ │ ┌─────────┼─────────┐ │ │ │ │ │ │ │ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ │ │ │MCP │ │MCP │ │MCP │ │ │ │Server│ │Server│ │Server│ │ │ │(搜索)│ │(DB) │ │(文件)│ │ │ └─────┘ └─────┘ └─────┘ │ │ 只需要 N+M = 6个集成代码 │ │ │ └─────────────────────────────────────────────────────────────────┘

3.2 MCP的核心概念

概念说明类比

3.3 MCP通信方式

传输方式说明适用场景
stdio通过标准输入/输出通信本地进程、CLI工具
SSE (HTTP)Server-Sent Events over HTTP远程服务、Web应用
Streamable HTTP可流式传输的HTTP（2025新标准）生产环境、云服务
🔬 MCP vs Function Calling：Function Calling是模型层的能力——LLM决定调用什么函数、传什么参数。MCP是协议层的标准——定义了工具如何注册、发现和连接。两者是互补关系：MCP Server暴露工具，Function Calling让LLM决定如何使用这些工具。

2.5 Structured Output：结构化输出

除了Function Calling，LLM还可以通过Structured Output返回结构化数据。这在Agent需要输出特定格式（如JSON、表格）时非常有用。

方式原理优势适用场景
JSON Mode强制LLM输出合法JSON简单直接数据提取、分类
Tool Calling通过工具定义约束输出格式格式100%符合Schema结构化决策
Pydantic模型用Python类定义输出结构类型安全、自动验证复杂数据结构

4. 工具编排与链式调用

实际Agent任务往往需要多个工具的组合使用。工具编排的模式包括：

┌─────────────────────────────────────────────────────────────────┐ │ 工具编排模式 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 1. 顺序链 (Sequential): │ │ Tool_A → Tool_B → Tool_C → 最终结果 │ │ 例: 搜索 → 提取 → 总结 │ │ │ │ 2. 并行调用 (Parallel): │ │ ┌─ Tool_A ─┐ │ │ ├─ Tool_B ─┤ → 合并结果 │ │ └─ Tool_C ─┘ │ │ 例: 同时搜索多个来源 │ │ │ │ 3. 条件分支 (Conditional): │ │ Tool_A → if 成功 → Tool_B │ │ → if 失败 → Tool_C │ │ 例: 尝试API调用，失败则回退到缓存 │ │ │ │ 4. 递归调用 (Recursive): │ │ Tool_A → 结果不完整 → Tool_A(更精确的参数) → ... │ │ 例: 搜索结果不满意，调整关键词重新搜索 │ │ │ └─────────────────────────────────────────────────────────────────┘

5. 工具描述的艺术

工具的描述质量直接影响LLM的选择准确率。好的工具描述包括：

# ❌ 差的工具描述
"name": "search"
"description": "搜索"

# ✅ 好的工具描述
"name": "web_search"
"description": "在互联网上搜索信息。适用于: (1)查找最新新闻或事件
(2)获取实时数据如天气、股价 (3)查找技术文档或教程。
不适用于: (1)数学计算(用calculator) (2)代码执行(用code_runner)"
💡 实践建议：工具描述应该包含三个要素：(1)功能说明 (2)适用场景 (3)不适用场景。这样LLM才能做出正确的工具选择。

6. 错误处理与重试策略

错误类型处理策略示例
工具不存在告知LLM可用工具列表，重新选择"没有find工具，请使用search或lookup"
参数错误返回错误信息，让LLM修正参数"city参数不能为空"
超时重试（指数退避）或换工具搜索超时→重试→换搜索引擎
权限不足告知LLM权限限制"没有写入/root的权限"
结果异常让LLM评估结果质量，决定是否重试"搜索结果为空，建议换个关键词"
⚠️ 安全警告：工具使用是Agent最危险的能力。一个能调用任意API的Agent，如果推理出错，可能造成不可逆的损失（如删除文件、发送错误邮件）。最佳实践：(1)最小权限原则 (2)关键操作需人类确认 (3)沙箱执行环境 (4)操作日志审计。

💡 参考答案与详解

解题思路: MCP Server的核心是定义工具（用装饰器注册）、实现工具逻辑、通过stdio或HTTP暴露服务。完整MCP Server需要处理工具发现、参数验证和错误处理。

# 完整可运行的MCP Server (server.py)
from mcp.server import Server
from mcp.server.stdio import stdio_server
import mcp.types as types
import json, math

server = Server("my-tools")

# ── 工具1: 天气查询 ──
@server.tool()
async def get_weather(city: str) -> str:
    """获取指定城市的天气信息。
    
    Args:
        city: 城市名称，如"北京"、"上海"
    """
    # 实际实现会调用天气API
    weather_db = {
        "北京": "晴，28°C，湿度45%，东北风3级",
        "上海": "多云，26°C，湿度60%，东南风2级",
        "深圳": "阵雨，30°C，湿度75%，南风1级",
    }
    result = weather_db.get(city)
    if result:
        return f"{city}天气: {result}"
    return f"未找到{city}的天气数据，请检查城市名称"

# ── 工具2: 安全计算器 ──
@server.tool()
async def calculate(expression: str) -> str:
    """计算数学表达式。支持加减乘除、幂运算、三角函数等。
    
    Args:
        expression: 数学表达式，如"2**10"、"math.sqrt(144)"
    """
    safe_globals = {"__builtins__": {}, "math": math}
    try:
        result = eval(expression, safe_globals)
        return f"{expression} = {result}"
    except Exception as e:
        return f"计算错误: {e}"

# ── 工具3: 文本分析 ──
@server.tool()
async def analyze_text(text: str) -> str:
    """分析文本的基本统计信息。
    
    Args:
        text: 要分析的文本内容
    """
    stats = {
        "字符数": len(text),
        "单词数": len(text.split()),
        "行数": len(text.splitlines()),
        "中文字符数": sum(1 for c in text if '\u4e00' <= c <= '\u9fff'),
    }
    return json.dumps(stats, ensure_ascii=False, indent=2)

# ── 资源: 提供工具列表 ──
@server.list_tools()
async def list_tools():
    return await server.list_tools()

# ── 启动服务器 (stdio通信模式) ──
async def main():
    async with stdio_server() as (read_stream, write_stream):
        await server.run(read_stream, write_stream,
                        server.create_initialization_options())

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

# 测试方法:
# 1. 安装: pip install mcp
# 2. 运行: python server.py
# 3. 在Claude Desktop配置中添加此server
# 4. 或用MCP Inspector测试: npx @modelcontextprotocol/inspector python server.py

关键知识点: (1) MCP Server通过@server.tool()装饰器注册工具，docstring自动成为工具描述 (2) safe_globals限制eval的执行环境，防止代码注入 (3) MCP支持三种通信方式：stdio（本地）、SSE（远程）、Streamable HTTP（生产） (4) MCP的核心价值是N+M集成（而非N×M），一次实现的工具可被所有支持MCP的LLM应用使用。

7. Agent成本优化

Agent的token消耗远高于普通LLM调用。一个复杂任务可能消耗数万到数十万tokens。成本优化是生产部署的关键。

策略方法节省比例
模型分层简单任务用小模型，复杂任务用大模型50-70%
缓存缓存工具结果和常见查询的回复30-50%
上下文压缩摘要历史对话，只保留关键信息20-40%
提前终止检测到Agent偏离目标时及时终止10-30%
批量处理将多个小任务合并为一次LLM调用20-40%
⚠️ 成本陷阱：Agent的隐性成本包括：(1) 重试——工具失败后重试消耗额外token (2) 规划开销——Plan-and-Execute的规划步骤本身消耗token (3) 评估——LLM-as-Judge评估也需要调用LLM。建议在开发阶段就建立成本监控，避免上线后才发现成本失控。

安全原则最小权限+人类确认+沙箱+日志审计
