👥 模块06：多智能体系统

协作模式、角色分配、A2A协议——当多个Agent协同工作

1. 背景与动机

单个Agent的能力有限。当任务复杂度超过一定阈值时，单Agent架构面临上下文窗口溢出、错误累积、角色混乱等问题。多智能体系统通过分工协作来解决这些问题。

2. 多Agent协作模式

2.1 协作模式分类

模式结构通信方式适用场景
流水线 (Pipeline)线性链单向传递有序任务（如：分析→设计→实现）
层级 (Hierarchy)树形结构上下级通信管理型任务（如：PM分配子任务）
辩论 (Debate)对等结构多方讨论需要多角度分析（如：方案评审）
投票 (Voting)对等结构独立输出+投票需要共识（如：代码审查）
专家混合 (MoE)星形结构路由器分发不同类型任务（如：路由到专业Agent）
群体 (Swarm)网状结构点对点分布式任务（如：大规模数据处理）

3. A2A协议：Agent-to-Agent

A2A（Agent-to-Agent Protocol）是Google于2025年4月发布的开放协议，旨在标准化不同Agent之间的通信方式。如果说MCP是Agent与工具之间的"USB-C"，那么A2A就是Agent与Agent之间的"HTTP"。

3.1 A2A的核心概念

概念说明
Agent CardAgent的"名片"，描述其能力、端点、认证方式（类似API的OpenAPI spec）
TaskAgent之间的任务请求，包含状态机（submitted→working→completed/failed）
MessageAgent之间的消息交换，支持文本、文件、结构化数据
Artifact任务的输出产物（如生成的代码、文档、图片）

3.2 A2A vs MCP

维度MCPA2A
连接对象LLM ↔ 工具/数据源Agent ↔ Agent
通信粒度单次函数调用多轮任务协作
状态管理无状态有状态（任务生命周期）
发现机制静态配置Agent Card动态发现
适用场景工具调用Agent协作
💡 关系理解：MCP和A2A不是竞争关系，而是互补关系。MCP解决"Agent如何使用工具"，A2A解决"Agent如何与其他Agent协作"。一个完整的多Agent系统通常同时使用两者。

4. 角色分配与专业化

多Agent系统的关键设计决策之一是如何为每个Agent分配角色：

4.1 角色定义要素

要素说明示例
身份 (Identity)Agent的名字和基本描述"你是一个资深前端工程师"
目标 (Goal)Agent需要达成的目标
技能 (Skills)Agent具备的能力"精通React、TypeScript、CSS"
约束 (Constraints)Agent的行为限制"不使用任何第三方UI库"
工具 (Tools)Agent可用的工具"code_runner, file_editor"

4.2 常见角色模式

模式角色组合适用场景
软件开发PM + 架构师 + 前端 + 后端 + 测试MetaGPT
研究分析研究员 + 分析师 + 写手 + 编辑GPT Researcher
内容创作策划 + 写手 + 编辑 + 发布内容工厂
客服系统路由 + 专业客服 + 升级处理企业客服
🔬 Stanford小镇实验的发现：2023年Stanford的"Generative Agents"论文让25个AI Agent在一个虚拟小镇中生活。令人惊讶的发现：(1) Agent自发形成了社交网络——某些Agent成为"社交中心"，其他Agent围绕它们建立关系。(2) 信息传播遵循现实世界的模式——重要消息通过社交网络传播，衰减速度与现实类似。(3) Agent展现了协同行为——它们自发组织了情人节派对，协调时间地点。这些"涌现行为"是多Agent系统最令人兴奋的研究方向。

4.2 A2A协议技术细节

A2A的核心是Task状态机——每个跨Agent任务都有明确的生命周期：

Task状态: submitted → working → completed | failed | canceled
状态含义触发条件可执行操作
submitted已提交，等待处理Agent Card发现+任务创建查询状态、取消
working正在执行目标Agent开始处理查询进度、补充信息
completed任务完成目标Agent返回结果获取Artifact
failed任务失败执行出错或超时查看错误、重试
💡 A2A vs MCP：MCP是Agent与工具之间的标准(单次调用，无状态)。A2A是Agent之间的标准(多轮协作，有状态机)。类比：MCP是USB接口，A2A是HTTP协议。一个完整的Agent系统同时使用两者。

5. 冲突解决与共识机制

多Agent系统不可避免地会遇到冲突——不同Agent给出矛盾的建议或行动。常见的解决策略：

策略方法适用场景
权威优先高级Agent的决策覆盖低级Agent层级结构
多数投票取多数Agent一致的意见对等结构
评委裁决由专门的评委Agent做最终决定辩论模式
置信度加权根据每个Agent的置信度加权需要量化判断
人类仲裁无法达成共识时请求人类介入关键决策
⚠️ 主要风险：2026年的研究《Beyond Individual Intelligence》指出，多Agent系统的最大风险不是单个Agent的错误，而是错误传播——一个Agent的错误可能通过协作网络放大到整个系统。设计多Agent系统时，必须考虑错误隔离和验证机制。

5.3 多Agent编排工程实现

# LangGraph多Agent编排示例
from langgraph.graph import StateGraph, MessagesState
from langchain_openai import ChatOpenAI

def researcher(state):
    llm = ChatOpenAI(model="gpt-4o", temperature=0)
    response = llm.invoke([
        {{"role": "system", "content": "你是技术研究员。"}},
        *state["messages"]
    ])
    return {{"messages": [response]}}

def writer(state):
    llm = ChatOpenAI(model="gpt-4o", temperature=0.7)
    response = llm.invoke([
        {{"role": "system", "content": "你是技术写手。"}},
        *state["messages"]
    ])
    return {{"messages": [response]}}

graph = StateGraph(MessagesState)
graph.add_node("researcher", researcher)
graph.add_node("writer", writer)
graph.set_entry_point("researcher")
graph.add_edge("researcher", "writer")
app = graph.compile()

💡 参考答案与详解

解题思路: CrewAI的核心是 Agent(角色) → Task(任务) → Crew(团队) → kickoff(执行)。关键是为每个Agent定义清晰的role、goal和backstory，并通过context参数建立任务依赖。

# 完整可运行的CrewAI多Agent系统
from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role="资深技术研究员", goal="搜索{topic}最新技术进展",
    backstory="10年经验技术研究员，擅长提取关键信息。",
    llm="gpt-4o", verbose=True
)
writer = Agent(
    role="技术写手", goal="将研究结果写成技术文章",
    backstory="擅长用简单语言解释复杂技术。", llm="gpt-4o"
)
research_task = Task(description="搜索{topic}最新进展", agent=researcher,
                     expected_output="5-10个关键发现的结构化报告")
write_task = Task(description="撰写2000字技术博客", agent=writer,
                  expected_output="Markdown格式技术文章", context=[research_task])
crew = Crew(agents=[researcher, writer], tasks=[research_task, write_task],
            process=Process.sequential, verbose=True)
print(crew.kickoff(inputs={"topic": "LLM Agent"}))

关键知识点: (1) CrewAI三要素: Agent(角色) + Task(任务) + Crew(编排) (2) context参数建立任务依赖 (3) Process.sequential顺序执行，也可用hierarchical (4) 实际项目应为Agent绑定搜索/代码等工具。
