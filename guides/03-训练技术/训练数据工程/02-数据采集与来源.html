🔍 02 · 数据采集与来源 — LLM 的"粮食"从哪来

网页、代码、书籍、多语言、合成数据 · 数据许可证与合规

一、背景与动机

LLM 需要海量文本数据来学习语言模式和世界知识。但这些数据从哪来？答案是：几乎所有人类公开发布的文本——网页、书籍、论文、代码、论坛帖子...

数据采集不是简单的"下载"，而是涉及爬虫技术、数据格式解析、版权合规、多语言覆盖等复杂工程问题。

💡 关键认知：公开可用的高质量文本数据正在接近"峰值"。Llama 3 用了 15T tokens，而估计互联网上有价值的英文文本总量约 20-30T tokens。合成数据和多语言数据将成为未来的关键补充。
⚠️ 注意事项：关于「一、背景与动机」的常见陷阱：不要盲目照搬网上的配置参数。不同的数据分布和模型规模下，LLM的最优配置可能完全不同。务必在自己的场景下做充分的验证实验。
🔍 深入理解：「一、背景与动机」的深层理解：从系统设计的角度看，LLM的设计体现了关注点分离的原则——每个组件只负责一个明确的功能，通过清晰的接口协作。这种设计哲学是构建可维护、可扩展系统的基础。

二、名字由来

Common Crawl：Common(公共的) + Crawl(爬取)。非营利组织，从 2008 年开始持续爬取互联网，数据免费开放。

Web Crawl：像蜘蛛(spider)一样在网页间沿着链接爬行，因此叫"crawl"。

Corpus：拉丁语"身体"，引申为"集合体"。语料库 = 语言材料的集合。

💡 实用技巧：动手实践Common Crawl数据时，先用warcio库读取一个WET文件（纯文本提取），感受实际数据格式。再对比WARC文件（原始HTML），理解文本提取的必要性。HuggingFace datasets库提供了FineWeb等数据集的流式加载，无需下载全部数据。
⚠️ 注意事项：Common Crawl的WET文件中经常残留HTML标签、JavaScript代码和乱码，文本提取并不完美。在使用前务必做基础质量检查：抽样查看100篇文档，检查是否有明显的提取错误。不要假设WET文件已经是"干净"的纯文本。
🔍 深入理解："Crawl"这个术语揭示了网页数据采集的本质：沿着超链接图进行广度优先遍历。这意味着爬虫的覆盖范围取决于种子URL和链接图的连通性——孤立页面（无入链）永远不会被爬到，这也是为什么Common Crawl对深网内容的覆盖率很低。

3.6 合成数据的兴起

💡 实用技巧：采集代码数据时，推荐用BigQuery查询GitHub数据（免费额度每月1TB），先按star数排序筛选高质量仓库，再按许可证过滤。对于学术论文，arXiv的API支持批量下载LaTeX源码，比PDF提取的文本质量高得多。
⚠️ 注意事项：从GitHub采集代码时，必须先过滤fork仓库（约占60%）、自动生成代码（如minified JS）、和数据文件（如.csv、.json）。BigCode的StarCoder Data项目有详细的过滤规则可参考。忽略这些预处理会导致大量低质量数据污染训练集。
🔍 深入理解：数据来源的多样性直接影响模型的泛化能力。Common Crawl主要覆盖"公开可爬取"的网页，但对数据库内容、付费墙后面的文章、社交媒体私密帖子等"深网"内容完全缺失。这意味着模型的知识天然偏向于公开网页上流行的观点和信息。

⚠️ 法律风险：纽约时报起诉 OpenAI 侵权，Books3 数据集被法院判定侵权。在采集数据时，必须仔细审查每个数据源的许可证。合规不是可选项，而是必要条件。
💡 实用技巧：选择数据集时，优先考虑有完整开源pipeline的数据集（如Dolma、FineWeb），而非仅有最终数据的数据集。开源pipeline允许你复现、修改和定制过滤规则。RedPajama v2提供了原始数据+质量信号，适合想自定义过滤策略的团队。
🔍 深入理解：数据许可证问题是AI行业最大的法律灰色地带。目前没有任何国家的法律明确规定"用受版权保护的数据训练AI模型"是否构成合理使用。纽约时报vs OpenAI的案件结果将对整个行业产生深远影响。建议在法律明朗前，优先使用CC0/CC-BY等明确允许的数据源。

五、使用场景

  • 预训练数据集构建：从多个数据源组合构建训练集
  • 领域数据采集：为特定领域(医疗/法律/金融)采集专业数据
  • 多语言数据构建：为低资源语言构建训练数据
  • 持续预训练：采集最新数据注入模型
💡 实用技巧：为特定领域（如医疗、法律、金融）采集数据时，推荐的来源包括：PubMed（生物医学摘要）、CourtListener（法律判决文书）、SEC EDGAR（金融报告）。这些数据源通常是开放获取的，且质量远高于通用网页数据。
⚠️ 注意事项：持续预训练时采集新数据，必须与已有训练数据做去重处理，否则会导致模型在重复数据上过拟合。同时注意新数据的时效性——新闻类数据可能很快过时，而科学知识相对稳定。建议按数据类型设定不同的更新频率。
🔍 深入理解：多语言数据采集的核心挑战是"资源诅咒"：高资源语言（英语）数据丰富但竞争激烈，低资源语言数据稀缺但机会大。Llama 3的策略是先用英语数据训练核心能力，再用合成多语言数据扩展——这本质上是"知识蒸馏"从高资源语言到低资源语言。

💡 实用技巧：数据采集方式的选择建议：小团队优先使用Common Crawl + 开源数据集（零成本），中等团队可以自建爬虫补充特定领域数据，大团队可以购买商业数据或大规模合成数据。不同方式的ROI差异很大，Common Crawl的性价比最高。
⚠️ 注意事项：自建爬虫时务必遵守robots.txt协议和网站的使用条款。高频爬取可能导致IP被封禁，甚至面临法律风险。建议设置合理的爬取间隔（如每秒1-2个请求），并使用User-Agent标识自己的爬虫身份。
🔍 深入理解：合成数据正在改变数据采集的经济学：传统采集的边际成本递减（越挖越难），而合成数据的边际成本几乎恒定。但合成数据的质量依赖于生成模型的能力，存在"模型坍塌"风险——多代合成后输出多样性递减。最佳实践是混合使用真实数据和合成数据。

七、关联技术

  • 数据清洗(下一章)：原始数据必须经过清洗才能使用
  • Tokenization：将原始文本切分为 token
  • 合成数据(模块05)：用 LLM 生成训练数据
  • 数据合规：版权、隐私、AI法规的合规要求
💡 实用技巧：数据采集后的第一件事是做"数据画像"：统计总文档数、平均长度、语言分布、主题分布、格式分布。这些统计信息直接决定后续清洗策略——例如如果90%是英语，语言过滤的阈值可以更宽松；如果大量是代码，需要单独的代码过滤pipeline。
⚠️ 注意事项：数据采集时务必记录完整的元数据：来源URL、采集时间、许可证信息、预处理步骤。当模型出现问题需要溯源时，完善的元数据记录可以大大缩短排查时间。建议将元数据与数据文件一起存储，而非分开管理。
🔍 深入理解：Tokenization方案的选择会影响数据采集策略：使用多语言tokenizer（如SentencePiece）时，中文等语言的token效率比英文低2-3倍，这意味着需要采集更多的中文数据才能达到与英文同等的训练效果。在规划多语言数据配比时必须考虑这个因素。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

💡 实用技巧：评估不同数据源的效果时，建议在固定模型架构和训练超参的条件下，单独替换数据源进行对比实验。例如分别用Common Crawl、FineWeb、DCLM训练1B模型，对比MMLU等基准测试结果。这种控制变量实验能清晰量化数据源的质量差异。
⚠️ 注意事项：Common Crawl的数据质量随时间波动很大——某些月份的数据可能包含大量垃圾内容（如爬虫陷阱、SEO攻击）。在使用前建议先在小样本上测试质量，不要假设每个月的数据质量都是一致的。FineWeb已经做了质量筛选，可以直接使用。
🔍 深入理解：数据采集的效果最终体现在下游任务性能上。Common Crawl直接使用的效果仅是基线，经过FineWeb pipeline过滤后MMLU提升3-5%，加入代码数据后代码能力大幅提升，加入合成数据后多语言能力显著改善。这说明数据来源的组合策略比单一来源更重要。

Common Crawl的数据获取流程涉及多个技术环节。首先是爬虫阶段：从种子URL开始，通过HTTP请求下载HTML页面，解析页面中的超链接，然后沿着链接继续爬取。Common Crawl使用的是Apache Nutch爬虫框架，这是一个成熟的分布式爬虫系统。爬取的数据存储为WARC格式，这是一种标准化的网页归档格式，包含HTTP响应头和HTML内容。每月的爬取量约为3-5B个页面，原始数据量约300-500TB。WARC文件的解析需要专门的工具，Python的warcio库可以高效地读取WARC文件。从HTML中提取纯文本需要去除JavaScript、CSS、HTML标签等非文本内容，常用工具有trafilatura和readability-lxml。

代码数据的采集需要特别关注质量控制。GitHub上约有2亿个代码仓库，但其中大量是fork、自动生成的代码或质量极低的代码。BigCode项目的StarCoder Data采用了严格的过滤策略：首先过滤掉fork仓库和已知的自动生成代码，然后检查代码的语法正确性，最后按许可证过滤，只保留MIT、Apache、BSD等宽松许可证的代码。学术论文数据的获取主要依赖arXiv和Semantic Scholar。arXiv提供约250万篇开放获取的论文，S2ORC提供了约1.3亿篇论文的元数据和全文引用图谱。这些学术数据对模型的科学推理能力有显著提升，特别是在数学推导、实验设计等方面。

多语言数据的采集是一个特别具有挑战性的领域。互联网上的文本数据严重偏向英语约占60-70%，其次是中文、西班牙语、德语等。对于低资源语言如斯瓦希里语、乌尔都语等，高质量文本数据极其稀缺。Llama 3通过合成数据和翻译模型大幅提升了多语言能力，其策略是先用英语高质量数据训练核心能力，再用合成的多语言数据扩展语言覆盖。数据采集的规模规划需要考虑目标模型的参数量，7B参数的模型通常需要7T到15T tokens的训练数据，原始采集量需要是最终使用量的3到5倍。此外，数据的时效性也是需要考虑的因素，训练数据截止日期之后发生的事件和知识模型无法学习到，因此持续预训练需要定期采集新数据来更新模型的知识。

数据采集过程中还需要特别注意数据格式的标准化。不同来源的数据格式差异很大，Common Crawl提供WARC格式，GitHub代码是纯文本文件，学术论文通常是PDF或LaTeX格式，对话数据可能是JSON格式。将这些异构数据统一转换为标准化的JSONL或Parquet格式是数据工程的重要环节。转换过程中需要注意编码问题（UTF-8是标准）、换行符差异（Windows的CRLF与Unix的LF）、以及特殊字符的处理。建立一套统一的数据格式规范并严格执行，可以大大减少后续处理中的问题。数据采集的自动化程度也直接影响工程效率，建议建立定期自动化的采集任务，每周或每月自动拉取最新数据并更新数据集版本。

九、学习路径与下一步

数据采集是数据工程的第一步。掌握了数据来源和采集方法后，下一步是学习如何清洗和过滤这些原始数据。建议按照推荐顺序继续学习数据清洗模块。

在实际的数据采集工程中，需要特别关注数据的版权合规性。不同国家和地区的版权法律存在显著差异，欧盟的版权指令要求AI训练数据必须获得权利人的明确授权，而美国的合理使用原则目前在法律上仍存在争议。纽约时报起诉OpenAI的案件、Books3数据集侵权判决等事件表明，版权合规正在成为数据工程的核心约束条件。建议在采集数据时建立完善的许可证审查流程，为每个数据源记录其许可证类型、使用限制和法律风险等级。

多语言数据的采集是一个特别具有挑战性的领域。互联网上的文本数据严重偏向英语（约占60-70%），其次是中文、西班牙语、德语等。对于低资源语言（如斯瓦希里语、乌尔都语等），高质量文本数据极其稀缺。Llama 3通过合成数据和翻译模型大幅提升了多语言能力，其策略是：先用英语高质量数据训练核心能力，再用合成的多语言数据扩展语言覆盖。这种方法虽然不完美，但在资源受限的情况下是目前最实用的方案。

代码数据的采集需要特别注意许可证问题。GitHub上的代码使用各种不同的开源许可证，其中GPL许可证具有"传染性"——如果训练数据中包含GPL代码，理论上衍生的模型可能需要开源。BigCode项目的StarCoder Data采用了严格的许可证过滤策略，只保留MIT、Apache、BSD等宽松许可证的代码。这一做法虽然减少了数据量，但有效降低了法律风险。

数据采集的规模规划需要考虑目标模型的参数量。根据当前的最佳实践，7B参数的模型通常需要7T-15T tokens的训练数据（过训练比例约1000-2000倍参数量）。这意味着需要从多个数据源组合采集数据，并且要考虑数据去重和质量过滤后的数据损失。通常原始采集量需要是最终使用量的3-5倍，以确保经过清洗和去重后仍有足够的高质量数据。

⚠️ 采集注意事项：在大规模数据采集中，务必建立数据血缘追踪机制。记录每个数据文件的来源URL、采集时间、许可证信息、预处理步骤等元数据。当模型出现问题需要溯源时，完善的元数据记录可以大大缩短排查时间。此外，定期检查数据源的可用性和许可证变更情况也是必要的维护工作。

在实际的数据采集工程中还需要考虑存储架构的选择，小规模数据可以使用本地文件系统，中等规模建议使用对象存储如Amazon S3或MinIO，大规模数据则需要分布式文件系统如HDFS，存储格式的选择也很关键，JSONL适合小规模和API交互场景，Parquet适合大规模存储和查询场景，WebDataset适合流式训练场景，数据采集完成后应该立即进行数据画像分析统计文档数量平均长度语言分布主题分布等基本特征为后续的清洗和过滤提供基线参考，完善的元数据记录可以大大缩短后续排查问题的时间。

数据采集的质量评估应该在采集阶段就开始进行而不是等到清洗阶段才发现问题，建议在采集时就记录每个数据源的质量信号如网页的PageRank值GitHub仓库的star数量论文的引用次数等，这些质量信号可以在后续的质量过滤中作为特征使用，同时数据采集应该建立完善的监控机制追踪每个数据源的可用性和数据质量变化。

数据采集的法律法规合规性审查应该在采集开始前完成而不是事后补救，建议建立数据合规检查清单涵盖版权许可个人隐私保护数据跨境传输限制等方面，每个数据源都应该有明确的合规评估记录，随着AI法规的不断完善数据合规的要求也会越来越严格提前建立合规体系可以避免未来的法律风险。

数据采集的频率规划也需要考虑模型更新的需求，如果需要定期更新模型则需要建立持续的数据采集机制而不是一次性的大规模采集。

数据采集完成后应该对采集的数据进行全面的质量画像分析，包括文档长度分布语言分布主题分布等基本统计，这些分析结果将为后续的清洗和过滤策略提供重要参考依据。

数据采集的规模估算公式可以帮助合理规划资源需求，目标tokens数除以平均文档token数再乘以三到五倍的安全系数就是需要采集的原始文档数量。

def evaluate_method_applicability(scenario_constraints):
    """
    评估技术方法在不同场景下的适用性
    
    Parameters:
    scenario_constraints: dict, 各场景的约束条件
    
    Returns:
    适用性评分和建议
    """
    dimensions = ["latency", "power", "area", "cost", "complexity"]
    
    for scenario, constraints in scenario_constraints.items():
        score = 0
        suggestions = []
        
        # 延迟约束
        if constraints.get("latency_budget_ms", 100) < 10:
            score += 20
        elif constraints.get("latency_budget_ms", 100) < 100:
            score += 15
        else:
            score += 10
        
        # 功耗约束
        if constraints.get("power_budget_w", 100) < 5:
            score += 20
            suggestions.append("⚠️ 极低功耗场景需特别注意泄漏电流")
        elif constraints.get("power_budget_w", 100) < 75:
            score += 15
        else:
            score += 10
        
        # 面积约束
        if constraints.get("area_budget_mm2", 500) < 50:
            score += 20
        elif constraints.get("area_budget_mm2", 500) < 200:
            score += 15
        else:
            score += 10
        
        print(f"场景: {scenario}")
        print(f"  适用性评分: {score}/60")
        if suggestions:
            print(f"  建议: {suggestions[0]}")
        print()
