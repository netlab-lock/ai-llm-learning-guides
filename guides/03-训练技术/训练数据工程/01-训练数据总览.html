📊 01 · 训练数据总览 — 数据决定模型上限

LLM 训练数据工程的全景视图 · 生命周期 · 规模演进 · 核心挑战

一、背景与动机

在 LLM 领域有一句名言："数据决定模型的上限，算法只是逼近这个上限。" 无论模型架构多么精巧、训练技巧多么先进，如果训练数据质量差、覆盖不全，模型的表现就不可能好。

GPT-3(2020)用 300B tokens 训练，Llama 3(2024)用 15T tokens——4年增长了 50 倍。但更重要的是质量的提升：FineWeb 用精心设计的 pipeline 从 100TB 原始网页中提取出 15TB 高质量数据，效果远超直接使用 Common Crawl。

💡 核心观点：训练数据工程不是"脏活累活"，而是 LLM 竞争的核心壁垒。数据的规模、质量、多样性、配比直接决定了模型的能力边界。
⚠️ 注意事项：关于「一、背景与动机」的常见陷阱：不要盲目照搬网上的配置参数。不同的数据分布和模型规模下，LLM的最优配置可能完全不同。务必在自己的场景下做充分的验证实验。
🔍 深入理解：「一、背景与动机」的深层理解：从系统设计的角度看，LLM的设计体现了关注点分离的原则——每个组件只负责一个明确的功能，通过清晰的接口协作。这种设计哲学是构建可维护、可扩展系统的基础。

二、名字由来

训练数据(Training Data)：用于训练模型参数的数据。区别于验证集(Validation)和测试集(Test)。

数据工程(Data Engineering)：将原始数据转化为可用训练集的系统性工程，包括采集、清洗、转换、存储、版本管理等全流程。

Corpus(语料库)：NLP 领域对训练数据集的传统称呼，来自拉丁语"body"(集合体)。

💡 实用技巧：区分术语时建议查阅原始论文的定义。"Training Data"强调用于优化参数，"Validation"用于调参和早停，"Test"仅用于最终评估。三者严格隔离是实验可信的基础。Corpus一词在HuggingFace datasets库中广泛使用，熟悉其API有助于快速上手。
⚠️ 注意事项：训练数据、验证集、测试集之间的数据泄漏是初学者最常犯的错误。例如，用同一份数据既做validation又做test，会导致模型选择过拟合验证集。务必在数据分割前就规划好三者的用途。
🔍 深入理解：从词源学角度，"Corpus"从拉丁语"身体"引申为"集合体"，暗示训练数据不是零散文档的堆积，而是一个有机整体——数据的配比、多样性、质量共同塑造了模型的"体质"。

3.4 Chinchilla 后的新认知

Chinchilla Scaling Law 指出模型和数据应该等比例扩展。但后续发现：

Chinchilla 最优: tokens ≈ 20 × 参数量
推理最优: tokens ≈ 100-200 × 参数量 (如 LLaMA-7B 用 1T tokens)

这意味着 数据量比模型大小更重要——小模型+更多高质量数据 > 大模型+少量数据。

💡 实用技巧：规划训练数据规模时，用公式"目标tokens ≈ 参数量 × 过训练倍数"估算。Llama 3的7B模型用了约2T tokens（过训练约285倍），而推理场景下可提高到1000-2000倍。建议先按Chinchilla比例做baseline，再逐步增加数据量对比效果。
⚠️ 注意事项：Chinchilla Scaling Law假设数据质量恒定，但实际中高质量数据的"等效token数"远大于低质量数据。不要盲目追求token数量，Phi系列模型用1/10的数据量达到10x模型的性能，证明了质量的杠杆效应。
🔍 深入理解："数据决定模型上限"背后有信息论支撑：模型本质上是对训练数据分布的压缩编码。数据量决定压缩的信息基础，数据质量决定信号与噪声的比例，数据多样性决定编码的覆盖范围。三者共同决定了模型能力的理论上限。

💡 实用技巧：处理数据配比时，建议先按Llama 3的经验设定基线：网页60%、代码17%、书籍5%、学术5%、百科3%、对话2%、多语言8%。然后针对目标能力做消融实验——例如想提升推理能力就逐步增加代码占比，观察效果变化。
⚠️ 注意事项：调整数据配比时一次只改一个变量。例如只调整代码占比从10%到17%，观察对代码能力、推理能力、通用能力的影响，再决定下一步调整方向。同时改多个配比会导致无法归因效果变化。
🔍 深入理解：数据质量与数据规模的权衡曲线并非线性：FineWeb从100TB原始网页中提取15TB高质量数据，效果远超使用全部100TB。这说明在质量-规模的权衡中，前期的质量提升投入回报率极高，但存在边际递减效应——过滤到一定程度后，继续收紧规则会开始丢失有价值的多样性数据。

五、使用场景

  • 预训练(Pretraining)：从零训练基座模型，需要 TB 级高质量数据
  • 持续预训练(Continual Pretraining)：在已有模型上继续训练，注入新知识或领域技能
  • 微调(SFT)：指令微调需要高质量的 prompt-response 对
  • 对齐(RLHF/DPO)：偏好数据(人类排序的多个回答)
  • 评测数据：构建抗污染的评测集
💡 实用技巧：不同训练阶段对数据的要求差异很大：预训练需要TB级无标注数据，SFT需要万级高质量prompt-response对，RLHF需要人类排序的偏好数据。建议先明确目标阶段，再规划数据采集策略，避免用预训练的标准去做SFT数据。
⚠️ 注意事项：持续预训练(Continual Pretraining)容易导致灾难性遗忘——新知识覆盖旧知识。建议在新数据中混入5-10%的旧数据，并在训练过程中监控旧任务的性能变化。
🔍 深入理解：SFT数据的"质量杠杆效应"比预训练更显著：LIMA论文证明仅用1000条精心筛选的SFT数据就能达到不错效果。这说明在微调阶段，每条数据的边际价值远高于预训练阶段，人工审核的投入回报率极高。

💡 实用技巧：选择数据格式时的实用建议：预训练大规模数据推荐Parquet格式（压缩率高、支持列式读取），SFT小规模数据用JSONL（便于人工查看和编辑），流式训练用WebDataset（避免随机IO）。格式选错会导致训练效率下降10倍以上。
⚠️ 注意事项：LLM预训练数据的标注方式是"自监督"(下一token预测)，这与传统NLP的人工标注有本质区别。不要用监督学习的数据质量标准（如每条都审核）来要求预训练数据，否则成本不可承受。预训练数据用统计过滤即可。
🔍 深入理解：数据工程的演进方向：传统NLP（MB级、人工标注、单任务）→ LLM预训练（TB级、自监督、全领域）→ 未来趋势（合成数据+多模态+自动化pipeline）。每个阶段的数据质量标准和处理方法都在变化，但"数据决定上限"的核心理念不变。

七、关联技术

  • Scaling Laws：数据规模与模型性能的定量关系
  • Tokenization：将文本切分为 token 的方法(BPE, SentencePiece)
  • 数据并行：分布式训练中的数据分割策略
  • 混合精度训练：FP16/BF16 对数据表示的影响
💡 实用技巧：Tokenization对数据工程的影响常被低估：BPE词表大小（如32K vs 128K）直接影响token效率，进而影响相同数据量的有效信息密度。建议先确定tokenization方案，再规划数据规模。混合精度训练(BF16)不影响数据准备，但会影响训练稳定性。
⚠️ 注意事项：数据并行训练中，各GPU上的数据分布必须一致，否则会导致模型参数不同步。使用不同的随机种子打乱数据时，确保所有worker使用相同的种子。数据顺序对最终模型性能有显著影响（GPT-3论文中有详细消融实验）。
🔍 深入理解：Scaling Law揭示了数据量、模型大小、计算量三者的幂律关系。但这个关系的系数取决于数据质量——高质量数据可以将幂律曲线整体上移。这意味着在数据工程上的投入可以减少对模型规模的依赖，用更小的模型+更好的数据达到同等效果。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

从工程实践的角度来看，数据采集的成本结构也需要仔细规划。以一个7B参数模型的训练数据准备为例，需要约10T tokens的数据。假设从Common Crawl采集原始数据约300TB，经过文本提取后约50TB，语言过滤后约30TB，质量过滤后约15TB，去重后约12TB，最终可用约10TB。这个过程中，数据下载的带宽成本约200美元，计算资源约1000美元，去重约2000美元，总计约3500美元。相比之下，训练一个7B模型的GPU成本可能高达数万到数十万美元，数据工程的成本仅占训练总成本的5-10%，但对模型质量的影响却远超这个比例。

数据配比的优化是一个需要反复实验的过程。从Llama系列的演进可以看出，代码数据的占比从Llama 1的4.5%逐步提升到Llama 3的17%，这一变化背后有深刻的工程洞察：代码数据不仅提升编程能力，还显著增强了逻辑推理和数学能力。这是因为代码具有严格的语法结构和逻辑关系，学习代码本质上是在学习结构化的思维模式。同时，代码中的注释和函数签名天然形成了指令-响应对，有助于提升模型的指令遵循能力。这一发现深刻改变了业界对数据配比的认知，15-17%的代码配比已经成为2024年后的行业标准。

合成数据的使用正在成为突破真实数据天花板的关键路径。Llama 3大量使用合成数据来增强多语言和代码能力，DeepSeek-V3在数学和推理领域也大量采用合成数据。合成数据的核心优势在于可以精确控制数据的难度分布、主题覆盖和质量标准。但需要注意模型坍塌风险——如果连续多代使用合成数据训练，模型的输出多样性会逐代递减，最终退化为重复的模板文本。预防坍塌的核心策略是始终保持至少30-50%的真实数据混入，并定期用新真实数据刷新训练集。

数据版本管理在工业级数据工程中至关重要。每次修改数据处理pipeline的参数或规则，都应该创建新的数据版本并记录变更内容。DVC提供了类似Git的数据版本控制能力，可以追踪数据文件的变化、记录处理参数、支持回滚到历史版本。建议同时使用DVC管理私有数据的版本，使用HuggingFace Hub托管开源数据集。数据质量监控是确保pipeline稳定运行的基础设施，一个好的监控系统应该能够实时追踪数据量变化、文本长度分布、语言分布、去重率和质量分数分布等关键指标。当任何指标出现异常时，监控系统应该自动发出告警，提醒工程师检查pipeline的某个环节是否出了问题。数据血缘追踪也很重要，当模型效果变差时，能够快速定位是哪个数据处理步骤引入了问题。

训练数据工程的成本优化可以从多个角度入手。首先，选择合适的计算框架可以显著降低成本，对于单机可以处理的数据Polars比Spark快十倍以上。其次，使用列式存储格式Parquet可以减少IO开销和存储成本，Parquet的压缩率通常是JSONL的三到五倍。第三，流式处理可以避免将全部数据加载到内存中，显著降低内存成本。第四，缓存中间结果可以避免重复计算。在典型的预训练数据准备流程中，去重环节是最大的成本中心，因此选择高效的去重算法和工具链对于控制成本至关重要。端到端的数据处理通常需要一到两周时间和数千美元的云计算成本。

从数据总览到工程实践的完整知识图谱

训练数据工程是一门实践性极强的学科。仅仅理解理论是不够的，还需要在真实数据上反复实践。建议在学习每个模块后，动手完成对应的练习题，并尝试在小规模数据上复现核心流程。

深入理解数据规模演进的规律对于工程决策至关重要。从GPT-2的10B tokens到Llama 3的15T tokens，数据规模增长了1500倍，但模型参数量只增长了约200倍。这意味着每个参数平均看到的数据量增加了约7.5倍。这种"过训练"策略的核心驱动力是推理成本优化：用更多数据训练更小的模型，使得最终部署时的推理成本更低。Chinchilla定律最初建议参数量和数据量等比例扩展，但在实际生产中，推理成本往往远超训练成本，因此用更多数据训练更小模型成为主流做法。

数据质量的"杠杆效应"在实践中得到了充分验证。Phi系列模型使用约1/10的数据量达到了10倍参数量模型的性能水平，这一现象深刻说明了数据质量的重要性远超数据规模。在实际工程中，我们应该将80%的精力放在数据质量提升上，而不是盲目扩大数据规模。FineWeb团队从100TB原始网页中提取15TB高质量数据的过程，充分展示了系统化数据清洗pipeline的价值。

数据多样性的工程实现需要精细化的领域分类和配比策略。不同领域的数据对模型能力的影响是不对称的：网页数据主要贡献通用语言能力，代码数据不仅提升编程能力还显著增强逻辑推理能力，学术论文贡献严谨的科学推理能力，书籍数据贡献长文本理解和深度知识。Llama 3将代码占比从4.5%提升到17%，结果发现通用能力不降反升，这一发现深刻改变了业界对数据配比的认知。

训练数据工程的成本结构分析对于资源有限的团队尤为重要。在典型的预训练数据准备流程中，数据下载约占总成本的5%，文本提取约占15%，语言过滤约占10%，质量过滤约占10%，去重约占50%（MinHash计算开销大），去污染约占5%，格式转换和打包约占5%。去重环节是最大的成本中心，因此选择高效的去重算法和工具链对于控制成本至关重要。

💡 实用建议：下一步建议：完成本章练习题后，继续学习02-数据采集与来源，了解Common Crawl等主要数据源的技术细节。建议同时安装HuggingFace datasets库，在本地环境中加载一个开源数据集（如FineWeb的子集），实际感受数据的格式和内容特征。
⚠️ 常见误区：初学者容易陷入"数据越多越好"的误区。研究表明，未经去重和质量筛选的大规模数据反而会降低模型性能。Common Crawl中约30-50%的内容是重复或近重复的，约90%是低质量内容。盲目扩大数据规模而不做质量控制，是训练数据工程中最常见也最昂贵的错误。
🔬 前沿思考：数据工程正在从"手工作坊"走向"工业化生产"。2024-2025年的趋势表明，合成数据将成为突破真实数据天花板的关键路径，但需要警惕模型坍塌风险。多模态数据工程（图像-文本、视频-文本配对数据）是下一个重要的研究方向。同时，数据合规和版权问题将越来越严格地约束数据工程的实践。

数据血缘追踪也很重要，当模型效果变差时能够快速定位是哪个数据处理步骤引入了问题。训练数据工程的成本优化可以从多个角度入手，首先选择合适的计算框架可以显著降低成本，对于单机可以处理的数据Polars比Spark快十倍以上，其次使用列式存储格式Parquet可以减少IO开销和存储成本，Parquet的压缩率通常是JSONL的三到五倍，第三流式处理可以避免将全部数据加载到内存中，第四缓存中间结果可以避免重复计算，在典型的预训练数据准备流程中去重环节是最大的成本中心，端到端的数据处理通常需要一到两周时间和数千美元的云计算成本，建议在小规模数据上完成端到端测试后再扩展到全量数据避免因pipeline错误导致原始数据丢失或资源浪费。

# 训练数据规模估算公式（实际工程中常用的规划方法）

目标tokens = 模型参数量 × 过训练倍数
# 示例: 7B模型, 过训练2000倍
target_tokens = 7e9 * 2000  # = 14T tokens

# 考虑清洗损耗率（通常保留20-40%）
采集量 = target_tokens / 平均每文档token数 / 保留率
# 例: 平均500 tokens/文档, 保留率30%
docs_needed = 14e12 / 500 / 0.3  # ≈ 93B 文档

# 成本估算
cost_per_tb = 50  # 美元/TB（含下载+处理）
total_tb = docs_needed * 500 * 4 / 1e12  # 约4.6 PB原始数据
storage_cost = total_tb * cost_per_tb  # 约23万美元（仅存储）

def evaluate_method_applicability(scenario_constraints):
    """
    评估技术方法在不同场景下的适用性
    
    Parameters:
    scenario_constraints: dict, 各场景的约束条件
    
    Returns:
    适用性评分和建议
    """
    dimensions = ["latency", "power", "area", "cost", "complexity"]
    
    for scenario, constraints in scenario_constraints.items():
        score = 0
        suggestions = []
        
        # 延迟约束
        if constraints.get("latency_budget_ms", 100) < 10:
            score += 20
        elif constraints.get("latency_budget_ms", 100) < 100:
            score += 15
        else:
            score += 10
        
        # 功耗约束
        if constraints.get("power_budget_w", 100) < 5:
            score += 20
            suggestions.append("⚠️ 极低功耗场景需特别注意泄漏电流")
        elif constraints.get("power_budget_w", 100) < 75:
            score += 15
        else:
            score += 10
        
        # 面积约束
        if constraints.get("area_budget_mm2", 500) < 50:
            score += 20
        elif constraints.get("area_budget_mm2", 500) < 200:
            score += 15
        else:
            score += 10
        
        print(f"场景: {scenario}")
        print(f"  适用性评分: {score}/60")
        if suggestions:
            print(f"  建议: {suggestions[0]}")
        print()
