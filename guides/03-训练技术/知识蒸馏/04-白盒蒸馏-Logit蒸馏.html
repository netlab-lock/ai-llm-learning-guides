🎯 04 · 白盒蒸馏 (Logit蒸馏)

白盒蒸馏可以访问 teacher 的 logits，因此可以进行更精细的知识迁移。在 LLM 时代，这通常意味着你有 teacher 模型的完整权重。

1. 白盒蒸馏的基础

1.1 什么是 Logit

Logit 是模型在 softmax 之前的原始输出分数：

1.2 白盒 vs 黑盒的信息量

维度白盒蒸馏黑盒蒸馏
信息粒度整个词表的概率分布只有采样结果
暗知识可以学习无法学习
训练效率更高较低
实现难度需要 teacher 权重只需要 API
💡 实用技巧：白盒蒸馏的核心前提是能访问 Teacher 的完整 logits。实操建议：先用 HuggingFace Transformers 的 output_hidden_states=True 参数获取 Teacher 的 logits 输出，确认维度与 Student 词表对齐后再写训练循环。如果 Teacher 和 Student 的 tokenizer 不同，需要先建立词表映射矩阵。
⚠️ 注意事项：白盒蒸馏最容易踩的坑是 Teacher 和 Student 的 tokenizer 不一致。如果两者词表不同，直接计算 KL 散度会得到无意义的结果。解决方案：(1) 选择同架构模型（如 Llama→Llama）；(2) 建立词表映射矩阵，将 Teacher logits 投影到 Student 词表空间。
🔍 深入理解：白盒蒸馏相比黑盒蒸馏的信息论优势在于：黑盒蒸馏每个 token 只获得 1 bit 信息（采样结果），而白盒蒸馏获得整个词表的概率分布（约 log₂|V| bits）。对于 32K 词表的模型，白盒蒸馏每个 token 的信息量是黑盒的 ~15 万倍，这就是白盒蒸馏数据效率更高的根本原因。

2. 序列级蒸馏 (Sequence-level KD)

2.1 原理

LLM 是自回归模型，蒸馏需要在序列级别进行：

序列级蒸馏公式

L_KD = Σ_t KL(p_teacher(·|x, y_<t), p_student(·|x, y_<t))

其中:
p_teacher(·|x, y_<t) = teacher 在位置 t 的词表分布
p_student(·|x, y_<t) = student 在位置 t 的词表分布
KL = KL 散度

2.2 序列级蒸馏的图解

2.3 KL 散度 vs 交叉熵

损失函数公式特点适用场景
KL 散度KL(p_t || p_s)匹配分布形状分布匹配
交叉熵H(p_t, p_s)最小化 teacher 分布下的 student 负对数似然标签学习
💡 实用技巧：序列级蒸馏的关键实现细节：(1) 使用 torch.nn.functional.kl_div 时输入应为 log-probabilities，target 为 probabilities；(2) 掩码 padding token 的 KL 损失，避免无效位置干扰训练；(3) 每个 token 位置独立计算 KL 后取平均，而非对整个序列求和后再归一化。
⚠️ 注意事项：序列级蒸馏中一个隐蔽的错误是使用 teacher-forced ground truth 作为 Student 的输入序列。正确做法是让 Student 自回归生成（on-policy），或者至少混入部分 Student 生成的序列（off-policy + on-policy 混合），否则 Student 从未在自己的错误分布上训练过。
🔍 深入理解：序列级蒸馏的本质是让 Student 在每个 token 位置都逼近 Teacher 的完整输出分布。KL 散度 KL(p_teacher ‖ p_student) 衡量的是「用 Student 分布编码 Teacher 分布所需的额外比特数」。而交叉熵 H(p_teacher, p_student) = H(p_teacher) + KL(p_teacher ‖ p_student)，在 Teacher 固定时两者等价。

3. On-policy vs Off-policy 蒸馏

3.1 Off-policy 蒸馏

Off-policy：用 teacher 生成的数据训练 student。

3.2 On-policy 蒸馏

On-policy：student 自己生成数据，teacher 打分。

3.3 效果对比

方法数据分布学习效率效果实现复杂度
Off-policyTeacher 分布较低好低
On-policyStudent 分布较高更好中
💡 实用技巧：On-policy vs Off-policy 的选择策略：起步阶段用 Off-policy（实现简单，只需预生成 Teacher 数据），验证流程正确后切换到 On-policy。On-policy 的实现关键是在训练循环中让 Student 先 model.generate() 生成序列，再将生成的序列输入 Teacher 获取 logits 分布。
⚠️ 注意事项：On-policy 蒸馏的训练不稳定风险：Student 生成的序列质量差时，Teacher 给出的 logits 可能包含大量低质量 token 的分布，导致 Student 学到错误模式。解决方案：(1) 使用 top-p/top-k 采样限制生成质量；(2) 设置 reward 阈值过滤低质量样本；(3) 混合 ground truth 序列稳定训练。
🔍 深入理解：On-policy vs Off-policy 的本质区别在于「分布偏移」(distribution shift)。Off-policy 下 Student 学习的是 Teacher 在 Teacher 分布上的输出，但推理时 Student 处于自己的分布上——训练和推理的输入分布不匹配。On-policy 通过让 Student 自己生成文本、Teacher 打分，消除了这个 gap，这也是 GKD 论文的核心贡献。

4. GKD (Generalized Knowledge Distillation)

4.1 原理

GKD (Agarwal et al. 2024) 是一个统一的蒸馏框架：

GKD 公式

L_GKD = Σ_t KL(p_teacher(·|x, ŷ_<t), p_student(·|x, ŷ_<t))

其中 ŷ_<t 是从 student 分布中采样的 token

关键: ŷ 从 student 分布采样 (on-policy)
而非从 teacher 分布采样 (off-policy)

4.2 GKD 的优势

  • 统一框架：包含了 on-policy 和 off-policy 作为特例
  • 分布匹配：student 学习在自己会遇到的分布上匹配 teacher
  • 灵活性：可以调整采样策略
  • 理论保证：有更好的理论收敛性

4.3 GKD 的采样策略

采样策略说明效果
Student 采样从 student 分布采样On-policy，效果最好
Teacher 采样从 teacher 分布采样Off-policy，效果较差
混合采样混合两种采样平衡效果和稳定性
💡 实用技巧：GKD 的实现建议：使用 TRL 库的 DistillationTrainer 可以快速上手。关键超参数：lmbda（混合比例，控制 KL 损失和 ground truth CE 损失的权重）、temperature（通常 2~5）、采样策略（推荐 Student 采样即 on-policy）。建议先用 lmbda=0.5, temperature=2.0 作为基线。
⚠️ 注意事项：GKD 的常见陷阱是温度参数选择不当。温度过低（T<1）会导致分布过于尖锐，KL 梯度消失；温度过高（T>10）会导致分布过于平滑，丧失有意义的区分信号。建议在 2~5 范围内网格搜索，以验证集上的 KL 散度和下游任务性能为指标。
🔍 深入理解：GKD 的理论优雅之处在于它将 off-policy（λ=0）和 on-policy（λ=1）统一为一个框架。λ 参数控制「从 Teacher 数据学」和「从自己数据学」的混合比例。实验表明 λ=1（纯 on-policy）效果最好，但训练更不稳定；实际工程中常用 λ=0.5~0.7 兼顾效果和稳定性。

5. Self-Imitation 蒸馏

5.1 原理

Self-Imitation 蒸馏让 student 从自己的好样本中学习：

💡 实用技巧：Self-Imitation 蒸馏的核心是「从自己的好样本中学习」。实现要点：(1) 用 Reward Model 或规则评分筛选 Student 的高质量生成样本；(2) 只对高分样本计算蒸馏损失，低分样本丢弃；(3) 维护一个 replay buffer 存储历史高分样本，避免每次重新生成。这种方法特别适合无法获取 Teacher 权重的场景。
⚠️ 注意事项：Self-Imitation 蒸馏的陷阱是「自我强化循环」：如果初始生成质量差，筛选出的「好样本」也只是相对好，Student 可能陷入局部最优。解决方案：定期引入外部信号（Teacher logits、人工标注、或更强的 Reward Model）打破循环，防止模式坍塌。
🔍 深入理解：Self-Imitation 蒸馏与 RLHF 中的 Rejection Sampling 有深层联系：两者都是「生成→筛选→学习」的循环。区别在于 RLHF 用人类偏好筛选，Self-Imitation 用 Reward Model 筛选。从强化学习视角看，Self-Imitation 本质上是一种 offline RL——从历史轨迹中筛选好的子集进行策略优化。

💡 实用技巧：选择白盒还是黑盒蒸馏的决策树：(1) 有 Teacher 权重且显存足够 → 白盒蒸馏，效果最好；(2) 只有 API 访问 → 黑盒蒸馏（CoT 蒸馏）；(3) 想要最佳效果 → 白盒 + 黑盒混合：用白盒蒸馏匹配分布，用黑盒蒸馏（CoT）学习推理过程。实际项目中，白盒蒸馏的数据效率通常是黑盒的 3-5 倍。
⚠️ 注意事项：不要认为白盒蒸馏一定优于黑盒蒸馏。对于 LLM 来说，CoT 蒸馏（黑盒，文本级）的效果常常优于纯 Logit 蒸馏（白盒），因为推理过程本身就是最重要的知识。白盒蒸馏的优势在于「暗知识」——Teacher 分布中非 top-1 token 的概率信息，这在分类任务中价值巨大，但在生成任务中可能不如 CoT。
🔍 深入理解：白盒 vs 黑盒蒸馏的本质区别是「监督信号的粒度」。白盒蒸馏在每个 token 位置提供 |V| 维的软标签，黑盒蒸馏只提供 1 个 token 的硬标签。但从信息论角度，最有价值的信息往往集中在 top-k 个 token 中（top-5 通常覆盖 90%+ 的概率质量），这解释了为什么「top-k logits 蒸馏」可以用很小的 k 实现接近全词表蒸馏的效果。

7. 实现细节

7.1 温度参数 T

温度 T 控制 softmax 分布的平滑程度：

温度缩放:

p_i = exp(z_i/T) / Σ exp(z_j/T)

T=1: 标准 softmax
T>1: 更平滑的分布（更多暗知识）
T<1: 更尖锐的分布（更接近 one-hot）

7.2 权重系数 α

总损失通常是 KD 损失和 CE 损失的加权组合：

L_total = α · L_KD + (1-α) · L_CE

L_KD = KL(teacher || student)
L_CE = CrossEntropy(student, ground_truth)

α 的选择:
α = 0.5: 平衡 KD 和 CE
α = 0.7: 更重视 KD
α = 0.3: 更重视 CE
💡 实用技巧：温度 T 和权重 α 的调优策略：(1) 先固定 α=0.5，搜索最优 T（推荐范围 2~5）；(2) 再固定最优 T，搜索 α（推荐范围 0.3~0.7）；(3) 记录每组参数的验证集 KL 散度和下游任务准确率。经验值：α=0.7 + T=3.0 是一个不错的起点，适用于大多数场景。
⚠️ 注意事项：实现细节中一个容易忽视的问题：KL 散度的数值稳定性。当 Student 分布接近 0 时，log(0) 会导致 NaN。解决方案：(1) 使用 torch.nn.functional.kl_div(log_target=False) 确保输入正确；(2) 对 Student logits 加 epsilon 平滑；(3) 使用 torch.clamp 限制概率下界为 1e-8。
🔍 深入理解：温度 T 的深层作用：T 控制 softmax 分布的「熵」。T→0 时分布退化为 one-hot（只保留 argmax 信息），T→∞ 时分布趋向均匀（丧失所有区分信息）。最优 T 使得分布的熵恰好处于「保留足够暗知识」和「不丧失区分度」的平衡点。Hinton 原始论文建议 T=5，但 LLM 场景中 T=2~3 通常更优。

8. 最新进展 (2025-2026)

8.1 在线白盒蒸馏

Teacher 和 Student 同时训练，实时蒸馏：

  • 优势：不需要预训练好的 Teacher
  • 方法：Teacher 和 Student 共享部分参数
  • 代表：Deep Mutual Learning

8.2 蒸馏 + RLHF

在蒸馏过程中加入人类反馈：

L_total = α · L_KD + β · L_RLHF + γ · L_SFT
通俗类比：On-policy vs Off-policy 蒸馏就像两种学习方式：Off-policy 是"看标准答案学习"（老师提前准备好答案），On-policy 是"做题后看批改"（自己先做，老师再纠正）。做题后看批改学得更扎实，因为老师针对的是你自己的错误，而不是泛泛的标准答案。这就是为什么 GKD（On-policy）通常优于传统蒸馏（Off-policy）。
常见误区：① 认为 Logit 蒸馏一定优于纯文本蒸馏——实际上，对于 LLM 来说，CoT 蒸馏（文本级）的效果常常优于 Logit 蒸馏，因为推理过程本身就是最重要的知识；② 忽略温度参数对 Logit 蒸馏的影响——温度 T 控制 softmax 的平滑程度，T=2~5 通常效果最好，但需要根据具体任务调优。
深入探讨：On-policy 蒸馏的核心优势是避免了"分布偏移"（distribution shift）问题。在 Off-policy 蒸馏中，学生学习的是老师在老师自己的分布上的输出，但学生实际生成时的分布与老师不同，导致训练和推理之间存在 gap。GKD 通过让学生自己生成文本、老师打分的方式，消除了这个 gap。但 On-policy 的代价是更高的计算成本——每一步都需要学生先生成文本，再用老师打分。这引出了一个工程上的权衡：在训练效率和最终效果之间找到平衡点。

9. 白盒蒸馏的高级技巧

9.1 分层蒸馏策略

分层蒸馏是白盒蒸馏中的重要技术，它不仅匹配最终输出层的概率分布，还匹配中间隐藏层的表示。这种方法可以让 Student 学到 Teacher 的内部知识表示，而不仅仅是输出行为。在 LLM 场景中，由于 Transformer 的每一层都在构建不同层次的语义表示，分层蒸馏可以让 Student 更好地理解语言的层次结构。

# 分层蒸馏实现
import torch
import torch.nn as nn

class LayerWiseDistillation(nn.Module):
    def __init__(self, teacher, student, layer_mapping):
        super().__init__()
        self.teacher = teacher
        self.student = student
        # 层映射: Teacher层 → Student层
        self.layer_mapping = layer_mapping  # e.g., {0:0, 4:1, 8:2, 11:3}
        # 维度对齐投影
        self.projectors = nn.ModuleDict({
            str(s): nn.Linear(teacher.config.hidden_size,
                             student.config.hidden_size)
            for _, s in layer_mapping.items()
        })

    def compute_layer_loss(self, teacher_hidden, student_hidden, layer_idx):
        """计算单层的蒸馏损失"""
        s_layer = self.layer_mapping[layer_idx]
        projected = self.projectors[str(s_layer)](teacher_hidden)
        loss = F.mse_loss(student_hidden, projected)
        return loss

    def forward(self, input_ids, labels):
        # 获取中间层表示
        teacher_out = self.teacher(input_ids, output_hidden_states=True)
        student_out = self.student(input_ids, output_hidden_states=True)
        # 计算分层损失
        layer_loss = 0
        for t_layer, s_layer in self.layer_mapping.items():
            layer_loss += self.compute_layer_loss(
                teacher_out.hidden_states[t_layer],
                student_out.hidden_states[s_layer],
                t_layer
            )
        return layer_loss / len(self.layer_mapping)

10. 白盒蒸馏的工程实践

10.1 显存优化策略

白盒蒸馏需要同时加载 Teacher 和 Student 两个模型，显存需求是单独训练 Student 的两倍以上。通过梯度检查点、混合精度训练和模型并行等技术，可以在有限的 GPU 资源上完成大规模的白盒蒸馏。

优化技术显存节省速度影响适用场景
FP16 混合精度~50%几乎无影响所有场景
梯度检查点~60%慢 20-30%显存不足
Teacher 量化~75%几乎无影响Teacher 过大
模型并行按 GPU 数通信开销多 GPU 环境
显存优化建议：如果显存不足以同时加载 Teacher 和 Student，推荐使用 Teacher 量化（INT8/INT4）来降低显存需求。对于 70B 的 Teacher，INT4 量化后只需约 35GB 显存，加上 7B 的 Student（约 14GB），单张 80GB A100 即可完成训练。如果只有消费级 GPU，可以使用 DeepSpeed ZeRO-3 进行模型并行。
工程陷阱：白盒蒸馏中一个容易忽视的问题是 Teacher 和 Student 的 Tokenizer 不一致。如果两者使用不同的词表，需要在 logits 层面做映射对齐，否则蒸馏损失的计算毫无意义。建议尽量选择相同架构、相同 Tokenizer 的 Teacher-Student 组合。

def evaluate_method_applicability(scenario_constraints):
    """
    评估技术方法在不同场景下的适用性
    
    Parameters:
    scenario_constraints: dict, 各场景的约束条件
    
    Returns:
    适用性评分和建议
    """
    dimensions = ["latency", "power", "area", "cost", "complexity"]
    
    for scenario, constraints in scenario_constraints.items():
        score = 0
        suggestions = []
        
        # 延迟约束
        if constraints.get("latency_budget_ms", 100) < 10:
            score += 20
        elif constraints.get("latency_budget_ms", 100) < 100:
            score += 15
        else:
            score += 10
        
        # 功耗约束
        if constraints.get("power_budget_w", 100) < 5:
            score += 20
            suggestions.append("⚠️ 极低功耗场景需特别注意泄漏电流")
        elif constraints.get("power_budget_w", 100) < 75:
            score += 15
        else:
            score += 10
        
        # 面积约束
        if constraints.get("area_budget_mm2", 500) < 50:
            score += 20
        elif constraints.get("area_budget_mm2", 500) < 200:
            score += 15
        else:
            score += 10
        
        print(f"场景: {scenario}")
        print(f"  适用性评分: {score}/60")
        if suggestions:
            print(f"  建议: {suggestions[0]}")
        print()
