1. 合成数据概述

合成数据（Synthetic Data）是指使用模型（通常是强模型）自动生成的训练数据，而非来自真实人类创作。随着高质量自然数据逐渐耗尽，合成数据已成为LLM训练的核心数据来源。Epoch AI的分析估计，高质量语言数据将在2026-2032年间被消耗殆尽，这使得合成数据不仅是一种技术选择，更是一种战略必然。

合成数据的核心价值在于其可控性和可扩展性。与自然数据相比，合成数据的难度、风格、主题、语言等属性可以被精确控制，这使得模型训练从"有什么数据用什么"的被动模式转变为"需要什么数据生成什么"的主动模式。此外，合成数据的生成成本远低于人工标注——使用GPT-4生成一条高质量指令数据的成本约为0.001-0.01美元，而人工标注同等质量的数据可能需要1-10美元。

📖 关键论文与里程碑

Microsoft的"Textbooks Are All You Need"（Gunasekar et al., 2023）是合成数据领域的里程碑工作，证明了用GPT-4生成的"教科书级"数据训练的1.3B模型（Phi-1）在代码生成上可媲美10x规模模型。随后的Phi-1.5、Phi-2、Phi-3系列进一步验证了这一范式。Meta的LLaMA-3报告也证实，合成数据在其训练中占据了约30%的比例。DeepSeek-V2和Qwen-2的技术报告同样强调了合成数据在特定能力维度（数学、代码、推理）提升中的关键作用。

💡 实用技巧：在开始合成数据项目前，建议先明确目标能力维度（如数学推理、代码生成、指令跟随等），然后选择最适合该维度的合成方法。不要试图用一种方法生成覆盖所有维度的数据——不同维度需要不同的生成策略和质量控制标准。例如，数学数据需要答案验证，代码数据需要测试用例，而对话数据更依赖模型打分。
⚠️ 注意事项：合成数据不是万能的。对于需要真实世界知识的任务（如时效性信息、小众领域知识），合成数据的质量往往不如人工收集的自然数据。此外，过度依赖合成数据可能导致"信息茧房"——模型只学会了生成模式而缺乏真正的知识理解。建议在合成数据中始终保留一定比例的高质量自然数据（至少20-30%）作为"锚点"。

2. Self-Instruct方法

Self-Instruct（Wang et al., 2023）是最早的系统化合成数据方法之一，也是后续诸多方法的基础。其核心创新在于让模型自己生成训练数据，从而突破人工标注的规模限制。

💡 Stanford Alpaca的突破

Stanford Alpaca项目使用Self-Instruct方法，仅用52K条GPT-3.5生成的指令数据微调LLaMA-7B，就获得了接近GPT-3.5的对话能力，成本仅约500美元。这一结果震动了整个AI社区，证明了合成数据的巨大潜力。然而，后续研究也发现Alpaca的数据质量存在较大波动——约20%的样本质量较低，这推动了更精细的质量控制方法的发展。

2.3 Self-Instruct的局限性

Self-Instruct虽然开创性地证明了合成数据的可行性，但也存在明显的局限：

  • 多样性不足：由于种子集较小（175条），生成的指令在主题和风格上存在明显的"模式塌缩"
  • 难度天花板：生成的指令大多为中低难度，难以覆盖需要深度推理的复杂任务
  • 知识盲区：无法生成模型本身不具备的知识，如最新的时事信息
  • 质量波动大：缺少有效的质量控制机制，产出数据的质量方差较大
🔍 深入理解：Self-Instruct的核心假设是"模型能够生成与训练数据质量相当的指令"。这在模型能力较弱时（如GPT-3.5级别）是有问题的——模型生成的数据质量不可能超过其自身能力上限。然而，当使用更强的模型（如GPT-4或Claude）来生成数据时，"学生"（被训练的小模型）可以从"老师"（生成数据的大模型）那里学到更高质量的模式。这就是知识蒸馏与Self-Instruct的交叉点——Self-Instruct本质上是一种"自我蒸馏"，而使用强模型生成数据则是一种"跨模型蒸馏"。详见01-数据飞轮方法论中关于蒸馏的讨论。

3. Evol-Instruct方法

Evol-Instruct（Xu et al., 2023）由WizardLM团队提出，核心创新是"指令进化"——通过多轮演化逐步增加指令复杂度。这种方法解决了Self-Instruct生成数据难度天花板的问题。

3.1 进化策略详解

进化策略描述具体操作示例变化
深度进化-增加约束添加具体约束条件添加时间/空间复杂度、代码风格、输出格式等要求「写排序算法」→「用O(nlogn)时间O(1)空间原地排序」
深度进化-增加推理要求逐步推理过程添加"让我们一步步思考"或要求解释思路「计算」→「先分析再计算，给出每步推理」
广度进化-扩展领域扩展任务类型和领域将技术问题转化为跨学科问题「写排序算法」→「设计数据库索引策略」
广度进化-增加变体生成同一任务的不同变体改变输入类型、输出要求、应用场景「Python排序」→「Rust排序」「GPU并行排序」
融合进化合并多个简单任务为复杂任务将2-3个相关任务组合为一个多步骤任务「排序」+「去重」→「排序并去重，输出统计信息」

3.2 Evol-Instruct的实现

Evol-Instruct进化实现：

def evolve_instruction(instruction, strategy="deepen", max_rounds=5):
    """对一条指令进行多轮进化"""
    current = instruction
    history = [instruction]
    
    for round_idx in range(max_rounds):
        if strategy == "deepen":
            prompt = f"""请将以下指令变得更复杂，增加更多约束条件和推理要求。
            
原始指令：{current}

更复杂的指令（保持任务本质不变，增加难度）："""
        
        elif strategy == "broaden":
            prompt = f"""请将以下指令扩展到更广泛的应用领域。
            
原始指令：{current}

扩展后的指令（改变应用场景，但保持类似的技能要求）："""
        
        elif strategy == "fuse":
            prompt = f"""请将以下指令与一个相关的不同任务融合，创建一个更复杂的复合任务。
            
原始指令：{current}

融合后的复合指令："""
        
        evolved = llm_generate(prompt)
        
        # 验证进化是否有效
        if is_valid_evolution(current, evolved):
            current = evolved
            history.append(evolved)
        else:
            # 如果进化失败，尝试另一种策略
            strategy = random.choice(["deepen", "broaden", "fuse"])
    
    return current, history

def is_valid_evolution(original, evolved):
    """验证进化后的指令是否合理"""
    # 检查1: 进化后的指令应该更长（更复杂）
    if len(evolved) < len(original) * 0.8:
        return False
    # 检查2: 不应该完全偏离原始任务
    similarity = compute_semantic_similarity(original, evolved)
    if similarity < 0.3:
        return False
    return True
Evol-Instruct进化过程可视化： Round 0: "写一个排序算法" │ │ 深度进化(+约束) ▼ Round 1: "用Python实现一个时间复杂度为O(nlogn)的排序算法" │ │ 深度进化(+推理) ▼ Round 2: "用Python实现O(nlogn)排序，解释每步选择该算法的原因， 并分析最好/最坏/平均时间复杂度" │ │ 广度进化(+领域) ▼ Round 3: "用Python实现适用于大规模数据(>10GB)的外部排序算法， 说明磁盘I/O优化策略，并分析内存使用情况" │ │ 融合进化(+复合) ▼ Round 4: "实现一个支持多字段排序、去重、分页的大规模数据处理 pipeline，要求支持CSV/JSON/Parquet格式， 内存使用不超过1GB，输出处理统计报告"
⚠️ 注意事项：Evol-Instruct的进化轮数需要谨慎控制。过多的进化会导致指令变得不自然、过于复杂，甚至偏离人类实际需求。WizardLM团队的实验表明，3-5轮进化通常是最优的，超过5轮后生成数据的质量开始下降，因为模型开始"过度装饰"指令而非增加实质性的难度。建议在每轮进化后都进行人工抽检，确保进化后的指令仍然是自然、合理、有用的。

4. Magpie方法

Magpie（Xu et al., 2024）提出了一种更优雅的合成数据生成方式：直接利用对齐模型的自回归特性生成指令。其关键洞察是：已对齐的LLM在接收到特定前缀（如chat template的开头）时，会自然地生成合理的用户指令。

4.1 工作原理

Magpie的核心思想极其简单：直接将chat template的前缀（如"<|user|>\n"）输入模型，让模型自回归地生成后续文本。由于对齐模型已经学会了"在用户角色下生成指令"的模式，它会自然地生成合理的用户指令。

Magpie指令生成原理：

# 传统方法：需要精心设计prompt
prompt = "请生成一个关于Python编程的指令"
instruction = model.generate(prompt)  # 生成质量依赖prompt设计

# Magpie方法：直接利用chat template
prefix = "<|user|>\n"  # 只需要chat template的开头
response = model.generate(prefix)  # 模型自然地生成用户指令
# 输出示例: "如何用Python实现一个高效的LRU缓存？请给出完整代码和解释。"

# 完整的Magpie流程
def magpie_generate(model, tokenizer, n_samples=10000):
    """使用Magpie方法生成指令数据"""
    results = []
    
    for _ in range(n_samples):
        # Step 1: 生成指令
        prefix = tokenizer.apply_chat_template([{"role": "user", "content": ""}])
        prefix = prefix.split("")[0] + ""  # 截取到user标记
        instruction = model.generate(prefix, max_new_tokens=512)
        
        # Step 2: 使用模型回答自己的指令
        response = model.chat(instruction)
        
        # Step 3: 质量过滤
        if is_high_quality(instruction, response):
            results.append({
                "instruction": instruction,
                "response": response
            })
    
    return results
⚠️ Magpie的前提条件

Magpie要求模型已经过对齐训练（SFT+RLHF），因此更适合用于生成第二轮及后续飞轮迭代的数据，而非从头构建数据集。生成的指令质量高度依赖基础模型的能力。此外，Magpie生成的指令分布会偏向模型"擅长"的领域——如果模型在代码任务上表现更好，生成的指令也会更多地涉及代码。这种分布偏差需要通过后续的采样策略来纠正。

4.2 Magpie的高级技巧

为了提升Magpie的生成质量和多样性，研究者提出了多种改进策略：

  • 多语言前缀：使用不同语言的chat template前缀，生成多语言指令数据
  • 难度引导：在前缀中注入难度标签（如"[困难]"），引导模型生成更高难度的指令
  • 领域引导：在前缀中添加领域提示（如"[数学]"、"[代码]"），控制生成数据的领域分布
  • 温度调节：使用不同温度生成指令，平衡多样性和质量
💡 实用技巧：使用Magpie生成数据时，建议先分析目标模型的指令分布特性。具体做法是生成1000条样本，统计主题分布、难度分布和长度分布。如果发现分布偏差（如代码类过多），可以通过加权采样或领域限制来纠正。此外，对于多语言场景，可以参考09-多语言数据工程中的语言平衡策略。

5. 拒绝采样与验证生成

拒绝采样（Rejection Sampling）是一种在数学和代码等有明确正确性标准的领域中极为有效的合成数据方法。其核心思想是：对同一问题生成多个回答，只保留那些通过验证的正确回答作为训练数据。

5.1 数学拒绝采样

数学问题的拒绝采样流程：

import sympy
from sympy.parsing.latex import parse_latex

def math_rejection_sampling(model, problem, n_samples=16, validator=None):
    """对数学问题进行拒绝采样"""
    solutions = []
    
    # 生成多个解答
    for _ in range(n_samples):
        response = model.generate(problem, temperature=0.7)
        answer = extract_final_answer(response)
        
        # 验证答案
        if answer is not None:
            is_correct = verify_math_answer(problem, answer)
            if is_correct:
                solutions.append({
                    "problem": problem,
                    "solution": response,
                    "answer": answer
                })
    
    # 返回通过验证的解答
    return solutions

def verify_math_answer(problem, predicted_answer):
    """验证数学答案的正确性"""
    # 方法1: 与标准答案比较（如果有）
    ground_truth = get_ground_truth(problem)
    if ground_truth:
        return compare_answers(predicted_answer, ground_truth)
    
    # 方法2: 使用SymPy符号验证
    try:
        predicted_expr = parse_latex(predicted_answer)
        # 简化并比较
        return sympy.simplify(predicted_expr - ground_truth_expr) == 0
    except:
        return False
    
    # 方法3: 多数投票（如果无法直接验证）
    return None

5.2 代码拒绝采样

代码数据的拒绝采样特别有效，因为可以直接通过运行测试用例来验证正确性。

代码问题的拒绝采样：

import subprocess
import tempfile
import os

def code_rejection_sampling(model, problem, test_cases, n_samples=20):
    """对代码问题进行拒绝采样"""
    valid_solutions = []
    
    for _ in range(n_samples):
        response = model.generate(problem, temperature=0.6)
        code = extract_code(response)
        
        if code and run_test_cases(code, test_cases):
            valid_solutions.append({
                "problem": problem,
                "code": code,
                "explanation": extract_explanation(response)
            })
    
    return valid_solutions

def run_test_cases(code, test_cases, timeout=5):
    """运行测试用例验证代码正确性"""
    with tempfile.NamedTemporaryFile(mode='w', suffix='.py', delete=False) as f:
        f.write(code)
        f.write('\n\n')
        f.write(test_cases)
        f.flush()
        
        try:
            result = subprocess.run(
                ['python3', f.name],
                capture_output=True,
                timeout=timeout
            )
            return result.returncode == 0
        except (subprocess.TimeoutExpired, Exception):
            return False
        finally:
            os.unlink(f.name)

🔍 深入理解：拒绝采样的理论基础来自蒙特卡洛采样——通过重复采样和筛选，从一个容易采样的分布（模型的输出分布）中提取满足特定条件（正确性）的样本。从信息论角度看，每次采样都提供了一次"独立验证"，多次采样的一致性是正确性的强信号。这一方法在数学领域的成功（如AlphaProof项目）表明，对于有明确验证标准的任务，"暴力采样+验证"的策略往往比"精心设计prompt一次生成"更可靠。这种范式的核心权衡是计算成本与数据质量——生成20个候选回答的成本是一次回答的20倍，但如果正确率从50%提升到95%，对于下游训练来说是非常值得的。

6. 合成数据质量控制

合成数据的质量控制是决定其价值的关键环节。质量控制不仅是过滤低质量数据，更是确保数据的多样性、准确性和训练有效性。

过滤方法原理适用场景优点缺点
奖励模型排序使用RM对回复打分，保留top-k对话、指令跟随通用性强RM本身可能有偏差
一致性检验生成多次，保留一致的答案数学、逻辑推理高精度计算成本高

6.1 质量评估指标体系

一个完善的合成数据质量评估体系应该包含以下维度：

  • 准确性（Accuracy）：答案/代码/推理是否正确。可通过自动验证（运行代码、数学验证）或模型评估。
  • 完整性（Completeness）：回答是否完整覆盖了指令的所有要求。可使用checklist评分。
  • 清晰度（Clarity）：回答是否清晰易懂、结构良好。可使用语言模型评分。
  • 多样性（Diversity）：数据集整体的覆盖范围。可通过聚类分析或n-gram多样性指标评估。
  • 难度分布（Difficulty Distribution）：数据的难度梯度是否合理。可使用困惑度或模型loss估计难度。
⚠️ 注意事项：使用奖励模型（RM）进行过滤时，需要注意RM本身的偏差。如果RM是在特定分布上训练的，它可能会偏好该分布风格的回答，导致合成数据的分布被RM"塑造"而非由任务需求决定。建议使用多个不同的RM进行交叉验证，或者结合规则过滤和模型过滤的混合策略。此外，定期使用人工评估来校准RM的评分标准。

🎯 练习题

1. 实现对比实验：分别使用Self-Instruct和Evol-Instruct生成1000条Python编程指令数据，比较两种方法在指令多样性（n-gram多样性指标）和难度分布（使用模型困惑度估计）上的差异。

2. 设计一个Magpie方案：使用LLaMA-3-70B-Instruct生成10万条中文指令数据，描述完整的pipeline和质量控制策略。要求覆盖数学、代码、常识推理、创意写作四个维度。

3. 实验设计：实现一个拒绝采样系统，对GSM8K中的100道数学题分别生成1、4、8、16、32个回答，分析采样次数与正确率的关系曲线，确定最优的采样-成本平衡点。

4. 挑战题：设计一个"自适应合成"系统——根据当前模型在各能力维度上的弱点，自动调整合成数据的生成策略和数量分配。例如，如果模型在数学上较弱但代码较强，则生成更多的数学数据。讨论这种自适应方法的潜在收益和风险。

🔍 深入理解：从更宏观的视角看，合成数据的发展正在重塑AI的训练范式。传统的"数据飞轮"（参见01-数据飞轮方法论）假设数据量是限制因素，而合成数据使得数据量不再是瓶颈。真正限制模型能力的变成了：(1)合成数据的质量控制能力；(2)评估方法的有效性；(3)训练算法对噪声的鲁棒性。这意味着未来的竞争焦点将从"谁有更多数据"转向"谁有更好的质量控制和评估体系"。这种转变对整个AI产业的格局将产生深远影响——数据不再是最稀缺的资源，而质量控制能力将成为新的核心竞争力。
