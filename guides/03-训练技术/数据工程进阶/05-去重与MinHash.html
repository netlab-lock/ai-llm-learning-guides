为什么去重如此重要

Web数据中存在大量重复内容，重复率通常在30%-50%之间。重复数据不仅浪费存储和计算资源，更严重的是会导致模型过拟合特定模式，降低泛化能力。Lee et al.（2022）的研究表明，去重可使模型在相同计算预算下获得等效于3倍数据量的性能提升。这一发现彻底改变了社区对数据工程的认识——"数据质量优于数据数量"从理论假设变成了有实验支撑的工程共识。

⚠️ 重复的多种形态

数据重复不仅指完全相同的文本副本，还包括多种形态：（1）跨网站转载：新闻被多个媒体转载，内容相同或仅有标题差异；（2）模板化内容：导航栏、版权声明、Cookie提示等出现在大量页面中；（3）近似重复：仅修改少量措辞（如同一篇文章的不同版本、机器翻译的不同结果）；（4）子串重复：一句话在多篇文档中出现（如常用引语、标准声明等）；（5）结构重复：不同内容但结构相同（如同一模板生成的多个产品页面）。去重需要处理所有这些情况。

💡 实用技巧：在实际项目中，建议在数据清洗pipeline的困惑度过滤之后、内容安全过滤之前执行去重。这样做的理由是：（1）困惑度过滤已经移除了大量低质量数据，减少了去重的计算量；（2）去重结果应该在内容安全过滤之前保留，因为重复检测本身不需要理解语义。此外，建议将去重结果持久化存储——去重是计算密集型操作，保存去重索引可以避免每次重新计算。
⚠️ 注意事项：去重不是越激进越好。过于激进的去重（如设置过低的相似度阈值）可能误删有价值的变体数据。例如，同一技术概念的不同解释、同一事件的不同报道视角、同一编程问题的不同解决方案等，这些虽然在文本上相似，但提供了不同的信息和观点。建议根据下游任务调整去重强度——通用预训练可以用较宽松的阈值（Jaccard > 0.8），微调数据可以用更激进的阈值（Jaccard > 0.5）。
🔍 深入理解：从信息论的角度看，去重的本质是提升数据集的"有效信息熵"。Shannon信息论告诉我们，一个符号的信息量与其出现概率的负对数成正比——P(x)越高，信息量I(x) = -log P(x)越低。重复的文本片段在数据集中出现概率极高，因此其信息量极低。去重移除了这些低信息量的样本，使得剩余数据集的平均信息量更高。这与"数据压缩"的本质相同——去重就是一种针对训练数据的"无损压缩"。从这个角度看，MinHash和LSH等去重算法本质上是在估计数据集的"冗余度"，并移除冗余部分。

精确去重：哈希方法

精确去重是最简单的去重方法，适用于检测完全相同的文档副本。其核心思想是对每条文档计算一个哈希值（如SHA-256），哈希值相同的即为重复。这种方法速度快、实现简单，但只能检测完全相同的文档，无法处理近似重复。

精确去重的Python实现：

import hashlib

def exact_dedup(documents):
    """精确去重：基于SHA-256哈希"""
    seen = set()
    unique = []
    for doc in documents:
        h = hashlib.sha256(doc.encode('utf-8')).hexdigest()
        if h not in seen:
            seen.add(h)
            unique.append(doc)
    return unique

# 时间复杂度: O(n), 空间复杂度: O(n) 仅存储哈希值
# 碰撞概率: 对于SHA-256，2^128个文档才有50%碰撞概率
# 局限: 无法处理近似重复（一字之差即不同）

精确去重的变体

除了全文哈希，精确去重还有多种变体，适用于不同的场景：

变体方法适用场景优缺点
全文哈希SHA-256(全文)完全相同副本检测简单高效，但无法处理近似重复
URL去重SHA-256(URL)同一页面的多次爬取极快，但不同URL可能有相同内容
行级哈希对每行计算哈希检测模板化内容可检测部分重复，粒度更细
N-gram哈希对n-gram集合哈希检测子串重复更精细但计算量更大
💡 实用技巧：精确去重虽然简单，但在大规模数据中实现高效的精确去重并不平凡。当数据量超过内存容量时，需要使用外部排序或磁盘哈希表。一个实用的优化是"分桶哈希"——先用文档的前N个字节的哈希值将文档分配到不同的桶中，再在每个桶内做精确去重。这样可以将内存需求降低到数据集的1/B（B是桶的数量）。
⚠️ 注意事项：精确去重对"格式变化"极其敏感——同一文档的不同格式（如HTML vs 纯文本、不同编码、不同换行符）会被视为不同的文档。建议在精确去重之前，先对文本做标准化处理：统一编码为UTF-8、统一换行符为\n、去除首尾空白、统一空格和标点。这样可以显著提升精确去重的召回率。
🔍 深入理解：精确去重的理论基础是"完美哈希"——一个理想的哈希函数应该将不同的输入映射到不同的输出（无碰撞）。SHA-256等密码学哈希函数在实践中几乎可以视为完美哈希，因为其碰撞概率极低（2^-128）。但从理论上说，由于输入空间远大于输出空间（无限 vs 2^256），碰撞不可避免。在实际应用中，SHA-256的碰撞概率可以忽略不计——即使处理10亿个文档，碰撞概率也仅为10^-20量级。

模糊去重：MinHash与LSH

MinHash是处理近似重复的核心技术，由Broder（1997）在AltaVista搜索引擎中首次提出。其基本思想是将文档表示为n-gram集合，通过多次随机哈希估计两个集合的Jaccard相似度。MinHash的巧妙之处在于：它将高维的集合相似度计算转化为低维的签名向量比较，大幅降低了计算复杂度。

MinHash的核心原理：

1. 文档 → n-gram集合的转换
   文档A: "the cat sat on the mat"
   → 3-gram集合 Sa = {"the cat sat", "cat sat on", "sat on the", "on the mat"}
   
   文档B: "the cat sat on a mat"
   → 3-gram集合 Sb = {"the cat sat", "cat sat on", "sat on a", "on a mat"}

2. Jaccard相似度定义
   Jaccard(A,B) = |Sa ∩ Sb| / |Sa ∪ Sb|
   
   Sa ∩ Sb = {"the cat sat", "cat sat on"}  → |intersection| = 2
   Sa ∪ Sb = {"the cat sat", "cat sat on", "sat on the", "on the mat", "sat on a", "on a mat"}
             → |union| = 6
   
   Jaccard(A,B) = 2/6 = 0.333

3. MinHash签名计算
   对于k个独立的随机哈希函数 h1, h2, ..., hk：
   
   minHash(S, hi) = min(hi(s) for s in S)
   
   签名向量 = [minHash(S, h1), minHash(S, h2), ..., minHash(S, hk)]

4. MinHash的核心定理
   P(minHash(A, hi) == minHash(B, hi)) = Jaccard(A, B)
   
   证明: 考虑Sa ∪ Sb中所有元素在hi下的哈希值排序。
   最小哈希值来自Sa ∩ Sb的概率恰好等于|Sa ∩ Sb|/|Sa ∪ Sb|。

5. 签名比较
   估计Jaccard相似度 = (签名向量中相同位置值相同的数量) / k
   k越大，估计越精确（方差 = J(1-J)/k）

LSH（Locality Sensitive Hashing）加速

即使有MinHash签名，两两比较所有文档对仍需O(n²)时间。LSH通过将签名分成b个band、每个band包含r行，将相似文档分到同一桶中，将复杂度降至近似O(n)。

┌──────────────────────────────────────────────────────────┐ │ MinHash + LSH 去重流程 │ ├──────────────────────────────────────────────────────────┤ │ │ │ 文档集合 {D1, D2, ..., Dn} │ │ │ │ │ ▼ │ │ [Step 1: N-gram提取] │ │ D1 → {g1, g2, g3, ...} │ │ D2 → {g1, g4, g5, ...} │ │ │ │ │ ▼ │ │ [Step 2: MinHash签名计算] │ │ k个哈希函数 → 签名向量 [s1, s2, ..., sk] │ │ D1 → [234, 891, 12, ..., 567] │ │ D2 → [234, 905, 12, ..., 103] │ │ │ │ │ ▼ │ │ [Step 3: LSH分桶] │ │ 将签名分成 b=16 bands, 每band r=8 行 │ │ │ │ Band 1: [234,891,12,345,678,901,234,567] → hash → 桶A │ │ Band 2: [891,12,345,678,901,234,567,123] → hash → 桶B │ │ ... │ │ Band 16: [...,567] → hash → 桶C │ │ │ │ 如果两个文档在任一band中落入同一桶 → 候选对 │ │ │ │ │ ▼ │ │ [Step 4: 候选对验证] │ │ 对候选对计算精确Jaccard相似度 │ │ 保留 Jaccard ≥ 阈值(如0.8) 的对 │ │ │ │ │ ▼ │ │ [输出: 去重后的文档集合] │ │ │ └──────────────────────────────────────────────────────────┘

LSH的关键参数是b（band数）和r（每band的行数）。当b×r=k时，相似度阈值约为(1/b)^(1/r)。这意味着我们可以通过调整b和r来控制去重的灵敏度——更小的阈值意味着更激进的去重（更多候选对），更大的阈值意味着更保守的去重（更少候选对）。

💡 LSH参数选择指南

LSH参数的选择直接影响去重的效果和效率。以下是常见的参数组合及其对应的阈值：

💡 实用技巧：在实际工程中，建议先用少量数据（如1%）测试不同的b/r参数组合，绘制"召回率-精确率"曲线来选择最优参数。一个实用的经验法则：如果你需要去除80%以上的近似重复，使用b=32, r=4（阈值≈0.67）；如果只需要去除明显的重复，使用b=16, r=8（阈值≈0.80）。此外，可以使用两阶段策略：先用宽松阈值做粗筛，再用严格阈值做精排。
⚠️ 注意事项：MinHash的一个重要局限是它将文档视为无序的n-gram集合——忽略了词序信息。这意味着"猫追狗"和"狗追猫"会被视为高度相似（如果它们共享大量n-gram）。在大多数去重场景中这不是问题（因为近似重复通常保留了词序），但在某些特殊场景（如检测改述paraphrase）中需要注意。此外，MinHash的内存消耗与k×n成正比（k是签名长度，n是文档数），对于超大规模数据集（如数十亿文档），需要使用压缩签名或分布式计算来降低内存需求。
🔍 深入理解：MinHash的数学之美在于它证明了一个深刻的概率论定理：两个集合的Jaccard相似度等于它们的MinHash签名在任一随机哈希函数下取相同值的概率。这个定理的证明简洁而优雅——考虑所有元素在哈希函数下的排序，最小哈希值来自交集的概率恰好等于交集大小与并集大小的比值。这个定理不仅有理论价值，还有重要的工程意义：它将复杂的集合相似度计算（需要遍历两个集合）转化为简单的哈希值比较（O(1)操作），使得大规模去重成为可能。

MinHash的数学原理与推导

本节深入推导MinHash的数学原理，包括Jaccard相似度的定义、MinHash签名的构造方法、以及LSH的阈值分析。

Jaccard相似度

给定两个集合A和B，Jaccard相似度定义为交集与并集的比值：

Jaccard(A, B) = |A ∩ B| / |A ∪ B|

性质：
  - 0 ≤ Jaccard(A,B) ≤ 1
  - Jaccard(A,A) = 1
  - Jaccard(A,B) = 0 当且仅当 A ∩ B = ∅
  - Jaccard(A,B) 是一个度量（满足三角不等式）

在去重中的应用：
  - 将文档表示为n-gram集合
  - Jaccard > 0.8 视为近似重复
  - Jaccard = 1.0 视为完全重复

MinHash签名的构造

MinHash签名构造的数学推导：

设 π 是 {1, 2, ..., N} 上的一个随机排列（N是所有可能的n-gram的数量）
对于集合 S，定义 h(S) = min{π(x) : x ∈ S}

定理: P(h(A) = h(B)) = Jaccard(A, B)

证明:
  设 U = A ∪ B，考虑 π 在 U 上的限制
  设 x* = argmin{π(x) : x ∈ U}  (并集中排列值最小的元素)
  
  情况1: x* ∈ A ∩ B
    则 h(A) = π(x*) = h(B)
    P(情况1) = |A ∩ B| / |A ∪ B| = Jaccard(A,B)
  
  情况2: x* ∈ A \ B
    则 h(A) = π(x*) < h(B) (因为B中不包含x*)
    P(情况2) = |A \ B| / |A ∪ B|
  
  情况3: x* ∈ B \ A
    则 h(B) = π(x*) < h(A)
    P(情况3) = |B \ A| / |A ∪ B|
  
  因此 P(h(A) = h(B)) = P(情况1) = Jaccard(A,B)  □

实际实现中，不使用真正的随机排列（计算量太大），
而是使用独立的哈希函数 h_i(x) = (a_i * x + b_i) mod p
其中 a_i, b_i 是随机整数，p 是大素数。

LSH阈值分析

LSH阈值推导：

设签名长度 k = b × r（b个band，每band r行）

两个文档在一个band中"碰撞"（所有r行都匹配）的概率：
  P_band = J^r  (J是Jaccard相似度)

两个文档在至少一个band中碰撞的概率：
  P_candidate = 1 - (1 - J^r)^b

当 b=16, r=8 时：
  P_candidate = 1 - (1 - J^8)^16
  
  J=0.5: P = 1 - (1-0.0039)^16 ≈ 0.061 (6.1%)
  J=0.7: P = 1 - (1-0.058)^16  ≈ 0.61  (61%)
  J=0.8: P = 1 - (1-0.168)^16  ≈ 0.95  (95%)
  J=0.9: P = 1 - (1-0.430)^16  ≈ 0.999 (99.9%)

阈值 t 使得 P_candidate = 0.5:
  1 - (1 - t^r)^b = 0.5
  (1 - t^r)^b = 0.5
  1 - t^r = 0.5^(1/b)
  t = (1 - 0.5^(1/b))^(1/r)

对于 b=16, r=8:
  t = (1 - 0.5^(1/16))^(1/8) = (1 - 0.9576)^(0.125) = 0.0424^0.125 ≈ 0.80

🔍 深入理解：LSH的S曲线特性使其成为一种优雅的"软阈值"机制——与硬阈值（直接判断Jaccard是否大于阈值）不同，LSH的概率性机制自然地处理了边界情况。在实际应用中，这种"软性"是有益的——它允许一些"接近阈值"的文档对以一定概率被选为候选对，避免了硬阈值可能导致的"临界点效应"。从优化理论的角度看，LSH的S曲线可以看是对理想阶梯函数（step function）的可微近似，这使得它在端到端可微系统中也能使用。

SimHash方法

SimHash（也称为locality-sensitive hashing for cosine similarity）是另一种近似去重方法，由Charikar（2002）提出。与MinHash不同，SimHash直接计算一个固定长度的指纹（如64位），通过Hamming距离判断相似性。SimHash特别适合超大规模场景（如搜索引擎），因为其比较操作可以在单个CPU指令中完成。

精度较高（k越大越准）中等（固定64位）MinHash可调精度

适用场景文档级去重网页级别、超大规模各有优势
LSH加速band技术多表位索引都可以实现亚线性搜索
SimHash算法详解：

1. 特征提取
   将文档转换为带权重的特征集合 {(f1, w1), (f2, w2), ...}
   其中 fi 是特征（如n-gram或词），wi 是权重（如TF-IDF值）

2. 哈希计算
   对每个特征计算哈希值: hash(fi) → 64位二进制向量
   
   加权累加: 对于每一位 j (0 ≤ j < 64):
     如果 hash(fi) 的第j位是1: V[j] += wi
     如果 hash(fi) 的第j位是0: V[j] -= wi

3. 生成指纹
   对于每一位 j:
     如果 V[j] > 0: fingerprint[j] = 1
     否则: fingerprint[j] = 0

4. 相似度判断
   两个文档的SimHash指纹的Hamming距离越小，越相似
   Hamming距离 ≤ 3 通常视为近似重复
   
   Hamming距离可以通过 POPCNT 指令高效计算:
   distance = popcount(fingerprint_A XOR fingerprint_B)
💡 实用技巧：SimHash的实现可以利用CPU的POPCNT指令来加速Hamming距离计算。在Python中，可以使用内置的bin(x).count('1')来计算汉明距离，但对于大规模数据，建议使用C扩展或NumPy的位操作来加速。此外，可以通过"多表位索引"（multiple tables with bit sampling）来实现SimHash的LSH加速——将64位指纹分成多组，每组作为一个独立的索引键。
⚠️ 注意事项：SimHash对权重的敏感度很高——如果特征权重分布不均匀（如少数特征权重极高），SimHash可能过度偏向这些高权重特征，忽略其他重要信息。建议在计算SimHash之前，对特征权重做归一化处理（如截断极端值、使用对数变换等）。此外，SimHash更适合较长的文档——对于短文本（如推文），由于特征数量少，SimHash的随机性较大，精度可能不佳。
🔍 深入理解：SimHash的数学原理可以从随机超平面的角度理解。在高维空间中，一个随机超平面将两个向量分到不同侧的概率与它们之间的角度成正比。SimHash的每一位本质上对应一个随机超平面——如果文档的特征向量在超平面的"正侧"，该位为1；否则为0。两个文档的Hamming距离反映了它们在多少个随机超平面上被分到了不同侧，这与余弦相似度成反比。这种"随机投影降维"的思想与Johnson-Lindenstrauss引理密切相关——在高维空间中的距离关系可以被低维随机投影近似保持。

Bloom Filter加速

Bloom Filter是一种空间高效的概率数据结构，用于快速判断一个元素是否在集合中。在去重流程中，Bloom Filter常用于预过滤阶段，快速排除明显不重复的文档对候选，仅对可能重复的文档进行精确比较。

Bloom Filter的工作原理：

1. 初始化: 创建一个m位的位数组，所有位初始化为0

2. 插入: 对元素x，使用k个独立的哈希函数
   对于 i = 1, 2, ..., k:
     位数组[h_i(x) mod m] = 1

3. 查询: 检查元素x是否在集合中
   如果所有 k 个位置都是1 → "可能存在"
   如果任一位置是0 → "一定不存在"

4. 误判率分析
   假设n个元素插入后，某一位仍为0的概率:
   p = (1 - 1/m)^(kn) ≈ e^(-kn/m)
   
   误判率: FPR = (1 - p)^k ≈ (1 - e^(-kn/m))^k
   
   最优哈希函数数量: k = (m/n) * ln(2) ≈ 0.693 * (m/n)
   
   示例: m/n = 10 bits/element, k=7 → FPR ≈ 0.8%

📖 去重粒度选择

去重可以在不同粒度进行：（1）文档级：整篇文档去重，适合去除完全转载；（2）段落级：去除重复段落，保留文档的独特部分；（3）n-gram级：去除重复短语，保留独特的表达。LLaMA论文使用了文档级和行级（trigram）两级去重。过细的粒度可能破坏文档结构，过粗的粒度可能遗漏大量子串重复。建议根据目标选择：通用预训练用文档级去重即可，高质量微调数据可以用段落级去重。

💡 实用技巧：Bloom Filter的参数选择需要权衡空间和误判率。对于去重预过滤场景，建议使用每元素10-15个bit（m/n = 10-15），对应误判率约0.8%-0.02%。这样可以确保绝大多数不重复的文档被快速排除，同时保持极低的误判率。Python的pybloom-live库提供了方便的Bloom Filter实现，支持自动调整大小。
⚠️ 注意事项：Bloom Filter不支持删除操作——一旦元素被插入，无法将其从BF中移除。这意味着如果需要动态更新去重集合（如在流式处理中），标准Bloom Filter可能不适用。解决方案是使用Counting Bloom Filter（用计数器替代位数组）或Cuckoo Filter（支持删除操作）。在大规模离线去重中，这个问题通常不显著，因为BF只需在处理过程中保持不变即可。
🔍 深入理解：Bloom Filter的核心优势在于其空间效率——它用固定大小的空间（与元素数量呈线性关系）实现了集合成员查询，而传统的哈希表需要与元素数量呈线性关系的空间来存储完整的键。这种空间效率的代价是允许一定的误判（false positive），但不允许漏判（false negative）。从信息论的角度看，Bloom Filter是一种"有损压缩"——它用b*m bit的空间编码了n个元素的存在信息，理论上需要n*log(m/n) bit的信息，Bloom Filter的效率接近这个理论下界。

去重对训练的影响

去重对模型训练的影响是多方面的，不仅体现在最终性能上，还体现在训练效率、数据利用效率等方面。以下是基于多个实验的系统性分析：

指标去重前去重后变化解释
数据量1.2T tokens627B tokens-48%移除重复数据
训练loss2.452.38-3%减少过拟合
下游准确率68.2%70.1%+2.8%更好的泛化
训练时间100%52%-48%数据量减半
GPU成本$100K$52K-48%训练时间减半
数据多样性较低较高+显著更多独特内容

从上表可以清晰地看到，去重不仅没有损害模型性能，反而在所有指标上都有所提升。这背后的原理是：重复数据导致模型在相同的模式上反复更新参数，造成隐式的过拟合。去重后，模型在更短的训练时间内看到了更多样的数据，学习效率更高。这一发现与Chinchilla定律的精神一致——数据的多样性比数据的绝对数量更重要。

💡 实用技巧：去重后建议重新调整学习率调度——由于数据量减少，每个epoch的step数也会减少，如果使用基于step的学习率调度（如cosine schedule），需要相应调整总step数。此外，去重后可以适当增加数据重复使用的次数（即多epoch训练），因为去重后的数据质量更高，多次训练不会导致严重的过拟合。
⚠️ 注意事项：去重的效果可能因模型规模而异。在小模型（如125M参数）上，去重的收益可能不显著——因为小模型本身的能力有限，数据质量的提升可能被模型容量的限制所掩盖。在大模型（如7B+参数）上，去重的收益通常更加明显。因此，在决定去重策略时，需要考虑目标模型的规模。
🔍 深入理解：去重对训练的影响可以从"隐式正则化"的角度理解。在传统的机器学习中，正则化（如L2正则、dropout）通过限制模型的复杂度来防止过拟合。去重通过减少训练数据中的冗余信息，起到了类似的正则化效果——模型被迫学习更一般化的模式，而非记忆特定的数据点。从梯度更新的角度看，重复数据会导致某些梯度方向被过度强调（因为它们在每个minibatch中出现多次），去重使得梯度方向更加多样，有助于模型探索更广阔的参数空间。这种"数据层面的正则化"与"算法层面的正则化"（如Adam优化器的动量项）是互补的。

去重粒度与工程实践

去重粒度的选择是工程实践中最重要的决策之一。不同粒度的去重适用于不同的场景，需要根据下游任务、数据特征和计算资源来选择。

去重粒度方法适用场景保留率计算复杂度
URL级URL哈希去除同一页面的多次爬取~85%极低 O(n)
文档级MinHash全文通用预训练数据~55-70%中等 O(n log n)
段落级MinHash段落高质量微调数据~70-80%较高
行级(trigram)精确哈希去除模板化内容~80-90%低
工业级去重工程实践示例（分层去重策略）：

# 第1层: URL去重（最快，O(n)）
seen_urls = set()
for doc in docs:
    url_hash = sha256(normalize_url(doc.url))
    if url_hash not in seen_urls:
        seen_urls.add(url_hash)
        yield doc  # 保留

# 第2层: 精确哈希去重（快，O(n)）
seen_hashes = set()
for doc in docs:
    doc_hash = sha256(normalize_text(doc.text))
    if doc_hash not in seen_hashes:
        seen_hashes.add(doc_hash)
        yield doc

# 第3层: MinHash近似去重（较慢，O(n log n)）
minhash_index = MinHashLSH(threshold=0.8, num_perm=128)
for doc in docs:
    mh = MinHash(num_perm=128)
    for ngram in get_ngrams(doc.text, n=5):
        mh.update(ngram.encode('utf-8'))
    if not minhash_index.query(mh):  # 无近似重复
        minhash_index.insert(doc.id, mh)
        yield doc

# 第4层: 行级trigram去重（补充）
seen_trigrams = set()
for doc in docs:
    unique_lines = []
    for line in doc.text.split('\n'):
        line_trigrams = set(get_ngrams(line, n=3))
        if not line_trigrams.intersection(seen_trigrams):
            unique_lines.append(line)
            seen_trigrams.update(line_trigrams)
    doc.text = '\n'.join(unique_lines)
    yield doc
💡 实用技巧：在实际工程中，建议使用Apache Spark或Dask来进行分布式去重。对于MinHash去重，可以使用datasketch库（Python）或MinHashLSH（Spark MLlib）来实现。关键的工程优化包括：（1）使用高效的序列化格式（如Apache Parquet）来存储中间结果；（2）对数据按域名或语言分区，减少跨节点通信；（3）使用Bloom Filter做预过滤，减少LSH查询次数。对于超大规模数据（如10TB+），建议使用MapReduce风格的去重流程——map阶段计算MinHash，reduce阶段合并和去重。
⚠️ 注意事项：分布式去重的一个关键挑战是"全局一致性"——在单机去重中，所有文档都在同一个进程中处理，可以保证全局一致性；但在分布式环境中，不同节点可能独立处理不同分区的数据，导致部分重复未被检测到。解决方案是使用"两轮MapReduce"——第一轮在每个分区内做局部去重，第二轮对跨分区的候选对做全局验证。此外，去重的顺序也很重要——建议先做质量过滤再做去重，避免在低质量数据上浪费去重的计算资源。
🔍 深入理解：去重工程的核心权衡是"精度 vs 效率"。精确去重（SHA-256哈希）速度快但只能检测完全重复；模糊去重（MinHash）可以检测近似重复但计算量大。在实际工程中，通常采用"分层去重"策略——先用快速的精确方法过滤掉明显的重复，再用精确的模糊方法处理剩余的近似重复。这种分层策略的核心原理是"early rejection"——尽可能在早期阶段排除不重复的文档，减少后续阶段的计算量。这一原理在计算机科学的很多领域都有应用，如编译器优化中的"死代码消除"、数据库查询中的"索引过滤"等。

🔍 深入理解：去重工具的选择反映了数据工程中的一个更广泛的权衡："通用性 vs 专用性"。通用工具（如datasketch）提供了灵活的API，可以适应各种场景，但可能在特定场景下效率不高；专用工具（如text-dedup）针对NLP数据去重做了大量优化，但灵活性较低。在实际项目中，建议根据项目阶段选择：原型阶段使用通用工具快速验证，生产阶段切换到专用工具追求效率。这种"原型-生产"的两阶段策略在工程实践中非常常见。
