➕ 03 · Task Arithmetic

Task Arithmetic (Ilharco et al. 2023) 引入了一个优雅的概念：Task Vector——微调带来的"知识增量"可以用向量表示，支持加法和减法运算。这一发现深刻揭示了大模型能力的可分解性和可组合性。

1. Task Vector 的概念

1.1 定义

τ = θ_finetuned - θ_pretrained 含义: 微调过程中，模型权重的变化量 这个变化量编码了微调带来的"知识增量"

Task Vector 的定义极其简单：它就是微调后模型权重减去预训练模型权重。这个简单的减法操作却蕴含着深刻的意义——它将微调过程中的"知识学习"从绝对权重空间中解耦出来，变成了一个独立的、可操作的"增量向量"。

1.2 直觉理解

Task Vector 的直觉: ┌──────────────────────────────────────────────────────────────┐ │ 预训练模型 θ_pre: "通用语言能力" │ │ │ │ │ │ 微调 (数学数据) │ │ ▼ │ │ 微调模型 θ_finetuned: "通用 + 数学能力" │ │ │ │ Task Vector τ = θ_finetuned - θ_pre │ │ 含义: "数学能力的增量" │ │ │ │ 关键洞察: 这个增量可以被提取、存储、组合、缩放! │ └──────────────────────────────────────────────────────────────┘

1.3 为什么 Task Vector 有效

Task Vector 之所以能够独立编码特定能力，根本原因在于微调的局部性。研究表明，微调主要改变模型中与目标任务相关的参数子集，而大部分参数变化很小。这意味着 Task Vector 是"稀疏的"——大部分元素接近零，只有少量元素有显著变化。

另一个重要原因是预训练的锚定效应。由于所有微调都从同一个预训练模型出发，Task Vector 之间具有可比性——它们都在同一个"参考系"下定义。这使得不同任务的 Task Vector 可以直接进行加减运算。

1.4 Task Vector 的数学性质

性质描述实际意义
可加性τ_A + τ_B 可以组合两种能力多任务融合
可减性θ - τ 可以去除特定能力能力遗忘/安全编辑
可缩放λ·τ 可以调节能力强度精细控制
稀疏性大部分元素接近零可以压缩存储
可迁移同一任务的不同模型的 τ 相似跨模型知识迁移
实用建议：在计算 Task Vector 时，建议使用 FP32 精度。由于 Task Vector 是两个大数相减的结果，低精度（如 FP16）可能导致数值不稳定。计算完成后可以转换回 FP16 存储以节省空间。
注意事项：Task Vector 的计算需要确保预训练模型和微调模型使用完全相同的架构和 tokenizer。即使 tokenizer 有微小差异，也可能导致权重不对齐，使 Task Vector 失去意义。
深入理解：从表示学习角度看，Task Vector 编码的不仅是"知识"，还包含了微调过程中对预训练表示的"重组"。预训练模型提供了一组通用的特征基，微调过程改变了这些基的组合方式，Task Vector 记录的就是这种组合方式的变化。

2. 任务加法

2.1 基本公式

θ_merged = θ_pre + Σ λ_i · τ_i 含义: 在预训练模型的基础上，叠加多个任务的知识增量 λ_i 控制第 i 个任务的"强度"

任务加法是 Task Arithmetic 最核心的操作。它在预训练模型的基础上，叠加多个任务的 Task Vector，每个向量乘以一个缩放系数 λ。

2.2 多任务融合示例

假设我们有三个微调模型：数学模型、代码模型和中文模型。通过任务加法，可以创建一个同时具备这三种能力的模型。关键在于为每个任务选择合适的缩放系数 λ_i，以平衡不同能力之间的强度。

任务加法 vs 简单平均: 简单平均: θ_merged = (θ_A + θ_B) / 2 = θ_pre + (τ_A + τ_B) / 2 任务加法: θ_merged = θ_pre + λ₁·τ_A + λ₂·τ_B 区别: 任务加法允许 λ₁ ≠ λ₂，提供更精细的控制 例如: λ₁=0.7, λ₂=0.3 → 更偏向任务A的能力
实用建议：在实际使用任务加法时，建议从 λ=0.5 开始，然后在 [0.3, 0.8] 范围内搜索最优值。不同任务的最优 λ 可能不同，因此建议分别调优每个 λ_i。
常见陷阱：当叠加的 Task Vector 数量超过 3-4 个时，可能出现"能力稀释"现象——每个任务的能力都不够强。此时可以适当增大 λ 值，或使用 DARE/TIES 等方法减少干扰。
深入理解：任务加法的成功暗示了大模型的"能力空间"可能是近似线性的。这意味着不同的能力可以被独立编码和组合，就像颜色可以用 RGB 分量独立表示一样。这一发现对模型编辑、遗忘学习等领域都有重要启示。

3. 任务减法与 Negation

3.1 能力去除

θ_modified = θ - λ · τ_undesired ← 去除特定能力

任务减法是 Task Arithmetic 的另一个强大应用。通过从模型中减去某个 Task Vector，可以有选择地"遗忘"模型的某种能力。这一技术在安全对齐、隐私保护等场景中有重要应用。

3.2 Negation 操作

θ_negated = θ_pre - λ · τ ← Negation (更激进) 含义: 不仅去除增量，还向"反方向"移动

Negation 比简单的任务减法更激进。简单减法只是"移除"微调带来的增量，而 Negation 则将模型推向任务向量的"反方向"。

3.3 应用场景

应用操作目标风险
安全对齐减去有害能力的 τ减少有害输出可能影响正常能力
隐私保护减去特定知识的 τ遗忘敏感信息可能不完全
偏差消除减去偏见的 τ减少输出偏见定义"偏见"本身有争议
风格调整减去旧风格，加新风格改变输出风格可能混合残留
实用建议：任务减法的效果通常不如任务加法稳定。建议在使用任务减法后，对模型进行全面的安全和性能评估。
安全警告：任务减法可以用于"解除"模型的安全对齐，这是一个严重的安全隐患。模型提供方应考虑采用更 robust 的安全防护机制。
深入探讨：Task Arithmetic 揭示了一个深刻的现象：LLM 的能力可以被分解为独立的"方向"。就像颜色可以用 RGB 三个分量表示，模型的能力也可以用多个 task vector 表示。这引出了一个前沿研究方向——"能力编辑"：通过加减 task vector，精确地添加或删除模型的特定能力。

4. λ 的选择与 Scaling

λ 值效果适用场景
λ < 0.3融合效果弱，能力保留不完全保守融合，避免干扰
λ = 0.3-0.5通常效果最好，平衡各任务大多数场景的默认选择
λ = 0.5-1.0能力较强但可能引入噪声对特定任务有强需求
λ > 1.0可能引入严重噪声，性能下降实验性场景
实用建议：不要一开始就追求完美方案，先跑通一个能工作的版本，再迭代优化。建议使用网格搜索，在 [0.1, 1.0] 范围内以 0.1 为步长评估不同 λ 值的效果。
常见误区：认为 λ 越大越好——实际上，λ 过大会导致"过拟合"到新增能力，原有能力退化。
深入理解：从优化角度看，λ 的选择本质上是在"新增能力"和"原有能力"之间做权衡。这与机器学习中经典的偏差-方差权衡有异曲同工之妙。

6. 学习建议与下一步

Task Arithmetic 是理解高级融合方法的基础。掌握了 Task Vector 的概念后，可以继续学习 DARE 和 TIES-Merging，这两种方法专门解决 Task Arithmetic 中的"干扰问题"。

6.1 Task Arithmetic 的实现细节

在实际实现 Task Arithmetic 时，有几个重要的技术细节需要注意。首先是权重存储格式：Task Vector 是两个大数相减的结果，数值范围通常很小。如果使用 FP16 存储原始模型权重，减法操作可能导致精度损失。建议在计算 Task Vector 时使用 FP32 精度，计算完成后再转换回 FP16 存储。

其次是逐层分析：不同层的 Task Vector 可能有不同的特征。一般来说，底层（靠近输入的层）的 Task Vector 变化较小，因为底层主要编码通用的语言特征；高层（靠近输出的层）的 Task Vector 变化较大，因为高层与具体任务更相关。了解这种分布特征有助于选择合适的融合策略。

第三是稀疏化处理：由于 Task Vector 的稀疏性，可以通过稀疏化来减少存储开销和计算成本。一种常见的方法是只保留绝对值最大的 top-k% 参数，将其余参数设为零。这与 TIES-Merging 的 Trim 步骤类似，但可以在 Task Vector 计算后立即进行。

层位置Task Vector 特征融合建议
Embedding 层变化较小，主要编码词汇表征通常保持不变或低权重融合
底层 Transformer (1-8)变化中等，编码通用语法和语义可以安全融合，干扰较小
中层 Transformer (9-20)变化较大，编码任务相关特征需要仔细调参，可能有干扰
高层 Transformer (21-32)变化最大，编码任务特定知识干扰最严重，建议使用 DARE/TIES
LM Head变化中等，编码输出分布建议单独处理，可能需要功能空间融合

6.2 Task Arithmetic 与 LoRA 的关系

Task Vector 和 LoRA（Low-Rank Adaptation）之间有有趣的联系。LoRA 也可以看作是一种"增量"——它用低秩矩阵 ΔW = B·A 来近似微调带来的权重变化。而 Task Vector 是完整的权重变化，不假设低秩结构。从这个角度看，LoRA 可以理解为 Task Vector 的低秩近似。

这种联系引出了一个有趣的问题：能否用 LoRA 来存储和操作 Task Vector？如果可以，那么 Task Arithmetic 的存储成本将大大降低（从全参数的 θ 维度降低到 LoRA 的 r×(d_in + d_out) 维度）。初步研究表明，这种"LoRA 级别的 Task Arithmetic"在某些场景下是可行的，但效果可能不如全参数版本。

Task Vector 与 LoRA 的关系: 全参数 Task Vector: LoRA 近似: ┌──────────────────┐ ┌──────────────────┐ │ τ ∈ R^{d×d} │ │ τ ≈ B·A │ │ (d×d 维矩阵) │ │ B ∈ R^{d×r} │ │ 存储: O(d²) │ │ A ∈ R^{r×d} │ │ 精度: 精确 │ │ 存储: O(2·d·r) │ │ │ │ 精度: 近似 │ └──────────────────┘ └──────────────────┘ 当 r << d 时: LoRA 存储远小于全参数 但可能丢失部分信息 示例 (7B 模型): 全参数: 7B × 2 bytes = 14 GB LoRA (r=16): 7B × 2 × 16/4096 ≈ 55 MB 压缩比: 250x

6.3 多任务融合的层次化策略

当需要融合多个任务时，简单的"一次性叠加"可能不是最优策略。层次化融合策略将融合过程分为多个阶段，每个阶段处理一组相关的任务。例如，先融合数学和代码能力（高兼容性），再融合中文能力（中等兼容性），最后融合安全对齐（需要谨慎处理）。

这种层次化策略的优势在于：(1) 可以在每个阶段独立调优参数；(2) 高兼容性的任务先融合，减少后续阶段的干扰；(3) 可以在每个阶段进行中间评估，及时发现和解决问题。

实用建议：在多任务融合时，建议按照任务兼容性从高到低的顺序进行融合。先融合最兼容的任务对，评估效果后再加入下一个任务。每一步都要进行中间评估，确保没有引入严重的问题。
安全警告：在使用 Task Arithmetic 进行能力编辑（如去除安全对齐）时，必须考虑伦理和法律风险。模型提供方应建立完善的审批流程，确保能力编辑操作不会被滥用。研究人员在发表相关论文时，也应充分讨论潜在的安全风险。
练习：计算三个 Task Vector 的融合。设 τ_math、τ_code、τ_chinese 分别为数学、代码、中文任务的 Task Vector。使用 λ₁=0.5, λ₂=0.4, λ₃=0.3 进行融合。分析 λ 的和不等于 1 时的影响。

7.1 Task Arithmetic 的变体与扩展

原始的 Task Arithmetic 论文提出了基本的加减运算框架，后续研究在此基础上提出了多种变体和扩展。其中一个重要的扩展是RegMean方法——它使用回归分析来估计最优的融合权重，而不是简单地使用预设的 λ 值。RegMean 的核心思想是：对于每个参数位置，找到使融合后输出与源模型输出最接近的权重。

另一个重要的扩展是Task-Driven Merging——它将任务性能作为融合的优化目标，使用梯度下降等优化算法来搜索最优的融合权重。与传统的网格搜索相比，这种方法可以更高效地找到好的融合策略，特别是当融合的模型数量较多时。

还有一个有趣的方向是Task Vector 的分解——将 Task Vector 分解为多个"子向量"，每个子向量编码不同方面的知识。例如，可以将数学能力的 Task Vector 分解为"计算能力"子向量和"推理能力"子向量，然后只融合需要的子向量。这种精细的分解可以实现更精确的能力控制。

7.2 Task Arithmetic 在不同模型规模下的表现

Task Arithmetic 的效果与模型规模密切相关。一般来说，模型越大，Task Arithmetic 的效果越好。这是因为大模型的参数空间更高维，Task Vector 更稀疏，不同任务之间的干扰更小。

模型规模Task Arithmetic 效果建议策略
125M效果有限，干扰严重建议使用简单平均或蒸馏
1.3B中等效果可以尝试，但需要仔细调参
7B效果较好推荐使用，配合 DARE/TIES
13B效果很好推荐使用，λ 可以更激进
70B+效果优秀最佳实践，可以融合更多任务

7.3 Task Arithmetic 实现代码

以下是 Task Arithmetic 的核心实现代码：

import torch

def compute_task_vector(pretrained_state, finetuned_state):
    """计算 Task Vector: τ = θ_finetuned - θ_pretrained"""
    task_vector = {}
    for key in pretrained_state:
        task_vector[key] = finetuned_state[key].float() - pretrained_state[key].float()
    return task_vector

def task_arithmetic_merge(pretrained_state, task_vectors, lambdas):
    """任务加法: θ_merged = θ_pre + Σ λ_i · τ_i"""
    merged = {}
    for key in pretrained_state:
        delta = sum(lam * tv[key] for lam, tv in zip(lambdas, task_vectors))
        merged[key] = pretrained_state[key].float() + delta
    return merged

def task_negation(pretrained_state, task_vector, lam=1.0):
    """任务 Negation: θ = θ_pre - λ · τ"""
    merged = {}
    for key in pretrained_state:
        merged[key] = pretrained_state[key].float() - lam * task_vector[key]
    return merged

# 使用示例
pretrained = torch.load("pretrained.pt")
finetuned_math = torch.load("finetuned_math.pt")
finetuned_code = torch.load("finetuned_code.pt")

tau_math = compute_task_vector(pretrained, finetuned_math)
tau_code = compute_task_vector(pretrained, finetuned_code)

merged = task_arithmetic_merge(
    pretrained,
    [tau_math, tau_code],
    [0.5, 0.4]
)

Task Arithmetic 的研究也揭示了大模型"知识组织"的一些深层规律。研究发现，不同类型的知识（如事实知识、推理能力、语言风格）在模型参数中有不同的编码方式。事实知识主要编码在模型的中间层和高层，以分布式的方式存储在注意力权重和前馈网络中。推理能力则更多地与模型的"计算图"相关——即信息在不同层之间的流动方式。语言风格主要体现在模型的输出层和解码策略中。

这种知识的分层组织为 Task Arithmetic 提供了重要的启示：不同类型的 Task Vector 可能需要不同的融合策略。例如，编码事实知识的 Task Vector 可以使用较大的 λ 值（因为事实知识的冲突较少），而编码推理能力的 Task Vector 可能需要较小的 λ 值（因为推理策略的冲突较多）。

此外，Task Arithmetic 还与"持续学习"(Continual Learning) 领域有深刻联系。在持续学习中，模型需要不断学习新知识而不遗忘旧知识。Task Arithmetic 提供了一种"事后整合"的方案——先独立学习每个任务，再通过向量运算将它们整合。这与持续学习中的"回放"(Replay) 和"正则化"(Regularization) 方法形成了互补。

def analyze_task_vector(task_vector):
    """分析 Task Vector 的统计特征"""
    all_params = torch.cat([v.float().flatten() for v in task_vector.values()])
    print(f"参数总数: {all_params.numel()}")
    print(f"均值: {all_params.mean().item():.6f}")
    print(f"标准差: {all_params.std().item():.6f}")
    print(f"最大值: {all_params.max().item():.6f}")
    print(f"最小值: {all_params.min().item():.6f}")
    print(f"零值比例: {(all_params.abs() < 1e-8).float().mean().item():.4f}")
    print(f"L2范数: {all_params.norm().item():.4f}")
