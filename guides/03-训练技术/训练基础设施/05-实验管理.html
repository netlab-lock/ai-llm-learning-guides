🧪 实验管理

1. 为什么需要实验管理

LLM 训练的实验成本极高（每次实验可能花费数千到数万美元），因此需要系统化的实验管理来避免重复实验、科学追踪结果、做出最优决策。本章将系统介绍超参数搜索方法、实验追踪系统、数据版本管理以及实验复现性的最佳实践。掌握这些技能不仅能节省大量计算资源，还能加速模型迭代周期，让团队在有限预算内找到最优的训练方案。

实验管理的必要性: 没有实验管理: 实验 #1: lr=1e-4, 结果不错... 但没记录 → 无法复现 实验 #2: lr=? 之前用的什么来着? → 重复犯错 实验 #3: 效果突然变好 → 不知道是哪个改动导致的 → 浪费大量GPU时间和人力成本 有实验管理: 实验 #1: lr=1e-4, bs=1024, loss=1.56 → ✓ 记录 实验 #2: lr=3e-4, bs=2048, loss=1.89 → ✗ 不如#1 实验 #3: lr=5e-5, bs=4096, loss=1.45 → ✓ 最佳 → 快速找到最优方案，可复现
💡 实用技巧：实验管理的第一步不是使用复杂工具，而是养成习惯：每次实验都记录完整的配置（包括随机种子、数据版本、环境信息）。即使是简单的文本文件也比没有记录好得多。推荐使用YAML配置文件 + Git版本控制作为最基础的实验管理方案。
⚠️ 注意事项：最常见的错误是"这次实验只是快速测试，不用记录"——结果发现效果很好，却无法复现。建议即使是5分钟的快速测试也要记录基本配置。一个简单的实验命名规范就能解决大部分问题，例如：{日期}_{模型}_{关键参数}。
🔍 深入理解：从科学方法论的角度看，实验管理的本质是"可证伪性"——每次实验都是对一个假设的检验。系统化的实验管理确保了：① 每个假设都有明确的记录；② 实验结果可以独立复现；③ 可以通过对比实验确定因果关系。这是从"炼丹"走向"工程"的关键一步。

2. 超参数搜索

2.1 搜索方法对比

超参数搜索是LLM训练中最耗时但也最重要的环节之一。不同的搜索方法在效率、成本和效果上差异显著。

2.1 搜索策略选择指南

超参数搜索策略的选择应基于可用计算预算和参数空间大小。对于参数少于3个的小规模搜索，Grid Search足够；对于3-10个参数，Random Search是最佳选择；对于计算预算有限但参数较多的场景，Bayesian优化更高效。在实际应用中，建议先用Random Search做粗筛（20-50次实验），找到最优区域后再用Bayesian优化做精调。

方法原理优势劣势适用场景
Grid Search遍历所有组合简单、完整指数级成本参数少 (< 3)
Random Search随机采样高效、可并行可能遗漏参数多
Bayesian基于历史优化智能、高效实现复杂计算预算有限
Hyperband早停+资源分配资源高效需要评估信号大规模实验
Population Based进化算法自动调整需要大量资源超大规模实验

2.2 关键超参数范围

以下是LLM训练中最关键的超参数及其推荐范围。合理的初始值可以大大缩小搜索空间。

超参数典型范围影响调优建议敏感度
Learning Rate1e-5 ~ 3e-3训练速度和稳定性先用 1e-4，观察 loss 曲线高
Batch Size64 ~ 8192梯度估计质量越大越稳定，但需要更多显存中
Warmup Steps100 ~ 2000训练初期稳定性通常 1-5% 总步数低
Weight Decay0.01 ~ 0.1正则化强度0.1 是常用默认值低
Gradient Clipping0.5 ~ 1.0防止梯度爆炸1.0 是常用默认值低
💡 实用技巧：超参数搜索的高效策略：先用小模型（如1B）快速筛选学习率和batch size的最优范围，再用大模型（如7B/13B）做精细调优。小模型上的最优超参数通常可以迁移到大模型（通过μP或线性缩放规则），节省90%以上的搜索成本。
⚠️ 注意事项：超参数搜索中最隐蔽的陷阱是"评估数据泄漏"——如果超参数选择是基于测试集的性能，那么最终的测试结果就是有偏的。建议将数据分为训练集、验证集和测试集三部分，超参数搜索只使用验证集。
🔍 深入理解：为什么Random Search通常比Grid Search更高效？Bergstra & Bengio (2012) 的研究表明，对于大多数机器学习问题，只有少数超参数对性能有显著影响（"低有效维度"）。Random Search在这个低维子空间上的覆盖密度远高于Grid Search。这解释了为什么在10次随机搜索中就能找到接近最优的超参数组合。

3. 实验追踪系统

实验追踪系统是实验管理的核心基础设施，它自动记录每次实验的配置、结果和元数据，使得团队能够系统化地对比不同实验方案的效果。

3.1 追踪系统架构与选型

实验追踪系统架构: ┌─────────────────────────────────────────────┐ │ 实验 #1: lr=1e-4, bs=1024, warmup=500 │ │ loss: 2.34 → 1.56 (1000 steps) │ │ GPU: 8x H100, 时间: 12h │ │ 状态: ✓ 完成 │ ├─────────────────────────────────────────────┤ │ 实验 #2: lr=3e-4, bs=2048, warmup=1000 │ │ loss: 2.34 → 1.89 (1000 steps) │ │ GPU: 8x H100, 时间: 10h │ │ 状态: ⚠ loss spike @ step 800 │ ├─────────────────────────────────────────────┤ │ 实验 #3: lr=5e-5, bs=4096, warmup=200 │ │ loss: 2.34 → 1.45 (1000 steps) │ │ GPU: 8x H100, 时间: 15h │ │ 状态: ✓ 最佳 │ ├─────────────────────────────────────────────┤ │ 实验 #4: lr=1e-4, bs=1024, wd=0.01 │ │ loss: 2.34 → 1.62 (1000 steps) │ │ 状态: 进行中 (step 650) │ └─────────────────────────────────────────────┘
实验追踪工具对比: ┌──────────────┬───────────┬───────────┬───────────┐ │ 工具 │ 部署方式 │ 协作能力 │ 适用场景 │ ├──────────────┼───────────┼───────────┼───────────┤ │ W&B │ 云端SaaS │ ★★★★★ │ 团队协作 │ │ MLflow │ 自托管 │ ★★★★ │ 企业内部 │ │ TensorBoard │ 本地 │ ★★ │ 个人实验 │ │ ClearML │ 混合 │ ★★★★ │ 端到端MLOps│ │ Aim │ 自托管 │ ★★★ │ 开源替代 │ └──────────────┴───────────┴───────────┴───────────┘
💡 实用技巧：实验追踪的最佳实践：① 使用统一的配置文件格式（YAML/JSON）记录所有超参数；② 每次实验自动生成唯一的实验ID；③ 在实验名中包含关键超参数信息（如lr1e4_bs1024）；④ 定期导出实验报告供团队review。
⚠️ 注意事项：实验追踪系统的一个常见问题：记录了太多指标导致查询缓慢。建议对指标进行分层记录——每10步记录详细指标（loss、lr），每100步记录汇总指标（avg_loss、throughput），每1000步记录评估指标（eval_loss、benchmark分数）。
🔍 深入理解：实验追踪系统的数据模型通常采用"实验-运行-指标"三层结构。实验（Experiment）定义了研究目标，运行（Run）是一次具体的训练执行，指标（Metric）是训练过程中的数值记录。这种层次结构使得可以轻松地对比同一实验的不同运行，或者跨实验分析趋势。

4. W&B Sweep 实战

W&B Sweep 是当前最易用的超参数搜索工具之一，它支持多种搜索策略并提供了直观的可视化界面和自动化的实验编排能力。

4.1 Sweep 配置与启动

# W&B 超参数搜索配置
sweep_config = {
    "method": "bayes",  # 贝叶斯优化
    "metric": {"name": "eval/loss", "goal": "minimize"},
    "parameters": {
        "learning_rate": {
            "min": 1e-5, "max": 3e-3,
            "distribution": "log_uniform"
        },
        "batch_size": {
            "values": [256, 512, 1024, 2048]
        },
        "warmup_steps": {
            "min": 100, "max": 2000
        },
        "weight_decay": {
            "values": [0.01, 0.05, 0.1]
        }
    }
}

# 启动 sweep
sweep_id = wandb.sweep(sweep_config, project="llm-hyperparam")
wandb.agent(sweep_id, function=train, count=20)  # 运行 20 次实验
💡 实用技巧：W&B Sweep 的一个高级功能：可以在搜索过程中动态调整搜索空间。如果发现learning_rate在某个范围内效果明显更好，可以手动缩小搜索范围到那个区域进行更精细的搜索。
⚠️ 注意事项：使用W&B Sweep时，一次只改一个变量。如果同时改多个参数，你无法确定是哪个改动带来了效果变化或引入了问题。建议先用Random Search做粗筛，再用Bayesian优化做精调。
🔍 深入理解：贝叶斯优化的核心原理：通过高斯过程（GP）对目标函数建模，在每次实验后更新后验分布，然后选择"期望提升"（Expected Improvement）最大的点作为下一次实验的参数。这比随机搜索更智能，但也更依赖于GP模型的准确性。在高维空间中（>10个超参数），GP模型可能失效，此时Random Search反而更可靠。

5. 数据版本管理

数据版本管理是实验可复现性的基础。没有数据版本管理，就无法确定模型效果的变化是由代码改动还是数据改动引起的。在LLM训练中，数据质量对模型性能的影响往往超过模型架构的影响，因此精确的数据版本管理是实验管理中不可或缺的环节。

5.1 数据版本管理工具选型

工具特点适用场景学习曲线
DVCGit 式的数据版本管理团队协作、可复现中
HuggingFace Hub模型和数据集托管开源分享低
Delta Lake数据湖版本管理大规模数据高
Git LFSGit 大文件存储小团队低
LakeFSGit式数据湖操作S3兼容存储中
💡 实用技巧：数据版本管理的最小可行方案：使用Git记录数据配置文件（包含数据路径、预处理参数），使用SHA256哈希值标识数据文件。这样即使没有DVC，也能通过配置文件+哈希值精确复现数据状态。
⚠️ 注意事项：数据版本管理中的隐蔽陷阱：数据泄漏。确保评估数据完全没有出现在训练集中。即使部分重叠也会导致评估结果严重失真。建议在数据准备阶段就进行严格的去重和泄漏检查。
🔍 深入理解：数据版本管理的工程哲学：在大规模数据场景下，简单但规模大往往胜过复杂但规模小。Chinchilla定律和Scaling Law都指向同一个方向——数据和计算的规模比精巧的算法设计更重要。因此，投资于数据质量（去重、清洗、过滤）的回报通常远高于投资于模型架构创新。

6. 实验复现性

6.1 实验复现性的关键要素

实验复现性是科学实验的基本要求，也是工业实践中保证质量的关键。一个实验要能被完全复现，需要记录以下六个维度的信息：代码版本、数据版本、环境配置、超参数、随机种子和硬件配置。缺少任何一个维度都可能导致复现失败。在LLM训练场景中，由于涉及的随机性来源更多（数据shuffle、dropout、分布式通信的非确定性），复现性保障比传统机器学习更加困难。

实验复现性的关键因素: ┌─────────────────────────────────────────────┐ │ 代码版本 │ │ └─ Git commit hash │ ├─────────────────────────────────────────────┤ │ 数据版本 │ │ └─ DVC hash / 数据集版本号 │ ├─────────────────────────────────────────────┤ │ 环境配置 │ │ └─ Docker image / requirements.txt │ ├─────────────────────────────────────────────┤ │ 超参数 │ │ └─ config.yaml / W&B config │ ├─────────────────────────────────────────────┤ │ 随机种子 │ │ └─ seed = 42 (固定所有随机源) │ ├─────────────────────────────────────────────┤ │ 硬件配置 │ │ └─ GPU 型号、数量、网络拓扑 │ └─────────────────────────────────────────────┘
通俗类比：实验管理就像"科学实验记录本"——每次实验的参数、结果、观察都要详细记录。没有记录本，你可能重复做同一个失败的实验；有了记录本，你可以快速找到最佳方案。W&B Sweep 就像"自动化实验助手"——它会智能地帮你选择下一组超参数，比手动 grid search 高效 10 倍。
常见误区：① 认为实验管理是"可选的"——当你训练了 100 个实验后，如果没有系统化管理，根本记不住哪个实验用了什么参数；② 忽略数据版本管理——模型效果变了，是因为改了代码还是改了数据？没有 DVC 等工具，很难回答这个问题。

7. 实验管理工程实践

将实验管理从理论转化为实践需要建立标准化的工作流程和工具链。以下是实验管理工程实践的关键步骤。一个成熟的实验管理流程应该覆盖实验的完整生命周期：从假设提出、实验设计、执行监控到结果分析和知识沉淀。每个阶段都有对应的工具支持和最佳实践。

7.1 配置管理与实验命名

首先是实验配置管理。建议使用YAML格式的配置文件来定义每次实验的所有参数，包括模型架构、训练超参数、数据配置和环境信息。配置文件应该支持继承和覆盖——基础配置定义默认值，实验特定的配置覆盖部分参数。这可以避免大量重复的配置文件，同时保持灵活性。Hydra是一个优秀的配置管理框架，它支持配置组合、命令行覆盖和配置验证。

其次是实验命名和组织规范。建议采用结构化的实验命名格式，例如：{项目}_{模型规模}_{日期}_{关键特征}。例如：llm7b_20240115_lr_sweep。实验结果应该按项目和日期组织存储，每个实验的结果目录包含：配置文件副本、训练日志、checkpoint路径和评估结果。这种组织方式使得即使在100个实验之后，也能快速找到特定实验的信息。

# Optuna超参数搜索 (替代W&B)
import optuna

def objective(trial):
    lr = trial.suggest_float("lr", 1e-5, 3e-3, log=True)
    bs = trial.suggest_categorical("batch_size", [256, 512, 1024])
    loss = train_model(lr, bs)
    return loss

study = optuna.create_study(direction="minimize")
study.optimize(objective, n_trials=20)
print(f"Best: {study.best_params}")

第三是实验比较和分析流程。建议建立定期的实验review机制——每周或每两周召开实验review会议，对比最近的实验结果，讨论发现和下一步方向。在review时，使用W&B的comparison功能可以直观地对比多个实验的loss曲线、超参数和评估指标。对于关键实验，建议进行消融实验（ablation study），逐一移除改动来确定每个改动的贡献。

最后是实验结果的归档和知识沉淀。对于有价值的实验（无论是成功还是失败），都应该编写简短的实验报告，记录实验假设、方法、结果和结论。这些报告是团队的宝贵知识资产，可以避免重复犯错，加速后续的实验设计。建议使用Notion或Confluence等协作工具来管理实验报告，便于搜索和引用。

实验预算与资源规划

大规模LLM训练的实验预算规划需要考虑GPU时租成本、数据存储成本和人力成本三个维度。一个典型的7B模型训练实验周期约为2-4周，使用64张A100的成本约为$50,000-100,000。建议在项目初期制定明确的实验预算分配：基线实验占30%、超参搜索占40%、消融实验占20%、意外开销预留10%。使用W&B的实验对比功能可以快速筛选出最有价值的实验方向，避免在低效方向上浪费算力。

实验结果可视化与团队协作

实验结果的有效呈现对于团队决策至关重要。除了自动化的指标对比外，建议建立结构化的实验评审机制：每个重要实验完成后，提交一份简短的实验报告，包含假设、方法、结果、结论和下一步计划。W&B Reports和Notion等工具可以支持这种协作式实验管理。定期的实验评审会议可以帮助团队对齐方向、避免重复劳动、及时发现有前景的实验方向。
