🖥️ 训练集群运维

1. 背景与动机

LLM 训练从单机多卡发展到多机多卡，再到万卡集群。当 GPU 数量超过 100 时，集群运维就成为决定训练效率的关键因素。一次配置错误可能导致数千 GPU 小时的浪费。本章将系统介绍集群调度系统（SLURM和Kubernetes）、存储架构、NCCL网络配置以及集群监控与故障排查的最佳实践。

集群规模演进: 单机多卡 (2-8 GPU): ┌─────────────────────┐ │ 1台服务器 × 8 GPU │ │ NVLink 互连 │ │ 适合: 7B模型调试 │ └─────────────────────┘ 小规模集群 (16-128 GPU): ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ │ Node1│ │ Node2│ │ Node3│ │ Node4│ │ 8 GPU│ │ 8 GPU│ │ 8 GPU│ │ 8 GPU│ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └────────┴────────┴────────┘ InfiniBand 互连 适合: 7B-70B模型训练 大规模集群 (1000+ GPU): ┌─────────────────────────────────────┐ │ 数百个节点 │ │ 多层交换机拓扑 (Spine-Leaf) │ │ InfiniBand NDR 400Gb/s │ │ 适合: 70B-405B模型训练 │ └─────────────────────────────────────┘
💡 实用技巧：集群运维的第一步是建立标准化的节点配置模板。使用Ansible或Terraform进行节点配置管理，确保所有节点的软件环境、驱动版本、网络配置完全一致。环境不一致是集群问题最常见的根因之一。
⚠️ 注意事项：集群运维中最隐蔽的问题是"配置漂移"——随着手动修改和软件更新，节点之间的配置逐渐变得不一致。建议定期运行配置一致性检查脚本，对比所有节点的关键配置项。
训练集群运维的核心概念与技术分析： 训练集群运维的关键指标是 GPU 有效利用率（MFU, Model FLOPs Utilization）。理论峰值利用率受限于模型的算术强度和通信开销，实际中 40-60% 的 MFU 已属优秀。运维优化方向包括：(1) 节点调度——使用 SLURM 或 Kubernetes 进行资源调度，合理分配亲和性规则以减少跨 NUMA 和跨交换机通信；(2) 故障管理——建立 GPU 健康检查脚本（检测 XID 错误、ECC 错误、温度异常），实现自动隔离故障节点；(3) 存储优化——使用 NVMe SSD 作为本地缓存加速数据加载，通过 NFS 或 Lustre 共享文件系统管理检查点；(4) 网络监控——持续监控 InfiniBand 端口的错误计数和带宽利用率，及时发现链路退化。

练习 1：写一个 SLURM 训练脚本，在 2 个节点上启动分布式训练（每节点 8 GPU）。配置 NCCL 环境变量。

练习 2：使用 nvidia-smi 和 ibstat 检查你的集群健康状态，记录 GPU 温度、ECC 错误和 IB 链路状态。

练习 3：编写一个自动化健康检查脚本，定期检查GPU状态、网络连通性和存储空间，并在异常时发送告警。
