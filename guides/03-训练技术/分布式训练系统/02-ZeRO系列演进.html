02 ZeRO 系列演进

什么是 ZeRO？

ZeRO（Zero Redundancy Optimizer）是微软 DeepSpeed 团队提出的训练优化技术，核心思想是：消除数据并行中的冗余存储。

传统数据并行中，每张 GPU 都存储完整的模型参数、梯度、优化器状态 —— 这是巨大的浪费。ZeRO 把这些状态分片到各张 GPU 上，每张只存 1/N，需要时再通信获取。

⚠️ 注意事项：ZeRO 的配置参数（如 reduce_bucket_size、overlap_comm）在不同模型规模和 GPU 数量下最优值不同。务必在你的实际硬件上做基准测试，不要直接套用网上示例的参数。
🔍 深入理解：「什么是 ZeRO？」的深层理解：从系统设计的角度看，GPU的设计体现了关注点分离的原则——每个组件只负责一个明确的功能，通过清晰的接口协作。这种设计哲学是构建可维护、可扩展系统的基础。

显存分析基础

混合精度训练中，模型的显存占用由三部分组成：

Ψ = 模型参数量（以元素个数计）

┌────────────────────────────────────────────┐ │ 组件 │ 精度 │ 显存(字节) │ ├────────────────────────────────────────────┤ │ 参数 (FP16) │ 2B/参数 │ 2Ψ │ │ 梯度 (FP16) │ 2B/参数 │ 2Ψ │ │ Adam 状态 │ │ │ │ ├ 一阶矩 m (FP32)│ 4B/参数 │ 4Ψ │ │ └ 二阶矩 v (FP32)│ 4B/参数 │ 4Ψ │ │ └ FP32 参数副本 │ 4B/参数 │ 4Ψ │ ├────────────────────────────────────────────┤ │ 总计 │ │ 16Ψ │ └────────────────────────────────────────────┘ 例：7B 模型 → 16 × 7×10⁹ = 112 GB（不含激活值）
💡 实用技巧：显存分析的核心公式 Ψ=16Ψ 是选择 ZeRO 策略的起点。建议先用 torch.cuda.memory_summary() 实测你模型的真实显存占用，再与理论值 16Ψ 对比。如果差距过大，说明有额外的激活值或碎片化开销，需要配合激活值检查点来解决。
⚠️ 注意事项：显存分析中常见的陷阱是忽略了 FP16 的边界条件：梯度溢出（overflow）和下溢（underflow）在混合精度训练中很常见。务必配合动态 loss scaling 使用，DeepSpeed 的 loss_scale: 0 就是自动动态调整。
🔍 深入理解：显存分析的 16Ψ 公式假设使用 Adam 优化器。如果换成 SGD（只有动量状态），显存只需 8Ψ；换成 Adafactor（分解二阶矩），可以进一步压缩。选择优化器时要考虑显存/收敛速度的权衡。

ZeRO-1：分片优化器状态

ZeRO-1（分片优化器状态）： GPU 0: [参数完整] [梯度完整] [优化器状态 1/N] GPU 1: [参数完整] [梯度完整] [优化器状态 1/N] GPU 2: [参数完整] [梯度完整] [优化器状态 1/N] 更新步骤： 1. 各卡独立前向+反向 → 得到本地梯度 2. AllReduce 梯度 → 各卡得到相同平均梯度 3. 各卡只更新自己负责的那部分参数 4. AllGather 更新后的参数 → 各卡同步 显存：2Ψ + 2Ψ + 16Ψ/N ≈ 4Ψ + 16Ψ/N
ZeRO-1 显存 = 4Ψ + 16Ψ/K_d
K_d = 数据并行度（GPU 数量）
8 GPU 时：4Ψ + 2Ψ = 6Ψ（比原始 16Ψ 节省 62.5%）
💡 实用技巧：ZeRO-1 的核心收益是将 Adam 的 m、v、master copy 三个 FP32 状态从 12Ψ 降到 12Ψ/N。实测时用 deepspeed --num_gpus N 启动，通过 deepspeed.runtime.zero.stage_1_and_2 的日志确认优化器状态确实被分片。注意：ZeRO-1 的通信量与标准 DDP 相同（2Ψ），几乎没有额外开销。
⚠️ 注意事项：ZeRO-1 的常见错误是跳过 baseline 直接上复杂方案。建议先用标准 DDP 跑一个 baseline，记录显存和吞吐量，再切换到 ZeRO-1 对比。没有 baseline 数据，你无法量化 ZeRO-1 的实际收益。
🔍 深入理解：ZeRO-1 的设计哲学是「只分片不参与计算的状态」——优化器状态只在参数更新时使用，不在前向/反向传播中使用。因此分片它们不会增加通信量。这个洞察可以推广：任何「计算时不需要，更新时才需要」的状态都可以安全分片。

ZeRO-2：分片优化器状态 + 梯度

ZeRO-2（分片优化器状态 + 梯度）： GPU 0: [参数完整] [梯度 1/N] [优化器状态 1/N] GPU 1: [参数完整] [梯度 1/N] [优化器状态 1/N] GPU 2: [参数完整] [梯度 1/N] [优化器状态 1/N] 反向传播时： - 某层梯度计算完 → ReduceScatter（各卡只保留自己负责的梯度部分） - 不需要的梯度立即释放 - 优化器只更新自己负责的参数 - AllGather 同步更新后的参数
ZeRO-2 显存 = 2Ψ + (2Ψ + 16Ψ)/K_d
8 GPU 时：2Ψ + 2.25Ψ = 4.25Ψ（比原始节省 73%）
💡 实用技巧：ZeRO-2 将梯度也分片，关键变化是反向传播时用 ReduceScatter 替代 AllReduce——每卡只保留自己负责的那部分梯度。实测收益：8 卡时显存从 6Ψ 降到 4.25Ψ。适合模型能放入单卡但显存余量不足以跑较大 batch size 的场景。
⚠️ 注意事项：ZeRO-2 的关键变量是梯度分桶大小（reduce_bucket_size）。太小会导致大量小通信操作，太大会增加峰值显存。建议从 5e8 开始，根据你的 GPU 数量和显存逐步调整。
🔍 深入理解：ZeRO-2 用 ReduceScatter 替代 AllReduce 是一个巧妙的观察：AllReduce = ReduceScatter + AllGather，而 ZeRO-2 只需要 ReduceScatter 的结果（每卡持有自己的梯度分片），AllGather 留到参数更新后才做。这一步优化消除了梯度的冗余存储。

ZeRO-3：分片一切

ZeRO-3（分片参数 + 梯度 + 优化器状态）： GPU 0: [参数 1/N] [梯度 1/N] [优化器状态 1/N] GPU 1: [参数 1/N] [梯度 1/N] [优化器状态 1/N] GPU 2: [参数 1/N] [梯度 1/N] [优化器状态 1/N] 前向传播： - 计算某层前 → AllGather 收集完整参数 - 计算完 → 释放非本地参数分片 反向传播： - 计算某层反向 → AllGather 收集完整参数 - 梯度计算完 → ReduceScatter 分片梯度 - 释放非本地参数分片
ZeRO-3 显存 = 16Ψ/K_d
8 GPU 时：2Ψ（比原始节省 87.5%）
代价：通信量增加 1.5×（多了参数的 AllGather）
💡 实用技巧：ZeRO-3 的关键代价是前向传播也需要 AllGather 收集完整参数，通信量从 2Ψ 增加到 3Ψ。实战中务必开启 overlap_comm: true 和 prefetch_bucket_size 参数，让 AllGather 与当前层计算重叠，可将实际性能损失控制在 5-15%。
⚠️ 注意事项：ZeRO-3 的隐蔽陷阱是参数持久化（persistence）。小参数（如 LayerNorm 的权重）频繁 AllGather 的开销可能大于收益。stage3_param_persistence_threshold: 1e6 让小于 1M 元素的参数保持完整不分片，是一个重要的优化。
🔍 深入理解：ZeRO-3 的理论显存为 16Ψ/N，当 N 足够大时，单卡显存可以忽略不计。但实际中激活值（activation）才是大头——7B 模型在 seq_len=2048 时激活值可能超过参数显存。ZeRO-3 解决了参数/梯度/优化器的显存，激活值需要靠梯度检查点（gradient checkpointing）来优化。

三个阶段对比

特性原始 DPZeRO-1ZeRO-2ZeRO-3
分片内容无优化器状态+梯度+参数
显存 (8GPU)16Ψ6Ψ4.25Ψ2Ψ
通信量2Ψ2Ψ2Ψ3Ψ
适用场景小模型中模型大模型超大模型
💡 实用技巧：选型口诀：模型能放入单卡选 ZeRO-1（零额外通信），显存紧张选 ZeRO-2（通信不变），单卡放不下参数必须 ZeRO-3。7B 模型在 8×A100 上，ZeRO-2 通常是最优平衡点——显存足够、通信量最小。超大模型（100B+）才需要 ZeRO-3 + Offload。
⚠️ 注意事项：对比三个阶段时，不要只看显存节省。通信量是关键：ZeRO-1/2 的通信量与 DDP 相同（2Ψ），ZeRO-3 多 50%（3Ψ）。在带宽受限的环境（如跨节点通信），ZeRO-3 的性能损失可能比显存节省更重要。
🔍 深入理解：ZeRO 的三个阶段体现了渐进式优化的思想：从优化器状态（最大冗余）→ 梯度（中等冗余）→ 参数（完全冗余），每一步都消除更多冗余但增加更多通信。这个演进模式在其他分布式系统中也很常见，如分布式数据库的分片策略。

ZeRO-Offload：CPU/NVMe 卸载

当 GPU 显存仍然不够时，可以把部分状态卸载到 CPU 内存甚至 NVMe SSD。

注意：CPU Offload 会显著降低训练速度（PCIe 带宽 vs NVLink），只在显存确实不够时使用。

💡 实用技巧：CPU Offload 适合「显存不够但训练速度可以妥协」的场景。关键参数：pin_memory: true 避免额外拷贝。PCIe 4.0 x16 带宽 32GB/s，7B 模型每步约 1.75 秒通信开销。NVMe Offload 只在 100B+ 且 CPU 内存也不够时使用，性能损失可达 50-100%。
🔍 深入理解：ZeRO-Offload 的设计动机是「用时间换空间」——CPU 内存（512GB-2TB）和 NVMe（数TB）远大于 GPU HBM（80GB），但带宽也低 1-2 个数量级。Offload 的关键是通过流水线化（pipelining）将数据传输与计算重叠，尽量隐藏传输延迟。

ZeRO++：通信优化

ZeRO++ 解决了 ZeRO-3 通信量增加的问题：

  • qwZ (Quantized Weights ZeRO)：用 INT8/FP16 传输权重（而非 FP32），通信量减半
  • hpZ (Hierarchical Partitioning ZeRO)：节点内用 ZeRO-3，节点间用 ZeRO-1，利用 NVLink 高带宽
  • pgZ (Pascal/Gradient ZeRO)：梯度分层通信优化
💡 实用技巧：ZeRO++ 的三个优化中，hpZ（层级分区）最实用：节点内用 NVLink 高速通信保持 ZeRO-3，节点间降为 ZeRO-1 只分片优化器状态。对于多节点训练，hpZ 可减少 50% 以上的跨节点通信量。qwZ（量化权重传输）在带宽受限的场景效果显著，通信量减半。
⚠️ 注意事项：ZeRO++ 的 hpZ 策略需要正确配置节点拓扑。确保 NCCL 能正确识别节点内/节点间的通信边界，通过 NCCL_SOCKET_IFNAME 和 NCCL_IB_DISABLE 等环境变量控制。错误的拓扑配置会导致 hpZ 退化为普通的 ZeRO-3。
🔍 深入理解：「ZeRO++：通信优化」的深层理解：从优化理论的角度看，FP16本质上是一个在约束条件下寻找最优解的问题。理解约束条件的来源和性质，比盲目调参更有价值。

实际配置示例

# DeepSpeed ZeRO-3 配置
{
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {
      "device": "cpu",
      "pin_memory": true
    },
    "offload_param": {
      "device": "none"       # 不卸载参数到 CPU
    },
    "overlap_comm": true,     # 通信与计算重叠
    "contiguous_gradients": true,
    "reduce_bucket_size": 5e8,
    "stage3_prefetch_bucket_size": 5e8,
    "stage3_param_persistence_threshold": 1e6
  },
  "fp16": {
    "enabled": true,
    "loss_scale": 0,          # 动态 loss scaling
    "initial_scale_power": 16
  },
  "train_batch_size": 32,
  "gradient_accumulation_steps": 4,
  "gradient_clipping": 1.0
}
通俗类比：ZeRO 就像"拼车"——传统数据并行中，每辆车（GPU）都装满所有行李（完整模型状态），浪费大量空间。ZeRO 的做法是：把行李分摊到所有车上，每辆车只装 1/N 的行李，需要时再互相传递。ZeRO-1 分摊"工具箱"（优化器状态），ZeRO-2 再分摊"行李"（梯度），ZeRO-3 连"车架"（参数）都分摊了——越来越省空间，但通信越来越多。
深入探讨：ZeRO 的核心洞察是"冗余消除"——在传统数据并行中，N 张 GPU 存了 N 份完全相同的优化器状态，这是巨大的浪费。但消除冗余的代价是通信：原本只需要 AllReduce 梯度，现在还需要在参数更新时获取其他 GPU 上的分片。ZeRO++ 的创新在于用"分层分区"（hierarchical partitioning）来平衡通信和内存：节点内用 NVLink 高速通信，节点间用低速网络，通过智能调度最小化跨节点通信。
⚠️ 注意事项：配置中容易忽略的陷阱：stage3_gather_16bit_weights_on_model_save: true 必须开启，否则保存的 checkpoint 是分片的，无法直接用于推理。overlap_comm: true 和 contiguous_gradients: true 应该始终开启。此外，不要只关注平均性能指标，P95/P99 延迟、最坏情况表现、资源峰值占用等指标同样重要。

ZeRO 系列的通信量与显存数学推导

理解 ZeRO 的关键在于掌握其通信量和显存占用的数学公式。本节将给出严格的推导过程，帮助读者建立直觉。

显存占用公式推导

设模型参数量为 Ψ，数据并行度为 N。在标准数据并行（DDP）中，每个 GPU 需要存储完整的模型参数、梯度和优化器状态。

DDP 每 GPU 显存: M_DDP = 2Ψ + 2Ψ + (4Ψ + 4Ψ + 4Ψ) = 16Ψ bytes 其中：参数(2Ψ, BF16) + 梯度(2Ψ, BF16) + 优化器状态(Adam: m=4Ψ + v=4Ψ + master=4Ψ, FP32)

ZeRO 的三个阶段分别对参数（Ψ）、梯度（G）和优化器状态（OS）进行分片：

ZeRO 阶段分片内容每 GPU 显存7B 模型(64GPU)显存节省
DDP (无分片)无16Ψ112 GB基线
ZeRO-1优化器状态2Ψ+2Ψ+12Ψ/N57.3 GB~4x
ZeRO-2优化器+梯度2Ψ+4Ψ/N+12Ψ/N15.6 GB~8x
ZeRO-3全部16Ψ/N1.75 GB~64x

通信量分析

ZeRO 的通信量分析需要考虑前向和反向传播中的 AllGather 和 ReduceScatter 操作。

DDP 通信量 = 2Ψ (一次 AllReduce = ReduceScatter + AllGather) ZeRO-1 通信量 = 2Ψ (与 DDP 相同) ZeRO-2 通信量 = 2Ψ (与 DDP 相同，但 ReduceScatter 替代 AllReduce) ZeRO-3 通信量 = 3Ψ (前向 AllGather Ψ + 反向 AllGather Ψ + 反向 ReduceScatter Ψ)

ZeRO-3 的通信量比 DDP 多 50%，但通过通信与计算的重叠，实际性能损失通常在 5-15% 以内。

ZeRO-Offload 与 ZeRO-Infinity

当 GPU 显存仍然不足时，ZeRO 可以将数据卸载到 CPU 内存（ZeRO-Offload）甚至 NVMe SSD（ZeRO-Infinity）。

ZeRO-3 + Prefetch 通信计算重叠： 时间 → GPU 0: [AG_L1][Compute_L1][RS_L1][AG_L2][Compute_L2][RS_L2] [AG_L2发起] ↑预取 GPU 1: [AG_L1][Compute_L1][RS_L1][AG_L2][Compute_L2][RS_L2] AG = AllGather (收集参数) RS = ReduceScatter (分发梯度) 关键: AG_L2 与 Compute_L1 重叠 无重叠总时间: T = L × (T_AG + T_compute + T_RS) 有重叠总时间: T ≈ L × max(T_compute, T_AG + T_RS) 节省: 当 T_compute > T_AG + T_RS 时接近零开销
ZeRO-Infinity 层次化存储架构： ┌────────────────────────────────────────────┐ │ GPU HBM (80GB × N) │ │ 当前层参数 + 激活值 + 梯度分片 │ │ 带宽: 3.35 TB/s (H100) │ ├────────────────────────────────────────────┤ │ CPU DRAM (512GB-2TB) │ │ 优化器状态 + 参数缓存 │ │ 带宽: 64 GB/s (PCIe 5.0) │ ├────────────────────────────────────────────┤ │ NVMe SSD (数TB, RAID) │ │ 参数分片存储 │ │ 带宽: 3-7 GB/s per SSD │ └────────────────────────────────────────────┘ 数据流: NVMe ←→ CPU ←→ GPU 预取策略: 提前 2-3 层将参数从 NVMe 读入 CPU 提前 1-2 层将参数从 CPU 传入 GPU

ZeRO 系列的适用场景选型指南

面对多种 ZeRO 变体，如何选择最适合自己场景的配置？以下是基于实践的选型建议。如果模型参数可以放入单 GPU 显存但显存余量不足，优先选择 ZeRO-1（仅分片优化器状态），通信开销几乎为零。如果显存更加紧张但通信带宽充裕，选择 ZeRO-2（额外分片梯度），通信量与标准数据并行相同。如果模型参数本身无法放入单 GPU，必须使用 ZeRO-3（全分片），代价是通信量增加约 50%。对于超大模型（100B+），可以组合使用 ZeRO-3 和 CPU/NVMe Offload，以存储换速度。

ZeRO 三阶段显存分片对比（4 GPU, 参数P=16GB）： DDP: GPU0:[16P+16G+48OS] GPU1:[16P+16G+48OS] 总:4×80=320GB ZeRO-1: GPU0:[16P+16G+12OS] GPU1:[同] 总:4×44+12×4=224GB ZeRO-2: GPU0:[16P+4G+12OS] GPU1:[同] 总:4×32+12×4=176GB ZeRO-3: GPU0:[4P+4G+12OS] GPU1:[同] 总:4×20=80GB 节省比: ZeRO-1=1.43x, ZeRO-2=1.82x, ZeRO-3=4x
ZeRO-3 前向传播参数收集时序： 时间 → GPU0: [AG_L1][L1计算][释放][AG_L2][L2计算][释放]... GPU1: [AG_L1][L1计算][释放][AG_L2][L2计算][释放]... ↓ NVLink/IB AllGather GPU0: [shard0+shard1+shard2+shard3] = 完整参数 显存峰值 = max(单层完整参数) + 激活值 比 DDP 低 N 倍 (N=GPU数) 优化: Prefetch 提前收集下一层参数 与当前层计算重叠，隐藏通信延迟
