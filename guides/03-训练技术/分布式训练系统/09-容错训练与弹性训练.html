09 容错训练与弹性训练

LLM 训练的故障现实

Llama 3 训练故障统计（Meta，2024）：

  • 16,384 张 H100 训练 54 天
  • 平均每 3 小时发生一次硬件故障
  • 总计约 400+ 次中断
  • 大部分是 GPU 硬件错误（ECC、XID 错误）

结论：故障是常态，不是异常。容错是 LLM 训练的必备能力。

⚠️ 注意事项：故障现实的工程启示：① Meta 的 Llama 3 经验表明，GPU 故障率约为每 3 小时一次（16384 卡集群），这意味着 checkpoint 间隔必须 ≤3 小时，否则故障恢复时丢失的工作量不可接受；② 故障类型分布：GPU 硬件错误（ECC/XID）占 ~60%，网络故障 ~20%，软件 bug ~15%，其他 ~5%；③ 不同集群的故障率差异很大——新硬件集群可能几天才出一次故障，老硬件集群可能每小时都有问题。
LLM 训练的故障现实的核心概念与技术分析： LLM 训练故障的统计分析基于硬件 MTBF（平均无故障时间）。以 A100 为例，单卡 MTBF 约 10,000 小时，但在大规模集群中，集群整体的 MTBF = 单卡 MTBF / (GPU 数量 × 故障率乘数)。对于 2048 卡集群，有效 MTBF 可能降至 5-10 小时。故障恢复的关键技术包括：(1) 异步检查点——使用 torch.distributed.checkpoint 将模型状态异步保存到共享存储，保存间隔通常设为 30-60 分钟；(2) 弹性训练——TorchElastic 支持动态调整 worker 数量，当节点故障时自动缩减集群规模继续训练；(3) 冗余通信——使用 NCCL 的故障检测机制和自动重路由，当某条 IB 链路故障时切换到备用路径。这些技术的共同目标是将有效训练时间占比从 70% 提升到 95% 以上。

容错训练的系统架构设计

在数千 GPU 的大规模集群中，硬件故障是常态而非异常。训练系统必须具备完善的容错机制，才能保证长周期训练任务的顺利完成。Google 的 TPU v4 集群数据显示，大规模训练中平均每几小时就会发生一次硬件故障。

故障分类与检测机制

分布式训练中常见的故障可以分为以下几类：

Checkpoint 策略优化

Checkpoint 是容错训练的基础。传统同步 checkpoint 需要暂停训练，导致训练效率降低。现代系统采用异步 checkpoint 技术，将 checkpoint 写入与训练计算重叠。

容错训练的监控与告警体系

一个完善的容错训练系统离不开全面的监控与告警体系。在大规模训练中，需要实时监控 GPU 状态、网络健康、存储性能、训练指标等多个维度。GPU 层面需要监控温度、功耗、ECC 错误和 XID 错误码；网络层面需要监控 IB 端口状态、丢包率和延迟；存储层面需要监控 IOPS 和带宽使用；训练层面需要监控 loss 曲线、梯度范数和学习率变化。当任何指标超出阈值时，系统应该自动触发告警并执行预定义的恢复策略。

训练恢复的时间成本分析

当发生节点故障时，从检测到恢复训练的总时间包括故障检测时间（通常30秒到2分钟）、checkpoint加载时间（取决于checkpoint大小和存储带宽，通常1到5分钟）、以及节点重新调度时间（Kubernetes环境下通常2到5分钟）。因此一次故障的总中断时间通常在5到15分钟之间。在大规模集群中，故障频率可能达到每天数次，因此快速恢复机制对于保证训练效率至关重要。
