05 3D 并行与混合并行

什么是 3D 并行？

3D 并行 = TP（张量并行） + PP（流水线并行） + DP（数据并行），三者组合使用以充分利用大规模 GPU 集群。

💡 实用技巧：3D 并行的核心约束：G = TP × PP × DP。经验法则——TP ≤ 8（节点内 NVLink 域），PP ≥ 1（跨节点只传激活值），DP = G/(TP×PP) 填满剩余。7B 模型用 TP=1,PP=1,DP=8 纯数据并行即可；70B 用 TP=8,PP=2,DP=4；405B 用 TP=8,PP=16,DP=8。先在 8 GPU 上验证 TP=8 的通信正确性。
⚠️ 注意事项：3D 并行的常见坑：① TP 跨节点会导致通信量剧增（AllGather/ReduceScatter 走 IB 带宽），所以 TP 必须限制在节点内 NVLink 域；② PP 的 microbatch 数过少会导致流水线 bubble 率 >50%，一般设 microbatch ≥ 4×PP_stages；③ DP 和 TP 的 GPU 拓扑不匹配会导致 AllReduce 走 PCIe 而非 NVLink，MFU 直接腰斩。
🔍 深入理解：3D 并行的深层逻辑：每种并行维度对应一类通信模式——TP 用 AllReduce/ReduceScatter 交换激活值（高频、小数据量），PP 用点对点通信传递微批次激活（中频、中数据量），DP 用 AllReduce 同步梯度（低频、大数据量）。理解这个通信层次有助于分析瓶颈：如果 TP 跨节点，高频小消息的延迟会主导训练时间。

如何决定 TP/PP/DP 的比例？

经验法则：

  • TP ≤ 节点内 GPU 数（8）— 需要最高带宽，必须在 NVLink 域内
  • PP ≥ 1 — 跨节点部署，带宽要求低（只传激活值）
  • DP = 总 GPU / (TP × PP) — 填满剩余维度
模型大小GPU数推荐配置理由
7B8TP=1, PP=1, DP=8单卡能放下，纯数据并行
70B64TP=8, PP=2, DP=4TP 切满节点内，PP 跨节点
405B1024TP=8, PP=16, DP=8深层流水线，大量数据并行
⚠️ 注意事项：TP/PP/DP 比例选择的常见坑：① TP 维度必须是 2 的幂且 ≤ 节点内 GPU 数（通常 8），否则 Megatron-LM 的列/行切分无法对齐；② PP 维度增加会线性增加 bubble 时间，除非用 1F1B 交错调度；③ DP 维度减小意味着每步有效 batch 变小，可能需要调整学习率。建议先用小规模（8 GPU）验证 TP=8 的通信正确性，再逐步扩展。
🔍 深入理解：TP/PP/DP 比例的核心权衡：TP 降低单 GPU 显存但增加通信（AllReduce 频率高），PP 降低通信量但增加 bubble 时间，DP 简单但梯度同步数据量大。Llama 3 405B 选择 TP=8,PP=16,DP=128 的原因是：TP=8 充分利用节点内 NVLink 带宽，PP=16 将模型层分布到 16 个 pipeline stage，DP=128 用数据并行填满剩余 GPU。PP 的 1F1B 交错调度将 bubble 率从 25% 降到 ~5%。

Expert Parallelism（EP）用于 MoE

MoE 模型额外需要一个并行维度：把不同专家分配到不同 GPU。

💡 实用技巧：EP 的通信模式是 AlltoAll（非 AllReduce），每个 token 通过 Router 选中 top-k 专家后，通过 AlltoAll 发送到对应专家所在 GPU。EP 的通信量 = 2×(N-1)/N × tokens × hidden_dim。监控重点：专家负载均衡度（理想情况各专家处理相近数量的 token）和 AlltoAll 延迟占总训练时间的比例。
⚠️ 注意事项：EP 的常见坑：① 专家负载不均衡会导致部分 GPU 成为瓶颈，需要用辅助损失（auxiliary loss）或容量因子（capacity factor）约束；② AlltoAll 通信的延迟与专家数和 GPU 数的乘积相关，EP 维度不宜超过 64；③ EP 与 TP 共享节点内通信，两者同时使用时 NVLink 带宽会被瓜分。建议先用 EP=1（纯 DP）做 baseline，再逐步增加 EP 维度观察 MFU 变化。
🔍 深入理解：EP 的深层设计考量：MoE 的稀疏激活（每次只激活 top-k 个专家）本质上是一种动态计算分配——给"简单"token 分配更少的计算。EP 的 AlltoAll 通信模式与 AllReduce 有本质区别：AllReduce 的通信量与数据大小成正比且带宽最优，而 AlltoAll 的延迟与 GPU 数成正比。这意味着 EP 的 scaling 上限比 DP 低，实际中 EP 维度通常不超过节点数。

Llama 3 405B 训练案例分析

Llama 3 405B 的训练配置（Meta，2024）：

  • 硬件：16,384 张 H100 80GB
  • 并行策略：TP=8, PP=16, DP=128
  • 序列长度：8192 tokens
  • 全局 batch size：约 16M tokens
  • 训练时间：54 天
  • MFU（Model FLOPs Utilization）：约 38-41%
⚠️ 注意事项：Llama 3 405B 训练的工程细节：① 16384 张 H100 上 MFU=38-41%，低于理论峰值的主因是 PP bubble 和通信开销；② 54 天训练中 400+ 次中断，checkpoint 间隔约 3 小时（≈10 分钟训练量）；③ 训练初期频繁调参，后期固定配置后故障率下降。复现时先用小规模验证 loss 曲线一致性，再扩展到大规模。
🔍 深入理解：Llama 3 的 3D 并行配置揭示了一个工程哲学：在超大规模训练中，稳定性比极致性能更重要。405B 模型选择 TP=8,PP=16,DP=128 而非更激进的 TP=16，是因为 TP 跨节点的通信可靠性远低于节点内 NVLink。PP=16 虽然 bubble 率更高，但 1F1B 调度将其控制在 ~5%，且 PP 的点对点通信比 TP 的 AllReduce 更容易容错。

通信量分析

每个 Transformer 层的通信量：

TP 通信：2 × 2 × h × s × b = 4hsb （前向+反向各 2 次 AllReduce）
PP 通信：h × s × b（传输激活值，单向）
DP 通信：2Ψ/DP（AllReduce 梯度，Ψ=参数量）

TP 通信量最大，但 NVLink 带宽 900 GB/s 足够
PP 通信量最小，可用较低带宽的 InfiniBand
DP 通信量中等，可与其他计算重叠

关键洞察：TP 放节点内（NVLink），PP 和 DP 跨节点（InfiniBand），这是 3D 并行的基本设计原则。

⚠️ 注意事项：通信量分析的常见误判：① 理论通信量（2D(N-1)/N）假设带宽完全饱和，实际受 IB 网络拥塞和 NCCL 算法开销影响，实测通常只有理论值的 70-85%；② 节点内 NVLink 通信（900 GB/s）和跨节点 IB 通信（400 Gb/s）的量纲不同（B vs b），换算后差距约 18 倍；③ 通信量分析必须结合 profiler 实测，不能只看理论公式。
🔍 深入理解：通信量分析的核心洞察：3D 并行的总通信量 = TP 通信（AllReduce 频率 × 激活大小）+ PP 通信（P2P 频率 × 微批次激活）+ DP 通信（AllReduce 频率 × 梯度大小）。当 TP=8,PP=16,DP=128 时，DP 通信量通常占总通信的 60-70%（因为梯度同步数据量最大），所以 DP 维度应优先放在 IB 带宽充足的维度上。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

3D 并行是将张量并行（TP）、流水线并行（PP）和数据并行（DP）组合使用的策略。正确配置三种并行度需要理解它们之间的数学关系和性能影响。

混合并行的工程实践经验

在实际的大模型训练项目中，3D并行的配置并非一次就能调好，需要经过多轮迭代优化。以下是来自工业界的宝贵实践经验。

首先，在集群规划阶段，需要充分评估网络拓扑。NVLink 和 InfiniBand 的带宽差异巨大（900 GB/s vs 50 GB/s），因此张量并行必须限制在单机内。其次，流水线并行的 micro-batch 数量需要根据显存大小和序列长度动态调整，通常需要进行多组实验找到最优配置。数据并行的梯度累积步数也需要精心设计，以平衡训练稳定性和吞吐量。

此外，混合并行中的通信重叠策略对性能有显著影响。在前向传播中，可以将 AllGather 通信与当前层的计算重叠；在反向传播中，ReduceScatter 可以与梯度计算并行执行。这种重叠技术通常能带来 10-30% 的性能提升。

并行度配置的实战经验总结

在大规模集群上进行3D并行训练时，张量并行度应尽量设置为单机GPU数量（通常为8），以充分利用NVLink的高带宽优势。流水线并行度的选择需要平衡气泡率和通信开销，一般建议micro-batch数量为流水线深度的4到8倍。数据并行度则由总GPU数自动确定。
