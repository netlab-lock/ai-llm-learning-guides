为什么 BF16 是 LLM 训练首选？
BF16 有和 FP32 相同的指数范围（8 bit 指数），不会出现 FP16 的溢出/下溢问题。虽然尾数精度只有 7 bit（vs FP16 的 10 bit），但对 LLM 训练足够。

⚠️ 注意事项：精度格式选择的常见坑：① BF16 虽然动态范围与 FP32 相同（8-bit 指数），但尾数精度只有 7 bit，在梯度累加时精度损失可能累积——所以优化器状态必须用 FP32；② FP16 的动态范围小（±65504），Loss Scaling 是必须的，但 scaling factor 过大会导致梯度溢出（Inf/NaN），过小会导致下溢为 0；③ TF32 在 A100/H100 上默认启用，但只有 Tensor Core 操作受影响，CPU 操作仍用 FP32。
🔍 深入理解：精度格式的演进逻辑：FP32→FP16（2017，HBM 带宽翻倍）→BF16（2020，A100 引入，无需 Loss Scaling）→FP8（2022，H100 引入，计算吞吐再翻倍）。每次精度降低都带来 ~2x 计算吞吐提升和 ~2x 显存节省，代价是精度损失。BF16 的出现是关键转折点——它保持了 FP32 的动态范围，消除了 FP16 最大的痛点（溢出/下溢），使得混合精度训练变得"无脑"。

混合精度训练原理

混合精度训练流程： ┌─────────────────────────────────────────────┐ │ 前向传播：FP16/BF16 参数 × FP16/BF16 输入 │ │ ↓ │ │ 损失计算：FP32 │ │ ↓ │ │ Loss Scaling：loss × 2^16（放大梯度） │ │ ↓ │ │ 反向传播：FP16/BF16 梯度 │ │ ↓ │ │ 梯度反缩放：grad / 2^16 │ │ ↓ │ │ 参数更新：FP32 master weights + FP32 Adam │ │ ↓ │ │ 权重转换：FP32 → FP16/BF16（用于下一迭代） │ └─────────────────────────────────────────────┘
💡 实用技巧：混合精度训练核心流程：前向用 BF16 计算，反向用 BF16 梯度，参数更新必须在 FP32 精度下进行。PyTorch 实现用 torch.autocast(device_type='cuda', dtype=torch.bfloat16) + GradScaler（BF16 不需要 scaler，FP16 必须）。FP32 master weights 是训练稳定性的关键，不可省略。
⚠️ 注意事项：混合精度训练的常见坑：① FP32 master weights 是训练稳定性的关键——如果用 FP16/BF16 直接更新参数，梯度的微小更新会被尾数精度截断（"stale weights" 问题），导致训练不收敛；② Loss Scaling 的动态调整（Dynamic Loss Scaling）会自动 halve/double scaling factor，但如果初始值设置不当，前几百步可能反复溢出；③ BatchNorm 的 running_mean/var 必须用 FP32 存储，否则统计量精度不够。
🔍 深入理解：混合精度训练的数学本质：前向传播和反向传播是"矩阵乘法密集型"操作，FP16/BF16 的 Tensor Core 可以将吞吐提升 2-8x（取决于 GPU 代次）。参数更新是"逐元素操作"，受显存带宽限制，精度降低带来的收益较小。所以最优策略是：计算密集部分用低精度（前向+反向），带宽密集部分用高精度（优化器状态+参数更新）。这就是"混合精度"的核心思想。

FP8 训练

Hopper/Blackwell GPU 的 FP8 Tensor Core 使 FP8 训练成为现实。

# 使用 NVIDIA Transformer Engine 进行 FP8 训练
import transformer_engine.pytorch as te
from transformer_engine.common.recipe import DelayedScaling

# 替换 nn.Linear 为 TE 的 Linear
model = te.TransformerLayer(
    hidden_size=4096,
    ffn_hidden_size=11008,
    num_attention_heads=32,
    fp8=True,
    fp8_recipe=DelayedScaling()
)

# FP8 训练循环
for batch in dataloader:
    with te.fp8_autocast(enabled=True):
        output = model(batch)
        loss = output.loss
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
💡 实用技巧：FP8 训练仅在 H100/H200（FP8 Tensor Core 1979 TFLOPS）上可用。使用 Transformer Engine 的 DelayedScaling recipe 自动调整 scaling factor，关注 fp8_meta 中的 amax 历史。前向用 E4M3（精度高，范围±448），反向用 E5M2（范围大，±57344）。FP8 可带来约 2x 计算吞吐提升。
⚠️ 注意事项：FP8 训练的常见坑：① FP8 E4M3 的范围只有 ±448，比 BF16（±3.4e38）小很多，权重和激活值需要 scaling 才能落在有效范围内；② Transformer Engine 的 DelayedScaling 用 amax history 窗口（默认 1024 步）来估算 scaling factor，窗口太大会响应迟钝，太小会导致 factor 震荡；③ FP8 训练的 loss 曲线通常比 BF16 有 1-2% 的波动，这是正常的，不要误以为训练不稳定。
🔍 深入理解：FP8 训练的核心挑战是"动态范围管理"——BF16 有 8-bit 指数（范围 ±3.4e38），FP8 E4M3 只有 4-bit 指数（范围 ±448）。Transformer Engine 用 per-tensor scaling 解决这个问题：将张量乘以一个 scaling factor，使值域映射到 FP8 的有效范围。scaling factor 的选择是 FP8 训练的关键——太大会溢出，太小会下溢。DelayedScaling 的本质是"指数移动平均"来平滑 amax 的波动。

激活重计算（Activation Checkpointing）

前向传播中保存的激活值是显存大户。激活重计算用时间换空间：不保存中间激活，反向传播时重新计算。

显存节省：
无重计算：O(L × B × S × H) 保存所有层的激活
有重计算：O(√L × B × S × H) 只保存 √L 个检查点
时间代价：约 33% 额外计算（重新前向一次） 
# PyTorch 激活重计算
from torch.utils.checkpoint import checkpoint

class TransformerLayer(nn.Module):
    def forward(self, x):
        # 使用 checkpoint 包装计算密集的部分
        attn_out = checkpoint(self.attention, x, use_reentrant=False)
        ffn_out = checkpoint(self.ffn, attn_out, use_reentrant=False)
        return ffn_out

# 或者对整个层做重计算
for layer in model.layers:
    layer = checkpoint_wrapper(layer)
💡 实用技巧：激活重计算用时间换空间：显存从 O(L) 降到 O(√L)，代价是约 33% 额外计算。PyTorch 实现用 torch.utils.checkpoint.checkpoint(fn, x, use_reentrant=False)。高级技巧：选择性检查点——只重计算 GeLU/Dropout 等轻量操作，保留 Attention softmax，可在 15-20% 额外计算下节省 ~40% 激活显存。
⚠️ 注意事项：激活重计算的常见坑：① use_reentrant=False 是 PyTorch 2.0+ 的必须设置——reentrant 模式会与 torch.compile 冲突，导致静默错误；② checkpoint 的粒度选择：太粗（整个 Transformer 层）节省显存多但重计算慢，太细（单个 Linear）节省显存少但重计算快，推荐每个 Transformer 层 2 个 checkpoint（Attention + FFN）；③ 重计算会改变 dropout 的随机行为，需要额外保存 dropout 的随机种子。
🔍 深入理解：激活重计算的理论基础来自 Chen et al. 2016 的"Sublinear Memory Cost"论文：对于 L 层网络，最优检查点策略是每隔 √L 层保存一个检查点，总显存 O(√L × B × S × H)，时间代价 ~33%。实际实现通常在每个 Transformer 层的边界做 checkpoint，等价于 √L 策略（因为层数 L 通常 32-128）。"选择性重计算"（Selective Checkpointing）是新方向——只重计算 Attention softmax（显存大但计算便宜），保留 FFN 的激活（计算贵但显存小）。

梯度累积

当显存不够放大的 batch size 时，用多个小 batch 的梯度累加来模拟。

# 梯度累积示例
accumulation_steps = 8  # 等效 batch size = micro_bs × 8
optimizer.zero_grad()

for step, batch in enumerate(dataloader):
    loss = model(**batch).loss
    loss = loss / accumulation_steps  # 归一化
    loss.backward()

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
💡 实用技巧：梯度累积的关键：loss 必须除以 accumulation_steps 做归一化，否则梯度会偏大导致训练不稳定。实现时在 (step+1) % accumulation_steps == 0 时才调用 optimizer.step()。与混合精度结合时，GradScaler 的 unscale 也要在累积完成后执行，否则梯度裁剪阈值会不准确。
⚠️ 注意事项：梯度累积的常见坑：① loss 必须除以 accumulation_steps 做归一化，否则有效学习率会放大 N 倍——这是最常见的 bug，且不容易从 loss 曲线发现；② 梯度累积改变了 DDP 的通信模式——累积期间不调用 AllReduce，只有最后一步才同步，这会增加单次通信的数据量；③ 与 GradScaler 结合时，scaler.unscale_(optimizer) 必须在所有累积步骤完成后才调用。
🔍 深入理解：梯度累积的数学等价性：累积 N 步等价于将 batch size 放大 N 倍（在梯度期望意义上），但方差会增加 √N 倍（因为每步的 mini-batch 梯度是独立采样的）。这意味着梯度累积不是真正的"大 batch 训练"——它在统计上等价于用更大方差的梯度更新。实践中这个差异通常可忽略（<1% 性能差异），但在训练稳定性敏感的场景下需要注意。

组合策略：显存计算

70B 模型在 8×H100 上训练的最优组合：

  • BF16 混合精度：参数 140GB → ZeRO-3 分片后每卡 17.5GB
  • 优化器状态 FP32：ZeRO-3 分片后每卡 ~35GB
  • 激活重计算：激活值从 ~50GB 降到 ~10GB
  • 梯度累积 4 步：micro_batch=1 等效 batch=32
  • 总计：~62.5GB/卡（在 80GB H100 内）
通俗类比：混合精度就像"用计算器做数学"——整数部分用精确的 FP32（就像用纸笔算），小数部分用近似的 FP16（就像用心算）。大部分时候近似就够了，但关键步骤（如累加梯度）必须用精确计算。Loss Scaling 就像"放大镜"——FP16 的精度太低，小梯度会被"四舍五入"为零，放大 1024 倍后就能保留这些小梯度了。
常见误区：① 认为 FP16 训练精度不够——实际上，混合精度训练（FP16 计算 + FP32 主权重）的最终模型质量与纯 FP32 训练几乎相同；② 忽略 BF16 的优势——BF16 的动态范围与 FP32 相同（8 bit 指数），不需要 Loss Scaling，训练更稳定，是目前大模型训练的首选精度。
深入探讨：显存优化的核心矛盾是"空间换时间"——Activation Checkpointing 节省显存但增加计算，Offload 节省显存但增加通信。一个有趣的前沿方向是"选择性重计算"——不是所有层都需要重计算，只重计算显存占用大但计算便宜的层（如注意力层），保留计算贵但显存占用小的层（如 FFN 层）。这种"智能调度"可以在显存和计算之间找到更好的平衡点。

混合精度训练的数学原理与工程实践

混合精度训练是降低显存占用和提升训练速度的核心技术。其核心思想是用低精度格式（FP16/BF16）存储大部分数据和计算，同时用高精度格式（FP32）维护关键的数值稳定性。

数值精度的数学分析

IEEE 754 浮点数标准定义了不同精度格式的表示范围和精度。理解这些差异对于选择合适的混合精度策略至关重要。

精度格式指数位尾数位数值范围精度(有效位)显存占用

Loss Scaling 的原理与实现

FP16 训练中最关键的问题是梯度下溢（underflow）。由于 FP16 的最小正规数为 ~6×10⁻⁸，很多小梯度值会被直接截断为零。Loss Scaling 通过放大 loss 值来放大梯度，避免下溢。反向传播完成后再缩小回来。

梯度检查点 (Activation Checkpointing) 工作原理： 无检查点 (存储所有激活值): Forward: [Act0]→[Act1]→[Act2]→[Act3]→[Act4]→Loss Backward: ←[Act4]←[Act3]←[Act2]←[Act1]←[Act0] 显存: O(L) 层激活值全存 有检查点 (只存检查点层的激活值): Forward: [Act0]→丢弃→[Act2]→丢弃→[Act4]→Loss ↓ ↓ Checkpoint Checkpoint Backward: 重新计算[Act4]→[Act3] 重新计算[Act2]→[Act1] 显存: O(√L) 只存 √L 个检查点 额外计算: ~33% (每层重算一次) 选择性检查点 (只重计算代价低的操作): 保留: Attention softmax (计算密集) 重算: GeLU, Dropout (计算轻量) 显存: 节省 ~40% 激活值 额外计算: ~15-20%
