1. 传统运营商网络优化

1.1 流量工程 (Traffic Engineering)

问题：如何在已知拓扑和流量矩阵下，优化路由使得链路利用率均衡？

背景:
  运营商网络有N个节点、E条链路, 每条链路有容量C[e]
  流量矩阵T[i][j] = 节点i到j的流量需求
  目标: 选择路由, 使得最大链路利用率最小化(避免拥塞)

OSPF/IS-IS权重优化:
  问题: 设置每条链路的OSPF权重w[e], 使得最短路路由下的
       最大链路利用率最小
  数学模型:
    min  max_e { load[e] / C[e] }
    s.t. load[e] = Σ_{(i,j)} T[i][j] × fraction[e in path(i,j)]
         path(i,j) = OSPF最短路(由权重w决定)
         w[e] ∈ {1,2,...,W_max}

  这是一个NP-hard组合优化问题!
  → 实践: 交替优化(先固定路径优化权重, 再固定权重优化路径)
  → 工具: Cisco WAN Automation Engine, Google OR-Tools

MPLS-TE (多协议标签交换流量工程):
  优势: 可以显式指定路径(不像OSPF只能走最短路)
  模型: 多商品流问题
    min  max_e { load[e] / C[e] }
    s.t. Σ_p f_p = T[i][j]  ∀(i,j)  (流量守恒)
         Σ_{p: e∈p} f_p ≤ C[e]      (容量约束)
         f_p ≥ 0                       (非负流量)
  → LP可解! 因为路径可以任意选
  → 实践: RSVP-TE信令协议建立显式路径

求解方法：多商品流LP (MPLS-TE) / 启发式+LP交替 (OSPF权重) / 模块一·LP

1.2 网络容量规划 (Capacity Planning)

问题：给定流量预测和可靠性要求，每条链路需要多大容量？

模型:
  min  Σ_e cost[e] × C[e]           (最小化总带宽成本)
  s.t. Σ_{p:e∈p} f_p ≤ C[e]        (容量约束)
       Σ_p f_p ≥ T[i][j]            (满足需求)
       C[e] ∈ {0, OC-3, OC-12, OC-48, OC-192}  (离散容量等级)

  这是固定成本网络设计问题 — NP-hard!
  → 实践: 先用LP松弛得到连续容量, 再向上取整到最近的容量等级
  → 长期规划: 考虑流量增长预测(随机规划)

求解方法：LP松弛+舍入 / Benders分解 / 模块六·随机规划

1.3 网络生存性 (Network Survivability)

问题：链路/节点故障时，如何保证业务不中断？

备份路径设计:
  主路径P₁: 正常工作时的路由
  备份路径P₂: 故障时切换到的路由
  约束: P₁和P₂不共享风险链路(SRLG约束)

  专用备份(1+1保护):
    流量同时在P₁和P₂传输, 接收端选更好的
    代价: 100%额外带宽, 但切换时间<50ms

  共享备份(1:N保护):
    多条主路径共享同一条备份路径(故障不同时发生)
    模型: min Σ_e backup[e]
          s.t. backup[e] ≥ f₂[p]  ∀p经过e, ∀主路径p的备份
               对不同主路径的备份, 共享带宽可重叠
    → 节省30-50%带宽

  无备份(恢复式):
    故障后重新计算路由 → 恢复时间长(秒级)但成本最低

生存性网络设计:
  min  Σ_e C[e]  (总带宽)
  s.t. 对每种单链路故障场景s, 存在可行路由满足所有需求
  → 需求不确定性下用鲁棒优化
  → 模块六·鲁棒优化

1.4 光网络波长分配 (WDM/RWA)

问题：光纤中每条光路(lightpath)分配哪个波长？

背景: WDM光纤可在同一根光纤上同时传输多个波长(λ₁,λ₂,...,λₙ)
  每条光路(i→j)需要分配一个波长
  约束: 同一根光纤上经过的光路不能使用相同波长(波长冲突)

  路由与波长分配(RWA):
    min  使用的波长数 (或最大波长编号)
    s.t. 每条光路分配一个波长
         同光纤上无波长冲突

  这本质是图着色问题!
  • 光路=顶点, 光纤共享=边 → 色数=最少波长数
  • 模块七·图着色

  波长连续性约束: 光路全程使用同一波长(无波长转换器)
  → 更困难! 可能需要更多波长

  实践: 先用启发式路由, 再用DSATUR着色

1.5 5G频谱分配与基站部署

问题：在哪些位置部署基站？每个基站分配哪些频段？

基站选址 (最小集合覆盖):
  候选位置集L, 用户集U, 每个位置l覆盖用户集Cover(l)
  min  |S|  (部署最少基站)
  s.t. ∪_{l∈S} Cover(l) = U  (所有用户被覆盖)
  → NP-hard, 贪心O(ln n)近似
  → 模块八·集合覆盖

频谱分配 (图着色):
  基站=顶点, 干扰关系=边
  分配频段=着色, 相邻基站不同色
  → 色数=最少频段数
  → 模块七·图着色

联合优化(选址+频谱+功率):
  min  Σ_deploy[l] + Σ_power[l]
  s.t. 覆盖约束 + 干扰约束 + 频谱约束
  → 大规模MIP, 通常用分解方法或启发式

1.6 网络功能虚拟化 (NFV) 编排

问题：将网络功能(Firewall, NAT, DPI, LB)部署到哪些服务器上？流量经过哪些功能？

服务功能链(SFC)编排:
  需求: 流量从A到B必须依次经过: 防火墙→DPI→负载均衡
  候选: 每个功能可部署在多台物理服务器上
  目标: 最小化端到端延迟, 满足资源约束

  模型: 带约束的最短路/多商品流
    min  Σ latency(e) × f(e)
    s.t. 流量从源到目的, 按顺序经过指定类型的功能节点
         每台服务器的CPU/内存不超限
         每个功能实例的吞吐量不超限

  → 整数规划(IP) / 约束规划(CP)
  → 模块一·MIP + 模块七·CP

弹性伸缩:
  当流量增加时, 自动增加功能实例
  当流量减少时, 自动减少实例节省成本
  → 在线优化 / MPC
  → 模块六·MPC

1.7 BGP对等互联与传输优化

问题：运营商之间如何选择对等互联(Peering)和传输(Transit)策略最小化成本？

背景:
  Transit(传输): 付费让大运营商帮你把流量送到全球 → $1-5/Mbps
  Peering(对等): 与另一个运营商免费互换流量 → $0 (但需要物理连接)
  IXP(互联网交换点): 多个运营商在同一点互连 → 降低peering成本

优化问题:
  min  Σ transit_cost[i] × transit_bw[i] + Σ peering_cost[j] × peering_link[j]
  s.t. 所有目的地可达(连通性约束)
       每条链路容量约束
       流量工程约束(偏好路径)

  → 多商品流 + 设施选址(选择在哪些IXP建立peering)
  → 模块一·LP

BGP路由策略优化:
  BGP是互联网的"路由协议", 但策略配置复杂:
  • Local Preference: 控制出站流量选哪条路径
  • AS-Path Prepending: 控制入站流量
  • MED (Multi-Exit Discriminator): 告诉邻居优先用哪个入口

  优化: 给定流量矩阵, 设置BGP策略使得:
    1. 总传输成本最低
    2. 链路利用率均衡
    3. 满足流量工程约束(某些流量必须走指定路径)

  → NP-hard(离散策略选择) + LP(连续流量分配)
  → 实践: 启发式 + 仿真验证

1.8 QoS保障与队列优化

问题：如何在有限带宽下保障不同业务的延迟/丢包/带宽要求？

队列调度优化:
  路由器有多个队列(语音/视频/数据/尽力而为)
  每个队列有不同的QoS要求:
    • 语音: 延迟<150ms, 丢包<1%
    • 视频: 延迟<300ms, 带宽≥2Mbps
    • 数据: 吞吐量优先
    • 尽力而为: 无保证

  加权公平队列(WFQ)优化:
    min  Σ w[i] × delay[i]
    s.t. Σ bandwidth[i] ≤ total_bandwidth
         delay[i] ≤ max_delay[i]  (延迟约束)
         loss[i] ≤ max_loss[i]    (丢包约束)

  排队论模型(M/M/1):
    平均延迟 = 1 / (μ - λ)    (μ=服务速率, λ=到达速率)
    当λ→μ时, 延迟→∞!  (不能让利用率接近100%)

  → 凸优化(延迟函数是凸的)
  → 模块二·凸优化

DiffServ (区分服务):
  将流量分为有限几个类别(通常3-4个)
  每个类别分配不同的队列和调度权重
  优化: 分类规则 + 权重分配
  → LP(连续) / IP(离散分类)

1.9 5G网络切片 (Network Slicing)

问题：如何在共享物理网络上为不同租户/业务创建隔离的虚拟网络？

背景: 5G需要在同一物理基础设施上支持:
  • eMBB (增强移动宽带): 高带宽, 视频流
  • URLLC (超可靠低延迟): 工厂自动化, 远程手术
  • mMTC (大规模物联网): 传感器, 海量设备

  每种业务需要不同的"切片"(虚拟网络)

切片映射优化:
  min  资源使用量(或最大化切片数)
  s.t. 每个切片的SLA约束(带宽/延迟/可靠性)
       物理资源约束(频谱/计算/存储)
       切片间隔离(一个切片的拥塞不影响其他)

  → 虚拟网络嵌入(VNE)问题 — NP-hard
  → 模块七

动态切片资源调整:
  流量随时间变化 → 切片资源需要动态调整
  → 在线优化 / MPC
  → 模块六·MPC

1.10 拥塞控制优化

问题：如何调整发送速率使得网络利用率最大且不丢包？

TCP拥塞控制:
  经典算法: 慢启动 → 拥塞避免 → 快重传 → 快恢复
  优化目标: 最大化吞吐量, 最小化丢包和延迟

  BBR (Google, 2016):
    建模: 吞吐量 = BDP / RTT (带宽×延迟积)
    实时估计: 带宽 × RTT_min
    发送速率 = 带宽估计 × 增益系数
    → 比传统Cubic在高延迟链路上快2-25%

  数据中心拥塞控制 (DCQCN/HPCC):
    RDMA网络需要极低延迟(<10μs)
    用ECN(显式拥塞通知)精确控制发送速率
    → 反馈控制 + 凸优化

  多路径拥塞控制 (MPTCP):
    同时使用多条路径(如WiFi+4G)
    优化: 流量在路径间的分配使得总吞吐量最大
    max  Σ rate[p]
    s.t. Σ rate[p] ≤ bottleneck[p]  (每条路径的瓶颈)
         总速率 ≤ 应用需求
    → 凸优化(速率分配是凹函数)
    → 模块二·凸优化

1.11 网络遥测与流量矩阵估计

问题：如何从有限的测量数据推断全网流量矩阵？

流量矩阵(Traffic Matrix): 每对(src,dst)之间的流量大小
  直接测量: 在每条链路上部署探针 → 成本太高!
  间接推断: 从链路负载和路由信息反推流量矩阵

  链路负载 = 路由矩阵 × 流量矩阵
  l = A × tm  (已知l和A, 求tm)
  → 欠定方程(方程数<未知数), 无穷多解

  优化: 加正则化
    min  ||A·tm - l||² + λ·regularizer(tm)
    s.t. tm ≥ 0

  正则化选择:
  • L2范数: 偏好均匀分布 → 过于平滑
  • L1范数: 偏好稀疏解 → 适合大象流检测
  • Poisson模型: 利用流量的统计特性
  → 模块二·凸优化

网络遥测(Network Telemetry):
  • INT(In-band Network Telemetry): 数据包携带路径信息
  • Streaming Telemetry: 实时推流网络状态(替代SNMP轮询)
  • AI/ML异常检测: 用优化方法检测网络异常

2.1 子网规划与IP地址分配

问题：如何划分子网使得地址浪费最少、路由表最简洁？

子网划分:
  给定N个子网, 各有n[i]台主机
  从IP地址池中分配CIDR块, 要求:
    • 每个子网地址数 ≥ n[i] (向上取2的幂)
    • 子网不重叠
    • 路由聚合后条目数最少

  这是装箱问题的变体!
  • 物品=子网需求, 箱子=地址空间
  • 但物品大小必须是2的幂(对齐约束)

2.2 跨区域流量优化

问题：多AZ/多Region部署下, 如何最小化跨区域数据传输成本？

背景: 云厂商对跨AZ/跨Region流量收费
  同AZ流量: 免费或极低
  跨AZ流量: $0.01-0.02/GB
  跨Region流量: $0.02-0.09/GB

优化目标:
  min  Σ cross_az_cost + Σ cross_region_cost
  s.t. 每个服务的延迟要求
       每个AZ的容量约束
       数据一致性要求(某些数据必须在同Region)

方法:
  • 服务亲和性: 将频繁通信的服务部署在同一AZ
  • 数据本地化: 将数据副本放在访问最频繁的AZ
  • 流量引导: 将用户请求路由到最近的AZ

  → 线性规划(连续流量分配)
  → 整数规划(离散部署决策)
  → 模块一

2.3 负载均衡优化

问题：如何将请求分配到多个后端实例，使得响应时间最短？

静态负载均衡 (加权轮询):
  给每个后端实例分配权重w[i], 按权重比例分配请求
  优化: min max_i { load[i] / capacity[i] }
  → 当所有实例相同时, 均匀分配最优
  → 实例不同时, 按容量比例分配

动态负载均衡 (最短队列/最短预期延迟):
  将请求发给当前队列最短的实例
  或发给预期响应时间最短的实例:
    E[response_time(i)] = queue_length[i] / service_rate[i] + network_latency[i]

  → 在线优化问题
  → 模块六·在线算法

全局负载均衡 (GSLB):
  DNS层面将用户引导到最近/最空闲的Region
  考虑: 延迟、健康状态、容量、成本
  → 多目标优化
  → 模块六·多目标

2.4 安全组与防火墙规则优化

问题：如何最小化安全组规则数量，同时满足所有访问需求？

规则合并:
  输入: N条访问需求 (源IP, 目的IP, 端口, 协议)
  输出: 最少的等价安全组规则

  技巧:
  • IP聚合: 10.0.1.0/24 + 10.0.2.0/24 → 10.0.0.0/22 (如果连续)

  • 规则冲突检测: 确保无冗余/矛盾规则

  → 集合覆盖问题: 用最少的规则覆盖所有需求
  → 模块八·集合覆盖

零信任网络优化:
  最小权限原则: 每个服务只允许访问它需要的资源
  → 最小割问题: 找到最小的权限集使得服务正常工作
  → 模块四·最小割

2.6 自动伸缩 (Auto-Scaling) 优化

问题：何时增加/减少实例？增减多少？

阈值策略(简单):
  CPU > 80% → 增加1个实例
  CPU < 20% → 减少1个实例
  → 简单但反应滞后, 可能震荡

预测式伸缩(进阶):
  用历史数据预测未来负载 → 提前伸缩
  模型: 队列论 M/M/c

3.1 CDN节点部署与内容分发

问题：在哪些位置部署CDN节点？每个节点缓存哪些内容？

节点选址 (k-Center问题):
  候选位置集L, 用户集U, 每对(l,u)有延迟d(l,u)
  选择k个位置部署节点, 最小化用户到最近节点的最大延迟:
    min  max_u { min_{l∈S} d(l,u) }
    s.t. |S| = k

  这是k-Center问题 — NP-hard!
  贪心2-近似: 每次选能覆盖最多未覆盖用户的点
  → 模块八·k-center

内容缓存策略:
  每个节点有有限缓存空间, 每个内容有大小和访问频率
  目标: 最大化缓存命中率(或最小化回源流量)

  • LRU/LFU: 经典启发式, 实践效果好
  • 最优化: 0/1背包(每个节点独立)
    max Σ hit_rate[c] × x[c]
    s.t. Σ size[c] × x[c] ≤ cache_size
    → 模块三·背包

  • 协同缓存: 多个节点联合决定谁缓存什么
    → 集合覆盖 / 设施选址
    → 模块八

3.2 P2P超节点拓扑设计

问题：如何选择超节点、设计超节点间的拓扑，使得查询效率最高？

超节点选择:
  从N个普通节点中选k个作为超节点
  选择标准:
    • 带宽大(能承载更多中继流量)
    • 在线时间长(稳定性)
    • 地理位置分散(覆盖广)

  → k-Center / k-Median问题
  → 模块八

超节点拓扑设计:
  超节点间需要形成连通图, 目标:
    • 查询跳数少(直径小)
    • 负载均衡(度数均匀)
    • 容错(连通度高)

  DHT (分布式哈希表)拓扑:
    • Chord: 环形拓扑, O(log n)跳
    • Kademlia: XOR度量, O(log n)跳, 非对称路由
    • 优化: 减少跨区域跳数(地理感知DHT)

  → 模块四·图论

3.3 边缘计算任务卸载

问题：用户的计算任务应该在本地、边缘节点还是云端执行？

任务卸载决策:
  每个任务i有:
    • 本地计算延迟 t_local[i]
    • 边缘计算延迟 t_edge[i] + 传输延迟 t_trans[i]
    • 云端计算延迟 t_cloud[i] + 传输延迟 t_trans_cloud[i]

  目标: min Σ latency[i]  (总延迟最小)
  约束: 边缘节点计算资源有限, 云端带宽有限

  → 整数规划: x[i] ∈ {local, edge, cloud}
  → 当只有edge/cloud两个选项时, 可贪心(优先选延迟低的)
  → 资源受限时, 是背包/调度问题

  联合优化(卸载+资源分配):
    min  Σ delay[i]
    s.t. Σ cpu[i] × x[i][edge] ≤ CPU_edge  (边缘算力)
         Σ bw[i] × x[i][cloud] ≤ BW_cloud  (云端带宽)
    → MIP, 规模大时用启发式

  → 模块一·MIP + 模块七·调度

3.4 分布式数据放置与复制

问题：数据副本放在哪些节点上？如何保证一致性？

数据放置优化:
  N个数据对象, M个存储节点, 每个节点有容量限制
  每个数据对象有访问模式(哪些区域访问最多)
  目标: 最小化平均访问延迟

  min  Σ access[i][r] × latency(r, replica_location[i])
  s.t. Σ size[i] × x[i][m] ≤ capacity[m]  (容量约束)
       Σ_m x[i][m] ≥ replication_factor[i]  (副本数约束)

  → 设施选址问题的变体
  → 模块七·选址

一致性vs可用性(CAP):
  强一致性: 所有副本同步写入 → 延迟高
  最终一致性: 异步复制 → 延迟低但可能读到旧数据

  Quorum协议: 读W_r + 写W_w > N (多数派)
  优化: 选择W_r, W_w, N使得延迟最低且满足一致性要求
  → 在线优化 / 随机优化

3.5 P2P网络中的激励机制

问题：如何激励节点贡献资源(带宽、存储)？

博弈论建模:
  每个节点选择"贡献"或"搭便车"(free-ride)
  贡献者付出成本但获得服务
  搭便车者不付出但享受服务

  纳什均衡: 没有节点能通过单方面改变策略获益
  → 可能所有人都搭便车(囚徒困境)

  激励机制设计:
  • Tit-for-Tat: 你给我多少, 我就给你多少
  • 信用系统: 贡献资源获得信用, 消耗资源花费信用
  • 声誉系统: 根据历史贡献分配优先级

  → 博弈论 / Minimax
  → 模块二·Minimax

3.6 一致性哈希与负载均衡

问题：如何在分布式系统中分配数据/请求，使得节点增减时迁移量最小？

传统哈希: hash(key) % N → 节点增减时几乎所有key需要重映射!

一致性哈希: 将节点和key映射到同一个环上
  key归属于顺时针方向最近的节点
  节点增减时只影响相邻节点的key → 迁移量 = O(K/N)

虚拟节点: 每个物理节点映射环上多个虚拟点
  → 更均匀的负载分布
  → 当节点容量不同时, 容量大的节点分配更多虚拟节点

优化问题:
  min  max_i { load[i] }         (负载均衡)
  s.t. 节点增减时的迁移量最小
       容量约束(load[i] ≤ capacity[i])

  → 在线优化(节点动态加入/离开)
  → 贪心: 每次将过载节点的key迁移到最空闲节点

进阶: 有界负载一致性哈希
  保证: load[i] ≤ (1+ε) × average_load
  → 当ε=0.25时, 每个节点负载不超过平均的125%

3.7 Gossip协议与信息传播优化

问题：如何在P2P/分布式系统中快速可靠地传播信息？

Gossip协议(流行病传播模型):
  每个节点周期性地随机选择k个邻居, 发送自己知道的新信息
  → 信息像病毒一样传播

  分析(流行病模型):
    已知节点数 = n(t), 总节点数N
    n(t+1) = n(t) + n(t) × (N-n(t))/N × k
    → S曲线: 慢→快→慢

  优化:
    • fanout(每次发送的邻居数): 太小→传播慢, 太大→带宽浪费
      最优fanout ≈ O(log N) (以高概率保证所有节点收到)
    • Push vs Pull: Push(主动发送)适合新信息, Pull(请求获取)适合同步
    • Anti-entropy: 周期性完整同步, 纠正遗漏

  应用: Cassandra/Redis Cluster的成员管理、故障检测

3.8 多播/广播优化

问题：如何将数据从一个源发送到多个目的地，使得总传输成本最小？

最小Steiner树:
  给定源节点s和目标节点集合T
  找总边权最小的树, 连通s和T中所有节点(可以引入中间节点)
  → NP-hard
  → 模块七·Steiner树
  → 近似比: 2-近似(基于MST) / 1.39-近似(Byrka et al.)

应用层多播(ALM):
  不用网络层多播(IP multicast), 而在应用层构建覆盖树
  优化: 构建延迟最小的分发树
  → 最小延迟生成树(Steiner树的变体)
  → 模块四·MST

视频会议/直播中的选择性转发(SFU):
  SFU选择性地将视频流转发给参与者
  优化: 选择哪些SFU节点, 如何路由, 使得延迟最小
  → 设施选址 + 路由优化
  → 模块七·选址

4. 数据中心网络 (DCN) 优化

通俗类比：数据中心网络像一座超级仓库的内部物流系统——货物（数据）需要从货架（服务器）快速送达出口（用户），关键是货架布局和传送带设计。
常见误区：常见错误是采用传统三层架构导致东西向流量瓶颈。现代DCN应采用Spine-Leaf架构，任何两个服务器间跳数不超过3跳（ToR→Spine→ToR）。
深入探讨：DCN核心指标：收敛时间<1秒、丢包率<10⁻⁶、尾延迟P99<1ms。Spine-Leaf架构：Leaf交换机下联48端口10G、上联6端口100G到Spine。ECMP负载均衡粒度：基于5元组哈希。RDMA网络PFC流控是关键。

4.1 Fat-tree/Clos拓扑设计

问题：如何用最便宜的交换机搭建无收敛比的大规模网络？

传统树形: 根交换机是瓶颈, 收敛比 1:N
Fat-tree: 每层全带宽, 无收敛

k-ary Fat-tree:
  k个Pod, 每Pod k台交换机(下层k/2 + 上层k/2)
  每台交换机 k 个端口: 下半连服务器, 上半连核心
  服务器总数: k³/4
  交换机总数: 5k²/4
  链路总数: k³

  例: k=48 → 27,648台服务器, 2,880台交换机

优化: 选择k使得 总端口≥服务器数, 总成本最低
  min  交换机数 × 单价 + 链路数 × 光模块价
  s.t. 端口数 ≥ 服务器数
       交换机端口数 ∈ 可用型号

变体:
  • Jellyfish(随机拓扑): 随机连接交换机, 比Fat-tree多25%端口利用率
  • 递归Fat-tree: 适合超大规模(>10万台)
  • VL2(Microsoft): 用Valiant负载均衡

4.2 ECMP等价多路径与流量调度

问题：服务器间有多条等价路径，如何均匀分配流量？

ECMP(Equal-Cost Multi-Path):
  同一TCP流走同一条路(避免乱序)
  不同流用哈希分配到不同路径
  问题: 哈希不均匀 → 某些链路过载

  优化: min max_link { load[link] / capacity[link] }
  → 多商品流LP, 但ECMP是哈希近似(非最优)

Conga/Protocols:
  数据中心内实时拥塞反馈
  每个交换机维护每条路径的拥塞度
  流量自动偏向拥塞度低的路径

大象流vs老鼠流:
  99%流量是老鼠流(<100KB) → 延迟敏感, 用ECMP
  1%流量是大象流(>100MB) → 吞吐敏感, 用显式路径(TE)
  Hedera: 检测大象流 → 全局调度
  → 模块一·多商品流LP

4.3 RDMA网络与无损传输

问题：如何在数据中心实现<2μs延迟的远程内存访问？

RDMA(Remote Direct Memory Access):
  绕过CPU, 直接读写远程内存, 延迟<2μs
  但要求网络"无损"(不能丢包, 否则重传代价极大)

无损保障机制:
  PFC(Priority Flow Control): 逐跳反压, 队列满时通知上游暂停
  DCQCN: 拥塞通知 + 速率调整
  问题: PFC风暴(级联反压) → 死锁

  优化: 在"无损"约束下最大化吞吐量
  → 凸优化(速率分配是凹函数)
  → 模块二·凸优化

AI训练网络优化:
  大模型训练需要All-Reduce通信(梯度同步)
  优化: 选择最优的All-Reduce算法(Ring/Tree/Hierarchical)
  → Ring All-Reduce: 2(N-1)/N × 数据量 (接近最优)
  → 拓扑感知: 将通信密集的GPU放在同一交换机下

4.4 数据中心网络虚拟化

问题：如何在物理网络上创建隔离的虚拟网络(VXLAN/NVGRE)？

VXLAN封装:
  原始帧 → 封装UDP/IP头 → 在物理网络传输
  问题: 封装开销(50字节/帧), MTU问题

优化:
  • VNI(VXLAN Network Identifier)分配: 最大化复用
  • BUM流量优化(广播/未知单播/多播): 减少泛洪
  → 用ARP代理/多播树替代泛洪
  → 模块四·多播树

4.5 光电路交换 (OCS) 与混合交换

问题：数据中心内如何用光交换减少延迟和功耗？

光电路交换(OCS):
  传统: 电交换(灵活但功耗高、延迟大)
  光交换: 直接用光信号交换, 延迟<100ns, 功耗极低
  但: 光交换的重配置时间长(μs~ms), 不能逐包交换

混合交换架构:
  大象流 → 光交换(长连接, 高吞吐)
  老鼠流 → 电交换(短连接, 低延迟)

  优化: 流量调度使得光交换利用率最高
  min  电交换负载(或功耗)
  s.t. 大象流分配到光路径
       光路径不冲突(时分复用)
       光交换重配置时间约束

  → 调度问题 + 多商品流
  → 模块一·MIP

4.6 可编程交换机与网内计算

问题：如何利用P4可编程交换机在数据平面执行计算？

P4可编程交换机:
  传统交换机: 固定逻辑(查路由表→转发)
  P4交换机: 可编程数据平面(自定义包处理逻辑)

网内计算(In-Network Computing):
  在交换机上执行聚合操作(sum/average/max)
  → 减少数据传输量(不需要把所有数据发到服务器)

  例: 分布式梯度下降
    传统: 所有worker → 参数服务器(All-Reduce)
    网内: 交换机直接做梯度聚合(NetReduce)
    → 减少50%以上通信量

  优化: 在哪里放置网内计算节点
  → 设施选址 + 流量优化
  → 模块七·选址

5.3 PON光纤接入规划

问题：如何规划无源光网络(PON)的分光器和光纤？

  分光比: 1:32(短距) 或 1:64(长距)

优化:
  • 选址: 分光器放在哪里(靠近用户 vs 靠近局端)
  • 分光比: 每个分光器分多少路
  • DBA(动态带宽分配): 在ONU之间分配上行带宽

  s.t. 每个ONU被覆盖

5.4 干扰管理与功率控制

问题：相邻小区之间如何协调干扰？

ICIC (Inter-Cell Interference Coordination):
  大小区在某些时隙不发数据(ABS), 让小基站用
  优化: ABS比例使得系统吞吐量最大
  → 凸优化(速率是干扰功率的凹函数)

功率控制:
  每个基站调整发射功率: 功率大→覆盖广但干扰大
  max  Σ log(rate[i])    (比例公平)
  s.t. rate[i] = f(power, interference)
       power[i] ≤ max_power
  → 凸优化(对数函数是凹的)
  → 模块二·凸优化

频谱共享(CBRS):
  三层频谱接入: 授权层 > 优先接入层 > 一般授权层
  优化: 动态频谱接入, 最大化总利用率
  → 在线决策 / 部分可观测MDP
  → 模块六·MDP

5.5 Massive MIMO波束赋形

  s.t. 发射功率 ≤ P_max

  → 凸优化(SDP松弛) / 迭代算法(WMMSE)

6.2 能耗均衡路由

问题：如何路由数据使得网络中最后一个节点死亡的时间最晚？

传统最短路: 近sink节点被频繁使用 → 早早耗尽电池

最大寿命路由:
  max  min_i { energy[i] / consumed[i] }   (最大最小公平)
  s.t. 所有数据被路由到sink
       流量守恒

  → 最大最小公平 = lexicographic optimization
  → 先最大化最弱节点的寿命, 再最大化次弱节点...
  → 模块六·多目标

多路径路由: 将流量分散到多条路径, 避免单条路径过载
  → 多商品流LP
  → 模块一·LP

6.3 数据聚合与休眠调度

问题：如何减少传输量并延长网络寿命？

数据聚合:
  相邻传感器数据相似 → 在中间节点聚合(求平均/最大值)后再传输
  优化: 选择聚合点使得总传输量最小
  → Steiner树(聚合点=Steiner点)
  → 模块八·Steiner树

休眠调度:
  轮流休眠: 每时刻只激活最少的传感器保持覆盖
  min  活跃传感器数(每时刻)
  s.t. 每时刻区域仍被覆盖
       每个传感器的总休眠时间尽量公平

  → 集合覆盖(每时刻选最少的活跃节点)
  → 公平性: 最大最小公平(所有节点寿命尽量一致)

6.4 能量收集传感网络

能量收集(Energy Harvesting):
  传感器从环境收集能量(太阳能、振动能、射频)
  能量到达是随机的(太阳能有昼夜/天气变化)

优化: 在能量约束下最大化数据收集量
  max  Σ data_rate[i]
  s.t. 能量消耗 ≤ 收集能量(在线约束)
       数据队列不溢出

  → 在线优化(不知道未来能量到达)
  → 模块六·在线算法

Lyapunov优化:
  将长期约束转化为逐时隙的在线决策
  → 不需要知道未来信息, 保证可行性
  → 理论性能接近离线最优

6.5 联邦学习中的通信优化

问题：如何在设备上训练模型，只传输梯度更新？

联邦学习(Federated Learning):

  问题: 大量设备同时上传梯度 → 上行带宽瓶颈

优化:
  • 设备选择: 选择哪些设备参与本轮训练
    max  模型质量(数据量 × 数据相关性)
    s.t. 选中设备数 ≤ k
         每设备通信资源 ≤ capacity
    → 子模优化(Submodular Optimization)

  • 梯度压缩: 只上传最重要的梯度分量
    → Top-K稀疏化(只传最大的K个梯度)
    → 量化(用更少的比特表示梯度)

  • 通信轮次优化: 减少全局通信频率
    → 增加本地训练步数(FedAvg)
    → 通信-计算权衡

7.1 拓扑控制与功率优化

问题：如何调整发射功率使得网络连通且总能耗最小？

目标: 最小化总功率, 同时保持连通

  min  Σ power[i]
  s.t. 图保持连通
       power[i] ≥ f(distance[i,j])  (覆盖最远邻居)

  功率 ∝ 距离^α, α≥2 (路径损耗模型)
  → 最小生成树(MST)就是最优拓扑!
  → 模块四·MST

  进阶: k-连通(容错) → k-MST / k-边连通生成子图
  → 模块七·NP-hard

7.2 按需路由与频谱感知

问题：移动节点如何高效发现路由？如何利用空闲频段？

按需路由(AODV/DSR):
  不维护全局路由表, 需要时再路由发现
  优化: 路由发现开销 vs 路由维护开销
  → 在线决策(什么时候主动发现 vs 被动等待)

频谱感知(Cognitive Radio):
  次级用户使用主用户未用的频段
  优化: 检测空闲频段 + 分配频段(干扰约束)
  → 图着色(频段=颜色, 干扰=边)
  → 模块七·图着色

网状网络容量(Gupta-Kumar定理):
  n个节点的无线网络, 每节点吞吐量 = Θ(W/√(n·log n))
  → 节点越多, 每节点带宽越少!
  突破: MIMO / 定向天线 / 中继

7.4 延迟容忍网络 (DTN)

问题：网络间歇连通，如何可靠传输数据？

  特征: 没有端到端路径, 节点移动带来"存储-携带-转发"机会
  应用: 星际通信、偏远地区、灾难救援

路由优化:
  洪泛: 复制给所有邻居 → 可靠但带宽浪费
  单副本: 只传给最有可能到达目的地的节点 → 节省但可能丢失

  优化: 选择最优的转发节点
    max  P(消息到达目的地)
    s.t. 副本数 ≤ k
         每节点存储 ≤ capacity

  → 部分可观测MDP(不知道节点未来移动)
  → 模块六·MDP

  → 洪泛的DTN版本
  → 本模块·Gossip

8.3 虚拟网络映射 (VNE)

  • 链路需求(带宽/延迟)

10.4 Layer 2扩容方案优化

问题：如何提升区块链的交易吞吐量(TPS)？

  • 分片: 水平扩展, 每个分片独立处理交易

Layer 2(链下)优化:
  • 状态通道: 双方在链下多次交易, 最终结算上链

    → 在线决策

  • Rollup: 将多笔交易打包后提交到主链
    优化: 批量大小(延迟 vs 成本)
    → 批量优化 / 背包
    → 模块三·背包

  • Plasma: 侧链定期向主链提交状态根
    优化: 挑战期长度(安全性 vs 延迟)
    → 鲁棒优化

11. 跨网络类型的优化模式

常见误区：误区是各网段独立优化。端到端延迟由最慢环节决定（木桶效应），瓶颈可能在网间互联点而非单一网络内部。需要全局视角分析。

以下优化模式在多种网络类型中反复出现，掌握一个就能触类旁通。

11.1 多商品流 (Multi-Commodity Flow)

出现场景：流量工程、数据中心调度、CDN分发、VPC跨AZ优化

形式: 多个(src,dst)对共享同一网络, 每对有流量需求
  min  Σ cost[e] × load[e]  (或 max min 利用率)
  s.t. Σ_{p:e∈p} f_p ≤ C[e]       (容量约束)
       Σ_p f_p = T[i][j]  ∀(i,j)   (需求满足)
       f_p ≥ 0

  特例:
  • 单商品流 = 最大流
  • 单源单汇 = 最短路
  • 所有需求=1 = 指派问题

  求解: LP(连续) / MIP(离散路径选择)
  → 模块一·LP

  实际工具: Gurobi/CPLEX(精确) / Google OR-Tools(启发式)

11.2 图着色变体 (Graph Coloring Variants)

出现场景：频率分配、WiFi信道、波长分配、寄存器分配、考试排期

统一形式: 给顶点分配"颜色", 相邻顶点不同色, 最小化颜色数

  不同网络中的"颜色":
  • 无线网络 → 频率/信道
  • 光网络 → 波长(λ)
  • 5G → 资源块(RB)
  • 调度 → 时间槽
  • 编译器 → 寄存器

  变体:
  • 列表着色: 每个顶点有可选颜色子集(频谱限制)
  • T-着色: 相邻顶点颜色差不在集合T中(干扰模型)
  • 在线着色: 顶点逐个到达, 即时分配颜色

  → 模块七·图着色
  → 模块八·着色近似

11.3 集合覆盖变体 (Set Cover Variants)

出现场景：基站选址、传感器覆盖、CDN节点、消防站、监控部署

统一形式: 选择最少的"设施"覆盖所有"需求点"

  不同网络中的"设施"和"需求":
  • 无线网络 → 基站 和 用户
  • IoT → 传感器 和 监测区域
  • CDN → 缓存节点 和 内容
  • 安防 → 摄像头 和 路段
  • 卫星 → 卫星 和 地面区域

  变体:
  • 带权集合覆盖: 每个设施有不同成本
  • k-覆盖: 每个需求点被至少k个设施覆盖
  • 部分覆盖: 允许覆盖p%的需求(鲁棒版本)

  → 模块八·集合覆盖

11.4 设施选址变体 (Facility Location Variants)

出现场景：数据中心选址、CDN部署、VPN Hub、仓库、基站

统一形式: 选择设施位置, 分配客户到设施, 最小化(建设+服务)总成本

  min  Σ fixed[j] × y[j] + Σ trans[i][j] × x[i][j]
  s.t. Σ x[i][j] = 1  (每个客户被服务)
       x[i][j] ≤ y[j]  (只有建设了才能服务)
       y[j] ∈ {0,1}

  不同网络中的应用:
  • 云 → 数据中心选址(AWS/Azure选Region)
  • CDN → 边缘节点选址
  • 电信 → 机房/POP点选址
  • 区块链 → 节点存储选址

  → 模块七·NP-hard
  → 模块一·MIP

11.5 最大最小公平 (Max-Min Fairness)

出现场景：带宽分配、功率控制、负载均衡、能耗均衡

核心思想: 先保障最差用户的资源, 再依次提升

  算法: 水填充(Water-Filling)
    1. 将资源平均分配给所有用户
    2. 达到最小需求的用户"退出"
    3. 剩余资源重新平均分配
    4. 重复直到资源耗尽

  数学形式: lexicographic maximization of sorted rates
    先max r_{(1)}(最小), 再max r_{(2)}(次小), ...

  应用:
  • 无线网络: 功率控制使得所有用户速率公平
  • 数据中心: 带宽分配使得所有流得到公平份额
  • IoT: 能耗均衡使得所有节点寿命一致

  → 模块六·多目标

11.6 在线/动态优化模式

出现场景：自动伸缩、路由、缓存、调度、一致性哈希

核心挑战: 决策时不知道未来输入

  常见模式:
  • 滑动窗口: 用最近W个时间步的数据做决策
  • 阈值策略: 当指标超过阈值时触发动作(如CPU>80%→扩容)
  • 预测+优化: 先用ML预测未来需求, 再用优化求解
  • MPC: 每步重新优化未来N步, 只执行第1步

  竞争比分析:
  • 贪心: 竞争比可能很差
  • 随机化: 通常能改善竞争比
  • 学习增强: 用ML预测改善在线决策

  → 模块六·在线算法
  → 模块五·竞争比

  1. 核心原理: 经典优化方法在网络领域的核心是将网络问题(路由、容量、调度)建模为数学优化问题并求解。网络优化的独特之处在于问题天然具有图结构——节点是设备/用户，链路是连接/关系，流量是商品。这使得图论算法(最短路、最大流、匹配)成为基础工具，而多商品流LP是流量工程的核心模型。网络优化还需处理实时性(在线算法)、不确定性(随机规划)和大规模(分解方法)等工程挑战。
    关键要点: 1. 网络问题的图结构是选择方法的关键；2. 多商品流LP是流量工程的通用建模框架；3. NP-hard网络问题(TSP/VRP/图着色)需要启发式；4. 实时网络决策需要在线算法；5. 网络生存性需鲁棒优化。
    实际案例: 运营商骨干网流量工程：100个节点、500条链路的网络，给定流量矩阵T[i][j]。用MPLS-TE建模为多商品流LP，Gurobi在秒级内求得使最大链路利用率最小化的路由方案。
    关联知识: 流量工程→多商品流LP(模块一)；路由优化→最短路/最大流(模块四)；不确定性→鲁棒优化(模块六)。
  2. 核心原理: 传统运营商网络优化覆盖从接入网到骨干网的完整链路。核心优化问题包括：(1)流量工程TE——设置OSPF权重或MPLS标签使链路利用率均衡(NP-hard+LP交替)；(2)容量规划——每条链路需要多大容量(固定成本网络设计，NP-hard)；(3)网络生存性——故障时保证业务不中断(备份路径设计，鲁棒优化)；(4)光网络波长分配RWA——本质是图着色问题(NP-hard)；(5)5G基站选址——最小集合覆盖(NP-hard，贪心O(ln n)近似)。每个问题都对应特定的理论模块。
    关键要点: 1. OSPF权重优化是NP-hard组合优化，实践中用启发式+LP交替；2. MPLS-TE的多商品流模型是LP可解的(路径可任意选)；3. 网络生存性的共享备份(1:N)可节省30-50%带宽；4. 波长分配RWA=图着色，色数=最少波长数；5. 5G基站选址+频谱分配是联合优化问题(大规模MIP)。
    实际案例: 某省运营商骨干网TE优化：100节点、300条链路。用MPLS-TE多商品流LP求解，最大链路利用率从85%降到62%，无需扩容即满足未来2年流量增长。
    关联知识: 流量工程→LP(模块一)；波长分配→图着色(模块七)；生存性→鲁棒优化(模块六)；5G选址→集合覆盖(模块八)。
  3. 核心原理: VPC网络优化是在云环境中合理规划虚拟网络以满足安全、性能和成本需求。核心优化问题：(1)子网规划——按AZ+业务分离划分子网，本质是带对齐约束的装箱问题(子网大小必须是2的幂)；(2)跨区域流量优化——最小化跨AZ/跨Region数据传输费用(LP/MIP)；(3)负载均衡——将请求分配到后端实例使响应时间最短(在线优化)；(4)安全组规则优化——最小化规则数量同时满足访问需求(集合覆盖)；(5)自动伸缩——根据负载动态调整资源(在线优化/MPC)。
    关键要点: 1. 子网CIDR规划应从/16起预留扩展空间；2. 跨AZ流量费用$0.01-0.02/GB，服务亲和性可显著降低成本；3. 动态负载均衡(最短队列/最短预期延迟)优于静态加权轮询；4. 安全组规则合并是集合覆盖问题；5. 自动伸缩需要结合预测(ML)+优化(MPC)。
    实际案例: 某互联网公司VPC优化：200个微服务分布在3个AZ。通过服务亲和性优化(将频繁通信的服务放在同一AZ)减少跨AZ流量40%，每月节省$5000+数据传输费。
    关联知识: 子网规划→装箱(模块一)；跨AZ优化→LP(模块一)；负载均衡→在线算法(模块六)；自动伸缩→MPC(模块六)。
  4. 核心原理: 超节点网络优化(如CDN、P2P、分布式存储)的核心是在大规模分布式系统中优化内容分发和资源分配。CDN优化：节点部署用k-Center问题建模(贪心2-近似，模块八)；缓存策略用背包问题(DP/贪心，模块三)；请求路由用最近节点分配(在线算法)。P2P网络：超节点选择用k-Median问题(近似算法)；一致性哈希实现负载均衡(在线优化)。分布式存储：数据放置用设施选址(MIP)；分片优化用图分割(最小割)。
    关键要点: 1. CDN节点部署=k-Center问题(最小化最大用户到节点距离)；2. 缓存命中率优化可用背包模型(容量=缓存大小，价值=访问频率)；3. 一致性哈希通过虚拟节点实现近似均匀负载分配；4. 超节点选择需同时考虑带宽、存储和计算能力(多目标)；5. 区块链分片=图分割(最小跨分片交易)。
    实际案例: 某视频CDN优化：100个候选节点位置，覆盖1000万用户。用k-Center贪心算法选出30个节点，最大用户到节点延迟从50ms降到18ms。缓存策略用背包模型，命中率从65%提升到88%。
    关联知识: k-Center→近似算法(模块八)；缓存策略→背包/DP(模块三)；一致性哈希→在线算法(模块六)；图分割→最小割(模块四)。
  5. 核心原理: 数据中心网络(DCN)优化覆盖从物理拓扑到虚拟化调度的完整栈。物理层：Fat-tree/Leaf-spine拓扑设计(组合设计)、光互连优化(图着色)。网络层：ECMP流量调度(多商品流LP)、RDMA拥塞控制(凸优化)、网络虚拟化VNE(子图同构，NP-hard)。应用层：K8s调度(多维背包+指派)、微服务部署(设施选址MIP)、服务链编排SFC(带约束最短路)。DCN优化的独特挑战是"极低延迟+极高带宽+动态负载"。
    关键要点: 1. Fat-tree拓扑提供无阻塞带宽但成本高(O(N²)交换机)；2. ECMP等价多路径是数据中心最常用的负载均衡；3. RDMA需要μs级拥塞控制(传统TCP不够)；4. K8s调度是多维背包(CPU/内存/GPU)问题；5. 网络功能虚拟化NFV将防火墙/NAT等部署为微服务链。
