背景与动机：为什么要拆开GPU看？

看一颗现代AI GPU的内部到底长什么样。

为什么要了解GPU内部架构？因为在集合通信中，数据的流动路径是：

数据流动路径： HBM（显存） ↔ L2 Cache ↔ L1 Cache ↔ SM（计算单元）↔ NVLink/PCIe接口 ↔ 外部

每一步都有带宽限制和延迟。如果你不知道这些组件在哪里、怎么连接的，就无法理解通信瓶颈出在哪里。

2.2 名字由来：SM、GPC、TPC这些缩写

名字由来：SM、GPC、TPC这些缩写

缩写全称名字由来
SMStreaming Multiprocessor"流式多处理器"——能同时处理大量"流式"（独立、可并行）的计算任务。"Streaming"来自GPU的图形渲染传统——像素着色就是一种流式处理。
GPCGraphics Processing Cluster"图形处理集群"——GPU内部最大的组织单元。名字保留了"Graphics"的历史痕迹，但现在主要用于计算。
TPCTexture Processing Cluster"纹理处理集群"——最初负责纹理采样和过滤。现在仍保留纹理单元，但对AI计算来说意义不大。
CUDA Core——CUDA是"Compute Unified Device Architecture"的缩写。CUDA Core就是能执行浮点/整数运算的基本计算单元，相当于CPU里的ALU（算术逻辑单元）。
Tensor Core——"张量核心"——专门为张量（多维数组）运算设计的硬件单元。核心操作是矩阵乘法（D = A × B + C），一个Tensor Core可以在一个时钟周期内完成一个4×4矩阵乘法。
HBMHigh Bandwidth Memory"高带宽内存"——通过3D堆叠实现超高带宽的内存技术。"High Bandwidth"是相对于传统GDDR/DDR内存而言的。

2.3 核心原理：GPU的层级结构

NVIDIA GPU内部有严格的层级组织，从大到小：

GPU层级结构（从大到小）： GPU（整颗芯片） │ ├── GPC 0（Graphics Processing Cluster） │ ├── TPC 0（Texture Processing Cluster） │ │ ├── SM 0（Streaming Multiprocessor） │ │ │ ├── 128个 FP32 CUDA Core │ │ │ ├── 128个 INT32 CUDA Core │ │ │ ├── 4个 Tensor Core（第四代） │ │ │ ├── 32个 LD/ST单元（加载/存储） │ │ │ ├── 4个 Special Function单元（sin/cos等） │ │ │ ├── 128KB 寄存器文件 │ │ │ └── 128KB L1/共享内存（可配置比例） │ │ ├── SM 1 │ │ └── ... │ ├── TPC 1 │ └── ... │ ├── GPC 1 │ └── ... │ ├── ... │ ├── L2 Cache（全局共享，50MB on H100） │ ├── 内存控制器（连接HBM） │ ├── NVLink接口（连接其他GPU） │ └── PCIe接口（连接CPU）

SM：GPU的基本计算单元

SM（Streaming Multiprocessor）是GPU的"工人"。每个SM能同时执行大量线程（H100的每个SM可以管理2048个活跃线程）。SM内部的关键组件：

组件H100每SM数量作用
FP32 CUDA Core128执行单精度浮点运算
INT32 CUDA Core128执行整数运算
Tensor Core（第四代）4执行矩阵乘法（FP16/BF16/FP8）
LD/ST单元32执行内存加载/存储操作
SFU4执行特殊函数（sin, cos, exp等）
寄存器文件256 KB最快的存储，每个线程私有
L1/共享内存256 KBSM内共享，可配置比例
💡 为什么Tensor Core只有4个，而CUDA Core有128个？

因为Tensor Core效率极高。一个第四代Tensor Core在一个时钟周期内可以完成一个4×4矩阵乘法（64次乘加运算）。而一个CUDA Core一个周期只能做1次乘加。所以4个Tensor Core = 4×64 = 256次乘加/周期，已经超过了128个CUDA Core的128次乘加/周期。

这就是为什么AI训练主要靠Tensor Core——不是靠堆CUDA Core的数量。

GPC和TPC：组织层级

GPC是GPU内部最大的组织单元。每个GPC包含多个TPC，每个TPC包含2个SM。GPC的作用是：

  • 资源隔离：每个GPC有自己的L1 Cache、指令缓存等
  • 并行调度：不同的GPC可以独立处理不同的任务
  • 光栅化：在图形渲染中，每个GPC有自己的光栅化引擎

2.4 技术细节：Compute Die vs IO Die

为什么要把IO功能拆出来？

在第一章讲过，现代芯片趋向于Chiplet架构。对于GPU来说，IO功能（PCIe接口、NVLink接口、内存控制器等）有两个特点：

  1. 不需要先进制程：IO电路用6nm/7nm就够了，不需要4nm/3nm
  2. 需要大量模拟电路：高速SerDes（串行/解串器）、PHY（物理层接口）是模拟电路，和数字计算电路的制造工艺不同

把IO功能拆到独立的IO Die上，可以：

  • Compute Die用最先进制程（4nm/3nm），最大化晶体管密度和计算性能
  • IO Die用成熟制程（6nm/7nm），成本更低，良率更高
  • 两种Die可以独立迭代——升级Compute Die不需要重新设计IO Die

2.7 关联技术：GPU内部架构如何影响通信

7 关联技术：GPU内部架构如何影响通信

  1. NVLink带宽为什么是900 GB/s？ → 因为H100的NVLink接口有18条链路，每条50 GB/s。18×50=900。这些链路的物理线路通过封装实现。
  2. 为什么HBM带宽比NVLink高？ → HBM通过TSV（3D封装）连接，几千根线；NVLink通过PCB/线缆连接，物理上只能放18条链路。
  3. 为什么GPUDirect RDMA能加速通信？ → 因为它让数据从HBM直接通过NVLink接口发出，不经过CPU内存——省了一次HBM→CPU内存的拷贝。
  4. 为什么B200要用双Die？ → 单Die的面积已经接近制造极限（~800mm²），要塞进更多晶体管只能拆成两个Die。

💡 为什么了解GPU内部架构对通信优化很重要？

知道GPU内部的分层架构和连接方式直接影响你的通信优化策略：

  • Blackwell的双Die设计：两个Compute Die之间通过NVLink-C2C连接（900GB/s），而GPU之间通过NVLink 5.0（1.8TB/s）。这意味着Die间通信（片内）实际上比GPU间通信（片间）更慢！优化时需要考虑数据放在哪个Die上。
  • HBM带宽 vs NVLink带宽：HBM带宽通常是NVLink带宽的4-5倍。如果你的AllReduce操作需要频繁读写HBM，HBM是瓶颈；如果数据量小但需要跨GPU传输，NVLink才是瓶颈。
  • IO Die的角色：所有NVLink/PCIe通信都经过IO Die。IO Die的调度策略影响通信延迟——这就是为什么NCCL的拓扑检测很重要，它会根据物理连接选择最优通信路径。

一句话：不理解GPU内部架构，就无法理解NCCL为什么这样调度通信。

2.9.2 内部工作机制：SM如何执行计算

前面我们介绍了SM的组件清单，但这些组件是如何协同工作来执行计算的？理解SM的内部执行模型，是理解GPU为什么能同时处理数千线程的关键。

SM内部的硬件组成

以H100为例，每个SM内部包含以下核心计算资源：

资源数量说明
FP32 CUDA Core128个单精度浮点运算单元，每个时钟周期执行1次FP32乘加
Tensor Core4个矩阵乘法加速单元，每个时钟周期完成1个4×4矩阵乘法（64次乘加）
Warp Scheduler4个调度器，每个周期从就绪队列中选择一个warp并发射指令
寄存器文件256 KB每个线程最多分配255个32位寄存器
共享内存 / L1256 KBSM内所有线程共享，可配置比例

Warp与SIMT执行模型

GPU执行计算的基本单位不是单个线程，而是Warp——一组32个线程。这32个线程被捆绑在一起，每时钟周期执行相同的指令。这就是SIMT（Single Instruction Multiple Thread）模型：

4个Warp Scheduler意味着SM可以同时管理4个Warp的执行。当一个Warp因为等待内存数据而暂停时（称为内存延迟隐藏），调度器立刻切换到另一个就绪的Warp继续执行。这就是GPU能隐藏长达数百时钟周期的HBM延迟的秘诀——SM始终保持忙碌。

💡 为什么GPU需要这么多线程？

CPU的设计哲学是"尽量让每个核心快"——用乱序执行、分支预测等复杂逻辑来提升单线程性能。GPU的设计哲学是"尽量让所有单元忙"——用大量线程来隐藏延迟。H100的一个SM可以管理2048个活跃线程（64个Warp × 32线程/Warp），这么多线程足以保证即使部分Warp在等内存，总有Warp可以执行计算。

🔬 AllReduce中的数据路径：

在执行AllReduce时，GPU需要从HBM中读取梯度数据，通过NVLink发送到其他GPU。这个过程的关键瓶颈是HBM读取带宽。以H100为例：

  • HBM带宽：3.35 TB/s —— 决定数据能多快从显存"搬出来"
  • NVLink带宽：900 GB/s —— 决定数据能多快传到其他GPU

如果HBM带宽不够，NVLink就会"饿着"——数据来不及从HBM搬到NVLink接口。这就是为什么HBM带宽是集合通信的基础瓶颈。

2.9.3 向前串联：GPU架构如何决定通信需求

前面几节我们了解了GPU的内部结构。现在让我们把这些知识串联起来，看看GPU架构如何从根本上决定了集合通信的行为模式。

Compute Die vs IO Die：两条独立的物理通路

这是理解GPU通信最关键的架构洞察：

💡 这就是"计算-通信重叠"的硬件基础！

因为计算路径（HBM→SM）和通信路径（HBM→IO Die→NVLink）使用不同的物理硬件，GPU可以同时进行计算和通信——SM在做矩阵乘法的同时，IO Die在通过NVLink发送数据。这就是计算-通信重叠（Compute-Communication Overlap）的物理基础，也是第10章要详细讲解的核心优化技术。

L2 Cache：通信的中转站

L2 Cache在集合通信中扮演着一个容易被忽视但至关重要的角色——数据中转缓冲：

NCCL发送数据的内部流程： 步骤1: NCCL在SM上运行一个kernel → SM从HBM读取梯度数据到寄存器 → SM对数据做必要的预处理（如Ring AllReduce中的reduce-scatter） 步骤2: SM将处理后的数据写回L2 Cache （不是写回HBM！L2比HBM快3-4倍） 步骤3: NVLink控制器从L2 Cache读取数据 → 通过NVLink链路发送到目标GPU 步骤4: 目标GPU收到数据后，NCCL的SM kernel从L2读取 → 写入目标GPU的HBM L2 Cache作为中转缓冲的优势： - L2读写带宽 ~12 TB/s，远高于HBM的3.35 TB/s - 避免了"写HBM再读HBM"的双重延迟 - 减少了HBM的读写争用（计算和通信共享HBM带宽）
⚠️ HBM带宽才是瓶颈，不是L2：

L2 Cache带宽高达~12 TB/s，而HBM带宽仅3.35 TB/s。在AllReduce中，瓶颈几乎总是HBM——数据必须从HBM"搬出来"才能被NVLink发送。L2 Cache的超高带宽确保了它不会成为额外的瓶颈。这也是为什么HBM带宽的每一代提升对集合通信都至关重要。

算力与带宽的精妙平衡

最后，让我们用一个具体数字来说明GPU架构的设计哲学——计算能力和通信能力的平衡：

🔬 这不是巧合，而是精心设计：

NVIDIA在设计GPU架构时，有意让计算吞吐量和通信吞吐量处于同一数量级。如果计算远快于通信，GPU就不得不空等数据传输（通信瓶颈）；如果通信远快于计算，NVLink的带宽就被浪费了（过度投资）。

这种平衡意味着：

  • 计算-通信重叠是真正有效的——两者耗时相近，重叠后总时间接近于较长的那个，而不是两者之和
  • 随架构代际演进，平衡被持续维护：B200的NVLink带宽翻倍到1.8TB/s，同时Tensor Core算力也翻倍到~4000 TFLOPS。Rubin更是同步提升到3.6TB/s和~8000 TFLOPS
  • 理解这个平衡，你就能预判优化的天花板：如果AllReduce已经和GEMM完美重叠，再优化通信也不会让整体更快——瓶颈已经转移到计算侧了

import time
import torch

def profile_inference(model, input_data, warmup=10, iterations=100):
    """推理性能分析：测量延迟、吞吐和显存"""
    # Warmup
    for _ in range(warmup):
        with torch.no_grad():
            model(input_data)
    torch.cuda.synchronize()
    
    # Measure latency
    latencies = []
    for _ in range(iterations):
        start = time.perf_counter()
        with torch.no_grad():
            output = model(input_data)
        torch.cuda.synchronize()
        latencies.append(time.perf_counter() - start)
    
    # Statistics
    avg_ms = sum(latencies) / len(latencies) * 1000
    p50 = sorted(latencies)[len(latencies)//2] * 1000
    p99 = sorted(latencies)[int(len(latencies)*0.99)] * 1000
    mem_mb = torch.cuda.max_memory_allocated() / 1024 / 1024
    
    print(f"Latency: avg={avg_ms:.2f}ms, p50={p50:.2f}ms, p99={p99:.2f}ms")
    print(f"Peak Memory: {mem_mb:.0f}MB")
    print(f"Throughput: {1000/avg_ms:.0f} req/s")
    return {"avg_ms": avg_ms, "p50": p50, "p99": p99, "mem_mb": mem_mb}

✅ 参考答案与详解
  1. 核心原理: H100理论FP16 TFLOPS=132 SM × 4 TC/SM × 256 FMA/TC(每TC每周期) × 2(乘和累加) × 1.83 GHz。注意：Hopper第4代TC每个周期完成256 FP16 FMA(不是64)，因为其MMA指令(m16n8k16)涵盖更大的计算量：每个Warp完成16×8×16×2=4096 FMA/指令，每个TC每周期处理256 FMA。计算：132×4×256×2×1.83G=132×4×256×2×1.83×10⁹=554×10¹²≈989 TFLOPS。其中2是因为一次乘法和一次加法各算一次FMA。与第3题类似但强调GPU内部结构理解
    关键要点: ①132 SM × 4 TC/SM ②每TC每周期256 FMA ③频率1.83 GHz ④总计989 TFLOPS
    实际案例: nsight compute实测H100 GEMM kernel约850 TFLOPS(86%利用率)
    关联知识: 与SM架构、TC内部结构和GPU算力规格直接相关
  2. 核心原理: AMD MI300X使用8个Compute Die(Chiplet)而非单Die的原因是：①制造良率——8个小Die(每个~150mm²)的良率远高于1个大Die(~600mm²)。缺陷概率随面积指数增长，大Die每个缺陷就报废整颗芯片。②成本——Chiplet方案使用成熟制程(6nm IO Die)降低制造成本，GPU核心用5nm聚焦计算性能。③模块化灵活性——同一IO Die设计可搭配不同数量Compute Die，灵活覆盖不同市场(HPC/AI/消费级)。④制造极限——单Die面积受限于光刻机reticle limit(~800mm²)，8小Die可集成更多晶体管(通过2.5D互连)。代价：Die间通信延迟增加、Infinity Fabric互连开销
    关键要点: ①Chiplet良率高 ②成本优化 ③模块化灵活 ④突破面积极限
    实际案例: MI300X: 8个Compute Die(5nm)+4个IO Die(6nm)，总192GB HBM3
    关联知识: 与2.5D封装、Infinity Fabric互连和Chiplet生态相关
  3. 核心原理: Ring AllReduce分两步：①Reduction阶段(N-1步)：每步从HBM读取数据→NVLink发送到下一GPU→归约；②Broadcast阶段(N-1步)：归约结果→NVLink发送→HBM写入。HBM带宽影响的是"从HBM读取/写入数据"环节，NVLink影响的是"跨GPU传输"环节。H100参数：HBM=3350GB/s，NVLink=900GB/s，HBM带宽是NVLink的3.7倍——说明当前HBM不是AllReduce瓶颈。如果HBM翻倍(6700GB/s)：Reduction步中HBM读取时间不变(因为NVLink仍是900GB/s，数据仍需以NVLink速率传输)，理论上AllReduce速度几乎不变。关键洞察：AllReduce性能由最慢环节(瓶颈)决定——当前是NVLink
    关键要点: ①AllReduce=HBM读取+NVLink传输 ②HBM已是NVLink的3.7倍 ③瓶颈在NVLink ④HBM翻倍不提升AllReduce
    实际案例: 实测HBM从2TB/s(A100)提升到3.35TB/s(H100)，AllReduce速度提升很小
    关联知识: 与Ring AllReduce算法、通信瓶颈分析和NCCL调优相关
