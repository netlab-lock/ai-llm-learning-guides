📡 08 集合通信原语

8.1 背景与动机：为什么需要标准化的通信操作？

前面七章讲的都是硬件和网络——数据在物理上怎么流动。从本章开始，进入算法层——数据在逻辑上怎么组织和传输。

在分布式训练中，GPU之间需要频繁交换数据。如果每个程序员都自己实现通信逻辑，会非常混乱。所以需要一组标准化的通信操作——就像数学里的"加减乘除"，所有复杂的通信模式都可以用这些基本操作组合出来。

🔬 历史背景： 集合通信的概念来自MPI（Message Passing Interface）——1994年发布的HPC通信标准。MPI定义了Broadcast、Reduce、AllReduce等基本操作，至今仍在HPC领域广泛使用。NCCL（2016年）把这些操作搬到了GPU上，并针对GPU的硬件特性（NVLink、GPUDirect等）做了深度优化。NCCL的API设计刻意模仿了MPI，让HPC程序员可以无缝迁移。

8.2 名字由来

集合通信（Collective Communication）：Collective = "集体的"。与Point-to-Point（点对点，只有两个参与者）不同，集合通信是所有参与者共同完成的操作——通信域内的每个GPU都必须参与。

每个原语的名字都很直观：

原语名字拆解含义
BroadcastBroad + cast"广泛传播"——一个节点广播给所有人
ReduceRe + duce（拉丁语ducere：引导）"归约"——把多个值"引导"合并成一个
AllReduceAll + Reduce"全归约"——所有人做Reduce，所有人拿结果
AllGatherAll + Gather"全收集"——所有人收集所有人的数据
ReduceScatterReduce + Scatter"归约分散"——先Reduce再把结果分散给各人
All-to-All——"全交换"——每个人给每个人发不同的数据

8.3 核心原理：六种原语详解

1. Broadcast（广播）：1→N

Broadcast：GPU0的数据发给所有人 初始：GPU0:[A,B,C,D] GPU1:[?,?,?,?] GPU2:[?,?,?,?] GPU3:[?,?,?,?] 结果：GPU0:[A,B,C,D] GPU1:[A,B,C,D] GPU2:[A,B,C,D] GPU3:[A,B,C,D] 通信量分析： GPU0发送：D × (N-1) = 3D 其他GPU接收：D 总网络流量：D × (N-1) = 3D

使用场景：训练开始时，rank 0加载模型参数，然后Broadcast到所有GPU。优化器状态同步。

2. Reduce（归约）：N→1

Reduce（求和）：所有GPU的数据汇总到GPU0 初始：GPU0:[1,2] GPU1:[3,4] GPU2:[5,6] GPU3:[7,8] 结果：GPU0:[16,20] 其他GPU不变 计算：[1+3+5+7, 2+4+6+8] = [16,20] 通信量分析： 其他GPU发送：D（每个GPU发送一份） GPU0接收：D × (N-1) = 3D 支持的操作：求和(SUM)、求最大(MAX)、求最小(MIN)、乘积(PROD)等

使用场景：损失函数汇总、梯度汇总（但通常用AllReduce代替，因为所有GPU都需要结果）。

3. AllReduce（全归约）：N→N ★最重要

AllReduce（求和）：所有人拿到相同的汇总结果 初始：GPU0:[1,2] GPU1:[3,4] GPU2:[5,6] GPU3:[7,8] 结果：GPU0:[16,20] GPU1:[16,20] GPU2:[16,20] GPU3:[16,20] 通信量分析（Ring实现）： 每个GPU发送：D × (N-1)/N 每个GPU接收：D × (N-1)/N 总网络流量：2 × D × (N-1)/N
💡 核心等式： AllReduce = ReduceScatter + AllGather 这个等式是理解AllReduce算法优化的关键。NCCL实际就是分两步实现AllReduce的——先ReduceScatter（每个GPU拿到部分汇总结果），再AllGather（收集所有部分结果）。

使用场景：分布式数据并行训练中的梯度同步——这是最频繁、最关键的集合通信操作。每一步训练都要做一次AllReduce。

4. AllGather（全收集）：N→N

AllGather：每个GPU收集所有GPU的数据片段 初始：GPU0:[A0] GPU1:[A1] GPU2:[A2] GPU3:[A3] 结果：GPU0:[A0,A1,A2,A3] GPU1:[A0,A1,A2,A3] GPU2:[A0,A1,A2,A3] GPU3:[A0,A1,A2,A3] 通信量分析（Ring实现）： 每个GPU发送：D × (N-1)/N 每个GPU接收：D × (N-1)/N 结果大小：每个GPU的数据从D/N变成D（扩大N倍）

使用场景：AllReduce的第二步；张量并行中收集各GPU的部分结果（比如每个GPU算了一部分注意力头，AllGather后所有GPU都有完整的注意力输出）。

5. ReduceScatter（归约分散）：N→N

ReduceScatter（求和 + 分散）： 初始（每个GPU有4个元素）： GPU0:[a0,a1,a2,a3] GPU1:[b0,b1,b2,b3] GPU2:[c0,c1,c2,c3] GPU3:[d0,d1,d2,d3] 结果（每个GPU只拿到一段的求和）： GPU0:[a0+b0+c0+d0] ← 只有第0段的求和 GPU1:[a1+b1+c1+d1] ← 只有第1段的求和 GPU2:[a2+b2+c2+d2] ← 只有第2段的求和 GPU3:[a3+b3+c3+d3] ← 只有第3段的求和 通信量分析（Ring实现）： 每个GPU发送：D × (N-1)/N 每个GPU接收：D × (N-1)/N 结果大小：每个GPU的数据从D变成D/N（缩小N倍）

使用场景：AllReduce的第一步；梯度累积后分发给各GPU更新不同层的参数。

6. All-to-All（全交换）：N→N

All-to-All：每个GPU给每个GPU发不同的数据 初始（每个GPU有N份数据）： GPU0:[d00,d01,d02,d03] ← d_ij 表示GPU0给GPUj的数据 GPU1:[d10,d11,d12,d13] GPU2:[d20,d21,d22,d23] GPU3:[d30,d31,d32,d33] 结果： GPU0:[d00,d10,d20,d30] ← 收集了所有GPU给自己的数据 GPU1:[d01,d11,d21,d31] GPU2:[d02,d12,d22,d32] GPU3:[d03,d13,d23,d33] 通信量分析： 每个GPU发送：D × (N-1)/N 每个GPU接收：D × (N-1)/N

使用场景：专家混合模型（MoE）中不同专家之间的数据交换——每个GPU持有一部分专家，需要把token路由到对应的专家。

8.4 技术细节

通信量公式汇总

原语每GPU发送量每GPU接收量结果大小变化
BroadcastD（root发送）D（非root接收）不变
ReduceD（非root发送）D（root接收）不变
AllReduceD(N-1)/ND(N-1)/N不变
AllGatherD(N-1)/ND(N-1)/ND → D×N
ReduceScatterD(N-1)/ND(N-1)/ND → D/N
All-to-AllD(N-1)/ND(N-1)/N不变

D = 每个GPU上的数据量，N = GPU数量。以上数据基于Ring实现。

通信域（Communicator）

所有集合通信操作都在一个通信域内进行。通信域定义了"哪些GPU参与通信"以及它们的编号（rank）。

# PyTorch示例
import torch.distributed as dist

# 初始化通信域（所有GPU，默认NCCL后端）
dist.init_process_group(backend='nccl')

# 获取当前GPU的rank和总GPU数
rank = dist.get_rank()        # 当前GPU编号：0, 1, 2, ...
world_size = dist.get_world_size()  # 总GPU数：8

# 在通信域内做AllReduce
tensor = torch.randn(1000).cuda()
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
# 所有GPU的tensor现在持有相同的求和结果

阻塞 vs 非阻塞

集合通信操作可以是阻塞的（同步）或非阻塞的（异步）：

类型特点PyTorch API
阻塞函数返回时操作已完成dist.all_reduce(tensor)
非阻塞函数立即返回一个Work对象，稍后等待完成work = dist.all_reduce(tensor, async_op=True)

非阻塞允许通信-计算重叠——在等待通信完成的同时执行其他计算。

8.5 使用场景：并行策略与原语的对应关系

并行策略主要使用的原语具体场景
数据并行（DDP）AllReduce每步训练后同步梯度
张量并行（TP）AllReduce + AllGather注意力头/MLP列切分后的结果汇聚
流水线并行（PP）点对点 Send/Recv层间传递激活值（不是集合通信）
序列并行（SP）AllGather + ReduceScatterLayerNorm/Dropout的序列维度分割
专家并行（EP）All-to-AllMoE中token路由到对应专家
上下文并行（CP）AllGather + ReduceScatter长序列的KV分块处理
🔬 为什么数据并行用AllReduce而不是Reduce？

如果只用Reduce，只有root GPU拿到汇总结果，其他GPU的梯度还是旧的。下一轮训练时，各GPU的模型参数会不一致 → 训练发散。AllReduce保证所有GPU拿到相同的梯度，用相同的梯度更新模型 → 参数保持一致。

8.8 实际效果

以70B模型、8 GPU为例，每步训练的通信量：

70B参数 × 2字节/参数(FP16) = 140GB 梯度 AllReduce通信量（Ring实现）： = 2 × 140GB × (8-1)/8 = 245GB 各互连的AllReduce时间： NVLink 4.0 (900 GB/s): 245GB / 900 ≈ 0.27秒 NVLink 5.0 (1800 GB/s): 245GB / 1800 ≈ 0.14秒 PCIe 5.0 (126 GB/s): 245GB / 126 ≈ 1.94秒 ← 不可接受！

这就是为什么集合通信的效率直接决定了大模型训练是否可行。

🆕 补充原语：Scatter与Gather

除了前面介绍的六种核心原语，还有两种在分布式系统中常用的点对点原语，它们经常被组合使用来构建集合操作：

原语方向说明通信量典型用途
Scatter1→N一个GPU把数据的不同部分发给不同GPUD（总量不变）数据分发、输入数据分配
GatherN→1多个GPU把数据发给一个GPUD结果收集、推理汇总
Scatter（1→N 分发）： Gather（N→1 收集）： GPU0: [A|B|C|D] GPU0: [_|_|_|_] ↓ ↓ ↓ ↓ ↑ ↑ ↑ ↑ GPU0 GPU1 GPU2 GPU3 GPU0 GPU1 GPU2 GPU3 [A] [B] [C] [D] [A] [B] [C] [D] 每个GPU只接收自己的部分 → GPU0: [A|B|C|D] 通信量 = D（不是D×N！） 通信量 = D

🆕 原语组合模式：实际训练中的通信模式

在实际的分布式训练框架（PyTorch DDP, DeepSpeed, Megatron-LM）中，集合通信原语被组合成更复杂的通信模式：

通信模式使用的原语所在框架目的

FSDP权重收集AllGatherPyTorch FSDP前向时收集完整权重
FSDP梯度分发ReduceScatterPyTorch FSDP反向后各GPU只保留自己的梯度部分
TP列切分AllReduceMegatron-LM张量并行的列切分层，输出需要AllReduce
TP行切分AllGather + ReduceScatterMegatron-LM张量并行的行切分层
PP通信P2P Send/RecvMegatron-LM流水线并行的层间激活传递
EP路由All-to-AllMegatron-CoreMoE的token分发到专家

💡 "通信量≠数据量"的直觉：

很多初学者会问："AllReduce不是要把所有GPU的数据汇总到每个GPU吗？那通信量不应该是D×N吗？"答案是否定的。Ring AllReduce的通信量是2D(N-1)/N ≈ 2D，不随N增长。这是因为Ring算法巧妙地把数据分成N份，每份只经过N-1次传递，而不是每次都传完整数据。理解这一点是理解所有集合通信算法的基础。

8.8.2 内部工作机制：NCCL如何执行一次AllReduce

当我们调用 dist.all_reduce(tensor) 时，背后到底发生了什么？以下是NCCL执行一次AllReduce的完整流程：

💡 关键细节：
  • 拓扑探测：NCCL在 init_process_group 时就探测了GPU间的连接拓扑（NVLink带宽、IB路径等），通信时直接查表，无需重新探测。
  • 多Channel并行：NCCL默认使用多个channel（通常2-32个），每个channel独立执行Ring/Tree的一个流水线，提高带宽利用率。
  • Kernel而非Memcpy：NCCL使用GPU kernel（而非cudaMemcpy）来做通信，因为kernel可以和计算kernel在同一个SM上调度，实现真正的通信-计算重叠。
  • Chunk大小：默认512KB-2MB，太小则kernel启动开销大，太大则延迟高、无法和其他kernel重叠。

8.8.3 向前串联：原语如何被上层框架使用

集合通信原语是底层"积木"，上层的分布式训练框架通过组合这些原语实现复杂的并行策略。以下是每种原语被哪些框架、在什么场景下使用：

原语使用者（框架）具体场景为什么用这个原语
AllReduce PyTorch DDP 梯度同步：每步反向传播后，所有GPU的梯度求平均 所有GPU需要相同的梯度来更新参数
DeepSpeed ZeRO-0/1 梯度平均：未分片模式下的梯度同步 与DDP类似，但配合其他ZeRO优化
Megatron-LM TP 列并行（Column-Linear）输出汇聚：每个GPU算了一部分列，需要AllReduce合并 列切分后，输出需要还原为完整张量
AllGather PyTorch FSDP 权重收集：前向/反向时，AllGather各GPU上的参数分片，组装成完整权重 参数分片存储，计算时需要完整权重
Megatron-LM TP 行并行（Row-Linear）输入收集：每个GPU只存部分列，前向时AllGather输入 行切分时，输入需要在所有GPU上完整
ReduceScatter PyTorch FSDP 梯度分片：反向传播后，ReduceScatter把完整梯度归约并分散，每个GPU只保留自己负责的参数分片的梯度 配合AllGather，实现全分片数据并行
梯度分区 DeepSpeed ZeRO-2/3：梯度分片存储，ReduceScatter后各GPU只更新自己的参数 减少显存：每个GPU只存1/N的梯度
All-to-All DeepSpeed-MoE MoE专家路由：token需要从源GPU发到目标专家所在的GPU 每个token的目标专家不同，通信是"个性化"的
Megatron-Core EP 专家并行（Expert Parallelism）：All-to-All分发token到专家，再All-to-All收回结果 无法用AllReduce替代——每个GPU发/收的数据不同
P2P Send/Recv Megatron-LM PP 流水线并行激活传递：前向时上一层GPU → Send → 下一层GPU Recv；反向时反向传递梯度 只有相邻层的两个GPU通信，不需要集体操作
DeepSpeed PP 流水线调度（1F1B等）：micro-batch在pipeline stage间传递 P2P更高效——避免不必要的同步
Barrier 训练框架 Epoch同步：确保所有GPU完成当前epoch后再开始下一个 防止快的GPU提前进入下一轮导致数据不一致
Checkpoint保存 保存检查点时：先Barrier确保所有GPU的梯度都已更新，再由rank 0保存 避免保存不一致的模型状态

✅ 参考答案与详解
  1. 核心原理: 手动执行4GPU AllReduce(求和)：初始GPU0:[1,2],GPU1:[3,4],GPU2:[5,6],GPU3:[7,8]。Ring AllReduce分2阶段：①ReduceScatter(3步)：Step1: GPU0发[1,2]给GPU1, GPU1发[3,4]给GPU2, GPU2发[5,6]给GPU3, GPU3发[7,8]给GPU0 → GPU0:[1+7,2+8]=[8,10], GPU1:[3+1,4+2]=[4,6], GPU2:[5+3,6+4]=[8,10], GPU3:[7+5,8+6]=[12,14]。Step2-3继续传递累直到每个GPU持有某一块的完整归约值。②AllGather(3步)：每个GPU将自己的归约块广播给所有GPU。最终所有GPU都得到[16,20]。验证RS+AG=AllReduce：RS后GPU0持有[16,0],GPU1:[0,20](举例)，AG后都得到[16,20]，结果一致
    关键要点: ①Ring RS+AG两阶段 ②RS每步归约并发 ③AG每步广播 ④结果数学等价
    实际案例: NCCL AllReduce默认使用Ring算法实现
    关联知识: 与Ring AllReduce算法和通信量分析相关
  2. 核心原理: 数据并行用AllReduce而不用Reduce的原因：①Reduce只将梯度汇总到rank 0——其他GPU得不到全局梯度，无法更新本地模型副本；②rank 0需做一次Broadcast将梯度发回所有GPU——这等价于AllReduce，但由软件手动实现效率低；③Reduce+Broadcast实现AllReduce的问题：a) 两次独立的通信操作，无法融合优化；b) rank 0可能成为瓶颈(需接收所有梯度再广播)；c) NCCL等库无法对分离的Reduce+Broadcast做ring/tree算法优化。AllReduce是原子操作——NCCL可以选择最优算法(tree/ring/DSML)实现单次操作，通信量=2(N-1)D/N(ring) vs Reduce+Broadcast=2(N-1)D(明显更多)
    关键要点: ①Reduce只汇总到1个GPU ②需额外Broadcast ③NCCL优化AllReduce更高效 ④通信量减半
    实际案例: PyTorch DDP自动使用NCCL AllReduce同步梯度
    关联知识: 与数据并行训练、DDP和梯度同步策略相关
  3. 核心原理: DeepSpeed ZeRO用ReduceScatter代替AllReduce的原理：ZeRO将模型参数(Optimizer States+Gradient+Parameter)分片到N个GPU，每个GPU只存1/N。传统AllReduce要求每个GPU持有完整梯度副本(显存=全量)。ReduceScatter将梯度归约并分散——每个GPU只持有1/N的归约后梯度。具体：N个GPU每个有完整梯度D → AllReduce后每GPU仍持有D(总显存N×D) → ReduceScatter后每GPU只持有D/N(总显存=D)。节省的显存=(N-1)/N。ZeRO Stage 1仅分区Optimizer States(节省3/4显存=OS占75%)，Stage 2分区OS+Gradient(节省更多)，Stage 3分区OS+G+P(最大化节省)。代价：ReduceScatter后需要AllGather恢复完整参数(但在forward/backward过程中按需执行)
    关键要点: ①参数分片存储 ②RS替代AR减少显存 ③节省=(N-1)/N ④按需AllGather恢复
    实际案例: ZeRO-3使200B模型在8×A100上可训练(否则需要64卡)
    关联知识: 与ZeRO优化器、显存管理和超大模型训练相关
  4. 核心原理: MoE(Mixture of Experts)需要All-to-All的原因：MoE中每个token根据门控网络路由到不同的Expert(FFN层)。不同GPU上的Expert容量不同，导致：某些GPU的Expert过载(需处理更多token)，某些GPU的Expert空闲。All-to-All实现全局token交换——每个GPU将属于远程Expert的token发给对应GPU，同时接收属于本地Expert的远程token。AllReduce不适用：AllReduce是"所有GPU拿相同结果"，而MoE是"每个GPU给不同GPU发不同数据"(个性化通信)。双向量：GPU0→Expert1的tokens, GPU0→Expert2的tokens...每个目标GPU收到的数据完全不同。这正是All-to-All的定义——每个参与者给每个其他参与者发不同的数据
    关键要点: ①MoE路由不均匀 ②token需跨GPU交换 ③每对GPU发不同数据(个性化) ④All-to-All=全局个性化交换
    实际案例: Mixtral 8x7B使用MoE，All-to-All占总通信量~40%
    关联知识: 与MoE模型架构、token路由和分布式训练通信模式相关

🎓 面试深度题

Q1: AllReduce和ReduceScatter+AllGather有什么关系？为什么说它们是等价的？

答案：AllReduce = ReduceScatter + AllGather。数学上证明通信量等价：

  • AllReduce（Ring实现）：每个GPU发送 2D(N-1)/N 的数据
  • ReduceScatter：每个GPU发送 D(N-1)/N 的数据（分N-1步，每步发D/N）
  • AllGather：每个GPU发送 D(N-1)/N 的数据（分N-1步，每步收D/N）
  • ReduceScatter + AllGather = D(N-1)/N + D(N-1)/N = 2D(N-1)/N

结果完全一致！实际执行效果也相同：每个GPU最终都持有完整的归约结果。

为什么要知道这个？因为很多框架（FSDP、DeepSpeed ZeRO）会拆开使用ReduceScatter和AllGather，而不是直接用AllReduce。理解等价性有助于看懂这些框架的设计。

Q2: 为什么PyTorch DDP用AllReduce，而FSDP用ReduceScatter+AllGather？

答案：根本区别在于模型参数的存储方式不同：

特性DDP（数据并行）FSDP（全分片数据并行）
每个GPU存储完整的模型参数模型参数的 1/N
梯度同步方式AllReduceReduceScatter + AllGather
内存占用高（完整模型）低（分片存储）

DDP：每个GPU都有完整模型，反向传播后每个GPU都有完整梯度，用AllReduce求平均后直接更新本地参数。

FSDP：每个GPU只存1/N的参数。反向传播时，需要先用AllGather收集完整参数做前向，反向时用ReduceScatter把梯度归约并分散，每个GPU只保留自己负责的那部分梯度，更新对应的参数分片。

关键洞察：FSDP用ReduceScatter+AllGather的原因是它需要在通信过程中同时完成参数的收集和梯度的分发，而AllReduce无法实现这种"分片"语义。

Q3: All-to-All通信量是多少？什么场景下会成为瓶颈？

答案：

通信量分析：假设每个GPU有D大小的数据，N个GPU。All-to-All要求每个GPU给其他N-1个GPU各发一份数据。

每个GPU发送量：D × (N-1)/N ≈ D（数据量接近D，因为每个GPU发D/N给其他N-1个GPU） 每个GPU接收量：D × (N-1)/N ≈ D 总网络流量：D × N × (N-1)/N = D(N-1)

瓶颈场景：

  1. MoE（Mixture of Experts）：最典型场景。每个token需要路由到不同专家，不同专家在不同GPU上。All-to-All完成token分发和结果收集。当专家数N很大时，通信量随N增长，容易成为瓶颈。
  2. 序列并行（CP）：超长序列切分后，KV块需要在GPU间传递。
  3. 大Embedding表：Embedding按行切分时，lookup操作需要All-to-All。

面试加分点：All-to-All的通信模式是"个性化"的（每个GPU发给不同GPU的数据不同），无法像AllReduce那样用Ring算法优化。所以All-to-All的实际带宽利用率通常低于AllReduce。

Q4: 如何判断集合通信是计算瓶颈还是通信瓶颈？

答案：核心方法是profiling + 分析计算/通信重叠率。

步骤2：分析trace中的关键指标

  • 计算/通信重叠率：理想的DDP训练中，梯度通信（AllReduce）应该和反向传播重叠执行。如果重叠率低（通信时间明显超出计算时间），说明是通信瓶颈。

  • NCCL kernel占比：如果NCCL kernel占总时间>30%，是通信瓶颈。

💼 工作实战：PyTorch DDP的梯度桶化与反向传播重叠

PyTorch DDP的核心优化技巧是梯度桶化（Gradient Bucketing）+ 反向传播重叠，这是理解分布式训练性能的关键。

问题：为什么需要梯度桶化？

如果等所有梯度都计算完再做AllReduce，会出现两个问题：

  1. 通信无法重叠：反向传播期间GPU只能计算，不能通信，利用率低。
  2. NCCL kernel效率低：每个参数单独做AllReduce，小tensor的通信开销远大于大tensor（带宽利用率低）。

解决方案：桶化 + 反向重叠

💡 桶化设计的三个关键参数：
  • bucket_size_mb（默认25MB）：每个桶的大小。太小→NCCL kernel启动开销大；太大→延迟高，第一个桶要等很久才开始通信。
  • find_unused_parameters：处理某些参数没参与loss计算的情况（如MoE中部分专家未激活）。
  • gradient_as_bucket_view：让梯度直接写入桶的buffer，避免额外的内存拷贝。
⚠️ 常见面试坑：桶化顺序

DDP的桶化是反向顺序：最后计算的梯度（最后一层的参数）放入第一个桶，最先开始AllReduce。这是因为反向传播从最后一层开始，最后一层的梯度最先计算完，可以最先开始通信。

如果不按反向顺序桶化，就必须等所有层的梯度都计算完才能开始通信，失去了重叠的意义。

实际工作中的性能调优

DDP训练性能瓶颈分析（实际profiling结果示例）： 理想情况（计算/通信完全重叠）： 反向传播时间：100ms AllReduce时间：60ms 实际步时间：100ms（通信被完全隐藏） 非理想情况（重叠不足）： 反向传播时间：100ms AllReduce时间：120ms 实际步时间：140ms（通信暴露40ms） 优化方向： 1. 增大bucket_size_mb → 减少NCCL kernel启动次数 2. 使用gradient_as_bucket_view → 减少内存拷贝 3. 调整find_unused_parameters=false → 减少额外遍历开销 4. 使用混合精度训练 → 减少通信数据量（FP16=2字节）
💡 面试加分回答：

DDP除了桶化，还有两个容易被忽略的优化：

  1. Static Graph优化（static_graph=True）：如果模型的计算图每次迭代都一样，DDP可以缓存通信顺序，避免每次迭代重新分析哪些参数参与了计算。
  2. 通信钩子（Comm Hook）：可以注册自定义的通信钩子，实现梯度压缩（如PowerSGD）、梯度累加（gradient accumulation不通信）等高级优化。
