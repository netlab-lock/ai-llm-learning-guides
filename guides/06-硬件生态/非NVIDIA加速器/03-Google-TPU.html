🟢 Google TPU

1. TPU 概述

TPU 概述

"tip">通俗类比：想象超市结账，如果只开一个收银台（单核CPU），队伍会很长。TPU就像同时开放了几十个专用快速通道（脉动阵列），专为处理购物车（张量）这种“批量计算”而设计，所以排队（计算延迟）时间大大缩短。
常见误区：易误认为TPU是“万能处理器”。它专为张量计算优化，处理控制流、IO等任务时效率反而不如通用CPU/GPU。编写程序时，应将“数据搬运”和“计算密集”的逻辑清晰分离。
深入探讨：TPUv1的设计指标是15倍于同期GPU的能效比。其核心是一个256x256的脉动阵列（MXU），单周期可执行65,536次乘加操作（MACs），峰值算力达92 TOPS（每秒万亿次操作，INT8精度）。

TPU (Tensor Processing Unit) 是 Google 自研的 AI 加速器，2016 年首次公开。TPU 的设计哲学与 GPU 完全不同：专为矩阵乘法优化的脉动阵列 (Systolic Array)。

2. TPU 代际演进

TPU 代际演进

通俗类比：就像手机从单核CPU发展到多核SoC，TPU的迭代不仅是“更快”，更是“更专用”和“更互联”。v1验证思路，v2引入浮点和HBM，v3优化云服务，v4/v5则通过3D封装和ICI网络，把“多个厨房”连成了“大型中央厨房”。
常见误区：不要仅凭峰值算力（TOPS/TFLOPS）选择代际。v2/v3的内存带宽和容量，以及v4/v5的ICI互联拓扑，对大规模模型的实际训练吞吐量影响可能比单芯片算力更大。
深入探讨：关键演进： • TPUv2/v3: 引入高带宽存储器(HBM)，解决数据饥饿问题。 • TPUv4: 采用3D封装，芯片间ICI带宽达4.8Tbps，可扩展至4096芯片的Pod。 • TPUv5e: 强调成本效益，针对大模型推理和训练优化，每芯片HBM带宽更高。

3. TPU 的核心设计：脉动阵列< 的核心设计：脉动阵列的工作原理，有助于在实际工程中做出更合理的技术选择和优化决策。

TPU 的核心设计：脉动阵列

通俗类比：脉动阵列就像一条“计算流水线”。数据（输入）和权重从阵列边缘“流入”，每个处理单元（PE）像流水线工人，接收上游的数据，做一次乘加，然后将结果和部分数据“流”给下一个邻居。数据在阵列中“脉动”前进，极大复用，减少内存访问。
常见误区：初学者易误以为脉动阵列能处理任意形状的计算。实际上，它对数据的重用模式和流动方式有严格要求。编程时，需要通过XLA编译器将高层张量操作“映射”成适合脉动阵列执行的、规整的数据流。
深入探讨：以一个N x N的脉动阵列计算C = A * B为例： • 权重矩阵B被预加载到每个PE中，固定不变。 • 输入矩阵A从左侧按对角线方式流入。 • 输出矩阵C从底部流出。整个过程仅需N个时钟周期，且每个权重被重用N次。
  时间 T=1:  a11 |  a12 |  a13 |
  输入A ->
  权重B:   b11 | b21 | b31 ... (固定)
           ----|-----|-----|---->
  T=2:     a12 | a13 | a21 |
           ... | ... | ... |
  输出C -> c11 | c12 | c13 | ...
对比维度脉动阵列通用计算单元 (如GPU SM)
数据重用硬件级高效，数据在PE间流动复用依赖软件管理的缓存/共享内存
控制复杂度固定数据流，控制简单，能效高灵活，但控制逻辑复杂，功耗较高
适用操作高度规整的矩阵乘法、卷积更通用的计算模式
策略通信量适用互联典型场景
张量并行(TP)高NVLink节点内多卡
流水线并行(PP)低PCIe/IB跨节点
数据并行(DP)中任意多数据源
专家并行(EP)高NVLinkMoE模型

脉动阵列的优势：极高的计算密度（每个时钟周期完成大量乘累加），劣势：灵活性差（只适合规则的矩阵运算）。

4. TPU Pod 与 ICI 互联

术选择和优化决策。

TPU Pod 与 ICI 互联

通俗类比：如果把单个TPU芯片比作一个“厨师”，那么TPU Pod就是一个“巨型厨房联盟”。ICI（Inter-Chip Interconnect）就是连接这些厨房的专用、高速内部通道。它不仅传输数据，更关键的是支持AllReduce等集合通信原语，让“厨师们”能高效协同完成一道“巨菜”（大模型）。
常见误区：ICI网络拓扑（如2D/3D环面）对通信效率影响巨大。编程时，必须注意计算任务和数据分片在Pod内的放置策略，避免过多跨ICI的通信，否则高性能互联会成为瓶颈。
深入探讨：TPUv4 Pod采用3D环面（torus）拓扑，4096个芯片通过ICI互联，总双向带宽超过4.8 Pbps。每个芯片可访问其他所有芯片的HBM内存，形成一个庞大的共享内存池（128TB）。这种架构使得AllReduce等集体通信操作非常高效。
  TPU 芯片 --ICI--> TPU 芯片 --ICI--> TPU 芯片
    |ICI|               |ICI|
  TPU 芯片  TPU 芯片  TPU 芯片
    |ICI|               |ICI|
  TPU 芯片 --ICI--> TPU 芯片 --ICI--> TPU 芯片

  (箭头代表高速、低延迟的ICI链路)
互联技术场景特点
ICI (TPU)Pod内芯片间专用，超低延迟，高带宽，支持硬件集合通信
NVLink (GPU)服务器内GPU间高带宽，但跨节点需InfiniBand/RoCE
PCIeCPU与加速器间通用标准，带宽和延迟相对较低

5. JAX/XLA 编程模型

JAX/XLA 编程模型

通俗类比：用JAX写代码就像写一份“高级食谱”（纯函数、函数式编程）。XLA编译器则是“首席厨师助手”，它会审阅你的食谱，然后为TPU厨房（硬件）量身定制一套最优的执行步骤（编译优化），比如合并几个简单的切菜步骤（算子融合）以提高效率。
常见误区：XLA的编译是“静态”的，一旦编译完成，计算图就固定了。这意味着代码中要避免频繁改变张量形状或控制流的动态模式，否则会触发反复编译，严重影响性能。调试时，也需要理解编译后的代码可能与原始Python代码差异很大。
深入探讨：XLA的关键优化： 1. **算子融合**：将多个小操作（如加法、激活函数）合并成一个内核，减少内存读写和启动开销。 2. **布局转换优化**：智能安排数据在HBM中的布局，最大化脉动阵列的数据重用。 3. **通信与计算重叠**：在Pod上，XLA可以调度让ICI通信与部分计算同时进行。
  用户代码 (JAX/Python)  -- JIT编译 -->  HLO (中间表示)
                                         |
                                         | XLA优化器
                                         V
  优化后的HLO  -->  TPU机器码  --> 在脉动阵列上执行
框架TPU支持方式特点
JAX原生支持，XLA为后端函数式，易于转换，研究前沿
TensorFlow通过XLA后端支持生态成熟，生产部署多
PyTorch通过`torch_xla`桥接需修改代码，兼容性持续改进
策略通信量适用互联典型场景

流水线并行(PP)低PCIe/IB跨节点
数据并行(DP)中任意多数据源

6. TPU 上的 LLM

TPU 上的 LLM

常见误区：不要忽视批处理大小（Global Batch Size）的设置。TPU的脉动阵列极爱规整、巨大的矩阵乘法。过小的批处理会浪费计算资源；但过大的批处理可能导致内存溢出或损害模型收敛性。需要用“梯度累积”等技巧找到平衡点。
深入探讨：TPU训练LLM的典型配置： • **模型并行 (Tensor Parallel)**：将一个巨大的注意力头/FFN层切分到同一个ICI环面内的多个TPU芯片上。 • **数据并行**：在不同ICI环面或Pod之间复制模型，处理不同数据批次。 • **流水线并行**：将模型的不同层（如Transformer层）分配到不同的TPU芯片组上。三者常混合使用。
  [数据批次1] --> | TP 芯片组 A (层1-10) | --> [输出]
                  |         ICI          |
  [数据批次2] --> | TP 芯片组 B (层11-20)| --> [输出]
  ...           (数据并行)  (流水线并行)
并行策略在TPU上的应用优势
张量并行 (TP)切分单层内的计算减少通信量，适合超大矩阵
流水线并行 (PP)将模型层分配到不同芯片可扩展至超深模型
数据并行 (DP)复制模型，处理不同数据扩展吞吐量最简单
策略通信量适用互联典型场景
张量并行(TP)高NVLink节点内多卡

数据并行(DP)中任意多数据源
专家并行(EP)高NVLinkMoE模型
  • PaLM (540B): Google 的大语言模型，TPU v4 Pod 训练
  • Gemini: Google 的多模态模型，TPU v5e 训练

7. TPU vs GPU

TPU 上的 LLM

">通俗类比：TPU像“专科手术刀”，GPU像“瑞士军刀”。GPU拥有大量通用核心（CUDA Core）和复杂的控制逻辑，灵活性强，能高效处理图形、物理模拟等多样任务。TPU则将晶体管预算集中给了超大脉动阵列，在“纯矩阵运算”这个特定场景下，追求极致的吞吐量和能效比。
常见误区：误区是简单地对比“谁更好”。选择取决于工作负载： • 训练/推理大型密集矩阵模型（如LLM）：TPU集群可能更优。 • 需要复杂控制流、稀疏计算或强大生态系统（CUDA）的任务：GPU通常是更稳妥的选择。
深入探讨：关键参数对比（以高端产品为例）： • **峰值算力 (FP16/BF16)**：TPUv5e ~393 TFLOPS；NVIDIA H100 SXM ~989 TFLOPS。 • **内存带宽**：TPUv5e HBM 820 GB/s；H100 HBM 3.35 TB/s。 • **设计哲学**：TPU优先考虑 **FLOPS/$** 和 **FLOPS/Watt**（能效）；GPU追求绝对性能和通用性。

深入探讨TPU 概述的技术本质： 从原理层面分析，TPU 概述的核心工作机制可以用排队论（Queuing Theory）建模。 设请求到达率为λ、服务率为μ，则系统利用率ρ = λ/μ。 为什么ρ不能接近1？因为当ρ→1时，平均等待时间趋向无穷大—— 这就是为什么生产环境需要保留30-50%的容量余量的数学本质。 从底层硬件看，GPU的计算吞吐和内存带宽是两个核心约束， 推导算术强度 = FLOPS / Bytes，决定了任务是计算受限还是内存受限。

  1. 核心原理: JAX on TPU的XLA编译流程是理解TPU编程模型的关键。JAX代码通过jit装饰器触发XLA编译：① Python代码被JAX的trace机制捕获，生成JAXPR中间表示；② JAXPR被转换为HLO（High Level Operations）IR；③ XLA编译器对HLO进行优化（算子融合、内存布局优化、脉动阵列调度）；④ 生成TPU可执行的二进制代码。关键特点：XLA是静态图编译，不支持Python级别的动态控制流，需要用jax.lax.cond/jax.lax.scan等原语替代。

    import jax.numpy as jnp
    from flax import linen as nn

    class TransformerBlock(nn.Module):
    features: int = 512
    @nn.compact
    def __call__(self, x):
    attn = nn.SelfAttention(num_heads=8)(x)
    x = nn.LayerNorm()(x + attn)
    ff = nn.Dense(self.features * 4)(x)
    ff = nn.gelu(ff)
    ff = nn.Dense(self.features)(ff)
    return nn.LayerNorm()(x + ff)

    @jax.jit # 触发XLA编译
    def train_step(params, x, y):
    def loss_fn(p):
    pred = TransformerBlock().apply({"params": p}, x)
    return jnp.mean((pred - y) ** 2)
    loss, grads = jax.value_and_grad(loss_fn)(params)
    return loss, grads
    首次调用train_step时，JAX会花数秒进行XLA编译（生成HLO→优化→TPU二进制），后续调用直接使用编译好的代码，速度极快。

  2. 核心原理: TPU v5e vs H100推理延迟对比需考虑多个维度：① 算力——TPU v5e BF16约393 TFLOPS，H100 BF16约1979 TFLOPS（约5倍差距）；② 内存带宽——TPU v5e约819 GB/s，H100约3.35 TB/s（约4倍差距）；③ 内存容量——TPU v5e 16GB HBM，H100 80GB HBM；④ 互联——TPU v5e通过ICI互联组成Pod，H100通过NVLink/NVSwitch互联。对于LLM推理的Decode阶段（内存瓶颈），延迟主要由模型参数量÷内存带宽决定。H100带宽是TPU v5e的4倍，因此Decode延迟约为TPU v5e的1/4。

    实际有效吞吐约为理论值的50-70%。

="tip">通俗类比：把这张表当成你的“TPU速查字典”。当遇到不懂的缩写或概念时，先来这里翻一翻。理解了这些基础“单词”，你才能读懂更复杂的“技术文章”（如XLA文档或Pod架构论文）。
常见误区：注意术语在不同语境下的差异。例如，“FLOPS”通常指每秒浮点运算次数，但TPU的“TOPS”常指每秒整数操作次数（如INT8）。对比时务必确认精度类型（FP32, BF16, INT8等）是否一致。
深入探讨：关键性能单位换算： • **MACs (乘加操作)**：一次MAC = 两次浮点运算（乘+加）。 • **FLOPS**：每秒浮点运算次数。1 TFLOPS = 10^12 FLOPS。 • **TOPS**：每秒万亿次操作。在INT8精度下，1 TOPS ≈ 1 TFLOPS (理论值)。 • **带宽**：GB/s（每秒传输千兆字节）。
  TPU术语关系链:
  脉动阵列 (MXU) --执行--> 矩阵乘法 (GEMM)
       |                     |
       |--> 产生             |--> 单位: MACs, FLOPS, TOPS
  数据流 <-----------------> 内存 (HBM) --带宽: GB/s

  1. 核心原理: 「1. TPU 概述」的核心在于理解其底层工作机制。从系统视角分析，该技术解决了LLM基础设施中的关键瓶颈问题。其数学本质可以归结为一个优化问题：在给定约束条件下最大化目标函数（如吞吐量、资源利用率或成本效率）。具体而言，需要掌握：问题形式化描述、算法设计思路、复杂度分析三个层面。从硬件视角看，性能通常受限于三个维度：计算能力（FLOPs）、内存带宽（GB/s）和通信带宽（Gb/s）。理解这三者的关系，就能推导出大多数优化策略。
    关键要点: ①核心原理的数学推导和物理意义；②关键参数的设计原则和调优方法；③与替代方案的对比分析；④实际部署中的工程考量；⑤性能指标的量化和评估方法。
    实际案例: 以一个生产环境部署场景为例：当请求QPS从100增长到1000时，系统瓶颈从计算受限转变为内存受限。通过Profiling发现Decode阶段HBM带宽利用率达到85%，此时应该采用KV Cache量化（减少带宽需求）或分离架构（将Prefill和Decode部署在不同GPU上）来优化。实测优化后TTFT降低40%，TPS提升2.5x。
  2. 核心原理: 在实际工程应用中，本章介绍的技术需要根据具体场景进行适配。关键技术决策包括：架构选择（一体vs分离）、精度选择（FP16/FP8/INT4）、缓存策略（LRU/LFU/Size-based）等。决策依据是请求特征（prompt长度分布、并发量、SLA要求）和硬件条件（GPU型号、网络带宽、存储类型）。标准化的评估方法是通过Profiling确定瓶颈类型：计算受限→量化/剪枝；内存受限→算子融合/缓存优化；通信受限→拓扑优化/通信压缩。
    关键要点: ①需求分析和场景匹配；②技术参数的调优方法；③性能评估和瓶颈定位；④成本-性能权衡分析；⑤生产环境的最佳实践。
    实际案例: 某RAG应用部署中，通过Profiling发现Prefill阶段占端到端延迟的70%。原因是system prompt包含5K token的检索文档。通过启用Prefix Caching，Prefill只需处理200 token的用户查询，TTFT从800ms降至120ms，节省85%。同时使用FP8量化进一步提升Decode阶段速度。
  3. 核心原理: 对比分析需要从多个维度展开：性能（延迟、吞吐）、精度（量化损失）、成本（GPU小时）、复杂度（运维难度）、适用场景（prompt长度、并发模式）。不同技术方案各有优劣：如一体架构简单但GPU利用率低，分离架构高效但引入传输开销。选择依据是业务需求：如果SLA要求TTFT<200ms且QPS>100，应选择分离架构+Prefix Caching；如果QPS<10且prompt短，一体架构即可满足。
    关键要点: ①多维度对比评估框架；②场景匹配和方案选择；③实验验证和性能benchmark；④渐进式优化路径；⑤技术债和可维护性考量。
    实际案例: 某团队对比了三种方案：A)一体架构+FP16 B)一体架构+FP8+Prefix Caching C)分离架构+FP8+Prefix Caching。方案A月度成本$50K，TTFT=500ms；方案B成本$30K，TTFT=200ms；方案C成本$25K，TTFT=150ms。考虑到运维复杂度和团队能力，选择了方案B作为过渡方案，半年后迁移到方案C。
