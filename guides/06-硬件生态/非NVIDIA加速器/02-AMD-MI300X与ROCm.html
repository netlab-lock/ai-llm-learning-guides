🔴 AMD MI300X 与 ROCm

1. AMD MI300X 架构

MI300X 是 AMD 2023 年底发布的 AI 专用 GPU，基于 CDNA 3 架构，是目前 NVIDIA H100/H200 的最强竞争者。

1.2 CDNA 3 架构原理

CDNA 3 架构的本质上采用了 Chiplet 设计，将计算芯片（XCD）和 I/O 芯片分离。MI300X 包含 8 个 XCD（加速器复合芯片），每个 XCD 包含 38 个 CU（Compute Unit），总计 304 个 CU。这种设计为什么重要？因为 Chiplet 允许 AMD 在同一封装内混合不同工艺节点——计算芯片用 5nm，I/O 芯片用 6nm，大幅提高了良率和性价比。

1.3 内存子系统详解

MI300X 的 192GB HBM3 内存是其最大的卖点。从数学本质上看，内存带宽决定了 LLM 推理的 decode 阶段性能。对于一个 70B 参数的模型（FP16），每个 decode token 需要读取 140GB 的权重数据。MI300X 的 5.3 TB/s 带宽意味着理论上每秒可以生成 5300/140 ≈ 37.8 个 token（单用户），而 H100 的 3.35 TB/s 只能达到 3350/140 ≈ 23.9 token/s。

深入探讨——内存带宽与推理性能的数学关系：LLM decode 阶段的性能上限由 Roofline 模型决定。对于矩阵-向量乘法（decode 的主要计算），算术强度 = 2×N / (2×N) = 1 FLOP/Byte（N 为参数量）。这意味着 decode 阶段完全是内存带宽瓶颈。具体公式：decode_tokens/s ≈ HBM_bandwidth / (2 × model_params × bytes_per_param)。对于 MI300X：5300 GB/s / (2 × 70B × 2B) = 5300/280 ≈ 18.9 tok/s（单用户实际值会因 overhead 更低）。这个公式解释了为什么 HBM 带宽比算力更重要——decode 阶段算力严重过剩。

1.4 Infinity Fabric 互联

MI300X 使用 Infinity Fabric 进行多卡互联，单向带宽 896 GB/s。Infinity Fabric 本质上是一个一致性互连协议，支持缓存一致性（Cache Coherence），这意味着多卡之间的数据同步不需要显式的内存拷贝。相比之下，NVLink 4.0 的单向带宽为 900 GB/s，两者基本持平。

MI300X vs H100 多卡互联拓扑: MI300X (8 XCD 单封装): XCD0 ←→ XCD1 ←→ XCD2 ←→ XCD3 ↕ ↕ ↕ ↕ I/O Die (Infinity Fabric, 统一内存) ↕ ↕ ↕ ↕ XCD4 ←→ XCD5 ←→ XCD6 ←→ XCD7 → 所有 XCD 共享 192GB 统一内存, 无需跨卡通信! H100 (多卡方案, 70B 模型): H100-0 (80GB) ←══NVLink══→ H100-1 (80GB) ↑ 张量并行: 每层切分到 2 卡 ↑ 需要 AllReduce 通信同步激活值 ↑ 通信开销: ~10-15% 额外延迟
通俗类比：MI300X 的核心优势——192GB HBM3 就像一个巨大的书桌。H100 的 80GB 像小书桌，放不下 70B 模型的全部资料，需要两张桌子拼起来（2 卡张量并行）。MI300X 的大书桌一张就够，省去了"两张桌子之间传递资料"的开销（跨卡通信），也省了一半的硬件成本。
常见误区：很多人看到 MI300X 的 192GB 就认为它能跑所有大模型。实际上 192GB 装 FP16 的 Llama-3-70B（140GB）后只剩约 52GB 给 KV Cache 和激活值，受限于显存余量，长序列推理时仍然可能 OOM。另外，192GB 不等于"能跑 405B 模型"——FP16 的 405B 需要 810GB，即使 INT4 也需要约 200GB。

2.2 HIP 兼容层

HIP (Heterogeneous-compute Interface for Portability) 是 AMD 的 CUDA 兼容层：

// CUDA 代码
cudaMalloc(&d_a, size);
kernel<<>>(d_a);

// HIP 代码 (几乎相同的 API)
hipMalloc(&d_a, size);
hipLaunchKernelGGL(kernel, grid, block, 0, 0, d_a);

// 使用 hipify 工具自动转换
# hipify-perl kernel.cu > kernel.hip

2.3 ROCm 安装与环境配置

ROCm 的安装过程比 CUDA 复杂得多，这也是其生态不成熟的一个重要体现。以下是 Ubuntu 22.04 上的典型安装流程：

# 1. 添加 ROCm 仓库
wget -qO - https://repo.radeon.com/rocm/rocm.gpg.key | sudo apt-key add -
echo "deb [arch=amd64] https://repo.radeon.com/rocm/apt/6.0 jammy main" | sudo tee /etc/apt/sources.list.d/rocm.list

# 2. 安装 ROCm
sudo apt update
sudo apt install rocm-hip-sdk rocm-hip-runtime-dev

# 3. 设置环境变量
echo 'export PATH=$PATH:/opt/rocm/bin' >> ~/.bashrc
echo 'export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/opt/rocm/lib' >> ~/.bashrc

# 4. 验证安装
rocm-smi          # 查看 GPU 状态
hipcc --version    # 查看 HIP 编译器版本

# 5. 安装 PyTorch (ROCm 版本)
pip install torch --index-url https://download.pytorch.org/whl/rocm6.0
常见误区：很多人以为安装了 ROCm 就能无缝运行所有 PyTorch 代码。实际上 ROCm 版 PyTorch 通常落后主线 1-2 个版本，而且部分自定义 CUDA kernel（尤其是 Triton 编写的）在 ROCm 上可能无法编译或运行异常。部署前务必在 ROCm 环境中做完整的兼容性测试。

2.5 Triton 在 ROCm 上的支持状态

Triton 是当前最热门的 GPU kernel 编写工具，但其在 ROCm 上的支持仍然不完善。核心问题在于 Triton 的后端代码生成最初是针对 NVIDIA GPU 的 PTX 指令集设计的，移植到 AMD 的 GCN/CDNA 指令集需要大量工作。

# 检查 Triton 是否支持 AMD GPU
import torch
print(torch.cuda.is_available())  # ROCm 下也返回 True
print(torch.version.hip)          # 查看 ROCm 版本

# Triton kernel 在 AMD 上可能遇到的问题
# 1. 某些 triton.atomic_add 操作不支持
# 2. autotune 配置空间需要针对 AMD GPU 重新设计
# 3. 部分 kernel 编译时间比 NVIDIA 长 2-5 倍
深入探讨——ROCm 的编译器栈工作机制：ROCm 使用 LLVM 作为编译后端，编译流程为：HIP 源码 → Clang 前端 → LLVM IR → AMDGPU 后端 → GCN/CDNA 指令。这个流程与 CUDA 的 NVCC 编译器（NVCC → PTX → SASS）有本质区别。为什么这很重要？因为 LLVM 是开源的，这意味着理论上 ROCm 的编译器可以受益于整个 LLVM 社区的优化，而 CUDA 的编译器优化完全由 NVIDIA 控制。但反过来说，AMD 也需要等待 LLVM 社区合并 AMDGPU 后端的改进，响应速度可能不如 NVIDIA 内部迭代快。

3.2 vLLM 在 MI300X 上的部署

vLLM 是目前在 MI300X 上最成熟的推理框架。以下是实际部署的关键参数：

# 在 MI300X 上启动 vLLM 服务
docker run -it --network=host --group-add=video \
    --ipc=host --cap-add=SYS_PTRIO \
    --device=/dev/kfd --device=/dev/dri \
    vllm/vllm-openai:rocm6.0 \
    --model meta-llama/Llama-3-70B-Instruct \
    --dtype float16 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 4096 \
    --tensor-parallel-size 1  # 单卡即可!
通俗类比：在 MI300X 上部署 70B 模型，就像用一辆大卡车搬家——一次就能装完所有东西。而在 H100 上部署同样的模型，像用两辆小卡车——需要两辆车同时出发，还得通过对讲机协调（张量并行通信），既多了协调开销，又多了硬件成本。

3.3 性能调优要点

在 MI300X 上获得最优性能需要注意以下原理和工程细节：

通俗类比：MI300X 的 304 个 CU 就像 304 条生产线。如果只开一条生产线（batch=1），其他 303 条都在晒太阳——这就是为什么单用户推理时 GPU 利用率极低。要让工厂高效运转，需要"攒够订单再一起生产"（增大 batch size），让每条生产线都有活干。
  • 内存对齐：CDNA 架构对内存对齐要求严格，确保 tensor 的起始地址是 256 字节对齐的
  • Wave 执行效率：MI300X 的 CU 数量（304）需要足够的并行任务来填满，避免出现"尾波"（tail wave）效应
  • XCD 间负载均衡：8 个 XCD 之间的任务分配需要均匀，否则部分 XCD 空闲而其他过载
Wave 执行效率示意: 理想情况 (高 occupancy): XCD 0: ████████████████████████████ 100% XCD 1: ████████████████████████████ 100% XCD 2: ████████████████████████████ 100% ... XCD 7: ████████████████████████████ 100% 尾波效应 (低 batch size): XCD 0: ████████████████████████████ 100% XCD 1: ████████████████████████████ 100% XCD 2: ████████████████████████████ 100% ... 4.2 性能差距的根源

MI300X 的理论算力高于 H100（FP16 高 32%），但实际 LLM 推理性能差距通常在 0-20% 之间。为什么会出现"纸面参数领先但实际性能不领先"的情况？

常见误区：很多人认为"算力高 = 推理快"。实际上 LLM 推理的 decode 阶段是内存带宽瓶颈，不是算力瓶颈。MI300X 的 HBM 带宽（5.3 TB/s）只比 H100（3.35 TB/s）高 58%，而不是算力差距的 2 倍。更关键的是，NVIDIA 的 CUDA 生态更成熟，kernel 优化更充分——同样的计算任务，手写优化的 CUDA kernel 可能比 ROCm 上的 kernel 快 10-20%，这直接抵消了硬件参数优势。

4.3 成本效益分析

选择 MI300X 还是 H100，本质上是一个成本效益计算问题：

# 成本效益对比公式
# 性价比 = (吞吐量 tok/s) / (硬件成本 $)

# MI300X (单卡 ~$15,000)
mi300x_perf = 80  # tok/s (INT4)
mi300x_cost = 15000
mi300x_ratio = mi300x_perf / mi300x_cost  # 0.0053 tok/s/$

# H100 (单卡 ~$30,000)
h100_perf = 100  # tok/s (INT4)
h100_cost = 30000
h100_ratio = h100_perf / h100_cost  # 0.0033 tok/s/$

# MI300X 性价比高 60%! 但需要考虑软件兼容性成本
深入探讨——为什么 NVIDIA 的软件优化能弥补硬件劣势：CUDA 生态经过 18 年的发展，积累了大量的手写优化 kernel。以 FlashAttention 为例，其 CUDA 实现针对 H100 的 TMA（Tensor Memory Accelerator）做了专门优化，利用异步内存拷贝实现计算和数据搬运的重叠。而 ROCm 上的 FlashAttention 移植版通常无法充分利用 MI300X 的特性，导致相同算法在不同平台上有 15-30% 的性能差距。这就是为什么硬件参数对比只是故事的一半——软件栈的成熟度同样重要。

5. 已知问题与坑点

5.1 软件生态问题

  • Triton 兼容性：许多 Triton kernel 在 ROCm 上有兼容性问题
  • NCCL → RCCL：集合通信库需要替换，性能略有差异
  • 驱动版本：ROCm 版本更新频繁，兼容性矩阵复杂
  • PyTorch 版本：ROCm 版本的 PyTorch 通常落后主线 1-2 个版本
  • 多卡互联：Infinity Fabric 的多卡扩展性略逊于 NVLink

5.2 硬件兼容性注意事项

MI300X 的部署有一些硬件层面的注意事项：

  • 散热需求：750W TDP 需要液冷或高风量散热方案
  • PCIe 兼容：需要 PCIe Gen5 x16 才能发挥全部互联带宽
  • 主板要求：需要支持 EPYC 9004 系列的主板才能启用全部 Infinity Fabric 特性

✅ 参考答案与详解
  1. 核心原理: MI300X 的 Chiplet 设计核心在于将计算芯片（XCD，5nm工艺）与 I/O 芯片（6nm工艺）分离制造，通过 Infinity Fabric 实现高带宽互连。这种设计的数学本质是：小芯片的良率随面积指数下降（Yield ∝ e^(-A×D)，A为芯片面积，D为缺陷密度），将一个大芯片拆分为多个小芯片后，总良率大幅提升。同时，不同工艺节点的混合使用（计算用先进制程、I/O用成熟制程）进一步降低了成本。192GB HBM3 的 decode 性能公式为 tokens/s ≈ HBM_bandwidth / (2 × model_params × bytes_per_param)，MI300X 的 5.3 TB/s 带宽在 70B 模型上理论值约 18.9 tok/s，单卡即可运行 70B FP16 模型（140GB），而 H100 80GB 需要 2 卡张量并行。
    关键要点: ①Chiplet 拆分使良率从单一大芯片的~30%提升至小芯片的~85%以上；②混合工艺节点（5nm计算+6nm I/O）降低制造成本约 25-40%；③Infinity Fabric 提供 896GB/s 多卡互联，支持缓存一致性统一内存；④192GB HBM3 使单卡可运行 70B FP16 模型，省去跨卡通信开销；⑤hipify 工具可自动转换~90% CUDA 代码，但 Warp 大小差异（32 vs 64）需手动调优。
    实际案例: 在 MI300X 上部署 Llama-3-70B，使用 vLLM 单卡（tensor-parallel-size=1）即可运行，实测 decode 吞吐约 30 tok/s（FP16）或 80 tok/s（INT4）。对比 H100 需要 2 卡（每卡 80GB），虽然双卡 H100 可达 40 tok/s，但硬件成本是 MI300X 的 2 倍。使用 hipify-perl 将 FlashAttention 的 CUDA kernel 转换后，ROCm 版本性能约为 CUDA 版本的 80-85%，主要差距来自 MIOpen 对 cuDNN 的替代尚未完全优化。
    关联知识: Chiplet 设计思想与 AMD EPYC CPU 的 CCD/IO Die 架构一脉相承，可参考 CPU 领域的 Chiplet 实践；Roofline 模型是分析 MI300X 性能瓶颈的核心工具；Infinity Fabric 的缓存一致性特性与 NVIDIA NVLink-C2C 有本质区别，前者支持跨卡统一地址空间，后者仅支持高带宽数据传输。
