2.2 合并访问 (Coalesced Access)

全局内存合并访问: 合并访问(好): 一个 Warp 的 32 个线程访问连续地址 Thread 0: addr 0 Thread 1: addr 1 Thread 2: addr 2 ... Thread 31: addr 31 → 合并为 1 次内存事务 (128 bytes) 非合并访问(差): 访问不连续的地址 Thread 0: addr 0 Thread 1: addr 100 Thread 2: addr 200 ... → 需要 32 次独立内存事务! 性能差异: 合并 vs 非合并可达 10-30× 速度差

2.3 Bank Conflict

Shared Memory 的 Bank 结构: Shared Memory 被分成 32 个 bank (每个 bank 4 字节宽) Bank 0: [0, 32, 64, ...] Bank 1: [1, 33, 65, ...] ... Bank 31: [31, 63, 95, ...] 无 Bank Conflict (好): 32 个线程访问 32 个不同的 bank → 1 个周期完成 Bank Conflict (差): 多个线程访问同一个 bank → 串行化 2-way conflict: 2 个周期 32-way conflict: 32 个周期 (最差情况) 解决方法: 1. Padding: 在数组末尾加 1 个元素改变对齐 2. Swizzle: 重新排列数据布局 3. 避免跨行访问

2.4 TMA (Tensor Memory Accelerator)

TMA: Hopper 的异步数据搬运引擎 传统数据搬运(同步): CPU/GPU 发出 load 指令 → 等待数据到达 → 使用数据 (计算单元在等待时闲置) TMA 数据搬运(异步): 1. 发出 TMA 异步拷贝指令 2. 计算单元继续做其他计算 3. 数据在后台从 HBM → Shared Memory 4. 数据到达后通知计算单元 5. 计算单元使用数据 TMA 支持: • 1D/2D/3D 张量拷贝 • 自动处理边界和填充 • 与 WGMMA 配合: TMA 加载 → WGMMA 计算 • 多级流水线: TMA 和 WGMMA 重叠执行

2.6 LLM 的内存瓶颈分析

LLM 推理的内存访问模式: Decode 阶段 (逐token生成): • 每个 token 只做一次前向传播 • 需要读取全部模型参数 (70B × 2 bytes = 140GB) • HBM 带宽: 3.35 TB/s • 理论最大: 3350 / 140 ≈ 24 tokens/s → 内存瓶颈! Prefill 阶段 (处理输入): • 一次处理整个输入序列 • 大量矩阵乘法 (batch × seq_len × hidden) • 计算量大，Tensor Core 利用率高 → 计算瓶颈! 优化方向: Decode: 量化(FP8/INT4减少参数大小), KV Cache优化 Prefill: FlashAttention, 算子融合

import time
import torch

def profile_inference(model, input_data, warmup=10, iterations=100):
    """推理性能分析：测量延迟、吞吐和显存"""
    # Warmup
    for _ in range(warmup):
        with torch.no_grad():
            model(input_data)
    torch.cuda.synchronize()
    
    # Measure latency
    latencies = []
    for _ in range(iterations):
        start = time.perf_counter()
        with torch.no_grad():
            output = model(input_data)
        torch.cuda.synchronize()
        latencies.append(time.perf_counter() - start)
    
    # Statistics
    avg_ms = sum(latencies) / len(latencies) * 1000
    p50 = sorted(latencies)[len(latencies)//2] * 1000
    p99 = sorted(latencies)[int(len(latencies)*0.99)] * 1000
    mem_mb = torch.cuda.max_memory_allocated() / 1024 / 1024
    
    print(f"Latency: avg={avg_ms:.2f}ms, p50={p50:.2f}ms, p99={p99:.2f}ms")
    print(f"Peak Memory: {mem_mb:.0f}MB")
    print(f"Throughput: {1000/avg_ms:.0f} req/s")
    return {"avg_ms": avg_ms, "p50": p50, "p99": p99, "mem_mb": mem_mb}

✅ 参考答案与详解
  1. 核心原理: H100的Decode推理是内存瓶颈，因为Decode阶段每个token需要读取全部模型参数(70B×2bytes=140GB)但计算量极小(一次矩阵向量乘)。算术强度=计算量/数据量≈2 FLOPS/byte(每个参数一次乘加)，远低于H100的算力/带宽比(989TFLOPS/3.35TB/s≈295)。最大decode速度=带宽/参数量×每参数字节数=3350GB/s÷140GB≈24tokens/s(单卡)。Prefill阶段处理整个输入序列，batch×seq_len×hidden的三维矩阵乘具有很高的算术强度(>295)，属于计算瓶颈。这就是为什么推理优化重点在decode阶段减少数据搬运(量化KV Cache、FP8推理)
    关键要点: ①Decode算术强度≈2 FLOPS/byte ②远低于H100拐点295 ③理论最大24 tokens/s ④Prefill是计算瓶颈
    实际案例: vLLM中70B模型decode实测约20tokens/s，接近理论上限
    关联知识: 与Roofline模型、KV Cache优化和量化推理相关
  2. 核心原理: Shared Memory分为32个bank(每个4字节宽)，地址按mod 32映射到bank。当同一Warp中多个线程访问同一bank的不同地址时发生bank conflict：2-way conflict需2周期，32-way需32周期。Padding通过在数组每行末尾添加元素(如float[32][32]→float[32][33])改变列到bank的映射，使原来冲突的线程访问不同bank。例如行优先存储的32×32矩阵，第0行第0列和第1行第0列都映射到bank 0(地址0和32，32 mod 32=0)。加1列padding后，第1行第0列地址变为33，映射到bank 1，消除冲突
    关键要点: ①32个bank按地址mod 32映射 ②同bank多线程访问串行化 ③Padding改变地址对齐 ④Swizzle更高级的重排
    实际案例: FlashAttention的Q/K tile使用padding消除32-way bank conflict
    关联知识: 与Shared Memory架构、FlashAttention实现和CUTLASS的swizzle相关
  3. 核心原理: Double Buffering的GEMM Shared Memory布局设计：将SMem分为两个buffer(A和B)，每个buffer存放一个tile的A矩阵和B矩阵数据。假设tile size为BLOCK_M×BLOCK_K和BLOCK_K×BLOCK_N，则每个buffer需要(BLOCK_M×BLOCK_K + BLOCK_K×BLOCK_N)×sizeof(float16)字节。以BLOCK_M=128, BLOCK_N=128, BLOCK_K=32为例：buffer大小=(128×32+128×32)×2B=16KB，两个buffer共32KB(在228KB SMem限制内)。执行流程：①TMA加载tile[i+1]到buffer B ②计算tile[i]使用buffer A ③交换buffer角色 ④循环。关键：加载和计算完全重叠，隐藏HBM→SMem延迟
    关键要点: ①双buffer交替使用 ②每buffer=(MK+KN)×2B ③加载/计算重叠 ④TMA硬件管理搬运
    实际案例: CUTLASS 3.x的mainloop使用double buffering，num_stages=2-4
    关联知识: 与TMA异步搬运、CUTLASS pipeline和GEMM优化相关
⚠️ 常见误区：Shared Memory越大越好：Shared Memory和L1 Cache共享物理SRAM(每SM共228KB)。如果把全部228KB分配给Shared Memory，L1 Cache就没了。某些访问模式下，让L1 Cache自动缓存可能比手动管理Shared Memory更高效。关键是根据具体workload做trade-off。
⚠️ 常见误区：合并访问只对全局内存重要：合并访问(Coalesced Access)确实主要针对全局内存(HBM)，但Shared Memory的Bank Conflict也是类似问题——都是多个线程同时访问同一物理资源导致的串行化。两者都需要关注。
🔬 深入探讨：TMA的2D/3D张量拷贝

TMA不只是一次简单的内存拷贝。它支持多维张量拷贝，可以高效地从不连续的内存布局中提取数据块。例如，矩阵在HBM中行主序存储，取出一个16×16的tile需要16次独立拷贝(每行一次)。TMA可以一次操作完成2D tile拷贝——硬件自动处理行间的stride跳转。这对FlashAttention尤其重要——Q、K、V矩阵的tile提取完全由TMA硬件完成。
🔬 深入探讨：L2 Cache Partitioning与Persistence

Hopper的L2 Cache支持cache partitioning。你可以通过CUDA API将L2 Cache的一部分"钉住"给特定数据，不让其他数据驱逐。这对Attention的KV Cache特别有用——推理时KV Cache被频繁访问，通过持久化在L2中可以避免每次decode都从HBM重新加载。

注意：L2 Cache是所有SM共享的(50MB)，持久化策略需要谨慎——过度持久化会导致其他数据频繁cache miss。

内存层次的数学本质与推导： GPU内存层次的设计原理可以用存储金字塔理论推导。 设第i层容量为C_i、带宽为B_i、延迟为L_i，则理想情况下： C_1 << C_2 << ... << C_n 且 B_1 >> B_2 >> ... >> B_n。 为什么要这样设计？数学本质是利用局部性原理—— 时间局部性（最近访问的数据 likely 被再次访问）和空间局部性（相邻数据 likely 被一起访问）。 从底层硬件看，A100的存储层次：寄存器(每SM 256KB, ~19TB/s) → 共享内存(每SM 164KB, ~19TB/s) → L2 Cache(40MB, ~5TB/s) → HBM(80GB, 2TB/s)。 每一级的工作机制是：容量×10, 带宽÷4, 延迟×5。 FlashAttention的原理正是利用这一层次结构——将注意力计算分块到SRAM中完成， 避免将N×N的注意力矩阵写入HBM，将IO从O(N²)降低到O(N²d²/M)。
