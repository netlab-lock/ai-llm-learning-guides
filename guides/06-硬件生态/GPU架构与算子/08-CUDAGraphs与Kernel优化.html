一、背景与动机

一、背景与动机

n 的生成需要执行几十个 kernel(Attention、FFN、LayerNorm、Embedding...)。每个 kernel 的启动开销约 5-10 微秒。对于 70B 模型，一轮推理可能有 100+ 个 kernel，启动开销就是 0.5-1 毫秒——而实际计算可能只有几毫秒。启动开销占比高达 20-30%！

CUDA Graphs 通过预录制→重放的方式，把整个 kernel 序列打包成一个图，一次启动就能执行所有 kernel，消除重复的启动开销。

3.2 CUDA Graphs 的执行对比

有无 CUDA Graphs 的执行对比: 无 CUDA Graphs: CPU: [准备1] [启动1] [等待] [准备2] [启动2] [等待] [准备3]... GPU: [执行1] [执行2] [执行3] ─────────────────────────────────────────────────────────→ 时间 CPU频繁介入, GPU有空闲 有 CUDA Graphs: CPU: [录制一次] GPU: [执行1][执行2][执行3][执行4][执行5]... ─────────────────────────────────────────────────────────→ 时间 CPU只启动一次, GPU连续执行, 无空闲

四、Stream 与 Event

4.1 CUDA Stream

CUDA Stream 并行执行: Stream 1: [Kernel A] ──→ [Kernel C] ──→ [Kernel E] Stream 2: [Kernel B] ──→ [Kernel D] ↑ 并行执行 ↑ 同一 Stream 内: 顺序执行 (FIFO) 不同 Stream 间: 可以并行 (如果无依赖) 用途: • 数据传输和计算重叠 Stream 1: [H2D拷贝] → [计算] Stream 2: [H2D拷贝] → [计算] • 多个推理请求并行 Stream 1: [请求1的推理] Stream 2: [请求2的推理]

4.2 CUDA Event

CUDA Event 同步机制: Stream 1: [Kernel A] ──[Record Event]──→ [继续计算] │ Stream 2: [Kernel B] ──[Wait Event]──→ [Event到达后继续] │ CPU: [Event同步] ──[记录时间]──→ [性能分析] 用途: 1. Stream间同步: 确保依赖关系 2. 性能计时: 精确测量kernel执行时间 3. CPU-GPU同步: 等待GPU完成

7.1 vLLM 的 CUDA Graphs 策略

# 一、背景与动机 - 性能基准测试与分析
import time
import torch

def profile_inference(model, input_data, warmup=10, iterations=100):
    """推理性能分析：测量延迟、吞吐和显存"""
    # Warmup
    for _ in range(warmup):
        with torch.no_grad():
            model(input_data)
    torch.cuda.synchronize()
    
    # Measure latency
    latencies = []
    for _ in range(iterations):
        start = time.perf_counter()
        with torch.no_grad():
            output = model(input_data)
        torch.cuda.synchronize()
        latencies.append(time.perf_counter() - start)
    
    # Statistics
    avg_ms = sum(latencies) / len(latencies) * 1000
    p50 = sorted(latencies)[len(latencies)//2] * 1000
    p99 = sorted(latencies)[int(len(latencies)*0.99)] * 1000
    mem_mb = torch.cuda.max_memory_allocated() / 1024 / 1024
    
    print(f"Latency: avg={avg_ms:.2f}ms, p50={p50:.2f}ms, p99={p99:.2f}ms")
    print(f"Peak Memory: {mem_mb:.0f}MB")
    print(f"Throughput: {1000/avg_ms:.0f} req/s")
    return {"avg_ms": avg_ms, "p50": p50, "p99": p99, "mem_mb": mem_mb}

原理与推导： 从原理层面理解，该技术的工作机制基于信息论的基本定理。 设信源熵为 H(X) = -Σ p(x)log p(x)，为什么这很重要？ 因为信息熵给出了无损压缩的理论下界——这也解释了为什么量化不可避免地 会带来信息损失。推导量化误差的上界：对于均匀量化， E[|x-q(x)|²] ≤ Δ²/12，其中Δ为量化步长。 这一公式揭示了精度与效率之间的数学本质关系： 步长越小（精度越高），误差越小，但需要的比特数越多。 从底层实现来看，现代加速器的Tensor Core正是利用这一原理， 通过混合精度计算在保持精度的同时最大化吞吐量。

⚠️ 常见误区：CUDA Graphs能加速训练：CUDA Graphs主要对推理有效。训练时kernel通常很大，启动开销占比很低(1-2%)，收益很小。而且训练时batch size、序列长度经常变化，Graph的固定形状限制更麻烦。最适合场景：推理decode阶段(每步相同kernel序列，每个kernel很小)。
⚠️ 常见误区：CUDA Graphs可以替代Kernel Fusion：两者解决不同问题。CUDA Graphs消除kernel启动开销(CPU→GPU通信)；Kernel Fusion消除HBM读写开销(中间结果不用写回HBM)。一个推理pipeline可能同时需要两者：先用Kernel Fusion减少kernel数量和HBM访问，再用CUDA Graphs消除剩余kernel的启动开销。
🔬 深入探讨：CUDA Graphs的内存管理

CUDA Graphs录制后，kernel的内存地址被固化在图中。这意味着：
① 不能用PyTorch的动态tensor(每次forward分配新tensor，地址变了对不上)
② 必须预分配固定内存池
③ vLLM的做法：预先分配KV Cache buffer池，Graph录制时使用固定地址
PyTorch提供torch.cuda.CUDAGraph()和torch.cuda.graph()上下文管理器自动处理。但注意：不要在graph中使用python控制流或调用cudaMalloc。
🔬 深入探讨：Occupancy的精确计算

以H100为例计算GEMM kernel的Occupancy：
• 每SM最大: 65536寄存器, 228KB SMem, 2048线程(64 Warp)
• 假设每线程128寄存器, 每Block 48KB SMem, 每Block 256线程(8 Warp)

寄存器限制: 65536 ÷ 128 = 512线程 → 2个Block → 16 Warp
SMem限制: 228KB ÷ 48KB = 4个Block → 32 Warp
瓶颈是寄存器: 16 ÷ 48 = 33% Occupancy

注意: Occupancy不是越高越好！有时50% Occupancy + 大tile > 100% Occupancy + 小tile，因为大tile = 更好的数据复用 = 更少HBM访问。
