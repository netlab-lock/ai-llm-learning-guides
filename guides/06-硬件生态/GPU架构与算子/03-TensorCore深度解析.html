一、背景与动机

一、背景与动机

ng>矩阵乘法：Attention 的 QK^T、FFN 的线性层、Embedding 的查表，本质都是矩阵运算。NVIDIA 从 Volta(2017) 开始在 GPU 中加入 Tensor Core——专用的矩阵乘法硬件单元，每个时钟周期可以完成一个小型矩阵乘累加(D=A×B+C)。

Tensor Core 的出现让 GPU 的矩阵运算性能提升了 10-100 倍，是 LLM 能够高效训练的关键硬件基础。

二、名字由来

二、名字由来

Tensor Core：Tensor(张量) + Core(核心)。张量是多维数组的推广，矩阵是2维张量。Tensor Core 专门做张量(矩阵)乘法。

MMA：Matrix Multiply Accumulate(矩阵乘累加)。Tensor Core 执行的基本指令。

WGMMA：Warp Group MMA(线程束组矩阵乘累加)。Hopper 新增的异步 MMA 指令，多个 Warp 协作完成更大的矩阵乘法。

三、核心原理

3.1 Tensor Core 工作原理

Tensor Core 的基本操作: D = A × B + C A (M×K) B (K×N) C (M×N) D (M×N) ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ │ │ × │ │ + │ │ = │ │ └─────┘ └─────┘ └─────┘ └─────┘ 最小操作尺寸(1st Gen TC, Volta): M=4, N=4, K=4 (FP16 输入, FP32 累加) 每个 Tensor Core 每时钟周期: 完成 4×4×4 = 64 次乘累加(FMA) 一个 SM 有 4 个 Tensor Core: 每时钟 4 × 64 = 256 次 FMA H100 有 132 个 SM, 时钟 1.83 GHz: 132 × 4 × 256 × 1.83G = 989 TFLOPS (FP16)

3.3 MMA 指令详解

MMA (Matrix Multiply Accumulate) 指令: PTX 指令: mma.sync.aligned.m16n8k16.row.col.f16.f16.f16.f16 mma: 矩阵乘累加 sync: 同步(所有线程完成后才继续) m16n8k16: 矩阵尺寸 M=16, N=8, K=16 row.col: A矩阵行主序, B矩阵列主序 f16.f16.f16.f16: A类型.B类型.C类型.D类型 一个 Warp(32线程) 协作完成 16×8×16 的矩阵乘法: A: 16×16 (每个线程持有 2 个元素) B: 16×8 (每个线程持有 1 个元素) C/D: 16×8 (每个线程持有 1 个元素) 每个线程做的事情: 1. 从寄存器读取自己负责的 A, B 元素 2. 计算部分乘积 3. 累加到 C 4. 写入 D

3.4 WGMMA (Hopper 新增)

WGMMA: Warp Group MMA (异步矩阵乘累加) 传统 MMA: 单个 Warp(32线程) 完成一个小矩阵乘法 需要同步等待结果 WGMMA (Hopper): 4个 Warp(128线程) 组成一个 Warp Group 协作完成更大的矩阵乘法(如 64×128×16) 异步执行: 发射指令后可以做其他事情 优势: • 更大的矩阵块 → 更好的数据复用 • 异步执行 → 隐藏内存延迟 • 与 TMA 配合 → 数据加载和计算重叠

3.5 Tensor Core 利用率

TC 利用率 = 实际 TFLOPS / 峰值 TFLOPS × 100%
目标: > 70% 为良好, > 85% 为优秀

影响利用率的因素：

因素影响优化方法
矩阵尺寸太小→TC空闲增大tile size
数据加载加载慢→TC等待Prefetch, double buffering
数据格式格式转换开销保持FP16/BF16格式
Bank conflict共享内存访问冲突Padding, swizzle

import time
import torch

def profile_inference(model, input_data, warmup=10, iterations=100):
    """推理性能分析：测量延迟、吞吐和显存"""
    # Warmup
    for _ in range(warmup):
        with torch.no_grad():
            model(input_data)
    torch.cuda.synchronize()
    
    # Measure latency
    latencies = []
    for _ in range(iterations):
        start = time.perf_counter()
        with torch.no_grad():
            output = model(input_data)
        torch.cuda.synchronize()
        latencies.append(time.perf_counter() - start)
    
    # Statistics
    avg_ms = sum(latencies) / len(latencies) * 1000
    p50 = sorted(latencies)[len(latencies)//2] * 1000
    p99 = sorted(latencies)[int(len(latencies)*0.99)] * 1000
    mem_mb = torch.cuda.max_memory_allocated() / 1024 / 1024
    
    print(f"Latency: avg={avg_ms:.2f}ms, p50={p50:.2f}ms, p99={p99:.2f}ms")
    print(f"Peak Memory: {mem_mb:.0f}MB")
    print(f"Throughput: {1000/avg_ms:.0f} req/s")
    return {"avg_ms": avg_ms, "p50": p50, "p99": p99, "mem_mb": mem_mb}

    ✅ 参考答案与详解
    1. 核心原理: Tensor Core的D=A×B+C操作中，A、B、C、D的数据类型可以部分不同，但受硬件指令支持限制。以H100第4代Tensor Core为例：A和B通常同为FP16/BF16/FP8/TF32，C和D通常同为FP32或FP16。具体组合由MMA指令编码决定(f16.f16.f32.f32表示A/B为FP16，C/D为FP32)。这种混合精度的意义在于：输入用低精度减少数据加载量，累加器用高精度(FP32)避免多次乘累加的舍入误差累积。Ampere新增TF32(19位：8指数+10尾数+1符号)，输入无需改代码即可用FP32加速
      关键要点: ①A/B类型相同 ②C/D类型可不同于A/B ③累加器精度≥输入精度 ④指令编码指定类型组合
      实际案例: FP16输入+FP32累加是LLM训练标配(C=FP32,D=FP32)
      关联知识: 与MMA指令、混合精度训练和loss scaling相关
    2. 核心原理: BF16(8位指数+7位尾数)与FP16(5位指数+10位尾数)的关键区别在于范围和精度的权衡。LLM训练中梯度和激活值动态范围极大(从10⁻¹⁰到10⁶)，FP16范围仅±65504，容易溢出(Inf/NaN)，需要loss scaling等技巧。BF16范围与FP32相同(±3.4×10³⁸)，永远不会溢出，训练更稳定。精度方面FP16更高(10位尾数vs 7位尾数)，但实验证明LLM训练对精度的敏感度远低于对范围的敏感度——BF16训练的模型精度与FP16相当。因此几乎所有大模型训练(LLaMA、GPT等)默认使用BF16
      关键要点: ①BF16范围=FP32(8位指数) ②FP16精度更高(10位尾数) ③LLM梯度范围大易溢出 ④BF16无需loss scaling
      实际案例: Meta LLaMA系列全部使用BF16训练，无需loss scaling
      关联知识: 与混合精度训练、loss scaling和分布式训练稳定性相关
    3. 核心原理: H100理论FP16 TFLOPS计算：132 SM × 4 TC/SM × 256 FMA/TC(4×4×4=64 operations×4 TC per clock via 4 MMA per TC per clock, but the actual per-TC per-clock throughput is 256 FP16 FMA on Hopper) × 2(乘和累加各一次) × 1.83 GHz = 132 × 4 × 256 × 2 × 1.83G = 989 TFLOPS。注：Hopper每个TC每时钟完成256 FP16 FMA(非64)，因为第4代TC的MMA尺寸更大(m16n8k16中M=16,N=8,K=16→每warp每指令完成16×8×16×2=4096 FMA，4个TC并行)。简化公式：算力强弱=SM数×TC数×每TC每周期FMA数×2(乘加)×频率
      关键要点: ①132 SM × 4 TC/SM = 528 TC ②每TC每时钟256 FMA ③频率1.83 GHz ④总计989 TFLOPS
      实际案例: 989 TFLOPS是理论峰值，实际GEMM kernel约850 TFLOPS(~86%利用率)
      关联知识: 与SM架构、时钟频率和Tensor Core利用率(nsight compute)相关
    ⚠️ 常见误区：Tensor Core不是万能加速器：很多人以为"用了Tensor Core就快了"，但实际上：① 如果矩阵太小(如<16×16)，Tensor Core利用率极低；② 如果数据没有正确对齐到MMA尺寸(m16n8k16)，会有padding浪费；③ 数据从HBM到寄存器的搬运如果跟不上，Tensor Core会饿死。Tensor Core的加速效果取决于整个数据通路的效率。
    ⚠️ 常见误区：BF16和FP16只是精度不同：两者都是16位，但结构完全不同：FP16是5位指数+10位尾数(精度高，范围小±65504)；BF16是8位指数+7位尾数(精度低，范围大±3.4×10³⁸)。在LLM训练中，梯度值经常很大，FP16容易溢出需要loss scaling。BF16的范围和FP32一样大，不需要loss scaling，训练更稳定。这就是为什么大多数LLM训练用BF16而不是FP16。
    🔬 深入探讨：Tensor Core的寄存器文件压力

    Tensor Core的操作数都存储在寄存器中。一个MMA m16n8k16操作需要：A矩阵8个寄存器(每线程)、B矩阵2个、C/D矩阵4个——合计14个寄存器/线程。如果做4-stage流水线的GEMM，需要4组操作数——56个寄存器。加上循环变量、地址计算等，一个GEMM kernel很容易用掉128-256个寄存器。

    H100每SM有65536个32位寄存器。如果每线程用256个，每SM最多256个线程，而最大是2048——Occupancy只有12.5%。这就是为什么寄存器压力是kernel优化的核心挑战。CUTLASS的tile size选择本质上就是在数据复用和寄存器压力之间找平衡。
    MMA m16n8k16 寄存器布局 (每个线程的职责): 一个 Warp(32线程) 协作完成 D[16×8] = A[16×16] × B[16×8] + C[16×8] 线程编号: t0 t1 t2 ... t15 t16 t17 ... t31 ───────────────── ───────────────────── A矩阵: 每线程持有8个FP16元素 (16行×16列 ÷ 32线程) t0持有A[0,0:2]和A[8,0:2] (行0和行8的前2列) → 8个FP16 = 4个寄存器 B矩阵: 每线程持有4个FP16元素 (16行×8列 ÷ 32线程) t0持有B[0,0:2]和B[8,0:2] → 4个FP16 = 2个寄存器 C/D矩阵: 每线程持有4个FP32元素 (16行×8列 ÷ 32线程) t0持有C[0,0:2]和C[8,0:2] → 4个FP32 = 4个寄存器 单次MMA寄存器需求: 4 + 2 + 4 = 10个寄存器/线程 如果做K维度累加: 需要2组A,B → 18个寄存器
    🔬 深入探讨：WGMMA的指令编码与Warp协作

    WGMMA(Warp Group MMA)是Hopper的关键创新。传统MMA由单个Warp(32线程)完成，WGMMA由4个Warp(128线程)协作：

    指令格式：wgmma.mma_async.m64n128k16.f16.f16.f16——4个Warp共同完成64×128×16的矩阵乘法。

    异步执行：WGMMA指令发射后不等结果就继续下一条。需要wgmma.commit_group和wgmma.wait_group显式同步。这允许在等待WGMMA结果的同时用TMA加载下一批数据。

    操作数来源：传统MMA的操作数必须在寄存器中，WGMMA的操作数可以直接来自Shared Memory——减少寄存器压力，提高Occupancy。
    FP8格式位分布范围精度LLM中的用途
    E4M31+4+3±448较高(3位尾数)权重存储、推理前向
    E5M21+5+2±57344较低(2位尾数)梯度(范围更重要)
