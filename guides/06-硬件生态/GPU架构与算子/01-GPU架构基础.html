一、背景与动机

一、背景与动机

需要约 10²⁵ 次浮点运算——如果用 CPU，需要几十年。GPU 将这个时间缩短到几天。为什么？因为 GPU 有数千个计算核心，能同时执行大量并行计算。

理解 GPU 架构是做 LLM 训练和推理优化的基础。不懂硬件，就无法理解为什么 FlashAttention 比标准 Attention 快 4 倍

import torch
import numpy as np

class 一、背景与动机Config:
    """配置类：定义一、背景与动机的关键参数"""

def benchmark(func, iterations=100):
    """性能基准测试"""
    import time
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iterations):
        func()
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iterations
    print(f"Average latency: {elapsed*1000:.2f}ms")
    return elapsed

💡 一句话总结：CPU 优化延迟(一个任务尽快完成)，GPU 优化吞吐(同时做大量任务)。LLM 的矩阵乘法天然适合 GPU 的大规模并行。

二、名字由来

二、名字由来

GPU：Graphics Processing Unit(图形处理单元)。最初为图形渲染设计，后来发现非常适合通用并行计算(GPGPU)。

CUDA：Compute Unified Device Architecture(统一计算设备架构)。NVIDIA 2006年推出的 GPU 通用计算平台。

SM：Streaming Multiprocessor(流式多处理器)。GPU 的基本计算单元，每个 SM 包含多个 CUDA Core 和 Tensor Core。

3.3 CUDA Core vs Tensor Core

通俗类比：CUDA Core 像是工厂里的通用工人——什么活都能干，但一次只做一个零件。Tensor Core 像是专用流水线——只能做矩阵乘法这一种活，但一次能同时处理一整箱零件。LLM 的 Attention 和 FFN 层都是大矩阵乘法，所以 Tensor Core 的吞吐量比 CUDA Core 高出几个数量级。

3.5 Warp 执行模型

Warp: GPU 的基本执行单位 一个 Warp = 32 个线程, 同步执行同一条指令(SIMT) Block (线程块, 如 256 线程) │ ├── Warp 0: Thread 0-31 → 同时执行同一条指令 ├── Warp 1: Thread 32-63 ├── Warp 2: Thread 64-95 └── ... Warp Divergence (线程束分化): if (threadIdx.x < 16) ← Warp 内部分支 A(); ← 前16个线程执行A else B(); ← 后16个线程执行B(串行化!) → 两个分支必须串行执行, 性能减半!

3.6 Roofline 模型

常见误区：很多人以为"买最强的 GPU 就能跑最快"。实际上 LLM 推理的 Decode 阶段是内存带宽瓶颈，不是算力瓶颈。A100（2 TB/s 带宽）和 H100（3.35 TB/s 带宽）在 Decode 阶段的性能差距主要来自带宽差异（1.67x），而不是算力差距（3.1x）。选购 GPU 时应优先考虑 HBM 带宽和容量，而不是 FP16 TFLOPS。

Roofline 模型帮助判断一个 kernel 是计算瓶颈还是内存瓶颈：

算术强度(Arithmetic Intensity) = 计算量(FLOPs) / 数据搬运量(Bytes)
性能上限 = min(峰值算力, 峰值带宽 × 算术强度)
Roofline 模型: 性能 (GFLOPS/s) ↑ │ ╱──────────── 峰值算力 (如 989 TFLOPS) │ ╱ │ ╱ │ ╱ │ ╱ ← 斜率 = 带宽 (如 3.35 TB/s) ├─────────────────────→ 算术强度 (FLOPS/Byte) ↑ 拐点 = 峰值算力/带宽 拐点左侧: 内存瓶颈 (Memory-bound) → 优化方向: 减少数据搬运, 用更快的内存 拐点右侧: 计算瓶颈 (Compute-bound) → 优化方向: 用Tensor Core, 提高计算效率 LLM 推理(Decode阶段): 内存瓶颈 每个token只做一次前向, 计算量小但要读全部参数 LLM 训练: 计算瓶颈 大量矩阵乘法, Tensor Core 利用率是关键

3.7 CUDA 编程模型

深入探讨——GPU 内存层次为什么如此重要：寄存器（~1 cycle）、共享内存（~20-30 cycles）、L2 Cache（~200 cycles）、HBM（~400-600 cycles）之间的延迟差异达到 500 倍。FlashAttention 的核心原理就是利用这个层次差异——将 Attention 计算分块（tiling），每块数据加载到共享内存中反复使用，避免反复访问 HBM。具体来说，标准 Attention 需要 O(N²) 的 HBM 访问（写中间矩阵 S、P），而 FlashAttention 通过 kernel 融合将 HBM 访问降为 O(N²d²/M)，其中 d 是头维度，M 是共享内存大小。
// CUDA 核函数示例: 向量加法
__global__ void vec_add(float* a, float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

GPU内存层次的底层原理与FlashAttention推导

GPU内存层次为什么如此重要？从数学本质看：寄存器延迟~1cycle、Shared Memory~20cycles、HBM~400cycles，差距达400倍。FlashAttention的核心原理就是利用这个层次：将Attention矩阵S=QK^T分块（tiling），每块大小匹配Shared Memory容量M，块内计算在SRAM中完成，避免将O(N²)的中间矩阵写回HBM。推导IO复杂度：标准Attention = O(N²d + N²)，FlashAttention = O(N²d²/M)，当M>>d²时IO大幅减少。工作机制：这正是FlashAttention比标准Attention快2-4倍的底层原因。

通俗类比：GPU 的内存层次就像一个厨房——寄存器是你的手边调料架（伸手就拿到，但只能放几瓶）；共享内存是操作台（放得下当前正在用的食材，同组厨师共享）；HBM 是冰箱（容量大但每次拿东西要走几步）。FlashAttention 就是"把冰箱里的食材一次性拿到操作台上，做完一整道菜再放回去"的策略——减少来回跑冰箱的次数。
  • LLM 训练：数据并行+张量并行+流水线并行
  • LLM 推理：Prefill(计算瓶颈)+Decode(内存瓶颈)
  • 微调：LoRA/QLoRA 在单卡上微调大模型
  • 量化：FP8/INT4 量化减少内存和计算

    深入探讨：从系统设计的角度看，一、背景与动机的本质是在性能、精度和资源三者之间寻找最优平衡点。理解这个三角约束关系，就能推导出大多数技术选择背后的逻辑。
    一、背景与动机的技术原理推导： 从底层硬件角度分析，一、背景与动机的性能瓶颈通常不在计算本身， 而在数据搬运。为什么？推导如下： GPU的计算吞吐（如A100 FP16: 312 TFLOPS）远超内存带宽（2TB/s）， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。 当实际算术强度低于此值时，任务是内存受限的。 数学本质上，优化的核心是提高算术强度—— 通过批处理（共享参数读取）和算子融合（减少中间结果写回）来实现。 这一原理适用于所有GPU加速的推理框架， 是理解一、背景与动机性能特征的底层基础。 
    # 一、背景与动机 性能测试示例
import torch
import time

def benchmark_一、背景与动机(model, input_data, warmup=10, iters=100):
    """一、背景与动机 性能基准测试"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    
    start = time.perf_counter()
    for _ in range(iters):
        output = model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    
    print(f"Average latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} requests/sec")
    return elapsed
    常见误区： 不要以为一、背景与动机的优化可以套用通用方案。不同硬件平台（A100 vs H100 vs 边缘GPU） 的最优配置可能完全不同。为什么？因为每个平台的算力/带宽比不同， 导致屋顶模型的转折点不同——在A100上最优的batch_size在H100上可能不是最优的。
    实用技巧： 在实际工程中应用一、背景与动机时，。 记录关键性能指标（延迟p50/p99、吞吐QPS、显存峰值）， 然后逐步扩大规模到生产配置。遇到性能瓶颈时， 用nsys或torch.profiler定位根因，避免盲目优化。
