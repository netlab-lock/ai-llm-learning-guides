📊 CUDA Graphs

1. 背景与动机

即使经过算子融合，LLM 推理仍然面临一个根本问题：kernel launch overhead。每个 CUDA kernel 的启动需要 CPU 向 GPU 发送指令，这个过程约 5-10μs。一个 Transformer 层可能有 20-30 个 kernel（经过融合后），Decode 阶段逐 token 生成时，launch overhead 累计可达数百微秒。

1.1 Kernel Launch 开销量化

问题量化 (Decode 阶段，batch=1): 一个 Transformer 层的 kernel 数 (融合后): - Q/K/V Linear: 3 kernels (GEMM + bias + activation) - Attention: 1 kernel (FlashAttention) - O Linear: 1 kernel - Residual Add: 1 kernel - FFN Gate/Up: 2 kernels - FFN Down: 1 kernel - Residual Add: 1 kernel - LayerNorm: 2 kernels (2 层) ───────────────── 总计: ~12 kernels/层 × 80 层 = ~960 kernels Launch overhead: 960 × 7μs = 6.7ms 实际计算: ~20ms (H100, 70B 模型) 开销占比: 6.7 / (6.7 + 20) ≈ 25% !!!

CUDA Graphs 的核心思想：预录制一次计算图，之后每次推理直接重放整个 graph，只需一次 launch。

1.2 CUDA Graphs 的执行图类比

CUDA Graphs 的核心思想（类比乐谱演奏） 没有 CUDA Graphs（逐指令指挥）： 指挥家(CPU) → "小提琴拉第一个音" → 等待完成 指挥家(CPU) → "大提琴拉第二个音" → 等待完成 指挥家(CPU) → "钢琴弹第三个音" → 等待完成 ... (每个音符都需要指挥家单独发指令) 问题：指挥家成为瓶颈，乐团大部分时间在等指令 使用 CUDA Graphs（录制后重放）： 第一次：指挥家完整指挥一遍整首曲子 → 录制为乐谱 后续：乐团直接按乐谱演奏 → 指挥家只需喊"开始" 优势：一次指令启动整首曲子，乐团全力演奏 技术映射： 指挥家 = CPU（发送 kernel launch 指令） 乐团 = GPU（执行 kernel） 乐谱 = CUDA Graph（录制的执行计划） 录制 = cudaStreamBeginCapture / EndCapture 演奏 = cudaGraphLaunch（一次 launch 整个图）

1.3 为什么算子融合不够

你可能会问：算子融合 不是已经减少了 kernel 数量吗？为什么还需要 CUDA Graphs？原因有二：

  • 融合有上限：共享内存和寄存器限制了单个 kernel 的融合程度（参见 算子融合-融合限制）
  • 跨 kernel 的 launch 开销：即使每个 kernel 融合得很好，80层 × 12个kernel/层 = 960 次 launch 的累积开销仍然不可忽视

2.3 固定内存的深层原因

为什么 CUDA Graphs 要求固定内存地址？ 正常执行模式： 第1次: kernel_a 在地址 0x1000 读取数据，写入地址 0x2000 第2次: kernel_a 在地址 0x3000 读取数据，写入地址 0x4000 ← 地址可以变 CUDA Graph 录制模式： 录制时: kernel_a 在地址 0x1000 读取，写入 0x2000 ← 记录的是地址 重放时: kernel_a 仍然去 0x1000 读取，写入 0x2000 ← 必须是同一地址 如果重放时地址变了 → 读到错误数据 → 结果错误！ 解决方案（vLLM 的做法）： 1. 分配固定缓冲区: input_buffer = torch.empty(...) 2. 录制时使用 input_buffer 3. 重放前: input_buffer.copy_(real_data) ← 只复制数据，不改地址 4. 重放: graph.replay() ← 内存地址不变，数据已更新
CUDA Graphs 与动态形状的冲突：LLM 推理中最常见的动态维度是 batch size 和序列长度。CUDA Graphs 的"固定形状"限制意味着：每次 batch size 变化，要么使用预录制的 graph（需要 graph 池），要么回退到普通执行模式。在高并发场景下，batch size 频繁变化会导致 graph 命中率下降，收益降低。vLLM 的解决方案是将 Decode 阶段（batch 固定后）用 CUDA Graph，Prefill 阶段（序列长度变化大）不用。

2.4 录制过程的底层机制

# CUDA Graph 录制的底层机制（C++ 视角）
# PyTorch 的 torch.cuda.graph() 是对以下 API 的封装

# 1. 开始录制
cudaStreamBeginCapture(stream, cudaStreamCaptureModeGlobal)

# 2. 提交 kernel（不真正执行，只记录）
my_kernel<<>>(args...)
another_kernel<<>>(args...)

# 3. 结束录制，获得 graph
cudaGraph_t graph
cudaStreamEndCapture(stream, &graph)

# 4. 实例化（生成可执行对象）
cudaGraphExec_t exec
cudaGraphInstantiate(&exec, graph, NULL, NULL, 0)

# 5. 重放（一次 launch）
cudaGraphLaunch(exec, stream)
cudaStreamSynchronize(stream)

3. CUDA Graphs 在 LLM 推理中的应用

3.1 Graph 池 (Graph Pool)

由于 CUDA Graphs 要求固定形状，而 LLM 推理的 batch size 和序列长度会变化，解决方案是维护一个 graph 池：

3.2 vLLM 的 CUDA Graph 实现

vLLM 的 CudaGraphRunner 组件负责管理 CUDA Graphs：

# vLLM 的 CUDA Graph 使用流程
class CudaGraphRunner:
    def __init__(self, model):
        self.graphs = {}      # batch_size → graph
        self.input_buffers = {}
        self.output_buffers = {}
    
    def capture(self, batch_size):
        # 1. 分配固定内存
        self.input_buffers[batch_size] = allocate_fixed(...)
        
        # 2. 录制 graph
        graph = torch.cuda.CUDAGraph()
        with torch.cuda.graph(graph):
            self.output = self.model(self.input_buffers[batch_size])
        
        # 3. 缓存
        self.graphs[batch_size] = graph
    
    def replay(self, batch_size, input_data):
        # 复制新数据到固定缓冲区
        self.input_buffers[batch_size].copy_(input_data)
        # 重放 graph（一次 launch）
        self.graphs[batch_size].replay()
        return self.output

3.3 SGLang 的 CUDA Graph 优化

SGLang 默认启用 CUDA Graphs，并针对 LLM 推理做了特殊优化：

  • RadixAttention：KV Cache 的 radix tree 管理与 CUDA Graph 兼容
  • Jump-forward decoding：确定性 token 的跳过，减少 graph 内的 kernel 数
  • Compressed FSM：结构化输出的 FSM 与 CUDA Graph 集成
vLLM vs SGLang 的 CUDA Graph 策略差异：vLLM 采用"按需录制"策略——首次遇到新 batch size 时才录制 graph，有冷启动开销但节省内存。SGLang 采用"预录制"策略——启动时就录制常用 batch size 的 graph，启动慢但运行时零开销。选择哪种策略取决于你的推理场景：如果是长时运行的服务，预录制更好；如果是按需启动的短任务，按需录制更好。

4. CUDA Graphs + torch.compile

torch.compile 的 reduce-overhead 模式自动启用 CUDA Graphs：

4.1 自动管理 vs 手动管理

# 方法 1: torch.compile 自动管理
model = torch.compile(model, mode="reduce-overhead")
# Inductor 会自动识别可以录制为 CUDA Graph 的子图
# 优点：简单，自动处理 graph break
# 缺点：可能不是最优的 graph 划分

# 方法 2: 手动管理（更精细的控制）
graph = torch.cuda.CUDAGraph()
with torch.cuda.graph(graph):
    static_output = model(static_input)

# 重放
static_input.copy_(real_input)
graph.replay()
# 优点：完全控制录制范围和内存
# 缺点：需要手动处理所有限制

4.2 CUDA Graphs 与算子融合的协同

编译优化的分层架构 ┌─────────────────────────────────────────────────┐ │ Layer 3: CUDA Graphs (消除 launch overhead) │ │ ┌───────────────────────────────────────────┐ │ │ │ Layer 2: 算子融合 (减少 HBM 访问) │ │ │ │ ┌───────────────────────────────────┐ │ │ │ │ │ Layer 1: Kernel 优化 │ │ │ │ │ │ (Tensor Core, 向量化, tiling) │ │ │ │ │ └───────────────────────────────────┘ │ │ │ └───────────────────────────────────────────┘ │ └─────────────────────────────────────────────────┘ 每一层解决不同的问题： Layer 1: 单个 kernel 的计算效率 Layer 2: 减少 kernel 数量（融合小 kernel） Layer 3: 减少 kernel launch 次数（graph 重放） 三层叠加的效果： 原始: 960 kernels × 7μs launch + 大量 HBM 访问 优化后: 1次 graph launch + 融合后少量 HBM 访问

4.3 实战：测量 CUDA Graph 的收益

import torch
import time

# 准备模型和数据
model = MyTransformer().cuda().eval()
input_data = torch.randn(1, 128, 4096, device="cuda")

# 方法 1: 无 CUDA Graph
with torch.no_grad():
    # warmup
    for _ in range(10): model(input_data)
    torch.cuda.synchronize()
    
    start = time.perf_counter()
    for _ in range(100): model(input_data)
    torch.cuda.synchronize()
    eager_time = (time.perf_counter() - start) / 100

# 方法 2: 有 CUDA Graph
graph = torch.cuda.CUDAGraph()
static_input = input_data.clone()
with torch.cuda.graph(graph):
    static_output = model(static_input)

# warmup
for _ in range(10):
    static_input.copy_(input_data)
    graph.replay()
torch.cuda.synchronize()

start = time.perf_counter()
for _ in range(100):
    static_input.copy_(input_data)
    graph.replay()
torch.cuda.synchronize()
graph_time = (time.perf_counter() - start) / 100

print(f"Eager: {eager_time*1000:.2f}ms, CUDA Graph: {graph_time*1000:.2f}ms")
print(f"Speedup: {eager_time/graph_time:.2f}x")

5. 性能收益

5.1 基准测试数据

场景无 CUDA Graphs有 CUDA Graphs提升
70B 模型 Decode (bs=1)~15ms/token~10ms/token1.5x
70B 模型 Decode (bs=32)~45ms/token~35ms/token1.3x
7B 模型 Decode (bs=1)~3ms/token~1.5ms/token2x
经验法则：模型越小、batch 越小，CUDA Graphs 的收益越大（因为 launch overhead 占比更高）。对于 7B 模型的单请求推理，CUDA Graphs 可以带来 2x 加速——这是因为小模型的计算时间短，launch overhead 占比高达 50%。

5.2 何时不该使用 CUDA Graphs

CUDA Graphs 不适合的场景：
  • Prefill 阶段：序列长度变化大，graph 命中率低，录制开销大于收益
  • 动态 shape 频繁变化：graph 池维护成本高，内存占用大
  • 需要 CPU-GPU 同步的操作：如 .item()、条件判断等，必须排除在 graph 外
  • 调试阶段：CUDA Graph 使调试变得困难（无法在 graph 内设断点）

5.3 CUDA Graphs 的内存开销

# CUDA Graph 的内存开销分析
# 每个 graph 需要：
# 1. 固定的输入/输出缓冲区（与模型大小相当）
# 2. Graph 元数据（kernel 参数、依赖关系）
# 3. 中间张量的固定内存

# 以 70B 模型为例：
# 模型参数: ~140GB (FP16)
# 每个 graph 的额外开销: ~几MB（元数据）+ 输入/输出缓冲区
# graph 池（16个 batch size）: ~16 × 几MB ≈ 几十MB
# 总额外开销: 相对于 140GB 模型可以忽略

# 但对于小模型（如 7B），graph 池的相对开销会更大
# 优化：只录制最常用的 batch size（如 1,2,4,8,16,32）
CUDA Graphs 的底层实现：CUDA Graph 并非简单的"录制宏"。在底层，NVIDIA 驱动会将录制的 kernel 序列转换为一种"执行计划"，包含：(1) kernel 之间的依赖图（DAG）；(2) 每个 kernel 的参数快照；(3) 内存地址绑定。重放时，GPU 的 command processor 直接从这个执行计划读取指令，完全绕过 CPU 的驱动层。这就是为什么重放只需一次 launch——GPU 自己按 DAG 顺序执行所有 kernel，CPU 只需要在开始时发一个"执行这个 graph"的命令。这种机制与 XLA 的"全图编译"思想有异曲同工之妙：都是将多个操作打包为一个执行单元，减少 CPU 参与。

6. 思考题

原理与推导： 从原理层面理解，该技术的工作机制基于信息论的基本定理。 设信源熵为 H(X) = -Σ p(x)log p(x)，为什么这很重要？ 因为信息熵给出了无损压缩的理论下界——这也解释了为什么量化不可避免地 会带来信息损失。推导量化误差的上界：对于均匀量化， E[|x-q(x)|²] ≤ Δ²/12，其中Δ为量化步长。 这一公式揭示了精度与效率之间的数学本质关系： 步长越小（精度越高），误差越小，但需要的比特数越多。 从底层实现来看，现代加速器的Tensor Core正是利用这一原理， 通过混合精度计算在保持精度的同时最大化吞吐量。
系统级原理分析： 从系统设计的底层逻辑来看，为什么分层架构是主流选择？ 推导如下：假设系统有 N 个组件，全连接的交互复杂度为 O(N²)， 而分层架构将其降低到 O(N·L)，其中 L 为层数。 这一数学本质上的简化使得系统可扩展性大幅提升。 在实际工作机制中，每一层只与相邻层通信，通过抽象接口解耦。 这类似于TCP/IP网络协议栈的分层设计原理——每层独立优化， 只要接口不变，层内改动不影响整体。从公式角度看， 模块间的耦合度 C = |接口依赖| / |总依赖|， 低耦合意味着 C → 0，这正是高内聚低耦合的数学本质。
