端侧设备内存有限（手机通常 6-12GB 可用），LLM 推理的内存管理至关重要。KV Cache 是 Transformer 推理中最大的内存消耗项，随序列长度线性增长，需要专门的优化手段。

KV Cache 压缩

传统 KV Cache 为每个注意力头存储完整的 Key 和 Value 向量。压缩策略包括：低秩分解（将 KV 向量投影到低维空间）、稀疏化存储（仅保留重要 token 的 KV）、量化存储（将 FP16 压缩为 INT4/INT8）。

# KV Cache 量化示例 (伪代码)
class QuantizedKVCache:
    def __init__(self, n_layers, n_heads, max_len, quant_bits=4):
        self.scale = torch.zeros(n_layers, n_heads, max_len)
        self.zero = torch.zeros(n_layers, n_heads, max_len)
        # INT4 存储，节省 75% 内存
        self.keys = torch.zeros(n_layers, n_heads, max_len,
                                head_dim // 2, dtype=torch.uint8)
        self.values = torch.zeros_like(self.keys)

    def append(self, layer, k, v):
        # 动态量化：逐 token 计算 scale/zero
        s, z = compute_affine(k, bits=4)
        self.keys[layer, :, self.pos] = quantize(k, s, z, bits=4)
        self.scale[layer, :, self.pos] = s
        self.pos += 1
⚠️ 常见误区：KV Cache 量化不会影响生成质量

KV Cache 量化（INT4/INT8）是降低内存的有效手段，但对长上下文场景的影响被严重低估。当序列长度超过 4K tokens 时，INT4 KV Cache 的累积量化误差会导致注意力权重分布偏移，表现为：长距离依赖丢失（"遗忘"早期上下文）、重复生成、逻辑断裂。实际测试中，8K 上下文下 INT4 KV Cache 的 perplexity 通常增加 0.3-0.5，而 INT8 仅增加 0.05-0.1。建议：短对话（<2K）用 INT4，长上下文（>4K）用 INT8，代码/数学任务避免 INT4。

滑动窗口注意力

滑动窗口注意力（Sliding Window Attention, SWA）将注意力范围限制在最近的 W 个 token 内，将内存复杂度从 O(n²) 降低到 O(n×W)。Mistral 系列模型原生支持此特性，在端侧部署中效果显著。

分页注意力（Mobile PagedAttention）

借鉴 vLLM 的 PagedAttention 思想，在移动端实现按页分配 KV Cache 内存，避免碎片化，支持动态序列长度管理。每页通常包含 16-64 个 token 的 KV 数据。

算子融合（Kernel Fusion）

将多个小算子合并为一个大 kernel，减少 GPU 内核启动开销和中间结果的显存读写。常见的融合模式包括：

  • QKV投影融合：将 Query、Key、Value 三个线性层合并为一个矩阵乘法
  • 注意力融合：Flash Attention 模式，将 Q×K^T、Softmax、×V 融合为单个 kernel
  • FFN融合：Gate × Up 融合，减少中间激活值的显存占用
  • LayerNorm + 线性层融合：消除中间 tensor 的读写
⚠️ 常见误区：算子融合越多性能越好

算子融合虽然能减少 GPU 内核启动开销和中间结果读写，但过度融合会适得其反。常见问题：1) 融合后的 kernel 寄存器使用量增加，导致 GPU occupancy（并发 warp 数）下降——当寄存器溢出到 local memory 时，性能可能比不融合更差；2) 某些融合（如 LayerNorm + Linear）在不同 GPU 架构上的最优策略不同——Ampere 上最优的融合方案在 Hopper 上可能不是最优；3) 端侧 GPU（如 Mali/Adreno）的寄存器文件更小，融合度需要更保守。建议使用框架的自动调优功能（如 TVM 的 AutoTVM）而非手动融合。

混合精度推理

在端侧设备上使用 FP16 和 INT8 混合精度，对精度敏感的层（如注意力输出投影、LM Head）保持 FP16，对其他层使用 INT8 量化。

# 混合精度配置策略
precision_config = {
    "embedding": "fp16",
    "q_proj": "int8",      # 注意力投影可用 INT8
    "k_proj": "int8",
    "v_proj": "int8",
    "o_proj": "fp16",      # 输出投影保持精度
    "gate_proj": "int8",   # FFN 门控
    "up_proj": "int8",
    "down_proj": "fp16",   # FFN 输出
    "norm": "fp32",        # LayerNorm 必须全精度
    "lm_head": "fp16"      # 最终输出层
}

持续推理会导致设备温度升高触发 SoC 热保护（降频），严重影响性能。优化策略：

  • 推理间歇期插入冷却时间，避免持续满载
  • 监控温度，高温时切换到更轻量的 INT4 模型
  • 调整推理线程数和 GPU 占用率控制发热

内存优化策略的技术原理与数学推导： 从底层硬件角度分析，内存优化策略的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了内存优化策略的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
