💡 量化建议

对于 3B 以下模型，推荐 Q4_K_M 量化；7B 模型推荐 Q4_K_M 或 AWQ。量化后务必用多个 prompt 测试质量。

⚠️ 常见误区：量化后不需要验证输出质量

很多开发者量化后只看文件大小是否缩小就认为成功了。实际上，不同量化方法对不同模型架构的兼容性差异很大——AWQ 对 Qwen 系列效果好，但对某些自定义架构可能出现精度骤降。量化后必须用至少 3-5 个不同类型的任务（问答、代码生成、中文理解）测试输出质量，关注重复生成、乱码、逻辑错误等量化特有的退化模式。

4.1 编译 llama.cpp for Android

export NDK=/path/to/android-ndk-r26b
cmake -B build-android \
  -DCMAKE_TOOLCHAIN_FILE=$NDK/build/cmake/android.toolchain.cmake \
  -DANDROID_ABI=arm64-v8a \
  -DANDROID_PLATFORM=android-28 \
  -DGGML_VULKAN=ON
cmake --build build-android -j$(nproc)

4.2 集成到 Android 项目

通过 llama.cpp 的 Java/JNI 绑定集成。将编译好的 libllama.so 和 libggml.so 放入 app/src/main/jniLibs/arm64-v8a/，将 GGUF 模型文件放入 assets/。

⚠️ 常见误区：Android 部署只需编译 .so 就能运行

很多开发者以为把 llama.cpp 编译为 Android .so 文件就完成了部署。实际上，端侧部署还面临几个关键挑战：1) 模型文件分发——GGUF 文件通常 2-4GB，不能打包进 APK，需要首次运行时下载或使用 OBB 扩展包；2) 内存限制——Android 对单进程内存有限制（通常为物理内存的 50%），需要通过 mmap 加载模型；3) ABI 兼容性——必须确保 NDK 版本与目标设备的 Android 版本匹配，arm64-v8a 是最低要求。

七、电池影响分析

端侧部署学习指南 · 06 端侧部署实战
一、完整部署流程的技术原理与数学推导： 从底层硬件角度分析，一、完整部署流程的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了一、完整部署流程的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
一、完整部署流程的系统架构分析： 从系统设计的原理出发，一、完整部署流程采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

✅ 参考答案与详解
  1. 核心原理: 端侧部署完整流程遵循"选择模型→量化压缩→GGUF转换→平台集成→性能调优→电池测试→发布"七步链路。核心原理在于：移动GPU算力通常只有桌面GPU的1/10-1/50，但模型大小只减少1/4-1/8，意味着端侧推理的算术强度更低、更偏向内存受限，因此INT4量化对端侧尤为关键——它同时减少模型大小(内存)和计算量(FLOPs)。本章以Qwen2.5-3B为例演示从原始模型到Android/iOS设备上可运行的全过程。性能优化需要从模型(量化)、框架(算子融合)、硬件(后端加速)三个层面协同考虑，用Core ML/NNAPI等硬件加速框架优先测试，如不达标再考虑自定义kernel。
    关键要点: ①七步流程为完整端侧部署标准路径；②端侧更偏向内存受限，INT4量化同时减少内存和计算量；③性能优化需模型+框架+硬件三层协同；④先用Core ML/NNAPI硬件加速，不达标再自定义kernel；⑤测试时注意热节流影响(30秒冷却间隔+3次取中位数)
    实际案例: Qwen2.5-3B 在 iPhone 15 Pro 上使用 Q4_K_M 量化，预填充280 tok/s、生成32 tok/s、内存2.1GB；在 Pixel 8 Pro 上生成22 tok/s、内存2.3GB
    关联知识: 算术强度转折点理论(A100: 156 FLOPs/Byte)解释了为何端侧更需要减少内存访问
  2. 核心原理: 模型量化是端侧部署的第一步压缩操作，将FP16/BF16权重转换为低精度表示(INT4/INT8)。对于3B以下模型推荐Q4_K_M量化，7B模型推荐Q4_K_M或AWQ。AWQ(Activation-aware Weight Quantization)量化使用zero_point量化、分组大小128、4bit位宽配置，通过识别重要权重通道保持精度。量化必须在转换前完成——先用量化工具(AUTO-GPTQ/AutoAWQ)在训练环境完成量化，再转为GGUF格式。量化后务必用至少3-5种不同类型的任务(问答/代码生成/中文理解)测试输出质量，关注重复生成、乱码、逻辑错误等量化特有退化模式——不同量化方法对不同模型架构的兼容性差异很大。
    关键要点: ①3B以下推荐Q4_K_M，7B推荐Q4_K_M或AWQ；②AWQ对Qwen系列效果好；③量化必须在GGUF转换前完成；④必须用3+种任务测试输出质量；⑤关注重复生成/乱码/逻辑错误等退化模式
    实际案例: Qwen2.5-3B 用 AutoAWQ 量化为 INT4，配置 zero_point=True, q_group_size=128, w_bit=4，可保存为本地量化模型供后续GGUF转换
    关联知识: 量化后的模型质量损失因架构而异，某些自定义架构可能出现精度骤降，需逐架构验证
  3. 核心原理: GGUF转换是将HuggingFace格式的模型(llama.cpp专用格式)转换为可在端侧推理引擎中加载的GGUF格式。标准流程为：(llama.cpp编译→)使用convert_hf_to_gguf.py脚本将量化后的模型(如qwen2.5-3b-awq)转换为GGUF文件。也可从FP16直接用llama-quantize量化为目标精度(Q4_K_M等)。转换后用llama-cli验证模型可正常推理。GGUF格式的优势在于：单文件包含权重+分词器+元数据、支持mmap内存映射(不按需加载)、向前兼容。转换过程需要注意：模型文件较大(Q4_K_M约2-4GB)、需要确保磁盘空间充足、转换后的验证步骤不可省略。
    关键要点: ①convert_hf_to_gguf.py标准转换脚本；②也可从FP16直接用llama-quantize；③转换后必须用llama-cli验证推理；④GGUF单文件：权重+分词器+元数据+mmap支持；⑤模型文件2-4GB需确保磁盘空间
    实际案例: qwen2.5-3b-awq → python convert_hf_to_gguf.py → qwen2.5-3b-q4k.gguf → ./llama-cli -m qwen2.5-3b-q4k.gguf -p "你好" -n 64 验证
    关联知识: GGUF转换是端侧部署的核心枢纽——连接上游的量化/蒸馏和下游的平台集成
  4. 核心原理: Android部署需要解决三个核心问题：(1)编译——通过Android NDK使用CMake交叉编译llama.cpp为arm64-v8a架构的.so文件，配合Vulkan后端实现GPU加速(CMake参数-DGGML_VULKAN=ON -DANDROID_ABI=arm64-v8a -DANDROID_PLATFORM=android-28)；(2)集成——将libllama.so和libggml.so放入jniLibs/arm64-v8a/，GGUF模型文件放入assets/，通过Java/JNI绑定调用；(3)分发——GGUF文件2-4GB不能打包进APK，需要首次运行时下载或使用OBB扩展包。还需通过mmap加载模型以满足Android单进程内存限制(通常为物理内存50%)。
    关键要点: ①NDK交叉编译arm64-v8a + Vulkan后端；②.so文件放入jniLibs，模型放assets，JNI调用；③2-4GB模型需OBB或首次下载，不能打包APK；④mmap加载满足内存限制；⑤ABI兼容性：arm64-v8a是最低要求
    实际案例: 在Android Studio中配置NDK r26b，编译生成libllama.so(约50MB)，GGUF模型首次启动时从CDN下载到应用私有目录后mmap加载
    关联知识: Vulkan后端是Android GPU推理首选，利用跨平台图形API支持各类Android GPU
  5. 核心原理: iOS部署利用llama.cpp的Swift Package Manager(SPM)支持和Metal后端，实现原生集成。核心步骤：(1)在Xcode中添加SPM依赖(github.com/ggerganov/llama.cpp的metal分支)；(2)使用Swift API调用：llama_model_load_from_file加载模型、llama_new_context_with_model创建上下文、llama_tokenize分词、llama_decode推理；(3)启用Metal后端利用Apple Silicon GPU加速；(4)使用mmap加载模型减少内存占用；(5)对于iPhone 15 Pro+可尝试Q5_K_S获得更好质量(设备内存充裕时)。电池优化策略包括：使用NPU替代GPU可降低50%功耗(3-5W vs 8-12W)、推理节流避免持续高负载、热管理降频保护设备。
    关键要点: ①SPM依赖：llama.cpp金属分支；②Swift原生API：load/tokenize/decode；③Metal后端利用Apple Silicon统一内存；④mmap加载降低内存峰值；⑤NPU比GPU省电50%(3-5W vs 8-12W)；⑥iPhone 15 Pro+可尝试Q5_K_S
    实际案例: 在iOS App中通过 SPM 集成 llama.cpp，加载 Qwen2.5-3B Q4_K_M GGUF(2.1GB)，iPhone 15 Pro 上生成32 tok/s，空闲功耗0.5W，推理时8-12W
    关联知识: Metal后端利用Apple Silicon统一内存架构实现CPU/GPU零拷贝，与Android Vulkan形成跨平台GPU方案对比
向前串联： 一、完整部署流程的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
学习路径建议： 建议按"概念→原理→实验→调优"四步走。先理解核心概念和术语， 再深入数学原理和工作机制，然后用最小化实验验证理解，最后在真实场景中调优参数。 每步都应记录关键指标（延迟/吞吐/精度），建立数据驱动的优化习惯。

端侧部署的性能优化要点

端侧部署的性能优化需要从模型、框架、硬件三个层面协同考虑。 从底层原理看，移动GPU的算力通常只有桌面GPU的1/10-1/50， 但模型大小只减少了1/4-1/8——这意味着端侧推理的算术强度更低，更偏向内存受限。 为什么INT4量化对端侧如此关键？因为它同时减少了模型大小（内存）和计算量（FLOPs）。

实用技巧： 端侧部署时建议先用Core ML / NNAPI等硬件加速框架测试， 如果性能不达标再考虑自定义kernel。硬件加速框架通常能覆盖80%的常见算子。
实用技巧： 在端侧设备上测量性能时，注意排除热节流（thermal throttling）的影响。 建议在每次测试之间留30秒冷却时间，或在设备散热良好的环境下测试。 连续运行3次取中位数，避免首次cold start的影响。
