def benchmark(func, iterations=100):
    """性能基准测试"""
    import time
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iterations):
        func()
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iterations
    print(f"Average latency: {elapsed*1000:.2f}ms")
    return elapsed
常见误区：很多人认为端侧部署就是"把云端模型搬下来"。实际上，端侧部署需要从模型设计、量化策略、推理引擎、内存管理到用户体验做全链路重新设计。直接搬一个FP16的7B模型到手机上，不仅跑不动，还会导致设备发热严重。

深入探讨：端侧部署最大的挑战不是技术，而是"约束下的权衡"。云端部署时，你可以用最好的GPU、最大的模型、最宽松的延迟要求。但在端侧，你必须在模型质量（更大更好）、推理速度（更小更快）、内存占用（更少更轻）、功耗散热（更低更凉）之间反复权衡。优秀的端侧AI工程师，本质上是一个"约束优化专家"。

七、核心挑战

7.1 内存瓶颈

LLM 权重占用大量内存。7B 模型 FP16 需要 14GB，即使 INT4 量化也需要约 3.5GB，加上 KV Cache 开销，手机内存压力很大。

7B模型内存占用分解（INT4量化） ┌─────────────────────────────────┐ │ 模型权重 (INT4) │ ~3.5 GB │ ← 量化后的主要开销 ├───────────────────┼───────────│ │ KV Cache │ 0.5~2 GB │ ← 随上下文长度增长 │ (seq_len=2048) │ │ ├───────────────────┼───────────│ │ 运行时缓冲区 │ ~0.5 GB │ ← 中间激活值 ├───────────────────┼───────────┤ │ 框架开销 │ ~0.2 GB │ ← 推理引擎本身 ├───────────────────┼───────────┤ │ 总计 │ 4.7~6.2GB│ ← 8GB设备的极限 └─────────────────────────────────┘

7.2 功耗与散热

持续推理会导致设备发热降频。旗舰手机在连续推理30秒后，CPU/GPU频率可能下降30-50%。需要在模型精度、推理频率和功耗之间做平衡。

7.3 算子兼容性

部分高级算子（如 Flash Attention、MoE 路由）在移动端 NPU 上支持不完整，需要算子融合或拆分。

7.4 模型更新与版本管理

端侧模型不像云端可以即时更新。需要设计 OTA（Over-The-Air）更新机制，在用户无感知的情况下推送新模型，同时保证旧版本的兼容性。

常见误区：不要以为在PC上跑通了端侧部署就能直接移植到手机上。PC有充足的内存和散热能力，手机完全不同——内存限制更严格、散热能力差10倍、NPU驱动兼容性也不一样。务必在目标真机上进行完整测试。

⚠️ 注意事项

端侧部署不是简单地把模型缩小——需要从模型选择、量化策略、推理引擎到用户体验全链路优化。建议从简单场景入手（如文本分类、关键词提取），逐步扩展到更复杂的生成任务。

  4. 核心原理: 端侧设备内存有限，严格约束了可部署的模型规模上限。设备类型与模型的匹配关系为：(1)旗舰手机(2024+，8-16GB内存)：可部署1B-8B模型，INT4/INT8量化，速度15-40 tok/s；(2)中端手机(6-8GB)：可部署0.5B-3B模型，INT4量化，速度8-20 tok/s；(3)平板/笔记本(16-32GB)：可部署7B-13B模型，INT4/FP16，速度20-60 tok/s；(4)边缘网关(4-8GB)：可部署0.5B-3B模型，INT4，速度10-30 tok/s；(5)智能手表(1-2GB)：仅能部署<0.5B模型，速度2-8 tok/s；(6)车载系统(8-16GB)：可部署1B-7B模型，速度15-35 tok/s。关键约束因素是INT4量化下每参数约0.5字节，加上KV Cache额外占用1-2GB（处理2048 token上下文时），8GB设备运行7B模型（4GB权重+1.5GB KV Cache+0.7GB运行时≈6.2GB）已接近极限。
    关键要点: (1) 旗舰手机1B-8B，中端0.5B-3B (2) INT4每参数约0.5字节 (3) KV Cache额外1-2GB (4) 8GB设备运行7B已近极限 (5) 内存是端侧部署的第一硬约束
    实际案例: 在8GB手机上部署Qwen2.5-7B INT4模型，需约6.2GB（含KV Cache），剩余内存仅1.8GB给操作系统和其他应用，非常紧张。
    关联知识: 与端侧部署工作流关联，模型选择阶段需根据设备内存确定规模上限，量化压缩阶段进一步减少内存占用。
  5. 核心原理: 完整的端侧部署工作流包含8个阶段，形成闭环迭代：(1)模型选择——评估任务需求（功能/精度/延迟）、匹配模型规模（参数量）、建立质量基线（FP16下的benchmark）；(2)量化压缩——INT4/INT8量化（减少4-8倍体积）、GPTQ/AWQ（优化量化质量）、剪枝蒸馏（进一步压缩）；(3)格式转换——GGUF（CPU友好）、ONNX（跨平台）、CoreML（Apple原生）、TF Lite（Google生态）；(4)推理集成——引擎选型（llama.cpp/MLC-LLM/Core ML等）、API对接、UI适配；(5)测试验证——质量评测（对比基线）、性能测试（延迟/吞吐）、功耗测试（发热/电池）；(6)上线监控——延迟监控、内存追踪、用户反馈收集；(7)持续优化——A/B测试、模型OTA更新、根据反馈迭代。文件强调端侧部署最大的挑战不是技术本身，而是在模型质量、推理速度、内存占用、功耗散热之间反复权衡。
    关键要点: (1) 8阶段全链路闭环 (2) 模型选择→量化→转换→集成→测试→部署→优化 (3) 量化减少4-8倍体积 (4) 多维度权衡（质量/速度/内存/功耗） (5) OTA更新机制
    实际案例: 部署Phi-3-mini到手机端：选择模型(3.8B)→INT4量化→GGUF格式→llama.cpp集成→真机测试(延迟/功耗)→OTA推送更新→根据用户反馈迭代。
    关联知识: 与推理框架选型关联，工作流第4阶段需要从llama.cpp/MLC-LLM/MediaPipe/Core ML等引擎中选择合适的推理框架。
