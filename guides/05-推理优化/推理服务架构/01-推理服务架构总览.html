1. LLM 推理的两个阶段

LLM 推理的两个阶段

💡 通俗类比：LLM推理的两个阶段就像餐厅的"备菜"和"上菜"——Prefill是厨师一次性把所有食材切好(并行处理所有输入token)，需要大量工作台(算力)；Decode是一道一道上菜(逐token生成)，每道菜快但需不断从冰箱(KV Cache)取食材。理解这个区别是设计推理架构的基础。
⚠️ 常见误区：不要用TTFT(首token延迟)来衡量整个推理性能。TTFT只反映Prefill效率，用户体感更依赖TPS(每秒token数)。TTFT=200ms但TPS=100的服务，比TTFT=100ms但TPS=20的服务体验好得多——后续每个token等10ms vs 50ms。

🎯 思考题
  1. 为什么LLM推理需要分为Prefill和Decode两个阶段？从计算模式、资源需求和性能瓶颈三个维度分析两者的本质差异。
  2. TTFT和TPS分别衡量什么？为什么Goodput比Throughput更能反映推理服务的真实质量？
  3. 一体架构和分离架构各自的优缺点是什么？在什么场景下分离架构反而会降低性能？
▶显示答案
✅ 参考答案与详解
  1. 核心原理: LLM推理分为Prefill和Decode两个阶段的根本原因在于两者的计算模式完全不同。Prefill阶段对输入的所有token并行执行矩阵-矩阵乘法(MNM)，生成初始KV Cache，算术强度高达O(seq_len)，属于compute-bound操作，GPU Tensor Core利用率可达70-90%。Decode阶段每步只处理1个新token，执行矩阵-矩阵乘法(MVM)，算术强度为O(1)，属于memory-bound操作，GPU利用率仅10-30%。这种本质差异意味着：Prefill需要高算力而Decode需要高带宽；混合部署会相互干扰；优化策略完全不同（Prefill优化batch效率，Decode优化KV Cache读取）。理解这个两阶段差异是设计推理架构的基础。
    关键要点: ①Prefill是compute-bound，Decode是memory-bound；②算术强度差异决定了GPU利用率差异；③两者的资源需求矛盾是分离架构的根本动机；④A100的算术强度转折点为156 FLOPs/Byte；⑤理解两阶段差异是推理优化的起点。
    实际案例: 以Llama-3-70B为例，处理2000 token的prompt：Prefill阶段需要约800ms（大矩阵乘法，GPU满载），Decode阶段每步约25ms（受限于HBM带宽读取140GB参数）。如果混合部署，一个长prompt的Prefill（800ms）会阻塞所有正在Decode的请求，导致这些用户的TPOT从25ms飙升到825ms——用户会明显感觉到"打字机效果"卡顿。
  2. 核心原理: TTFT（Time to First Token）衡量从请求到达到第一个token输出的延迟，反映Prefill阶段的效率，典型值100ms-2s。TPS（Tokens Per Second）衡量用户每秒接收到的token数，反映Decode阶段的生成速度，典型值30-100 tok/s。Goodput（有效吞吐量）是在SLA约束下的有效服务吞吐量——只有满足TTFT和TPS双重SLA的请求才算"有效"。Goodput比Throughput更真实的原因：推理请求长度差异巨大（从短对话到长文档），部分长时间请求可能超时但仍占用资源；超时的请求对用户无价值但消耗了GPU时间；Goodput直接关联用户体验和收入。一个系统Throughput为5000 tok/s但P99延迟为12秒（SLA要求10秒），Goodput可能仅为3000 tok/s。
    关键要点: ①TTFT对应Prefill，TPS对应Decode；②Goodput = 满足SLA的请求速率；③高Throughput不等于高Goodput；④SLA约束决定了有效服务边界；⑤Goodput是商业决策的核心依据。
    实际案例: 某翻译API的Throughput为5000 tok/s，但P99延迟为12秒（SLA要求10秒），Goodput仅为3000 tok/s。虽然系统每秒处理5000个token，但其中40%的请求超时，用户实际体验差。优化后P99降至8秒，Goodput提升到4800 tok/s。
  3. 核心原理: 一体架构将Prefill和Decode部署在同一GPU上，优点是简单、无网络传输开销、适合小规模部署。缺点是阶段干扰严重（Prefill阻塞Decode）、GPU利用率低（阶段切换导致空闲）、无法独立优化。分离架构将Prefill和Decode部署在不同GPU上，优点是消除阶段干扰、可动态调配资源比例、各自独立优化。缺点是引入KV Cache网络传输开销（2.6GB需52ms@IB NDR）、系统复杂度高。分离架构在以下场景反而更慢：短prompt（<100 token）时Prefill计算时间（~5ms）小于KV Cache传输时间（~20ms）；单节点部署时NVLink传输虽快但分离增加了调度复杂度；网络条件差（100GbE）时传输延迟过高。
    关键要点: ①一体架构简单但低效；②分离架构高效但复杂；③分离收益严格依赖于prompt长度；④KV Cache传输是分离架构的核心开销；⑤适用场景：长prompt(>1K)+多节点+SLA严格。
    实际案例: DistServe论文显示：在代码补全场景（prompt 2-8K tokens），分离架构P99 TGBT比一体改善3.2倍。但在短对话场景（prompt<100 tokens），KV Cache传输固定开销（~20ms）超过Prefill计算时间（~5ms），分离比一体慢50%。

import time
import random

def simulate_inference(prompt_tokens, output_tokens, gpu_type="A100"):
    """模拟LLM推理过程"""
    # 模拟Prefill: compute-bound, 与prompt长度成正比
    prefill_time = prompt_tokens * 0.04  # 0.04ms per token
    # 模拟Decode: memory-bound, 每步固定时间
    decode_time_per_token = 10  # 10ms per token
    decode_time = output_tokens * decode_time_per_token
    return prefill_time, decode_time

def simulate_disaggregated(prompt_tokens, output_tokens, network_bandwidth_gbps=400):
    """模拟分离架构"""
    prefill_time = prompt_tokens * 0.04
    # KV Cache传输: ~2.56MB/token * prompt_tokens
    kv_cache_size_mb = 2.56 * prompt_tokens / 1024  # GB
    transfer_time = (kv_cache_size_mb * 8 / network_bandwidth_gbps) * 1000  # ms
    decode_time = output_tokens * 10
    return prefill_time, transfer_time, decode_time

解题思路：通过简化的数学模型模拟两种架构的性能。一体架构的总时间 = Prefill时间 + Decode时间。分离架构的总时间 = Prefill时间 + KV Cache传输时间 + Decode时间。关键发现是存在一个"盈亏平衡点"——当prompt足够长时，分离架构消除阶段干扰的收益超过传输开销。

LLM 推理的两个阶段的技术原理与数学推导： 从底层硬件角度分析，LLM 推理的两个阶段的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了LLM 推理的两个阶段的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
LLM 推理的两个阶段的系统架构分析： 从系统设计的原理出发，LLM 推理的两个阶段采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。 
# LLM 推理的两个阶段 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

✅ 参考答案与详解
  1. 核心原理: LLM推理的两个阶段指的是Prefill（预填充）和Decode（解码）。Prefill阶段将输入的prompt所有token并行处理，通过一次大矩阵乘法生成初始KV Cache，属于计算密集型（compute-bound），其延迟指标为TTFT（首Token时间）。Decode阶段逐token自回归生成，每步只处理1个新Token但需读取全部历史KV Cache，属于内存密集型（memory-bound），其指标为TPS（每秒Token数）。两者的根本区别在于算术强度不同——Prefill执行矩阵-矩阵乘法（MNM），算术强度高，GPU Tensor Core满载；Decode执行矩阵-向量乘法（MVM），算术强度低，受限于HBM带宽。
    关键要点: ①Prefill是compute-bound，Decode是memory-bound；②Prefill延迟取决于输入长度，Decode每步几乎恒定；③批处理对Prefill效率提升明显，对Decode则受KV Cache内存限制；④GPU利用率Prefill远高于Decode；⑤两个阶段的资源需求截然不同是分离架构的动机。
    实际案例: 以聊天机器人为例，用户输入"请用2000字解释Transformer架构"（约200 token Prompt），Prefill阶段需要在数百ms内完成所有200个token的并行投影和注意力计算；进入Decode后，模型每步生成1个新token（如"Transformer"→"是"→"一种"…），每步约10-30ms，用户感知到的是流畅的逐字输出。若Prefill耗时过长，用户等待首字时间过长（TTFT恶化）；若Decode TPS过低，用户感觉"卡顿"。
    关联知识: 这两个阶段的本质差异直接引出后续的Chunked Prefill（通过将Prefill分块避免长Prompt阻塞Decode）、Disaggregated Serving（将Prefill和Decode部署在不同GPU上以各自优化）、以及KV Cache优化（Decode阶段性能高度依赖KV Cache读写效率）等技术方向。
  2. 核心原理: 推理服务的核心指标构成了一套完整的性能评估体系。TTFT（Time to First Token）衡量从请求到达到第一个token输出的延迟，反映系统响应速度，通常为100ms-2s。TPS（Tokens Per Second）衡量用户每秒接收到的token数，决定"打字机效果"的流畅度，典型值为30-100 tok/s。吞吐量（Throughput）是系统每秒处理的总token数，为1K-10K tok/s。Goodput（有效吞吐量）是最关键的隐藏指标——在SLA约束下的有效吞吐量，因为超时的请求即使最终被处理也不算有效服务。GPU Util反映算力利用率，$/1M tokens衡量成本效率。
    关键要点: ①TTFT和TPS分别对应两个阶段（Prefill和Decode）；②Goodput比Throughput更真实反映服务质量；③SLA约束决定了有效服务边界；④吞吐量不等于用户体验（40倍吞吐差异下体验可能相近）；⑤成本效率是最终商业决策依据。
    实际案例: 考虑一个在线翻译服务的场景：用户提交10K token的英文技术文档翻译为中文。系统A：TTFT=500ms、TPS=50，但P99延迟15秒（SLA为10秒），Goodput可能只有理论吞吐的60%。系统B：TTFT=800ms、TPS=40，但P99延迟8秒，Goodput接近100%。虽然系统A的TPS更高，但用户实际感知上系统B更稳定可靠。
    关联知识: Goodput的概念源自网络工程的"有效吞吐量"，在推理服务中特别重要，因为LLM请求的长度分布差异巨大（从短对话到长文档），导致部分长时间请求占用大量资源但可能超时，反而拉低整体服务效率。本指南后续的成本优化章节将基于这些指标建立量化分析模型。
  3. 核心原理: 推理架构的演进是一个逐步解决瓶颈的过程，每个阶段都针对前一阶段的核心痛点。2022年单GPU推理时代，模型放一块GPU逐请求处理，简单但GPU利用率极低。2023年vLLM引入Continuous Batching（连续批处理）和PagedAttention，实现了动态batch——不同长度的请求可以动态加入和退出batch，解决了静态batch的GPU空闲浪费问题。2024年以Splitwise（Microsoft）、DistServe（PKU）、Mooncake（Moonshot）为代表的分离架构将Prefill和Decode部署到不同GPU，消除了阶段间的干扰。2025年进一步发展为KV Cache感知路由（LMCache、llm-d），实现分布式KV Cache的智能调度。2026年正朝着Serverless+多模型共享GPU+语义缓存的方向发展。
    关键要点: ①演进驱动力是持续提升GPU利用率；②从静态batch到动态batch解决了请求长度差异问题；③从一体到分离解决了阶段干扰问题；④KV Cache成为架构设计的核心对象；⑤Serverless是终极形态——按需扩缩容。
    实际案例: 以Moonshot AI的Mooncake为例，它采用以KV Cache为中心的分离架构：当用户请求到达时，调度器优先将请求路由到已有该请求前缀KV Cache的Decode GPU，避免了Prefill重复计算。实验显示在长Prompt（>1K token）场景下，Mooncake的TTFT比一体架构降低37%。
    关联知识: 这条演进路线与训练系统的分布式演化（从数据并行→模型并行→流水线并行→专家并行）形成有趣的对照——推理系统的核心对象从"计算"转向"KV Cache"，反映了推理工作负载的本质是内存密集型的自回归生成。
  4. 核心原理: 一体架构与分离架构的核心差异在于是否将Prefill和Decode阶段部署在同一GPU上。一体架构中，Prefill和Decode共享同一GPU，Prefill的compute-bound特性和Decode的memory-bound特性相互干扰——正在Prefill时长达数秒的计算会阻塞所有Decode请求，导致已在线用户的TPOT飙升。分离架构将Prefill部署在高算力GPU（A100）、Decode部署在高带宽GPU（H100），各自独立优化。分离架构的核心收益包括：GPU利用率提升（各阶段专用资源）、资源可动态调配（如聊天场景多Decode少Prefill，则GPU比例为1:4）、消除阶段干扰（在线用户不受新用户Prefill影响）。但分离架构也引入了KV Cache跨网络传输的开销（约20ms/256MB，100Gbps网络）。
    关键要点: ①一体架构简单但低效（阶段切换浪费）；②分离架构高效但复杂；③分离架构的价值严格依赖于Prompt长度；④KV Cache传输是分离架构的最大开销；⑤适用场景：长Prompt（>1K token）+多节点+SLA严格。
    实际案例: 以DistServe（PKU 2024）为例，它为每个请求分配独立的Prefill和Decode实例。在代码补全场景中（Prompt通常2-8K tokens），分离架构的P99 TGBT比一体架构改善3.2倍。但在短对话场景（Prompt<100 tokens）下，KV Cache传输的固定开销（~20ms）可能超过Prefill计算时间（~5ms），分离反而比一体慢50%。
    关联知识: 分离架构的引入直接依赖于KV Cache的可传输性——正是因为KV Cache是独立于GPU的纯数据结构，才可能在GPU间传输。这与Ring Attention中的KV环形传递有本质区别：Ring Attention是在Prefill阶段将KV分布到多GPU并行计算，而分离架构是在Prefill和Decode之间传输完整的KV Cache。
  5. 核心原理: 本指南的学习路线按照"基础原理→核心技术→系统实现→生产实战"的认知递进逻辑设计。首先学习Disaggregated Prefill-Decode（分离架构），这是现代推理系统的架构基础，理解Prefill/Decode的本质差异后，自然引出为什么需要以及如何实现分离部署。其次学习KV Cache调度与路由，因为分离架构的核心挑战是如何高效管理分布式KV Cache——如何在多节点间存储、检索、传输KV Cache。然后学习多模型与Serverless，将推理优化从单模型扩展到多模型共享资源的更复杂场景，实现成本最优。接着学习成本优化，基于前述技术指标建立量化成本模型。最后学习监控与可观测性，将优化从实验室延伸至生产环境。
    关键要点: ①从架构到优化的递进逻辑；②每章以前章为基础（KV Cache知识贯穿全路线）；③最终目标是生产可用的推理系统；④覆盖技术栈完整（架构、调度、共享、成本、监控）；⑤强调动手实践和量化分析。
    实际案例: 一个典型的学习路径是：先通过本章节理解Prefill/Decode差异→阅读Disaggregated Prefill-Decode章节了解Splitwise/DistServe实现→学习KV Cache调度理解如何优化分布式缓存→在实际部署多模型（如同时部署Llama-3和Qwen2.5）时应用Serverless推理→通过成本分析选择最优部署方案→建立完整的监控体系。
    关联知识: 本指南的学习路线与业内主流推理框架的演化高度对应——vLLM（连续批处理）→ TensorRT-LLM（分离部署）→ SGLang（KV Cache路由）→ KServe（多模型Serverless），反映了技术栈从单一到完整的自然演进。
