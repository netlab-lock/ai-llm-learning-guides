1. 背景与动机

背景与动机

💡 通俗类比：KV Cache感知路由就像快递站的智能分拣——传统Round Robin路由相当于不管包裹去哪，轮流扔到不同车上(每个GPU都要重新Prefill)。Cache感知路由相当于先看包裹地址(hash)，如果某辆车已经装了同路线的包裹(KV Cache命中)，就直接放到那辆车上，省去重新分拣的步骤。
⚠️ 常见误区：Prefix Caching不是万能的。如果每个请求的prompt都完全不同(无共享前缀)，Prefix Caching的命中率为零，反而增加了hash计算和路由查询的开销。它真正有效的场景是：共享system prompt的多轮对话、使用相同few-shot示例的批量请求、RAG场景中频繁引用相同文档片段。

2. KV Cache 感知路由

⚠️ 常见误区：Cache Affinity调度可能导致负载不均衡。如果大量请求都命中同一GPU的缓存(比如都共享同一个热门system prompt)，该GPU会过载而其他GPU空闲。解决方案是使用Soft Affinity(优先缓存命中但允许负载均衡)，并在路由表中维护GPU负载信息。

2.1 Prefix Caching 路由

将包含相同前缀的请求路由到同一 GPU，使 KV Cache 可以复用：

  • 精确匹配：token 序列完全相同的前缀
  • 前缀匹配：共享 system prompt 的请求
  • 实现方式：维护 token hash → GPU 映射表

2.2 LMCache (2025)

LMCache 是一个分布式 KV Cache 存储系统，与 vLLM 集成：

  • 分布式存储：KV Cache 存储在 CPU 内存或远程存储中
  • Cache 感知路由：根据缓存命中率路由请求
  • 3-10x 延迟节省：在多轮对话和 RAG 场景中

2.3 llm-d (K8s 原生)

llm-d 是 Kubernetes 原生的 LLM 推理调度器：

  • Inference Gateway：K8s 的 AI-aware 负载均衡
  • KV Cache 感知：将请求路由到有缓存的 pod
  • 集成 LMCache：分布式缓存 + 智能路由

Cache 预热是指在服务启动时预加载常用的 KV Cache：

  • 系统 prompt 预热：提前计算 system prompt 的 KV Cache
  • 热门查询预热：对高频查询提前 Prefill
  • 文档预热：RAG 场景中预计算常用文档的 KV Cache
背景与动机的技术原理与数学推导： 从底层硬件角度分析，背景与动机的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了背景与动机的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
背景与动机的系统架构分析： 从系统设计的原理出发，背景与动机采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

背景与动机的性能优化策略

优化背景与动机性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

# 背景与动机 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed
