在一体架构中，Prefill 和 Decode 共享同一组 GPU。这导致一个根本问题：阶段干扰 (Phase Interference)。

阶段干扰示意: 时间 → GPU 0: [====Prefill====][--Decode--][====Prefill====][--Decode--] GPU 1: [--Decode--][====Prefill====][--Decode--][====Prefill====] ↑ Prefill 抢占 Decode 的 GPU 资源 ↑ Decode 的低效使用拖累 Prefill 问题: - Prefill 突发: 长 prompt 的 Prefill 会阻塞正在 Decode 的请求 - Decode 空闲: Decode 阶段 GPU 利用率低，但占着显存 - 资源竞争: Prefill (compute-bound) 和 Decode (memory-bound) 争抢不同资源

2. 分离架构的原理

分离架构的原理

⚠️ 常见误区：分离架构的核心成本不是GPU，而是KV Cache传输。以Llama-3-70B为例，seq_len=1024时KV Cache约2.6GB，通过InfiniBand NDR(400Gb/s)传输需要约52ms。如果Prefill计算本身只需30ms，传输就成了瓶颈。务必评估你的网络条件再决定是否采用分离架构。

3. 三大分离架构论文

💡 通俗类比：三大分离架构论文的关系就像同一道菜的三种做法——Splitwise(微软)是"分灶做饭"(硬件级分离)，DistServe(北大)是"按客人满意度优化"(Goodput优化)，Mooncake(月之暗面)是"把食材放公共冰箱"(KV Cache为中心)。三者殊途同归，都证明了分离架构的价值。

3.1 Splitwise (Microsoft, 2024)

核心思想：将 LLM 推理集群分为 Prefill 机器和 Decode 机器，通过高速网络传输 KV Cache。

  • 机器配置：Prefill 机器用高算力 GPU（如 H100 SXM），Decode 机器用高带宽 GPU
  • KV Cache 传输：通过 RDMA（远程直接内存访问）传输，延迟约 1-5ms
  • 调度器：根据请求的输入/输出长度预估，路由到合适的机器

3.2 DistServe (Peking University, OSDI 2024)

核心思想：以 Goodput（满足 SLA 的有效吞吐）为优化目标，而非单纯的吞吐量。

  • Goodput 定义：在 TTFT SLA 和 TPS SLA 约束下，系统能稳定服务的请求速率
  • 核心发现：分离架构比一体架构提升最高 2x Goodput
  • 调度策略：根据 SLA 约束动态分配 Prefill/Decode 资源

3.3 Mooncake (Moonshot AI, 2024)

核心思想：以 KV Cache 为中心的架构，将 KV Cache 视为一等公民。

  • KV Cache Pool：分布式 KV Cache 存储池，跨 GPU/节点共享
  • KV Cache 路由：根据缓存命中率路由请求，减少重复 Prefill
  • KV Cache 传输优化：异步传输、流水线传输、增量传输

4.1 传输量分析

KV Cache 大小估算 (Llama-3-70B, FP16): 每层 KV Cache: 2 × hidden_dim × seq_len × 2 bytes = 2 × 8192 × seq_len × 2 bytes 80 层总计: 80 × 2 × 8192 × seq_len × 2 bytes = 2.56 MB × seq_len seq_len=128: ~328 MB seq_len=1024: ~2.6 GB seq_len=8192: ~21 GB seq_len=128K: ~328 GB (!!!)

何时使用分离架构？ ├─ 请求特征 │ ├─ 长 prompt (> 1K tokens) → 分离有收益 │ └─ 短 prompt (< 100 tokens) → 一体架构更优 │ ├─ 规模 │ ├─ 单节点 (8 GPU) → 一体架构足够 │ └─ 多节点 (32+ GPU) → 分离架构收益明显 │ ├─ SLA 要求 │ ├─ TTFT < 200ms + TPS > 50 → 分离 + 专用 Prefill GPU │ └─ 宽松 SLA → 一体架构即可 │ └─ 预算 ├─ 充足 → 分离 + 高速互联 (IB NDR) └─ 有限 → 一体架构 + CUDA Graphs
背景与动机的技术原理与数学推导： 从底层硬件角度分析，背景与动机的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了背景与动机的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
背景与动机的系统架构分析： 从系统设计的原理出发，背景与动机采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。 
# 背景与动机 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed
