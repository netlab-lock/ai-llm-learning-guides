⚠️ 常见误区：并行度越高性能越好

一个常见的错误认知是"卡越多越快"。实际上，每增加一种并行维度都会引入额外的通信开销：TP 需要每层 AllReduce（NVLink 带宽决定上限），PP 有 pipeline bubble（空泡率可达 20-30%），DP 需要梯度同步。当通信开销超过计算收益时，增加并行度反而降低性能。正确做法是先分析模型的算术强度（FLOPs/Byte），判断是计算受限还是带宽受限，再选择并行策略。例如 7B 模型在单卡 A100 上已经是带宽受限，增加 TP 只会增加通信而不会加速。

🤔 我该用哪种并行？
  1. 模型能放进单卡吗？→ 能：用 DP（多副本）或单卡推理
  2. 单卡放不下，同机多卡能放下吗？→ 能：用 TP（同机 NVLink 最快）
  3. 同机放不下，需要跨机？→ TP(同机) + PP(跨机)
  4. 是 MoE 模型？→ 加入 EP
  5. 上下文超长（128K+）？→ 加入 CP
  6. 训练场景？→ 用 ZeRO + DP
⚠️ 常见误区：TP 和 SP 可以随意组合

很多人以为张量并行(TP)和序列并行(SP)可以自由组合来同时解决模型大和序列长的问题。实际上，Megatron 的 SP-1 只处理 TP 切分不到的区域（如 LayerNorm、Dropout），它依赖于 TP 的存在。而 Ulysses SP-2 是独立的序列维度切分，与 TP 正交但需要额外的 AllToAll 通信。错误地将 SP-1 用在非 TP 场景会导致运行时错误，将 SP-2 与 TP 组合时需要确保 head 数能被并行度整除。

一、为什么需要并行？的技术原理与数学推导： 从底层硬件角度分析，一、为什么需要并行？的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了一、为什么需要并行？的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
一、为什么需要并行？的系统架构分析： 从系统设计的原理出发，一、为什么需要并行？采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。 
# 一、为什么需要并行？ - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed
