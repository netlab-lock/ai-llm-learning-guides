2.1 问题：TP 的"盲区"

TP 的盲区：LayerNorm 和 Dropout TP 只能切分"大矩阵乘法"（Attention 投影、MLP） 但 LayerNorm 和 Dropout 不是矩阵乘法，无法用 TP 切分！ 传统 TP: [Attention (TP)] → AllReduce → [LayerNorm] → [MLP (TP)] → AllReduce ↑ 这里的 LayerNorm 每卡都有完整序列 浪费内存！

2.2 SP-1 的解决方案

SP-1: 对 LayerNorm/Dropout 按序列维度切分 传统 TP: Attention(TP) → AllReduce → LayerNorm(完整序列) → MLP(TP) → AllReduce ↑ 每卡有完整序列 ↑ 每卡有完整序列 SP-1: Attention(TP) → ReduceScatter → LayerNorm(序列的1/N) → AllGather → MLP(TP) → ReduceScatter → ... ↑ 每卡只有 1/N 序列 ↑ 每卡只有 1/N 序列 关键变化: AllReduce = ReduceScatter + AllGather 拆开后，中间的 LayerNorm 只需处理 1/N 的序列！
SP-1 的好处：
  • LayerNorm 的内存和计算量 ÷ N（N = TP 度数）
  • Dropout 的 mask 内存也 ÷ N
  • 总通信量不变（AllReduce = ReduceScatter + AllGather）
  • 是 Megatron-LM 的标准配置

3.1 问题：超长序列的 Attention 计算

问题：128K tokens 的 Attention Attention = softmax(Q × K^T) × V Q: [128K × d] K: [128K × d] V: [128K × d] Q × K^T: [128K × 128K] ← 这个矩阵太大了！ 128K × 128K × 2 bytes = 32 GB（比很多 GPU 显存还大） 单卡算不下 → 需要把序列切分到多卡！

3.2 DeepSpeed Ulysses 方案

Ulysses: 用 AllToAll 交换 QKV Step 1: 每卡持有序列的 1/N GPU 0: [token 0-31K] → Q0, K0, V0 GPU 1: [token 32K-63K] → Q1, K1, V1 GPU 2: [token 64K-95K] → Q2, K2, V2 GPU 3: [token 96K-127K] → Q3, K3, V3 Step 2: AllToAll 交换 → 每卡获得所有 token 的部分 QKV GPU 0: Q[所有token, head 0-1], K[所有token, head 0-1], V[所有token, head 0-1] GPU 1: Q[所有token, head 2-3], K[所有token, head 2-3], V[所有token, head 2-3] ... Step 3: 每卡独立计算自己负责的 head 的 Attention GPU 0: Attention_head_0,1 = softmax(Q_01 × K_01^T) × V_01 ... Step 4: AllToAll 合并结果 每卡只需存 1/N 的 KV Cache！

3.3 Ring Attention 方案

Ring Attention: 环形传递 KV 每卡持有序列的一段，逐块传递 KV: 时间步1: GPU 0 计算 [Q_local × K_0^T]，同时把 K_0,V_0 传给 GPU 1 时间步2: GPU 0 计算 [Q_local × K_1^T]，同时把 K_1,V_1 传给 GPU 2 ... 关键: 通信和计算重叠！ 计算当前块的 Attention 时，同时传递下一块的 KV → 理论上可以完全隐藏通信延迟 适合极长序列（1M+ tokens）

一、⚠️ 先搞清楚：两种不同的 SP的技术原理与数学推导： 从底层硬件角度分析，一、⚠️ 先搞清楚：两种不同的 SP的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了一、⚠️ 先搞清楚：两种不同的 SP的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
一、⚠️ 先搞清楚：两种不同的 SP的系统架构分析： 从系统设计的原理出发，一、⚠️ 先搞清楚：两种不同的 SP采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

一、⚠️ 先搞清楚：两种不同的 SP的性能优化策略

优化一、⚠️ 先搞清楚：两种不同的 SP性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

一、⚠️ 先搞清楚：两种不同的 SP的工程实践与最佳实践

在生产环境中部署一、⚠️ 先搞清楚：两种不同的 SP时，需要考虑以下工作机制： 资源规划（GPU显存预算公式：M_total = M_model + M_kv×B + M_overhead， B为并发数）、容错机制（健康检查、自动重启、灰度发布）、 监控告警（延迟p99、错误率、GPU利用率）。 为什么这些因素如此关键？因为推理服务是持续运行的， 任何微小的效率问题在高并发下都会被放大—— 从数学本质看，总成本 = 单次成本 × QPS × 时间， 当QPS达到百万级时，1%的优化意味着年节省数十万元。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

常见问题与解决方案

实际工程中遇到的大多数问题都可以归为三类：性能问题（延迟/吞吐不达标）、 精度问题（输出质量下降）、稳定性问题（偶发OOM/超时）。 为什么要这样分类？因为每类问题的诊断工作机制完全不同： 性能问题用profiler，精度问题用eval benchmark，稳定性问题用日志分析。
