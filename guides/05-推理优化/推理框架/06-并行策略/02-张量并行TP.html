矩阵切分有两种方式，在 Transformer 中配合使用：

2.1 列切分（Column Parallel）

列切分: 按"列"把矩阵切成左右两半 输入 X: [n × d] (n 个 token，每个 d 维) 权重 W: [d × d'] 切成 [d × d'/2] 和 [d × d'/2] GPU 0: Y_left = X × W_left ← 计算左半部分输出 GPU 1: Y_right = X × W_right ← 计算右半部分输出 结果: [Y_left, Y_right] ← 两半拼起来就是完整结果！ 不需要 AllReduce！（结果天然互补）

2.2 行切分（Row Parallel）

行切分: 按"行"把矩阵切成上下两半 输入 X: [n × d] 切成 [n × d/2] 和 [n × d/2] 权重 W: [d × d'] 切成 [d/2 × d'] 和 [d/2 × d'] GPU 0: Y_left = X_left × W_top ← 左输入 × 上半权重 GPU 1: Y_right = X_right × W_bottom ← 右输入 × 下半权重 结果: Y = Y_left + Y_right ← 需要 AllReduce 求和！

一、核心思想

从系统架构的底层原理来看，一、核心思想由多个协同工作的组件构成。 为什么需要这样的分层设计？推导表明， 将复杂系统分解为独立模块可以将维护成本从O(N²)降低到O(N)， 每个模块只需关注自身的工作机制，通过标准接口与外部交互。

一、核心思想

衡量一、核心思想性能的核心指标包括延迟（latency）、吞吐（throughput）和资源利用率。 从数学本质看，这三者之间存在固有权衡关系： 吞吐 = batch_size / latency，而资源利用率 = 实际计算时间 / 总时间。 为什么不能同时优化三者？因为提高batch_size会增加延迟， 降低延迟会减少batch_size进而降低吞吐——这就是为什么需要 针对具体场景选择优化目标的底层原因。

一、核心思想的技术原理与数学推导： 从底层硬件角度分析，一、核心思想的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了一、核心思想的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
一、核心思想的系统架构分析： 从系统设计的原理出发，一、核心思想采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

一、核心思想的性能优化策略

优化一、核心思想性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

一、核心思想的工程实践与最佳实践

在生产环境中部署一、核心思想时，需要考虑以下工作机制： 资源规划（GPU显存预算公式：M_total = M_model + M_kv×B + M_overhead， B为并发数）、容错机制（健康检查、自动重启、灰度发布）、 监控告警（延迟p99、错误率、GPU利用率）。 为什么这些因素如此关键？因为推理服务是持续运行的， 任何微小的效率问题在高并发下都会被放大—— 从数学本质看，总成本 = 单次成本 × QPS × 时间， 当QPS达到百万级时，1%的优化意味着年节省数十万元。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

✅ 参考答案与详解
  1. 核心原理: 张量并行（TP）的核心思想是将每一层的权重矩阵"切开"分配到多张GPU上，每张GPU只存储和计算部分权重，最后通过集合通信合并结果。文件用"切蛋糕"做类比：一块大蛋糕一个人吃不下（单卡显存不够），切成两半两个人各吃一半（TP=2）。技术实现上，Transformer层的权重矩阵W从[d×d']被切分为两个[d×d'/2]（列切分）或两个[d/2×d']（行切分），每张GPU只存储一半权重并计算部分结果。TP的本质是"层内并行"——在同一层内部将计算和显存需求分散到多张GPU，显存节省为线性（TP=N则每卡显存约1/N），延迟也接近线性降低（TP=2延迟约减半）。
    关键要点: ①将权重矩阵切分到多张GPU；②每卡只存1/N权重；③每次计算后需AllReduce合并；④显存和延迟均接近线性改善。
    实际案例: 在单台8×H100服务器上部署LLaMA-70B推理，使用TP=8将80层Transformer的权重均匀切分到8张GPU，每卡只存约17.5GB权重，使推理成为可能。
    关联知识: 与Megatron-LM的TensorParallel实现、NVLink通信架构、Transformer权重矩阵维度（hidden_dim ×intermediate_dim）等直接相关。
  2. 核心原理: 矩阵切分的两种方式——列切分和行切分——是TP的技术基石。列切分（Column Parallel）按列将W切成左右两半[d×d'/2]，输入X完整地送到每张GPU，每张GPU计算X×W_left和X×W_right得到输出的左右两半，结果天然互补无需AllReduce。行切分（Row Parallel）按行将W切成上下两半[d/2×d']，输入X也被切分为左右两半，每张GPU计算X_left×W_top和X_right×W_bottom得到部分结果，需要AllReduce求和才能得到最终输出。在Transformer中两者配合使用：Q/K/V投影和Gate/Up投影使用列切分（按head自然切分，无通信），输出投影和Down投影使用行切分（需要AllReduce），每层恰好2次AllReduce。
    关键要点: ①列切分按列切、无AllReduce；②行切分按行切、需AllReduce；③Transformer中两者交替使用；④每层固定2次AllReduce（Attention输出+MLP输出）。
    实际案例: Llama-2-13B的hidden_dim=5120，80层模型使用TP=2时，每层产生2次AllReduce（传输batch×seq×5120×2bytes的数据），总计160次AllReduce。
    关联知识: 与多头注意力的并行head分配（GPU0负责head0-15，GPU1负责head16-31）、NCCL AllReduce操作的原生优化、通信量计算公式（2×batch×seq×dim×sizeof(dtype)）紧密相关。
  3. 核心原理: TP对通信带宽有极高要求，这是其最大限制。文件以Llama-70B（hidden_dim=8192, TP=4）为例精确计算：每次AllReduce数据量≈128MB，80层×2次=160次AllReduce，总通信量约20GB/请求。不同互联方式的带宽差异巨大：NVLink(H100) 900GB/s仅需0.02秒，PCIe Gen5 64GB/s需0.3秒，而100GbE网络12GB/s需1.7秒——完全不可用。这个计算的底层逻辑是：TP的AllReduce发生在每一层（80层×2次=160次），频率极高（每层都有），所以必须依赖NVLink这种高带宽低延迟的NVLink。跨机器的网络带宽（即使100GbE RDMA）也比NVLink低两个数量级。
    关键要点: ①每层2次AllReduce（80层=160次）；②每次AllReduce数据量与hidden_dim成正比；③跨机器网络带宽低两个数量级；④TP必须使用NVLink不能跨机。
    实际案例: 某团队尝试在两台各4×A100的服务器上做TP=8（跨机），发现延迟比单机TP=8高出10倍以上，根本原因是跨机走网络而不是NVLink。
    关联知识: 与GPU服务器内部拓扑（NVSwitch全互联、PCIe tree）、RDMA/RoCE网络、AllReduce算法（Ring AllReduce的通信量=2×(N-1)/N×data_size）等知识直接相关。
  4. 核心原理: TP的特点总结可归纳为四句话：显存线性节省、延迟接近线性降低、通信模式固定（每层AllReduce）、必须NVLink互联。KV Cache也按TP切分——每张GPU只存储分配到的attention head对应的KV Cache，这也是一种显存节省。最佳实践明确指出：TP只在同机NVLink互联的卡上使用，TP=2/4最常见，跨机器时用TP(同机)+PP(跨机)组合。值得注意的是，TP对延迟的降低效果在prefill阶段（并行处理所有输入token）最明显，但在decode阶段（逐token生成）由于batch size通常为1，计算量小，AllReduce的通信开销占比增大，TP的收益相对降低。
    关键要点: ①显存节省线性（TP=N→1/N）；②延迟降低接近线性；③KV Cache也按TP切分；④最佳配置TP=2/4同机NVLink。
    实际案例: vLLM部署LLaMA-70B时使用--tensor-parallel-size=2（双卡H100 NVLink），相比单卡无法运行，TP=2使推理延迟从不可能降至约200ms/token。
    关联知识: 与推理框架的TP配置参数（vLLM的--tp、TRT-LLM的--tp_size）、KV Cache内存管理（PagedAttention中KV按head分配到不同GPU）、推理阶段差异（prefill vs decode的并行度）密切相关。
向前串联： 一、核心思想的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
学习路径建议： 建议按"概念→原理→实验→调优"四步走。先理解核心概念和术语， 再深入数学原理和工作机制，然后用最小化实验验证理解，最后在真实场景中调优参数。 每步都应记录关键指标（延迟/吞吐/精度），建立数据驱动的优化习惯。
