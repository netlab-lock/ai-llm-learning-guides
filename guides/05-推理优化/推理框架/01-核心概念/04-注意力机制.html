👁️ 注意力机制

本节目标：理解 MHA、MQA、GQA、MLA 四种注意力架构的原理和推理影响。注意力头的共享策略直接决定了 KV Cache 的大小。

标准的 Multi-Head Attention，每个头独立计算 Q、K、V：

MHA (Multi-Head Attention): 输入 x ├──→ W_q1 → Q1 ──┐ ├──→ W_k1 → K1 ──┤→ Head 1: Attention(Q1,K1,V1) ├──→ W_v1 → V1 ──┘ ├──→ W_q2 → Q2 ──┐ ├──→ W_k2 → K2 ──┤→ Head 2: Attention(Q2,K2,V2) ├──→ W_v2 → V2 ──┘ ... ├──→ W_qh → Qh ──┐ ├──→ W_kh → Kh ──┤→ Head h: Attention(Qh,Kh,Vh) └──→ W_vh → Vh ──┘ │ Concat → W_o → 输出 KV Cache: 存储所有 h 个头的 K, V Llama-2-7B: h=32, 每层存 32 个 K 头 + 32 个 V 头

多头注意力 (MHA)的技术原理与数学推导： 从底层硬件角度分析，多头注意力 (MHA)的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了多头注意力 (MHA)的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
多头注意力 (MHA)的系统架构分析： 从系统设计的原理出发，多头注意力 (MHA)采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

多头注意力 (MHA)的性能优化策略

优化多头注意力 (MHA)性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

# 多头注意力 (MHA) - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed
