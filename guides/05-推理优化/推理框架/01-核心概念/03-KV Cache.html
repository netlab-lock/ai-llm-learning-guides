💾 KV Cache

本节目标：理解 KV Cache 的原理、内存计算公式、以及为什么它是推理优化的核心。KV Cache 是理解 KV Cache 压缩、前缀缓存 等后续技术的基础。
🗺️ 学习路径：本节 → KV Cache 压缩 → 前缀缓存 → Ring Attention（多GPU KV 分布）→ llama.cpp KV 实现

1. 为什么需要 KV Cache？

💡 计算复杂度的本质：自回归解码的核心瓶颈在于注意力计算。每生成一个新 token，标准实现需要将新 token 的 Q 与所有历史 token 的 K 做点积。如果不缓存，每步都要重新投影所有历史 token 的 K 和 V，这意味着每步的计算量线性增长，总复杂度为 O(N²)。KV Cache 的本质是用空间换时间——将已计算的 K、V 存起来，避免重复计算。

1.1 自回归解码的计算瓶颈

在自回归解码中，每生成一个新 token，模型需要对所有历史 token 计算注意力。如果每次都从头算，计算量会越来越大。理解这个瓶颈的数学本质对于后续学习所有优化技术至关重要。

没有 KV Cache —— 每步都重算所有 K/V：

Step 1: Q=[x1],         K=[Wk·x1],              V=[Wv·x1]              → y1
Step 2: Q=[x2],         K=[Wk·x1, Wk·x2],       V=[Wv·x1, Wv·x2]      → y2
Step 3: Q=[x3],         K=[Wk·x1, Wk·x2, Wk·x3], V=[...]              → y3
  ...
Step N: Q=[xN],         K=[Wk·x1...Wk·xN],      V=[Wv·x1...Wv·xN]     → yN

总 FLOPs: Σ(i=1 to N) O(i·d) = O(N²·d)  ← 巨大浪费！
         ↓
有 KV Cache —— 只算新 token 的 Q/K/V：

Step 1: Q1=Wq·x1, K1=Wk·x1, V1=Wv·x1       cache: K=[K1], V=[V1]
Step 2: Q2=Wq·x2, K2=Wk·x2, V2=Wv·x2       cache: K=[K1,K2], V=[V1,V2]
Step 3: Q3=Wq·x3, K3=Wk·x3, V3=Wv·x3       cache: K=[K1,K2,K3], V=[...]
  ...
每步: 1 次 Q 投影 + 1 次 K/V 投影 + 1 次注意力 (Q_new × K_cache^T)
总 FLOPs: O(N·d² + N²·d)  ← 投影部分线性，注意力部分仍为 O(N²)
🔍 数学本质推导：设模型隐藏维度为 d，序列长度为 N，注意力头数为 h，每头维度 d_k = d/h。对于第 t 步解码：
• Q 投影: h × d_k 次乘法（1 个 token）
• K/V 投影: 2 × h × d_k 次乘法（1 个 token）
• 注意力: Q_t × K_cache^T → h × t × d_k 次乘法
• 总计: 每步 O(h × d_k + h × t × d_k) = O(d + t·d)
• 全部 N 步: Σ_{t=1}^{N} O(d + t·d) = O(N·d + N²·d)
• 关键结论：KV Cache 将 K/V 投影的 O(N²·d) 降为 O(N·d)，但注意力计算仍为 O(N²·d)。这就是为什么即使有 KV Cache，长序列推理仍然很慢——后续的 Ring Attention 和 FlashAttention 试图进一步优化注意力计算本身。

1.2 为什么 K 和 V 能缓存而 Q 不能？

这是一个经常被忽视但非常关键的问题。在注意力机制 Attention(Q,K,V) = softmax(QK^T/√d_k)V 中：

  • K 和 V：在 decode 阶段，每个已生成 token 的 K 和 V 一旦计算完成就不会改变（投影矩阵 W_k、W_v 是固定的，输入 x_i 也是固定的）。所以 K_i 和 V_i 可以缓存。
  • Q：每一步只需要当前 token 的 Q（Q_new = W_q × x_new），不需要历史 token 的 Q。Q 的作用是"查询"，而 K/V 是"被查询的数据库"。
⚠️ 深层含义：Q 不需要缓存这个事实揭示了注意力机制的不对称性——Q 是"提问者"，K/V 是"知识库"。这种不对称性是 Ring Attention 的理论基础：每个 GPU 持有不同的 Q，但需要共享所有 K/V。

2. KV Cache 的工作原理

2.1 Prefill 与 Decode 两阶段

Transformer 推理分为两个截然不同的阶段，KV Cache 在两个阶段中的角色完全不同：

Prefill 阶段（prompt 处理）:
┌──────────────────────────────────────────────────┐
│  输入: [t0, t1, t2, ..., tn]  (n 个 prompt tokens)│
│                                                    │
│  所有 tokens 并行 → 一次前向传播                    │
│  ↓                                                 │
│  生成 KV Cache:                                    │
│    K_cache = [K_0, K_1, ..., K_n]  shape: [n, d_k]│
│    V_cache = [V_0, V_1, ..., V_n]  shape: [n, d_v]│
│  ↓                                                 │
│  输出: 第一个生成 token 的 logits                   │
└──────────────────────────────────────────────────┘
            ↓ KV Cache 传递给 Decode 阶段
Decode 阶段（逐 token 生成）:
┌──────────────────────────────────────────────────┐
│  每步输入: 1 个新 token                             │
│  每步计算:                                         │
│    Q_new = W_q × x_new     ← 只算 1 个 token      │
│    K_new = W_k × x_new     ← 只算 1 个 token      │
│    V_new = W_v × x_new     ← 只算 1 个 token      │
│    K_cache.append(K_new)   ← 追加到缓存            │
│    V_cache.append(V_new)   ← 追加到缓存            │
│    attn = softmax(Q_new × K_cache^T / √d_k) × V_cache│
└──────────────────────────────────────────────────┘
💡 Prefill 与 Decode 的计算特征差异：Prefill 阶段是计算密集型（compute-bound）——大量 token 并行处理，GPU 算力被充分利用。Decode 阶段是内存带宽密集型（memory-bound）——每步只算 1 个 token 的 Q，但需要从显存读取整个 KV Cache。这种不对称性是 Chunked Prefill 技术的出发点。

2.2 每层的 KV Cache 结构

在标准 MHA（Multi-Head Attention）模型中，每一层 Transformer 都有独立的 KV Cache。一个 L � 层模型就有 L 对 KV Cache：

// KV Cache 数据结构（伪代码）
struct KVCache {
    // 每层一个 K 缓冲区和一个 V 缓冲区
    K: Tensor[num_layers][max_seq_len][num_kv_heads][head_dim]  // FP16
    V: Tensor[num_layers][max_seq_len][num_kv_heads][head_dim]  // FP16
    seq_len: int  // 当前已缓存的 token 数
};

// Decode 步骤 t 的注意力计算（单层伪代码）
fn decode_attention_step(Q_new, KVCache, layer_idx):
    K_cache = KVCache.K[layer_idx][:seq_len]   // [seq_len, num_kv_heads, head_dim]
    V_cache = KVCache.V[layer_idx][:seq_len]

    // Q 与所有 K 做点积
    scores = Q_new @ K_cache.T / sqrt(head_dim)  // [num_heads, seq_len]

    // GQA: Q 有 num_heads 个头，K 只有 num_kv_heads 个头
    // 每个 Q 头查询对应的 KV 头（通过 head mapping）

    attn_weights = softmax(scores)
    output = attn_weights @ V_cache              // [num_heads, head_dim]
    return output

2.3 GQA 对 KV Cache 的影响

GQA（Grouped Query Attention）通过让多个 Q 头共享一组 K/V 头来减少 KV Cache 大小。这是现代大模型（Llama-3、Qwen2.5 等）的标准做法。

注意力类型Q 头数KV 头数KV Cache 大小代表模型
MHA3232基准GPT-2, Llama-1
GQA-83281/4 基准Llama-2-70B, Llama-3
GQA-43241/8 基准部分小模型
MQA3211/32 基准研究中
MLA128压缩1/64 基准DeepSeek-V2/V3
🔍 GQA 的数学本质：在标准 MHA 中，K 和 V 的投影矩阵 W_k、W_v 的形状为 [d_model, d_model]（num_heads × head_dim = d_model）。GQA 将 W_k 和 W_v 的形状从 [d_model, d_model] 缩小为 [d_model, num_kv_heads × head_dim]。这意味着 KV Cache 从 num_heads 缩小到 num_kv_heads，但注意力计算时，每个 Q 头通过重复（repeat）或广播（broadcast）对应的 KV 头来保持相同的计算逻辑。GQA 的压缩率 = num_heads / num_kv_heads。详见 注意力机制。

3. KV Cache 内存计算

3.1 精确计算公式

KV Cache 内存 = 2 × L × n_kv × d_k × S × b
其中: L=层数, n_kv=KV头数, d_k=每头维度, S=序列长度, b=数据类型字节

这个公式的推导原理：

  • 2：K 和 V 两个独立的缓存
  • L：Transformer 层数（每层有独立的注意力，所以每层需要独立的 KV Cache）
  • n_kv：KV 注意力头数（GQA 时少于 Q 头数）
  • d_k：每个头的维度（通常 128）
  • S：当前序列长度（随 decode 过程线性增长）
  • b：数据类型字节数（FP16=2, FP8=1, INT8=1, INT4=0.5）

3.2 实际模型计算示例

模型层数 LKV头数 n_kvd_k2K ctx (FP16)8K ctx128K ctx
Llama-2-7B (MHA)32321281.0 GB4.0 GB64 GB
Llama-2-70B (GQA)8081280.5 GB2.0 GB32 GB
Llama-3-8B (GQA)3281280.25 GB1.0 GB16 GB
Qwen2.5-72B (GQA)8081280.5 GB2.0 GB32 GB

⚠️ 关键结论：128K 上下文时，KV Cache 可达 32-64 GB，远超模型参数本身（INT4 量化后仅 3.5-18 GB）。KV Cache 是长上下文推理的最大内存瓶颈，比模型权重本身还大！这就是为什么 KV Cache 压缩 是如此重要的研究方向。

3.3 KV Cache 与模型参数的内存对比

// 以 Llama-3-8B 为例的内存分析
// 模型参数: 8B × 2 bytes (FP16) = 16 GB
// 模型参数: 8B × 0.5 bytes (INT4) = 4 GB

// KV Cache (FP16):
//   2 × 32 layers × 8 kv_heads × 128 dim × seq_len × 2 bytes
//   = 2 × 32 × 8 × 128 × seq_len × 2
//   = 131072 × seq_len bytes
//   = 128 KB × seq_len

//   2K:    128 KB × 2048  = 256 MB
//   8K:    128 KB × 8192  = 1 GB

//   128K:  128 KB × 131072 = 16 GB  ← 超过 INT4 模型的 4 GB!

// KV Cache / 模型参数 比率:
//   FP16 模型 + FP16 KV: KV/Param = 16/16 = 1.0 (2K ctx)
//   INT4 模型 + FP16 KV: KV/Param = 16/4 = 4.0 (2K ctx!)
//   INT4 模型 + 128K ctx: KV/Param = 16/4 = 4.0 → 瓶颈在 KV

4. 多请求时的 KV Cache 管理

4.1 并发请求的内存竞争

服务多个用户请求时，每个请求都需要独立的 KV Cache。这导致了一个核心矛盾：并发数 vs 序列长度。

GPU 显存分配 (以 80GB A100 + Llama-3-8B INT4 为例):

┌────────────────────────────────────────┐
│         模型参数 (INT4)                 │  4 GB (固定)
├────────────────────────────────────────┤
│  Request 1 KV Cache (2K ctx)           │  256 MB
├────────────────────────────────────────┤
│  Request 2 KV Cache (2K ctx)           │  256 MB
├────────────────────────────────────────┤
│  ...                                   │
├────────────────────────────────────────┤
│  Request N KV Cache (2K ctx)           │  256 MB
├────────────────────────────────────────┤
│  框架开销 + 激活值                       │  ~4 GB
├────────────────────────────────────────┤
│  空闲显存                               │
└────────────────────────────────────────┘

可用显存: 80 - 4 (模型) - 4 (开销) = 72 GB
并发 2K ctx: 72 GB / 0.25 GB = ~288 并发 ✅
并发 8K ctx: 72 GB / 1 GB = ~72 并发
并发 128K ctx: 72 GB / 16 GB = ~4 并发 ← 严重瓶颈！
🔍 为什么 KV Cache 内存管理如此困难：与模型参数（固定大小、可预分配）不同，KV Cache 有三个独特的管理难题：
① 动态增长：每生成一个 token，KV Cache 增长一行，无法预知最终长度
② 生命周期不确定：用户可能随时停止生成，释放内存
③ 大小不一：不同请求的 prompt 长度和生成长度差异巨大
这三个特性导致了严重的内存碎片化问题。传统的预分配策略（为每个请求预留最大长度的空间）会浪费大量显存。这就是 PagedAttention 要解决的核心问题。

4.2 KV Cache 的生命周期

// KV Cache 生命周期管理（简化伪代码）
fn handle_request(prompt):
    // 阶段 1: Prefill — 分配 KV Cache
    kv_cache = allocate_kv_cache(num_layers, num_kv_heads, head_dim, max_seq_len)
    tokens = tokenize(prompt)
    logits, kv_cache = prefill(tokens, kv_cache)  // 填充 prompt 的 K/V

    // 阶段 2: Decode — KV Cache 持续增长
    for i in range(max_new_tokens):
        next_token = sample(logits)
        if next_token == EOS:
            break
        logits, kv_cache = decode_step(next_token, kv_cache)  // 追加 K/V

    // 阶段 3: 完成 — 释放 KV Cache
    free_kv_cache(kv_cache)  // 归还显存给内存池

• 原始 FP16 KV: 16 GB × 100 = 1600 GB → 不可能

✅ 参考答案与详解
  1. 核心原理: KV Cache之所以被称为"用空间换时间"，是因为它通过占用额外的GPU显存（空间）来存储已计算的K和V值，从而避免每次decode步骤中的重复计算（时间）。从数学角度分析：不使用KV Cache时，第t步需要重新计算所有t个token的K和V投影，投影部分的总FLOPs为Σ_{t=1}^{N} 2×t×d×d_k = O(N²·d·d_k)，即二次复杂度。使用KV Cache后，每个token的K和V只计算一次，投影总FLOPs降为Σ_{t=1}^{N} 2×d×d_k = O(N·d·d_k)，即线性复杂度。注意力计算部分仍为O(N²·d_k)。如果不用KV Cache，以N=1000为例，投影部分的计算量将增大约500倍！这使得推理延迟随序列长度急剧恶化，在工程上不可接受。
    关键要点: ①无KV Cache时投影计算为O(N²·d)，有KV Cache时为O(N·d)；②K和V能被缓存的根本原因是它们在计算后不再变化（W_k/W_v固定，x_i固定）；③Q不能缓存是因为每一步只需要当前token的Q；④K和V的角色不对称——Q是"提问者"，K/V是"知识库"；⑤即使有KV Cache，注意力计算本身仍为O(N²)，这是FlashAttention要解决的问题。
    实际案例: 以Llama-3-8B生成长度N=2048序列为例，无KV Cache时投影需重算约2048²×4096×128 ≈ 2.2×10¹²次操作，有KV Cache仅需2048×4096×128×2 ≈ 2.2×10⁹次操作，相差1000倍。实测中，无KV Cache的推理延迟将从每步10ms增长到全过程约10秒——完全无法实用。
    关联知识: "空间换时间"是计算机科学中最基本的权衡思想。类似的技术包括：数据库索引（用存储空间换取查询速度）、视频编码的参考帧缓存（用内存换取编码效率）、操作系统的页缓存（用RAM换取磁盘IO速度）。KV Cache的特殊之处在于它的"空间"需求是动态增长的——每生成一个token就多占一行KV，这带来了独特的内存管理挑战。
  2. 核心原理: 根据KV Cache内存公式：KV Cache = 2 × L × n_kv × d_k × S × b。对于Llama-3-8B（32层、8 KV头、d_k=128），128K上下文：(a) FP16精度：2 × 32 × 8 × 128 × 131072 × 2 = 17,179,869,184 bytes = 16 GB；(b) FP8量化：仅需1 byte/值，所以为16 × (1/2) = 8 GB；(c) MLA压缩到d_compress=512：MLA仅缓存压缩后的c_kv表示（512维），而非完整的K和V（8×128=1024维），因此为2 × 32 × 512 × 131072 × 2 = 8,589,934,592 bytes ≈ 0.25 GB（注意：MLA重新计算K/V时使用了额外的上投影矩阵，但这部分不计入Cache存储）。
    关键要点: ①FP16下128K上下文KV Cache为16GB，超过INT4量化模型本身（4GB）；②FP8量化将KV Cache减半至8GB，是性价比最高的方案；③MLA压缩到512维后仅0.25GB，实现64倍压缩；④KV Cache与上下文长度成正比（128K是2K的64倍内存）；⑤在实际部署中，16GB的KV Cache意味着一个80GB A100只能在KV Cache上放5个128K并发请求。
    实际案例: 在多轮对话场景中，假设系统需要支持100个并发用户，每用户8K上下文（FP16），Llama-3-8B的KV Cache需求为：16×(8K/128K)×100 = 100GB。使用FP8量化后降至50GB，再结合GQA和PagedAttention可优化至约45GB。2张A100即可满足需求，而纯FP16方案需要4张。
    关联知识: KV Cache的内存占比取决于上下文长度与量化精度的乘积效应。在训练中不是问题（K/V在反向传播后丢弃），但推理中KV Cache是唯一随时间线性增长的数据结构。这就是为什么"KV Cache压缩"成为推理优化的核心课题——量化、驱逐、共享、MLA四大策略从不同维度解决这一问题。
  3. 核心原理: Prefill是compute-bound而Decode是memory-bound的根本原因是算术强度（Arithmetic Intensity = FLOPs / Bytes transferred）不同。Prefill阶段：输入N个token执行矩阵-矩阵乘法（GEMM），对于C=A×B（[M,K]×[K,N]），AI = 2MNK / (MK+KN+MN)×dtype_bytes。当M=N=较大值（如4096），AI很高（>156 for A100），远超GPU的FLOPs/Byte比值（A100为312T/2T=156），因此是compute-bound——GPU算力是瓶颈，增加更多算力就能加速。Decode阶段：输入1个token执行矩阵-向量乘法（GEMV），M=1，AI = 2KN/(K+N+1)×dtype_bytes ≈ 2×d_model/(d_model+1) × dtype_bytes ≈ 2，远低于156，因此是memory-bound——HBM带宽是瓶颈。
    关键要点: ①算术强度决定了是compute-bound还是memory-bound；②Prefill的AI ∝ N（正比于序列长度），长Prompt更compute-bound；③Decode的AI ≈ 2，与序列长度无关；④GPU的算力/带宽比值决定了转折点（A100为156，H100更高）；⑤这一差异影响所有批处理策略——Decode阶段batch增大无法提升单token延迟。
    实际案例: 对Llama-3-8B来说，Prefill 2048 tokens的算术强度约为2048，远超A100的转折点156，因此Prefill A100 2048 tokens只需约50ms且GPU利用率>90%。Decode阶段AI≈2，远低于156，A100的理论Decode速度受HBM带宽限制：2TB/s / (16GB模型参数) ≈ 125 tok/s。这就是为什么Decode阶段增加batch size对延迟改善有限——瓶颈在带宽而非算力。
    关联知识: 这一计算/内存的本质差异直接决定了Chunked Prefill的设计理念——将Prefill分块，使每个chunk的计算时间与Decode单次时间相当，从而实现两者的交错执行。在Disaggregated Serving中，Prefill GPU（compute-bound）选择高算力卡如A100，Decode GPU（memory-bound）选择高带宽卡如H100（3.35TB/s），可以实现硬件与工作负载的最优匹配。
  4. 核心原理: 计算分析：Llama-3-70B INT4量化模型35GB，32K上下文的KV Cache（FP16）= 2 × 80 × 8 × 128 × 32768 × 2 = 10,737,418,240 bytes ≈ 10 GB/请求。单卡可用显存 = 80 - 35 = 45 GB，扣除框架开销约4 GB后为41 GB，可支持41/10 ≈ 4并发请求。4GPU总计16并发，这对于生产API服务远远不足。优化方案：①PagedAttention：将KV Cache分页管理（消除内存碎片后利用率>95%），使16并发提升至约17-18；②Ring Attention：将KV Cache分布到多GPU，每GPU仅存1/G的KV。4GPU全部用于Ring Attention后，KV Cache降至2.5GB/卡/请求，释放大量显存，理论上可支持更多并发；③组合：先用Ring Attention将模型分布到4GPU（TP=4），每卡模型仅8.75GB，剩余71GB用于KV Cache，支持约6-7并发/卡，总计24-28并发。
    关键要点: ①单GPU仅4并发是严重的生产瓶颈；②PagedAttention可减少碎片但无法根本解决KV Cache过大的问题；③Ring Attention的KV分布可线性增加并发数；④TP+KV分布的组合是最优工程方案；⑤实际部署还需考虑KV Cache的量化压缩（FP8再减半）。
    实际案例: 综合部署方案：4×A100-80GB部署Llama-3-70B（INT4），32K上下文+FP8 KV Cache。模型8.75GB/卡（TP=4），KV Cache：2×80×8×128×32768×1 = 5.2GB/请求/卡。可用显存71GB，支持约13并发/卡，总计52并发。再叠加PagedAttention（消除碎片），实际可达50+并发。若需100并发，则需8卡（8×13≈104并发）。
    关联知识: 本题涉及KV Cache量化、张量并行（TP）、Ring Attention、PagedAttention四大技术的组合，是推理优化的综合应用。在实际生产环境中，通常采用TP（降低显存压力）+ FP8 KV Cache（降低KV内存）+ PagedAttention（消除碎片）的三重组合。Ring Attention主要用于超长序列（>256K），对于32K上下文，TP配PAGEDAttension已经足够。
