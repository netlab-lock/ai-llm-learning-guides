🎯 核心概念总览

本模块目标：掌握 LLM 推理的 6 个核心概念——从文本如何变成 token，到模型如何一步步生成输出，再到如何衡量推理系统的性能。本总览页是整个模块的"导航地图"，帮你建立全局认知后再逐个击破。

1. 模块知识地图

常见误区： 学习模块知识地图时，很多人只看概念不动手实验。建议每学完一个知识点就写验证代码， 在实际运行中观察行为是否符合预期。理论和实践的差距往往比想象中大。
常见误区： 不要以为模块知识地图可以跳过基础直接上手高级内容。基础概念是后续所有优化和应用的前提， 跳过基础会导致后续学习事倍功半。

1.3 核心概念依赖关系

深入探讨：从系统设计的底层原理看，模块知识地图的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

2.3 概念间的交叉关联

核心概念交叉引用图: 01-Tokenizer ─────────────────────────────────────────┐ │ │ │ 输入: 文本→token │ ▼ │ 02-自回归解码 ◄──────────────────────────────────┐ │ │ │ │ │ │ Prefill/Decode │ 采样输出 │ │ ▼ ▼ │ │ 03-KV Cache 05-采样策略 │ │ │ │ │ │ │ 内存优化 │ 输出质量 │ │ ▼ │ │ │ 04-注意力机制 │ │ │ │ │ │ │ │ MHA/MQA/GQA │ │ │ ▼ ▼ │ │ 06-性能指标 ◄─────────────┴──────────────────────┘ │ │ │ │ TTFT/TPOT/TPS │ ▼ │ ┌──────────────────────────────────────────────────────┘ │ 下游应用: │ ├── ../03-注意力优化/ (FlashAttention等) │ ├── ../04-量化技术/ (减少KV Cache内存) │ ├── ../编译优化/ (torch.compile加速) │ ├── ../09-部署实战/ (监控与运维) │ └── ../10-学习资源/ (深入学习) └──────────────────────────────────────────────────────
深入探讨：这 6 个概念并非孤立存在，而是形成了一个紧密的"优化闭环"。例如：Tokenizer 的词表大小（01）直接影响 KV Cache 的内存占用（03），因为更大的词表意味着更大的 embedding 层和更大的 logits 输出；注意力机制的选择（04，如 GQA vs MHA）决定了 KV Cache 的压缩比（03）；采样策略（05）的复杂度影响 Decode 阶段的延迟（06）；而性能指标（06）又反过来指导我们应该优先优化哪个环节。理解这些交叉关联，是从"知道概念"到"能做优化"的关键跃迁。

3.2 常见学习误区与避坑

常见误区：
  • 跳过基础直接看优化：不理解 KV Cache 的原理就去看 PagedAttention，只会知其然不知其所以然
  • 只看不练：看完概念后不写代码验证，一周后就忘了
  • 忽略交叉关联：单独理解每个概念，但不知道它们之间如何相互影响
  • 混淆训练和推理：用训练的思维去理解推理，导致对延迟和吞吐的要求理解偏差

3.3 从概念到实践的桥梁

实用建议：学完本模块后，建议通过以下路径将概念转化为能力：
  1. 阅读源码：在 vLLM 源码中搜索 PagedAttention、Sampler 等关键词，看概念如何落地

  4. 对比框架：用 vLLM 和 SGLang 部署同一模型，对比 TTFT 和 TPOT 的差异

# 练习 1: Tokenizer 实验
from transformers import AutoTokenizer
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3-8B-Instruct")
tokens = tokenizer.encode("你好，世界！Hello World!")
print(f"文本长度: {len('你好，世界！Hello World!')} 字符")
print(f"Token 数量: {len(tokens)}")
print(f"压缩比: {len('你好，世界！Hello World!') / len(tokens):.2f} 字符/token")

# 练习 2: KV Cache 内存计算
def calc_kv_cache_memory(num_layers, num_kv_heads, head_dim, seq_len, batch_size, dtype_bytes=2):
    """计算 KV Cache 内存占用 (bytes)"""
    # 每层的 K 和 V 各需要: batch × seq_len × num_kv_heads × head_dim
    memory = 2 * num_layers * batch_size * seq_len * num_kv_heads * head_dim * dtype_bytes
    return memory / (1024**3)  # 转换为 GB

# Llama-3-8B: 32层, 8个KV头(GQA), 128维头
memory_gb = calc_kv_cache_memory(32, 8, 128, seq_len=4096, batch_size=1)
print(f"Llama-3-8B, seq_len=4096, batch=1: KV Cache = {memory_gb:.2f} GB")

💡 思考题：
  1. 为什么 LLM 推理的 Decode 阶段是 memory-bound 而不是 compute-bound？从矩阵乘法的形状（M=1 的 GEMV）和 GPU 的 Roofline 模型来分析。
  2. 如果一个模型的词表从 32K 扩大到 128K，对推理的哪些环节会产生影响？影响有多大？（提示：Tokenizer 编码效率、embedding 层大小、logits 计算、采样策略的计算量）
  3. KV Cache 的内存占用与序列长度是线性关系。如果要支持 128K 长度的上下文，KV Cache 的内存挑战有多大？有哪些技术可以缓解？（提示：GQA、MQA、MLA、KV Cache 量化、PagedAttention）
  4. TTFT（首 token 延迟）和 TPOT（每 token 延迟）分别由哪些因素决定？在什么场景下用户更在意 TTFT，什么场景下更在意 TPOT？
  5. 如果要设计一个新的注意力机制（比如 MLA），你需要在 KV Cache 大小、计算复杂度、推理延迟之间做哪些权衡？
