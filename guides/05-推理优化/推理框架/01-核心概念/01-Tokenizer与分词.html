🔤 Tokenizer 与分词

本节目标：理解文本如何变成模型可处理的数字序列。Tokenizer 的选择直接影响推理效率（token 数量 = 推理步数）。

1. 为什么需要 Tokenizer？

为什么需要 Tokenizer？

ong>通俗类比：就像图书馆的索引系统。图书馆不会逐页阅读，而是先用索引卡（Tokenizer）将书名、作者（文本）转化为唯一的编码（Token ID），这样管理员（模型）才能高效地存取和理解。
常见误区：不要误以为Tokenizer在理解语义。它的任务只是将文本分割并映射为数字ID，就像查字典找页码，而不是理解单词含义。理解语义是模型后续的工作。
深入探讨：一个典型Tokenizer的词表大小（V）通常在3万到10万个Token之间。每个Token会被映射为一个d维的向量（嵌入），例如d=768。因此，嵌入层的参数量就是 V * d，例如3万*768 ≈ 2304万个参数。这占用了模型相当一部分存储空间。

神经网络只能处理数字，不能直接处理文本。Tokenizer 负责：

  1. 编码：文本 → token ID 序列（推理输入）
  2. 解码：token ID 序列 → 文本（推理输出）
输入: "Hello, 你好世界!" Tokenizer 编码: "Hello" → [15496] (完整单词) "," → [11] (标点) " 你好" → [233, 162] (中文子词) "世界" → [1917, 101] (中文子词) "!" → [0] (标点) 最终: [15496, 11, 233, 162, 1917, 101, 0] 7 个 token → 模型需要推理 7 步

2. 主流分词算法

通俗类比：就像编辑一份食谱。BPE像自下而上合并常见配料组合（如“葱姜蒜”）；WordPiece像按特定语法规则（最大似然）切分步骤；Unigram则从一个大词表开始，逐步剔除不重要的词汇。
常见误区：不要认为分词算法对所有语言一视同仁。BPE和WordPiece在处理中文、日文等无空格语言时，效果和逻辑与处理英文完全不同，通常需要专门为这些语言设计预处理步骤。
深入探讨：BPE算法核心是迭代合并。初始词表是所有单个字符。训练时，统计所有相邻Token对的出现频率，将频率最高的对合并为新Token。例如，语料中’s’和’t’相邻频率极高，则合并为新Token‘st’。词表大小 = 初始字符数 + 合并操作次数。
Step1: a b c a b d
Step2: ab c ab d (合并a,b)
Step3: ab c abd (合并b,d)

2.1 BPE (Byte Pair Encoding)

最常用，GPT 系列、Llama、Qwen 都使用 BPE 或其变体：

  1. 初始词表 = 所有单个字符（或字节）
  2. 统计训练语料中最频繁的相邻 token 对
  3. 将最频繁的对合并为新 token，加入词表
  4. 重复步骤 2-3，直到词表达到目标大小
BPE 训练示例: 语料: "low lower lowest" 初始词表: {'l','o','w','e','r','s','t'} Step 1: 最频繁对 ('l','o') → 合并为 'lo' Step 2: 最频繁对 ('lo','w') → 合并为 'low' Step 3: 最频繁对 ('e','r') → 合并为 'er' Step 4: 最频繁对 ('low','e') → 合并为 'lowe' ... 最终: "lowest" → ['low', 'est'] 或 ['low', 'e', 'st']

2.2 Byte-level BPE

GPT-2/3/4 使用。初始词表是 256 个字节（byte），而非 Unicode 字符：

  • 优势：可以编码任何字节序列，不会出现 OOV（未登录词）
  • 中文处理：每个中文字符通常被编码为 2-3 个 byte token

2.3 SentencePiece

Llama、Mistral 使用。直接在原始文本上训练，不依赖预分词：

  • Unigram 或 BPE 算法
  • 将空格也作为 token 处理（用 ▁ 表示）
  • 对多语言支持更好

3. 词表大小的影响

。

词表大小的影响

通俗类比：就像城市地图的比例尺。词表太小（比例尺大），一个街区要用很多张地图拼（Token序列很长），查找麻烦；词表太大（比例尺小），地图册变得厚重无比（嵌入矩阵庞大），携带和翻阅成本高。
常见误区：切忌盲目追求超大词表。词表并非越大越好，它会急剧增加模型首层的参数量（嵌入层）和最后一层的计算量（输出层logits），可能超出硬件内存限制，导致训练和推理速度变慢。
深入探讨：设词表大小为V，嵌入维度为d，则嵌入矩阵参数量为V*d。当V从3万增至10万时，仅嵌入层参数就增加约 (10万-3万)*768 = 5376万个参数，约占用205MB显存（FP32）。此外，更大的词表可能导致序列长度缩短，但上下文信息密度可能不均匀。
小词表：I love AI -> [I] [lo] [ve] [A] [I] (5个Token)
大词表：I love AI -> [I] [love] [AI] (3个Token)

深入探讨：从系统设计的底层原理看，为什么需要 Tokenizer？的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。
