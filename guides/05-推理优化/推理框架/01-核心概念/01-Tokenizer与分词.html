🔤 Tokenizer 与分词

本节目标：理解文本如何变成模型可处理的数字序列。Tokenizer 的选择直接影响推理效率（token 数量 = 推理步数）。

1. 为什么需要 Tokenizer？

为什么需要 Tokenizer？

ong>通俗类比：就像图书馆的索引系统。图书馆不会逐页阅读，而是先用索引卡（Tokenizer）将书名、作者（文本）转化为唯一的编码（Token ID），这样管理员（模型）才能高效地存取和理解。
常见误区：不要误以为Tokenizer在理解语义。它的任务只是将文本分割并映射为数字ID，就像查字典找页码，而不是理解单词含义。理解语义是模型后续的工作。
深入探讨：一个典型Tokenizer的词表大小（V）通常在3万到10万个Token之间。每个Token会被映射为一个d维的向量（嵌入），例如d=768。因此，嵌入层的参数量就是 V * d，例如3万*768 ≈ 2304万个参数。这占用了模型相当一部分存储空间。

神经网络只能处理数字，不能直接处理文本。Tokenizer 负责：

  1. 编码：文本 → token ID 序列（推理输入）
  2. 解码：token ID 序列 → 文本（推理输出）
输入: "Hello, 你好世界!" Tokenizer 编码: "Hello" → [15496] (完整单词) "," → [11] (标点) " 你好" → [233, 162] (中文子词) "世界" → [1917, 101] (中文子词) "!" → [0] (标点) 最终: [15496, 11, 233, 162, 1917, 101, 0] 7 个 token → 模型需要推理 7 步

2. 主流分词算法

通俗类比：就像编辑一份食谱。BPE像自下而上合并常见配料组合（如“葱姜蒜”）；WordPiece像按特定语法规则（最大似然）切分步骤；Unigram则从一个大词表开始，逐步剔除不重要的词汇。
常见误区：不要认为分词算法对所有语言一视同仁。BPE和WordPiece在处理中文、日文等无空格语言时，效果和逻辑与处理英文完全不同，通常需要专门为这些语言设计预处理步骤。
深入探讨：BPE算法核心是迭代合并。初始词表是所有单个字符。训练时，统计所有相邻Token对的出现频率，将频率最高的对合并为新Token。例如，语料中’s’和’t’相邻频率极高，则合并为新Token‘st’。词表大小 = 初始字符数 + 合并操作次数。

Step2: ab c ab d (合并a,b)
Step3: ab c abd (合并b,d)

2.1 BPE (Byte Pair Encoding)

最常用，GPT 系列、Llama、Qwen 都使用 BPE 或其变体：

  1. 初始词表 = 所有单个字符（或字节）
  2. 统计训练语料中最频繁的相邻 token 对
  3. 将最频繁的对合并为新 token，加入词表
  4. 重复步骤 2-3，直到词表达到目标大小
BPE 训练示例: 语料: "low lower lowest" 初始词表: {'l','o','w','e','r','s','t'} Step 1: 最频繁对 ('l','o') → 合并为 'lo' Step 2: 最频繁对 ('lo','w') → 合并为 'low' Step 3: 最频繁对 ('e','r') → 合并为 'er' Step 4: 最频繁对 ('low','e') → 合并为 'lowe' ... 最终: "lowest" → ['low', 'est'] 或 ['low', 'e', 'st']

2.2 Byte-level BPE

GPT-2/3/4 使用。初始词表是 256 个字节（byte），而非 Unicode 字符：

  • 优势：可以编码任何字节序列，不会出现 OOV（未登录词）
  • 中文处理：每个中文字符通常被编码为 2-3 个 byte token

2.3 SentencePiece

Llama、Mistral 使用。直接在原始文本上训练，不依赖预分词：

  • Unigram 或 BPE 算法
  • 将空格也作为 token 处理（用 ▁ 表示）
  • 对多语言支持更好

3. 词表大小的影响

词表大小的影响

通俗类比：就像城市地图的比例尺。词表太小（比例尺大），一个街区要用很多张地图拼（Token序列很长），查找麻烦；词表太大（比例尺小），地图册变得厚重无比（嵌入矩阵庞大），携带和翻阅成本高。
常见误区：切忌盲目追求超大词表。词表并非越大越好，它会急剧增加模型首层的参数量（嵌入层）和最后一层的计算量（输出层logits），可能超出硬件内存限制，导致训练和推理速度变慢。
深入探讨：设词表大小为V，嵌入维度为d，则嵌入矩阵参数量为V*d。当V从3万增至10万时，仅嵌入层参数就增加约 (10万-3万)*768 = 5376万个参数，约占用205MB显存（FP32）。此外，更大的词表可能导致序列长度缩短，但上下文信息密度可能不均匀。
小词表：I love AI -> [I] [lo] [ve] [A] [I] (5个Token)
大词表：I love AI -> [I] [love] [AI] (3个Token)

✅ 参考答案与详解
  1. 核心原理: Tokenizer存在的根本原因是神经网络只能处理数值张量，不能直接处理文本字符串。Tokenizer承担"翻译官"的角色，将人类语言文本转换为模型可计算的token ID序列。具体来说，Tokenizer包含编码（Encode）和解码（Decode）两个方向：编码将输入文本切分为子词单元并映射为词表中的整数ID，解码则将模型输出的token ID序列还原为人类可读的文本。Tokenizer的选择直接影响推理效率，因为token数量等于模型的推理步数——同样的文本，分词效率高的Tokenizer产生更少的token，从而减少Decode阶段的迭代次数，降低延迟和计算成本。此外，Tokenizer的词表构成决定了模型对不同语言和领域的处理效率，比如词表中包含大量中文词元的Tokenizer处理中文时token数更少。Tokenizer本身不做语义理解，只是机械的分割和映射操作。
    关键要点: ①神经网络只能处理数字，Tokenizer是文本到数字的桥梁 ②编码（文本→ID）和解码（ID→文本）是两个方向 ③token数量=推理步数，直接影响延迟和成本 ④Tokenizer不做语义理解，只是分割+映射 ⑤词表构成决定多语言和领域处理效率
    实际案例: 以处理"Hello, 你好世界!"为例，Tokenizer编码过程为："Hello"→[15496]（完整单词，1个token），","→[11]（标点，1个token），" 你好"→[233, 162]（中文子词，2个token），"世界"→[1917, 101]（中文子词，2个token），"!"→[0]（标点，1个token）。总共7个token意味着模型需要7步Decode来生成等长输出。如果换用一个中文效率更高的Tokenizer（如Qwen的152K词表），同样的中文文本可能只需3-4个token，推理步数减少近一半。
    关联知识: Tokenizer与Embedding层紧密相连——Embedding层的参数矩阵大小为V×d（V为词表大小，d为嵌入维度），如Llama 3的128K词表×4096维=525M参数。Tokenizer的选择还影响上下文窗口的有效利用——同样128K token的上下文窗口，分词效率高的Tokenizer能装入更多实际文本内容。
  2. 核心原理: 主流分词算法有三大类。BPE（Byte Pair Encoding）是最常用的方法，采用自下而上的策略：初始词表为所有单个字符或字节，然后迭代统计语料中最频繁的相邻token对，将最高频对合并为新token加入词表，重复直到词表达到目标大小。GPT系列和Llama都使用BPE或其变体。WordPiece采用最大似然估计来决定合并哪些对，可能产生##前缀标记子词续接（如"playing"→"play"+"##ing"），BERT使用此方法。Unigram则采用自上而下策略，从一个大词表开始，计算每个token被移除后对整体似然的影响，逐步剪枝至目标大小，T5和XLNet使用此方法，它是一个概率模型，可能产生多个分词结果。此外，Byte-level BPE（GPT-2/3/4使用）以256个字节为初始词表，可以编码任何字节序列避免OOV问题，但中文效率较低（1个汉字通常需要2-3个byte token）。SentencePiece（Llama、Mistral使用）直接在原始文本上训练，将空格也作为token处理（用▁表示），对多语言支持更好。
    关键要点: ①BPE自下而上合并，最简单高效，GPT/Llama/Qwen都使用 ②WordPiece用最大似然合并，产生##前缀，BERT使用 ③Unigram自上而下剪枝，概率模型，T5使用 ④Byte-level BPE以256字节为基础避免OOV但中文效率低 ⑤SentencePiece不依赖预分词，多语言友好
    实际案例: 以训练BPE处理语料"low lower lowest"为例：初始词表为{'l','o','w','e','r','s','t'}，Step1统计发现(l,o)频率最高，合并为'lo'；Step2统计(lo,w)频率最高，合并为'low'；Step3(e,r)合并为'er'；Step4继续合并。最终"lowest"被分词为['low','est']或['low','e','st']。这个过程就像学习语言中的常见词缀和组合——高频组合被"固化"为独立token。实际的GPT-4训练会在数十GB语料上执行数万次这样的合并操作，最终构建约10万个token的词表。
    关联知识: 分词算法的选择影响推理的多个环节。BPE的确定性分词（同一输入始终产生同一token序列）有利于Prefix Caching——相同前缀的请求可以复用KV Cache。而Unigram的概率分词可能导致同一输入产生不同token序列，不利于缓存复用。中文分词效率差异很大：Llama 2的32K词表中中文token较少，一个汉字可能被拆成2-3个byte token；而Qwen的152K词表专门扩充了中文词元，一个常用汉字通常只需1个token。
  3. 核心原理: 词表大小（V）对模型有双重影响。一方面，词表越大，每个token能表示的信息越多，生成同样文本需要的token数越少（序列更短），从而减少Decode阶段的推理步数，降低延迟。另一方面，词表越大，Embedding层（V×d参数矩阵）和LM Head层（d×V矩阵乘法）的计算量和内存占用越大。以具体数据说明：GPT-2词表50K，Llama 2为32K，Llama 3扩充到128K，Qwen 2.5为152K，DeepSeek-V3为129K。Llama 3从32K扩充到128K，仅Embeding层参数就增加了(128K-32K)×4096=393M参数，约1.5GB显存（FP32）。但好处是Llama 3的多语言token效率大幅提升。LM Head的计算是[batch_size×hidden_dim]×[hidden_dim×vocab_size]的矩阵乘法，词表从32K增大到128K使这一步的计算量增加4倍。因此词表大小是一个需要在token效率和计算/内存开销之间权衡的设计决策。通常50K-150K是当前主流范围，中文为主的模型倾向选择更大词表。
    关键要点: ①词表大→token效率高→序列短→Decode步数少 ②词表大→Embedding层和LM Head层参数量和计算量大 ③词表从32K增到128K使LM Head计算量增加4倍 ④当前主流范围50K-150K，中文模型倾向更大词表 ⑤词表大小需要在token效率和计算开销之间权衡
    实际案例: 以处理"请帮我解释一下深度学习的原理"这句话为例。Llama 2（32K词表，英文优化）可能需要18-20个token来编码这段中文文本；Qwen 2.5（152K词表，中文优化）可能只需8-10个token。在Decode阶段，假设TPS=30，Llama 2生成等长回复需要约0.6秒，Qwen只需约0.3秒。但Qwen的LM Head矩阵乘法（3584×152K）在每个Decode步的计算量也更大。综合来看，中文场景下大词表的收益（减少50%推理步数）远超其额外计算开销。
    关联知识: 词表大小与GQA/MLA等注意力变体也有关系——更大的词表意味着每个序列更短，KV Cache总量减少，有利于缓解长上下文的内存压力。此外，词表大小还影响量化精度——当词表很大时，LM Head层的输出logits范围更大，INT8量化的精度损失可能更明显。Flash Attention的优化效果也与序列长度相关——大词表缩短序列后，Flash Attention带来的加速比可能降低，因为短序列的优化空间较小。
  4. 核心原理: 特殊Token是词表中保留的具有特殊控制功能的token，不对应任何实际文本内容，但在推理流程中起着至关重要的作用。主要包括：BOS（Begin of Sequence）标记序列开始，告诉模型"从这里开始处理"；EOS（End of Sequence）标记序列结束，告诉模型"停止生成"，这是控制生成终止的关键信号；PAD（Padding）用于将不同长度的序列填充到相同长度以便批处理；UNK（Unknown）表示未登录词（现代BPE/Byte-level方案已基本消除此问题）。此外还有ChatML模板中的特殊token，如<|im_start|>和<|im_end|>用于标记多轮对话中不同角色（system/user/assistant）的边界。这些特殊token在推理中直接影响生成质量和系统行为——例如不正确处理EOS会导致模型无法停止生成（"无限生成"问题），不正确处理BOS会导致模型无法理解对话上下文的开始位置。
    关键要点: ①BOS标记序列开始，EOS控制生成终止 ②PAD用于批处理对齐，UNK处理未知词（现代方案已少用） ③ChatML等模板用特殊token标记对话角色边界 ④不正确处理EOS会导致无限生成问题 ⑤特殊token的ID和处理方式因模型而异，切换模型时必须检查
    实际案例: 以OpenAI ChatML格式为例，一次多轮对话的token序列为：<|im_start|>system\n你是一个助手<|im_end|><|im_start|>user\n你好<|im_end|><|im_start|>assistant\n。模型看到assistant标记后开始生成，当生成到<|im_end|>时停止。如果推理引擎没有正确实现EOS停止逻辑，模型会在生成完回答后继续"自言自语"生成虚假的user和assistant对话，造成严重的安全问题。这就是为什么推理框架必须严格检查特殊token的处理。
    关联知识: 特殊Token的设计直接影响Prefix Caching的效果——系统提示（system prompt）的特殊token边界决定了缓存粒度。如果两个请求共享相同的system prompt，它们的BOS到第一个<|im_end|>之间的token序列完全相同，可以复用KV Cache，这就是SGLang RadixAttention的核心优化。此外，特殊token在投机解码中也有作用——draft model生成候选token序列时，需要在EOS处特殊处理以确保不会跳过停止信号。
  5. 核心原理: 推理中的Tokenizer优化主要从三个维度展开。第一是减少token数量：选择对目标语言/领域分词效率更高的Tokenizer，如中文场景优先选择Qwen（152K词表）而非Llama 2（32K词表），可以减少50%以上的token数，直接减少Decode步数。还可以通过prompt压缩技术减少输入token数。第二是加速Tokenizer本身的执行：Tokenizer编码/解码操作在CPU上运行，对于短文本开销可忽略，但对于超长输入（如10万字文档），分词本身可能耗时数十毫秒到数秒。优化手段包括使用Rust/C++实现的高性能Tokenizer库（如HuggingFace Tokenizers的Rust后端）、预编译词表的Trie树结构加速匹配、以及批量化处理多个请求的编码操作。第三是LM Head层优化：词表越大，最后一层logits计算越慢。优化方法包括LM Head与最后Linear层的融合、使用Top-K/Top-P截断减少softmax计算范围、以及将LM Head计算与采样过程融合。此外，对大词表可以使用分层softmax或product quantization近似替代全词表softmax，将O(V)计算降低到O(log V)。
    关键要点: ①选择高效Tokenizer可减少50%以上token数 ②Rust/C++实现可大幅加速Tokenizer执行 ③LM Head层是词表大时的计算瓶颈 ④Top-K截断和分层softmax可优化logits计算 ⑤Tokenizer优化贯穿编码、模型计算、解码三个环节
    实际案例: 以vLLM中的实际优化为例。vLLM使用Rust实现的Tokenizer（tokenizers库），单线程编码速度可达每秒数十万token，远快于Python实现。在LM Head优化方面，vLLM对LM Head输出先做Top-K=50的截断，只对概率最高的50个token计算精确概率，而不是对全部128K token做softmax，将softmax计算量降低了2560倍。在批处理场景下，多个请求的Tokenizer编码操作可以并行执行，使用padding对齐后批量送入模型，减少CPU-GPU同步等待。
    关联知识: Tokenizer优化与KV Cache管理密切相关——更高效的Tokenizer产生更短的序列，直接减少KV Cache内存占用，缓解内存墙压力。Tokenizer优化还与Prompt Caching关联——相同的system prompt经过相同的Tokenizer产生相同的token序列，可以跨请求复用KV Cache。此外，Tokenizer的选择还影响量化方案——不同Tokenizer产生的token分布不同，对量化精度的影响也不同。在边缘设备部署（如手机端llama.cpp）中，Tokenizer的CPU开销占比更为显著，需要特别优化。
