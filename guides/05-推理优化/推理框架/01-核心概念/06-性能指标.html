1.1 TTFT (Time to First Token)

从请求发送到收到第一个输出 token 的时间 = Prefill 耗时

  • 用户感知的"等待时间"，最关键体验指标
  • 取决于：prompt 长度、模型大小、GPU 算力
  • 典型值：100ms - 5s

1.2 TPOT (Time Per Output Token)

生成每个输出 token 的平均时间 = Decode 速度

  • 影响用户看到"文字流出"的速度
  • 取决于：模型大小、GPU 带宽、batch size
  • 典型值：10-50ms/token

1.3 TPS (Tokens Per Second)

每秒生成的 token 数 = 1000 / TPOT

  • 人类阅读速度：~5 tokens/s
  • 理想 TPS：> 30 tokens/s（流畅体验）

1.4 E2E Latency (端到端延迟)

E2E Latency = TTFT + TPOT × num_output_tokens
请求时间线: |-- TTFT --|--- TPOT ---|--- TPOT ---|--- TPOT ---| | Prefill | Token 1 | Token 2 | Token 3 | | 200ms | 30ms | 30ms | 30ms | E2E Latency = 200 + 30×3 = 290ms (3 tokens) E2E Latency = 200 + 30×100 = 3200ms (100 tokens)

3.1 GPU 利用率 (Compute Utilization)

GPU 实际用于计算的时间占比。Decode 阶段通常只有 5-20%。

3.2 显存利用率 (Memory Utilization)

已用显存 / 总显存。理想值 90%+（留少量余量）。

3.3 MFU (Model FLOPS Utilization)

MFU = 实际 FLOPS / GPU 峰值 FLOPS

训练时 MFU 可达 40-60%，推理时 Prefill 阶段可达 30-50%，Decode 阶段通常 < 10%。

5. 指标关系总结

指标关系总结

深入探讨：从系统设计的底层原理看，核心延迟指标的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

LLM 推理框架系统学习指南 · 模块 01 · 2026
核心延迟指标的技术原理与数学推导： 从底层硬件角度分析，核心延迟指标的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了核心延迟指标的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
核心延迟指标的系统架构分析： 从系统设计的原理出发，核心延迟指标采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

核心延迟指标的性能优化策略

优化核心延迟指标性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

# 核心延迟指标 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

✅ 参考答案与详解
  1. 核心原理: 核心延迟指标体系包含四个维度：TTFT（首 token 延迟）、TPOT（每 token 延迟）、TPS（每秒 token 数）和 E2E Latency（端到端延迟）。TTFT 衡量从请求发送到收到第一个输出 token 的时间，本质上是 Prefill 阶段的耗时，因为 Prefill 阶段会并行处理所有输入 token，其计算量与 prompt 长度成正比。TPOT 衡量 Decode 阶段生成每个 token 的平均时间，受模型大小和 GPU 显存带宽制约（Decode 是内存受限操作）。TPS = 1000/TPOT，人类阅读速度约5 tokens/s，流畅体验需 TPS > 30。E2E Latency = TTFT + TPOT × num_output_tokens，是用户感知的总等待时间。这四个指标形成了从局部到整体的完整延迟画像。
    关键要点: ① TTFT 反映 Prefill 效果，是用户感知等待时间的关键指标；② TPOT 反映 Decode 速度，受显存带宽限制（典型10-50ms/token）；③ TPS = 1000/TPOT，>30 tokens/s 为流畅体验；④ E2E Latency = TTFT + TPOT × output_length，随输出长度线性增长；⑤ Prefill 是计算受限，Decode 是内存受限，两者瓶颈不同。
    实际案例: 部署 LLaMA-3.1-8B 模型时，用户输入一个200 token 的 prompt，测量 TTFT=200ms（Prefill 阶段），TPOT=30ms/token。若模型输出100个 token，则 E2E Latency = 200 + 30×100 = 3200ms（3.2秒）。用户首先等待200ms 看到第一个字，然后以33 tokens/s 的速度"流式"看到后续内容，体验流畅。若 prompt 增长到2000 token，TTFT 可能升至2秒，用户明显感到"卡顿"。
    关联知识: TTFT 与 Prefill 阶段的 KV Cache 计算、Flash Attention 优化密切相关；TPOT 受 Continuous Batching 和 KV Cache 管理策略影响；这些指标是推理框架（如 vLLM、TGI）选型和调优的核心依据，与后续"推理优化"章节的量化、剪枝等技术直接关联。
  2. 核心原理: 吞吐量指标衡量推理系统的"产能"，分为 Request Throughput（每秒完成请求数，requests/s）和 Token Throughput（每秒处理 token 数，tokens/s）。吞吐量与延迟之间存在根本性权衡：增大 batch size 可以提高吞吐量（因为多请求共享模型参数的读取，提升算术强度和 GPU 利用率），但同时会增加每个请求的排队等待和计算时间，导致延迟上升。在实时对话场景中优先保证低延迟（小 batch），在批量处理场景中优先最大化吞吐量（大 batch），在 API 服务场景中需要在两者之间找到满足 SLA 的平衡点。
    关键要点: ① Request Throughput 衡量系统服务请求的能力；② Token Throughput 衡量系统处理 token 的总产能；③ 吞吐量与延迟是此消彼长的权衡关系；④ 增大 batch size 提升吞吐但增加延迟；⑤ 不同场景对两者的优先级不同（实时对话→低延迟，批处理→高吞吐）。
    实际案例: 使用 vLLM 部署 LLaMA-3.1-8B，测试 ShareGPT 数据集。当 batch_size=1 时，TPS=34.8 tokens/s，Request Throughput=4.42 req/s，延迟低但 GPU 利用率仅5-20%。当 batch_size 增大到32时，Token Throughput 可提升至约2000 tokens/s（提升3倍），但 TTFT 从185ms 升至800ms+。对于实时聊天应用，应控制 batch_size 保持 TTFT < 500ms；对于夜间批量摘要任务，则可放开 batch size 最大化吞吐。
    关联知识: 吞吐量优化与 Continuous Batching（vLLM 的 PagedAttention）、Dynamic Batching 策略密切相关；Token Throughput 的计算方式（含/不含输入 token）会影响指标含义；与资源效率指标（GPU利用率、MFU）联动分析，可以定位吞吐量瓶颈是在计算还是内存。
  3. 核心原理: 资源效率指标包括 GPU 利用率（Compute Utilization）、显存利用率（Memory Utilization）和 MFU（Model FLOPS Utilization）。GPU 利用率衡量 GPU 实际用于计算的时间占比，Decode 阶段通常只有5-20%，因为每个 token 只需一次矩阵-向量乘法，大量时间花在等待显存数据加载上。显存利用率 = 已用显存/总显存，理想值90%+，剩余留给系统开销。MFU = 实际 FLOPS / GPU 峰值 FLOPS，是衡量模型实际利用硬件能力的核心指标。训练时 MFU 可达40-60%（大 batch + 大矩阵乘法），推理 Prefill 阶段可达30-50%，而 Decode 阶段通常低于10%，根本原因是 Decode 的算术强度（FLOPs/Byte）远低于 GPU 的算术强度转折点（A100 约156 FLOPs/Byte），属于典型的内存受限操作。
    关键要点: ① GPU 利用率在 Decode 阶段极低（5-20%），因为是内存受限操作；② 显存利用率目标90%+，剩余留给系统开销；③ MFU 是衡量硬件利用效率的终极指标；④ Prefill 和 Decode 的 MFU 差异巨大（30-50% vs <10%）；⑤ 屋顶模型（Roofline Model）可判断任务是计算受限还是内存受限。
    实际案例: 在 A100-80GB 上运行 LLaMA-70B 推理时，Prefill 阶段处理1024 token 的 prompt，GPU 利用率可达45%，MFU 约35%；进入 Decode 阶段后，GPU 利用率骤降至8%，MFU 仅约5%。这意味着 A100 的312 TFLOPS 算力中，Decode 阶段实际只用了约15 TFLOPS。通过增大 batch size（如同时处理32个请求），可以将 Decode 的 GPU 利用率提升至50%+，MFU 提升至15-20%，因为多个请求共享权重的读取，提高了算术强度。
    关联知识: 资源效率指标与屋顶模型（Roofline Model）直接相关——算术强度转折点决定任务是计算受限还是内存受限；GPU 利用率低的优化手段包括算子融合（减少中间结果写回）、Continuous Batching（提升 Decode 阶段 batch size）、量化（减少数据搬运量）；MFU 的提升是推理框架设计的核心目标，与 PagedAttention、Speculative Decoding 等技术的动机直接相关。
  4. 核心原理: 基准测试方法是量化评估推理系统性能的标准化流程。核心要素包括：选择代表性数据集（如 ShareGPT 模拟真实对话分布）、设置请求速率（request_rate 模拟并发负载）、运行足够数量的请求（通常200+以确保统计显著性）、收集关键指标（TTFT、TPOT、TPS、Throughput）。vLLM 的 benchmark_serving.py 是业界常用工具，它模拟真实用户请求模式，测量端到端的服务性能。基准测试的关键原则是：①使用真实分布的数据（避免均匀长度的 synthetic 数据）；②控制变量（同一模型、同一硬件、同一数据集）；③多次运行取均值和分位数；④报告 P50/P99 延迟而非仅平均值。
    关键要点: ① 使用真实分布数据集（如 ShareGPT）而非 synthetic 数据；② 控制 request_rate 模拟不同负载场景；③ 运行足够请求数（200+）确保统计可靠性；④ 关注 P50/P99 延迟，而非仅平均值；⑤ 标准化测试流程便于跨框架对比。
    实际案例: 使用 vLLM benchmark_serving.py 对 LLaMA-3.1-8B 进行基准测试：以 request_rate=10（每秒10个请求）发送200个来自 ShareGPT 的请求，输入平均214 token，输出平均156 token。结果：Request Throughput=4.42 req/s，Output Token Throughput=690.3 tok/s，Mean TTFT=185.3ms，Mean TPOT=28.7ms，Mean TPS=34.8 tok/s。通过将 request_rate 从10提升到50，可观察到 TTFT 和 TPOT 的变化趋势，找到系统性能拐点。
    关联知识: 基准测试结果直接指导推理框架选型（vLLM vs TGI vs TensorRT-LLM）；测试中发现的瓶颈（如 TTFT 过高→Prefill 优化、TPOT 过高→Decode 优化、Throughput 不足→Batch 策略优化）对应不同的优化路径；基准测试应与 Roofline 分析结合，判断性能瓶颈是计算受限还是内存受限，从而选择正确的优化方向。
  5. 核心原理: 指标关系总结揭示了推理系统性能指标之间的内在联系和权衡关系。核心关系包括：①延迟-吞吐量权衡：增大 batch size 提升吞吐但增加延迟，存在一个"平衡点"；②延迟内部关系：E2E Latency = TTFT + TPOT × output_length，TTFT 和 TPOT 可独立优化；③资源-性能关系：GPU 利用率和 MFU 反映硬件利用效率，低利用率意味着有优化空间；④阶段差异：Prefill 是计算受限（高 MFU），Decode 是内存受限（低 MFU），需要不同的优化策略。理解这些关系的关键是屋顶模型（Roofline Model）——当实际算术强度低于转折点（A100: 156 FLOPs/Byte, H100: 295 FLOPs/Byte）时，任务是内存受限的，增加计算单元无法提升性能，应优化数据搬运。
    关键要点: ① 延迟与吞吐量是核心权衡，需根据场景选择平衡点；② Prefill 和 Decode 的性能特征截然不同（计算受限 vs 内存受限）；③ 屋顶模型是判断瓶颈类型的理论基础；④ TTFT、TPOT、GPU Util、MFU 需联动分析而非孤立看待；⑤ 不同场景（实时对话/批处理/API服务）对指标优先级不同。
    实际案例: 综合分析一个推理服务的指标：TTFT=200ms（可接受），TPOT=30ms（流畅），但 GPU Util 仅15%（极低），MFU=8%（极低）。这说明系统处于"低负载/低效率"状态——虽然用户体验尚可，但硬件资源严重浪费。通过增大 batch size（Continuous Batching），可将 GPU Util 提升至60%、MFU 提升至25%，同时保持 TTFT < 500ms、TPOT < 40ms，在不牺牲用户体验的前提下将吞吐量提升4倍，大幅降低单请求成本。
    关联知识: 指标关系总结是推理系统调优的方法论基础——它将 TTFT/TPOT/Throughput/GPU Util/MFU 等孤立指标整合为一个分析框架；与 Continuous Batching、PagedAttention、Speculative Decoding 等优化技术的选型直接相关；是后续"推理优化"模块中量化、剪枝、蒸馏等技术决策的量化依据。
向前串联： 核心延迟指标的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
