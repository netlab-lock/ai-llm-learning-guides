1. Transformer 的核心组件

Transformer 的核心组件

ng>通俗类比：Transformer像一座多层办公楼：自注意力是每层的‘圆桌会议室’，让信息自由交流；前馈网络是独立的‘分析师工位’，对信息做深度加工。每一层都这样‘开会-分析’。
常见误区：初学者常混淆‘层’与‘组件’。一层Transformer包含自注意力和前馈网络两个子层，前者负责上下文关联，后者负责非线性变换，二者缺一不可。
深入探讨：以GPT-3为例，它拥有96层Transformer。每一层中的自注意力子层包含96个注意力头。模型总参数量达1750亿，其中大部分存储于各层FFN的权重矩阵中。

2. Self-Attention 机制详解

通俗类比：自注意力就像一个高效的会议室：每个词（成员）都会生成‘询问’(Query)、‘提供资料’(Key)和‘实际观点’(Value)。通过Q与所有K的匹配度，来决定最终吸收多少来自各个词的V。
常见误区：容易忽略对注意力分数的缩放（除以根号d_k）。这防止了因维度d_k过大导致的点积结果过大，使softmax梯度饱和，从而导致训练困难。
深入探讨：计算公式为：Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V。对于序列长度为L，head维度为d_k的输入，计算复杂度为O(L^2 * d_k)。这就是长序列计算成本高的根源。

Attention 是 Transformer 的核心，也是推理优化的重点：

2.1 标准 Self-Attention 计算

Attention(Q, K, V) = softmax(QK^T / √d_k) × V

计算步骤分解：

  1. Q、K、V 投影：将输入分别乘以三个权重矩阵 W_q, W_k, W_v
  2. 注意力分数：Q × K^T，得到 [seq_len, seq_len] 的矩阵
  3. 缩放：除以 √d_k，防止梯度消失
  4. Softmax：归一化为概率分布
  5. 加权求和：用注意力权重对 V 加权

3. FFN（前馈网络）

优化决策。

FFN（前馈网络）

通俗类比：FFN像是一个生产线上的‘质检与精加工’环节：自注意力层混合了上下文信息后，FFN对每个位置（词）独立地进行两次线性变换和一次非线性激活，提取更高级的特征模式。
常见误区：误以为FFN对序列全局建模。FFN是逐位置（position-wise）独立的，即序列中每个token的FFN计算互不影响，这与自注意力层的全局交互形成鲜明对比。
深入探讨：标准FFN结构为：FFN(x) = max(0, xW1 + b1)W2 + b2。其中W1将模型维度d_model（如768）升维到d_ff（如3072，通常是4倍），W2再降回d_model。这带来了主要的参数量。
[输入x] -> [线性层W1, b1] -> [ReLU/GELU] -> [线性层W2, b2] -> [输出]
(d_model)      (d_ff)                     (d_ff)       (d_model)
激活函数公式特点
ReLUmax(0, x)简单，可能产生‘死神经元’
GELUx * Φ(x)平滑，性能更佳，主流选择
SILU/Swishx * sigmoid(x)非单调，类似GELU

每个 Transformer Block 的 FFN 通常包含模型 2/3 的参数：

FFN 结构 (以 Llama 为例): 输入 x: [batch, seq_len, d_model] │ ├──→ W_gate (d_model → d_ffn) ──→ SiLU 激活 │ │ └──→ W_up (d_model → d_ffn) ─────────┤ │ 逐元素相乘 │ ▼ W_down (d_ffn → d_model) │ ▼ 输出: [batch, seq_len, d_model] 其中 d_ffn = 2/3 × 4 × d_model (SwiGLU 变体) Llama-7B: d_model=4096, d_ffn=11008

3.1 推理中的 FFN 特点

  • 参数量大：FFN 占模型参数的 ~2/3，是量化的主要目标
  • 计算密集：Prefill 阶段 FFN 是主要计算量
  • MoE 变体：Mixtral 等模型用多个专家 FFN + 路由，只激活部分专家

4. 模型规模与计算量

模型规模与计算量

通俗类比：模型规模好比汽车的排量和重量：参数量（Params）像发动机排量，决定了模型的‘理论能力上限’；计算量（FLOPs）像油耗和行驶阻力，决定了实际处理数据时需要消耗多少计算资源（时间和电费）。
常见误区：不要只关注参数量，而忽略推理时的计算量（FLOPs）。一个参数少但序列长的模型，其计算成本可能远超一个参数多但序列短的模型。二者需权衡。
深入探讨：GPT-3 (175B) 的一次前向传播，对于长度为1024的序列，约需3.14 * 10^14 FLOPs。训练它消耗了约3.14 * 10^23 FLOPs，相当于在上千个GPU上运行数月。
计算量 FLOPs ≈ 2 * 参数量 * 序列长度

5. 不同架构变体

通俗类比：Transformer架构变体如同不同菜系：Encoder-Only（如BERT）是‘凉菜师傅’，擅长理解；Decoder-Only（如GPT）是‘主菜大师’，擅长生成；Encoder-Decoder（如T5）是‘全能套餐’，擅长序列到序列任务。
常见误区：没有‘万能’架构。选择取决于任务：理解用BERT类，生成用GPT类，翻译/摘要用T5类。盲目追求最新架构可能不如选择合适架构有效。
深入探讨：Decoder-Only（GPT）采用因果掩码（Causal Mask），确保每个位置只能关注之前的位置。Encoder-Only（BERT）采用双向掩码。这从根本上决定了它们的训练和推理方式。
Encoder-Only: [输入] => [双向Transformer堆叠] => [输出]
Decoder-Only: [输入] => [带掩码的单向Transformer堆叠] => [输出]
Enc-Dec:      [输入] => [Encoder] => [Decoder] => [输出]
架构类型代表模型掩码方式擅长任务
Encoder-OnlyBERT, RoBERTa双向（无掩码）分类、NER、理解
Decoder-OnlyGPT, LLaMA, PaLM因果（向左看）文本生成、续写
Encoder-DecoderT5, BARTEnc双向，Dec因果翻译、摘要、问答

5.1 仅解码器（Decoder-Only）——当前主流

  • GPT 系列、Llama、Qwen、Mistral 等
  • 特点：单向注意力（causal mask），自回归生成
  • 推理优化最成熟，本指南主要覆盖此类

5.2 编码器-解码器（Encoder-Decoder）

  • T5、BART 等
  • 特点：双向编码 + 单向解码
  • 推理时需要分别处理编码和解码阶段

5.3 混合专家（MoE）

  • Mixtral、DeepSeek-V2/V3、Qwen MoE
  • 特点：多个专家 FFN，每次只激活 top-k 个
  • 推理挑战：专家路由、负载均衡、All-to-All 通信

6. 代码示例：一次前向传播

代码示例：一次前向传播

前向传播就像按食谱做菜：输入张量是‘食材’，经过一层层‘处理’（线性变换、激活函数、注意力），最终输出结果张量。每一层的处理都遵循固定的‘配方’（模型结构）。

常见误区：初学者最常犯的错误是张量形状不匹配。务必时刻关注输入输出的维度（如 batch_size, seq_len, d_model），这是调试深度学习代码的关键。
深入探讨：在PyTorch中，一个标准的TransformerLayer调用大致为：`output = self.attn(x, x, x)[0]; output = self.ffn(output)`。注意注意力层的输入Q、K、V通常都是同一个x（自注意力）。
import torch

# 假设 x 形状为 (batch=4, seq_len=10, d_model=512)
x = torch.randn(4, 10, 512)
# 通过一层Transformer
layer = TransformerLayer(d_model=512, nhead=8)
out = layer(x) # 输出形状仍为 (4, 10, 512)

多头注意力MHA(Q, K, V)(B, L, d) -> (B, L, d)
FFNLinear -> Act -> Linear(B, L, d) -> (B, L, d_ff) -> (B, L, d)

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

# 加载模型
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.1-8B",
    torch_dtype=torch.float16, device_map="auto")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.1-8B")

# 编码输入
inputs = tokenizer("The capital of France is", return_tensors="pt").to("cuda")

# 一次前向传播 = Prefill 阶段
with torch.no_grad():
    outputs = model(**inputs)
    # outputs.logits shape: [1, seq_len, vocab_size]
    next_token = outputs.logits[:, -1, :].argmax(dim=-1)
    # → token_id for " Paris"

# Decode 阶段（逐 token）
for i in range(20):
    with torch.no_grad():
        outputs = model(next_token.unsqueeze(0))
        next_token = outputs.logits[:, -1, :].argmax(dim=-1)
    print(tokenizer.decode(next_token), end="")
⚠️ 注意：上面的代码是为了演示原理，实际生产中请使用 vLLM、TGI 等框架，它们已经实现了 KV Cache 管理、连续批处理等优化。
