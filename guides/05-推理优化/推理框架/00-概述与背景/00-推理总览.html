🔍 LLM 推理总览

本节目标：理解什么是 LLM 推理、为什么推理比训练更"烧钱"、推理系统的核心挑战是什么。

1. 什么是 LLM 推理？

什么是 LLM 推理？

>通俗类比：LLM推理就像在餐厅点菜吃饭。训练是厨师花数月学习并掌握烹饪技艺（成本高昂且一次性）。推理是厨师为你现场烹饪和上菜（每次服务都产生成本，但更贴近用户价值）。
常见误区：最常见的误区是将训练和推理混为一谈。训练是“一次性”创造模型的过程，而推理是模型被反复使用、持续产生计算和内存成本的“现在进行时”过程。
深入探讨：以GPT-3（175B参数）为例，单次前向推理（生成约100个token）需约3.5x10^14次浮点运算（FLOPs）。在A100 GPU上，理论峰值约312 TFLOPS，但受内存带宽瓶颈限制，实际利用率常低于50%。生成一个token的延迟（TTFT + 每token延迟）是核心体验指标。

LLM 推理（Inference）是指使用已训练好的大语言模型对输入文本生成响应的过程。简单来说：

  • 训练（Training）：教模型"知识"——用海量文本数据调整数十亿参数
  • 推理（Inference）：让模型"干活"——用学到的参数回答用户问题

2. 推理的成本真相

推理的成本真相

通俗类比：推理成本像餐厅的“水电煤气费+厨师工资”，是持续消耗的。虽然单次做饭成本低，但客流量大（高QPS）时，总成本（电费、机器折旧、人力）会变得惊人，远超“学艺”（训练）成本。
常见误区：最大的成本陷阱是只关注GPU采购价，而忽视了“总拥有成本（TCO）”中的电力、冷却、运维和数据中心租金。一个未优化的推理服务，其长期电费可能超过硬件本身。
深入探讨：推理成本主要由三部分驱动：1) 模型FLOPs（计算量）；2) 内存带宽（数据搬运）；3) 批量大小（Batch Size）带来的硬件利用率。公式简估：单请求成本 ≈ (模型参数量 x 2 x Token数) / (GPU峰值算力 x 利用率) x (电价 + 硬件摊销)。

很多人以为训练是 LLM 最贵的部分，但事实恰恰相反：

阶段成本占比时间特征硬件需求
训练~10-20%一次性，数周到数月数千张 GPU 集群
推理~80-90%持续性，7×24小时根据 QPS 弹性扩展
💡 直观比喻：训练就像建一座工厂（一次性投入大），推理就像工厂每天运营（持续消耗电费和人工）。对于 ChatGPT 这样的产品，推理成本每天可能高达数百万美元。

2.1 推理成本的三大来源

  1. GPU 计算：每次前向传播都需要大量矩阵运算，GPU 是最大开销
  2. 内存占用：模型参数 + KV Cache 占用大量显存，显存 = 钱
  3. 延迟要求：用户期望秒级响应，不能靠"慢慢算"来节省资源

3. 推理的两个阶段

的技术选择和优化决策。

推理的两个阶段

通俗类比：预填充阶段像厨师根据你的宴客需求（提示词）备齐所有食材（计算所有K-V），计算密集。解码阶段像按顺序一道道上菜（逐个生成token），内存带宽密集，是体验流畅度的关键。
常见误区：易犯错误是用处理“预填充”的配置去优化“解码”。例如，为追求大批次吞吐，可能使首个token延迟（TTFT）过长，用户失去耐心。二者需要不同的优化策略。
深入探讨：预填充（Prefill）: 并行处理所有输入token，计算复杂度为O(n²)，瓶颈在算力。解码（Decode）: 自回归，每步生成一个token，需反复读取整个KV缓存，瓶颈在内存带宽。KV缓存大小 ≈ 2 * 层数 * 头数 * 头维度 * 序列长度 * 精度字节数。
预填充 (并行计算)
[“今天” “天气”] -> [K1,V1], [K2,V2]
解码 (逐步生成)
“今天天气” -> (K1,V1),(K2,V2) + “很” -> (K3,V3)
阶段主要瓶颈优化方向
预填充 (Prefill)算力 (Compute-bound)提高算术强度，使用Flash Attention
解码 (Decode)内存带宽 (Memory-bound)增加批处理大小，量化压缩KV

4. 推理系统的核心挑战

通俗类比：一个优秀的推理系统要同时做好三件事：让顾客（用户）等得不耐烦（低延迟），让餐厅翻台率（吞吐量）高，同时老板（服务方）还要有钱赚（低成本）。这几乎是“不可能三角”。
常见误区：片面追求单个指标会损害整体。例如，为极致降低延迟而使用独占资源，会导致吞吐低下、成本飙升。系统设计必须在三者间找到业务场景的最优平衡点。
深入探讨：核心挑战可量化为：1) 延迟（Latency）: TTFT + TPOT * 生成长度。2) 吞吐（Throughput）: Tokens/秒 或 请求/秒。3) 成本（Cost）: $/百万Token。三者关系：吞吐 ↑ 通常通过增大批大小实现，但这会增加排队延迟（延迟 ↑），并因更好的硬件利用率而降低成本（成本 ↓）。

4.1 内存墙（Memory Wall）

现代 GPU 的算力增长速度远超显存带宽增长：

GPU算力 (TFLOPS)显存带宽 (TB/s)算力/带宽比
A100312 (FP16)2.0156
H100990 (FP16)3.35296
H200990 (FP16)4.8206

这意味着在 decode 阶段，GPU 大部分时间在等待数据从显存传输，而不是在计算。这就是为什么内存优化（KV Cache 量化、PagedAttention）如此重要。

4.3 长上下文的挑战

当上下文长度从 4K 增长到 128K 甚至 1M 时：

  • KV Cache 内存从 GB 级增长到 TB 级
  • Attention 计算量从 O(n²) 爆炸式增长
  • Prefill 时间从秒级增长到分钟级

4.4 多模态推理

当输入不再只是文本，还包括图像、音频、视频时：

  • 输入 token 数量剧增（一张图 ≈ 256-1024 tokens）
  • 需要视觉编码器等额外模型组件
  • 不同模态的计算特征不同，调度更复杂

5. 推理优化的全景

推理优化的全景

通俗类比：优化推理就像管理一个高效餐厅：优化菜单（模型架构）、改进厨房布局（计算图优化）、培训厨师（算子优化）、使用预制菜（KV缓存）、让快餐和正餐分流（请求调度）。
常见误区：避免“局部优化”思维。例如，仅优化算子速度，却忽略内存瓶颈或调度策略，整体提升可能微乎其微。必须进行端到端、分层次的优化。
深入探讨：优化全景从底向上分层：1) 硬件层：GPU选型、NVLink/互联。2) 算子层：融合算子（如Flash Attention）、内核优化。3) 模型层：量化（INT8/INT4）、剪枝、蒸馏。4) 系统层：批处理策略、连续批处理、KV缓存管理。5) 架构层：推测解码、MoE。

6. 本模块小结

本模块小结

常见误区：不要认为学完概念就掌握了推理优化。真正的挑战在于理解不同技术在不同场景（如长文本、高并发）下的适用性与权衡，避免纸上谈兵。
深入探讨：本模块核心数字回顾：单次推理FLOPs：~2 * 参数量 * Token数。瓶颈：预填充在算力，解码在带宽。KV缓存大小公式。成本公式：TCO = 硬件 + 能源 + 运维。优化金字塔：算子 -> 模型 -> 系统 -> 架构。
