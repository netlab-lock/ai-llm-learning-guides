llama.cpp 由 Georgi Gerganov 于 2023 年 3 月创建，最初目标是在纯 CPU 上以 4-bit 量化运行 Meta 的 LLaMA 模型。如今已发展为支持数十种模型架构、多后端加速（CUDA/Metal/Vulkan/SYCL）、完整 HTTP 服务器的全栈推理框架。

1.1 核心设计哲学

  • 零依赖、可移植：核心库仅需 C/C++ 编译器，无 Python/PyTorch 依赖
  • GGUF 自描述格式：单一文件包含模型权重、分词器、元数据，免去 HuggingFace 转换
  • 内存映射 (mmap)：通过操作系统虚拟内存实现零拷贝模型加载
  • 逐层 offload：灵活控制每一层放 CPU 还是 GPU，精准管理显存
  • 社区驱动量化：imatrix 量化、K-Quant 等社区创新持续提升量化质量

GGUF（GPT-Generated Unified Format）是 llama.cpp 的模型文件格式，取代了早期的 GGML/GGJT 格式。它是一个自描述的、可内存映射的二进制格式。

2.1 文件结构

// GGUF v3 文件布局
struct gguf_file {
    // ── 头部 ──
    uint32_t magic;          // 0x46475547 = "GGUF" (little-endian)
    uint32_t version;        // 格式版本号 (当前 v3)
    uint64_t n_tensors;      // 张量数量
    uint64_t n_kv;           // 元数据键值对数量

    // ── 元数据 KV 对 ──
    for (i = 0; i < n_kv; i++) {
        gguf_string  key;     // UTF-8 字符串 (长度 + 数据)
        uint32_t     type;    // GGUF_TYPE_*
        union value;          // 根据类型的值
    }

    // ── 张量信息 ──
    for (i = 0; i < n_tensors; i++) {
        gguf_string  name;    // 张量名如 "blk.0.attn_q.weight"
        uint32_t     n_dims;  // 维度数
        uint64_t[]   dims;    // 各维度大小
        uint32_t     type;    // GGML_TYPE_* (量化类型)
        uint64_t     offset;  // 相对于数据区起始的偏移
    }

    // ── 对齐填充 ──
    padding_to_alignment(32);

    // ── 张量数据 ──
    // 实际的量化权重数据，按 offset 索引
};

⚠️ 常见误区：GGUF 格式完全兼容所有版本

GGUF 虽然是向前兼容的格式，但不同版本的 llama.cpp 对 GGUF 的元数据字段支持程度不同。一个常见的坑是：用最新版 llama.cpp 转换的 GGUF 文件可能包含旧版不支持的元数据键（如新的 tokenizer 类型），导致旧版工具加载失败或行为异常。另一个误区是认为 GGUF 文件可以直接从 safetensors 转换——实际上某些自定义架构（如非标准的 MoE 路由）需要先修改 convert 脚本中的架构映射，否则张量名不匹配会导致转换后的模型输出全为垃圾。

2.4 K-Quant 原理

K-Quant 的核心思想：对模型中不同层使用不同的量化精度。敏感层（如第一层、最后一层、attention 的 Q/K 投影）使用更高位宽（Q5/Q6），不敏感层（中间 FFN）使用更低位宽（Q3/Q4）。具体分配方案通过在验证集上测量 perplexity 来确定。

// K-Quant 层分配示例 (32 层 LLaMA 模型)
// Q4_K_M 分配策略:
//   Layer  0:    Q6_K   (首层最敏感)
//   Layer  1-3:  Q5_K   
//   Layer  4-28: Q4_K   (中间层)
//   Layer 29-30: Q5_K   
//   Layer 31:    Q6_K   (末层最敏感)
//   Output:      Q6_K   (lm_head)

// imatrix 量化更进一步:
// 通过校准数据计算每个权重通道的重要性分数
// 重要性高的通道保留更多精度
// 需要先运行 llama-imatrix 工具生成 importance matrix

2.5 Q4_K_M 量化数学详解

Q4_K_M 的核心：将 FP16 权重以 32 个为一组进行非对称 4-bit 量化，每组保存一个 scale（FP16）和一个 bias/min（FP16），以及 32 个 4-bit 整数。配合 K-Quant 的层间混合精度分配，实现 4.8 bits/param 的平均位宽。

2.5.1 量化过程 (Quantization)

// Q4_K_M 量化: 以 32 个权重为一个 block
// 输入: float w[32] (FP16/FP32 权重)
// 输出: uint8_t q[16] (4-bit 量化值, 每字节存 2 个) + 2 个 FP16 元数据

// 步骤 1: 计算 block 的 min 值 (bias)
bias = min(w[0], w[1], ..., w[31]);    // 非对称量化的零点

// 步骤 2: 计算 scale
d = max(|w[i] - bias|) / 7.0;          // 4-bit 量化范围 [0, 7] (3-bit 用于 magnitude)
                                          // 实际 Q4_K 使用 [0, 15] 范围, 但 super-block 有额外 scale

// 步骤 3: 量化每个权重
for (i = 0; i < 32; i++) {
    w_q[i] = round((w[i] - bias) / d);  // 映射到 [0, 15] 整数
    w_q[i] = clamp(w_q[i], 0, 15);     // 钳位到 4-bit 范围
}

// 存储格式 (Q4_K block):
//   [d: FP16]      — 主 scale
//   [dmin: FP16]   — bias/min 值
//   [q: uint8 × 16] — 32 个 4-bit 量化值 (每字节 2 个)
//   总计: 4 + 16 = 20 bytes / 32 weights = 0.625 bytes/weight = 5 bits/weight

2.5.2 反量化过程 (Dequantization)

// 在矩阵乘法内核中，需要将量化权重恢复为浮点
// SIMD 优化的反量化: 同时处理多个 block

// 反量化公式:
w[i] = w_q[i] × d + dmin;               // 恢复原始浮点值

// 伪代码 (简化):
for (i = 0; i < 32; i++) {
    // 从 packed bytes 提取 4-bit 值
    if (i % 2 == 0)
        w_q = q[i/2] & 0x0F;           // 低 4 位
    else
        w_q = (q[i/2] >> 4) & 0x0F;    // 高 4 位

    w_fp16[i] = w_q * d + dmin;         // FP16 运算
}

// AVX-512 优化: 一次反量化 32 个权重
// 1. 加载 16 字节 packed 数据
// 2. VPSRLW + VPAND 拆分高低 4-bit
// 3. VPMOVZXWD 扩展到 32-bit
// 4. VCVTDQ2PS 转 FP32
// 5. VFMADD (fused multiply-add): w_q * d + dmin

2.5.3 Super-Block 结构 (Q4_K)

// Q4_K 实际使用两层 scale 结构:
// Super-block: 256 个权重 (8 个 sub-block × 32 weights)

struct block_q4_K {            // 总计 144 bytes / 256 weights
    half    d;                  // 2B: super-block scale
    half    dmin;               // 2B: super-block min
    uint8_t scales[12];        // 12B: 8 个 sub-block 的 6-bit scale (packed)
    uint8_t qs[128];           // 128B: 256 个 4-bit 量化值
};                             // 平均: 144/256 × 8 = 4.5 bits/weight
                               // Q4_K_M 通过层混合使平均达到 4.8 bits

// 反量化: w = (q × scale[sub_block] × d) + (dmin × dmin_scale)
// 两层 scale 提供更细粒度的精度控制

3.3 Flash Attention 支持

llama.cpp 从 2024 年起逐步集成了 Flash Attention 优化：

  • CUDA 后端：通过 GGML_USE_CUDA_FLASH_ATTN 编译选项启用
  • Metal 后端：原生支持 Flash Attention（Apple Silicon 优化）
  • CPU 后端：通过 GGML_USE_FLASH_ATTN 启用（效果有限）
  • 命令行：-fa 或 --flash-attn
⚠️ 常见误区：KV Cache 量化对所有任务影响相同

KV Cache 量化（Q8_0/Q4_0）对不同类型任务的影响差异很大。在短上下文（<2K tokens）的问答任务中，Q4_0 量化几乎无感知；但在长上下文推理（>8K tokens）、多轮对话、代码补全等场景中，Q4_0 会导致注意力权重精度下降，表现为长距离依赖丢失、代码括号匹配错误、重复生成等问题。建议对长上下文场景使用 Q8_0（精度损失 <0.1 PPL），仅在内存极端受限时才使用 Q4_0。

4. 后端架构详解

💡 后端抽象层的设计哲学：llama.cpp 的 GGML 后端抽象层采用策略模式——同一套张量运算接口，底层根据编译选项自动分发到不同的硬件后端（CPU/CUDA/Metal/Vulkan/SYCL）。这种设计的代价是无法针对每个后端做极致优化，但换来了无与伦比的可移植性：一个 C/C++ 项目覆盖从树莓派到 A100 的所有硬件。
⚠️ 后端选择的性能陷阱：不要以为编译了 CUDA 后端就一定比 CPU 快。对于小 batch（如单请求 decode），CPU 的 SIMD 向量化可能比 GPU 的 kernel launch 开销更高效。经验法则：batch size < 8 时 CPU 可能更快，batch size > 32 时 GPU 优势明显。这就是为什么 llama.cpp 支持逐层 offload——让敏感的层用 GPU，不敏感的层留在 CPU。

4.1 CPU 后端

4.3 Metal 后端 (Apple Silicon)

Metal 后端针对 Apple M 系列芯片深度优化：

  • 利用统一内存架构 (UMA)，CPU 和 GPU 共享内存，零拷贝
  • 内置 Flash Attention 实现
  • 支持 M 系列的矩阵乘法协处理器
  • -ngl 99 即可将全部计算 offload 到 GPU（无需手动管理显存/内存边界）

4.4 Vulkan 后端

Vulkan 是跨平台 GPU 计算 API，支持 NVIDIA/AMD/Intel/ARM GPU：

  • 编译：cmake -B build -DGGML_VULKAN=ON
  • 优势：一个 binary 支持所有 GPU 厂商
  • 劣势：相比 CUDA/Metal 有一定性能损失
  • 适用：嵌入式设备、Windows AMD GPU、无 CUDA 环境

5.3 mmap 底层 OS 原理

核心机制：mmap 并不立即读取文件数据，而是在进程虚拟地址空间建立与文件的映射关系。实际数据加载由操作系统按需通过 page fault 机制完成。

5.3.1 Page Fault 触发机制

// mmap 调用后，内核只修改页表，不加载数据
// 当 CPU 首次访问映射地址时，触发 page fault:

// 用户态:
mmap(NULL, file_size, PROT_READ, MAP_SHARED, fd, 0);
// 返回虚拟地址 ptr, 此时页表项全部标记为"未映射"

// 当执行: float val = *(float*)(ptr + offset);
// CPU 查页表 → 发现该页未映射 → 触发缺页异常

// 内核态处理流程:
1. CPU 保存现场, 切换到内核模式
2. 内核查找 VMA (vm_area_struct) → 找到该地址属于 mmap 区域
3. 分配一个物理页框 (page frame, 4KB)
4. 从磁盘读取对应的 4KB 数据到物理页框 (DMA)
5. 更新页表: 虚拟页 → 物理页框, 标记为 "已映射 + 只读"
6. 刷新 TLB, 恢复用户态执行 → 指令重新执行, 读取成功

// 后续访问同一页时: 页表命中, 直接访问, 零开销
// 典型 page fault 延迟: NVMe SSD ~5-20μs, SATA SSD ~50-100μs, HDD ~5-10ms

5.3.2 Readahead 预读策略

// Linux 内核对 mmap 有智能预读 (readahead) 机制:

// 当检测到顺序访问模式时，内核预读更多页面:
// 初始预读窗口: 128KB (32 页)
// 每次 page fault 后，异步预读后续页面
// 预读窗口指数增长: 128KB → 256KB → 512KB → ... → max_readahead

// 但 llama.cpp 的 tensor 访问模式:
// - 按层顺序访问 (第0层→第1层→...): 触发顺序预读 ✓
// - 每层内访问多个张量 (Q/K/V/O/FFN): 随机但局部 ✓
// - 总体是分段顺序访问，预读机制能较好适配

// llama.cpp 主动调用 madvise 优化:
madvise(ptr, file_size, MADV_SEQUENTIAL);   // 告诉内核: 顺序访问
madvise(ptr, file_size, MADV_WILLNEED);     // 告诉内核: 即将使用
// MADV_SEQUENTIAL → 内核加大预读窗口, 访问后尽快释放页面
// MADV_WILLNEED  → 内核立即开始预读指定范围

5.3.3 mmap vs read() 性能对比

为什么不总是用 mmap?
  • NFS/网络文件系统：page fault 延迟不可控（可能数百 ms），此时 --no-mmap + --mlock 更好
  • 需要确定性性能：page fault 引入不可预测的延迟抖动，实时场景可预加载
  • 写入场景：mmap MAP_SHARED 写入需要 msync() 刷盘，复杂度高

7.2 并发处理架构

7.2.1 Slot 架构详解

// llama-server 的核心并发模型: Slot 槽位架构
// 源码: server/server.cpp → struct server_slot

struct server_slot {
    int32_t     id;              // slot 编号 (0 ~ n_parallel-1)
    llama_seq_id seq_id;         // 对应的 KV Cache 序列 ID
    llama_pos   n_past;          // 当前已处理的 token 数
    int32_t     n_decoded;       // 已生成的 token 数
    int32_t     n_prompt_tokens; // prompt 的 token 数
    std::vector<llama_token> prompt_tokens;
    std::vector<llama_token> generated_tokens;

    // 状态机:
    //   IDLE → 正在等待新请求
    //   PROCESSING_PROMPT → 正在 prefill
    //   GENERATING → 正在 decode 逐 token
    //   FINISHED → 生成完毕, 等待回收

    bool is_idle() const;        // 是否空闲可接受新请求
    bool has_prompt_in_progress(); // 是否正在 prefill
};

// n_parallel (-np) 决定并发 slot 数
// 每个 slot 独占一段 KV Cache 空间
// KV Cache 总容量被均分: slot_size = ctx_size / n_parallel

7.2.2 请求调度策略

// 调度主循环: server.cpp → server_context::update_slots()

// 策略: FCFS (First Come First Serve) + 批处理合并

// 1. 检查是否有空闲 slot
for (auto& slot : slots) {
    if (slot.is_idle() && !queue.empty()) {
        slot.prompt_tokens = queue.pop_front();  // FCFS: 先到先服务
        slot.state = PROCESSING_PROMPT;
    }
}

// 2. Prefill 阶段: 合并多个 slot 的 prompt 并行计算
//    所有 slot 中正在 prefill 的 prompt 会合并到一个 batch
//    → 大矩阵乘, GPU 利用率高
batch = build_batch(slots_with_pending_prompt);

// 3. Decode 阶段: 每个 active slot 各自 decode 1 个 token
//    → 并行 decode: 3 个 slot = 1 次前向传播产出 3 个 token
for (auto& slot : slots) {
    if (slot.state == GENERATING) {
        batch.add(slot.last_token, slot.seq_id, slot.n_past);
    }
}

// 4. Continuous Batching: 新请求可随时加入空闲 slot
//    不必等所有 slot 都完成 → 低延迟 + 高吞吐

# KV Cache 量化: 大幅减少内存占用
./llama-cli -m model.gguf -c 32768 \
  -ctk q8_0 -ctv q8_0    # K/V Cache 都用 8-bit (内存减半)

./llama-cli -m model.gguf -c 65536 \
  -ctk q4_0 -ctv q4_0    # 更激进: 4-bit Cache (内存减为 1/4)
  -fa                     # 配合 Flash Attention

# 内存估算公式:
# KV Cache = 2 × n_layers × n_kv_heads × head_dim × ctx_len × dtype_size
# LLaMA-3-8B, ctx=32K, FP16: 2 × 32 × 8 × 128 × 32768 × 2 = 32 GB
# LLaMA-3-8B, ctx=32K, Q8_0: 2 × 32 × 8 × 128 × 32768 × 1 = 16 GB
# LLaMA-3-8B, ctx=32K, Q4_0: 2 × 32 × 8 × 128 × 32768 × 0.5 = 8 GB

// - 理论峰值: memory_bandwidth / model_bytes_per_param

Q1: GGUF 格式相比 HuggingFace safetensors 的优势是什么？为什么 llama.cpp 不直接用 safetensors？

A: GGUF 是自描述格式，单文件包含模型权重、分词器、架构配置元数据，无需额外的 config.json/tokenizer.json。safetensors 是纯权重格式，不包含量化元数据和分词器。GGUF 支持原生量化格式（Q4_K_M 等），safetensors 存储的是 FP16/BF16。GGUF 支持内存映射直接从磁盘加载，safetensors 虽然也支持 mmap，但缺少量化数据的 block 结构信息。此外，GGUF 的 tensor info 区域记录了精确的 offset，使得 mmap 后可以直接通过指针访问任意张量，无需反序列化。

Q2: llama.cpp 的 K-Quant 量化（如 Q4_K_M）和 GPTQ/AWQ 有什么本质区别？

A: 本质区别在于量化粒度和校准方法。GPTQ/AWQ 是基于 Hessian 矩阵的逐层量化，需要校准数据集，量化后仍是 FP16 权重经过 transformation。K-Quant 是静态的、基于规则的混合精度量化：对不同层按预设方案分配不同位宽（敏感层高精度，不敏感层低精度），不依赖校准数据。imatrix 量化则介于两者之间，使用校准数据计算重要性矩阵但不修改权重。GPTQ/AWQ 通常在 GPU 上有专门的 kernel 加速（如 Marlin kernel），而 K-Quant 在 CPU 上更高效。

Q3: mmap 加载模型时，如果系统内存不足会怎样？如何优化？

A: 当物理内存不足时，OS 会触发页面换出：将不活跃的页面写回磁盘（对于 mmap 的只读文件，直接丢弃，需要时从文件重新加载）。这会导致page fault 延迟，表现为推理时偶尔出现延迟尖峰。优化方案：(1) --mlock 将所有页面锁定在物理内存（需要足够 RAM）；(2) 使用更小的量化版本减少文件大小；(3) 将模型放在 NVMe SSD 上（page fault 延迟从 HDD 的 ~10ms 降到 SSD 的 ~0.1ms）；(4) 预热：首次运行时用 vmtouch -t model.gguf 预加载到页面缓存。

Q4: 为什么 llama.cpp 的并发性能不如 vLLM？根本原因是什么？

A: 根本原因是内存管理策略不同。vLLM 的 PagedAttention 将 KV Cache 分成固定大小的 page（类似操作系统的虚拟内存页），通过 page table 动态管理，支持非连续内存分配、copy-on-write 和内存共享（如 beam search 的多个候选共享前缀 KV）。llama.cpp 的 KV Cache 是预分配的连续内存，每个 slot 占用固定空间，无法动态伸缩，内存利用率低。此外，vLLM 使用 CUDA Graph 和连续批处理深度优化 GPU 利用率，而 llama.cpp 更注重通用性和可移植性，在高并发场景下的 GPU 利用率不够充分。简单说，llama.cpp 优化的是单请求延迟，vLLM 优化的是系统吞吐。
