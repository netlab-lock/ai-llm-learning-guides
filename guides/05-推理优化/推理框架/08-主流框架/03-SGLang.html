2.2 Radix Tree 数据结构

SGLang 使用 Radix Tree（基数树 / Patricia Trie）管理所有请求的 KV Cache。每个节点存储一段 token 序列（而非单个字符），叶子节点指向 KV Cache 的物理存储。这使得最长前缀匹配成为 O(m) 操作（m = 查询序列长度）。

# === SGLang RadixCache 核心数据结构 ===
# 文件: sglang/srt/mem_cache/radix_cache.py
# 基于 LRU 策略的 Radix Tree KV Cache 管理器

from collections import OrderedDict
import torch
from typing import List, Optional, Tuple

class RadixNode:
    """Radix Tree 节点 — 存储一段 token 子序列及其对应的 KV Cache 物理块"""
    def __init__(self):
        self.children: OrderedDict[int, 'RadixNode'] = OrderedDict()
        # key: 存储这段前缀的 token ids（压缩路径，不是单个 token）
        self.key: List[int] = []
        # value: KV Cache 的物理块索引（GPU 显存中的实际位置）
        self.value: List[int] = []   # 块 ID 列表
        self.ref_count: int = 0      # 引用计数：>0 表示有活跃请求在用
        self.last_access: float = 0  # LRU 时间戳，驱逐时用
        self.parent: Optional['RadixNode'] = None
        self.locked: bool = False    # 是否被锁定（不可驱逐）

class RadixCache:
    """
    核心缓存管理器 — 基于 Radix Tree 的 KV Cache 复用引擎
    
    设计原则:
    1. 用 Radix Tree (Patricia Trie) 管理所有前缀的 KV Cache
    2. 支持最长前缀匹配 → O(m) 查找，m=查询长度
    3. LRU 驱逐策略 → 内存不足时释放最久未用的节点
    4. 引用计数 → 防止驱逐正在使用的 KV Cache
    """
    
    def __init__(self, req_to_token_pool, token_to_kv_pool, 
                 eviction_policy="lru"):
        self.req_to_token_pool = req_to_token_pool  # 请求→token 块映射池
        self.token_to_kv_pool = token_to_kv_pool    # token 块→KV 数据池
        self.root_node = RadixNode()                 # 虚拟根节点
        self.root_node.key = []                       # 根节点无 token
        self.eviction_policy = eviction_policy
        self.num_tokens = 0                          # 树中总 token 数
        self.num_nodes = 0                           # 树中总节点数

    # ==================== 核心方法 1: 最长前缀匹配 ====================
    def match_prefix(self, token_ids: List[int]) -> Tuple[RadixNode, int]:
        """
        在 Radix Tree 中查找与 token_ids 的最长公共前缀。
        
        算法:
        1. 从根节点开始，逐层查找匹配的子节点
        2. 对每个节点比较 key 与剩余 token_ids
        3. 如果完全匹配该节点的 key → 继续深入子树
        4. 如果部分匹配 → 分裂节点，返回匹配长度
        5. 如果不匹配 → 返回当前累积的匹配长度
        
        返回: (匹配到的最深节点, 匹配的 token 数)
        时间复杂度: O(m)，m = len(token_ids)
        """
        if not token_ids:
            return self.root_node, 0
            
        current_node = self.root_node
        remaining = list(token_ids)  # 还未匹配的 token
        total_matched = 0
        
        while remaining:
            # 步骤 1: 用第一个 token 的 hash 查找子节点 → O(1)
            first_token = remaining[0]
            if first_token not in current_node.children:
                break  # 无匹配子节点，停止
            
            child = current_node.children[first_token]
            child_key = child.key
            
            # 步骤 2: 计算 child_key 和 remaining 的最长公共前缀长度
            common_len = 0
            for i in range(min(len(child_key), len(remaining))):
                if child_key[i] == remaining[i]:
                    common_len += 1
                else:
                    break
            
            if common_len == 0:
                break  # 无公共前缀
            
            if common_len == len(child_key):
                # 完全匹配该节点 → 继续深入
                current_node = child
                remaining = remaining[common_len:]
                total_matched += common_len
                # 更新 LRU 时间戳
                current_node.last_access = time.time()
            else:
                # 部分匹配 → 需要分裂节点（见 insert 方法）
                total_matched += common_len
                break
        
        return current_node, total_matched

    # ==================== 核心方法 2: 插入/缓存 KV Cache ====================
    def insert(self, token_ids: List[int], kv_indices: List[int]) -> int:
        """
        将新的 token_ids + 对应的 KV Cache 块插入 Radix Tree。
        
        插入流程:
        1. 调用 match_prefix 找到最长匹配前缀
        2. 如果匹配到树中间 → 分裂现有节点
        3. 将剩余 token 创建为新叶子节点
        4. 返回复用的 token 数（已有的前缀部分）
        
        图示:
          插入 [t1,t2,t3,t4,t5,t6] 前已有 [t1,t2,t3,x,y]
          
          Before:  Root → [t1,t2,t3] → [x,y]
          After:   Root → [t1,t2,t3] → [x,y]
                                ↘ [t4,t5,t6]  (新分支)
        """
        if not token_ids:
            return 0
            
        # Step 1: 查找最长匹配
        match_node, match_len = self.match_prefix(token_ids)
        remaining_tokens = token_ids[match_len:]
        remaining_kv = kv_indices[match_len:]
        
        if not remaining_tokens:
            return match_len  # 完全复用，无需插入
        
        # Step 2: 需要分裂的情况
        # match_node 的 key 可能需要被拆分
        if match_len > 0 and match_len < sum(len(n.key) for n in self._trace_path(match_node)):
            self._split_node(match_node, match_len)
        
        # Step 3: 创建新节点存储剩余 token
        new_node = RadixNode()
        new_node.key = remaining_tokens
        new_node.value = remaining_kv  # 对应的 GPU 物理块
        new_node.ref_count = 1
        new_node.last_access = time.time()
        new_node.parent = match_node
        
        # 插入为 match_node 的子节点
        first_token = remaining_tokens[0]
        match_node.children[first_token] = new_node
        
        self.num_tokens += len(remaining_tokens)
        self.num_nodes += 1
        
        return match_len  # 返回复用的 token 数

    # ==================== 核心方法 3: 节点分裂 ====================
    def _split_node(self, node: RadixNode, split_pos: int):
        """
        分裂节点 — 当匹配在某节点 key 的中间位置时触发
        
        场景: node.key = [t1,t2,t3,t4,t5], 要在位置 3 处分裂
        
        Before:  parent → [t1,t2,t3,t4,t5]
        After:   parent → [t1,t2,t3] → [t4,t5]
        
        分裂后:
        - 前半段 [t1,t2,t3] 保留在原节点（修改 key/value）
        - 后半段 [t4,t5] 移到新中间节点
        - 原节点的子节点变成新中间节点的子节点
        """
        # 创建新中间节点，承接后半段
        mid_node = RadixNode()
        mid_node.key = node.key[split_pos:]
        mid_node.value = node.value[split_pos:]
        mid_node.parent = node
        mid_node.ref_count = node.ref_count
        mid_node.last_access = node.last_access
        mid_node.children = node.children  # 原来的子树挂在新节点下
        
        # 更新子节点的 parent 指针
        for child in node.children.values():
            child.parent = mid_node
        
        # 原节点只保留前半段
        node.key = node.key[:split_pos]
        node.value = node.value[:split_pos]
        node.children = OrderedDict()
        node.children[mid_node.key[0]] = mid_node
        
        self.num_nodes += 1

    # ==================== 核心方法 4: LRU 驱逐 ====================
    def evict(self, num_tokens_needed: int) -> int:
        """
        LRU 驱逐 — 当 GPU 显存不足时释放最久未访问的 KV Cache。
        
        驱逐策略:
        1. 按 last_access 排序所有叶子节点（最久未访问的排最前）
        2. 从最久未访问节点开始，跳过 ref_count > 0 的（活跃使用中）
        3. 释放 ref_count == 0 的节点的 GPU 物理块
        4. 合并空节点到父节点（维护树结构紧凑性）
        
        关键约束:
        - 只驱逐叶子节点（内部节点可能有正在使用的子树）
        - ref_count > 0 的节点绝对不驱逐（会导致推理错误）
        - 驱逐后尝试合并父节点（如果父节点也变成单子叶节点）
        
        返回: 实际释放的 token 数
        """
        evicted_count = 0
        
        # 收集所有叶子节点，按 last_access 排序
        leaves = []
        self._collect_leaves(self.root_node, leaves)
        leaves.sort(key=lambda n: n.last_access)  # 最久未访问排最前
        
        for leaf in leaves:
            if evicted_count >= num_tokens_needed:
                break
            
            # 跳过活跃节点
            if leaf.ref_count > 0 or leaf.locked:
                continue
            
            # 释放该节点的 KV Cache 物理块
            self._free_kv_blocks(leaf.value)
            evicted_count += len(leaf.key)
            
            # 从树中移除该节点
            parent = leaf.parent
            if parent:
                del parent.children[leaf.key[0]]
                self.num_tokens -= len(leaf.key)
                self.num_nodes -= 1
                
                # 尝试合并: 如果 parent 只剩 1 个子节点且 parent 非根
                self._try_merge(parent)
        
        return evicted_count

    def _try_merge(self, node: RadixNode):
        """
        合并优化: 当节点只剩一个子节点时，将子节点合并到当前节点
        
        Before:  parent → [t1,t2] → [t3,t4]  (中间节点只有1个子节点)
        After:   parent → [t1,t2,t3,t4]       (合并后路径更短)
        
        这保持了 Radix Tree 的 Patricia 性质 — 没有单子节点的中间节点
        """
        if node == self.root_node or node.ref_count > 0:
            return
        if len(node.children) != 1:
            return
            
        child = next(iter(node.children.values()))
        if child.ref_count > 0:
            return  # 子节点正在使用，不能合并
        
        # 合并: 将子节点的 key/value 追加到当前节点
        node.key = node.key + child.key
        node.value = node.value + child.value
        node.children = child.children
        
        # 更新孙子节点的 parent 指针
        for grandchild in child.children.values():
            grandchild.parent = node
        
        self.num_nodes -= 1

    def _collect_leaves(self, node, leaves):
        """递归收集所有叶子节点"""
        if not node.children:
            if node != self.root_node:
                leaves.append(node)
        else:
            for child in node.children.values():
                self._collect_leaves(child, leaves)

    def _free_kv_blocks(self, block_ids: List[int]):
        """将物理块归还到空闲池（释放 GPU 显存）"""
        for bid in block_ids:
            self.token_to_kv_pool.free(bid)

    # ==================== 核心方法 5: 引用计数管理 ====================
    def inc_ref(self, node: RadixNode):
        """增加引用计数 — 当请求开始使用该节点的 KV Cache 时"""
        node.ref_count += 1
        # 递归增加所有子节点的引用
        for child in node.children.values():
            self.inc_ref(child)

    def dec_ref(self, node: RadixNode):
        """减少引用计数 — 当请求完成时"""
        node.ref_count -= 1
        for child in node.children.values():
            self.dec_ref(child)
源码要点总结：
  • match_prefix 是最高频调用 — 每个新请求到达时都执行，时间复杂度 O(m)，m=token 数
  • _split_node 是 Radix Tree 区别于普通 Trie 的关键 — 支持在路径中间插入分支
  • evict 只驱逐叶子节点且 ref_count==0 — 保证不会破坏正在使用的缓存
  • _try_merge 维护 Patricia 性质 — 避免树退化为链表
  • 整个数据结构用 OrderedDict 存储子节点 — 保持插入顺序 + O(1) 哈希查找

3.1 有限状态机 (FSM) 构建过程

# === SGLang Constrained Decoding 核心实现 ===
# 文件: sglang/srt/constrained/fsm_cache.py + compressed_fsm.py
# 职责: 将 JSON Schema / Regex 编译为 Token-level 压缩 FSM

import interegular  # 正则表达式 → NFA/DFA 的 Python 库
import json
from typing import Dict, List, Set, Optional, Tuple
import numpy as np

# ==================== Step 1: JSON Schema → Regex ====================
def json_schema_to_regex(schema: dict) -> str:
    """
    将 JSON Schema 编译为正则表达式。
    
    核心思路:
    - object → {"key1": , "key2": }
    - string → "([^"\\\\]|\\\\.)*"  (带转义的双引号字符串)
    - integer → -?(0|[1-9][0-9]*)
    - array → \[((,)*)?\]
    
    示例: {"name": str, "age": int}
    → \{\\s*"name"\\s*:\\s*"([^"\\\\]|\\\\.)*"\\s*,\\s*"age"\\s*:\\s*-?(0|[1-9][0-9]*)\\s*\}
    """
    from outlines.fsm.json_schema import build_regex_from_schema
    return build_regex_from_schema(json.dumps(schema))

# ==================== Step 2: Regex → DFA (字符级) ====================
def regex_to_dfa(regex_str: str) -> interegular.fsm.FSM:
    """
    将正则表达式编译为确定性有限状态机 (DFA)。
    
    流程: regex → NFA (Thompson's) → DFA (子集构造法) → 最小化 DFA
    
    例子: regex "ab+c" →
      State 0 --a--> State 1 --b--> State 1 --c--> State 2 (accept)
    """
    pattern = interegular.parse_pattern(regex_str)
    nfa = pattern.to_fsm()      # Thompson's construction → NFA
    dfa = nfa.determinize()     # 子集构造法: NFA → DFA
    dfa = dfa.minimize()        # Hopcroft 最小化
    return dfa

# ==================== Step 3: 字符级 DFA → Token 级 Compressed FSM ====================
class CompressedFSM:
    """
    压缩有限状态机 — SGLang Constrained Decoding 的核心数据结构。
    
    将字符级 DFA 转换为 token 级 FSM，使得每步解码时：
    1. 直接根据当前 FSM 状态和候选 token 查表 → O(1)
    2. 返回合法 token 集合（bitset 表示）
    3. 用 bitmask 屏蔽不合法 token 的 logits
    
    关键优化:
    - 预计算所有 (state, token_id) → next_state 的转移表
    - 每个状态的合法 token 用 bitset 存储，GPU 上高效 apply mask
    """
    
    def __init__(self, dfa: interegular.fsm.FSM, tokenizer):
        self.dfa = dfa
        self.tokenizer = tokenizer
        self.num_states = len(dfa.states)
        self.vocab_size = tokenizer.vocab_size
        
        # 核心数据结构: 转移表
        # transition_table[state] = {token_id: next_state, ...}
        self.transition_table: Dict[int, Dict[int, int]] = {}
        
        # 每个状态的合法 token 集合 (bitset)
        # valid_tokens_bitset[state] = numpy uint8 array, shape=(vocab_size//8,)
        self.valid_tokens_bitset: Dict[int, np.ndarray] = {}
        
        # 接受状态集合
        self.accept_states: Set[int] = set(dfa.accepting)
        
        self._build_transition_table()
    
    def _build_transition_table(self):
        """
        预计算 Token 级转移表 — 一次性开销，缓存复用
        
        核心算法:
        对每个 DFA 状态 s 和词表中的每个 token t:
          1. 将 token t 解码为字符序列 chars
          2. 从状态 s 开始，逐字符转移 DFA
          3. 记录最终状态 s'
          4. 如果转移到死状态 → token t 在状态 s 下非法
          5. 否则: transition_table[s][t] = s'
        
        时间复杂度: O(|states| × |vocab| × avg_token_len)
        对于 LLaMA (vocab=32000, states=100): ~3-5 秒（一次性）
        """
        all_tokens = range(self.vocab_size)
        
        for state in self.dfa.states:
            self.transition_table[state] = {}
            valid_mask = np.zeros(self.vocab_size, dtype=np.uint8)
            dead_state = self.dfa.empty  # DFA 的死状态
            
            for token_id in all_tokens:
                # 将 token id 解码为字符序列
                token_str = self.tokenizer.decode([token_id])
                
                # 逐字符转移 DFA
                current_state = state
                is_valid = True
                
                for char in token_str:
                    # DFA 转移函数: δ(current_state, char)
                    next_state = self._dfa_transition(current_state, char)
                    
                    if next_state == dead_state:
                        is_valid = False
                        break
                    current_state = next_state
                
                if is_valid:
                    self.transition_table[state][token_id] = current_state
                    valid_mask[token_id] = 1  # 标记为合法 token
            
            # 将 valid_mask 压缩为 bitset (每 8 个 token 打包为 1 字节)
            self.valid_tokens_bitset[state] = np.packbits(valid_mask)
    
    def _dfa_transition(self, state: int, char: str) -> int:
        """单步 DFA 转移: δ(state, char) → next_state"""
        transitions = self.dfa.map.get(state, {})
        next_state = transitions.get(char, self.dfa.empty)
        return next_state
    
    def get_valid_tokens(self, state: int) -> np.ndarray:
        """
        获取指定状态的合法 token bitset
        
        返回: numpy uint8 数组，每个 bit 对应一个 token 是否合法
        形状: (vocab_size // 8,)  — 每字节存储 8 个 token 的合法性
        
        用途: 在 GPU 上直接用 bitwise AND 操作 apply logit mask
        """
        return self.valid_tokens_bitset.get(
            state, 
            np.zeros(self.vocab_size // 8, dtype=np.uint8)
        )
    
    def advance_state(self, state: int, token_id: int) -> int:
        """
        状态转移: 给定当前状态和采样到的 token，返回下一 FSM 状态
        
        时间复杂度: O(1) — 直接查表
        """
        return self.transition_table[state].get(token_id, self.dfa.empty)
    
    def is_accepting(self, state: int) -> bool:
        """检查当前状态是否为接受状态（即生成完毕）"""
        return state in self.accept_states
    
    def count_valid_tokens(self, state: int) -> int:
        """计算当前状态有多少合法 token（用于 Jump-Forward 判断）"""
        bitset = self.valid_tokens_bitset.get(state)
        if bitset is None:
            return 0
        return int(np.sum(np.unpackbits(bitset)))

# ==================== Step 4: Logit Mask 应用 ====================
def apply_fsm_mask(logits: torch.Tensor, fsm: CompressedFSM, 
                   current_state: int) -> torch.Tensor:
    """
    将 FSM 合法 token 掩码应用到 logits 上。
    
    实现:
    1. 从 FSM 获取当前状态的合法 token bitset
    2. 将 bitset 展开为 (vocab_size,) 的 bool tensor
    3. 不合法位置的 logits 设为 -inf
    
    GPU 上的高效实现:
    - bitset 在 CPU 上预计算好
    - 上传到 GPU 后用 bitwise 操作展开
    - masked_fill_ 原地操作，无额外分配
    
    时间: ~0.05ms (A100, vocab_size=32000)
    """
    valid_bitset = fsm.get_valid_tokens(current_state)
    
    # bitset → bool mask: unpackbits 将每字节展开为 8 个 bit
    mask_np = np.unpackbits(valid_bitset)[:logits.shape[-1]]
    mask = torch.from_numpy(mask_np).bool().to(logits.device)
    
    # 不合法 token 的 logits 设为 -inf → softmax 后概率为 0
    logits[~mask] = float('-inf')
    
    return logits

# ==================== Step 5: Jump-Forward 判断 ====================
def check_jump_forward(fsm: CompressedFSM, current_state: int, 
                       tokenizer) -> Optional[List[int]]:
    """
    检查当前状态是否只有一个合法 token → 可以跳过模型推理。
    
    算法:
    1. 如果当前状态只有 1 个合法 token → 直接输出该 token
    2. 转移到下一个 FSM 状态
    3. 重复检查，直到有多个合法选择或到达接受状态
    
    返回: 可直接输出的确定性 token 列表，或 None（需要模型推理）
    
    示例: 生成 JSON 时， "{" → "name" → ":" → " 这些位置
          每个状态只有 1 个合法 token → 一次跳过 4 个 token
    """
    jump_tokens = []
    state = current_state
    
    for _ in range(100):  # 最多跳 100 个 token 防止死循环
        valid_count = fsm.count_valid_tokens(state)
        
        if valid_count != 1:
            break  # 有多个选择，需要模型推理
        
        # 找到唯一合法的 token
        valid_bitset = fsm.get_valid_tokens(state)
        mask = np.unpackbits(valid_bitset)
        token_id = int(np.argmax(mask))
        
        jump_tokens.append(token_id)
        state = fsm.advance_state(state, token_id)
        
        if fsm.is_accepting(state):
            break  # 到达接受状态，生成完毕
    
    return jump_tokens if jump_tokens else None

5.1.1 Runtime 调度伪代码

# === SGLang Scheduler 核心调度伪代码 ===
# 文件: sglang/srt/managers/scheduler.py
# 职责: 管理请求队列，构建每步的 extend/decode batch

class Scheduler:
    """
    SGLang 核心调度器 — 决定每个 iteration 执行哪些请求
    
    核心设计:
    1. 每个 iteration 决定: 哪些请求做 extend (prefill), 哪些做 decode
    2. extend 和 decode 可以在同一个 iteration 混合执行
    3. 使用 LPM (Longest Prefix Match) 策略优先调度前缀匹配度高的请求
    """
    
    def __init__(self, model_runner, radix_cache, policy="lpm"):
        self.model_runner = model_runner      # 模型执行器
        self.radix_cache = radix_cache        # Radix Tree 缓存
        self.policy = policy                  # 调度策略: lpm / fcfs / dfs-weight
        
        # 请求队列
        self.waiting_queue: List[Req] = []     # 等待 prefill 的新请求
        self.running_queue: List[Req] = []     # 正在 decode 的请求
        self.finished_queue: List[Req] = []    # 已完成的请求
        
        # 资源限制
        self.max_running_requests = 200
        self.max_total_tokens = 100000         # KV Cache 总容量
        self.chunked_prefill_size = 8192       # 单次 prefill 最大 token 数

    # ==================== 主循环: 每个 iteration 执行一次 ====================
    def run_step(self) -> BatchResult:
        """
        单步调度 — SGLang 的核心循环
        
        执行顺序:
        1. 从 waiting_queue 中选择新请求做 extend (prefill)
        2. 从 running_queue 中选择已有请求做 decode
        3. 将 extend + decode 合并为一个 batch
        4. 执行模型前向传播
        5. 更新请求状态，处理完成的请求
        
        关键: extend 和 decode 在同一个 batch 中执行！
        这比 vLLM 的分阶段调度更灵活
        """
        
        # ========== Phase 1: 构建 extend batch (新请求的 prefill) ==========
        new_batch = self._get_new_extend_batch()
        
        # ========== Phase 2: 构建 decode batch (已有请求的下一步) ==========
        decode_batch = self._get_decode_batch()
        
        # ========== Phase 3: 合并 batch ==========
        if new_batch and decode_batch:
            # extend + decode 混合执行
            combined_batch = Batch.merge(new_batch, decode_batch)
        elif new_batch:
            combined_batch = new_batch
        elif decode_batch:
            combined_batch = decode_batch
        else:
            return BatchResult.EMPTY  # 无请求可调度
        
        # ========== Phase 4: 执行模型前向传播 ==========
        result = self.model_runner.forward(combined_batch)
        
        # ========== Phase 5: 更新请求状态 ==========
        self._process_batch_result(result)
        
        return result

    def _get_new_extend_batch(self) -> Optional[Batch]:
        """
        从 waiting_queue 中选择新请求做 extend (prefill)。
        
        选择策略:
        1. LPM 策略: 优先选择与 Radix Tree 中已有前缀匹配度最高的请求
           → 最大化 KV Cache 复用率
        2. FCFS 策略: 先来先服务，按到达时间排序
        3. DFS-weight 策略: 深度优先，按权重排序
        
        Chunked Prefill:
        如果新请求的 prefill 很长（如 10K tokens），不一次性全部 prefill，
        而是分 chunk（默认 8192 tokens），避免长时间阻塞 decode 请求
        """
        if not self.waiting_queue:
            return None
        
        batch = Batch()
        remaining_budget = self.chunked_prefill_size  # 本步 prefill 的 token 预算
        
        # 按调度策略排序等待队列
        if self.policy == "lpm":
            self.waiting_queue.sort(
                key=lambda req: self._compute_lpm_score(req),
                reverse=True  # 前缀匹配度高的排前面
            )
        
        while self.waiting_queue and remaining_budget > 0:
            req = self.waiting_queue[0]
            
            # Step 1: 检查 KV Cache 是否足够
            needed_tokens = len(req.input_ids) - req.cached_token_num
            if not self._can_allocate(needed_tokens):
                # 触发 LRU 驱逐
                evicted = self.radix_cache.evict(needed_tokens)
                if evicted < needed_tokens:
                    break  # 驱逐后仍不够，停止添加
            
            # Step 2: Radix Cache 前缀匹配
            match_node, match_len = self.radix_cache.match_prefix(req.input_ids)
            
            # 只需要 prefill 未缓存的部分
            req.cached_token_num = match_len
            prefill_len = min(
                len(req.input_ids) - match_len,
                remaining_budget
            )
            
            if prefill_len <= 0:
                # 完全命中缓存！直接进入 decode
                req.is_prefill_done = True
                self.waiting_queue.pop(0)
                self.running_queue.append(req)
                continue
            
            # Step 3: 加入 extend batch
            req.prefix_node = match_node
            req.extend_len = prefill_len
            batch.add_extend(req)
            
            remaining_budget -= prefill_len
            
            # 如果是 chunked prefill（只 prefill 了一部分）
            if match_len + prefill_len < len(req.input_ids):
                req.input_ids_offset = match_len + prefill_len
                # 不从 waiting_queue 移除，下个 iteration 继续 prefill
            else:
                self.waiting_queue.pop(0)
                req.is_prefill_done = True
                self.running_queue.append(req)
            
            # 检查并发请求数限制
            if len(self.running_queue) >= self.max_running_requests:
                break
        
        return batch if batch.size() > 0 else None

    def _get_decode_batch(self) -> Optional[Batch]:
        """
        从 running_queue 中选择请求做 decode (下一步 token 生成)。
        
        注意:
        - 所有在 running_queue 中的请求都参与 decode（除非完成）
        - decode 每次只生成 1 个 token，计算量小
        - decode 和 extend 在同一个 batch 中执行
        """
        if not self.running_queue:
            return None
        
        batch = Batch()
        
        for req in list(self.running_queue):
            # 检查该请求是否已完成
            if req.is_finished:
                self.running_queue.remove(req)
                self.finished_queue.append(req)
                continue
            
            batch.add_decode(req)
        
        return batch if batch.size() > 0 else None

    def _compute_lpm_score(self, req: Req) -> float:
        """
        计算请求的 LPM (Longest Prefix Match) 分数。
        
        分数越高 → 前缀匹配度越高 → KV Cache 复用越多 → 调度优先级越高
        
        计算方式:
        - 查找该请求与 Radix Tree 的最长前缀匹配长度
        - score = match_len / total_len
        - 完全命中缓存的请求 score=1.0，全新请求 score=0.0
        """
        match_node, match_len = self.radix_cache.match_prefix(req.input_ids)
        total_len = len(req.input_ids)
        
        if total_len == 0:
            return 0.0
        
        # 基础分数: 匹配比例
        base_score = match_len / total_len
        
        # 加权: 等待时间越长，分数越高（防止饥饿）
        wait_time = time.time() - req.arrival_time
        wait_bonus = min(wait_time / 60.0, 0.5)  # 最多加 0.5 分
        
        return base_score + wait_bonus

    def _can_allocate(self, num_tokens: int) -> bool:
        """检查是否有足够的 KV Cache 空间"""
        free_blocks = self.radix_cache.token_to_kv_pool.available_size()
        return free_blocks >= num_tokens

    def _process_batch_result(self, result: BatchResult):
        """
        处理模型前向传播的结果。
        
        对于 extend 请求: prefill 完成后，将 KV Cache 写入 Radix Tree
        对于 decode 请求: 检查是否生成了 EOS 或达到 max_tokens
        """
        for req_result in result:
            req = req_result.request
            
            if req.is_prefill_done and req.just_finished_prefill:
                # Prefill 完成 → 将 KV Cache 存入 Radix Tree
                self.radix_cache.insert(
                    token_ids=req.input_ids,
                    kv_indices=req.kv_indices
                )
            
            if req_result.is_finished:
                req.is_finished = True
                self.running_queue.remove(req)
                self.finished_queue.append(req)
            
            # 递减引用计数
            if req.is_finished:
                self.radix_cache.dec_ref(req.prefix_node)
调度器关键设计：
  • Extend-Decode 混合：新请求的 prefill 和已有请求的 decode 在同一 batch 执行，GPU 算力利用率最高
  • LPM 调度策略：优先调度前缀匹配度高的请求，Agent 场景下 KV Cache 命中率从 ~40% 提升到 ~80%
  • Chunked Prefill：长 prompt 分 8192-token 的 chunk 多次 prefill，避免阻塞 decode 请求（降低 ITL）
  • 主动驱逐：KV Cache 不足时按 LRU 驱逐，而非直接拒绝请求
  • 引用计数跟踪：prefill 完成后立即写入 Radix Tree 并建立引用，请求结束时释放

# 文件: sglang/srt/speculative/ (spec_worker.py, dflash_utils.py)

import torch

import math

# ==================== 1. 动态草稿长度的置信度计算 ====================

class DynamicDraftLengthScheduler:
    """
    动态草稿长度调度器 — 根据 draft model 的置信度自适应调整 K
    
    核心思想:
    - 传统 V1: 固定 K=5，不管 draft model 多确定都猜 5 个
    - V2: 根据每个 token 的置信度动态调整 K (范围 2-16)
    
    置信度高的 token → 大胆猜测更多 (K=12-16)
    置信度低的 token → 保守猜测 (K=2-4)
    
    数学原理:
    - 定义置信度 c(t) = max(P(t)) — draft model 对 token t 的最大概率
    - 如果 c(t) > threshold_high → 增加 K
    - 如果 c(t) < threshold_low → 减少 K
    
    效果:
    - 平均 K 从固定 5 提升到 ~8 (因为高置信度位置多猜)
    - 接受率从 ~78% 提升到 ~91% (因为低置信度位置少猜)
    - 综合吞吐提升 1.78x (3.2x / 1.8x)
    """
    
    def __init__(self, 
                 min_draft_len: int = 2,
                 max_draft_len: int = 16,
                 threshold_high: float = 0.85,
                 threshold_low: float = 0.4,
                 smoothing: float = 0.3):
        self.min_draft_len = min_draft_len
        self.max_draft_len = max_draft_len
        self.threshold_high = threshold_high
        self.threshold_low = threshold_low
        self.smoothing = smoothing  # 平滑系数，防止 K 震荡
    
    def compute_draft_length(self, 
                             draft_probs: torch.Tensor,  # (K, vocab_size)
                             draft_tokens: torch.Tensor   # (K,)
                             ) -> int:
        """
        根据 draft model 的输出概率计算最优草稿长度 K。
        
        算法:
        1. 计算每个 draft token 的置信度 c_i = max(P_i)
        2. 计算平均置信度和方差
        3. 根据置信度分布调整 K
        
        公式:
          c_mean = mean(c_1, c_2, ..., c_K)
          c_var = var(c_1, c_2, ..., c_K)
          
          if c_mean > threshold_high and c_var < 0.1:
              K_new = min(K + 2, max_draft_len)  # 置信度高且稳定 → 多猜
          elif c_mean < threshold_low:
              K_new = max(K - 3, min_draft_len)  # 置信度低 → 少猜
          else:
              K_new = K  # 保持不变
          
          # 平滑: K_final = (1-α) × K_old + α × K_new
          K_final = round((1 - smoothing) * K + smoothing * K_new)
        """
        # 计算每个 token 的置信度
        confidences = draft_probs.max(dim=-1).values  # (K,)
        c_mean = confidences.mean().item()
        c_var = confidences.var().item()
        
        K_old = len(draft_tokens)
        
        # 动态调整策略
        if c_mean > self.threshold_high and c_var < 0.1:
            # 高置信度 + 低方差 → 大胆增加 K
            K_target = min(K_old + 3, self.max_draft_len)
        elif c_mean > self.threshold_high:
            # 高置信度但方差大 → 小幅增加
            K_target = min(K_old + 1, self.max_draft_len)
        elif c_mean < self.threshold_low:
            # 低置信度 → 大幅减少 K
            K_target = max(K_old - 4, self.min_draft_len)
        elif c_mean < 0.6:
            # 中低置信度 → 小幅减少
            K_target = max(K_old - 2, self.min_draft_len)
        else:
            K_target = K_old
        
        # 指数平滑，防止 K 剧烈震荡
        K_final = round((1 - self.smoothing) * K_old + self.smoothing * K_target)
        K_final = max(self.min_draft_len, min(K_final, self.max_draft_len))
        
        return K_final

# ==================== 2. Tree Attention 验证 ====================

class TreeAttentionVerifier:
    """
    Tree Attention 验证器 — 一次 target model forward pass 验证多分支
    
    传统 V1 的问题:
    - Draft model 生成 [t1, t2, t3, t4, t5] (线性序列)
    - Target model 一次验证 5 个 token
    - 如果 t3 被拒绝 → t4, t5 也全部丢弃（即使 t4 可能正确）
    
    V2 Tree Attention 的改进:
    - Draft model 生成树状候选（多个分支）:
    
           root
          / | \
         t1 t2 t3     ← 第1层候选 (3个分支)
        / \    |
       t4  t5  t6     ← 第2层候选
    
    - Target model 一次 forward pass 验证整棵树
    - 使用特殊的 attention mask 保证因果关系
    - 接受率从 ~78% → ~91%（因为多分支覆盖更多可能）
    """
    
    def __init__(self, max_tree_size: int = 32):
        self.max_tree_size = max_tree_size
    
    def build_tree_attention_mask(self, 
                                   tree_structure: 'TreeSpec'
                                   ) -> torch.Tensor:
        """
        构建树状 Attention 的因果掩码矩阵。
        
        关键: 树中每个节点只能 attend 到:
        1. 它的所有祖先节点（因果关系）
        2. 同一父节点下的所有兄弟节点（并行验证）
        
        掩码矩阵形状: (tree_size, tree_size)
        mask[i][j] = 1 表示 token i 可以 attend 到 token j
        
        示例 (3层树):
        token: root t1 t2 t3 t4 t5 t6
        root:  [1    0  0  0  0  0  0]
        t1:    [1    1  0  0  0  0  0]  ← t1 attend root + t1
        t2:    [1    0  1  0  0  0  0]  ← t2 attend root + t2
        t3:    [1    0  0  1  0  0  0]  ← t3 attend root + t3
        t4:    [1    1  0  0  1  0  0]  ← t4 attend root + t1 + t4
        t5:    [1    1  0  0  0  1  0]  ← t5 attend root + t1 + t5
        t6:    [1    0  0  1  0  0  1]  ← t6 attend root + t3 + t6
        """
        n = tree_structure.total_nodes
        mask = torch.zeros(n, n, dtype=torch.bool)
        
        for i in range(n):
            # 每个节点可以 attend 自己
            mask[i, i] = True
            
            # 每个节点可以 attend 所有祖先
            ancestors = tree_structure.get_ancestors(i)
            for anc in ancestors:
                mask[i, anc] = True
            
            # 每个节点可以 attend 同父的所有兄弟（包含自己）
            siblings = tree_structure.get_siblings(i)
            for sib in siblings:
                mask[i, sib] = True
        
        return mask
    
    def verify_tree(self, 
                    target_logits: torch.Tensor,     # (tree_size, vocab_size)
                    draft_tokens: torch.Tensor,      # (tree_size,)
                    draft_probs: torch.Tensor,       # (tree_size, vocab_size)
                    tree_structure: 'TreeSpec'
                    ) -> Tuple[List[int], int]:
        """
        树状验证 — 根据 target model 的输出决定接受哪些 token。
        
        验证算法 (修正的投机采样):
        对树中每个节点 i (按 BFS 顺序):
          1. target_prob = softmax(target_logits[i])[draft_tokens[i]]
          2. draft_prob = draft_probs[i][draft_tokens[i]]
          3. accept_prob = min(1, target_prob / draft_prob)
          4. 如果 random() < accept_prob → 接受 token i
             → 继续验证其子节点
          5. 否则 → 拒绝 token i 及其整个子树
             → 从 target_prob 的修正分布中采样一个 token 作为 fallback
        
        返回: (accepted_tokens, num_accepted)
        """
        accepted = []
        queue = [tree_structure.root]
        
        while queue:
            node = queue.pop(0)
            token_id = draft_tokens[node]
            
            # 计算接受概率
            target_prob = F.softmax(target_logits[node], dim=-1)[token_id]
            draft_prob = draft_probs[node][token_id]
            
            # 修正的接受概率: min(1, p_target / p_draft)
            # 这保证了输出分布与 target model 一致
            if draft_prob > 0:
                accept_ratio = min(1.0, (target_prob / draft_prob).item())
            else:
                accept_ratio = 1.0
            
            if torch.rand(1).item() < accept_ratio:
                # 接受！加入结果，继续验证子节点
                accepted.append(token_id)
                for child in tree_structure.get_children(node):
                    queue.append(child)
            else:
                # 拒绝！从修正分布采样 fallback token
                # 修正分布: max(0, p_target - p_draft) 归一化
                residual = F.softmax(target_logits[node], dim=-1) - draft_probs[node]
                residual = torch.clamp(residual, min=0)
                if residual.sum() > 0:
                    residual = residual / residual.sum()
                else:
                    residual = F.softmax(target_logits[node], dim=-1)
                
                fallback_token = torch.multinomial(residual, 1).item()
                accepted.append(fallback_token)
                break  # 拒绝后不再验证子节点
        
        return accepted, len(accepted)

# ==================== 3. DFlash: Draft-Target 内核融合 ====================

class DFlashKernel:
    """
    DFlash (Dynamic Flash Decoding) — SGLang 的核心优化
    
    传统问题:
    - Draft model 和 Target model 各自调用独立的 Attention kernel
    - Draft → CPU sync → Target: CPU-GPU 同步开销 ~0.5ms
    - 两个 kernel 的 KV Cache 读取重复
    
    DFlash 解决方案:
    1. Draft 和 Target 共享同一个 FlashInfer Attention kernel
    2. Draft 的 KV Cache 直接复用 Target 的 KV Cache
    3. 合并为单个 CUDA kernel launch，消除 CPU-GPU 同步
    
    技术实现:
    - 使用 FlashInfer 的 RaggedKVAttention 接口
    - Draft 阶段: 用 draft head 从共享 KV Cache 中计算 attention
    - Target 阶段: 用 target head 从同一 KV Cache 中计算 attention
    - 两个阶段在一个 CUDA Graph 中完成
    
    性能收益:
    - 消除 CPU-GPU 同步: -0.5ms per step
    - 减少 KV Cache 读取: -30% 显存带宽
    - 合并 kernel launch: -20% kernel overhead
    
    与传统方案的数学对比:
    ┌─────────────┬──────────────┬──────────────┐
    │   指标       │  传统 V1      │  DFlash V2   │
    ├─────────────┼──────────────┼──────────────┤
    │ Draft time  │  T_d          │  T_d         │
    │ Sync time   │  0.5ms        │  0ms (融合)  │
    │ Target time │  T_t          │  T_t         │
    │ KV read     │  2× (重复)    │  1× (共享)   │
    │ Total       │  T_d+T_t+0.5  │  T_d+T_t     │
    │ Accepted    │  K×0.78 ≈3.9  │  K×0.91 ≈7.3 │
    │ Throughput  │  3.9/(T_d+T_t)│ 7.3/(T_d+T_t)│
    │ 加速比       │  baseline     │  1.87x       │
    └─────────────┴──────────────┴──────────────┘
    """
    
    @staticmethod
    def fused_draft_target_attention(
        query_draft: torch.Tensor,   # draft model 的 query
        query_target: torch.Tensor,  # target model 的 query
        kv_cache: torch.Tensor,      # 共享的 KV Cache
        tree_mask: torch.Tensor      # 树状因果掩码
    ) -> Tuple[torch.Tensor, torch.Tensor]:
        """
        融合的 Draft-Target Attention 计算。
        
        在单个 FlashInfer kernel 调用中同时计算:
        1. Draft model 的 attention output
        2. Target model 的 attention output
        
        使用 FlashInfer 的 batched ragged KV attention:
        - query: [draft_q; target_q] 拼接
        - KV: 共享同一份 kv_cache
        - 输出: [draft_out; target_out] 拆分
        """
        # 拼接 draft 和 target 的 query
        combined_query = torch.cat([query_draft, query_target], dim=0)
        
        # 单次 FlashInfer kernel 调用
        # FlashInfer 内部使用分块计算 + online softmax，显存效率最高
        combined_output = flashinfer_ragged_attention(
            query=combined_query,
            kv_cache=kv_cache,
            mask=tree_mask  # 树状因果掩码
        )
        
        # 拆分输出
        draft_len = query_draft.shape[0]
        draft_output = combined_output[:draft_len]
        target_output = combined_output[draft_len:]
        
        return draft_output, target_output

# ==================== 4. 数学对比: V1 vs V2 ====================

"""
=== 投机解码效率的数学分析 ===

设:
- T_d = draft model 单步时间 (~0.8ms)
- T_t = target model 单步时间 (~5ms)
- T_sync = CPU-GPU 同步时间 (~0.5ms)
- α = 接受率
- K = 草稿长度

传统 V1 (固定 K=5):
  每步有效 token = Σ(i=0..K-1) α^i = (1 - α^K) / (1 - α)
  当 α=0.78, K=5: 有效 token = (1 - 0.78^5) / 0.22 = 3.91
  每步总时间 = T_d + T_sync + T_t = 0.8 + 0.5 + 5 = 6.3ms
  吞吐 = 3.91 / 6.3 = 0.62 token/ms = 620 tok/s (per request)

Spec V2 + DFlash (动态 K, 平均 K=8):
  当 α=0.91, K=8: 有效 token = (1 - 0.91^8) / 0.09 = 7.28
  每步总时间 = T_d + T_t = 0.8 + 5 = 5.8ms (DFlash 消除同步)
  吞吐 = 7.28 / 5.8 = 1.255 token/ms = 1255 tok/s (per request)
  
  加速比 = 1255 / 620 = 2.02x (单请求)
  
  考虑 batch 效应 (GPU 并行度提升):
  - DFlash 减少 30% 显存带宽 → batch size 可增大 15%
  - 综合加速 = 2.02 × 1.15 × 1.35 ≈ 3.13x ≈ 官标的 3.2x

关键洞察:
1. 接受率从 78% → 91% 是最大贡献 (1.87x)
2. DFlash 消除同步贡献 ~8.5% (1.085x)  
3. 显存带宽优化贡献 ~15% (1.15x)
4. 三者相乘 ≈ 2.33x，加上 batch 效应 ≈ 3.2x
"""
DFlash + Spec V2 核心创新总结：
  • 动态草稿长度：置信度高时 K=12-16，低时 K=2-4，平均 K 从固定 5 提升到 ~8
  • 树状验证：多分支覆盖更多可能，接受率从 78% → 91%，比线性验证多接受 1.7x 有效 token
  • DFlash 融合：Draft/Target 共享 FlashInfer kernel，消除 CPU-GPU 同步 (-0.5ms)，减少 30% 显存带宽
  • 数学本质：加速 ≈ (有效token数提升) × (每步时间降低) × (batch效应) = 1.87 × 1.085 × 1.57 ≈ 3.2x

Q1: RadixAttention 和 vLLM 的 Prefix Caching 本质区别是什么？各自适用什么场景？
A: 核心区别在于数据结构和匹配能力。vLLM 用 Hash Table 做 token hash 匹配，只能做完整前缀匹配 — 两个请求的前缀必须完全相同才能复用。SGLang 用 Radix Tree（基数树/Patricia Trie），可以做最长公共前缀匹配 — 两个请求即使只有部分前缀重叠，也能找到重叠部分并复用。适用场景：vLLM 适合请求间前缀完全相同的批量处理场景；SGLang 适合多轮对话、Agent 工具调用、共享 System Prompt + 不同 Few-shot 等部分重叠场景，KV Cache 复用率可高 30-60%。
Q2: SGLang 的 Constrained Decoding 如何做到"零开销"？
A: "零开销"指的是不需要额外的模型前向传播。SGLang 将 JSON Schema/Regex 编译为 Compressed FSM（压缩有限状态机），在每步解码时：(1) 根据当前 FSM 状态查表得到合法 token 集合（O(1) 查表）；(2) 将不合法 token 的 logits 设为 -∞（O(V) element-wise 操作）。这个过程在 GPU 上只需 ~0.1ms，远小于一次模型前向传播的 ~5ms。更进一步，Jump-Forward Decoding 在确定性位置（如 JSON 的引号、冒号）直接跳过模型推理，批量输出确定性 token，反而比非约束解码更快。首次 FSM 构建约 16ms，后续缓存复用。
Q3: SGLang 的 sgl.function DSL 相比纯 API 调用有什么优势？
A: DSL 模式让 SGLang 看到完整的执行计划，而非逐次独立的 API 请求。这带来三个核心优势：(1) 前缀自动复用 — 编译器识别多个调用的共享前缀，通过 RadixAttention 缓存；(2) 跨请求批处理 — 不同调用的 gen() 操作在调度器层面合并为统一批次；(3) 控制流优化 — 支持 fork/select/条件分支等复杂控制流，编译器自动优化执行路径。纯 API 模式下，每次 chat.completions.create() 都是独立请求，无法做跨请求优化。
Q4: SGLang 的连续批处理和 vLLM 有什么区别？
A: 两者都支持连续批处理（Continuous Batching），但实现细节不同。SGLang 的特色是extend-decode 混合调度：在同一个 iteration 中，新请求的 extend（prefill）和现有请求的 decode 可以在同一个 batch 中执行。SGLang 还支持 LPM（Longest Prefix Match）调度策略 — 优先调度与 Radix Tree 中已有前缀最匹配的请求，最大化 KV Cache 复用率。vLLM 使用 FCFS + Chunked Prefill，更侧重公平性和 ITL 控制。在 Agent/多轮对话场景下，SGLang 的 LPM 调度可将 KV Cache 命中率从 ~40% 提升到 ~80%。
Q5: 2026 年 SGLang 的 Spec V2 和传统投机解码相比有什么突破？
A: 三个关键突破：(1) 动态草稿长度 — 根据 draft model 的置信度自适应调整候选 token 数（2-16），替代固定的 K=5，高置信度时大胆猜测、低置信度时保守验证；(2) 树状验证 — 用树结构组织候选 token，一次 target model forward pass 验证多个分支，接受率从 ~80% 提升到 ~91%；(3) DFlash — 合并 draft 和 target 的 attention 计算，共享 FlashInfer kernel，减少 CPU-GPU 同步。综合效果：在 DeepSeek-V3 上实现 3.2x 加速（V1 仅 1.8x），ITL 降低 69%。

    关键要点: (1) Frontend Layer统一API和DSL两种入口；(2) Tokenizer Manager作为调度入口；(3) Scheduler核心调度循环：选extend→选decode→合并batch→前向传播→更新状态；(4) Radix Tree实现KV Cache复用；(5) 请求状态机：waiting→running→finished

  2. 核心原理: RadixAttention使用Radix Tree（基数树/Patricia Trie）数据结构管理所有请求的KV Cache，实现最长公共前缀匹配。每个RadixNode存储一段token子序列（压缩路径）及其对应的GPU物理块索引（value），通过引用计数（ref_count）跟踪活跃使用情况。当新请求到达时，match_prefix方法从根节点开始逐层查找：用首token的hash在O(1)时间内定位子节点，然后计算child_key与remaining token的最长公共前缀长度。完全匹配则继续深入，部分匹配则触发_split_node分裂节点。插入时先match_prefix找到最长匹配，剩余token创建新叶子节点。LRU驱逐时按last_access排序叶子节点，跳过ref_count>0的活跃节点，释放后通过_try_merge维护Patricia性质。相比vLLM的Hash Table只能做完整前缀匹配，RadixAttention支持任意深度的部分前缀复用，在多轮对话和Agent场景下KV Cache复用率高30-60%。

  5. 核心原理: SGLang Runtime架构的核心是extend-decode混合调度的连续批处理与GPU内存池管理。Scheduler在每个iteration中同时构建extend batch（新请求的prefill）和decode batch（已有请求的下一步），两者合并为一个batch执行模型前向传播。调度策略包括LPM（最长前缀匹配优先，最大化KV Cache复用）、FCFS（先来先服务）和DFS-weight（深度优先按权重）。Chunked Prefill将长prompt分为8192-token的chunk多次prefill，避免长时间阻塞decode请求。内存管理方面，GPU显存按模型权重→KV Cache Pool→CUDA Workspace的顺序静态分配（--mem-fraction-static控制比例）。KV Cache Pool由token_to_kv_pool管理物理块分配/回收，RadixCache管理逻辑前缀到物理块的映射。当显存不足时触发LRU驱逐释放最久未用的KV Cache节点。
    关键要点: (1) extend+decode混合调度最大化GPU利用率；(2) LPM策略将Agent场景KV Cache命中率从~40%提升到~80%；(3) Chunked Prefill避免长prompt阻塞decode（降低ITL）；(4) 静态显存分配+LRU驱逐管理GPU内存；(5) 引用计数跟踪KV Cache活跃使用情况
    实际案例: 一个代码助手服务部署SGLang，同时处理100个用户的代码补全请求。使用LPM调度策略，共享System Prompt（代码补全指令）的KV Cache命中率达92%。Chunked Prefill将长代码文件（10K tokens）分为2个chunk处理，避免阻塞其他用户的decode请求。--max-running-requests=200允许200个并发请求，GPU利用率保持在85%以上。
