TensorRT-LLM (TRT-LLM) 是 NVIDIA 官方推出的 LLM 推理引擎，核心理念是编译时优化：在模型部署前，通过 Graph 级别的优化和 Auto-tuning，生成针对特定硬件的最优执行计划。

与 vLLM/SGLang 的根本区别： vLLM/SGLang 是运行时调度优化（动态批处理、PagedAttention），而 TRT-LLM 是编译时 + 运行时双层优化。编译阶段将整个模型的计算图固化为 TensorRT engine，运行时再叠加 batching/scheduling 策略。

2.1 Graph 优化（图级别）

① 算子融合 (Kernel Fusion)

TRT 自动识别可融合的算子模式，将多个小 kernel 合并为一个大 kernel，减少：

  • Kernel launch overhead：每次 CUDA kernel 启动约 5-10μs，融合后减少次数
  • Global memory 访问：中间结果保留在寄存器/共享内存中
  • 带宽瓶颈：减少 HBM 读写次数

关键融合模式（LLM 场景）

融合模式说明收益
QKV GEMM FusionQ、K、V 三个线性层合并为一次 GEMM减少 2 次 kernel 启动 + 2 次内存读取
Attention FusionQK^T → Scale → Mask → Softmax → ×V 融合FlashAttention 级别效果
MLP FusionGate × Up → SiLU → Down 融合减少中间张量物化
LayerNorm+LinearRMSNorm 后紧跟 Linear 融合减少一次 global memory round-trip
Add+Norm Fusion残差加法 + LayerNorm 融合避免单独 kernel

算子融合的 CUDA Kernel 实现细节

以下展示 TRT-LLM 中两个最关键的融合 kernel 的实现逻辑（伪代码），帮助理解编译时优化的实际执行方式：

① QKV Fusion Kernel — 三次 GEMM 合并为一次

传统实现对 Q、K、V 分别做三次独立的 GEMM（矩阵乘法），每次都需要从 HBM 加载输入 X。融合后将三个权重矩阵拼接，一次 GEMM 完成计算：

// QKV Fusion Kernel 伪代码 (CUDA)
// 核心思想: W_qkv = concat(W_q, W_k, W_v) along output dim
// 一次 GEMM: Y = X @ W_qkv, 然后 split 得到 Q, K, V

__global__ void fused_qkv_gemm_kernel(
    const half* X,        // [batch, seq_len, hidden_dim]
    const half* W_qkv,    // [hidden_dim, 3 * hidden_dim]  (已预拼接)
    half* Q, half* K, half* V,  // 输出 (split 后)
    int M, int N, int K_dim   // M=seq_len, N=3*hidden, K=hidden
) {
    // 使用 Shared Memory Tiling 减少 HBM 访问
    __shared__ half smem_A[TILE_M][TILE_K];  // 输入 tile
    __shared__ half smem_B[TILE_K][TILE_N];  // 权重 tile

    float acc[TILE_M][TILE_N] = {0.0f};

    // 外层循环: 沿 K 维度分块
    for (int k = 0; k < K_dim; k += TILE_K) {
        // 协作加载 tile 到 shared memory
        cooperative_load(smem_A, X, M, k, TILE_M, TILE_K);
        cooperative_load(smem_B, W_qkv, k, N, TILE_K, TILE_N);
        __syncthreads();

        // 内层: 寄存器级矩阵乘 (每个 thread 计算小块)
        for (int kk = 0; kk < TILE_K; kk++) {
            for (int i = 0; i < TILE_M; i++) {
                for (int j = 0; j < TILE_N; j++) {
                    acc[i][j] += __half2float(smem_A[i][kk])
                               * __half2float(smem_B[kk][j]);
                }
            }
        }
        __syncthreads();
    }

    // 写回: 按 output_offset split 为 Q, K, V
    int q_offset = 0;
    int k_offset = hidden_dim;
    int v_offset = 2 * hidden_dim;
    write_split_output(Q, K, V, acc, q_offset, k_offset, v_offset);
}

内存访问模式分析

② Attention Fusion Kernel — FlashAttention 风格分块计算

将 QK^T → Scale → Mask → Softmax → ×V 的整个 Attention 计算融合为单个 kernel，中间结果不物化到 HBM：

// Fused Attention Kernel 伪代码 (FlashAttention 风格)
// 关键: O = softmax(Q @ K^T / sqrt(d_k)) @ V
// 分块计算，中间 S=P@V 在 SRAM 中完成，不写回 HBM

__global__ void fused_attention_kernel(
    const half* Q, const half* K, const half* V,
    half* O,                    // 输出 [batch, heads, seq, d]
    float scale,                // 1/sqrt(d_k)
    int seq_len, int d          // 序列长度, head_dim
) {
    extern __shared__ half smem[];  // SRAM: ~100KB on A100

    // Q 分块: 每个 thread block 处理 Q 的一小行块
    int q_block_start = blockIdx.x * BLOCK_M;

    float acc[BLOCK_M][HEAD_DIM] = {0};   // 累加器 (寄存器)
    float row_max[BLOCK_M];                // Online softmax: 行最大值
    float row_sum[BLOCK_M];                // Online softmax: 行指数和

    // 外层循环: 沿 K/V 的序列维度分块
    for (int kv_start = 0; kv_start < seq_len; kv_start += BLOCK_N) {
        // Step 1: 加载 Q_block, K_block, V_block 到 SRAM
        load_q_block(smem, Q, q_block_start, BLOCK_M, d);
        load_kv_block(smem + BLOCK_M * d, K, V, kv_start, BLOCK_N, d);
        __syncthreads();

        // Step 2: 计算 S = Q_block @ K_block^T * scale
        half S[BLOCK_M][BLOCK_N];
        for (int i = 0; i < BLOCK_M; i++)
            for (int j = 0; j < BLOCK_N; j++)
                S[i][j] = dot_product(q_smem[i], k_smem[j], d) * scale;

        // Step 3: Apply mask (causal / padding)
        apply_causal_mask(S, q_block_start, kv_start, BLOCK_M, BLOCK_N);

        // Step 4: Online Softmax (数值稳定)
        for (int i = 0; i < BLOCK_M; i++) {
            float new_max = max(row_max[i], reduce_max(S[i]));
            float correction = exp(row_max[i] - new_max);
            row_sum[i] = row_sum[i] * correction + sum(exp(S[i] - new_max));
            row_max[i] = new_max;
            // 修正累加器
            for (int k = 0; k < d; k++)
                acc[i][k] *= correction;
        }

        // Step 5: P @ V 累加 (在寄存器中完成，不物化 P 到 HBM)
        for (int i = 0; i < BLOCK_M; i++)
            for (int j = 0; j < BLOCK_N; j++)
                for (int k = 0; k < d; k++)
                    acc[i][k] += exp(S[i][j] - row_max[i]) * v_smem[j][k];

        __syncthreads();
    }

    // Step 6: 归一化并写回 HBM (只写一次)
    for (int i = 0; i < BLOCK_M; i++)
        for (int k = 0; k < d; k++)
            O[...] = __float2half(acc[i][k] / row_sum[i]);
}
FlashAttention 的核心洞察： 传统 Attention 需要物化 O(N×N) 的注意力矩阵到 HBM（N=seq_len 时可能达数 GB）。FlashAttention 通过分块（tiling）和在线 softmax（online softmax），将中间结果全部保留在 SRAM（~20MB on A100）和寄存器中，HBM 读写量从 O(N²) 降低到 O(N²d²/M)，其中 M 是 SRAM 大小。TRT-LLM 将此作为 plugin 集成到编译后的 engine 中。

② 常量折叠 (Constant Folding)

在编译时将所有可静态计算的子图预计算完毕：

# 原始图

cos_emb = cos(position_ids * theta)
sin_emb = sin(position_ids * theta)

# 常量折叠后：cos_emb 和 sin_emb 在 build 时预计算并烘焙进 engine

③ 死代码消除 (Dead Code Elimination)

移除不影响输出的计算节点。例如训练时的 dropout、gradient checkpointing 标记等在推理时都是死代码。

2.3 Transformer → TRT Engine 的转换路径

TRT-LLM 的模型定义层为每种模型架构（Llama、GPT、Falcon 等）提供了专门的 TensorRT 网络定义代码，而非简单的 ONNX 导出。这使得它能精细控制每个算子的实现：

# TRT-LLM 模型定义示例 (简化)
class LlamaAttention:
    def forward(self, hidden_states, past_kv):
        # 显式定义 QKV 投影 — 可指定精度和 kernel
        q = self.q_proj(hidden_states, dtype='fp16')
        k = self.k_proj(hidden_states, dtype='fp16')
        v = self.v_proj(hidden_states, dtype='fp16')

        # GQA/MQA 的 K/V 膨胀 — 在 TRT 图中显式处理
        k = repeat_kv(k, n_rep=self.num_kv_groups)
        v = repeat_kv(v, n_rep=self.num_kv_groups)

        # 指定使用 FlashAttention plugin
        attn_out = self.flash_attn(q, k, v, use_plugin=True)

        # KV Cache 操作 — 直接写入预分配的 TRT tensor
        update_kv_cache(k, v, past_kv, sequence_length)
        return self.o_proj(attn_out)

5.2 实现细节

# TRT-LLM In-flight Batching 的请求状态机

状态转换:
  WAITING ──▶ CONTEXT_INIT ──▶ GENERATION ──▶ COMPLETE
     │              │                │
     │         (Prefill)        (逐 token)
     │              │                │
     │         所有 input       每次 decode
     │         tokens 一次      一个 token
     │         处理完毕
     │
     └── 被调度器选中时开始

# 关键：Context phase 一次性处理所有 input tokens
# Generation phase 每步生成一个 token
# 两个 phase 可以在同一个 batch 中并行（不同请求）
CUDA Graph 深度集成： TRT-LLM 为 Context phase 和 Generation phase 分别录制 CUDA Graph。Context phase 的 CUDA Graph 接受变长输入（通过 padding + mask），Generation phase 的 CUDA Graph 固定为 batch_size 个请求各生成 1 个 token。这消除了 CPU-GPU 同步开销，实现接近硬件极限的利用率。

Ring AllReduce 算法详解

TRT-LLM 的 Tensor Parallelism 在 MLP 层的 Row Parallel 部分需要 AllReduce 操作。NCCL 底层使用 Ring AllReduce 算法：

通信量公式推导

TRT-LLM TP 通信优化三板斧：
1. Fused AllReduce+LayerNorm：将 AllReduce 的结果直接传给 LayerNorm kernel，避免写回 HBM 再读取（节省一次 HBM round-trip）
2. Latency-optimized AllReduce kernel：对 decode 阶段的小消息（~512KB），TRT-LLM 使用自研的 latency-optimized kernel 而非 NCCL 默认的 bandwidth-optimized kernel，延迟降低 30-50%
3. NCCL in CUDA Graph：将 NCCL 通信操作录制进 CUDA Graph，避免 CPU 调度开销和 CPU-GPU 同步，通信与计算完美 overlap

Q1: TensorRT-LLM 的编译优化和 PyTorch torch.compile 有什么本质区别？

A: 核心区别在于优化深度和持久化。torch.compile 是 JIT 编译，运行时动态生成优化代码，受限于 tracing 时间窗口。TRT-LLM 是 AOT (Ahead-of-Time) 编译，可以花费数小时进行 kernel-level auto-tuning，遍历数百种 tactic 选择最优实现。TRT-LLM 的优化粒度更细（单个 CUDA kernel 级别），而 torch.compile 主要在 graph 级别做融合。此外，TRT-LLM 将优化结果序列化为 engine 文件，部署时零编译开销；torch.compile 每次启动都需要 warmup。TRT-LLM 还有 NVIDIA 专有的 plugin 生态（FlashAttention plugin、GEMM plugin），这些是 torch.compile 无法获得的。但 torch.compile 的优势是通用性和易用性——不需要手动管理 build 流程。

Q2: TRT-LLM 的 In-flight Batching 如何在同一 iteration 中混合 Prefill 和 Decode 请求？

A: TRT-LLM 将请求分为两类 phase：Context phase（处理 input tokens 的 prefill）和Generation phase（逐 token decode）。在同一 iteration 中，Scheduler 会将 context phase 的请求和 generation phase 的请求打包在一起。具体实现上：(1) 为两类 phase 分别构建不同的 CUDA Graph；(2) Context phase 的请求使用变长 attention（通过 cuSeqLen 数组指定每个请求的长度）；(3) Generation phase 的请求每个只处理 1 个新 token，但 batch 内有多个请求；(4) 两组请求共享同一个 TRT engine 执行调用，但使用不同的 kernel config。调度器会动态调整 batch composition，确保 GPU 利用率最大化。新完成的请求立即释放 KV Cache 空间，新请求可以立即加入。

Q3: FP8 量化相比 INT8 SmoothQuant 的优势是什么？在什么场景下选择哪个？

A: FP8 (E4M3) 的核心优势是硬件原生支持。在 H100 上，FP8 Tensor Core 的算力是 FP16 的 2 倍 (989 TFLOPS vs 495 TFLOPS)，而 INT8 Tensor Core 的算力与 FP16 相同。这意味着 FP8 不仅量化精度损失小（因为浮点格式天然适合神经网络权重分布），而且计算吞吐更高。FP8 还不需要像 INT8 那样复杂的校准过程（SmoothQuant 需要 alpha 超参搜索）。选择建议：H100/H200 上首选 FP8（精度高+速度快）；A100 上只能用 INT8 SmoothQuant 或 INT4 AWQ（无 FP8 硬件支持）；对精度极其敏感的场景用 FP16；需要极致压缩比时用 INT4 AWQ/GPTQ。

Q4: TRT-LLM 的 Tensor Parallelism 和 vLLM 的 TP 实现有什么区别？对性能有何影响？

A: 实现层面，两者都基于 NCCL 的 AllReduce/AllGather，但 TRT-LLM 有三个关键优势：(1) Fused AllReduce——TRT-LLM 将 AllReduce 与后续的 LayerNorm/RMSNorm 融合为一个 kernel，减少一次 global memory 读写，而 vLLM 的 AllReduce 和后续计算是分离的；(2) 自定义通信 kernel——TRT-LLM 对小消息量的 AllReduce 使用 latency-optimized kernel（而非 NCCL 默认的 bandwidth-optimized kernel），在 LLM decode 阶段（hidden_size 维度的向量）延迟更低；(3) CUDA Graph 内通信——TRT-LLM 将 NCCL 通信操作录制进 CUDA Graph，避免 CPU 介入调度通信。但 vLLM 的 TP 也有优势：实现简单透明，调试方便，且社区优化持续推进。在 H100 NVLink 环境下，TRT-LLM 的 TP 效率比 vLLM 高约 5-15%。

架构总览与设计哲学的技术原理与数学推导： 从底层硬件角度分析，架构总览与设计哲学的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了架构总览与设计哲学的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
架构总览与设计哲学的系统架构分析： 从系统设计的原理出发，架构总览与设计哲学采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

✅ 参考答案与详解
  1. 核心原理: 架构总览与设计哲学的核心理念是编译时+运行时双层优化。与vLLM/SGLang纯运行时调度优化不同，TensorRT-LLM在部署前通过TRT Builder对模型计算图进行离线优化，包括算子融合（Kernel Fusion）、常量折叠、死代码消除等图级别优化，再通过Auto-tuning为每个算子选择最优CUDA kernel，最终固化为.engine文件。运行时叠加In-flight Batching和KV Cache管理。编译阶段将整个模型的计算图转化为高度优化的TensorRT引擎，运行时不再需要动态编译或调度决策。这种方式特别适合硬件固定的生产环境，例如云端GPU推理服务中同一型号GPU集群部署时可以充分利用编译时优化的优势。
    关键要点: (1) 编译时图优化：算子融合减少kernel launch和HBM访问；(2) Auto-tuning：为每个GEMM选择最优tactic；(3) 运行时In-flight Batching：动态请求级批处理；(4) KV Cache Manager：高效GPU显存管理；(5) Multi-GPU并行：TP/PP/EP通信
    实际案例: 一个典型的云端LLM推理服务部署在8xH100 GPU集群上，使用TensorRT-LLM将Llama-70B模型编译为TensorRT engine，编译阶段对QKV GEMM进行融合、对FP8 kernel进行auto-tuning，运行时通过In-flight Batching动态组合多个用户的推理请求，实现端到端低延迟高吞吐。
    关联知识: 与推理优化中的量化技术（FP8/INT4）、并行策略（TP/PP/EP）、CUDA Graph捕获、vLLM/SGLang等运行时框架形成互补或竞争关系。
  2. 核心原理: TRT编译优化原理通过三个层次的图优化实现极致推理性能：(1) 算子融合(Kernel Fusion)：将多个连续小kernel合并为一个大kernel，例如QKV GEMM Fusion将Q、K、V三个线性层的权重矩阵拼接为W_qkv，一次GEMM完成计算，减少2次kernel launch和2次内存读取；Attention Fusion将QK^T→Scale→Mask→Softmax→×V融合为FlashAttention级别效果。(2) 常量折叠：编译时预计算所有可静态确定的子图（如RoPE的cos/sin表烘焙进engine）。(3) 死代码消除：移除推理时不参与输出的计算节点（如dropout、gradient checkpointing）。配合Auto-tuning对每个GEMM操作遍历所有可用Tactic（cuBLAS/cuBLASLt/FP8 Kernel/INT8预量化Kernel等），通过Profiling选择latency最低的方案序列化到engine文件。
    关键要点: (1) QKV Fusion减少3次GEMM为1次，节省66% HBM读写；(2) Attention Fusion实现FlashAttention级别效果，HBM读写量从O(N²)降至O(N²d²/M)；(3) builder_optimization_level 0-5级别越高优化越充分但build越慢；(4) 融合后的最优tactic烘焙进engine，运行时零开销；(5) 编译优化是一次性投入，生成的engine可重复部署
    实际案例: 在H100 GPU上部署LLaMA-7B模型时，通过builder_optimization_level=5进行编译，Auto-tuning自动选择FP8 GEMM kernel（Hopper架构的FP8 Tensor Core可达989 TFLOPS，是FP16的2倍），QKV GEMM Fusion将三次矩阵乘合并，整体推理延迟从FP16的12ms降至FP8优化后的4.5ms。
    关联知识: 与CUDA编程（shared memory tiling、cooperative launch）、FlashAttention算法（分块+online softmax）、FP8量化格式（E4M3/E5M2）、TensorRT Builder API直接相关。
  3. 核心原理: Build流程是将HuggingFace模型转换为TensorRT可执行文件的离线过程。步骤包括：首先从PyTorch模型通过torch.export/torch.compile进行图捕获（TRT-LLM为每种模型架构如Llama/GPT/Falcon提供专门的TensorRT网络定义代码，而非简单ONNX导出），然后通过TensorRT Builder API进行图级别优化（算子融合、常量折叠、死代码消除），接着通过Auto-tuning对每个算子的所有Tactic进行profiling选择最优kernel，最后将最优执行计划序列化为.engine/.plan文件到磁盘。Runtime流程则加载engine文件到TRT Runtime，通过Context Manager管理执行上下文，Runtime层包含In-flight Batching引擎（处理动态请求的批处理组合）、KV Cache Manager（管理GPU显存中的KV Cache分配与释放）、CUDA Graph Executor（减少kernel launch开销）和Multi-GPU Manager（通过NCCL处理TP/PP/EP通信）。
    关键要点: (1) Build阶段决定推理上限（编译优化充分性）；(2) Runtime阶段决定服务灵活性（动态批处理能力）；(3) Mode Config定义模型的超参数（num_layers, hidden_size, num_heads等）；(4) engine文件可跨平台部署（相同GPU架构）；(5) build耗时与模型大小成正比（70B模型level=5需数小时）
    实际案例: 一个生产环境的CI/CD流水线中，模型工程师将训练好的Falcon-180B模型check-in到模型仓库后，自动化脚本触发trtllm-build命令（optimization_level=5, max_batch_size=64, FP8量化），build过程在H100上运行约6小时完成。生成的engine文件被推送到Kubernetes推理集群，TRT-LLM Runtime通过trtllm-serve加载该engine对外提供OpenAI兼容API。
    关联知识: 与ONNX格式、TensorRT Builder API、模型序列化部署、CI/CD for ML、trtllm-serve（基于FastAPI的推理服务框架）相关联。
  4. 核心原理: FP8/INT4量化是TensorRT-LLM在权重量化和KV Cache压缩上的核心能力。FP8量化利用Hopper/Ada Lovelace架构的FP8 Tensor Core（E4M3格式范围[-448,+448]），通过per-tensor或per-channel缩放因子将FP16/BF16权重映射到FP8范围。相比FP16，FP8不仅精度损失极小（per-channel量化时outlier只影响所在通道），而且H100的FP8 Tensor Core算力达989 TFLOPS（FP16的2倍）。INT4量化通过SmoothQuant/AWQ/GPTQ等方法将权重压缩至4-bit，配合--use_weight_only和--weight_only_precision=int4_awq参数启用。KV Cache通过--fp8_kv_cache或int4量化进一步压缩显存占用。量化数学推导：per-tensor量化scale = max(|W|) / 448，q = round(clamp(W/scale, -448, 448))；per-channel量化则为每个输出通道独立计算scale，避免全局outlier导致的精度损失。
    关键要点: (1) FP8 E4M3格式范围[-448,+448]，per-channel量化精度 > per-tensor；(2) INT4 AWQ/GPTQ需要校准数据集（SmoothQuant通过将激活难度迁移到权重来实现无损量化）；(3) FP8 KV Cache可减少50% KV Cache显存；(4) build参数--fp8_kv_cache启用KV Cache FP8量化；(5) 量化在build阶段完成，运行时直接使用
    实际案例: 部署DeepSeek-V3（671B MoE模型）到8xH100集群时，使用FP8权重+FP8 KV Cache量化，模型显存从1342GB（FP16）降至约671GB（FP8），8张H100（640GB）配合TP=8恰好容纳。推理延迟从FP16的45ms/tok降至FP8的18ms/tok，同时吞吐量提升2.5倍。
    关联知识: 与量化校准算法（SmoothQuant迁移策略、AWQ的通道缩放搜索、GPTQ的Hessian矩阵方法）、混合精度推理、显存规划策略相关联。
  5. 核心原理: In-flight Batching（持续批处理/飞行中批处理）是TensorRT-LLM区别于静态批处理的核心运行时优化技术。传统static batching必须等待一个batch中所有请求都完成后才能处理下一批，导致GPU算力大量空闲。In-flight Batching允许新请求在任意时刻加入当前正在执行的batch——当某个请求完成生成时，新请求立即填入其slot，GPU无需空闲等待。实现机制上，In-flight Batching Engine维护一个请求队列，在每次iteration中动态组合：已完成的请求被移出batch并返回结果，新请求通过Context Manager加入batch。每个请求独立维护自己的KV Cache空间和序列状态（pos/seq_id）。配合Request Scheduler的请求优先级策略，实现动态、细粒度的请求调度。这种机制在真实生产环境中（请求到达率动态变化）的吞吐量可提升2-5倍。
    关键要点: (1) 请求级别的动态批处理——每个请求可以独立开始和结束；(2) 与static batching相比，GPU利用率从40-60%提升到80-95%；(3) 需要配合高效的KV Cache Manager管理显存碎片；(4) max_batch_size参数定义最大并行请求数；(5) 适合请求到达率变化大的在线推理场景
    实际案例: 一个在线客服系统使用TensorRT-LLM部署，白天高峰期QPS从50到500波动。使用In-flight Batching后，系统在QPS=500时通过max_batch_size=64维持平均延迟120ms/tok，而使用静态批处理的同等配置延迟超过400ms/tok（因为短请求要等长请求完成）。夜间低QPS时In-flight Batching仍然高效运行，无需switch到低功耗模式。
    关联知识: 与vLLM的Continuous Batching、SGLang的Chunked Prefill共享相同的设计目标，但实现细节不同——TRT-LLM的batch调度在engine编译时部分固化（如max_batch_size绑定了显存预分配），灵活性不如纯运行时方案。
