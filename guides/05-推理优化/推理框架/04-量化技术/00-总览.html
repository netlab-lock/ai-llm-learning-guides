🔢 量化技术总览

本模块目标：掌握模型量化的核心原理和主流方法——从基础的 PTQ/QAT 到 GPTQ、AWQ、GGUF、FP8 等具体实现。
通俗类比：量化就像把高清照片压缩成 JPEG。原图（FP32/FP16）每个像素用 16-32 位存储，量化后（INT8/INT4）只用 4-8 位。压缩后图片看起来差不多（精度损失很小），但文件大小缩小了 2-8 倍，传输和加载都更快。

为什么量化如此重要

为什么量化如此重要

MA-70B 有 700 亿参数，以 FP16 存储需要 140GB 显存，远超单张 A100 的 80GB。量化可以将显存需求压缩到原来的 1/2 甚至 1/4，使得大模型可以在消费级 GPU 上运行。同时，更小的数据类型意味着更快的内存读取速度，推理吞吐量也能显著提升。

常见误区：量化不是万能药。INT4 量化在大多数任务上精度损失可控（1-3%），但在数学推理、代码生成等需要高精度的任务上，精度下降可能非常明显。建议先用 FP8/INT8 验证任务可接受，再考虑更激进的 INT4 量化。
深入探讨：量化的数学本质是将连续的浮点数范围映射到离散的整数集合。对称量化公式为 x_q = round(x / scale)，其中 scale = max(|x|) / (2^(bits-1) - 1)。非对称量化增加一个 zero_point 偏移：x_q = round(x / scale) + zp。量化的误差来源主要有两方面：(1) 截断误差——超出范围的值被钳位；(2) 舍入误差——连续值映射到离散值的精度损失。
量化粒度对比 (Per-tensor vs Per-channel vs Per-group): Per-tensor (整个tensor共享一个scale): 整个权重矩阵 → 1 个 scale 值 ┌─────────────────────────┐ │ W: 4096 × 4096 │ scale = max(|W|) / 127 │ 所有元素用同一个 scale │ → 精度损失最大，但存储开销最小 └─────────────────────────┘ Per-channel (每个输出通道一个scale): 每行一个 scale → 4096 个 scale 值 ┌─────────────────────────┐ │ Row 0: scale_0 │ → 精度较好，存储开销小 │ Row 1: scale_1 │ → GPTQ/AWQ 常用 │ ... │ │ Row 4095: scale_4095 │ └─────────────────────────┘ Per-group (每组元素一个scale): 每 128 个元素一个 scale → 4096×32 个 scale ┌─────────────────────────┐ │ [0:127] → scale_0 │ → 精度最高 │ [128:255] → scale_1 │ → 存储开销较大 │ ... │ → GGUF Q4_K_M 使用 └─────────────────────────┘ 精度: Per-group > Per-channel > Per-tensor 存储: Per-group > Per-channel > Per-tensor

def benchmark(func, iterations=100):
    """性能基准测试"""
    import time
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iterations):
        func()
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iterations
    print(f"Average latency: {elapsed*1000:.2f}ms")
    return elapsed

常见误区：很多人认为 INT4 量化后模型速度一定会提升。实际上，量化本身引入了反量化（dequantization）开销——每次计算前需要将 INT4 权重转换回 FP16 进行矩阵乘法。在 compute-bound 的场景下，这个额外开销可能抵消内存节省带来的收益。量化真正的加速主要来自 decode 阶段的 memory-bound 瓶颈缓解。

学习建议

量化技术的学习路径：先理解量化基础（对称/非对称、per-tensor/per-channel/per-group 粒度），然后分别学习 GPTQ（侧重数学推导）和 AWQ（侧重工程实现），再了解 GGUF 格式和 CPU 推理场景，最后关注 FP8 等硬件原生支持的新格式。实际动手时，推荐从 AutoGPTQ 或 AutoAWQ 开始，对一个小模型（如 LLaMA-7B）做量化实验。

通俗类比：FP8 就像是硬件厂商直接给你造了一个"刚好够用"的尺码——不再需要你自己裁剪（手动量化），GPU 原生支持 FP8 运算，既保留了足够的精度，又获得了接近 INT8 的速度。这比你自己量化的 INT8 更稳定、更高效。
常见误区：不要盲目追求最低位数。Q2 量化虽然显存极小，但在复杂任务上精度损失严重。实际部署建议：生产环境用 FP8 或 INT8，消费级 GPU 用 INT4 (AWQ/GPTQ)，纯 CPU 推理用 GGUF Q4_K_M。

  1. 核心原理: 使用 AutoGPTQ 对 LLaMA-3-8B 进行 INT4 量化的核心流程包括三步：配置量化参数（bits=4, group_size=128, damp_percent=0.01）、加载模型并用校准数据集（通常 128 条 WikiText-2 样本）执行量化、保存量化模型。GPTQ 算法基于 Optimal Brain Surgeon (OBS) 框架，通过逐层最小化权重重建误差来实现量化。其核心思想是利用 Hessian 矩阵的逆来指导量化顺序——每次量化一个权重时，利用 Hessian 信息调整其余未量化权重以补偿量化误差。WikiText-2 perplexity 是衡量语言模型质量的标准指标，量化前 LLaMA-3-8B 的 perplexity 约为 5.7-6.0，INT4 量化后通常升至 6.0-6.5，精度损失约 3-8%，这个结果表明 INT4 量化在绝大多数应用场景中是可接受的。
    关键要点: ①GPTQ 使用 OBS/Hessian 逆矩阵逐列量化权重，每量化一列就用剩余列补偿误差 ②group_size=128 是精度与效率的最佳平衡点 ③desc_act=True 按激活值重要性排序量化列，优先保留对输出影响大的权重 ④128 条校准数据足以估计 Hessian 统计量 ⑤perplexity 上升幅度取决于模型架构和任务复杂度
    实际案例: 在实践中，执行以下 Python 代码可完成量化实验：首先从 HuggingFace 加载 LLaMA-3-8B 和 WikiText-2 测试集，用 lm-evaluation-harness 测量 FP16 baseline 的 perplexity；然后用 AutoGPTQForCausalLM 配置 BaseQuantizeConfig(bits=4, group_size=128) 并调用 model.quantize(calibration_dataset)；最后对量化模型再测 perplexity 并对比。典型结果：FP16 约 5.8，INT4 约 6.1，精度损失约 5%。
    关联知识: 此实验与后续 02-GPTQ 章节的 OBS 数学推导直接相关，与 03-AWQ 章节的激活感知方法形成对比，与 01-量化基础中的 scale/zero_point 公式一脉相承。Perplexity 指标还与推理框架选型中的延迟-质量权衡决策密切相关。
  2. 核心原理: 对称量化假设数据分布关于零点对称，公式为 x_q = round(x / scale)，其中 scale = max(|x|) / (2^(bits-1) - 1)。其映射关系是线性且过零点的：浮点零精确映射到整数零，无需存储零点偏移。对于 INT8，映射范围为 [-max, max] → [-127, 127]。非对称量化引入零点偏移 zero_point，公式为 x_q = round(x / scale) + zp，其中 scale = (max - min) / (2^bits - 1)，zp = round(-min / scale)。其映射范围为 [min, max] → [0, 255]（无符号 INT8），能更精确地利用全部 256 个量化级别。GPTQ 通常选择非对称量化的原因：(1) 神经网络权重在 BatchNorm/激活函数后往往呈现非零均值的分布，非对称量化能更高效地利用量化范围；(2) 非对称量化能精确表示零值（用于稀疏激活和注意力掩码），避免截断误差；(3) 额外存储一个 zero_point 的开销极小（每 group 仅多一个整数），但精度提升显著；(4) GPTQ 论文中的实验表明，非对称量化在 4-bit 场景下比对称量化 perplexity 低 0.2-0.5 个点。
    关键要点: ①对称量化：q = round(r/s)，零点对应零，只需存 scale ②非对称量化：q = round(r/s) + zp，存 scale + zero_point，但量化范围利用率更高 ③映射关系图上，对称量化是一条过原点的直线，非对称量化是一条不过原点的直线（截距为 zp） ④GPTQ 选择非对称量化因为在 INT4 仅有 16 个离散值的极端情况下，充分利用量化范围至关重要 ⑤实际实现中 GPTQ 使用 unsigned INT4 (0-15) 存储，配合 zero_point 实现非对称映射
    实际案例: 以 LLaMA 的注意力 QKV 投影层权重为例，ReLU 激活后的 Q 权重分布通常集中在 [-0.15, 0.35] 之间（非对称分布），如果用对称量化，scale 需按 max(|x|) = 0.35 计算，负半轴 [-0.35, 0) 的量化范围被浪费；而非对称量化用 [0, 255] 完整映射 [-0.15, 0.35]，精度损失更小。
    关联知识: 与 01-量化基础中的量化公式推导直接对应，与 02-GPTQ 的误差补偿算法深度耦合，与 AWQ 的激活感知权重保护策略形成互补。在 GGUF 格式中，Q4_K_M 也采用非对称量化以获得更好的精度。
  3. 核心原理: 这个问题需要从 Roofline 模型角度理解 Prefill 和 Decode 两个阶段的计算特征差异。Roofline 模型的核心公式是：实际性能 = min(峰值算力, 峰值带宽 × 算术强度)。以 A100 为例，FP16 峰值算力 312 TFLOPS，HBM 带宽 2 TB/s，算术强度转折点 = 312T / 2T = 156 FLOPs/Byte。在 Prefill 阶段，模型并行处理所有输入 token，执行的是大规模矩阵乘法（batch_size × seq_len 维度），矩阵尺寸巨大，算术强度远超 156，处于 Roofline 的计算受限区域。此时瓶颈是 GPU 计算能力而非内存带宽，量化虽然减少了权重大小，但 INT4 矩阵乘法需要反量化为 FP16 再计算，增加了额外计算开销，所以加速效果不明显甚至可能更慢。而在 Decode 阶段，每次只处理 1 个 token（或很小的 batch），矩阵乘法退化为矩阵-向量乘法（GEMV），矩阵维度为 1 × hidden_dim，算术强度极低（远低于 156），处于 Roofline 的内存带宽受限区域。此时瓶颈是 HBM 带宽而非计算能力。INT4 权重只有 FP16 的 1/4 大小，内存读取量减少 4 倍，直接转化为 2-4 倍的吞吐提升。这就是量化加速的秘密——它不加速计算，而是缓解了内存带宽瓶颈。
    关键要点: ①Prefill 是 compute-bound（算术强度高），Decode 是 memory-bound（算术强度低） ②Roofline 转折点 = 峰值算力 / 峰值带宽，A100 为 156 FLOPs/Byte，H100 为 295 FLOPs/Byte ③INT4 量化将权重从 2 字节压缩到 0.5 字节，对 memory-bound 的 Decode 阶段收益最大 ④Decode 阶段的 GEMV 操作算术强度约 1-2 FLOPs/Byte，远低于转折点 ⑤实际加速还需考虑反量化开销和 Tensor Core 对 INT4 的硬件支持程度
    实际案例: 以 A100-80G 推理 LLaMA-70B 为例：Prefill 阶段处理 2048 tokens 时，FP16 耗时约 50ms，INT4 量化后耗时约 55ms（反而慢了 10%，因为反量化开销）；Decode 阶段生成 1 token 时，FP16 耗时约 25ms（受限于 140GB 权重的 70GB/s 读取），INT4 量化后仅需约 8ms（35GB 权重读取），加速约 3 倍。
    关联知识: Roofline 模型是推理优化的通用分析框架，与批处理技术中的 Prefill-Decode 分离策略直接相关（vLLM 的 Chunked Prefill 正是基于此洞察），与连续批处理中的动态 batch 调度、KV Cache 管理也密切相关。理解此原理有助于在生产部署中选择合适的量化策略。

反量化计算过程

# GPTQ 反量化计算过程 (PyTorch 伪代码)
import torch

def dequantize_gptq(W_int4, scales, zeros, group_size=128):
    # W_int4: 量化后的 INT4 权重 [out_features, in_features]
    # scales: 缩放因子 [out_features, num_groups]
    # zeros: 零点 [out_features, num_groups]
    W_float = torch.zeros_like(W_int4, dtype=torch.float16)
    for i in range(W_int4.shape[1] // group_size):
        start = i * group_size
        end = start + group_size
        W_float[:, start:end] = (W_int4[:, start:end] - zeros[:, i:i+1]) * scales[:, i:i+1]
    return W_float  # 反量化为 FP16 进行矩阵乘法

FP8 与新格式的数学原理

FP8 格式本质上是 IEEE 754 浮点数的压缩版本，分为 E4M3（4 位指数 + 3 位尾数）和 E5M2（5 位指数 + 2 位尾数）两种变体。为什么需要两种？因为前向传播需要较高精度（E4M3），而反向传播的梯度需要较大动态范围（E5M2）。

FP8 两种格式的位宽分布: E4M3 格式 (精度优先): ┌───┬────┬───┐ │ S │ EEEE │ MMM │ → 动态范围: ±448, 最小精度: 0.002 └───┴────┴───┘ 1位 4位 3位 用于: 权重存储、前向传播 E5M2 格式 (范围优先): ┌───┬─────┬──┐ │ S │ EEEEE │ MM │ → 动态范围: ±57344, 最小精度: 0.0625 └───┴─────┴──┘ 1位 5位 2位 用于: 梯度、反向传播 对比 FP16: 动态范围 ±65504, 最小精度: 0.00006 → E4M3 精度够用，E5M2 范围够用 → H100 的 FP8 Tensor Core 原生支持两者
