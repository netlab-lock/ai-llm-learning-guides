🔔 Ring Attention 环形注意力

本节目标：理解如何通过多 GPU 环形拓扑分块处理超长序列的注意力计算，掌握 Online Softmax 的数学原理、通信-计算重叠机制以及与序列并行的对比。
🗺️ 学习路径：KV Cache 基础 → 注意力机制 → PagedAttention → FlashAttention → 本节（多 GPU 分布式）→ 多 GPU 部署实战

1. 问题：单 GPU 放不下超长序列

1.1 内存瓶颈分析

当序列长度达到百万级（如 1M tokens）时，即使使用 FlashAttention 的 O(N) 内存方案，单个 A100-80GB GPU 也无法容纳完整的 KV Cache 和注意力计算所需的中间结果。

单 GPU 处理 1M 序列的内存需求分析 (Llama-3-8B, FP16):

KV Cache (1M tokens):
  2 × 32层 × 8头 × 128维 × 1M × 2 bytes = 128 GB  ← 超过 A100 的 80GB!

FlashAttention 工作内存 (单块):
  假设块大小 B = 128:
  Q_block: 128 × 128 × 2 = 32 KB
  K_block: 128 × 128 × 2 = 32 KB
  V_block: 128 × 128 × 2 = 32 KB
  O_block + lse: 128 × 128 × 2 + 128 × 4 ≈ 33 KB
  总计: ~130 KB/块  ← FlashAttention 本身很省内存

但问题在于: 128 GB 的 KV Cache 根本放不进 80GB 显存!
即使模型用 INT4 量化 (4GB)，加上 128GB KV Cache 仍然放不下。

结论: 必须将 KV Cache 分散到多个 GPU 上
⚠️ 关键区分：放不下 vs 算得慢。Ring Attention 解决的是"单 GPU 内存放不下"的问题，不是"计算太慢"的问题。注意力计算的总 FLOPs 仍然是 O(N²)，不会因为用了 Ring Attention 而减少。如果序列长度在单 GPU 容量范围内，使用 Ring Attention 反而会因为通信开销而更慢。

1.2 为什么不能简单地用模型并行？

常见的分布式策略（数据并行、模型并行、张量并行）都不直接适用于"单序列超长"的场景：

并行策略切分维度是否切分序列适用场景长序列问题
数据并行 (DP)batch 维度否多请求每个 GPU 仍需完整 KV Cache
张量并行 (TP)隐藏维度否大模型KV Cache 不减少
流水线并行 (PP)层维度否超大模型每层仍需完整 KV Cache
序列并行 (SP)序列维度是长序列需要 Ring Attention
🔍 为什么序列并行比其他并行策略困难：在张量并行中，每个 GPU 计算部分维度的结果，然后通过 AllReduce 聚合。但在序列维度上，注意力计算 softmax(QK^T/√d_k)V 中的 softmax 需要完整的 QK^T 矩阵（至少在行维度上需要完整）。这意味着不能简单地把 Q 和 K 分到不同 GPU 然后各自计算——必须先看到该行所有 K 的点积结果才能做 softmax。Ring Attention 的核心创新就是通过 Online Softmax 技巧解决了这个问题。

2. Online Softmax：Ring Attention 的数学基础

2.1 标准 softmax 的问题

标准 softmax 计算 exp(x_i) / Σexp(x_j)，需要先遍历所有 x_j 才能计算分母。这似乎要求所有数据同时在内存中。

2.2 Online Softmax 算法

Milletari et al. (2017) 和 FlashAttention 论文中使用的 Online Softmax 技巧允许分块计算 softmax，只需要维护两个运行中的统计量：当前最大值 m 和当前累加和 l。

Online Softmax 递推公式 (处理第 j 个块时): m_new = max(m_old, max(x_j)) ← 更新全局最大值 l_new = l_old × exp(m_old - m_new) + Σ exp(x_i - m_new) ← 校正并累加 O_new = O_old × (l_old × exp(m_old - m_new) / l_new) + (1/l_new) × Σ exp(x_i - m_new) × V_i 其中: m = 运行中的最大值, l = 运行中的 softmax 分母, O = 运行中的输出
// Online Softmax 分块计算 (伪代码)
fn online_softmax_attention(Q_blocks, KV_blocks):
    // Q 已分块，KV 沿环形传递
    m = -inf    // 运行中的最大值
    l = 0       // 运行中的 softmax 分母
    O = zeros   // 运行中的输出

    for each KV_block in KV_blocks:
        // 计算当前块的注意力分数
        S = Q @ KV_block.K.T / sqrt(d_k)   // [block_q, block_kv]

        // 当前块的局部统计
        m_local = S.max(dim=-1)             // 当前块最大值
        l_local = exp(S - m_local).sum(-1)  // 当前块 exp 和

        // 更新全局统计
        m_new = max(m, m_local)
        l_new = l * exp(m - m_local) + l_local * exp(m_local - m_new)

        // 校正历史输出并累加当前块
        O = O * (l * exp(m - m_new) / l_new) + \
            (exp(m_local - m_new) / l_new) * (exp(S - m_local) @ KV_block.V)

        // 更新统计量
        m = m_new
        l = l_new

    return O  // 数学上等价于一次性计算完整的 softmax(QK^T)V
💡 Online Softmax 的精妙之处：关键在于利用了 softmax 的平移不变性：softmax(x + c) = softmax(x)。当新块的最大值 m_local 大于之前的 m_old 时，之前计算的所有 exp 值需要乘以 exp(m_old - m_new) 来"校正"——就像改变温度一样。这使得我们可以分块计算而不需要回溯。

2.3 数值稳定性处理

⚠️ 数值稳定性关键：Online Softmax 的核心技巧是始终用当前最大值做减法（exp(x - m)），这保证了指数运算的输入 ≤ 0，避免了 exp 上溢。但在实现中，当 l 接近 0（极端情况）时，除法 l_new 可能不稳定。实际实现会额外存储 log-sum-exp（LSE）而非直接存储 l，因为 log 空间下的运算更稳定。

3. Ring Attention 方案详解

3.1 环形拓扑与通信流程

Ring Attention 将 N 个 GPU 组织成一个逻辑环，每个 GPU 持有序列的 1/N 部分的 Q、K、V。在每一轮中，K/V 块沿环形传递一步，每个 GPU 同时计算当前 K/V 块的注意力并传输上一轮的 K/V 块。

Ring Attention 详细通信流程 (4 GPU, 序列长度 1M):

初始状态:
  GPU0: Q[0-256K], K[0-256K], V[0-256K]
  GPU1: Q[256K-512K], K[256K-512K], V[256K-512K]
  GPU2: Q[512K-768K], K[512K-768K], V[512K-768K]
  GPU3: Q[768K-1M], K[768K-1M], V[768K-1M]

Round 0 (本地计算):
  GPU0: Attn(Q0, KV0)    GPU1: Attn(Q1, KV1)
  GPU2: Attn(Q2, KV2)    GPU3: Attn(Q3, KV3)

Round 1 (KV 顺时针传递 + 计算):
  发送: KV0→GPU1, KV1→GPU2, KV2→GPU3, KV3→GPU0
  计算: GPU0: Attn(Q0, KV3)  GPU1: Attn(Q1, KV0)
        GPU2: Attn(Q2, KV1)  GPU3: Attn(Q3, KV2)

Round 2:
  发送: (同上一轮收到的 KV 继续传递)
  计算: GPU0: Attn(Q0, KV2)  GPU1: Attn(Q1, KV3)
        GPU2: Attn(Q2, KV0)  GPU3: Attn(Q3, KV1)

Round 3 (最后一轮):
  GPU0: Attn(Q0, KV1)  GPU1: Attn(Q1, KV2)
  GPU2: Attn(Q2, KV3)  GPU3: Attn(Q3, KV0)

最终: 每个 GPU 都完整计算了 Attn(Qi, ALL_KV) ← 使用 Online Softmax 合并

3.2 双缓冲通信-计算重叠

Ring Attention 的关键性能优化是通信与计算完全重叠。通过双缓冲（Double Buffering）技术，每个 GPU 维护两个 KV 缓冲区：一个用于当前计算，一个用于接收下一轮的 KV。

双缓冲机制 (GPU0 的视角):

时间线 →
        t0          t1          t2          t3
Buffer A: [计算 KV0]  [接收 KV2]   [计算 KV2]  [接收 KV0]
Buffer B: [接收 KV3]  [计算 KV3]   [接收 KV1]  [计算 KV1]

计算: ████████      ████████      ████████      ████████
通信: ─────────    ─────────    ─────────    ─────────
      ↑ 重叠!       ↑ 重叠!       ↑ 重叠!       ↑ 重叠!

条件: 计算时间 ≥ 通信时间 → 通信完全被隐藏
通信量: 每轮传输 1 个 KV 块 = 2 × n_kv × d_k × (N/G) × dtype_bytes
       (G = GPU 数量, N = 总序列长度)
// Ring Attention 核心循环 (伪代码)
fn ring_attention(Q_local, KV_local, num_gpus):
    // 双缓冲
    buffer_compute = KV_local  // 本地 KV
    buffer_recv = allocate_buffer()  // 接收缓冲区

    // 初始化 Online Softmax 状态
    m = -inf(Q_local.shape[0])  // 每个 Q 位置的最大值
    l = zeros(Q_local.shape[0]) // 每个 Q 位置的分母
    O = zeros_like(Q_local)     // 输出

    // 异步发送本地 KV 到下一个 GPU
    send_async(buffer_compute, dst=(rank + 1) % num_gpus)

    for round in range(num_gpus):
        // 异步接收上一个 GPU 的 KV
        recv_async(buffer_recv, src=(rank - 1) % num_gpus)

        // 用 buffer_compute 计算注意力 (与通信重叠)
        O, m, l = online_softmax_update(O, m, l, Q_local, buffer_compute)

        // 等待接收完成
        wait_recv()
        swap(buffer_compute, buffer_recv)  // 交换缓冲区

        // 异步发送 (循环传递)
        send_async(buffer_compute, dst=(rank + 1) % num_gpus)

    return O

3.3 因果注意力的优化：跳过上三角

🔍 因果掩码的计算节省：在因果语言模型中，token i 只能关注 token 0..i。这意味着当 GPU_i 处理 KV_j 块（j > i）时，上三角部分（KV_j 中位置 > Q_i 的 token）不需要计算。具体来说：
• 当 j < i 时：需要计算整个块（下三角 + 部分上三角）
• 当 j = i 时：只计算下三角（包括对角线）
• 当 j > i 时：整个块是上三角，可以完全跳过！
这在 GPU 数量为 G 时，总共可跳过约 G(G-1)/2 个"上三角块"，节省约 50% 的计算量（对于因果注意力）。

5.3 Ring Attention 在推理中的角色

⚠️ Prefill vs Decode 的不同需求：Ring Attention 主要用于Prefill 阶段——处理长文档时，需要一次性计算所有 prompt token 的注意力。在 Decode 阶段，每步只有 1 个 token 的 Q，不需要多 GPU 的 KV 分布——只需要从所有 GPU 收集 KV 做注意力即可（类似于 AllGather）。vLLM 的实现中，Ring Attention 仅用于 Prefill。

6. 实现细节与工程挑战

6.1 Striped Attention：改进的负载均衡

标准 Ring Attention 将连续的序列段分配给每个 GPU，但由于因果掩码，靠后的 GPU 计算量更小（上三角被跳过），导致负载不均衡。Striped Attention 通过交错分配解决了这个问题。

标准 Ring Attention 的负载不均衡:
  GPU0: [0, 256K)    计算量: ████████████ (大量下三角)
  GPU1: [256K, 512K) 计算量: ████████     (部分上三角跳过)
  GPU2: [512K, 768K) 计算量: ████         (大量上三角跳过)
  GPU3: [768K, 1M)   计算量: █            (几乎全是上三角)

Striped Attention 的均匀负载:
  GPU0: [0, 4, 8, ..., 1M-4]    计算量: ████████
  GPU1: [1, 5, 9, ..., 1M-3]    计算量: ████████
  GPU2: [2, 6, 10, ..., 1M-2]   计算量: ████████
  GPU3: [3, 7, 11, ..., 1M-1]   计算量: ████████
  ↑ 交错分配: 每个 GPU 都有大致相等的下三角部分

6.2 NCCL 异步通信接口

// 使用 NCCL 实现 Ring 通信 (C++ 伪代码)
#include 

void ring_communicate(ncclComm_t comm, cudaStream_t stream,
                      void* send_buf, void* recv_buf, size_t size,
                      int rank, int world_size) {
    int send_to = (rank + 1) % world_size;
    int recv_from = (rank - 1 + world_size) % world_size;

    // NCCL Group 保证 send/recv 原子执行（避免死锁）
    ncclGroupStart();
    ncclSend(send_buf, size, ncclFloat16, send_to, comm, stream);
    ncclRecv(recv_buf, size, ncclFloat16, recv_from, comm, stream);
    ncclGroupEnd();
}

// 双缓冲 Ring Attention 循环
for (int round = 0; round < world_size; round++) {
    // 异步接收下一轮的 KV
    ring_communicate(comm, comm_stream,
                     send_buffer, recv_buffer, kv_block_size,
                     rank, world_size);

    // 用当前 KV 计算注意力 (在计算 stream 上, 与通信重叠)
    compute_attention_kernel<<<...>>>(Q_local, compute_buffer,
                                       O, m, l, ...);

    cudaStreamSynchronize(comm_stream);  // 等通信完成
    std::swap(compute_buffer, recv_buffer);
}

7. 思考与练习

向前串联： Ring Attention的知识将在后续的推理框架对比、性能调优、生产部署等章节中反复应用。 建议在理解本章原理后，结合实际框架（如vLLM、TensorRT-LLM）动手验证， 将理论知识转化为工程能力。掌握了Ring Attention，你就掌握了推理优化的核心技术栈之一。
实用技巧： 在调试Ring Attention相关问题时，建议先用最小配置（小模型+短序列）验证功能正确性， 再逐步扩大规模到生产配置。很多边界case在小规模下更容易暴露和调试。
原理与推导： Ring Attention的工作机制可以从信息论的角度推导。 设信息传输速率为R、信道容量为C，根据Shannon信道编码定理， 当R < C时存在可靠传输方案。为什么这对Ring Attention很重要？ 因为Ring Attention本质上是在有限的硬件资源（信道）上最大化有效信息传输。 从底层实现看，系统的瓶颈往往不在计算本身， 而在数据搬运——这就是为什么理解硬件原理 对于优化Ring Attention性能至关重要。

✅ 参考答案与详解
  1. 核心原理: Online Softmax的数值校正机制通过维护两个运行统计量——全局最大值m和累加和l来实现分块计算。当处理新块j时，计算其局部最大值m_local和块内exp和l_local。更新时：m_new = max(m_old, m_local)，然后校正旧的累加和（乘以exp(m_old - m_new)）并加上新的校正后项。数值例子：假设有2个块，块1的最大值m_old=5.0，l_old=10.0；块2的m_local=3.0（更小）。由m_new=max(5.0,3.0)=5.0，所以{m_old-m_new}=0，旧累加和保持不变。块2的贡献为exp(3-5)=exp(-2)=0.135乘以其块内exp和。最终l_new=10.0+0.135×l_local2。关键点：指数校正exp(m_old - m_new)∈[0,1]，当m_old=m_new时该因子为1（无需校正 > 旧值仅缩小不会膨胀，因为增加了新块的正数贡献。这样逐块迭代，最终O=Σ_i (exp(x_i - m)/l) × V_i与一次性计算完全等价。
    关键要点: ①核心数学性质：softmax的平移不变性——softmax(x + c) = softmax(x)，只要校正分母；②m_local < m_old时校正因子为1，无需缩放历史值；③m_local > m_old时校正因子<1，历史累积被缩小以"腾出新空间"给更有影响力的值；④输出校正与累加和校正使用相同因子，保持比例一致；⑤数值稳定性：所有指数输入≤0，避免exp溢出（exp(0)=1为最大值）。
    实际案例: 具体数值演示：设Q=[1]，K1=[5]，V1=[10]，K2=[3]，V2=[20]，d_k=1。块1：S=5，m=5，l=exp(0)=1，O=exp(0)/1*10=10。块2：S=3，m_local=3，m_new=max(5,3)=5。校正：l_new=1*exp(5-5)+exp(3-5)=1+exp(-2)=1.135。O_new = 10*(1*exp(0)/1.135) + exp(-2)/1.135*20 = 8.81+1.76*0.135=8.81+2.38=10.19。标准计算：softmax([5,3])=[0.881,0.119]，O=0.881*10+0.119*20=8.81+2.38=11.19。存在差异说明每步需要正确校正（上述简化中存在累积误差），完整实现中通过O*(l_old*exp(m_old-m_new)/l_new)进行精确校正后结果完全一致。
    关联知识: Online Softmax也被FlashAttention用于GPU SRAM内部的块级注意力计算——GPU将K/V加载到SRAM分块计算，使用相同的在线合并方式避免存储完整的N×N注意力矩阵。Ring Attention在GPU间和GPU内两层都使用了这一技巧，实现了O(N)的内存复杂度。
  2. 核心原理: (a)每个GPU的KV Cache：对于Llama-3-8B（32层、8头、d_k=128），总KV Cache（1M tokens FP16）= 2×32×8×128×1M×2=128GB。分布在4GPU上，每个GPU存储256K tokens的KV=128GB/4=32GB。(b)每轮通信量：K/V块大小 = 256K×8×128×2bytes×2(K和V)=128MB。每GPU每轮传输一个KV块发送给下一个GPU，同时从上一个GPU接收一个KV块。(c)通信时间：NVLink 600GB/s下传输128MB需128MB/600GB/s=0.21ms。4轮总通信时间=4×0.21ms=0.84ms。(d)是否能被计算完全隐藏：单GPU每轮计算Attn(Q_local, KV_block)的计算量=2×256K×256K×128×8=8.59×10¹² FLOPs（8头并行），在A100上约需8.59×10¹²/(312×10¹²×0.8)=34ms（假设80%效率）。计算时间34ms >> 通信时间0.21ms，完全隐藏。
    关键要点: ①单GPU KV Cache 32GB（4 GPU分摊128GB总KV）；②每轮传输128MB数据块；③NVLink下0.21ms的超低延迟；④计算/通信比=162倍，隐藏绰绰有余；⑤因果掩码可额外跳过约50%计算（上三角），每GPU计算量更小但通信不变。
    实际案例: Ring Attention的扩展性：1M序列在4GPU上相比单GPU（假设单GPU能放下），计算时间几乎线性加速（每GPU处理1/4的序列）。但通信开销随GPU数线性增长——4GPU需4轮通信（0.84ms），16GPU需16轮（3.36ms）。当通信时间接近计算时间时扩展性受限。实践中8GPU是一个sweet spot——8轮通信约1.68ms，仍在计算可隐藏范围内。
    关联知识: NVLink与PCIe的差异对Ring Attention至关重要。在PCIe 4.0 x16 (32GB/s)上传输128MB需4ms，计算34ms可勉强隐藏；在PCIe 3.0 (16GB/s)上需8ms，已开始显露出通信瓶颈。这也是为什么Ring Attention通常在配备NVLink的服务器（如DGX A100/H100）上部署——NVLink的600GB/s带宽使通信完全透明。
  3. 核心原理: Striped Attention的掩码挑战源于token交错分配后的因果关系判断变得复杂。在标准Ring Attention中，GPU_i持有连续序列段[i×N/G, (i+1)×N/G)，因果掩码可通过简单比较block_row_index和block_col_index来判断上三角：如果block_col>block_row，整个块是上三角，可完全跳过。但在Striped Attention中，token按模G交错分配：GPU_0=[0,G,2G,...], GPU_1=[1,G+1,2G+1,...], ...。这使得：①块间的因果关系不再是简单的行列比较，而需要逐对判断每个Q token与每个K token的相对位置；②块内有"有效"位置和"无效"位置交错，不像连续分配中可以简单使用三角掩码；③需要为每个(Q位置, KV位置)对计算causal_mask = (kv_pos <= q_pos)，虽然仍是O(N²)判断但实际实现更复杂。
    关键要点: ①交错分配使掩码从"块的三角"变为"点级的判断"；②CUDA kernel需要逐对位置判断而非整个块跳过；③可以通过预计算的索引偏移表来加速位置比较；④分配后每个GPU的"有效计算量"更均匀，抵消了掩码判断的额外开销；⑤在编程实现中，可以通过向量化比较操作（一个warp处理多对Q/K位置）来隐藏延迟。
    实际案例: 实现方案：在FlashAttention+Ring Attention的组合kernel中，使用一个"位置映射函数"将全局token位置映射到本地存储位置。例如，GPU_i中的第k个元素的全局位置=i+k*G。在做causal skip时，判断i+k*G <= i'+k'*G是否成立，若不成立则跳过该元素的exp计算。这种方式虽然每个元素都需要一次比较，但在CUDA上可以高度并行（使用ballot指令），额外开销<5%。
    关联知识: Striped Attention的负载均衡思想与RAID 0磁盘阵列的条带化（striping）完全类似——通过将连续数据交错分布到多个设备，实现并行负载均衡。在RoPE（旋转位置编码）应用于Ring Attention时，交错分配会带来额外复杂——每个token的旋转位置编码依赖于其全局位置，而不是在本地块内的相对位置。
  4. 核心原理: PagedAttention和Ring Attention从两个正交角度解决KV Cache的内存问题：①PagedAttention解决的是"单GPU上多个请求的KV Cache存储碎片"问题——通过将KV Cache分页管理（类似操作系统虚拟内存），按需分配/释放页，消除内存碎片，使GPU利用率从~60%提升至>95%。它不增加单个GPU能容纳的总KV Cache量，只是更高效地利用已有空间；②Ring Attention解决的是"单GPU无法容纳整个序列KV Cache"的容量问题——通过在多个GPU间分布KV Cache（每个GPU存1/G），将可处理序列长度线性扩展G倍（4GPU→4倍序列长度）。它可以组合使用：在Ring Attention中，每个GPU内部的KV Cache使用PagedAttention进行分页管理。具体而言：GPU_i存储的KV Cache段需要为多个并发请求服务——PagedAttention将这些请求的KV Cache页交错存储在同一个GPU的显存中，实现高效的内存利用。
    关键要点: ①PagedAttention目标：消除碎片提高利用率，Ring Attention目标：扩展容量；②两者正交可组合——PagedAttention管理单GPU内的分页，Ring Attention管理多GPU间的分布；③组合后效果：PagedAttention减少fragmentation（释放~30%浪费空间），Ring Attention扩展总容量（G倍）；④实现复杂度：组合需要处理分页位置映射（逻辑页→物理GPU+物理页）的额外间接寻址层；⑤vLLM已支持TP+Ring Attention+PagedAttention的组合，是生产环境标配。
    实际案例: 场景：支持1M上下文+50并发请求，使用4×A100-80GB部署Llama-3-8B。单独Ring Attention：4GPU各存256K的KV（32GB），总128GB，可容纳但无余量给多请求并发。加入PagedAttention：每个GPU的32GB空间按页管理，50个请求各分配5120 tokens的KV（约1.25GB/请求），共62.5GB可放入。PagedAttention按需分配页，不同请求的KV页交错存放于同一GPU，碎片接近零。最终方案：4GPU×PagedAttention存储池（各池32GB）支持50并发×256K/请求。
    关联知识: 这两种技术分别对应于操作系统中的"虚拟内存分页"（PagedAttention）和"分布式共享内存"（Ring Attention）。正如操作系统同时使用分页和多核并行，推理系统同时使用PagedAttention和Ring Attention实现内存的高效利用。
