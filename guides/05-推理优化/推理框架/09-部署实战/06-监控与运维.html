📊 监控与运维

本节目标：了解如何监控推理服务的健康状态和性能。监控系统就像医院的体检中心——定期检查各项指标，在"病入膏肓"之前就发现隐患。
通俗类比（医院体检）：监控系统就像医院的体检体系——nvidia-smi 是基础体检（量血压、测体温），DCGM 是深度体检（CT、核磁共振），Prometheus 是体检报告存档系统（记录每次数据），Grafana 是医生的看诊界面（一眼看出趋势），告警系统 是异常指标的"紧急呼叫"——指标超标就自动打120。没有这套体系，你就像一个从不体检的人，直到"爆缸"才知道出了问题。

1. GPU 监控

1.1 基础体检：nvidia-smi

# 实时 GPU 状态（每秒刷新）
watch -n 1 nvidia-smi

# 详细信息（CSV格式，便于脚本解析）
nvidia-smi --query-gpu=utilization.gpu,utilization.memory,memory.used,memory.total,temperature.gpu,power.draw --format=csv -l 1

# 只看显存使用
nvidia-smi --query-gpu=memory.used,memory.total --format=csv,noheader
# 输出示例: 15234 MiB, 81920 MiB

1.2 深度体检：DCGM 与 Nsight

# 安装 DCGM
pip install dcgm-client

# 使用 dcgmi 查看详细 GPU 指标
dcgmi dmon -e 100,101,102,103,150,151 -d 1000
# 100 = GPU 利用率, 101 = SM 活跃率
# 102 = Tensor Core 活跃率, 103 = 显存带宽利用率
# 150 = FP16 Tensor Core 利用率, 150 = 显存温度

# 导出 Prometheus 格式（需配合 DCGM Exporter）
# docker run --gpus all -d -p 9400:9400 nvcr.io/nvidia/k8s/dcgm-exporter:3.3.8-3.6.0-ubuntu22.04
常见误区：nvidia-smi 显示的 GPU 利用率是"有 kernel 在运行的时间百分比"，不代表实际算力利用率。一个只用了 5% CUDA Core 的 kernel 也会让 nvidia-smi 显示 100% GPU 利用率。要真正了解算力利用率，需要用 Nsight Compute 或 DCGM 工具。

1.3 GPU 指标的正确解读

深入探讨：NVIDIA DCGM (Data Center GPU Manager) 是数据中心级的 GPU 监控工具，比 nvidia-smi 提供更细粒度的指标：SM 活跃率、Tensor Core 利用率、显存带宽利用率、NVLink 吞吐等。对于推理服务，最值得关注的是 FP16/FP8 Tensor Core 利用率和 HBM 带宽利用率——前者反映 compute-bound 程序的效率，后者反映 memory-bound 程序的瓶颈。
GPU 监控指标层次（由浅入深）: ┌─────────────────────────────────────────────────────┐ │ Level 0: nvidia-smi (基础体检) │ │ ├── GPU 利用率 (有 kernel 在跑的时间%) │ │ ├── 显存使用量 (已用/总量) │ │ ├── 温度 / 功耗 │ │ └── ⚠ 不代表实际算力利用率! │ ├─────────────────────────────────────────────────────┤ │ Level 1: DCGM (深度体检) │ │ ├── SM 活跃率 (真实计算单元使用率) │ │ ├── Tensor Core 利用率 (矩阵计算效率) │ │ ├── HBM 带宽利用率 (内存瓶颈检测) │ │ ├── NVLink / PCIe 吞吐量 │ │ └── ECC 错误计数 (硬件健康) │ ├─────────────────────────────────────────────────────┤ │ Level 2: Nsight Compute (CT扫描) │ │ ├── 每个 kernel 的详细 profiling │ │ ├── Warp 执行效率 / Stall 原因分析 │ │ ├── 内存访问模式 (coalesced vs scattered) │ │ └── Roofline 模型对比 │ └─────────────────────────────────────────────────────┘

3.2 核心面板设计

  • 导入 vLLM 官方 Grafana Dashboard
  • 监控：QPS、延迟 P50/P99、GPU 利用率、显存使用
  • 设置告警：显存 > 95%、延迟 P99 > 5s
  • 推荐面板：Request Rate、Token Throughput、Queue Depth、GPU Memory Timeline

3.3 Dashboard 面板布局建议

深入探讨：一个好的 Grafana Dashboard 应该遵循"黄金信号"原则（参考 Google SRE）：延迟（Latency）、流量（Traffic）、错误（Errors）、饱和度（Saturation）。对于 LLM 推理服务，建议第一行放"概览"面板（QPS、总错误率、P99 延迟），第二行放"性能"面板（TTFT、TPOT、Token/s），第三行放"资源"面板（GPU 利用率、显存、温度），第四行放"队列"面板（等待请求数、调度延迟）。这样从上到下就是"先看症状、再查病因"的诊断路径。

4. 核心监控指标详解

4.1 延迟指标体系

指标类别具体指标正常范围告警阈值说明
延迟TTFT (首 token 延迟)< 500ms> 2s用户感知的响应速度
延迟TPOT (每 token 延迟)< 50ms> 200ms生成速度
延迟P99 端到端延迟< 3s> 10s尾部延迟

4.2 Prometheus 查询示例

# PromQL 查询示例

# P99 延迟（过去5分钟）

# 每秒 token 生成量

# KV Cache 使用率趋势
vllm:gpu_cache_usage_perc

# 请求排队时间中位数
histogram_quantile(0.5, rate(vllm:time_in_queue_seconds_bucket[5m]))

4.3 延迟异常的根因分析

常见误区：很多人只关注平均延迟而忽略 P99 延迟。平均延迟可能是 1 秒，但 P99 可能是 10 秒——这意味着每 100 个请求就有 1 个用户等 10 秒。对于面向用户的服务，P99 延迟比平均延迟更重要。建议同时监控 P50、P95、P99 三个分位。
深入探讨：LLM 推理服务的延迟异常通常有几种根因：(1) 显存接近满载导致频繁的 KV Cache 换入换出；(2) 一个超长 prompt 的 Prefill 阻塞了其他 Decode 请求；(3) GPU 温度过高触发降频（thermal throttling）；(4) 调度器积压了过多请求。通过同时监控应用层指标（vLLM metrics）和硬件层指标（DCGM），可以快速定位是哪种原因。例如，如果 P99 延迟上升但 GPU 利用率没有变化，大概率是调度问题；如果 GPU 利用率也下降了，可能是温度降频。
延迟异常诊断决策树: P99 延迟上升 │ ├── GPU 利用率也上升? │ ├── YES → 计算瓶颈 │ │ ├── Tensor Core 利用率高 → 正常饱和，需扩容 │ │ └── SM 利用率高但 TC 低 → kernel 未优化 │ │ → 参考 torch.compile 优化 │ └── NO ↓ │ ├── GPU 利用率下降? │ ├── YES → 硬件问题 │ │ ├── 温度 > 85°C → 降频，检查散热 │ │ └── ECC 错误 → GPU 硬件故障 │ └── NO ↓ │ ├── 显存接近满载? │ ├── YES → KV Cache 压力 │ │ → 参考 KV Cache 优化 │ └── NO ↓ │ └── 等待队列增长? ├── YES → 调度瓶颈 │ → 检查 max_num_seqs 配置 └── NO → 网络问题，检查 API 网关

5. 告警策略

5.1 分级告警体系

合理的告警策略应该分级别：

级别条件响应时间通知方式
P0 - 严重服务不可用 / OOM 崩溃5 分钟电话 + 短信
P1 - 紧急P99 延迟 > 10s / GPU 温度 > 85°C15 分钟即时消息
P2 - 警告显存 > 90% / 等待队列 > 501 小时邮件
P3 - 通知GPU 利用率持续 < 20%次日日报

5.2 Alertmanager 配置示例

# prometheus/rules/vllm_alerts.yml
groups:
  - name: vllm_alerts
    rules:
      - alert: VllmHighP99Latency
        expr: histogram_quantile(0.99, rate(vllm:e2e_request_latency_seconds_bucket[5m])) > 10
        for: 2m
        labels:
          severity: P1
        annotations:
          summary: "vLLM P99 延迟超过 10 秒"
          description: "当前 P99 = {{ $value }}s，超过阈值 10s"

      - alert: VllmGpuMemoryFull
        expr: vllm:gpu_cache_usage_perc > 0.95
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "GPU KV Cache 使用率超过 95%"

      - alert: VllmQueueBacklog
        expr: vllm:num_requests_waiting > 100
        for: 3m
        labels:
          severity: P1
        annotations:
          summary: "等待队列积压超过 100"

5.3 告警疲劳的应对

通俗类比（医院分诊）：告警级别就像医院的分诊——P0 是心脏骤停（立即抢救），P1 是高烧不退（紧急处理），P2 是血压偏高（预约复查），P3 是体检指标异常（下次注意）。如果所有问题都当 P0 处理，你很快就会"告警疲劳"，真正的紧急情况反而被忽略。好的告警策略就像一个好的分诊台——让正确的信息在正确的时间到达正确的人。
常见误区：不要设置过多告警规则。一个常见的错误是给每个指标都设告警，结果每天收到几十条通知，很快就会忽略所有告警。建议只对直接影响用户体验的指标设置告警（延迟、可用性），其他指标放在日报中观察趋势即可。

5.4 预测性监控与容量规划

深入探讨：高级运维还应该关注"预测性监控"——通过分析历史指标趋势，预测未来的容量需求。例如，如果过去 30 天的请求量每周增长 10%，那么在 5 周后当前集群就会达到容量上限。通过 Prometheus 的 predict_linear 函数可以做简单的线性预测，提前扩容避免突发流量导致的服务降级。
# Prometheus 预测性查询示例

# 预测未来 7 天的 QPS 趋势
predict_linear(vllm:num_requests_running[7d], 7 * 24 * 3600)

# 预测显存何时会满
predict_linear(vllm:gpu_cache_usage_perc[30d], 30 * 24 * 3600) > 1.0

# 基于预测的告警：预计 7 天内显存将耗尽
- alert: VllmMemoryCapacityWarning
  expr: predict_linear(vllm:gpu_cache_usage_perc[7d], 7*24*3600) > 0.98
  labels:
    severity: P3
  annotations:
    summary: "按当前趋势，7天内 GPU 显存将耗尽"

6. 实战运维 Checklist

6.1 部署前检查清单

运维 Checklist：部署推理服务前，确保以下监控已就绪：
  1. Prometheus 已配置采集 vLLM + DCGM 指标
  2. Grafana Dashboard 已导入并验证数据
  3. 告警规则已配置（至少 P0 和 P1 级别）
  4. 告警通知渠道已测试（确认能收到通知）
  5. 日志收集已配置（ELK / Loki）
  6. 健康检查端点 /health 已配置到负载均衡器

6.2 日常巡检与容量规划

深入探讨：建议每天生成一份"推理服务日报"，包含：(1) 过去 24 小时的 QPS 趋势；(2) P50/P95/P99 延迟分布变化；(3) GPU 利用率和显存使用趋势；(4) 告警事件汇总；(5) 与上周同期的对比。这份日报不仅是运维的"体检报告"，也是容量规划的数据基础。结合 多GPU部署 策略和 性能指标 体系，可以科学地做出扩容决策。
💡 思考题：
  1. 如果 nvidia-smi 显示 GPU 利用率 100%，但推理吞吐量很低，可能是什么原因？你会如何进一步诊断？（提示：参考 DCGM 的 SM 活跃率和 Tensor Core 利用率）
  2. 为什么说 P99 延迟比平均延迟更重要？在什么场景下平均延迟更有参考价值？
  3. 如果你的推理服务每天收到 100 万次请求，P99 = 10s，意味着每天有多少用户等待超过 10 秒？这对用户体验和业务有什么影响？
  4. 如何设计一个"自愈"系统——当检测到 GPU 温度过高时自动将流量切换到其他节点？
向前串联：本节的监控体系是 下一节：故障排查 的基础——有了完善的监控数据，排查问题时才有迹可循。同时，监控到的性能数据也需要结合 性能指标 的定义来正确解读。如果你发现持续的性能瓶颈，可能需要参考 编译优化 或 多GPU部署 来提升服务能力。

深入探讨GPU 监控的核心机制： 从底层原理看，GPU 监控的设计为什么如此高效？推导其关键在于将计算复杂度和通信开销进行了巧妙的平衡。从数学本质角度，这一技术的核心工作机制可以用信息论的基本定理来解释。，负责将计算映射到具体的 硬件单元（GPU SM、Tensor Core等）；中间是调度层，负责任务分配和资源管理；上层是应用接口层， 提供用户友好的API。这种分层设计的数学本质是将一个复杂的优化问题分解为多个子问题， 每一层独立优化，最终通过协同工作达到全局最优。根据Amdahl定律，系统的整体加速比 S = 1 / ((1-p) + p/s)，其中 p 为可并行化比例，s 为并行加速倍数。 当 p=0.95 且 s=100 时，理论加速比约为 16.8 倍。
