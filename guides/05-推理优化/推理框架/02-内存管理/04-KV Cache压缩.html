🗜️ KV Cache 压缩

本节目标：理解 KV Cache 压缩的三大策略——量化、驱逐、共享，以及架构层面的 MLA 压缩，掌握各技术的数学原理和工程权衡。
🗺️ 学习路径：KV Cache 基础 → 本节（压缩技术）→ PagedAttention（存储优化）→ 前缀缓存（共享优化）→ llama.cpp 量化实现

1. KV Cache 量化

1.1 量化的数学原理

KV Cache 量化的本质是将 FP16 浮点数映射到低精度整数表示。量化公式为：

对称量化: x_q = round(x / scale) scale = max(|x|) / (2^(b-1) - 1) 非对称量化: x_q = round((x - zero_point) / scale) scale = (max - min) / (2^b - 1) 反量化: x_hat = x_q × scale (对称) 或 x_hat = x_q × scale + zero_point (非对称)

量化引入的误差来源于 round 操作和有限的表示范围（截断）。对于 KV Cache 来说，K 和 V 的分布特征不同，这影响了量化策略的选择。

💡 K 和 V 的分布差异：实验表明，K 向量的值分布通常比 V 更集中（方差更小），因为 K 经过 softmax 前的点积运算会被归一化。因此，K 的量化损失通常比 V 更小。一些实现（如 TensorRT-LLM）会对 K 和 V 使用不同的量化参数。

2. KV Cache 驱逐 (Eviction)

2.1 H2O (Heavy-Hitter Oracle) 的注意力统计原理

H2O 的核心发现是：在注意力计算中，只有少数 token 的 K 会获得持续的高注意力权重，这些 token 被称为"Heavy Hitter"。通过在线统计每个 token 累积获得的注意力权重，可以动态识别并保留这些重要 token。

H2O 重要性分数: score(t) = Σ_{i=t+1}^{N} α_{i,t} 其中 α_{i,t} 是 decode 步骤 i 对 token t 的注意力权重 保留策略: 保留 score 最高的 top-k 个 token 的 KV
// H2O 驱逐算法（简化伪代码）
fn h2o_eviction(kv_cache, scores, budget):
    // scores: 每个 token 累积的注意力权重
    // budget: 最多保留的 KV Cache 条目数

    if kv_cache.seq_len <= budget:
        return kv_cache  // 未满，无需驱逐

    // 计算每个 token 的重要性
    importance = cumulative_attention_scores  // 在线累积

    // 选择 top-k 重要 token
    top_k_indices = top_k(importance, budget)

    // 只保留这些 token 的 KV
    kv_cache.K = kv_cache.K[top_k_indices]
    kv_cache.V = kv_cache.V[top_k_indices]
    return kv_cache
⚠️ H2O 的局限性：H2O 需要在每个 decode 步骤后更新所有 token 的重要性分数，这带来 O(N) 的额外开销。更重要的是，驱逐是不可逆的——一旦某个 token 的 KV 被丢弃，后续 decode 步骤中如果需要关注这个被丢弃的 token，就无法恢复其精确值，只能用近似（如用残差信息）。这就是为什么 H2O 更适合推理场景而非训练。

2.2 StreamingLLM 的 Attention Sink 发现

StreamingLLM 的关键洞察来自一个惊人的实验发现：序列最前面的几个 token（通常是 BOS token 或系统提示）会吸收大量注意力权重，即使这些 token 的语义并不重要。这种现象被称为"Attention Sink"。

Attention Sink 现象可视化:

注意力权重分布 (某个 decode 步骤):
token:  [BOS] [t1] [t2] [t3] ... [t100] [t101] [t102]
权重:   0.35  0.02 0.01 0.01 ... 0.01   0.15   0.45
        ^^^^                                    ^^^^^^
        第一个 token 吸收了大量注意力!            最近的 token
        (Attention Sink)                        (局部相关性)

StreamingLLM 保留策略:
[BOS, t1, t2, t3, | ... 丢弃 ... | t100, t101, t102]
 保留前 N 个           丢弃中间        保留最近 M 个
 (Attention Sink)                    (局部上下文)

总 KV Cache 大小: N + M (固定, 不随序列增长)
典型设置: N=4, M=2040 → 总共 2048 个 token 的 KV
🔍 Attention Sink 的数学本质：为什么第一个 token 会吸收注意力？在因果注意力中，softmax(QK^T/√d_k) 必须为每个查询 token 分配一个概率分布（所有注意力权重之和 = 1）。当所有历史 token 都与当前查询不相关时（即 Q·K 都很小），softmax 的输出会近似均匀分布——但因为第一个 token 可以被所有 token 看到（因果掩码允许），它成为了"默认选择"，吸收了模型想要"忽略"历史时的注意力概率。这个发现表明 softmax 注意力机制中存在一个结构性的"注意力泄漏"。

3. KV Cache 共享

3.1 前缀共享的工作机制

在多请求场景下，很多请求共享相同的系统提示（system prompt）。前缀缓存通过哈希匹配识别相同前缀，复用已计算的 KV Cache，避免重复 Prefill。

// 前缀共享的 KV Cache 管理
// 场景: 3 个请求共享相同的 system prompt (1024 tokens)

Request A: [system(1024)] + [user_query_A(256)]
Request B: [system(1024)] + [user_query_B(128)]
Request C: [system(1024)] + [user_query_C(512)]

// 无前缀共享:
// KV Cache 总量 = (1024+256) + (1024+128) + (1024+512) = 3968 tokens
// Prefill 计算: 3 × 1024 = 3072 tokens (重复计算!)

// 有前缀共享:
// 共享 KV: [system(1024)] → 1 份, 3 个请求引用
// 独立 KV: [query_A(256)] + [query_B(128)] + [query_C(512)]
// KV Cache 总量 = 1024 + 256 + 128 + 512 = 1920 tokens (节省 52%!)
// Prefill 计算: 1024 + 256 + 128 + 512 = 1920 tokens (节省 37%)

3.2 Beam Search 与并行采样中的共享

Beam Search 中的 KV Cache 共享:

Step 3: beam 候选
  Beam 1: [t0, t1, t2a]     ← 分叉点
  Beam 2: [t0, t1, t2b]     ← 共享前缀 [t0, t1]
  Beam 3: [t0, t1, t2c]     ← 共享前缀 [t0, t1]

共享 KV Cache:
  ┌─────────────────────┐
  │ 共享: KV[t0, t1]     │  ← 引用计数 = 3, 只存一份
  ├─────────────────────┤
  │ Beam1: KV[t2a]       │
  │ Beam2: KV[t2b]       │
  │ Beam3: KV[t2c]       │
  └─────────────────────┘

内存节省: 3 × 2 tokens → 2 + 3 = 5 tokens (vs 不共享: 9 tokens)
💡 Copy-on-Write 策略：前缀缓存通常使用引用计数 + Copy-on-Write（COW）来管理共享的 KV Cache。当一个请求完成时，减少引用计数但不立即释放内存。只有当引用计数降为 0 时才释放。如果某个请求需要修改共享部分（理论上不会发生在标准推理中），则复制一份再修改。这种策略与操作系统中的虚拟内存 COW 机制完全相同。

4. MLA (Multi-Latent Attention)

4.1 MLA 的低秩压缩原理

DeepSeek-V2/V3 提出的 MLA 从模型架构层面压缩 KV Cache，其核心思想是将 K 和 V 投影到一个低维压缩空间，只缓存压缩后的表示。

标准 MHA: K = W_k × x V = W_v × x KV Cache = [K, V] 每 token: 2 × n_heads × d_k 维 MLA (DeepSeek-V2/V3): c_kv = W_down × x ← 降维: d_model → d_compress K = W_k_up × c_kv ← 升维恢复 (注意力计算时) V = W_v_up × c_kv ← 升维恢复 (注意力计算时) KV Cache = [c_kv] 每 token: d_compress 维 DeepSeek-V3 参数: d_model = 7168, n_heads = 128, d_k = 128 标准 KV: 2 × 128 × 128 = 32768 维/层 MLA: d_compress = 512 维/层 压缩率: 32768 / 512 = 64×

4.2 MLA 的数学推导

// MLA 注意力计算的等价性证明 (简化版)
// 关键问题: 缓存 c_kv 而非 K, V, 如何计算注意力?

// 标准注意力:
// attn = softmax(Q × K^T / √d_k) × V
//      = softmax((W_q × x_q) × (W_k × x_kv)^T / √d_k) × (W_v × x_kv)

// MLA 注意力 (缓存 c_kv):
// K = W_k_up × c_kv,  V = W_v_up × c_kv
// attn = softmax(Q × (W_k_up × c_kv)^T / √d_k) × (W_v_up × c_kv)
//      = softmax(Q × W_k_up^T × c_kv^T / √d_k) × W_v_up × c_kv

// 优化: Q' = Q × W_k_up^T  (可以预先合并到 Q 的投影中)
// attn = softmax(Q' × c_kv^T / √d_k) × W_v_up × c_kv

// 等价性: 结果与标准注意力完全一致!
// 只是把 W_k_up, W_v_up 的计算从"预处理 KV"移到"实时计算 Q 和输出"
🔍 MLA 的数学本质——低秩分解：MLA 的本质是将 KV 投影矩阵 W_k（形状 [d_model, n_heads×d_k]）分解为两个矩阵的乘积：W_k = W_k_up × W_down，其中 W_down 将输入从 d_model 降到 d_compress，W_k_up 再升回 n_heads×d_k。这类似于 SVD 分解中的低秩近似——如果 d_compress < n_heads×d_k，就实现了压缩。关键洞察是：在注意力计算中，K 和 V 总是与 Q 做点积，所以升维矩阵可以合并到 Q 的计算中，从而避免实际存储高维的 K 和 V。

4.3 MLA vs GQA 的架构对比

⚠️ MLA 的 RoPE 兼容性问题：RoPE（旋转位置编码）作用于 Q 和 K，如果 K 是从压缩表示 c_kv 恢复的，RoPE 的旋转矩阵会破坏低秩结构。DeepSeek-V2 的解决方案是为 Q 和 K 各自添加独立的 RoPE 分量——即 Q = [Q_nope; Q_rope]，K = [K_nope; K_rope]，其中 K_nope 来自 MLA 压缩，K_rope 是独立的位置编码向量（也缓存在 KV Cache 中）。这增加了少量 KV Cache 开销，但保持了位置编码的正确性。

5.2 技术选型决策树

KV Cache 压缩技术选型:

你的模型是否可以重训/微调?
├── 是 → 考虑架构级压缩

│

组合使用: GQA + FP8 + 前缀缓存 = 最佳实践
