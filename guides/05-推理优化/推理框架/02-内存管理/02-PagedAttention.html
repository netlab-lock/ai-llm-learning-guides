class BlockManager:
    def __init__(self, num_blocks, block_size):
        self.block_size = block_size
        self.free_blocks = list(range(num_blocks))  # 空闲 Block 池
        self.block_tables = {}  # request_id → [物理Block列表]

    def allocate(self, request_id):
        # 分配一个新 Block
        block_id = self.free_blocks.pop()
        self.block_tables.setdefault(request_id, []).append(block_id)
        return block_id

    def append_token(self, request_id, token_kv):
        # 检查最后一个 Block 是否已满
        blocks = self.block_tables[request_id]
        if blocks[-1].is_full():
            self.allocate(request_id)  # 分配新 Block
        blocks[-1].append(token_kv)

    def free(self, request_id):
        # 释放所有 Block
        for block_id in self.block_tables.pop(request_id):
            self.free_blocks.append(block_id)

传统 KV Cache 分配的问题的技术原理与数学推导： 从底层硬件角度分析，传统 KV Cache 分配的问题的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了传统 KV Cache 分配的问题的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
传统 KV Cache 分配的问题的系统架构分析： 从系统设计的原理出发，传统 KV Cache 分配的问题采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

传统 KV Cache 分配的问题的性能优化策略

优化传统 KV Cache 分配的问题性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

import torch

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

  1. 核心原理: 传统KV Cache分配采用预分配策略，为每个请求预留最大序列长度（如2048个token）的连续内存空间。然而实际生成的token数远小于最大长度（例如仅生成512个token），导致60-80%的预分配内存被浪费。由于每个请求的KV Cache必须占据一段连续的物理显存地址空间，不同请求之间无法共享内存区域，即使它们有相同的前缀内容。随着请求不断到达和完成，内存中会留下大量大小不一的空洞（外部碎片），这些碎片虽然总量可观但因不连续而无法被新请求使用，最终导致GPU显存利用率仅为20-40%，严重限制了系统的并发服务能力。
    关键要点: ①预分配最大长度导致大量内部浪费；②连续地址分配要求阻止了请求间内存共享；③外部碎片随运行时间累积不断恶化；④内存利用率通常仅20-40%；⑤直接限制了可服务的并发请求数量。
    实际案例: 在A100-80GB GPU上运行Llama-70B，传统方式下每个请求按max_len=4096预分配约2.5GB KV Cache空间，若实际平均仅使用1024个token，则每请求浪费约1.875GB。理论上GPU可服务约32个并发请求，但因碎片化实际容量降至约20个。
    关联知识: 这一问题直接催生了PagedAttention技术（第2节），与GPU内存层次结构（模块01）密切相关，同时也是后续KV Cache压缩技术（模块04）的核心动机之一。
  2. 核心原理: PagedAttention借鉴了操作系统虚拟内存的分页机制来管理GPU显存中的KV Cache。在OS中，物理内存被划分为固定大小的页（Page），进程使用虚拟地址，通过页表（Page Table）将虚拟地址翻译为物理地址，进程无需占用连续的物理内存。类似地，PagedAttention将GPU显存划分为固定大小的Block（如每Block容纳16个token的KV数据），通过Block Table将请求的逻辑token位置映射到物理Block地址，因此请求无需连续的显存空间。Block按需分配——只有当token实际生成时才分配新的Block，而非预先保留最大长度所需的空间。当GPU显存不足时，Block还可以被驱逐到CPU内存，这类似于OS的页面交换（Swap）机制。
    关键要点: ①固定大小Block分配消除了外部碎片；②Block Table提供逻辑到物理地址的间接映射；③按需分配消除了最大长度预分配的浪费；④CPU Offloading能力对应OS的Swap机制；⑤概念对等关系：Page→Block，Page Table→Block Table，Swap→CPU驱逐。
    实际案例: vLLM通过BlockManager维护空闲Block池。当请求A需要存储35个token的KV数据（block_size=16），仅分配3个Block（48个槽位，最后一个Block仅浪费13个槽位）。传统方式则需预分配2048个槽位，浪费率从约98%降至约27%，内存利用率从1.3%提升至68.7%。
    关联知识: 该技术直接建立在操作系统虚拟内存概念之上，与GPU内存层次结构（模块01）中HBM/显存管理紧密相关，并为后续的Copy-on-Write和前缀缓存（第03节）奠定基础。
  3. 核心原理: PagedAttention的工作原理围绕三个核心组件展开：Block Pool（物理Block池）、Block Table（地址映射表）和Block Manager（分配管理器）。KV Cache被切分为固定大小的Block（如每Block 16个token），所有物理Block组成GPU上的Block Pool。每个请求维护一张Block Table，记录其逻辑token位置到物理Block ID的映射关系。例如，请求A生成了35个token，其Block Table为[2, 5, 8]：逻辑位置0-15映射到物理Block 2，位置16-31映射到Block 5，位置32-34映射到Block 8（部分填充）。请求B有20个token，Block Table为[1, 4]。两个请求的Block在物理池中交错分布，完全无需连续。内存利用率计算为(35+20)/(3×16+2×16)=55/80=68.7%，而传统方式为55/(2048+2048)=1.3%，差距高达50倍以上。在实际的注意力计算中，修改后的CUDA Kernel通过Block Table进行间接寻址：对于逻辑位置i的token，访问block_table[i//block_size][i%block_size]来获取对应的KV数据。
    关键要点: ①Block Table是核心数据结构，实现逻辑到物理的间接映射；②允许最后一个Block部分填充（内部碎片仅存在于每个请求的最后一个Block）；③多请求的Block在物理池中交错分布，无需连续；④注意力Kernel被修改为通过Block Table间接寻址，额外开销仅约2%；⑤分配采用池化方式（空闲链表），单次分配O(1)时间复杂度。
    实际案例: 在vLLM的实际实现中，注意力Kernel接受Block Table作为输入参数。以Llama-13B为例，batch中同时处理多个请求时，每个请求有独立的Block Table，Kernel通过gather操作根据Block Table从非连续的物理Block中收集KV数据，再执行标准的注意力计算。这种设计使GPU可以在同一batch中混合处理长短不同的请求，充分利用已分配的每个Block。
    关联知识: Block Table机制与实现伪代码（第5节）中的BlockManager直接对应，注意力Kernel的修改涉及GPU内存层次（模块01）中的共享内存和全局内存访问模式优化，该机制也是前缀缓存（第03节）的实现基础。
  4. 核心原理: PagedAttention带来五大核心优势：第一，按需分配——Block仅在token实际生成时才分配，而非预先保留最大长度空间，从根本上消除了预分配浪费。第二，消除碎片——所有Block大小固定相同，不存在外部碎片问题，仅在每个请求的最后一个Block中存在少量内部碎片（最多浪费block_size-1个槽位）。第三，高利用率——内存利用率从传统方式的20-40%提升到95%以上，因为几乎所有分配的内存都被实际token数据填充。第四，更多并发——同样的GPU显存现在可以服务2-4倍的并发请求，直接提升系统吞吐量。第五，Copy-on-Write——共享相同前缀（如系统提示词）的多个请求可以共享同一组物理Block的引用，只有当某个请求生成了不同的token时才复制对应的Block，极大节省了前缀重复计算和存储的开销。这五大优势相互叠加、层层递进：更少的浪费→更多的可用内存→更高的并发→更高的吞吐。
    关键要点: ①按需分配消除了最大长度预分配的根本问题；②固定Block大小彻底解决外部碎片；③内存利用率从20-40%跃升至95%以上；④并发请求数提升2-4倍；⑤Copy-on-Write机制实现前缀级别的内存共享。
    实际案例: 文件中的性能对比数据显示，vLLM在Llama-13B上从无PagedAttention的约800 tok/s提升至约2400 tok/s（3倍提升），在Llama-70B上从约200 tok/s提升至约600 tok/s（同样3倍提升）。在生产环境中，一个原本只能处理50个并发请求的GPU服务系统，采用PagedAttention后可处理150-200个并发请求，显著降低了单请求的GPU成本。
    关联知识: Copy-on-Write优势直接支撑前缀缓存技术（第03节），高利用率与GPU内存层次优化（模块01）紧密关联，这些综合优势也是vLLM在推理框架竞争中脱颖而出的技术基础。
  5. 核心原理: 实现伪代码展示了BlockManager类的设计，它是PagedAttention的核心实现组件。BlockManager维护两个关键数据结构：free_blocks（一个存储可用物理Block ID的列表，即空闲Block池）和block_tables（一个字典，映射request_id到其对应的物理Block ID列表）。allocate()方法从空闲池中pop一个Block ID并追加到请求的block_table中，实现O(1)的分配操作。append_token()方法在每次新token生成时被调用，它首先检查请求最后一个Block是否已满（is_full()），如果已满则调用allocate()分配新Block，然后将token的KV数据追加到当前Block中——这是一种惰性分配策略，只有在真正需要时才扩展空间。free()方法在请求完成时被调用，将该请求占用的所有Block一次性归还到空闲池中，实现批量释放。这种设计与OS的页帧分配器（Page Frame Allocator）的空闲链表机制高度类似，简洁高效。
    关键要点: ①空闲池（free list）实现O(1)的Block分配和释放；②惰性分配策略：新Block仅在当前Block填满时才分配，最大化Block利用率；③block_table使用列表结构，支持高效的随机访问和顺序遍历；④free()方法执行批量释放（所有Block一次归还），避免逐个释放的开销；⑤整体设计模仿OS页帧分配器的空闲链表机制，借鉴了成熟的操作系统内存管理思想。
    实际案例: 在流式生成场景中，每生成一个新token就调用一次append_token()。以block_size=16为例，前16个token填充第一个Block，第17个token触发allocate()获取新Block。假设请求共生成128个token，则整个过程仅触发8次allocate()调用（第一次在初始化时，后续7次在Block填满时）。当请求正常完成或被中止时，free()一次将8个Block全部归还。Block管理的per-token开销仅为一次is_full()检查和一次append操作，相比注意力计算的开销可以忽略不计（通常<0.5%的总推理时间）。
    关联知识: BlockManager与注意力Kernel的接口是Block Table（Kernel读取它进行间接寻址），与前缀缓存（第03节）结合时需要增加引用计数机制来支持Block共享，其整体设计模式与操作系统中的伙伴系统（Buddy System）和slab分配器有异曲同工之处。
