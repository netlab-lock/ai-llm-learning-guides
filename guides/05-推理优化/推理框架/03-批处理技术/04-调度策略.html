def schedule(self, waiting_queue, running_batch):
    available_blocks = self.block_manager.num_free_blocks()
    for request in waiting_queue:
        needed = self.estimate_blocks(request)
        if needed <= available_blocks:
            running_batch.add(request)
            available_blocks -= needed
        else:
            break  # 内存不足，停止接受

5. 预占 (Preemption)

预占 (Preemption)

通俗类比：FCFS (先来先服务)

深入探讨：从系统设计的底层原理看，FCFS (先来先服务)的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

FCFS (先来先服务)的技术原理与数学推导： 从底层硬件角度分析，FCFS (先来先服务)的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了FCFS (先来先服务)的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
FCFS (先来先服务)的系统架构分析： 从系统设计的原理出发，FCFS (先来先服务)采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

FCFS (先来先服务)的性能优化策略

优化FCFS (先来先服务)性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

# FCFS (先来先服务) - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

✅ 参考答案与详解
  1. 核心原理: FCFS（First Come First Served，先来先服务）是最基本的请求调度策略，严格按照请求到达的先后顺序依次处理。在LLM推理场景中，请求进入等待队列后，调度器从队列头部取出请求加入运行batch，先到达的请求一定先被处理。FCFS的优点是公平性——任何请求都不会因为其他请求的"优先级更高"而被延迟，且实现极为简单，只需一个FIFO队列即可。但其致命缺陷是"队头阻塞"（Head-of-Line Blocking）：如果先到达的请求需要生成很长的输出（如max_tokens=4096），那么后续所有短请求（如只需生成10个token）都必须等待它完成才能被调度，导致短请求的延迟被严重拉长。这在网络路由、磁盘I/O等场景中同样是经典问题。
    关键要点: ①严格按FIFO顺序处理，实现最简单；②保证绝对公平，无饥饿现象；③存在严重的队头阻塞问题；④长请求会拖慢所有后续短请求的响应时间。
    实际案例: 在一个API推理服务中，用户A发送了一个需要生成2000 token的长文翻译请求，随后用户B发送了一个只需生成50 token的简单问答请求。FCFS策略下，用户B必须等待用户A的请求全部完成（可能需要十几秒），即使用户B的请求只需几百毫秒就能完成——这就是队头阻塞导致的不公平延迟。
    关联知识: FCFS是操作系统进程调度、网络包调度中的经典策略；在LLM推理中，FCFS常作为基准策略用于对比其他调度策略的性能；连续批处理（Continuous Batching）在一定程度上缓解了FCFS的队头阻塞问题，因为每个step都有机会处理新请求。
  2. 核心原理: SJF（Shortest Job First，最短作业优先）调度策略的核心思想是优先处理预估输出长度最短的请求，以最小化平均响应时间。在LLM推理场景中，"作业长度"对应的是请求需要生成的token数量，通常用用户指定的max_tokens作为预估值。SJF的数学基础是：在已知作业长度的情况下，按长度升序排列处理可以使所有请求的平均等待时间最小化（这是运筹学中的经典结论）。其优势是短请求能获得极快的响应——大部分用户的问题只需要简短回答，这些请求不应被少数长请求阻塞。但SJF的致命缺陷是"饥饿"（Starvation）：如果持续有短请求到达，长请求可能永远无法被调度执行。此外，SJF需要预先知道请求长度，而LLM推理中实际生成长度往往远小于max_tokens，仅靠max_tokens估算可能不够精确。
    关键要点: ①优先处理预估输出最短的请求，最小化平均等待时间；②短请求响应速度快；③存在长请求饥饿风险；④需要可靠的输出长度预估机制。
    实际案例: 在ChatGPT类产品的API服务中，大部分请求是简短问答（预估输出50-200 token），少数请求是长文生成（预估输出2000-4096 token）。使用SJF策略，短问答请求优先被处理，平均首token延迟（TTFT）显著降低，用户体验更好。但需要配合防饥饿机制（如等待超过一定时间的请求自动升级优先级），避免长文生成请求被无限期延迟。
    关联知识: SJF是操作系统调度理论中的经典算法；在LLM推理框架中，vLLM和TensorRT-LLM支持基于max_tokens的优先级调度；与FCFS相比，SJF在平均延迟上更优但公平性较差；实际生产中常采用SJF+防饥饿的混合策略，如Aging机制。
  3. 核心原理: 内存感知调度（Memory-Aware Scheduling）是根据当前GPU可用内存（具体来说是PagedAttention中的空闲block数量）来决定是否接受新请求进入运行batch的策略。LLM推理的内存瓶颈主要来自KV Cache——每个请求在生成过程中需要存储所有已生成token的Key和Value张量，且这些内存随生成长度线性增长。内存感知调度器维护一个BlockManager，实时追踪已分配和空闲的block数量。当等待队列中的新请求到来时，调度器通过estimate_blocks()估算该请求需要的block数（通常基于max_tokens），只有当空闲block足够时才将其加入batch，否则停止接受新请求。这种策略的核心价值是避免OOM（Out of Memory）错误——在高并发场景下，如果不做内存检查就盲目接受请求，GPU显存会在某个时刻耗尽，导致服务崩溃。
    关键要点: ①基于可用block数量而非固定batch_size决定并发度；②通过estimate_blocks()预估请求的内存需求；③避免OOM，保证系统稳定性；④在内存允许范围内最大化并发以提高吞吐。
    实际案例: 在vLLM部署中，A100 80GB GPU上运行LLaMA-13B模型，模型权重约占26GB，剩余54GB用于KV Cache。每个block（16个token）约占0.5MB。当内存感知调度器检测到空闲block仅剩100个（约50MB）时，如果新请求的max_tokens=2048需要128个block，调度器会拒绝该请求直到有足够空间释放——而非让系统进入OOM状态。当一个长请求生成完毕，其block被回收后，等待中的新请求才能被调度。
    关联知识: 内存感知调度依赖PagedAttention的block管理机制；与连续批处理配合，实现了"计算调度+内存管理"的双重优化；在高并发场景下，内存感知调度是防止系统崩溃的安全阀；vLLM中的Scheduler._schedule_prefills()就是内存感知调度的典型实现。
  4. 核心原理: SLA-Aware（Service Level Agreement感知）调度是根据请求的服务质量要求来分配不同优先级的调度策略。与FCFS或SJF的"一刀切"不同，SLA-Aware调度认识到不同类型请求有不同的延迟敏感度。核心做法是将请求分为多个优先级：P0级（最高优先级）要求TTFT（Time To First Token，首token延迟）低于200ms，这类请求需要优先执行Prefill以尽快输出第一个token；P1级要求TPOT（Time Per Output Token，每token生成时间）低于50ms，需要保持decode batch较小以确保生成速度；P2级是尽力而为（Best-Effort），利用P0和P1不使用的空闲资源来处理。这种分层策略确保关键业务请求（如实时对话、流式输出）获得优先保障，同时不会完全牺牲低优先级请求。调度器在每个决策点都要权衡不同优先级之间的资源分配。
    关键要点: ①按SLA要求将请求分为P0/P1/P2多个优先级；②P0优先Prefill确保低TTFT；③P1控制batch大小确保低TPOT；④P2利用剩余资源尽力服务；⑤不同优先级之间需要资源隔离和公平性平衡。
    实际案例: 在一个企业级AI平台中，P0请求来自实时客服对话系统（用户正在等待回复，TTFT必须<200ms），P1请求来自代码补全工具（需要流式输出，TPOT<50ms保证打字般的体验），P2请求来自批量文档摘要任务（可以排队等待）。当GPU负载较高时，调度器优先保障P0的Prefill请求，适当减少batch size以满足P1的TPOT要求，P2请求则在低峰期被调度。这确保了核心业务的SLA不被后台任务拖累。
    关联知识: SLA-Aware调度源于云计算和数据中心的服务质量管理理念；在LLM推理中，TTFT和TPOT是两个关键的SLA指标，分别对应Prefill和Decode阶段的性能；与FCFS相比，SLA-Aware更灵活但实现更复杂，需要请求标签、优先级队列和资源隔离机制；vLLM的调度器支持priority字段，可实现基本的SLA-Aware调度。
  5. 核心原理: 预占（Preemption）是当GPU资源不足时，暂停（或驱逐）低优先级或长时间运行的请求，将资源让给更需要的请求的调度策略。这是解决资源竞争的"最后手段"。在LLM推理中，预占的实现有两种方式：Swap（交换）是将被暂停请求的KV Cache从GPU显存复制到CPU内存（通过PCIe传输），释放GPU显存空间给新请求，当资源恢复后再将KV Cache从CPU拷回GPU继续生成——这避免了重新计算但引入了额外的数据传输开销。Recompute（重计算）是直接丢弃被暂停请求的KV Cache，完全释放内存，当资源恢复后重新对该请求执行Prefill来重建KV Cache——这避免了CPU-GPU数据传输但需要重新计算所有token的注意力。调度器需要根据PCIe带宽、请求已生成长度等因素决定使用Swap还是Recompute。
    关键要点: ①预占是资源不足时的应急调度机制；②Swap方式：KV Cache搬到CPU，保留状态但有传输开销；③Recompute方式：丢弃KV Cache后重新Prefill，节省带宽但有计算开销；④调度器需根据请求长度和带宽选择预占方式；⑤预占应尽量少触发，过度预占会导致性能下降。
    实际案例: 假设A100 GPU上运行模型，当前batch中有一个长请求R1已生成3000 token（KV Cache占6GB），此时一个P0高优先级请求R2到达但没有空闲block。使用Swap方式：调度器将R1的6GB KV Cache通过PCIe（带宽32GB/s）拷贝到CPU内存（约200ms），释放GPU空间给R2执行Prefill；R2完成后，R1的KV Cache再拷回GPU继续生成（另外200ms）。使用Recompute方式：直接丢弃R1的KV Cache，给R2腾出空间；R2完成后，R1重新Prefill重建3000 token的KV Cache（可能需要500ms+）。当请求较长时Swap更划算，请求较短时Recompute更高效。
    关联知识: 预占机制借鉴了操作系统的虚拟内存swap思想和CPU调度中的抢占式调度；在vLLM的实现中，Scheduler通过evict()函数执行预占，配合BlockManager完成KV Cache的释放和恢复；预占与SLA-Aware调度紧密配合——高优先级请求可以抢占低优先级请求的资源；预占策略的设计直接影响系统的尾延迟（P99 latency）和稳定性。
