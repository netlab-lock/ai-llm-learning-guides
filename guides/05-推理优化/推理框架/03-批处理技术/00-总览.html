📦 批处理技术总览

本模块目标：理解推理框架如何通过批处理技术提高 GPU 利用率和吞吐量。从静态批处理到连续批处理，再到 Chunked Prefill 和预填充-解码分离。
通俗类比：批处理就像餐厅后厨出菜。如果厨师每次只给一位客人做一道菜（单请求推理），灶台利用率极低。但如果把多位客人点的同类型菜品一起炒（批量推理），一个灶台就能同时服务多人。但也不能无限堆单——超过灶台容量，所有人的等待时间都会变长。

推理过程分为两个阶段：Prefill（预填充）阶段是 compute-bound（计算密集），可以充分利用 GPU 算力；Decode（解码）阶段是 memory-bound（内存带宽密集），逐 token 生成时大部分时间花在读取 KV Cache 上。两个阶段对批处理策略的需求完全不同，这也是为什么需要多种批处理技术的原因。

常见误区：很多人以为"批处理越大越好"，实际上批大小受显存限制——每个请求都需要独立的 KV Cache，batch size 过大会导致 OOM。此外，过大的 batch size 会使单个请求的延迟（尤其是 TTFT）急剧上升，因为 Prefill 阶段需要计算更多的 token。
深入探讨：从 GPU 架构角度看，批处理之所以有效，是因为 GPU 的 CUDA Core 数量远超单个请求的并行需求。A100 有 6912 个 CUDA Core，而单个 decode step 的矩阵-向量乘法只能用到其中很小一部分。通过 batching，将多个请求的计算合并为矩阵-矩阵乘法，就能更充分地利用这些核心。同时，Tensor Core 对大矩阵的运算效率也远高于小矩阵。

通俗类比：静态批处理像固定班次的大巴——乘客必须等满一车才发车；动态批处理像网约车拼车——有人就走，随时可以接新乘客；连续批处理则是地铁——每到一站都有人上下，永不停歇。
常见误区：连续批处理（Continuous Batching）并不是"无限并行"。它需要配合 PagedAttention 等内存管理技术，因为新请求随时可能加入，必须有动态的 KV Cache 分配机制。如果只做连续批处理而没有优化内存管理，很容易在请求频繁加入/退出时产生内存碎片。
深入探讨：vLLM 的 Continuous Batching 实现依赖三个核心组件：(1) Scheduler 负责在每个 iteration 决定哪些请求参与计算；(2) PagedAttention 负责动态分配和回收 KV Cache 块；(3) Block Manager 管理物理显存块到逻辑块的映射。这三个组件的协同工作使得请求可以在任意 iteration 级别加入或退出，而不需要等待整个 batch 完成。
Continuous Batching 的 iteration 级调度: Iteration 1: [Req A: step 3] [Req B: step 1] [Req C: step 5] ↓ GPU 并行执行 ↓ Iteration 2: [Req A: step 4] [Req B: step 2] [Req C: 完成,退出!] ↓ 新请求 D 加入 ↓ Iteration 3: [Req A: step 5] [Req B: step 3] [Req D: step 1] ↓ Req A 完成 ↓ Iteration 4: [Req B: step 4] [Req D: step 2] [Req E: step 1] ← 新加入! 关键: 每个 iteration 独立调度, 请求随时进出 对比静态批处理: 必须等所有请求完成才能处理下一批
# vLLM Continuous Batching 的简化调度逻辑
class Scheduler:
    def schedule(self):
        batch = []
        # 1. 优先调度正在运行的请求 (decode)
        for req in self.running:
            if self.block_manager.can_allocate(req):
                batch.append(req)
        
        # 2. 如果还有显存空间, 调度新请求 (prefill)
        while self.waiting and self.has_free_memory():
            new_req = self.waiting.pop(0)
            self.block_manager.allocate(new_req)
            batch.append(new_req)
        
        # 3. 检查完成的请求, 释放资源
        for req in batch:
            if req.is_finished():
                self.block_manager.free(req)
                self.running.remove(req)
        
        return batch

学习建议

建议按照知识地图的顺序学习：先理解静态和动态批处理的区别，再学习连续批处理的工作原理，然后深入 Chunked Prefill 如何缓解长 prompt 对延迟的影响，最后了解预填充-解码分离的前沿架构。每一层递进都建立在前一层的基础上。

通俗类比：Chunked Prefill 就像把一个大项目的报告分成多段，每次会议只汇报一小段，这样其他紧急任务（decode 请求）不用等整个大报告讲完才能处理。它解决了"一个超长 prompt 把整个 batch 卡住"的问题。
常见误区：很多人混淆了"吞吐量"和"延迟"。批处理主要提升吞吐量（每秒处理的请求数），但通常会增加单个请求的延迟（因为要等待和共享资源）。SLA-aware 调度策略的目标就是在吞吐量和延迟之间找到最佳平衡点，而不是盲目追求最大 batch size。
深入探讨：预填充-解码分离（PD 分离）是 2024 年的重要研究方向。DistServe 和 Splitwise 的核心思想是将 Prefill 和 Decode 阶段放到不同的 GPU 上执行——Prefill 节点使用高算力 GPU（如 H100 SXM），Decode 节点使用高带宽内存的 GPU。这种异构部署可以独立优化两个阶段，避免 Prefill 的大计算量干扰 Decode 的稳定输出。代价是需要高速网络连接传输 KV Cache，PCIe Gen5 或 NVLink 成为关键约束。

# SLA-aware 调度: 在延迟和吞吐间找平衡
class SLAScheduler:
    def __init__(self, ttft_sla_ms=500, tpot_sla_ms=50):
        self.ttft_sla = ttft_sla_ms   # 首 token 延迟 SLA
        self.tpot_sla = tpot_sla_ms   # 每 token 延迟 SLA
    
    def should_add_request(self, new_req, current_batch):
        # 估算加入新请求后对现有 batch 的影响
        estimated_tpot = self.estimate_tpot(current_batch + [new_req])
        return estimated_tpot <= self.tpot_sla
    
    def estimate_tpot(self, batch):
        # 基于 batch size 估算每 token 延迟
        # 简化公式: TPOT ≈ base_latency × batch_size / num_sms
        return 5.0 * len(batch) / 132  # H100 有 132 个 SM

为什么批处理是推理优化的核心的关键技术要点

掌握为什么批处理是推理优化的核心需要理解其底层原理和工作机制。 从数学本质看，该技术的核心是在性能、精度和资源三者之间寻找最优平衡。 为什么这个三角约束如此重要？因为任何优化都不能同时最大化三者—— 提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32）， 减少资源又会降低性能。理解这个原理，就能推导出大多数技术选择背后的逻辑。

常见问题与解决方案

实际工程中遇到的大多数问题都可以归为三类：性能问题（延迟/吞吐不达标）、 精度问题（输出质量下降）、稳定性问题（偶发OOM/超时）。 为什么要这样分类？因为每类问题的诊断工作机制完全不同： 性能问题用profiler，精度问题用eval benchmark，稳定性问题用日志分析。
