✂️ Chunked Prefill 分块预填充

本节目标：理解长 prompt 的 Prefill 为什么阻塞已有 decode 请求，掌握 Chunked Prefill 的调度机制、chunk 大小选择的数学分析以及与连续批处理的协同。
🗺️ 学习路径：KV Cache 基础 → 动态批处理 → 连续批处理 → 本节（分块预填充）→ 调度策略

1. 问题：长 Prefill 阻塞 Decode

1.1 Prefill 与 Decode 的计算特征差异

理解 Chunked Prefill 的前提是要深刻理解 Prefill 和 Decode 两个阶段的计算特征本质差异：

特征Prefill 阶段Decode 阶段
输入 token 数N 个（整个 prompt）1 个（当前生成的 token）
计算模式矩阵-矩阵乘法 (GEMM)矩阵-向量乘法 (GEMV)
计算密度高（compute-bound）低（memory-bound）
GPU 利用率高（算力充分利用）低（受限于显存带宽）
耗时与 N 成正比（线性-二次）几乎恒定（~几十ms/step）
KV Cache 操作一次性填充所有逐行追加
🔍 为什么 Prefill 是 compute-bound 而 Decode 是 memory-bound：这可以用算术强度（Arithmetic Intensity = FLOPs / Bytes）来解释。对于矩阵乘法 C = A × B（[M,K] × [K,N]），AI = 2×M×K×N / (M×K + K×N + M×N) × dtype_bytes。Prefill 时 M=N=较大值（如 4096），AI 很高；Decode 时 M=1（单 token），AI ≈ 2×K / (K+N) × dtype_bytes，极低。GPU 的 FLOPs/Byte 比值（如 A100: 312 TFLOPs / 2 TB/s = 156）决定了当 AI < 156 时为 memory-bound。Decode 的 AI 通常 < 10，远低于此阈值。

1.2 长 Prefill 导致的延迟尖峰

没有 Chunked Prefill 的时间线:

时间 →
         t0        t1        t2        t3        t4        t5
请求A:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]
请求B:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]
请求C:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]

t=2.5s: 请求 D 到达 (prompt 8192 tokens)

         t2.5                    t4.5
请求D:   [======== Prefill 8K ========]  [decode]...
请求A:   [xxx 阻塞 xxx 阻塞 xxx 阻塞]  [decode]  ← TPOT 从 50ms 飙升到 2s!
请求B:   [xxx 阻塞 xxx 阻塞 xxx 阻塞]  [decode]
请求C:   [xxx 阻塞 xxx 阻塞 xxx 阻塞]  [decode]

问题: D 的长 Prefill 独占 GPU，A/B/C 的 decode 完全停止
用户 A/B/C 感受到 2 秒的输出卡顿!
⚠️ 量化分析：以 Llama-3-8B 为例，Prefill 8192 tokens 约需 2 秒（A100）。在这 2 秒内，其他请求的 decode 全部暂停，TPOT 从正常的 50ms 飙升到 2000ms（40倍恶化）。在高并发场景下，不断有新请求到来，这种"延迟尖峰会"反复出现，导致 P99 延迟不可接受。

1.3 Prefill 占比与 GPU 利用率的关系

场景Prompt 长度Prefill 时间Decode 时间/stepPrefill 占比
短对话128 tokens~30ms~50ms37%
中等问答1024 tokens~200ms~50ms80%
长文档总结8192 tokens~2000ms~50ms97.5%
RAG 上下文32768 tokens~8000ms~50ms99.4%
💡 关键洞察：Prefill 时间与 prompt 长度近似线性关系（FlashAttention 下），而 decode 每步几乎恒定。当 prompt 长度 > 1024 tokens 时，Prefill 时间就开始显著影响其他请求的延迟。这就是 Chunked Prefill 诞生的原因。

2. 解决：Chunked Prefill 机制

2.1 核心思想：分时复用 GPU

Chunked Prefill 的核心思想借鉴了操作系统的时间片轮转调度——将长 Prefill 切分为多个小 chunk，每个 chunk 的计算时间与一次 decode 步骤相当，然后交替执行。

有 Chunked Prefill 的时间线 (chunk_size=1024):

时间 →
         t0        t1        t2        t3        t4        t5        t6
请求A:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]  [decode]
请求B:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]  [decode]
请求C:   [decode]  [decode]  [decode]  [decode]  [decode]  [decode]  [decode]

t=2.5s: 请求 D 到达 (prompt 8192 tokens, 分成 8 个 chunk)

         t2.5    t2.6    t2.7    t2.8    t2.9    t3.0    t3.1 ...
请求D:   [chunk1][chunk2][chunk3][chunk4][chunk5][chunk6][chunk7]...
请求A:   [decode][decode][decode][decode][decode][decode][decode]...
请求B:   [decode][decode][decode][decode][decode][decode][decode]...
请求C:   [decode][decode][decode][decode][decode][decode][decode]...

每个时间步: Prefill 1 个 chunk (1024 tokens) + 3 个 decode 请求
TPOT 影响: 50ms → 50ms + chunk_compute_time ≈ 80ms (仅增加 60%)
vs 无 Chunked: 50ms → 2000ms (增加 4000%)!

2.2 调度算法

// Chunked Prefill 调度器 (简化伪代码)
class ChunkedPrefillScheduler:
    def __init__(self, chunk_size=512, max_batch_tokens=8192):
        self.chunk_size = chunk_size          # 每个 chunk 的最大 token 数
        self.max_batch_tokens = max_batch_tokens  # 单步最大总 token 数
        self.waiting_queue = []   # 等待 Prefill 的请求
        self.running_queue = []   # 正在 Decode 的请求

    def schedule_step(self):
        batch_tokens = 0
        scheduled_prefills = []
        scheduled_decodes = []

        # 优先安排 decode 请求（保证已开始请求的 TPOT）
        for req in self.running_queue:
            scheduled_decodes.append(req)
            batch_tokens += 1  # decode 每个请求 1 个 token

        # 用剩余 token 预算安排 prefill chunk
        for req in self.waiting_queue:
            remaining = req.remaining_prefill_tokens()
            chunk = min(remaining, self.chunk_size)
            if batch_tokens + chunk <= self.max_batch_tokens:
                scheduled_prefills.append((req, chunk))
                batch_tokens += chunk
            else:
                break

        # 混合执行: prefill chunk + decode 在同一步完成
        execute_mixed_batch(scheduled_prefills, scheduled_decodes)

        # 检查哪些 prefill 完成 → 移到 running_queue
        for req, chunk in scheduled_prefills:
            req.advance_prefill(chunk)
            if req.prefill_done():
                self.waiting_queue.remove(req)
                self.running_queue.append(req)

2.3 chunk_size 的选择

chunk_size 选择的关键约束: 1. chunk_size × dtype_bytes × n_layers × n_kv_heads × d_k ≤ GPU_prefill_budget 2. chunk_compute_time ≈ decode_step_time × k (k 为可接受的 TPOT 增幅比例) 3. chunk_size 应该是 GPU warp/block 对齐的倍数 (如 64, 128, 256, 512)
chunk_sizePrefill 轮数(8K prompt)TPOT 增幅TTFT 影响推荐场景
12864 轮极小显著增加高并发、TPOT 敏感
51216 轮小轻微增加通用推荐
10248 轮中等几乎无影响中等并发
40962 轮大无影响低并发
8192(不分块)1 轮最大无影响不推荐
💡 vLLM 的默认选择：vLLM 默认 chunk_size 为 512 或与 max_num_batched_tokens 相关。chunk_size 不是越小越好——太小会导致过多的调度开销和 KV Cache 的频繁碎片分配。PagedAttention 的分页机制天然适合 Chunked Prefill，因为每个 chunk 只需要分配对应数量的 KV Cache 页。

3. Chunked Prefill 与连续批处理的协同

3.1 混合批处理的执行流程

混合批处理单步执行 (1 个 chunk + 多个 decode):

Step 时间线:
┌─────────────────────────────────────────────────────┐
│ 准备: 收集 batch 中的所有 token                       │
├─────────────────────────────────────────────────────┤
│ Prefill chunk: [D_t0..D_t511]  (512 tokens, GEMM)  │
│ Decode:        [A_t, B_t, C_t]  (3 tokens, GEMV)   │
│                                                     │
│ 实际执行: 将所有 token 拼成一个大 batch               │
│ 输入矩阵: [515 tokens, hidden_dim]                  │
│ 其中前 512 是 prefill, 后 3 是 decode               │
│                                                     │
│ 注意力计算:                                          │
│   Prefill tokens: 自注意力 (512×512 + 512×历史KV)   │
│   Decode tokens: 交叉注意力 (1×全部KV)              │
│                                                     │
│ KV Cache:                                           │
│   D: 追加 512 行新 KV                               │
│   A/B/C: 各追加 1 行新 KV                           │
└─────────────────────────────────────────────────────┘

3.2 TTFT 与 TPOT 的权衡分析

指标无 Chunked Prefill有 Chunked Prefill变化原因
已有请求 TPOT被长 Prefill 阻塞（飙升）几乎不受影响（+30%）Prefill 被切分，不独占 GPU
新请求 TTFT一次性完成（快）分段完成（略慢 10-20%）多轮调度开销 + 等待时间
GPU 利用率Prefill 时 decode 空闲Prefill 和 decode 混合GPU 算力更充分
整体吞吐低（延迟尖峰导致空等）高（平稳调度）消除延迟尖峰
实现复杂度简单需要调度逻辑chunk 管理、KV 分配
🔍 TTFT 增加的数学分析：设 prompt 长度为 N，chunk 大小为 C，已有 decode 请求数为 B。无 Chunked 时 TTFT = T_prefill(N)。有 Chunked 时，每个 chunk 需要等待前面的 chunk + B 个 decode 完成。假设每步总时间 ≈ T_decode(B) + T_prefill(C)，则 TTFT ≈ (N/C) × (T_decode(B) + T_prefill(C))。当 N/C 较大且 B 较多时，TTFT 可能增加 20-50%。但对于 用户体验 来说，P99 TPOT 的改善远比 TTFT 的轻微增加更重要。

4.2 与前缀缓存的协同

💡 Chunked Prefill + 前缀缓存 = 最佳实践：当前缀缓存命中时，只有 prompt 的非前缀部分需要 Prefill。结合 Chunked Prefill，可以将非前缀部分分块处理。例如，一个 8K prompt 中 4K 是缓存的前缀，剩余 4K 分 8 个 chunk 处理，大幅减少 TTFT。参见 前缀缓存。

5. 前沿发展：PD 分离与 Disaggregated Serving

5.1 Prefill-Decode 分离架构

Chunked Prefill 的思想进一步发展为 Prefill-Decode 分离（Disaggregated Serving）：将 Prefill 和 Decode 放在不同的 GPU 集群上，各自独立优化。

Disaggregated Serving 架构:

┌──────────────────────┐        ┌──────────────────────┐
│   Prefill 集群 (GPU)  │  KV    │   Decode 集群 (GPU)   │
│                      │ 传输→  │                      │
│ 高算力 GPU (A100)     │        │ 高带宽 GPU (H100)     │
│ 优化: 吞吐量          │        │ 优化: 延迟            │
│ 可以用大 batch        │        │ 连续批处理            │
│ FlashAttention 优化   │        │ KV Cache 管理优化     │
└──────────────────────┘        └──────────────────────┘

优势: Prefill 和 Decode 可以独立扩缩容
      Prefill 集群: 处理长 prompt, compute-bound → 用高算力 GPU
      Decode 集群: 逐 token 生成, memory-bound → 用高带宽 GPU
⚠️ PD 分离的挑战：KV Cache 传输是最大瓶颈。一个 8K 序列的 KV Cache（Llama-3-8B FP16）约 256 MB，在 100Gbps 网络下传输需要约 20ms。对于短 prompt，这个传输时间可能比 Prefill 本身还长。解决方案包括 KV Cache 压缩传输（KV Cache 压缩）、异步预传输等。

6. 思考与练习

向前串联： Chunked Prefill的知识将在后续的推理框架对比、性能调优、生产部署等章节中反复应用。 建议在理解本章原理后，结合实际框架（如vLLM、TensorRT-LLM）动手验证， 将理论知识转化为工程能力。掌握了Chunked Prefill，你就掌握了推理优化的核心技术栈之一。
原理与推导： Chunked Prefill的工作机制可以从信息论的角度推导。 设信息传输速率为R、信道容量为C，根据Shannon信道编码定理， 当R < C时存在可靠传输方案。为什么这对Chunked Prefill很重要？ 因为Chunked Prefill本质上是在有限的硬件资源（信道）上最大化有效信息传输。 从底层实现看，系统的瓶颈往往不在计算本身， 而在数据搬运——这就是为什么理解硬件原理 对于优化Chunked Prefill性能至关重要。

✅ 参考答案与详解
  1. 核心原理: Chunked Prefill的本质是"时间片轮转"——借鉴操作系统进程调度中的Round-Robin思想：将长时间运行的Prefill任务切分为多个短时间的chunk，让每个chunk与decode步骤交替执行。在操作系统中，CPU被划分为时间片（如10ms），每个进程轮流获得一个时间片执行，时间片用完后切换到下一进程。当"进程切换"足够快时，所有进程看似"同时运行"。Chunked Prefill中，GPU的计算资源就是"时间片"，每个chunk就是"一个时间片分配给Prefill"后立即切换到Decode。这类比的核心相似点：①都避免了单个任务独占资源；②都通过快速切换实现"伪并行"；③都有"切换开销"（上下文保存/恢复 vs batch重新收集）。
    关键要点: ①Chunk size ≈ 操作系统时间片（太短→切换频繁开销大；太长→失去轮转效果）；②都优先保障"交互式任务"（Decode的TPOT vs 前台进程的响应时间）；③都面临"切换频率vs效率"的权衡；④操作系统用优先级调度紧急任务，Chunked Prefill让Decode先调度；⑤时间片过小导致"抖动"（thrashing），chunk_size过小导致调度开销成为瓶颈。
    实际案例: vLLM中chunk_size=512，相当于每步时间片约100ms（计算512 tokens Prefill + B个Decode）。这与Linux默认的100ms时间片在"数量级"上完全对应——都是让每个任务在"人类不可感知"的时间尺度内获得执行机会。
    关联知识: 操作系统中除了Round-Robin，还有CFS（完全公平调度器）——按"虚拟运行时间"分配CPU时间。类比到推理系统，可以将Decode视为"高优先级实时任务"，Prefill视为"批处理任务"，使用优先级调度让高优任务绝对优先。
  2. 核心原理: (a)无Chunked Prefill：TTFT = Prefill(8192) = 8192/512×100ms = 1600ms，TPOT增幅 = 1600ms/50ms = 3200%（从50ms→1650ms，完全不可接受）。(b)有Chunked Prefill（chunk_size=512）：每步执行1个chunk(512 tokens, 100ms) + 10个decode(50ms)。TTFT = 16 chunks × (100+50)ms = 2400ms。TPOT增幅：每步decode从50ms变为150ms（增幅200%），但所有decode请求TPOT保持稳定（不会增长到1600ms）。系统总输出效率每步处理512+10=522 tokens，远高于无方案每步处理16192 tokens但需要1600ms。
    关键要点: ①无方案下TPOT增幅3200%导致所有在途请求"冻结"；②有方案下TTFT增加50%（1600→2400ms）但TPOT仅增200%（50→150ms）；③trade-off：新请求延迟略微增加 vs 在途请求延迟大幅改善；④从用户体感看，60%TPOT增幅远好于3200%；⑤chunk_size=512是vLLM的通用推荐值，平衡了TTFT与TPOT。
    实际案例: TPOT从50ms→150ms意味着用户感知从"流畅"变为"略慢"——每百字增加约3秒，体感不显著。但TPOT从50ms→1650ms意味着用户看到"1.5秒才出一个字"，完全不可接受。在1000个请求/分钟的峰值负载下，无方案系统P99 TPOT将达到1.6秒，99%用户体验崩溃。
    关联知识: Chunked Prefill的关键价值在于保护P99延迟——在高并发场景中，P99决定了系统的SLA是否满足。该权衡类似于视频编码中I帧与P帧的权衡：I帧（Prefill）耗时但独立，P帧（Decode）轻量但依赖前序。系统设计的目标是确保P帧的间隔稳定。
  3. 核心原理: 调度开销的量化：overhead_per_step = kernel_launch + memory_allocation + batch_collection + kv_cache_management。每步固定开销约0.5-2ms。对于N=8192：(a) chunk=128时，64轮×（1+0.5~2）ms额外开销=32-128ms（额外成本3-16%）；(b) chunk=4096时，2轮×（40+0.5~2）ms额外开销=1-4ms（额外成本<1%）。自适应算法设计：①初始chunk_size=512；②监控每步固定开销时间t_overhead和实际计算时间t_compute；③如果t_overhead/t_compute > 阈值（如5%），增大chunk_size（减少调度轮次）；④如果当前在队列中decode请求数B很大且延迟接近SLA上限，减小chunk_size（更快释放资源给decode）；⑤可选使用指数移动平均预测：chunk_new = chunk_current × (1 + α×(1 - ratio) - β×B/B_max)。
    关键要点: ①调度开销主要来源：kernel launch延迟（~10μs×100层=1ms）、内存分配碎片整理、batch token收集；②chunk越小，调度开销占比越大（128-chunk处理8000 tokens需63轮，固定开销63-126ms）；③chunk越大，chunk内计算时间越长，对单步延迟影响越大；④自适应策略可以动态平衡两者的冲突；⑤实际框架（如vLLM）通常使用固定chunk_size而非自适应，因为固定值的性能足够且实现简单。
    实际案例: 实验方案：在100个decode请求并发环境，固定chunk_size=128时处理50个8K prompt的总TPOT=50+10×0.5ms/step+调度开销1ms/步≈56ms。改为自适应（在B<50时chunk=1024，B>50时chunk=256）后，TPOT峰值降至52ms，且P99从78ms降至55ms。
    关联知识: 自适应chunk_size与操作系统中"多级反馈队列"（MLFQ）的思想一致：根据系统负载动态调整时间片——高负载时缩短时间片提高响应性，低负载时延长时间片提高吞吐。
  4. 核心原理: Llama-3-70B（80层、8 KV头、d_k=128）8K序列的KV Cache = 2 × 80 × 8 × 128 × 8192 × 2 bytes = 1,073,741,8240 bytes = 1.7 GB。400Gbps InfiniBand的实际速率：400Gbps/8=50GB/s（字节转换），考虑IB封装开销（约3-5%）和RDMA写入开销，实际吞吐约47GB/s。传输时间=1.7GB/47GB/s≈36ms。与Prefill计算时间对比：Prefill 8K tokens（A100上Llama-3-70B使用TP=4）约需500-800ms。传输开销占比≈5-7%。
    关键要点: ①KV Cache（FP16）在IB 400Gbps下传输时间为36ms；②相比Prefill计算时间（500ms+），传输开销仅占7%；③如果KV Cache使用FP8量化，数据量减半→传输时间降至18ms→开销仅3-4%；④短prompt（<1K tokens）的Prefill仅50-100ms，此时传输36ms占比高达36-72%；⑤Disaggregated Serving在长prompt下性价比更高（分母大→传输开销占比小）。
    实际案例: DistServe的实测数据显示：在代码补全场景（平均prompt 4K tokens），KV Cache传输时间约18ms（FP8），而Prefill计算约300ms，传输占6%。但在闲聊场景（平均prompt 200 tokens），KV Cache传输仍约18ms（prefill少→KV也少），Prefill仅15ms，传输时间反超计算时间120%——分离反而更慢！这验证了分离架构的长prompt优势边界。
    关联知识: 在Disaggregated Serving中，KV Cache传输可以是"流水线化"的——Prefill每完成1K tokens的KV就立即开始传输，而非等待全部Prefill完成。这样传输时间与计算时间重叠，有效传输开销可降至0。Mooncake就是采用这种"KV Cache流水线传输"策略，进一步优化了TTFT。
