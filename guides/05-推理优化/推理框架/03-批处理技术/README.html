前面提到，Decode 阶段是 Memory-bound 的——GPU 大部分时间在等数据从显存读取。如果一次只处理 1 个请求，GPU 利用率可能只有 10-20%。

批处理的核心思想：同时处理多个请求，让 GPU 一次加载的模型权重被多个请求"共享"，从而提高 GPU 利用率和总吞吐量。

静态批处理的致命问题：
  • 一个 batch 中，完成早的请求必须等完成晚的请求
  • GPU 在等待期间完全空闲
  • 请求长度差异越大，浪费越严重
  • 典型场景下 GPU 利用率可能只有 30-50%

3.1 核心思想

不要等所有请求完成！每当一个请求完成，立即用新请求填补空位。这就是"连续"的含义——batch 在时间维度上是连续流动的。

连续批处理 (Continuous Batching) 时间 → Step1 Step2 Step3 Step4 Step5 Step6 Step7 Step8 请求 A: ██ ██ ██ ██ ██ ██ ██ ██ ← 完成 请求 B: ██ ██ ██ ██ (完成) 请求 C: ██ ██ ██ ██ ██ ██ (完成) ───────────────────────────────────────────────────── 请求 D: ██ ██ ██ ██ ██ ← 新加入 请求 E: ██ ██ ██ ██ ← 新加入 GPU 利用率：始终满载！每个 Step 都在处理请求。 关键：以 "iteration"（迭代步）为单位调度，而不是以 "request" 为单位。

3.2 为什么叫 "Iteration-level" Batching

连续批处理也叫 Iteration-level Batching，因为调度决策在每个 decode iteration（解码迭代步）都会重新做出：

  • 每一步检查：哪些请求完成了？→ 移出 batch
  • 每一步检查：队列中有新请求？→ 移入 batch
  • 每一步检查：GPU 内存够不够？→ 决定 batch 大小

4.1 问题：Prefill 和 Decode 的计算特征不同

Prefill 是 Compute-bound（计算密集），Decode 是 Memory-bound（内存带宽密集）。如果把新请求的 Prefill 和正在进行的 Decode 混在一起，Prefill 的大计算量会阻塞 Decode，导致正在生成的请求卡住。

4.2 Chunked Prefill（分块预填充）

将长 prompt 的 Prefill 分成多个 chunk，每个 chunk 与 Decode 请求混合执行：

Chunked Prefill 长 prompt (1000 tokens) 被分成 4 个 chunk: Step 1: [Prefill chunk 1] + [Decode A] + [Decode B] + [Decode C] Step 2: [Prefill chunk 2] + [Decode A] + [Decode B] + [Decode C] Step 3: [Prefill chunk 3] + [Decode A] + [Decode B] + [Decode C] Step 4: [Prefill chunk 4] + [Decode A] + [Decode B] + [Decode C] Step 5: [Decode D (新)] + [Decode A] + [Decode B] + [Decode C] 好处： 1. Decode 请求不会被长 Prefill 阻塞（TPS 稳定） 2. Prefill 的计算被"打散"到多个 Step，与其他请求共享 GPU 3. TTFT 可能略增，但整体系统更公平、更高效

5.1 先来先服务（FCFS）

最简单：按请求到达顺序处理。公平但不考虑请求特征。

5.2 最短作业优先（SJF）

优先处理短请求（输出 token 少的），减少平均延迟。但可能导致长请求饿死。

5.3 内存感知调度

根据当前 GPU 内存余量决定是否接受新请求，避免 OOM（内存溢出）。

5.4 Preemption（抢占）

当内存不足时，暂停低优先级请求，释放其 KV Cache，等内存充裕时恢复。这是 vLLM 和 SGLang 都支持的特性。

一、为什么需要批处理？的技术原理与数学推导： 从底层硬件角度分析，一、为什么需要批处理？的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了一、为什么需要批处理？的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
一、为什么需要批处理？的系统架构分析： 从系统设计的原理出发，一、为什么需要批处理？采用分层架构来管理复杂性。 为什么分层是主流选择？推导如下： 设系统有N个组件，全连接交互复杂度为O(N²)，分层后降低到O(N·L)，L为层数。 数学本质上，这是用少量层间通信开销换取大幅降低的组件间耦合。 在实际工作机制中，每一层只与相邻层通信，通过标准接口解耦—— 这类似于TCP/IP协议栈的设计原理，每层独立优化，层内改动不影响整体。

一、为什么需要批处理？的性能优化策略

优化一、为什么需要批处理？性能需要系统性的方法论。首先用profiling工具定位瓶颈—— 从底层看，GPU推理的瓶颈通常表现为三种模式： 1) 计算受限（GPU利用率>80%，SM活跃）； 2) 内存受限（带宽利用率高，但计算单元空闲）； 3) 通信受限（多卡场景，NCCL等待时间长）。 为什么要区分这三种模式？因为优化策略完全不同： 计算受限→减少FLOPs（量化/剪枝）；内存受限→减少数据搬运（算子融合/缓存）； 通信受限→优化通信拓扑（梯度压缩/异步通信）。 这一原理是推理性能调优的基础框架。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

✅ 参考答案与详解
  1. 核心原理: LLM推理的Decode阶段是Memory-bound的——GPU大部分时间在等待数据从HBM显存传输到计算单元，而非在做计算。以A100为例，FP16峰值算力312 TFLOPS，HBM带宽仅2TB/s，算术强度转折点为156 FLOPs/Byte。Decode阶段的算术强度远低于此阈值，因此GPU计算单元大量空闲。如果一次只处理1个请求，GPU利用率可能仅10-20%。批处理的核心思想是将多个请求合并为一个batch同时处理，让GPU一次加载的模型权重被多个请求"共享"。这样，相同的显存带宽读取一次权重，却可以同时服务N个请求，等效地将权重读取的带宽成本分摊到N个请求上，从而大幅提升GPU利用率和总吞吐量。这类似于超市收银台——单个顾客结账效率很低，多个顾客排队(batch)处理才能充分利用收银员(GPU)的时间。
    关键要点: ①Decode阶段是Memory-bound，GPU利用率低 ②批处理通过共享权重读取分摊带宽成本 ③batch越大，GPU利用率越高(受限于显存容量) ④批处理是所有后续优化技术(连续批处理、PagedAttention)的基础
    实际案例: 使用LLaMA-7B模型在A100上测试：batch_size=1时吞吐约50 tokens/s，GPU利用率仅12%；batch_size=32时吞吐达到1200 tokens/s，GPU利用率提升到78%。总吞吐提升24倍，而单个请求的延迟仅增加15%，这就是批处理的威力。
    关联知识: 批处理与Continuous Batching、PagedAttention、Chunked Prefill紧密相关。批处理是基础思想，Continuous Batching解决静态批处理的等待浪费问题，PagedAttention解决动态batch的内存管理问题，Chunked Prefill解决Prefill与Decode混合调度的问题。
  2. 核心原理: 静态批处理(Static Batching)是最简单的批处理实现方式：收集一组请求，凑满一个batch后一起处理，所有请求完成后才开始下一批。其工作流程是：调度器等待batch填满→所有请求同时开始Prefill→所有请求进入Decode循环→等待batch中最慢的请求完成→释放整个batch→开始处理下一批。这种方案的最大问题是"气泡(bubble)"现象——由于不同请求的输出长度差异巨大(有的10个token就完成，有的需要500个token)，完成早的请求必须等待完成晚的请求，期间GPU完全空闲。从图示可以看到，请求A、B、C的完成时间不同，但都要等到最后才能释放batch，等待期间GPU资源被严重浪费。请求长度差异越大，浪费越严重，典型场景下GPU利用率可能只有30-50%。
    关键要点: ①以整个batch为单位进行调度，粒度过粗 ②短请求必须等待长请求完成，产生大量GPU空闲时间 ③实现简单但效率低下 ④GPU利用率仅30-50% ⑤被Continuous Batching全面取代
    实际案例: 假设一个batch包含3个请求：请求A输出10个token(完成时间20ms)，请求B输出50个token(完成时间100ms)，请求C输出200个token(完成时间400ms)。在静态批处理下，三个请求都必须等待400ms才能释放。GPU在20-400ms期间有380ms在"空等"请求B和C完成，利用率不足50%。如果改用Continuous Batching，请求A在20ms完成后立即让出位置给新请求D，GPU始终保持满载。
    关联知识: 静态批处理是理解Continuous Batching的必经之路——只有理解了静态批处理的"气泡"问题，才能深刻理解Continuous Batching为何要以iteration为调度单位。同时，静态批处理的内存分配模式也是理解PagedAttention动机的关键——静态批处理需要为每个请求预分配最大可能的KV Cache内存，导致大量内存浪费。
  3. 核心原理: 连续批处理(Continuous Batching)也叫Iteration-level Batching，是现代推理框架(vLLM、SGLang、TensorRT-LLM)的核心调度技术。其核心思想是：不再以"整个batch"为调度单位，而是以每个decode iteration(解码迭代步)为单位进行调度决策。在每一步中，调度器执行三个检查：①哪些请求生成了结束标记(EOS)？→ 立即移出batch，释放其KV Cache内存；②队列中是否有等待的新请求？→ 如果GPU内存足够，立即移入batch填补空位；③当前batch大小是否接近显存上限？→ 控制并发数避免OOM。这样batch在时间维度上是"连续流动"的——有请求完成就立即替换，GPU每个step都在高效工作，完全没有空闲等待。相比静态批处理，吞吐量可提升2-10倍。
    关键要点: ①以iteration为调度粒度，而非整个batch ②请求完成即释放，新请求立即填补，GPU零空闲 ③吞吐量提升2-10倍 ④需要PagedAttention等精细内存管理配合 ⑤是vLLM/SGLang/TRT-LLM的核心技术
    实际案例: 某API服务使用vLLM部署Llama-2-13B模型。在静态批处理模式下，每秒处理80个请求，平均延迟120ms；切换到Continuous Batching后，同样硬件每秒处理350个请求(提升4.4倍)，平均延迟反而降至80ms(提升1.5倍)。关键改进在于：短请求(如翻译"Hello"→"你好")完成后立即让位，不再等待同batch中的长请求(如写一篇500字的文章)。
    关联知识: Continuous Batching与PagedAttention是"天作之合"——Continuous Batching需要频繁地分配和释放KV Cache内存，PagedAttention通过分页机制避免了内存碎片。同时与Chunked Prefill配合：Continuous Batching负责Decode阶段的高效调度，Chunked Prefill解决Prefill与Decode混合执行时的阻塞问题。
  4. 核心原理: Prefill和Decode的计算特征截然不同——Prefill是Compute-bound(计算密集)，需要并行处理所有输入token进行注意力计算；Decode是Memory-bound(内存带宽密集)，每个step只生成一个token，瓶颈在读取KV Cache。当新请求的Prefill与正在进行的Decode请求混合在同一batch中执行时，Prefill的大量矩阵运算会长时间占用GPU计算资源，阻塞Decode请求的逐token生成，导致正在流式输出的用户感受到卡顿(TPOT飙升)。Chunked Prefill(分块预填充)的解决方案是：将长prompt的Prefill分成多个小chunk(如每256 tokens一个chunk)，每个chunk与Decode请求混合执行。从图示可以看到，1000 tokens的prompt被分成4个chunk，每个step中既有Prefill chunk处理，又有Decode请求继续生成。这样Decode请求不会被长Prefill阻塞(TPS保持稳定)，Prefill的计算被"打散"到多个step与其他请求共享GPU。虽然TTFT(首token延迟)可能略有增加，但整体系统更公平、更高效。
    关键要点: ①Prefill是Compute-bound，Decode是Memory-bound，混合执行互相干扰 ②Chunked Prefill将长Prefill切分为小chunk分步执行 ③Decode请求不再被长Prefill阻塞，TPOT保持稳定 ④TTFT可能略增但系统整体更公平 ⑤是现代推理框架处理长prompt的标配技术
    实际案例: 一个RAG系统中，检索到的文档片段拼接后prompt约2000 tokens。不使用Chunked Prefill时，每次新请求加入都会导致正在进行的Decode请求TPOT从30ms飙升到180ms(持续约50ms的Prefill时间)。使用Chunked Prefill(每chunk 256 tokens)后，Prefill被分成8个chunk分散到8个step中执行，Decode请求的TPOT波动从30-180ms缩小到30-50ms，用户几乎感觉不到卡顿。
    关联知识: Chunked Prefill与Continuous Batching深度配合——Continuous Batching提供iteration级调度框架，Chunked Prefill在此框架内将Prefill请求切分为多个可调度的chunk。同时与Prefill/Decode分离架构(Disaggregated Serving)互补：如果采用分离架构，Prefill集群内部也可以使用Chunked Prefill来优化长prompt的处理。
  5. 核心原理: 调度策略决定了在Continuous Batching的每个iteration中，哪些请求被优先处理、哪些请求被暂停或驱逐。文中介绍了四种主要调度策略。FCFS(先来先服务)：按请求到达顺序处理，简单公平但不考虑请求特征——可能导致短请求被迫等待长请求。SJF(最短作业优先)：优先处理输出token数少的短请求，减少平均等待时间，但可能导致长请求"饿死"。内存感知调度：根据当前GPU显存余量决定是否接受新请求，避免OOM——这是生产环境中最关键的保障机制。Preemption(抢占)：当内存不足时，暂停低优先级请求并释放其KV Cache，等内存充裕时恢复(vLLM和SGLang都支持)。这四种策略可以组合使用：例如先用SJF排序，再用内存感知过滤，最后用Preemption作为安全网。
    关键要点: ①FCFS公平但不优化延迟指标 ②SJF优化平均延迟但可能饿死长请求 ③内存感知调度是防OOM的安全保障 ④Preemption通过暂停/恢复机制应对突发高负载 ⑤实际系统通常组合多种策略
    实际案例: 某在线API服务部署了vLLM，设置了如下组合调度策略：新请求进入FCFS队列→内存感知模块检查当前显存使用率(阈值90%)→如果未满则按SJF优先级排序(短请求优先)→当显存超过95%时触发Preemption，暂停最近最少活跃的请求释放其KV Cache。这套策略将P99延迟从500ms优化到150ms，同时保证了零OOM崩溃。特别在流量高峰时(如促销活动QPS翻倍)，Preemption机制确保了系统不会因内存溢出而宕机。
    关联知识: 调度策略与PagedAttention紧密配合——Preemption需要PagedAttention支持KV Cache的高效回收和恢复。同时与Continuous Batching的iteration级调度框架配合：调度策略在每个iteration被调用，决定当前step的batch组成。此外，调度策略的选择也受到SLA(服务等级协议)的影响——如果SLA要求严格的最大延迟，FCFS可能比SJF更合适，因为SJF可能让长请求无限延迟。
