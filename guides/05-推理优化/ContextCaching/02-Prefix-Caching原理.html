1. 核心思想

核心思想

s="tip">通俗类比：把前缀缓存想象成餐厅的“招牌菜单”。客人点相同的开胃菜（系统提示+历史对话），厨房不用从头备料，直接把准备好的半成品端上，节省了宝贵的备餐时间。
常见误区：常见的误解是认为整个请求都能被缓存。实际上，只有请求序列最前面相同的“前缀”部分可以复用，动态变化的用户查询部分无法被缓存。
深入探讨：原理基于Transformer的注意力机制。计算一次前缀的KV缓存后，将其存储在GPU显存中。后续请求以相同前缀开始时，可直接复用这些KV值，避免了重复计算。假设单次前缀计算耗时为T，则N个请求的缓存命中可节省(N-1)*T的计算时间。

缓存未命中完整计算高延迟 (T_prefix)
缓存命中跳过计算，加载KV低延迟 (T_load << T_prefix)

2. vLLM Automatic Prefix Caching (APC)

通俗类比：vLLM的APC就像一位智能前台。当两位客人点了相同的套餐前菜（前缀）时，前台会默默记住第二位客人的要求，并悄悄通知厨房直接上第一份的半成品，无需再次询问。
常见误区：启用APC需在引擎初始化时设置`enable_prefix_caching=True`。若未启用，即使请求完全相同，vLLM也不会自动利用缓存，可能导致不必要的重复计算。
深入探讨：APC的核心是维护一个全局的“块映射表”。它将前缀序列切分为固定大小的块（如16个token），为每个块计算哈希值作为标识。新请求到来时，vLLM会前向扫描其前缀的哈希，若在映射表中找到匹配项，则直接复用对应块的KV缓存。
vLLM APC 工作流程:
1. 扫描前缀 -> 计算哈希
2. 哈希匹配？ -> 复用KV块
3. 哈希不匹配 -> 计算新KV块
4. 存储新KV块到缓存池
配置项作用默认值
enable_prefix_caching启用/禁用自动前缀缓存False
block_size缓存块的大小（token数）16
num_prefix_blocks系统预留的缓存块数量自动调整

2.1 Hash-based 匹配

# vLLM 的 KV Cache Block 结构
class KVCacheBlock:
    block_id: int           # Block 编号
    block_hash: int         # 内容哈希 (tokens + 前缀)
    ref_cnt: int            # 引用计数
    prev: KVCacheBlock      # LRU 双向链表
    next: KVCacheBlock

# Hash 计算方式
def compute_block_hash(block_tokens, prefix_tokens):
    """每个 block 的 hash 唯一标识其内容和位置"""
    # 包含所有前缀 token，确保相同内容在相同位置有相同 hash
    return sha256(block_tokens + prefix_tokens)

# 示例:
# Block 1: tokens=["Hello", " world"] + prefix=[] → hash_1
# Block 2: tokens=["!"] + prefix=["Hello", " world"] → hash_2
# 如果另一个请求也有相同的 ["Hello", " world"] 前缀
# → Block 1 的 hash 匹配 → 复用 KV Cache

2.3 LRU Eviction 策略

# vLLM 的 LRU 驱逐
class BlockPool:
    def __init__(self, num_blocks):
        self.blocks = [KVCacheBlock(i) for i in range(num_blocks)]
        self.free_queue = DoublyLinkedList()
        self.hash_table = {}  # hash → block

    def allocate(self, block_hash):
        """分配一个 block，优先复用缓存"""
        # 1. 检查缓存命中
        if block_hash in self.hash_table:
            block = self.hash_table[block_hash]
            block.ref_cnt += 1
            return block

        # 2. 从 free_queue 获取空闲 block
        block = self.free_queue.pop_tail()
        block.block_hash = block_hash
        block.ref_cnt = 1
        self.hash_table[block_hash] = block
        return block

    def free(self, block):
        """释放一个 block"""
        block.ref_cnt -= 1
        if block.ref_cnt == 0:
            self.free_queue.push_head(block)

    def evict(self):
        """驱逐最久未使用的 block"""
        block = self.free_queue.pop_tail()
        if block.block_hash in self.hash_table:
            del self.hash_table[block.block_hash]
        return block

3. SGLang RadixAttention

通俗类比：RadixAttention像一个基于“订单树”的厨房管理法。所有订单（请求序列）按共同的前缀被组织成一棵树。结点共享前菜（KV缓存），新订单只需从树上“生长”出自己的枝叶。
常见误区：RadixAttention的树结构会引入额外的内存管理开销。对于请求前缀高度不一致、共享度很低的场景，其维护树结构的开销可能超过缓存带来的收益。
深入探讨：它使用Radix Tree（基数树）来管理缓存。每个树节点对应一个序列片段（token子串）及其KV缓存。匹配前缀时，沿树进行最长前缀匹配（LPM）。例如，两个请求`[A,B,C,D]`和`[A,B,X,Y]`会共享树路径`[A,B]`的KV，然后在`B`节点分叉。
Radix树示例 (Root: 空)
        [A,B]
        /    \
    [C,D]   [X,Y]
     ↑         ↑
请求1的缓存  请求2的缓存

3.2 Radix Tree 操作

# Radix Tree 的核心操作

class RadixNode:
    def __init__(self):
        self.children = {}  # token → RadixNode
        self.kv_cache = None  # 缓存的 KV Cache
        self.ref_cnt = 0

class RadixTree:
    def __init__(self):
        self.root = RadixNode()

    def insert(self, tokens, kv_cache):
        """插入 token 序列和对应的 KV Cache"""
        node = self.root
        for token in tokens:
            if token not in node.children:
                node.children[token] = RadixNode()
            node = node.children[token]
        node.kv_cache = kv_cache
        node.ref_cnt += 1

    def longest_prefix_match(self, tokens):
        """查找最长公共前缀"""
        node = self.root
        match_len = 0
        cached_kv = None

        for i, token in enumerate(tokens):
            if token in node.children:
                node = node.children[token]
                match_len = i + 1
                if node.kv_cache is not None:
                    cached_kv = node.kv_cache
            else:
                break

        return match_len, cached_kv

    def evict_lru(self):
        """驱逐最久未使用的叶节点"""

        # 释放其 KV Cache

4. Cache-aware Scheduling

Cache-aware Scheduling

通俗类比：这就像一位经验丰富的餐厅领班，不仅按客人到来顺序排号，还会看哪批客人点了相似的套餐（共享前缀），把他们安排在同一时段让厨房集中处理，提升整体出餐速度。
常见误区：Cache-aware Scheduling可能让个别请求的等待时间（排队延迟）增加。因为它为了最大化缓存命中率，可能会让一个拥有“热门前缀”的请求稍微等待，凑齐一批相似请求一起处理。
深入探讨：调度器在为新批次（batch）选择请求时，会优先考虑那些能与当前批次中已有请求共享前缀的待处理请求。这需要调度器维护待处理请求队列的“前缀特征索引”。例如，批次中已有前缀`P1`，则拥有`P1`的请求`R2`比拥有完全不同前缀`P2`的请求`R3`有更高优先级。
传统FIFO调度：[R1, R2, R3, R4] -> 依次处理
Cache-aware调度：
队列: [R1(P1), R2(P2), R3(P1)]
当前批次: [R1(P1)]
-> 优先选择 R3(P1) 加入批次，提高缓存复用
调度目标FIFO (先进先出)Cache-aware
主要依据请求到达时间到达时间 + 前缀匹配度
优化目标公平性系统吞吐量（吞吐量）
对延迟影响可预测可能增加部分请求的排队延迟

5. Eviction 策略对比

Eviction 策略对比

通俗类比：缓存淘汰策略就像厨房备料区的管理。空间有限时，是优先清理最久没用的食材（LRU），还是用量最少的食材（LFU），或是考虑食材贵重程度（成本）？这直接关系到能否快速应对新订单。
常见误区：在LLM服务中，简单的LRU策略可能不够用。因为一次长对话可能产生一个很长的缓存，它可能很久没被访问（LRU中优先级高），但它占据的内存空间巨大，不清理会严重影响新请求的缓存能力。
深入探讨：常见策略：1. **LRU (Least Recently Used)**：淘汰最久未被访问的缓存块。2. **LFU (Least Frequently Used)**：淘汰被访问频率最低的缓存块。3. **Cost-aware**：结合缓存块大小（token数）和生成成本，淘汰`大小 * 计算成本`最高的块。vLLM默认使用基于块的LRU。
内存缓存池 (容量=3块): [A, B, C]
新请求需要缓存 D，但池满。
LRU: A最久未用 -> 淘汰A，放入D -> [B, C, D]
LFU: B被访问最少 -> 淘汰B，放入D -> [A, C, D]
策略淘汰依据优点缺点
LRU最久未使用的时间简单，对时间局部性友好可能淘汰大而有价值的缓存
LFU历史访问频率对访问模式稳定的应用好历史数据可能过时，新缓存易被淘汰
Cost-aware大小与计算成本最大化缓存价值计算和评估开销较大

6. 前缀匹配的边界情况

通俗类比：处理前缀匹配就像处理客人的“特殊要求备注”。客人说“套餐A，但酱料换成上次那种”，系统不仅要匹配套餐A（前缀），还要正确关联“上次那种”这个上下文（历史），否则缓存可能失效或用错。
常见误区：最容易出错的是`从句前缀`和`时间敏感前缀`。例如，两个请求的系统提示词完全相同，但一个要求以JSON格式输出，另一个则无要求。若不将格式指令严格包含在前缀中，会导致缓存污染。
深入探讨：1. **严格前缀**：两个序列必须从第一个token开始就完全一致。2. **从句前缀**：系统可能支持在序列中间进行匹配，但需要复杂的逻辑。3. **带时间戳的前缀**：若前缀中包含动态生成的时间，每次都会变化，导致缓存完全失效。应将其从缓存的前缀中剥离。
有效缓存：
前缀 = [System, Context, ...]
请求A: [P, Q1]
请求B: [P, Q2] -> 命中P
无效缓存（污染）：
前缀 = [System] + [格式指令]
若[格式指令]未被正确纳入前缀，则缓存的是不完整的前缀。
边界情况示例处理建议
动态前缀前缀包含当前时间戳将其从缓存键中排除，或生成静态前缀
条件分支前缀末尾有可选内容确保所有分支都计算缓存，或使用Radix树管理
嵌入式变量系统提示中含{{用户名}}将用户名视为前缀的一部分，为不同用户分别缓存
