1. 为什么需要 Context Caching？

为什么需要 Context Caching？

类比：就像餐饮中的“老汤底”或“预制菜”，每次点单不用从头熬汤、备料，直接用现成的核心部分，快速出餐。Context Caching就是为LLM推理准备的“数字老汤底”。
常见误区：最大的误区是认为缓存万能。它适用于“固定核心+可变查询”的模式。如果输入每次都完全不同（如随机生成文本），缓存命中率会极低，反而增加了管理开销。
深入探讨：对于LLM，处理长Prompt的核心成本在于计算其所有token的Key-Value（KV）向量（称为Prefill阶段）。一个万token的Prompt，Prefill可能消耗数百毫秒和大量算力。缓存其KV向量后，后续请求可跳过此阶段，直接进入更便宜的Decoding阶段，实现“计算预算”的复用。

2. 典型场景

典型场景

通俗类比：想象一个健身房，会员（系统指令）每天来，但锻炼项目（用户查询）每天变。你不会每天重新介绍器械用法，而是把会员信息（Prompt）登记好（缓存），直接开始指导锻炼（回答问题）。
常见误区：避免在高度动态、个性化极强的场景（如实时聊天中无重复话题）强行套用缓存。识别场景是“模式稳定”的关键，盲目应用会导致缓存失效频繁，收益为负。
深入探讨：典型场景可通过“重复率”量化：1）系统指令+工具定义重复率≈100%（如智能体框架）。2）RAG中检索到的文档上下文，在针对同一文档集的连续查询中，重复率>80%。3）多轮对话中，历史上下文在窗口内复用率随对话深入递减。

3. 技术分类

技术分类

通俗类比：好比交通方式：Prompt Caching是“固定班车线路”（端到端缓存）；Prefix Caching是“地铁接驳”（复用最长公共前缀）；Logit/State Caching是“中途加油站”（保存中间计算结果）。
常见误区：切勿混淆“缓存粒度”。粗粒度缓存（如整个Prompt）命中率低但收益高；细粒度缓存（如Prefix）命中灵活但管理复杂。选择错误会导致要么存不住，要么存得少。
深入探讨：1. **Prefix Caching**：仅缓存公共前缀的KV向量。当新请求与缓存前缀匹配时，从匹配点开始计算。节省计算量 = 节省的token数 × (Prefill FLOPs/token)。2. **Prompt Caching**：将整个Prompt作为缓存单元。适用于精确重复的查询。3. **Logit Caching**：缓存生成每个token后的输出logits向量，用于后续投机解码或快速验证，空间开销大。

Prefill Savings = 3 * Cost(token)

4. 各框架支持

各框架支持

通俗类比：就像不同酒店集团的会员积分体系：VLLM是“万豪旅享家”，原生深度集成Prefix Caching；HuggingFace TGI是“希尔顿荣誉客会”，通过配置项灵活支持；各家API是“精品店”，直接提供托管的缓存服务。
常见误区：注意框架间的“缓存粒度”和“API语义”不同。例如，A框架的`enable_prefix_caching`可能只对前缀生效，而B框架的`cache_prompt`是全量缓存。直接迁移代码会导致缓存失效或错误。
深入探讨：工程实现对比：1）**VLLM**：在PagedAttention基础上原生实现Prefix Caching，通过哈希匹配前缀，对用户几乎透明。2）**TGI**：通过`--prefix-caching`或`--prompt-cache`参数开启，将缓存序列化到磁盘或Redis。3）**SGLang**：支持RadixAttention，将缓存组织为Radix树，支持更灵活的共享与更新。选择时需权衡推理吞吐、缓存命中率、运维复杂度。

5. 代码示例

代码示例

常见误区：代码中最易犯的错误是缓存键（Cache Key）设计不当。如果用动态时间戳或随机数作为键的一部分，会导致几乎每次请求都未命中缓存。键必须基于可复用内容的哈希值。
深入探讨：以VLLM为例，代码级缓存是透明的。当`enable_prefix_caching=True`时，引擎内部会自动处理。关键配置是`block_size`（如16），它决定了KV缓存的内存页大小。命中率 = (缓存匹配的token数) / (请求总token数)。可监控`vllm:cache_hit`指标进行优化。

# vLLM 自动前缀缓存
from vllm import LLM, SamplingParams

# 启用前缀缓存
llm = LLM(
    model="meta-llama/Llama-3-8B-Instruct",
    enable_prefix_caching=True  # 关键参数
)

# 共享系统 prompt 的多个请求
system_prompt = "你是一个专业的客服助手..."  # 2000 tokens

# 请求 1: 计算 system prompt 的 KV Cache
response1 = llm.generate([
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": "问题1"}
])

# 请求 2: 复用 system prompt 的 KV Cache
response2 = llm.generate([
    {"role": "system", "content": system_prompt},  # 缓存命中！
    {"role": "user", "content": "问题2"}
])
# 请求 2 的 TTFT 大幅降低

6. API 级别 Prompt Caching

代码示例

俗类比：就像自动售货机：你投入的硬币（Prompt）中，一部分是固定的“机器识别码”（缓存部分），另一部分是“商品选择”（查询）。识别码被识别后，你只需为选择付费，且投币过程更快。
常见误区：使用API缓存时，最常见的错误是误判“缓存失效”。API提供商通常有时间窗口（如5-30分钟无活动）或最小长度限制。在闲时测试导致缓存被清除，而在生产高峰时发现缓存未命中。
深入探讨：以Anthropic Claude API为例：1）缓存将Prompt中标记的部分（至少2048 tokens）存储5分钟。2）首次请求计费：缓存部分按标准价，后续请求缓存部分按标准价的10%计费（即90%折扣）。3）命中率取决于请求模式。例如，一个8000 token的Prompt，其中6000 token被缓存，首次花费：6000*Price + 2000*Price；后续花费：6000*Price*0.1 + 2000*Price，节省明显。

    实际案例: 某RAG应用部署中，通过Profiling发现Prefill阶段占端到端延迟的70%。原因是system prompt包含5K token的检索文档。通过启用Prefix Caching，Prefill只需处理200 token的用户查询，TTFT从800ms降至120ms，节省85%。同时使用FP8量化进一步提升Decode阶段速度。
  3. 核心原理: 对比分析需要从多个维度展开：性能（延迟、吞吐）、精度（量化损失）、成本（GPU小时）、复杂度（运维难度）、适用场景（prompt长度、并发模式）。不同技术方案各有优劣：如一体架构简单但GPU利用率低，分离架构高效但引入传输开销。选择依据是业务需求：如果SLA要求TTFT<200ms且QPS>100，应选择分离架构+Prefix Caching；如果QPS<10且prompt短，一体架构即可满足。
    关键要点: ①多维度对比评估框架；②场景匹配和方案选择；③实验验证和性能benchmark；④渐进式优化路径；⑤技术债和可维护性考量。
    实际案例: 某团队对比了三种方案：A)一体架构+FP16 B)一体架构+FP8+Prefix Caching C)分离架构+FP8+Prefix Caching。方案A月度成本$50K，TTFT=500ms；方案B成本$30K，TTFT=200ms；方案C成本$25K，TTFT=150ms。考虑到运维复杂度和团队能力，选择了方案B作为过渡方案，半年后迁移到方案C。
