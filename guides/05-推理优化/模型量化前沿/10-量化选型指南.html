面对众多量化方法，如何选择最适合的方案？本文提供一个系统化的决策框架，帮助你根据硬件条件、质量要求和使用场景做出最优选择。

核心原则：没有"最好"的量化方法，只有"最适合"的量化方法。选择取决于你的硬件、预算、质量容忍度和部署场景。

NVIDIA GPU 推理

GPU 推理是目前最主流的 LLM 部署方式：

  • H100/B200：首选 FP8（TensorRT-LLM），质量几乎无损，速度提升 1.5-2x
  • A100/A800：推荐 GPTQ 或 AWQ 4-bit，平衡质量和速度
  • RTX 4090/3090：推荐 GPTQ 4-bit，vLLM 或 TGI 部署
  • RTX 3060/4060（12GB）：GPTQ 4-bit 或 AWQ 4-bit，模型选择受限

CPU 推理

当 GPU 资源不足或需要在纯 CPU 环境运行时：

  • 首选 GGUF：llama.cpp 针对 CPU 深度优化，支持 AVX2/AVX-512
  • 推荐配置：Q4_K_M 精度与速度的平衡最好
  • 内存需求：Q4_K_M 约为模型参数量（B）× 0.6 GB

边缘设备与移动端

  • 手机（8GB RAM）：GGUF Q4_0 或 Q3_K，3B-7B 模型
  • 嵌入式设备：GGUF Q2_K，1B-3B 模型
  • Apple Silicon：GGUF 配合 Metal 加速，效果优秀

低延迟场景（实时对话、交互式应用）

  • 首选：AWQ 4-bit + vLLM/TensorRT-LLM
  • 替代：GPTQ 4-bit + vLLM
  • 原因：AWQ 的权重整理更适合 GPU 并行计算

高吞吐场景（批量处理、异步任务）

  • 首选：GPTQ 4-bit + vLLM（连续批处理）
  • 替代：AWQ 4-bit + vLLM
  • 关键：使用 vLLM 的 PagedAttention 和连续批处理

内存受限场景

  • 首选：GGUF Q4_K_M（CPU）或 GPTQ 3-bit（GPU）
  • 替代：AQLM/QuIP# 2-bit

常见误区：选型时不要先选量化方法再找框架，而应该反过来——先确定部署框架，再选该框架支持的量化方法。比如你选了GPTQ量化，但目标框架是llama.cpp（不支持GPTQ），就白费功夫了。框架兼容性是第一优先级。

常见场景推荐

场景 1：企业 API 服务（高并发、高质量）

推荐方案：AWQ 4-bit + vLLM
原因：AWQ 质量最优，vLLM 吞吐量最高。配合 A100/H100 集群可支撑大规模并发。

场景 2：内部工具（中等并发、可接受轻微退化）

推荐方案：GPTQ 4-bit + TGI 或 vLLM
原因：GPTQ 生态成熟，工具链完善，社区支持充分。

场景 3：本地开发和测试

推荐方案：GGUF Q4_K_M + Ollama 或 llama.cpp
原因：无需 GPU，安装简单，跨平台支持好。

场景 4：边缘设备部署

推荐方案：GGUF Q3_K_M + llama.cpp
原因：内存占用最低，CPU 优化最好，支持各种硬件。

场景 5：极致性能（有 H100）

推荐方案：FP8 + TensorRT-LLM
原因：FP8 硬件原生支持，速度最快，质量几乎无损。

量化前

  1. 明确需求：确定质量、速度、内存的优先级
  2. 评估硬件：确认 GPU 型号、显存大小、是否支持 FP8
  3. 选择基准模型：在 FP16 下测试模型在目标任务上的表现
  4. 准备评估数据：收集目标任务的测试集用于量化前后对比

量化时

  1. 校准数据：使用 128-512 条与目标场景匹配的数据
  2. group_size：128 是通用选择，质量要求高时用 64
  3. 对称 vs 非对称：非对称量化通常更好，但增加少量存储
  4. 测试验证：量化后立即在测试集上评估，不要跳过这步

部署后

  1. 监控质量：持续监控模型输出质量，设置告警
  2. A/B 测试：对比量化版本与原始版本的用户满意度
  3. 备份原始模型：保留 FP16 模型以便回滚
  4. 记录配置：记录量化参数、校准数据、评估结果

常见错误：

  • 跳过量化后评估，直接上线导致质量问题
  • 使用不匹配的校准数据（如用英文数据量化中文模型）
  • 忽视 group_size 参数的影响
  • 在不支持 FP8 的硬件上使用 FP8 量化
  • 过度追求极致压缩比而忽视质量退化

深入探讨：从系统设计的底层原理看，决策框架的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

量化选型的核心决策流程：

  1. 确定硬件平台（GPU 型号、是否有 CPU 推理需求）
  2. 确定质量要求（生产/内部/原型）
  3. 确定部署框架（vLLM/TGI/TensorRT-LLM/llama.cpp）
  4. 选择量化方法（AWQ/GPTQ/GGUF/FP8）
  5. 执行量化并评估
  6. 部署并监控
常见误区：选型时不要只看"精度指标"（如PPL），还要考虑"工程指标"：量化速度（GPTQ量化70B可能需要数小时）、框架兼容性（某些格式只在特定框架中可用）、社区活跃度（遇到问题能否快速解决）。一个PPL稍差但工具链完善的方法，往往比PPL最优但社区冷清的方法更实用。

决策框架的技术原理与数学推导： 从底层硬件角度分析，决策框架的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了决策框架的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。 
# 决策框架 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed
