核心原则：没有"最好"的量化方法，只有"最适合"的量化方法。选择取决于你的硬件、预算、质量容忍度和部署场景。

NVIDIA GPU 推理

GPU 推理是目前最主流的 LLM 部署方式：

  • H100/B200：首选 FP8（TensorRT-LLM），质量几乎无损，速度提升 1.5-2x
  • A100/A800：推荐 GPTQ 或 AWQ 4-bit，平衡质量和速度
  • RTX 4090/3090：推荐 GPTQ 4-bit，vLLM 或 TGI 部署
  • RTX 3060/4060（12GB）：GPTQ 4-bit 或 AWQ 4-bit，模型选择受限

CPU 推理

当 GPU 资源不足或需要在纯 CPU 环境运行时：

  • 首选 GGUF：llama.cpp 针对 CPU 深度优化，支持 AVX2/AVX-512
  • 推荐配置：Q4_K_M 精度与速度的平衡最好
  • 内存需求：Q4_K_M 约为模型参数量（B）× 0.6 GB

边缘设备与移动端

  • 手机（8GB RAM）：GGUF Q4_0 或 Q3_K，3B-7B 模型
  • 嵌入式设备：GGUF Q2_K，1B-3B 模型
  • Apple Silicon：GGUF 配合 Metal 加速，效果优秀

常见误区：选型时不要先选量化方法再找框架，而应该反过来——先确定部署框架，再选该框架支持的量化方法。比如你选了GPTQ量化，但目标框架是llama.cpp（不支持GPTQ），就白费功夫了。框架兼容性是第一优先级。

常见场景推荐

场景 1：企业 API 服务（高并发、高质量）

推荐方案：AWQ 4-bit + vLLM
原因：AWQ 质量最优，vLLM 吞吐量最高。配合 A100/H100 集群可支撑大规模并发。

场景 2：内部工具（中等并发、可接受轻微退化）

推荐方案：GPTQ 4-bit + TGI 或 vLLM
原因：GPTQ 生态成熟，工具链完善，社区支持充分。

场景 3：本地开发和测试

推荐方案：GGUF Q4_K_M + Ollama 或 llama.cpp
原因：无需 GPU，安装简单，跨平台支持好。

场景 4：边缘设备部署

推荐方案：GGUF Q3_K_M + llama.cpp
原因：内存占用最低，CPU 优化最好，支持各种硬件。

场景 5：极致性能（有 H100）

推荐方案：FP8 + TensorRT-LLM
原因：FP8 硬件原生支持，速度最快，质量几乎无损。

量化前

  1. 明确需求：确定质量、速度、内存的优先级
  2. 评估硬件：确认 GPU 型号、显存大小、是否支持 FP8
  3. 选择基准模型：在 FP16 下测试模型在目标任务上的表现
  4. 准备评估数据：收集目标任务的测试集用于量化前后对比

量化时

  1. 校准数据：使用 128-512 条与目标场景匹配的数据
  2. group_size：128 是通用选择，质量要求高时用 64
  3. 对称 vs 非对称：非对称量化通常更好，但增加少量存储
  4. 测试验证：量化后立即在测试集上评估，不要跳过这步

部署后

  1. 监控质量：持续监控模型输出质量，设置告警
  2. A/B 测试：对比量化版本与原始版本的用户满意度
  3. 备份原始模型：保留 FP16 模型以便回滚
  4. 记录配置：记录量化参数、校准数据、评估结果

常见错误：

  • 跳过量化后评估，直接上线导致质量问题
  • 使用不匹配的校准数据（如用英文数据量化中文模型）
  • 忽视 group_size 参数的影响
  • 在不支持 FP8 的硬件上使用 FP8 量化
  • 过度追求极致压缩比而忽视质量退化

深入探讨：从系统设计的底层原理看，决策框架的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

量化选型的核心决策流程：

  1. 确定硬件平台（GPU 型号、是否有 CPU 推理需求）
  2. 确定质量要求（生产/内部/原型）
  3. 确定部署框架（vLLM/TGI/TensorRT-LLM/llama.cpp）
  4. 选择量化方法（AWQ/GPTQ/GGUF/FP8）
  5. 执行量化并评估
  6. 部署并监控
常见误区：选型时不要只看"精度指标"（如PPL），还要考虑"工程指标"：量化速度（GPTQ量化70B可能需要数小时）、框架兼容性（某些格式只在特定框架中可用）、社区活跃度（遇到问题能否快速解决）。一个PPL稍差但工具链完善的方法，往往比PPL最优但社区冷清的方法更实用。

决策框架的技术原理与数学推导： 从底层硬件角度分析，决策框架的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了决策框架的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。 
# 决策框架 - 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    """测量推理延迟和吞吐"""
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    print(f"Throughput: {1/elapsed:.0f} req/s")
    return elapsed

  1. 核心原理: 量化选型的核心原则是"没有最好的量化方法，只有最适合的量化方法"。选择取决于四个维度的综合权衡：硬件条件（GPU型号、显存大小、是否支持FP8）、预算约束（量化时间成本）、质量容忍度（生产/内部/原型）和部署场景（API服务/本地工具/边缘设备）。文件提出了六步系统决策流程：确定硬件平台→确定质量要求→确定部署框架→选择量化方法→执行量化并评估→部署并监控。这个框架强调工程指标（量化速度、框架兼容性、社区活跃度）与精度指标（PPL）同等重要。特别指出"框架兼容性是第一优先级"——不要先选量化方法再找框架，而应该反过来先确定部署框架。
    关键要点: (1) 没有"最好"只有"最适合" (2) 四维度：硬件/预算/质量/场景 (3) 六步系统化决策流程 (4) 工程指标与精度指标并重 (5) 框架兼容性是第一优先级
    实际案例: 企业API服务选型：有A100集群→选AWQ 4-bit+vLLM，因为AWQ质量最优（⭐⭐⭐⭐⭐）且vLLM吞吐量最高，可支撑大规模并发。
    关联知识: 与第9章极限量化关联——2-bit方法虽PPL低但工具链不成熟，决策框架将其归类为"原型验证"场景而非生产环境。
  2. 核心原理: 六种主流量化方法在六个维度上各有优劣：(1)GPTQ(4/3/2-bit)：生态最成熟、易用性最高（⭐⭐⭐⭐）、硬件兼容性最广（⭐⭐⭐⭐⭐），适合大多数场景；(2)AWQ(4-bit)：质量最优（⭐⭐⭐⭐⭐）、速度较快（⭐⭐⭐⭐），适合低延迟场景；(3)GGUF(2-8-bit)：内存效率最高（⭐⭐⭐⭐⭐）、易用性最好（⭐⭐⭐⭐⭐）、跨平台兼容性最强（⭐⭐⭐⭐⭐），适合CPU推理和边缘设备；(4)FP8(8-bit)：质量几乎无损（⭐⭐⭐⭐⭐）、速度最快（⭐⭐⭐⭐⭐），但硬件兼容性差（⭐⭐），仅支持H100/B200；(5)QuIP#和AQLM(2-bit)：压缩比最高但速度较慢（⭐⭐）。选择时需在这六个维度中做权衡——没有方法在所有维度都最优。
    关键要点: (1) GPTQ生态最成熟、兼容性最广 (2) AWQ质量最优适合低延迟 (3) GGUF通用性最好支持CPU (4) FP8最快但需H100 (5) 2-bit方法压缩比高但速度慢
    实际案例: 本地开发测试选GGUF Q4_K_M+Ollama，因为易用性最高（⭐⭐⭐⭐⭐）且无需GPU，安装简单跨平台。
    关联知识: 与硬件考虑章节关联——不同硬件适合不同方法（H100→FP8，A100→GPTQ/AWQ，CPU→GGUF），方法总览为硬件匹配提供全景视图。
  3. 核心原理: 硬件平台决定了可选的量化方法范围，是选型的第一约束条件。NVIDIA GPU方面：H100/B200首选FP8（TensorRT-LLM），质量几乎无损且速度提升1.5-2x；A100/A800推荐GPTQ或AWQ 4-bit，平衡质量和速度；RTX 4090/3090推荐GPTQ 4-bit+vLLM/TGI部署；RTX 3060/4060（12GB）模型选择受限。CPU推理首选GGUF格式（llama.cpp），Q4_K_M精度与速度平衡最好，内存需求约为参数量×0.6GB，支持AVX2/AVX-512指令集加速。边缘设备方面：手机（8RAM）用GGUF Q4_0/Q3_K运行3B-7B模型；嵌入式设备用GGUF Q2_K运行1B-3B模型；Apple Silicon用GGUF配合Metal加速。
    关键要点: (1) H100首选FP8，A100推荐4-bit GPTQ/AWQ (2) CPU首选GGUF Q4_K_M，约参数量×0.6GB (3) 边缘设备用GGUF低比特（Q4_0/Q3_K/Q2_K） (4) Apple Silicon用GGUF+Metal (5) GPU型号决定模型规模和量化精度上限
    实际案例: 在骁龙8 Elite手机上部署Qwen2.5-7B，使用GGUF Q4_0约需4GB内存，推理速度可达15-20 tok/s，满足移动端对话需求。
    关联知识: 与质量要求决策树关联——硬件约束是决策树的第一层分支条件，不同硬件对应不同的量化推荐路径。
  4. 核心原理: 质量要求决策树将场景分为三个层级，每个层级对应不同的量化策略：(1)高要求（生产环境，对输出精度敏感）：有H100/Blackwell→FP8（硬件原生支持，质量几乎无损）；有A100/4090→AWQ 4-bit（质量最优⭐⭐⭐⭐⭐）；只有消费级GPU→GPTQ 4-bit g128（group_size=128平衡质量与速度）。(2)中等要求（内部工具，允许轻微退化）：GPU部署→GPTQ 4-bit g128或AWQ 4-bit；CPU部署→GGUF Q4_K_M（通用性最好）；混合部署→GGUF Q4_K_M。(3)低要求（原型验证，极低内存环境）：GPU→GPTQ 3-bit；CPU→GGUF Q3_K_M；极限压缩→QuIP# 2-bit或AQLM 2-bit。决策树的核心逻辑是：质量要求越高，选择越保守、越成熟的方法。
    关键要点: (1) 三级质量要求：高/中/低 (2) 高要求选FP8/AWQ/GPTQ 4-bit g128 (3) 中等要求选GPTQ/GGUF Q4_K_M (4) 低要求选3-bit/2-bit方法 (5) 质量越高选择越保守
    实际案例: 内部代码辅助工具（中等质量要求），使用GPTQ 4-bit g128+vLLM部署，可接受轻微质量退化换取更高吞吐和更低延迟。
    关联知识: 与延迟需求决策关联——质量要求和延迟需求是两个独立的决策维度，需要组合考虑（高质量+低延迟→AWQ+vLLM）。
  5. 核心原理: 延迟需求决策根据应用场景的时间特性分为三类：(1)低延迟场景（实时对话、交互式应用）：首选AWQ 4-bit+vLLM/TensorRT-LLM，因为AWQ的权重整理（weight shuffling）更适合GPU并行计算，可最小化首token时间（TTFT）；替代方案为GPTQ 4-bit+vLLM。(2)高吞吐场景（批量处理、异步任务）：首选GPTQ 4-bit+vLLM（连续批处理），利用PagedAttention和连续批处理最大化QPS；替代方案为AWQ 4-bit+vLLM。关键是用vLLM的PagedAttention减少内存碎片，用连续批处理提高GPU利用率。(3)内存受限场景：首选GGUF Q4_K_M（CPU）或GPTQ 3-bit（GPU），替代方案为AQLM/QuIP# 2-bit。核心权衡在于：低延迟优化TTFT（首token延迟），高吞吐优化QPS（整体吞吐量）。
    关键要点: (1) 低延迟选AWQ+vLLM/TensorRT-LLM (2) 高吞吐选GPTQ+vLLM+连续批处理 (3) 内存受限选GGUF Q4_K_M/3-bit (4) 低延迟优化TTFT，高吞吐优化QPS (5) PagedAttention和连续批处理是高吞吐关键
    实际案例: 实时对话应用（如AI客服）选AWQ 4-bit+vLLM，首token延迟可控制在100ms以内，用户体验流畅。
    关联知识: 与部署框架选择关联——vLLM适合高吞吐，TensorRT-LLM适合低延迟，llama.cpp适合内存受限的CPU/边缘场景。
向前串联： 决策框架的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
