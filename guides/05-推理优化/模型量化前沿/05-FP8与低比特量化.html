FP8 是 2023 年以来最受关注的量化格式，由 NVIDIA 在 Hopper 架构（H100）中首次引入硬件原生支持。FP8 提供了两种变体，分别针对不同的计算阶段：

两种 FP8 格式：E4M3 用于权重和激活值（前向传播），提供更高精度；E5M2 用于梯度（反向传播），提供更大数值范围。推理时通常只使用 E4M3。

值得注意的是，RTX 4090 虽然支持 FP8，但其 FP8 性能远低于 H100。在消费级硬件上，FP8 的优势主要体现在减少显存占用，而非推理速度。

常见误区：不要以为所有NVIDIA GPU都支持FP8。只有Hopper(H100)及以后的架构才原生支持FP8 Tensor Core。A100虽然强大，但完全没有FP8硬件支持，用软件模拟FP8反而比原生FP16更慢。购买硬件前务必确认FP8支持情况。

其他硬件平台

  • AMD MI300X：支持 FP8，但软件生态不如 NVIDIA 成熟
  • Intel Gaudi：FP8 支持有限，主要依赖 BF16
  • Google TPU：不支持 FP8，使用 BF16 和 INT8

vLLM 中的 FP8

# vLLM 支持 FP8 量化推理
from vllm import LLM, SamplingParams

# 方式1: 使用 FP8 权重（需要预量化模型）
llm = LLM(
    model="meta-llama/Llama-3-70B",
    quantization="fp8",          # 启用 FP8
    gpu_memory_utilization=0.9,
    max_model_len=4096
)

# 方式2: 使用 FP8 KV Cache（减少 KV 缓存内存）
llm = LLM(
    model="meta-llama/Llama-3-70B",
    kv_cache_dtype="fp8_e5m2",   # KV 缓存使用 FP8
    quantization=None            # 权重保持原始精度
)

TensorRT-LLM 中的 FP8

# TensorRT-LLM FP8 量化流程
import tensorrt_llm

# 1. 转换模型为 FP8
python convert_checkpoint.py \
    --model_dir ./llama3-70b \
    --output_dir ./llama3-70b-fp8 \
    --dtype float16 \
    --use_fp8

# 2. 构建引擎
trtllm-build \
    --checkpoint_dir ./llama3-70b-fp8 \
    --output_dir ./llama3-70b-fp8-engine \
    --gemm_plugin float16 \
    --max_batch_size 64 \
    --max_input_len 2048

推荐：对于 H100/Blackwell 用户，TensorRT-LLM 的 FP8 实现通常能获得最佳性能，推理速度比 FP16 提升约 1.5-2 倍。

3-bit 量化

3-bit 量化将每个权重映射到 8 个离散值（对称量化为 [-7, 7]）。目前主要支持 GPTQ 和 GGUF 格式：

  • GPTQ 3-bit：AutoGPTQ 支持，group_size=128 时质量可接受
  • GGUF Q3_K：llama.cpp 支持，混合使用 3-bit 和 4-bit
  • 质量损失：PPL 增加约 1-3 点，对简单任务影响不大

2-bit 量化

2-bit 量化将每个权重映射到仅 4 个离散值，属于极限压缩：

  • 传统方法：GPTQ 2-bit 质量损失严重，不推荐生产使用
  • 新研究：QuIP#、AQLM 等方法通过向量量化显著改善 2-bit 质量
  • 应用场景：边缘设备、极低内存环境、原型验证

实际建议：对于大多数生产场景，4-bit 仍然是最佳平衡点。除非有严格的内存限制，不建议使用 3-bit 或更低比特宽度。FP8 在支持的硬件上是 4-bit 的有力替代方案。

实践建议

深入探讨：从系统设计的底层原理看，FP8 格式概述的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

选择指南

  1. 有 H100/Blackwell + 注重质量：使用 FP8（TensorRT-LLM 或 vLLM）
  2. 有 H100 + 注重吞吐量：使用 GPTQ/AWQ 4-bit
  3. 只有消费级 GPU：使用 GPTQ/AWQ 4-bit
  4. 需要 CPU 推理：使用 GGUF Q4_K_M
  5. 极致压缩需求：考虑 QuIP# 2-bit 或 AQLM
常见误区：FP8不是FP16的简单截断。E4M3格式的数值范围仅±448，如果权重或激活值中存在超出此范围的outlier，直接截断会导致严重的精度崩溃。必须配合动态scaling或Transformer Engine的自动混合精度机制使用，切勿手动将FP16模型简单转换为FP8。

FP8 格式概述的技术原理与数学推导： 从底层硬件角度分析，FP8 格式概述的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了FP8 格式概述的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。
