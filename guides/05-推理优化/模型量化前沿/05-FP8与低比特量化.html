FP8 是 2023 年以来最受关注的量化格式，由 NVIDIA 在 Hopper 架构（H100）中首次引入硬件原生支持。FP8 提供了两种变体，分别针对不同的计算阶段：

两种 FP8 格式：E4M3 用于权重和激活值（前向传播），提供更高精度；E5M2 用于梯度（反向传播），提供更大数值范围。推理时通常只使用 E4M3。

值得注意的是，RTX 4090 虽然支持 FP8，但其 FP8 性能远低于 H100。在消费级硬件上，FP8 的优势主要体现在减少显存占用，而非推理速度。

常见误区：不要以为所有NVIDIA GPU都支持FP8。只有Hopper(H100)及以后的架构才原生支持FP8 Tensor Core。A100虽然强大，但完全没有FP8硬件支持，用软件模拟FP8反而比原生FP16更慢。购买硬件前务必确认FP8支持情况。

其他硬件平台

  • AMD MI300X：支持 FP8，但软件生态不如 NVIDIA 成熟
  • Intel Gaudi：FP8 支持有限，主要依赖 BF16
  • Google TPU：不支持 FP8，使用 BF16 和 INT8

vLLM 中的 FP8

# vLLM 支持 FP8 量化推理
from vllm import LLM, SamplingParams

# 方式1: 使用 FP8 权重（需要预量化模型）
llm = LLM(
    model="meta-llama/Llama-3-70B",
    quantization="fp8",          # 启用 FP8
    gpu_memory_utilization=0.9,
    max_model_len=4096
)

# 方式2: 使用 FP8 KV Cache（减少 KV 缓存内存）
llm = LLM(
    model="meta-llama/Llama-3-70B",
    kv_cache_dtype="fp8_e5m2",   # KV 缓存使用 FP8
    quantization=None            # 权重保持原始精度
)

TensorRT-LLM 中的 FP8

# TensorRT-LLM FP8 量化流程
import tensorrt_llm

# 1. 转换模型为 FP8
python convert_checkpoint.py \
    --model_dir ./llama3-70b \
    --output_dir ./llama3-70b-fp8 \
    --dtype float16 \
    --use_fp8

# 2. 构建引擎
trtllm-build \
    --checkpoint_dir ./llama3-70b-fp8 \
    --output_dir ./llama3-70b-fp8-engine \
    --gemm_plugin float16 \
    --max_batch_size 64 \
    --max_input_len 2048

推荐：对于 H100/Blackwell 用户，TensorRT-LLM 的 FP8 实现通常能获得最佳性能，推理速度比 FP16 提升约 1.5-2 倍。

3-bit 量化

3-bit 量化将每个权重映射到 8 个离散值（对称量化为 [-7, 7]）。目前主要支持 GPTQ 和 GGUF 格式：

  • GPTQ 3-bit：AutoGPTQ 支持，group_size=128 时质量可接受
  • GGUF Q3_K：llama.cpp 支持，混合使用 3-bit 和 4-bit
  • 质量损失：PPL 增加约 1-3 点，对简单任务影响不大

2-bit 量化

2-bit 量化将每个权重映射到仅 4 个离散值，属于极限压缩：

  • 传统方法：GPTQ 2-bit 质量损失严重，不推荐生产使用
  • 新研究：QuIP#、AQLM 等方法通过向量量化显著改善 2-bit 质量
  • 应用场景：边缘设备、极低内存环境、原型验证

实际建议：对于大多数生产场景，4-bit 仍然是最佳平衡点。除非有严格的内存限制，不建议使用 3-bit 或更低比特宽度。FP8 在支持的硬件上是 4-bit 的有力替代方案。

深入探讨：从系统设计的底层原理看，FP8 格式概述的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

选择指南

  1. 有 H100/Blackwell + 注重质量：使用 FP8（TensorRT-LLM 或 vLLM）
  2. 有 H100 + 注重吞吐量：使用 GPTQ/AWQ 4-bit
  3. 只有消费级 GPU：使用 GPTQ/AWQ 4-bit
  4. 需要 CPU 推理：使用 GGUF Q4_K_M
  5. 极致压缩需求：考虑 QuIP# 2-bit 或 AQLM
常见误区：FP8不是FP16的简单截断。E4M3格式的数值范围仅±448，如果权重或激活值中存在超出此范围的outlier，直接截断会导致严重的精度崩溃。必须配合动态scaling或Transformer Engine的自动混合精度机制使用，切勿手动将FP16模型简单转换为FP8。

FP8 格式概述的技术原理与数学推导： 从底层硬件角度分析，FP8 格式概述的性能瓶颈通常在于内存带宽而非计算能力。 推导：以A100为例，FP16峰值算力312 TFLOPS，HBM带宽2TB/s， 算术强度转折点 = 312T/2T = 156 FLOPs/Byte。为什么这很重要？ 因为当实际算术强度低于此值时，任务是内存受限的——增加更多计算单元不会提升性能。 这一数学本质决定了FP8 格式概述的所有优化策略都应围绕减少内存访问展开。 从工作机制看，算子融合（减少中间结果写回）和批处理（共享参数读取） 是提升算术强度的两大核心手段。

✅ 参考答案与详解
  1. 核心原理: FP8 是 NVIDIA 在 Hopper 架构中首次引入硬件原生支持的 8-bit 浮点量化格式，提供 E4M3（4位指数+3位尾数，范围±448，精度更高）和 E5M2（5位指数+2位尾数，范围±57344，范围更大）两种变体。E4M3 用于权重和激活值，E5M2 用于梯度。推理时通常只使用 E4M3，相比 FP16 内存减半，在 H100 上计算速度翻倍，且精度损失极小（PPL 增加 <0.1），是目前精度损失最小的量化方案之一。其数学本质是利用较少的尾数位表达浮点数值，通过指数位保持动态范围。
    关键要点: ①E4M3 精度高于 E5M2（尾数多1位）；②E5M2 范围远大于 E4M3（±57344 vs ±448）；③推理时通常只用 E4M3；④必须配合动态 scaling 使用，防止 outlier 导致精度崩溃；⑤FP8 不是 FP16 的简单截断。
    实际案例: 在 H100 上使用 vLLM 部署 Llama 3 70B，设置 quantization="fp8"，推理速度比 FP16 提升约1.5-2倍，显存从 140GB 降至约 70GB，而 MMLU 准确率几乎无变化。
    关联知识: FP8 与前章的 BF16、FP16 同属浮点格式家族；其硬件依赖性在第07章框架集成中进一步讨论。
  2. 核心原理: NVIDIA GPU 对 FP8 的硬件原生支持始于 Hopper 架构（H100），后续 Ada Lovelace（RTX 4090）和 Blackwell（B100/B200）也支持 FP8 Tensor Core。H100 的 FP8 吞吐量约 2000 TFLOPS，Blackwell 更是达到约 4500 TFLOPS。而 Ampere 架构（A100）完全不具备 FP8 硬件支持，软件模拟反而比原生 FP16 更慢。其他硬件平台中，AMD MI300X 支持 FP8 但软件生态不如 NVIDIA，Intel Gaudi 主要依赖 BF16，Google TPU 使用 BF16 和 INT8。
    关键要点: ①FP8 需要 Hopper 及以上架构才有原生 Tensor Core 支持；②A100 不支持 FP8；③RTX 4090 支持 FP8 但性能远低于 H100；④Blackwell B100/B200 的 FP8 吞吐量约 4500 TFLOPS；⑤AMD/Intel/TPU 的 FP8 支持各异。
    实际案例: 某团队计划部署 Llama 3 70B，原计划使用 8×A100（无 FP8 支持），FP16 推理吞吐为 2000 tok/s。改用 4×H100 后开启 FP8，吞吐提升至约 8000 tok/s，总成本降低约 40%。
    关联知识: 硬件支持差异是量化选型的核心约束，第07章的选型决策树中将"是否有 H100+"作为关键决策节点。
  3. 核心原理: FP8 推理实现需要在推理框架中完成模型转换和引擎构建两大步骤。vLLM 提供两种方式：一是直接使用 FP8 量化权重（quantization="fp8"），二是仅对 KV Cache 使用 FP8（kv_cache_dtype="fp8_e5m2"）而保持权重为原始精度。TensorRT-LLM 则需要先用 convert_checkpoint.py 将模型转换为 FP8 格式，再用 trtllm-build 构建推理引擎。核心优化在于 FP8 GEMM kernel 的实现——在 H100 的 FP8 Tensor Core 上，单卡算力可达约 2000 TFLOPS。
    关键要点: ①vLLM 支持 FP8 权重和 FP8 KV Cache 两种模式；②TensorRT-LLM 需要两步流程（转换+构建）；③FP8 KV Cache 可减少 KV 缓存内存占用约 50%；④FP8 推理在 H100 上比 FP16 快 1.5-2 倍；⑤TensorRT-LLM 性能通常优于 vLLM 的 FP8 实现。
    实际案例: 在 H100 上部署 Llama 3 70B，使用 TensorRT-LLM 的 FP8 引擎，最大 batch_size=64 时吞吐达到 4000 tok/s，而同等条件下 FP16 仅约 2200 tok/s。
    关联知识: FP8 推理实现是第07章"高性能 Kernel 深度解析"的基础。
  4. 核心原理: FP8 与 4-bit 整数量化代表了两种不同的量化哲学。FP8 是浮点格式，通过指数位保持动态范围，精度损失极小（PPL +0.1），压缩比为 2 倍（vs FP16），在 H100+ 上推理速度最快。而 GPTQ/AWQ 4-bit 是整数量化，压缩比更高（4 倍），精度损失略大（PPL +0.15~0.3），但硬件兼容性更广（所有 NVIDIA GPU）。FP8 的软件生态目前仍在发展中，而 4-bit 生态（GPTQ/AWQ/GGUF）已非常成熟。
    关键要点: ①FP8 压缩比 2x，4-bit 压缩比 4x；②FP8 精度损失远小于 4-bit；③FP8 推理速度最快（原生硬件）；④4-bit 硬件兼容性更广；⑤FP8 生态仍在发展中。
    实际案例: 某 API 服务商在 H100 集群上对比 FP8 和 AWQ-4bit 部署 Llama 3 8B：FP8 吞吐 3200 tok/s（PPL +0.02），AWQ-4bit 吞吐 3600 tok/s（PPL +0.10）。虽然 4-bit 更快，但 FP8 精度更优。
    关联知识: 此对比是第08章"混合精度量化策略"中 W4A8（权重 INT4 + 激活 INT8）的理论基础。
  5. 核心原理: 超低比特量化（3-bit 和 2-bit）是模型压缩的极限探索。3-bit 将每个权重映射到 8 个离散值（对称量化为 [-7,7]），GPTQ 和 GGUF（Q3_K）均支持，PPL 增加约 1-3 点。2-bit 将权重映射到仅 4 个离散值，传统 GPTQ 2-bit 质量损失严重（PPL +8.0），但 QuIP# 和 AQLM 等新方法通过向量量化显著改善了 2-bit 质量（QuIP# 在 70B 模型上 MMLU 约 75%，vs FP16 的 79%）。
    关键要点: ①3-bit 映射到 8 个离散值，PPL +1~3；②2-bit 映射到 4 个离散值；③QuIP# 通过随机正交旋转+E8 格量化改善 2-bit 质量；④AQLM 通过加性量化实现极低比特；⑤4-bit 仍是大多数生产场景的最佳平衡点。
    实际案例: 在边缘设备（如手机端）部署 Llama 3 8B，使用 GGUF Q3_K_M 格式将模型压缩至约 3.5GB，可在 4GB 内存的手机上运行简单对话任务。
    关联知识: 超低比特量化是第09章"2bit 与极限量化"的前导，QuIP# 和 AQLM 的详细原理将在该章深入讨论。
