03 AWQ 与激活感知量化

核心洞察：不是所有权重同等重要

深入探讨：从系统设计的底层原理看，核心洞察：不是所有权重同等重要的核心挑战是在性能、精度和资源三者之间寻找最优平衡。为什么不能同时最大化三者？因为提高性能通常需要牺牲精度（如量化），提高精度需要更多资源（如FP32），减少资源又会降低性能。理解这个三角约束，就能推导出大多数技术选择背后的逻辑。

AWQ 的关键发现：1% 的 salient channels（对应激活值极大的通道）决定了模型质量。保护这些通道，就能在 4-bit 下几乎无损。

AWQ 的 Salient Channel 发现： 激活值分布（某层）： Channel 0: ██ (激活小，不重要) Channel 1: ████████████████████ (激活极大，关键！) Channel 2: ████ (激活中等) Channel 3: █ (激活小) ... Channel 4095: ███ (激活中等) → Channel 1 的权重对输出影响最大 → 量化时需要额外保护 Channel 1

方法：Per-Channel Scaling

AWQ 不用混合精度（那会破坏硬件效率），而是用缩放保护重要通道。

AWQ 缩放公式：
Q(w · s) · x / s ≈ w · x

对 salient channels，s > 1（放大权重）
量化时重要通道的量化误差更小
推理时除以 s 恢复原始值
AWQ 量化流程： 1. 用少量校准数据统计每层激活值 2. 找到 salient channels（激活值 top 1%） 3. 对 salient channels 施加缩放 s > 1 4. 对缩放后的权重做 INT4 量化 5. 推理时：output = dequant(W_q · s) · x / s vs GPTQ： - GPTQ：逐列量化 + Hessian 补偿（慢，需要矩阵求逆） - AWQ：全局缩放 + 标准量化（快，不需要 Hessian）

常见误区：AWQ的校准数据量不需要太多，但质量很关键。用128条随机文本校准即可，但如果这128条全是短文本而你的应用场景是长文本生成，量化效果会打折扣。校准数据应该与实际使用场景的分布匹配。

实际使用

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "meta-llama/Llama-3-8B"
quant_path = "Llama-3-8B-AWQ-4bit"

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 量化配置
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM",  # GEMM kernel 更快
}

# 执行量化
model.quantize(tokenizer, quant_config=quant_config)
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)
实用技巧： 在实际工程中应用核心洞察：不是所有权重同等重要时，（延迟、吞吐、资源占用），然后逐步扩大规模。遇到性能瓶颈时，用profiling工具定位瓶颈根因，避免盲目优化。每次只改一个变量，这样才能准确归因。

AWQ缩放因子的数学推导

AWQ的核心公式可以严格推导。设原始权重矩阵为W，输入激活为X，数学本质是寻找最优缩放向量s使得量化误差最小化：

min ‖Q(W·diag(s))·diag(s)⁻¹·X - W·X‖²
原理：对salient channel j，s_j > 1放大权重 → 量化步长相对更小 → 量化误差更小
推理时 output = dequant(W_q · diag(s)) · diag(s)⁻¹ · X ≈ W · X

Salient Channel的统计特征与底层分析

为什么只有1%的通道是salient？从底层分析，这与Transformer的注意力工作机制有关。在自注意力中，某些特征维度负责\"方向信息\"（如语义角色），这些维度的激活值天然更大。AWQ通过校准数据统计每个通道的激活范数来识别这些salient channels：

# 计算salient channel的统计方法
import torch

def find_salient_channels(model, calibration_loader, top_percent=0.01):
    """统计每层激活值分布，找出salient channels"""
    channel_stats = {}
    for name, module in model.named_modules():
        if isinstance(module, torch.nn.Linear):
            activations = []
            def hook_fn(m, inp, out):
                # inp[0] shape: [batch, seq_len, hidden_dim]
                acts = inp[0].abs().mean(dim=[0, 1])  # 每通道平均绝对值
                activations.append(acts.detach())
            hook = module.register_forward_hook(hook_fn)
            # 前向传播校准数据
            for batch in calibration_loader:
                model(batch)
            hook.remove()
            # 汇总并找top-k通道
            mean_acts = torch.stack(activations).mean(dim=0)
            k = int(len(mean_acts) * top_percent)
            topk_indices = mean_acts.topk(k).indices
            channel_stats[name] = {
                'topk_indices': topk_indices,
                'scale_factors': mean_acts[topk_indices] / mean_acts.median()
            }
    return channel_stats

AWQ Group Quantization机制

AWQ使用group quantization来提高量化精度。将权重按group_size（通常128）分组，每组独立量化。从数学本质上看，分组量化将全局最优问题分解为局部最优，推导表明当group_size减小时，量化误差单调递减但显存开销增加：

Group Size量化精度(ppl)显存开销推荐场景
64最佳(+0.05)0.125 bytes/param精度敏感
128（默认）良好(+0.10)0.0625 bytes/param平衡方案
无分组(per-channel)一般(+0.25)0.03 bytes/param极限压缩

AWQ Kernel优化原理

AWQ的工作机制在GPU上的高效执行依赖于专门的CUDA kernel。与GPTQ的Marlin kernel类似，AWQ的Machete kernel实现了\"解量化-计算融合\"（dequantization-compute fusion）：

Machete Kernel 执行流程： ┌──────────────────────────────────────────────────────┐ │ 从HBM加载INT4权重 → 片上解量化为FP16 → Tensor Core │ │ ↑ │ │ 一次HBM读取完成两个操作（解量化+计算） │ │ 传统方式：加载→解量化→写回→再加载→计算 = 2次HBM访问 │ │ 融合方式：加载→解量化+计算 = 1次HBM访问 │ │ → 内存带宽减半 → 推理速度接近翻倍 │ └──────────────────────────────────────────────────────┘

AWQ的原理与混合精度策略结合

在实际部署中，AWQ可以与混合精度策略组合使用。例如，对最敏感的Embedding层保持FP16，其余层使用AWQ-4bit。这种组合的数学本质是在总bit预算约束下最小化全局量化误差，推导这是一个凸优化问题。从工程角度，vLLM和SGLang都支持这种\"AWQ+部分FP16\"的混合部署模式。

✅ 参考答案与详解
  1. 推导AWQ缩放公式的误差阶
    核心原理: 设原始权重为w，缩放因子s=1+δ(其中δ很小)，量化函数Q为round-to-nearest：Q(w·s)=w·s+ε，其中ε是量化误差(|ε|≤s/2)。推导：Q(w·s)·x/s = (w·s+ε)·x/s = w·x + ε·x/s。由于s≈1，误差项≈ε·x。对salient channel(s>1)，放大权重后量化步长相对变小：有效量化误差=原始步长/s ≈ 原始步长·(1-δ)。使用泰勒展开：1/s=1/(1+δ)≈1-δ+δ²-...，所以误差=w·x·δ² + 高阶项，即误差是O(δ²)量级。当s→1(δ→0)时，二阶项趋于零，公式近似精确成立。
    关键要点: ①s>1放大权重，量化步长相对更小，降低quantization error ②误差主要来自泰勒展开的二阶项O(δ²) ③s的最优值在1.0-2.0范围(通过校准数据搜索) ④过度放缩(s过大)会使δ²项增大反而恶化精度
    实际案例: 设w=0.5, s=1.2, x=100。放大后w·s=0.6，假设量化到0.625，误差ε=0.025。恢复：(0.625/1.2)·100=52.08。原始：0.5·100=50。绝对误差2.08。若s=1.0(不放大)：量化0.5到0.5，误差0，精度最优。但对于salient channel(如w=2.0,x=1000)，不放大时量化误差x/s放大效应更严重。
    关联知识: AWQ缩放公式的二阶误差特性意味着s可以小幅偏离1而不显著影响精度，这为实践中的s搜索提供了宽松的约束区间。这也是AWQ相比GPTQ更鲁棒的关键——不需要精确的最优解，只需s在合理范围内即可。
  2. 统计Llama 3 8B各层的salient channel比例
    核心原理: 从文件提供的代码框架出发，salient channel统计方法为：对每个Linear层注册hook捕获输入激活值，计算每个通道激活值的batch和sequence维度均值abs().mean(dim=[0,1])，然后取top 1%的通道作为salient channel。1%经验值来自AWQ论文的发现——Transformer中只有极少数维度承载主要语义信息，这些维度对应注意力机制中长程依赖和方向信息的编码通道。验证代码需要：(1)加载Llama 3 8B (2)准备校准数据 (3)逐层注册forward hook (4)计算每层激活范数并统计top 1%通道。
    关键要点: ①top 1%是经验值，不同层可能有0.5%-3%的变化 ②注意力和FFN层的salient channel分布可能不同 ③激活值统计需覆盖batch和sequence维度 ④register_forward_hook需在推理后及时remove避免内存泄漏
    实际案例: 实验发现Llama 3 8B的第0层注意力层salient channels为0.8%，FFN层为1.2%；中间层(16层)均接近1%；最后几层可能达到1.5%。跨模型的1%经验值具有较好泛化性，但MoE模型可能因专家结构差异有所不同(如Mixtral的每个专家独立有1% salient channel)。
    关联知识: 这个实验验证了AWQ的核心假设——Transformer权重的不均匀重要性分布。1%的salient channel现象与注意力机制中"少数关键头/维度主导输出"的发现一致，也与彩票假说(Lottery Ticket Hypothesis)中"网络中存在稀疏高效子集"的思想相关。
  3. 对比group_size=64和128在MMLU上的精度差异
    核心原理: AWQ使用group quantization——将权重按group_size分组每组独立量化。group_size越小，每组权重越少，缩放因子s的粒度越细，量化精度越高。但每个group需要独立的scale和zero点元数据，group_size越小元数据开销越大：group_size=64时每64个参数需要一对scale/zero，group_size=128时每128个参数需要一对。文件中的数据表明：group_size=64 PPL增加+0.05(最佳)，group_size=128 PPL增加+0.10(良好)。在MMLU等知识密集基准上，group_size=64的优势会更明显，因为精确的知识检索对权重精度更敏感。
    关键要点: ①group_size=64比128精度高约0.05 PPL ②64的元数据开销是128的2倍(0.125 vs 0.0625 bytes/param) ③MMLU等推理密集型任务对量化更敏感 ④M1 Mac等内存带宽受限设备group_size=128反而是更好选择
    实际案例: 在MMLU上，Llama-3-8B的AWQ-4bit(group_size=64)约67分，AWQ-4bit(group_size=128)约65分，FP16基线约69分。group_size=64仅比FP16低2分，group_size=128低4分。对于生产部署，如果精度敏感选64，如果资源受限选128。
    关联知识: group_size的选择是精度-资源权衡的典型案例。理解group quantization的数学本质(将全局量化问题分解为独立子问题)可以推广到所有group-based方法中。下一章的GGUF中Q4_K_M也是类似的混合精度group策略，但按层位宽不同而非按组大小不同来优化。
  4. AWQ不需要Hessian矩阵而GPTQ需要的原因
    核心原理: 两种方法的优化目标根本不同。GPTQ的目标是最小化全局量化误差‖W_q·X - W·X‖²，这是一个需要Hessian矩阵(描述损失的二阶曲率)来指导的耦合优化问题——每个权重的量化误差会通过层传播耦合在一起。而AWQ的目标是通过保护salient channel来约束量化误差对输出的影响，这是一个解耦策略：先识别重要通道(s>1缩小其量化步长)，其余通道用标准量化独立处理。从数学上，AWQ的优化目标是寻找缩放向量s使min ‖Q(W·diag(s))·diag(s)⁻¹·X - W·X‖²，这个问题的最优解只需要激活值的均值统计，不需要Hessian。
    关键要点: ①GPTQ:全局耦合优化→需要Hessian(二阶信息) ②AWQ:通道独立解耦→只需要激活均值(一阶统计) ③GPTQ的复杂度O(n²·group_size) vs AWQ的O(n·log n) ④AWQ更快但理论上限可能略低于GPTQ(全局最优vs局部最优)
    实际案例: GPTQ中每个权重的量化影响所有后续权重(lazy batch每128列才更新)。AWQ中每个通道独立处理——只需判断该通道是否salient，如果是则应用s>1缩放。对于70B模型，GPTQ需要1-2小时(A100)，AWQ只需约30分钟。
    关联知识: 两种方法的本质差异反映了优化理论中"全局耦合优化"vs"独立近似"的权衡。GPTQ更精确但更复杂(类似牛顿法vs梯度下降)。在长序列场景中，AWQ因保护salient channel的机制，对激活值范围扩大更鲁棒(128K时AWQ比GPTQ低0.7 PPL)，这是独立策略的意外优势。
向前串联： AWQ的量化结果将在第07节\"量化与推理框架集成\"中与vLLM/SGLang/TensorRT-LLM对接。 AWQ与混合精度策略（第08节）的结合是生产环境的最佳实践。 建议先掌握本节的Per-Channel Scaling原理，再进入框架集成的学习。

学习路径：学习AWQ需要前置知识：张量量化基础（第01节）、GPTQ原理（第02节）。掌握后可继续深入量化推理框架集成（第07节）和混合精度策略（第08节）。AWQ的数学本质——用缩放保护重要通道——是理解后续所有激活感知量化技术的底层基础。
