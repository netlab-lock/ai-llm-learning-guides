1. 为什么需要驱逐

为什么需要驱逐

通俗类比：仓库空间有限，不能无限囤货。就像超市货架需要定期清理滞销品给新品腾位置，KV Cache驱逐就是在有限显存中淘汰不重要的历史信息，保留关键内容。
常见误区：误以为驱逐等于粗暴丢弃信息会严重降质。实际上优秀驱逐策略是智能筛选，只淘汰对当前生成几乎无影响的token，保留关键上下文信息。
深入探讨：以Llama-70B为例，单token的KV Cache约占2.5MB（128层×8头×128维×2×2字节）。生成4K token需10GB，生成128K token需320GB，远超单卡80GB容量，驱逐是必选项而非可选项。

驱逐后：[K1][K50][K200]...[K999]  显存30%

Transformer推理中，KV Cache的内存占用与序列长度呈严格的线性关系。当处理长上下文（如128K、1M tokens）时，KV Cache会迅速耗尽GPU显存。然而，并非所有token的KV对对生成质量同等重要——这为"智能驱逐"提供了理论基础。

图1: KV Cache内存增长与驱逐效果——长序列下内存必然溢出，驱逐是唯一出路

📐 线性增长困境

对于 LLaMA-70B (80层 × 64头 × d=128, FP16)，每个token的KV Cache占用 2 × 80 × 64 × 128 × 2B = 2.5MB。128K序列需要 320GB，远超单卡容量。

💰 内存预算约束

实际部署中，GPU显存还需容纳模型权重、激活值、系统开销。分配给KV Cache的预算通常只有总显存的 30-50%，必须精打细算。

⚡ 非均匀重要性

实验表明，删除大部分token的KV对后，模型输出质量下降很小。少量"重要"token承载了绝大部分注意力信息。

🔍 稀疏性观察

注意力矩阵天然稀疏——典型情况下，80%以上的注意力权重集中在不到20%的token上，这为激进压缩提供了可能。

直觉层

想象你在阅读一本1000页的书并做笔记。你的笔记本（显存）只够记200页的要点。你需要一个策略来决定：哪些页的笔记保留，哪些可以丢弃。关键洞察是：不是每页都同等重要——目录页（初始token）、核心论点页、和最近读的页（最近token）往往最重要。

机制层

KV Cache驱逐的核心公式：budget = N_max × 2 × L × H × d × dtype_bytes。当序列长度 n 超过预算允许的最大 N_max 时，必须丢弃 n - N_max 个token的KV对。驱逐策略决定了丢弃哪些，其目标是 最小化注意力分布的KL散度变化。

公式层

KV Cache内存: M = 2 × n_layers × n_heads × d_head × seq_len × bytes_per_element
驱逐目标: min KL(Attn(Q, K_full, V_full) || Attn(Q, K_kept, V_kept))
等价于: min Σ |α_i - α'_i| 其中 α 是原始注意力权重分布

2. 注意力稀疏性观察<助于在实际工程中做出更合理的技术选择和优化决策。

注意力稀疏性观察

通俗类比：读一本500页的书，你不会每页都仔细看——目录、摘要、结论最重要，中间细节可能跳过。注意力稀疏性就是模型也只关注少数关键token，大部分注意力得分接近零。
常见误区：不要误以为所有token同等重要。实验表明80%以上的注意力集中在不到20%的token上，忽略这个规律就无法理解为何驱逐策略可行且有效。

驱逐策略的理论基础在于Transformer注意力的天然稀疏性。通过仔细观察真实模型的注意力分布，研究者发现了多个关键规律。

图2: 注意力权重热力图——初始token（注意力汇聚）和局部窗口占据绝大部分注意力

关键发现

🪣 Attention Sink 现象

StreamingLLM发现：无论输入内容如何，序列最前面的1-4个token始终承受不成比例的高注意力权重。即使这些token在语义上不重要（如BOS token），模型也会将"无用"注意力倾倒到它们身上。

📊 注意力集中度

在典型层中，超过80%的注意力权重集中在不到5%的token上。这意味着理论上可以丢弃95%的KV Cache而只损失少量信息。

📍 位置依赖模式

靠近序列末尾的token自然获得更多关注（局部性）。不同层呈现不同模式：浅层更局部化，深层更全局化。

🎭 头间差异显著

同一层的不同注意力头呈现截然不同的稀疏模式。有的头专注局部上下文，有的头追踪远距离依赖——需要差异化驱逐策略。

直觉层

注意力矩阵就像一张"重要性地图"。大部分区域是"荒漠"（低注意力），但有两块"绿洲"：左侧边缘（初始token/注意力汇聚点）和右下角对角线附近（最近的token）。驱逐策略就是学会只保留绿洲，丢弃荒漠。

3. 基于注意力分数的驱逐 (H2O)

通俗类比：H2O就像图书馆的借阅排行榜——长期无人借阅的书移到仓库，热门书和新书留在前台。根据历史注意力分数淘汰冷门token，保留热门token和最近token。
常见误区：关键陷阱：只看单次注意力分数会误判重要性。需要累积历史注意力分数（如指数衰减求和），才能准确评估token的真实重要性，避免昙花一现的误导。
深入探讨：H2O保留策略：budget = recent_tokens + heavy_hitters。设预算B，保留最近r个token + 累积注意力分数最高的(B-r)个token。论文推荐r=512，B=2048时Llama-2-7B的perplexity仅增加1.2%。

H2O (Heavy-Hitter Oracle) 是最经典的KV Cache驱逐算法之一。核心思想简洁而优雅：保留历史上获得累计注意力分数最高的token，驱逐分数最低的token。

H2O 核心算法

图3: H2O驱逐过程——维护一个top-k集合，新token加入时淘汰累计注意力分数最低的token

三层次理解

直觉层

H2O就像一个"热门排行榜"。每个token都在竞争注意力——被关注越多，排名越高，越不容易被淘汰。当排行榜满了（达到内存预算），新来的token必须把排名最低的挤出去。

机制层

累积分数更新：每生成一个新token，用其attention score更新所有保留token的累积分数。
Top-K维护：使用最小堆维护当前保留的token集合，堆顶即为待驱逐的候选。
驱逐时机：当新token的KV加入后总长度超过budget时，移除堆顶元素。
多头独立：每个注意力头维护独立的驱逐决策，因为不同头关注的token集合差异很大。

公式层

score_i(t) = Σ_{s=t_prev}^{t} α_s,i   (token i 在时刻 t 的累积注意力分数)
α_s,i = softmax(Q_s · K_i^T / √d)   (标准注意力权重)
evict = argmin_{i ∈ kept} score_i(t)   (选择分数最低的驱逐)

复杂度：每个step的驱逐操作 O(log k)（堆操作），总计 O(n log k) 

# H2O 算法伪代码
class H2OCache:
    def __init__(self, budget):
        self.budget = budget
        self.kept_indices = []      # 当前保留的token索引
        self.cum_scores = {}        # {idx: cumulative_attention_score}
        self.min_heap = []          # 最小堆：(score, idx)

    def update(self, new_attn_scores):
        # new_attn_scores: [n_kept] — 新token对所有保留token的注意力
        for i, idx in enumerate(self.kept_indices):
            self.cum_scores[idx] += new_attn_scores[i]
            heap_update(self.min_heap, self.cum_scores[idx], idx)

        # 加入新token自身
        new_idx = len(all_tokens)
        self.kept_indices.append(new_idx)
        self.cum_scores[new_idx] = 0.0

        # 超出预算则驱逐
        while len(self.kept_indices) > self.budget:
            score, evict_idx = heappop(self.min_heap)
            self.kept_indices.remove(evict_idx)
            del self.cum_scores[evict_idx]
            free_kv(evict_idx)  # 释放GPU内存
💡 质量-内存权衡：实验表明，在LLaMA上使用仅 20% 的KV Cache预算，H2O在多数benchmark上仅损失 1-3% 的精度。在预算为50%时几乎无损。

4. 基于重要性评估的驱逐

通俗类比：就像高考录取不仅看单科成绩，还要综合评估各科加权总分。重要性驱逐通过多维度指标（注意力、位置、语义）给每个token打综合分，淘汰低分token。
常见误区：过度依赖单一指标（如只看注意力分数）会遗漏重要信息。位置信息、语义角色（主语/宾语）、特殊token（[SEP]）都需纳入评估，否则连贯性严重下降。
深入探讨：ScissorHands方法：importance = attention_score × position_decay × semantic_weight。位置衰减用1/√(distance)，语义权重通过小模型预计算。实测LLaMA-7B上保留25%token即可维持97%的原始精度。

除了简单的累积注意力分数，更先进的方法通过profiling、观察、分层策略来评估token重要性，实现更精细的驱逐决策。

4.1 FastGen: 基于Profiling的驱逐策略

FastGen的核心洞察：不同的注意力头扮演不同的角色（如局部头、全局头、稀疏头），需要不同的KV Cache管理策略。

  1. 离线Profiling阶段：用少量代表性输入分析每个头的注意力模式，归类为：Special Token头、Local头、Sliding Window头、Pivot Token头等
  2. 策略分配：Special Token头 → 保留特殊token(BOS/SEP)；Local头 → 只保留最近的w个token；Pivot Token头 → 保留注意力权重最大的k个token
  3. 在线执行：各头按分配的策略独立管理KV Cache，组合成复合缓存结构

4.2 SnapKV: 基于观察的压缩

SnapKV的发现：在Prefill阶段完成后，通过观察注意力权重模式，可以一次性选择最重要的token子集，无需逐步驱逐。

4.3 PyramidInfer: 金字塔KV Cache

PyramidInfer提出按层差异化分配KV Cache预算：浅层（信息冗余度高）分配更少的KV存储，深层（信息更关键）分配更多。

5. StreamingLLM与注意力汇聚

通俗类比：注意力汇聚就像音乐会的开场和返场——开场定基调，返场最重要，中间部分可以精简。模型的前几个token（如[BOS]）承载了异常高的注意力，必须保留。
常见误区：StreamingLLM最大误区：以为只保留最近token就够了。实验表明丢弃初始sink token会导致perplexity暴增10倍以上，模型输出完全崩溃，前4个token必须始终保留。

StreamingLLM揭示了一个深刻的现象：LLM推理中存在"注意力汇聚"(Attention Sink)机制，并据此提出了一种极简但有效的KV Cache管理策略。

注意力汇聚的核心发现

图5: StreamingLLM策略——保留少量Sink tokens + 滑动窗口，实现无限长度流式推理

三层次理解

直觉层

想象一条传送带上的物品（tokens）。你有两个关注点：①传送带最开头放的"指示牌"（sink tokens），你时不时需要看它来保持方向感；②传送带末端刚到的新物品（recent tokens），你正在处理它们。中间那些已经走远的旧物品，你几乎不需要再回头看。

机制层

KV Cache结构：[Sink KVs (固定4个)] + [Recent Window KVs (滑动窗口大小w)]
窗口滑动：当新token进入，最老的recent token被淘汰，但sink tokens永远不淘汰
总预算：sink_size + window_size，通常 4 + (4092) = 4096 tokens
位置编码：使用相对位置编码(RoPE)时，需保持位置索引连续（不因驱逐而跳变）

公式层

KV_kept = [KV_0, KV_1, ..., KV_s | KV_{n-w}, ..., KV_{n-1}, KV_n]
其中 s = sink数量(通常4)，w = 窗口大小

注意力汇聚原因：softmax要求 Σα_i = 1。当模型"不想关注"任何token时，它会将多余注意力dump到固定的sink token上，避免干扰。
α_sink = 1 - Σ_{i∈relevant} α_i   (吸收"无用"注意力)

⚠️ 关键限制：StreamingLLM的滑动窗口策略无法外推到训练时未见过的序列长度。它只能让推理"持续运行"而不崩溃，但无法真正利用超出窗口的长距离上下文信息。这意味着它不适合需要"回忆"很久以前内容的任务（如长文档问答）。

6. 稀疏注意力与KV Cache

通俗类比：稀疏注意力就像用聚光灯而非泛光灯照亮舞台——只聚焦关键区域，其余保持昏暗。不是每个token都要和所有历史token交互，只计算相关的子集即可大幅节省计算。
常见误区：稀疏模式选错会导致严重信息丢失。全局attention捕获长程依赖，局部窗口捕获上下文，两者缺一不可。纯局部窗口会让模型丧失跨段落理解能力。
深入探讨：典型稀疏模式：Longformer采用local+global，窗口大小w=512，每256token设一个global attention。BigBird用random+window+global组合。实测稀疏度75%（只计算25%attention）时性能损失<1%。

稀疏注意力从另一个角度解决KV Cache问题：不是先存满再驱逐，而是一开始就只计算和存储必要的注意力连接。这将KV Cache优化从"事后清理"转向"事前规划"。

图6: 四种稀疏注意力模式——从固定模式到查询驱动的动态稀疏

稀疏注意力对KV Cache的影响

📐 Longformer: 局部+全局

每个token只关注窗口内的w个邻居 + 少量全局token。KV Cache仅需存储活跃的局部窗口和全局token，复杂度从O(n²)降至O(n×w)。

🎲 BigBird: 三重稀疏

组合局部窗口 + 全局token + 随机连接。随机连接保证了理论上的图连通性，局部性保证了效率，全局token提供了信息桥梁。

🔍 Quest: 查询感知

不预设固定的稀疏模式，而是根据当前query动态决定哪些key block值得计算。通过轻量级的block选择器，预估每个KV block的重要性，只计算top-k个block。

⚡ 动态稀疏的意义

固定模式假设所有输入有相同的注意力结构——这显然不对。动态稀疏让模型根据输入内容自适应调整，在质量和效率间找到更好的平衡。

# Quest: 查询感知的KV Block选择
def quest_attention(Q, K_blocks, V_blocks, top_k):
    """
    Q: [n_heads, 1, d]  当前query
    K_blocks: [n_blocks, block_size, d]  分块的Key
    V_blocks: [n_blocks, block_size, d]  分块的Value
    """
    # Step 1: 轻量级预估——用每个block的极值key近似
    K_max = K_blocks.max(dim=1)  # [n_blocks, d]
    K_min = K_blocks.min(dim=1)  # [n_blocks, d]

    # Step 2: 估算每个block的最大可能注意力分数
    scores_upper = (Q @ K_max.T + Q @ K_min.T) / 2
    scores_range = abs(Q @ (K_max - K_min).T) / 2
    block_scores = scores_upper + scores_range  # 上界估计

    # Step 3: 只计算top-k blocks的精确注意力
    selected = topk(block_scores, k=top_k)
    K_selected = K_blocks[selected]  # 只取选中的blocks
    V_selected = V_blocks[selected]

    # Step 4: 标准注意力计算（只在选中的blocks上）
    attn = softmax(Q @ K_selected.T / sqrt(d))
    return attn @ V_selected

7. KV Cache合并技术

通俗类比：KV Cache合并就像打包行李箱——把多件衣服卷成紧实的一卷，占用空间更小但功能不变。合并技术将多个KV向量压缩为更少的表示，减少显存同时保留关键信息。
常见误区：合并过度会损失细节信息。压缩比过高（如4:1以上）会导致生成质量明显下降。建议从2:1开始测试，逐步提升压缩比，找到精度与显存的最佳平衡点。
深入探讨：CLA(Cache-Linking Attention)将相邻层KV Cache合并，如第l层和第l+1层共享同一份KV。实测Llama-2-7B合并相邻2层后显存减少50%，perplexity增加仅0.8%。D2O方法通过PCA降维进一步压缩。

          ↓ 合并共享 ↓
Merged:    [K1'][K2'][K3'][K4']  4份共用
显存: 8份 → 4份 = 节省50%

合并(Merging)是驱逐的替代方案：与其直接丢弃不重要的KV对，不如将相似的KV对合并为一个"代表"。这能在压缩内存的同时保留更多信息。

7.1 CaM (Cache Merging)

图7: 驱逐直接丢弃信息，合并将相似token的信息融合后保留

合并方法分类

🎯 K-Means 合并

将KV向量聚类，每个簇用中心点代表。合并后的KV = 簇中心的加权平均。适合token高度相似的场景（如重复模式）。

📏 相似度合并

计算KV对之间的余弦相似度，合并最相似的对。需要O(n²)预计算但可以增量更新。保留了更多语义信息。

⚖️ 加权平均合并

KV_merged = Σ w_i · KV_i，权重w由注意力分数决定。高注意力token在合并后的"代表"中占更大比重。

🏗️ Token Merging (ToMe)

源自视觉领域的Token Merging方法。将token序列分为两半，计算跨半相似度，用贪心匹配合并最相似的token对。

# KV Cache K-Means 合并伪代码
def merge_kv_cache_kmeans(K, V, target_size):
    """
    K: [seq_len, d]  当前Key缓存
    V: [seq_len, d]  当前Value缓存
    target_size: 目标压缩大小
    """
    # 对K向量做K-Means聚类
    kmeans = KMeans(n_clusters=target_size)
    labels = kmeans.fit_predict(K)  # [seq_len] → 簇标签

    # 每个簇内加权平均
    K_merged = []
    V_merged = []
    for c in range(target_size):
        mask = (labels == c)
        # 可选: 按注意力权重加权
        weights = attn_weights[mask] / attn_weights[mask].sum()
        K_merged.append((K[mask] * weights[:, None]).sum(dim=0))
        V_merged.append((V[mask] * weights[:, None]).sum(dim=0))

    return stack(K_merged), stack(V_merged)  # [target_size, d]
直觉层

驱逐像是"删除旧照片"来腾空间，合并像是"把多张相似照片拼成一张全景图"。全景图不如每张单独照片精确，但保留了更多整体信息。特别当多张照片拍的是同一场景时，合并几乎不损失信息。

场景选型推荐

💬 流式对话机器人

推荐：StreamingLLM + H2O
理由：对话天然需要流式处理，StreamingLLM提供零开销基础保障，H2O在窗口内精细筛选重要token。

📄 长文档问答

推荐：SnapKV + PyramidInfer
理由：长文档有明确的Prefill阶段，SnapKV可以一次性压缩。PyramidInfer的分层策略适配文档中不同层级的信息。

🔧 多任务服务

推荐：FastGen
理由：不同任务的注意力模式差异大，FastGen的头级profiling和差异化策略最适合异构工作负载。

📐 信息密集型任务

推荐：CaM (合并)
理由：如摘要、翻译等需要保留全局信息的任务，合并比驱逐更能保留细节。适合对质量要求极高、可接受一定延迟的场景。

组合策略

🚀 最佳实践：分层组合

在实际生产中，最优方案通常是多种策略的组合：

  • Prefill阶段：SnapKV一次性压缩prompt的KV Cache
  • Decode阶段：StreamingLLM的sink + window作为基线保障
  • 窗口内优化：H2O在窗口内做精细化选择，替换简单的FIFO
  • 层间差异化：PyramidInfer为不同层分配不同预算
  • 周期性合并：每隔N步做一次K-Means合并，进一步压缩

    关键要点: ①KV Cache内存公式M=2×n_layers×n_heads×d_head×seq_len×bytes ②128K序列需320GB，远超单卡容量 ③GPU显存预算仅30-50%可分配给KV Cache ④非均匀重要性观察是驱逐的理论基础 ⑤驱逐目标是最小化KL散度变化
    实际案例: 在部署Llama-2-70B处理长文档摘要任务时，当输入超过16K tokens（KV Cache约40GB）就开始逼近显存上限，必须启用驱逐策略。使用H2O算法保留20%的KV Cache预算，在多数benchmark上仅损失1-3%的精度。
    关联知识: 驱逐机制与注意力稀疏性观察（第2节）直接相关——正是因为注意力矩阵天然稀疏（80%以上权重集中在不到20%的token上），驱逐才可行。同时与StreamingLLM（第5节）的滑动窗口策略形成互补方案。
  3. 核心原理: 注意力稀疏性观察是驱逐策略的实证基础。通过分析真实Transformer模型的注意力权重热力图，研究者发现三个关键规律：①Attention Sink现象——序列最前面的1-4个token（如BOS token）始终承受不成比例的高注意力权重，即使这些token在语义上不重要，模型也会将"无用"注意力倾倒到它们身上；②注意力集中度极高——典型层中超过80%的注意力权重集中在不到5%的token上，这意味着理论上可以丢弃95%的KV Cache而只损失少量信息；③位置依赖模式——靠近序列末尾的token自然获得更多关注（局部性），且不同层呈现不同模式：浅层更局部化，深层更全局化。此外，同一层的不同注意力头呈现截然不同的稀疏模式，有的头专注局部上下文，有的头追踪远距离依赖。
    关键要点: ①Attention Sink：前1-4个token承载异常高注意力 ②top-20%的token累计贡献超过95%注意力分数 ③注意力矩阵零值比例达60%-80% ④浅层局部化、深层全局化的层间差异 ⑤不同头的稀疏模式差异显著，需差异化策略
    实际案例: H2O论文实测数据：在LLaMA模型上，保留最近token+高注意力token的组合，perplexity仅上升0.5%-2%，而显存节省70%以上。注意力热力图显示，初始token列和最近token对角线区域是最亮的"绿洲"，中间大量区域是可安全驱逐的"荒漠"。
    关联知识: 注意力稀疏性直接支撑了StreamingLLM（第5节）的sink token机制——正是因为初始token天然获得高注意力，保留它们才能维持模型稳定性。同时也为H2O（第3节）的累积注意力分数排序提供了合理性依据。
  4. 核心原理: H2O（Heavy-Hitter Oracle）是最经典的KV Cache驱逐算法，核心思想是保留历史上获得累计注意力分数最高的token（Heavy Hitters），驱逐分数最低的token。其工作流程为：维护一个大小为budget的top-k集合，每生成一个新token时，用其attention score更新所有保留token的累积分数（score_i(t) = Σ α_s,i），然后将新token加入集合。当总长度超过budget时，使用最小堆找到累积分数最低的token并驱逐。H2O的budget由两部分组成：最近r个token（保持局部连贯性，通常占25%）+ 累积注意力分数最高的(B-r)个Heavy Hitters（保留全局关键信息，通常占75%）。每个注意力头独立维护驱逐决策，因为不同头关注的token集合差异很大。算法复杂度为O(n log k)，其中n为序列长度，k为budget大小。
    关键要点: ①累积分数而非单次分数评估重要性 ②最小堆实现O(log k)的驱逐操作 ③budget=recent(r)+heavy_hitters(B-r)双组件结构 ④多头独立驱逐决策 ⑤在LLaMA上20%预算仅损失1-3%精度
    实际案例: 论文推荐配置：r=512（最近token数），B=2048（总预算），在Llama-2-7B上perplexity仅增加1.2%。实现上使用最小堆维护token的累积注意力分数，堆顶即为待驱逐候选。当新token加入导致超出预算时，弹出堆顶元素并释放其对应的GPU显存中的K/V张量。
    关联知识: H2O与FastGen（第4节）都是基于注意力分数的驱逐，但FastGen通过离线profiling将注意力头分类后采用差异化策略，而H2O是在线动态决策。H2O的累积分数思想也启发了后续SnapKV的Prefill后一次性观察压缩方法。
  5. 核心原理: 基于重要性评估的驱逐超越了单纯的注意力分数，采用多维度指标综合评估每个token的重要性。核心方法包括：①FastGen——通过离线Profiling分析每个注意力头的模式（Special Token头、Local头、Sliding Window头、Pivot Token头），为不同类型的头分配不同的KV Cache管理策略；②SnapKV——在Prefill阶段完成后，通过观察注意力权重模式一次性选择最重要的token子集（窗口池化选择Top-K），无需逐步驱逐，实现零额外延迟开销；③PyramidInfer——按层差异化分配KV Cache预算，浅层信息冗余度高分配更少（20%），深层信息更关键分配更多（100%）。多维度评估公式：importance = attention_score × position_decay × semantic_weight，其中位置衰减用1/√(distance)，语义权重通过小模型预计算。
    关键要点: ①FastGen：逐头profiling差异化策略，压缩率50-70% ②SnapKV：Prefill后一次性压缩，零额外延迟，压缩率60-80% ③PyramidInfer：金字塔式层级预算分配，压缩率50-75% ④多维度评估：注意力×位置衰减×语义权重 ⑤ScissorHands方法：保留25%token即可维持97%精度
    实际案例: PyramidInfer的实际部署：对于32层模型，Layer 0-6分配20%预算，Layer 7-13分配35%，Layer 14-27分配60%，Layer 28-32分配100%。这种差异化分配基于实验观察——浅层注意力更均匀（信息冗余），深层注意力更集中（信息关键），因此深层需要保留更多KV对。
    关联知识: 重要性评估方法与H2O（第3节）形成递进关系——H2O用单一累积注意力分数，重要性评估加入位置和语义维度。PyramidInfer的层级差异化思想与稀疏注意力（第6节）中不同层的局部/全局模式差异相呼应。SnapKV的一次性压缩思路与KV Cache合并技术（第7节）有相似的"批处理"思想。
