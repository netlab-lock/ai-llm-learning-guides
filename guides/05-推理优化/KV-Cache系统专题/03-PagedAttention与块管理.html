Module 03: PagedAttention与块管理

从操作系统虚拟内存思想到LLM推理KV Cache的革命性分页管理

1. 从操作系统虚拟内存到PagedAttention

通俗类比：操作系统管理内存就像图书馆管理员管理书架——虚拟内存让每本书（数据）都有一个逻辑位置，实际可能存放在不同书架（物理内存）。PagedAttention借鉴了这个思想。
常见误区：不要误以为PagedAttention只是简单分块，它需要处理块的动态分配、回收和跨请求共享，否则会导致内存碎片化问题。
深入探讨：操作系统虚拟内存将进程地址空间划分为固定大小的页（通常4KB），通过页表映射到物理内存帧。PagedAttention将KV-cache划分为固定大小的块（如16个token），每块独立管理，物理位置灵活。

1.1 OS Paging概念类比

🎯 直觉层

想象你是一个图书馆管理员。书架空间有限，每位读者要借阅的书数量不同。如果你给每位读者预留整个书架——很快就会没空间。聪明的做法是：把书架分成固定大小的格子，读者按需申请格子，归还后格子可以给别人用。这就是操作系统的分页（Paging）思想，也是PagedAttention的核心。

⚙️ 技术层

操作系统虚拟内存的分页机制：

  • 页（Page）：固定大小的虚拟地址空间块（通常4KB）
  • 页框（Frame）：物理内存中的固定大小块
  • 页表（Page Table）：虚拟页号 → 物理帧号的映射
  • 进程看到连续的虚拟地址空间，实际物理内存可以不连续
  • 页表由MMU（内存管理单元）在每次内存访问时查询
🔬 专家层

PagedAttention (Kwon et al., 2023) 将OS虚拟内存思想引入GPU上的KV Cache管理。关键洞察：KV Cache的生命周期与OS进程相似——创建时增长、运行时读写、完成时释放——因此可以用相同机制管理。但GPU没有MMU硬件支持，需要在CUDA kernel中软件实现页表查找。

1.2 为什么连续分配在KV Cache中失败

核心问题：每个序列的长度未知——生成前不知道最终产生多少token。连续分配必须按最大长度预留空间，导致巨大浪费。

❌ 连续分配

序列A (实际512 tokens, 预留2048):  [████████░░░░░░░░░░░░░░░░]
序列B (实际128 tokens, 预留2048):  [███░░░░░░░░░░░░░░░░░░░░░]
序列C (实际1024 tokens, 预留2048): [█████████████░░░░░░░░░░░]

利用率: (512+128+1024)/(2048*3) = 27%
内存浪费: 73%

✅ 分页分配

物理块池: [A0][A1][A2][B0][C0][C1][C2][A3][空][空][空][空]

序列A: 5 blocks = 512 tokens
序列B: 1 block = 128 tokens
序列C: 4 blocks = 1024 tokens

利用率: 1664/(1664+128*3) ≈ 81%  (仅最后块有内部碎片)
内存浪费: ~19%

1.3 PagedAttention论文的关键洞察

vLLM (Kwon et al., SOSP 2023) 的核心贡献：

  1. 发现现有LLM推理系统KV Cache内存浪费率高达60-80%
  2. 提出将KV Cache分成固定大小的块（block），通过块表映射实现非连续存储
  3. 块可以按需分配和释放，几乎消除外部碎片
  4. 引入Copy-on-Write机制支持beam search和parallel sampling
  5. 系统吞吐量提升2-4倍

2. PagedAttention核心机制

通俗类比：酒店管理客房——前台（块管理器）记录每个客人（序列）住在哪间房（KV块），退房（序列完成）后房间可以立刻分配给新客人，避免空置浪费。
常见误区：实现时务必处理好序列长度增长时的块分配时机：预分配太少会导致频繁系统调用，预分配太多会浪费内存。
深入探讨：核心机制：将每个序列的KV-cache存储在固定大小的块中，每个块包含一定数量token的K和V向量。注意力计算时，通过块表（Block Table）查找逻辑块到物理块的映射，支持非连续内存分配。

2.1 Block: 固定大小的KV Cache块

🎯 直觉层

一个Block就像一个"集装箱"——里面可以装固定数量token的KV向量。不管你是1个token还是16个token，都用同样大小的箱子。最后一个箱子可能没装满，但箱子本身大小不变。

⚙️ 技术层

Block定义：

  • 一个block存储 block_size 个连续token的K和V向量
  • 每个block的shape: [2, num_heads, block_size, head_dim]（K和V合并存储）
  • 典型block_size = 16（vLLM默认）
  • 物理上是一段连续GPU显存
# Block的数据结构
class KVCacheBlock:
    def __init__(self, block_size, num_heads, head_dim, dtype):
        # shape: [2, num_heads, block_size, head_dim]
        # 2 = key + value
        self.data = torch.zeros(
            (2, num_heads, block_size, head_dim),
            dtype=dtype, device="cuda"
        )
        self.ref_count = 0    # 引用计数
        self.block_size = block_size
        self.num_filled = 0   # 已填充的token数

    def is_full(self):
        return self.num_filled == self.block_size

    def append(self, key, value, slot):
        # slot: block内的位置 (0 ~ block_size-1)
        self.data[0, :, slot, :] = key   # K
        self.data[1, :, slot, :] = value  # V
        self.num_filled += 1

2.2 Block Table: 逻辑块到物理块的映射

⚙️ 技术层

每个序列维护一个Block Table（块表），记录该序列的逻辑块编号到物理块地址的映射。推理时，Attention kernel通过查表获取非连续存储的KV数据。

2.3 物理块的分配与释放

# 简化的Block Manager实现
class BlockManager:
    def __init__(self, num_blocks, block_size):
        self.block_size = block_size
        # 空闲块列表: 初始所有物理块都空闲
        self.free_blocks = list(range(num_blocks))
        # 每个序列的block table: seq_id → [phys_block_ids]
        self.block_tables = {}
        # 引用计数: 每个物理块被多少序列引用
        self.ref_counts = [0] * num_blocks

    def allocate(self, seq_id):
        """为序列分配一个新物理块"""
        if not self.free_blocks:
            raise OutOfMemoryError("No free blocks!")
        phys_block = self.free_blocks.pop(0)
        self.ref_counts[phys_block] = 1
        if seq_id not in self.block_tables:
            self.block_tables[seq_id] = []
        self.block_tables[seq_id].append(phys_block)
        return phys_block

    def free(self, seq_id):
        """释放序列的所有物理块"""
        for phys_block in self.block_tables[seq_id]:
            self.ref_counts[phys_block] -= 1
            if self.ref_counts[phys_block] == 0:
                self.free_blocks.append(phys_block)
        del self.block_tables[seq_id]

    def can_allocate(self, seq_id):
        """检查是否能为序列分配新块"""
        return len(self.free_blocks) > 0

3. 块管理数据结构

通俗类比：快递分拣中心使用分区货架管理包裹——每个货架有编号（物理块地址），分拣员通过运单号（序列ID）查找对应货架位置，类似块管理器使用哈希表管理块。
常见误区：为每个序列维护独立的块表会带来额外内存开销，对于超长序列需要优化查找效率，避免块管理成为性能瓶颈。
深入探讨：关键数据结构：块管理器包含一个空闲块列表（链表或位图）、一个已分配块字典（哈希表映射物理块地址到块状态）。每个序列有自己的块表，存储逻辑块索引到物理块地址的映射。

3.1 空闲块池（Free Block Pool）

⚙️ 技术层

空闲块池跟踪所有未被使用的物理块。实现方式包括：

  • 栈（Stack）：最简单，O(1)分配/释放，但块地址可能不连续
  • 队列（Queue）：FIFO顺序，可能提高局部性
  • Bitmap：适合需要找到连续空闲块的场景
  • vLLM使用Python collections.deque实现
# vLLM风格的空闲块管理
from collections import deque

class FreeBlockPool:
    def __init__(self, num_gpu_blocks, num_cpu_blocks=0):
        # GPU上的空闲块
        self.gpu_free = deque(range(num_gpu_blocks))
        # CPU (用于swap/offload) 的空闲块
        self.cpu_free = deque(range(num_cpu_blocks))

    def acquire_gpu_block(self):
        if not self.gpu_free:
            raise GPUMemoryError("GPU blocks exhausted")
        return self.gpu_free.popleft()

    def release_gpu_block(self, block_id):
        self.gpu_free.append(block_id)

    @property
    def num_free_gpu(self):
        return len(self.gpu_free)

    def can_swap_in(self, num_blocks):
        return len(self.gpu_free) >= num_blocks

3.2 每序列Block Table

⚙️ 技术层

每个活跃序列有一个Block Table，记录其逻辑块到物理块的映射。在GPU上，block table被传输给CUDA kernel以进行地址查找。

# Block Table 数据结构
class SequenceBlockTable:
    def __init__(self, max_blocks):
        # 逻辑到物理的映射
        self.logical_to_physical = {}  # {logical_idx: physical_block_id}
        self.max_blocks = max_blocks

    def append_block(self, physical_block_id):
        logical_idx = len(self.logical_to_physical)
        self.logical_to_physical[logical_idx] = physical_block_id

    def get_physical_blocks(self):
        # 返回有序物理块列表, 传给CUDA kernel
        return [self.logical_to_physical[i]
                for i in range(len(self.logical_to_physical))]

    def to_tensor(self):
        # 转为GPU tensor供kernel使用
        blocks = self.get_physical_blocks()
        return torch.tensor(blocks, dtype=torch.int32, device="cuda")

3.3 引用计数（Reference Counting）

当多个序列共享同一个物理块时（如beam search），需要引用计数来决定何时真正释放物理块。只有引用计数降为0时，块才归还到空闲池。

# 引用计数机制示例
class RefCountedBlockManager:
    def __init__(self, num_blocks):
        self.free_pool = deque(range(num_blocks))
        self.ref_counts = [0] * num_blocks
        self.block_tables = {}  # seq_id → BlockTable

    def allocate_block(self, seq_id):
        """分配新块, 引用计数=1"""
        block = self.free_pool.popleft()
        self.ref_counts[block] = 1
        self.block_tables[seq_id].append_block(block)
        return block

    def share_block(self, block, new_seq_id):
        """共享块给另一个序列(用于fork)"""
        self.ref_counts[block] += 1
        self.block_tables[new_seq_id].append_block(block)

    def free_block(self, block):
        """减少引用计数, 为0则释放"""
        self.ref_counts[block] -= 1
        if self.ref_counts[block] == 0:
            self.free_pool.append(block)
        elif self.ref_counts[block] < 0:
            raise RuntimeError("Negative ref count!")

4.1 CUDA Kernel如何访问非连续块

⚙️ 技术层

核心挑战：KV Cache物理上不连续，但Attention计算需要访问所有历史token的K和V。CUDA kernel通过以下方式解决：

  1. 将block table传入kernel（作为int数组）
  2. 对于每个query token，计算需要访问哪些逻辑块
  3. 通过block table查表得到物理块地址
  4. 从物理块加载K/V数据进行attention计算
// PagedAttention CUDA Kernel 核心逻辑 (简化)
__global__ void paged_attention_kernel(
    float* __restrict__ output,        // [num_seqs, num_heads, head_dim]
    const float* __restrict__ q,        // query
    const float* __restrict__ k_cache,  // 所有物理块的K
    const float* __restrict__ v_cache,  // 所有物理块的V
    const int* __restrict__ block_table, // [num_seqs, max_blocks]
    const int* __restrict__ seq_lens,    // 每个序列的长度
    int block_size,
    int num_heads,
    int head_dim
) {
    int seq_idx = blockIdx.x;
    int head_idx = blockIdx.y;
    int seq_len = seq_lens[seq_idx];

    // 该序列的block table起始位置
    const int* seq_block_table = block_table + seq_idx * max_blocks;

    // 遍历序列中所有token的K/V
    for (int token_idx = threadIdx.x; token_idx < seq_len;
         token_idx += blockDim.x) {

        // 1. 计算逻辑块号和块内偏移
        int logical_block = token_idx / block_size;
        int block_offset  = token_idx % block_size;

        // 2. 通过block table查找物理块地址 ★关键★
        int physical_block = seq_block_table[logical_block];

        // 3. 计算物理地址
        int k_addr = ((physical_block * num_heads + head_idx)
                     * block_size + block_offset) * head_dim;
        int v_addr = k_addr + total_blocks * num_heads
                     * block_size * head_dim;  // V在K之后

        // 4. 加载K, V并计算attention score
        float score = dot(q_vec, &k_cache[k_addr]);
        // ... softmax, accumulate output ...
    }
}

4.2 性能开销分析

间接寻址开销：与连续KV Cache相比，PagedAttention每次访问需要额外的查表操作（1次int load + 计算物理地址）。但实测显示，对于典型LLM工作负载，这个开销仅为1-3%，因为：

  • block table通常很小（几百个int），可以完全缓存在L1/共享内存中
  • K/V数据访问的延迟远大于查表延迟
  • block_size足够大时，一次加载的KV数据可以复用多次

4.3 Warp-level Block Table缓存

🔬 专家层

优化策略：将block table加载到共享内存或寄存器中，避免全局内存访问。由于同一warp内的线程访问相同的逻辑块，block table命中率极高。

// Warp-level block table caching优化
__shared__ int shared_block_table[MAX_BLOCKS];

// 只由warp的第一个线程加载block table到共享内存
if (threadIdx.x == 0) {
    for (int i = 0; i < num_blocks_this_seq; i++) {
        shared_block_table[i] = seq_block_table[i];
    }
}
__syncthreads();

// 后续所有查表操作都在共享内存中完成 (极快)
int physical_block = shared_block_table[logical_block];

5. 内存效率分析

通俗类比：拼图游戏——传统方法为每个玩家（序列）预留整个拼图底板（连续内存），PagedAttention像共享拼图盒，玩家按需拿取拼图块，盒子空间利用率更高。
常见误区：不能只关注总内存节省，还要考虑内存访问模式：PagedAttention虽然省内存，但可能导致更多的内存访问分散，需要权衡。
深入探讨：内存效率公式：传统方法浪费率 = (预留空间 - 实际使用) / 预留空间，通常50-70%。PagedAttention浪费率 ≈ 块内未填充token比例，对于块大小B，平均浪费(B-1)/2B。典型值：B=16时浪费约3%，B=64时浪费<1%。

理论模型

传统连续分配的浪费：

  • 外部碎片：不同长度序列之间无法利用的空隙
  • 预留浪费：按最大长度预留，实际序列通常短得多
  • 设平均序列长度为 L，最大长度为 M，则浪费率 ≈ 1 - L/M

  • 仅最后一个block可能未填满
  • 平均浪费 = block_size/2 tokens per sequence
  • 浪费率 ≈ block_size / (2 × avg_seq_len)
  • 对于 block_size=16, avg_seq_len=512，浪费率仅 1.6%

6.1 自动前缀检测

🎯 直觉层

很多请求共享相同的开头——比如系统提示（system prompt）。如果能把这些共享部分的KV Cache缓存下来，新请求就不用重复计算了。Prefix Caching就是利用PagedAttention的块级共享来实现这一点。

⚙️ 技术层

Prefix Caching的工作流程：

  1. 将每个序列的KV Cache分成固定大小的block
  2. 对每个block计算哈希值（基于block内容）
  3. 新请求到来时，逐block计算哈希，查找是否有匹配的缓存block
  4. 匹配到的block直接复用，未匹配的从匹配点开始计算
# Prefix Caching 哈希匹配逻辑
import hashlib

class PrefixCache:
    def __init__(self):
        # hash → (physical_block_id, ref_count)
        self.hash_to_block = {}

    def compute_block_hash(self, token_ids, block_idx, block_size):
        # 包含block位置信息, 防止不同位置相同内容误匹配
        start = block_idx * block_size
        end = min(start + block_size, len(token_ids))
        block_tokens = tuple(token_ids[start:end])
        # 加入前缀hash以保证前缀一致性
        prefix_hash = self._compute_prefix_hash(token_ids, start)
        return hash((prefix_hash, block_tokens))

    def lookup_and_share(self, token_ids, block_idx, block_size):
        """查找缓存的block, 找到则增加引用计数"""
        h = self.compute_block_hash(token_ids, block_idx, block_size)
        if h in self.hash_to_block:
            phys_block, _ = self.hash_to_block[h]
            self.hash_to_block[h] = (phys_block, _ + 1)
            return phys_block  # 命中!
        return None  # 未命中, 需要计算

    def _compute_prefix_hash(self, token_ids, up_to):
        # 递归hash: H(block_i) = hash(H(block_{i-1}), tokens_i)
        if up_to == 0:
            return hash(())
        prev = self._compute_prefix_hash(token_ids, up_to - block_size)
        return hash((prev, tuple(token_ids[max(0,up_to-block_size):up_to])))

7.1 Chunked Prefill with Paged KV

⚙️ 技术层

长prompt的prefill阶段会占用大量显存和计算时间，导致decode请求被阻塞（延迟spike）。Chunked prefill将长prompt分成多个chunk，与decode请求交错执行：

  • 将prompt分成固定大小的chunk（如256 tokens/chunk）
  • 每个chunk的KV Cache写入分页块
  • 在chunk之间插入decode步骤
  • 减少TTFT (Time To First Token) 的方差

7.2 PagedAttention v2优化

PagedAttention v2 (FlashAttention风格优化)：

  • Flash-Decoding：将KV序列长度维度并行化，减少warp间的同步
  • Block粒度融合：将多个逻辑块的加载合并，减少kernel launch开销
  • 流水线化：block table预取与KV加载重叠执行
  • 混合精度：block table用int16代替int32，减少传输开销

8. Copy-on-Write与并行采样

通俗类比：复印店的服务——客户请求多份相同文件（同一提示的多个采样）时，店员（COW机制）先给出原稿引用，实际需要修改时才复印，节省纸张（内存）和时间。
常见误区：实现COW时必须正确处理引用计数和写时复制的线程安全性，否则会导致内存泄漏或数据损坏，特别是在多并发环境下。
深入探讨：并行采样实现：为同一提示生成多个回答时，共享前缀部分的KV块（COW），只在每个采样分支生成新token时分配新块。引用计数：每个物理块维护引用数，最后一个引用释放时回收块。典型内存节省：N个采样节省约(N-1)*前缀大小的内存。

8.1 Beam Search的KV Cache共享

🎯 直觉层

Beam search维护多个候选序列（beam），它们通常共享大部分前缀。没有CoW时，每个beam都要复制完整的KV Cache。有了CoW，共享部分只需一份物理数据，只有分支点之后的新token才需要新block。

⚙️ 技术层

Beam Search + CoW的典型场景：beam_width=4，生成100个token。

  • 无CoW: 4个beam × 100 tokens = 400 tokens的KV Cache
  • 有CoW: 假设前50个token共享，后50个token不同
  • 有CoW: 50 (共享) + 4×50 (独立) = 250 tokens → 节省37.5%
  • beam越宽、共享前缀越长，节省越多

# Parallel Sampling: 从一个prompt生成N个不同回答
def parallel_sample(prompt, N=8, max_tokens=256):
