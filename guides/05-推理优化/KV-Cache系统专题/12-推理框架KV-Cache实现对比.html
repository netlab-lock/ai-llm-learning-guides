vLLM — PagedAttention 先驱

通俗类比：PagedAttention就是给KV-Cache设计的“虚拟内存系统”。它把KV-Cache切成固定大小的“页”，按需分配物理块，就像操作系统管理内存，彻底解决了内存碎片和浪费问题。
常见误区：使用vLLM时，`block_size`参数（页大小）不是越大越好。太大可能导致小请求浪费内部碎片，太小则增加页表管理开销。需要根据模型和场景调优。
深入探讨：传统KV-Cache为每个序列预分配`seq_len`的连续内存。PagedAttention将其拆分为物理块（如16个token一块），通过页表映射。序列生成时按需申请新块，生成后立即释放。一个物理块可被多个序列共享（如系统提示），显存利用率从平均~50%提升至接近100%。

🧠 三层理解

第一层 · 核心概念

vLLM 使用 PagedAttention 技术，将 KV Cache 划分为固定大小的物理块（Block），通过块表（Block Table）实现逻辑到物理的映射。类似操作系统的虚拟内存分页机制，消除了内存碎片，使 KV Cache 内存利用率接近 100%。

第二层 · 工作原理

当新请求到达时，BlockSpaceManager 从预分配的 GPU 内存池中分配空闲块。每个请求维护一个 Block Table，记录逻辑块号→物理块号的映射。Attention 计算时，通过块表间接寻址获取 KV Cache。当内存不足时，调度器（Scheduler）可以选择 Swap（换出到 CPU）或 Recompute（重计算）来腾出空间。

第三层 · 深入实现

PagedAttention kernel 在 CUDA 层面实现：Q 向量与所有相关 KV Block 计算注意力分数，使用 max_len 和 block_size 参数控制块边界。vLLM 的 Block 大小默认 16 tokens。Prefix Caching 通过哈希前缀 token 实现 Block 共享，Copy-on-Write 机制支持 beam search 等共享场景。

SGLang — RadixAttention 创新

通俗类比：RadixAttention就像餐厅的“共享菜谱”缓存。如果多个客人点了有相同前菜（前缀）的套餐，厨房只需要准备一次前菜，后续订单直接复用，大大加速了上菜速度。
常见误区：RadixAttention的优势在长前缀共享时才明显。如果你的请求都是独立的短对话，其性能可能与优化后的vLLM相当。避免为所有场景盲目使用。
深入探讨：SGLang构建一个内存中的Radix树来索引KV-Cache。树的键是token序列的前缀。当新请求到来时，沿树查找最长公共前缀(LCP)并复用其KV-Cache，只为差异部分（LCP之后的token）计算新的KV-Cache。这在批量处理具有相同系统提示或少量few-shot示例的请求时，速度提升可达3-10倍。
Radix树 (前缀树) 示例:
         [root]
        /     \ 
     "你好"   "请问"
      |          |
   [KV-Cache]  [KV-Cache]
     / \          | \ 
  "世界" "中国"  "今天" "明天"
    |     |       |     |
  [KVC] [KVC]   [KVC] [KVC]

新请求"你好中国" -> 直接复用"你好"的KVC，只计算"中国"部分。

UC Berkeley · 基于 Radix Tree 的 KV Cache 管理 · 结构化生成优化

🧠 三层理解

第一层 · 核心概念

SGLang 的 RadixAttention 使用 基数树（Radix Tree）管理所有请求的 KV Cache 前缀。树的每个节点对应一组 token 的 KV Cache，天然支持前缀共享。多轮对话、few-shot 等场景下前缀命中率极高。

第二层 · 工作原理

当新请求到达时，SGLang 在 Radix Tree 中匹配最长公共前缀，命中部分直接复用 KV Cache，只需计算新增 token。树的 eviction 策略基于 LRU。对于结构化输出（JSON/正则），压缩有限状态机（Compressed FSM）将多个状态合并，减少冗余计算。

第三层 · 深入实现

Radix Tree 的每个节点存储：(token_ids, kv_cache_ref, ref_count, last_access_time)。树的分支点是 token 序列的分歧处。Eviction 时选择 ref_count=0 且 LRU 的叶子节点。DP 场景下，KV Transfer 通过 NCCL 将 KV Cache 从一个 worker 传输到另一个，实现跨 DP 实例的前缀共享。

TensorRT-LLM — NVIDIA 原生优化

通俗类比：TensorRT-LLM就像米其林大厨使用的“定制化厨具套装”。它与NVIDIA的“厨房”（GPU硬件、CUDA、cuDNN）深度集成，每一个操作都针对硬件特性手工打磨，以追求极致性能。
常见误区：TensorRT-LLM的灵活性较低。模型需要先“编译”成TensorRT引擎，这个过程耗时较长，且对模型结构有修改限制。不适合频繁更换模型或快速实验的场景。
深入探讨：其优化栈包括：1. **量化**：FP8/INT4等低精度计算，配合NVIDIA GPU的Tensor Core。2. **FlashAttention/FlashInfer**：优化注意力计算内存访问。3. **算子融合**：将多个小算子合并为一个大kernel，减少GPU kernel launch开销和显存读写。4. **多GPU通信**：深度优化NVLINK/NVSwitch下的张量并行(TP)和流水线并行(PP)。

🧠 三层理解

第一层 · 核心概念

TensorRT-LLM 使用 PagedKVCache（类似 PagedAttention）管理 KV Cache，结合 TensorRT 的图优化和 CUDA kernel 调优，在 NVIDIA GPU 上实现极致推理性能。支持 FP8 KV Cache 和 Inflight Batching（动态批处理）。

第二层 · 工作原理

PagedKVCache 将 KV Cache 分为固定大小的 token blocks，通过 KVCacheManager 统一管理分配和回收。Inflight Batching 在每个 decode step 都重新调度 batch，将新请求的 prefill 和已有请求的 decode 混合执行。KV Cache Reuse 允许跨请求共享相同前缀的 KV Cache。

第三层 · 深入实现

TensorRT-LLM 的 attention kernel 使用 fused multi-head attention，直接在 kernel 内处理 paged KV 访问。FP8 KV Cache 使用 E4M3 格式，在 H100 的 FP8 Tensor Core 上实现近乎无损的精度。Multi-GPU 通过 NCCL AllReduce 实现 TP，PP 通过 pipeline schedule 减少 bubble。

LMDeploy — 连续批处理实践

通俗类比：连续批处理就像餐厅的“流水线”：A桌在等主菜时，B桌可以开始点单和上冷盘。GPU不用干等一个请求的全部token生成完，可以同时处理多个请求的不同阶段。
常见误区：连续批处理对请求的完成时间有要求。一个“慢”请求（输出很长）可能阻塞后续“快”请求的资源，需要配合请求优先级或抢占机制使用，否则会损害尾部延迟。
深入探讨：传统静态批处理中，一个批次内所有请求必须等最慢的结束才能开始下一批。连续批处理允许在任一请求生成结束时，立即从队列取出新请求插入当前批次，填满GPU计算单元。LMDeploy实现了高效的迭代级调度器，并配合vLLM的PagedAttention管理KV-Cache，使得GPU利用率常保持在90%以上。
静态批处理: [Req1, Req2, Req3] -> 计算... 等Req3结束 -> [Req4, Req5, Req6]
连续批处理: [Req1, Req2, Req3] -> 计算... Req1结束，立即插入Req4 -> [Req2, Req3, Req4]
处理模式GPU利用率平均延迟实现复杂度
静态批处理低，尤其是长短请求混合时高（受最慢请求影响）低
连续批处理高，接近满载较低，请求交错执行高，需精密调度

🧠 三层理解

第一层 · 核心概念

LMDeploy 的 TurboMind 引擎使用连续批处理（Continuous Batching）配合 KV Cache 管理，实现高效的 LLM 推理。支持 Persistent KV Cache，可在多轮对话中保持 KV Cache 不释放。

第二层 · 工作原理

TurboMind 将 KV Cache 分配为固定大小的 slot，每个请求占用一个或多个 slot。Persistent KV Cache 模式下，完成的请求不立即释放 slot，而是标记为可复用，后续相同前缀的请求可直接复用。W4A16 量化主要压缩模型权重，间接减少每层 KV 计算开销。

第三层 · 深入实现

TurboMind 使用 paged KV Cache，每个 block 包含 32 个 token 的 KV。Attention kernel 针对 Turing/Ampere/Hopper 架构分别优化。Persistent KV Cache 通过 session_id 管理，支持手动控制 session 的生命周期。

DeepSpeed-FastGen — SplitFuse 技术

通俗类比：SplitFuse技术就像分子料理里的“拆解与重组”：把长任务（长文本预填充）拆成小块，和短任务（解码）混合在一起处理，让厨师（GPU）始终有活干，避免忙闲不均。
常见误区：SplitFuse会引入额外的调度和管理开销。对于请求长度分布非常均匀、且都很短的场景，其收益可能不明显，甚至略低于高度优化的连续批处理方案。
深入探讨：SplitFuse的核心是**动态拆分**和**融合**。它将长的Prompt预填充(Prefill)计算拆分成多个小块，并将这些块与短的解码(Decode)请求的小块融合成一个“批次”送给GPU执行。这样，GPU计算单元（Tensor Core）不会因为一个长Prefill而独占，解码请求也不会被长期阻塞，显著改善了在混合负载下的尾部延迟（P99 Latency）。

🧠 三层理解

第一层 · 核心概念

DeepSpeed-FastGen 的 SplitFuse 技术将长 prompt 的 prefill 和短请求的 decode 动态拆分并融合到同一个 step 中执行，避免传统方案中 prefill 和 decode 的互相干扰。

第二层 · 工作原理

传统方法将 prefill 和 decode 分开调度：长 prefill 会阻塞 decode 请求的响应。SplitFuse 将长 prefill 切分为多个小 chunk，与 decode token 交替组合成 batch。这样每个 step 的计算量更均匀，GPU 利用率更高，decode 延迟更低。

第三层 · 深入实现

SplitFuse 调度器维护两个队列：pending prefill queue 和 active decode queue。每个 step 中，调度器按 max_new_tokens 预算从两个队列中选取 token，组合成固定大小的 micro-batch。KV Cache 采用连续内存分配，配合 DeepSpeed ZeRO 进行跨 GPU 内存管理。

  • 混合长度负载处理优秀

  • 适合分布式训练+推理一体场景
  • 微软官方支持

多维度对比分析

通俗类比：对比框架就像评比不同菜系的餐厅。粤菜（SGLang）精致高效，川菜（TensorRT-LLM）火爆极致，本帮菜（LMDeploy）均衡实用。你要看你的“食客”（负载）喜欢什么。
常见误区：切勿只看单一的基准测试得分（如固定长度文本的吞吐）。一定要在你自己的业务数据（真实提示词长度分布、并发数）上进行测试，结果可能完全不同。
深入探讨：关键对比维度：1. **吞吐量（Tokens/s）**：单位时间生成的总token数。2. **Time-to-First-Token (TTFT)**：用户发送请求到接收第一个token的延迟，影响体验。3. **Inter-Token Latency (ITL)**：生成每个后续token的平均间隔。4. **显存占用**：相同并发数下的显存使用量。不同框架在这些指标上表现各异，且与硬件强相关。

选型指南与推荐

通俗类比：选框架就像为餐厅选“主厨”：追求极致上菜速度（在线服务）选TensorRT-LLM；追求处理复杂多变订单（研发实验）选vLLM/SGLang；追求稳定和易用性（快速部署）选LMDeploy。
常见误区：避免“技术追新”陷阱。最前沿的技术（如最新论文实现）可能在生产环境不稳定。优先考虑你团队的技术栈熟悉度（如是否熟悉TensorRT编译）、运维成本和社区支持力度。
深入探讨：决策树参考：1. **硬件是NVIDIA A100/H100且追求极限性能** -> TensorRT-LLM。2. **请求有大量相同前缀（多轮对话、批量分类）** -> SGLang。3. **通用高吞吐场景，需要稳定、易用、开源** -> vLLM 或 LMDeploy。4. **负载混合度极高，对P99延迟敏感** -> 考虑DeepSpeed-FastGen或其思想。5. **非NVIDIA硬件（如国产GPU）** -> 选择对目标硬件有适配的框架（如LMDeploy对某些国产硬件的适配）。

📈 高吞吐量场景（大批量推理、离线处理）

需求：最大化 token/s，可以容忍较高的首 token 延迟

🥇 TensorRT-LLM 🥈 vLLM 🥉 SGLang
  • TensorRT-LLM 的 TensorRT 编译优化 + Inflight Batching 提供最高吞吐
  • vLLM 的 PagedAttention + FP8 量化是简单易用的高吞吐方案
  • 配置建议：--gpu-memory-utilization 0.95 --max-num-seqs 512 --kv-cache-dtype fp8_e4m3

⚡ 低延迟场景（实时交互、聊天机器人）

需求：最小化 TTFT 和 TPOT，用户体验优先

🥇 TensorRT-LLM 🥈 SGLang 🥉 vLLM
  • TensorRT-LLM 的 kernel 优化提供最低的单请求延迟
  • SGLang 的 Chunked Prefill + 灵活调度策略减少排队延迟
  • vLLM 的 Chunked Prefill 模式也可以有效降低 TTFT
  • 配置建议：--enable-chunked-prefill --max-num-batched-tokens 4096

📜 长上下文场景（文档分析、代码理解）

需求：支持 32K-128K+ 上下文长度，内存效率优先

🥇 vLLM 🥈 SGLang 🥉 TensorRT-LLM
  • PagedAttention 的内存效率对长序列至关重要
  • KV Cache 量化（FP8）可以将可用上下文长度翻倍
  • Chunked Prefill 避免长 prompt 造成的 OOM 和延迟尖峰
  • 配置建议：--block-size 16 --kv-cache-dtype fp8_e4m3 --enable-chunked-prefill --max-num-batched-tokens 16384

🤖 多 Agent 场景（多轮对话、工具调用、Autogen/CrewAI）

需求：大量前缀重复的短请求，前缀缓存命中率优先

🥇 SGLang 🥈 vLLM 🥉 LMDeploy
  • SGLang 的 RadixAttention 对多轮对话前缀命中率最高（85-95%）
  • SGLang 的结构化输出优化（JSON/正则 FSM）对 Agent 工具调用友好
  • vLLM 的自动 Prefix Caching 是更简单的替代方案
  • 配置建议（SGLang）：--schedule-policy lpm --mem-fraction-static 0.85

🏭 生产部署场景（K8s、微服务、SLA 保障）

需求：稳定性、可运维性、监控、容错

🥇 vLLM 🥈 TensorRT-LLM 🥉 LMDeploy
  • vLLM 生态最成熟：OpenAI API 兼容、丰富的部署示例、活跃社区
  • TensorRT-LLM + Triton Inference Server 是 NVIDIA 官方推荐方案
  • vLLM 的 PD 分离支持灵活的生产架构
  • 配置建议：--enable-prefix-caching --preemption-mode swap --swap-space 8

🇨🇳 中文模型场景（InternLM、Qwen、ChatGLM）

需求：中文大模型的最佳推理性能

🥇 LMDeploy 🥈 vLLM 🥉 SGLang
  • LMDeploy 的 TurboMind 对 InternLM 系列有深度优化
  • W4A16 量化开箱即用，部署简单
  • Persistent KV Cache 适合中文对话场景

🎓 总结

KV Cache 管理是推理框架的核心差异点。各框架在不同维度上各有优势：

  • vLLM — PagedAttention 开创者，生态最成熟，通用场景首选
  • SGLang — RadixAttention 创新，前缀缓存最强，多轮/Agent 场景首选
  • TensorRT-LLM — NVIDIA 深度优化，极致性能首选
  • LMDeploy — 中文模型优化，InternLM 生态首选
  • DeepSpeed-FastGen — SplitFuse 技术，混合长度场景

⚠️ 以上性能数据为参考值，实际表现因硬件配置、模型规模、负载特征等因素而异。建议在目标场景下进行实测对比。
