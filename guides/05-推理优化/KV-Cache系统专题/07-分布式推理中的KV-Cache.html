Section 01

分布式推理概述

分布式推理概述

lass="tip">通俗类比：分布式推理就像一家大型快递分拣中心。一个包裹（模型参数）太大，单个工人（GPU）搬不动，所以需要多个工人协作，将工作（计算）拆分到不同区域（设备）同步进行，以实现快速送达（推理）。
常见误区：易犯‘盲目扩展’的错误。误以为只要增加更多GPU就能线性提升速度，忽略了通信、调度和内存同步带来的额外开销，有时效率反而下降。
深入探讨：一个70B参数模型，FP16精度下约需140GB显存。单卡A100（80GB）无法加载，必须使用至少2张卡进行模型并行。分布式推理的典型延迟由 `max(计算时间, 通信时间)` 决定。

为什么需要分布式推理？当模型参数和 KV Cache 超过单 GPU 显存时，必须将计算和状态分布到多个设备上。

直觉层

一辆卡车装不下，就用一个车队

想象你要搬运一栋房子的全部家具。一辆卡车装不下，你需要一个车队。但关键问题是：怎么分配家具到不同卡车上？到达目的地后怎么组装？分布式推理就是这个"分配与组装"的过程，而 KV Cache 就是那些需要在搬运过程中保持完好、并能快速取用的"贵重物品"。

原理层

四种并行策略的核心思想

分布式推理的本质是将模型的不同维度切分到多个 GPU 上：

  • 张量并行 (TP)：将单层的权重矩阵沿某个维度切分 — 每个 GPU 只计算矩阵的一部分
  • 流水线并行 (PP)：将模型的不同层分配到不同 GPU — 每个 GPU 负责一段连续的层
  • 专家并行 (EP)：MoE 模型特有，不同专家放在不同 GPU — 每个 token 只激活部分专家
  • 序列并行 (SP)：将序列长度维度切分 — 不同 GPU 处理同一序列的不同片段
实现层

通信拓扑决定并行策略的选择

实际部署中，选择哪种并行策略取决于硬件通信拓扑：

  • 同一节点内 GPU 间 NVLink（600~900 GB/s）→ 适合 TP
  • 跨节点 InfiniBand（400~800 Gb/s）→ 适合 PP 或 EP
  • PCIe（64 GB/s）→ 仅适合 PP，不适合 TP
  • 实际集群通常采用 TP+PP+EP 混合策略（如 Megatron-LM 3D 并行）
图 7-1: 3D 并行（TP + PP）中 KV Cache 的分布全景，展示不同并行策略如何切分 KV Cache
🔑 核心洞察：并行策略决定了 KV Cache 的切分方式

张量并行沿 注意力头维度 切分 KV Cache，流水线并行沿 层维度 切分，专家并行 不切分 KV Cache（但影响路由模式），序列并行沿 序列长度维度 切分。理解这些切分维度是优化分布式 KV Cache 的基础。

Section 02

张量并行中的 KV Cache

张量并行中的 KV Cache

通俗类比：张量并行就像一个合唱团唱高音部。每个歌手（GPU）只唱自己的一个声部（注意力头的一个子集），最终所有声部和谐地混合在一起，形成完整合唱（注意力输出）。
常见误区：错误地认为每个GPU都能独立处理完整的KV Cache。实际上，每个GPU只负责自己‘声部’对应的KV Cache片段，需要All-Reduce通信才能获得全局注意力结果。
深入探讨：对于有 `H` 个注意力头的模型，使用 `T` 张GPU张量并行时，每张GPU只处理 `H/T` 个头的KV Cache。通信量是 `(seq_len, 2*H/T, d_head)` 形状的张量进行All-Reduce。

张量并行将注意力头分配到不同 GPU，每个 GPU 只存储和计算自己负责的头的 KV Cache。

直觉层

一本大字典，每人读几页

想象一本 1000 页的字典（模型权重），4 个人一起查字典。每个人只需要带自己负责的 250 页。KV Cache 也是同理 — 如果模型有 32 个注意力头，用 TP=4，那么每个 GPU 只存储 8 个头的 KV Cache，显存占用直接降到 1/4。

原理层

注意力计算的切分与合并

Multi-Head Attention 天然适合张量并行：

  • KV Cache 存储：每个 GPU 存储 num_kv_heads/TP 个头的 K 和 V
  • 注意力计算：每个 GPU 独立计算自己负责的头的 Q·K^T 和 Attn·V
  • 输出合并：通过 AllReduce 将各 GPU 的输出部分求和
  • Block Table：每个 GPU 维护独立的 block table，管理自己的 KV Cache blocks
实现层

GQA 与 TP 的交互

对于 GQA 模型（如 LLaMA-70B 有 8 个 KV 头，64 个 Q 头）：

  • TP=8 时：每个 GPU 只有 1 个 KV 头，但有 8 个 Q 头
  • KV Cache 大小 = (num_kv_heads/TP) × seq_len × d_head × 2(K+V) × dtype_bytes
  • 如果 num_kv_heads < TP，需要复制 KV 头或使用特殊的 TP 划分策略
  • Block table 在每个 GPU 上独立分配，无需跨 GPU 同步
图 7-2: 张量并行 KV Cache 分布 — 每个 GPU 存储和计算一部分注意力头的 KV
TP 模式下每个 GPU 的 KV Cache 显存 MemGPU_i = 2 × (num_kv_heads / TP) × num_layers × seq_len × d_head × bytes_per_element

📋 Block Table 管理（TP 模式）

  • 每个 GPU 维护独立的 block table
  • Block table 大小 = max_num_blocks × (num_kv_heads/TP)
  • 物理 block 只存储该 GPU 负责的头
  • 逻辑 block 到物理 block 的映射每个 GPU 独立
  • PagedAttention 在 TP 模式下自然并行化

⚡ 性能特征

  • KV Cache 读取带宽 = 单 GPU / TP（天然降低）
  • Attention 计算量 = 总量 / TP
  • 增加通信：AllReduce 输出 (2 × hidden_size × batch)
  • TP=8 理论加速 ~7.5x（考虑通信开销）
  • KV Cache 命中率不受 TP 影响
# vLLM: TP 模式下的 KV Cache 配置 class TensorParallelKVCacheManager: def __init__(self, tp_size, tp_rank, num_kv_heads, ...): self.tp_size = tp_size self.tp_rank = tp_rank # 每个 GPU 只负责部分 KV 头 self.num_kv_heads_per_gpu = num_kv_heads // tp_size self.head_range = ( tp_rank * self.num_kv_heads_per_gpu, (tp_rank + 1) * self.num_kv_heads_per_gpu ) # Block table 独立管理 self.block_table = BlockTable( num_blocks=max_blocks, num_heads=self.num_kv_heads_per_gpu, head_dim=head_dim ) def allocate_kv_cache(self, num_tokens): # 每个 GPU 分配自己份额的 KV Cache k_cache = torch.empty( num_tokens, self.num_kv_heads_per_gpu, self.head_size, dtype=self.dtype, device=f"cuda:{self.tp_rank}" ) v_cache = torch.empty_like(k_cache) return k_cache, v_cache
Section 03

流水线并行中的 KV Cache

流水线并行中的 KV Cache

通俗类比：流水线并行就像汽车装配线。车身（输入数据）依次经过底盘车间（GPU0）、喷漆车间（GPU1）、总装车间（GPU2）。每个车间只负责自己的环节（模型层），中间产物（激活值）需要传递。
常见误区：忽略‘流水线气泡’（Bubble）问题。由于各阶段计算时间不均或等待前一阶段数据，导致部分GPU闲置，这是流水线并行效率损失的主要来源。
深入探讨：假设模型有L层，均分为S个阶段，每个阶段L/S层。理想吞吐量提升S倍，但实际因流水线气泡，加速比为 `S / (1 + (S-1)/M)`，M是微批次大小。
时间 ->
[GPU0] [Layer1-4]  [Layer1-4] ...
[GPU1]   [Layer5-8]  [Layer5-8] ...
[GPU2]      [Layer9-12] [Layer9-12] ...
         ^气泡^    ^气泡^

流水线并行将模型的不同层分配到不同 GPU，KV Cache 自然按层分割，无需跨阶段通信。

直觉层

接力赛跑，每人跑一段

流水线并行就像接力赛：第一个选手跑 0-100 米，第二个跑 100-200 米。每个选手只需要记住自己那段的记忆（KV Cache）。第一个选手不需要知道 100-200 米发生了什么，因为那已经是下一个选手的事了。所以 KV Cache 天然被层范围切分，互不干扰。

原理层

KV Cache 的层归属

  • 层切分：PP Stage i 拥有第 [i×L/PP, (i+1)×L/PP) 层的 KV Cache
  • 无需跨阶段 KV 通信：每层的自注意力只使用本层的 KV Cache
  • Activation 传递：需要传递的是激活值（hidden states），不是 KV Cache
  • 每个 Stage 的 KV Cache 容量 = Total / PP（按层数均分）
实现层

Micro-batch 与 KV Cache 生命周期

  • 1F1B 调度：每个 Stage 最多同时处理 bubble+1 个 micro-batch
  • 每个 micro-batch 有自己的 KV Cache 空间
  • Pipeline bubble 期间：KV Cache 空间被占用但未充分利用
  • 实际显存需求 = micro_batches_in_flight × per_microbatch_kv_size
  • vLLM 的 continuous batching 与 PP 的 micro-batch 需要协调
图 7-3: 流水线并行中 KV Cache 按层自然切分 — 跨 Stage 只需传递 Activation
🔑 PP 的优势：KV Cache 零通信

与 TP 需要 AllReduce 不同，PP 模式下 KV Cache 完全不需要跨阶段通信。每个 Stage 只需传递 hidden states（大小为 [batch, seq, hidden_dim]），而 KV Cache 的通信量为零。这是 PP 在长序列场景下的重要优势。

⏱️ Pipeline Schedule 与 KV Cache

  • GPipe: 先完整前向，KV Cache 全部生成后再反向
  • 1F1B: 前向反向交替，KV Cache 峰值较低
  • Interleaved 1F1B: 更多 bubble 但 KV Cache 更分散
  • 推理阶段只有前向，Schedule 影响 decode 调度策略

💾 PP 中 KV Cache 的显存压力

  • Stage 0 和 Stage N-1 通常显存压力相同
  • 中间 Stage 如果有额外的 activation，可能显存更紧
  • KV Cache 总量 ÷ PP = 每 Stage 显存需求
  • 但每 Stage 的 micro-batch 数量不同影响峰值
Section 04

专家并行 (MoE) 中的 KV Cache

专家并行 (MoE) 中的 KV Cache

通俗类比：MoE就像公司的专家顾问团。门控网络（路由器）是项目总监，根据项目类型（输入token），指派最合适的专家（少数FFN层）去处理，其他专家休息。这样公司（模型）能力大但日常开销（计算量）小。
常见误区：误以为MoE中所有专家的KV Cache都需要被激活和缓存。实际上，每个token只路由到少数专家（如2/8），因此只需存储和缓存被激活专家对应的Key和Value。
深入探讨：在一个 `N` 选 `K` 的MoE层中，每个token的KV Cache仅需为 `K` 个专家分别存储。显存需求约为稠密模型的 `K/N` 倍，但通信和负载均衡是新的瓶颈。

MoE 模型中，KV Cache 是 per-layer 的，不受 Expert Routing 影响，但专家的稀疏激活模式带来独特考量。

直觉层

KV Cache 是公区，专家是私室

MoE 模型就像一栋大楼，KV Cache 是大厅里的公共信息板（每层都有），而专家是不同的房间。不管哪个专家被激活，公共信息板都在那里。每个 token 进来后会根据路由算法去不同的"房间"（专家），但它们共用同一块"信息板"（KV Cache）。

原理层

KV Cache 与 Expert Routing 的关系

  • Attention 层不区分专家：Self-Attention 对所有 token 一视同仁
  • Expert 只影响 FFN：路由发生在 FFN 层，不影响 K/V 的生成
  • KV Cache 与 dense 模型相同：MoE 的 KV Cache 大小 = dense 版本
  • EP 不切分 KV Cache：EP 只切分 FFN 权重和 Expert 参数
实现层

DeepSeek-V3 的 EP 实现

  • DeepSeek-V3: 671B 参数，37B 激活（每 token 路由 8/256 专家）
  • EP=64：256 个专家分布在 64 个 GPU 上，每 GPU 4 个专家
  • AllToAll 通信：将 token 路由到对应专家的 GPU
  • KV Cache 存储在每个 GPU 的 Attention 计算位置
  • KV Cache 管理独立于 Expert 路由
图 7-4: MoE 专家并行 — KV Cache 在 Attention 层共享，路由只影响 FFN
🔑 MoE 中 KV Cache 的关键特征

MoE 模型的 KV Cache 与同等参数量的 Dense 模型相比，大小取决于 Attention 层的参数（不是 Expert FFN 参数）。DeepSeek-V3 的 671B 参数中，KV 相关参数很小（MLA 压缩后更小），所以 KV Cache 并不是 MoE 模型的显存瓶颈 — 真正的瓶颈是 Expert 权重的分布。

序列并行与 KV Cache

序列并行与 KV Cache

通俗类比：序列并行就像处理一篇超长论文。将长序列（论文）按顺序切成几段（章节），分给不同的人（GPU）并行阅读并记笔记（计算KV Cache），最后汇总所有人的笔记（All-Gather）以理解全文。
常见误区：易混淆‘序列并行’与‘张量并行’。前者是沿序列维度切分输入，后者是沿隐藏层维度切分参数。两者常结合使用（SP+TP），但通信模式不同。
深入探讨：对于超长序列（如128K），单卡KV Cache显存可能不够。序列并行将序列分成 `S` 段，每段长度 `L/S`，KV Cache显存需求降为 `1/S`。但需要在前向计算中多次进行 `All-Gather` 同步。

当序列长度超过单 GPU 的 KV Cache 容量时，需要将序列沿长度维度切分到多个 GPU。

直觉层

一本超长小说，每人读几章

想象一本 100 万字的小说，一张桌子（GPU 显存）放不下所有笔记。4 个人分工：第一个人读第 1-25 万字并做笔记（KV Cache），第二个人读第 25-50 万字...最后，每个人需要参考其他人的笔记来理解完整故事，所以需要定期交换笔记。

原理层

两种序列并行策略

  • Ring Attention：KV 块在 GPU 间环形传递，每个 GPU 依次处理所有 KV 块的注意力
  • Ulysses (DeepSpeed)：使用 AllToAll 将序列维度转置为头维度，然后正常计算
  • KV Cache 切分：每个 GPU 存储 seq_len/SP 个位置的 KV Cache
  • 通信模式：Ring 是 P2P 逐步传递，Ulysses 是 AllToAll 一步完成
实现层

Ring Attention 的 KV Cache 管理

  • 每个 GPU 维护 KV Cache[seq_start:seq_end, heads, d_head]
  • Ring 通信：每步将本地 KV 发送给下一个 GPU，同时接收上一个 GPU 的 KV
  • 注意力计算需要 SP 轮（每个 GPU 的 KV 都要参与计算一次）
  • 显存额外需求：1 份远程 KV 缓冲区（recv buffer）
  • 计算/通信可以重叠：计算当前 KV 块的同时接收下一个 KV 块

Section 06

TP 与 KV Cache 的通信开销

TP 与 KV Cache 的通信开销

通俗类比：TP中的通信开销就像合唱团排练时的对音。每次唱完一句（一层计算），都需要停下来，交换各声部信息（通信）并校准音高（同步结果），才能开始下一句。对音越频繁，整体进度越慢。
常见误区：低估通信带宽的影响。在低速网络（如多机之间）下，频繁的All-Reduce通信会成为严重瓶颈，导致张量并行的效果大打折扣，甚至不如单机。
深入探讨：对于一个隐藏维度 `d=4096`，序列长度 `seq=2048` 的张量，使用FP16，每次All-Reduce的数据量为 `2*seq*d*2 bytes = 33.5 MB`。在100Gbps网络下，仅通信延迟就需约2.7ms。

张量并行的 AllReduce 通信是推理延迟的重要组成部分，理解通信模式和优化策略至关重要。

直觉层

人越多，开会越慢

TP 就像一个团队分工合作。分工后每个人干得快了，但需要频繁开会（AllReduce）同步结果。TP=2 两个人开会很快，TP=8 八个人开会就慢多了。关键是找到"分工效率"和"开会成本"的平衡点。

原理层

AllReduce 通信量分析

  • Attention 层：AllReduce 输出 → 2 × (TP-1)/TP × batch × seq_len × hidden_size
  • FFN 层：同样需要 AllReduce → 又一次 2 × (TP-1)/TP × batch × seq_len × hidden_size
  • 每层通信量 ≈ 4 × hidden_size × batch × seq × bytes（Ring AllReduce）
  • Decode 阶段：seq=1，通信量 = 4 × hidden_size × batch × 2B ≈ 很小
  • Prefill 阶段：seq 很大，通信量显著
实现层

NCCL AllReduce 优化

  • Ring AllReduce：2×(N-1)/N × data_size，带宽利用率高
  • Tree AllReduce：延迟低但带宽利用率低
  • 计算/通信 Overlap：NCCL 支持异步执行，可与下一层计算重叠
  • TP=8 Intranode：NVLink 600+ GB/s，通信延迟 < 50μs
  • TP=2 跨节点：IB 400G = 50 GB/s，通信延迟 ~100-500μs
图 7-6: TP 模式下每层推理的通信模式与不同 TP 值的开销对比
TP AllReduce 通信量（每层） Comm = 2 × (TP-1)/TP × batch_size × seq_len × hidden_size × dtype_bytes
🔑 Decode vs Prefill 通信量差异巨大

Decode 阶段每次只处理 1 个 token，AllReduce 通信量仅为 2×hidden_size×batch×2B（~0.5MB for hidden=8192, batch=32），通常 < 100μs。而 Prefill 阶段处理完整序列（如 4096 tokens），通信量是 Decode 的数千倍。这就是为什么 TP 主要受限于 Prefill 阶段的通信。

多节点 KV Cache 管理

多节点 KV Cache 管理

通俗类比：多节点KV Cache管理就像图书馆的分布式馆藏。热门书籍（常用KV Cache）放在总馆（主节点内存），普通书籍分布在各个分馆（从节点）。需要有智能的检索系统（调度策略）快速找到并调取。
常见误区：忽视‘缓存一致性’和‘内存碎片’问题。在动态请求下，不同节点的KV Cache版本可能不一致，且频繁分配/释放会导致显存碎片化，降低有效容量。

当 KV Cache 分布在多个节点上时，互连带宽成为关键瓶颈。选择正确的并行策略和互连拓扑至关重要。

直觉层

省内快递 vs 跨省快递

同一个节点内的 GPU 就像同一个城市的快递 — NVLink 是高速公路，速度极快（600+ GB/s）。跨节点就像跨省快递 — InfiniBand 虽然也很快（50-100 GB/s），但比 NVLink 慢 10 倍。所以"省内能解决的事别跨省"：把需要高频通信的放在同一节点内。

原理层

互连带宽决定并行策略

  • NVLink (节点内): 600-900 GB/s → 适合 TP（高频 AllReduce）
  • InfiniBand (跨节点): 50-100 GB/s → 适合 PP（低频 P2P）或 EP
  • PCIe (备用): 64 GB/s → 仅适合 PP
  • 核心原则：通信频率高的并行放在带宽高的互连上
实现层

典型集群配置

  • 8×H100 节点: TP=8 intranode, PP=4 across 4 nodes
  • KV Cache 主要在节点内（TP 切分），不跨节点
  • PP 跨节点：只有 hidden states 传递，不含 KV Cache
  • EP 跨节点：token routing 通过 IB，但 KV Cache 仍在本地
  • NVLink SHARP: 节点内 allreduce 硬件加速

DeepSeek-V3/V4 分布式 KV Cache 实践

DeepSeek-V3 是 671B 参数的 MoE 模型，使用 MLA 压缩 + EP + TP 的混合并行策略，是分布式 KV Cache 的工业级实践。

直觉层

一个超级工厂的流水线设计

DeepSeek-V3 就像一个拥有 256 个工位（专家）的超级工厂。MLA 就是压缩包装技术 — 把巨大的 KV Cache 压缩到极小的尺寸。EP 就是把 256 个工位分配到 64 个车间（GPU），每个车间 4 个工位。TP 则是每个工位内部的精密分工。

原理层

MLA + EP + TP 的协同

  • MLA (Multi-Latent Attention)：将 KV 压缩到低维 latent，每 token 仅需存储 512 维 latent（而非传统 2×128×num_heads 维）
  • EP=64：256 个专家分布在 64 个 GPU，每 GPU 4 个专家
  • TP=16 (节点内)：Attention 的 QKV 投影和输出投影用 TP 切分
  • KV Cache 极小：MLA 使 KV Cache 缩小 ~93%（与传统 MHA 相比）
  • KV Cache 管理：latent cache 存储在 TP group 内的每个 GPU 上
实现层

DeepSeek-V3 部署架构

  • 8 节点 × 8 GPU = 64 GPU 集群
  • TP=16（2 节点），EP=64（所有 GPU），PP=4（可选）
  • KV Cache latent: [seq_len, 512] 每层每 token
  • 传统等效: [seq_len, num_heads(128) × d_head(128) × 2] = 每层 32K 维
  • 压缩比: 512 / 32768 = 1.56%（压缩了 64 倍！）
  • KV Cache 显存: 几乎不再瓶颈，主要瓶颈在 Expert 权重分布
图 7-8: DeepSeek-V3 分布式推理架构 — MLA 压缩使 KV Cache 几乎可忽略
# DeepSeek-V3 MLA KV Cache 核心概念 class MLAKVCache: """ DeepSeek-V3 MLA: 只存储压缩后的 latent，而非完整的 K/V 传统 MHA: 存储 K[num_heads, d_head] + V[num_heads, d_head] = 2 × 128 × 128 = 32768 维 MLA: 存储 latent[d_compress] = 512 维 压缩比: 512 / 32768 = 1.56% (64× 压缩) """ def __init__(self, d_compress=512, ...): self.d_compress = d_compress # latent 维度 def store(self, token_hidden_state): # 只存储压缩后的 latent latent = self.W_compress(token_hidden_state) # [batch, seq, 512] self.cache.append(latent) def retrieve(self): # 从 latent 重建完整的 K, V (on-the-fly) latent = torch.stack(self.cache) K = self.W_k_up(latent) # [batch, seq, num_heads, d_head] V = self.W_v_up(latent) # [batch, seq, num_heads, d_head] return K, V

1. 分布式推理概述

四种并行策略（TP/PP/EP/SP）从不同维度切分模型。通信拓扑（NVLink/IB/PCIe）决定哪些策略适合放在哪里。实际部署通常采用 3D 混合并行。

2. 张量并行中的 KV Cache

TP 沿注意力头维度切分 KV Cache。每 GPU 存储 num_kv_heads/TP 个头。需要 AllReduce 同步输出，但 KV Cache 读取带宽天然降低。

3. 流水线并行中的 KV Cache

PP 沿层维度切分，KV Cache 零通信！每个 Stage 独立管理自己的 KV Cache Pool。是跨节点部署的首选策略。

4. 专家并行中的 KV Cache

MoE 的 KV Cache 是 per-layer 的，不受 Expert Routing 影响。EP 不切分 KV Cache，只切分 FFN 专家权重。

5. 序列并行与 KV Cache

Ring Attention 和 Ulysses 将序列长度维度切分到多 GPU。每 GPU 只存储 1/SP 的序列位置的 KV Cache。适合超长序列。

6. TP 通信开销

Decode 阶段 AllReduce 通信量小（~0.5MB），Prefill 阶段大（数百 MB）。TP 应限制在节点内（NVLink 600+ GB/s）。

7. 多节点 KV Cache 管理

核心原则：让 KV Cache 留在本地。TP 在节点内，PP 跨节点。KV Cache 不需要跨节点传输是最佳状态。

8. DeepSeek-V3 实践

MLA 压缩 64× 使 KV Cache 从 34GB 降至 0.5GB。分布式推理的主要挑战从 KV Cache 管理转变为 Expert 权重分布和 AllToAll 通信优化。

🎯 核心公式速查

TP 模式 KV Cache/GPU: 2 × (num_kv_heads / TP) × num_layers × seq_len × d_head × dtype_bytes
PP 模式 KV Cache/Stage: 2 × num_kv_heads × (num_layers / PP) × seq_len × d_head × dtype_bytes
MLA KV Cache/Token: d_compress × num_layers × dtype_bytes (e.g., 512 × 61 × 2B = 61KB)
TP AllReduce/Layer: 2 × (TP-1)/TP × batch × seq × hidden × dtype_bytes
Ring Attention 通信: SP 轮 × (seq_len / SP) × hidden_size × dtype_bytes
