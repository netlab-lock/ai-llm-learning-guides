1. 为什么量化 KV Cache

通俗类比：量化KV Cache如同给厨房储物柜“瘦身”——柜子（显存）大小固定，压缩食材（KV Cache）就能存更多菜谱（上下文），同时降低存取成本。
常见误区：切勿只关注模型权重压缩而忽视KV Cache，它占推理显存大头。忽视它，如同只优化厨具却任由食材乱堆，厨房照样混乱。
深入探讨：KV Cache显存计算公式：`Mem = 2 * L * d * b * s * p`（2: K和V，L: 层数，d: 维度，b: 批次，s: 序列长，p: 精度字节）。以L=32, d=4096, b=32, s=4096, p=2(FP16)为例，单次推理占用高达32GB，远超模型权重本身。

1.1 核心动机：内存瓶颈

💡 直觉层

KV Cache 是 FP16 存储的，每个 token 每层需要存 2×d_model 个参数。如果我们把精度从 16-bit 降到 4-bit，就能把 KV Cache 缩小到原来的 1/4，相当于免费多了 4 倍的上下文长度。

⚙️ 技术层

Decode 阶段是 Memory-bound 的，每个 token 需要从 HBM 读取全部 KV Cache。降低 KV Cache 的比特宽度 = 降低 HBM 带宽需求 = 提升 decode 吞吐。量化带来的带宽节省是直接的：FP16→INT8 带宽减半，FP16→INT4 带宽降到 1/4。

📐 数学层

对于 L 层、d_model 维、seq_len 长度的模型，FP16 KV Cache 大小 = 2 × 2 × L × d_model × seq_len bytes。以 LLaMA-70B (L=80, d=6144) 为例，4096 tokens 的 KV Cache = 2×2×80×6144×4096 ≈ 7.5 GB。INT4 量化后降至 ≈1.9 GB。

🔑 关键洞察：KV Cache 量化与 Weight 量化不同。Weight 量化在 prefill 阶段计算，一次性完成；KV Cache 量化需要在每个 decode step 对新产生的 KV 向量实时量化，并在 attention 计算时实时反量化。这决定了它对 kernel 的要求：低延迟的 quantize/dequantize 操作。

2. 量化基础回顾

通俗类比：量化基础就像学习“压缩打包”技巧：用更小的箱子（低精度）装东西，关键在于如何选择箱子（量化粒度）并填写装箱单（scale/zero-point）。
常见误区：切忌将对称量化与非对称量化混淆。对称量化假设数据对称分布（如中心0），非对称更灵活。选错如同用圆筒装长方形蛋糕，空间浪费或损坏。
深入探讨：量化通用公式：`q = round(x / scale) + zero_point`。其中 `scale = (max(x) - min(x)) / (q_max - q_min)`，`zero_point = round(-min(x)/scale) - q_min`。INT8量化范围通常为[-128, 127]或[0, 255]。
  原始浮点值: -1.2, 0.5, 3.7
  INT8对称:   -60,  25, 185
  计算: q = round(x / 0.02) + 128
量化类型数据分布要求典型应用
对称量化零中心对称权重、激活值
非对称量化任意分布KV Cache（值域常非对称）

2.1 Symmetric vs Asymmetric 量化

对称量化 (Symmetric)

// 对称量化公式
q = round(x / scale)
x_hat = q × scale

// scale = max(|x|) / (2^(b-1) - 1)
// 零点 zp = 0，映射关于原点对称

优点：计算简单，无零点偏移开销
缺点：分布不对称时（如 ReLU 后的激活）浪费表示范围

非对称量化 (Asymmetric)

// 非对称量化公式
q = round((x - zp) / scale)
x_hat = q × scale + zp

// scale = (max(x) - min(x)) / (2^b - 1)
// zp = min(x)，引入额外存储

优点：更好利用表示范围，适合非对称分布
缺点：需要存储零点，计算稍复杂

图 2.1: 对称量化 vs 非对称量化 —— KV Cache 中 Key 和 Value 的分布决定了哪种更优

2.3 动态 vs 静态量化

🟢 动态量化 (Dynamic)

在运行时根据实际数据计算 scale/zp。

  • ✅ 适配实时分布变化
  • ✅ 不需要校准数据
  • ❌ 有计算开销（每个 token 需计算 min/max）

KV Cache 默认使用动态量化，因为 KV 向量在 decode 时动态生成。

🟡 静态量化 (Static)

使用预先确定的 scale/zp，来自校准阶段。

  • ✅ 无运行时开销
  • ❌ 需要校准数据集
  • ❌ 无法适应分布变化

对 KV Cache 不适用，因为 KV 分布随序列动态变化。

// 常用量化误差度量

SNR = 10 × log₁₀(var(x) / MSE) // 信噪比 (dB)
CosSim = (x · x̂) / (||x|| × ||x̂||) // 余弦相似度

3. KV Cache 特定量化方案

入理解KV Cache 特定量化方案的工作原理，有助于在实际工程中做出更合理的技术选择和优化决策。

KV Cache 特定量化方案

通俗类比：KV Cache量化如同处理“特殊食材”：K（键）和V（值）性格不同（K常更敏感），需定制方案，不能粗暴地一刀切。
常见误区：避免对K和V使用完全相同的量化参数。V通常比K对量化更鲁棒，忽略此差异可能导致注意力计算精度显著下降。
深入探讨：研究发现：将Key量化至INT8精度损失约0.1-0.5%，而Value量化至INT4仍可保持<1%损失。常见方案：K用INT8，V用INT4（K8V4），实现精度与效率的最佳平衡。

📌 KIVI: A Tuning-Free 2bit Quantization for KV Cache (2024)

Liu et al. | ICML 204 | arXiv: 2402.02750
💡 核心思想

发现 K 和 V 在不同维度上的分布特征不同：Key 在 channel 维度有显著 outlier，而 Value 在 token 维度有 outlier。因此对 K 采用 per-channel 量化，对 V 采用 per-token 量化。

⚙️ 算法

1) Key: per-channel 对称量化到 INT2，每个 channel 一个 scale
2) Value: per-token 对称量化到 INT2，每个 token 一个 scale
3) 最近 token 保留 FP16 不量化（保护最新上下文）
4) 非对称量化采用 asymmetric 方式减少 INT2 损失

📊 结果

2-bit KV Cache 在 LLaMA-7B/13B/70B 上质量损失极小。相比 FP16，KV Cache 内存降低 8×。在 1-shot MMLU 上精度损失 < 0.5%。支持 4× 更长的上下文。

// KIVI: 量化策略选择
// Key: outlier 在 channel 维度 → per-channel 量化
K_quant[i, :] = round(K[i, :] / scale_K[i]) // i = channel index
scale_K[i] = max(|K[i, :]|) / 3 // INT2: [-2, +2] → /3

// Value: outlier 在 token 维度 → per-token 量化
V_quant[:, j] = round(V[:, j] / scale_V[j]) // j = token index
scale_V[j] = max(|V[:, j]|) / 3

// 保留最近 w 个 token 为 FP16 (w=128 常用)
Cache = [INT2_K[:, :-w], FP16_K[:, -w:]] & [INT2_V[:, :-w], FP16_V[:, -w:]]

📌 KVQuant: Towards 10 Million Context Length LLM Inference (2024)

Hooper et al. | arXiv: 2401.18079
💡 核心思想

系统性地研究 KV Cache 量化，提出 per-channel 量化 + outlier 保护方案。使用非对称量化 + 学习的最优 group size 来最小化量化误差。额外提出了 Per-Channel-Key + Per-Token-Value 的混合策略。

⚙️ 关键技术

1) Per-Channel Key 量化：发现 Key 在 channel 维度分布差异大，per-channel 量化 INT4 效果最佳
2) Per-Token Value 量化：Value 在 token 维度变化大，per-token 量化更优
3) Pre-RoPE Key 量化：在 RoPE 旋转之前量化 Key，减少旋转引入的 outlier
4) Norm 常量吸收：将 LayerNorm 的 scale 吸收到量化参数中

📊 结果

INT3 KV Cache 在 LLaMA-70B 上达到接近无损质量。支持 10M context length 的 3-bit KV Cache 量化。Per-channel key + per-token value 的组合比纯 per-token 方案精度提升 0.5-1.0 perplexity。

📌 Gear: An Efficient KV Cache for LLM Inference (2024)

Kang et al. | arXiv: 2403.05527
💡 核心思想

采用低秩矩阵 + 量化残差的混合压缩方案。将 KV Cache 分为两部分：低秩近似（捕获主要模式）和量化残差（存储差值）。残差天然接近零，非常适合低位量化。

⚙️ 算法

1) 对历史 KV Cache 做 SVD 低秩分解: KV ≈ U × V
2) 计算残差: R = KV - U × V
3) 残差 R 用 INT4 量化（因分布集中在 0 附近，量化误差小）
4) Attention 计算时: KV ≈ U×V + dequant(R)

📊 结果

在 LLaMA-2-7B 上，3.12× 压缩比下 perplexity 损失 < 0.1。比纯 INT4 量化精度更高，因为低秩近似保留了主信息。额外计算开销约 5-10%（SVD 计算可以增量更新）。

📌 QServe: W4A8KV4 Quantization for LLM Serving (2024)

Lin et al. | MLSys 2025 | arXiv: 2405.04532
💡 核心思想

提出 W4A8KV4 统一量化方案：4-bit Weight + 8-bit Activation + 4-bit KV Cache。系统性地优化了 GPU kernel 以实现端到端加速。

⚙️ 算法

1) Weight: INT4 对称 per-group 量化 (group=128)
2) Activation: INT8 动态量化，per-token scale
3) KV Cache: INT4 对称量化，per-channel for Key, per-token for Value
4) SmoothQuant 的 smooth 技术处理 outlier
5) 自研 CUDA kernel 实现 fused quantized attention

📊 结果

在 L40S GPU 上，QServe-LLaMA-70B 达到 1.0-1.3× 吞吐提升。相比 FP16 基线，KV Cache 内存降低 4×，质量损失在 benchmark 上 < 1%。首个实现 W4A8KV4 端到端加速的系统。

4. Per-Token vs Per-Channel 量化

通俗类比：Per-Token量化像按“每页书”压缩（每页有独立压缩参数），Per-Channel像按“每行字”压缩。选择取决于哪种方式能让压缩误差更均匀。
常见误区：Per-Channel量化可能忽略不同token间特征分布差异，对KV Cache效果通常不佳。KV Cache更适用Per-Token量化，因为每个token的K/V向量应独立处理。
深入探讨：Per-Token: 每个token的整个d维向量共享一组`scale`和`zero_point`。公式：`scale_t = max(abs(K_t)) / 127`。Per-Channel: 每个特征维度（channel）在所有token上共享参数。对于KV Cache，Per-Token量化能更好地保留单个token的注意力模式。
Per-Token (推荐):  [t1_scale, t1_zp]用于整个K[t1,:]
Per-Channel:      [d1_scale]用于所有K[:,d1]列

4.1 为什么 Key 和 Value 应该用不同粒度

📊 实验观察：Key 矩阵在 channel 维度（特征维度）存在显著的 outlier channels，某些 channel 的值比其他 channel 大 10-100×。而 Value 矩阵在 token 维度存在 outlier tokens，某些 token 的 V 向量范数远大于平均值。这决定了最优的量化粒度选择。
图 4.1: Key 的 outlier 在 channel 维度，Value 的 outlier 在 token 维度

4.2 Per-Token 量化实现

def per_token_quantize_int8(kv_cache, # [num_heads, seq_len, head_dim]
                               axis=1):
    # axis=1 表示 per-token 量化 (对每个 token 独立量化)
    # 计算每个 token 的 absmax 作为 scale
    abs_max = kv_cache.abs().amax(dim=axis, keepdim=True)
    scale = abs_max / 127.0

    # 量化到 INT8
    kv_int8 = (kv_cache / scale).round().clamp(-128, 127).to(torch.int8)

    # 反量化
    kv_dequant = kv_int8.float() * scale

    return kv_int8, scale, kv_dequant

4.3 Per-Channel 量化实现

def per_channel_quantize_int4(kv_cache, # [num_heads, seq_len, head_dim]
                                axis=2,  # 对 head_dim 维度
                                group_size=128):
    # Per-group 量化：每 group_size 个 channel 共享一个 scale
    H, S, D = kv_cache.shape
    assert D % group_size == 0

    # reshape 为 [H, S, D//G, G]
    kv_grouped = kv_cache.reshape(H, S, D // group_size, group_size)

    # 每组的 absmax
    abs_max = kv_grouped.abs().amax(dim=-1, keepdim=True)
    scale = abs_max / 7.0  # INT4: [-8, 7] → /7 对称

    # 量化到 INT4
    kv_int4 = (kv_grouped / scale).round().clamp(-8, 7).to(torch.int8)

    return kv_int4, scale

4.4 Activation-Aware 量化

核心思想：不仅考虑 KV Cache 自身的分布，还考虑与 Query 的注意力交互。某些 KV 向量虽然自身值小，但在与特定 Query 交互时贡献大（即 attention score 高），这些 KV 应该用更高精度。

// Activation-Aware KV 量化
// 1. 计算每个 KV token 的 "重要性分数"
importance[j] = Σ_i |Q_i · K_j| / √d // 累积 attention score

// 2. 根据重要性分配精度
if importance[j] > threshold:
    store KV[:, j] as FP16 // 高重要性 → 高精度
else:
    store KV[:, j] as INT4 // 低重要性 → 低精度

// KIVI 的 "recent tokens stay FP16" 是此思想的特例

5. 量化 Kernel 实现

通俗类比：量化Kernel就像厨房的“高效解压机”：硬件上需要专门设计（CUDA内核），才能快速将压缩的KV数据解压（反量化）并计算，避免成为新瓶颈。
常见误区：切勿只优化量化过程而忽略反量化开销。反量化常与注意力计算融合，若实现不当，计算延迟可能抵消显存节省带来的收益。
深入探讨：高效Kernel设计关键：1) 融合反量化与注意力计算，减少中间数据搬运；2) 使用向量化指令（如CUDA的`__ldg`）提升访存效率；3) 针对INT4/INT8设计专用矩阵乘（GEMM）内核。例如，FlashAttention-2已集成INT8 KV Cache支持。

6. 与其他压缩方法的组合

7.5 实际部署 Checklist

✅ KV Cache 量化部署检查清单：
  • 硬件确认：GPU 是否支持 FP8 (SM89+)? 是否支持 INT8 Tensor Core (SM80+)?
  • 框架支持：vLLM 支持 FP8/INT8 KV；TensorRT-LLM 支持 FP8 KV；TGI 支持有限
  • 精度验证：在目标 benchmark 上对比 FP16 baseline，PPL 增加应 < 0.5
  • 内存测量：确认实际内存节省符合预期（scale/zp 有额外开销）
  • 延迟基准：量化后 decode 延迟应降低（非增加，否则方案有问题）
  • 长序列测试：在接近最大上下文长度时质量是否仍可接受
  • 边界情况：短序列 (< 128 tokens) 时量化 overhead 可能 > 收益

7.6 常见问题与排错

❌ NaN / 乱码输出

原因：INT4 量化溢出，scale 计算不正确

解决：增大 group_size；使用 per-token 代替 per-tensor；检查 outlier

⚠️ PPL 突然升高

原因：某些层的 KV 分布极端 outlier

解决：混合精度：前几层 FP16，后面层 INT4；或增大保护窗口

ℹ️ 内存未按预期减少

原因：Scale/zp 元数据开销；或框架未完全优化

解决：检查 scale 存储格式；使用 per-group 量化减少元数据

ℹ️ 量化后速度没提升

原因：Batch size 太小，计算而非带宽是瓶颈

解决：量化 KV 主要在大 batch / 长序列时提升明显；单请求短序列几乎无提升

7.7 完整量化实验代码示例

import torch
import time

def benchmark_kv_quantization(model, tokenizer, prompts, dtype_list):
    """对比不同 KV Cache 量化方案的质量和性能"""
    results = {}

    for dtype in dtype_list:
        # 构造输入
        inputs = tokenizer(prompts, return_tensors="pt", padding=True).to("cuda")

        # 设置 KV Cache 数据类型
        # vLLM 方式: kv_cache_dtype=dtype
        # 手动方式: 在生成时对 KV Cache 做量化

        torch.cuda.synchronize()
        start = time.perf_counter()

        # 生成
        outputs = model.generate(
            **inputs,
            max_new_tokens=256,
            do_sample=False,
        )

        torch.cuda.synchronize()
        elapsed = time.perf_counter() - start

        # 统计
        mem_used = torch.cuda.max_memory_allocated() / 1e9  # GB
        tokens_generated = outputs.shape[1] - inputs.input_ids.shape[1]
        throughput = tokens_generated / elapsed

        results[dtype] = {
            "memory_gb": mem_used,
            "time_s": elapsed,
            "tokens_per_sec": throughput,
        }
        print(f"[{dtype}] Mem={mem_used:.2f}GB | "
              f"Time={elapsed:.2f}s | "
              f"Throughput={throughput:.1f} tok/s")

    return results

# 运行对比
results = benchmark_kv_quantization(
    model, tokenizer,
    prompts=["Explain quantum computing in detail..."],
    dtype_list=["float16", "fp8", "int8"]
)

  1. 核心原理: 本文件系统讲解了KV Cache量化技术，这是长序列推理中以精度换空间的关键杠杆。KV Cache显存公式为 Mem=2×L×d×b×s×p，以LLaMA-70B（L=32, d=4096）为例，batch=32、seq=4096时FP16 KV Cache高达32GB。量化将精度从FP16降至INT8/INT4/INT2，分别实现2×/4×/8×压缩。文件详细介绍了量化基础（对称/非对称、粒度、动态/静态）、KV Cache特定方案（KIVI、KVQuant、Gear、QServe）、Per-Token vs Per-Channel粒度选择、量化Kernel实现等内容。核心发现是Key在channel维度有outlier（适合per-channel量化），Value在token维度有outlier（适合per-token量化）。
    关键要点: ① Decode阶段是Memory-bound的，降低KV Cache比特宽度直接降低HBM带宽需求；② KV Cache量化需在每个decode step实时量化/反量化，不同于权重的一次性量化；③ FP8是硬件原生最佳选择（H100支持），INT4是内存受限场景的利器；④ KIVI采用K-per-channel+V-per-token的混合策略实现2-bit量化。
    实际案例: 以LLaMA-70B为例，4096 tokens的FP16 KV Cache约7.5GB，INT4量化后降至约1.9GB，INT2进一步降至约0.94GB。KIVI在LLaMA-70B上实现2-bit KV Cache，MMLU精度损失<0.5%，支持4×更长上下文。QServe的W4A8KV4方案在L40S GPU上实现1.0-1.3×吞吐提升。
    关联知识: KV Cache量化与架构级压缩（GQA/MQA，前一章）可叠加使用。GQA-8+INT4可实现32×总压缩比。量化技术也可与下一章的驱逐/稀疏化策略组合，进一步优化长序列推理。
  2. 核心原理: 量化KV Cache的核心动机是解决推理阶段的内存瓶颈。Decode阶段是Memory-bound的——每个token需要从HBM读取全部KV Cache，降低比特宽度直接降低带宽需求并提升吞吐。具体来说，FP16→INT8带宽减半，FP16→INT4带宽降到1/4。KV Cache与权重量化不同：权重量化在prefill阶段一次性完成，而KV Cache量化需要在每个decode step对新产生的KV向量实时量化，并在attention计算时实时反量化，这对kernel提出了低延迟quantize/dequantize的要求。不同精度的权衡是：FP16(无损)→FP8/INT8(几乎无损，2×压缩)→INT4(轻微损失，4×压缩)→INT2(显著损失，8×压缩)。
    关键要点: ① KV Cache显存 = 2×L×d×b×s×p，随序列长度线性增长，是长上下文推理的首要瓶颈；② 量化带来的带宽节省直接转化为decode吞吐提升；③ KV Cache量化是实时的（每个decode step），不同于权重的一次性量化；④ FP8(E4M3)是当前硬件（H100）的最优无损方案，INT4是极端场景的实用选择。
    实际案例: 以LLaMA-70B（L=80, d=6144）为例，4096 tokens的FP16 KV Cache约7.5GB。INT4量化后降至约1.9GB，使得单张A100-80GB可以同时服务更多请求或支持更长上下文。在生产环境中，INT4量化使得原本需要多卡的部署方案可以缩减到单卡，显著降低推理成本。
    关联知识: 量化动机与前一章的GQA/MQA压缩动机相同——都是解决KV Cache的内存/带宽瓶颈。量化可与架构压缩叠加，如GQA-8(8×压缩)+INT4(4×压缩)=32×总压缩。量化还可与驱逐策略（下一章）互补：量化压缩每个token的存储，驱逐减少token数量。
  3. 核心原理: 量化基础包括四个核心概念：(1) 对称量化 vs 非对称量化：对称量化假设数据关于零点对称，公式为 q=round(x/scale)，scale=max|x|/(2^(b-1)-1)，零点zp=0；非对称量化更灵活，q=round((x-zp)/scale)，适合非对称分布的KV Cache。(2) 量化粒度：Per-Tensor（全局一个scale，最粗糙）、Per-Token（每个token一个scale，KV Cache常用）、Per-Channel（每个特征维度一个scale，精度最高）、Per-Group（每g个维度共享scale，group=128常用）。(3) 动态 vs 静态量化：KV Cache默认使用动态量化，因为KV向量在decode时动态生成，静态量化无法适应分布变化。(4) 量化误差度量：MSE、SNR、余弦相似度（CosSim>0.99通常质量损失可忽略）、最大绝对误差。
    关键要点: ① 对称量化计算简单但分布不对称时浪费表示范围，非对称量化更灵活但需存储零点；② Per-Token量化是KV Cache的常用方案，Per-Channel精度最高但实现复杂；③ KV Cache必须使用动态量化（运行时计算scale），因为分布随序列变化；④ CosSim>0.99是质量损失可接受的经验阈值。
    实际案例: 对称量化示例：scale=max|x|/127，将FP16值映射到[-128,127]的INT8范围。非对称量化示例：scale=(max-min)/255，zp=min，将值映射到[0,255]。在KV Cache中，Key的分布通常不对称（存在正值outlier），因此非对称量化更优。INT8量化后CosSim通常>0.999，INT4后>0.98。
    关联知识: 量化基础是理解后续KIVI、KVQuant等特定方案的前置知识。对称/非对称的选择直接影响量化精度，粒度选择（Per-Token vs Per-Channel）是第4节的核心议题。动态量化的实时性要求影响了第5节的Kernel实现设计。
  4. 核心原理: KV Cache特定量化方案针对Key和Value的不同分布特征设计了差异化策略。KIVI（2024）发现Key在channel维度有outlier，Value在token维度有outlier，因此采用K-per-channel + V-per-token的混合策略，并将最近w个token保留FP16不量化（保护最新上下文），实现2-bit量化且质量损失极小。KVQuant（2024）提出Pre-RoPE Key量化（在RoPE旋转前量化Key减少outlier）和Norm常量吸收技术，INT3即可达到接近无损。Gear（2024）采用低秩矩阵+量化残差的混合方案，对KV Cache做SVD分解，残差天然接近零非常适合低位量化。QServe（2024）提出W4A8KV4统一方案，系统优化GPU kernel实现端到端加速。
    关键要点: ① K对量化更敏感（outlier在channel维度），V更鲁棒，K8V4是常见最佳实践；② KIVI的"最近token保留FP16"策略保护了最新上下文质量；③ KVQuant的Pre-RoPE量化技术避免了RoPE旋转引入的outlier；④ Gear的低秩+残差方案在3.12×压缩比下perplexity损失<0.1。
    实际案例: KIVI在LLaMA-7B/13B/70B上实现2-bit KV Cache，MMLU精度损失<0.5%，支持4×更长上下文。KVQuant在LLaMA-70B上实现INT3 KV Cache接近无损质量。QServe-LLaMA-70B在L40S GPU上实现W4A8KV4端到端加速，KV Cache内存降低4×，质量损失<1%。
    关联知识: 这些方案与前一章的GQA/MQA架构压缩可叠加使用。KIVI和KVQuant的K/V差异化量化策略基于对分布特征的深入分析，这与量化基础（第2节）中的粒度选择直接相关。Gear的低秩方法与MLA的低秩压缩思想有异曲同工之处。
  5. 核心原理: Per-Token和Per-Channel是两种核心量化粒度，选择取决于数据的outlier分布特征。Per-Token量化：每个token的整个d维向量共享一组scale和zero_point，公式为 scale_t=max(abs(K_t))/127，参数量O(b×s)，适合序列间差异大的数据。Per-Channel量化：每个特征维度在所有token上共享参数，参数量O(d)，适合特征间差异大的数据。关键发现：Key矩阵在channel维度有显著outlier（某些channel值比其他大10-100×），因此Per-Channel量化更优；Value矩阵在token维度有outlier（某些token的V向量范数远大于平均值），因此Per-Token量化更优。这一观察是KIVI和KVQuant采用混合策略的核心依据。此外还有Activation-Aware量化思想：不仅考虑KV自身分布，还考虑与Query的注意力交互，高attention score的token应使用更高精度。
    关键要点: ① Key适合Per-Channel量化（outlier在channel维度），Value适合Per-Token量化（outlier在token维度）；② Per-Token是KV Cache的默认常用方案，实现简单且效果好；③ Per-Channel精度最高但可能忽略token间分布差异；④ Activation-Aware量化根据attention score分配精度，是更精细的策略。
    实际案例: KIVI采用K-per-channel+V-per-token混合策略，在INT2精度下实现极小质量损失。KVQuant验证了Per-Channel-Key + Per-Token-Value的组合比纯Per-Token方案精度提升0.5-1.0 perplexity。Activation-Aware方法中，KIVI的"最近token保留FP16"是其特例——最新上下文的attention score通常最高。
    关联知识: Per-Token vs Per-Channel的选择直接影响第3节中各种方案（KIVI、KVQuant等）的设计。这一粒度选择也与量化基础（第2节）中的Per-Tensor/Per-Group粒度形成完整的粒度谱系。在第5节的Kernel实现中，不同粒度需要不同的CUDA kernel设计来高效支持。
