MODULE 08 · KV-Cache系统专题

Prefill/Decode分离与PD架构

为什么同一块GPU不能高效同时做Prefill和Decode？如何通过分离架构将吞吐提升1.5-2倍？ Mooncake、Splitwise、DistServe如何设计KV Cache传输？生产级部署怎么做？

⚡ Prefill Pool 🔄 Decode Pool 📦 KV Cache Transfer 🧠 Conductor/Scheduler

1为什么要分离Prefill和Decode

通俗类比：就像印刷厂排版和印刷是两个工种——排版需要大量人力审稿校对，印刷只需机器高速运转。混在一起既浪费排版师的时间，也让印刷机空转等待。
常见误区：最常见的误区是认为Prefill和Decode只是"前后顺序"不同，实际上是计算模式完全不同的两种任务，强行混用会严重浪费GPU算力。
深入探讨：Prefill阶段是compute-bound，A100-80G上GPT-3级别模型Prefill吞吐可达2000 tokens/s；Decode阶段是memory-bound，同样硬件仅约30-50 tokens/s。两者计算密度相差40-60倍。

1.1 两阶段的根本差异

LLM推理分为两个截然不同的阶段，它们对GPU资源的需求完全相反：

维度 Prefill（预填充） Decode（解码）
输入 完整prompt (数百~数千tokens) 单个token（逐token生成）
计算模式 大矩阵乘法 GEMM 向量-矩阵乘 GEMV
瓶颈 Compute-bound（计算密集） Memory-bound（内存带宽密集）
GPU利用率 FLOPS利用率高 (40-60%) 带宽利用率高，FLOPS低 (5-10%)
Batch特征 单请求即可填满GPU 需要大批量才能提升利用率
延迟目标 TTFT (Time To First Token) TPOT (Time Per Output Token)
KV Cache 生成全部KV Cache 增量追加1层KV
💡 直觉层

Prefill像"一次性读完整本书做笔记"——需要大量算力。Decode像"一边翻笔记一边写下一个字"——需要快速翻阅（内存带宽）。让同一个工人交替做这两件事，效率必然低下。

⚙️ 机制层

Prefill阶段的GEMM运算充分利用GPU的Tensor Core（A100可达312 TFLOPS），算术强度(arithmetic intensity)极高。Decode阶段每个token只做GEMV，数据搬运量远大于计算量，瓶颈在HBM带宽（A100仅2TB/s）。两者交替执行时，Prefill的长计算会阻塞Decode的低延迟生成，造成Decode请求的TPOT抖动。

🔧 工程层

在vLLM的实际部署中，当一个大prompt的Prefill到来时，正在Decode的请求会被"抢占"，TPOT从稳定的30ms飙升到数百ms甚至秒级。这就是为什么SGLang引入了chunked prefill——把Prefill拆成小块与Decode交错执行。但chunked prefill只是妥协，PD分离才是根本解法。

🔥 核心洞察：Prefill和Decode对GPU资源的需求是"互补"的——Prefill吃算力不吃带宽，Decode吃带宽不吃算力。硬塞在同一块GPU上，必然有一半资源在闲置。分离后各自使用最适合的硬件，是资源利用率最大化的自然选择。

2Prefill/Decode分离架构

通俗类比：就像快餐店把备菜区和出餐区分开——备菜区在后厨大批量切配（Prefill），出餐区在前台快速组装打包（Decode），两个区域各用各的设备和人。
常见误区：分离架构不是简单的"切两刀"——需要精心设计KV Cache的传输机制、负载均衡策略和故障容错方案，否则传输开销可能抵消分离带来的收益。
深入探讨：典型PD分离架构中，Prefill集群使用高算力GPU（如A100/H100），Decode集群可使用带宽优化型GPU。KV Cache大小估算：2 × n_layers × n_heads × d_head × seq_len × dtype_bytes，LLaMA-70B单请求约400MB/1K tokens。

2.2 各组件职责

💡 直觉层

就像一个高效的出版社：编辑部(Prefill Pool)快速审稿做笔记，印刷厂(Decode Pool)快速排版出书，图书馆(KV Pool)管理所有笔记的存取。调度员(Conductor)决定每份稿件的流转路径。

⚙️ 机制层

API Gateway接收请求，判断是否需要Prefill（检查KV Cache是否已存在）。Conductor维护全局状态：各Pool的负载、KV Cache分布、请求队列。它决定：(1) 新请求发给哪个Prefill节点；(2) Prefill完成后KV传给哪个Decode节点；(3) 是否可以复用已有KV Cache。KV Pool可以是独立的分布式存储层，也可以是Decode节点的本地缓存。

🔧 工程层

实际部署中，Conductor通常是无状态的RPC服务，状态存在Redis/etcd中。Prefill Pool的GPU配置可能是4×A100(高算力)，Decode Pool可能是8×L4(高带宽比)。KV Pool可以利用CPU内存(便宜大容量)做二级缓存，GPU HBM做一级缓存。

3KV Cache传输是核心挑战

通俗类比：就像快递站分拣——包裹(KV Cache)本身又大又重，如果传送带太慢，前端打包再快后端也得干等。传输速度直接决定了整条流水线的效率。
常见误区：只关注计算优化而忽略KV Cache传输，是PD分离最大的性能陷阱。一个LLaMA-70B请求的KV Cache可达数GB，10Gbps网络传输需数秒，完全抵消Prefill的快速完成优势。
深入探讨：LLaMA-70B：80层×64头×128维×FP16，1K tokens KV Cache约800MB，4K tokens约3.2GB。100Gbps网络传输3.2GB需0.26秒，但序列化/反序列化开销可能再增加50%-100%。目标是将总传输延迟控制在Prefill时间的10%以内。

3.1 KV Cache的规模

KV Cache的传输是PD分离架构中最大的工程挑战。让我们量化这个问题：

模型 Layers Heads×Dim 4K Context KV大小 32K Context KV大小
LLaMA-7B 32 32×128 ~0.5 GB ~4 GB
LLaMA-13B 40 40×128 ~1 GB ~8 GB
LLaMA-70B 80 64×128 ~5 GB ~40 GB
GPT-4级 (推测) 120 128×128 ~15 GB ~120 GB

* 以FP16精度计算，公式：2 × 2 × num_layers × num_kv_heads × head_dim × seq_len × batch_size

3.2 传输时间分析

图4: 不同网络带宽下KV Cache传输时间（LLaMA-70B, 4K context, FP16）
⚠️ 关键约束：KV Cache传输时间必须远小于端到端生成时间才有意义。对于4K context、100 token输出的请求，总生成时间约3秒（Prefill 500ms + Decode 30ms×100）。如果KV传输需要400ms(100GbE)，那13%的延迟被浪费在网络传输上。如果用NVLink(16ms)，影响可以忽略不计。

3.3 KV Cache大小计算公式

// KV Cache大小（字节）
// = 2(K+V) × 2(FP16) × num_layers × num_kv_heads × head_dim × seq_len × batch_size

def kv_cache_size_bytes(model, seq_len, batch_size=1):
  # 以LLaMA-70B为例
  num_layers = 80
  num_kv_heads = 8  # GQA, 8个KV头
  head_dim = 128
  bytes_per_elem = 2  # FP16

  size = 2 * bytes_per_elem * num_layers * num_kv_heads * head_dim * seq_len * batch_size
  return size

# LLaMA-70B, seq_len=4096, batch=1
# = 2 × 2 × 80 × 8 × 128 × 4096 = 1,342,177,280 bytes ≈ 1.25 GB
# 注：实际可能因实现细节略有不同，通常引用~5GB包含更多开销

4Mooncake (月之暗面) 架构

通俗类比：就像城市的"中心厨房+配送站"模式——中心厨房集中备料(Prefill)，各社区配送站(Decode节点)通过高速物流网络获取成品，既保证出餐速度又能覆盖全城。
常见误区：Mooncake的分布式KV Cache池看似灵活，但Cache亲和性调度很关键——频繁跨节点读取KV Cache会导致延迟抖动，冷启动时Cache未命中率高会影响首token延迟。
深入探讨：Mooncake采用KV Cache为中心的调度，将Prefill/Decode集群解耦。核心指标：TTFT（首token延迟）和TPOT（每token输出延迟）。Mooncake将KV Cache存储在分布式内存池中，命中本地Cache时传输延迟<1ms，跨节点约2-5ms。月之暗面报告系统吞吐提升约2-3倍。

4.2 Conductor调度逻辑

# Mooncake Conductor 调度伪代码
class Conductor:
  def schedule(self, request):
    # Step 1: 查找KV Cache复用机会
    prefix_match = self.prefix_tree.find_longest_prefix(request.prompt)
    if prefix_match:
      # 找到前缀匹配 → 只需Prefill剩余部分
      remaining_tokens = request.prompt[prefix_match.length:]
      prefill_node = self.select_prefill_with_affinity(prefix_match.node)
    else:
      remaining_tokens = request.prompt
      prefill_node = self.select_least_loaded_prefill()

    # Step 2: 选择Decode节点（考虑KV Cache亲和性）
    decode_node = self.select_decode_node(
      prefer_with_kv_cache=prefix_match,
      load_balancing="least_connections"
    )

    # Step 3: 路由执行
    prefill_node.submit(remaining_tokens)
    decode_node.reserve_slot(request)
    return prefill_node, decode_node

5Splitwise (Microsoft) 架构

通俗类比：就像工厂的"原料预处理车间"和"组装车间"——预处理车间用大型设备批量加工原材料(Prefill)，组装车间用精密工具逐件装配(Decode)，各自用最适合的设备配置。
常见误区：Splitwise在同一物理节点上混部Prefill和Decode时，需要注意GPU SM资源隔离——如果Prefill任务占用过多SM，会导致Decode任务延迟飙升，影响输出token的稳定性。
深入探讨：Splitwise的核心创新是在同一集群内实现PD混部。通过SM级别的资源隔离（MPS/MIG），Prefill占用70% SM，Decode占用30% SM。在H100集群上，Splitwise相比vLLM基线提升吞吐约1.5-2.1倍，TTFT降低40%-60%。关键是利用Prefill的计算空隙为Decode服务。

5.1 设计理念

Splitwise将PD分离推进到了硬件选型层面——不仅软件分离，硬件配置也针对各阶段优化。

⚙️ 机制层

Prefill Machine配置高算力GPU（如H100 SXM），利用Tensor Core做大规模GEMM。Decode Machine配置高带宽GPU（如H100 NVL），利用高HBM带宽做大批量GEMV。两组机器通过高速网络（InfiniBand 400Gbps）连接。

5.2 调度算法

# Splitwise调度策略
class SplitwiseScheduler:
  def route_request(self, request):
    # 估算Prefill计算量
    prefill_flops = self.estimate_prefill_cost(request.num_input_tokens)

    # 检查Prefill池的当前负载
    prefill_loads = [p.current_flops_utilization for p in self.prefill_pool]
    best_prefill = self.prefill_pool[argmin(prefill_loads)]

    # 估算Decode资源需求
    decode_memory = self.estimate_decode_memory(
      request.num_input_tokens + request.max_output_tokens
    )

    # 检查Decode池容量
    best_decode = self.select_decode_with_capacity(decode_memory)

    return best_prefill, best_decode

6DistServe (北京大学)

通俗类比：就像医院的"分诊挂号"和"科室诊疗"——分诊台快速完成信息录入(Prefill)，各科室独立接诊(Decode)，通过合理分诊让每个科室的医生都不空闲也不过载。
常见误区：DistServe的干扰感知调度虽然精细，但调度器本身也有开销——如果请求到达速率过高，调度决策延迟可能成为瓶颈，需要预留10%-15%的CPU资源给调度逻辑。
深入探讨：DistServe提出干扰感知调度，核心公式：Decode延迟 = base_latency × (1 + α×prefill_interference)。其中α为干扰系数，通常0.3-0.8。DistServe通过预测干扰程度动态调整PD任务比例，在LLaMA-13B上实现了SLO达成率从78%提升到99%的显著改善。

6.1 Goodput优化目标

DistServe的核心创新是引入Goodput（有效吞吐）作为优化目标——不仅要跑得快，还要满足SLO。

⚙️ 机制层

Goodput = 满足SLO的请求数 / 总时间。DistServe的目标函数：
maximize Goodput = Σ[1(SLO_met_i)] / T_total
它不仅要最大化吞吐，还要最大化满足SLO约束（TTFT ≤ T_max, TPOT ≤ T_max）的请求比例。

6.2 KV Cache路由策略

# DistServe KV Cache路由策略
class DistServeRouter:
  def route(self, request):
    # 策略1: 最少KV传输
    if self.config.routing == "min_transfer":
      # 选择KV Cache传输距离最短的Decode节点
      return self.find_nearest_decode(prefill_node)

    # 策略2: 负载均衡
    elif self.config.routing == "load_balance":
      return self.find_least_loaded_decode()

    # 策略3: SLO感知
    elif self.config.routing == "slo_aware":
      # 估算请求的剩余SLO预算
      slo_budget = request.slo_deadline - time.now()
      # 如果SLO紧张，选最快Decode节点
      if slo_budget < self.transfer_estimate:
        return self.find_fastest_decode()
      return self.find_least_loaded_decode()

6.3 DistServe的关键贡献

  • Goodput指标：首次将SLO满足率纳入PD分离的优化目标
  • 解耦资源分配：Prefill和Decode池可以独立扩缩容
  • KV路由优化：考虑网络拓扑的KV Cache传输路径选择
  • 抢占策略：当SLO可能被违反时，优先处理高优先级请求

7Prefill-Decode调度算法

通俗类比：就像机场的跑道分配——起飞(Prefill)需要长时间占用跑道，降落(Decode)需要快速响应。调度员必须平衡两种需求，不能让飞机在跑道上排队太久。
常见误区：调度算法最易犯的错误是"短视"——只看当前负载不看预测趋势。当大批长文本请求涌入时，如果未预留足够Prefill资源，会导致后续请求排队时间指数级增长。
深入探讨：常用调度策略：FIFO（公平但延迟高）、最短作业优先SJF（降低平均延迟但可能饥饿）、抢占式优先级（保证SLO但开销大）。工程实践中，Decode请求通常设更高优先级，因为用户对输出中断更敏感。典型阈值：Prefill排队>2s时触发扩容，Decode延迟>100ms时触发抢占。

7.2 KV Cache亲和性调度

⚙️ 机制层

KV Cache亲和性调度的核心思想是：如果某个Decode节点已经缓存了请求的prefix KV Cache，优先将请求路由到该节点，避免重复传输。

实现方式：每个Decode节点维护一个本地KV索引（通常是prefix hash → 存储位置的映射）。Scheduler查询所有Decode节点的索引，选择匹配度最高的节点。匹配度 = 匹配的token数 × 节点可用容量的加权分数。

# KV亲和性调度伪代码
def select_decode_with_affinity(self, request, decode_nodes):
  scores = []
  for node in decode_nodes:
    # 查询本地KV缓存匹配
    match_len = node.kv_index.longest_prefix_match(request.prompt_hash)
    transfer_saved = match_len * self.kv_bytes_per_token

    # 综合评分 = 节省的传输量 × 可用容量
    affinity_score = transfer_saved / self.max_transfer_size
    capacity_score = node.available_slots / node.total_slots
    load_score = 1.0 - node.current_load

    total = (
      self.w_affinity * affinity_score +
      self.w_capacity * capacity_score +
      self.w_load * load_score
    )
    scores.append((node, total))

  return max(scores, key=lambda x: x[1])[0]

8KV Cache传输优化

通俗类比：就像搬家时用压缩打包——先用真空袋压缩衣物(KV Cache量化)，再用大货车一次运完(批量传输)，比一件件搬运快得多。压缩率和打包方式决定了搬家效率。
常见误区：过度压缩KV Cache会导致精度损失——INT4量化虽能将传输量减半，但在长文本生成任务中可能导致第50+token开始出现语义漂移，需要在压缩率和质量间找平衡点。
深入探讨：优化手段：1)KV Cache量化（FP16→INT8减少50%体积，INT4减少75%）；2)异步流水线传输（Prefill进行中即开始传输已计算层的KV）；3)差分传输（仅传输新增token的KV增量）；4)RDMA零拷贝（绕过CPU直接GPU-to-GPU）。综合优化可将有效传输延迟降低80%-90%。

8.2 KV Cache压缩传输

⚙️ 机制层

量化压缩：将KV Cache从FP16压缩到INT8甚至INT4，传输量减少50-75%。Decode端反量化恢复精度。
稀疏化：利用注意力稀疏性，只传输注意力权重较大的KV条目。通常保留top-k注意力头的KV。
选择性传输：研究表明不同层的KV重要性不同。浅层和深层的KV更关键，中间层可以省略或压缩。

# KV Cache压缩传输示例
def compress_kv_for_transfer(kv_cache, method="int8"):
  if method == "int8":
    # Per-channel量化
    scales = kv_cache.abs().max(dim=-1, keepdim=True) / 127.0
    quantized = (kv_cache / scales).to(torch.int8)
    return quantized, scales  # 传输量 = 原始的 ~50%

  elif method == "sparse_topk":
    # 只传输top-k重要的KV heads
    importance = kv_cache.abs().mean(dim=[0, 2, 3])  # 按head排序
    topk_indices = importance.topk(k=4).indices
    return kv_cache[:, topk_indices], topk_indices  # 传输量 = 原始的 50%

  elif method == "layer_drop":
    # 只传输关键层
    important_layers = [0, 1, 2, 39, 40, 41, 78, 79]  # 浅层+深层
    return kv_cache[important_layers]

8.4 RDMA与GPU Direct

🔧 工程层

传统TCP传输：GPU HBM → CPU内存 → 网卡发送 → 对端网卡 → CPU内存 → GPU HBM。经历4次内存拷贝。

RDMA (GPUDirect RDMA)：GPU HBM → 网卡直接发送 → 对端网卡 → GPU HBM。仅2次拷贝，CPU不参与数据搬运。

实测效果：在100Gbps网络下，RDMA可将KV传输延迟从400ms降到200ms（减少50%）。在400Gbps InfiniBand + GPUDirect下，LLaMA-70B 4K context的KV传输仅需100ms。

✅ 参考答案与详解
  1. 核心原理: Prefill和Decode在计算模式上存在根本性差异，这是分离的核心驱动力。Prefill阶段处理完整的prompt序列（数百至数千tokens），执行大规模矩阵乘法GEMM运算，属于Compute-bound类型，GPU算力利用率可达40%-60%，A100上吞吐约2000 tokens/s。而Decode阶段逐token生成输出，执行向量-矩阵乘GEMV，属于Memory-bound类型，GPU算力利用率仅5%-10%，同样硬件吞吐仅30-50 tokens/s。两者混合在同一GPU上执行时，Prefill的长计算会阻塞Decode的低延迟生成，导致TPOT（Time Per Output Token）从30ms飙升到数百ms甚至秒级。两者对GPU资源的需求是"互补"的——Prefill吃算力不吃带宽，Decode吃带宽不吃算力，硬塞在同一块GPU上必然有一半资源闲置。
    关键要点: ①Prefill是Compute-bound，Decode是Memory-bound，计算密度相差40-60倍；②混合执行导致Decode请求的TPOT严重抖动；③分离后综合GPU利用率从约40%提升至约85%；④吞吐可提升1.5-2倍，TTFT P99降低40%，TPOT抖动降低60%。
    实际案例: 在vLLM的实际部署中，当一个大prompt的Prefill到来时，正在Decode的请求会被"抢占"，TPOT从稳定的30ms飙升。SGLang引入了chunked prefill作为临时妥协方案（把Prefill拆成小块与Decode交错执行），但PD分离才是根本解法。例如在月之暗面(Kimi)的生产环境中，采用Mooncake架构分离Prefill和Decode后，系统吞吐提升约2-3倍。
    关联知识: 这一概念与SGLang的chunked prefill机制直接相关——chunked prefill是混合执行下的折中方案，而PD分离是根本解决方案。同时关联KV Cache传输问题，因为分离后KV Cache必须在Prefill和Decode集群间传输，这成为新的核心挑战。
  2. 核心原理: Prefill/Decode分离架构将LLM推理系统分为三个核心组件：Prefill Pool（预填充集群）、Decode Pool（解码集群）和KV Cache Pool/KV传输层。请求的完整生命周期为：用户请求→API Gateway（请求分类与路由）→Conductor/Scheduler（全局调度决策）→Prefill Pool执行完整prompt的KV Cache计算→KV Cache传输到Decode Pool→Decode Pool逐token生成并流式返回。每个组件的职责明确：API Gateway判断是否需要Prefill（检查KV Cache是否已存在），Conductor维护全局状态（各Pool负载、KV Cache分布、请求队列），决定新请求发给哪个Prefill节点、Prefill完成后KV传给哪个Decode节点、是否可复用已有KV Cache。Prefill Pool配置高算力GPU（如H100/A100），优化TTFT；Decode Pool配置高带宽GPU，优化吞吐。
    关键要点: ①架构分为API Gateway→Conductor→Prefill Pool→KV Pool→Decode Pool五个层次；②Conductor是无状态RPC服务，状态存储在Redis/etcd中；③KV Pool可利用CPU内存做二级缓存，GPU HBM做一级缓存；④总延迟=Prefill Time+KV Transfer Time+Decode Time×num_tokens，KV Transfer Time必须远小于Decode Time。
    实际案例: 以LLaMA-70B部署为例，Prefill集群使用4×A100-80G配置（高算力），Decode集群使用8×L4配置（高带宽比）。请求到达后，Prefill在500ms内完成4K context的KV Cache计算，通过100Gbps网络传输约5GB的KV Cache（约400ms），然后Decode集群以30ms/token的速度逐token生成。生产中通常Prefill Pool和Decode Pool独立扩缩容。
    关联知识: 与Mooncake的KVCache-centric设计和Splitwise的硬件感知PD分离直接相关。同时与Continuous Batching技术深度关联——Decode Pool内部使用continuous batching动态管理请求队列。vLLM v0.6+和SGLang均已支持disaggregated prefill的工程实现。
  3. 核心原理: KV Cache传输是PD分离架构中最大的工程挑战，因为KV Cache的数据量巨大。以LLaMA-70B为例：80层×64个KV头×128维×FP16，1K tokens的KV Cache约800MB，4K tokens约3.2-5GB，32K context约40GB。在不同网络带宽下，传输延迟差异巨大：10Gbps网络传输5GB需4.0秒，100Gbps需400ms，400Gbps需100ms，NVLink仅需16ms。而单token的Decode时间约30ms，如果KV传输耗时400ms（100GbE），则13%的延迟被浪费在网络传输上。此外，序列化/反序列化开销可能再增加50%-100%。KV Cache传输时间必须控制在Prefill时间的10%以内才有实际意义，因此网络带宽和传输优化策略至关重要。
    关键要点: ①KV Cache大小公式：2(K+V)×2(FP16)×num_layers×num_kv_heads×head_dim×seq_len×batch_size；②100Gbps网络传LLaMA-70B 4K context的KV约需400ms，NVLink仅16ms；③优化手段包括量化压缩（INT8减50%，INT4减75%）、流水线传输（层间重叠，节省30-60%等待）、RDMA零拷贝（减少50%传输延迟）、前缀复用（节省50%-80%传输量）；④目标是总传输延迟控制在端到端生成时间的10%以内。
    实际案例: LLaMA-70B处理4K context、100 token输出的请求，总生成时间约3秒（Prefill 500ms+Decode 30ms×100）。使用RDMA/GPUDirect RDMA（100Gbps），KV传输约200ms，占比约7%可接受。若用普通10Gbps TCP，传输需4秒完全抵消分离收益。Mooncake采用流水线KV传输——Prefill进行中即开始传输已计算层的KV，将有效传输延迟降低80%-90%。
    关联知识: 直接关联GQA（Grouped-Query Attention）技术——GQA通过减少KV头数（如LLaMA-70B从64头减至8个KV头）天然降低KV Cache传输量。同时关联NIXL传输后端标准化工作，以及INT8/INT4 KV Cache量化技术。
  4. 核心原理: Mooncake是月之暗面(Kimi)团队提出的生产级PD分离架构，其核心创新是以KVCache为中心（KVCache-centric）设计整个系统，而非以GPU为中心。架构由三部分组成：Conductor（中央调度器）维护全局视图（请求队列、KV分布、节点负载、前缀树），Prefill Pool使用4×A100-80G实例进行连续Prefill批处理，Decode Pool使用8×L4/S4实例进行大batch continuous batching。KV Cache存储在分布式内存池中（CPU Memory+GPU缓存），通过前缀索引（Hash Tree）和LRU驱逐机制管理。Conductor的核心调度逻辑：①查找KV Cache复用机会（通过全局前缀树找到最长前缀匹配）；②选择具有Cache亲和性的Prefill节点执行剩余tokens的Prefill；③选择Decode节点时考虑KV Cache亲和性（prefer_with_kv_cache）和负载均衡。Mooncake的性能结果：吞吐提升1.5倍，TTFT降低42%，长上下文吞吐提升2.5倍，P99 TPOT抖动消除。
    关键要点: ①以KV Cache为中心而非GPU为中心的架构设计；②Conductor维护全局前缀树实现跨请求的KV Cache复用；③KV Cache存储在分布式CPU内存池中，命中本地Cache延迟<1ms，跨节点约2-5ms；④支持流水线KV传输、增量KV传输、KV压缩传输三种优化。
    实际案例: 在月之暗面(Kimi)的生产环境中，用户进行多轮长对话时，共享的system prompt（约2K tokens）的KV Cache只计算一次，后续所有请求复用该KV Cache，Prefill仅计算增量部分。例如5轮对话场景下，前缀复用可节省约80%的KV传输量。Conductor的前缀树调度确保同一用户的连续请求路由到同一Decode节点，避免KV Cache重新计算。
    关联知识: Mooncake与Splitwise形成对比——Mooncake用同构硬件+软件分离，Splitwise用异构硬件分离。Mooncake的前缀树调度与SGLang的RadixAttention理念一致，但Mooncake将其扩展到跨节点的分布式场景。与DistServe的Goodput优化目标不同，Mooncake更关注长上下文和多轮对话场景的吞吐优化。
  5. 核心原理: Splitwise是Microsoft提出的PD分离架构，其核心创新是将分离推进到硬件选型层面——不仅软件分离，硬件配置也针对各阶段优化。Splitwise在同一集群内实现PD混部，通过SM级别的资源隔离（使用NVIDIA MPS或MIG技术），Prefill占用约70%的SM（如H100的132个SM中分配92个），Decode占用约30%的SM。设计上分为Prefill Machine和Decode Machine：Prefill Machine配置高算力GPU（如H100 SXM），利用Tensor Core做大规模GEMM运算；Decode Machine配置高带宽GPU（如H100 NVL），利用高HBM带宽做大批量GEMV运算。两组机器通过高速网络（InfiniBand 400Gbps）连接。调度算法分两步：①估算Prefill计算量（FLOPS），选择当前负载最低的Prefill节点；②估算Decode资源需求（KV Cache内存），选择有足够容量的Decode节点。KV Cache直接GPU-to-GPU传输，利用高带宽网络最小化传输延迟。
    关键要点: ①硬件级PD分离——Prefill用H100 SXM（高算力），Decode用H100 NVL（高带宽）；②SM级资源隔离实现同节点混部，Prefill占用70% SM，Decode占用30% SM；③KV Cache传输走GPU直传（不经过CPU内存池），延迟更低；④相比vLLM基线提升吞吐约1.5-2.1倍，TTFT降低40%-60%。
    实际案例: 在Azure的大规模推理集群中，Splitwise被用于部署GPT-4级别模型。Prefill Machine配置8×H100 SXM（高算力，312 TFLOPS），Decode Machine配置8×H100 NVL（高HBM带宽，3.35 TB/s）。对于输入4K tokens、输出200 tokens的典型API请求，Prefill阶段在200ms内完成（利用Tensor Core的高算力），KV Cache通过InfiniBand 400Gbps直传到Decode节点（约100ms），Decode阶段以每token 25ms的速度生成（利用高带宽）。相比同构部署，整体延迟降低35%。
    关联知识: Splitwise与Mooncake的核心区别在于：Splitwise用异构硬件（不同GPU型号），Mooncake用同构硬件+软件分离。Splitwise的KV传输走GPU直传更高效但灵活性较差，Mooncake的分布式KV Pool更灵活但传输开销更大。Splitwise与NVIDIA MPS/MIG技术直接关联，SM级隔离依赖这些底层GPU虚拟化能力。在工程选型上，Splitwise更适合大规模异构集群场景。
