1. KV Cache对集群调度的影响

通俗类比：KV Cache就像后厨的‘半成品’备料。调度系统就像餐厅大堂经理，他的决策（谁先上菜、拼桌）直接取决于后厨有多少备料以及备料速度。没有这个意识，经理会做出错误安排。
常见误区：易犯错误：调度器只看‘GPU计算力’，无视‘显存’。导致接到大量请求，但KV Cache迅速占满显存，后续请求无法生成任何Token，整个集群‘爆厨’。

1.1 KV Cache是推理服务的核心状态

关键认知: 在传统Web服务中，请求是无状态的——任何节点都可以处理任何请求。但LLM推理服务是有状态的：每个正在进行的请求都持有KV Cache，这是调度决策的根本差异。

1.2 调度决策必须考虑KV Cache位置

1.3 GPU内存容量决定并发数

GPU内存是LLM推理服务的硬约束。调度器必须精确计算每个请求的KV Cache内存需求:

KV Cache内存 = 2 × num_layers × num_heads × head_dim × seq_len × dtype_bytes

以LLaMA-70B (80GB A100) 为例:

# LLaMA-70B 参数
num_layers    = 80
num_kv_heads  = 8     # GQA
head_dim      = 128
dtype_bytes   = 2     # FP16

# 每个token的KV Cache大小
per_token_kv = 2 * 80 * 8 * 128 * 2 = 327,680 bytes ≈ 0.31 MB

# 模型权重占用 ~140GB (需要2×A100张量并行)
# 单卡模型权重: ~70GB
# 可用于KV Cache: 80GB - 70GB - 激活(~2GB) - 系统(~1GB) = ~7GB

# 最大并发序列长度(假设每请求4K tokens)
# 7GB / (0.31MB * 4096) ≈ 5.5 → 约5个并发请求
⚠️ 关键约束: 这就是为什么在LLM推理中，GPU内存容量直接决定了并发数。调度器如果不感知KV Cache的内存占用，就会导致OOM（Out of Memory）崩溃。

1.4 KV Cache生命周期与请求生命周期绑定

T0: 请求到达 — 调度器分配GPU slot，KV Cache开始分配内存
T1: Prefill阶段 — 一次性计算所有prompt token的KV Cache，写入GPU显存
T2: Decode阶段循环 — 每生成一个token，KV Cache增长一个位置
T3: 生成完成 — 输出完整，KV Cache可以释放
T4: 内存回收 — 释放的KV Cache内存归还给调度器池

2. 连续批处理(Continuous Batching)中的KV Cache

通俗类比：连续批处理像餐厅的‘流水线作业’：厨师A切菜，厨师B炒菜，厨师C摆盘。每个‘请求’(菜)在不同阶段被不同的资源处理。KV Cache是那盘菜的‘状态’，必须精准传递给下一个厨师。
常见误区：易犯错误：在连续批处理中，错误地为已完成生成的请求保留其全部KV Cache，导致显存无法释放给新请求，批处理“队列”无法持续流动，吞吐量下降。
深入探讨：实现细节：调度器维护两个队列：等待队列（Waiting）和运行队列（Running）。当一个请求完成时，其KV Cache显存被立即回收，并从运行队列移除。新请求从等待队列中以最小KV Cache开销加入运行队列。

2.2 运行时添加/移除请求的机制

Continuous batching的核心是每个decode iteration都可以调整batch组成:

class ContinuousBatchScheduler:
    def __init__(self, max_batch_size, gpu_memory_pool):
        self.running_requests = []      # 当前正在decode的请求
        self.waiting_queue = []         # 等待接入的请求
        self.max_batch_size = max_batch_size
        self.memory_pool = gpu_memory_pool
        self.kv_cache_manager = KVCacheManager(gpu_memory_pool)

    def schedule_iteration(self):
        """每个decode iteration调用一次"""

        # Step 1: 检查是否有请求完成 → 释放KV Cache
        completed = []
        for req in self.running_requests:
            if req.is_finished():
                completed.append(req)
        for req in completed:
            self.running_requests.remove(req)
            self.kv_cache_manager.free(req.kv_cache_blocks)  # 释放内存

        # Step 2: 从等待队列接入新请求
        while (self.waiting_queue and
               len(self.running_requests) < self.max_batch_size and
               self.memory_pool.available() > self.estimate_prefill_memory()):

            new_req = self.waiting_queue.pop(0)

            # 分配KV Cache内存 (PagedAttention: 分配物理blocks)
            new_req.kv_cache_blocks = self.kv_cache_manager.allocate(
                num_tokens=new_req.prompt_length
            )

            # 执行prefill: 一次性计算所有prompt token的KV
            self.prefill(new_req)

            self.running_requests.append(new_req)

3. KV Cache感知的调度策略

通俗类比：这就像餐厅经理的‘智能点菜系统’。系统不仅看厨师忙不忙，还看这道‘菜’(请求)需要多少‘备料’(KV Cache)。优先安排‘小份菜’(短序列)可以快速腾出备料台。
常见误区：易犯错误：采用简单的先来先服务（FCFS）。一个需要长上下文的‘大订单’会立即占据大部分显存，阻塞后面多个‘小订单’，导致整体吞吐量和平均延迟恶化。
深入探讨：经典策略：最短作业优先（SJF）或最短剩余时间优先（SRTF）。在KV Cache语境下，可近似为‘剩余Token数最少优先’。更先进的策略会结合‘当前已使用KV Cache大小’和‘剩余Token数’来估算总显存占用时间。
优先级计算：
P = f(剩余Token数, 当前KV长度)
例如: P = (剩余Token) / (1 + 当前KV长度)

3.1 Cache-Aware Routing

通俗类比：Cache-aware路由像手机的"最近通话"：打给同一个人时直接从通讯录缓存找，而不是重新搜索。
核心思想: 如果用户Alice的上一轮对话的KV Cache还在Node A上，那么Alice的新请求应该优先路由到Node A，而不是随机分配。这可以避免重新计算prefill，将首token延迟(TTFT)从数百毫秒降到个位数毫秒。 
class CacheAwareRouter:
    def __init__(self):
        # 用户 → 节点映射表 (最近KV Cache所在节点)
        self.cache_location = {}  # user_id → node_id
        # 节点 → KV Cache信息
        self.node_kv_status = {}  # node_id → {cache_hit_rate, memory_usage, ...}

    def route(self, request):
        user_id = request.user_id
        prompt_hash = hash_prefix(request.prompt)  # prompt前缀哈希

        # 策略1: 精确cache匹配 — 同一用户、同一前缀
        if user_id in self.cache_location:
            node_id = self.cache_location[user_id]
            if self.node_kv_status[node_id].has_prefix(prompt_hash):
                # Cache命中! 直接路由到该节点
                return node_id

        # 策略2: 前缀匹配 — 不同用户但共享系统prompt
        for node_id, status in self.node_kv_status.items():
            if status.has_prefix(prompt_hash):
                if status.memory_usage < 0.85:  # 内存有余量
                    return node_id

        # 策略3: 降级到负载均衡 — 选择内存最空闲的节点
        return min(self.node_kv_status.items(),
                   key=lambda x: x[1].memory_usage)[0]

4. 抢占与优先级

通俗类比：抢占就像餐厅给VIP客人‘加塞’，让他的菜‘插队’上。这要求厨师能立即停止手头工作，把‘半成品’(KV Cache)妥善存到冰箱(换出)，然后处理VIP的菜，之后再恢复(换入)。
常见误区：易犯错误：抢占频率过高。每次抢占（换入换出）都涉及大量显存拷贝（CPU<->GPU），开销巨大。频繁抢占会导致GPU大部分时间在‘搬运’而非‘计算’，性能断崖式下跌。
深入探讨：技术实现：抢占通常通过Swap（换出到CPU内存）或Recomputation（丢弃后重新计算）实现。Swap的开销是显存带宽，约数百GB/s。一个70B模型的KV Cache，换入换出可能耗时100ms以上，相当于生成数十个Token的时间。

4.1 Preemption策略: Swap vs Recomputation

通俗类比：抢占就像急诊分诊：重症患者(高优先级)可以插队，轻症患者被暂时移出诊室(swap到CPU)。

当GPU内存不足且有高优先级请求到达时，调度器需要抢占低优先级请求的KV Cache。两种主要策略:

class PriorityScheduler:
    def __init__(self):
        self.priority_queues = {
            0: deque(),  # P0: 实时
            1: deque(),  # P1: 标准
            2: deque(),  # P2: 批量
            3: deque(),  # P3: 后台
        }
        self.running = SortedList(key=lambda r: r.priority)

    def on_new_request(self, request):
        """新请求到达时的调度决策"""
        # 首先尝试直接接入
        if self.can_admit(request):
            self.admit(request)
            return

        # 内存不足 — 尝试抢占更低优先级的请求
        victims = self.find_preemption_victims(request.priority)
        if victims and self.can_preempt_and_admit(victims, request):
            for victim in victims:
                self.preempt(victim)  # swap或recompute
            self.admit(request)
            return

        # 无法抢占 — 排队等待
        self.priority_queues[request.priority].append(request)

    def preempt(self, victim):
        """选择抢占策略"""
        if self.should_swap(victim):
            self.swap_to_cpu(victim)     # KV Cache → CPU内存
            victim.state = SWAPPED
        else:
            self.recompute_later(victim)  # 丢弃KV Cache
            victim.state = PENDING_RECOMPUTE

    def should_swap(self, request):
        """根据KV Cache大小和PCIe带宽决定swap还是recompute"""
        kv_size_gb = request.estimate_kv_size() / 1e9
        swap_time_ms = kv_size_gb * 1000 / PCIE_BANDWIDTH_GBPS  # ~25 GB/s
        recompute_time_ms = request.estimate_prefill_time()

        # 如果swap比recompute快，并且CPU有足够内存
        return (swap_time_ms < recompute_time_ms and
                self.cpu_memory_available() > kv_size_gb)

5. 内存预算管理

通俗类比：管理GPU显存就像管理你的‘钱包’。KV Cache是‘硬性开销’，必须预留。剩余的钱(显存)才能用来‘招待新客人’(加载新请求)。预算做不好，就会‘破产’(OOM)。
常见误区：易犯错误：为每个请求静态分配其理论最大长度的KV Cache空间。对于短请求，这会造成巨大的内存浪费，严重降低集群整体吞吐量。
深入探讨：动态管理策略：采用‘池化’或‘按需分配’。例如，为所有运行中的请求设置一个总KV Cache池，每个请求按需增长。常用算法是‘预留-分配-释放’。一个实践公式：最大并发数 ≈ (GPU显存 - 模型权重) / (单请求平均KV Cache大小 * 安全系数)。

class AdmissionController:
    def __init__(self, gpu_memory_pool, config):
        self.pool = gpu_memory_pool
        self.high_watermark = config.get('kv_high_watermark', 0.95)
        self.low_watermark = config.get('kv_low_watermark', 0.80)
        self.max_seq_len = config.get('max_sequence_length', 32768)

    def can_admit(self, request) -> bool:
        """判断是否可以接受新请求"""
        usage = self.pool.usage_ratio()

        # 已超过high watermark — 拒绝所有新请求
        if usage > self.high_watermark:
            return False

        # 预估该请求的KV Cache峰值内存
        peak_kv = self.estimate_peak_kv(request)

        # 检查加上新请求后是否超过限制
        if usage + peak_kv / self.pool.total_size > self.high_watermark:
            # 尝试抢占低优先级请求
            if not self.can_preempt_to_fit(peak_kv):
                return False

        # 检查单请求最大长度限制
        if request.estimated_output_len > self.max_seq_len:
            return False

        return True

    def estimate_peak_kv(self, request):
        """估算请求的KV Cache峰值内存"""
        total_tokens = request.prompt_len + request.estimated_output_len
        return (2 * num_layers * num_kv_heads * head_dim *
                total_tokens * dtype_bytes)

class KVCacheAwareLoadBalancer:
    def __init__(self, nodes):
        self.nodes = nodes
        self.cache_index = {}  # prefix_hash → {node_id: last_seen_time}

    def select_node(self, request):
        """KV Cache感知的节点选择"""
        candidates = []

        for node in self.nodes:
            score = self.compute_score(node, request)
            candidates.append((score, node))

        # 选择综合得分最高的节点
        candidates.sort(reverse=True, key=lambda x: x[0])
        return candidates[0][1]

    def compute_score(self, node, request):
        """计算节点得分（越高越好）"""
        # Factor 1: KV Cache命中率 (权重: 0.4)
        cache_score = 0
        prefix_hash = request.prefix_hash
        if prefix_hash in node.kv_cache:
            cache_score = 1.0  # 完美命中
        elif self.partial_prefix_match(node, prefix_hash):
            cache_score = 0.5  # 部分命中

        # Factor 2: GPU内存余量 (权重: 0.3)
        memory_score = 1.0 - node.kv_cache_usage_ratio

        # Factor 3: 待处理请求数 (权重: 0.2)
        queue_score = 1.0 / (1 + node.pending_requests)

        # Factor 4: 当前batch大小 (权重: 0.1)
        batch_score = 1.0 - (node.current_batch_size / node.max_batch_size)

        return (0.4 * cache_score +
                0.3 * memory_score +
                0.2 * queue_score +
                0.1 * batch_score)

    def get_metrics(self):
        """收集所有节点的指标"""
        return {
            node.id: {
                'kv_cache_usage': node.kv_cache_usage_ratio,
                'kv_cache_hit_rate': node.cache_hit_rate_1min,
                'pending_requests': node.pending_requests,
                'active_sequences': node.current_batch_size,
                'gpu_utilization': node.gpu_compute_util,
                'ttft_p99': node.ttft_p99_ms,
            }
            for node in self.nodes
        }

# vLLM Scheduler 核心逻辑 (简化)
class Scheduler:
    def __init__(self, scheduler_config, cache_config):
        self.block_manager = BlockManager(cache_config)  # KV Cache block管理
        self.waiting = deque()     # 等待prefill的请求
        self.running = deque()     # 正在decode的请求
        self.swapped = deque()     # 被swap到CPU的请求

    def schedule(self) -> SchedulerOutput:
        """每个iteration调用一次，返回本批次要执行的请求"""

        scheduled_prefills = []
        scheduled_decodes = []

        # Phase 1: 优先处理decode请求（维持用户体验）
        while self.running:
            req = self.running[0]
            if not self.block_manager.can_allocate(req):
                # 内存不足 → 抢占
                if not self._preempt(req):
                    break  # 无法抢占更多
            scheduled_decodes.append(req)
            self.running.popleft()

        # Phase 2: 尝试接入新的prefill请求
        while self.waiting:
            req = self.waiting[0]
            # 检查是否能分配KV Cache blocks
            if not self.block_manager.can_append_slot(req):
                break  # 内存不足，停止接入

            # 预算检查
            num_new_tokens = self._get_num_new_tokens(req)
            if num_new_tokens > self._get_available_budget():
                break

            self.block_manager.allocate(req)
            scheduled_prefills.append(req)
            self.waiting.popleft()

        # Phase 3: 尝试恢复swap出去的请求
        while self.swapped:
            req = self.swapped[0]
            if not self.block_manager.can_swap_in(req):
                break
            self.block_manager.swap_in(req)
            scheduled_decodes.append(req)
            self.swapped.popleft()

        return SchedulerOutput(
            scheduled_prefill=scheduled_prefills,
            scheduled_decode=scheduled_decodes,
        )

    def _preempt(self, victim):
        """vLLM默认使用recomputation策略"""
        if self.scheduler_config.preemption_mode == 'swap':
            self.block_manager.swap_out(victim)
            self.swapped.append(victim)
        else:
            # Recomputation: 释放blocks，稍后重新prefill
            self.block_manager.free(victim)
            victim.status = WAITING
            self.waiting.appendleft(victim)  # 高优先级重新排队
        self.running.remove(victim)
        return True

# SGLang RadixAttention 调度核心 (概念简化)
class SGLangScheduler:
    def __init__(self):
        self.tree_cache = RadixTree()       # 前缀树管理KV Cache
        self.token_to_kv_pool = []          # KV Cache内存池

    def schedule(self):
        # 1. 前缀匹配: 在Radix Tree中查找最长匹配前缀
        for req in self.waiting:
            prefix_indices = self.tree_cache.match_prefix(req.input_ids)
            # prefix_indices包含已缓存的KV Cache block引用
            req.prefix_len = len(prefix_indices)

            # 只需要计算prefix之后的token
            remaining = req.input_ids[req.prefix_len:]
            req.needed_blocks = self.estimate_blocks(remaining)

        # 2. 按前缀长度排序（优先处理共享前缀的请求）
        self.waiting.sort(key=lambda r: -r.prefix_len)

        # 3. 调度决策
        scheduled = []
        for req in self.waiting:
            if self.can_schedule(req):
                self.tree_cache.insert(req.input_ids, req.kv_indices)
                scheduled.append(req)

        return scheduled

class RadixTree:
    """Radix Tree管理KV Cache前缀共享"""
    def __init__(self):
        self.root = RadixNode()

    def match_prefix(self, token_ids):
        """查找最长匹配前缀，返回KV Cache索引"""
        node = self.root
        match_len = 0
        for i, token_id in enumerate(token_ids):
            if token_id in node.children:
                node = node.children[token_id]
                match_len = i + 1
            else:
                break
        return node.kv_indices[:match_len]

    def insert(self, token_ids, kv_indices):
        """插入新的前缀和对应的KV Cache索引"""
        node = self.root
        for token_id, kv_idx in zip(token_ids, kv_indices):
            if token_id not in node.children:
                node.children[token_id] = RadixNode()
            node = node.children[token_id]
        node.kv_indices = kv_indices
        node.ref_count += 1  # 引用计数，用于回收

✅ 参考答案与详解
  1. 核心原理: 在传统Web服务中，请求是无状态的——任何节点都可以处理任何请求，迁移成本几乎为零。但LLM推理服务是有状态的：每个正在进行的请求都持有KV Cache，这是调度决策的根本差异。KV Cache对集群调度的影响体现在五个维度：①请求状态：KV Cache绑定到特定GPU，不能随意迁移；②资源模型：GPU内存（固定且昂贵），KV Cache占用显著（LLaMA-70B单请求约2.3GB/4K tokens）；③负载单位：不再是简单的请求数/QPS，而是Token数×序列长度+KV Cache大小；④迁移成本：KV Cache重新计算代价高昂；⑤并发模型：GPU批处理+连续调度，而非进程/线程池。调度器如果不感知KV Cache的内存占用，就会导致OOM崩溃——一个80G显存的GPU，即使有空间，也可能因KV Cache碎片无法调度新请求。
    关键要点: ①KV Cache是有状态的，绑定到特定GPU，迁移需重计算；②KV Cache大小公式：2×num_layers×num_kv_heads×head_dim×seq_len×dtype_bytes；③GPU内存容量直接决定并发数（LLaMA-70B单卡约5个并发/4K context）；④调度器必须预估每个请求的KV Cache开销；⑤KV Cache生命周期与请求生命周期完全绑定（T0到达→T1 Prefill→T2 Decode循环→T3完成→T4回收）。
    实际案例: 以LLaMA-70B部署在A100-80G上为例：模型权重占用约70GB，剩余约7GB可用于KV Cache。每个token的KV Cache约0.31MB，4K context的单请求需约1.25GB。因此最大并发约5个请求。调度器维护KV Cache使用率监控，当使用率超过95%（high watermark）时拒绝新请求，80%-95%之间正常接收，低于80%时积极接收。如果不做这种准入控制，第6个请求会导致GPU OOM。
    关联知识: 与PagedAttention技术直接相关——PagedAttention通过分页管理KV Cache（类似OS虚拟内存），解决了内存碎片问题，提高KV Cache内存利用率。同时关联Continuous Batching——每个decode iteration调度器需检查是否有请求完成并释放KV Cache，然后从等待队列接入新请求。
  2. 核心原理: 连续批处理(Continuous Batching)是LLM推理的核心调度技术，其与KV Cache的交互是性能关键。传统Static Batching中，一个batch必须等所有请求全部完成才能释放资源——短请求完成后GPU空闲等待长请求，新请求必须等整个batch完成，GPU利用率仅30-50%。Continuous Batching的核心是每个decode iteration都可以调整batch组成：①检查是否有请求完成→释放其KV Cache；②从等待队列接入新请求→分配KV Cache内存并执行prefill；③为所有running请求执行一步decode。调度器维护三个队列：waiting（等待prefill）、running（正在decode）、swapped（被swap到CPU的请求）。KV Cache在PagedAttention体系下的管理类似OS虚拟内存：分配(Allocate)——新请求prefill完成后从free block pool分配物理block；增长(Grow)——每个decode step如果当前block已满则分配新block；释放(Free)——请求完成或被抢占时归还physical block。错误地为已完成请求保留其全部KV Cache会导致显存无法释放，批处理队列无法持续流动。
    关键要点: ①每个iteration独立调度决策，请求完成即释放KV Cache，新请求立即填入；②调度器维护waiting/running/swapped三个队列；③KV Cache操作包括Allocate/Grow/Free/Swap Out/Swap In五种；④与PagedAttention协作，logical block→physical block映射支持Copy-on-Write共享前缀；⑤GPU利用率从Static Batching的30-50%提升至Continuous Batching的70-95%。
    实际案例: 在vLLM的实现中，Scheduler.schedule()每个iteration调用一次。Phase 1优先处理decode请求（维持用户体验），Phase 2尝试接入新prefill请求（检查block_manager.can_append_slot），Phase 3尝试恢复swap出去的请求。例如：当前running队列有3个请求，其中请求A在iteration 5完成，其KV Cache（占用4个physical block）立即归还给free pool。此时waiting队列中有请求D，调度器为其分配2个block并执行prefill，请求D加入running队列。整个过程对用户透明，请求D无需等待其他请求完成。
    关联知识: 与SGLang的RadixAttention深度关联——SGLang在Continuous Batching基础上引入前缀树管理KV Cache，自动发现可共享前缀的请求，共享前缀的请求可指向相同物理block（Copy-on-Write）。同时关联PagedAttention的block table映射机制，以及Swap In/Out策略。
  3. 核心原理: KV Cache感知的调度策略是将KV Cache的状态纳入调度决策的核心维度，而非仅看GPU计算负载。其核心思想包含三个方面：①Cache-Aware Routing——如果用户Alice的上一轮对话的KV Cache还在Node A上，新请求应优先路由到Node A，避免重新计算prefill，将TTFT从数百毫秒降到个位数毫秒；②Prefix-Aware Scheduling——大量请求共享相同system prompt，前缀感知调度可以将共享system prompt的KV Cache只计算一次，所有请求共享同一份物理KV blocks（引用计数管理），3个请求共享2048 token system prompt时，prefill从8704 tokens降到2944 tokens，加速2.96倍；③Cache Affinity与实现方案——一致性哈希（user_id→hash→节点）、前缀路由表（维护prefix_hash→node[]映射）、LRU缓存路由（每个节点维护最近KV Cache的key）、RADIX Attention（用Radix Tree管理KV Cache前缀）。简单先来先服务(FCFS)会导致长上下文大订单占据大部分显存，阻塞后面多个小订单，整体吞吐和平均延迟恶化。
    关键要点: ①Cache-Aware Routing通过维护cache_location映射表实现session亲和；②Prefix-Aware Scheduling利用前缀共享大幅减少重复计算和KV传输；③实现方案从简单到复杂：一致性哈希→LRU缓存→前缀路由表→RADIX Attention；④调度评分综合考虑cache命中(0.4权重)+内存余量(0.3)+队列深度(0.2)+batch大小(0.1)；⑤SGLang的Radix Tree自动发现可共享前缀，是最先进的前缀感知方案。
    实际案例: 以ChatGPT类服务为例，所有请求共享相同的system prompt（约2K tokens）。在Prefix-Aware调度下，system prompt的KV Cache只计算一次，后续请求（如Alice问"今天天气"，Bob问"帮我写代码"）仅需prefill各自的user prompt部分（分别256和512 tokens）。无前缀共享时总prefill量为3×(2048+user_tokens)=8704 tokens（约870ms），有前缀共享时仅2048+256+512=2816 tokens（约282ms），加速3.1倍。同时Cache-Aware Routing确保Alice的第二轮对话路由到同一节点，复用已有KV Cache，TTFT从850ms降到12ms。
    关联知识: 与SGLang的RadixAttention直接对应——Radix Tree管理KV Cache前缀，支持自动前缀发现和共享。与Mooncake的Conductor前缀树调度理念一致但实现方式不同。同时关联vLLM的prefix caching功能（--enable-prefix-caching参数）。这些调度策略的量化指标：KV Cache命中率目标>60%（多轮对话场景），目标告警阈值<30%时需调优路由策略。
  4. 核心原理: 抢占(Preemption)是在GPU内存不足且有高优先级请求到达时，调度器强制暂停低优先级请求以腾出资源的机制。两种主要策略：①Swap（换出）——将低优先级请求的KV Cache通过PCIe复制到CPU内存，释放GPU内存服务高优先级请求，高优先级完成后将KV Cache换回GPU继续decode（无需重计算）。Swap时间=KV Cache大小(GB)÷PCIe带宽(GB/s)，PCIe4约25GB/s，LLaMA-7B 4K context的KV约0.5GB换出需20ms，LLaMA-70B约5GB需200ms。②Recomputation（重算）——直接丢弃KV Cache释放GPU内存，高优先级完成后重新为被抢占请求执行prefill。Recomputation时间与seq_len成正比，LLaMA-7B 4K context约50ms，LLaMA-70B约200ms。选择策略的依据：如果swap时间关键要点: ①Swap保留状态恢复快但需要CPU内存和PCIe带宽；②Recompute不需额外内存但消耗GPU计算资源；③选择依据：swap_time vs recompute_time的量化比较；④LLaMA-7B推荐Swap（2.5x快），LLaMA-400B推荐Recompute（1.4x快）；⑤抢占频率过高（>10/min）是告警信号，需调优优先级配置或扩容。
    实际案例: 以混合SLA服务为例：实时对话（P0，TTFT<100ms）和文档批处理（P2，TTFT<5s）共享同一GPU集群。当P0请求到达但GPU内存不足时，调度器选择KV Cache最小的P2请求作为victim，将其5GB KV Cache通过PCIe swap到CPU内存（约200ms），然后为P0请求分配GPU内存并执行prefill。P0请求完成后，调度器将P2的KV Cache从CPU换回GPU（200ms），P2继续decode。如果victim的KV Cache很小（如LLaMA-7B的0.5GB，swap仅20ms），swap比重算（50ms）快2.5倍，首选swap。
    关联知识: 与vLLM的preemption_mode配置直接对应（swap或recompute）。同时关联Priority Scheduler的设计——多级优先级队列+抢占决策引擎。Swap操作涉及CPU-GPU间PCIe传输，与KV Cache传输优化（RDMA/GPUDirect）技术同源但场景不同。在SGLang中，Radix Cache天然支持部分swap——已共享前缀的KV Cache不需要swap，仅需swap增量部分。
  5. 核心原理: 内存预算管理是确保GPU显存不被KV Cache耗尽（OOM）的核心机制。GPU显存分为四个区域：模型权重（固定，约占40-50%）、KV Cache池（动态，约占30-35%）、激活值/系统开销（预留，约10%）、空闲缓冲（约10-15%）。最大并发数的计算公式：最大并发数≈(GPU显存-模型权重-激活-系统开销)÷(单请求平均KV Cache大小×安全系数)。以LLaMA-70B在A100-80G为例：模型权重约40GB（张量并行后单卡），KV Cache可用约25GB，激活约5GB，系统约2GB，空闲约8GB。单请求4K context的KV约1.25GB，理论最大并发约20个。但需设置high watermark（95%=23.75GB）和low watermark（80%=20GB）的内存水位控制：超过high watermark触发抢占或拒绝新请求，low~high之间正常运行，低于low时积极接受新请求。内存管理策略分三种：静态预分配（按最大序列长度分配固定空间，简单但浪费）、动态分页（类似OS内存分页，按块分配，如vLLM的PagedAttention）、层级管理（热数据在GPU，冷数据在CPU/磁盘）。长时间运行还会遇到内存碎片问题——PagedAttention缓解但不能完全消除。
    关键要点: ①GPU显存预算公式：模型权重+KV Cache池+激活+系统+空闲；②最大并发数≈可用KV内存÷单请求KV峰值；③High/Low Watermark控制（95%/80%）实现平滑准入控制；④PagedAttention分页管理减少碎片；⑤Admission Controller预估每个请求的peak KV（prompt_len+estimated_output_len），加上新请求后不超过high watermark才准入。
    实际案例: 部署LLaMA-70B在2×A100-80G（张量并行）上：单卡模型权重约70GB/2=35GB，可用KV Cache约80-35-5-2=38GB。设置high watermark=95%（36.1GB），low watermark=80%（30.4GB）。当前已用16.25GB（65%），系统正常运行。新请求到达，预估peak KV=1.25GB（4K input+4K output），加上后总使用17.5GB/38GB=46%，远低于high watermark，准入。若连续接入15个长上下文请求（每个32K context，peak KV约10GB），使用率达95%，此时新请求被拒绝，同时调度器尝试抢占最不活跃的低优先级请求。如果使用静态预分配（每个请求分配max_seq_len=32K的KV空间），则15个请求就需150GB，远超38GB容量，而动态分页下实际使用量可能仅30GB。
    关联知识: 与PagedAttention技术直接对应——vLLM的BlockManager管理GPU物理block的分配和释放，block table维护logical→physical映射。同时关联Swap In/Out机制——当内存紧张时可将冷数据swap到CPU内存（PCIe4约25GB/s）。碎片整理策略包括Block Compaction（合并半满block）和Defragmentation（重新排列活跃block）。FP8 KV Cache（--kv-cache-dtype=fp8）可将KV Cache体积减半，H100及以上GPU推荐使用。
