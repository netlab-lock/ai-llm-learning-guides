01 内存预算规划

通俗类比：这就像在餐厅开业前，规划好厨房要买多大的冰箱（GPU显存）、留多少空间给食材（模型权重）和临时碗碟（KV Cache）。规划错了，要么冰箱塞不下，要么放不进更多客人（请求）。
常见误区：最大陷阱是只计算模型和KV Cache，忽略系统碎片、内存对齐等“运营杂费”。实际可用显存永远比标称值小，建议预留10%-15%的冗余空间。
深入探讨：核心公式：`可用KV缓存 = 总显存 - 模型显存 - 系统开销`。例如，70B模型（FP16约140GB），在80GB A100上，单卡无法加载。必须使用模型并行或量化，再为KV Cache计算预算。量化到INT4后模型约35GB，则KV缓存空间约为45GB。

 │     15-30GB         ~5GB             剩余所有空间 │
 └──────────────────────────────────────────────────────────────────────────┘

KV Cache调优的第一步是理解GPU内存预算。所有优化策略都建立在精确的内存计算之上。

GPU总内存分配公式

📐 GPU内存分配等式
Total_GPU_Mem = Model_Weights + KV_Cache + Activations + System_Overhead

其中：
  Model_Weights = param_count × bytes_per_param
  KV_Cache = 2 × n_layers × n_heads × d_head × seq_len × batch_size × dtype_bytes
  Activations ≈ 取决于batch_size和seq_len，通常 1-4 GB
  System_Overhead ≈ CUDA context + cuDNN workspace ≈ 0.5-1.5 GB

模型权重占用计算

模型 参数量 FP16 权重 INT8 权重 INT4 权重

LLaMA-13B13B26 GB13 GB6.5 GB

KV Cache可用内存推导

🔢 KV Cache可用内存
KV_Cache_Available = Total_GPU_Mem × util_ratio - Model_Weights - Activations - Overhead

// util_ratio: GPU内存利用率，通常 0.85-0.95
// 保守估计用 0.90，激进用 0.95

A100-80GB 典型部署内存分配图

最大并发数与序列长度计算

📊 并发与序列长度公式
// 已知序列长度，求最大并发数
max_concurrent = KV_Cache_Available / KV_Per_Token_Per_Layer × n_layers × seq_len)

KV_Per_Token = 2 × n_kv_heads × d_head × dtype_bytes

// 已知并发数，求最大序列长度
max_seq_len = KV_Cache_Available / (KV_Per_Token × n_layers × batch_size)

实际计算示例: LLaMA-70B on A100-80GB

计算过程 # LLaMA-70B 模型参数 模型: LLaMA-70B (80层, 64 KV heads, d_head=128) GPU: NVIDIA A100-80GB (80 GB HBM2e) 精度: FP16 权重 + FP16 KV Cache # Step 1: 模型权重 (FP16, 无量化) Model_Weights = 70B × 2 bytes = 140 GB # → 单卡放不下！需要张量并行 TP=2 (2×A100) Model_Weights_per_GPU = 140 / 2 = 70 GB # Step 2: 每GPU可用内存 Available = 80 × 0.92 - 70 - 2 - 1 = 0.6 GB # 紧张！ # Step 3: 使用INT8量化权重 Model_Weights_INT8 = 70 / 2 = 35 GB (per GPU with TP=2) Available = 80 × 0.92 - 35 - 2 - 1 = 35.6 GB # Step 4: 每token KV Cache开销 KV_Per_Token = 2 × 64 × 128 × 2 bytes = 32,768 bytes = 32 KB KV_Per_Token_All_Layers = 32 KB × 80 = 2,560 KB = 2.5 MB # Step 5: 最大并发 (seq_len=2048) max_batch = 35.6 GB / (2.5 MB × 2048) = 35.6 GB / 5.12 GB ≈ 6.9 → 6 # Step 6: 最大序列长度 (batch_size=4) max_seq = 35.6 GB / (2.5 MB × 4) = 3,645,440 tokens → 但受模型限制为 4096/8192
⚠️ 关键洞察: 对于70B模型，INT8量化是单节点(2×A100)部署的关键。FP16权重+FP16 KV Cache几乎不留空间给并发。

02 vLLM KV Cache调优

通俗类比：vLLM的PagedAttention就像把餐厅的大圆桌（连续显存）换成了灵活的四人小桌（分页）。客人（序列）来了可以拼桌，走了桌位立刻回收，接待能力（吞吐量）大幅提升。
常见误区：切忌将`gpu_memory_utilization`设得过高（如0.99）。系统和驱动需要缓冲，设置过高极易触发OOM（内存溢出），导致服务崩溃。建议从0.9开始尝试。
深入探讨：关键参数：`--gpu-memory-utilization 0.9`（默认0.9），`--max-model-len`（单请求最大长度），`--max-num-seqs`（最大并行序列数）。PagedAttention通过`block_size`（默认16）管理显存页，其管理开销远低于连续内存分配。
 传统连续分配：
 [Seq1: 长长的连续内存块] [Seq2: 更长的连续内存块] -> 碎片化，空间浪费

 vLLM PagedAttention：
 [Block][Block][Block] (Seq1的3个页)   [Block][Block] (Seq2的2个页)
 -> 灵活拼接，空间利用率高，可回收复用
参数作用调优建议
gpu_memory_utilization控制KV Cache可用显存比例设为0.85-0.95，留有余地
max_num_seqs最大并发请求数根据显存和延迟要求调整，过大易OOM
block_sizeKV Cache分页大小一般保持默认16，无需修改

vLLM vLLM是当前最流行的推理引擎之一，其PagedAttention机制使KV Cache管理极为高效。

核心参数详解

参数 默认值 说明 调优建议
--gpu-memory-utilization 0.90 GPU显存使用比例，剩余部分留给KV Cache 生产环境0.88-0.93，留余量防OOM
--max-num-seqs 256 最大并发请求数 根据GPU内存调，A100-80G通常24-128
--max-num-batched-tokens None (auto) 单步最大批处理token数 增大可提升吞吐但增加延迟
--kv-cache-dtype auto (FP16) KV Cache数据类型 fp8节省50%内存，精度损失极小
--enable-prefix-caching False 启用前缀缓存(自动复用共同前缀) 多轮对话/Agent场景强烈推荐
--max-model-len 模型默认 最大上下文长度 降低可容纳更多并发
--enforce-eager False 禁用CUDA Graph 调试时开，生产关闭以提升性能
--tensor-parallel-size 1 张量并行度 多GPU必设，等于GPU数

03 SGLang调优

通俗类比：SGLang的RadixAttention就像餐厅后厨的“预制菜缓存”。常被点的“宫保鸡丁”（公共前缀）的备料被缓存，新订单来了直接取用，不用从头切配，出菜（生成）速度飞快。
常见误区：使用压缩功能时，务必注意其适用的场景。对于随机性强、前缀共享少的对话，压缩收益可能很低，甚至引入额外开销。需先用少量请求测试压缩比。
深入探讨：SGLang通过`--mem-fraction-static`（默认0.8）分配静态缓存，并支持对KV Cache进行`int8`量化压缩。其RadixAttention缓存命中率在多轮对话、代码生成等场景下可达30%-70%，能显著降低TTFT（首token延迟）。
 请求1: [你][好][吗][？] --> 生成[我][很][好]
 请求2: [你][好][吗][？]今天[吃][什][么]
 ↓ RadixAttention缓存
 直接复用 [你][好][吗][？] 的KV，从[今天]开始计算
特性原理最佳适用场景
RadixAttention基于基数树缓存公共前缀KV多轮对话、代码补全
KV Cache压缩对存储的KV值进行INT8量化长序列生成，显存瓶颈场景
连续批处理类似vLLM的高效批处理高并发在线服务
维度训练(Training)推理(Inference)

SGLang SGLang以RadixAttention（基数树KV Cache管理）为核心特色，特别适合多轮对话和Agent场景。

核心参数

参数 默认值 说明 调优建议
--mem-fraction-static 0.88 静态分配的GPU内存比例 越大KV Cache越多，但留余量防OOM
--max-running-requests None 最大同时运行请求数 限制并发防OOM，配合mem-fraction调
--schedule-policy lpm 调度策略: lpm/fcfs/random lpm(最长前缀匹配)适合Agent场景
--schedule-conservativeness 1.0 调度保守度 <1更激进(高吞吐)，>1更保守(低OOM)
--chunked-prefill-size 8192 分块预填充大小 减小可降低长prompt的首token延迟
--enable-cache-report False 报告缓存命中信息 调试时开启
--dp-size 1 数据并行大小 多副本提升吞吐

04 TensorRT-LLM调优

通俗类比：TensorRT-LLM就像一位“预制菜大师”。它在开始营业（推理）前，会花很长时间编译、优化计算图（好比精心预制并包装好半成品）。一旦开始，每道菜（推理请求）都出得极快、极稳定。
常见误区：编译优化是“一次性的”，但不同模型、精度、算子版本都需要重新编译。务必保存好编译好的`engine`文件，否则换模型或升级后环境不匹配，会报错或性能暴跌。
深入探讨：其核心是`图融合`（Graph Fusion）和`Kernel自动调优`。例如，将多个连续的`MatMul+Add+LayerNorm`算子融合成一个优化的Kernel。在A100上，对LLaMA-70B的推理，经过优化的TensorRT-LLM engine相比PyTorch原生实现，吞吐量可提升3-5倍。
 原始计算图 (PyTorch):
 Input -> MatMul -> Add -> LayerNorm -> ... -> Output

 TensorRT-LLM优化后:
 Input -> [Fused_MatMulAddLayerNorm_Kernel] -> ... -> Output
 └─── 多个操作合并为一个高效GPU Kernel ───┘
优化技术说明性能收益
算子融合合并多个小算子为一个大算子减少显存读写和Kernel启动开销
Kernel自动调优为特定硬件选择最优计算Kernel最大化GPU计算单元利用率
In-flight Batching动态批处理，持续利用GPU提高吞吐量，降低延迟

05 监控与诊断

通俗类比：这就像给餐厅装监控和仪表盘。光看“客流量”（QPS）不够，还要看“上菜速度”（延迟）、“厨房拥挤度”（GPU利用率）和“冰箱库存”（显存使用）。数据是调优的眼睛。

⏱️ 端到端延迟分解
Total_Latency = Queue_Time + Prefill_Time + Decode_Time + Overhead

TTFT (Time To First Token) = Queue_Time + Prefill_Time
TPOT (Time Per Output Token) ≈ Decode_Time per step
TPS (Tokens Per Second) = 1 / TPOT
Total_Time ≈ TTFT + TPOT × output_length

06 常见问题与排查

通俗类比：当餐厅“上菜慢”或“菜品变味”（推理报错）时，需要系统性排查。是“厨师”（模型）问题、“菜单”（配置）问题，还是“冰箱”（显存）问题？本节就是你的故障检修手册。
常见误区：遇到OOM（内存溢出）时，不要盲目增大`gpu_memory_utilization`或减小`max_model_len`。应首先检查是否有异常大的请求、确认批处理大小、并查看是否存在显存泄漏（监控曲线持续上升）。
深入探讨：高频问题排查清单：1) **OOM**: 检查单请求最大长度、并发数、模型精度。2) **高延迟**: 检查是否开启了流式输出、批处理是否饱和、是否存在显存交换（Swap）。3) **结果乱码/重复**: 检查分词器配置、推理终止条件（`eos_token`）、是否使用了错误的对齐方式（如聊天模板）。

问题1: OOM - KV Cache内存不足

🔍 症状

CUDA out of memory
请求被拒绝
KV Cache使用率持续 >95%

🔎 根因

1. 并发数过高
2. max_seq_len设置过大
3. FP16 KV Cache消耗过多
4. 长上下文请求堆积

✅ 解决方案

1. 降低--max-num-seqs
2. 启用--kv-cache-dtype fp8
3. 降低--max-model-len
4. 启用前缀缓存
5. 降低--gpu-memory-utilization到0.85

问题2: 吞吐量低于预期

🔍 症状

tokens/s远低于benchmark
GPU利用率 <60%
请求数少但延迟高

🔎 根因

1. max_num_seqs太小导致批处理不足
2. CUDA Graph未启用
3. 序列长度差异大(straggler)
4. TP配置不当

✅ 解决方案

1. 增大--max-num-seqs
2. 关闭--enforce-eager(启用CUDA Graph)
3. 限制max_model_len减少straggler
4. 尝试continuous batching参数调优

问题3: 延迟突增

🔍 症状

TTFT P99从200ms突增到2s+
间歇性延迟飙升
排队请求数波动大

🔎 根因

1. 长prompt请求阻塞预填充
2. KV Cache碎片化
3. 调度器不公平
4. 突发流量

✅ 解决方案

1. 启用chunked prefill分块处理
2. 使用prefix caching减少重复计算
3. 配置请求超时和速率限制
4. SGLang用lpm调度策略

问题4: Prefix Cache命中率低

🔍 症状

prefix_cache_hit_rate < 20%
多轮对话无加速效果
TTFT无明显降低

🔎 根因

1. prompt格式不一致
2. KV Cache被频繁淘汰
3. 系统prompt过短
4. 随机化(temperature>0)破坏缓存

✅ 解决方案

1. 统一prompt模板格式
2. 增大KV Cache空间减少淘汰
3. 将共同前缀(system prompt)拉长
4. 确保tokenization一致

问题5: 长上下文请求失败

🔍 症状

请求报错 "max context length exceeded"
或 silently truncated
长文档问答质量下降

🔎 根因

1. max_model_len设置过小
2. 并发+长上下文超出KV Cache预算
3. max_input_len限制

✅ 解决方案

1. 增大--max-model-len
2. 减少并发数让出KV Cache空间
3. 启用FP8 KV Cache
4. 使用INT4量化减少权重占用

08 最佳实践清单

通俗类比：KV-Cache调优就像优化厨房调料架——常用调料放最前排（热门缓存），少用调料存后排（冷数据），这样厨师能快速取用，提升出餐效率。
常见误区：常见错误是只盲目扩大缓存容量而不监控命中率，导致内存浪费；或忽略数据热度变化，缓存了低频数据，反而降低整体性能。
深入探讨：工程调优中，关键指标是缓存命中率，目标通常设为90%以上。使用公式：缓存效率 = (缓存命中次数 / 总请求次数) × 100%。建议采用LRU（最近最少使用）策略，并根据实时负载动态调整缓存大小，例如设置弹性缓存窗口，内存使用率控制在70-80%为佳。

📋 部署前 Checklist

  • 计算模型权重大小，确认GPU内存足够
  • 规划KV Cache预算: 可用内存 - 权重 - 激活 - 系统开销
  • 选择合适的精度: FP16/INT8/INT4 权重 + FP16/FP8 KV Cache
  • 确定目标并发数和最大序列长度
  • 选择推理引擎 (vLLM / SGLang / TensorRT-LLM)
  • 配置张量并行 (TP) / 流水线并行 (PP) 如需多GPU
  • 启用Prefix Caching (如有多轮对话需求)
  • 准备压力测试脚本和监控配置
  • 设置请求超时和速率限制
  • 测试OOM边界: 逐步加压到KV Cache耗尽，记录安全上限

📊 上线后监控 Checklist

  • KV Cache使用率: 告警阈值 85%, 危险阈值 95%
  • Prefix Cache命中率: 记录基线，异常下降及时告警
  • TTFT P50/P99: 设置SLA (如 P99 < 500ms)
  • TPOT P50: 设置SLA (如 P50 < 30ms)
  • 排队深度 (waiting requests): 告警阈值 > 10
  • GPU内存使用: 告警阈值 > 95%
  • GPU温度: 告警阈值 > 83°C
  • 请求错误率: OOM拒绝 / 超时 / 5xx
  • Prometheus指标正常采集
  • Grafana面板覆盖所有关键指标

⚡ 性能优化 Checklist

  • FP8 KV Cache: 一行配置节省50%内存
  • Prefix Caching: 多轮对话场景必开
  • Chunked Prefill: 长prompt场景降低TTFT
  • Continuous Batching: 确保引擎默认启用
  • CUDA Graph: 生产环境确保启用
  • INT4/INT8量化: 大模型减少权重占用
  • 适当降低max_model_len减少KV Cache压力
  • 使用GQA模型天然减少KV Cache (如 LLaMA-3 8B: 8 KV heads)
  • 批处理大小调优: 逐步增加直到throughpt饱和
  • 数据并行(DP): 如有多余GPU，DP比TP扩展性更好

🔧 故障排查 Checklist

  • 确认OOM: 检查nvidia-smi、dmesg、应用日志
  • 检查KV Cache使用率趋势: 是渐进增长还是突变?
  • 确认是否为长上下文请求导致: 过滤input_len > 阈值的请求
  • 检查Prefix Cache命中率: 下降是否与部署变更相关?
  • 对比变更前后的性能指标: 排查配置/代码变更影响
  • 检查GPU硬件状态: ECC错误、温度、功耗
  • 检查网络: 多GPU通信延迟 (NVLink vs PCIe)
  • 复现问题: 用固定seed和输入复现，确认是否稳定复现
  • 回滚验证: 回退到上一个正常版本确认问题是否消失
  • 收集诊断信息: metrics截图、日志、配置文件、硬件信息
✅ 总结: KV Cache调优的核心思路是 内存预算 → 参数配置 → 监控验证 → 持续优化。记住公式：KV可用内存 = 总内存 × 利用率 - 权重 - 激活 - 开销，一切优化都围绕最大化这块"蛋糕"。
