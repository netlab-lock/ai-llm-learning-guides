1. 自回归生成与重复计算问题

通俗类比：餐厅每次上新菜，都要把之前所有顾客点过的菜重新切配一遍。KV Cache就是把前序菜品的切配结果记在小票上，下次直接取用，省时省力。
常见误区：新手常误以为缓存是“存储对话历史文本”。KV Cache缓存的是模型内部的中间计算状态（Key和Value张量），不是原始文字。
深入探讨：假设模型有32层，隐藏维度为4096。生成第100个token时，若不缓存，仅注意力层的Key/Value计算就需要对前99个token重复计算32*2次。单次浮点运算(FLOPs)量级约为 99 * 4096 * 4096 * 32 * 2 ≈ 10.6 TFLOPs。缓存后，此计算量降至约 1 * 4096 * 4096 * 32 * 2 ≈ 0.1 TFLOPs。
无Cache:
[T1, T2, T3] -> 注意力 -> T4
[T1, T2, T3, T4] -> 注意力 -> T5  (T1-T3重复计算)

有Cache:
[T1, T2, T3] -> 注意力 -> T4 (Cache: K3, V3)
[T4] + Cache(K3,V3) -> 注意力 -> T5 (仅计算T4)

1.1 通俗类比：抄写笔记的困境

💡 通俗类比

想象你在考试中抄写课堂笔记。每次你要写一个新字时，老师说："请从第一个字开始，重新读一遍所有已写的字，然后才写下一个。"

写了10个字时，你需要读前面10个字才能写第11个。写了100个字后，每次写新字都要重读100个字。这显然极其低效！

KV Cache就像是你桌上的笔记本——你把之前写的字都记在本子上（缓存起来），写新字时只需看一眼本子，不需要从头重写。这就是KV Cache的核心思想：用空间换时间，避免重复计算。

1.2 没有KV Cache的自回归生成

自回归语言模型（如GPT、LLaMA）逐token生成文本。在时刻 t，模型需要根据已生成的所有token x₁, x₂, ..., xₜ₋₁ 来预测下一个token xₜ。

关键问题：在标准Transformer的Self-Attention层中，每生成一个新token，都需要重新计算整个序列的Key和Value。

// 没有KV Cache：每步都重新计算所有token的K, V // 生成第t个token时的计算量： Step t: 需要计算 Q = X[1:t] @ W_q // 重新计算所有t个token的Query K = X[1:t] @ W_k // 重新计算所有t个token的Key ← 重复！ V = X[1:t] @ W_v // 重新计算所有t个token的Value ← 重复！ Attention = softmax(Q @ K^T / √d_k) @ V // 总计算量（生成T个token）： Total FLOPs ∝ Σ(t=1 to T) t² ≈ T³/3 // 三次方增长！

1.3 使用KV Cache的增量计算

🔑 核心思想

观察到：在生成第 t 个token时，x₁ 到 xₜ₋₁ 的Key和Value与上一步完全相同！因此，我们可以将它们缓存起来，每一步只计算新token xₜ 的 K 和 V。

// 有KV Cache：增量计算 // 初始化：处理prompt时缓存所有K, V K_cache = K_prompt // [1:t₀, d_k] V_cache = V_prompt // [1:t₀, d_v] // 生成第t个token（t > t₀）： q_t = x_t @ W_q // 只计算1个token的Query ✓ k_t = x_t @ W_k // 只计算1个token的Key ✓ v_t = x_t @ W_v // 只计算1个token的Value ✓ K_cache = concat(K_cache, k_t) // 追加到缓存 V_cache = concat(V_cache, v_t) // 追加到缓存 attn = softmax(q_t @ K_cache.T / √d_k) @ V_cache // 总计算量（生成T个token）： Total FLOPs ∝ T × T ≈ T² // 二次方，大幅改善！

1.4 逐步数值示例：3个token的注意力计算

让我们用一个极简例子（d_model=4, d_k=2）来追踪每一步的计算。

📐 数值设定

假设词嵌入矩阵（3个token，维度4）：

X = [[1, 0, 1, 0], // token_1: "我" [0, 1, 0, 1], // token_2: "爱" [1, 1, 0, 0]] // token_3: "你" W_q = [[1, 0], [0, 1], [1, 0], [0, 1]] // [4, 2] W_k = [[1, 1], [0, 0], [1, 1], [0, 0]] // [4, 2] W_v = [[1, 0], [0, 0], [0, 1], [0, 0]] // [4, 2]

Step 1: 没有KV Cache — 生成token_3时

1计算所有3个token的 Q, K, V
Q = X @ W_q = [[1, 0], [0, 1], [1, 1]] // [3, 2] K = X @ W_k = [[2, 2], [0, 0], [1, 1]] // [3, 2] ← token_1,2的K重复计算了！ V = X @ W_v = [[1, 0], [0, 0], [1, 0]] // [3, 2] ← token_1,2的V重复计算了！ // Attention scores: Q @ K^T / √2 Q @ K^T = [[2, 0, 1], [0, 0, 1], [2, 0, 2]] // token_1: [2, 0, 1] // token_2: [0, 0, 1] // token_3: [2, 0, 2] ÷ √2 = [[1.41, 0, 0.71], [0, 0, 0.71], [1.41, 0, 1.41]] // softmax（每行）： Attn = [[0.56, 0.15, 0.29], // token_1的注意力分布 [0.27, 0.27, 0.46], // token_2的注意力分布 [0.38, 0.12, 0.50]] // token_3的注意力分布 // 输出 = Attn @ V Output = [[0.61, 0], [0.46, 0], [0.69, 0]] // [3, 2]

Step 2: 使用KV Cache — 生成token_3时

✓假设我们已经在生成token_1和token_2时缓存了它们的K和V
K_cache = [[2, 2], [0, 0]] // 缓存: token_1和token_2的Key [2, 2] V_cache = [[1, 0], [0, 0]] // 缓存: token_1和token_2的Value [2, 2] // 只计算token_3的Q, K, V q_3 = [1, 1, 0, 0] @ W_q = [1, 1] // 只算1个！ k_3 = [1, 1, 0, 0] @ W_k = [1, 1] // 只算1个！ v_3 = [1, 1, 0, 0] @ W_v = [1, 0] // 只算1个！ // 追加到缓存 K_cache = [[2, 2], [0, 0], [1, 1]] // concat → [3, 2] V_cache = [[1, 0], [0, 0], [1, 0]] // concat → [3, 2] // 只计算token_3的注意力 q_3 @ K_cache^T = [1,1] @ [[2,0,1],[2,0,1]] = [2, 0, 2] ÷ √2 = [1.41, 0, 1.41] softmax = [0.38, 0.12, 0.50] Output_3 = [0.38, 0.12, 0.50] @ V_cache = [0.69, 0] // ✓ 结果完全一致！但节省了2/3的QKV投影计算

2. Self-Attention 数学推导

通俗类比：把每个词想象成一个仓库。Query是发货请求单，Key是仓库的门牌号，Value是仓库里的货物。注意力计算就是根据请求单，去所有仓库提货并按相关性（点积）配货。
常见误区：易混淆点：计算 Attention(Q, K, V) 时，矩阵乘法顺序是 (Q K^T) V，而非 (Q V) K^T。Q和K^T相乘得到的注意力分数矩阵形状是 [seq_len_q, seq_len_k]。
深入探讨：标准自注意力公式：Attention(Q, K, V) = softmax(Q K^T / √d_k) V。其中 √d_k 是缩放因子，防止点积结果过大导致softmax梯度消失。以BERT-base为例，d_k = 64，点积最大理论值约为 √64 = 8，softmax后仍能保持梯度。
输入X [seq, d_model]
   |       |       |
   W_Q     W_K     W_V
   |       |       |
   Q       K       V
   |       |
   Q * K^T [seq_q, seq_k]
   |
   softmax & scale
   |
   Attn Score [seq_q, seq_k]
   |
   Score * V -> Output
计算步骤输入张量形状输出张量形状核心操作
生成Q,K,V[batch, seq, d_model]三个 [batch, seq, d_k]线性投影

2.1 通俗类比：图书馆查资料

💡 通俗类比

Self-Attention就像你在图书馆查资料：

  • Query（查询）：你脑中的问题 — "这段话和哪些内容有关？"
  • Key（索引）：每本书的标签/目录 — 表示"我包含什么内容"
  • Value（内容）：每本书的实际内容 — 表示"我能提供的信息"

你拿着问题(Q)，去匹配所有书的标签(K)，找到相关性最高的书，然后提取它们的内容(V)组合起来。

KV Cache意味着：之前读过的书，你把它们的标签和关键内容抄在便签纸上（缓存），下次查新问题时不用重新翻那些书。

2.2 Q, K, V 投影

给定输入序列矩阵 X ∈ ℝ^{n×d_model}（n个token，每个维度d_model），通过三个可学习的权重矩阵将输入投影到不同的表示空间：

// === QKV 投影 === Q = X · W_Q // Query: [n, d_model] × [d_model, d_k] → [n, d_k] K = X · W_K // Key: [n, d_model] × [d_model, d_k] → [n, d_k] V = X · W_V // Value: [n, d_model] × [d_model, d_v] → [n, d_v] // 其中通常 d_k = d_v = d_model / num_heads

2.3 Attention Score 计算

// 标准 Scaled Dot-Product Attention Attention(Q, K, V) = softmax( Q · K^T / √d_k ) · V // 分步分解： // Step 1: 相似度矩阵 S = Q · K^T // [n, d_k] × [d_k, n] → [n, n] // S[i,j] = q_i · k_j = token_i对token_j的原始关注度 // Step 2: 缩放 S_scaled = S / √d_k // 防止点积过大导致softmax饱和 // Step 3: 归一化（因果掩码） A = softmax(S_scaled + Mask) // [n, n], 每行和为1 // Mask: 上三角为-∞（因果注意力：不能看到未来token） // Step 4: 加权求和 Output = A · V // [n, n] × [n, d_v] → [n, d_v]

2.4 Prefill 阶段 vs Decode 阶段的数学差异

这是理解KV Cache的核心：两个阶段做的是同一个注意力计算，但矩阵形状完全不同。

PREFILL 预填充阶段

// 输入: 整个prompt, 长度 = n Q = Xprompt @ W_q // [n, d_k] ← n个query K = Xprompt @ W_k // [n, d_k] ← n个key V = Xprompt @ W_v // [n, d_v] ← n个value Scores = Q @ K^T // [n, n] ← 完整矩阵 Output = softmax(Scores/√d_k) @ V // 计算类型: 矩阵×矩阵 (GEMM) // 并行度: 极高 ✓

DECODE 解码阶段（逐token）

// 输入: 仅1个新token q_t = x_t @ W_q // [1, d_k] ← 1个query k_t = x_t @ W_k // [1, d_k] ← 1个新key v_t = x_t @ W_v // [1, d_v] ← 1个新value K_cache ← concat(K_cache, k_t) // [t, d_k] V_cache ← concat(V_cache, v_t) // [t, d_v] Scores = q_t @ K_cache^T // [1, t] ← 向量×矩阵 Output = softmax(Scores/√d_k) @ V_cache // [1, d_v] // 计算类型: 向量×矩阵 (GEMV) // 并行度: 低 ✗

3. KV Cache 的数学定义

通俗类比：就像出门前，把“钥匙”和“钱包”（Key和Value）提前放好在固定位置（缓存区）。每生成一个新词，只需把新的“钥匙”和“钱包”放到指定位置，出门时直接拿走一整串，不用每次都翻箱倒柜找。
常见误区：常见的实现误区是将整个输入序列的KV一次性计算并缓存。在自回归解码中，应只缓存新生成token对应的单个KV向量，并与历史缓存拼接，实现增量更新。
深入探讨：缓存更新公式（第t步）： K_cache_t = [K_cache_{t-1}; K_t] (沿序列维度拼接) V_cache_t = [V_cache_{t-1}; V_t] 其中K_t, V_t是仅对第t个输入token x_t计算的结果，形状为 [batch, 1, d_k]。缓存总形状随序列长度线性增长：[batch, num_heads, t, d_k]。

3.1 通俗类比：渐长的购物清单

💡 通俗类比

KV Cache就像你不断追加的购物清单。每次你想到一个新东西（新token），就在清单末尾写上它的标签（K）和内容（V）。清单越来越长，但你永远不需要擦掉之前写过的条目。

代价是什么？清单会占用越来越多的纸（显存）。当清单长得塞不进一张纸时（显存不够），你就需要想办法压缩它——这就是后续模块要讲的KV Cache压缩技术。

3.2 形式化定义

📐 数学定义

在时刻 t（已生成t个token），对于单个注意力层中的单个注意力头：

// KV Cache 的递推定义 // 初始状态（处理完prompt后）： K_cache^{(0)} = [k_1, k_2, ..., k_{t₀}] // prompt的所有Key V_cache^{(0)} = [v_1, v_2, ..., v_{t₀}] // prompt的所有Value // 递推更新（每生成一个新token）： K_cache^{(t)} = [K_cache^{(t-1)}; k_t] // 追加新Key V_cache^{(t)} = [V_cache^{(t-1)}; v_t] // 追加新Value // 矩阵形式： K_cache ∈ ℝ^{t × d_k} // t行d_k列 V_cache ∈ ℝ^{t × d_v} // t行d_v列

3.3 多层多头的完整内存公式

实际的Transformer模型有 L 层，每层有 H 个注意力头。每层的每个头都需要独立的KV Cache。

// 总内存公式（所有层、所有头） Memory_KV = 2 × L × H × d_head × seq_len × dtype_bytes // 其中： 2 // K和V各一份 L // Transformer层数 H // 每层的注意力头数 d_head // 每个头的维度 (通常 d_model / H) seq_len // 序列长度（prompt + 生成的token） dtype_bytes // 数据类型字节数 (FP16=2, FP32=4, INT8=1) // 注意: 2 × H × d_head = H × d_k + H × d_v = d_model (当 d_k = d_v 时) // 等价写法： Memory_KV = 2 × L × d_model × seq_len × dtype_bytes

4. Prefill vs Decode 两阶段深度分析

通俗类比：Prefill是大型厨房一次性处理全部食材（用户输入），可以并行切配所有蔬菜。Decode是精致摆盘阶段，每放上一道菜（生成一个词），就要根据盘子（缓存）里的现有布局来决定下一道菜的位置，是串行的。
常见误区：一个关键工程细节：在Prefill阶段，为了计算注意力，模型也需要使用初始的KV Cache（来自输入提示）。但在实现上，这通常是一次性生成的完整KV张量，而非逐步缓存。
深入探讨：性能对比：假设输入提示有1000个token，模型需生成500个token。 - Prefill阶段：一次性并行计算1000个token的KV，并进行一次完整注意力计算。计算密集型。 - Decode阶段：串行生成500个token，每步计算1个token的Q和注意力。此阶段是访存密集型，因为每步需读取庞大的缓存（约1000+当前步长）来计算注意力。
Prefill阶段 (计算密集型并行):
输入: [T1, T2, ..., T1000]
输出: [O1, O2, ..., O1000], Cache_all

Decode阶段 (访存密集型串行):
输入: T1001
加上: Cache_all (1000个KV)
输出: O1001, Cache_all+[K1001,V1001]
输入: T1002 ...
输出: ...

4.1 通俗类比：演讲准备 vs 现场回答

💡 通俗类比

Prefill就像你准备演讲稿：你可以同时查看所有资料，并行处理大量信息，效率很高（计算密集）。

Decode就像现场回答观众提问：每次只能回答一个问题，而且每个答案都要参考之前的全部对话（内存密集）。你的瓶颈不是思考速度，而是翻笔记本找资料的速度。

4.3 Roofline分析：为什么Decode是内存瓶颈

GPU的计算能力可以用 Roofline模型来理解：每个操作要么受限于算力（FLOPS），要么受限于内存带宽（Bandwidth），取决于算术强度（Arithmetic Intensity）。

// 算术强度 = 计算量 / 内存访问量 (FLOPs / Byte) // 临界点 = 峰值FLOPS / 峰值带宽 (硬件决定) // H100 SXM 参数： 峰值算力 = 989 TFLOPS (FP16 Tensor Core) 峰值带宽 = 3.35 TB/s (HBM3) // 临界算术强度 = 989 / 3.35 ≈ 295 FLOPs/Byte 临界点 ≈ 295 // 算术强度 > 295 → 计算密集 (Compute-bound) // 算术强度 < 295 → 内存密集 (Memory-bound)

Decode阶段的算术强度计算

// Decode: 生成1个token，序列长度为t，隐藏维度d // QKV投影（以一个线性层为例）： 计算量 = 2 × d × d_model (矩阵×向量) ≈ 2d² 内存读取 = d × d_model × 2 bytes (权重) + d_model × 2 (输入) ≈ 2d² + 2d ≈ 2d² bytes // Attention计算（最关键的一步）： 计算量 = 2 × t × d_k (q @ K_cache^T) ≈ 2·t·d_k 内存读取 = t × d_k × 2 bytes (K_cache) ≈ 2·t·d_k // 算术强度 = 2·t·d_k / (2·t·d_k + 权重) ≈ 很小的值 // 以LLaMA-7B为例: d_model = 4096, d_k = 128, t = 2048 计算量 = 2 × 2048 × 128 = 524,288 FLOPs 内存读取 = 2048 × 128 × 2 = 524,288 bytes 算术强度 = 524,288 / 524,288 = 1 FLOP/Byte // 远小于临界点295！ // 结论: Decode极端内存密集，GPU算力严重浪费

4.4 Prefill的计算细节

// Prefill处理完整prompt (n个token) // 核心操作: Q @ K^T — 这是一个 [n, d] × [d, n] 的矩阵乘法 计算量 = 2 × n × n × d_k // 两个矩阵相乘的FLOPs 内存读取 = n × d_k × 2 (Q) + n × d_k × 2 (K) = 4·n·d_k bytes 算术强度 = 2·n²·d_k / (4·n·d_k) = n / 2 // 当 n=2048 时，算术强度 = 1024 >> 295 → Compute-bound ✓ // 当 n=8192 时，算术强度 = 4096 >> 295 → 高度Compute-bound ✓

5. KV Cache 的代价

通俗类比：KV Cache就像你为了做饭快，提前把所有可能的配菜都洗好切好摆在台面上。速度是快了，但厨房台面（显存）被占得满满当当，可能连放新盘子的空间都没有了。
常见误区：最大序列长度（Max Sequence Length）设置得过大，是导致KV Cache占用显存“爆表”的常见原因。应根据实际应用场景谨慎设置，而非盲目追求最大值。
深入探讨：显存占用公式： 大小 ≈ 2 * 层数 * 隐藏维度 * 序列长度 * 精度字节数 * 批大小 以LLaMA-70B为例：2 * 80 * 8192 * 序列长度 * 2字节(FP16) * 1 序列长度为4096时，缓存占用约为 2*80*8192*4096*2 / 1e9 ≈ 10.7 GB。若批大小为8，则需约85.6 GB，远超单卡80GB显存。

5.1 通俗类比：仓库租赁费用

💡 通俗类比

KV Cache就像租仓库存放货物：

  • 空间代价（Memory Cost）：仓库越大越贵——显存是GPU上最昂贵的资源
  • 搬运代价（Bandwidth Cost）：每次取货都要搬运——把KV Cache从显存读入计算单元需要时间
  • 时间代价（Latency）：仓库越大，找东西越慢——KV Cache越大，每生成一个token的延迟越高

5.3 带宽代价分析

Decode阶段每生成一个token，需要读取整个KV Cache来计算注意力。这意味着内存带宽是核心瓶颈。

// Decode每步的内存读取量（单层、单头）： 读取量 = (seq_len × d_k + seq_len × d_v) × dtype_bytes = 2 × seq_len × d_head × dtype_bytes // 所有层所有头： 读取量 = 2 × L × H × seq_len × d_head × dtype_bytes // 示例: LLaMA-7B, seq_len = 4096 读取量 = 2 × 32 × 32 × 4096 × 128 × 2 bytes = 2,147,483,648 bytes = 2 GB per token // H100带宽 = 3.35 TB/s 时间 = 2 GB / 3.35 TB/s ≈ 0.6 ms per token (仅KV Cache读取)

6. 多头注意力中的 KV Cache 变体

通俗类比：MHA是每个部门（注意力头）都有自己的会议记录本和档案柜。MQA是所有部门共享同一套记录本和档案柜，但用不同颜色标签（Query）做标记。GQA是将部门分组，每组共享一套。
常见误区：MQA/GQA的核心优势在于极大减少了KV Cache的显存占用和解码时的内存带宽需求，但可能会损失一定的模型表达能力。选择时需要权衡速度、成本和模型效果。
深入探讨：显存对比（以LLaMA-7B架构，隐藏维度4096为例）： - MHA: 32头，每头d_k=128。缓存大小 = 2 * 32 * 128 * L * 2 bytes = 16384 * L bytes。 - GQA-8 (8组，每组4头共享KV): 缓存大小 = 2 * 8 * 128 * L * 2 bytes = 4096 * L bytes。相比MHA减少75%。 - MQA: 缓存大小 = 2 * 1 * 128 * L * 2 bytes = 512 * L bytes。减少96.875%。

7.1 核心概念回顾

自回归生成中的重复计算

没有KV Cache时，每生成一个新token都要重新计算所有先前token的Key和Value，导致 O(T³) 的总计算量。

KV Cache的缓存机制

缓存已计算的K和V矩阵，每步只计算新token的QKV，将总计算量降至 O(T²)。

内存公式

Memory = 2 × L × H × d_head × seq_len × dtype，线性增长，对长序列是巨大挑战。

Prefill vs Decode

Prefill是GEMM（计算密集），Decode是GEMV（内存密集）。Decode的算术强度仅约1 FLOP/Byte，远低于GPU临界点~295。

MHA/GQA/MQA权衡

GQA通过分组共享KV头，在质量和效率之间取得最佳平衡，已被LLaMA-2/3和Mistral等主流模型采用。
