🔀 11 · Seq2Seq — 编码器-解码器架构

2014 · Sutskever/Vinyals/Le (Google) & Cho 等 · 机器翻译里程碑 · Transformer 的直接前身

一、背景与动机

机器翻译的输入和输出都是变长序列，但长度不一定相等。例如 "I love you" → "我爱你"(3→3)，"I love deep learning" → "我喜欢深度学习"(4→4)。普通 RNN 无法处理输入输出长度不同的问题。

2014年，两个团队独立提出了解决方案：编码器-解码器(Encoder-Decoder)架构——用一个 RNN 把输入序列"压缩"成固定长度向量，再用另一个 RNN 从这个向量"解压"出输出序列。

💡 核心思想：编码器读完输入后把所有信息压缩到一个向量(context vector)中，解码器从这个向量开始逐词生成输出。就像翻译员先听完整句话，再逐词翻译。

二、名字由来

💡 通俗类比：Seq2Seq(Sequence-to-Sequence)就是"序列到序列"——输入一个序列，输出一个序列。翻译("I love you"→"我爱你")、摘要(长文→短文)、对话(问题→回答)都是Seq2Seq。
⚠️ 常见误区：Seq2Seq 不是一种"模型"，而是一种"框架"。编码器和解码器可以是 RNN、LSTM、GRU、甚至 Transformer。"Encoder-Decoder"架构是更准确的说法。
🔬 深入探讨：Seq2Seq 的提出(2014年，Sutskever et al.)标志着 NLP 从"特征工程"转向"端到端学习"——不再需要人工设计特征，模型直接从原始文本学习输入到输出的映射。这个范式转变影响了整个 NLP 领域。

Seq2Seq：Sequence-to-Sequence(序列到序列)。简称 S2S。

Encoder-Decoder：编码器(压缩输入) + 解码器(生成输出)。这个框架后来成为 NLP 的通用范式——Transformer 就是 Encoder-Decoder 架构。

三、核心原理

💡 通俗类比：编码器就像"听完整段话后在脑中形成印象"，解码器就像"根据印象逐词翻译"。问题是：印象(上下文向量)是一个固定长度的向量——如何把一整段话压缩成一个向量而不丢失信息？这就是注意力机制要解决的。
⚠️ 常见误区：原始 Seq2Seq 的"信息瓶颈"问题很严重——把整个输入序列压缩成一个固定向量(通常256-1024维)。长句子的信息损失巨大。注意力机制(2014)正是为了解决这个问题而生的。
🔬 深入探讨：Teacher Forcing 是 Seq2Seq 训练的重要技巧——解码时用"真实标签"而非"模型预测"作为下一步输入。这加速了训练但导致"暴露偏差"(Exposure Bias)——训练和推理的输入分布不一致。Scheduled Sampling 是一种折中方案。

3.2 信息瓶颈问题

Seq2Seq 的致命弱点：所有输入信息都被压缩到一个固定长度的向量中(如 256 或 512 维)。对于长句子，这个向量无法承载所有信息——这就是信息瓶颈。

3.3 Teacher Forcing

训练时，解码器的输入不是自己上一步的预测，而是真实的上一个目标词(Teacher Forcing)。这加速训练但导致训练和推理的不一致(Exposure Bias)。

四、技术细节

💡 通俗类比：Beam Search(束搜索)就像"同时走多条路"——不是只走最可能的一条路(贪心)，而是同时探索K条最可能的路径，最终选最好的。K=5通常就足够好了。
⚠️ 常见误区：Beam Search 不保证找到全局最优解！它是贪心搜索的近似——在每一步保留K个候选，但可能在早期就排除了正确的路径。在实践中，beam_size=4-5是最佳平衡点。
🔬 深入探讨：BLEU 分数是机器翻译的标准评估指标——比较候选翻译和参考翻译的 n-gram 重叠度。但 BLEU 有很多缺陷：不考虑同义词、不理解语法正确性、对短翻译有偏。人类评估仍然是金标准，但昂贵且不可扩展。

五、使用场景

💡 通俗类比：Seq2Seq 的应用场景就像"各种翻译任务"——不只是语言翻译，还包括代码生成(自然语言→代码)、语音识别(音频→文字)、图像描述(图片→文字)。任何"一种序列→另一种序列"的转换都可以用 Seq2Seq。
⚠️ 常见误区：不要以为 Seq2Seq 只能处理"等长"的输入输出。翻译中"I love you"(3词)变成"我爱你"(3字)长度相近，但摘要任务可能把1000词压缩成100词——长度可以完全不同。
🔬 深入探讨：ChatGPT 本质上是一个超大规模的 Seq2Seq 模型——输入是用户的问题(序列)，输出是回答(序列)。从2014年的"固定向量瓶颈"到2023年的"万能对话助手"，Seq2Seq 的框架没变，但规模和细节发生了革命性变化。
  • 机器翻译：Google 翻译 2016 年用 Seq2Seq+Attention 替代统计方法
  • 文本摘要：长文章→短摘要
  • 对话系统：输入用户消息，输出回复
  • 代码生成：自然语言→代码
  • 语音识别：音频特征→文字

六、相似技术对比

💡 通俗类比：Seq2Seq vs Transformer 就像"老式翻译"vs"现代翻译"——老式翻译先听完整句再翻译(信息瓶颈)，现代翻译边听边对照原文翻译(注意力机制)。后者信息保留更完整。
⚠️ 常见误区：不要把"编码器-解码器"和"Seq2Seq"画等号。BERT 只有编码器(不生成序列)，GPT 只有解码器(自回归生成)，T5 才是编码器-解码器的 Transformer。架构选择取决于任务类型。
🔬 深入探讨：Seq2Seq 的信息瓶颈问题，在数学上等价于"把高维数据压缩到低维空间"——这就是"表征学习"的核心挑战。注意力机制的解决方案是"不压缩"——让解码器直接访问编码器的所有输出。这种"跳过压缩"的思想影响深远。

七、关联技术

💡 通俗类比：注意力机制和 Seq2Seq 的关系就像"翻译时查字典"——翻译每个词时，不是只看一遍原文(固定向量)，而是"回头看"原文中最相关的部分。这让翻译质量大幅提升。
⚠️ 常见误区：很多人以为 Transformer "取代"了 Seq2Seq。实际上 Transformer 仍然遵循编码器-解码器(或仅解码器)的 Seq2Seq 范式——改变的是内部实现(RNN→注意力)，不是整体框架。
🔬 深入探讨：从 Seq2Seq(2014)到 Transformer(2017)只有3年，但从"固定向量"到"全注意力"是质的飞跃。这3年间的关键里程碑：注意力机制(2014)、Pointer Network(2015)、ByteNet(2017)、Transformer(2017)——每一步都在让信息流动更自由。
  • Attention(模型12)：Seq2Seq 的直接升级，解决信息瓶颈
  • Transformer(模型14)：抛弃 RNN，纯注意力的 Encoder-Decoder
  • T5(模型17)：Encoder-Decoder Transformer，统一所有 NLP 任务

八、实际效果

💡 通俗类比：Seq2Seq 在2014年让机器翻译"从能用到好用"——BLEU 分数提升了约5-10分(翻译质量的显著提升)。但离人类翻译仍有差距——差距在于"理解"和"流畅度"。

⚠️ 常见误区：Seq2Seq 的"信息瓶颈"不只是"向量太小"

很多人以为瓶颈问题只是"256维向量装不下长句子"。实际上问题更深层：

  • 位置信息丢失：RNN 编码器的最后隐藏态主要编码了句子末尾的信息，开头的信息在传递过程中被"稀释"
  • 语义粒度不匹配：一个固定向量无法同时编码"主语是谁"、"做了什么"、"在哪里"、"什么时候"等多个语义维度
  • Attention 的本质解法：不是让向量更大，而是让解码器在生成每个词时能"回看"输入的所有位置——用动态加权替代固定压缩
🔬 深入探讨：Teacher Forcing 与 Exposure Bias

Teacher Forcing 是 Seq2Seq 训练的核心技巧，但它引入了一个根本性矛盾：

阶段解码器输入问题
训练时真实的上一个目标词 (y_{t-1})模型从未见过自己的错误
推理时自己预测的上一个词 (ŷ_{t-1})错误会累积——雪崩效应

Exposure Bias（曝光偏差）：训练时模型永远看到"正确答案"，但推理时必须处理自己的错误。就像一个学生只在有老师提示的情况下做题，考试时没人提示就懵了。

缓解方法：

  • Scheduled Sampling：训练时逐渐从"用真实词"过渡到"用模型预测词"
  • 强化学习：直接优化 BLEU 等评估指标而非交叉熵
  • 自回归预训练：GPT 的语言模型预训练天然没有 Exposure Bias

一、背景与动机的技术原理与数学推导： 从编码器-解码器原理看，编码器将变长输入压缩为固定长度上下文向量，解码器基于该向量生成输出。 推导：将整个源序列压缩为维度d的向量h，当源序列长度T≫d时，h无法承载全部信息——这就是信息瓶颈。 注意力机制通过让解码器在每步直接访问编码器所有隐状态来解决此问题。 这一数学本质是将信息瓶颈从O(d)扩展到O(T·d)， 本质上是解码器对编码器输出的动态加权聚合。

🔬 深入探讨：从 Seq2Seq(2014)到 Transformer(2017)只有3年，但从"固定向量"到"全注意力"是质的飞跃。这3年间的关键里程碑：注意力机制(2014)、Pointer Network(2015)、ByteNet(2017)、Transformer(2017)——每一步都在让信息流动更自由。
  • Attention(模型12)：Seq2Seq 的直接升级，解决信息瓶颈
  • Transformer(模型14)：抛弃 RNN，纯注意力的 Encoder-Decoder
  • T5(模型17)：Encoder-Decoder Transformer，统一所有 NLP 任务

  • 位置信息丢失：RNN 编码器的最后隐藏态主要编码了句子末尾的信息，开头的信息在传递过程中被"稀释"

三、核心原理

💡 通俗类比：编码器就像"听完整段话后在脑中形成印象"，解码器就像"根据印象逐词翻译"。问题是：印象(上下文向量)是一个固定长度的向量——如何把一整段话压缩成一个向量而不丢失信息？这就是注意力机制要解决的。
⚠️ 常见误区：原始 Seq2Seq 的"信息瓶颈"问题很严重——把整个输入序列压缩成一个固定向量(通常256-1024维)。长句子的信息损失巨大。注意力机制(2014)正是为了解决这个问题而生的。
🔬 深入探讨：Teacher Forcing 是 Seq2Seq 训练的重要技巧——解码时用"真实标签"而非"模型预测"作为下一步输入。这加速了训练但导致"暴露偏差"(Exposure Bias)——训练和推理的输入分布不一致。Scheduled Sampling 是一种折中方案。

3.2 信息瓶颈问题

Seq2Seq 的致命弱点：所有输入信息都被压缩到一个固定长度的向量中(如 256 或 512 维)。对于长句子，这个向量无法承载所有信息——这就是信息瓶颈。

3.3 Teacher Forcing

训练时，解码器的输入不是自己上一步的预测，而是真实的上一个目标词(Teacher Forcing)。这加速训练但导致训练和推理的不一致(Exposure Bias)。

四、技术细节

💡 通俗类比：Beam Search(束搜索)就像"同时走多条路"——不是只走最可能的一条路(贪心)，而是同时探索K条最可能的路径，最终选最好的。K=5通常就足够好了。
⚠️ 常见误区：Beam Search 不保证找到全局最优解！它是贪心搜索的近似——在每一步保留K个候选，但可能在早期就排除了正确的路径。在实践中，beam_size=4-5是最佳平衡点。
🔬 深入探讨：BLEU 分数是机器翻译的标准评估指标——比较候选翻译和参考翻译的 n-gram 重叠度。但 BLEU 有很多缺陷：不考虑同义词、不理解语法正确性、对短翻译有偏。人类评估仍然是金标准，但昂贵且不可扩展。

九、工程实现与优化技巧

💡 实践建议：实现 Seq2Seq 时，编码器的最终隐藏状态应作为解码器的初始状态。这个"信息传递"是整个架构的关键——编码器将整个输入序列压缩为一个固定长度的向量(context vector)，解码器基于这个向量逐步生成输出。

9.1 Beam Search 解码策略

贪心解码(greedy decoding)在每一步选择概率最高的词，但这不保证全局最优。Beam Search 维护 k 个最优候选序列(beam)，每一步对每个候选扩展所有可能的下一个词，保留总概率最高的 k 个。beam_size 通常为 4-10，越大结果越好但计算量越大。

Beam Search 的一个已知问题是生成的句子倾向于短而通用。Length Penalty 通过对较长序列给予奖励来缓解这个问题。Google 神经机器翻译系统使用了长度归一化：score(Y|X) = log P(Y|X) / |Y|^α，其中 α 通常取 0.6-0.7。

⚠️ 常见陷阱：Beam Search 在机器翻译中效果很好，但在开放域对话和故事生成中可能导致"退化重复"——模型反复生成安全但无趣的短句。这时需要使用 Nucleus Sampling (Top-p) 或 Temperature Sampling 来增加多样性。

9.2 Copy Mechanism 与 UNK 处理

标准 Seq2Seq 模型有一个严重问题：无法生成词汇表之外的词(OOV 问题)。在翻译人名、地名、专业术语时，模型只能输出 UNK 标记。Copy Mechanism(复制机制)允许模型从源序列中直接"复制"词到输出，解决了这个问题。

Pointer Network 是 Copy Mechanism 的一种实现：在每个解码步，模型计算一个"生成概率" pg，以概率 pg 从词汇表生成，以概率 1-pg 从输入序列中复制。CopyNet 和 Google 的 GNMT 都采用了类似的策略。

🔬 深入探讨：Copy Mechanism 的数学表达为 P(yₜ) = pg · P_gen(yₜ) + (1-pg) · Σ_{i:xᵢ=yₜ} αₜᵢ，其中 P_gen 是生成分布，αₜᵢ 是注意力权重。这统一了"生成"和"复制"两种模式，pg 通过 sigmoid 函数动态调整。这个思想后来被 GPT 等大模型的 in-context learning 所继承。

9.3 Scheduled Sampling 实现

Scheduled Sampling 的具体实现：训练时以概率 ε 使用真实标签(teacher forcing)，以概率 1-ε 使用模型上一步的预测作为输入。ε 从 1.0 开始，每个 epoch 按线性或指数衰减到最小值(如 0.1)。

Scheduled Sampling 训练流程: Epoch 1-5: ε=1.0 [真实]→[真实]→[真实]→[真实] (完全Teacher Forcing) Epoch 6-10: ε=0.8 [真实]→[真实]→[预测]→[真实] (偶尔用模型预测) Epoch 11-15: ε=0.5 [真实]→[预测]→[真实]→[预测] (混合模式) Epoch 16-20: ε=0.2 [预测]→[预测]→[真实]→[预测] (主要用模型预测) 推理时: ε=0 [预测]→[预测]→[预测]→[预测] (完全自主生成) 效果: 减少训练/推理分布不一致(exposure bias)

十、Seq2Seq 的演进与现代变体

10.1 从 LSTM Seq2Seq 到 Transformer

Seq2Seq 架构的演进路径清晰：最初用 RNN 作为编码器和解码器(2014)，然后加入注意力机制(2015)，再用 LSTM 替代普通 RNN 提升长距离依赖能力，最终 Transformer(2017)完全去掉了 RNN，只用注意力机制。每一步都是对前一步瓶颈的突破。

LSTM Seq2Seq 的核心瓶颈在于"信息压缩"——整个输入序列被压缩为一个固定长度的向量。注意力机制缓解了这个问题，但 RNN 的串行计算仍然是训练速度的瓶颈。Transformer 通过自注意力机制同时解决了信息压缩和并行化两个问题。

⚠️ 历史教训：Seq2Seq 的"信息瓶颈"问题告诉我们一个重要原则——不要将变长信息压缩到固定维度。这个教训影响了后续很多架构设计，如 Transformer 的注意力机制、Memory Network 的外部记忆等。

10.2 现代编码器-解码器模型

虽然纯解码器模型(GPT 系列)在大语言模型领域占据主导，但编码器-解码器架构并未消亡。T5、BART、mBART 等模型证明了编码器-解码器在翻译、摘要等任务上的优势。Flan-T5 和 UL2 等模型进一步展示了编码器-解码器在指令微调中的潜力。

编码器-解码器的优势在于：编码器可以双向理解输入(不像 GPT 只能从左到右)，解码器可以利用编码器的完整表示。这种"理解"和"生成"的分离在很多任务上比纯解码器更高效。

架构代表模型优势劣势
编码器-解码器T5, BART, mBART双向理解+序列生成参数量大(两倍)
纯解码器GPT, LLaMA, PaLM统一架构，Scaling 好单向理解
纯编码器BERT, RoBERTa理解能力强无法生成

# Seq2Seq 编码器-解码器
import torch, torch.nn as nn

class Encoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.lstm = nn.LSTM(embed_dim, hidden_dim, batch_first=True)
    def forward(self, x):
        _, (h, c) = self.lstm(self.embedding(x))
        return h, c

class Decoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.lstm = nn.LSTM(embed_dim, hidden_dim, batch_first=True)
        self.fc = nn.Linear(hidden_dim, vocab_size)
    def forward(self, x, h, c):
        out, (h, c) = self.lstm(self.embedding(x), (h, c))
        return self.fc(out), h, c
# Beam Search 解码
import heapq

def beam_search(model, src, beam_size=5, max_len=50):
    h, c = model.encoder(src)
    beams = [(0.0, [], h, c)]
    for _ in range(max_len):
        candidates = []
        for score, seq, h, c in beams:
            last = seq[-1] if seq else sos_token
            out, h_new, c_new = model.decoder(last, h, c)
            log_probs = torch.log_softmax(out, dim=-1)
            topk = log_probs.topk(beam_size)
            for i in range(beam_size):
                new_score = score + topk.values[0, i].item()
                new_seq = seq + [topk.indices[0, i].item()]
                candidates.append((new_score, new_seq, h_new, c_new))
        beams = heapq.nlargest(beam_size, candidates, key=lambda x: x[0])
    return beams[0]
# Teacher Forcing 比率衰减
def get_teacher_forcing_ratio(epoch, total_epochs, start=1.0, end=0.1):
    return start - (start - end) * epoch / total_epochs

# 训练时使用
tf_ratio = get_teacher_forcing_ratio(epoch, 20)
if random.random() < tf_ratio:
    input_token = target[t]  # 使用真实标签
