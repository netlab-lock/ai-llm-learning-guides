🔲 2. 张量并行 (Tensor Parallelism)

2.1 核心思想与数学原理

张量并行（Tensor Parallelism, TP）是将单个矩阵运算切分到多张GPU上并行执行的策略。与数据并行不同，张量并行切分的是模型本身——具体来说，是模型中的大型权重矩阵。这种并行方式的核心动机是：当模型的单层参数量太大，无法放进单张GPU的显存时，必须将该层的计算分摊到多张GPU上。

💡 通俗类比
张量并行就像把一块超大的蛋糕切成几份，每个厨师负责做其中一份。关键问题是：每份蛋糕的原料（输入数据）是相同的，但各自处理不同的部分。做完后需要把结果拼起来（AllReduce）。因为每步都需要拼接，厨师之间必须坐得很近（NVLink），否则传递太慢。

张量并行的数学基础是矩阵分块乘法。设一个线性层的计算为 Y = XW，其中 X 是输入矩阵（形状 [b, d]），W 是权重矩阵（形状 [d, n]）。如果我们将 W 按列分成两块 W = [W1, W2]，则：

Y = X · [W1, W2] = [X·W1, X·W2] = [Y1, Y2]

这意味着每张GPU可以独立计算 Y1 和 Y2，最后将结果拼接起来。这种按列切分的方式不需要任何通信（除了最后的拼接），因为输入 X 在所有GPU上是相同的。但是，当下一层需要完整的 Y 作为输入时，就需要一次 AllReduce 操作来聚合结果。

FFN 层的张量并行 (Megatron-LM 风格): 原始计算: Y = GeLU(X · W1) · W2 其中 W1: [d, 4d], W2: [4d, d] ┌─────────────────────────────────────────────────────────────┐ │ GPU 0 (前半): │ │ A1 = X · W1[:, :2d] ← 列切分 W1 的左半 │ │ B1 = GeLU(A1) ← 激活函数 │ │ Y0 = B1 · W2[:2d, :] ← 行切分 W2 的上半 │ │ │ │ GPU 1 (后半): │ │ A2 = X · W1[:, 2d:] ← 列切分 W1 的右半 │ │ B2 = GeLU(A2) ← 激活函数 │ │ Y1 = B2 · W2[2d:, :] ← 行切分 W2 的下半 │ │ │ │ 结果: Y = Y0 + Y1 ← AllReduce (求和) │ │ │ │ 通信量: 每层 2 × AllReduce (前向), 2 × AllReduce (反向) │ │ 显存: 每卡只存一半权重 │ └─────────────────────────────────────────────────────────────┘ 关键: W1 按列切分, W2 按行切分, 这样只需 1 次 AllReduce 而非 2 次 (GeLU 前后各一次)

2.2 注意力层的张量并行

Multi-Head Attention（MHA）天然适合张量并行，因为每个注意力头可以独立计算。在标准的 Transformer 架构中，注意力层包含四个权重矩阵：W_Q、W_K、W_V（投影矩阵）和 W_O（输出投影矩阵）。张量并行的切分策略如下：

  • W_Q、W_K、W_V：按列切分。每张 GPU 持有部分注意力头的投影权重。例如，对于 32 头注意力，2 张 GPU 各持有 16 个头的 Q、K、V 投影。
  • W_O：按行切分。每张 GPU 持有 W_O 的部分行，与各自的注意力输出相乘后，通过 AllReduce 聚合得到最终输出。
💡 关键洞察
Multi-Head Attention 天然适合张量并行——每个头可以独立计算，这和数学上的分块矩阵乘法完全一致。Q、K、V 的投影矩阵可以按列切分（每卡处理部分列），而输出投影 W_O 需要按行切分。这种"列切+行切"的交替模式保证了每层只需要一次 AllReduce，而不是三次。Megatron-LM 的论文详细描述了这个优化。
# Multi-Head Attention 张量并行示例
# 假设: 64个头, TP=2, 每卡32个头

# GPU 0:
Q0 = X @ W_Q[:, :d_head*32]     # 32个头的Q
K0 = X @ W_K[:, :d_head*32]     # 32个头的K
V0 = X @ W_V[:, :d_head*32]     # 32个头的V
Attn0 = softmax(Q0 @ K0.T / √d) @ V0  # 32个头独立计算
Y0 = Attn0 @ W_O[:d_head*32, :] # 行切分 W_O

# GPU 1:
Q1 = X @ W_Q[:, d_head*32:]
K1 = X @ W_K[:, d_head*32:]
V1 = X @ W_V[:, d_head*32:]
Attn1 = softmax(Q1 @ K1.T / √d) @ V1
Y1 = Attn1 @ W_O[d_head*32:, :]

# 结果: Y = AllReduce(Y0 + Y1)
# 每层仅 1 次 AllReduce (前向), 1 次 AllReduce (反向)

Flash Attention 与张量并行的配合

Flash Attention 是一种内存高效的注意力计算算法，通过分块（tiling）技术避免存储完整的注意力矩阵。在张量并行场景下，Flash Attention 的分块策略与 TP 的头切分完美配合：每个GPU只需要计算自己负责的那些头的注意力，使用 Flash Attention 的分块算法在 SRAM 中完成 softmax 和矩阵乘法，无需额外的通信。这意味着 Flash Attention 和张量并行是正交的优化——两者可以同时使用，效果叠加。

🔬 深入探讨

列切分与行切分的数学证明

为什么 FFN 层需要"列切+行切"的交替模式？这是一个精妙的数学优化。考虑两层连续的线性变换：

Y = f(X · W1) · W2

如果 W1 按列切分：X · [W1a, W1b] = [X·W1a, X·W1b]，得到两个分片结果。激活函数 f（如 GeLU）是逐元素操作，可以分别应用。现在关键问题是 W2 如何切分。

如果 W2 也按列切分：[f(X·W1a), f(X·W1b)] · [W2a, W2b]，结果需要 4 个子矩阵的拼接，通信复杂度增加。但如果 W2 按行切分：[f(X·W1a), f(X·W1b)] · [W2a; W2b] = f(X·W1a)·W2a + f(X·W1b)·W2b，结果是一个简单的求和，只需一次 AllReduce！

这种"列切-行切"的交替模式确保了每对连续层只需要一次 AllReduce，通信量与 GPU 数量无关。Megatron-LM 论文中的定理证明了这是最优的切分策略。

2.3 通信需求与硬件约束

张量并行的通信模式与数据并行截然不同。TP 需要每层 2 次 AllReduce（FFN + Attention 各一次），通信量大但每步的通信量较小（只传输激活值，不传输梯度）。这使得 TP 对通信带宽极其敏感——任何带宽不足都会直接导致GPU空闲等待。

⚠️ 常见误区
张量并行不是"越多卡越快"。通信量随 GPU 数线性增长，而计算量只是被分摊。当 GPU 数量超过单机 NVLink 互联的上限（通常 8 卡），跨机器的网络带宽（InfiniBand ~50 GB/s vs NVLink ~600 GB/s）会成为严重瓶颈。因此张量并行通常限制在单机 8 卡以内，跨机器用流水线并行。
张量并行的通信模式 (每层前向传播): ┌────────────────────────────────────────────────────────┐ │ 时间轴 → │ │ │ │ GPU 0: [计算 Q0K0V0] → [AllReduce] → [计算 Attn0] │ │ → [计算 Y0] │ │ ↕ │ │ AllReduce │ │ ↕ │ │ GPU 1: [计算 Q1K1V1] → [AllReduce] → [计算 Attn1] │ │ → [计算 Y1] │ │ │ │ FFN 层同理: [计算 GeLU(X·W1)] → AllReduce → 输出 │ │ │ │ 每层总计: 2 × AllReduce (前向) + 2 × AllReduce (反向)│ │ 通信量: 4 × batch_size × seq_len × hidden_size × 2B │ │ │ │ ⚠ AllReduce 是同步操作, 最慢的 GPU 决定整体速度 │ └────────────────────────────────────────────────────────┘

NVLink (A100): 600 GB/s vs PCIe 4.0: 32 GB/s — 快 ~20x

2.4 Megatron-LM 的 TP 实现

NVIDIA 的 Megatron-LM 是张量并行的参考实现。Megatron-LM 的 TP 实现包含以下关键优化：

  • 融合内核（Fused Kernel）：将 GeLU/层归一化与矩阵乘法融合，减少 GPU kernel 启动开销和中间结果的显存读写。
  • 通信-计算重叠：在 AllReduce 传输数据的同时，GPU 可以继续计算不依赖于 AllReduce 结果的部分。
  • 序列并行（Sequence Parallelism）：将 LayerNorm 和 Dropout 沿序列维度切分，进一步减少显存占用。
  • Ring Attention：对于非常长的序列，将 KV 沿序列维度分布在不同 GPU 上，通过 Ring 通信模式计算注意力。
💡 实践建议
使用 Megatron-LM 进行张量并行训练时，需要注意以下几点：(1) TP 度必须能整除注意力头数和 FFN 中间层维度；(2) 确保 GPU 之间有 NVLink 互联（通过 nvidia-smi topo -m 检查）；(3) 使用 --sequence-parallel 开启序列并行以节省显存；(4) 对于超过 8 卡的训练，TP 限制在单机内，跨机器使用流水线并行。

张量并行的扩展：2D 和 2.5D 并行

标准的张量并行是一维切分——将权重矩阵沿一个维度分成 N 份。研究者还提出了 2D 和 2.5D 并行方案，将矩阵沿两个维度切分，以减少通信量。

2D 并行：将权重矩阵按行和列同时切分，组织成 √N × √N 的 GPU 网格。通信量从 O(N) 降低到 O(√N)，但需要更复杂的通信模式（行列交替通信）。

2.5D 并行：在 2D 的基础上增加一个"复制"维度，进一步减少通信量。理论上可以将通信量降低到 O(N^(1/3))，但实现复杂度高，实际使用较少。

2.5 张量并行的最新进展

近年来，张量并行技术出现了几个重要的新方向：

Ring Attention：由 UC Berkeley 提出，用于处理超长序列（百万 token 级别）。Ring Attention 将 KV 沿序列维度分布在多张 GPU 上，每张 GPU 只持有部分 KV。计算注意力时，KV 块在 GPU 之间以 Ring 方式传递，每个 GPU 对当前的 KV 块计算局部注意力并累积结果。通信可以与计算重叠，因此理论上序列长度可以无限扩展。

Context Parallelism：类似 Ring Attention，但由 NVIDIA 集成到 Megatron-LM 中。它将序列维度切分到多张 GPU 上，支持多种注意力计算模式（Ring、All-to-All 等），并与 TP、PP、DP 兼容。

Expert Parallelism (EP)：针对 MoE（Mixture of Experts）模型的特殊并行方式。每个专家可以放在不同的 GPU 上，输入通过路由机制分配到对应的专家 GPU。EP 本质上是一种特殊的张量并行——将 MoE 层的权重按专家维度切分。

🔬 深入探讨

Ring Attention 的工作原理

Ring Attention 解决的核心问题是：当序列长度 L 非常大时，注意力矩阵 QK^T 的大小为 O(L²)，单卡显存无法容纳。Ring Attention 通过将 KV 分布在 N 张 GPU 上，每张 GPU 只持有 L/N 的 KV 块。

计算过程：(1) 每张 GPU 持有完整的 Q（L/N 行）和初始的 KV 块；(2) 使用 Flash Attention 的在线 softmax 算法计算当前 KV 块的注意力；(3) 将 KV 块发送给下一张 GPU（Ring 通信）；(4) 接收上一张 GPU 的 KV 块，继续计算并累积结果；(5) 经过 N-1 轮后，每张 GPU 完成所有 KV 的注意力计算。

关键优化：KV 的传输可以与当前块的注意力计算重叠，只要通信时间 ≤ 计算时间，就不会有额外开销。这要求每个 KV 块的计算时间 ≥ 传输时间，即 L/N × d ≥ L × d × sizeof(dtype) / bandwidth，简化后得到：GPU 数量 N ≤ bandwidth × sizeof(dtype)。对于 NVLink 600 GB/s 和 BF16（2 bytes），N ≤ 300，足够支持大多数场景。

⚠️ 张量并行的限制
张量并行有几个重要的限制需要了解：(1) TP 度必须能整除注意力头数（num_attention_heads）和 FFN 中间层维度（通常为 hidden_size × 4 / TP 度也必须是整数）；(2) TP 引入的 AllReduce 是同步操作，任何负载不均衡都会导致 GPU 空闲；(3) 对于 MoE 模型，TP 只能切分注意力层，MoE 层的专家需要用 EP 切分；(4) TP 不适用于 CPU 内存不足的场景——它只减少 GPU 显存，不减少 CPU 内存需求。

# 练习参考: 手动实现 2-GPU 张量并行
import torch
import torch.distributed as dist

def tensor_parallel_linear(x, weight_full, rank, world_size):
    # 按列切分权重
    chunk_size = weight_full.shape[1] // world_size
    weight_chunk = weight_full[:, rank*chunk_size:(rank+1)*chunk_size]
    
    # 各 GPU 独立计算
    y_local = x @ weight_chunk  # [batch, chunk_size]
    
    # AllReduce 聚合结果
    dist.all_reduce(y_local, op=dist.ReduceOp.SUM)
    return y_local

💡 张量并行的调试技巧
在实际调试张量并行训练时，最常遇到的问题是梯度不一致和通信死锁。建议使用以下方法排查：(1) 先用 TP=1 跑通基线，确认单卡逻辑正确；(2) 设置 NCCL_DEBUG=INFO 和 NCCL_DEBUG_SUBSYS=ALL 查看详细通信日志；(3) 使用 torch.distributed.barrier() 在关键位置同步，定位哪一步出了问题；(4) 对比 TP=1 和 TP=2 的每层输出，找出第一个不一致的层。常见错误包括：权重切分维度不正确、AllReduce 在错误的进程组上执行、以及未正确处理 bias 的切分方式。此外，使用 torch.profiler 可以可视化 TP 通信与计算的时间线，帮助识别通信瓶颈和空闲等待。在混合精度训练中还需要特别注意：FP16/BF16 的 AllReduce 精度损失可能在深层网络中累积，建议对关键的梯度聚合操作使用 FP32 精度。此外，torch.compile 对 TP 模型的支持仍在改进中，目前建议先验证兼容性再启用编译优化。随着 H100 和 H200 等新一代 GPU 的普及，NVLink 带宽进一步提升到 900 GB/s，TP 的通信开销将进一步降低，使得张量并行在更大规模的训练中仍然保持实用价值。

这些数据也解释了为什么在实际训练中，TP 度通常设为 8（单机 8 卡）而非更大。如果需要更多 GPU 参与计算，应该通过 DP 和 PP 扩展，而非增加 TP 度。DeepSeek-V3 的 TP=1 配置是一个极端案例——通过 MLA 架构减少了注意力层的参数量，使得单卡就能容纳注意力层的计算，从而完全消除了 TP 通信开销。在实际部署中，建议先用小模型（如 1B 参数）验证 TP 配置的正确性和性能，再扩展到目标模型规模。使用 torch.profiler 或 NVIDIA Nsight Systems 可以精确分析 TP 通信和计算的时间占比，帮助找到最优的并行配置。此外，NCCL 的环境变量如 NCCL_DEBUG=INFO 可以输出详细的通信日志，帮助诊断网络问题和通信瓶颈。

# 核心思想与数学原理 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
深入探讨： 从张量并行的核心原理出发，将矩阵运算拆分到多个GPU上并行计算。为什么MLP层可以按列或按行切分？推导：对于Y=XW，将W按列切分为[W₁,W₂]，则Y=X[W₁,W₂]=[XW₁,XW₂]，两张GPU分别计算后拼接即可。注意力层的Q/K/V矩阵同样可按列切分。数学本质上，张量并行将单个矩阵乘法的时间复杂度从O(m·n)降至O(m·n/N)，但要求GPU间高速互联（如NVLink），因为每层前向后需要一次All-Reduce同步。
常见误区： 学习张量并行时，很多人只看矩阵切分图不实际测量通信量。建议分析张量并行中每层前向和反向的All-Reduce通信量，理解通信量与隐藏维度和GPU数量的关系。
常见误区： 不要以为张量并行就是简单的矩阵切分可以直接上手多卡训练。切分维度选择、通信原语选择（All-Reduce vs Reduce-Scatter+All-Gather）等细节影响性能。
