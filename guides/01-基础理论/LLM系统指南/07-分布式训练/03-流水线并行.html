🔄 3. 流水线并行 (Pipeline Parallelism)

3.1 核心思想与动机

流水线并行（Pipeline Parallelism, PP）是将模型的不同层分配到不同GPU上，数据在GPU之间依次传递的并行策略。这种并行方式的核心动机是：当模型层数太多，单张GPU无法容纳所有层时，必须将模型"竖着切"成多个阶段（stage），每个阶段放在不同的GPU上。从数学原理上分析，流水线并行将模型的L层均匀分配到P个stage，每个stage处理L/P层，通过推导可知总通信量与P成正比但每步通信量与P无关。

💡 通俗类比
流水线并行就像工厂的流水线——每个工位（GPU）负责一道工序（几层模型），半成品在工位之间传递。问题是：如果只有一个产品在流水线上，大部分工位都在空等。解决方案是同时放入多个产品（micro-batch），让每个工位都有活干。这就是 GPipe 的核心思想。

与张量并行相比，流水线并行有一个关键优势：通信量小。TP 每层需要 AllReduce（通信量与 batch_size × seq_len × hidden_size 成正比），而 PP 只需要在层间传递激活值（通信量同样与 batch_size × seq_len × hidden_size 成正比，但只在相邻 stage 之间传递，而非所有 GPU 之间）。这使得 PP 非常适合跨机器的场景，因为跨机器的通信带宽远低于机器内（NVLink）。

3.2 Pipeline 调度策略

朴素的流水线并行存在严重的 GPU 利用率问题：如果只有一个 micro-batch 在流水线上，那么任何时候只有一个 GPU 在工作，其余 GPU 都在空闲。这种空闲时间被称为"流水线气泡"（pipeline bubble）。为了减少气泡，研究者提出了多种调度策略。

GPipe 调度

GPipe（Google, 2019）是最直观的调度策略：将一个大 batch 分成 M 个 micro-batch，依次送入流水线。当所有 micro-batch 的前向传播完成后，再依次进行反向传播。其工作机制可以通过数学公式表达：气泡比例 = (P-1)/(P+M-1)，这个公式的推导基于流水线的时序分析，证明了当M趋向无穷时气泡比例趋近于零。

GPipe 调度 (P=4 stages, M=8 micro-batches): 时间 → GPU 0: [F0][F1][F2][F3][F4][F5][F6][F7][B7][B6][B5][B4][B3][B2][B1][B0] GPU 1: [F0][F1][F2][F3][F4][F5][F6][F7][B7][B6][B5][B4][B3][B2][B1][B0] GPU 2: [F0][F1][F2][F3][F4][F5][F6][F7][B7][B6][B5][B4][B3][B2][B1][B0] GPU 3: [F0][F1][F2][F3][F4][F5][F6][F7][B7][B6][B5][B4][B3][B2][B1][B0] 气泡比例: (4-1)/(4+8-1) = 3/11 ≈ 27% F = 前向传播, B = 反向传播 数字表示 micro-batch 编号 问题: 所有前向完成才开始反向, 显存峰值高 (需保存所有激活值)

1F1B 调度

1F1B（One Forward One Backward）是 Megatron-LM 和 PipeDream 提出的优化调度策略。其核心思想是：一旦流水线被填满，每完成一个 micro-batch 的前向传播后立即进行一个 micro-batch 的反向传播。这样可以尽早释放激活值，将显存峰值从 O(M) 降低到 O(P)。

1F1B 调度 (P=4 stages, M=8 micro-batches): 时间 → GPU 0: [F0][F1][F2][F3][B0][F4][B1][F5][B2][F6][B3][F7][B4][B5][B6][B7] GPU 1: [F0][F1][F2][F3][B0][F4][B1][F5][B2][F6][B3][F7][B4][B5][B6][B7] GPU 2: [F0][F1][F2][F3][B0][F4][B1][F5][B2][F6][B3][F7][B4][B5][B6][B7] GPU 3: [F0][F1][F2][F3][B0][F4][B1][F5][B2][F6][B3][F7][B4][B5][B6][B7] 气泡比例: 与 GPipe 相同 = (P-1)/(P+M-1) 但显存峰值: O(P) 而非 O(M) ← 关键优势! 前向传播和反向传播交替执行, 及时释放激活值
⚠️ 常见误区
流水线并行的"气泡"（bubble）是不可避免的——每个 micro-batch 在进入和离开流水线时都会造成 GPU 空闲。气泡比例 ≈ (pipeline_stages - 1) / (pipeline_stages + micro_batches - 1)。因此 micro-batch 数量应该远大于 pipeline stage 数量，才能让气泡比例可接受。1F1B 调度策略可以进一步减少气泡的显存峰值。

Interleaved 1F1B 调度

Megatron-LM 进一步提出了 Interleaved 1F1B 调度：将每个 GPU 分配多个不连续的层组（model chunk），而非一个连续的层组。例如，4 个 GPU、2 个 model chunk 时，GPU 0 持有 Layer 0-3 和 Layer 16-19，GPU 1 持有 Layer 4-7 和 Layer 20-23，以此类推。这种方式可以将气泡比例降低到 (P-1) / (P + kM - 1)，其中 k 是 model chunk 数量。

💡 调度策略选择
选择流水线调度策略时的实用指南：
• GPipe：实现简单，适合 micro-batch 数量远大于 stage 数的场景
• 1F1B：显存效率最高，适合显存紧张的场景
• Interleaved 1F1B：气泡最少，但实现复杂，适合超多 stage 的场景
• Virtual Pipeline：Interleaved 的另一种称呼，Megatron-LM 中使用此术语

3.3 气泡比例的数学分析

流水线气泡是衡量流水线效率的关键指标。以下是各种调度策略的气泡比例公式推导：

GPipe / 1F1B 气泡比例 = (P - 1) / (P + M - 1)
Interleaved 1F1B 气泡比例 = (P - 1) / (P + kM - 1)
其中: P = 流水线阶段数, M = micro-batch 数, k = model chunk 数

以 P=16, M=64 为例：GPipe 气泡比例 = 15/79 ≈ 19%；Interleaved (k=4) 气泡比例 = 15/(16+256-1) ≈ 5.7%。可以看到，增加 model chunk 数量可以显著减少气泡，但代价是更多的 P2P 通信和更复杂的调度逻辑。

1F1B(P-1)/(P+M-1)O(P × 层大小)中Megatron-LM
Interleaved 1F1B(P-1)/(P+kM-1)O(kP × 层大小)高Megatron-LM

🔬 深入探讨

Zero Bubble Pipeline Parallelism

2024 年的研究提出了 Zero Bubble Pipeline Parallelism，目标是将气泡比例降至接近 0%。其核心思想是将反向传播拆分为两个独立的操作：B（计算输入梯度 dX）和 W（计算权重梯度 dW）。传统方法中 B 和 W 是绑定在一起的，但 Zero Bubble 方法发现可以将 W 延迟执行——只要在参数更新之前完成即可。

通过精心调度 F、B、W 三种操作的执行顺序，Zero Bubble 方法可以将气泡填充到几乎为零。实验表明，在 8 个 stage、16 个 micro-batch 的配置下，Zero Bubble 方法的气泡比例从 1F1B 的 30% 降低到不到 2%。但代价是实现复杂度极高，需要修改反向传播的底层逻辑，目前尚未在主流框架中广泛使用。

Zero Bubble 方法的数学证明依赖于一个关键观察：在 1F1B 调度中，气泡主要产生在流水线的启动和排空阶段。如果将 W 操作从 B 中分离出来，W 可以在气泡期间执行，从而"填充"气泡。总操作数为 P × M × 3（F、B、W 各 P × M 次），而时间槽总数为 P × M × 3 - (P-1) × M × 2 ≈ P × M × 3，因此气泡比例趋近于 0。

3.4 通信与显存分析

流水线并行的通信模式与张量并行完全不同。PP 只需要在相邻 stage 之间传递激活值（前向）和梯度（反向），通信量与 batch_size × seq_len × hidden_size 成正比，但与模型层数无关。

⚠️ 通信注意事项
流水线并行的 P2P 通信虽然量小，但延迟敏感。每个 micro-batch 在进入下一个 stage 之前必须等待当前 stage 的计算完成。如果 stage 之间的网络延迟高（如跨数据中心），流水线效率会急剧下降。此外，P2P 通信的启动开销（kernel launch + 网络握手）对小消息的影响很大——当 hidden_size 较小或 batch_size 较小时，通信开销可能超过计算时间。

切分方式切分矩阵（列/行）切分层（stage）
通信频率每层 2 次 AllReduce每 micro-batch 每 stage 边界 1 次 P2P
通信量大（激活值 × 层数 × 2）小（仅层间输出 × micro-batch 数）

PP 的显存优化

流水线并行的显存优化主要集中在两个方面：(1) 减少流水线气泡导致的显存浪费；(2) 优化激活值的存储策略。

激活值存储策略：在 GPipe 中，所有 micro-batch 的激活值需要在前向传播完成后才能开始反向传播，显存峰值为 O(M × 激活大小)。1F1B 通过交替执行前向和反向，将显存峰值降低到 O(P × 激活大小)。Activation Checkpointing 可以进一步将激活值存储降低到 O(P)（只保存每个 stage 的输入激活值）。

虚拟流水线（Virtual Pipeline）：将每个 GPU 分配多个不连续的层组后，每个 GPU 只需同时保存较少的激活值（因为不同 micro-batch 的不同层组交替执行），但通信次数增加。

💡 实践建议
使用流水线并行时的关键实践建议：
1. stage 数量：通常设为单机 GPU 数量（4 或 8），跨机器的 stage 通过网络连接
2. micro-batch 数量：至少为 stage 数量的 4-8 倍，以保持气泡比例 < 20%
3. 层分配：尽量让每个 stage 的计算量均匀（不均匀会导致木桶效应）
4. 通信优化：使用 --overlap-p2p-communication 开启通信-计算重叠
5. 与 TP 结合：单机内用 TP=8，跨机器用 PP，这是最常见的 3D 并行配置

P2P 通信的实现细节

流水线并行的 P2P 通信使用 NCCL 的 Send/Recv 操作。在 Megatron-LM 中，P2P 通信有两种模式：

  • ISend/IRecv（异步）：发送和接收操作立即返回，通过 Wait 操作等待完成。允许通信与计算重叠。
  • Send/Recv（同步）：发送和接收操作阻塞直到完成。实现简单但无法重叠。

Megatron-LM 默认使用异步 P2P，并通过 --overlap-p2p-communication 开启通信-计算重叠。在 1F1B 调度的稳定阶段，当前 micro-batch 的计算可以与下一个 micro-batch 的 P2P 通信同时进行，从而进一步减少气泡。

🔬 深入探讨

流水线并行与张量并行的通信对比

从通信模式的角度，PP 和 TP 是互补的：

TP 的通信特征：每次 AllReduce 的通信量 = 2 × (N-1)/N × batch_size × seq_len × hidden_size × sizeof(dtype)。对于 32 层的模型，前向传播需要 32 × 2 = 64 次 AllReduce，反向传播同样 64 次，总计 128 次 AllReduce。通信量大但可以利用 NVLink 的高带宽。

PP 的通信特征：每个 micro-batch 在每个 stage 边界只需要一次 P2P Send + 一次 P2P Recv。对于 M 个 micro-batch、P 个 stage，前向传播需要 M × (P-1) 次 P2P 通信，反向传播同样 M × (P-1) 次。通信量小但延迟敏感。

在 3D 并行中的分工：TP 放在 NVLink 互联的单机内（通信量大但带宽高），PP 跨机器（通信量小但带宽低），DP 也跨机器但通信可以与计算重叠。这种分工充分利用了不同层级网络的特性。

# 练习参考: 手动 2-stage 流水线
import torch
import torch.distributed as dist

def pipeline_forward(model_chunk, x, stage, world_size):
    if stage > 0:
        # 接收上一个 stage 的输出
        recv_buf = torch.empty_like(x)
        dist.recv(recv_buf, src=stage - 1)
        x = recv_buf
    
    # 计算当前 stage 的层
    x = model_chunk(x)
    
    if stage < world_size - 1:
        # 发送给下一个 stage
        dist.send(x, dst=stage + 1)
    
    return x

从数据可以看出几个重要规律：(1) 气泡比例与 P/M 比值正相关——保持 M >> P 是减少气泡的关键；(2) Interleaved 1F1B 可以将气泡降低 2-3 倍，但实现复杂度高；(3) 当 PP=16 时，即使 M=64 气泡仍接近 20%，需要 M=128 才能降到 10% 左右；(4) 显存峰值与调度策略密切相关——1F1B 的 O(P) 远优于 GPipe 的 O(M)。

实际训练中的另一个重要考量是流水线的负载均衡。如果各 stage 的计算时间不一致（例如某些层有 MoE 而其他层没有），最慢的 stage 会成为瓶颈。解决方法包括：(1) 手动调整每 stage 的层数，使计算量尽量均衡；(2) 使用动态调度策略，在运行时调整 micro-batch 的分配；(3) 对于 MoE 模型，将 MoE 层和 Dense 层交替放置，使每个 stage 的平均计算量相近。

# 核心思想与动机 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
深入探讨： 从流水线并行的核心原理出发，将模型按层划分到不同GPU上，形成流水线式的前向-反向计算。为什么朴素流水线并行效率低？推导：若模型有L层分到N个GPU，每个GPU处理L/N层。朴素方式下同一时刻只有一张GPU在计算，GPU利用率仅为1/N。GPipe通过micro-batch将数据切分为M份，使流水线上同时有M个micro-batch在不同阶段运行。数学本质上，流水线气泡比为(N-1)/(M+N-1)，当M≫N时气泡可忽略不计。
常见误区： 学习流水线并行时，很多人只看时序图不实际计算气泡比。建议根据模型层数和micro-batch数计算理论气泡比，并与实际训练的GPU利用率对比验证。
常见误区： 不要以为流水线并行就是简单地把模型切分到不同GPU上。切分点选择、micro-batch大小、调度策略（1F1B vs Interleaved）等都影响效率。
