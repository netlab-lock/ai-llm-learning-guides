📊 1. 数据并行与 FSDP

1.1 数据并行的基本原理

数据并行（Data Parallelism, DP）是分布式训练中最基本也是最广泛使用的并行策略。其核心思想非常直观：将训练数据分成多份，每份交给一张GPU处理，每张GPU持有完整的模型副本，独立完成前向传播和反向传播后，通过集合通信操作同步梯度，最后各自更新参数。这种并行方式的优势在于实现简单、扩展性好，并且不会改变模型的计算逻辑，因此训练结果与单卡训练完全等价。

💡 通俗类比
数据并行就像一个餐厅有 8 个厨师，每人拿到一模一样的菜谱（完整模型副本），但处理不同的订单（不同数据批次）。做完后大家互相交流心得（AllReduce 同步梯度），确保每个人的菜谱更新一致。简单高效，但前提是每个厨师的厨房（GPU 显存）要放得下完整的菜谱。

在标准的数据并行训练中，每张GPU需要存储三部分内容：模型参数（以FP16/BF16存储，占2字节/参数）、梯度（与参数同等大小）、以及优化器状态（Adam优化器需要额外存储一阶动量和二阶动量，各占4字节/参数）。以一个7B参数的模型为例，单卡显存占用的详细计算如下：模型参数为7B × 2 bytes = 14 GB，梯度为7B × 2 bytes = 14 GB，Adam优化器状态为7B × 4 bytes × 2 = 56 GB，总计约84 GB。这意味着即使是7B模型，单张A100 80GB显卡也只能勉强放下，而更大的模型则完全无法使用标准DDP训练。

AllReduce 的实现原理

AllReduce是数据并行中最核心的通信原语。其作用是将所有GPU上的梯度张量进行归约（通常是求平均），然后将结果广播回所有GPU。最常见的实现算法是Ring AllReduce，其工作原理如下：假设有N张GPU，每张GPU持有一个大小为M的梯度张量。Ring AllReduce将每个张量分成N份，通过N-1轮的ReduceScatter操作和N-1轮的AllGather操作完成归约。总通信量为2(N-1)/N × M，当N足够大时趋近于2M，即与GPU数量无关，实现了带宽最优。

🔬 深入探讨

Ring AllReduce 的数学分析

Ring AllReduce 的通信复杂度分析如下：设共有 N 张 GPU，每张 GPU 持有大小为 M 的数据。在 ReduceScatter 阶段，数据被分成 N 块，每块大小为 M/N。每轮传输中，每张 GPU 向环中下一张 GPU 发送一块数据，同时接收上一张 GPU 的一块数据并累加。经过 N-1 轮后，每张 GPU 持有完整归约结果的 1/N。AllGather 阶段类似，经过 N-1 轮后每张 GPU 持有完整的归约结果。

总通信量 = 2 × (N-1) × (M/N) ≈ 2M（当 N 较大时）。这个结果表明 Ring AllReduce 的通信量与 GPU 数量几乎无关，是带宽最优的算法。然而，Ring AllReduce 的延迟为 2(N-1) × α，其中 α 是每次传输的启动延迟。对于小消息，延迟可能成为瓶颈，此时树形（Tree）AllReduce 更优。

NCCL（NVIDIA Collective Communications Library）在实际实现中会根据消息大小、GPU 数量和拓扑结构自动选择最优的 AllReduce 算法。对于大于几MB的消息，NCCL 通常选择 Ring 或 Tree 算法的混合变体；对于小消息，则可能使用简单的广播方式。NCCL 2.x 引入了对 NVLink、NVSwitch 和 InfiniBand 的原生支持，能够自动检测硬件拓扑并优化通信路径。

DDP 的梯度同步策略

PyTorch 的 DistributedDataParallel（DDP）是数据并行的标准实现。DDP 提供了两种梯度同步策略：

  • 同步数据并行（Synchronous DP）：所有GPU必须完成梯度计算后才能进行参数更新。优点是训练结果确定且等价于单卡大Batch训练；缺点是存在木桶效应，最慢的GPU决定了整体速度。
  • 异步数据并行（Asynchronous DP）：各GPU独立更新参数，无需等待其他GPU。优点是无木桶效应；缺点是梯度过时（stale gradient）问题，可能导致训练不稳定。

现代大模型训练几乎都采用同步数据并行，因为训练稳定性远比速度重要。DDP 还引入了一个重要优化：梯度分桶（Gradient Bucketing）。DDP 将参数的梯度分成多个桶（默认25MB），当一个桶的所有梯度就绪后立即启动该桶的 AllReduce，而不是等待所有梯度计算完成。这种流水线化的处理方式可以将通信与计算重叠，显著提高GPU利用率。

💡 实践建议
在使用 DDP 时，有几个关键的性能优化技巧：(1) 设置 find_unused_parameters=False（默认值），除非模型有动态计算图；(2) 使用 gradient_as_bucket_view=True 减少内存拷贝；(3) 确保数据加载器使用 DistributedSampler，避免各GPU处理相同数据；(4) 使用梯度累积来等效增大 Batch Size，减少通信频率。这些优化综合起来可以将训练吞吐量提升 10-30%。

1.2 ZeRO (Zero Redundancy Optimizer)

ZeRO（Zero Redundancy Optimizer）是微软 DeepSpeed 团队提出的显存优化技术，其核心思想是通过分片（Sharding）消除数据并行中的显存冗余。在标准DDP中，每张GPU都存储完整的模型参数、梯度和优化器状态，这造成了N-1倍的冗余存储（N为GPU数量）。ZeRO通过将这些状态分散存储到不同GPU上，将单卡显存需求降低到原来的1/N，代价是增加了少量通信开销。

⚠️ 常见误区
很多人以为 ZeRO 和 DDP 效果一样。实际上 DDP 需要每张 GPU 存放完整模型副本（参数+梯度+优化器状态），而 ZeRO 将这些分片存储。对于 70B 模型，DDP 每卡需要 ~560GB（不可能），ZeRO-3 只需 ~560/N GB。但 ZeRO 的代价是更多通信——Stage 3 每次前向/反向都需要 AllGather 收集完整参数。
ZeRO 三阶段显存分片示意 (N=4 GPU, 参数 P, 梯度 G, 优化器 O): ┌──────────────────────────────────────────────────────────────────┐ │ Stage 0 (DDP): 每卡存 [P, G, O] 完整副本 │ │ │ │ GPU0: [P0-5][G0-5][O0-5] GPU1: [P0-5][G0-5][O0-5] │ │ GPU2: [P0-5][G0-5][O0-5] GPU3: [P0-5][G0-5][O0-5] │ │ 冗余: (N-1)/N = 75% │ ├──────────────────────────────────────────────────────────────────┤ │ Stage 1: 分片优化器状态 │ │ │ │ GPU0: [P0-5][G0-5][O0-1] GPU1: [P0-5][G0-5][O2-3] │ │ GPU2: [P0-5][G0-5][O4-5] GPU3: [P0-5][G0-5][空] │ │ 节省: 优化器显存 → 1/N │ ├──────────────────────────────────────────────────────────────────┤ │ Stage 2: + 分片梯度 │ │ │ │ GPU0: [P0-5][G0-1][O0-1] GPU1: [P0-5][G2-3][O2-3] │ │ GPU2: [P0-5][G4-5][O4-5] GPU3: [P0-5][G0-1][空] │ │ 节省: 优化器+梯度 → 各1/N │ ├──────────────────────────────────────────────────────────────────┤ │ Stage 3 (FSDP): + 分片参数 │ │ │ │ GPU0: [P0-1][G0-1][O0-1] GPU1: [P2-3][G2-3][O2-3] │ │ GPU2: [P4-5][G4-5][O4-5] GPU3: [P0-1][G0-1][O0-1] │ │ 节省: 全部 → 1/N | 代价: 前向需 AllGather 获取完整参数 │ └──────────────────────────────────────────────────────────────────┘

ZeRO 各阶段的通信与显存分析

ZeRO 的三个阶段在通信量和显存节省之间提供了不同的权衡点。以下是每个阶段的详细数学分析：

ZeRO Stage 1：将优化器状态（Adam 的一阶动量和二阶动量）分片到 N 张 GPU 上。每个参数的优化器状态从 8 bytes（2 × FP32 动量）降至 8/N bytes。通信量与 DDP 相同：每步需要 2M 字节的 AllReduce（M 为参数量 × 2 bytes，FP16）。显存节省主要体现在优化器状态上，对大模型的参数和梯度存储无帮助。

ZeRO Stage 2：在 Stage 1 的基础上，将梯度也分片存储。每张GPU只需要存储 1/N 的梯度，通过 ReduceScatter 聚合梯度后只保留自己负责的那部分。通信量仍为 2M 字节/步（ReduceScatter + AllGather 与 AllReduce 等价）。显存节省显著：以 7B 模型为例，Stage 2 在 8 GPU 上的单卡显存 = 14GB(参数) + 14/8 GB(梯度) + 56/8 GB(优化器) = 14 + 1.75 + 7 = 22.75 GB。

ZeRO Stage 3 / FSDP：将参数也分片存储。前向传播时通过 AllGather 收集完整参数，计算完立即释放；反向传播时再次 AllGather。通信量增加到 3M 字节/步（多了一次 AllGather）。以 7B 模型为例，Stage 3 在 8 GPU 上的单卡显存 = 84/8 = 10.5 GB。

⚠️ 关键注意
ZeRO Stage 3 的通信量虽然增加了50%（3M vs 2M），但在实际训练中，由于通信可以与计算重叠，额外的开销通常只有 10-20%。真正的瓶颈在于 AllGather 的延迟——每次前向和反向都需要等待所有参数收集完毕才能开始计算。对于非常大的模型，这种同步等待可能成为性能瓶颈。因此，在跨节点训练中，通常建议使用 Stage 2 而非 Stage 3。

1.3 FSDP 实战详解

PyTorch 的 Fully Sharded Data Parallel（FSDP）是 ZeRO Stage 3 的官方 PyTorch 实现。与 DeepSpeed 的 ZeRO 相比，FSDP 更加"PyTorch 原生"，无需修改模型代码，只需用 FSDP 包装器包裹模型即可。FSDP 的关键设计决策包括：分片粒度（auto_wrap_policy）、混合精度策略、以及通信与计算的重叠方式。

💡 实践建议
FSDP 的关键是选择合适的 auto_wrap_policy。通常按 Transformer 层来包装（每层一个 FSDP 单元），这样通信粒度适中。混合精度（BF16）几乎是必须的——既节省显存又不影响训练质量。如果显存仍然不够，可以开启 activation_checkpointing 用计算换存储。
# PyTorch FSDP 完整使用示例
import torch
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import MixedPrecision, ShardingStrategy
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy
from transformers import LlamaForCausalLM, LlamaConfig
from transformers.models.llama.modeling_llama import LlamaDecoderLayer
import functools

# 1. 创建模型
config = LlamaConfig(
    hidden_size=4096,
    num_hidden_layers=32,
    num_attention_heads=32,
    intermediate_size=11008,
)
model = LlamaForCausalLM(config)

# 2. 定义自动包装策略 (按 Transformer 层包装)
auto_wrap_policy = functools.partial(
    transformer_auto_wrap_policy,
    transformer_layer_cls={LlamaDecoderLayer},
)

# 3. 混合精度策略
mp_policy = MixedPrecision(
    param_dtype=torch.bfloat16,   # 参数用 BF16
    reduce_dtype=torch.bfloat16,  # 梯度归约用 BF16
    buffer_dtype=torch.bfloat16,  # Buffer 用 BF16
)

# 4. FSDP 包装
model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3
    mixed_precision=mp_policy,
    auto_wrap_policy=auto_wrap_policy,
    device_id=torch.cuda.current_device(),
    limit_all_gathers=True,  # 限制同时进行的 AllGather 数量
)

# 5. 启动: torchrun --nproc_per_node=8 train_fsdp.py

FSDP 的前向与反向传播流程

FSDP 的前向传播过程如下：当模型的某个 FSDP 单元（如一个 Transformer 层）需要计算时，FSDP 首先通过 AllGather 操作从所有 GPU 收集该单元的完整参数。AllGather 完成后，使用完整参数执行正常的前向计算。计算完成后，释放非本地的参数分片以节省显存。这一过程对每个 FSDP 单元依次进行。

反向传播过程类似：当需要计算某个 FSDP 单元的梯度时，再次通过 AllGather 收集完整参数，然后执行反向计算。梯度计算完成后，通过 ReduceScatter 操作将梯度分片，每张 GPU 只保留自己负责的那部分梯度。这样，整个训练步的通信量为：每个参数被 AllGather 两次（前向+反向），ReduceScatter 一次，总计约 3 × 参数量的通信。

🔬 深入探讨

FSDP 通信量的精确计算

FSDP 的通信量可以精确建模。设模型参数量为 Φ（以元素个数计），参数精度为 b 字节（BF16 则 b=2），GPU 数量为 N。对于 ZeRO Stage 3（FULL_SHARD），每个训练步的通信量为：

前向 AllGather: Φ × b 字节（每个参数被收集一次）

反向 AllGather: Φ × b 字节（再次收集参数用于梯度计算）

反向 ReduceScatter: Φ × b 字节（梯度分片）

总通信量 = 3 × Φ × b

对比 DDP 的通信量 = 2 × Φ × b（AllReduce = ReduceScatter + AllGather），FSDP 的通信量增加了 50%。但 FSDP 的通信可以更好地与计算重叠——因为每次只收集一层的参数，计算完立即释放，而不是一次性 AllReduce 所有梯度。在实践中，FSDP 的实际开销通常只有 10-30% 的额外时间。

一个重要的优化是 forward_prefetch，它允许 FSDP 在当前层计算时预先 AllGather 下一层的参数。类似地，backward_prefetch 在反向传播时预先收集下一层的参数。这些预取操作可以将通信完全隐藏在计算中，使 FSDP 的性能接近 DDP。

Activation Checkpointing（激活检查点）

对于非常大的模型，即使使用 FSDP 分片参数，显存仍然可能不够。这是因为前向传播中保存的中间激活值（用于反向传播）也会占用大量显存。Activation Checkpointing（也叫 Gradient Checkpointing）通过丢弃部分前向传播的中间激活值，在反向传播时重新计算，从而用计算时间换取显存空间。

PyTorch 提供了 torch.utils.checkpoint 模块来实现这一功能。与 FSDP 配合使用时，通常对每个 Transformer 层应用 Activation Checkpointing。这样，每个层的激活值只在需要时才重新计算，显存占用可以减少 60-70%，代价是增加约 33% 的计算时间（因为反向传播需要额外的前向计算）。

# Activation Checkpointing + FSDP
from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper,
    CheckpointImpl,
    apply_activation_checkpointing,
)

# 对每个 Transformer 层应用 Activation Checkpointing
apply_activation_checkpointing(
    model,
    checkpoint_wrapper_fn=checkpoint_wrapper,
    check_fn=lambda submodule: isinstance(submodule, LlamaDecoderLayer),
)

# 效果: 显存减少 60-70%, 计算时间增加 ~33%
# 适用于: 模型太大，即使用 FSDP 也放不进显存的情况

1.4 通信开销与性能优化

FSDP 的通信开销是实际部署中最重要的考量因素。通信量比 DDP 多50%（3× vs 2× 参数量），因为每次前向传播都需要 AllGather 收集完整参数，计算完立即释放。但在大模型场景下，FSDP 让原本不可能跑的模型变得可能——这是"能不能跑"和"快不快"的区别。

💡 性能优化清单
FSDP 性能优化的关键技巧：
1. 使用 BF16 混合精度（非 FP16，避免梯度 underflow）
2. 开启 forward_prefetch 和 backward_prefetch
3. 设置 limit_all_gathers=True 防止显存 OOM
4. 按 Transformer 层粒度包装（而非按参数或按模块）
5. 使用梯度累积减少通信频率
6. 确保数据加载不成为瓶颈（num_workers > 0, pin_memory=True）

跨节点通信的带宽分析

实际训练中，通信带宽是决定 FSDP 性能的关键因素。以 A100 80GB GPU 为例，NVLink 3.0 提供 600 GB/s 的双向带宽，而跨节点的 InfiniBand NDR 只有 50 GB/s（400 Gbps）。这意味着跨节点的 FSDP 通信比节点内慢约 12 倍。

不同通信硬件的带宽对比: NVLink 4.0 (H100): ████████████████████████████████████████ 900 GB/s NVLink 3.0 (A100): ██████████████████████████████ 600 GB/s NVSwitch (H100): ████████████████████████████████████████ 900 GB/s InfiniBand NDR: ███ 50 GB/s InfiniBand HDR: ██ 25 GB/s RoCE v2 (100GbE): █ 12.5 GB/s PCIe 5.0 (x16): ████ 64 GB/s (单向) 性能影响: FSDP 跨节点训练时, 通信时间占总训练时间的 30-60% 节点内 FSDP 训练时, 通信时间仅占 5-15%
⚠️ 生产环境注意
在生产环境中，跨节点 FSDP 的性能可能远低于理论值。常见问题包括：(1) 网络拥塞导致实际带宽只有理论值的 50-70%；(2) 多个 AllGather 操作竞争带宽；(3) CPU 端的通信调度开销。建议在跨节点训练时使用梯度累积来减少通信频率，或考虑使用 ZeRO Stage 2 代替 Stage 3 以减少通信量。此外，NCCL 的环境变量如 NCCL_IB_TIMEOUT、NCCL_SOCKET_IFNAME 等需要根据集群网络配置进行调优。

1.5 实际案例：LLaMA-3 的训练配置

Meta 在训练 LLaMA-3 405B 模型时使用了 16,384 张 H100 GPU。其训练配置是分布式并行策略的典型范例：张量并行度 TP=8（单机 8 卡通过 NVLink 互联），流水线并行度 PP=16（16 个流水线阶段跨机器），数据并行度 DP=122（122 组数据并行）。总计 8 × 16 × 122 = 15,616 张 GPU 用于训练，剩余 GPU 用于容错备份。

在这个配置中，FSDP 被用于数据并行维度上的梯度同步。由于 DP 组内的 GPU 跨越多个节点，通信通过 InfiniBand 进行。为了减少跨节点通信，LLaMA-3 团队使用了 ZeRO Stage 1（仅分片优化器状态），而非 Stage 3。这是因为对于如此大规模的训练，跨节点的 AllGather 开销会严重影响训练吞吐量。

DeepSeek-V3 的创新并行策略

DeepSeek-V3 在 2,048 张 H800 GPU 上训练时，采用了一种非常独特的并行策略。由于其 Multi-head Latent Attention（MLA）机制大幅减少了注意力层的 KV Cache 和参数量，DeepSeek-V3 将张量并行度设为 TP=1（即不做张量并行），而使用 EP（Expert Parallelism）来并行化 MoE 层的专家计算。具体配置为：PP=32（32 个流水线阶段），DP=64（64 组数据并行）。

这种配置的优势在于：(1) TP=1 消除了张量并行的通信开销，这对于 MLA 架构特别有利；(2) EP 允许每个专家在不同的 GPU 上计算，天然适合 MoE 架构；(3) 较大的 DP 度允许使用较大的全局 Batch Size（约 60M tokens），有利于训练稳定性。但代价是流水线气泡比例较高（约 32/(32+micro_batches-1)），需要足够多的 micro-batch 来填充流水线。

深入探讨： 从数据并行的核心原理出发，每个GPU持有一份完整模型副本，处理不同的mini-batch数据后同步梯度。为什么需要FSDP？推导：标准数据并行要求每张GPU存完整模型，当模型超过单卡显存时无法工作。FSDP将参数、梯度、优化器状态分片到N张GPU，前向和反向时按需All-Gather收集参数，计算后Reduce-Scatter同步梯度。数学本质上，FSDP将单卡显存需求从O(P)降至O(P/N)，代价是增加了N倍的通信量，通过计算-通信重叠可部分掩盖此开销。

✅ 参考答案与详解
  1. 核心原理: 数据并行的核心思想是将训练数据分成多份，每张 GPU 持有完整模型副本，处理不同的 mini-batch 数据后通过 AllReduce 同步梯度，最后各自更新参数。Ring AllReduce 是最核心的通信原语：将数据分成 N 份，通过 N-1 轮 ReduceScatter 和 N-1 轮 AllGather 完成归约，总通信量为 2(N-1)/N × M，当 N 足够大时趋近 2M，与 GPU 数量无关，实现带宽最优。标准 DDP 中每张 GPU 需存储三部分：模型参数（FP16, 2B/参数）、梯度（同等大小）、优化器状态（Adam 需 8B/参数）。以 7B 模型为例，单卡需 84GB（14GB 参数 + 14GB 梯度 + 56GB 优化器），因此出现了 ZeRO 优化。ZeRO 通过分片消除冗余：Stage 1 分片优化器状态，Stage 2 分片梯度，Stage 3（FSDP）分片参数，将单卡显存需求降至原来的 1/N。
    关键要点: ① AllReduce 通信量与 GPU 数几乎无关，是带宽最优算法；② DDP 的梯度分桶（默认 25MB）实现通信-计算重叠；③ ZeRO 三阶段分别分片优化器/梯度/参数，Stage 3 通信量增加 50%；④ FSDP 是 ZeRO Stage 3 的 PyTorch 原生实现，通信量 3× 参数量；⑤ Activation Checkpointing 可减少 60-70% 激活值显存，代价是 33% 额外计算。
    实际案例: Meta 训练 LLaMA-3 405B 使用 16,384 张 H100，配置为 TP=8、PP=16、DP=122。在数据并行维度使用 ZeRO Stage 1（仅分片优化器），因为跨节点 AllGather 在 16K GPU 规模下开销太大。DeepSeek-V3 则更激进，使用 TP=1（MLA 架构减少注意力参数）、PP=32、DP=64 的配置，在 2,048 张 H800 上训练。在 2 张 GPU 上对比 DDP 和 FSDP 训练 LLaMA-7B：DDP 每卡需 ~84GB（无法放入单张 A100 80GB），FSDP 仅需 ~10.5GB，但训练速度慢约 15-20%。
    关联知识: 数据并行与张量并行、流水线并行共同构成 3D 并行策略。TP 放在 NVLink 互联的单机内处理大通信量，PP 跨机器处理小通信量的 P2P 传递，DP 也跨机器但通信可与计算重叠。FSDP 的 auto_wrap_policy 选择（按 Transformer 层包装）直接影响通信粒度和效率。混合精度（BF16）几乎是 FSDP 的必须配置，既省显存又不影响训练质量。
  2. 核心原理: 数据并行在实际工程中最主要的性能瓶颈是通信开销和木桶效应。AllReduce 虽然带宽最优，但在跨节点场景下 NVLink 600 GB/s 与 InfiniBand 50 GB/s 的带宽差距高达 12 倍，导致跨节点 FSDP 通信时间占总训练时间的 30-60%（节点内仅 5-15%）。NCCL 的 Ring AllReduce 对小消息的延迟为 2(N-1) × α，可能成为瓶颈。DDP 的木桶效应意味着最慢的 GPU 决定整体速度。FSDP 的 AllGather 操作需要在每次前向和反向时收集完整参数，同步等待可能导致 GPU 空闲。网络拥塞在生产环境中可能使实际带宽只有理论值的 50-70%。
    关键要点: ① 跨节点通信是最大瓶颈，建议使用梯度累积减少通信频率；② DDP 设置 find_unused_parameters=False 避免不必要开销；③ FSDP 使用 forward_prefetch 和 backward_prefetch 预取参数隐藏延迟；④ NCCL 环境变量（NCCL_IB_TIMEOUT 等）需根据集群网络调优；⑤ 数据加载也可能成为瓶颈，需设置 num_workers>0 和 pin_memory=True。
    实际案例: 在 8 卡 A100 节点上训练 LLaMA-7B 时，使用 DDP 的吞吐量约为 120 samples/s，改用 FSDP（FULL_SHARD）后降至 105 samples/s（通信增加 50% 的影响被计算-通信重叠部分抵消）。但 FSDP 将显存从 84GB/卡降至 10.5GB/卡，使得 70B 模型可以在同一集群上训练。通过开启 forward_prefetch、梯度累积（gradient_accumulation_steps=4）和 BF16 混合精度，FSDP 的吞吐量提升至 115 samples/s，接近 DDP 水平。
    关联知识: 性能诊断工具推荐 PyTorch Profiler，可分析 AllGather 和 ReduceScatter 的耗时占比。与 Activation Checkpointing 配合可进一步优化显存（减少 60-70% 激活值，代价 33% 额外计算）。对于跨节点训练，可考虑使用 ZeRO Stage 2 代替 Stage 3 以减少通信量（2× vs 3× 参数量），或者使用 DeepSpeed 的通信优化策略。
  3. 核心原理: 数据并行的替代方案包括：① ZeRO（三阶段分片）——消除 DDP 的 N-1 倍冗余存储；② 张量并行——切分模型权重矩阵而非数据；③ 流水线并行——切分模型层到不同 GPU。DDP 适合模型能放进单卡的场景，实现简单、扩展性好、训练结果与单卡等价。ZeRO Stage 1 适合优化器显存占比大的情况（如 Adam 优化器占总显存 2/3）。ZeRO Stage 2 适合中等规模模型，通信量与 DDP 相同。ZeRO Stage 3/FSDP 适合超大模型，通信量增加 50% 但显存降至 1/N。异步数据并行无木桶效应但有梯度过时问题，训练不稳定，现代大模型几乎都用同步方案。关键权衡是"能不能跑"vs"快不快"——FSDP 让原本不可能的模型变得可能，即使牺牲部分速度。
    关键要点: ① DDP 通信量 2× 参数量，FSDP 3× 参数量，但 FSDP 通信可更好重叠；② ZeRO Stage 2 通信量与 DDP 相同（2×），但显存节省显著；③ 跨节点训练建议用 Stage 2 而非 Stage 3；④ FSDP + Activation Checkpointing 是超大模型的标准方案；⑤ NCCL 自动选择 Ring/Tree AllReduce 算法，对大消息用 Ring，小消息用 Tree。
    实际案例: 训练 LLaMA-70B（需 ~560GB 显存）的方案对比：方案 A——DDP：完全不可能（每卡需 560GB）；方案 B——FSDP（FULL_SHARD）+ 8×A100 80GB：每卡 ~70GB，可行但紧张；方案 C——FSDP + Activation Checkpointing：每卡 ~45GB，余量充足；方案 D——FSDP（SHARD_GRAD_OP，即 ZeRO-2）+ 8×A100：每卡 ~90GB，不可行。最终选择方案 C，训练吞吐量比方案 B 低约 30%（Activation Checkpointing 的额外计算），但稳定性最好。
    关联知识: 数据并行的选择与模型规模、GPU 数量、网络拓扑密切相关。在 3D 并行中，DP 度 = 总 GPU 数 / (TP 度 × PP 度)。NCCL 的拓扑感知能力（自动检测 NVLink/PCIe/InfiniBand）对性能影响很大。DeepSpeed 的 ZeRO-Offload/ZeRO-Infinity 可将优化器状态卸载到 CPU/NVMe，进一步扩展可训练模型规模，但速度代价较大。
