🏗️ 5. Decoder-Only 架构

5.1 三种 Transformer 架构

Transformer 架构自 2017 年提出以来，衍生出了三种主要的架构变体：Encoder-Only、Encoder-Decoder 和 Decoder-Only。每种架构都有其特定的优势和适用场景。在现代大语言模型（LLM）的发展历程中，Decoder-Only 架构逐渐成为绝对主流。理解这三种架构的区别和各自的优劣势，有助于理解为什么行业选择了 Decoder-Only 作为标准架构。

💡 通俗类比：三种架构像三种"考试形式"
Encoder-Only（BERT）像"完形填空"——看完整篇文章再回答问题，擅长理解但不擅长写作。Encoder-Decoder（T5）像"翻译考试"——先读原文（Encoder），再写译文（Decoder）。Decoder-Only（GPT）像"续写作文"——只看前面的内容，一个字一个字往下写。续写最通用：完形填空和翻译都可以转化为续写格式（通过 prompt）。

5.2 为什么 Decoder-Only 胜出？

Decoder-Only 架构的胜出并非偶然，而是多种因素共同作用的结果。从理论角度看，Next Token Prediction（下一个 token 预测）是最通用的训练目标——几乎所有 NLP 任务都可以转化为文本生成格式。从工程角度看，Decoder-Only 架构只需要一套参数、一个训练流程，大大简化了模型的开发和部署。从 Scaling Law 的角度看，Decoder-Only 架构的性能与参数量之间存在可预测的幂律关系，这使得研究者可以准确预测更大模型的性能，从而做出合理的投资决策。

特性Encoder-OnlyEncoder-DecoderDecoder-Only
生成能力❌✅✅
架构简单度中复杂 (两套)简单 (一套)
训练效率高中高
Scaling 效率中中最好
In-Context Learning弱中强

核心原因: CLM (Next Token Prediction) 是最通用的训练目标。只需要一个 Decoder 就能做所有事情 — 问答、翻译、代码、推理 — 只需要在 prompt 中定义任务格式。这种统一性不仅简化了模型设计，还使得模型可以通过 In-Context Learning 来处理从未见过的任务，而无需任何额外的训练。

⚠️ 常见误区：Decoder-Only 不是"唯一正确"的架构
Decoder-Only 胜出主要是因为工程优势（简单、可扩展、统一接口），而不是因为它在理论上比 Encoder-Decoder 更强。T5 在翻译/摘要等 seq2seq 任务上仍然有优势。BERT 在句子嵌入/分类任务上仍然高效。选择 Decoder-Only 是一个"足够好且最通用"的工程权衡，不是"唯一正确"的选择。
🔬 深入探讨：Scaling Law 与涌现能力
Scaling Law (Kaplan et al., 2020; Chinchilla, 2022) 表明：模型的 loss 与参数量 N、数据量 D、计算量 C 之间存在幂律关系 L ∝ N^(-α)。这意味着增加参数量可以"可预测地"降低 loss。更惊人的是"涌现能力"——某些能力（如思维链推理、代码生成）在模型规模超过某个阈值后突然出现，而不是线性增长。这解释了为什么大家疯狂扩大模型规模：大模型不仅是"更好的小模型"，而是"具备新能力的物种"。涌现能力的机制至今仍是研究热点。一种观点认为，涌现是评估指标的假象——当使用连续指标（如 BPC）而非离散指标（如准确率）时，能力的增长是连续的。另一种观点认为，涌现确实存在——它反映了模型在特定规模下"理解"了某种抽象模式。无论如何，Scaling Law 和涌现能力共同构成了 LLM 发展的理论基础：扩大规模是有回报的，而且某些关键能力只在足够大的规模下才会出现。

5.3 Causal Mask (因果掩码)

因果掩码是 Decoder-Only 架构的核心机制。它通过在注意力计算中屏蔽未来位置的信息，确保模型在预测下一个 token 时只能看到之前的 token。这模拟了人类写作的真实过程——你只能基于已经写下的内容来决定下一个词。

因果掩码矩阵 (5×5): t1 t2 t3 t4 t5 t1 [ ✓ ✗ ✗ ✗ ✗ ] t1 只看 t1 t2 [ ✓ ✓ ✗ ✗ ✗ ] t2 看 t1-t2 t3 [ ✓ ✓ ✓ ✗ ✗ ] t3 看 t1-t3 t4 [ ✓ ✓ ✓ ✓ ✗ ] t4 看 t1-t4 t5 [ ✓ ✓ ✓ ✓ ✓ ] t5 看全部 实现: ✗ 位置的注意力分数设为 -∞ softmax(-∞) = 0 → 完全屏蔽
💡 通俗类比：因果掩码像"单向玻璃"
因果掩码就像审讯室的单向玻璃：每个 token 只能看到自己左边的 token（过去），看不到右边的（未来）。这保证了模型在预测下一个 token 时，不会"偷看"答案。训练时我们有完整的文本，但通过掩码强制模型只能用前面的信息来预测——这模拟了推理时的真实场景（生成时确实看不到未来）。
⚠️ 常见误区：训练和推理的掩码行为不同
训练时：一次输入整个序列，用因果掩码并行计算所有位置的 loss（teacher forcing）。推理时：一次只生成一个 token，每次把新生成的 token 追加到输入中。虽然效果相同，但计算方式完全不同——训练是并行的（矩阵乘法），推理是串行的（自回归）。这也是为什么训练和推理的优化策略不同。

5.4 参数量计算

理解参数量的计算对于模型设计和资源规划至关重要。一个 LLM 的总参数量由多个组件决定：嵌入层、注意力层、FFN 层和归一化层。其中 FFN 层通常占总参数的 60-65%，注意力层占 30-35%，嵌入层和归一化层占剩余的 5%。

# LLaMA-2 7B 参数量估算
d = 4096, n_layers = 32, vocab = 32000

每层:
  Attention: 4 × d² = 4 × 4096² = 67.1M
  FFN (SwiGLU): 3 × d × (8/3 × d) = 134.2M
  RMSNorm: 2 × d = 8K
  每层小计: ~201.5M

总计:
  32层 × 201.5M = 6.45B
  + Embedding: 32000 × 4096 = 131M
  + Output head: (与 Embedding 共享)
  ≈ 6.6B → 约 7B ✓
🔬 深入探讨：参数量 vs 计算量 (FLOPs)
参数量和计算量是两个不同的概念。一个 token 的前向传播计算量 ≈ 2 × 参数量（矩阵乘法的乘加操作）。对于序列长度 n、batch size b，总计算量 ≈ 6 × n × b × 参数量（前向+反向+激活重计算）。LLaMA-2 7B 训练 2T tokens 的总计算量 ≈ 6 × 7B × 2T = 8.4 × 10²² FLOPs ≈ 84 ZFLOPs。这就是为什么 LLM 训练需要数千块 GPU。FLOPs 与 GPU 时间的换算关系也很重要。以 A100（312 TFLOPS BF16）为例，84 ZFLOPs 需要 84 × 10²¹ / (312 × 10¹²) ≈ 2.7 × 10⁸ 秒 ≈ 8.5 年（单卡）。使用 1024 张 A100，理论上需要约 3 天，但实际训练效率通常只有理论峰值的 40-50%（由于通信开销、内存限制等），因此实际需要约 6-7 天。

5.5 现代 LLM 的标准配置

经过多年的研究和实践，现代 LLM 已经形成了一套相对统一的标准配置。这些配置经过了大量的实验验证，代表了当前最优的工程实践。不同模型之间的差异主要在于模型规模（层数、维度）和训练数据，而不是核心架构组件的选择。理解这些标准配置的原理和工作机制，有助于快速理解任何新的 LLM 架构。从数学角度看，每个组件的选择都有其严格的理论依据：RoPE 通过旋转矩阵的数学性质实现了优雅的相对位置编码，SwiGLU 通过门控机制实现了更高效的特征选择，RMSNorm 通过简化归一化步骤提升了计算效率。这些设计的共同特点是通过数学推导和实验验证找到了性能与效率的最优平衡点。

💡 通俗类比：标准配置像"乐高积木"
现代 LLM 的标准配置就像一套成熟的乐高积木：每个组件（Decoder-Only、GQA、RoPE、SwiGLU、RMSNorm）都是经过大量实验验证的"最优积木块"。不同模型的差异主要在于：积木的数量（层数）、大小（d_model）、和排列方式（如 MoE），而不是积木本身的类型。这也是为什么不同团队的模型架构越来越相似——大家都在用同一套"标准积木"。

5.6 Decoder-Only 与 KV Cache

Decoder-Only 架构在推理阶段的一个关键优势是它天然适合 KV Cache。由于因果掩码的存在，每个 token 只能看到之前的 token，这意味着之前计算过的 Key 和 Value 可以被缓存，避免重复计算。这种缓存策略可以将推理速度提升数倍，但代价是需要额外的显存来存储缓存。KV Cache 的大小与序列长度、层数、头数和维度成正比，是长上下文推理的主要瓶颈之一。

KV Cache 推理过程: Step 1: 输入 "今天" 计算: K1, V1 (位置1) 缓存: [K1, V1] 输出: "天气" Step 2: 输入 "天气" (追加) 读取缓存: [K1, V1] 计算: K2, V2 (位置2, 只算新token) 缓存: [K1, V1, K2, V2] 输出: "很" Step 3: 输入 "很" (追加) 读取缓存: [K1, V1, K2, V2] 计算: K3, V3 (位置3) 缓存: [K1, V1, K2, V2, K3, V3] 输出: "好" 优势: 每步只计算新token的K/V，无需重复计算历史 代价: 缓存大小随序列长度线性增长

5.7 自回归生成的数学原理

Decoder-Only 架构的核心数学原理是自回归生成。给定一个输入序列 x₁, x₂, ..., xₙ，模型的目标是预测下一个 token xₙ₊₁ 的概率分布。这个过程可以用链式法则来推导：P(x₁, x₂, ..., xₙ) = P(x₁) · P(x₂|x₁) · P(x₃|x₁,x₂) · ... · P(xₙ|x₁,...,xₙ₋₁)。模型通过最小化负对数似然（NLL）来训练，这等价于最大化每个位置预测正确 token 的概率。在推理阶段，模型从起始 token 开始，一次生成一个 token，每次将新生成的 token 追加到输入序列中，然后重复这个过程直到生成终止 token 或达到最大长度。

# 自回归生成的原理与实现
import torch

def autoregressive_generate(model, prompt_ids, max_new_tokens, temperature=1.0):
    """
    自回归生成的原理：每次只生成一个token
    数学公式: P(x_t | x_1, ..., x_{t-1})
    """
    generated = prompt_ids.clone()

    for _ in range(max_new_tokens):
        # 前向传播: 计算所有位置的logits
        # 注意: 推理时可以用KV Cache避免重复计算
        logits = model(generated)  # [1, seq_len, vocab_size]

        # 只取最后一个位置的logits (预测下一个token)
        next_logits = logits[:, -1, :] / temperature

        # 采样策略
        probs = torch.softmax(next_logits, dim=-1)
        next_token = torch.multinomial(probs, num_samples=1)

        # 追加新token到序列
        generated = torch.cat([generated, next_token], dim=-1)

        # 检查终止条件
        if next_token.item() == EOS_TOKEN_ID:
            break

    return generated

# 关键概念: KV Cache 在推理中的作用
# 没有KV Cache: 每步重新计算整个序列的K和V → O(n²)
# 有KV Cache:   每步只计算新token的K和V → O(n)
# 这就是为什么KV Cache是推理优化的关键技术

自回归生成的工作机制决定了 Decoder-Only 架构的许多重要特性。首先，由于每个 token 的生成依赖于之前的所有 token，推理过程必然是串行的，无法像训练那样并行计算。其次，KV Cache 通过缓存已计算的 Key 和 Value 向量来避免重复计算，这是推理优化的核心技术。最后，不同的采样策略（贪心、温度采样、top-k、top-p）会影响生成文本的质量和多样性，这些策略的选择取决于具体的应用场景。

# KV Cache 的实现原理
class CachedAttention(nn.Module):
    """
    KV Cache 的工作机制：
    1. 第一次前向传播: 计算所有token的K和V，存入缓存
    2. 后续前向传播: 只计算新token的K和V，与缓存拼接
    3. 这样避免了重复计算，将复杂度从O(n²)降到O(n)
    """
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.d_k = d_model // n_heads
        self.n_heads = n_heads
        self.W_Q = torch.nn.Linear(d_model, d_model, bias=False)
        self.W_K = torch.nn.Linear(d_model, d_model, bias=False)
        self.W_V = torch.nn.Linear(d_model, d_model, bias=False)
        self.W_O = torch.nn.Linear(d_model, d_model, bias=False)

    def forward(self, x, kv_cache=None):
        # x: [batch, 1, d_model] (推理时只有1个新token)
        B, L, _ = x.shape
        Q = self.W_Q(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)
        K_new = self.W_K(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)
        V_new = self.W_V(x).view(B, L, self.n_heads, self.d_k).transpose(1, 2)

        if kv_cache is not None:
            # 拼接历史缓存和新的K/V
            K_past, V_past = kv_cache
            K = torch.cat([K_past, K_new], dim=2)
            V = torch.cat([V_past, V_new], dim=2)
        else:
            K, V = K_new, V_new

        # 更新缓存
        new_cache = (K, V)

        # 计算注意力 (不需要因果掩码，因为Q只有1个位置)
        scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.d_k ** 0.5)
        attn = torch.softmax(scores, dim=-1)
        out = torch.matmul(attn, V)
        out = out.transpose(1, 2).contiguous().view(B, L, -1)
        return self.W_O(out), new_cache

深入探讨： Decoder-Only 架构的"因果性"是其成功的关键。因果掩码使得每个 token 只能看到之前的 token，这与自回归生成的顺序天然匹配。从信息论角度看，因果注意力实现了一种"信息流控制"——信息只能从过去流向未来，不能反向流动。这种单向性虽然限制了模型的表达能力，但换来了训练和推理的一致性：训练时的注意力模式与推理时完全相同，避免了"训练-推理不匹配"问题。
常见误区： 不要以为 Decoder-Only 是"阉割版"的 Encoder-Decoder。实际上，Decoder-Only 通过因果注意力隐式地实现了编码和解码的统一——每一层同时扮演编码器（处理已知 token）和解码器（预测下一个 token）的角色。这种统一设计简化了架构，也使得 scaling 更加高效。
常见误区： 参数量计算时不要混淆"总参数"和"激活参数"。对于 MoE 模型（如 DeepSeek-V3 的 671B 总参数/37B 激活），总参数决定显存需求，激活参数决定推理 FLOPs。两者差距可达 10 倍以上，是 MoE 架构的核心优势。
相关章节： Decoder-Only 是现代 LLM 的标准架构选择。理解因果掩码、KV Cache、参数量计算这三个关键点，是掌握模型选型、推理优化和训练配置的基础。建议结合 Scaling Law 和 MoE 章节深入学习。
