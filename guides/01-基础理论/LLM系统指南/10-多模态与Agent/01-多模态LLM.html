👁️ 1. 多模态 LLM

💡 通俗类比
传统 LLM 就像一个只通过文字认识世界的人——他读了无数的书，知识渊博，但从未亲眼看过这个世界。多模态 LLM 则是给这个人装上了"眼睛"和"耳朵"——他不仅能读书，还能看图片、听声音、甚至看视频。两种方式各有利弊：纯文字学习（预训练）知识量大但缺乏直觉感知；加上视觉后，对世界的理解更立体，但需要额外的"视觉编码器"作为"眼睛"。这个"眼睛"的质量直接决定了模型的"视觉智商"。

1.1 两种架构范式

方案1: 后接视觉编码器 (LLaVA 范式) ┌──────────┐ ┌──────────┐ ┌──────────┐ │ 图像 │ → │ ViT/CLIP │ → │ 投影层 │ └──────────┘ └──────────┘ └──────────┘ │ 拼接文本token │ ┌──────────┐ │ LLM │ → 文本输出 └──────────┘ 特点: LLM 不变，只加一个"视觉适配器" 优势: 复用已有 LLM，训练成本低 劣势: 视觉和语言的融合不够深入 方案2: 原生多模态 (Gemini 范式) ┌──────────┐ ┌──────────────┐ │ 图像 │ → │ │ └──────────┘ │ 统一Tokenizer │ → LLM → 输出 ┌──────────┐ → │ │ │ 文本 │ └──────────────┘ └──────────┘ 特点: 从预训练阶段就是多模态 优势: 模态融合更深，理解更全面 劣势: 训练成本极高

两种架构代表了多模态 LLM 发展的两条路径：

  • 后接方案（LLaVA 范式）：先训练一个强大的视觉编码器（通常是 CLIP ViT），再通过一个简单的投影层（MLP 或 cross-attention）将视觉特征映射到 LLM 的 token 空间。LLM 本身不需要修改，只需要在少量多模态数据上微调。这种方式成本低、速度快，是 2023-2024 年的主流。
  • 原生方案（Gemini 范式）：从预训练阶段就将图像、音频、视频等模态和文本一起训练。图像被 tokenizer 转化为离散 token，与文本 token 一起进入 Transformer。这种方式需要海量的多模态数据和巨大的计算资源，但融合效果更好。
⚠️ 常见误区
很多人以为"原生多模态"一定比"后接"好。实际上，两者各有优劣。后接方案的最大优势是可以复用已有的强大 LLM——你可以给 GPT-4、LLaMA-3 等模型"装上眼睛"，而不需要从头训练。2024 年的很多优秀多模态模型（如 LLaVA-NeXT、InternVL-2）都是后接方案。原生方案的优势在需要深度理解视觉-语言关系的任务上更明显，如复杂图表理解、视频推理等。选择哪种方案取决于你的资源和需求。在实际工程选型中，如果你的场景主要是图片问答、文档理解等，后接方案已经足够好；如果需要处理复杂的视频内容或多模态交叉推理，原生方案更有优势。值得注意的是，两种方案的推理成本也有显著差异——后接方案因为视觉编码器独立，可以对视觉特征做预计算和缓存，适合需要反复查询同一图片的场景。

在后接方案中，视觉编码器的质量直接决定了多模态模型的"视觉智商"。最常用的是 CLIP（Contrastive Language-Image Pre-training）的视觉编码器：

投影层的设计

投影层是连接视觉编码器和 LLM 的"桥梁"。它的设计对模型性能有重要影响：

投影层设计方案对比: 1. 线性投影 (LLaVA-v1): visual_tokens → [Linear] → llm_tokens 最简单,效果有限 2. MLP 投影 (LLaVA-v1.5): visual_tokens → [Linear → GELU → Linear] → llm_tokens 多一层非线性,效果明显提升 3. Cross-Attention (Flamingo): llm_tokens ← [Attention(Q=llm, KV=visual)] ← visual_tokens LLM 主动"查看"视觉特征,更灵活 4. Perceiver Resampler (Flamingo): 可变数量 visual_tokens → [Resampler] → 固定数量 query_tokens 控制视觉信息的"带宽"
  • 线性投影：最简单的方案，一个矩阵乘法。LLaVA-v1 使用的就是这种。参数量极小，但表达能力有限，适合快速原型验证。
  • MLP 投影：两层 MLP + GELU，比线性投影多一些非线性变换能力。LLaVA-v1.5 使用这种方案，效果明显提升。实际测试中 MLP 投影在大多数基准上比线性投影提升 5-15%。
  • Cross-Attention：让 LLM 通过注意力机制直接"查看"视觉特征。Flamingo 和 Qwen-VL 使用这种方案，允许更灵活的视觉信息提取。Cross-Attention 层通常每隔几层插入一个，而非每一层都加。
  • Perceiver Resampler：将可变数量的视觉 token 压缩为固定数量的"查询 token"。Flamingo 使用这种方案，可以控制视觉信息的"带宽"。Resampler 的查询数量通常设为 64-128，是一个重要的超参数。

1.3 核心技术挑战

挑战一：高分辨率理解

早期多模态模型只能处理低分辨率图像（224×224），这严重限制了对细节的理解——例如无法阅读小字、无法看清图表中的数据点。解决方案包括：

  • 动态分辨率：将高分辨率图像切成多个小块（tile），每块独立编码后拼接。InternVL-2 和 LLaVA-NeXT 使用这种方式，可以处理 4K 分辨率的图像。
  • 多尺度编码：同时在多个分辨率下编码图像，融合不同尺度的信息。
  • 更大的视觉编码器：使用更高分辨率训练的 ViT，如 SigLIP-SO400M（384×384）。

挑战二：视觉幻觉

多模态模型经常产生"视觉幻觉"——描述图像中并不存在的物体或属性。例如，看到一张猫的照片，模型可能说"这只猫正在玩毛线球"，但实际上图中没有毛线球。这是因为模型倾向于根据语言先验"脑补"细节。

🔬 深入探讨
视觉幻觉的根本原因是多模态模型中的"模态不平衡"。LLM 的语言先验非常强——它知道"猫通常会玩毛线球"，所以即使图像中没有毛线球，它的语言生成部分也会倾向于输出这个描述。解决这个问题的方法包括：(1) 使用更强的视觉编码器，让视觉信号更强于语言先验；(2) 在训练数据中加入"负样本"——告诉模型"图中没有的东西不要说"；(3) 使用 RLHF 或 DPO，对幻觉进行惩罚。GPT-4o 和 Gemini 的幻觉率比早期模型低很多，说明这些方法有效。评估视觉幻觉的常用基准是 POPE（Polling-based Object Probing Evaluation）和 CHAIR（Caption Hallucination Assessment with Image Relevance）。

挑战三：视频理解

视频理解比图像理解难一个数量级，因为视频不仅有空间维度，还有时间维度。一分钟的 30fps 视频有 1800 帧，即使每帧只产生 256 个 token，总 token 数就达到 46 万——远超大多数 LLM 的上下文窗口。解决方案包括：

  • 关键帧采样：不是处理每一帧，而是均匀或智能地采样关键帧（如每秒 1 帧）。智能采样策略基于帧间差异（如光流计算或特征距离）来选择信息量最大的帧，避免处理几乎相同的相邻帧。
  • 时间压缩：对连续帧的视觉 token 进行时间维度的压缩，减少冗余。常用方法包括时间平均池化、时间注意力聚合、以及基于光流的帧对齐后合并。时间压缩可以将视频 token 数量减少 4-8 倍，同时保留关键的时序信息。
  • 滑动窗口：对长视频分段处理，每段独立理解后再整合。这种方式可以处理任意长度的视频，但需要设计好窗口之间的信息传递机制，确保全局理解不会丢失。最新研究探索了"记忆增强"方式——维护一个不断更新的视频记忆，存储之前片段的关键信息。
⚠️ 视频理解不仅仅是"多看几帧图片"
视频中有大量时间冗余——相邻帧之间的变化很小，直接处理每一帧会浪费大量计算。更重要的是，视频理解需要"时序推理"能力——理解事件的先后顺序、因果关系。例如，"杯子被打翻了，水洒了一地"需要模型理解"打翻"导致"洒了"的因果关系。当前的多模态模型在时序推理上仍有很大提升空间。实际工程中，视频多模态推理的延迟控制是最大挑战——需要在理解质量和实时性之间找到平衡点，通常通过关键帧智能选择和视觉 Token 缓存来实现。
💡 多模态模型的实际部署建议
在生产环境中部署多模态 LLM 需要关注以下要点：(1) 视觉编码器的预计算——对于反复查询同一图片的场景，可以预先计算视觉特征并缓存，避免重复编码。(2) 动态分辨率策略——根据图片的实际复杂度选择分辨率，简单图片使用低分辨率以节省 Token 预算。(3) 批处理优化——将多个图片查询合并为一个批次处理，提高 GPU 利用率。(4) 降级策略——当视觉编码失败时，模型应能回退到纯文本模式继续工作。(5) 内容安全——多模态模型更容易生成不当内容，需要额外的过滤层。此外，建议在生产环境中监控视觉 Token 的平均长度和分布，以便持续优化资源分配。多模态服务的 SLA 指标应包括图片处理延迟的 P99 值和视觉幻觉率。在大规模部署时，还需要考虑图片 CDN 的缓存策略和预处理流水线的设计，确保从图片上传到模型响应的端到端延迟在可接受范围内。定期更新视觉编码器的版本也是保持服务质量的关键。

1.4 对比学习与视觉语言对齐

🔬 CLIP 对比学习的核心机制
CLIP（Contrastive Language-Image Pre-training）是多模态 LLM 的基石之一。它的训练目标是让匹配的图文对在嵌入空间中距离更近，不匹配的图文对距离更远。具体来说，CLIP 同时训练一个图像编码器和一个文本编码器，将它们的输出映射到同一个嵌入空间。对于一个 batch 中的 N 个图文对，CLIP 计算 N×N 的相似度矩阵，然后用对比损失（InfoNCE Loss）来优化——匹配对的相似度应该最大，不匹配对的相似度应该最小。这种训练方式使得 CLIP 学到了非常强大的视觉语言对齐能力。CLIP 的训练数据来自互联网上的 4 亿个图文对（WebImageText），这些数据覆盖了极其广泛的视觉和语言概念，使得 CLIP 的视觉编码器具有很强的泛化能力。
CLIP 对比学习示意图: 图像编码器 (ViT) 文本编码器 (Transformer) │ │ ▼ ▼ 图像嵌入 I₁,I₂,...,Iₙ 文本嵌入 T₁,T₂,...,Tₙ │ │ └─────────┬─────────────────┘ │ 计算相似度矩阵 ┌──────────────────┐ │ I₁·T₁ I₁·T₂ ···│ 对角线 = 匹配对 (最大化) │ I₂·T₁ I₂·T₂ ···│ 非对角线 = 不匹配 (最小化) │ ··· │ └──────────────────┘ │ InfoNCE Loss 优化
# CLIP 对比学习损失函数实现
import torch
import torch.nn.functional as F

def clip_loss(image_embeds, text_embeds, temperature=0.07):
    """
    InfoNCE 对比损失: 让匹配的图文对相似度最大
    image_embeds: [B, D] 图像嵌入
    text_embeds:  [B, D] 文本嵌入
    """
    # L2 归一化
    image_embeds = F.normalize(image_embeds, dim=-1)
    text_embeds = F.normalize(text_embeds, dim=-1)

    # 计算 N×N 相似度矩阵
    logits = torch.matmul(image_embeds, text_embeds.T) / temperature
    # 对角线为正样本 (匹配对)
    labels = torch.arange(logits.size(0), device=logits.device)

    # 双向对比损失: image→text + text→image
    loss_i2t = F.cross_entropy(logits, labels)
    loss_t2i = F.cross_entropy(logits.T, labels)
    return (loss_i2t + loss_t2i) / 2

# 多模态投影层: 将视觉特征映射到 LLM 的嵌入空间
class VisualProjection(torch.nn.Module):
    """将 ViT 输出投影到 LLM 隐藏维度"""
    def __init__(self, vision_dim=1024, llm_dim=4096):
        super().__init__()
        self.proj = torch.nn.Sequential(
            torch.nn.Linear(vision_dim, llm_dim),
            torch.nn.GELU(),
            torch.nn.Linear(llm_dim, llm_dim),
        )

    def forward(self, vit_output):
        # vit_output: [B, num_patches, vision_dim]
        # → [B, num_patches, llm_dim]
        return self.proj(vit_output)

import torch
from transformers import LlavaForConditionalGeneration, AutoProcessor

model = LlavaForConditionalGeneration.from_pretrained(
    "llava-hf/llava-v1.6-mistral-7b-hf",
    torch_dtype=torch.float16, device_map="auto"
)
processor = AutoProcessor.from_pretrained("llava-hf/llava-v1.6-mistral-7b-hf")

# 处理图像输入: 图像 → ViT → 视觉 Token
from PIL import Image
image = Image.open("example.jpg")
prompt = "USER: <image>\n描述这张图片的内容</s>\nASSISTANT:"
inputs = processor(prompt, image, return_tensors="pt").to("cuda")

# 生成回答: 视觉 Token + 文本 Token 一起进入 LLM
output = model.generate(**inputs, max_new_tokens=512, do_sample=True)
response = processor.decode(output[0], skip_special_tokens=True)
print(response)

# 视觉 Token 压缩: 使用 Perceiver Resampler 降维
# 原始 576 视觉 Token → 压缩为 64 个 query Token
class PerceiverResampler(torch.nn.Module):
    def __init__(self, dim=4096, num_queries=64, num_heads=8):
        super().__init__()
        self.query = torch.nn.Parameter(torch.randn(1, num_queries, dim))
        self.cross_attn = torch.nn.MultiheadAttention(dim, num_heads, batch_first=True)

    def forward(self, visual_tokens):
        # query attend to visual tokens
        out, _ = self.cross_attn(self.query.expand(visual_tokens.size(0), -1, -1),
                                 visual_tokens, visual_tokens)
        return out  # [B, 64, dim]
🔬 深入探讨
多模态训练的一个关键问题是"灾难性遗忘"——当 LLM 在多模态数据上微调时，可能会遗忘之前学到的纯文本能力。这是因为多模态数据的分布与纯文本数据不同，微调会改变模型的权重分布。解决方法包括：(1) 混合训练——在多模态数据中混入一定比例的纯文本数据（通常 20-30%）；(2) 低秩适配（LoRA）——只微调少量参数，减少对原始权重的干扰；(3) 渐进式训练——先用少量多模态数据微调，再逐步增加。经验表明，混合训练是最简单有效的方案。

1.7 多模态推理与部署

视觉 Token 的效率优化

多模态模型的推理效率瓶颈主要在视觉 Token 的处理上。一张 384×384 的图像经过 ViT 编码后通常产生 576-1024 个视觉 Token，而动态分辨率方案（如将高分辨率图像切分为多个 tile）可能产生 2000-6000 个视觉 Token。这些视觉 Token 与文本 Token 一起进入 LLM，显著增加了推理的计算和内存开销。优化方案包括：(1) 视觉 Token 压缩——通过 Perceiver Resampler 或 Q-Former 将视觉 Token 压缩到固定数量（如 64-256 个）；(2) 视觉 Token 剪枝——根据注意力权重动态丢弃不重要的视觉 Token；(3) 视觉 Token 量化——将视觉 Token 的精度从 FP16 降低到 INT8 或 INT4。这些优化技术可以将多模态推理的速度提升 2-5 倍，同时保持 95% 以上的性能。

多模态模型的量化策略

多模态模型的量化面临独特的挑战——视觉编码器和 LLM 的权重分布差异很大，需要使用不同的量化策略。视觉编码器的权重通常分布更均匀，适合使用对称量化；LLM 的权重分布则更稀疏，适合使用非对称量化。此外，视觉编码器对量化误差更敏感——轻微的量化误差可能导致视觉特征的语义偏移，进而影响 LLM 对图像的理解。因此，多模态模型的量化通常采用"选择性量化"策略——对视觉编码器使用更保守的量化方案（如 INT8），对 LLM 使用更激进的量化方案（如 INT4）。

实时多模态推理

GPT-4o 展示了实时多模态推理的可能性——模型可以实时处理摄像头和麦克风输入，实现类似人类的多模态感知和交互。实时多模态推理的技术挑战包括：(1) 延迟控制——从摄像头捕获图像到模型输出响应的延迟需要控制在几百毫秒以内；(2) 流式处理——图像和音频需要以流式方式输入模型，而非一次性输入；(3) 上下文管理——实时交互需要维护一个不断更新的多模态上下文，这对 KV Cache 管理提出了更高要求。目前，实时多模态推理主要通过"感知压缩"来实现——不是处理每一帧图像，而是只在检测到显著变化时才处理新帧。

多模态模型的评估比纯文本模型更复杂，因为需要同时评估视觉理解和语言生成能力。主流的多模态评估基准包括：(1) MMBench——综合性多模态评估基准，涵盖感知、推理、知识等多个维度，采用循环评估策略减少选项偏差。(2) MMMU（Massive Multi-discipline Multimodal Understanding）——聚焦大学级别的多学科知识理解，包含图表、公式、示意图等专业视觉内容，是目前难度最高的多模态基准之一。(3) DocVQA——文档视觉问答，评估模型对扫描文档、表格、发票等的理解能力，对企业应用特别重要。(4) ChartQA——图表理解能力评估，包括柱状图、折线图、饼图等常见图表类型。(5) RealWorldQA——真实世界场景理解，评估模型在日常生活中的视觉推理能力。评估时需要注意的问题包括：不同基准之间的分数不可直接比较（因为评估方法和数据分布不同）、多模态模型在不同语言上的表现差异很大（中文能力通常弱于英文）、以及视觉幻觉的评估需要专门的基准（如 POPE）。

多模态架构的技术原理与性能分析： 从底层硬件角度分析，多模态LLM的性能瓶颈通常在于视觉编码器的计算量和视觉Token的内存占用。一张384×384的图像经ViT编码后产生576个视觉Token，动态分辨率方案（如4K图像切分为多个tile）可能产生2000-6000个视觉Token。这些视觉Token与文本Token一起进入LLM，使得单次推理的KV Cache和注意力计算量成倍增加。推导：设视觉Token数为V，文本Token数为T，则多模态推理的注意力计算量为O((V+T)²)，当V远大于T时（如处理高分辨率图片），计算量主要由视觉Token主导。为什么这很重要？因为这意味着多模态推理的优化重点应放在视觉Token的压缩和筛选上——通过Perceiver Resampler将视觉Token从576压缩到64，注意力计算量可降低80%以上。从工作机制看，视觉Token预计算缓存（对同一图片复用编码结果）和动态分辨率选择（根据图片复杂度调整tile数量）是提升多模态推理效率的两大核心手段。

实用技巧： 在实际工程中部署多模态LLM时，建议先在100张代表性图片上测试视觉编码器的处理延迟和Token产出量，建立视觉Token的平均长度基线。然后测试不同投影层方案（线性投影vs MLP vs Cross-Attention）在MMBench和DocVQA上的准确率差异，选择最优方案。部署时实现视觉特征缓存层，对相同图片的重复查询复用预计算的视觉Token，可将多模态推理延迟降低50%以上。
深入探讨： 从系统设计的原理出发，多模态LLM采用「编码-对齐-生成」三层架构来管理跨模态理解的复杂性。编码层负责将不同模态（图像、音频、视频）转换为统一的Token表示，对齐层负责将不同模态的Token映射到共享语义空间，生成层（LLM）负责基于对齐后的多模态Token进行推理和文本生成。为什么这种分层设计有效？推导：设模态数为M，每种模态的编码器参数为P_m，直接训练统一编码器的参数量为∏P_m（指数级），而分层设计的总参数量为ΣP_m + P_align（线性级）。数学本质上，分层设计用模块化的编码器和轻量的对齐层替代了巨大的统一模型，使得多模态能力可以"插件式"地添加到已有LLM上——这正是LLaVA范式的核心优势。

✅ 参考答案与详解
  1. 核心原理: 多模态 LLM 的两种架构范式代表了视觉-语言融合的两条技术路线。后接方案（LLaVA 范式）的核心原理是"编码-对齐-生成"三层架构：先用预训练的视觉编码器（如 CLIP ViT）将图像编码为视觉 Token，再通过投影层（MLP 或 Cross-Attention）将视觉特征映射到 LLM 的嵌入空间，最后将视觉 Token 与文本 Token 拼接后送入 LLM 生成回答。这种方式不需要修改 LLM 本身，只需训练投影层和少量微调，成本极低。原生方案（Gemini 范式）则从预训练阶段就将图像、音频、视频与文本一起训练，图像被统一 Tokenizer 转化为离散 Token 与文本 Token 共享同一语义空间，融合更深入但训练成本极高。投影层的设计对后接方案至关重要——线性投影（LLaVA-v1）最简单但表达有限，MLP 投影（LLaVA-v1.5）增加非线性变换提升 5-15%，Cross-Attention（Flamingo）让 LLM 主动"查看"视觉特征更灵活，Perceiver Resampler 将可变数量视觉 Token 压缩为固定数量（64-128 个）控制"带宽"。实际应用场景：企业文档理解场景使用后接方案（如 InternVL-2.5），给已有 LLaMA-3 模型"装上眼睛"，只需在百万级文档图文对上微调投影层，即可实现高精度的表格、图表、发票理解能力。
    关键要点: ① 后接方案可复用已有强大 LLM（GPT-4、LLaMA-3），训练成本低，是 2023-2024 年主流；② 原生方案在深度视觉-语言推理任务（复杂图表、视频推理）上更有优势；③ 视觉编码器质量直接决定"视觉智商"——SigLIP（384×384）比 CLIP ViT-L/14（224×224）细粒度识别更好；④ 动态分辨率方案将高分辨率图像切成多个 tile 独立编码，可处理 4K 分辨率但 Token 数量成倍增加；⑤ 后接方案的视觉特征可预计算缓存，适合反复查询同一图片的场景。
    实际案例: LLaVA-NeXT 使用 SigLIP 视觉编码器（384×384）+ MLP 投影层 + LLaMA-3 7B，在 MMBench 上达到与原生多模态模型 Gemini 相近的性能，但训练成本仅为后者的 1/100。对于需要处理 4K 文档图片的场景，采用动态分辨率将图像切成 6 个 512×512 tile，产生 6×256=1536 个视觉 Token，配合 Perceiver Resampler 压缩到 64 个 query Token，兼顾了分辨率和推理效率。
    关联知识: 两种范式与 CLIP 对比学习（视觉-语言对齐的基石）、LoRA 低秩微调（减少灾难性遗忘）、以及推理优化（视觉 Token 预计算缓存、动态分辨率策略）紧密关联。理解 CLIP 的 InfoNCE 损失函数如何将匹配图文对拉近是理解所有多模态对齐技术的前提。
  2. 核心原理: 多模态 LLM 在工程中的核心性能瓶颈在于视觉 Token 的处理开销。一张 384×384 图像经 ViT 编码后产生 576-1024 个视觉 Token，动态分辨率方案（4K 图像切 tile）可能产生 2000-6000 个视觉 Token，与文本 Token 一起进入 LLM 后，注意力计算量为 O((V+T)²)，当视觉 Token 数 V 远大于文本 Token 数 T 时，计算量主要由视觉 Token 主导。视觉幻觉是另一大瓶颈——模型根据语言先验"脑补"图中不存在的物体，原因是 LLM 的语言先验强于视觉信号。诊断方法：在 100 张代表性图片上测试视觉编码器的处理延迟和 Token 产出量，建立基线；用 POPE 和 CHAIR 基准量化幻觉率。解决方案：(1) Perceiver Resampler 将视觉 Token 从 576 压缩到 64，注意力计算量降低 80% 以上；(2) 视觉 Token 剪枝——根据注意力权重动态丢弃不重要 Token；(3) 视觉特征缓存——对同一图片预计算复用编码结果，延迟降低 50%+；(4) 选择性量化——视觉编码器用 INT8（对量化误差敏感），LLM 用 INT4。
    关键要点: ① 视觉 Token 是多模态推理的主要瓶颈，压缩和筛选是优化重点；② 视觉编码器对量化误差比 LLM 更敏感——轻微量化误差可能导致语义偏移；③ 视觉幻觉需用负样本训练+RLHF/DPO 惩罚来缓解；④ 视频理解需要关键帧采样+时间压缩（4-8x Token 减少）+滑动窗口；⑤ 生产环境需监控视觉 Token 平均长度分布和幻觉率 P99 指标。
    实际案例: 部署 Qwen2.5-VL 处理电商商品图片问答时，原始 384×384 图像产生 576 个视觉 Token，通过 Perceiver Resampler 压缩到 128 个后，单次推理延迟从 2.3 秒降至 0.8 秒，MMBench 准确率仅下降 1.2%。对于同一商品图片的多次查询，预计算视觉特征缓存使后续查询延迟降至 0.3 秒。
    关联知识: 性能瓶颈分析与 KV Cache 优化（PagedAttention）、模型量化（GPTQ/AWQ/FP8）、推理框架选型（vLLM 支持多模态批处理）、以及视觉编码器选型（CLIP vs SigLIP vs DINOv2 在不同任务上的表现差异）密切相关。
  3. 核心原理: 两种范式的替代方案对比需要从"成本-性能-灵活性"三个维度分析：(1) 后接方案（LLaVA）vs 原生方案（Gemini）——后接方案训练成本低（百万级数据+少量 GPU 时间）、可复用已有 LLM、视觉特征可缓存复用，适合图片问答、文档理解等场景；原生方案融合更深、理解更全面，适合复杂视频推理和多模态交叉推理，但需要海量多模态数据和巨大计算资源。(2) 投影层方案对比——线性投影最简单适合原型验证，MLP 投影性价比最高（5-15% 提升），Cross-Attention 最灵活但增加模型复杂度，Perceiver Resampler 可控制 Token 带宽。(3) 视觉编码器对比——CLIP ViT-L/14（224×224）经典通用，SigLIP（384×384）细粒度更强，InternViT-6B（448×448）超大编码器精度最高但计算量大，DINOv2 自监督空间理解强但缺乏语言对齐。选择指南：资源有限用后接方案+MLP 投影+SigLIP，追求极致理解用原生方案，需要视频理解用原生方案+关键帧采样。
    关键要点: ① 后接方案在 2024 年仍然极具竞争力（LLaVA-NeXT、InternVL-2 表现优秀）；② 原生方案的优势在需要深度模态融合的任务上才明显；③ 视觉编码器选择比 LLM 选择更关键——不同编码器在 OCR、图表、空间推理上差异巨大；④ 训练策略三阶段（预训练对齐→指令微调→偏好优化）对两种范式都适用；⑤ 混合训练（多模态数据中混入 20-30% 纯文本数据）是防止灾难性遗忘的最简单有效方案。
    实际案例: 对比三种典型方案：(a) 快速原型用 LLaVA-v1.5（CLIP ViT+MLP 投影+LLaMA-7B），训练仅需 1 天 8×A100；(b) 企业文档理解用 InternVL-2.5（InternViT-6B+MLP+Qwen-72B），MMBench 85%+，支持 4K 分辨率；(c) 实时多模态交互用 Gemini 风格原生方案，支持音视频文本实时推理，延迟<500ms。
    关联知识: 替代方案对比与 CLIP 对比学习原理、LoRA/QLoRA 高效微调、视觉 Token 压缩技术、以及 Agent 系统中的多模态感知模块紧密关联——多模态 LLM 是 Agent"眼睛"的技术基础。
