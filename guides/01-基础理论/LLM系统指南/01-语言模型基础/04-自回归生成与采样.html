🎲 4. 自回归生成与采样策略

4.1 Prefill 与 Decode

LLM 推理的两个阶段: 阶段1: Prefill (预填充) — 并行，快 输入: "今天天气" (4个 token) 4个 token 一次性并行处理 计算所有位置的 K/V → 存入 KV Cache 输出: 第一个预测 token "很" 耗时: ~10ms (取决于输入长度) 阶段2: Decode (解码) — 串行，慢 每次只输入 1 个新 token，复用 KV Cache 第1步: 输入 "很" → 预测 "好" (复用 KV1~KV4) 第2步: 输入 "好" → 预测 "," (复用 KV1~KV5) 第3步: 输入 "," → 预测 "适" (复用 KV1~KV6) ...直到 <EOS> 或最大长度 耗时: 每步 ~10-50ms
⚠️ 关键性能差异
Prefill 是并行的（快），Decode 是串行的（慢）。生成 100 个 token 需要 100 次前向传播。这就是为什么 LLM 推理需要 KV Cache 和各种优化技术。
💡 通俗类比：Prefill vs Decode 就像"读题"和"答题"
Prefill 像是"读题" — 你可以一次读完整个题目（并行处理所有输入 token）。Decode 像是"手写答题" — 你只能一个字一个字地写（每次生成一个 token）。读题很快，但写字很慢，尤其是长篇大论的回答。
KV Cache 的作用 — 避免重复计算: 没有 KV Cache (浪费): 第1步: 处理 [今天天气] → 预测 "很" ← 算了4个token的K/V 第2步: 处理 [今天天气很] → 预测 "好" ← 重算前4个的K/V！ 第3步: 处理 [今天天气很好] → 预测 "," ← 又重算前5个的K/V！ 复杂度: O(1+2+3+...+n) = O(n²) 有 KV Cache (高效): 第1步: 处理 [今天天气] → KV1~4存入Cache → 预测 "很" 第2步: 只处理 [很]，从Cache读KV1~4 → 预测 "好" 第3步: 只处理 [好]，从Cache读KV1~5 → 预测 "," 复杂度: O(n) — 每步只算1个新token
🔬 深入探讨：KV Cache 的内存瓶颈
KV Cache 虽然避免了重复计算，但代价是巨大的内存开销。对于 LLaMA-2 70B，处理 4K 上下文的 KV Cache 需要约 2.5GB 显存（每增加一个请求就多 2.5GB）。这就是为什么长上下文和高并发推理如此昂贵 — 不是计算不够，而是显存放不下 KV Cache。MQA (Multi-Query Attention) 和 GQA (Grouped-Query Attention) 等技术就是为了解决这个问题。

4.2 Temperature (温度)

模型输出的是 logits（未归一化的分数），需要转为概率分布。温度参数 τ 控制分布的"尖锐程度"：

# 温度缩放
scaled_logits = logits / temperature

# 然后 softmax
probabilities = softmax(scaled_logits)
温度效果概率分布形状适用场景
τ → 0贪心，总选最高概率尖锐 (接近 one-hot)事实问答、代码
τ = 0.3稍有随机性较尖锐翻译、摘要
τ = 1.0按原始概率采样原始分布通用对话
τ = 1.5更随机平坦创意写作
💡 通俗类比：Temperature 就像"创意旋钮"
温度低 (τ→0) = 严谨的会计，每一步都选最稳妥的选项，结果可预测但可能死板。温度高 (τ=1.5) = 天马行空的艺术家，经常选意想不到的词，结果可能有创意但容易出错。调 Temperature 就是在"可靠性"和"创造性"之间找平衡。
不同温度下同一个 logits 的概率分布: 原始 logits: [猫:3.0, 狗:2.5, 鱼:1.0, 鸟:0.5] τ=0.1 (极低): 猫 ██████████████████████████████████████ 92% 狗 ████ 7% 鱼 · 0.5% 鸟 · 0.1% τ=1.0 (原始): 猫 ████████████████ 38% 狗 ████████████ 28% 鱼 ████ 18% 鸟 ███ 16% τ=2.0 (高): 猫 █████████ 29% 狗 ████████ 27% 鱼 ██████ 23% 鸟 █████ 21%
🔬 深入探讨：Temperature = 0 真的是"贪心"吗？
严格来说，Temperature → 0 才是贪心（趋近于 one-hot）。实际 API 中 temperature=0 通常等价于 greedy decoding — 总是选概率最高的 token。但注意：贪心解码不保证找到全局最优序列，因为局部最优不一定组合成全局最优。Beam Search 通过保留多个候选序列来缓解这个问题，但计算开销更大。

4.3 Top-k 采样

只从概率最高的 k 个 token 中采样：

# Top-k 过滤
top_k_logits = logits.copy()
top_k_logits[rank > k] = -inf   # 只保留前 k 个
probs = softmax(top_k_logits)

问题: k 是固定的。当分布很尖锐时 (如 "1+1=")，k=50 太宽；当分布很平坦时，k=50 又太窄。

Top-k 采样的问题: 情况1: 分布尖锐 ("1+1=" → 应该是 "2") 原始: [2:95%, 3:3%, 4:1%, ...] Top-k(50): [2:95%, 3:3%, 4:1%, ..., 第50个:0.01%] → k=50 太宽，可能选到荒谬的答案 情况2: 分布平坦 (开放性问题) 原始: [A:8%, B:7%, C:6%, ..., Z:2%] Top-k(50): 保留前50个 → k=50 可能太窄，排除了合理的低概率选项
⚠️ 常见误区：Top-k 越大越"安全"？
不是。Top-k 太大会引入噪声（低概率的荒谬 token），太小会限制多样性。而且"安全"的 k 值取决于具体上下文 — 数学计算需要小 k，创意写作需要大 k。这就是为什么 Top-p 采样更常用 — 它能自适应调整候选集大小。

4.4 Top-p (Nucleus) 采样

自适应选择候选集：从概率最高的 token 开始累加，直到累计概率达到 p：

# Top-p 过滤
sorted_probs = sort(probs, descending=True)
cumulative = cumsum(sorted_probs)
# 找到累积概率超过 p 的位置
cutoff_index = first_index_where(cumulative >= p)
probs[rank > cutoff_index] = 0   # 只保留前面的
probs = probs / sum(probs)       # 重新归一化
💡 通俗类比：Top-p 就像"90% 信心区间"
Top-p=0.9 的意思是：只从模型"有 90% 信心"的候选词中选择。当模型很确定时（如 "1+1="），候选集可能只有 1-2 个词；当模型不确定时（如开放式问题），候选集会更大。它能自动适应不同上下文的需求。
Top-p 自适应候选集: 情况1: 分布尖锐 ("1+1=") P("2") = 0.95 ← 累计 0.95 > 0.9 → 候选集只有 ["2"] P("3") = 0.03 P("4") = 0.01 → 自动收紧到 1 个选项 情况2: 分布平坦 (开放问题) P("A") = 0.15 ← 累计 0.15 P("B") = 0.12 ← 累计 0.27 P("C") = 0.10 ← 累计 0.37 P("D") = 0.09 ← 累计 0.46 P("E") = 0.08 ← 累计 0.54 P("F") = 0.07 ← 累计 0.61 P("G") = 0.06 ← 累计 0.67 P("H") = 0.05 ← 累计 0.72 P("I") = 0.04 ← 累计 0.76 P("J") = 0.04 ← 累计 0.80 P("K") = 0.04 ← 累计 0.84 P("L") = 0.04 ← 累计 0.88 P("M") = 0.04 ← 累计 0.92 > 0.9 → 候选集 13 个词 → 自动扩展到更多选项
🔬 深入探讨：Top-p 的数学直觉
Top-p 本质上是在做"截断分布" — 把概率质量集中在最可能的 token 上，然后重新归一化。这保证了采样结果的概率分布仍然是合法的（和为 1）。一个有趣的观察：Top-p 的有效 k 值随上下文动态变化。在确定性高的位置，有效 k≈1（接近贪心）；在不确定的位置，有效 k 可能很大。这比固定 k 的 Top-k 优雅得多。

4.5 组合策略

实践中通常同时使用 Top-k + Top-p + Temperature：

# 典型的采样流程
1. Temperature: logits = logits / temperature
2. Top-k: 保留前 50 个 token
3. Top-p: 在 Top-k 基础上，进一步用 p=0.9 过滤
4. 从最终分布中采样
采样流水线: 原始 logits (50K 个 token) │ ▼ ┌──────────────┐ │ Temperature │ logits / τ │ τ = 0.7 │ ← 调节随机性 └──────────────┘ │ ▼ ┌──────────────┐ │ Top-k │ 只保留前 50 个 │ k = 50 │ ← 硬性截断 └──────────────┘ │ 从 50K → 50 个候选 ▼ ┌──────────────┐ │ Top-p │ 累积概率 < 0.9 │ p = 0.9 │ ← 软性截断 └──────────────┘ │ 从 50 → 可能 10~30 个 ▼ ┌──────────────┐ │ 采样 │ 按概率随机选 1 个 └──────────────┘ │ ▼ 输出 token
💡 通俗类比：组合策略就像"多层过滤网"
Temperature 是调节水流速度（控制随机性），Top-k 是粗滤网（去掉明显杂质），Top-p 是细滤网（只保留精华）。三层过滤后，水质（输出质量）才有保证。不同场景需要不同组合 — 写代码需要收紧（低温度、小 k、小 p），写故事需要放松（高温度、大 k、大 p）。
⚠️ 常见误区：先 Top-p 再 Top-k？
顺序很重要！标准做法是先 Temperature → Top-k → Top-p。如果反过来，Top-p 在 Temperature 之前应用的话，温度缩放会改变已筛选的分布形状，效果不可控。各框架（HuggingFace、vLLM、TGI）都是按 Temperature → Top-k → Top-p 的顺序实现的。
场景TemperatureTop-kTop-p理由
代码生成0.0~0.2100.9需要确定性，减少错误
事实问答0.011.0贪心，确保准确
通用对话0.7500.9平衡准确性和自然感
创意写作1.0~1.51000.95鼓励多样性和创造力
翻译0.3200.85需要准确但允许灵活性

🔬 深入探讨：Speculative Decoding — 用小模型加速大模型
传统 Decode 每步只能生成 1 个 token，串行瓶颈明显。Speculative Decoding 的思路是：用一个轻量"草稿模型"快速生成多个候选 token，再用大模型一次性验证。如果草稿正确（大模型也同意），就一次"吃掉"多个 token，显著减少前向传播次数。这在不改变输出质量的前提下，可以实现 2-3x 的推理加速。

4.7 推理优化技术概览

LLM 推理的瓶颈主要在于 Decode 阶段的串行特性。以下介绍几种关键的推理优化技术。

Continuous Batching（连续批处理）

传统的静态批处理要求一个批次中的所有请求同时开始、同时结束。但 LLM 的请求长度差异很大 — 一个请求可能只需要生成 10 个 token，另一个需要生成 1000 个。静态批处理会导致短请求等待长请求结束，造成严重的资源浪费。

Continuous Batching 允许在 Decode 过程中动态地添加新请求和移除已完成的请求。当一个请求生成了 EOS token，立即释放其资源给新请求。这种方式可以将 GPU 利用率从 30-50% 提升到 80-90%。

静态批处理 vs 连续批处理: 静态批处理 (Static Batching): 时间 → 请求1: [============] 完成 请求2: [====================] 完成 请求3: [========] 完成 GPU: [████████░░░░░░░░████] 利用率低 (大量等待) 连续批处理 (Continuous Batching): 时间 → 请求1: [====]完成 请求4: [======]完成 请求2: [========]完成 请求5: [====]完成 请求3: [====]完成 请求6: [======] GPU: [████████████████████] 利用率高 (无空闲)

PagedAttention 与 vLLM

KV Cache 的内存管理是一个重要问题。传统方式为每个请求预分配一段连续内存，大小等于最大上下文长度。但实际生成长度通常远小于最大长度，导致大量内存浪费。

PagedAttention 借鉴了操作系统的虚拟内存思想：将 KV Cache 分成固定大小的"页"（block），按需分配。当一个请求完成时，其页面可以立即被回收分配给新请求。这种方式将 KV Cache 的内存利用率从约 20-40% 提升到接近 100%。

🔬 深入探讨：PagedAttention 的工作原理

PagedAttention 将每个请求的 KV Cache 分成多个固定大小的块（block），每个块存储固定数量的 token 的 KV 向量。这些块不需要在物理内存中连续存放 — 通过一个"块表"（block table）记录逻辑块到物理块的映射。当需要读取某个位置的 KV 向量时，先查块表找到物理地址，再读取数据。

这种设计带来了三个关键优势：(1) 内存碎片化几乎为零；(2) 可以实现 Copy-on-Write — 当多个请求共享相同的前缀（如系统提示）时，它们的 KV Cache 可以共享同一组物理块，只在出现差异时才复制；(3) 内存分配是 O(1) 的，不需要预先知道请求的最终长度。

4.8 对比解码与引导生成

除了基本的采样策略外，还有一些高级的解码技术用于控制生成质量。

对比解码（Contrastive Decoding）

对比解码的核心思想是：用一个大模型和一个小模型的 logits 差值来采样。直觉是 — 大模型和小模型都认为好的 token 一定是"简单"的；只有大模型认为好而小模型认为不好的 token，才是"有信息量"的。

logits_final = logits_large - α · logits_small

这种方法可以减少模型生成"退化"内容（如重复、套话），提高生成质量。

引导生成（Guided Generation）

在很多应用场景中，我们需要模型的输出严格遵循某种格式（如 JSON、SQL、正则表达式）。引导生成通过在采样时屏蔽不符合格式要求的 token，确保输出100%格式正确：

引导生成的工作原理: 目标格式: JSON 期望输出: {"name": "xxx", "age": xx} 已生成: {"name": " 下一步只能选: 任何合法的 JSON 字符串内容字符 已生成: {"name": "Alice" 下一步只能选: " (引号，因为 JSON 字符串必须闭合) 已生成: {"name": "Alice" 下一步只能选: , (逗号) 或 } (闭合大括号) → 屏蔽所有其他 token 已生成: {"name": "Alice", "age": 下一步只能选: 数字字符 (0-9) 结果: 100% 合法的 JSON 输出！
💡 引导生成的实际应用

引导生成在以下场景中非常有用：(1) 结构化数据提取 — 确保输出是合法的 JSON；(2) SQL 生成 — 确保生成的 SQL 语法正确；(3) 代码生成 — 确保括号匹配、缩进正确；(4) 分类任务 — 将输出限制为预定义的类别标签。主流框架如 Outlines、Guidance、SGLang 都支持引导生成。在 vLLM 中，可以通过 guided_decoding 参数启用 JSON Schema 或正则表达式约束。

⚠️ 引导生成可能降低输出质量

引导生成保证了格式正确性，但可能降低内容质量。原因是：当某些"好"的 token 被格式约束屏蔽后，模型被迫选择次优的 token，这可能导致后续生成偏离最优路径。例如，模型本来想输出 "The answer is 42"，但格式约束要求输出 JSON，模型可能被迫生成 {"answer": "42"}，但后续的推理可能因为格式转换而偏离原始思路。在使用引导生成时，需要在格式保证和内容质量之间找到平衡。

🎯 练习：实现一个简单的 Top-p 采样器

用 PyTorch 从零实现 Top-p 采样，并与 Top-k 和贪心解码对比：

import torch
import torch.nn.functional as F

def greedy_decode(logits):
    """贪心解码：总是选概率最高的"""
    return logits.argmax(dim=-1)

def top_k_sample(logits, k=50, temperature=1.0):
    """Top-k 采样"""
    logits = logits / temperature
    top_k_vals, top_k_ids = logits.topk(k)
    # 将不在 top-k 中的 token 设为 -inf
    probs = F.softmax(top_k_ids, dim=-1)
    idx = torch.multinomial(probs, 1)
    return top_k_ids.gather(-1, idx)

def top_p_sample(logits, p=0.9, temperature=1.0):
    """Top-p (Nucleus) 采样"""
    logits = logits / temperature
    sorted_logits, sorted_ids = logits.sort(descending=True)
    cumulative_probs = F.softmax(sorted_logits, dim=-1).cumsum(dim=-1)
    
    # 找到累积概率超过 p 的位置
    cutoff_mask = cumulative_probs - F.softmax(sorted_logits, dim=-1) >= p
    sorted_logits[cutoff_mask] = float('-inf')
    
    probs = F.softmax(sorted_logits, dim=-1)
    idx = torch.multinomial(probs, 1)
    return sorted_ids.gather(-1, idx)

# 对比测试
logits = torch.randn(1, 1000)  # 模拟模型输出
print("Greedy:", greedy_decode(logits).item())
print("Top-k:", top_k_sample(logits, k=10).item())
print("Top-p:", top_p_sample(logits, p=0.9).item())

实用技巧： 自回归生成的实际工程调优需要关注多个关键指标。首先，TTFT（首token延迟）主要由Prefill阶段决定，建议使用FlashAttention加速长prompt处理。其次，ITL（逐token延迟）由Decode阶段决定，可通过增大batch size和使用GQA减少KV Cache带宽来优化。在生产环境中，建议使用Continuous Batching替代静态批处理以提升GPU利用率至90%以上。温度参数的选择至关重要：事实问答类任务建议temperature=0，通用对话建议0.7，创意写作建议1.0-1.5。Top-p通常设为0.9，Top-k设为50即可覆盖大多数场景。此外，Repetition Penalty设为1.1-1.2可以有效减少重复生成。在多用户服务场景中，还需要合理设置max_model_len以控制KV Cache显存占用，避免因单个超长请求导致OOM。
深入探讨： 自回归生成的核心数学原理在于链式法则的分解：P(x₁,...,xₙ) = ∏P(xᵢ|x₁,...,xᵢ₋₁)。这一分解使得生成过程可以逐token进行，但也引入了暴露偏差（Exposure Bias）——训练时模型看到的是真实前文，推理时看到的是自己生成的可能有误的前文。推导：设模型在第t步的误差为εₜ，则累积误差随序列长度L呈指数增长：总误差 ≈ Σεₜ·(1+ε)^(L-t)。这就是为什么长文本生成质量会逐渐下降。数学本质上，Beam Search通过保留k条候选路径来缓解这一问题，但计算开销线性增加。Speculative Decoding则利用小模型的快速草稿和大模型的一次性验证，在不改变输出分布的前提下将延迟降低2-3倍。

✅ 参考答案与详解
  1. 核心原理: Prefill 和 Decode 是 LLM 推理的两个阶段，分别对应"读题"和"答题"过程。Prefill（预填充）阶段将用户输入的所有 token 一次性并行处理——4 个 token 同时输入 Transformer，计算所有位置的 Key 和 Value 向量并存入 KV Cache，输出第一个预测 token。这个阶段是并行计算，耗时约 10ms（取决于输入长度），充分利用了 GPU 的并行能力。Decode（解码）阶段则是逐 token 串行生成——每次只输入上一步预测的 1 个新 token，从 KV Cache 中读取之前所有位置的 K/V 向量，计算注意力并预测下一个 token。生成 100 个 token 需要 100 次前向传播，每次约 10-50ms。KV Cache 是关键优化：没有 KV Cache 时，每步需要重新计算所有已生成 token 的 K/V，复杂度 O(n²)；有 KV Cache 后，每步只计算 1 个新 token 的 K/V，复杂度降为 O(n)。例如生成"今天天气很好，适合出去走走"，Prefill 处理"今天天气"4 个 token 并预测"很"，然后 Decode 逐个输入"很"→"好"→","→"适"→...，每步复用之前的 KV Cache。实际应用场景：ChatGPT 的流式输出就是 Decode 阶段的逐 token 生成，用户看到的"打字机效果"正是每个 Decode 步的输出。
    关键要点: ① Prefill：并行处理所有输入 token，快（~10ms）② Decode：串行逐 token 生成，慢（每步 10-50ms）③ KV Cache 避免重复计算 K/V，将 Decode 复杂度从 O(n²) 降至 O(n) ④ Prefill 受限于输入长度（O(n²) 注意力），Decode 受限于生成长度（串行瓶颈）⑤ KV Cache 内存开销大：70B 模型 4K 上下文约需 2.5GB/请求
    实际案例: 在 vLLM 推理框架中，Prefill 和 Decode 被明确分离管理。Prefill 阶段使用 Flash Attention 加速长 prompt 处理，Decode 阶段使用 PagedAttention 管理 KV Cache 内存，通过 Continuous Batching 让短请求不必等待长请求结束。
    关联知识: Prefill/Decode 的分离直接影响推理系统的调度策略（Continuous Batching）和内存管理（PagedAttention）。KV Cache 的大小与模型层数、注意力头数、序列长度和维度相关，GQA/MQA 技术通过减少 KV head 数来压缩 Cache。Speculative Decoding 通过小模型快速生成草稿 + 大模型批量验证来突破 Decode 的串行瓶颈。
  2. 核心原理: Prefill 和 Decode 在工程中的性能瓶颈各有不同。Prefill 阶段的瓶颈是计算密集型——需要处理整个输入序列的 O(n²) 注意力计算，长 prompt（如 10K token）会导致 TTFT（首 token 延迟）显著增加。诊断方法是监控 TTFT 指标，如果超过 1 秒说明 Prefill 需要优化。解决方案：Flash Attention 通过分块计算和减少 HBM 访问将注意力加速 2-4 倍；对于超长 prompt 可以使用 Prompt Caching 缓存系统提示的 KV Cache。Decode 阶段的瓶颈是内存带宽密集型——每生成一个 token 需要读取全部 KV Cache，以 A100 为例，HBM 带宽 2TB/s，70B 模型 KV Cache 约 20GB/请求，单次 Decode 读取需 10ms。当 batch_size=32 时总 KV Cache=640GB，读取需 320ms 远超计算时间。诊断方法是监控 ITL（逐 token 延迟）和 GPU 利用率——如果 GPU 利用率低于 50% 说明内存带宽是瓶颈。解决方案：(1) GQA/MQA 减少 KV head 数，LLaMA-3 用 GQA 将 KV Cache 压缩 4-8 倍；(2) PagedAttention 将 KV Cache 分页管理，内存利用率从 20-40% 提升到接近 100%；(3) Continuous Batching 动态调度请求，GPU 利用率从 30-50% 提升到 80-90%；(4) KV Cache 量化（FP8）将 Cache 大小减半；(5) Speculative Decoding 用小模型加速大模型 Decode 2-3 倍。
    关键要点: ① Prefill 瓶颈：O(n²) 计算密集，影响 TTFT ② Decode 瓶颈：内存带宽密集，影响 ITL 和吞吐量 ③ KV Cache 是核心内存消耗者，batch_size 受限于 KV Cache 总量 ④ 诊断指标：TTFT、ITL、GPU 利用率、显存占用 ⑤ 优化组合：Flash Attention + GQA + PagedAttention + Continuous Batching
    实际案例: vLLM 的 benchmark 显示，使用 PagedAttention + Continuous Batching 后，LLaMA-13B 在 A100 上的吞吐量从 500 tokens/s 提升到 2500 tokens/s（5 倍提升）。Speculative Decoding 在代码生成任务中可实现 2.5 倍加速，因为代码的可预测性高，草稿模型的接受率可达 70-80%。
    关联知识: Prefill 的计算瓶颈与 Transformer 的 O(n²) 注意力复杂度直接相关，Flash Attention 的分块计算策略是关键优化。Decode 的带宽瓶颈与模型参数量和 KV Cache 大小相关，Scaling Law 表明更大模型性能更好但推理更贵，这就是"推理最优训练"（过度训练小模型）策略的由来。
  3. 核心原理: Prefill/Decode 架构的替代方案和互补技术主要包括不同的解码策略和推理优化方法。在解码策略层面：(1) 贪心解码（Greedy Decoding）每步选概率最高的 token，temperature=0 等价于贪心，确定性强但可能陷入局部最优——因为局部最优的 token 序列不一定组成全局最优序列；(2) Beam Search 保留 beam_width 条候选序列同时扩展，选累积概率最高的完整序列，比贪心更接近全局最优，但计算开销线性增加（beam_width=5 则计算量 5 倍），且生成的文本可能过于"安全"而缺乏多样性；(3) Top-k 采样只从前 k 个 token 中采样，但 k 固定无法适应不同上下文——数学题"k=50 太宽"，开放问题"k=50 太窄"；(4) Top-p（Nucleus）采样自适应选择候选集，累计概率达到 p 即截止，确定性高时候选集仅 1-2 个词，不确定时扩展到更多，是当前最主流的采样策略。在推理优化层面：(1) Speculative Decoding 用轻量草稿模型快速生成多个候选 token，大模型一次性验证，不改变输出分布的前提下加速 2-3 倍；(2) 对比解码（Contrastive Decoding）用大模型和小模型的 logits 差值采样，过滤"退化"内容；(3) 引导生成（Guided Generation）在采样时屏蔽不符合格式要求的 token，确保输出 100% 符合 JSON/SQL 等格式。实践中通常组合使用：Temperature → Top-k → Top-p 三层过滤，代码生成用 τ=0.2/k=10/p=0.9，创意写作用 τ=1.5/k=100/p=0.95。
    关键要点: ① 贪心解码快但可能局部最优；Beam Search 更优但计算开销大 ② Top-p 自适应候选集，优于固定 k 的 Top-k ③ Speculative Decoding 不改变输出质量，加速 2-3 倍 ④ 引导生成保证格式正确但可能降低内容质量 ⑤ 采样顺序固定：Temperature → Top-k → Top-p
    实际案例: 在 SQL 生成场景中，使用引导生成（JSON Schema 约束）确保输出 100% 是合法 JSON，但测试发现内容准确率下降约 5%——因为格式约束迫使模型选择次优 token。解决方案是在关键推理步骤放宽格式约束，只在最终输出时启用引导生成。
    关联知识: 采样策略的选择与语言模型的本质密切相关——模型输出的是概率分布而非确定答案，采样策略决定了如何从分布中选择 token。Temperature 的数学本质是对 logits 做缩放后 softmax，Top-p 的本质是截断分布后重新归一化。这些策略的组合效果直接影响生成质量，是 Prompt Engineering 和模型部署调优的核心知识。

深入探讨： 从推理优化的角度看，自回归生成的核心挑战在于Decode阶段的内存带宽瓶颈。每生成一个token，模型需要读取全部KV Cache，这意味着生成速度受限于HBM带宽而非计算能力。推导：以A100为例，HBM带宽2TB/s，70B模型的KV Cache约20GB/请求，单次Decode读取需10ms。若batch_size=32，总KV Cache=640GB，读取需320ms，远超计算时间。这就是为什么GQA（减少KV head数）、KV Cache量化（FP8减半）和PagedAttention（优化内存布局）是推理优化的三大核心技术。Speculative Decoding通过"草稿+验证"模式将多次串行Decode合并为一次批量验证，从另一个角度突破了带宽瓶颈。
常见误区： 学习自回归生成时，很多人只关注采样策略的理论描述而不实际测试不同参数的效果。建议用同一个prompt分别设置temperature=0/0.3/0.7/1.0/1.5生成10次，观察输出的确定性和多样性变化；对比Top-k=10/50/100和Top-p=0.8/0.9/0.95的组合效果，理解"多层过滤"的实际行为；用不同beam_width运行Beam Search对比输出质量。在实际运行中观察不同参数如何影响生成结果的连贯性、创造性和准确性。
常见误区： 不要以为自回归生成只需设置好temperature和Top-p就万事大吉。理解Prefill和Decode的性能差异是优化推理系统的基础——不了解KV Cache的内存开销就无法合理规划并发数，不理解Decode的串行特性就无法理解为什么投机解码能加速。同样，不理解采样策略的数学原理（如Top-p的累积概率截断机制），就无法在实际应用中做出合理的参数选择。掌握这些基础概念是正确使用推理框架和调优参数的前提。
