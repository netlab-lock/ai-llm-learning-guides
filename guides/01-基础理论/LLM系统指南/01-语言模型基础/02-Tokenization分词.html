✂️ 2. Tokenization (分词)

2.1 为什么需要分词？

LLM 不直接处理文字字符串，而是处理 token（整数 ID）。Tokenization 就是把文字切成 token 的过程。

分词的三种方案对比: 方案1: 按字 (Character-level) "Hello" → ['H', 'e', 'l', 'l', 'o'] → [72, 101, 108, 108, 111] 优点: 词汇表极小 (~256) 缺点: 序列极长，语义信息弱 方案2: 按词 (Word-level) "Hello world" → ['Hello', 'world'] → [15339, 995] 优点: 语义清晰 缺点: OOV 问题严重 (未登录词) "unhappiness" 如果不在词表中 → <UNK> 方案3: 子词 (Subword) ← 主流方案 "unhappiness" → ['un', 'happiness'] → [432, 8921] 优点: 平衡词汇表大小和覆盖率 缺点: 需要训练分词算法
💡 通俗类比：分词就像切蛋糕
按字切 = 切成碎末，每块太小没有意义；按词切 = 一整块太大，遇到新形状就装不下；子词切 = 切成合适的块，既有意义又能灵活组合。你不需要为每种形状的蛋糕准备专用模具 — 标准的子词"切法"就能应对任何形状。
⚠️ 常见误区：token = 一个"字"？
中文的一个 token 不一定是一个汉字。在英文为主的 tokenizer 中，"你好"可能被切成 3-4 个 token（因为是 UTF-8 字节序列）。这就是为什么中文在 GPT-2 上比英文贵 2-3 倍 — 同样的语义消耗更多 token，推理成本更高。选择合适的 tokenizer（如 Qwen 系列）可以大幅降低中文成本。
分词方案词汇表大小序列长度OOV 问题代表模型
字符级~256极长无早期实验
词级100K+短严重Word2Vec

2.2 BPE (Byte Pair Encoding)

GPT 系列使用的分词方法。

训练过程

# BPE 训练算法
1. 初始化: 词汇表 = 所有单字符 (如 a-z, 0-9, 标点)
2. 统计语料中所有相邻 token 对的出现频率
3. 找到频率最高的 pair (如 ('e', 's'))
4. 合并为新 token 'es'，加入词汇表
5. 在语料中执行所有 'e','s' → 'es' 的替换
6. 重复 2-5，直到词汇表达到目标大小
BPE 训练示例: 初始语料 (频率): "low" 5次 → l o w "lower" 2次 → l o w e r "newest"6次 → n e w e s t "widest"3次 → w i d e s t 第1轮: 最频繁 pair 是 (e, s) = 6+3 = 9次 合并: e s → es "newest" → n e w es t "widest" → w i d es t 第2轮: 最频繁 pair 是 (es, t) = 9次 合并: es t → est "newest" → n e w est "widest" → w i d est 第3轮: 最频繁 pair 是 (l, o) = 5+2 = 7次 合并: l o → lo "low" → lo w "lower" → lo w e r ...最终词汇表包含: lo, w, est, es, e, r, n, i, d, ...
💡 通俗类比：BPE 就像"常用缩写"的自动发现
想象你在看大量英文文本，发现 "e" 和 "s" 总是挨在一起（因为英语复数），你就会把 "es" 当作一个整体。BPE 就是这个过程的自动化 — 统计哪些字符对最常一起出现，把它们合并。训练完成后，常见的词（如 "the"）会成为单个 token，罕见词（如 "antidisestablishmentarianism"）会被拆成多个子词。
🔬 深入探讨：BPE 的贪心策略有何缺陷？
BPE 是贪心算法 — 每轮选最频繁的 pair，不保证全局最优。例如，"est" 和 "es" 都被加入了词表，但"es" 可能只是 "est" 的中间产物，单独存在的意义不大。Unigram 算法 (SentencePiece 支持) 用概率模型优化全局切分，理论上更优但实现更复杂。实践中两者效果差异不大。

2.3 Byte-level BPE

标准 BPE 以 Unicode 字符为基本单位，但 Unicode 字符集太大（15万+）。Byte-level BPE 以字节 (256 种) 为基本单位：

  • 可以处理任何语言、emoji、特殊字符
  • 永远不会出现 OOV — 最坏情况退回单字节
  • GPT-2+、DeepSeek、LLaMA-3 都使用
Byte-level BPE 编码示例: 原始文本: "Hello你好👋" UTF-8 字节: 48 65 6C 6C 6F E4 BD A0 E5 A5 BD F0 9F 91 8B H e l l o 你(3字节) 好(3字节) 👋(4字节) BPE 合并后可能变成: [Hello] [你好] [👋] ← 如果训练语料中有中文 或 [Hell][o] [你][好][👋] ← 如果训练语料中文很少
⚠️ 常见误区：同一个词总是被切成同样的 token？
不一定！BPE 的切分取决于词汇表。同一个中文词在 GPT-2（英文为主）中可能被切成 5-6 个 byte token，而在 Qwen（中文优化）中只占 1-2 个 token。这就是为什么不同模型的推理成本差异巨大。

2.4 WordPiece

BERT 使用的方法，和 BPE 类似但选择合并对的标准不同：

方法选择标准
BPE选出现频率最高的 pair
WordPiece选使语言模型似然增加最大的 pair
💡 通俗类比：BPE vs WordPiece 的区别
BPE 像是"投票制" — 谁出现得多就合并谁；WordPiece 像是"考试制" — 谁合并后模型预测得更准就选谁。WordPiece 理论上更精准，但计算开销更大。
🔬 深入探讨：WordPiece 的 ## 前缀是什么意思？
BERT 的 tokenizer 中，非词首的子词会加 "##" 前缀。例如 "playing" → ["play", "##ing"]。这告诉模型"##ing"不是一个独立的词，而是某个词的续接部分。BPE 不需要这种标记 — 它通过合并规则隐式地处理了这个问题。

2.5 SentencePiece

LLaMA、T5、Gemma 使用的方法：

  • 不依赖预分词（不需要先按空格/标点切分）
  • 直接处理原始文本流
  • 支持 BPE 和 Unigram 两种子词算法
  • 内置 Byte-level fallback
💡 通俗类比：SentencePiece 就像"通用水龙头"
传统分词需要先"预处理"（按空格切词），就像洗菜前要先切块。SentencePiece 直接处理原始文本流，就像一个万能水龙头 — 不管你放进去什么形状的食材，它都能处理。这对中文、日文等不用空格分词的语言特别重要。
⚠️ 常见误区：SentencePiece 是一种新算法？
不是。SentencePiece 是一个工具库，它支持 BPE 和 Unigram 两种算法。它最大的贡献是"语言无关的预处理" — 把空格也当作普通字符处理（用特殊符号 ▁ 表示词首），所以不需要任何语言特定的规则。

2.7 分词算法的数学原理

虽然 BPE 和 WordPiece 在实践中表现相似，但它们背后的数学模型有本质区别。理解这些区别有助于在特定场景下做出更好的选择。

BPE 的信息论视角

BPE 的合并策略可以理解为一种贪心压缩算法。每次合并最频繁的 pair，相当于最大化压缩率。从信息论的角度看，BPE 试图最小化编码整个语料库所需的总比特数。如果我们将每个 token 视为一个编码单元，那么 BPE 的目标等价于：

minimize ∑i len(token_i) × freq(token_i)

其中 len(token_i) 是 token 的编码长度，freq(token_i) 是出现频率。这本质上是一个霍夫曼编码的变体，但约束条件不同（BPE 要求词汇表大小固定）。

BPE 合并过程的树状表示: ┌─────────────────┐ │ 初始字符集 │ │ a,b,c,...,z,空格 │ └────────┬────────┘ │ 统计 pair 频率 ▼ ┌─────────────────┐ │ 合并最频繁 pair │ │ (e,s) → "es" │ └────────┬────────┘ │ 更新语料 ▼ ┌─────────────────┐ │ 合并最频繁 pair │ │ (es,t) → "est" │ └────────┬────────┘ │ 更新语料 ▼ ... │ ▼ ┌─────────────────┐ │ 词汇表达到上限 │ │ 停止合并 │ └─────────────────┘ 每一步的合并都是贪心的 — 选当前最好的，不保证全局最优。 但实践证明，这种贪心策略的效果非常好。

Unigram 算法的概率模型

与 BPE 的"自底向上"合并策略不同，Unigram 算法采用"自顶向下"的剪枝策略。它假设每种分词方式都有一个概率，目标是最大化语料库在所有分词方式下的似然：

L = ∑s∈D log P(s) = ∑s∈D log ∑x∈S(s) P(x)

其中 S(s) 是句子 s 的所有可能分词方式，P(x) 是某种分词方式的概率。Unigram 算法通过 EM 算法迭代优化 token 的概率，然后剪掉概率最低的 token，直到词汇表达到目标大小。

🔬 深入探讨：BPE vs Unigram 的本质区别

BPE 是确定性算法 — 给定相同的训练数据和词汇表大小，BPE 总是产生相同的词汇表和分词结果。而 Unigram 是概率性算法 — 同一个词可能有多种分词方式，每种方式都有一个概率。例如 "unhappiness" 在 Unigram 中可能以 60% 的概率分为 ["un", "happiness"]，以 30% 的概率分为 ["unhappi", "ness"]，以 10% 的概率分为 ["un", "happi", "ness"]。这种概率性分词在数据增强和正则化方面有独特优势，因为模型训练时可以看到同一文本的不同分词方式，相当于一种隐式的数据增强。

2.8 分词的工程实践

在实际项目中，分词不仅是学术问题，更是工程问题。以下是几个关键的工程考量：

特殊 Token 的设计

每个 tokenizer 都有一组特殊 token，用于标记序列的边界和特殊结构：

💡 分词器的版本兼容性

在生产环境中使用 LLM 时，务必确保分词器版本与模型版本完全匹配。即使同一个模型系列（如 LLaMA），不同版本的分词器也可能不同。LLaMA-1 用 SentencePiece BPE（32K 词表），LLaMA-3 用 tiktoken BPE（128K 词表）。如果用错了分词器，模型输出将完全无意义。HuggingFace 的 AutoTokenizer.from_pretrained() 会自动下载正确版本的分词器配置，这是最安全的方式。

分词对推理性能的影响

分词效率直接影响推理成本和延迟。同一个语义内容，用不同的 tokenizer 分词，token 数量可能相差数倍：

不同 tokenizer 对同一段中文的分词效率: 输入: "人工智能正在改变我们的生活方式和工作方式" GPT-2 (英文为主, 50K词表): → 约 25-30 个 token → 每个中文字符被拆成 2-3 个 byte token → 推理成本: 高 (30次前向传播) LLaMA-3 (多语言优化, 128K词表): → 约 12-15 个 token → 常见中文词被合并 → 推理成本: 中等 (15次前向传播) Qwen2.5 (中文优化, 151K词表): → 约 8-10 个 token → 中文词汇高度压缩 → 推理成本: 低 (10次前向传播) 结论: 选择合适的 tokenizer 可以节省 2-3 倍的推理成本！
⚠️ 分词边界对模型理解的影响

分词边界会显著影响模型对文本的理解。例如 "乒乓球拍卖完了" 这句话，如果分词为 ["乒乓球", "拍卖", "完了"]，模型会理解为"乒乓球拍被拍卖完了"；如果分词为 ["乒乓球拍", "卖完", "了"]，模型会理解为"乒乓球拍卖完了"。虽然现代大模型通过上下文可以处理大部分歧义，但在关键场景（如医疗、法律文本）中，分词歧义仍可能导致严重问题。

自定义分词器的场景

在某些场景下，使用现成的分词器可能不够。以下情况需要考虑自定义分词器：

  • 专业领域：医学、法律、化学等领域有大量专业术语，通用分词器可能将其拆成无意义的碎片
  • 新语言支持：如果目标语言在训练语料中占比很小，通用分词器的效率会很低
  • 代码生成：代码有特殊的语法结构（缩进、括号、关键字），专用分词器可以提高效率
  • 压缩需求：如果需要在极长的上下文中工作，更高效的分词器可以节省宝贵的上下文窗口

🎯 练习：实现一个简单的 BPE 分词器

从零实现 BPE 算法，加深对分词过程的理解：

def train_bpe(corpus, vocab_size):
    """从零训练一个 BPE 分词器"""
    # 初始化：将所有文本拆成字符
    tokens = [list(word) for word in corpus.split()]
    vocab = set()
    for word_tokens in tokens:
        vocab.update(word_tokens)
    
    merges = []
    while len(vocab) < vocab_size:
        # 统计所有相邻 pair 的频率
        pair_counts = {}
        for word_tokens in tokens:
            for i in range(len(word_tokens) - 1):
                pair = (word_tokens[i], word_tokens[i+1])
                pair_counts[pair] = pair_counts.get(pair, 0) + 1
        
        if not pair_counts:
            break
        
        # 找到最频繁的 pair
        best_pair = max(pair_counts, key=pair_counts.get)
        new_token = best_pair[0] + best_pair[1]
        merges.append(best_pair)
        vocab.add(new_token)
        
        # 执行合并
        new_tokens = []
        for word_tokens in tokens:
            i = 0
            new_word = []
            while i < len(word_tokens):
                if (i < len(word_tokens) - 1 and 
                    word_tokens[i] == best_pair[0] and 
                    word_tokens[i+1] == best_pair[1]):
                    new_word.append(new_token)
                    i += 2
                else:
                    new_word.append(word_tokens[i])
                    i += 1
            new_tokens.append(new_word)
        tokens = new_tokens
    
    return vocab, merges

2.9 分词质量对下游任务的影响

分词质量直接影响模型的训练效率和推理质量。一个设计良好的分词器应该满足以下条件：(1) 常见词汇被编码为单个 token；(2) 罕见词汇可以被分解为有意义的子词；(3) 不存在无法编码的字符（无 OOV）；(4) 词汇表大小适中，不会导致 Embedding 层过大。

分词粒度与模型性能的关系

分词粒度（即每个 token 平均包含多少字符）对模型性能有显著影响。粒度太细（接近字符级），序列会很长，模型需要更多层才能捕获长距离依赖；粒度太粗（接近词级），词汇表会很大，Embedding 层参数过多，且罕见词的 Embedding 训练不充分。

分词粒度的权衡: 粒度细 (字符级) 粒度粗 (词级) ←───────────────────────────────────────→ 字符级 子词级 (BPE) 词级 | | | | 词汇表: ~256 | 词汇表: 32K~150K | 词汇表: 100K+ | 序列: 极长 | 序列: 适中 | 序列: 短 | OOV: 无 | OOV: 无 | OOV: 严重 | 语义: 弱 | 语义: 平衡 | 语义: 强 | | | | 训练: 慢 | 训练: 适中 | 训练: 快(但需大词表) | 推理: 慢 | 推理: 适中 | 推理: 快(但OOV多) | | | ↑ 最佳平衡点 (主流方案)
🔬 深入探讨：多语言分词的公平性问题

以英文为主训练的分词器对非英文语言存在系统性不公平。研究表明，GPT-2 的分词器处理同样语义的文本时，印地语、阿拉伯语、中文等语言的 token 数量是英文的 3-10 倍。这不仅意味着更高的推理成本（按 token 计费的 API），还意味着更短的有效上下文窗口（同样 4K token，中文能表达的内容远少于英文）。LLaMA-3 将词表从 32K 扩大到 128K，Qwen 系列进一步优化到 151K，都是为了解决这个公平性问题。在选择模型时，如果主要使用中文，应该优先选择中文分词效率高的模型（如 Qwen、DeepSeek），不仅因为成本更低，还因为更大的有效上下文窗口能处理更复杂的任务。

分词错误的传播效应

分词处于 LLM 处理管线的最前端，分词阶段的错误会向后传播并放大。以下是几种常见的分词错误及其影响：

错误类型示例影响解决方案
过度切分"ChatGPT" → ["Chat", "G", "PT"]语义碎片化，理解困难添加领域专用词表
错误合并"unhappy" → ["unhappy"]（整体）丢失构词信息调整合并策略
语言混淆"Hello你好" → 混合编码跨语言理解困难使用多语言优化分词器
特殊字符emoji、数学符号被拆成字节序列过长，语义丢失扩大词表覆盖特殊字符

深入探讨： 从BPE分词的核心原理出发，BPE（Byte Pair Encoding）算法通过迭代合并最高频的相邻字符对来构建子词词表。为什么BPE优于固定词表？推导：设语料中出现N种不同字符组合，BPE通过频率驱动的合并策略使词表大小V与覆盖文本量之间达到最优平衡，等价于信息熵最小化。数学本质上，BPE是在子词粒度上寻找最优压缩编码，使常见词保持完整、罕见词被拆解为可组合的子词片段，将OOV（未登录词）率降至接近零。
