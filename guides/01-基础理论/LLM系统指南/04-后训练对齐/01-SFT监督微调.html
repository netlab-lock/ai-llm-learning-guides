📝 1. SFT (监督微调)

1.1 为什么需要 SFT？

预训练模型只是"文字续写器" — 它会续写文本，但不会按指令回答问题。SFT（Supervised Fine-Tuning，监督微调）是后训练对齐的第一步，其核心目标是将预训练语言模型从一个通用的"文本补全器"转变为一个能够理解并遵循人类指令的"对话助手"。这个过程涉及数据准备、模型架构调整、训练策略设计等多个技术环节，是大语言模型工程中最基础也是最关键的步骤之一。SFT 的质量直接决定了后续 RLHF、DPO、GRPO 等对齐方法的效果上限。如果 SFT 阶段没有建立好基本的指令遵循能力，后续的偏好优化就如同在沙滩上建城堡——基础不牢，地动山摇。

预训练模型 vs SFT 后的模型: 预训练模型: 用户: "请翻译成英文: 今天天气很好" 模型: "今天天气很好。明天可能会下雨。最近气温..." ← 续写，不听指令 SFT 后: 用户: "请翻译成英文: 今天天气很好" 模型: "The weather is very nice today." ← 按指令回答
💡 通俗类比：预训练 vs SFT

预训练就像让一个人读完所有百科全书 — 他知识渊博，但你问他问题时他只会继续背书。SFT 就像给他上"客服培训课" — 教他听问题、组织语言、给出有用的回答。知识量没变，但表达方式完全变了。换句话说，预训练阶段学习的是语言的统计规律和世界知识，而SFT阶段学习的是如何利用这些知识来服务人类的需求。

⚠️ 常见误区：SFT 不是"教新知识"
SFT 的主要目的不是注入新知识，而是激活预训练中已有的知识，并教会模型以"问答"格式输出。如果你需要模型掌握训练数据中没有的知识，应该用 RAG 或继续预训练，而不是 SFT。混淆这两者会导致"知识注入幻觉"——模型表面上能回答新问题，实际上是在"编造"看似合理的答案。

1.2 SFT 数据格式

SFT 数据格式决定了模型如何理解输入和输出的关系。选择合适的数据格式不仅影响训练效果，还影响模型在推理时的指令遵循能力。以下是主流数据格式的详细对比和选择建议。在实际项目中，建议先确定目标格式，然后将所有数据统一转换为该格式后再开始训练。

# 标准 SFT 数据格式
{
  "instruction": "请将以下文本翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is very nice today."
}

# 多轮对话格式
{
  "conversations": [
    {"role": "user", "content": "什么是机器学习？"},
    {"role": "assistant", "content": "机器学习是人工智能的一个分支..."},
    {"role": "user", "content": "它和深度学习有什么区别？"},
    {"role": "assistant", "content": "深度学习是机器学习的子集..."}
  ]
}
💡 通俗类比：数据格式像"模板"

SFT 数据格式就像给模型提供"填空模板"：instruction 是题目，input 是素材，output 是标准答案。模型学会的是：看到这类模板，就按这个格式回答。模板越一致，模型学得越快越准。实践中，数据格式的统一性对模型的指令遵循能力影响极大——格式混乱的数据会让模型困惑，不知道该以何种方式回答。

1.3 SFT 训练细节

# SFT 的损失函数: 只在 assistant 回答部分计算 loss
# user 的部分不参与 loss 计算

token:  [user问] [assistant答] [user问] [assistant答]
loss:    xxxxxx    ✓✓✓✓✓✓       xxxxxx    ✓✓✓✓✓✓
         ↑不计算    ↑计算loss     ↑不计算    ↑计算loss

# 这叫做 "loss mask" — 只让模型学习如何回答
⚠️ 常见误区：为什么不在 user 部分计算 loss？
如果在 user 部分也计算 loss，模型会学会"生成问题"而不是"回答问题"。这在多轮对话中尤其有害 — 模型可能学会自问自答。loss mask 确保模型只学习 assistant 的回答模式。此外，user 部分通常是模板化的（如"请帮我..."），如果让模型学习这些模板，会导致模型在推理时也倾向于生成类似的请求文本。
🔬 深入原理：SFT 的 Loss 计算与梯度传播机制
在 PyTorch 中，loss mask 的标准实现方式是将 user token 的 label 设为 -100（CrossEntropyLoss 的 ignore_index）。每个 token 位置的 loss 只在 label != -100 时计算。这意味着一个 batch 中实际参与 loss 计算的 token 可能远少于总 token 数，直接影响梯度的方差和训练稳定性。 具体来说，假设 batch 中有 32 个样本，每个样本平均 1024 个 token，但只有 512 个 token 是 assistant 的回答。那么实际上只有 50% 的 token 参与梯度计算。这带来的问题是： 1. 梯度方差增大——不同 batch 的有效 token 数量差异大 2. 学习率需要相应调整——通常比全 token 训练时稍大 3. 需要使用 gradient accumulation 来稳定训练 一个常见的实现细节是：在计算 loss 时，对有效 token 数进行归一化（而非对总 token 数归一化），以确保不同 batch 的 loss 量级一致。Hugging Face 的 Trainer 默认使用这种策略（loss_type="token_mean"），而有些框架使用"sample_mean"策略，后者对长短不一的样本更公平。

1.3.1 SFT 的注意力机制与位置编码

在多轮对话中，模型应该只能看到当前轮次之前的对话历史，而不能"偷看"未来的回复。这通过因果注意力掩码（Causal Attention Mask）来实现，确保自回归生成的正确性。

多轮对话的注意力掩码: 用户轮1 助手轮1 用户轮2 助手轮2 [u1] [a1] [u2] [a2] [u1]: ✓ ✗ ✗ ✗ ← 只看到自己 [a1]: ✓ ✓ ✗ ✗ ← 看到u1和自己 [u2]: ✓ ✓ ✓ ✗ ← 看到前面所有 [a2]: ✓ ✓ ✓ ✓ ← 看到全部历史 ✓ = 可以看到 ✗ = 被遮蔽 这就是标准的 causal mask，确保自回归生成的正确性。
💡 实践技巧：Flash Attention 2 与 SFT 的结合
在 SFT 训练中使用 Flash Attention 2 可以显著降低显存占用（从 O(n²) 降到接近 O(n)），同时保持训练速度。对于长对话（4096+ tokens），Flash Attention 几乎是必需的。使用方法很简单：在训练配置中设置 attn_implementation="flash_attention_2" 即可。需要注意的是，Flash Attention 要求 CUDA 11.6+ 和特定的 GPU 架构（Ampere 或更新）。

1.3.2 LoRA 与全参数微调的对比

在 SFT 中，选择全参数微调（Full Fine-Tuning）还是参数高效微调（PEFT，如 LoRA）是一个关键决策。全参数微调更新模型的所有参数，效果最好但显存需求巨大；LoRA 只更新低秩分解的增量矩阵，显存需求小但效果可能略逊。

对比维度全参数微调LoRA (rank=64)QLoRA (4-bit)
可训练参数占比100%~0.5-2%~0.5-2%
7B 模型显存需求~120GB (bf16)~24GB~12GB
训练速度基准~1.2x 快~0.8x 慢
最终效果最好接近全参数略逊 LoRA
适用场景资源充足、追求最优资源有限、快速迭代消费级 GPU
🔬 深入原理：LoRA 的数学原理与秩的选择
LoRA（Low-Rank Adaptation）的核心思想是：将权重更新矩阵 ΔW 分解为两个低秩矩阵的乘积 ΔW = BA，其中 B ∈ R^(d×r)，A ∈ R^(r×k)，r 远小于 d 和 k。这样，原本需要 d×k 个参数的更新，现在只需要 r×(d+k) 个参数。 直觉上，LoRA 假设微调过程中的权重变化是"低秩"的——即模型只需要在一个低维子空间中调整就能适应新任务。这个假设在实践中被证明是成立的，因为 SFT 本质上只是调整模型的"输出风格"，而非改变其核心知识表示。 秩 r 的选择是一个重要超参数： - r=8-16：适合简单的风格转换任务 - r=32-64：适合复杂的指令遵循训练 - r=128-256：接近全参数微调，但显存优势减小 实践中，r=64 是一个很好的默认值。可以通过观察 LoRA 矩阵的奇异值衰减速度来判断是否需要更大的秩——如果前几个奇异值包含了大部分能量，说明低秩假设成立。

  • Alpaca: 用 GPT-4 生成 52K 条 SFT 数据，效果也不错
  • 关键: 数据多样性、回答质量、格式一致性
SFT 数据质量金字塔: ▲ / \ / 质量 \ ← 最重要: 回答准确性、无幻觉 /--------\ / 多样性 \ ← 覆盖不同任务类型、领域 /------------\ / 格式一致性 \ ← 统一的对话模板、长度分布 /----------------\ / 数量 (够用即可) \ ← 1K~50K 条通常足够 /____________________\
⚠️ 常见误区：数据越多越好？
LIMA 论文证明 1000 条高质量数据 > 10万条低质量数据。低质量 SFT 数据会导致：模型学到错误的回答模式、格式不一致降低指令遵循能力、甚至注入错误知识。清洗数据比堆数据更重要。一个实用的数据质量检查清单包括：答案是否准确、是否包含幻觉、格式是否统一、长度是否合理、是否覆盖多种任务类型。

1.4.1 数据生成与清洗策略

在实际项目中，SFT 数据通常来自三个渠道：人工标注、模型生成（如用 GPT-4 生成）、以及从开源数据集中筛选。无论哪种来源，数据清洗都是必不可少的步骤。数据清洗的具体流程包括：首先进行去重处理，移除完全相同或高度相似的样本；然后进行质量筛选，通过自动化规则和人工抽检确保回答质量；最后进行格式规范化，确保所有样本遵循统一的对话模板。这三个步骤缺一不可，任何一个环节的疏忽都可能导致最终模型效果不佳。

数据来源优点缺点典型做法
人工标注质量最高、可控成本高、速度慢专业标注团队，多轮审核
模型生成（Self-Instruct）成本低、可扩展可能有错误、多样性有限GPT-4 生成 + 人工筛选
开源数据集免费、即用质量参差不齐OASST、ShareGPT 等筛选
用户日志真实场景隐私问题、噪声多脱敏 + 质量过滤
🔬 深入原理：数据去污染（Decontamination）与评测基准保护
一个容易被忽视但极其重要的问题是数据去污染。如果 SFT 数据中包含了评测基准（如 MMLU、GSM8K、HumanEval）的题目或答案，模型在这些基准上的表现就会虚高——这被称为"评测基准泄漏"（Benchmark Contamination）。 去污染的标准做法是： 1. 计算 SFT 数据与评测集之间的 n-gram 重叠（通常用 13-gram） 2. 移除与评测集有高重叠度的样本 3. 使用 MinHash 或 SimHash 进行高效的近似匹配 4. 对于多语言评测集，需要对每个语言版本分别去污染 OpenAI 和 Meta 在发布模型时都会详细说明其去污染流程。如果你自己构建 SFT 数据集，强烈建议在训练前进行去污染检查，否则你的模型评测结果将不可信。

💡 通俗类比：SFT 超参数的"火候"

SFT 就像炒菜的最后一步"收汁" — 火候（学习率）太大容易糊（过拟合），太小收不住（欠拟合）。通常用小火（低 LR）快炒（少 epoch），因为预训练模型已经有很好的基础，SFT 只需要微调"表达方式"。对于 LoRA 微调，由于可训练参数少，可以适当使用较大的学习率（如 2e-4）来加速收敛。需要注意的是，学习率和 epoch 数之间存在交互关系——较大的学习率配合较少的 epoch 数通常效果更好，而较小的学习率则需要更多 epoch 来充分收敛。实践中建议先用默认参数训练一个基线模型，然后根据训练曲线和评测结果进行针对性调整。

🔬 深入原理：学习率调度与梯度裁剪策略
SFT 通常使用 cosine decay 或 linear decay 调度。一种高效的策略是：先用较大学习率快速收敛（warmup 5%~10% 步数），然后 cosine 衰减到初始值的 10%。总步数少时（如 1 epoch），可以直接用 constant LR + warmup。 梯度裁剪（Gradient Clipping）在 SFT 中同样重要。通常设置 max_grad_norm=1.0，防止个别异常样本导致梯度爆炸。在使用 loss mask 时，由于有效 token 数量变化大，梯度方差也更大，建议使用更严格的裁剪阈值（如 0.5）。 另一个高级技巧是使用 "Packing" 策略：将多个短样本拼接到一个序列中，用特殊 token 分隔。这可以显著提高 GPU 利用率（避免大量 padding），但需要注意在不同样本之间正确设置注意力掩码（防止跨样本的注意力）。

1.6 SFT 的评估方法

SFT 训练完成后，如何评估模型的效果？除了标准的 benchmark 评测外，还需要关注以下几个维度。评估的目的是确认模型是否真正学会了指令遵循能力，而不仅仅是记住了训练数据中的模式。一个全面的评估体系应该覆盖模型能力的多个方面，包括指令遵循的准确性、回答的质量和完整性、多轮对话的连贯性、以及输出格式的规范性。在评估过程中，建议同时使用自动评测和人工评测两种方式，因为自动评测虽然高效但可能遗漏一些主观质量维度。

⚠️ 重要提醒：过拟合检测与应对策略
SFT 最容易犯的错误是过拟合。由于 SFT 数据量通常较少（1K-50K），模型很容易"记住"训练数据而非"学会"回答模式。过拟合的信号包括：训练 loss 持续下降但评测分数不升、模型开始输出与训练数据高度相似的回答、对措辞变化过度敏感。应对策略：使用 early stopping、增大 dropout、减少 epoch 数、增加数据多样性。

1.7.2 SFT 数据质量检查清单

在开始 SFT 训练之前，务必完成以下数据质量检查：

数据质量是 SFT 成功的基石。根据 LIMA 论文的研究，仅 1000 条精心准备的高质量数据就能训练出效果优秀的模型。相反，10 万条低质量数据反而会让模型学到错误的回答模式。因此，投入时间在数据质量上远比增加数据量更有价值。以下是经过实践验证的数据质量检查流程：首先进行格式一致性检查，确保所有样本遵循统一的对话模板；然后进行内容准确性验证，通过抽检和自动化工具确保回答事实正确；接着进行多样性评估，确保覆盖多种任务类型和难度级别；最后进行安全性审查，移除包含敏感信息或有害内容的样本。

  • 准确性验证：抽检 5% 的数据，确认回答事实正确且无幻觉
  • 格式一致性：确保所有数据使用统一的对话模板（如 ChatML）
  • 长度分布：检查回答长度分布是否合理，移除过短（<10字）和过长（>4096字）的样本
  • 多样性检查：确保覆盖多种任务类型，避免某一类任务过度集中
  • 去污染检查：确保训练数据不包含评测基准的题目或答案
  • 去重处理：使用 MinHash 或精确匹配移除重复或近似重复的样本
  • 安全性审查：移除包含有害、偏见或隐私信息的样本

✅ 参考答案与详解
  1. 核心原理: SFT（监督微调）的核心原理是将预训练语言模型从一个"文本续写器"转变为"指令遵循助手"。预训练模型通过海量语料学会了语言的统计规律和世界知识，但它的行为模式是"续写"——给定前文预测下一个token。SFT通过在高质量的"指令-回答"对上进行有监督学习，教会模型识别指令格式并生成符合人类期望的回答。关键的技术细节包括：(1) Loss Mask机制——只在assistant回答部分计算交叉熵损失，user输入部分通过label设为-100被忽略，确保模型只学习"回答"而非"生成问题"；(2) 数据格式统一——使用Alpaca、ShareGPT或ChatML等标准化格式；(3) 学习率远小于预训练（1e-5~5e-5），通常只训练1-3个epoch避免过拟合。
    关键要点: ①SFT不注入新知识，而是激活预训练中已有的知识并教会模型以问答格式输出 ②Loss Mask是SFT的核心机制，只在response部分计算loss防止模型学会自问自答 ③数据质量远比数量重要，LIMA论文证明1000条高质量数据优于10万条低质量数据 ④LoRA等PEFT方法可以将显存需求从120GB降至12-24GB，使消费级GPU也能进行SFT ⑤超参数选择上，小学习率+少epoch是关键，预训练模型已有良好基础，SFT只需微调表达方式
    实际案例: Stanford Alpaca项目使用GPT-4生成的52K条SFT数据对LLaMA-7B进行微调，仅需几百美元成本就获得了接近ChatGPT的指令遵循能力。在实际工程中，一个典型的SFT流程是：准备Alpaca格式数据→配置LoRA(rank=32, alpha=64, target_modules=["q_proj","v_proj"])→设置lr=2e-4, epochs=3→训练后在MT-Bench上评估。如果loss不下降，首先检查loss mask是否正确覆盖了assistant回答部分。
    关联知识: SFT是所有后训练对齐方法的第一步和基础。RLHF在SFT基础上训练奖励模型并用PPO优化；DPO在SFT模型作为reference的基础上直接优化偏好；GRPO+RLVR也需要SFT先做格式对齐（如DeepSeek-R1的阶段1冷启动SFT）。SFT的质量直接决定了后续所有对齐方法的效果上限——如果SFT阶段没有建立好基本的指令遵循能力，后续的偏好优化就如同在沙滩上建城堡。
  2. 核心原理: SFT在实际工程中最常见的性能瓶颈包括：(1) 训练loss不下降——通常是loss mask配置错误或学习率过小，需要检查user/assistant token的mask是否正确实现（label设为-100），或适当增大学习率；(2) 过拟合——SFT数据量通常较少（1K-50K），模型容易"记住"而非"学会"，表现为训练loss持续下降但评测分数不升，解决方案包括early stopping、增大dropout、减少epoch数；(3) 显存不足——7B模型全参数微调需要约120GB显存，可通过LoRA(约24GB)或QLoRA(约12GB)大幅降低需求；(4) 输出格式混乱——训练数据格式不一致导致模型不知道该以何种方式回答。
    关键要点: ①使用梯度检查点和Flash Attention 2可显著降低显存占用，Flash Attention将注意力复杂度从O(n²)降到接近O(n) ②Packing策略将多个短样本拼接到一个序列中提高GPU利用率，但需正确设置跨样本的注意力掩码 ③使用cosine decay学习率调度配合gradient clipping(max_grad_norm=1.0)稳定训练 ④数据去污染（decontamination）是必要的，防止评测基准泄漏导致分数虚高 ⑤监控loss曲线的同时关注评测指标，loss下降不等于效果提升
    实际案例: 某团队在微调Qwen2-7B时发现loss快速下降到接近0但MT-Bench分数反而下降，诊断发现是SFT数据中存在大量重复样本导致过拟合。解决方案是先用MinHash去重（移除约40%的重复样本），再将epoch从3减为1，最终MT-Bench提升1.5分。另一个常见案例是loss mask配置错误——将整个对话历史都mask掉只计算最后一轮loss，导致模型无法学会多轮对话中的上下文一致性。
    关联知识: SFT的故障排查技能直接迁移到RLHF/DPO/GRPO的训练中。例如loss mask机制在DPO中同样重要（需要分别mask prompt部分），学习率调度策略在GRPO训练中也需要类似调整（GRPO通常用更小的lr=1e-6）。Flash Attention 2和梯度检查点等显存优化技术在所有后训练方法中通用。
  3. 核心原理: SFT的主要替代方案包括：(1) 继续预训练——在特定领域语料上继续自回归训练，适合注入新领域知识但不擅长指令遵循；(2) RAG（检索增强生成）——通过外部知识库补充模型知识，不需要修改模型参数，适合知识密集型任务；(3) RLHF/DPO——在SFT基础上进一步对齐人类偏好，效果更好但成本更高。三者的核心权衡在于：SFT是"教模型如何回答"，继续预训练是"教模型新知识"，RAG是"给模型查资料的能力"。混淆这三者是最常见的错误——如果你需要模型掌握训练数据中没有的知识，应该用RAG或继续预训练而不是SFT。
    关键要点: ①SFT的主要目的是激活已有知识并教会问答格式，而非注入新知识 ②继续预训练适合领域适配（如医疗、法律），SFT适合任务适配（如对话、摘要） ③RAG适合需要实时更新知识的场景，SFT适合需要固定行为模式的场景 ④实际项目中三者通常组合使用：先继续预训练注入领域知识，再SFT建立指令遵循能力 ⑤全参数微调效果最好但资源需求大(120GB)，LoRA是性价比最高的选择(24GB)，QLoRA可在消费级GPU上运行(12GB)
    实际案例: 构建一个医疗问答助手的典型流程是：先在医疗文献上继续预训练注入医学知识→用高质量医疗问答对进行SFT建立指令遵循能力→用RAG接入最新的医学指南确保知识时效性→最后用DPO对齐医生的偏好（如回答风格、详细程度）。如果只做SFT而不做继续预训练，模型虽然会遵循指令但缺乏深度医学知识；如果只做继续预训练，模型虽然有知识但不会以问答形式回答。
    关联知识: SFT与后续对齐方法（RLHF、DPO、GRPO）形成递进关系。SFT建立基础的指令遵循能力，是后续所有方法的"地基"。在DeepSeek-R1的四阶段训练流程中，阶段1的冷启动SFT为后续GRPO+RLVR训练提供了格式基础。ORPO方法甚至尝试将SFT和DPO合并为一步，直接在偏好数据上同时学习格式和偏好。SFT数据质量检查清单（准确性验证、格式一致性、长度分布、多样性检查、去污染检查、去重处理、安全性审查）同样适用于后续所有训练阶段的数据准备。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
