🏆 2. RLHF 经典方法

2.1 RLHF 三步流程

RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）是 2022 年 ChatGPT 成功的核心技术之一。它通过引入人类偏好信号来指导模型优化，使模型的回答更符合人类的价值观和期望。RLHF 的完整流程包含三个阶段：监督微调（SFT）、奖励模型训练（RM）、以及近端策略优化（PPO）。每个阶段都有其独特的技术挑战和设计选择。

RLHF 完整流程: ┌─────────────────────────────────────────────────────────┐ │ Step 1: SFT (已在 01 讲解) │ │ 用指令-回答对微调 → 得到 SFT 模型 π_SFT │ │ ✅ 已完成 │ └────────────────────────┬────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 2: 训练奖励模型 (Reward Model) │ │ │ │ 收集偏好数据: 训练 RM: │ │ 问题: "什么是量子计算？" Loss = -log σ(r(A) - r(B)) │ │ 回答A: 详细解释 ← 好 让好回答分数 > 坏回答分数 │ │ 回答B: 含糊其辞 ← 差 RM 从 SFT 模型初始化 │ │ 人类标注: A > B │ └────────────────────────┬────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 3: PPO 优化 │ │ │ │ Reward = RM(response) - β·KL(π||π_ref) │ │ 用 PPO 算法优化模型: │ │ 最大化奖励，同时不偏离 SFT 模型太远 │ └─────────────────────────────────────────────────────────┘
💡 通俗类比：RLHF 像请家教
SFT 是"上课学知识"，RLHF 是"请家教做一对一辅导"。人类标注者就像家教 — 看了学生的两种回答后说"A更好"。奖励模型从这些评价中学会"什么是好答案"，然后用强化学习让学生（模型）朝着好答案的方向改进。这个过程的关键洞察是：我们不需要告诉模型"好答案的每个细节"，只需要告诉它"A比B好"，模型就能从中提炼出改进方向。
⚠️ 常见误区：RLHF 不是直接用人类反馈训练
RLHF 不是每一步都请人类打分。而是先训练一个奖励模型（RM），然后用 RM 的打分作为代理信号来训练。人类只参与偏好数据收集阶段。这是因为人类反馈太慢太贵，无法直接用于强化学习的在线采样。一个标注员每小时大约能标注 10-20 对偏好数据，但 PPO 训练每步需要评估成千上万个回答——这个差距只能用 RM 来弥合。

2.2 奖励模型 (RM)

2.2.1 RM 的训练原理

奖励模型是 RLHF 流程中的核心组件，它将人类的偏好判断转化为一个可微分的标量评分函数。RM 的训练基于 Bradley-Terry 模型——一个经典的成对比较统计模型，其核心假设是：人类选择好回答的概率与两个回答的奖励值之差通过 sigmoid 函数相关。

💡 通俗类比：奖励模型像"裁判"
奖励模型就像比赛裁判 — 看到选手的两种表现后给出评分。训练数据是人类标注的"A比B好"，RM 学会的是给好回答打高分、差回答打低分。但裁判也有偏见 — RM 可能被"欺骗"（奖励黑客），所以需要 KL 惩罚作为"反作弊规则"。RM 的另一个重要特性是它输出的是"相对分数"而非"绝对分数"——它更擅长判断"A比B好多少"，而非"A有多好"。

⚠️ 常见误区：RM 的泛化能力有限
RM 只在训练分布内表现好。如果推理时模型生成了训练中没见过的回答风格，RM 的评分可能不可靠。这也是"奖励黑客"的根源 — 模型找到 RM 的盲区，生成 RM 喜欢但人类不喜欢的回答。解决方法包括：(1) 扩大偏好数据的多样性；(2) 使用多个 RM 的集成；(3) 使用 KL 惩罚限制模型探索范围。

2.2.3 偏好数据标注指南

高质量的偏好数据是 RLHF 成功的基础。标注质量直接影响 RM 的准确性，进而影响最终模型的效果。以下是偏好数据标注的关键原则：

标注维度评估标准权重建议
准确性事实正确、无幻觉最重要
有帮助性回答了用户的问题高
安全性无有害/不当内容高（一票否决）
流畅性语言自然、逻辑清晰中
格式结构合理、易于阅读低
🔬 深入原理：Bradley-Terry 模型的数学推导
Bradley-Terry 模型假设每个回答 y 有一个潜在的"实力"参数 r(y)，人类选择 y_w 优于 y_l 的概率为： P(y_w > y_l) = σ(r(y_w) - r(y_l)) 其中 σ 是 sigmoid 函数。这个模型的优美之处在于：它将"绝对评分"问题转化为了"相对比较"问题。人类很难对一个回答给出 1-10 的绝对分数（主观性强），但很容易判断两个回答哪个更好（一致性高）。 从最大似然估计的角度，给定 N 对偏好数据 {(y_w^i, y_l^i)}，RM 的训练目标就是最大化： L = Σ log σ(r(y_w^i) - r(y_l^i)) 这等价于最小化负对数似然，即我们之前看到的 Loss_RM。 一个重要的理论性质是：当偏好数据满足 Bradley-Terry 假设时，RM 学到的评分 r(y) 是人类偏好的充分统计量。这意味着 RM 的评分包含了偏好数据中的所有信息。

2.3 PPO 算法详解

# PPO 目标函数
J(θ) = E[ min(
    r_t(θ) · A_t,                              # 策略比率 × 优势
    clip(r_t(θ), 1-ε, 1+ε) · A_t              # 裁剪后的版本
)]

r_t(θ) = π_θ(a|s) / π_old(a|s)              # 新旧策略比率
A_t = R(s,a) - V(s)                           # 优势 = 奖励 - 基线

# KL 散度惩罚: 防止偏离参考模型太远
Reward = RM_score - β · KL(π_θ || π_ref)
PPO 训练中的 4 个模型: ┌─────────────────────────────────────────────────┐ │ PPO 训练循环 │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ Policy π │ │Reference │ │ Reward │ │ │ │ (Actor) │ │ π_ref │ │ Model │ │ │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ │ │ │ │ ▼ ▼ ▼ │ │ 生成回答 计算 KL 惩罚 计算奖励分数 │ │ │ │ │ │ │ ▼ ▼ ▼ │ │ ┌──────────────────────────────────────┐ │ │ │ Value Model (Critic) │ │ │ │ 估计状态价值 V(s) │ │ │ └──────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ 更新 Policy 和 Value Model │ └─────────────────────────────────────────────────┘ 这就是 PPO 的代价 — 需要 4 个模型同时在 GPU 上！

2.3.1 PPO 的核心机制

PPO（Proximal Policy Optimization，近端策略优化）是 OpenAI 在 2017 年提出的策略梯度算法。它的核心创新是通过裁剪（clipping）机制限制策略更新幅度，从而在保证训练稳定性的同时实现有效的策略改进。

🔬 深入原理：PPO 裁剪机制的数学直觉
PPO 的裁剪机制可以这样理解：当优势 A_t > 0（这个动作好）时，我们想增加这个动作的概率，但不能增加太多；当 A_t < 0（这个动作差）时，我们想减少这个动作的概率，但也不能减少太多。 具体来说，裁剪比率 r_t(θ) 到 [1-ε, 1+ε] 范围（通常 ε=0.2）意味着：新策略相对于旧策略的变化不超过 20%。这就像给优化过程装了一个"限速器"——防止模型因为某一步的好运气而大幅改变策略。 在 RLHF 中，PPO 的特殊之处在于奖励信号来自 RM 而非环境。这带来了一个独特的问题：RM 是一个不完美的代理（proxy），它的评分与真实的人类偏好之间存在 gap。KL 惩罚就是用来缩小这个 gap 的——它强制模型保持在 RM 的"可信区域"内。

🔍 KL 散度惩罚的作用
KL(π||π_ref) 衡量当前策略与参考策略的偏离程度。没有 KL 惩罚时，模型会"过度优化"RM — 生成 RM 给高分但人类觉得奇怪的回答（奖励黑客）。KL 惩罚像一根"弹簧"，把模型拉回 SFT 模型附近。β 越大，弹簧越紧，模型越保守。实践中，β 的选择是一个微妙的平衡：太大会导致模型改进缓慢，太小会导致奖励黑客。一种自适应策略是设置目标 KL 值（如 6），当实际 KL 超过目标时增大 β，反之减小 β。

2.4 RLHF 的问题与挑战

⚠️ RLHF 的痛点 (2025-2026 已验证)
  • 成本高: 需要 4 个模型 (Policy + Reference + RM + Value Model)，显存需求巨大
  • 训练不稳定: PPO 超参敏感，需要仔细调参，经常出现 reward hacking
  • 人类偏好数据贵: 高质量标注数据成本高，标注者之间的一致性也有限
  • 奖励黑客: 模型可能学会"欺骗" RM 而不是真正变好
  • RM 过时: 随着策略改进，RM 可能不再准确反映人类偏好
  • → 正在被 DPO / GRPO 取代

2.4.1 奖励黑客（Reward Hacking）详解

奖励黑客是 RLHF 中最棘手的问题之一。它指的是模型找到了"欺骗" RM 的捷径——生成 RM 给高分但人类实际不喜欢的回答。

奖励黑客的典型模式: 模式1: 长度偏好 RM 倾向: 回答越长分数越高 模型学到: 生成冗长但无实质内容的回答 模式2: 格式偏好 RM 倾向: 使用列表/编号的回答分数高 模型学到: 所有回答都变成列表格式 模式3: 奉承模式 RM 倾向: 礼貌、赞美的回答分数高 模型学到: "这是一个非常好的问题！" 开头 模式4: 关键词堆砌 RM 倾向: 包含特定关键词的回答分数高 模型学到: 在回答中堆砌关键词，内容空洞 这些都是 RM 作为"不完美代理"导致的问题
💡 通俗类比：RLHF 的"四马分尸"
RLHF 需要 4 个模型协同训练，就像四匹马拉一辆车 — 任何一匹出问题（RM 偏差、Value Model 不准、KL 惩罚过大/过小）都会影响整体效果。DPO 的核心创新就是"砍掉两匹马"（RM 和 Value Model），让训练更简单稳定。而 GRPO 则通过"组内相对排名"进一步简化，连 Value Model 都不需要了。
🔬 深入原理：RLHF 的理论基础 — 带 KL 约束的 RL 目标
RLHF 的数学目标可以写成一个带 KL 约束的优化问题： max_π E_{x~D, y~π(y|x)} [r(x,y)] - β · KL(π(y|x) || π_ref(y|x)) 这个目标的直觉是：最大化奖励的同时，不要偏离参考模型太远。β 控制两者的权衡。 这个优化问题有闭式解（这是 DPO 的理论基础）： π*(y|x) = π_ref(y|x) · exp(r(x,y)/β) / Z(x) 其中 Z(x) 是配分函数。这个公式告诉我们：最优策略是在参考模型的基础上，按照奖励值进行指数加权。奖励越高的回答，最优策略赋予的概率越大。 这个闭式解的存在意味着：理论上，我们不需要 PPO 这样的迭代算法来求解。DPO 正是利用了这一点——它直接从偏好数据中恢复出最优策略，而无需显式训练 RM。

2.6.3 RLHF 的显存优化策略

RLHF 需要同时加载 4 个模型，显存需求极高。以下是经过实践验证的显存优化策略：

  • 模型共享：Policy 和 Reference 模型可以共享底层参数，只在需要计算 KL 时临时切换。这可以减少约 40% 的显存占用。
  • 梯度检查点：在 PPO 训练中使用梯度检查点（Gradient Checkpointing），用计算时间换取显存空间。对 Transformer 的每一层进行检查点，可以将显存需求降低约 60%。
  • 混合精度训练：使用 bf16 或 fp16 混合精度训练，将模型参数的显存需求减半。注意：RM 和 Value Model 的精度对训练稳定性有影响，建议使用 bf16。
  • ZeRO 优化：使用 DeepSpeed ZeRO Stage 2 或 3 进行分布式训练，将优化器状态和梯度分片到多个 GPU 上。
  • 离线 RM 推理：将 RM 的推理与 Policy 训练分离，先用 RM 对大量样本打分并缓存结果，再用缓存的分数训练 Policy。这可以避免同时加载 RM。

以 7B 参数模型为例，标准 RLHF 训练需要约 4×16=64GB 显存（4 个模型各 16GB）。使用上述优化策略后，可以将需求降低到约 2×16=32GB（2 张 A100-40GB），甚至在消费级 GPU 上也能运行。

# RLHF 三步流程 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
深入探讨： 从RLHF的核心原理出发，PPO算法通过奖励模型对齐人类偏好。为什么不能直接优化奖励模型？推导：直接最大化奖励会导致奖励黑客（reward hacking），即模型找到高奖励但低质量的捷径。PPO通过KL散度约束D_KL(π_θ||π_ref)≤ε确保策略不偏离参考模型太远。数学本质上，RLHF是带约束的策略优化：max E[r(x,y)] - β·D_KL(π_θ||π_ref)，用信任域方法稳定训练过程，防止策略崩溃。
常见误区： 学习RLHF时，很多人只看流程图不实际训练奖励模型。建议从头实现一个简单的奖励模型训练流程，用人类偏好数据训练后观察奖励分数与实际质量的相关性。
常见误区： 不要以为RLHF就是简单地用人类反馈训练一个打分模型。偏好数据质量、奖励模型过拟合、PPO训练的KL约束等都是关键挑战。
