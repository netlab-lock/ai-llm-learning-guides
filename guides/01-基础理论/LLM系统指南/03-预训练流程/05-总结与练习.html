💡 通俗类比
预训练就像让一个学生从零开始自学整个图书馆。不是让学生死记硬背每一本书（过拟合），而是让学生广泛阅读、理解规律（泛化能力）。数据清洗相当于图书馆管理员筛选高质量书籍，去重相当于移除重复的副本，数据配比相当于合理安排文学、科学、历史等不同科目的阅读比例。学习率就像学生每天学习的"强度"——一开始需要循序渐进（Warmup），后期逐渐减少（Cosine Decay），就像复习阶段不需要高强度学习一样。
⚠️ 常见误区
① "数据越多越好"——错误！Phi 系列用精心筛选的少量高质量数据，效果超过了用万亿低质量 token 训练的模型。数据质量永远优先于数量。② "学习率越高训练越快"——过高的学习率会导致梯度爆炸、训练不稳定甚至发散。Warmup 阶段的存在正是为了渐进式地让优化器找到合适的更新步长。③ "FP16 和 BF16 一样"——FP16 的指数位只有 5 位，范围窄容易溢出；BF16 有 8 位指数，范围和 FP32 一样大，是训练的首选。④ "预训练数据可以不关注版权"——实际上主流公司（OpenAI、Google、Meta）都在努力确保数据合规，这也是为什么高质量开源数据集越来越重要。
🔬 深入探讨
为什么 FLOPs 的估算公式是 6ND 而不是 2ND？前向传播中每个参数在每个 token 上约需 2 次浮点运算（乘加），反向传播约需前向的 2 倍（因为需要计算参数梯度和输入梯度），所以总 FLOPs ≈ 2ND(前向) + 4ND(反向) = 6ND。但这是一个近似值，实际上还需要考虑注意力层的额外开销（约 2Nd² 的序列长度相关项）、嵌入层、LayerNorm 等。对于万亿 token 级别的训练，这些"额外开销"加起来也不可忽视。DeepSeek-V3 在训练时采用了 FP8 混合精度，通过细粒度的 per-tile 量化和高精度累加器，将 FLOPs 的实际硬件效率提升了约 1.5-2 倍，而精度损失可以忽略不计。这种工程创新使得 DeepSeek-V3 仅用 2.788M H800 GPU 小时就完成了 14.8T token 的训练。

数据流水线全景

深入探讨：现代 LLM 的数据流水线远比"爬取→清洗→训练"复杂得多。以 FineWeb-Edu 为例，其流水线包括：① Common Crawl 的 WARC 文件解析和文本提取（trafilatura）；② 基于规则的过滤（语言检测、长度过滤、重复行过滤、特殊字符比例）；③ 基于 ML 的质量评分（用 LLaMA-3 70B 给网页教育质量打分 0-5 分，只保留 3 分以上）；④ MinHash 去重（Jaccard 相似度阈值 0.75，去掉近似重复文档）；⑤ 按主题和语言配比。整个流水线处理了 15 万亿 token 的原始数据，最终得到约 1.3T 高质量教育类 token。这个过程体现了"数据工程"在 LLM 时代的核心地位——模型架构已经高度趋同，数据质量成为竞争的决定性因素。

⚠️ 常见误区
① "去重只做精确去重就够了"——实际上近似去重（MinHash/SimHash）同样重要，大量网页存在轻微修改的转载内容，精确去重完全无法检测。建议同时使用精确和近似去重。② "多语言数据等比例混合"——英语内容通常占比最高（50-70%），因为高质量非英语数据较少。简单等比例会导致模型在低资源语言上反而表现更差（因为高噪声数据会干扰学习）。③ "训练数据越多 token 越好，不需要在意数据配方"——数据配比（mixture）对模型能力有决定性影响。增加代码数据比例可以提升推理能力，增加数学数据可以增强数学能力。这个比例需要精心调试。
💡 通俗类比
数据配比就像烹饪中的调味料比例。盐、糖、酱油、醋各有各的用途，但比例不同会做出完全不同的菜。同样，代码、数学、自然语言、多语言数据的不同配比，会训练出擅长不同能力的模型。比如 LLaMA 系列从 v1 到 v3 逐步增加了代码数据比例，使其编程能力大幅提升。DeepSeek 系列则特别强调了数学和代码数据的占比，这直接贡献了其强大的推理能力。

动手练习

🎯 练习1: 数据去重
写一个 Python 脚本，用 hash 做精确去重：读取一个文本文件，去除完全相同的行。 
import hashlib

def dedup_file(input_path, output_path):
    seen = set()
    with open(input_path) as f_in, \
         open(output_path, 'w') as f_out:
        for line in f_in:
            h = hashlib.md5(line.encode()).hexdigest()
            if h not in seen:
                seen.add(h)
                f_out.write(line)
    print(f"去重完成: {len(seen)} 行保留")
🎯 练习2: 学习率可视化
用 matplotlib 画出 Warmup=2000, Total=100000, Peak=3e-4, Min=3e-5 的 cosine decay 曲线。 
import numpy as np, matplotlib.pyplot as plt

steps = np.arange(100001)
warmup, total, peak, mn = 2000, 100000, 3e-4, 3e-5
lr = np.where(steps < warmup,
    peak * steps / warmup,
    mn + 0.5*(peak-mn)*(1+np.cos(np.pi*(steps-warmup)/(total-warmup))))
plt.plot(steps, lr); plt.xlabel("Steps")
plt.ylabel("Learning Rate"); plt.title("Cosine Decay")
plt.savefig("lr_schedule.png", dpi=150)
🎯 练习3: FLOPs 计算
计算 LLaMA-3 8B (15T tokens) 的训练 FLOPs，估算需要多少张 A100 训练多少天。 
# FLOPs ≈ 6 × N × D
N = 8e9        # 8B 参数
D = 15e12      # 15T tokens
flops = 6 * N * D  # = 7.2e23 FLOPs

# A100 BF16 算力: 312 TFLOPS, MFU ≈ 45%
gpu_flops = 312e12 * 0.45  # 实际有效算力
gpu_seconds = flops / gpu_flops
days = gpu_seconds / 86400
print(f"需要 {days:.0f} GPU·天 ≈ {days/1000:.0f} 千卡×{days/1000:.0f} 天")
# 输出: 约 60,000 GPU·天 ≈ 1000 卡 × 60 天
💡 通俗类比
FLOPs 估算公式 6ND 的来源很好理解：一个参数在前向传播时要做一次乘法和一次加法（≈2 FLOPs），反向传播需要计算参数梯度（≈2 FLOPs）和输入梯度（≈2 FLOPs），总共 6 FLOPs per token per parameter。这就像一道菜每份需要 6 个步骤，要做 D 份菜、每份用 N 种食材，总步骤数就是 6×N×D。

预训练工程实践深度解析

🔬 深入探讨：预训练的数据配方（Data Mixture）科学
数据配方是预训练中最被低估的超参数之一。以 LLaMA-3 为例，其数据配方为：网页文本 50%、代码 17%、数学/科学 15%、多语言 10%、书籍 5%、其他 3%。Meta 团队测试了超过 100 种不同的混合比例。关键发现：(1) 代码数据比例与推理能力正相关——增加代码数据可以提升数学和逻辑推理表现；(2) 多语言数据需要精心平衡——过多低质量多语言数据会拖累英语能力；(3) 数据重复超过 4 次后收益递减。DeepSeek-V3 进一步发现，在训练后期逐步增加高质量数据的比例（课程学习）可以在相同 token 数下获得更好的模型质量。

预训练流程的学习已完成！建议后续路径：① 回顾本章 5 个核心概念（CLM自回归、数据流水线、AdamW优化器、训练稳定性、FLOPs估算）→ ② 进入后训练对齐，学习如何将预训练模型转化为 AI 助手 → ③ 实践：尝试用开源数据在小模型上做预训练实验。理解预训练是理解整个 LLM 技术栈的基础——它决定了模型的"先天能力"，而后训练（SFT/RLHF）决定了模型的"后天表现"。建议在学习后训练之前，先通过小规模实验加深对预训练的理解。
⚠️ 预训练资源规划提醒
在开始预训练之前务必做好资源规划：(1) 计算 FLOPs 和 GPU 小时——使用 6ND 公式估算；(2) 评估显存需求——包括模型参数、梯度、优化器状态和 KV Cache；(3) 规划存储——checkpoint 通常需要 TB 级别存储；(4) 考虑容错——万卡级别训练平均每 1-2 小时遇到一次硬件故障。建议从小规模实验开始逐步扩大规模。具体来说，一个 7B 模型的完整训练需要约 84GB 显存（BF16 参数 14GB + 梯度 14GB + 优化器状态 56GB），需要至少 2 张 80GB A100。训练 2T tokens 约需 170K A100 小时，按云服务价格约 34 万美元。这些数字可以帮助你评估项目的可行性和预算需求。

🔬 深入探讨：预训练的计算效率优化
预训练的计算效率（MFU，Model FLOPs Utilization）是衡量训练系统优劣的核心指标。A100 的理论 BF16 算力为 312 TFLOPS，但实际训练中 MFU 通常只有 40-55%。影响 MFU 的主要因素包括：(1) 通信开销——AllReduce、AllGather 等集合通信操作占据了 10-30% 的计算时间；(2) 流水线气泡——PP 并行中的 GPU 空闲时间约占 5-20%；(3) 内存带宽瓶颈——注意力计算是 memory-bound 操作，GPU 算力利用率低；(4) 优化器更新开销——AdamW 的状态更新和权重衰减计算约占 5%。DeepSeek-V3 通过 DualPipe 调度、FP8 训练和 MoE 架构，将 MFU 提升到了约 55%，这在万卡级别的训练中是非常优秀的水平。对于中小规模的训练，MFU 可以更高（60-70%），因为通信开销相对较小。此外，梯度累积（Gradient Accumulation）策略可以有效减少通信频率——将多个 micro-batch 的梯度累积后再做一次 AllReduce，原理上是将通信开销从每步 O(N_params) 降低到 O(N_params / accum_steps)。在实践中，累积步数通常设为 4-8 步，在不损失数学等价性的前提下显著提升训练吞吐量。
💡 实践建议
预训练的核心在于数据质量和超参数调优。建议初学者从以下步骤开始：(1) 使用开源数据集（如 FineWeb-Edu 的子集）在 125M-1B 模型上做预训练实验；(2) 对比不同学习率和 batch size 对训练 loss 的影响；(3) 使用 WandB 或 TensorBoard 记录训练过程中的各项指标；(4) 在标准评测基准（如 MMLU、HellaSwag）上评估模型质量。这些实践经验将帮助你深入理解预训练的每个环节。

常见误区： 学习预训练流程时，很多人只关注单个环节（如优化器或数据处理）而不理解整个流程的系统性。建议动手完成以下综合实验：(1) 用开源数据集（如FineWeb-Edu子集）在125M模型上做完整预训练，从数据处理到训练到评估的全流程；(2) 对比不同数据配比（10%/30%/50%代码数据）对模型推理能力的影响；(3) 实现FLOPs计算脚本，估算不同模型规模的训练成本；(4) 用MMLU或HellaSwag评估训练好的模型质量。这些实验将帮助你理解预训练的每个环节如何影响最终模型能力。
常见误区： 不要以为预训练只需要"有钱有卡"就能做好。数据清洗的质量直接决定了模型是否会输出垃圾；去重策略的选择决定了模型是否会"背诵"训练数据；学习率调度的设计决定了训练是否能稳定收敛；混合精度训练的配置决定了显存是否够用。这些基础环节中的任何一个出错，都可能导致数百万美元的训练资源浪费。DeepSeek-V3仅用557万美元就训练了671B参数模型，靠的不是堆硬件，而是对每个基础环节的精确优化。
