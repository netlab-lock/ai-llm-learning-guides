💡 通俗类比：数据来源像"食材采购"
训练 LLM 就像开一家高级餐厅：Common Crawl 是"菜市场"——量大便宜但品质不一，需要仔细挑选；Wikipedia 是"有机超市"——品质有保证但种类有限；Books 是"进口食材"——品质高但成本也高；GitHub 是"调味料"——少量就能给模型加上代码能力。好的"厨师"（数据工程师）知道如何搭配这些食材，做出最美味的"菜"（训练数据）。
⚠️ 常见误区：数据越多不一定越好
盲目增加数据量可能适得其反。Common Crawl 中大量是垃圾网页（广告、模板文本、重复内容），如果不过滤直接训练，模型会学到垃圾。研究表明，用 1/10 的高质量数据训练的效果可能优于用全部低质量数据。数据配比也很关键——代码数据太少，模型不会写代码；代码太多，自然语言能力下降。

2.2 数据处理流水线

完整的数据处理流水线: 原始爬取数据 ↓ 1. 语言识别 (fastText 分类器) ↓ 过滤非目标语言 2. 文本清洗 ↓ 去除 HTML 标签、特殊字符、广告 ↓ 规范化: Unicode 标准化、全角→半角 3. 去重 ↓ 精确去重: SHA256 hash 完全相同的文档 ↓ 模糊去重: MinHash + LSH 相似度 > 0.8 4. 质量过滤 ↓ 规则: 过短(<50字)/过长、特殊字符比例过高 ↓ 模型: 用 perplexity 过滤 (困惑度太高=低质量) ↓ 分类器: 训练质量分类器 (好/坏文本) 5. PII 脱敏 ↓ 去除个人信息: 姓名、邮箱、电话、身份证号 6. 混合配比 ↓ 不同来源按比例混合 ↓ 如: 网页60% + 代码15% + 书籍10% + 学术10% + 其他5% 7. Tokenization + 打包 ↓ 把文档拼接成固定长度的训练序列 最终训练数据
💡 通俗类比：数据处理像"自来水厂"
原始数据就像河水——混着泥沙、垃圾、细菌。数据处理流水线就是自来水厂的处理流程：先过滤大颗粒（去 HTML），再沉淀（去重），再消毒（质量过滤），最后检测达标（PII 脱敏）。最终输出的"自来水"（训练数据）才是模型可以安全"饮用"的。跳过任何一步，模型都可能"生病"（输出垃圾或泄露隐私）。
去重算法对比: 精确去重 (SHA256 Hash): 文档A: "今天天气很好" → hash: a3f2b1... 文档B: "今天天气很好" → hash: a3f2b1... ← 完全相同，删除 文档C: "今天天气不错" → hash: c7d9e4... ← 不同，保留 优点: 快速、无误判 缺点: 只能检测完全相同的文档 模糊去重 (MinHash + LSH): 文档A: "今天天气很好，适合出去玩" → MinHash签名: [3,7,2,9,1] 文档B: "今天天气很好，适合出去散步" → MinHash签名: [3,7,2,9,4] 相似度: Jaccard ≈ 0.85 > 0.8 ← 高度相似，删除 优点: 能检测近似重复 缺点: 有少量误判，计算量大

2.3 去重的重要性

⚠️ 数据重复的危害
  • 模型会"背诵"重复内容，降低泛化能力
  • 训练时 loss 异常下降（虚假的"学好了"）
  • 生成时可能出现大段重复文本
  • 研究表明：3-4 次重复就会显著影响模型质量
🔬 深入探讨：去重的"度"在哪里？
去重不是越狠越好。过度去重会：1）损失长尾知识（罕见但有价值的信息）；2）降低某些领域的训练数据量；3）引入数据选择偏差。实践中，精确去重阈值通常设为 100% 相同，模糊去重阈值设为 80% Jaccard 相似度。但这个阈值需要根据具体场景调整——代码去重可以更严格（因为代码模板很多），而多语言数据可能需要更宽松（因为翻译文本天然相似）。DeepSeek-V3 使用了多阶段去重：先精确去重，再按语言分组做模糊去重，最后用质量过滤保留高质量数据。

数据配比没有"标准答案"。LLaMA-2 比 LLaMA-1 大幅增加了代码比例（4.5%→15%），因为发现代码能力对推理很重要。DeepSeek-V3 进一步优化了配比。最佳配比取决于：1）目标能力（代码/数学/多语言）；2）模型规模（大模型更需要多样数据）；3）训练 token 数（重复训练需要更高质量数据）。通常需要多轮实验才能找到最优配比。在实践中，可以先训练小规模模型（如 1B 参数）来快速验证不同配比的效果，然后将最优配比应用到大规模训练中。此外，配比的调整应该是动态的——随着训练的进行，逐步增加高质量数据和合成数据的比例，这种"课程学习"式的配比策略已被证明可以提升最终模型的质量。

2.5 数据质量 > 数据数量

💡 Phi 系列的启示
微软的 Phi-1 (1.3B) 用精心筛选的 "教科书级" 数据 (~7B tokens) 训练，在代码任务上媲美 10 倍大的模型。这证明：数据质量比数据数量更重要。
🔬 深入探讨：什么是"高质量数据"？
"高质量"没有统一定义，但通常包含以下维度：1）正确性：事实准确、逻辑自洽；2）信息密度：每句话携带的信息量大（教科书 > 聊天记录）；3）多样性：覆盖不同领域、风格、难度；4）结构化：有清晰的组织结构（如问答对、代码注释）；5）时效性：知识是否过时。Phi 系列的成功表明，精心筛选的高质量数据可以"四两拨千斤"——用 1/100 的数据量达到同等效果。这也是为什么 synthetic data（合成数据）越来越重要。

2.6 工具

  • NVIDIA NeMo Curator: GPU 加速的大规模数据处理框架，支持分布式数据清洗、去重和质量过滤。它可以利用多 GPU 并行处理 PB 级数据，是目前工业界最常用的数据处理工具之一。NeMo Curator 还集成了基于 Nemo Retrieval 的语义去重功能。
  • DataComp-LM: 数据策展技术的评测基准，提供了标准化的数据集和评估流程，帮助研究者比较不同的数据筛选和配比策略。它定义了从 Common Crawl 中筛选高质量训练数据的标准流程。
  • dolma: AI2 的开源数据工具，提供了从数据收集、清洗到打包的完整工具链。dolma 的设计哲学是"可复现性"——所有数据处理步骤都有详细的日志和配置，确保训练数据可以被精确复现。
💡 通俗类比：工具选择像"厨房设备"
NeMo Curator 像"工业级厨房设备"——处理 PB 级数据，需要 GPU 加速，适合大公司。dolma 像"家用厨房"——开源免费，适合学术研究和小团队。DataComp-LM 像"美食评分标准"——帮你评估不同数据处理方法的效果。选择工具取决于你的"餐厅规模"（数据量）和"预算"（GPU 资源）。
⚠️ 常见误区：工具不能替代数据直觉
再好的工具也只是自动化了数据处理流程。真正决定模型质量的是：选择什么数据源、如何定义"质量"、配比怎么调。这些需要对数据的深入理解和大量实验。不要以为"用了 NeMo Curator 就万事大吉"——工具是手段，数据直觉才是核心竞争力。在实践中，数据工程师需要持续关注以下几个方面：数据分布的变化（新数据源的加入可能导致分布偏移）、数据质量的退化（爬虫质量随时间下降）、以及数据合规性的更新（隐私法规的变化可能要求重新处理历史数据）。建议建立数据质量监控仪表盘，实时追踪各数据源的去重率、质量过滤通过率和 token 分布变化，以便及时发现和处理数据异常。

2.7 Tokenization 深度解析

Tokenization 是将原始文本转换为 token ID 序列的过程，是数据处理流水线的最后一步，也是影响模型性能的关键环节。不同的 tokenizer 设计直接影响模型的词表大小、多语言能力、以及训练效率。

主流 Tokenization 算法

算法原理代表模型优点缺点
BPE (Byte Pair Encoding)反复合并最高频字节对GPT-2/3/4, LLaMA简单高效，无 OOV依赖训练语料
WordPiece类似 BPE，用似然而非频率BERT语言学更合理计算稍慢
Unigram (SentencePiece)从大词表逐步剪枝T5, mBERT支持语言无关需要预分词
Byte-level BPE直接在字节级做 BPEGPT-2, LLaMA天然支持多语言中文效率较低
BPE Tokenization 训练过程: 初始词表: 所有单个字节/字符 {'a', 'b', 'c', 'd', 'e', ...} Step 1: 统计所有相邻字节对频率 "ab" → 5000次, "bc" → 3000次, "cd" → 2000次, ... 合并最高频对 "ab" → 新token "ab" Step 2: 重新统计，再次合并 "abc" → 2800次, "de" → 2500次, ... 合并 "abc" → 新token "abc" Step 3-N: 重复，直到词表达到目标大小 (如 32000 或 128000) 编码过程 (贪心匹配): 输入: "预训练数据处理" 分词: ["预", "训", "练", "数", "据", "处", "理"] (每个字) 或者: ["预训练", "数据", "处理"] (子词级) ↑ 词表越大，每个 token 承载信息越多，序列越短 ↑ 但 Embedding 层参数 = vocab_size × hidden_dim

现代 LLM 的词表大小选择是一个权衡：词表越大，每个 token 平均承载的信息越多，序列越短，推理更快；但 Embedding 层的参数量也越大，且稀有 token 的训练不充分。GPT-2 使用约 50K 词表，LLaMA 使用 32K，LLaMA-3 和 DeepSeek-V3 使用 128K——词表大小的趋势是越来越大。

💡 中文 Tokenization 的特殊挑战
中文没有天然的空格分隔符，且字和词的关系复杂。英文 "understanding" 可以拆成 "under" + "stand" + "ing"，但中文"理解"通常不会被拆分。这意味着：1）中文在 BPE tokenizer 中的效率通常低于英文（同样的文本，中文需要更多 token）；2）以英文为主训练的 tokenizer 对中文的编码效率很差（GPT-2 的 tokenizer 将一个中文字符拆成 2-3 个字节 token）；3）LLaMA-3 的 128K 词表大幅改善了中文效率，但仍然不如专门针对中文优化的 tokenizer。在实际应用中，如果目标场景以中文为主，建议在预训练阶段扩展 tokenizer 的中文词表——具体做法是收集大规模中文语料，运行 BPE 训练生成额外的中文子词 token，然后将这些 token 合并到基座 tokenizer 中，重新训练 Embedding 层。这种方法可以在不改变模型架构的情况下显著提升中文编码效率。
不同 Tokenizer 对中文的编码效率对比: 文本: "大语言模型正在改变世界" GPT-2 (50K词表, 英文为主): → 约 25-30 tokens (每个中文字符拆成2-3个字节token) 效率: ~0.4 字符/token LLaMA-1 (32K词表, 多语言): → 约 10-12 tokens 效率: ~1.0 字符/token LLaMA-3 (128K词表, 多语言增强): → 约 5-7 tokens 效率: ~1.8 字符/token 中文专用 (如 ChatGLM, 65K词表): → 约 4-5 tokens 效率: ~2.5 字符/token 启示: 词表设计直接影响训练成本和推理速度！
⚠️ Tokenizer 不匹配的严重后果
使用不匹配的 tokenizer 会导致严重问题：1）用英文 tokenizer 训练的模型在中文任务上表现极差——因为每个中文字符被拆成多个字节 token，模型需要更多的预测步骤才能生成同样的内容；2）不同 tokenizer 的 token ID 不可互换——即使两个 tokenizer 的词表大小相同，相同文本的 token ID 序列也完全不同；3）混用 tokenizer 会导致微调时出现大量 UNK token。因此，从预训练到微调到推理，必须全程使用同一个 tokenizer。在模型合并（model merging）场景中，tokenizer 不匹配是最常见的失败原因之一。如果你需要将两个使用不同 tokenizer 的模型合并，通常需要先对齐它们的 embedding 空间——这是一个非常复杂的过程，效果往往不理想。最好的做法是从一开始就确保所有模型使用同一个 tokenizer。在选择 tokenizer 时，还需要考虑特殊 token 的设计——系统提示词模板、工具调用格式、对话轮次分隔符等都需要在 tokenizer 中预定义对应的特殊 token。

🔬 深入探讨：Synthetic Data（合成数据）的崛起
当高质量自然数据日趋稀缺时，合成数据成为重要补充。主要方法包括：1）LLM 生成：用 GPT-4 或 Claude 生成高质量问答对、教程、代码等。Phi 系列模型大量使用 GPT-4 生成的"教科书级"数据。2）数据增强：对已有数据做改写、翻译、摘要等变换。3）Self-Instruct：模型自己生成指令和回答，用于训练。4）蒸馏：用大模型的输出训练小模型。合成数据的风险在于：可能放大模型偏见、导致"模型坍塌"（所有模型趋向相似）、以及质量难以保证。但实践表明，合理使用合成数据可以显著提升模型能力——DeepSeek-V3 的训练中就大量使用了合成数据。

2.9 数据配比的动态调整策略

🔬 动态数据配比的前沿实践
传统的固定数据配比在大规模预训练中逐渐显露出局限性——不同阶段的训练需要不同的数据分布。训练初期，模型更需要"简单"的数据来快速学习基础语言能力；训练后期，则需要"复杂"的数据来提升推理和知识能力。动态数据配比策略应运而生：(1) 基于 Loss 的自适应采样——监控模型在不同数据源上的 loss 变化，loss 下降快的数据源减少采样比例，loss 仍然高的数据源增加采样比例。这种方法可以让模型将更多计算资源投入到"还没学好"的领域。(2) DoReMi 算法——先训练一个小型参考模型，用参考模型的 loss 差异来指导数据采样权重。研究表明 DoReMi 可以在不增加计算量的情况下显著提升模型质量。(3) 课程学习（Curriculum Learning）——按照从易到难的顺序组织训练数据。例如先训练教科书级别的高质量数据，再训练网页级别的多样化数据。DeepSeek-V3 和 Qwen-2.5 系列都采用了某种形式的动态数据配比策略。
动态数据配比示意图: 训练阶段 网页数据 代码数据 学术数据 高质量合成数据 ─────────────────────────────────────────────────────────── 初期 (0-30%) 70% 10% 15% 5% ↑ 基础语言学习为主 中期 (30-70%) 55% 20% 15% 10% ↑ 增加代码和推理数据 后期 (70-100%) 40% 20% 15% 25% ↑ 增加高质量合成数据提升精细能力 关键: 根据各数据源的 loss 变化动态调整

🎯 练习1: 实现 MinHash 去重
写一个 Python 脚本，使用 MinHash + LSH 对文本集合做近似去重。 
import hashlib
from collections import defaultdict

def minhash(text, num_hashes=128):
    """计算文本的 MinHash 签名"""
    tokens = set(text.split())
    signature = []
    for i in range(num_hashes):
        min_val = float('inf')
        for token in tokens:
            h = int(hashlib.md5(f"{i}_{token}".encode()).hexdigest(), 16)
            min_val = min(min_val, h)
        signature.append(min_val)
    return signature

def jaccard_est(sig1, sig2):
    """估计 Jaccard 相似度"""
    matches = sum(1 for a, b in zip(sig1, sig2) if a == b)
    return matches / len(sig1)

docs = ["今天天气很好", "今天天气不错", "明天会下雨"]
sigs = [minhash(d) for d in docs]
print(f"Doc0 vs Doc1 相似度: {jaccard_est(sigs[0], sigs[1]):.2f}")
🎯 练习2: BPE 分词实现
手动实现一个简化版的 BPE 分词算法，理解其核心原理。 
def simple_bpe(text, num_merges=10):
    """简化的 BPE 训练"""
    tokens = list(text.encode('utf-8'))
    vocab = {bytes([i]): i for i in range(256)}

    for i in range(num_merges):
        # 统计相邻对频率
        pairs = {}
        for j in range(len(tokens) - 1):
            pair = (tokens[j], tokens[j+1])
            pairs[pair] = pairs.get(pair, 0) + 1

        if not pairs: break
        best = max(pairs, key=pairs.get)
        print(f"Merge {i+1}: {best} (freq={pairs[best]})")

        # 合并
        new_tokens = []
        j = 0
        while j < len(tokens):
            if j < len(tokens)-1 and (tokens[j], tokens[j+1]) == best:
                new_tokens.append(best)
                j += 2
            else:
                new_tokens.append(tokens[j])
                j += 1
        tokens = new_tokens
    return tokens
🎯 练习3: 数据质量评分
设计一个简单的数据质量评分函数，综合考虑文本长度、重复率和信息密度。 
import re

def quality_score(text):
    """简单数据质量评分 (0-1)"""
    score = 1.0

    # 长度过滤 (太短或太长扣分)
    if len(text) < 50: score *= 0.3
    elif len(text) > 10000: score *= 0.8

    # 重复率 (重复行占比)
    lines = text.strip().split('\n')
    unique = len(set(lines))
    dup_rate = 1 - unique / max(len(lines), 1)
    score *= (1 - dup_rate)

    # 特殊字符比例
    special = len(re.findall(r'[^a-zA-Z0-9\u4e00-\u9fff\s]', text))
    ratio = special / max(len(text), 1)
    if ratio > 0.3: score *= 0.5

    return round(score, 2)

LLM 系统学习 · 02-训练数据处理

返回总目录

相关章节： 数据来源与以下内容密切相关：01-CLM训练目标 | 03-优化器与学习率 | 04-训练稳定性与精度。建议结合学习。
实用技巧： 训练数据处理的工程实践建议：在构建数据流水线时，建议按照以下步骤进行。首先用小规模数据（100GB级别）测试每个处理步骤的正确性——验证语言识别准确率、去重率、质量过滤通过率。然后用中等规模（1TB级别）测试流水线的吞吐量和稳定性。关键监控指标包括：各阶段的数据保留率（去重后剩余比例、质量过滤通过比例）、token分布变化（是否符合预期的配比）、以及处理速度（是否能在合理时间内完成TB级数据处理）。建议使用NVIDIA NeMo Curator或dolma等成熟工具，避免重复造轮子。最后务必在1B模型上用处理后的数据做小规模预训练实验，验证数据质量。
深入探讨： 训练数据处理的核心挑战在于如何从海量原始数据中提取高质量训练信号。推导：设原始数据为D_raw，经过质量过滤后保留率为r_q，去重后保留率为r_d，则最终数据量D_final = D_raw × r_q × r_d。以Common Crawl为例，原始数据约250B页，语言过滤保留约30%（r_l=0.3），质量过滤保留约20%（r_q=0.2），去重保留约70%（r_d=0.7），最终得到约10B页高质量数据——仅占原始数据的4%。数学本质上，这4%的数据决定了模型的全部能力上限。因此数据处理流水线的每一步都需要精心设计——过度过滤会损失有价值的长尾知识，过滤不足会引入噪声降低模型质量。数据配比的调整更是一门"艺术"，需要通过大量小规模实验来找到最优方案。

向前串联： 数据来源的知识将在后续的推理框架选型、性能基准测试、生产部署中反复应用。 建议结合实际框架动手验证本章所学内容。
深入探讨： 从数据工程的角度看，训练数据处理的核心挑战在于数据质量与覆盖范围的平衡。过度追求质量（只保留"教科书级"数据）会导致模型缺乏多样性和鲁棒性；过度追求覆盖范围（不过滤直接使用所有数据）会让模型学到大量噪声。推导：设数据质量评分函数为Q(d)，信息密度为I(d)，则最优数据集应最大化 ΣQ(d)·I(d) 同时保证领域覆盖的多样性。实践中，Phi系列证明了"少而精"的路线可行（7B tokens高质量数据媲美万亿级低质量数据），但LLaMA-3和DeepSeek-V3证明了"多而精"的路线更优（15T tokens高质量数据）。关键在于建立系统化的质量评估和动态配比机制。
常见误区： 学习训练数据处理时，很多人只了解流程概念而不实际操作数据处理工具。建议动手实现以下实验：(1) 用Python实现一个简单的MinHash去重算法，对比精确去重和模糊去重的检测能力差异；(2) 用fastText语言识别模型对混合语言文本做分类，观察识别准确率；(3) 用一个预训练小模型计算不同文本的perplexity，理解"高质量"文本与"低质量"文本的困惑度差异；(4) 手动实现BPE分词算法，理解tokenizer的训练过程。这些实验能帮助你建立对数据处理各环节的直觉判断能力。
常见误区： 不要以为训练数据处理只是"清洗一下就能用"。数据去重的阈值选择（Jaccard 0.75 vs 0.85）直接影响模型是否会"背诵"训练数据；质量过滤的标准定义决定了模型学到的是"教科书知识"还是"网络噪音"；数据配比中代码数据的比例直接影响模型的推理能力（代码越多推理越强，但自然语言能力会下降）。这些看似简单的数据处理决策，在万亿token规模下会被放大到决定模型命运的程度。理解每一步处理的原理和权衡，是构建高质量训练数据集的基础。
