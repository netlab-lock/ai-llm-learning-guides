🔄 2. 连续批处理 (Continuous Batching)

2.1 背景：LLM 推理的"一问一答"特性

💡 通俗类比
连续批处理就像医院急诊分诊台——静态批处理是挂号窗口，必须凑够一批人才开始看病；连续批处理是急诊室，来一个看一个，看完立刻接下一个，床位永远不空。
⚠️ 常见误区
很多人以为连续批处理只是"多线程"。实际上它解决的是GPU显存槽位的动态复用问题——即使你有100个线程，如果batch里有个超长请求，其他请求仍然要等它完成才能被替换。
🔬 深入探讨
连续批处理的调度粒度精确到每个iteration（每生成一个token），这比传统batch调度精细几个数量级。vLLM的iteration-level scheduling实现中，调度器在每一步都检查running队列和waiting队列，确保GPU槽位的利用率最大化。

LLM 推理有一个独特的特点：每生成一个 token，都要完整跑一遍模型前向传播。一个 200 字的回答，需要 200+ 次前向传播。这意味着：

  • 单次请求的 GPU 利用率极低 — 一个请求只用到 GPU 算力的一小部分
  • 必须把多个请求"批"在一起 — 才能喂饱 GPU 的计算能力
💡 为什么要批处理？
GPU 的设计哲学是"大量简单核心并行计算"。一个请求只用到一小部分核心，大部分空闲。批处理让多个请求共享 GPU 的并行计算能力，就像餐厅一张大桌同时坐多桌客人。

2.2 静态批处理（Static Batching）：朴素方案

💡 通俗类比
静态批处理像电影院一场一场放映——不管观众多少，每场必须坐满才开演，散场后才能进下一批。连续批处理像自助餐——随时来随时吃，走一个补一个。
⚠️ 常见误区
静态批处理的GPU利用率不是恒定的，而是随时间递减。一个batch中最早完成的短请求会留下空槽位，但这些槽位无法被新请求填充，导致GPU利用率从100%逐步下降到30%甚至更低。
🔬 深入探讨
静态批处理的"木桶效应"在实际生产中非常严重。假设batch中有100个请求，其中1个请求需要生成5000个token，其余99个只需100个token，那么这99个请求完成后，GPU的99%槽位都会空闲，等待那1个长请求完成。

最直觉的做法：攒一批请求，凑够 batch_size 个，一起送进 GPU。

请求A: 生成 10 个token (短回答) 请求B: 生成 100 个token (长回答) 请求C: 生成 5 个token (超短回答) 静态批处理: ┌─────────────────────────────────────────────┐ │ Step 1: A₁ B₁ C₁ (三请求同时生成第1个token) │ │ Step 2: A₂ B₂ C₂ │ │ ... │ │ Step 5: A₅ B₅ C₅ ← C 在这里就完成了！ │ │ Step 6: A₆ B₆ [空] ← C 的槽位闲置 │ │ ... │ │ Step 10: A₁₀ B₁₀ [空] ← A 也完成了 │ │ Step 11: [空] B₁₁ [空] ← 只剩B，大量空闲 │ │ ... │ │ Step 100: [空] B₁₀₀ [空] ← 独自跑完 │ └─────────────────────────────────────────────┘

2.3 静态批处理的三个致命问题

💡 通俗类比
三个问题可以类比为：木桶效应像"一个慢食客拖累整桌人"；GPU利用率暴跌像"电影院只剩一个观众但全场灯还开着"；排队延迟爆炸像"高速收费站只开一条车道"。
⚠️ 常见误区
很多人认为排队延迟只影响用户体验。实际上在高并发场景下，排队延迟会导致请求超时、连接断开，甚至引发级联故障。一个被长请求阻塞的batch可能导致整个推理服务的P99延迟飙升到数十秒。
🔬 深入探讨
GPU利用率暴跌的根本原因是静态批处理将"调度"和"计算"耦合在一起——调度必须等计算完成。连续批处理的创新在于将两者解耦：调度器独立运行，每步都可以重新分配资源。

问题一：木桶效应

整个 batch 必须等最慢的请求完成，才能处理下一批。C 完成后它的"槽位"空了，但新请求不能插进来。

问题二：GPU 利用率暴跌

时间轴 → 请求A: ████████░░░░░░░░░░░░ (10步完成) 请求B: ████████████████████████████████... (100步) 请求C: █████░░░░░░░░░░░░░░░ (5步完成) GPU利用率: ████████████░░░░░░░░░░░░░░░░░░░░ ↑ C完成后开始下降 ↑ A完成后进一步下降

问题三：排队延迟爆炸

新来的请求必须等当前整批完成才能进入。如果一个 batch 里有个超长回答，后面所有请求都被堵住。

2.4 连续批处理的解决方案

核心思想极其简单：不要等整批结束，每一步（iteration）都可以动态替换已完成的请求。

💡 生活类比
静态批处理像餐厅大桌 — 必须等所有人吃完才能上下一轮，吃得快的人干等。
连续批处理像回转寿司 — 吃完一个走一个，空位立刻坐新人，座位永远不空。

2.5 关键区别：调度粒度

💡 通俗类比
调度粒度的区别就像"按天发工资"和"按小时发工资"——静态批处理按天发，不管你今天工作几小时都一样；连续批处理按小时发，干一小时结一小时，效率更高。
⚠️ 常见误区
很多人误以为连续批处理的调度开销很高。实际上，现代推理引擎（如vLLM）的调度逻辑非常轻量——只是一次链表操作和状态检查，耗时在微秒级别，远小于GPU计算的毫秒级延迟。
🔬 深入探讨
调度粒度的选择本质上是在"调度开销"和"资源利用率"之间做权衡。粒度越细，利用率越高，但调度开销也越大。对于LLM推理，由于每次前向传播的计算量很大（毫秒级），每步调度的开销可以忽略不计，所以极细粒度是合理的。
对比静态批处理连续批处理
调度粒度整个 batch 结束后每个 iteration（每生成一个 token）
GPU 利用率随时间递减 (30-60%)始终高位 (80-95%)
吞吐量低高 2-5x
排队延迟高 (等整批完成)低 (每步可插入)
短请求体验差 (被长请求拖累)好 (完成后立刻让位)
公平性差好

2.6 实现原理

💡 通俗类比
调度器的工作原理像机场登机口调度员——每架飞机起飞后（请求完成），立刻检查下一架飞机是否就绪（waiting queue），然后分配登机口（GPU槽位）。
⚠️ 常见误区
调度策略的选择对实际性能影响巨大。FCFS（先来先服务）看似公平，但在混合长短请求的场景下效率最低。SJF（短作业优先）能最大化吞吐量，但可能导致长请求永远得不到服务（饥饿问题）。
🔬 深入探讨
现代推理引擎的调度器通常采用多级策略：先按优先级分组，再在每组内用SJF调度，同时设置最大等待时间保证公平性。这种设计借鉴了操作系统进程调度的经典思想。
连续批处理的调度器 (Scheduler): 每个 iteration: 1. 检查 batch 中哪些请求已完成 (生成了 EOS token) 2. 从队列中取出新请求，填满空出的槽位 3. 把当前 batch 的所有 token 送入 GPU 执行一步 4. 返回步骤 1 关键数据结构: running_requests: 正在生成的请求列表 (固定大小) waiting_queue: 等待进入的请求队列 (FIFO 或优先级) 调度策略: FCFS (先来先服务): 最简单，但可能被长请求占满 SJF (短作业优先): 预估输出长度，优先调度短请求 Fairness: 保证每个请求的最大等待时间

💡 通俗类比
是否需要连续批处理，就像问"餐厅是否需要翻台管理"——客流量大时必须有，一个人吃饭时无所谓。
⚠️ 常见误区

2.9 连续批处理的局限

连续批处理解决的是调度问题（谁进 batch），但不能解决所有问题：

⚠️ 连续批处理没解决的问题
长 prompt 阻塞：一个请求的 prefill 阶段（处理长输入）会"霸占"GPU，其他 decode 请求被拖慢。
→ 这需要 Chunked Prefill 来解决（下一讲）。

KV Cache 内存碎片：多个请求的 KV Cache 大小不一，频繁分配释放导致内存碎片。
→ 这需要 PagedAttention 来解决（第 08 讲）。

2.10 连续批处理的高级话题：前缀缓存

前缀缓存（Prefix Caching）是连续批处理的重要扩展。当多个请求共享相同的 system prompt 时，前缀缓存允许第一个请求计算并缓存 system prompt 的 KV Cache，后续请求直接复用，避免重复计算。SGLang 的 RadixAttention 将这个思想推广到了任意前缀——用 Radix Tree 组织所有请求的 KV Cache，自动发现和复用共享前缀。

前缀缓存的实现原理

前缀缓存的核心挑战是「如何高效地发现共享前缀」。朴素方法是逐 token 比较，但时间复杂度是 O(n²)。SGLang 的 RadixAttention 使用基数树（Radix Tree）数据结构，将 token 序列作为 key，KV Cache 的物理地址作为 value。当新请求到达时，沿着 Radix Tree 匹配最长公共前缀，只需计算未匹配部分的 KV。这个方法的时间复杂度是 O(n)，且支持前缀的自动淘汰（LRU 策略）。在多轮对话场景中，system prompt 和历史对话通常是共享前缀，前缀缓存可以将 TTFT 降低 50-80%。

抢占式调度与内存管理

当 GPU 显存不足时，推理引擎需要「抢占」某些请求来释放内存。vLLM 支持两种抢占策略：(1) Swap——将被抢占请求的 KV Cache 从 GPU 内存换出到 CPU 内存，等有空间后再换入；(2) Recompute——丢弃被抢占请求的 KV Cache，等有空间后从头重新计算。Swap 的恢复速度快但需要 CPU 内存，Recompute 不需要额外内存但恢复时间长。vLLM 的调度器会根据当前的内存压力和请求的进度自动选择最优策略。

# vLLM 的连续批处理调度器核心逻辑 (简化)
class ContinuousBatchScheduler:
    def __init__(self, max_num_seqs, gpu_memory):
        self.max_num_seqs = max_num_seqs
        self.running = []      # 正在生成的请求
        self.waiting = []      # 等待进入的请求
        self.gpu_memory = gpu_memory
    
    def schedule_step(self):
        # 1. 检查已完成的请求，释放槽位
        completed = [r for r in self.running if r.is_finished()]
        for r in completed:
            self.running.remove(r)
            self.gpu_memory.free(r.kv_cache)
        
        # 2. 从等待队列取出新请求
        while self.waiting and len(self.running) < self.max_num_seqs:
            new_req = self.waiting.pop(0)
            if self.gpu_memory.can_allocate(new_req):
                self.gpu_memory.allocate(new_req)
                self.running.append(new_req)
            else:
                # 内存不足，尝试抢占
                if self.running:
                    victim = self.running.pop()  # 最后一个
                    self.gpu_memory.swap_out(victim)
                    self.waiting.insert(0, victim)
                break
        
        # 3. 返回当前 batch 的所有请求
        return self.running
💡 实践技巧
在高并发场景下，连续批处理的性能高度依赖于 max_num_seqs 参数的设置。太小（如 8）会导致 GPU 利用率不足，太大（如 256）会导致每个请求分到的计算资源不足。经验公式：max_num_seqs ≈ GPU_memory / (model_size_per_request × safety_factor)。对于 A100-80GB 运行 LLaMA-70B-GPTQ-INT4，建议 max_num_seqs=32-64。
⚠️ 常见误区
连续批处理的一个隐含问题是「尾延迟」——当 batch 中有超长请求时，虽然其他请求不被阻塞，但整体的 GPU 利用率会因为长请求的 KV Cache 占用大量内存而下降。解决方案包括：(1) 设置 max_model_len 限制单个请求的最大长度；(2) 使用 KV Cache 压缩（GQA/MLA）减少每个请求的内存占用；(3) 对超长请求使用 Prefill-Decode 分离。

2.11 连续批处理与 PagedAttention 的协同优化

连续批处理的效率高度依赖于 KV Cache 的管理策略。在 vLLM 的 PagedAttention 实现中，KV Cache 被划分为固定大小的页（Page），每个请求的 KV Cache 由多个不连续的页组成。这种分页管理消除了内存碎片，支持 Copy-on-Write（多个请求共享相同 prompt 前缀的 KV 页），并支持换入换出。vLLM 的基准测试表明，PagedAttention 相比连续内存分配，内存利用率提升了 2-4 倍。

调度器性能开销的量化分析

连续批处理的调度器在每个 iteration 都执行以下操作：(1) 检查 running 队列中是否有已完成的请求（O(n) 时间）；(2) 从 waiting 队列取出新请求填满空槽位（O(1) 到 O(k) 时间）；(3) 构造当前 batch 的输入张量（O(n) 时间）；(4) 触发 GPU 前向传播。其中前三步是 CPU 操作，第四步是 GPU 操作。实测表明，调度器的 CPU 开销在微秒级别（10-100us），远小于 GPU 计算的毫秒级延迟（5-50ms），因此调度开销可以忽略不计。但在极端高并发场景（>1000 QPS）下，调度器可能成为瓶颈，此时需要使用异步调度或多线程调度来分担 CPU 压力。

2.12 连续批处理在长上下文场景下的挑战

当处理长上下文（32K-128K tokens）请求时，连续批处理面临独特的挑战。长上下文请求的 KV Cache 可能占用大量 GPU 内存，导致同一 batch 中能容纳的请求数量大幅减少。例如，一个 128K token 的请求在 LLaMA-70B 上需要约 40GB 的 KV Cache（FP16），加上模型权重，一张 A100-80GB 只能同时处理 1-2 个这样的请求。这种情况下，连续批处理的优势被大幅削弱——batch size 太小，GPU 利用率很低。解决方案包括：(1) 使用 GQA/MLA 减少 KV Cache 大小；(2) 使用 KV Cache 量化（FP8/INT8）进一步压缩；(3) 使用 Context Parallelism 将长序列分散到多卡；(4) 使用 Chunked Prefill 保证 decode 不被阻塞。这些技术的组合使用是当前长上下文推理的最佳实践。

2.13 连续批处理的性能调优指南

连续批处理的性能调优主要涉及三个参数：max_num_seqs（最大并发请求数）、max_model_len（最大序列长度）和 scheduler_delay_factor（调度延迟因子）。max_num_seqs 设置过大可能导致每个请求分到的计算资源不足（TPOT 增加），设置过小会导致 GPU 利用率不足。建议的初始值是 GPU 内存除以单个请求的 KV Cache 大小。max_model_len 限制单个请求的最大长度，防止超长请求占用过多内存。scheduler_delay_factor 控制调度器在等待新请求和立即处理之间的平衡——较大的值会让调度器等待更多请求以组成更大的 batch，提高吞吐量但增加延迟。

连续批处理在多模态模型中的应用

连续批处理不仅适用于纯文本 LLM，也适用于多模态模型（如 LLaVA、GPT-4V）。但多模态模型的连续批处理面临额外挑战：图像 token 的数量远多于文本 token（一张图片可能对应 576-1024 个 token），导致不同请求的序列长度差异更大。这加剧了 padding 浪费和 GPU 利用率不均的问题。解决方案包括：(1) 使用动态分辨率——根据请求的图像数量动态调整图像分辨率和 token 数量；(2) 使用独立的图像编码器——将图像预处理与文本推理分离，避免图像编码阻塞文本 decode；(3) 使用 Chunked Prefill——将大量图像 token 切块处理，避免阻塞其他请求的 decode。

✅ 参考答案与详解
  1. 核心原理: LLM推理的"一问一答"特性指每生成一个token都需要完整跑一遍模型前向传播，一个200字的回答需要200+次前向传播。这导致单次请求的GPU利用率极低——GPU的设计哲学是"大量简单核心并行计算"，单请求只用到一小部分核心。连续批处理的核心原理是将"调度"和"计算"解耦：调度器独立运行在CPU上，每生成一个token（每个iteration）就重新检查running队列和waiting队列，动态替换已完成的请求，确保GPU槽位始终满载。这与静态批处理形成鲜明对比——静态批处理必须等整批完成才能处理下一批，已完成请求的槽位空闲但无法被新请求填充。连续批处理的调度粒度精确到每个iteration（微秒级调度开销 vs 毫秒级GPU计算），使得GPU利用率从静态批处理的30-60%提升到80-95%。
    关键要点: ①连续批处理解决的是"GPU显存槽位的动态复用"问题，不是简单的多线程——即使有100个线程，如果batch中有超长请求，其他请求仍要等它完成才能被替换；②调度粒度是核心区别：静态批处理按"整个batch结束"调度，连续批处理按"每个iteration"调度；③"木桶效应"是静态批处理的致命问题——batch中最慢的请求决定整批完成时间，短请求完成后槽位空闲浪费；④所有现代推理引擎（vLLM/SGLang/TensorRT-LLM/llama.cpp）都内置连续批处理，已成为基础设施；⑤连续批处理不能解决长prompt阻塞（需Chunked Prefill）和KV Cache内存碎片（需PagedAttention）。
    实际案例: 用vLLM部署LLaMA-3-8B，发送100个并发请求（长度从10到500 token不等）。设置max_num_seqs=1模拟静态批处理：GPU利用率从100%逐步下降到30%，P99延迟超过30秒（长请求阻塞短请求）。设置max_num_seqs=64启用连续批处理：GPU利用率稳定在85-95%，吞吐量提升5-8倍，P99延迟降至3秒以内。短请求（10 token）完成后立刻让位给新请求，不再被长请求拖累。
    关联知识: 连续批处理是推理优化四层架构中调度层的基础技术，与Chunked Prefill（解决长prompt阻塞）、PagedAttention（解决KV Cache碎片）、FlashAttention（加速注意力计算）共同构成推理优化的四大基础设施。同时与推理总览中的TTFT/TPOT指标直接相关——连续批处理主要改善吞吐量和P99延迟。
  2. 核心原理: 连续批处理在工程中遇到的性能瓶颈主要包括：(1)长prompt阻塞——一个请求的Prefill阶段（处理长输入）会"霸占"GPU，其他decode请求被拖慢，这是连续批处理无法解决的，需要Chunked Prefill将长prompt切成小块与decode交替执行；(2)KV Cache内存碎片——多个请求的KV Cache大小不一，频繁分配释放导致内存碎片，需要PagedAttention的分页管理解决；(3)尾延迟问题——当batch中有超长请求时，其KV Cache占用大量内存导致batch中能容纳的请求数减少，整体GPU利用率下降；(4)调度策略选择——FCFS公平但效率低，SJF吞吐量最高但可能导致长请求饥饿。诊断方法：用nvitop监控GPU利用率和显存使用，如果利用率低说明batch size太小或调度有问题，如果显存碎片化严重说明需要PagedAttention。
    关键要点: ①max_num_seqs参数设置对性能影响巨大——太小（如8）导致GPU利用率不足，太大（如256）导致每个请求分到的计算资源不足（TPOT增加），经验公式：max_num_seqs ≈ GPU_memory / (model_size_per_request × safety_factor)；②调度策略的选择对实际性能影响显著——FCFS+SJF混合策略加最大等待时间保障是大多数生产环境的选择；③前缀缓存（Prefix Caching）是连续批处理的重要扩展——多轮对话中system prompt和历史对话是共享前缀，SGLang的RadixAttention用基数树自动发现和复用，可将TTFT降低50-80%；④抢占式调度（Swap/Recompute）解决显存不足问题——Swap将KV Cache换出到CPU内存恢复快但需额外内存，Recompute丢弃后重算不需额外内存但恢复慢；⑤长上下文场景（32K-128K）下连续批处理优势被削弱，需配合GQA/MLA+KV Cache量化+Context Parallelism。
    实际案例: 某API服务部署LLaMA-70B-GPTQ-INT4在A100-80GB上，max_num_seqs=32。遇到问题：(1)偶尔出现128K长请求导致batch中只能容纳1-2个请求，GPU利用率暴跌到20%；(2)高并发时显存碎片导致OOM。解决方案：设置max_model_len=32768限制单请求最大长度，超长请求返回提示；启用PagedAttention消除显存碎片，内存利用率提升2-4倍；配置FCFS+SJF混合调度策略，短请求优先但保证长请求最大等待时间不超过30秒。最终GPU利用率稳定在80%+，P99延迟<5秒。
    关联知识: 连续批处理的瓶颈与Chunked Prefill（解决长prompt阻塞）、PagedAttention（解决KV Cache碎片）、GQA/MLA（减少KV Cache大小）直接对应。前缀缓存与SGLang的RadixAttention设计相关。同时与推理总览中的推理延迟分解（Prefill vs Decode）和SLA设计紧密关联。
  3. 核心原理: 连续批处理的替代方案主要是在"调度粒度"和"架构设计"两个维度上的不同选择。对比分析：(1)静态批处理——最简单但GPU利用率仅30-60%，适合单用户本地推理（如Ollama单请求模式）；(2)连续批处理——GPU利用率80-95%，是所有现代推理引擎的标配，适合多用户并发服务；(3)Prefill-Decode分离（DistServe）——将Prefill和Decode彻底解耦到不同GPU，消除两类计算的干扰，适合大规模部署但需要更多GPU资源；(4)Chunked Prefill——将长prompt切成固定大小的chunk与decode交替执行，是连续批处理的增强，解决长prompt阻塞问题。在调度策略维度：FCFS（先来先服务）公平但吞吐量低；SJF（短作业优先）吞吐量最高但公平性差；FCFS+SJF混合加最大等待时间保障是最佳平衡。
    关键要点: ①连续批处理 vs 静态批处理的核心区别是调度粒度（iteration-level vs batch-level）和GPU利用率（80-95% vs 30-60%），吞吐量提升2-10倍；②连续批处理的收益与请求长度方差正相关——方差越大，静态批处理的浪费越严重，连续批处理的收益越明显；③场景选择：多用户API服务必须用连续批处理，本地单用户（Ollama）不需要，批量离线推理推荐用；④各引擎实现差异：vLLM用iteration-level scheduling+PagedAttention，SGLang用RadixAttention强调前缀复用，TensorRT-LLM用In-flight Batching+NVIDIA硬件优化；⑤连续批处理的调度开销可忽略——CPU端仅10-100微秒，远小于GPU计算的5-50毫秒。
    实际案例: 某团队部署代码助手服务，对比三种方案：方案A用llama.cpp静态批处理（max_num_seqs=1），方案B用vLLM连续批处理（max_num_seqs=64），方案C用vLLM连续批处理+Chunked Prefill。场景：100个并发请求，长度从10到2000 token不等。结果：方案A吞吐量最低（50 req/s），长请求阻塞导致P99>30秒；方案B吞吐量提升6x（300 req/s），P99降至5秒；方案C在方案B基础上进一步降低P99到2秒（长prompt不再阻塞decode）。方案B的性价比最优——方案C的额外收益有限但配置更复杂。
    关联知识: 替代方案分析与推理总览中的推理引擎技术架构对比直接对应——vLLM（PagedAttention核心）、SGLang（RadixAttention核心）、TensorRT-LLM（NVIDIA硬件优化核心）的技术路线差异决定了它们在不同场景下的优劣。同时与推理总览中的适用场景速查表关联——连续批处理是调度层基础，需与计算层（FlashAttention/量化）和内存层（GQA/PagedAttention）组合使用才能达到最佳效果。

import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
