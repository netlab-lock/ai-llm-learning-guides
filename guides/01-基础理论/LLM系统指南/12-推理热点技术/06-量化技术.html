📐 6. 量化技术

6.1 为什么要量化？

💡 通俗类比
量化就像「把高清照片压缩成 JPEG」——原始照片（FP16 权重）很大很清晰，JPEG（量化后）文件小很多但肉眼几乎看不出区别。关键是找到「压缩多少还能保持质量」的平衡点。INT4 量化就像中等质量 JPEG，几乎无损；INT2 就像极低质量 JPEG，肉眼可见失真。
⚠️ 常见误区
不要以为量化只是「简单地四舍五入」。现代量化方法（如 GPTQ、AWQ）非常精密——GPTQ 用 Hessian 信息补偿量化误差，AWQ 根据激活值分布保护重要通道。直接截断（round-to-nearest）的效果远不如这些方法。选对量化方法比选对精度更重要。

大模型的参数通常是 FP16（16位浮点），一个 70B 模型需要 140GB 显存。量化把参数从高精度压缩到低精度，直接减小模型体积。量化是目前最直接、最有效的模型压缩方法，几乎所有生产部署的大模型都使用了某种形式的量化技术。

不同精度的存储开销 (70B 模型): FP32: 70B × 4 bytes = 280 GB FP16: 70B × 2 bytes = 140 GB ← 训练常用 INT8: 70B × 1 byte = 70 GB ← 量化 INT4: 70B × 0.5 byte = 35 GB ← 激进量化 INT3: 70B × 0.375 byte ≈ 26 GB ← 极限量化 量化让 70B 模型从"需要 2 张 A100"变成"1 张 A100 就能跑"

6.2 量化的基本原理

🔬 深入原理
量化的数学本质是「有损压缩」——将连续的浮点空间映射到离散的整数空间。对称量化的数学公式为：q = round(x / scale)，其中 scale = max(|x|) / (2^(b-1) - 1)，b 为目标位数。反量化过程为：x_hat = q × scale。量化误差 e = x - x_hat 的方差 Var(e) 与位数 b 呈指数关系：Var(e) ∝ 2^(-2b)。这意味着每增加1位精度，量化误差减小4倍。非对称量化引入零点偏移 z：q = round(x / scale) + z，scale = (max(x) - min(x)) / (2^b - 1)。对称量化假设数据关于零对称，非对称量化则允许零点偏移，适应性更强但计算开销略大。现代量化方法的关键创新不在映射函数本身，而在「如何选择缩放因子和零点」——好的选择让量化噪声对最终输出的影响最小化。
量化 = 把连续的浮点数映射到离散的整数 FP16 值: [0.125, -0.5, 0.875, -0.25, ...] ↓ 量化 (INT4, 16个离散值) INT4 值: [2, 6, 15, 8, ...] ↓ 反量化 (恢复近似值) 恢复值: [0.13, -0.48, 0.88, -0.24, ...] 精度损失: 有，但通常很小 收益: 存储减少 4x (INT4 vs FP16)

6.3 两种量化范式

🔬 深入原理
PTQ 和 QAT 的根本区别在于「量化误差的补偿方式」：PTQ 在量化后通过数学方法（如 Hessian 补偿）修正误差，QAT 在训练过程中让模型「适应」量化噪声。QAT 理论上更优，但成本太高（需要完整训练流程）。PTQ 的成功说明了一个重要事实：预训练好的模型对量化噪声有很强的鲁棒性，不需要显式地「训练适应」。从优化理论看，PTQ 的工作原理是在已训练好的权重空间中寻找「最近的」低精度点，而 QAT 则是让优化过程直接在低精度空间中搜索最优解。两者的数学推导都涉及二阶泰勒展开：L(W+ΔW) ≈ L(W) + g^T ΔW + 0.5 ΔW^T H ΔW，其中 H 是 Hessian 矩阵，量化误差 ΔW 的影响由 H 决定。

6.4 GPTQ：逐层量化

💡 通俗类比
GPTQ 就像「精密拆卸」——不是把整台机器拆了重装，而是逐个零件检查、测量、替换，每替换一个零件后立刻用数学方法补偿它对其他零件的影响。这样一步步下来，虽然每个零件都有微小误差，但整体误差被控制在最小范围内。

最经典的 PTQ 方法。核心思想：逐层量化，用 Hessian 信息最小化量化误差。 GPTQ 的核心创新是 OBS（Optimal Brain Surgeon）方法的逐列应用——量化一列后，用 Hessian 信息将误差「分散」到后续列，而非累积。

GPTQ 算法 (简化): 对每一层: 1. 收集校准数据 (几百个样本) 的激活值 2. 计算 Hessian 矩阵 H = 2 × X^T × X 3. 按列处理权重矩阵: - 量化当前列 - 用 Hessian 信息补偿后续列 (OBS 方法) → 量化误差被"分散"到后续列，而非累积 4. 得到量化后的权重 特点: - 需要校准数据 (几百个样本即可) - INT4 量化精度损失很小 - 量化过程较慢 (几小时) - 推理时速度提升明显

6.5 AWQ：激活感知量化

💡 通俗类比
AWQ 就像「给重要员工加薪后再全员降薪」——发现某些通道（员工）的激活值（贡献）特别大，先给它们乘一个放大因子（加薪），然后统一量化（降薪），推理时再除回来。这样重要通道的量化误差相对更小，整体质量更好。
⚠️ 工程注意事项
AWQ 的缩放因子选择至关重要。如果缩放因子太大，放大后的权重可能溢出INT4的表示范围；如果太小，保护效果不明显。AWQ 使用网格搜索在少量候选值中选择最优缩放因子，搜索空间虽然不大（通常10-20个候选值），但需要在校准数据上评估每个候选值的量化质量。这个过程通常需要几分钟到几十分钟，远快于GPTQ的逐列补偿。

核心发现：不是所有权重同等重要，对应大激活值的权重通道更敏感。 AWQ 的激活感知思想揭示了一个重要规律：权重的重要性不取决于权重本身，而取决于权重与激活值的交互。

AWQ：激活感知权重量化流程: 关键观察: 激活值分布不均匀 通道1: 激活值=0.01 → 量化误差影响小 通道2: 激活值=10.0 → 量化误差影响大！ AWQ 做法: ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐ │① 找显著通道│ → │② 乘缩放因子│ → │③ 量化权重 │ → │④ 推理时除回│ │ 激活值大的 │ │ s 放大显著 │ │ INT4 量化 │ │ 恢复原始量级│ │ 通道 │ │ 通道权重 │ │ │ │ │ └───────────┘ └───────────┘ └───────────┘ └───────────┘ vs GPTQ: AWQ 更快（不需要逐列补偿）· 精度相当或更好 vLLM/TRT-LLM 原生支持 AWQ INT4: LLaMA-2 70B PPL=3.80 (vs FP16基线3.32, 仅+0.48)

6.6 GGUF：llama.cpp 的量化格式

💡 选择建议
如果你不确定该用哪种GGUF量化类型，Q4_K_M 是最安全的默认选择——它在大多数模型上都能保持接近FP16的质量，同时将模型大小减少约75%。只有在显存极度紧张（如单张8GB显卡跑13B模型）时才需要考虑Q3_K_M或更低精度。Q2_K虽然更小，但质量退化明显，不推荐用于生产环境。

面向本地部署的量化方案，支持多种精度。GGUF 的设计哲学是「面向部署者」——不追求极致的压缩率，而是提供多种精度选择让用户根据自己的硬件做权衡。

6.7 FP8 / MXFP：新硬件原生支持

🔬 深入原理
FP8 的出现标志着一个趋势：硬件厂商开始为 AI 推理「定制」数据格式。传统 IEEE 浮点标准并非为神经网络设计——FP16 的动态范围对权重来说太大，INT8 的均匀量化对不同量级的值不够灵活。FP8 的 E4M3/E5M2 格式在「范围」和「精度」之间做了 AI 专用的权衡。从数学角度看，FP8 E4M3 的表示范围是 [-448, 448]，精度约为 0.002（在 1.0 附近），而 INT8 的范围是 [-128, 127]，精度在归一化后约为 0.004。FP8 在小值区域（神经网络权重的主要分布区域）提供更高精度，这是其优于INT8的根本原因。MXFP 更进一步，让每组 32 个值共享缩放因子——这是「分组量化」思想在硬件层面的实现，缩放因子本身使用FP8表示。
FP8 (H100 原生支持): 两种格式: E4M3: 4位指数 + 3位尾数 → 精度更高，推理常用 E5M2: 5位指数 + 2位尾数 → 范围更大，训练常用 vs INT8: INT8: 均匀量化，需要校准 FP8: 浮点格式，天然适配不同量级的值 → FP8 更简单，不需要复杂的校准过程 MXFP (Microscaling FP): DeepSeek-V4 首次大规模使用 每组 32 个值共享一个缩放因子 在 FP8 基础上进一步压缩 → 训练和推理都能用

6.9 量化与其他技术的关联

⚠️ 量化误差累积
量化是推理优化「技术栈」中最底层的优化——它直接作用于模型权重，与所有上层优化（注意力优化、批处理优化、调度优化）正交。这意味着量化收益可以与任何其他优化叠加。但要注意「量化误差的累积效应」——如果同时使用量化+稀疏注意力+KV Cache 压缩，每一层的误差可能逐层放大。在极端优化场景下，需要做端到端的质量验证，不能只看单项指标。

  • 投机解码 — 草稿模型通常用量化版本，进一步降低成本
  • PagedAttention — 量化后的 KV Cache 更小，能放更多请求
  • GQA/MLA — 和量化是正交优化，可以叠加

6.11 SmoothQuant：激活值量化

SmoothQuant 是 MIT 和 NVIDIA 联合提出的量化方法，专门解决激活值量化的难题。传统量化只量化权重（Weight），但激活值（Activation）的量化更加困难——激活值中存在异常值（outlier），某些通道的激活值可能是其他通道的 100 倍以上，导致量化精度严重下降。SmoothQuant 的核心思想是「平滑」激活值中的异常值：通过数学变换将激活值的量化难度「转移」到权重上。

SmoothQuant 的数学原理

SmoothQuant 的数学变换非常优雅。设激活值为 X，权重为 W，输出 Y = X × W。引入对角缩放矩阵 S，使得 Y = (X × S^(-1)) × (S × W) = X' × W'。关键在于选择 S 使得 X' 和 W' 都容易量化。SmoothQuant 的策略是让 S 的每个对角元素 s_j = max(|X_j|)^α / max(|W_j|)^(1-α)，其中 α 是平滑因子（通常 0.5）。这样，激活值中的异常值被 S^(-1) 缩小，权重中的对应通道被 S 放大，两者的量化难度趋于平衡。

🔬 深入探讨
SmoothQuant 的平滑因子 α 的选择原理。α 控制量化难度在激活值和权重之间的分配：α=0 表示所有难度在激活值上（退化为标准 INT8 权重量化），α=1 表示所有难度在权重上。最优的 α 使得 X' 和 W' 的量化误差之和最小。从数学上，这个问题可以表述为：min_α E[|Q(X × diag(s^(-1))) - X × diag(s^(-1))|²] + E[|Q(W × diag(s)) - W × diag(s))|²]，其中 Q 是量化函数。实验表明 α=0.5 在大多数模型上都是接近最优的选择，这与信息论中的「等量化难度分配」原理一致。
# 使用 AutoAWQ 进行 INT4 量化的完整示例
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "meta-llama/Llama-3-8B-Instruct"
quant_path = "Llama-3-8B-Instruct-AWQ-INT4"

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 配置量化参数
quant_config = {
    "zero_point": True,     # 使用零点（非对称量化）
    "q_group_size": 128,    # 每128个权重共享缩放因子
    "w_bit": 4,             # 目标位数: INT4
    "version": "GEMM",      # 使用GEMM内核优化
}

# 执行量化 (需要校准数据)
model.quantize(tokenizer, quant_config=quant_config)

# 保存量化模型
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)
print(f"量化完成! 模型保存在 {quant_path}")
💡 实践技巧
选择量化方案时的决策树：(1) 如果使用 H100/A100 → 优先考虑 FP8（原生支持，无需校准）；(2) 如果需要 INT4 压缩 → AWQ 是当前最佳选择（速度快、精度好）；(3) 如果是本地 CPU 部署 → GGUF + llama.cpp；(4) 如果需要极致压缩 → 考虑 GPTQ（逐列补偿精度更好）。不推荐 INT2 量化用于生产环境——精度损失太大。
⚠️ 常见误区
量化模型的评测不能只看 PPL（困惑度）。PPL 反映的是语言建模的整体质量，但可能掩盖特定任务上的严重退化。例如，一个 INT4 量化的模型在 PPL 上只增加 0.5，但在数学推理（GSM8K）上的准确率可能下降 10%。建议在量化后运行完整的 Benchmark 评测（MMLU、GSM8K、HumanEval），确认在你的目标任务上没有严重退化。

6.11 SmoothQuant：激活值量化的突破

SmoothQuant 是 MIT 和 NVIDIA 联合提出的量化方法，专门解决激活值量化的难题。传统量化方法只量化权重，但激活值中的异常值（outlier）使得激活值量化极其困难。SmoothQuant 的核心思想是引入数学变换，将激活值中的异常值平滑转移到权重上，使得激活值和权重都变得容易量化。

SmoothQuant 的数学原理

SmoothQuant 的变换公式为 Y = (X times S 的逆) 乘以 (S 乘以 W) = X_prime 乘以 W_prime，其中 S 是对角缩放矩阵。关键在于选择 S 使得 X_prime 和 W_prime 都容易量化。SmoothQuant 的策略是让 S 的每个对角元素等于激活值最大值的 alpha 次方除以权重最大值的 1-alpha 次方，其中 alpha 是平滑因子（通常取 0.5）。这样，激活值中的异常值被 S 的逆缩小，权重中的对应通道被 S 放大，两者的量化难度趋于平衡。实验表明，SmoothQuant INT8 在 OPT-175B 和 BLOOM-176B 上实现了几乎无损的量化效果，推理速度比 FP16 快 1.56 倍。

量化感知微调（QAF）的实践方法

量化感知微调是介于 PTQ 和 QAT 之间的方法。它先用 PTQ 量化模型，然后在量化后的模型上进行少量微调（通常 1000-5000 步），让模型适应量化噪声。QLoRA 是这种方法的代表——它在 4-bit 量化基础上叠加 LoRA 微调，在极低的计算成本下恢复量化损失的质量。QLoRA 的关键创新包括：(1) 4-bit NormalFloat（NF4）数据类型，对正态分布的权重最优；(2) 双重量化——对量化常数本身也进行量化，进一步节省内存；(3) 分页优化器——在 GPU 和 CPU 之间自动换页，避免 OOM。使用 QLoRA 微调 70B 模型只需要单张 A100-80GB 即可完成。

量化精度与推理速度的权衡分析

量化的推理速度收益取决于硬件对低精度计算的支持程度。在 NVIDIA A100 上，INT8 的 Tensor Core 运算速度是 FP16 的 2 倍，但 INT4 没有原生硬件支持，需要通过 INT8 模拟实现，实际加速约 1.5 倍。在 NVIDIA H100 上，FP8 有原生支持，速度是 FP16 的 2 倍且精度更好。因此，选择量化精度时不能只看理论压缩比，还要考虑目标硬件的计算能力。建议的决策路径：H100/A100 优先考虑 FP8（如果模型支持），A100 其次考虑 INT8 或 INT4（AWQ），消费级 GPU 使用 GGUF 格式。量化后的推理速度还需要考虑反量化开销——某些量化方法（如 GPTQ）的反量化操作较慢，可能抵消部分计算加速。

量化方法的选型决策流程

选择量化方法需要考虑目标硬件、精度要求和部署复杂度三个因素。在 NVIDIA H100 上部署时，FP8 是最佳选择——它有原生硬件支持，不需要复杂的校准过程，精度接近 FP16。在 NVIDIA A100 上部署时，AWQ INT4 是性价比最高的选择——它比 GPTQ 更快且精度相当。在消费级 GPU（如 RTX 4090）或 CPU 上本地部署时，GGUF 格式配合 llama.cpp 是最成熟的方案。对于精度要求极高的场景（如医疗、金融），建议使用 FP16 或 INT8，不要使用 INT4 以下的精度。量化后的模型必须在目标任务上进行质量验证，不能只看 PPL 指标。

量化与推理引擎的集成方式

不同的推理引擎对量化的支持方式不同。vLLM 支持 GPTQ、AWQ 和 FP8 三种量化格式，通过 model 参数自动检测量化类型。TensorRT-LLM 支持 INT8 和 FP8 量化，使用 TensorRT 编译器将量化模型编译为优化的 CUDA 内核。llama.cpp 使用 GGUF 格式，支持从 Q2_K 到 Q8_0 的多种精度级别。在选择量化方案时，需要同时考虑量化方法和推理引擎的兼容性——某些量化格式只被特定引擎支持。例如，AWQ 格式主要被 vLLM 支持，而 GGUF 格式主要被 llama.cpp 支持。混合使用不同量化方案（如权重用 INT4、激活用 FP8）需要引擎级别的支持。

混合精度量化的工程实践策略

混合精度量化是指对模型的不同层或不同组件使用不同的量化精度。常见的策略包括：(1) 首尾高精度——第一层和最后一层使用 FP16 或 INT8，中间层使用 INT4，因为首尾层对精度最敏感；(2) 注意力高精度——注意力层（QKV 投影和输出投影）使用 INT8，FFN 层使用 INT4，因为注意力层的量化误差更容易传播；(3) 关键通道保护——对 AWQ 识别出的显著通道保持高精度，其他通道使用低精度。这些策略可以在不显著增加模型大小的情况下提升量化质量。DeepSeek-V3 的训练就使用了 FP8 混合精度——前向传播使用 FP8，梯度累积使用 FP32，优化器状态使用 FP32。

✅ 参考答案与详解
  1. 核心原理: 量化的核心原理是将模型权重从高精度浮点数（FP16/BF16）压缩到低精度整数或浮点数（INT8/INT4/FP8），以减少显存占用和内存带宽需求。量化的数学本质是有损压缩——将连续的浮点空间映射到离散的整数空间。对称量化公式为q = round(x / scale)，其中scale = max(|x|) / (2^(b-1) - 1)；反量化为x_hat = q × scale。量化误差方差Var(e)与位数b呈指数关系Var(e) ∝ 2^(-2b)，每增加1位精度误差减小4倍。量化对推理加速效果显著的根本原因在于：LLM推理的decode阶段是memory-bound（每次只生成1个token但需读取全部模型权重），INT4量化将权重大小压缩为FP16的1/4，理论上可将内存带宽需求降至1/4，直接转化为4倍的吞吐提升。以70B模型为例，FP16需要140GB（2张A100），INT8需70GB（1张A100），INT4仅需35GB（半张A100），量化让70B模型从"需要2张A100"变成"1张A100就能跑"。现代量化方法（GPTQ、AWQ）非常精密——GPTQ用Hessian信息补偿量化误差，AWQ根据激活值分布保护重要通道，直接截断（round-to-nearest）的效果远不如这些方法。
    关键要点: ① INT4是部署的甜点精度——GPTQ/AWQ INT4在LLaMA-2 70B上PPL仅增加0.48-0.53 ② GPTQ用Hessian逐列补偿，AWQ用激活感知缩放，两者精度相当但AWQ更快 ③ 量化误差累积效应需注意——同时使用量化+稀疏注意力+KV Cache压缩时可能逐层放大 ④ FP8是新趋势——H100原生支持，比INT8更简单不需要复杂校准 ⑤ 量化是正交优化，可与FlashAttention、GQA/MLA、投机解码叠加使用
    实际案例: 某公司部署LLaMA-3 70B在线对话服务。FP16需要2张A100-80GB，每月GPU成本约$4000。使用AWQ INT4量化后，模型仅需35GB，单张A100即可运行且剩余45GB用于KV Cache，每月成本降至$2000。质量验证：WikiText-2 PPL从3.32增至3.80（+0.48），MMLU准确率从78.5%降至77.8%（-0.7%），GSM8K从72.3%降至70.1%（-2.2%），业务方确认可接受。
    关联知识: 量化与FlashAttention-3的关系——FA-3原生支持FP8，在H100上计算再提速；与投机解码的关系——草稿模型通常用量化版本进一步降低成本；与PagedAttention的关系——量化后的KV Cache更小，能放更多请求；与GQA/MLA的关系——两者是正交优化可以叠加，GQA减少KV头数，量化减少每个值的位数。
  2. 核心原理: 量化在实际工程中的首要性能瓶颈是精度退化——尤其在特定任务上可能比PPL指标反映的更严重。PPL反映的是语言建模的整体质量，但可能掩盖特定任务上的严重退化。例如INT4量化在PPL上只增加0.5，但在数学推理（GSM8K）上的准确率可能下降10%，在代码生成（HumanEval）上可能下降5-8%。诊断方法：(1) 运行完整Benchmark评测（MMLU、GSM8K、HumanEval）而非只看PPL；(2) 在目标任务上做A/B测试对比量化前后的输出质量；(3) 分析不同层的量化误差分布——首尾层对精度最敏感，中间层容忍度更高。第二个瓶颈是推理速度未必如理论预期——INT4在A100上没有原生硬件支持，需要通过INT8模拟实现，实际加速约1.5倍而非理论的4倍。反量化操作的开销也可能抵消部分计算加速——GPTQ的逐列反量化比AWQ的批量反量化更慢。第三个瓶颈是量化过程本身的时间成本——GPTQ需要几小时的校准计算，AWQ需要几十分钟的网格搜索。解决策略：H100优先考虑FP8（原生支持无需校准），A100考虑AWQ INT4，消费级GPU用GGUF+llama.cpp。混合精度量化（首尾高精度+中间低精度）可在不显著增加模型大小的情况下提升质量。
    关键要点: ① 不能只看PPL，必须在目标任务上做端到端质量验证 ② 量化精度选择要考虑目标硬件的计算能力 ③ 混合精度量化（首尾层INT8+中间层INT4）是实用的工程策略 ④ SmoothQuant通过数学变换将激活值的量化难度转移到权重上 ⑤ QLoRA在4-bit量化基础上叠加LoRA微调可恢复量化损失
    实际案例: 某医疗AI公司对LLaMA-2 70B进行INT4量化部署用于病历摘要。PPL测试仅增加0.5（可接受），但实际部署后发现医学术语的准确率下降了12%（不可接受）。最终采用混合精度方案：注意力层（QKV投影和输出投影）使用INT8，FFN层使用INT4，首尾层保持FP16。优化后PPL仅增加0.2，医学术语准确率下降控制在3%以内。
    关联知识: 量化精度与硬件的关系——A100的INT8 Tensor Core速度是FP16的2倍，但INT4无原生支持需INT8模拟；H100的FP8原生支持是FP16的2倍且精度更好；SmoothQuant的平滑因子α=0.5在大多数模型上接近最优，符合"等量化难度分配"原理；QLoRA的NF4数据类型对正态分布权重最优，双重量化可进一步节省内存。
  3. 核心原理: 量化技术的替代方案和横向对比：(1) GPTQ（逐层量化）——用Hessian矩阵信息逐列补偿量化误差，基于OBS方法。需要校准数据（几百个样本）和几小时的量化时间。INT4精度损失很小，推理速度快。代表方案：AutoGPTQ库。(2) AWQ（激活感知量化）——发现对应大激活值的权重通道更敏感，先给显著通道乘缩放因子放大，然后统一量化，推理时除回来。比GPTQ更快（不需要逐列补偿），精度相当或更好。vLLM/TRT-LLM原生支持。(3) GGUF（本地部署格式）——面向llama.cpp生态，支持Q2_K到Q8_0多种精度。Q4_K_M是推荐默认选择。设计哲学是"面向部署者"，提供多种精度让用户根据硬件权衡。(4) SmoothQuant（激活值量化）——通过数学变换Y = (X×S^(-1))×(S×W)将激活值中的异常值平滑转移到权重上，使激活值和权重都容易量化。INT8推理速度比FP16快1.56倍。(5) FP8/MXFP——H100原生支持的浮点格式，E4M3精度更高适合推理，E5M2范围更大适合训练。不需要复杂校准。MXFP每组32个值共享缩放因子，DeepSeek-V4首次大规模使用。(6) BitNet（1-bit量化）——研究前沿，将权重量化为{-1, 0, 1}三值，用加法替代乘法实现极致加速。(7) QLoRA——4-bit量化+LoRA微调，在极低计算成本下恢复量化损失。
    关键要点: ① 选择决策：H100→FP8, A100→AWQ INT4, 消费级GPU→GGUF ② GPTQ精度略好但更慢，AWQ更快且被更多推理引擎支持 ③ SmoothQuant是唯一同时量化权重和激活值的PTQ方法 ④ INT2量化不推荐用于生产——PPL从3.32增至6.80 ⑤ 混合精度量化策略：首尾高精度+中间低精度、注意力高精度+FFN低精度
    实际案例: 某推理平台同时维护三种量化模型：(1) 在8×H100集群上部署FP8版本用于高并发在线服务（延迟优先，每token 15ms）；(2) 在4×A100集群上部署AWQ INT4版本用于批处理任务（吞吐优先，成本降低50%）；(3) 在消费级RTX 4090上用GGUF Q4_K_M版本做本地demo演示。三种方案覆盖从高配到低配的完整硬件谱系，质量差异在业务可接受范围内。
    关联知识: 量化与推理引擎的集成方式——vLLM支持GPTQ/AWQ/FP8自动检测，TensorRT-LLM支持INT8/FP8编译优化，llama.cpp使用GGUF格式；量化与模型微调的关系——QLoRA将4-bit量化与LoRA微调结合，单张A100即可微调70B模型；量化误差的数学分析基于二阶泰勒展开L(W+ΔW) ≈ L(W) + g^TΔW + 0.5ΔW^THΔW，Hessian矩阵H决定量化误差ΔW的影响。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters

    return elapsed
