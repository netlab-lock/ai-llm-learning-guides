✂️ 3. Chunked Prefill 与 Prefill-Decode 分离

3.1 问题：连续批处理还不够

💡 通俗类比
连续批处理虽然解决了槽位空闲问题，但长prompt的prefill就像高速公路上的"大货车"——它占满了整条车道，后面的"小轿车"（decode请求）只能跟在后面慢慢等。
⚠️ 常见误区
很多人混淆了"连续批处理"和"Chunked Prefill"——前者解决的是"槽位复用"问题，后者解决的是"长prompt阻塞"问题。两者是互补关系，不是替代关系。
🔬 深入探讨
Prefill阻塞decode的根本原因是两者的计算特征完全不同：prefill是compute-bound（大量矩阵乘法），decode是memory-bound（读取KV Cache）。当它们混在同一batch时，prefill的计算密集特性会"饿死"decode的访存需求。

上一讲的连续批处理解决了"槽位空闲"的问题，但还有一个严重问题没解决：长 prompt 的 prefill 会阻塞 decode。

连续批处理下，长 prompt 仍然有问题: 请求A: prompt 有 8000 tokens → prefill 需要计算 8000 tokens 请求B: 正在 decode 请求C: 正在 decode Step 1: A_prefill_chunk1 B_decode C_decode Step 2: A_prefill_chunk2 (B、C 等待...) Step 3: A_prefill_chunk3 (B、C 等待...) ... Step 500: A_prefill_chunk500 (B、C 全程等待！) Step 501: A_decode B_decode C_decode ← 终于可以继续了 问题: A 的 prefill 期间，B 和 C 的 decode 被完全阻塞 → B、C 用户感受到明显的延迟抖动
⚠️ 为什么 prefill 会阻塞 decode？
Prefill 和 Decode 的计算特征完全不同：
Prefill: 一次处理大量 token → 矩阵乘法密集 → 计算密集型 (Compute-bound)
Decode: 每次只处理 1 个 token → 读取大量 KV Cache → 访存密集型 (Memory/IO-bound)

当它们混在同一个 batch 时，prefill 的大计算量会"霸占"GPU，decode 请求只能排队等。

3.2 Chunked Prefill：把长 prompt 切块

💡 通俗类比
Chunked Prefill就像把一顿大餐分成多道小菜上桌——不再等一整桌菜做完才开饭，而是做好一道上一道，同时其他菜也在做，食客（decode请求）不会饿着等。
⚠️ 常见误区
chunk_size的选择不是越大越好。太大的chunk会让decode请求等待时间过长，太小的chunk会增加调度开销和GPU碎片化。经验值512-2048 tokens是经过充分调优的平衡点。
🔬 深入探讨
Chunked Prefill的数学本质是将一个大的compute-bound任务分解为多个小任务，然后与memory-bound任务交替执行。这与操作系统的时间片轮转调度思想完全一致——只不过这里的"进程"是prefill chunk和decode step。

核心思想：不要一次性处理整个 prompt，把它切成小块，和 decode 请求交替执行。

3.3 Chunked Prefill 的调度逻辑

💡 通俗类比
调度逻辑就像餐厅厨房的"排菜"规则——先做快的菜（decode），再切大的菜（prefill chunk），保证每桌客人都能及时上菜。
⚠️ 常见误区
很多人以为Chunked Prefill只需要简单地把prompt切成固定大小的块。实际上还需要考虑：chunk之间的依赖关系（KV Cache需要累积）、与decode请求的配比、以及chunk完成后的状态转换。
🔬 深入探讨
chunk_size的选择存在一个理论最优值：当chunk的计算时间等于decode step的计算时间时，两者的GPU利用率都能达到最大。这个最优值取决于模型大小、GPU型号和batch size。
每个 iteration 的调度决策: 1. 从 waiting_queue 取出新请求 2. 如果新请求有长 prompt: → 切成 chunk_size 大小的块 → 只处理第一个 chunk，剩余放回队列 3. 把当前 chunk 和正在 decode 的请求组成 batch 4. GPU 执行一步 5. 如果某个请求的 prefill 还没完成 → 下一步继续处理下一个 chunk 6. 如果 prefill 完成 → 该请求进入 decode 阶段 chunk_size 的选择: 太小: prefill 碎片化，GPU 效率下降 太大: decode 请求仍然被阻塞太久 经验值: 512 ~ 2048 tokens

3.4 效果

💡 通俗类比
Chunked Prefill的效果就像把"堵车"变成"顺畅通行"——延迟抖动从"过山车"变成"平路"，用户体验显著改善。
⚠️ 常见误区
延迟抖动（Jitter）的降低比平均延迟的降低更重要。在实时对话场景下，用户对延迟的一致性非常敏感——如果每次回复的延迟差异很大（有时1秒有时10秒），用户体验会比稳定的5秒延迟更差。
🔬 深入探讨
Chunked Prefill的吞吐量提升（20-50%）主要来自两个方面：decode请求不再被阻塞（减少空闲时间）和GPU利用率提升（每步batch size更大）。这两个因素的乘积效应导致了显著的性能提升。
指标无 Chunked Prefill有 Chunked Prefill
Decode 请求延迟被长 prefill 阻塞 (可能数秒)每步都有机会执行 (ms 级)
延迟抖动 (Jitter)严重轻微
GPU 利用率prefill 期间 batch size 小每步 batch size 较大

3.5 Prefill-Decode 分离：更激进的方案

💡 通俗类比
Prefill-Decode分离就像把"厨师"和"服务员"分开——厨师（Prefill节点）专门做菜，服务员（Decode节点）专门上菜，各司其职效率更高。
⚠️ 常见误区
Prefill-Decode分离的最大代价是KV Cache传输。一个8000 token的prompt，其KV Cache可能有几百MB到几GB，需要在节点间传输。这个传输延迟可能抵消分离带来的收益。
🔬 深入探讨
Prefill-Decode分离的理论基础是"异构计算最优"——prefill和decode的计算特征完全不同（compute-bound vs memory-bound），使用不同配置的GPU（高算力 vs 高带宽）能实现更优的总体效率。

Chunked Prefill 让 prefill 和 decode 交替执行，但它们仍然在同一组 GPU 上。更激进的方案是把它们拆到不同的 GPU 上。

Prefill-Decode 分离 (DistServe / Splitwise): 传统方式: GPU0-7: 既做 Prefill 又做 Decode → 两种计算互相干扰 分离方式: GPU0-3 (Prefill 节点): 专门处理 prompt 的 prefill GPU4-7 (Decode 节点): 专门生成 token 请求流程: 1. 用户请求到达 → 发送到 Prefill 节点 2. Prefill 节点处理完 prompt → 把 KV Cache 传输到 Decode 节点 3. Decode 节点逐 token 生成 → 返回结果 两种计算完全解耦，互不干扰！

3.6 Prefill-Decode 分离的优劣

💡 通俗类比
分离方案就像"专业分工"——让算力强的GPU做prefill，让带宽大的GPU做decode，就像让力气大的人搬箱子，让速度快的人送快递。
⚠️ 常见误区
Prefill-Decode分离的部署复杂度被严重低估。你需要：独立的资源池管理、KV Cache传输的网络优化、故障时的跨节点恢复机制、以及动态负载均衡。这些都是工程上的巨大挑战。
🔬 深入探讨
Prefill-Decode分离的弹性优势在实际生产中价值巨大：白天对话请求多（decode密集），可以给decode节点分配更多GPU；晚上批量处理多（prefill密集），可以给prefill节点分配更多GPU。这种动态调配能力是单体架构无法实现的。

资源利用可以独立扩缩容需要更多 GPU
弹性Prefill 和 Decode 按需配比架构复杂度增加

3.7 Chunked Prefill vs Context Parallelism (CP)

💡 通俗类比
Chunked Prefill和Context Parallelism的区别就像"把大蛋糕切成小块分着吃"（时间维度）和"多个人同时吃一块大蛋糕"（空间维度）——前者是调度策略，后者是并行策略。
⚠️ 常见误区
很多人误以为有了CP就不需要Chunked Prefill。实际上CP只加速了prefill的计算速度，但没有消除prefill对decode的阻塞。两者解决的是不同层面的问题，可以叠加使用。
🔬 深入探讨
Chunked Prefill和CP的组合使用是当前长序列推理的最优实践：CP在空间维度将序列切分到多卡并行计算，Chunked Prefill在时间维度将prefill和decode交替执行。两者协同可以实现"长序列+低延迟"的双重优化。

这两个技术名字有点像，但解决的问题完全不同：

Chunked PrefillContext Parallelism

解决什么长 prompt 阻塞 decode序列太长单卡算不下
做法把 prompt 切块，和 decode 交替执行把序列切到多卡并行计算
需要多卡？不需要必须多卡
通信开销无有 (卡间交换 KV)

💡 CP 能解决 Chunked Prefill 的问题吗？
CP 能间接缓解——它让 prefill 变快（多卡并行），阻塞时间缩短。但 CP 没有消除阻塞——prefill 和 decode 仍然是串行的。Chunked Prefill 是从调度层面消除阻塞。两者可以叠加使用：CP 加速每个 chunk 的计算，Chunked Prefill 保证 decode 不被阻塞。

3.8 实际部署中的组合

💡 通俗类比
现代推理引擎的优化技术就像"瑞士军刀"的各个工具——每个工具解决一个问题，组合起来才能应对所有场景。
⚠️ 常见误区
技术组合的顺序很重要：先有连续批处理（解决调度），再有Chunked Prefill（解决阻塞），然后是PagedAttention（解决内存），最后是FlashAttention（加速计算）。乱序组合可能导致收益相互抵消。
🔬 深入探讨
vLLM、SGLang、TensorRT-LLM的优化组合各有侧重：vLLM以PagedAttention为核心，SGLang以RadixAttention为核心，TensorRT-LLM以NVIDIA硬件优化为核心。选择时需要根据具体场景（长序列/高并发/低延迟）权衡。
现代推理引擎的典型组合: 连续批处理 + Chunked Prefill ← 解决长 prompt 阻塞 + PagedAttention ← 解决 KV Cache 碎片 + FlashAttention ← 解决注意力计算瓶颈 + 投机解码 ← 解决一次一个 token 的瓶颈 + 量化 ← 解决模型太大的问题 vLLM: 连续批处理 + Chunked Prefill + PagedAttention + FA SGLang: 连续批处理 + Chunked Prefill + RadixAttention + FA TRT-LLM: 连续批处理 + Chunked Prefill + FP8 量化 + FA-3

3.9 Chunked Prefill 的数学建模

为了定量理解 Chunked Prefill 的收益，我们可以建立一个简化的排队论模型。设 prefill 的计算时间为 T_p，decode 每步的时间为 T_d，chunk_size 为 C。在没有 Chunked Prefill 时，一个长 prompt（N 个 token）的 prefill 需要 N/C × T_p 的时间，在此期间所有 decode 请求被阻塞。使用 Chunked Prefill 后，每 C 个 token 的 prefill 与 decode 交替执行，decode 请求的最大等待时间从 N/C × T_p 降低到 T_p。

最优 chunk_size 的理论分析

chunk_size 的选择需要平衡两个因素：(1) GPU 利用率——chunk 越大，矩阵乘法的规模越大，GPU 的算力利用率越高；(2) decode 延迟——chunk 越大，decode 请求的等待时间越长。理论最优值是让 chunk 的计算时间等于 decode batch 的计算时间：T_prefill(chunk_size) = T_decode(batch_size)。对于典型的 LLaMA-70B 模型在 A100 上，这个最优值约在 512-1024 tokens 之间。实际部署中建议通过 A/B 测试找到具体场景的最优值。

Chunked Prefill 对首 token 延迟的影响

Chunked Prefill 会略微增加首 token 延迟（Time To First Token, TTFT），因为 prefill 被切分为多个 chunk，每个 chunk 之间有调度开销和与 decode 的切换开销。实测表明，对于 4K token 的 prompt，TTFT 增加约 5-15%；对于 32K token 的 prompt，TTFT 增加约 3-8%（因为长 prompt 的 prefill 本身就很慢，chunk 间开销占比小）。总体而言，TTFT 的小幅增加是值得的——换来的是 decode 延迟的显著降低和整体吞吐量的提升。

# vLLM 中配置 Chunked Prefill 的示例
from vllm import LLM, SamplingParams

# 启用 Chunked Prefill (vLLM 0.4+ 默认启用)
llm = LLM(
    model="meta-llama/Llama-3-70B-Instruct",
    max_num_seqs=64,          # 最大并发请求数
    max_model_len=32768,       # 最大序列长度
    enable_chunked_prefill=True,  # 启用 Chunked Prefill
    # chunk_size 隐含在 max_num_seqs 和 max_model_len 中
)

# 发送长短混合的请求
prompts = [
    "Hello" * 10,      # 短 prompt
    "Hello" * 1000,    # 中等 prompt  
    "Hello" * 8000,    # 长 prompt
]
params = SamplingParams(temperature=0.7, max_tokens=100)
outputs = llm.generate(prompts, params)
💡 实践技巧
在生产环境中配置 Chunked Prefill 时，建议先用默认参数（chunk_size=512 或 1024），然后通过监控 P99 延迟和吞吐量来微调。如果 P99 延迟过高，可以减小 chunk_size；如果吞吐量不够，可以增大 chunk_size。vLLM 和 SGLang 都支持通过配置文件动态调整，不需要重启服务。
⚠️ 常见误区
Chunked Prefill 不是万能的。在以下场景中它的收益有限：(1) 所有 prompt 长度都很短（<512 tokens）——没有长 prompt 需要切块；(2) batch size 很小（<4）——没有足够的 decode 请求需要保护；(3) 请求模式是纯 prefill（如批量 embedding 生成）——没有 decode 阶段。在这些场景中，Chunked Prefill 的调度开销反而会降低性能。

3.10 Chunked Prefill 的工程实践与性能调优

动态 chunk_size 调整策略

固定的 chunk_size 无法适应动态变化的请求模式。当请求队列中短请求较多时，应减小 chunk_size 以降低 decode 延迟；当长请求较多时，应增大 chunk_size 以提高 GPU 利用率。SGLang 实现了自适应 chunk_size 策略：它监控当前队列的平均 prompt 长度和 decode 请求数量，动态调整 chunk_size。具体公式为 chunk_size 等于最小值（最大允许 chunk_size，平均 prompt 长度除以当前 decode 请求数加一）。这个公式确保每个 chunk 的计算时间与 decode batch 的计算时间大致匹配。实验表明，自适应策略比固定策略的吞吐量高 10-15%。

vLLM 中配置 Chunked Prefill 的代码示例

from vllm import LLM, SamplingParams

# 配置 Chunked Prefill (vLLM 0.4+ 默认启用)
llm = LLM(
    model="meta-llama/Llama-3-70B-Instruct",
    max_num_seqs=64,
    max_model_len=32768,
    enable_chunked_prefill=True,
)

# 发送长短混合请求测试
prompts = [
    "Hello" * 10,      # 短 prompt
    "Hello" * 1000,    # 中等 prompt
    "Hello" * 8000,    # 长 prompt
]
params = SamplingParams(temperature=0.7, max_tokens=100)
outputs = llm.generate(prompts, params)
for o in outputs:
    print(f"Prompt len: {len(o.prompt_token_ids)}, "
          f"Output len: {len(o.outputs[0].token_ids)}")

Prefill-Decode 分离的 KV Cache 传输优化

Prefill-Decode 分离架构的最大瓶颈是 KV Cache 的跨节点传输。一个 8000 token 的 prompt，使用 LLaMA-70B 模型（GQA, g=8），其 KV Cache 大小约为 2.5GB（FP16）。在 100Gbps 网络下传输需要约 200ms，这个延迟可能抵消分离带来的部分收益。优化方案包括：(1) KV Cache 压缩——在传输前用 INT8 或 FP8 量化 KV Cache，将传输量减半；(2) 流式传输——prefill 完成一个 chunk 就立即传输对应的 KV，不需要等整个 prompt 处理完；(3) RDMA 直接内存访问——绕过 CPU 和操作系统内核，实现 GPU 内存之间的直接传输，延迟降低 50-70%。DistServe 论文的实验表明，使用 RDMA + FP8 压缩的组合方案，KV Cache 传输延迟可以控制在 50ms 以内。

Chunked Prefill 的实现细节与注意事项

实现 Chunked Prefill 时需要注意几个关键细节。首先是 chunk 之间的 KV Cache 累积——每个 chunk 计算完后需要将新的 KV 追加到该请求的 KV Cache 中，这需要正确管理 KV Cache 的偏移量。其次是 chunk 与 decode 请求的 padding——不同 chunk 的长度可能不同，需要 padding 到相同长度才能组成 batch。padding 会浪费计算资源，因此 chunk_size 的选择应该尽量与 GPU 的最优矩阵大小对齐（通常是 2 的幂次）。最后是 chunk 完成后的状态转换——prefill 全部完成后，请求需要从 waiting 状态转移到 running 状态，开始 decode 阶段。

Context Parallelism 与 Ring Attention 的实现原理

Context Parallelism（CP）将长序列切分到多张 GPU 上并行计算。Ring Attention 是 CP 的一种高效实现：每张 GPU 持有序列的一个分片，通过环形通信模式依次传递 KV 向量。在每次迭代中，GPU_i 向 GPU_{i+1} 发送自己分片的 KV，同时接收 GPU_{i-1} 的 KV，并用接收到的 KV 计算注意力。经过 N 次迭代后（N 为 GPU 数量），每个 GPU 都看到了完整的序列。Ring Attention 的通信与计算可以重叠——在等待接收下一个分片的 KV 时，可以先计算当前分片的注意力，从而隐藏大部分通信延迟。

# 问题：连续批处理还不够 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
深入探讨： 从Chunked Prefill的核心原理出发，将长prompt的预填充阶段切分为多个chunk分批处理。为什么需要切分prefill？推导：长prompt的prefill会独占GPU计算资源很长时间，导致decode请求被阻塞（排队延迟增大）。切分为chunk后，每个chunk的计算时间缩短，在chunk之间可以插入decode请求的处理，显著降低排队延迟。数学本质上，chunk大小C的选择影响吞吐-延迟权衡：C太大失去chunking意义，C太小则kernel启动开销占比增大。经验上C=512-2048 token是较优区间。
常见误区： 学习Chunked Prefill时，很多人只看理论分析不测量实际延迟分布。建议对比有无chunking的TTFT（首token延迟）和decode延迟，观察chunk大小对排队延迟的影响。
常见误区： 不要以为Chunked Prefill就是简单地把prompt切分就可以跳过调度策略。chunk大小选择、与decode请求的交织策略等直接影响系统吞吐和延迟。
