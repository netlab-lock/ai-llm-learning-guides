📐 01 · 什么是 Scaling Law

入门 概念 直觉建立

1. 一句话定义

Scaling Law（缩放定律/规模法则）描述的是：当模型的某个维度（参数量、数据量、计算量）按比例增大时，模型性能（通常用损失函数衡量）会以可预测的幂律关系持续改善。

Loss ∝ (Scale)^(-α)

其中 Scale 可以是参数量 N、数据量 D 或计算量 C，α 是一个小正数（通常 0.05~0.34）。

2. 为什么 Scaling Law 重要？

Scaling Law 之所以被视为现代 AI 研究中最重要的发现之一，是因为它揭示了一个令人震惊的事实：

💡 通俗类比：Scaling Law 就像「摩尔定律」之于芯片行业——它不是一个物理定律，而是一个经验观察到的趋势。摩尔定律告诉芯片工程师「每两年晶体管数量翻一倍」，让整个行业有了可预测的路线图。Scaling Law 告诉 AI 研究者「算力每翻一倍，损失按固定比例下降」，让 AI 的发展变得可规划、可投资。

这意味着：

  • 投资可预测：公司可以提前计算「投入 X 算力能得到什么水平的模型」
  • 路线图可规划：OpenAI 在 2020 年就基于 Scaling Law 预测了 GPT-4 级别模型的算力需求
  • 竞争格局清晰：谁拥有更多算力和数据，谁就能训练出更强的模型

对工业界的影响

Scaling Law 直接影响了数十亿美元的投资决策。当 OpenAI 向微软展示其 Scaling Law 研究时，微软理解了这一趋势的商业含义——只要持续投入算力，AI 的能力就会按照可预测的曲线提升。这直接促成了微软对 OpenAI 的 130 亿美元投资。可以说，Scaling Law 不仅是一个科学发现，更是 AI 行业的「投资纲领」。

3. 直觉理解：类比生活

🏋️ 健身类比

想象你在健身：你投入的训练时间翻倍，力量不会翻倍——但会按照一个可预测的曲线增长。这就是幂律关系。初期进步飞快，后期进步缓慢但持续。没有人能在一周内从新手变成奥运冠军，但只要坚持训练，力量就会持续增长——增长的速度遵循一个数学规律。

💰 经济学类比

就像规模经济：工厂生产 100 万个零件的单位成本比生产 1 万个低，但不是线性降低——而是按照某种规律递减。Scaling Law 告诉我们，AI 模型也遵循类似的规律。投入翻倍的算力，不会得到翻倍的性能——但会得到可预测的、持续的改善。

💡 通俗类比：学语言也是一个很好的类比。认识 1000 个单词，你只能读懂简单的文章；认识 5000 个单词，你能读懂大部分新闻；认识 10000 个单词，你能读懂学术论文。每增加 1000 个单词，你的理解能力都在提升——但提升的幅度在递减。从 1000 到 2000 的提升感远大于从 9000 到 10000。这就是幂律关系在日常生活中的体现。

Scaling Law 的核心直觉： 投入 (算力/数据/参数) 产出 (模型性能) ───────────────────── ───────────────────── 翻 1 倍 损失下降 ~7% 翻 10 倍 损失下降 ~25% 翻 100 倍 损失下降 ~40% 翻 1000 倍 损失下降 ~50% 关键：每次翻倍都在改善，但改善幅度递减 好消息：永远不会「停止改善」 坏消息：改善的成本越来越高 ┌─────────────────────────────────────────────┐ │ 投入少 → 进步快（低垂的果实） │ │ 投入多 → 进步慢（高处的果实需要更多努力） │ │ 投入无限 → 理论上趋近完美（但永远达不到） │ └─────────────────────────────────────────────┘

4. Scaling Law 的三个核心变量

几乎所有 Scaling Law 研究都围绕三个变量展开：

变量符号含义直观理解
模型参数量N (Parameters)神经网络中可学习参数的数量模型的「脑容量」——能存储多少知识
训练数据量D (Dataset)训练时使用的 token 数量模型读过多少书
训练计算量C (Compute)训练消耗的 FLOPs训练投入的总「能量」
Scaling Law 三大变量关系： ┌──────────┐ │ 模型性能 │ │ (Loss) │ └─────┬────┘ │ ┌───────────────┼───────────────┐ │ │ │ ┌─────┴─────┐ ┌─────┴─────┐ ┌─────┴─────┐ │ N 参数量 │ │ D 数据量 │ │ C 计算量 │ │ (大脑) │ │ (教材) │ │ (电费) │ └───────────┘ └───────────┘ └───────────┘ 三者的关系：C ≈ 6 × N × D (训练一个 N 参数模型处理 D 个 token 大约需要 6ND FLOPs) 核心权衡： - 参数多 + 数据少 → 过拟合（记住了但不理解） - 参数少 + 数据多 → 欠拟合（看了很多但记不住） - 两者平衡 + 算力足 → 最优性能（Chinchilla 最优）

三者之间的数学关系

这三个变量不是独立的，它们之间存在紧密的数学联系。最著名的近似关系是：

C ≈ 6 × N × D

其中 C 是训练的总浮点运算次数（FLOPs），N 是模型参数量，D 是训练 token 数。这个公式的直觉是：每个参数在每个 token 上大约需要 6 次浮点运算（前向传播 2 次 + 反向传播 4 次）。这个关系是 Chinchilla 论文（模块 04）的核心基础。

⚠️ 常见误区：很多人以为 Scaling Law 就是「模型越大越好」。这是错误的！Scaling Law 说的是在给定算力预算下，存在一个参数量和数据量的最优配比。如果你只有 10²⁵ FLOPs 的算力，训练一个 1T 参数但只用少量数据的模型，效果可能还不如一个 10B 参数但用充足数据训练的模型。参见 模块 04: Chinchilla Scaling Laws 了解最优配比的详细讨论。

5. 历史脉络：三个时代

时代时间核心观点代表关键论文
第一代2020模型越大越好，参数优先Kaplan / OpenAIScaling Laws for Neural LMs
第二代2022参数与数据要平衡Chinchilla / DeepMindTraining Compute-Optimal LLMs
第三代2024-2025推理时也能缩放o1, R1 / Test-timeScaling LLM Test-Time Compute

第一代：Kaplan 时代（2020）

OpenAI 的 Kaplan 等人在 2020 年发表了具有里程碑意义的论文，首次系统性地展示了神经语言模型中的 Scaling Law。他们的核心发现包括三条独立的幂律曲线：模型参数量 N、数据集大小 D、训练计算量 C 分别与损失 Loss 之间存在幂律关系。Kaplan 的一个关键建议是：在固定算力预算下，应该优先增大模型参数量（N），而非增加训练数据量（D）。这意味着大模型即使「吃不饱」数据，也比小模型吃饱数据效果更好。

第二代：Chinchilla 修正（2022）

DeepMind 的 Hoffmann 等人通过 Chinchilla 实验修正了 Kaplan 的结论。他们发现 Kaplan 的实验设计存在系统性偏差——所有实验都在固定数据量（而非固定算力）的条件下进行，导致高估了参数量的作用。Chinchilla 的核心发现是：参数量和数据量应该等比例增长。具体来说，最优配比约为 N:D ≈ 1:20（每 1 个参数需要约 20 个训练 token）。这意味着 GPT-3（175B 参数）如果按 Chinchilla 最优来训练，应该使用约 3.5T tokens 的数据——而非实际使用的 300B tokens。详见 模块 04。

第三代：推理时缩放（2024-2025）

OpenAI 的 o1 模型和 DeepSeek 的 R1 模型引入了第三条缩放轴——推理时计算量（Test-time Compute）。传统 Scaling Law 只关注训练时的算力投入，但 o1 证明了在推理阶段投入更多计算（让模型「思考更久」）也能显著提升性能。这开辟了一个全新的维度：即使模型参数量不变，通过延长推理时间也能获得更好的结果。详见 模块 08。

6. Scaling Law 不是什么

⚠️ 常见误解：

  • ❌ Scaling Law 不是说「模型越大一定越好」——它说的是「在给定约束下，存在可预测的最优策略」
  • ❌ Scaling Law 不是万能的——它描述的是统计趋势，不是对每个任务都成立
  • ❌ Scaling Law 不意味着「只要堆算力就行」——数据质量、架构设计同样重要
  • ❌ Scaling Law 不是物理定律——它是经验观察，可能在某个规模范围内失效
  • ❌ Scaling Law 不能告诉你具体某个任务上的表现——它只描述总体损失的趋势

⚠️ 常见误区：「Scaling Law 已经被证伪了」。这种说法源于对 Scaling Law 的误解。Scaling Law 从未声称能在所有任务上给出精确预测——它描述的是预训练损失（cross-entropy loss）与规模之间的统计关系。在这一范围内，Scaling Law 始终成立。被证伪的是某些人对 Scaling Law 的过度引申——比如「只要模型够大就能解决任何问题」。关于 Scaling Law 的局限性，详见 模块 09: 局限性与批评。

8. Scaling Law 的数学形式

Scaling Law 最通用的数学形式为：

L(S) = (S₀ / S)^α + L∞

其中各参数的含义如下：

符号名称典型值含义
L(S)损失函数1.5 ~ 3.5 nats模型在给定规模下的交叉熵损失
S规模变量N, D, 或 C可以是参数量、数据量或计算量
S₀特征规模10⁸ ~ 10¹⁴幂律关系的特征常数
α缩放指数0.05 ~ 0.34幂律的斜率，越大表示规模增长越有效
L∞不可约损失~1.69 nats即使完美模型也无法突破的损失下限

🔬 深入探讨：为什么性能遵循幂律而非指数或对数？

幂律关系（Loss ∝ Scale^(-α)）的出现不是巧合，有多种理论解释：

理论核心思想类比
统计力学神经网络学习类似物理系统趋向平衡态，幂律是临界现象的普遍特征水在 100°C 沸腾——临界点附近的标度律
信息论数据中的信息量与模型容量的关系本质上是幂律的压缩文件大小与压缩算法复杂度的关系
组合数学任务复杂度与所需参数之间的关系遵循组合幂律解锁 n 位密码需要尝试 10^n 种组合
经验规律在 10 个数量级的范围内观察到的稳定规律地震震级与频率的关系（Gutenberg-Richter 定律）

关键洞察：幂律意味着"边际收益递减"——每翻倍算力，性能提升的绝对量越来越小。但好消息是，收益永远不会降到零——总有提升空间。

从数学角度看，幂律关系在对数-对数坐标系下表现为直线。如果我们将 log(Loss) 对 log(Scale) 作图，幂律关系就是一条斜率为 -α 的直线。这一特性使得我们可以通过小规模实验来拟合幂律参数，然后外推到大规模——尽管外推的可靠性存在争议（详见 模块 09）。

论文 "Scaling Laws for Neural Language Models" (Kaplan et al., 2020) 通过在超过 7 个数量级的范围内进行系统实验，确认了幂律关系的稳定性。他们发现，从 768 参数的小模型到 1.5B 参数的大模型，log(Loss) vs log(N) 的图始终呈现良好的线性关系，拟合的 R² 值超过 0.99。

🔬 深入探讨：Scaling Law 与贝叶斯最优

从贝叶斯统计的角度来看，Scaling Law 可以理解为模型后验分布与真实数据分布之间 KL 散度的收敛过程。假设真实数据分布为 P*(x)，模型学到的分布为 P_θ(x)，则交叉熵损失可以分解为：

L = H(P*, P_θ) = H(P*) + D_KL(P* || P_θ)

其中 H(P*) 是数据的固有熵（不可约损失），D_KL 是模型分布与真实分布之间的 KL 散度。Scaling Law 描述的是 D_KL 随模型规模增大而按幂律下降的过程。这意味着：

  • 模型规模越大，学到的分布越接近真实分布
  • 收敛速度遵循幂律，意味着存在「最后一公里问题」——接近完美分布的最后几步极其困难
  • 不可约损失 H(P*) 代表了数据本身的随机性，这是任何模型都无法消除的

这一分解也解释了为什么不同的数据集有不同的 Scaling 行为：数据本身的熵 H(P*) 不同，导致幂律曲线的渐近线不同。高质量、低噪声的数据集 H(P*) 更低，因此同样的模型规模能达到更低的损失。

10. Scaling Law 的计算实操

理解 Scaling Law 不仅需要理论知识，还需要掌握实际计算方法。以下是一个典型的算力估算流程：

# 估算训练一个语言模型所需的算力 (FLOPs)
# 基于 Kaplan et al. (2020) 的近似公式

def estimate_training_flops(N_params, D_tokens):
    """
    估算训练所需的总浮点运算次数
    N_params: 模型参数量 (例如 175e9 表示 175B)
    D_tokens: 训练 token 数 (例如 300e9 表示 300B tokens)
    返回: 总 FLOPs
    """
    # 经典公式: C ≈ 6 × N × D
    # 每个参数在每个 token 上约需 6 次浮点运算
    # (前向传播 2 次矩阵乘法 + 反向传播 4 次)
    C = 6 * N_params * D_tokens
    return C

# GPT-3 的计算
gpt3_flops = estimate_training_flops(175e9, 300e9)
print(f"GPT-3 训练算力: {gpt3_flops:.2e} FLOPs")
# 输出: GPT-3 训练算力: 3.15e+23 FLOPs

# 用 Kaplan Scaling Law 预测损失
# Loss(N) ≈ (N_c / N)^0.076 + 1.69
import math
N_c = 8.8e13  # 特征参数量
alpha_N = 0.076
L_inf = 1.69   # 不可约损失

for N in [1e8, 1e9, 1e10, 1e11, 1e12]:
    loss = (N_c / N)**alpha_N + L_inf
    print(f"N = {N:.0e} → Loss = {loss:.3f}")
# 输出:
# N = 1e+08  → Loss = 3.320
# N = 1e+09  → Loss = 2.985
# N = 1e+10  → Loss = 2.696
# N = 1e+11  → Loss = 2.447
# N = 1e+12  → Loss = 2.233

如何用 Scaling Law 做资源规划

在实际工业应用中，Scaling Law 最重要的用途之一是资源规划。假设你是一个 AI 公司的 CTO，想训练一个在 MMLU 基准上达到 85% 准确率的模型。你可以：

  1. 先在小规模（如 1B、3B、7B 参数）上训练几个模型，测量损失
  2. 用这些数据点拟合幂律参数（α 和 S₀）
  3. 用拟合的公式外推到目标准确率对应的损失值
  4. 反推出需要的模型规模和数据量
  5. 计算所需的总算力和成本

当然，这种方法的可靠性取决于外推的距离——距离越远，误差越大（详见 模块 09）。但在 1-2 个数量级的范围内，预测通常是相当可靠的。

# 一句话定义 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

# 一句话定义 - 性能基准测试
import torch, time
def benchmark(model, x, n=100):
    for _ in range(10): model(x)
    torch.cuda.synchronize()
    t0 = time.perf_counter()
    for _ in range(n): model(x)
    torch.cuda.synchronize()
    print(f"Latency: {(time.perf_counter()-t0)/n*1000:.2f}ms")
