1. 第三条缩放轴的发现

2024 年之前，Scaling Law 只有两个维度：训练时的参数量 N 和数据量 D。2024 年 8 月，一个革命性的发现出现了：

不仅训练时可以缩放，推理时也可以！

2. 什么是推理时 Scaling？

推理时 Scaling（Test-Time Compute Scaling）的核心思想是：

让模型在回答问题时「思考更久」——生成更多中间推理步骤、搜索更多候选答案、进行自我验证——可以提升最终答案的质量。这个思想的革命性在于：它打破了「模型能力在训练后就固定了」的传统观念。即使是一个已经训练好的模型，通过在推理时投入更多计算，也能获得更好的表现。

类比：人类的 System 1 vs System 2

System 1 (快思考): System 2 (慢思考): "2 + 2 = ?" "17 × 23 = ?" → 立即回答: 4 → 让我想想... → 0.1 秒 → 17 × 20 = 340 → 17 × 3 = 51 → 340 + 51 = 391 → 用了 5 秒，但答对了 传统 LLM: 推理模型 (o1/R1): 只有 System 1 有 System 1 + System 2 快速生成第一个答案 深思熟虑后给出答案 难题正确率: ~12% 难题正确率: ~83% (AIME 数学竞赛) (AIME 数学竞赛)

🔬 推理时 Scaling 的理论基础：为什么「想更久」能提升性能？从计算复杂性的角度看，许多数学推理问题属于 NP-hard 或更高的复杂性类别。对于这类问题，直觉式的单次前向传播（System 1）本质上是在做「贪婪搜索」——它只考虑最可能的下一步，而不会回溯或探索其他可能性。推理时 Scaling 通过引入搜索、验证和回溯机制，将推理过程从贪婪搜索升级为更彻底的搜索策略，从而能够发现那些「不直觉但正确」的推理路径。从某种意义上说，推理时 Scaling 是在用计算换正确率——这与 AlphaGo 用 MCTS 击败人类棋手的原理相同。

4. 推理时 Scaling 的方法

主要有四种策略来消耗更多推理时计算：

策略一：Best-of-N 采样

问题 → 模型 → 答案1 → 模型 → 答案2 → 模型 → 答案3 → ... → 模型 → 答案N ↓ Verifier 选出最佳答案 计算量: N × 单次推理成本 性能提升: ∝ N^γ (γ ≈ 0.1-0.3)

最简单的方法：生成 N 个候选答案，用验证器（Verifier）选最好的。N 越大，计算越多，质量越高。验证器可以是：(1) 一个独立的判别模型；(2) 模型自身的置信度评分；(3) 外部工具（如代码执行器）；(4) 多数投票（majority voting）。

策略二：思维链搜索（Chain-of-Thought Search）

问题: "计算 137 × 28" 思考路径1: 137 × 28 → 137 × 30 - 137 × 2 → 4110 - 274 → 3836 ✓ 思考路径2: 137 × 28 → 100×28 + 37×28 → 2800 + 1036 → 3836 ✓ 思考路径3: 137 × 28 → 140×28 - 3×28 → 3920 - 84 → 3836 ✓ 搜索更多路径 → 更可能找到正确路径 关键: 不是所有路径都正确，搜索 + 验证是核心 
# 思维链搜索的简化实现
import random

def cot_search(model, question, n_paths=5, max_steps=10):
    """Chain-of-Thought 搜索：生成多条推理路径，选择最佳"""
    paths = []
    for i in range(n_paths):
        # 从问题开始，逐步生成推理步骤
        reasoning = [question]
        for step in range(max_steps):
            next_step = model.generate_next_step(
                reasoning, temperature=0.7 + 0.3 * random.random()
            )
            reasoning.append(next_step)
            # 如果模型认为推理完成，停止
            if model.is_complete(reasoning):
                break
        # 提取最终答案并评分
        answer = extract_answer(reasoning)
        confidence = model.score_reasoning(reasoning)
        paths.append({
            'reasoning': reasoning,
            'answer': answer,
            'confidence': confidence,
            'n_steps': len(reasoning)
        })
    
    # 选择置信度最高的路径
    best = max(paths, key=lambda p: p['confidence'])
    return best

# 实验：不同路径数对准确率的影响
# N=1:  ~45% (单次推理)
# N=5:  ~58% (5条路径搜索)
# N=20: ~68% (20条路径搜索)
# N=100: ~76% (100条路径搜索)

策略三：自我验证与修正

模型生成答案后，再花时间验证和修正。这是一种「元认知」能力——模型不仅需要解决问题，还需要能够判断自己的答案是否正确。具体流程如下：

  1. 生成初始答案——快速给出一个「第一直觉」的答案
  2. 检查答案是否合理——代入原题验证、检查边界条件、测试特殊情况
  3. 如果发现错误，修正并重新推理——可能需要回溯到更早的推理步骤，甚至完全推翻之前的思路
  4. 重复直到满意或用完计算预算——设置最大迭代次数，避免无限循环
自我验证的完整流程： 问题 → 生成答案A → 验证A │ ┌────┴────┐ │ │ A 正确 A 有误 │ │ ▼ ▼ 输出 A 分析错误原因 │ ┌────┴────┐ │ │ 小错误 大错误 │ │ ▼ ▼ 修正 A 重新推理 │ │ ▼ ▼ 验证 A' 生成答案B → 验证B → ...

💡 自我验证的关键：自我验证的有效性取决于模型能否准确地识别自己的错误。研究表明，模型在「判断一个答案是否正确」方面的准确率，通常高于「直接生成正确答案」的准确率。这是因为验证是一个相对简单的任务（只需检查已有答案的正确性），而生成是一个更复杂的任务（需要从零开始构造答案）。这种不对称性是 Best-of-N 和自我验证策略有效的根本原因。

策略四：树搜索（Tree Search / MCTS）

将推理过程组织成树结构，用类似 AlphaGo 的方法搜索最优路径。每个节点代表一个推理状态（如当前的思维链），每条边代表一个推理步骤。通过评估函数（如模型的置信度或验证器的评分）来指导搜索方向。

树搜索推理过程示意： 问题 ├── 路径A: 步骤1 → 步骤2 → 步骤3 → 答案₁ (得分: 0.9) ✓ │ ├── 路径B: 步骤1 → 步骤2' → ✗ 死胡同 (剪枝) │ ├── 路径C: 步骤1' → 步骤2'' → 步骤3'' → 答案₂ (得分: 0.6) │ └── 路径D: 步骤1'' → 步骤2''' → 步骤3''' → 答案₃ (得分: 0.85) 搜索预算: 探索 N 个节点 最终选择: 答案₁ (得分最高)

5. OpenAI o1：推理时 Scaling 的里程碑

⚠️ 常见误区：不要以为推理时 Scaling 只是「让模型多想一会儿」就能提升质量。关键在于如何思考——o1 通过强化学习训练出高质量的内部推理策略（思维链、自我验证、回溯），而不是简单地生成更多 token。如果只是随机生成更多文本，质量不会提升反而可能下降。换句话说，推理时 Scaling 需要「有质量的思考」，而不仅仅是「更多的思考」。o1 的核心贡献不在于「让模型想更久」，而在于「教会模型如何有效地思考」。

OpenAI o1（2024 年 9 月）是第一个大规模商用的推理时 Scaling 模型：

特性传统 GPT-4o1
思考方式直接输出答案内部推理后输出
推理 token~100-500~1,000-50,000
推理成本低高（5-50 倍）
数学竞赛~12% (AIME)~83% (AIME)
编程竞赛~50th percentile~89th percentile

关键发现：o1 的性能随着推理 token 数量的增加而持续提升，遵循幂律关系！这验证了推理时 Scaling Law 的存在。更具体地说，当 o1 在内部推理中使用 1000 个 token 时，其 AIME 正确率约为 50%；当推理 token 增加到 10000 时，正确率提升到约 70%；当推理 token 增加到 50000 时，正确率进一步提升到约 83%。这种「更多思考 → 更好结果」的幂律关系，与训练时的 Scaling Law 形式上完全一致。

6. DeepSeek-R1：开源验证

DeepSeek-R1（2025 年 1 月）用纯强化学习（GRPO）训练，证明了推理时 Scaling 不仅是 OpenAI 的专利，开源社区同样可以实现。R1 的成功具有里程碑意义——它表明推理能力是大语言模型的一种「涌现能力」，可以通过适当的训练方法自然激发，而不需要昂贵的人工标注数据。

GRPO（Group Relative Policy Optimization）是 DeepSeek 提出的一种强化学习算法，其核心创新在于：不需要单独训练一个价值函数（critic），而是使用同一组采样结果的相对排名来估计优势函数。这种方法大幅降低了训练复杂度，同时保持了与 PPO 相当的性能。在 R1 的训练中，GRPO 的奖励信号非常简单——只需要判断最终答案是否正确，不需要标注中间推理步骤。

  • 推理能力可以通过 RL 自然涌现，不需要人工标注的思维链
  • 开源模型也能实现推理时 Scaling
  • 性能与 OpenAI o1 相当（在数学和编程任务上）
  • 训练成本远低于 o1（据估计约为 o1 的 1/10）
DeepSeek-R1 的训练流程： Base Model (DeepSeek-V3) │ ▼ GRPO 强化学习训练 │ ├── 自发学会 Chain-of-Thought ├── 自发学会自我验证 ├── 自发学会「等一下，让我重新想想」 │ ▼ R1 模型：推理时消耗更多 token → 更好结果 关键洞察: 推理策略是「涌现」的，而非人工设计的 GRPO 的奖励信号: 只需要答案正确性，不需要思维链标注

⚠️ R1 的局限性：虽然 R1 证明了推理能力可以通过 RL 涌现，但它也暴露了一些问题：(1) 推理过程的「冗余」——R1 的思维链中包含大量重复和自我确认的 token，效率远低于 o1；(2) 推理 token 的不可控——用户无法精确控制模型「思考多久」或「思考什么」；(3) 推理成本的不可预测——同一个问题，R1 可能用 1000 个 token 解决，也可能用 50000 个 token 解决，成本波动很大。这些问题目前是推理时 Scaling 研究的热点方向。

7. 推理时 Scaling 的幂律

研究表明，推理时计算与性能之间也存在幂律关系：

Performance ∝ (Inference Compute)^γ

其中 γ 取决于任务类型。γ 值的大小直接反映了推理时 Scaling 的「性价比」——γ 越大，每增加一单位推理计算带来的性能提升越多。

任务类型γ (近似)含义实用价值
数学推理~0.1-0.2推理计算翻倍 → 性能提升 7-14%非常高
代码生成~0.05-0.1推理计算翻倍 → 性能提升 3-7%高
逻辑推理~0.08-0.15推理计算翻倍 → 性能提升 5-10%高
常识问答~0.03-0.06推理计算翻倍 → 性能提升 2-4%中等
通用问答~0.02-0.05推理计算翻倍 → 性能提升 1-3%有限
创意写作~0.01-0.03推理计算翻倍 → 性能提升 0.7-2%较低

🔬 γ 值的差异来源：为什么数学推理的 γ 值（0.1-0.2）远高于通用问答（0.02-0.05）？原因在于任务的「可搜索性」。数学推理具有明确的正确答案和清晰的推理路径——搜索更多路径几乎总是有帮助的。而通用问答（如「你最喜欢的颜色是什么？」）没有唯一正确答案，搜索更多候选答案不会显著提升质量。这启示我们：推理时 Scaling 的价值与任务的「可验证性」成正比——越是能明确判断对错的任务，推理时 Scaling 的收益越大。

8. 训练时 vs 推理时 Scaling 的权衡

固定总计算预算（训练 + 推理）下的最优策略： 方案 A: 大模型 + 少推理 训练: ████████████████ (高) 推理: ██ (低) 总成本: ██████████████████ 方案 B: 小模型 + 多推理 训练: ████████ (中) 推理: ██████████ (中高) 总成本: ██████████████████ 同等总成本下，哪个效果更好？ 答案：取决于任务难度！ 简单任务 → 方案 A（大模型快思考就够） 困难任务 → 方案 B（小模型深思考更有效）

🔬 Snell et al. (2024) 的核心发现：对于困难问题，将算力从「训练更大模型」转移到「让小模型思考更久」可能更有效。具体来说，一个 14B 模型 + 充分推理，在某些数学任务上能超越 70B 模型的直接输出。这从根本上改变了「堆算力」的方向——从训练场转移到推理场。

这一发现的深远影响在于：它暗示了 AI 能力的提升不再只依赖于更大的训练集群和更多的训练数据。一个训练成本较低但推理策略精良的模型，可能在实际应用中超越一个训练昂贵但推理简单的模型。这对 AI 行业的成本结构和竞争格局具有重大意义——训练阶段的「算力军备竞赛」可能部分转移到推理阶段的「策略优化竞赛」。

⚠️ 推理时 Scaling 的经济性悖论：推理时 Scaling 引入了一个有趣的经济悖论：如果一个模型每次推理需要 50 倍的计算，但训练成本只需要 1/10，那么在什么使用量下总成本会打平？假设训练成本为 C_train，单次推理成本为 C_inf，使用次数为 Q。传统模型的总成本 = C_train + Q × C_inf；推理时 Scaling 模型的总成本 = C_train/10 + Q × 50×C_inf。当 Q > C_train / (49×C_inf × 10) 时，推理时 Scaling 模型的总成本更高。这意味着推理时 Scaling 模型更适合「低频高难」的使用场景，而不是「高频低难」的场景。

10. 推理时 Scaling 的实际应用场景

推理时 Scaling 并非适用于所有场景。它的价值与任务的难度、可验证性和容错率密切相关。以下是几个典型的应用场景分析：

应用场景是否适合推理时Scaling原因
数学竞赛/奥赛非常适合有明确正确答案，可验证，难度高
代码竞赛非常适合可通过测试用例验证，搜索空间大
科学证明适合逻辑严密，可逐步验证
日常对话不适合无唯一正确答案，延迟敏感
文本翻译不太适合质量提升有限，成本过高
客服问答不适合需要低延迟，问题通常简单
推理时 Scaling 的「投入产出」曲线： 性能 ↑ │ ····· 接近上限 │ ····· │ ····· │ ···· │ ··· │ ·· │ · │ · │ · │· └──────────────────────────────→ 推理计算量 关键观察: - 初期回报高（每单位计算带来显著提升） - 后期回报递减（接近任务的理论上限） - 存在「甜蜜点」——投入产出比最高的计算量 - 不同任务的曲线形状不同（γ 值不同）

💡 推理蒸馏的意义：推理蒸馏（Inference Distillation）是将推理时 Scaling 的成果「固化」到模型权重中的关键技术。具体做法是：先用推理时 Scaling 生成高质量的思维链数据，然后用这些数据对小模型进行微调。这样，小模型就学会了「像大模型一样思考」，在推理时不需要额外的计算就能获得接近大模型的性能。DeepSeek-R1-Distill 系列证明了这一方法的有效性：经过蒸馏的 7B 模型在数学推理上接近了未蒸馏的 70B 模型。

13. 代码示例：Best-of-N 采样

以下代码演示了推理时 Scaling 最简单的实现方式——Best-of-N 采样。通过生成 N 个候选答案并选择最佳答案，可以直观地看到推理时计算量与性能之间的关系。

import numpy as np

def best_of_n_sampling(model, question, n_samples=10, verifier=None):
    """
    Best-of-N 采样实现
    - model: 语言模型
    - question: 输入问题
    - n_samples: 采样数量（推理时计算量的代理指标）
    - verifier: 验证器函数，返回答案的得分
    """
    candidates = []
    for i in range(n_samples):
        # 生成一个候选答案（使用不同的随机种子）
        answer = model.generate(question, temperature=0.8, seed=i)
        # 用验证器评分
        if verifier:
            score = verifier(question, answer)
        else:
            score = answer.confidence  # 使用模型自身的置信度
        candidates.append((answer, score))
    
    # 选择得分最高的答案
    best_answer, best_score = max(candidates, key=lambda x: x[1])
    return best_answer, best_score

    log_acc = np.log10([r[1] for r in results])
    gamma = np.polyfit(log_n, log_acc, 1)[0]
    print(f"幂指数 γ ≈ {gamma:.3f}")
    return results, gamma

🔬 推理时 Scaling 与模型能力的耦合效应：一个有趣的研究问题是：推理时 Scaling 的效果是否依赖于模型的基础能力？换句话说，一个更强的基座模型是否能从推理时 Scaling 中获益更多？Snell et al. (2024) 的实验表明，答案是「是」——在相同的推理计算预算下，基座能力更强的模型能获得更大的性能提升。这意味着训练时 Scaling 和推理时 Scaling 不是简单的替代关系，而是互补关系：先训练一个足够强的基座模型，再通过推理时 Scaling 进一步挖掘其潜力，是目前最有效的策略。这一发现也解释了为什么 OpenAI 的策略是先投资训练 o1 的基座模型（GPT-4 级别），再在其上叠加推理时 Scaling。

11. 小结

✅ 本章要点：

  1. 第三条缩放轴：除了训练时的参数量 N 和数据量 D，推理时的计算量也是独立的缩放维度
  2. System 1 vs System 2：推理时 Scaling 让模型从「快思考」升级到「慢思考」，深度推理难题
  3. 幂律关系：推理计算与性能之间存在幂律关系，数学任务 γ≈0.1-0.2
  4. 训练 vs 推理权衡：困难任务上，小模型+充分推理 可能优于 大模型+少量推理
  5. 成本结构变革：从「按 token 计费」转向「按推理复杂度计费」的新商业模式

# 第三条缩放轴的发现 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
