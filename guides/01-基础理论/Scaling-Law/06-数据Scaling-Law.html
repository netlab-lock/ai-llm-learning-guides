📚 06 · 数据 Scaling Law

进阶 数据 实践导向

数据是大语言模型训练的三大要素之一（参数、数据、算力），但在很长一段时间里，数据维度被严重简化——人们只关心「多少 tokens」，而忽略了数据的质量、多样性、重复率、混合比例等关键因素。本模块将系统性地探讨数据维度的 Scaling Law，揭示为什么「数据比参数更重要」正在成为新的共识。

1. 为什么数据维度值得单独研究？

Kaplan 和 Chinchilla 的 Scaling Law 都将数据简化为一个数字——token 数量 D。但实际上，数据远比「多少」复杂：

  • 数据质量：同样 1T tokens，高质量网页 vs 低质量垃圾，效果天差地别。Microsoft 的 Phi 系列证明，1B 模型 + 教科书级数据可以超越 10B 模型 + 普通数据
  • 数据混合：代码、数学、多语言的比例如何影响能力？不同任务对数据混合比例的敏感度不同
  • 数据重复：用同一批数据反复训练，效果会怎样变化？什么时候重复变得有害？
  • 数据枯竭：高质量互联网数据用完了怎么办？合成数据能否替代真实数据？
  • 数据过滤：如何从海量原始数据中筛选出高质量子集？过滤策略如何影响最终性能？
数据维度的多层结构： ┌─────────────────────────────────────────────┐ │ 传统 Scaling Law 只看这一层： │ │ D = token 数量 │ │ L = f(N, D) │ └─────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────┐ │ 实际上需要考虑更多维度： │ │ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ │ │ 数据质量 q │ │ 数据混合 m │ │ 数据重复 r │ │ │ └───────────┘ └───────────┘ └───────────┘ │ │ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ │ │ 数据新鲜度 │ │ 数据多样性 │ │ 数据噪声 │ │ │ └───────────┘ └───────────┘ └───────────┘ │ │ │ │ L = f(N, D, q, m, r, ...) │ └─────────────────────────────────────────────┘ 传统: 2 个变量 → 简单幂律 现实: 6+ 个变量 → 复杂多维优化

Chinchilla 的 Scaling Law 只优化了训练成本。但 MosaicML 在 2024 年的研究中指出，实际部署中必须考虑推理成本：

小模型 + 超量数据 > 大模型 + Chinchilla最优数据

这就是为什么 LLaMA-3 的 8B 模型用了 15T tokens（1875 tokens/param），远超 Chinchilla 的 20:1 建议。虽然训练成本更高，但推理时 8B 模型比 70B 快 8 倍，长期来看更划算。

💡 实践指导：如果你的模型会被大量用户使用（高推理负载），应该选择更小的模型 + 更多训练数据。具体来说，tokens/param 比例应该根据预期推理量来调整：推理量越大，比例应该越高。一个粗略的经验法则是：对于面向消费者的产品，tokens/param 应该在 100-1000 之间，而不是 Chinchilla 建议的 20。

3. 数据质量 Scaling Law

2024 年的多项研究表明，数据质量的影响可能比数据数量更大。这一发现具有深远的实践意义——它意味着与其无脑收集更多数据，不如花精力清洗和筛选现有数据。

发现来源含义证据强度
高质量数据可以 10x 补偿数据量不足Phi 系列 (Microsoft)1B 模型 + 高质量数据 ≈ 10B 模型 + 普通数据强（多个模型验证）
数据去重提升 5-10% 性能多篇论文重复数据是隐性浪费强（广泛复现）
数据混合比例影响能力分布Data Mixing Laws (ICLR 2025)代码比例↑ → 推理能力↑，但语言能力↓中（特定任务）
困惑度过滤显著提升质量GPT-3, LLaMA过滤低质量网页可提升 3-5% 性能强（工业界标准）

💡 Phi 系列的启示：Microsoft 的 Phi-1 (1.3B) 用精心筛选的「教科书级」数据训练，在代码生成上超越了很多 10x 大的模型。Phi-2 (2.7B) 进一步证明了这一路线的可行性——在多个基准上超越了 LLaMA-2-7B。这证明了「数据质量 > 数据数量」的核心理念。Phi 系列的成功也启发了整个行业对数据质量的重新审视。

数据质量的量化方法

如何量化「数据质量」？这是一个至今未完全解决的难题。目前工业界使用的主要方法包括：

方法原理优点局限
困惑度过滤用小模型计算文本困惑度，过滤低质量文本简单高效，大规模可用只能识别「不自然」的文本，无法判断事实正确性
分类器打分训练质量分类器（如用 Wikipedia 为正例）可以学习复杂的质量模式依赖「什么是高质量」的定义——存在偏见
去重率MinHash/SimHash 检测近似重复去重后数据量减少，需要补充新数据去重粒度难以确定
信息密度单位 token 携带的信息量（互信息）理论最优雅计算成本极高，难以大规模应用
多维度评分综合考虑流畅度、信息量、多样性、安全性最全面实现复杂，各维度权重难以确定

🔬 深入探讨：数据质量的幂律关系

如果我们把数据质量 q 纳入损失函数，可以得到一个更一般的公式：

L(N, D, q) = A / N^α + B / (D × q)^β + E(q)

其中 q ∈ (0, 1] 表示数据质量（1 为完美质量）。这个公式暗示了一个重要结论：提升数据质量 q 的效果等价于增加数据量 D。如果 q 翻倍，等价于 D 翬倍——这意味着 2 倍质量的数据可以替代 2 倍数量的数据。

Phi 系列的经验表明，高质量数据的质量因子 q 可以比普通数据高 5-10 倍。这意味着 1T 高质量 tokens 的训练效果可能等价于 5-10T 普通 tokens。这一发现对数据策略有深远影响：与其大规模爬取互联网数据并粗暴训练，不如投入资源进行数据清洗和筛选。

关键发现：Microsoft 的 Phi 系列证明，1B 模型 + "教科书级"数据 ≈ 10B 模型 + 普通数据。这意味着数据质量的幂律指数可能比数据数量的指数大 2-3 倍——提升数据质量比增加数据量更"划算"。

4. 数据混合比例 Scaling

Data Mixing Laws (ICLR 2025) 发现，数据混合比例对模型能力有重要影响：

Loss = f(mixture_weights, N, D)

即损失不仅取决于 N 和 D，还取决于数据的混合比例。不同的混合比例会导致模型在不同能力维度上的表现差异。

⚠️ 数据混合的陷阱：增加某类数据的比例会提升该领域能力，但可能削弱其他能力。例如，将代码比例从 10% 提升到 30% 会显著提升推理能力，但可能降低自然语言生成的质量。这种「能力替代」效应在模块 05：涌现能力与相变中有更详细的讨论。在实际操作中，需要根据目标应用场景来平衡数据混合比例。

5. 数据枯竭问题

一个日益严峻的问题：高质量互联网数据正在被耗尽。随着各大公司对训练数据的争夺加剧，数据枯竭已经成为制约 AI 发展的关键瓶颈之一。

全球高质量文本数据估算（2025 年）： 英文网页: ~5-10T tokens ████████████████████████ 代码 (GitHub): ~5-10T tokens ████████████████████████ 书籍/学术: ~1T tokens ████████ 其他语言: ~5T tokens ████████████████████████ 期刊论文: ~0.5T tokens ████ ───────────────────────────────────────────── 总计: ~15-25T tokens LLaMA-3 已用: 15T tokens ← 几乎用尽了公开英文数据！ 数据枯竭时间线： 2023: 数据充足，各公司自由使用 2024: 开始出现数据争夺，版权诉讼增加 2025: 高质量英文数据接近耗尽 2026: 大多数公开数据已被使用 2027+: 必须依赖合成数据和多模态数据

⚠️ 数据墙（Data Wall）：按照当前的 Scaling 速度，高质量文本数据可能在 2026-2028 年被耗尽。这意味着未来的 Scaling 将面临「数据瓶颈」，而不仅仅是算力瓶颈。解决方案包括：合成数据、多模态数据、数据增强、以及更高效的数据利用方法。数据枯竭问题也在模块 10：前沿与展望中有讨论。

6. 代码数据的特殊作用

一个有趣的发现：在训练数据中加入代码能显著提升模型的推理能力（不仅仅是编程能力）。这一发现最初是经验性的，但后来的研究给出了理论解释。

实验结果解释来源
纯文本 vs 文本+代码+代码的模型推理更强代码是结构化思维的训练多篇论文
代码比例 10% vs 25%25% 的推理更好更多结构化训练信号LLaMA-3
数学+代码 vs 纯文本前者的数学更强符号推理能力迁移Phi 系列
代码预训练 → 文本微调优于纯文本预训练代码提供了更好的「思维基底」多篇论文

💡 代码数据的独特价值：代码之所以能提升推理能力，是因为代码本质上是「结构化的逻辑推理」。每一行代码都要求精确的逻辑、明确的因果关系、严格的类型约束。这些特性恰好是自然语言所缺乏的。当模型在大量代码上训练后，它学会了「如何进行结构化思考」，这种能力可以迁移到自然语言推理任务上。这就是为什么几乎所有现代大模型（GPT-4、Claude、Gemini、LLaMA）都在训练数据中包含了大量代码。

7. 数据重复与退化

当数据不足时，不得不重复使用同一批数据。研究表明数据重复对模型性能有显著的负面影响：

  • 少量重复（2-3 次）：影响很小，几乎可以忽略
  • 中度重复（4-10 次）：性能开始下降，但可以通过正则化（如 dropout）部分缓解
  • 大量重复（>10 次）：严重过拟合，性能急剧下降，模型开始「记住」数据而不是「理解」数据

数据重复的数学模型

我们可以用一个简单的模型来估算重复数据的有效性。每次重复的有效性会衰减，因此重复数据的有效 token 数远小于实际 token 数：

# 数据重复的影响估算（简化模型）
def effective_tokens(D_unique, D_total, decay=0.9):
    """估算重复数据的有效 token 数
    
    Args:
        D_unique: 唯一 token 数
        D_total: 总 token 数（包含重复）
        decay: 每次重复的衰减因子
    Returns:
        有效 token 数
    """
    repeats = D_total / D_unique
    if repeats <= 1:
        return D_total
    # 每次重复的有效性衰减
    effective = D_unique * (1 + sum(decay**i for i in range(1, int(repeats))))
    return effective

# 例：1T unique tokens 重复 10 次
eff = effective_tokens(1e12, 1e13, decay=0.9)
print(f"有效 tokens: {eff/1e12:.2f}T")  # 约 6.5T，远小于 10T

# 不同衰减因子的影响
for d in [0.5, 0.7, 0.9, 0.95]:
    eff = effective_tokens(1e12, 1e13, decay=d)
    print(f"decay={d}: 有效 tokens = {eff/1e12:.2f}T")

⚠️ 数据重复的实际影响：一项关键研究发现，当数据重复 4 次以上时，模型的有效参数效率开始显著下降——这意味着重复数据训练出来的模型，性能不如用等量唯一数据训练的模型。更严重的是，过度重复会导致模型在特定 token 模式上过拟合，降低其泛化能力。在数据枯竭的背景下，这是一个必须认真对待的问题。

8. 合成数据的 Scaling

合成数据（用 AI 生成训练数据）正在成为解决数据枯竭问题的关键方向。合成数据不仅能量大，还可以精确控制质量、领域分布、难度等属性。

方法案例效果风险
用大模型生成教科书Phi-1 (Microsoft)1.3B 模型超越 10B+生成质量依赖教师模型
用强模型蒸馏弱模型多篇论文小模型获得大模型的能力可能损失多样性
用代码执行验证代码+数学训练自动过滤错误数据只能用于可验证领域
自我对弈生成AlphaProof (DeepMind)数学奥赛金牌需要可靠的验证器
多模型协作生成Self-Play 方法多样性和质量兼顾计算成本高

🔬 深入探讨：模型坍缩问题

合成数据面临的核心风险是「模型坍缩」（Model Collapse）：当模型在自己生成的数据上训练时，会逐渐失去多样性，最终收敛到一个退化的分布。这个问题可以用信息论来理解：

1. 真实数据分布包含丰富多样的模式，信息熵较高

2. 模型生成的数据只能捕捉到真实分布的一部分，信息熵较低

3. 在生成数据上训练的模型，学到的分布信息熵更低

4. 多代迭代后，信息熵趋向于零——模型坍缩

解决方案：

  • 始终保留一部分真实数据（至少 10-20%）
  • 使用验证器（如代码执行、数学证明检查器）过滤错误的合成数据
  • 控制合成数据的多样性——不要用同一个模型生成所有数据
  • 定期用新的真实数据「刷新」训练集

9. 数据过滤的 Scaling 效应

数据过滤的质量直接影响最终模型的性能。不同的过滤策略会产生截然不同的结果：

过滤策略过滤后数据量性能影响适用场景
无过滤100%基线数据充足且质量高
语言过滤~60%+2-3%多语言数据集
质量过滤（困惑度）~40%+5-8%网页数据
去重~70%+3-5%所有数据集
综合过滤~20%+10-15%追求极致性能

一个关键观察是：激进的过滤（保留 20% 的数据）可能比保留所有数据更好。这暗示了一个反直觉的结论：减少数据量可能反而提升性能——前提是去掉的是低质量数据。这一发现与模块 04：Chinchilla的结论并不矛盾，因为 Chinchilla 的 D 指的是「有效数据量」而非「原始数据量」。

10. 多模态数据的 Scaling

随着多模态模型（视觉-语言、音频-语言）的兴起，数据 Scaling 的维度进一步扩展：

多模态数据与纯文本数据有本质区别。文本是离散的符号序列，而图像和视频是连续的高维信号。这意味着多模态数据的信息密度远高于纯文本——一张图片包含的信息可能相当于数百甚至数千个 token。然而，信息密度高并不意味着学习效率也高：模型需要从高维信号中提取有意义的特征，这个过程需要更多的计算和更多的数据。

另一个关键差异是模态间的对齐问题。文本和图像之间的对应关系远没有文本内部的语义关系那么明确。一幅图可能有无数种合理的文字描述，而一段文字也可能对应无数种合理的图像。这种一对多的关系增加了模型学习的难度，也增加了对高质量配对数据的需求。

  • 图文配对数据：LAION-5B 包含 50 亿个图文对，但质量参差不齐。如何筛选高质量配对？
  • 视频数据：视频包含时间维度的信息，信息密度远高于静态图片。但视频数据的标注成本极高
  • 音频数据：语音、音乐、环境声音的数据 Scaling 行为尚不清楚
  • 跨模态对齐：不同模态的数据量如何平衡？视觉编码器的参数量如何影响多模态性能？

💡 多模态数据策略：目前的共识是，多模态模型需要的不是简单的「更多数据」，而是「更好的跨模态对齐数据」。图文配对的质量远比数量重要——1000 万高质量图文对可能比 10 亿低质量配对更有效。这一规律与纯文本领域的「质量 > 数量」趋势一致。

11. 代码复现：数据质量影响估算

import numpy as np

# 数据质量对损失的影响模型
def loss_with_quality(N, D, q, A=406.4, alpha=0.34, B=410.7, beta=0.28, E=1.69):
    """带质量因子的损失函数
    N: 参数量, D: 数据量, q: 质量因子 (0,1]
    """
    return A / N**alpha + B / (D * q)**beta + E

# 比较不同质量下的损失
N = 7e9  # 7B 参数
D = 1e12  # 1T tokens

print("7B 模型，1T tokens，不同数据质量的损失:")
for q in [0.1, 0.3, 0.5, 0.7, 0.9, 1.0]:
    L = loss_with_quality(N, D, q)
    print(f"  q={q:.1f}: Loss = {L:.3f}")

# 等价关系：多少低质量数据 = 1T 高质量数据？
q_low = 0.3
q_high = 1.0
# D_low * q_low = D_high * q_high
# D_low = D_high * q_high / q_low
D_high = 1e12
D_low_needed = D_high * q_high / q_low
print(f"\n要达到 1T 高质量数据的效果，需要 {D_low_needed/1e12:.1f}T 低质量数据")

12. 学习路径与进阶阅读

数据 Scaling 的历史演变

数据在 AI 训练中的地位经历了三次重大转变。第一次是 2012 年 ImageNet 竞赛，证明了大规模标注数据的价值。第二次是 2018 年 BERT 的出现，展示了无监督预训练数据的强大潜力。第三次是 2022 年 Chinchilla 论文，将数据量提升到了与参数量同等重要的地位。每一次转变都深刻改变了研究者对数据的看法和资源配置策略。

在 2024-2025 年，我们正在经历第四次转变：从「数据量」到「数据质量」的认知升级。这一转变的标志是 Phi 系列的成功和数据枯竭问题的加剧。研究者开始意识到，盲目收集更多数据不如精心筛选和清洗现有数据。这一认知转变可能会催生一系列新的数据处理技术和工具。

# 为什么数据维度值得单独研究？ 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
