ML CI/CD的特点

ML CI/CD的特点

>通俗类比：传统CI/CD像标准化快餐：配方（代码）固定，出餐快。ML CI/CD像定制大餐：食材（数据）每次可能不同，火候（训练）需要实时调整，口味（模型）要反复品尝（评估）。
常见误区：最常见的误区是套用软件CI/CD的思路，只关注代码质量，而忽略了数据质量监控、模型性能评估和可复现性检查，导致部署的模型“悄悄”变差。
深入探讨：核心区别在于‘三件套’：数据版本控制（如DVC）、模型注册表和自动化评估。评估指标需自定义，如F1、AUC、延迟。数据验证规则（如Great Expectations）应像代码测试一样被纳入流水线，通常设置准确率阈值>95%的数据才能流入下一流程。
代码提交 -> 数据验证 -> 模型训练 -> 性能评估 -> 注册 -> 部署
    |            |            |            |         ^
    v            v            v            v         |
  触发器    数据检查器    训练器     评估器    注册表
特征传统 CI/CDML CI/CD
触发事件代码提交代码提交、数据更新、定期重训练
核心制品应用二进制包、容器镜像训练好的模型文件、数据集版本、评估报告
“成功”标准编译通过、测试通过测试通过 **且** 模型性能不退化、数据质量达标
关键组件构建服务器、制品库实验追踪工具、模型注册表、数据版本库
方面要点注意事项
核心概念ML CI/CD的特点的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化
常见问题踩坑记录和解决方案参考社区issue和讨论

传统CI/CD关注代码测试和部署，ML CI/CD还需要处理数据验证、模型评估和训练流水线。关键是将"数据+代码+模型"三者的变更纳入统一的自动化流程。

💡 CI/CD在ML中的含义

CI（持续集成）= 代码/数据变更时自动运行测试 + 验证；CD（持续部署）= 通过评估的模型自动部署到生产环境。这在LLM场景中通常被称为CT（Continuous Training）。

训练流水线

训练流水线

通俗类比：并行策略就像分工做菜。张量并行（TP）= 一道菜由多个厨师同时切不同的食材；流水线并行（PP）= 不同厨师负责不同工序（洗菜→切菜→炒菜），形成流水线；数据并行（DP）= 多个厨师各自独立做相同的菜。
常见误区：不要盲目追求最大并行度。并行会引入通信开销（厨师之间传递食材的时间）。需要平衡计算与通信比例，对于小模型或慢网络，并行可能反而降低效率。
深入探讨：常用策略组合：DP+TP+PP（3D并行）。以GPT-3 175B为例，典型配置为：128块GPU，使用8路TP（在同一高速互联的节点内），16路DP（跨节点），1路PP。气泡率（Bubble Ratio）是衡量流水线效率的关键，理想流水线并行气泡率应趋近于 (p-1)/(m+p-1)，其中p是阶段数，m是微批次数。
张量并行(TP): GPU1, GPU2, GPU3 同时处理矩阵分块
             [矩阵A] -> [分块A1 on GPU1] + [分块A2 on GPU2] + [分块A3 on GPU3]
                         |___________________________________|
                                      同步聚合

流水线并行(PP): GPU1(前层) -> GPU2(中层) -> GPU3(后层)
             Batch1: [P1->P2->P3]   Batch2: [P1->P2->P3]   Batch3: ...
             (微批次在“流水线”中交替前进)
策略原理适用场景核心开销
数据并行 (DP)数据分割，模型复制，梯度同步模型可放入单卡显存All-Reduce 通信
张量并行 (TP)层内并行，分割矩阵计算层内计算密集，节点内高带宽互联All-Reduce (行/列切分)
流水线并行 (PP)层间并行，流水线执行模型深度大，跨节点通信微批次间的激活传输、流水线气泡
策略通信量适用互联典型场景
张量并行(TP)高NVLink节点内多卡
流水线并行(PP)低PCIe/IB跨节点
数据并行(DP)中任意多数据源
专家并行(EP)高NVLinkMoE模型

一个完整的LLM训练流水线通常包含以下阶段：

阶段任务输出

数据预处理清洗、分词、格式转换训练数据集

# pipeline.yaml - 训练流水线定义
stages:
  - name: validate_data
    script: scripts/validate_data.py
    inputs:
      - data/raw/train.jsonl
    checks:
      - min_samples: 10000
      - schema_valid: true

  - name: preprocess
    script: scripts/preprocess.py
    inputs:
      - data/raw/train.jsonl
    outputs:
      - data/processed/train.bin

  - name: train
    script: scripts/train.py
    resources: { gpu: 8, nodes: 4 }
    inputs:
      - data/processed/train.bin
    outputs:
      - models/checkpoint-final/

  - name: evaluate
    script: scripts/evaluate.py
    gates:
      - metric: "mmlu_score > 0.65"
      - metric: "perplexity < 15"

GitHub Actions for ML

GitHub Actions for ML

通俗类比：它就像一个自动化的快递流程：你打包代码（提交），仓库（物流中心）自动触发流水线：检查包裹质量（测试）-> 工厂加工（训练模型）-> 质检（评估）-> 送达用户（部署）。整个过程无需你手动搬运。
常见误区：注意权限和资源限制。Runner的免费额度有限，大型模型训练会很快耗尽。务必使用 `actions/cache` 缓存大型依赖（如Hugging Face模型），避免每次运行都从头下载，浪费时间和资源。
深入探讨：一个典型的ML工作流YAML关键步骤：`jobs.train` 中，使用 `docker build` 构建包含训练脚本的环境；通过 `python train.py --data-path ${{ inputs.data-path }}` 训练；利用 `actions/upload-artifact` 保存模型文件和评估报告；`needs: [train]` 确保部署作业在训练评估通过后运行。

# .github/workflows/ml-pipeline.yml
name: ML Pipeline
on:
  push:
    paths: ['src/**', 'data/**', 'configs/**']
  workflow_dispatch:
    inputs:
      train_config:
        description: '训练配置文件路径'
        default: 'configs/default.yaml'

jobs:
  data-validation:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate dataset
        run: python scripts/validate_data.py --data data/train.jsonl

  model-training:
    needs: data-validation
    runs-on: [self-hosted, gpu]
    steps:
      - uses: actions/checkout@v4
      - name: Train model
        run: |
          srun --gpus=8 python train.py \
            --config ${{ github.event.inputs.train_config }}

  model-evaluation:
    needs: model-training
    runs-on: [self-hosted, gpu]
    steps:
      - name: Run benchmarks
        run: python evaluate.py --model models/latest/ --bench mmlu,hellaswag
⚠️ GPU Runner注意

ML训练需要self-hosted GPU Runner。确保Runner配置了NVIDIA驱动、CUDA环境和足够的磁盘空间。建议使用标签（如gpu、a100）区分不同类型的Runner。

Kedro与ZenML<与ZenML的工作原理，有助于在实际工程中做出更合理的技术选择和优化决策。

Kedro与ZenML

通俗类比：Kedro像一个功能全面的“万能料理机”，提供标准化的项目结构和节点化流程，让你专注于“菜谱”逻辑。ZenML则像更现代的“模块化集成灶”，天生为MLOps设计，方便插入各种云服务和实验跟踪“配件”。
常见误区：选择框架时，不要被“特性列表”迷惑。Kedro的学习曲线更平缓，适合数据处理和模型原型开发；ZenML的MLOps集成更深，适合团队协作和生产部署。关键是评估团队当前阶段的核心需求。
深入探讨：Kedro的核心是`pipeline`和`node`，通过`catalog.yml`管理数据IO，自带缓存机制。ZenML的核心是`step`和`pipeline`，通过`Stack`管理基础设施（跟踪器、实验器、部署器）。例如，在ZenML中切换实验跟踪器：zenml experiment-tracker register mlflow_tracker --flavor=mlflow，然后将该组件加入当前Stack。
Kedro 项目结构:               ZenML 核心概念:
my_project/                  Stack (基础设施集合)
├── conf/   (配置)            ├── Orchestrator (编排器，如Local, Kubeflow)
├── data/   (数据IO)          ├── Experiment Tracker (实验跟踪，如MLflow)
├── src/    (管道代码)        ├── Model Deployer (模型部署，如Seldon, Sagemaker)
    ├── pipelines/            └── Artifact Store (制品存储，如S3, GCS)
    └── nodes/ (节点，处理单元)

# ZenML流水线示例
from zenml import pipeline, step

@step
def data_loader() -> Dataset:
    return load_training_data()

@step
def trainer(data: Dataset) -> Model:
    model = train_llm(data, config)
    return model

@step
def evaluator(model: Model) -> float:
    score = run_benchmarks(model)
    return score

@pipeline
def llm_training_pipeline():
    data = data_loader()
    model = trainer(data)
    score = evaluator(model)

llm_training_pipeline.run()

数据与模型验证

数据与模型验证

通俗类比：这就像飞机起飞前的安全检查。数据验证是核对乘客名单（特征是否齐全、格式正确）、检查行李（数据分布是否异常）。模型验证是测试飞机的各个系统（模型在各项指标上的表现）和模拟飞行（在真实场景中的表现）。
常见误区：不要只验证模型在“测试集”上的准确率。这存在巨大漏洞：无法发现数据泄露（测试集信息已“污染”训练集），也无法发现模型在特定子群体（如少数族裔）上的性能偏差（公平性问题）。
深入探讨：验证应包含三个层次：1) **数据验证**：使用`pandas-profiling`或`Great Expectations`设置断言，如“用户年龄列不应为空且范围在[0,120]”。2) **模型性能验证**：不仅看整体准确率（95%），还要分析混淆矩阵（哪类错误多）、FPR（误报率）。3) **业务指标验证**：将模型输出映射到业务指标，如“信用评分模型A的F1提升，对应坏账率预期下降0.5%”。

✅ 验证检查清单
  • 数据完整性：样本数、字段完整性、编码格式
  • 数据分布：类别平衡、文本长度分布、异常值检测
  • 模型基准：MMLU、HellaSwag、TruthfulQA等标准基准
  • 回归检测：与上一版本对比，确保无性能退化
  • 安全评估：有害内容生成率、偏见检测

✅ 参考答案与详解
  1. 核心原理: ML CI/CD与传统CI/CD的核心区别在于"三件套"：数据版本控制（DVC）、模型注册表和自动化评估。传统CI/CD触发事件仅代码提交，ML CI/CD还需处理数据更新和定期重训练（CT，Continuous Training）。训练流水线通常包含：数据验证→数据预处理→模型训练→模型评估→模型注册→模型部署六阶段，每个阶段有明确的质量门禁。分布式训练策略包括数据并行（DP，模型复制+梯度同步）、张量并行（TP，层内矩阵分块）和流水线并行（PP，层间流水线执行），3D并行组合DP+TP+PP是大规模训练标准方案。数据验证应使用Great Expectations设置断言（如"用户年龄列不为空且范围[0,120]"），模型验证需分析混淆矩阵、FPR和子群体公平性。
    关键要点: ①GitHub Actions ML工作流需self-hosted GPU Runner，使用actions/cache缓存Hugging Face模型避免重复下载；②Kedro通过pipeline/node+catalog.yml管理数据IO，ZenML通过step/pipeline+Stack管理基础设施（跟踪器、部署器、编排器）；③质量门禁示例：mmlu_score>0.65且perplexity<15才允许进入下一阶段；④验证三层：数据验证（Schema/统计分布）、模型性能验证（准确率/混淆矩阵）、公平性验证（demographic parity/equal opportunity）；⑤流水线气泡率=(p-1)/(m+p-1)，p是阶段数，m是微批次数
    实际案例: 使用ZenML构建完整流水线：@step定义data_loader/trainer/evaluator三个步骤，@pipeline定义llm_training_pipeline，配置MLflow作为Experiment Tracker、S3作为Artifact Store。在GitHub Actions中设置on: [push, schedule]触发器，data-validation Job使用Great Expectations验证数据，model-training Job通过srun --gpus=8训练，model-evaluation Job运行MMLU/HellaSwag基准测试。
    关联知识: 与实验追踪（第5章）的指标记录、模型版本管理（第6章）的Registry注册、灰度发布（第8章）的部署策略形成完整的MLOps闭环。
向前串联： ML CI/CD的特点的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
学习路径建议： 学习ML CI/CD的特点建议按"概念→原理→实验→调优"四步走。先理解核心概念和术语， 再深入数学原理和工作机制，然后用最小化实验验证理解，最后在真实场景中调优参数。 每步都应记录关键指标（延迟/吞吐/精度），建立数据驱动的优化习惯。
