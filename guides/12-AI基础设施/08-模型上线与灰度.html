部署策略概述

部署策略概述

"tip">通俗类比：部署策略就像点餐方式。蓝绿是“打包两份饭，挑一份吃”，金丝雀是“先尝一小口，再决定要不要整份”，A/B测试则是“盲测，看哪份更受欢迎”。
常见误区：最常见的误区是只关注部署动作本身，忽略了配套的监控、告警和快速回滚机制。没有回滚计划的部署，如同不带降落伞跳伞。
深入探讨：核心指标是切换速度与影响范围。蓝绿需秒级切换DNS，金丝雀可控制1%-10%流量观察数分钟至数小时。回滚目标通常是分钟级内完成，将用户影响降至最低。
策略选择树：
用户请求 -> 评估需求
          |
          +--> 零停机？--> 蓝绿
          +--> 小范围验证？--> 金丝雀
          +--> 效果对比？--> A/B
策略核心目标资源开销适用阶段
直接替换简单快速低非核心、低频服务
蓝绿零停机，快速回滚高（需2倍环境）核心服务，要求高可用
金丝雀控制风险，逐步验证中新功能或重大更新上线
A/B数据驱动决策中高（需分流与统计）效果优化，长期迭代
框架核心特性适用场景学习曲线
vLLMPagedAttention通用推理服务★★
TensorRT-LLMNVIDIA深度优化NVIDIA GPU★★★★
TGIHF生态快速部署★★
SGLangRadixAttention复杂推理★★★
Ollama一键运行本地开发★

LLM模型上线不同于传统软件部署：模型文件巨大、GPU资源昂贵、推理延迟敏感。需要谨慎选择部署策略，在降低风险和控制成本之间找到平衡。

策略风险资源开销回滚速度适用场景

蓝绿部署中高(2倍)极快关键服务

A/B测试低中快效果评估

蓝绿部署

蓝绿部署

通俗类比：想象你有两条完全相同的高铁轨道（蓝轨和绿轨）。平时用蓝轨运行（生产），新版本铺设在绿轨上。测试无误后，瞬间将所有列车调度到绿轨。若出问题，立刻调回蓝轨。
常见误区：最大的陷阱是“环境不完全镜像”。数据库Schema变更、配置文件差异、或依赖服务的网络策略不一致，都可能导致绿环境上线后瞬间崩溃。
深入探讨：工程上，蓝绿部署要求环境100%对等。切换通常依赖负载均衡器或DNS。DNS切换有TTL缓存延迟（秒到分钟级），负载均衡器切换更快（秒级）。资源成本是长期的，需要为闲置环境付费。
用户 -> [负载均衡器] -> [蓝环境 (v1.0)] (当前生产)
              |        ^
              +--------+
              |        | (回滚)
              +->[绿环境 (v1.1)] (待命/验证中)
步骤操作关键检查点
1. 部署在绿环境部署v1.1部署成功，健康检查通过
2. 验证内部流量或自动化测试核心功能、性能指标无异常
3. 切换修改负载均衡指向流量平滑转移，监控无报警
4. 观察/回滚监控线上指标，或一键切回蓝错误率、延迟在阈值内

蓝绿部署维护两个完全相同的环境（蓝色=当前版本，绿色=新版本），通过负载均衡器瞬间切换流量：

# 蓝绿部署配置示例
# blue-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: llm-service-blue
spec:
  replicas: 4
  template:
    spec:
      containers:
      - name: model
        image: registry/llm:v2.1  # 当前版本

# green-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: llm-service-green
spec:
  replicas: 4
  template:
    spec:
      containers:
      - name: model
        image: registry/llm:v2.2  # 新版本

# 流量切换（修改Service selector）
kubectl patch svc llm-service -p \
  '{"spec":{"selector":{"version":"green"}}}'
⚠️ 蓝绿部署的代价

蓝绿部署需要2倍的GPU资源（两个版本同时运行）。对于大模型来说，这意味着额外的硬件成本。建议只在关键模型版本升级时使用。

金丝雀发布

通俗类比：就像新厨师上台前，先让他在角落给几位老顾客做菜。反馈好（监控正常），再让他服务更多桌子。反馈差（指标异常），立刻让他停手，影响范围很小。
常见误区：易犯错误是“观察期太短”或“流量比例跳升太快”。1%的流量可能无法暴露问题，但直接从1%跳到50%，风险就不可控了。需设置明确的梯度放量和观察时长。
深入探讨：典型金丝雀流程：1% -> 5% -> 20% -> 50% -> 100%。每级观察10-30分钟。核心监控指标：错误率（如HTTP 5xx）、延迟P99/P999、业务成功率（如下单成功率）。任何指标异常即触发回滚。
总流量 (100%)
   |
   +---> [路由规则] --(99%流量)--> [旧模型池 (v1)]
   |
   +---> [路由规则] --(1%流量) ---> [新模型池 (v2)] <-- 监控重点
阶段新版本流量观察重点动作决策
1. 冒烟1%服务是否启动，核心API可访问失败则立即停止，成功则进入下一阶段
2. 小流量验证5-10%错误率、延迟基线指标劣化超阈值则回滚
3. 梯度放量20%-50%指标稳定性、有无内存泄漏等平稳则继续放量
4. 全量发布100%全量负载下的表现完成发布，旧版本下线
方面要点注意事项
核心概念金丝雀发布的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化
常见问题踩坑记录和解决方案参考社区issue和讨论

金丝雀发布将新版本先暴露给一小部分流量，逐步扩大比例：

# 使用Istio实现金丝雀流量分割
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: llm-service
spec:
  hosts:
  - llm-service
  http:
  - route:
    - destination:
        host: llm-service
        subset: stable
      weight: 90       # 90%流量到稳定版
    - destination:
        host: llm-service
        subset: canary
      weight: 10       # 10%流量到金丝雀

# 逐步扩大比例: 10% → 30% → 50% → 100%
# 观察指标: 错误率、延迟P99、用户满意度
💡 金丝雀观察指标

关键监控指标：推理延迟P99 < 目标阈值、错误率 < 0.1%、输出质量评分无退化。每个阶段至少观察30分钟至数小时。

A/B测试

A/B测试

通俗类比：像超市里卖酸奶，把两种新口味（A和B）摆在一起，让顾客自选。统计哪种口味的复购率更高，数据说话，而不是经理个人喜好决定哪种口味全面上架。
常见误区：最大的坑是“流量分配不均”和“没有达到统计显著性”。不能随机让不同地区或时段的用户测试不同版本，这会导致混杂因素。必须确保用户分组随机，且样本量足够大（通常每组需数千次交互）。
深入探讨：统计显著性通常用p值衡量，p<0.05被认为结果可信。计算所需样本量的公式涉及预期提升幅度、基线转化率和统计功效。例如，将转化率从10%提升到11%（10%相对提升），在95%置信度下，每组需要约14,000次曝光。
原始流量
   |
   +-> [分流器] -> (50%用户) -> [模型A] -> 记录结果
   |                         |
   +-> [分流器] -> (50%用户) -> [模型B] -> 记录结果 -> [分析平台]
要素说明示例
实验单元被随机分配的对象用户ID、会话ID
指标衡量效果的首要标准点击率、停留时长、业务收入
分流将用户随机分配到不同组基于哈希的均匀分流
显著性结果非偶然的统计置信度p < 0.05 (95%置信)
方面要点注意事项
核心概念A/B测试的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化
常见问题踩坑记录和解决方案参考社区issue和讨论

模型服务工具

模型服务工具

通俗类比：模型服务工具就像一个瑞士军刀套装。vLLM是那把主刀（高效切水果），Triton是多功能螺丝刀（适配各种模型），TensorRT-LLM是精密镊子（追求极致精细速度）。
常见误区：常见误区是“盲目追求最先进框架”而忽略团队技术栈和模型特性。用PyTorch写的模型，突然换成TensorRT-LLM，可能面临复杂的图转换和调试。适合的才是最好的。
深入探讨：vLLM核心创新是PagedAttention，将KV Cache分页管理，显存利用率提升2-4倍，吞吐量提升数倍。TensorRT-LLM通过算子融合、量化（FP8/INT8）、KV Cache优化，在NVIDIA GPU上可实现比原生PyTorch快3-10倍的推理速度。
应用层
   |
   v
[API网关/负载均衡]
   |
   v
[模型服务工具] <-- 核心：高效推理
   |       (vLLM/Triton/TGI)
   |
   v
[GPU硬件驱动]
工具核心优势适用场景典型吞吐提升
vLLM高吞吐，易用，支持连续批处理通用文本生成，对延迟要求不是极端苛刻相比HF原生，提升2-24倍
TensorRT-LLM极致低延迟，NVIDIA深度优化对延迟敏感的实时交互场景相比PyTorch，提升3-10倍
TGI (Text Generation Inference)功能全面，开箱即用，支持多种模型快速部署Hugging Face模型生态相比原生，显著提升

核心概念模型服务工具的基本原理理解底层机制比记忆参数更重要

✅ 影子模式

影子模式（Shadow Mode）让新模型在后台处理真实流量但不返回结果给用户，用于对比新旧模型的实际表现。这是最安全的模型验证方式，适合重大版本升级前的最后验证。
