K8s在AI场景中的角色

K8s在AI场景中的角色

通俗类比：K8s在AI场景中就像是物流调度中心，不再是简单的应用托管。它负责统筹计算、存储、网络等资源，将GPU、CPU、内存等打包，智能地分配给不同的AI训练或推理任务。
常见误区：最常见的错误是试图直接用原生K8s管理AI任务。AI任务（如模型训练）生命周期、资源需求与Web应用差异巨大，缺乏作业队列、GPU感知调度等能力，会导致效率低下甚至失败。
深入探讨：在AI基础设施中，K8s扮演的是编排层。它通过其声明式API管理Pod、Service等资源对象。对于AI，它需要与GPU设备插件（Device Plugin）、集群调度器（如Volcano）以及分布式计算框架（如Ray、PyTorch Operator）深度集成，构成完整的AI作业生命周期管理平台。

Kubernetes已成为云原生AI工作负载的标准编排平台。相比Slurm，K8s提供了更好的容器支持、服务发现和弹性伸缩能力。但对于大规模训练任务，原生K8s调度器存在不足，需要扩展调度器。

💡 K8s vs Slurm

K8s适合混合工作负载（训练+推理+Web服务），Slurm适合纯HPC训练场景。两者可以共存：K8s管理推理服务，Slurm管理训练任务。

GPU设备插件

GPU设备插件

通俗类比：GPU设备插件就像K8s世界里的“硬件翻译官”。它向K8s集群报告“我这里有一块NVIDIA A100，显存80GB”，并让K8s能够像管理内存一样，将一块GPU作为可分配的资源（如`nvidia.com/gpu: 1`）分配给容器。
常见误区：易犯错误是忘记在每个GPU节点上部署设备插件（如`nvidia-device-plugin-daemonset`），或者插件版本与GPU驱动/CUDA版本不兼容，导致节点上虽然有GPU，但K8s无法识别和调度。
深入探讨：设备插件通过K8s的`DevicePlugin` gRPC接口注册设备。当Pod请求`nvidia.com/gpu: 2`时，插件会确保该Pod能访问宿主机的2块物理GPU卡。它通常挂载`/dev/nvidia*`设备文件和必要的库。现代插件还支持MIG（多实例GPU）模式，将一块A100划分为多个独立的GPU实例。

NVIDIA GPU Device Plugin是K8s管理GPU资源的核心组件，它通过K8s设备插件框架将GPU暴露为可调度资源：

# 安装NVIDIA Device Plugin
kubectl apply -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/v0.14.0/nvidia-device-plugin.yml

# Pod请求GPU资源
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  containers:
  - name: training
    image: nvidia/cuda:12.0-runtime
    resources:
      limits:
        nvidia.com/gpu: 4  # 请求4张GPU

调度策略

合理的技术选择和优化决策。

调度策略

通俗类比：AI调度策略就像智能交通调度员。它不仅要“把车（Pod）停到空车位（节点）”，还要考虑“目的地”（任务队列）、“车辆类型”（GPU型号）和“乘客优先级”（作业优先级），甚至安排“车队”（分布式作业）同时出发。
常见误区：盲目使用K8s默认调度器调度GPU任务，容易导致资源碎片化。例如，两个任务各需1张GPU，节点上有2张，但默认调度器可能将它们调度到不同节点，留下一个节点有1张闲置GPU，无法调度需要2张GPU的大任务。
深入探讨：高级调度策略包括：1. **Bin-Packing**：尽量将任务打包到少数节点，提高利用率。2. **Gang Scheduling**：确保分布式作业的所有Pod同时启动，避免资源死锁。3. **抢占（Preemption）**：高优先级任务可抢占低优先级任务的GPU资源。4. **拓扑感知**：考虑GPU间的NVLink连接，将紧密通信的任务Pod调度在同节点同拓扑下。
作业队列：[A(需2GPU), B(需1GPU), C(需1GPU)]
       ↓ 调度器决策
策略1(默认): A→Node1, B→Node2, C→Node1 (资源分散)
策略2(Bin-Packing): B→Node1, C→Node1, A→Node2 (打包)
调度策略目标适用场景
默认/Random基础调度，简单小规模、异构性不强的集群
Bin-Packing提高单节点资源利用率降低成本，支持大规模大任务
Gang Scheduling作业完整性，避免死锁大规模分布式训练（如GPU集群All-Reduce）
批处理策略GPU利用率延迟代表框架
Static Batching低(~30%)高早期框架
Continuous Batching高(~80%)低vLLM, TGI
Chunked Prefill很高(~85%)更低Sarathi, vLLM

节点选择器与容忍度

# 使用nodeSelector选择GPU节点
spec:
  nodeSelector:
    nvidia.com/gpu.present: "true"
    gpu-type: "a100-80gb"

# 使用tolerations容忍GPU节点taint
  tolerations:
  - key: "nvidia.com/gpu"
    operator: "Exists"
    effect: "NoSchedule"
⚠️ 常见陷阱

原生K8s调度器不理解GPU拓扑，可能将同一Pod的4张GPU分配到不同NUMA节点甚至不同PCIe交换机下，严重影响训练性能。需要使用拓扑感知调度器。

Volcano调度器

Volcano调度器

通俗类比：Volcano是K8s专为AI、HPC等高性能计算场景打造的“超级调度员”。它内置了作业队列、Gang调度、资源预留等高级功能，就像为繁忙的快递站配备了专业的智能分拣管理系统。
常见误区：使用Volcano时，若未正确定义`Queue`（队列）和`PodGroup`（Pod组），或者队列配额（Capability）设置不合理，会导致作业无法调度或资源分配不公，无法发挥其优势。
深入探讨：Volcano核心组件：1. **vc-scheduler**：实现Gang Scheduling、Fair-share等算法。2. **vc-controller-manager**：管理Queue和PodGroup的生命周期。3. **vc-admission**：进行准入校验。它将一组相关的Pod（如一个训练作业的Worker）作为一个`PodGroup`整体调度，要么全部成功，要么全部不调度，避免了资源碎片和作业饥饿。
用户创建 PyTorchJob (由Operator转为PodGroup)
      ↓
Volcano Scheduler
      ├── 检查Queue配额
      ├── 执行Gang调度 (所有Worker Pod需同时满足)
      └── 绑定Pod到节点
特性原生K8s调度器Volcano调度器
调度粒度单个PodPodGroup (一组Pod)
队列管理无支持多队列，可设优先级、配额
作业调度顺序逐个调度，易死锁Gang Scheduling，确保作业完整性
典型负载Web服务，微服务TensorFlow/PyTorch分布式训练，大数据Spark作业
批处理策略GPU利用率延迟代表框架
Static Batching低(~30%)高早期框架
Continuous Batching高(~80%)低vLLM, TGI
Chunked Prefill很高(~85%)更低Sarathi, vLLM

Volcano是CNCF孵化的Kubernetes原生批处理调度器，专为AI/HPC工作负载设计：

# Volcano Job示例
apiVersion: batch.volcano.sh/v1alpha1
kind: Job
metadata:
  name: distributed-training
spec:
  minAvailable: 4  # 至少4个Pod
  schedulerName: volcano
  policies:
    - event: PodEvicted
      action: RestartJob
  tasks:
  - replicas: 4
    name: worker
    template:
      spec:
        containers:
        - name: worker
          image: training:latest
          resources:
            limits:
              nvidia.com/gpu: 8
Volcano特性说明
Gang Scheduling保证所有Pod同时启动，避免死锁
Queue管理支持队列优先级和资源配额
公平调度多租户资源公平分配
拓扑感知优化GPU/NUMA亲和性

KubeRay与分布式训练

KubeRay与分布式训练

通俗类比：KubeRay是运行在K8s上的“分布式计算指挥家”。它让Ray（一个分布式计算框架）能够轻松在K8s集群上部署和管理。就像指挥家协调乐团（Worker Pod）共同演奏复杂交响乐（分布式训练任务）。
常见误区：在K8s上运行分布式Ray集群时，常见错误是未正确配置Pod间网络策略，导致Worker之间无法通过主机名或Pod IP高效通信，严重拖慢训练速度。同时，共享存储（如NFS、PVC）挂载不一致也会导致数据读取失败。
深入探讨：KubeRay Operator管理三个核心自定义资源：`RayCluster`（定义集群规模）、`RayJob`（提交一次性作业）、`RayService`（部署长期服务）。一个典型的`RayCluster`包含一个Head Pod（负责调度和监控）和多个Worker Pod（执行计算任务）。训练时，Worker Pod通过K8s Service或Pod DNS发现彼此，形成计算集群。
KubeRay Operator (部署在K8s)
      ↓ 监听并管理
RayCluster CRD
      ├── Head Pod (协调者)
      │      ↕ gRPC通信
      ├── Worker Pod 1 (GPU)
      ├── Worker Pod 2 (GPU)
      └── Worker Pod N (GPU)
组件角色K8s资源体现

Ray Worker执行具体任务（Actor/Task）一组Pod，通过Ray Autoscaler动态伸缩
KubeRay Operator生命周期管理控制器Deployment，运行控制循环，调和CR状态
批处理策略GPU利用率延迟代表框架

Continuous Batching高(~80%)低vLLM, TGI
Chunked Prefill很高(~85%)更低Sarathi, vLLM

KubeRay是运行Ray集群的Kubernetes Operator，适合分布式ML工作负载：

# Ray Cluster CRD
apiVersion: ray.io/v1
kind: RayCluster
metadata:
  name: ml-cluster
spec:
  headGroupSpec:
    serviceType: ClusterIP
    rayStartParams:
      dashboard-host: "0.0.0.0"
  workerGroupSpecs:
  - groupName: gpu-workers
    replicas: 4
    rayStartParams:
      resources: '"{\"gpu\": 8}"'
    template:
      spec:
        containers:
        - name: ray-worker
          image: rayproject/ray-ml:latest
          resources:
            limits:
              nvidia.com/gpu: 8
✅ 调度器选择建议

小规模集群（<100 GPU）使用NVIDIA Device Plugin + 原生调度器即可；中大规模集群建议使用Volcano或Kueue（K8s原生批处理调度器）；分布式训练优先考虑KubeRay。

✅ 参考答案与详解
  1. 核心原理: Kubernetes在AI场景中扮演编排层角色，通过声明式API管理Pod和Service。核心组件包括：NVIDIA Device Plugin通过DevicePlugin gRPC接口注册GPU设备，支持MIG模式将A100划分为多个独立实例；Volcano调度器实现Gang Scheduling（所有Pod同时启动避免死锁）、Queue管理和拓扑感知；KubeRay Operator管理RayCluster/RayJob/RayService三种CRD。原生K8s调度器不理解GPU拓扑，可能将同一Pod的4张GPU分配到不同NUMA节点，需要使用拓扑感知调度器。Volcano的PodGroup机制将一组相关Pod作为整体调度，要么全部成功要么全部不调度，避免资源碎片和作业饥饿。
    关键要点: ①GPU Device Plugin将GPU暴露为nvidia.com/gpu extended resource；②Bin-Packing策略打包任务到少数节点，Gang Scheduling确保分布式作业所有Pod同时启动；③Volcano的minAvailable控制最小可用Pod数，policies定义驱逐后的重启策略；④KubeRay的Head Pod负责调度监控，Worker Pod执行计算，通过K8s Service发现彼此；⑤tolerations配合GPU节点的NoSchedule taint实现精确调度
    实际案例: 使用Volcano Job部署4节点×2GPU分布式训练：设置minAvailable=8、schedulerName=volcano，每个worker Pod请求nvidia.com/gpu: 2。配置nodeSelector确保调度到A100节点。同时部署KubeRay Operator管理RayCluster，4个Worker Pod各请求8张GPU，通过rayStartParams配置GPU资源。
    关联知识: 与Slurm（第2章）对比——K8s适合混合工作负载，Slurm适合纯HPC训练。容器化（第4章）提供镜像管理基础，Volcano Queue机制与FinOps（第10章）的成本分配关联。
向前串联： K8s在AI场景中的角色的知识将在后续的推理框架选型、性能基准测试、生产环境调优中反复应用。 建议在本章学习后，立即用真实模型（如LLaMA-7B）进行动手实验， 将概念理解转化为可测量的性能数据（延迟、吞吐、显存占用）， 这些数据将成为后续优化决策的量化依据。
学习路径建议： 学习K8s在AI场景中的角色建议按"概念→原理→实验→调优"四步走。先理解核心概念和术语， 再深入数学原理和工作机制，然后用最小化实验验证理解，最后在真实场景中调优参数。 每步都应记录关键指标（延迟/吞吐/精度），建立数据驱动的优化习惯。
