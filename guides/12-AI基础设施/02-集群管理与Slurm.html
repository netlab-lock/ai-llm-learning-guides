Slurm概述

Slurm概述

tip">通俗类比：Slurm就像机场的航班调度塔。它不负责造飞机（写程序），但负责安排哪架飞机（作业）何时用哪条跑道（计算节点），确保所有航班高效有序地起飞。
常见误区：误区：认为Slurm就是“输入命令，马上运行”。实际上，它是资源预约系统，作业提交后状态通常是PD（排队中），需要等待资源可用，耐心是关键。
深入探讨：Slurm（Simple Linux Utility for Resource Management）的核心是一个高度可扩展的资源管理器。其slurmctld守护进程负责集中管理所有计算节点、作业队列和资源分配策略，通常能在毫秒级完成一次调度决策。
User -> srun/sbatch -> [Slurmctld] -> Schedule -> [Compute Nodes]
        ^                                  |
        |___________________________(Job Complete)
特性传统手动执行Slurm管理
资源分配手动指定，易冲突自动、协调、防冲突
作业管理终端关闭即丢失持久化队列，可查询、取消
扩展性非常困难轻松扩展至数万节点
优化层次技术加速比精度影响
算法层量化(INT4/INT8)2-4x<3%
系统层FlashAttention2-9x0%
系统层连续批处理2-5x0%
架构层投机解码2-4x0%

Slurm（Simple Linux Utility for Resource Management）是最流行的HPC工作负载管理器，广泛应用于AI/ML集群。全球TOP500超级计算机中约70%使用Slurm进行作业调度。它提供了作业提交、资源分配、队列管理等核心功能。

核心架构

核心架构

通俗类比：理解Slurm架构就像了解一个物流网络：中央调度中心（slurmctld）负责全局指挥，各个分拣站（slurmd）负责本地执行。清晰的分工是高效协作的基础。
常见误区：误区：只关注slurmctld主控节点，而忽略计算节点上slurmd服务的健康状态。若某个节点的slurmd挂掉，该节点便无法接收新作业，成为集群的“黑洞”。
深入探讨：核心组件包括：1) slurmctld：主控守护进程，运行在管理节点上。2) slurmd：节点守护进程，每台计算节点运行一个。3) slurmdbd：可选的数据库守护进程，用于记录历史作业和计费信息。它们之间通过高效的RPC协议通信。
slurmctld (Head)
    |    \
(slurmd) (slurmd)
(Node1) (Node2)
组件运行位置主要职责
slurmctld管理节点资源调度、作业队列管理、决策
slurmd所有计算节点启动/监控/终止本地任务，报告节点状态
slurmdbd数据库服务器存档作业历史、管理账户与资源限制

常见问题踩坑记录和解决方案参考社区issue和讨论
组件角色说明
slurmctld控制守护进程中央管理器，负责作业调度和资源分配
slurmd计算守护进程每个节点上运行，执行实际计算任务
slurmdbd数据库守护进程记录作业历史、用户账户和QoS信息
Munge认证服务提供集群节点间的安全认证

作业提交流程

# 提交批处理作业
#!/bin/bash
#SBATCH --job-name=train_llm
#SBATCH --partition=gpu
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=8
#SBATCH --gres=gpu:8
#SBATCH --time=48:00:00
#SBATCH --output=logs/%j.out

srun python train.py --config config.yaml
💡 关键概念

Partition（分区）是Slurm中的资源分组，类似于队列。不同分区可配置不同的节点数、时间限制和QoS策略。

GPU分配策略

GPU分配策略

通俗类比：GPU分配就像分配剧院里的舞台。你可以申请独占整个剧院（Exclusive模式），或者只申请一个舞台（CUDA_VISIBLE_DEVICES）。策略选择取决于你的“演出”（计算任务）规模。
常见误区：易犯错误：在脚本中申请了`--gres=gpu:1`，但代码里却默认使用所有GPU（如`torch.distributed.launch`）。这会导致程序崩溃或资源浪费。务必通过环境变量约束代码。
深入探讨：在Slurm脚本中，常用指令如`#SBATCH --gres=gpu:a100:2`（申请2张A100 GPU）。作业启动后，Slurm会设置`CUDA_VISIBLE_DEVICES`环境变量，确保程序只能看到被分配的GPU卡，实现资源隔离。
Node
|--- CPU(s)
|--- Memory
|---- GPU:0 <---- Job A sees this
|---- GPU:1 <---- Job A sees this
|---- GPU:2 <---- Job B sees this
策略命令示例适用场景
共享--gres=gpu:1小规模实验，充分利用GPU计算力
独占节点--exclusive大规模分布式训练，需全部GPU和高速互联
MPS共享配合MPS守护进程多个小任务共享单GPU，提高利用率
GPU显存FP16算力显存带宽适用

公平共享调度

>

公平共享调度

通俗类比：Fairshare就像小区公共设施（如健身房）的使用权分配。你过去使用得越少（Share越小），现在排队时就越靠前。这保证了资源不会被长期霸占，对所有人公平。
常见误区：误区：误以为Fairshare是“插队券”。它是在满足作业优先级（如作业大小、等待时间）后的第二级排序依据。滥用（如频繁提交小作业“刷”优先级）可能违反集群使用政策。
深入探讨：Fairshare的计算基于“消耗份额”与“公平份额”的比值。公式通常为：`Fairshare因子 = 账户公平份额 / (近期消耗份额 + 衰减)`。其中，消耗份额按时间衰减（如半衰期7天），鼓励用户近期少用资源，从而获得未来更高的调度优先权。
Pending Jobs (High Priority First)
        |
        V
[Job1: 8CPU] --> Compare Fairshare --> Node Pool
[Job2: 8CPU] --> Factor: A > B --> |
                                   V
                              Execute Job1
调度算法主要依据目标
FIFO提交时间简单，先来先服务
Backfill小作业填充空隙提高集群整体利用率
Fairshare历史资源消耗保证长期公平，防止资源垄断
批处理策略GPU利用率延迟代表框架
Static Batching低(~30%)高早期框架
Continuous Batching高(~80%)低vLLM, TGI
Chunked Prefill很高(~85%)更低Sarathi, vLLM

公平共享（Fairshare）确保集群资源在多个用户/组之间合理分配。Slurm使用优先级因子来实现公平共享，包括：过去使用量、作业大小、等待时间等维度。

# 查看账户使用情况
sshare -A my_lab --format=Account,User,RawShares,RawUsage

# 配置fairshare权重（slurm.conf）
PriorityType=priority/multifactor
PriorityWeightFairshare=1000
PriorityWeightAge=100
PriorityWeightJobSize=200
⚠️ 注意事项

Fairshare分数会随时间衰减。长期不使用的账户会逐渐恢复高优先级，避免资源被垄断。建议设置合理的衰减周期（通常1-2周）。

数组作业

数组作业

通俗类比：数组作业就像工厂的生产线模板。你定义好一个产品模板（作业脚本），然后告诉工厂生产100个（--array=1-100），系统会自动用不同编号（$SLURM_ARRAY_TASK_ID）去加工，批量处理。

常见误区：易犯错误：在数组脚本中直接使用相同的输入/输出文件名。这样会导致100个任务同时读写同一个文件，造成数据损坏。必须根据任务ID区分输入输出路径。
深入探讨：数组作业的关键环境变量是`SLURM_ARRAY_TASK_ID`。在提交时使用`#SBATCH --array=0-99%10`表示创建索引0到99的100个子作业，并且最多同时运行10个（并发限制）。这极大地简化了参数扫描类任务的管理。
sbatch --array=1-4 job.sh
        |
        V
Task1 (ID=1) -> Runs on Node A
Task2 (ID=2) -> Runs on Node B
Task3 (ID=3) -> Waits (Concurrent=2)
Task4 (ID=4) -> Waits
特性数组作业循环提交多个独立作业
管理复杂度低（一个sbatch命令）高（需循环或脚本提交）
依赖关系内部任务独立，可并行需显式设置依赖
状态查询可整体查询，如`JobArray_123_4`需查询多个作业ID
方面要点注意事项
核心概念数组作业的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化
常见问题踩坑记录和解决方案参考社区issue和讨论

数组作业允许批量提交大量相似任务，非常适合超参数搜索、数据预处理等场景：

# 提交数组作业（100个子任务）
#SBATCH --array=0-99
#SBATCH --gres=gpu:1

python train.py --lr=${LR_LIST[$SLURM_ARRAY_TASK_ID]}

# 管理数组作业
squeue -u $USER              # 查看运行状态
scancel 12345_50             # 取消特定子任务
sacct -j 12345 --format=JobID,State,Elapsed

替代方案

替代方案

常见误区：误区：盲目追求技术新颖而选择替代方案。必须评估：1) 学习成本和团队技能栈；2) 与现有科学计算软件（如MPI）的兼容性；3) 服务商的GPU实例库存和竞价策略稳定性。
深入探讨：常见替代方案对比：Kubernetes (K8s) + KubeFlow：适合云原生AI流水线，但直接管理HPC的InfiniBand网络和GPU拓扑更复杂。云服务商（AWS Batch, Azure Batch）：完全托管，但长期成本可能高于自建集群，且数据传输受带宽限制。
On-Premise Cluster: [Slurm] <-> [Storage]
Hybrid Cloud: [Slurm] <-> [Cloud HPC]
Full Cloud: [K8s/Cloud Batch] <-> [Cloud Storage]
方案优势劣势
Slurm高性能，对传统HPC支持极佳运维复杂，弹性伸缩较慢
Kubernetes弹性，微服务化，云生态丰富对裸金属、高速网络支持门槛高
云批处理免运维，按需付费，快速启动长期成本高，数据迁移、锁定问题
