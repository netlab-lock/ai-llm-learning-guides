可观测性三大支柱

可观测性三大支柱

ip">通俗类比：类比经营一家餐厅：日志（Log）是详细的服务员工作记录；指标（Metric）是营业额、翻台率等实时看板；追踪（Trace）是单个顾客从点菜到结账的完整流程。三者结合才能知道哪道菜（系统）有问题。
常见误区：只重指标不重日志。看到“餐厅营业额（QPS）下降”却不看日志，就像只看到营业额跌了，却不去翻服务员的记录本，无法得知是“服务员态度差（报错）”还是“厨房出菜慢（延迟高）”。
深入探讨：三大支柱数据量级差异巨大：日志是文本流，1个实例可达1GB/天；指标是时间序列，是压缩的数值；追踪是稀疏的DAG图。生产环境中常采用分级策略：日志只存ERROR以上，核心指标15s采样，仅对慢请求（>1s）开启全链路追踪。
  [应用] --> (日志): 详细错误栈
    |--> (指标): error_rate=5%, p99=800ms
    |--> (追踪): trace_id=abc -> svc1 -> svc2(慢)
支柱数据形式核心用途存储开销
日志 (Logging)非结构化/半结构化文本事件审计、错误根因分析高
指标 (Metrics)带时间戳的数值型时间序列趋势监控、实时告警低
追踪 (Tracing)带时间戳的请求DAG图分布式链路性能分析中
方面要点注意事项
核心概念可观测性三大支柱的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化
常见问题踩坑记录和解决方案参考社区issue和讨论

AI系统的可观测性需要监控基础设施、模型和业务三个层面。不同于传统服务，ML系统还需要关注模型质量和数据漂移。

层面关注指标工具
基础设施GPU利用率、显存、网络IO、磁盘DCGM, Prometheus, Grafana

GPU监控

GPU监控

通俗类比：类比大型厨房：CUDA Core是几千个只能炒蛋炒饭的小灶台，GPU显存是备料台。监控要盯紧：灶台使用率（GPU Utilization）、备料台占用（显存占用）、以及每个灶台等待上料的时间（GPU Stall）。哪个满载或堵塞都得知道。
常见误区：只看GPU Utilization高就以为好事。如果Utilization 100%但显存占用很低，很可能是你的“小灶台”一直在空转等CPU“主厨”备料，这是一种资源浪费（CPU瓶颈）。需要结合显存占用和FP32/FP16单元利用率综合判断。
深入探讨：关键指标：1. GPU Utilization (nvidia-smi)：>70%算健康。2. Memory Usage：显存占用不能超过物理显存，否则会触发昂贵的CPU-Swap。3. SM Occupancy：理论峰值=活跃Warps / 最大Warps，低于30%可能因线程块配置不当。4. Tensor Core利用率（如可用）：针对矩阵计算的核心。
  CPU (主厨) --> [数据] --> GPU显存 (备料台)
                        |
                        V
   GPU Core 0 (灶台0) <-- |
   GPU Core 1 (灶台1) <-- |-- 共享“菜谱” (Kernel)
   ... (几千个灶台)     <-- |

⚠️ GPU监控陷阱

nvidia-smi显示的GPU利用率可能具有误导性。GPU在等待数据加载时仍显示高利用率。应使用DCGM的SM Active和Tensor Core Active指标获取更准确的利用率数据。

模型性能指标

通俗类比：评估模型就像评厨师：不仅要菜做得快（延迟），还要做得准（准确率），且不能浪费太多食材（计算成本）。不能只看一个维度，快而不准（延迟低但胡言乱语）或准但巨慢（准确但用户等不起）都不行。
常见误区：混淆吞吐量(Throughput)与延迟(Latency)。追求高吞吐量（如用大Batch）可能大幅提升单个请求的延迟。对于在线服务，用户感知的是延迟。需根据场景权衡：离线批量处理重吞吐，在线服务重P99延迟。
深入探讨：核心公式：1. 延迟(Latency)：P50（中位数）， P99（99分位，长尾指标）。2. 吞吐量(Throughput)：QPS = 并发数 / 平均延迟(s)。3. 成本效率：Tokens/$ = 生成的总Token数 / 模型推理成本。4. 首Token延迟(TTFT)：流式场景下的关键体验指标。
  用户请求 --> |[队列]| --> [模型服务器] --> 响应
                     |
                     V
        测量: 请求到达 -> 首Token返回 (TTFT)
                  请求到达 -> 完整响应返回 (Latency)
指标类别具体指标公式/定义关注场景
延迟P50, P99 Latency, TTFT百分位响应时间在线交互服务
吞吐量QPS, Tokens/s单位时间处理的请求数或Token数批量处理，成本优化
资源效率GPU利用率，Tokens/$Token输出量 / 推理成本成本控制与优化

漂移检测

漂移检测

通俗类比：就像依赖天气预报。如果气候模式（数据分布）变了，但你还在用去年的老模型（天气模型）预测，必然不准。漂移检测就是持续比对“新天气数据”和“老模型训练时的天气数据”，发现偏移就报警或换模型。
常见误区：将概念漂移与数据漂移混为一谈。数据漂移（输入分布变了）不一定导致性能下降。只有当数据漂移导致了模型性能下降（概念漂移），才需要紧急干预。误报会消耗大量人力。
深入探讨：1. 数据漂移检测：常用PSI（群体稳定性指标）。公式：PSI = Σ (实际占比 - 预期占比) * ln(实际占比 / 预期占比)。通常PSI > 0.25认为显著漂移。2. 概念漂移检测：直接监控模型输出指标（如准确率）的滑动窗口变化。当连续N个窗口指标低于基线阈值β时触发告警。
  训练数据 (分布P) --> 训练 --> 模型
  实时数据 (分布Q) --> 推理 --> 输出
        |                      |
        V                      V
  [计算 PSI(P,Q)]    [计算 准确率变化]
漂移类型检测对象常用方法应对策略
数据漂移输入特征(X)的分布PSI, KL散度, KS检验检查数据源，评估是否影响模型
概念漂移特征与标签的关系 P(Y|X)监控模型性能指标(准确率等)的滑动窗口触发模型重训练或在线学习
方面要点注意事项
核心概念漂移检测的基本原理理解底层机制比记忆参数更重要

模型上线后，输入数据分布可能随时间变化（数据漂移），导致模型性能下降。需要建立漂移检测机制：

# 数据漂移检测示例
from evidently.report import Report
from evidently.metric_preset import DataDriftPreset

report = Report(metrics=[DataDriftPreset()])
report.run(
    reference_data=train_data,   # 训练数据分布
    current_data=production_data # 生产数据分布
)

# 输出漂移报告
drift_results = report.as_dict()
if drift_results['metrics'][0]['result']['dataset_drift']:
    send_alert("数据漂移检测到！需要重新训练模型")
💡 漂移类型
  • 数据漂移：输入数据分布变化（如用户行为变化）
  • 概念漂移：输入与输出的关系变化（如市场趋势变化）
  • 模型退化：模型本身性能随时间下降

Prometheus + Grafana

Prometheus + Grafana

通俗类比：Prometheus像一个极其高效、主动的“档案管理员”：它定期“抄写”（拉取）服务器贴在门上（/metrics端点）的“工作数据纸条”，并存入自己的“卡片柜”（时序数据库）。Grafana则是“可视化设计师”，能从这个卡片柜里取出任意数据，拼成美观的仪表盘。
常见误区：认为Prometheus是“推（Push）”模型。它是“拉（Pull）”模型，客户端（Exporter）被动等待Prometheus来抓取。如果客户端在NAT网关后或防火墙内，可能需要Pushgateway做中转，但这不是标准用法。
深入探讨：1. 核心架构：Prometheus Server (拉取，存储，查询)， Exporters (如node_exporter, nvidia_gpu_exporter)， Alertmanager (告警管理)。2. 存储：本地TSDB，单机默认保留15天，长期存储需接Thanos或Mimir。3. 查询：强大的PromQL，例如 `rate(http_requests_total[5m])` 计算QPS。
  [应用/主机] --(Exporter)--> [ :9100/metrics ]
                               ^
                               | (定期Pull)
                               |
                       [Prometheus Server] --(Grafana查询)--> [仪表盘]
                               | --(触发告警规则)--> [Alertmanager]
组件角色关键特点
Prometheus Server数据采集、存储、查询引擎Pull模型，内置TSDB，PromQL
Exporter指标暴露端点如node_exporter, dcgm_exporter, mysql_exporter
Grafana可视化与看板支持PromQL数据源，高度可定制图表
Alertmanager告警去重、分组、路由对接Slack, Email, 企微等通知渠道
方面要点注意事项
核心概念Prometheus + Grafana的基本原理理解底层机制比记忆参数更重要
实践应用在实际项目中的使用方式先跑通demo再优化

告警策略

告警策略

常见误区：告警疲劳（Alert Fatigue）。设置了海量低优先级告警，导致值班人员麻木，错过真正重要的P0告警。必须定期“修剪”无效告警，确保每条告警都需要且能够立即采取行动。
深入探讨：1. 分级策略：P0：核心服务不可用，全员电话通知。P1：性能严重下降（P99 > 2s），通知主值班。P2：非关键指标波动（如CPU > 80%持续5分钟），工作时间群通知。2. 设计原则：每条告警必须可操作（Actionable）。3. 静默与聚合：通过Alertmanager将相同根因的告警聚合。
  监控规则 (Prometheus Rule) 
      |
      V
  持续满足条件? --是--> 触发告警 --> Alertmanager 
      |                                   |
      no                                  V
      |                            分组，去重，路由
      V                                   |
  状态正常                               V
                                  [发送通知：P0/P1/P2]
告警等级触发条件示例通知方式响应要求
P0 (紧急)服务完全不可用，持续5分钟无数据电话 + 短信 + 即时通讯立即响应，全力处理
P1 (严重)错误率>10%持续10分钟， P99延迟>5s即时通讯 + 邮件30分钟内响应处理
P2 (警告)CPU > 90%持续15分钟， 磁盘>85%即时通讯/群组工作时间排查

✅ 参考答案与详解
  1. 核心原理: AI系统可观测性需要监控基础设施、模型和业务三个层面。三大支柱：日志（Logging，非结构化文本，用于事件审计和错误根因分析）、指标（Metrics，时间序列数值，用于趋势监控和实时告警）、追踪（Tracing，请求DAG图，用于分布式链路性能分析）。GPU监控的关键指标包括GPU Utilization（>70%健康）、Memory Used（<90%物理显存）、SM Occupancy（>30%）和Tensor Core Utilization。注意nvidia-smi显示的利用率可能具有误导性——GPU等待数据加载时仍显示高利用率，应使用DCGM的SM Active指标。模型性能指标核心公式：QPS=并发数/平均延迟，Tokens/$=生成总Token数/推理成本。漂移检测区分数据漂移（输入特征分布变化，用PSI衡量，PSI>0.25认为显著）和概念漂移（特征与标签关系变化，监控模型输出指标的滑动窗口）。Prometheus采用Pull模型（定期拉取/metrics端点），Grafana通过PromQL查询可视化。
    关键要点: ①TTFT（首Token延迟）<200ms、TPS（每秒Token数）>30 tok/s是推理服务关键指标；②DCGM Exporter导出GPU指标到Prometheus，需配置custom-counters.csv自定义计数器；③告警分级：P0（服务不可用，5分钟响应，自动回滚+电话通知）、P1（P99延迟>目标3倍，15分钟响应）、P2（GPU利用率<30%，1小时响应）；④PSI=Σ(实际占比-预期占比)*ln(实际占比/预期占比)，>0.25触发告警；⑤Prometheus本地TSDB默认保留15天，长期存储需接Thanos或Mimir
    实际案例: 配置Prometheus抓取AI服务指标：设置scrape_configs分别监控llm-inference（端口8080，15s间隔）、dcgm-exporter（端口9400）和gpu-node（端口9100）。在Grafana中创建"黄金指标"仪表板，一屏展示请求延迟、错误率、吞吐量、GPU利用率。使用Evidently库实现数据漂移检测，当PSI超过0.25时通过Alertmanager发送告警到Slack。
