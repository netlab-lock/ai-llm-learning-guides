一、Prompt 工程

有效的 Prompt 是高效使用 AI Coding 工具的关键。以下是经过验证的 Prompt 策略：

1.1 结构化 Prompt

// 好的 Prompt 结构
// 1. 上下文：这是一个 React 电商项目，使用 TypeScript + Zustand
// 2. 任务：实现商品购物车的加减数量功能
// 3. 约束：需要支持库存上限校验，使用乐观更新
// 4. 输出：完整的组件代码和对应的单元测试

// 差的 Prompt
// 写一个购物车组件

1.2 渐进式细化

  • 第一步：描述高层次需求，获取架构方案
  • 第二步：针对具体模块，生成实现代码
  • 第三步：添加边界处理、错误处理、性能优化
  • 第四步：生成测试用例和文档
💡 Prompt 模板

为团队建立 Prompt 模板库：通用组件模板、API 端点模板、测试模板等。统一的 Prompt 模式提升一致性和效率。

二、上下文管理

AI 的输出质量直接取决于提供的上下文质量：

上下文类型提供方式影响
项目规范CLAUDE.md / .cursorrules代码风格一致性
相关文件@file 引用 / 打开的标签功能正确性
API 文档Paste / @docs接口使用正确
错误日志终端输出 / 粘贴调试准确性
设计文档链接引用架构一致性
# CLAUDE.md 示例
## 项目约定
- 使用 TypeScript strict 模式
- 组件使用函数式 + Hooks
- 状态管理使用 Zustand
- API 层使用 React Query
- 测试使用 Vitest + Testing Library
- 提交信息遵循 Conventional Commits
📁 上下文窗口管理

当上下文过长时，AI 的注意力会分散。遵循"最小充分上下文"原则——只提供完成任务所需的最少信息。

三、TDD + AI

测试驱动开发 (TDD) 与 AI 工具的结合是提升代码质量的最佳实践：

写测试 → AI 生成实现 → 运行测试 → AI 修复失败 → 重构优化

TDD 工作流

  1. 先写测试：自己或让 AI 生成测试用例，定义预期行为
  2. AI 生成实现：将测试作为上下文，让 AI 生成通过测试的代码
  3. 迭代修复：运行测试，将失败信息反馈给 AI 修复
  4. 重构：测试通过后，让 AI 优化代码质量
# TDD + AI 示例
# Step 1: 写测试
def test_parse_email():
    assert parse_email("user@example.com") == {"user": "user", "domain": "example.com"}
    assert parse_email("invalid") is None

# Step 2: 给 AI prompt
# "根据以上测试用例，实现 parse_email 函数，处理各种边界情况"

# Step 3: 运行测试，如有失败反馈给 AI

四、AI 代码审查

AI 生成的代码需要系统性审查：

审查维度检查点工具辅助
功能正确性边界条件、空值处理、类型安全单元测试覆盖
安全性注入漏洞、敏感信息泄露、权限校验Snyk, CodeQL
性能N+1 查询、内存泄漏、不必要的重渲染Profiler
可维护性命名规范、代码重复、过度工程ESLint, SonarQube
许可证引入的代码/库的许可证兼容性FOSSA
⚠️ 审查黄金法则

AI 生成的代码等同于初级开发者提交的代码——需要同等严格的审查标准。不要因为"AI 写的"就降低审查要求。

五、安全考虑

  • 敏感信息：不要将 API Key、密码、私钥等放入 AI 的上下文中
  • 代码泄露：了解工具的隐私政策，企业代码是否用于模型训练
  • 供应链：AI 推荐的依赖可能包含已知漏洞，需要检查
  • 注入攻击：AI 生成的用户输入处理代码需要特别审查
  • 合规性：AI 生成的代码可能包含受版权保护的片段

六、反模式

忽略测试AI 代码不写测试TDD，AI 生成+人工补充测试

Prompt 模糊"帮我写个 API"明确输入输出、约束、边界
✅ 团队规范建议

1. 建立团队级 CLAUDE.md / .cursorrules 配置
2. AI 生成的 PR 必须标注 [AI-assisted]，适用相同审查标准
3. 定期评估 AI 工具的 ROI，调整投入
4. 建立安全审查清单，特别关注 AI 代码

常见误区：1) 盲目接受 AI 代码——AI 代码 = 初级开发者代码，必须同等审查标准；2) 上下文塞太多——「最小充分上下文」原则，给太多信息反而让 AI 注意力分散；3) 期望一次生成完美——AI 擅长「初稿」，你需要迭代细化；4) 不写测试就用 AI 代码——没有测试 = 没有安全网，AI 可能生成表面正确但边界 case 错误的代码；5) 忽略团队规范——个人用 AI 效果好不代表团队也一样，需要统一规范确保一致性。

九、效率度量体系

如何量化 AI Coding 工具的效率提升？以下是推荐的度量框架：

指标定义测量方法基准值目标值
编码速度每小时有效代码行数Git diff 统计30行/小时60行/小时
任务完成时间从开始到PR提交的时间项目管理工具基准值减少30-50%
首次通过率PR 首次审查通过的比例GitHub PR 统计40%60%
测试覆盖率新增代码的测试覆盖Istanbul/nyc60%80%
Bug 密度每千行代码的 Bug 数Bug 追踪系统基准值降低20%
AI 使用率AI 辅助代码占总代码比例AI 工具统计0%40-60%
深入探讨：TDD + AI 是当前最有效的 AI Coding 工作流。核心逻辑：测试是「规格说明书」，AI 根据规格生成实现。这解决了 AI 最大的问题——「不知道你想要什么」。有了测试，AI 的输出可以自动验证，失败了就反馈给 AI 修复，形成闭环。团队级最佳实践：1) 建立 CLAUDE.md 统一规范；2) AI PR 标注 [AI-assisted]；3) 安全审查清单特别关注注入漏洞和敏感信息泄露；4) 定期评估 AI 工具 ROI。更深层的思考是：AI Coding 的价值不仅在于「写代码更快」，而在于「释放认知资源」——当开发者不再需要记忆 API 语法、编写样板代码时，他们可以将更多精力投入到架构设计、业务理解、用户体验等更高价值的工作中。这种「认知卸载」效应是 AI Coding 工具的真正价值所在，但很难用简单的效率指标来衡量。建议团队在评估 AI 工具时，除了量化指标外，也要关注开发者的主观感受和工作满意度。
💡 效率度量建议：1) 不要只看「代码行数」——AI 辅助下可能写了更多代码，但不一定是更好的代码；2) 关注「首次通过率」比「编码速度」更重要——AI 生成的代码如果审查反复修改，效率反而下降；3) 建立 A/B 对比——让部分开发者使用 AI 工具，部分不使用，对比 1 个月后的各项指标；4) 定期收集开发者反馈——效率提升可能是无形的（如减少上下文切换、降低认知负荷），这些不会体现在量化指标中。

✅ 参考答案与详解
  1. 核心原理: 「1.1 结构化 Prompt」的核心在于理解其底层工作机制。从系统视角分析，该技术解决了LLM基础设施中的关键瓶颈问题。其数学本质可以归结为一个优化问题：在给定约束条件下最大化目标函数（如吞吐量、资源利用率或成本效率）。具体而言，需要掌握：问题形式化描述、算法设计思路、复杂度分析三个层面。从硬件视角看，性能通常受限于三个维度：计算能力（FLOPs）、内存带宽（GB/s）和通信带宽（Gb/s）。理解这三者的关系，就能推导出大多数优化策略。
    关键要点: ①核心原理的数学推导和物理意义；②关键参数的设计原则和调优方法；③与替代方案的对比分析；④实际部署中的工程考量；⑤性能指标的量化和评估方法。
    实际案例: 以一个生产环境部署场景为例：当请求QPS从100增长到1000时，系统瓶颈从计算受限转变为内存受限。通过Profiling发现Decode阶段HBM带宽利用率达到85%，此时应该采用KV Cache量化（减少带宽需求）或分离架构（将Prefill和Decode部署在不同GPU上）来优化。实测优化后TTFT降低40%，TPS提升2.5x。
  2. 核心原理: 在实际工程应用中，本章介绍的技术需要根据具体场景进行适配。关键技术决策包括：架构选择（一体vs分离）、精度选择（FP16/FP8/INT4）、缓存策略（LRU/LFU/Size-based）等。决策依据是请求特征（prompt长度分布、并发量、SLA要求）和硬件条件（GPU型号、网络带宽、存储类型）。标准化的评估方法是通过Profiling确定瓶颈类型：计算受限→量化/剪枝；内存受限→算子融合/缓存优化；通信受限→拓扑优化/通信压缩。
    关键要点: ①需求分析和场景匹配；②技术参数的调优方法；③性能评估和瓶颈定位；④成本-性能权衡分析；⑤生产环境的最佳实践。
    实际案例: 某RAG应用部署中，通过Profiling发现Prefill阶段占端到端延迟的70%。原因是system prompt包含5K token的检索文档。通过启用Prefix Caching，Prefill只需处理200 token的用户查询，TTFT从800ms降至120ms，节省85%。同时使用FP8量化进一步提升Decode阶段速度。
  3. 核心原理: 对比分析需要从多个维度展开：性能（延迟、吞吐）、精度（量化损失）、成本（GPU小时）、复杂度（运维难度）、适用场景（prompt长度、并发模式）。不同技术方案各有优劣：如一体架构简单但GPU利用率低，分离架构高效但引入传输开销。选择依据是业务需求：如果SLA要求TTFT<200ms且QPS>100，应选择分离架构+Prefix Caching；如果QPS<10且prompt短，一体架构即可满足。
    关键要点: ①多维度对比评估框架；②场景匹配和方案选择；③实验验证和性能benchmark；④渐进式优化路径；⑤技术债和可维护性考量。
    实际案例: 某团队对比了三种方案：A)一体架构+FP16 B)一体架构+FP8+Prefix Caching C)分离架构+FP8+Prefix Caching。方案A月度成本$50K，TTFT=500ms；方案B成本$30K，TTFT=200ms；方案C成本$25K，TTFT=150ms。考虑到运维复杂度和团队能力，选择了方案B作为过渡方案，半年后迁移到方案C。

🔬 深入原理：理解 AI Coding 工具的工作机制有助于更高效地使用它们。代码补全模型的核心原理基于 Transformer 架构的因果语言建模：模型将代码视为 token 序列，通过注意力机制学习 token 之间的依赖关系。当开发者输入部分代码时，模型计算每个候选 token 的条件概率分布，选择概率最高的 token 作为补全建议。这一工作机制的数学推导基于极大似然估计（MLE），目标函数为最大化训练数据的对数似理：L = Σ log P(x_t | x_1, ..., x_{t-1})。理解这一原理后，开发者可以更好地设计 Prompt：提供更多的上下文信息（如函数签名、类型注释、注释）可以显著提升模型的预测准确率，因为这些信息有效地缩小了条件概率的搜索空间。
