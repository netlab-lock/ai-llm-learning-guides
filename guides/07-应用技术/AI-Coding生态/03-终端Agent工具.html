一、Claude Code

Anthropic 推出的终端 AI 编程代理，以强大的代码理解和推理能力著称。

# 安装
npm install -g @anthropic-ai/claude-code

# 使用
claude                           # 启动交互模式
claude "帮我实现用户认证模块"       # 直接执行任务
claude /review                   # 代码审查
claude /cost                     # 查看 token 用量

核心能力

  • 代码库理解：自动索引项目结构，理解依赖关系
  • 文件编辑：直接读写文件，支持多文件批量编辑
  • 终端执行：运行构建、测试、Git 操作
  • 安全控制：可配置的权限系统，敏感操作需确认
🔧 Claude Code 特色

Claude Code 的优势在于推理能力强，能处理复杂的跨文件重构任务。支持 CLAUDE.md 项目配置文件，可定义编码规范和项目约定。

二、OpenHands

开源的 AI 软件工程代理平台（原名 OpenDevin），可在沙箱环境中自主执行编程任务。

# 使用 Docker 部署
docker run -it --pull=always \
  -e SANDBOX_RUNTIME_CONTAINER_IMAGE=docker.all-hands.dev/all-hands-ai/runtime:latest \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -p 3000:3000 \
  --add-host host.docker.internal:host-gateway \
  docker.all-hands.dev/all-hands-ai/openhands:latest

架构特点

  • 沙箱执行：在 Docker 容器中安全执行代码，防止意外修改
  • 多模型支持：GPT-4o、Claude、本地模型均可
  • Web UI：提供浏览器界面，可视化交互
  • 扩展性：插件系统支持自定义能力扩展

三、Aider

终端中的 AI 结对编程工具，专注于代码编辑的精确性和可控性。

# 安装
pip install aider-chat

# 使用
aider                            # 启动交互模式
aider --model claude-3.5-sonnet  # 指定模型
aider --watch-files              # 监视文件变更

# 在 Aider 中
/add src/auth.py tests/          # 添加文件到上下文
/ask 解释一下这个认证流程          # 提问模式
/code 实现密码重置功能            # 编辑模式

独特功能

  • Git 集成：每次修改自动创建 commit，便于回滚
  • 编辑模式：支持 whole-file、diff、udiff 三种编辑格式
  • Map 模式：自动生成代码库地图，帮助 AI 理解项目结构
  • 多模型：支持 20+ 种 LLM，包括本地模型
💡 Aider 优势

Aider 的 Git 自动提交功能非常适合需要精确版本控制的场景。每次 AI 修改都有对应的 commit，便于代码审查和回滚。

四、Continue.dev

开源的 AI 代码助手，同时支持 IDE 插件和终端使用。特色是可连接本地模型。

  • 支持 VS Code 和 JetBrains 插件
  • 可连接 Ollama、llama.cpp 等本地推理服务
  • 自定义上下文提供器和工具

七、终端 Agent 的工作循环原理

终端 Agent 工具的核心是一个「思考-行动-观察」循环（ReAct Loop）。理解这个循环的工作原理，有助于开发者更好地与 Agent 高效协作，并在 Agent 出错时快速定位问题所在。

终端 Agent 的 ReAct 循环： ┌──────────────────────────────────────────────────────────┐ │ │ │ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ │ │ 思考 │────→│ 行动 │────→│ 观察 │ │ │ │ (Think) │ │ (Act) │ │ (Observe) │ │ │ └───────────┘ └───────────┘ └─────┬─────┘ │ │ ↑ │ │ │ └─────────────────────────────────────┘ │ │ 循环直到任务完成 │ │ │ │ Think: 分析当前状态，规划下一步行动 │ │ Act: 读取文件/编辑代码/执行命令/搜索代码 │ │ Observe: 获取行动结果，判断是否需要继续 │ └──────────────────────────────────────────────────────────┘

以 Claude Code 处理一个 Bug 修复任务为例，其循环过程大致如下：

  1. Think：分析 Issue 描述，确定需要查看哪些文件
  2. Act：读取 Issue 中提到的源文件
  3. Observe：获取文件内容，发现异常代码
  4. Think：分析异常原因，制定修复方案
  5. Act：编辑文件，修复 Bug
  6. Observe：运行测试，验证修复是否正确
  7. Think：测试通过，生成提交信息
  8. Act：创建 Git commit

这个循环的关键在于「观察」环节——Agent 需要根据行动结果动态调整策略。如果测试失败，Agent 会分析失败原因并尝试不同的修复方案，而不是简单地重试。这种动态调整能力是区分「好的 Agent」和「差的 Agent」的关键标准。

7.1 循环深度与成本控制

Agent 循环的深度（循环次数）直接影响成本和延迟。一个简单的 Bug 修复可能只需要 3-5 次循环，而复杂的重构任务可能需要 20-50 次循环。每次循环都会消耗模型的输入/输出 tokens，因此循环深度与成本呈线性关系。

任务复杂度典型循环次数Token 消耗估算成本耗时
简单修复（1-2行）3-5次~5K tokens$0.01-0.0530秒-1分钟
中等功能（多文件）10-20次~30K tokens$0.1-0.32-5分钟
复杂重构（跨模块）20-50次~100K tokens$0.5-2.05-15分钟
端到端任务（完整功能）50-100次~300K tokens$2-515-30分钟
通俗类比：终端 Agent 的工作循环就像「一个程序员接到需求后的思考过程」——先理解需求（Think），然后写代码（Act），再跑测试看结果（Observe），如果测试失败就再想再改（循环）。区别在于：人可能会卡住去 Stack Overflow，Agent 则会直接尝试下一种方案。所以 Agent 更「执着」，但也可能在错误的方向上反复尝试——这就是为什么需要设置最大循环次数限制。

八、工具调用机制

终端 Agent 的强大之处在于它能调用多种外部工具来完成任务。这些工具构成了 Agent 的「能力边界」——Agent 能做什么，取决于它能调用哪些工具。

工具类型具体工具用途风险级别
文件读取read_file, glob, grep查看代码、搜索文件🟢 低
文件编辑write_file, patch修改代码、创建文件🟡 中
终端执行terminal (bash/sh)运行命令、构建、测试🔴 高
Git 操作git add/commit/push版本控制、代码提交🟡 中
网络请求curl, API 调用下载依赖、调用服务🟡 中
浏览器puppeteer, playwrightUI测试、截图验证🟡 中
终端 Agent 工具调用流程： ┌──────────────┐ │ 用户: "修复 │ │ 登录页面bug" │ └──────┬───────┘ ▼ ┌──────────────┐ ┌────────────────────────────────────┐ │ Agent 思考 │ │ 可用工具列表： │ │ 分析任务 │ │ 1. read_file - 读取文件 │ └──────┬───────┘ │ 2. write_file - 写入文件 │ │ │ 3. patch - 精确编辑 │ ▼ │ 4. terminal - 执行命令 │ ┌──────────────┐ │ 5. search_files - 搜索文件 │ │ 选择工具 │───→│ 6. ... │ │ 构造参数 │ └────────────────────────────────────┘ └──────┬───────┘ ▼ ┌──────────────┐ │ 执行工具调用 │ │ 获取结果 │ └──────┬───────┘ ▼ ┌──────────────┐ │ 观察结果 │──→ 继续循环或完成任务 └──────────────┘

8.1 权限控制策略

由于终端 Agent 可以执行任意命令，权限控制至关重要。主流工具采用了不同的权限策略：

工具默认策略配置方式特色
Claude Code高风险操作需确认allowedTools 配置支持按工具粒度控制
Aider只编辑指定文件/add 命令控制精确的文件范围控制
OpenHands沙箱隔离Docker 容器完全隔离执行环境
Cline每次操作需确认自动/手动模式切换最保守的默认策略
常见误区：1) 终端 Agent 权限太大——它能访问整个文件系统和终端，一个误操作可能 rm -rf /，必须用沙箱或权限控制；2) 不看 Agent 的 Git 提交——Aider 每次改代码都自动 commit，但你得 Review 这些 commit，不能盲目信任；3) 以为 OpenHands 能解决所有 Issue——它在 SWE-bench 上解决率约 53%，将近一半的 Issue 它搞不定；4) 忽略 Agent 的 token 消耗——一个复杂任务可能消耗 $2-5 的 API 费用，团队使用时需要注意预算控制。

九、安全控制策略

终端 Agent 的安全控制是企业采纳的关键障碍。没有完善的安全机制，企业IT部门不会允许Agent访问生产代码库和终端运行环境。以下是推荐的多层安全防护策略：

多层安全防护架构： ┌──────────────────────────────────────────────────────────┐ │ 用户层 │ │ 任务描述审查 │ 结果人工确认 │ 操作日志审计 │ ├──────────────────────────────────────────────────────────┤ │ Agent层 │ │ 权限配置 │ 工具白名单 │ 最大循环次数限制 │ ├──────────────────────────────────────────────────────────┤ │ 系统层 │ │ 沙箱执行 │ 文件系统隔离 │ 网络访问控制 │ ├──────────────────────────────────────────────────────────┤ │ CI/CD层 │ │ 自动测试 │ 安全扫描 │ 代码审查门禁 │ └──────────────────────────────────────────────────────────┘
安全层措施实现方式适用场景
权限控制限制 Agent 可执行的命令Claude Code allowedTools所有场景
沙箱执行在 Docker 容器中运行OpenHands sandbox不信任的任务
文件隔离只允许编辑指定目录Aider /add 命令精确编辑场景
操作确认高风险操作需人工确认Cline auto-approve 配置生产环境

深入探讨：终端 Agent 的安全模型与 IDE Agent 有本质区别。IDE Agent 是「人在回路」——每个补全都由用户手动接受（按 Tab），风险极低；终端 Agent 是「半自主」——一旦启动，它会连续执行多个操作，中间可能不会暂停。这意味着终端 Agent 的安全模型必须是「默认不信任」的——每个操作都需要验证权限，而不是「默认信任」。Claude Code 的 allowedTools 配置就是这种理念的体现：你必须明确声明 Agent 可以使用哪些工具，未声明的工具默认禁止。这种「最小权限原则」在安全领域是经典做法，但在 AI Agent 领域还处于早期实践阶段。另一个值得关注的趋势是「沙箱化」——OpenHands 通过 Docker 容器将 Agent 的执行环境完全隔离，即使 Agent 执行了危险操作（如 rm -rf），也不会影响宿主系统。未来，沙箱化可能成为终端 Agent 的标配安全机制。

十、CI/CD 集成实践

终端 Agent 的一个重要应用场景是 CI/CD 流水线的自动化。以下是典型的集成模式：

# GitHub Actions 集成 Claude Code 示例
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: Run AI Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude /review \
            --pr ${{ github.event.pull_request.number }} \
            --focus security,performance \
            --output review.md
      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });
💡 CI/CD 集成建议：1) 将 AI 审查作为 PR 检查的一个环节，而非唯一审查手段；2) 设置超时限制（如 5 分钟），避免 Agent 无限循环导致 CI 卡住；3) 将 AI 审查结果以 Comment 形式发布，便于团队讨论；4) 对 AI 审查设置「通过/失败」阈值，如发现严重安全漏洞则阻止合并。
向前串联： 一、Claude Code的知识将在后续的推理框架选型、性能基准测试、生产部署中反复应用。 建议结合实际框架动手验证本章所学内容。
终端 Agent 工具的技术原理与深度分析： 核心概念：终端 Agent 工具是AI-Coding生态领域的关键技术方向，其理论基础涉及信息论、优化理论和系统工程的交叉。 推导：从信息论角度，任何信息处理系统的吞吐量受限于信道容量——Shannon定理指出C=B×log₂(1+S/N)， 其中B为带宽、S/N为信噪比。在终端 Agent 工具的语境下，"信道"可以理解为模型的上下文窗口或推理管线， "信噪比"对应有用信息与噪声（无关上下文、冗余计算）的比例。 为什么这很重要？因为理解终端 Agent 工具的理论限制，才能在工程实践中做出正确的权衡—— 盲目追求指标上限往往导致过度设计，而理解根本约束才能找到最高效的优化路径。 从工程实践看，终端 Agent 工具的优化通常遵循"先测量、再分析、后优化"的循环： 首先建立完善的评测基准（覆盖质量、延迟、成本三个维度），然后通过profiling定位瓶颈环节， 最后针对性地优化——是计算受限就优化算法，是内存受限就优化数据布局，是通信受限就优化网络拓扑。 这一系统化方法论适用于AI-Coding生态领域的所有技术问题。

11.1 终端 Agent 的 Prompt 工程

与终端 Agent 协作时，任务描述的质量直接决定了输出效果。一个好的任务描述应该包含上下文、目标、约束和期望输出四个要素。上下文告诉 Agent 项目的技术栈和当前状态，目标明确要完成的具体功能，约束限制实现方式和依赖范围，期望输出定义交付物的形式。以下是经过验证的 Prompt 策略和实际案例：

# 好的任务描述示例
claude "在 src/auth/ 目录下实现 JWT 刷新令牌功能：
1. 创建 refresh-token.ts 实现令牌刷新逻辑
2. 修改 login.ts 添加 refresh token 生成
3. 添加 /api/auth/refresh 路由
4. 编写对应的单元测试
约束：使用现有的 jwt 库，不引入新依赖"

# 差的任务描述示例
claude "帮我做认证功能"  # 太模糊，AI 不知道具体需求
