⚙️ RAG 工程实践

1. 背景与动机

前面几章讲了向量检索的理论基础，本章聚焦于 如何构建一个生产级 RAG 系统。RAG 的核心挑战不是算法，而是工程——数据处理、检索策略、Prompt 设计的每一个环节都影响最终效果。

2. Chunking 策略详解

2.1 固定大小 Chunking

# 最简单的 chunking 实现
def fixed_size_chunk(text, chunk_size=512, overlap=64):
    tokens = encode(text)  # 文本 → token
    chunks = []
    for i in range(0, len(tokens), chunk_size - overlap):
        chunk = tokens[i:i + chunk_size]
        chunks.append(decode(chunk))
    return chunks

# 优点: 实现简单、可预测
# 缺点: 可能在句子中间切断，破坏语义

2.2 递归 Chunking

# LangChain 的 RecursiveCharacterTextSplitter
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,
    separators=["\n\n", "\n", ". ", " ", ""]
    # 先尝试按段落切，段落太长则按句子，句子太长则按词
)

chunks = splitter.split_text(document)
递归 Chunking 的关键技巧：1) separators 的顺序很重要——优先按段落、其次按句子、最后按词切分，保持语义完整性；2) chunk_overlap 设为 chunk_size 的 10-15%，避免信息在边界丢失；3) 对于结构化文档（如 Markdown、HTML），先按标题分割再递归切分，效果显著提升。

2.3 语义 Chunking

根据语义相似度自动检测断点，其工作原理是：先将文档按句子分割，计算相邻句子的 Embedding 相似度，在相似度骤降处（语义断点）切分。这种方法的数学推导基于余弦相似度的阈值检测——当 cos(s_i, s_{i+1}) < threshold 时判定为语义断点。

2.4 Chunk 大小选择

大小检索精度上下文完整性适用场景
128 tokens高差精确问答（FAQ）
256 tokens中高中通用 RAG
512 tokens中好文档理解
1024 tokens中低很好长文档摘要
2048 tokens低极好全文理解
经验法则：chunk 大小应该是模型上下文窗口的 1/4 到 1/8。例如，8K 上下文的模型，chunk 用 512-1024 tokens。

3.1 Hybrid Search 实现

# 使用 Milvus 实现 Hybrid Search
from pymilvus import Collection

# 1. Dense 检索 (向量)
dense_results = collection.search(
    data=[query_embedding],
    anns_field="dense_vector",
    param={"metric_type": "COSINE", "params": {"nprobe": 16}},
    limit=20
)

# 2. Sparse 检索 (BM25)
sparse_results = collection.search(
    data=[query_sparse],
    anns_field="sparse_vector",
    param={"metric_type": "IP"},
    limit=20
)

# 3. RRF 融合
def rrf_fusion(results_lists, k=60):
    scores = {}
    for results in results_lists:
        for rank, hit in enumerate(results):
            doc_id = hit.id
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: -x[1])

6. 生产部署架构

⚙️ RAG 工程实践的技术原理与深度分析： 核心概念：⚙️ RAG 工程实践是向量数据库与RAG领域的关键技术方向，其理论基础涉及信息论、优化理论和系统工程的交叉。 推导：从信息论角度，任何信息处理系统的吞吐量受限于信道容量——Shannon定理指出C=B×log₂(1+S/N)， 其中B为带宽、S/N为信噪比。在⚙️ RAG 工程实践的语境下，"信道"可以理解为模型的上下文窗口或推理管线， "信噪比"对应有用信息与噪声（无关上下文、冗余计算）的比例。 为什么这很重要？因为理解⚙️ RAG 工程实践的理论限制，才能在工程实践中做出正确的权衡—— 盲目追求指标上限往往导致过度设计，而理解根本约束才能找到最高效的优化路径。 从工程实践看，⚙️ RAG 工程实践的优化通常遵循"先测量、再分析、后优化"的循环： 首先建立完善的评测基准（覆盖质量、延迟、成本三个维度），然后通过profiling定位瓶颈环节， 最后针对性地优化——是计算受限就优化算法，是内存受限就优化数据布局，是通信受限就优化网络拓扑。 这一系统化方法论适用于向量数据库与RAG领域的所有技术问题。

通俗类比：RAG工程就像"开图书馆"——收集书籍（文档入库）→编目录（chunk+embedding）→读者来借书（检索）→馆员推荐相关书籍（重排序）→读者写读后感（LLM生成回答）。每个环节都有坑：书太厚读者看不完（chunk太大）、目录编得不好找不到书（embedding质量差）、推荐了不相关的书（检索不准）、读后感不忠实于原书（幻觉）。
常见误区：1) 只关注检索不关注生成——检索到了相关文档但LLM不看就回答，等于白检索；2) 忽略chunk策略——按固定长度切分不如按语义切分（Semantic Chunking），效果差距可达20%+；3) 不做Hybrid Search——纯向量检索漏掉关键词精确匹配，BM25+向量混合是标配；4) 跳过重排序——Cross-Encoder重排序可以把Precision@5提升15-30%。
深入探讨：RAG工程的最佳实践Pipeline：1) 文档预处理（OCR/解析/清洗）→2) 智能Chunk（Semantic Chunking按语义断句）→3) Embedding（选MTEB高分模型）→4) 混合检索（BM25+向量+RRF融合）→5) 重排序（Cross-Encoder精排）→6) 上下文压缩（只保留最相关段落）→7) LLM生成（带Faithfulness检查）。Query Rewrite（查询改写）是被低估的优化手段——"怎么了"→"系统故障原因分析"，检索效果差距巨大。

7. Query Rewrite 详解

Query Rewrite 是提升 RAG 检索质量的关键步骤。用户原始查询往往是口语化的、模糊的，直接用于检索效果不佳。通过 LLM 改写查询，可以显著提升检索召回率。

Query Rewrite 示例: 原始查询: "这个东西怎么老是出问题?" ↓ LLM 改写 改写1: "系统频繁故障的常见原因分析" 改写2: "软件系统稳定性优化方案" 改写3: "设备故障排查与维护指南" ↓ 并行检索 (每个改写查询取 Top-10) ↓ RRF 融合 → 最终 Top-5 效果对比: 直接检索 Recall@5: 0.45 Query Rewrite Recall@5: 0.72 (+60% 提升!)
Query Rewrite 注意事项：1) 改写后的查询可能偏离原始意图——需要在 Prompt 中强调"保持原始查询的核心语义"；2) 改写数量不宜过多——3-5 个子查询是最佳平衡点，过多会增加延迟和成本；3) 对于简单事实查询（如"Python创始人是谁"），改写反而可能引入噪声，建议用 Adaptive RAG 判断是否需要改写。

8. 上下文压缩与窗口管理

当检索到的文档总 token 数接近模型上下文窗口时，需要进行上下文压缩：

压缩策略原理压缩率信息损失适用场景
截断直接截取前 N 个 token可控高紧急情况
提取式摘要提取关键句子50-70%中长文档
LLM 压缩用 LLM 生成压缩版30-50%低高质量需求
Recomp训练专门的压缩模型20-40%极低生产环境
上下文窗口管理经验：1) 预留 20-30% 的上下文窗口给 LLM 的回答生成；2) 当检索结果总 token 超过窗口 70% 时触发压缩；3) 优先保留与查询最相关的片段，而不是简单截断；4) 使用"Lost in the Middle"策略——将最相关的文档放在上下文的开头和结尾，中间放次相关的。
深入探讨——RAG 系统的可观测性：生产级 RAG 系统必须具备完整的可观测性（Observability），包括三个维度：1) 指标（Metrics）——端到端延迟、检索召回率、LLM 幻觉率、用户满意度评分，建议用 Prometheus + Grafana 搭建监控大盘；2) 日志（Logging）——记录每次查询的完整链路：原始查询→改写查询→检索结果→重排序结果→最终 Prompt→LLM 输出，便于离线分析和问题排查；3) 追踪（Tracing）——用 OpenTelemetry 或 LangSmith 追踪每个步骤的耗时和中间结果，定位性能瓶颈。没有可观测性的 RAG 系统就像"黑盒"——出了问题根本无法定位是检索环节还是生成环节的故障。建议在系统上线前就埋好所有监控点，而不是出了问题再补。
