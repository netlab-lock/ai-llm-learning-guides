🔍 向量检索基础

1. 为什么需要向量检索

传统的关键词搜索（如 BM25）依赖精确的词汇匹配，无法理解语义。向量检索通过将文本映射到高维向量空间，利用向量之间的距离来衡量语义相似度，从而实现真正的语义搜索。

BM25算法的核心公式是 TF-IDF 的改进版：score(D,Q) = Σ IDF(qi) × (TF(qi,D) × (k1+1)) / (TF(qi,D) + k1 × (1 - b + b × |D|/avgdl))。其中TF是词频，IDF是逆文档频率。BM25的优势在于快速、可解释，但致命缺陷是无法处理同义词、跨语言查询和语义模糊的情况。

深入探讨——Embedding的数学原理：Embedding模型的核心是将离散的文本token映射到连续的向量空间。以BERT为例：输入文本→Tokenizer分词→WordPiece→[CLS] + tokens + [SEP]→12层Transformer编码→取[CLS]或Mean Pooling→768维向量。训练目标通常是对比学习：让语义相似的文本对向量距离近（正样本对），语义不同的文本对向量距离远（负样本对）。InfoNCE损失函数：L = -log(exp(sim(q,k+)/τ) / Σexp(sim(q,ki)/τ))，其中τ是温度参数，控制分布的尖锐程度。MTEB（Massive Text Embedding Benchmark）是评测Embedding模型的标准基准，覆盖8个任务类型（分类、聚类、检索、重排等）、56个数据集。参考文献：Muennighoff et al., "MTEB: Massive Text Embedding Benchmark", ACL 2023.

3. 距离度量详解

3.1 余弦相似度

余弦相似度 = cos(θ) = A·B / (|A| × |B|) A = [1, 0, 1], B = [1, 1, 0] A·B = 1×1 + 0×1 + 1×0 = 1 |A| = √(1²+0²+1²) = √2 |B| = √(1²+1²+0²) = √2 cos(θ) = 1 / (√2 × √2) = 0.5 特点: - 值域: [-1, 1] - 1 = 完全相同方向 - 0 = 正交 (无关) - -1 = 完全相反 - 对向量长度不敏感 (只看方向)

3.2 欧氏距离

欧氏距离 = d(A,B) = √Σ(aᵢ-bᵢ)² A = [1, 0, 1], B = [1, 1, 0] d = √((1-1)² + (0-1)² + (1-0)²) = √(0+1+1) = √2 ≈ 1.41 特点: - 值域: [0, +∞) - 0 = 完全相同 - 越大越不同 - 对向量长度敏感

3.3 内积（点积）

内积 = IP(A,B) = Σ aᵢ·bᵢ = A·B A = [1, 0, 1], B = [1, 1, 0] IP = 1×1 + 0×1 + 1×0 = 1 关键性质: - 当 |A| = |B| = 1 (归一化) 时: 内积 = 余弦相似度 - 适用: Embedding模型输出已归一化时 - 优势: 计算最快 (无需开方)

3.4 度量选择对比

度量公式适用场景特点性能
余弦相似度cos(A,B)文本相似度最常用，对长度不敏感中等
欧氏距离√Σ(aᵢ-bᵢ)²聚类、异常检测直观，对长度敏感中等
内积Σ aᵢ·bᵢ归一化向量计算最快最快
曼哈顿距离Σ|aᵢ-bᵢ|稀疏向量对异常值鲁棒快
Jaccard距离1-|A∩B|/|A∪B|集合相似度适用于稀疏特征快
常见误区：1) 以为向量维度越高越好——768维通常够用，1536维提升有限但成本翻倍；2) 忽略chunk策略——切太大检索不精确，切太小丢失上下文，500-1000 tokens是常见最佳范围；3) 混淆精确检索和近似检索——ANN牺牲少量精度换100倍速度提升，生产环境几乎都用ANN；4) 所有文本用同一种距离度量——归一化后的向量用内积最快，未归一化的用余弦相似度最合适；5) 忽略Embedding模型与距离度量的匹配——有些模型输出已L2归一化（如text-embedding-ada-002），此时余弦相似度=内积，直接用内积即可。

4. 精确搜索 vs 近似搜索

4.1 ANN算法分类

类型代表算法原理构建时间查询时间内存
基于树Annoy随机投影树森林快中低
基于图HNSW分层可导航小世界图慢极快高

基于量化PQ向量压缩（乘积量化）慢快极低

通俗类比：精确搜索就像"在图书馆里逐本翻找"——保证找到但很慢。HNSW就像"图书馆的索引卡片系统"——先按大类找，再按小类找，最后在书架上精确定位。IVF就像"图书馆分区"——先把图书馆分成100个区，找书时只在最可能的2-3个区里找，牺牲一点准确率但速度快得多。PQ就像"用缩略图代替原图"——把768维向量压缩成64维，大幅减少内存但损失一些精度。

5. RAG 全流程

5.1 Chunking策略详解

策略原理优点缺点适用场景
固定大小按token/字符数切分简单、可控可能切断语义通用场景
递归切分按分隔符层级递归保持语义完整大小不均匀结构化文档
语义切分根据Embedding变化检测语义最完整计算成本高高质量RAG
句子级按句子切分语义最细粒度上下文太少精确匹配场景
段落级按段落切分上下文充足粒度太粗长文档问答
重叠切分固定大小+overlap避免边界丢失存储冗余大多数场景
深入探讨——Chunking对检索质量的影响：Chunk Size是RAG系统最关键的超参数之一。根据LlamaIndex 2024年的基准测试：Chunk Size从128→256→512→1024 tokens，Recall@10分别为0.72→0.81→0.86→0.84，呈现先升后降的趋势。原因是：太小的chunk丢失上下文（"他"不知道指谁），太大的chunk引入噪声（无关信息稀释了关键信息）。最佳实践是使用Overlapping Chunking（重叠切分），overlap通常设为chunk_size的10-20%。例如chunk_size=512, overlap=64 tokens。另一个关键参数是top_k：检索太少可能遗漏关键文档，检索太多引入噪声。典型值top_k=5-10，配合Reranker精排到top_3-5。参考文献：LlamaIndex Documentation, "Chunking Strategies for RAG", 2024.

7.2 存储优化策略

向量存储优化层次: ┌─────────────────────────────────────────────────────┐ │ Level 1: 原始向量 (float32) │ │ 大小: N × D × 4 bytes │ │ 例: 100万 × 768 × 4 = 3 GB │ ├─────────────────────────────────────────────────────┤ │ Level 2: float16 半精度 │ │ 大小: N × D × 2 bytes = 1.5 GB │ │ 精度损失: <0.5% 召回率下降 │ ├─────────────────────────────────────────────────────┤ │ Level 3: Scalar Quantization (SQ8) │ │ 大小: N × D × 1 byte = 750 MB │ │ 精度损失: <1% 召回率下降 │ ├─────────────────────────────────────────────────────┤ │ Level 4: Product Quantization (PQ) │ │ 大小: N × M bytes (M通常64-128) = 64-128 MB │ │ 精度损失: 3-5% 召回率下降 │ ├─────────────────────────────────────────────────────┤ │ Level 5: Binary Quantization (BQ) │ │ 大小: N × D/8 bytes = 96 MB │ │ 精度损失: 5-10% 召回率下降 │ └─────────────────────────────────────────────────────┘ 推荐策略: - 内存充足 → float32 (最高精度) - 内存紧张 → SQ8 (最佳平衡) - 极端受限 → PQ (最大压缩)

7.3 批量查询优化

向量数据库的批量查询比逐条查询效率高得多。原因：1）CPU/GPU可以利用SIMD指令并行计算多个查询的距离；2）索引的内存访问模式更友好（缓存命中率高）；3）网络开销被摊薄。实测数据：单条查询QPS=5000，批量100条查询等效QPS=50000（10倍提升）。建议在生产环境中将多个并发查询聚合成批量处理。

import numpy as np
import faiss

# 向量检索批量查询示例
d = 768  # 向量维度
index = faiss.IndexHNSWFlat(d, 32)  # HNSW索引, M=32

# 批量添加向量
vectors = np.random.randn(100000, d).astype('float32')
index.add(vectors)

# 批量查询 (比逐条快10-100倍)
queries = np.random.randn(100, d).astype('float32')
distances, indices = index.search(queries, k=10)

print(f"返回形状: {indices.shape}")  # (100, 10)
print(f"每条查询最近的3个结果: {indices[0, :3]}")
向量检索常见错误：1) 在查询时才调用Embedding模型——应该在请求入口处一次性编码，避免重复调用；2) 不设置超时——向量检索可能因为数据量大而超时，应设置合理的查询超时（如100ms）并有降级策略；3) 忽略向量维度对齐——不同Embedding模型的向量维度不同，混用会导致维度不匹配错误；4) 不监控召回率——生产环境应定期抽检召回率，当索引更新或数据分布变化时可能下降；5) 忽略归一化——有些Embedding模型输出未归一化，直接用内积计算相似度会得到错误结果，应先检查模型文档是否需要手动L2归一化。

8.2 多模态向量检索

多模态Embedding（如CLIP）将文本和图像映射到同一个向量空间，使得跨模态检索成为可能。例如输入文字"一只在草地上奔跑的狗"，可以找到语义匹配的图片，即使图片中不包含任何文字。CLIP模型的向量维度为512（ViT-B/32）或768（ViT-L/14），支持4亿图文对的训练数据覆盖了广泛的视觉概念。在实际应用中，多模态检索已广泛用于电商商品搜索（文字搜商品图）、设计素材检索（描述搜设计图）、医疗影像分析（症状描述搜CT图）等场景。

深入探讨——向量维度的诅咒与祝福：向量维度是Embedding设计的核心权衡。维度越高（如3072），表达能力越强，但存储和计算成本也越高。根据"维度诅咒"理论（Bellman, 1961），在高维空间中，所有点之间的距离趋于相等，最近邻的概念变得模糊。但实践表明，对于768-3072维的Embedding向量，这个问题并不严重，因为Embedding模型学习到的有效维度远低于原始维度（数据分布在一个低维流形上）。实验证据：text-embedding-3-large支持维度缩减（Matryoshka Representation Learning），从3072维缩减到256维，Recall@10仅下降约3%，说明有效信息集中在前几百维。参考文献：Kusupati et al., "Matryoshka Representation Learning", NeurIPS 2022.

# 为什么需要向量检索 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

核心理念：向量检索是RAG系统的"眼睛"——如果检索不到正确的文档，LLM再强也无法生成正确答案。因此，投入时间理解向量检索的原理、选择合适的Embedding模型和索引策略，是构建高质量RAG系统的第一步。
生产注意事项：1) 索引构建是离线操作，查询是在线操作——不要在查询时建立索引；2) 向量数据库的内存消耗约为：向量数 × 维度 × 4字节（float32），100万×768维≈3GB；3) Embedding模型的选择比索引算法的选择影响更大——先选好模型再优化索引；4) 混合检索（向量+BM25）是当前最佳实践，纯向量检索可能漏掉关键词精确匹配的结果。
