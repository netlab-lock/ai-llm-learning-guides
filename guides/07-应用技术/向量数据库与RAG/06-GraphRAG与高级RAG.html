2. GraphRAG (Microsoft)

2.1 核心思想

GraphRAG 由 Microsoft Research 于 2024 年提出，核心思想是：用 LLM 从文档中提取 实体和关系，构建 知识图谱，然后在图谱上做检索。与传统 RAG 的向量相似度匹配不同，GraphRAG 通过图结构捕捉实体间的复杂关系，特别适合需要跨文档推理和全局分析的场景。

GraphRAG 知识图谱示例: 实体节点: [张三] ──CEO of──→ [A公司] │ │ │ ├──develops──→ [产品X] │ │ │ └──competes with──→ [B公司] │ ├──graduated from──→ [B大学] │ └──published──→ [论文Y] │ └──cites──→ [论文Z] 社区检测 (Leiden 算法): Community 1: [张三, A公司, B大学, 产品X, 论文Y] → 摘要: "张三是 A 公司 CEO，毕业于 B 大学， 公司开发了产品X，张三发表了论文Y" Community 2: [B公司, 论文Z] → 摘要: "B 公司是 A 公司的竞争对手..." 检索时: 根据查询找到相关社区，再从社区中提取相关实体和关系

2.2 构建流程

GraphRAG 的构建流程包含多个阶段，其核心原理是利用 LLM 的信息抽取能力从非结构化文本中提取结构化知识。整个工作机制分为四个步骤：实体抽取、关系抽取、社区检测和层次化组织。

2.4 Local Search 详解

Local Search 是 GraphRAG 中最常用的检索模式，其工作机制是：从查询中识别实体，沿知识图谱的边遍历相关实体和关系，获取相关社区摘要，最后用 LLM 综合生成回答。这种方法的原理类似于图上的 BFS/DFS 遍历，但加入了语义相关性排序。

2.6 Global Search 详解

Global Search 采用 Map-Reduce 模式，其工作原理是遍历所有社区摘要，让 LLM 对每个社区生成中间分析结果，最后综合所有中间结果生成全局答案。这种方法的数学推导本质上是分治策略——将全局问题分解为多个局部子问题，再合并结果。

3. Agentic RAG (2026 热点)

Agentic RAG 是 2026 年最热门的 RAG 方向，核心思想是让 Agent 自主决定如何检索。与传统 RAG 的固定 Pipeline 不同，Agentic RAG 赋予 LLM 推理和决策能力——Agent 可以分析问题复杂度、选择合适的检索工具、评估中间结果并决定是否需要进一步检索。这种"思考-行动-观察"的循环使得 RAG 系统能够处理更复杂的多步推理任务。

Agentic RAG vs 传统 RAG: 传统 RAG (固定 Pipeline): 查询 → 检索 → 生成 → 输出 特点: 简单快速，但无法处理复杂问题 Agentic RAG (动态决策): 查询 → Agent 思考 → 选择工具 → 执行 → 观察 ↑ │ └────── 需要更多信息? ──────┘ 特点: 灵活强大，但延迟和成本更高 适用场景对比: 简单事实问答 → 传统 RAG 足够 多步推理 → Agentic RAG 全局分析 → GraphRAG 复杂研究 → Agentic RAG + GraphRAG

4. Self-RAG

4.1 Self-RAG 原理

Self-RAG (Asai et al., 2023) 让 LLM 自己判断是否需要检索，以及检索结果是否有用。其核心原理是通过特殊的"反思 token"（Retrieve、IsRel、IsSup、IsUse）来控制检索流程。

4.2 Self-RAG 代码示例

# Self-RAG 的简化实现
def self_rag(query, llm, retriever, max_iterations=3):
    # Step 1: LLM 判断是否需要检索
    retrieve_decision = llm.generate(
        f"问题: {query}\n需要检索外部知识吗? (Yes/No)"
    )
    
    if "No" in retrieve_decision:
        return llm.generate(query)  # 直接回答
    
    # Step 2: 检索并评估相关性
    for _ in range(max_iterations):
        docs = retriever.search(query, top_k=5)
        
        # 评估检索结果相关性
        relevance = llm.generate(
            f"以下文档与问题相关吗?\n问题: {query}\n文档: {docs}"
        )
        
        if "相关" in relevance:
            # Step 3: 生成并验证
            answer = llm.generate(f"基于{docs}回答{query}")
            support = llm.generate(
                f"回答有文档支持吗?\n回答: {answer}\n文档: {docs}"
            )
            if "有支持" in support:
                return answer
    
    return llm.generate(query)  # 降级到直接回答

5. Adaptive RAG

Adaptive RAG 的核心思想是根据查询的复杂度动态选择最优的检索策略，而不是对所有查询使用相同的 Pipeline。通过分类器判断查询类型，可以同时优化检索质量和系统效率。

多跳推理中迭代检索"A 的 CEO 毕业大学?"
全局概述高GraphRAG"领域趋势?"
无需检索低直接生成"1+1=?"

6. RAG-Fusion

RAG-Fusion 通过 LLM 生成多个子查询来扩展检索覆盖范围，再用 RRF 融合所有结果。与简单的 Query Rewrite 不同，RAG-Fusion 特别强调子查询的多样性和结果融合的权重分配。

# 传统 RAG 的局限 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

通俗类比：GraphRAG就像给RAG装了一个"关系地图"——普通RAG只能找"和你问的最像的文档"（点对点），GraphRAG能找"和你问的相关的所有实体和关系"（网络关系）。就像查案——普通警察只看直接证据（向量检索），侦探会画人物关系图（知识图谱）找出隐藏的关联。GraphRAG对"多跳推理"问题效果提升显著。
常见误区：1) 以为GraphRAG能替代普通RAG——GraphRAG适合"关系密集型"问题（多实体、多跳推理），简单事实问答用普通RAG更高效；2) 忽略图构建成本——构建知识图谱需要实体抽取+关系抽取，LLM调用成本是普通RAG的5-10倍；3) 混淆GraphRAG和知识图谱——GraphRAG=知识图谱+向量检索+社区聚类，比纯知识图谱更灵活。
深入探讨：GraphRAG的核心流程：1) 实体抽取（从文档中识别实体）→2) 关系抽取（识别实体间关系）→3) 图构建（构建知识图谱）→4) 社区检测（Leiden算法聚类相关实体）→5) 社区摘要（为每个社区生成摘要）→6) 检索时先找相关社区再找相关实体。Agentic RAG是更高级的形态——Agent自主决定是否需要检索、用什么策略检索、检索几次，比固定Pipeline更灵活但成本更高。

8. GraphRAG 实战代码

# 使用 Microsoft GraphRAG 库构建知识图谱
import graphrag
from graphrag.query import GlobalSearch, LocalSearch
from graphrag.query.llm.oai.chat_openai import ChatOpenAI

# 初始化 LLM
llm = ChatOpenAI(model="gpt-4o", api_key="your-key")

# Local Search: 针对具体实体的问题
local_search = LocalSearch(
    llm=llm,
    context_data="entity_data",  # 已构建的图谱数据
    community_level=2
)
result = local_search.search("张三的教育背景是什么?")
print(result.response)

# Global Search: 针对全局性问题
global_search = GlobalSearch(
    llm=llm,
    context_data="community_data",  # 社区摘要数据
    map_prompt="分析以下社区摘要中的关键趋势...",
    reduce_prompt="综合所有分析结果..."
)
result = global_search.search("这个领域的主要发展趋势?")
print(result.response)
GraphRAG 部署建议：1) 图构建是一次性成本（但较大），构建完成后查询成本与普通 RAG 相当；2) 建议先用小规模文档（100-1000 篇）验证效果，再扩展到全量数据；3) 社区层级（community_level）设为 2-3 级，过细会导致摘要过多、过粗会丢失细节；4) 定期增量更新图谱——新增文档只需抽取新实体和关系，不需要重建整个图谱。

9. RAG 架构选型对比

架构适用场景构建成本查询成本多跳推理全局问题
传统 RAG简单事实问答低低❌❌
Hybrid RAG关键词+语义混合低中❌❌
Agentic RAG复杂多步推理中高✅部分
GraphRAG关系密集型分析高中✅✅
GraphRAG + Agent全场景覆盖极高高✅✅
RAG 架构选型原则：1) 80% 的场景用传统 RAG + Hybrid Search 就够了——不要过度工程化；2) 只有当你的问题确实涉及多实体关系推理时才考虑 GraphRAG；3) Agentic RAG 适合需要多轮检索和推理的复杂场景（如研究报告生成）；4) 始终以"简单方案能否解决"为起点，逐步升级架构复杂度。
高级 RAG 的成本陷阱：1) GraphRAG 的图构建需要大量 LLM 调用——1000 篇文档的构建成本约 $50-200（取决于模型）；2) Agentic RAG 每次查询可能触发 3-10 次 LLM 调用，延迟 5-30 秒；3) Multi-hop RAG 的迭代检索可能导致"错误传播"——第一步检索错误会影响后续所有步骤；4) 建议在开发环境中用小模型（如 GPT-4o-mini）做原型验证，生产环境再切换大模型。
深入探讨——高级 RAG 的技术演进路线：RAG 技术从 2023 年的 Naive RAG（简单检索+生成）到 2024 年的 Advanced RAG（Hybrid Search + Reranking + Query Rewrite）再到 2025-2026 年的 Modular RAG 和 Agentic RAG，经历了三代演进。核心趋势是：1) 从固定 Pipeline 到动态决策——Agent 根据查询复杂度自动选择检索策略（简单查询直接检索、复杂查询迭代检索、全局问题用 GraphRAG）；2) 从单一数据源到多源融合——向量库 + 知识图谱 + SQL 数据库 + API 的统一检索；3) 从无状态到有状态——支持多轮对话的上下文管理和记忆机制；4) 从人工评估到自动评测——用 LLM-as-Judge 自动评估 Faithfulness、Relevance 和 Answer Quality。未来的 RAG 系统将是"自适应"的——根据查询特征、数据特征和性能要求自动选择最优的检索和生成策略。
