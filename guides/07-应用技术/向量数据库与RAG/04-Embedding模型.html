📐 Embedding 模型

Embedding 模型是 RAG 系统的"语义引擎"——它将文本转化为稠密向量，决定了"什么算相似"。选择合适的 Embedding 模型对检索质量有决定性影响。本章从模型原理、训练方法、评测体系到选型实战，全面讲解 Embedding 模型的技术栈。

1. Embedding 模型的作用与原理

1.1 为什么需要 Embedding

在 RAG 系统中，用户查询和文档需要被转化为同一种"语言"才能进行相似度计算。Embedding 模型就是这个翻译官——它将自然语言文本映射到高维向量空间，使得语义相近的文本在向量空间中也相近。

Embedding 模型在 RAG 中的位置: 用户查询: "什么是 Transformer 架构?" ↓ ┌──────────────────────────────────┐ │ Embedding 模型 (BGE-M3) │ │ 文本 → 1024维向量 │ └──────────────────────────────────┘ ↓ 查询向量: [0.12, -0.34, 0.56, ..., 0.78] ↓ ↑ 同一个模型 ↓ ┌──────────────────────────┐ ↓ │ 文档 Chunk 的向量 (预计算) │ ↓ │ "Transformer 是一种..." │ ↓ │ → [0.15, -0.31, 0.52,...] │ ↓ └──────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ 向量数据库 (Milvus / Qdrant) │ │ 计算余弦相似度 → 返回 Top-K │ └──────────────────────────────────┘ ↓ 检索结果: 最相关的文档片段

3. Embedding 训练方法

3.1 对比学习 (Contrastive Learning)

现代 Embedding 模型的核心训练方法是对比学习——通过让模型区分"相似"和"不相似"的文本对，学习高质量的语义表示。

对比学习训练过程: 正样本对 (语义相似): Query: "如何训练大模型" Doc: "LLM 训练完整指南" → 让这两个向量的余弦相似度 → 1.0 负样本对 (语义不同): Query: "如何训练大模型" Doc: "今天天气真不错" → 让这两个向量的余弦相似度 → 0.0 InfoNCE 损失函数: L = -log( exp(sim(q, k⁺)/τ) / Σᵢ exp(sim(q, kᵢ)/τ) ) 其中: q = 查询向量 k⁺ = 正样本向量 kᵢ = 所有样本向量 (1正 + N-1负) τ = 温度参数 (通常 0.01-0.1) sim = 余弦相似度 含义: 拉近正样本对，推远负样本对

3.2 硬负样本挖掘 (Hard Negative Mining)

随机负样本太容易区分，模型学不到细粒度语义。硬负样本是"看起来像但实际不相关"的样本，对提升模型辨别力至关重要：

硬负样本类型获取方法难度效果
BM25 高分但语义不相关BM25 检索 Top-100 中排除正样本中提升关键词区分
同主题不同问题同一领域的其他问题高提升细粒度语义
LLM 生成的对抗样本用GPT生成与正样本相似但错误的文本极高大幅提升泛化
跨语言翻译对同一句子的不同语言翻译高提升跨语言能力

3.3 多任务训练

BGE-M3 等先进模型采用多任务训练策略，同时在多种任务上训练：

  • 检索任务（Retrieval）：query-document 匹配
  • 分类任务（Classification）：文本分类
  • 聚类任务（Clustering）：相似文本聚类
  • 重排序任务（Reranking）：精排相关性
# 使用 FlagEmbedding 微调 BGE 模型
from FlagEmbedding import FlagModel
import torch

# 加载预训练模型
model = FlagModel(
    'BAAI/bge-large-zh-v1.5',
    query_instruction_for_retrieval="为这个句子生成表示以用于检索相关文章：",
    use_fp16=True
)

# 编码文本
queries = ["什么是Transformer?", "如何训练大模型?"]
documents = ["Transformer是...", "训练大模型需要..."]

q_embeddings = model.encode_queries(queries)   # 查询编码 (带指令)
d_embeddings = model.encode(documents)          # 文档编码 (不带指令)

# 计算相似度
similarities = q_embeddings @ d_embeddings.T
print(similarities)
# [[0.85, 0.12],   "什么是Transformer?" 与两个文档的相似度
#  [0.15, 0.82]]   "如何训练大模型?" 与两个文档的相似度

4. MRL (Matryoshka Representation Learning)

4.1 MRL 原理

MRL（套娃表示学习）是 OpenAI 在 text-embedding-3 系列中引入的技术。核心思想是：训练时让模型同时学习不同维度的表示，推理时可以灵活选择维度，无需重新训练。

MRL 训练过程: 训练阶段 (同时优化多个维度): 原始输出: [0.12, -0.34, 0.56, 0.78, ..., 0.45] (3072维) ↓ 截断 ↓ 截断 ↓ 截断 256维损失: L_256 ← 前256维也要能区分正负样本 512维损失: L_512 ← 前512维也要能区分正负样本 1024维损失: L_1024 ← 前1024维也要能区分正负样本 3072维损失: L_3072 ← 完整维度的损失 总损失: L = L_256 + L_512 + L_1024 + L_3072 推理阶段 (灵活选择维度): 需要高精度 → 使用 3072 维 需要低成本 → 使用 256 维 (存储减少12倍!) 平衡方案 → 使用 1024 维

4.2 MRL 代码示例

# MRL 维度截断示例
import numpy as np

# 原始 3072 维 embedding
embedding_full = np.random.randn(3072).astype(np.float32)

# MRL 支持直接截断 (前 N 维即可)
embedding_256  = embedding_full[:256]   # 存储: 1 KB
embedding_1024 = embedding_full[:1024]  # 存储: 4 KB
embedding_3072 = embedding_full         # 存储: 12 KB

# 归一化后计算相似度 (需要对截断后的向量重新归一化)
embedding_256  = embedding_256 / np.linalg.norm(embedding_256)
embedding_1024 = embedding_1024 / np.linalg.norm(embedding_1024)

# 典型精度对比 (MTEB Retrieval):
# 256 维: 54.8  (存储减少12倍, 精度下降约9%)
# 512 维: 58.2  (存储减少6倍, 精度下降约6%)
# 1024维: 62.1  (存储减少3倍, 精度下降约3%)
# 3072维: 64.6  (完整精度)
常见误区：1) 以为截断维度等同于重新训练小模型——MRL 截断的是前 N 维，而不是 PCA 降维，前 N 维已经包含了最重要的语义信息；2) 忽略截断后需要重新归一化——截断后向量的 L2 范数不再是 1，必须重新归一化才能用余弦相似度；3) 在所有场景都用最低维度——256 维在简单问答中够用，但在法律、医疗等需要精细区分的场景可能不够。

6.2 指令前缀的使用

# 非对称检索: query 加指令，doc 不加
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-zh-v1.5")

# 查询编码: 加指令前缀
query = "为这个句子生成表示以用于检索相关文章：什么是Transformer?"
q_emb = model.encode(query)

# 文档编码: 不加指令
doc = "Transformer是一种基于自注意力机制的深度学习架构..."
d_emb = model.encode(doc)

# BGE-M3 不需要手动加前缀 (自动处理)
model_m3 = SentenceTransformer("BAAI/bge-m3")
q_emb_m3 = model_m3.encode("什么是Transformer?")  # 无需加前缀
d_emb_m3 = model_m3.encode(doc)
深入探讨：Embedding 模型的训练核心是"对比学习"——让相似文本的向量靠近，不相似文本的向量远离。关键技术包括：1) InfoNCE损失函数——拉近正样本、推远负样本，温度参数τ控制分布的"锐度"；2) Hard Negative Mining——用BM25高分低语义相关的样本或LLM生成的对抗样本提升辨别力；3) MRL——训练时同时优化多个维度的损失，使低维度也具有良好的语义区分能力；4) Instruction Tuning——为query添加任务指令前缀，使模型能区分检索、分类、聚类等不同任务的编码需求。

8.2 微调代码示例

# 使用 sentence-transformers 微调 Embedding 模型
from sentence_transformers import SentenceTransformer, InputExample, losses
from torch.utils.data import DataLoader

# 准备训练数据 (query, positive_doc, negative_doc)
train_examples = [
    InputExample(texts=["医疗诊断AI", "AI辅助医学影像诊断系统", "股票投资策略"]),
    InputExample(texts=["机器学习模型部署", "模型上线运维指南", "烹饪食谱大全"]),
]

# 加载预训练模型
model = SentenceTransformer("BAAI/bge-large-zh-v1.5")

# 配置训练
train_dataloader = DataLoader(train_examples, batch_size=16, shuffle=True)
train_loss = losses.TripletLoss(model=model)  # 三元组损失

# 微调
model.fit(
    train_objectives=[(train_dataloader, train_loss)],
    epochs=3,
    warmup_steps=100,
    output_path="./bge-medical-finetuned"
)
常见误区：1) 微调数据太少——至少需要几千条高质量样本，数据量不足会导致过拟合；2) 忘记冻结底层——小数据微调时应冻结 Transformer 底层，只训练顶层和投影层；3) 不做评估——微调后必须在测试集上验证，确保没有退化。建议保留 10-20% 数据作为验证集。
Embedding 模型部署优化：1) 使用 ONNX Runtime 或 TensorRT 加速推理，速度可提升 2-5 倍；2) 批量编码时 batch_size 设为 32-128，充分利用 GPU 并行；3) 对于高并发场景，部署多个模型实例并通过负载均衡分发请求；4) 使用 FP16 或 INT8 量化可将模型大小减半且精度损失 <1%。
深入探讨——Embedding 空间的几何结构：高质量的 Embedding 空间具有"语义流形"特性——相关概念在高维空间中形成连续的子流形。例如，所有编程语言相关的查询会聚集在一个区域，所有医学相关的查询聚集在另一个区域。模型的"泛化能力"本质上是在这些流形上做插值——训练时没见过的查询，只要落在正确的流形附近，也能被正确检索。这就是为什么 Hard Negative Mining 如此重要：它帮助模型定义流形的边界，防止不同语义区域重叠。MRL 的成功也在于此：前 N 维已经编码了流形的主要结构，高维只是提供更精细的边界区分。

  3. 核心原理: 对比分析需要从多个维度展开：性能（延迟、吞吐）、精度（量化损失）、成本（GPU小时）、复杂度（运维难度）、适用场景（prompt长度、并发模式）。不同技术方案各有优劣：如一体架构简单但GPU利用率低，分离架构高效但引入传输开销。选择依据是业务需求：如果SLA要求TTFT<200ms且QPS>100，应选择分离架构+Prefix Caching；如果QPS<10且prompt短，一体架构即可满足。
    关键要点: ①多维度对比评估框架；②场景匹配和方案选择；③实验验证和性能benchmark；④渐进式优化路径；⑤技术债和可维护性考量。
    实际案例: 某团队对比了三种方案：A)一体架构+FP16 B)一体架构+FP8+Prefix Caching C)分离架构+FP8+Prefix Caching。方案A月度成本$50K，TTFT=500ms；方案B成本$30K，TTFT=200ms；方案C成本$25K，TTFT=150ms。考虑到运维复杂度和团队能力，选择了方案B作为过渡方案，半年后迁移到方案C。
