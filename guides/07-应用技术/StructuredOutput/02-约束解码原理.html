⚙️ 02 · 约束解码原理

中级 核心

1. 核心思想

约束解码的核心：在每一步解码时，根据当前语法状态，屏蔽不合法的 token。这保证了输出的格式正确性，而不需要后处理或重试。

logits[illegal] = -inf → softmax → 只有合法 token 有非零概率

直觉理解

想象你在走迷宫: 自由生成 = 在每个路口随意选方向 (可能走错) 约束解码 = 在每个路口只开放通往终点的路 (保证到达) LLM 的词表 = 50,000 个 token 在 JSON 的某个位置, 可能只有 100 个 token 是合法的 约束解码 = 把 49,900 个非法 token 的概率设为 0

3. FSM 的关键数据结构

from typing import Dict, List, Set

class FSMState:
    """FSM 的一个状态"""
    def __init__(self, state_id: int):
        self.state_id = state_id
        self.transitions: Dict[str, int] = {}  # 字符 → 下一状态
        self.is_final: bool = False  # 是否为接受状态
        self.token_cache: Dict[int, List[int]] = {}  # 缓存合法 token

class ConstrainedFSM:
    """基于 FSM 的约束解码器"""
    def __init__(self, states: List[FSMState]):
        self.states = states
        self.current_state = 0

    def get_valid_token_ids(self, tokenizer, vocab_size: int) -> List[int]:
        """获取当前状态下所有合法的 token ID"""
        # 先检查缓存
        if self.current_state in self.states[self.current_state].token_cache:
            return self.states[self.current_state].token_cache[self.current_state]

        valid = []
        for token_id in range(vocab_size):
            token_str = tokenizer.decode([token_id])
            if self._can_consume(token_str):
                valid.append(token_id)

        # 缓存结果
        self.states[self.current_state].token_cache[self.current_state] = valid
        return valid

    def _can_consume(self, text: str) -> bool:
        """检查文本是否能从当前状态消费"""
        state = self.current_state
        for ch in text:
            if ch not in self.states[state].transitions:
                return False
            state = self.states[state].transitions[ch]
        self.current_state = state
        return True

    def advance(self, token_str: str):
        """根据选中的 token 更新状态"""
        for ch in token_str:
            self.current_state = self.states[self.current_state].transitions.get(ch, -1)

4. CFG 引导解码（上下文无关文法）

为什么需要 CFG？

FSM 只能处理正则语言（Type-3 语法），无法处理递归结构：

下推自动机 (Pushdown Automata, PDA)

GBNF 格式 (llama.cpp)

# GBNF (GGML BNF) 文法定义
# llama.cpp 使用的 CFG 格式

root   ::= object
object ::= "{" ws members "}"
members ::= pair ("," ws pair)*
pair   ::= string ":" ws value
value  ::= string | number | object | array | "true" | "false" | "null"
string ::= "\"" chars "\""
chars  ::= [^"\\]+
number ::= "-"? ("0" | [1-9] [0-9]*) ("." [0-9]+)?
array  ::= "[" ws values "]"
values ::= value ("," ws value)*
ws     ::= [ \t\n]*

# 使用方式 (llama.cpp):
# ./main -m model.gguf --grammar-file json.gbnf

组件开销优化方法

FSM 状态更新每步 ~0.001ms查表操作
总体+5-15% 延迟Compressed FSM 可降至 +2-5%

🔬 为什么开销这么小？

约束解码的主要操作是修改 logits（一个向量加法），这在 GPU 上几乎免费。真正的开销来自 CPU 端的 FSM 状态查询和 token mask 计算。XGrammar 通过预计算 99% 的 token mask，将这个开销降到了几乎为零。

8. 实际实现：Transformers + Outlines

from transformers import LogitsProcessor
import interegular  # 正则→FSM 编译库

class FSMLogitsProcessor(LogitsProcessor):
    """Transformers 的 FSM 约束解码器"""

    def __init__(self, regex_pattern: str, tokenizer):
        # 编译正则为 FSM
        self.fsm = interegular.parse_pattern(regex_pattern).to_fsm()
        self.current_state = self.fsm.initial
        self.tokenizer = tokenizer

        # 预计算 token→FSM 转移表
        self.token_to_next_state = {}
        vocab = tokenizer.get_vocab()
        for token_str, token_id in vocab.items():
            for state in self.fsm.states:
                next_state = self._try_consume(state, token_str)
                if next_state is not None:
                    self.token_to_next_state[(state, token_id)] = next_state

    def __call__(self, input_ids, scores):
        """每步解码时调用"""
        # 获取当前状态的合法 token
        valid_mask = scores.new_full(scores.shape, float('-inf'))
        for token_id in range(scores.shape[-1]):
            key = (self.current_state, token_id)
            if key in self.token_to_next_state:
                valid_mask[token_id] = 0  # 合法

        # 应用 mask
        scores = scores + valid_mask
        return scores

    def advance(self, token_id):
        """根据选中的 token 更新状态"""
        key = (self.current_state, token_id)
        if key in self.token_to_next_state:
            self.current_state = self.token_to_next_state[key]

通俗类比：约束解码就像给AI装了一个"语法检查器"——每次AI要输出一个token时，检查器会说"这个位置只能放数字"或"这个位置必须是引号"。就像填答题卡——你只能在规定位置涂黑，不能涂到外面去。这样AI的输出就100%符合JSON格式，不可能出现"少了一个引号"这种低级错误。
常见误区：1) 以为约束解码很慢——现代实现（如Outlines）通过预编译状态机，额外开销仅5-15%；2) 以为约束解码会降低质量——它只约束格式不约束内容，模型的语义理解能力不受影响；3) 忽略正则vs JSON Schema的选择——简单模式用正则更高效，复杂嵌套结构用JSON Schema更清晰。
通俗类比——交通信号灯：约束解码就像给AI的输出通道装上了交通信号灯系统。在每个十字路口（token生成位置），红灯代表"此路不通"（非法token被设为-logits=-inf），绿灯代表"可以通行"（合法token保持原始概率）。FSM就是那个智能交通控制系统，它知道每个路口该亮什么灯。没有信号灯（无约束），车辆（token）可能逆行、闯红灯，导致交通事故（JSON格式错误）。有了信号灯，交通100%有序。
实用技巧——调试约束解码问题：当约束解码的输出不符合预期时，排查步骤：1) 检查正则/Schema是否正确表达了你的意图（用regex101.com测试正则）；2) 确认tokenizer的行为——同一个词在不同tokenizer下可能被切分为不同token；3) 检查FSM状态转移日志，看哪一步偏离了预期路径；4) 对于中文等多字节字符，确保tokenizer的special tokens不会干扰FSM状态。
性能陷阱——状态爆炸：复杂JSON Schema（尤其是含oneOf/anyOf/嵌套数组）编译出的FSM可能有数万个状态。每个状态需要预计算合法token集合（遍历50000词表），初始化时间可能长达数秒。解决方案：1) 尽量简化Schema，避免不必要的嵌套；2) 使用缓存——相同Schema只编译一次；3) 生产环境用SGLang/XGrammar的压缩FSM，可减少90%状态数；4) 对实时性要求高的场景，考虑预热（warmup）阶段提前编译所有常用Schema。
深入探讨——约束解码的理论基础与前沿：约束解码的核心是"有限状态机（FSM）"——将JSON Schema编译成FSM，每生成一个token就推进状态，只允许合法的下一个token。数学上，这等价于在每一步对logits向量施加一个二值掩码M：logits' = logits + M，其中M[i]=0（合法）或M[i]=-inf（非法）。Outlines框架的创新在于：1) 预编译FSM并缓存，避免每次重新编译；2) 用index加速token到FSM状态的映射；3) 支持正则和JSON Schema两种约束语法。前沿研究方向包括：① SGLang的Compressed FSM（将线性路径合并，减少状态数90%+）；② XGrammar的Context-Independent/Dependent分离（99% token预计算，10x加速）；③ 基于CFG的递归约束（下推自动机处理嵌套JSON）。Willard & Louf (2023) 的论文证明，预计算Token Index后，约束解码的额外延迟开销仅为5-15%，远低于后处理+重试方案。
