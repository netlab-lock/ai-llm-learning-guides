⚡ 04 · SGLang 的结构化输出

进阶 性能

1. SGLang 的约束解码方案

SGLang（LMSYS 开发）实现了独立的约束解码系统，不依赖 Outlines。核心创新是 Compressed FSM + Jump-forward Decoding。参考：LMSYS Blog。

1.1 SGLang 的定位

SGLang 不仅仅是一个约束解码库，它是一个完整的 LLM 推理引擎，类似于 vLLM。其特点是将约束解码深度集成到推理引擎中，而非像 Outlines 那样作为外部库挂载。这种深度集成带来了性能优势：约束解码可以利用引擎内部的 KV Cache、调度器等基础设施。

SGLang 架构全景: ┌─────────────────────────────────────────────────────────┐ │ SGLang 推理引擎 │ ├─────────────────────────────────────────────────────────┤ │ │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ 前端 DSL │ │ RadixAttention│ │ 调度器 │ │ │ │ (sgl.function)│ │ (KV Cache复用)│ │ (批处理) │ │ │ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ │ │ │ │ │ │ │ ┌──────▼─────────────────▼──────────────────▼───────┐ │ │ │ 推理核心 (CUDA Kernel) │ │ │ └──────┬────────────────────────────────────────────┘ │ │ │ │ │ ┌──────▼──────────────────────────────────────────┐ │ │ │ 约束解码引擎 │ │ │ │ ┌─────────────┐ ┌──────────┐ ┌───────────┐ │ │ │ │ │Compressed │ │ XGrammar │ │ Regex/EBNF│ │ │ │ │ │FSM │ │ (CFG) │ │ 支持 │ │ │ │ │ └─────────────┘ └──────────┘ └───────────┘ │ │ │ └─────────────────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────┘ ▲ ▲ ▲ │ │ │ JSON Schema EBNF Grammar 自定义正则

2. Compressed FSM

2.1 普通 FSM 的问题

普通 FSM 的冗余: 生成 {"name": "Alice", "age": 25} 普通 FSM: 每个字符一个状态 s0 --'{'→ s1 --'"'→ s2 --'n'→ s3 --'a'→ s4 s4 --'m'→ s5 --'e'→ s6 --'"'→ s7 --':'→ s8 s8 --'"'→ s9 --'A'→ s9 --'l'→ s9 --'i'→ s9 ... 问题: s2→s3→s4→s5→s6 是一条「确定性路径」 只有一种合法的 next token ("name") 但普通 FSM 还是每步都要查询

普通 FSM 的核心问题是：JSON 格式中大量的字符是结构性字符（引号、冒号、花括号、逗号），这些字符在给定上下文中是唯一确定的。但普通 FSM 并不利用这一特性，仍然在每个 token 生成后进行状态查询，浪费了大量计算资源。

2.2 压缩策略

Compressed FSM: 合并非分支路径 s0 --'{'→ s1 s1 --'"name"':→ s7 (7 个字符合并为 1 条边!) s7 --'"'→ s9 s9 --"Alice"→ s10 (5 个字符合并!) s10 --'",'→ s12 s12 --'"age"':→ s17 s17 --25→ s20 s20 --'}'→ 终态 状态数: 10 (vs 普通 FSM 的 ~25) 查询次数: 大幅减少 压缩原理: ┌────────────────────────────────────────────────────┐ │ 普通FSM路径: s --"→ s --n→ s --a→ s --m→ s --e→ s│ │ 压缩后路径: s --"name"→ s │ │ │ │ 条件: 这段路径上每个节点都只有1个出边 │ │ 即「确定性路径」→ 可以安全合并 │ └────────────────────────────────────────────────────┘

2.3 压缩算法

# Compressed FSM 构建算法
def compress_fsm(fsm):
    """合并非分支的连续路径"""
    compressed = {}

    for state in fsm.states:
        # 检查是否为「确定性路径」
        # 即只有一个合法的 next transition
        if len(state.transitions) == 1:
            char, next_state = list(state.transitions.items())[0]
            # 沿着确定性路径一直走
            path = [char]
            current = next_state
            while len(current.transitions) == 1:
                char, current = list(current.transitions.items())[0]
                path.append(char)
            # 合并为一条边
            compressed[state] = {''.join(path): current}
        else:
            # 分支点，保持不变
            compressed[state] = state.transitions

    return compressed

2.4 压缩率分析

Compressed FSM 的压缩率取决于 JSON 结构的"确定性比例"。对于典型的 API 响应 JSON：

JSON 类型原始状态数压缩后状态数压缩率加速比
简单对象 (2-3字段)~25~868%3x
数组 [obj, obj, ...]~60~1575%4x
深层嵌套 (4层)~200~8060%2.5x
纯值对象 {"a":1,"b":2}~30~680%5x
通俗类比：SGLang 的 Compressed FSM 就像"高速公路 ETC 车道"——普通车道（逐字符检查）每辆车都要停下来验证身份，ETC 车道（Compressed FSM）提前缓存了车辆信息，通过时只需扫一下瞬间放行。JSON 中的结构性字符（引号、冒号、花括号）就像"已知车牌"，可以直接跳过验证；只有"值"部分（如用户名、年龄）才是真正需要模型决策的。

3. Jump-forward Decoding

Compressed FSM 的最大优势：对于确定性路径，直接跳过解码，用 prefill 填充。

3.1 工作原理

Jump-forward Decoding 详细流程: 输入: JSON Schema + 用户 prompt 当前状态: s0 (开始) Step 1: 正常解码 (分支点) LLM forward → 输出 '{' (分支点，必须解码) FSM: s0 → s1 Step 2: Jump-forward (确定性路径) FSM 分析: s1 → '"name":' → s7 是确定性路径 直接 prefill '"name":' (不经过 LLM decode!) FSM: s1 → s7 节省: 7 次 LLM forward Step 3: 正常解码 (分支点) LLM forward → 输出 'Alice' (值部分，必须解码) FSM: s9 → s10 Step 4: Jump-forward (确定性路径) FSM: s10 → '",' → '"age":' 是确定性路径 直接 prefill '","age":' FSM: s10 → s17 节省: 6 次 LLM forward Step 5: 正常解码 (分支点) LLM forward → 输出 '25' (值部分，必须解码) FSM: s18 → s20 Step 6: Jump-forward (确定性路径) FSM: s20 → '}' 是确定性路径 直接 prefill '}' FSM: s20 → 终态 总结: 只需 3 次 LLM forward (vs 普通 ~20 次) 加速: ~6.7x

3.2 为什么 Jump-forward 不影响生成质量？

一个关键问题是：跳过 LLM decode 直接 prefill 确定性 token，会不会影响后续生成的质量？答案是不会，原因如下：

1) 确定性 token 的值在给定 Schema 下是唯一合法的，模型不可能生成其他值。因此直接填充等价于模型"必然"会生成的 token。

2) Prefill 操作会正确更新 KV Cache，后续的 LLM forward 可以看到完整的上下文。

3) 确定性 token 不携带语义信息（它们只是 JSON 结构），跳过解码不会丢失任何语义。

3.3 Re-tokenization 机制

# 问题: Jump-forward 生成的文本需要重新 tokenize
# 但 tokenizer 可能产生不同的分割

text = ',"name":'
# 可能的 tokenization:
tokenization_1: [',"', 'name', '":']  # 3 tokens
tokenization_2: [',', '"name', '":']  # 3 tokens (不同!)
tokenization_3: [',"name":']          # 1 token (合并!)

# SGLang 的解决方案:
# 1. 预计算所有可能的 tokenization
# 2. 选择与 FSM 状态兼容的分割
# 3. 如果不兼容，回退到普通解码

# 关键 insight: 确定性路径上的文本长度通常较短(5-20字符)
# re-tokenization 的开销远小于跳过的 LLM forward 开销
# 实测: 额外开销约 4%, 但节省了 60-80% 的 forward 次数

3.4 Jump-forward 的适用条件

Jump-forward 可用性判断: 当前FSM状态 s: │ ├── 出边数 = 1 (确定性路径) → 可以 Jump-forward │ │ │ └── 合并后的文本长度 > 1 token? │ ├── 是 → Jump-forward + Prefill │ └── 否 → 正常解码 (不值得优化) │ ├── 出边数 > 1 (分支点) → 必须解码 │ │ │ └── 典型场景: │ ├── JSON 值位置 (string/number/boolean/null) │ ├── 数组元素分隔 (逗号或右方括号) │ └── 对象结束 (逗号或右花括号) │ └── 出边数 = 0 (终态) → 生成完成 统计: 典型JSON中 ~60-80% 的token是确定性的 → Jump-forward 可以跳过这些token的解码

5. XGrammar 集成 (SGLang v0.4+)

XGrammar 是 CMU 开发的高效 CFG 约束解码库，SGLang 从 v0.4 开始将其作为默认的 grammar 后端。XGrammar 的核心创新是 Context-Independent Grammar Splitting——将 CFG 分解为与上下文无关的部分和与上下文相关的部分，前者可以预计算并缓存。

# 启动 SGLang 服务时指定 XGrammar 后端
python -m sglang.launch_server \
    --model meta-llama/Llama-3-8B-Instruct \
    --grammar-backend xgrammar

# 在 SGLang 程序中使用 JSON schema
import sglang as sgl

@sgl.function
def extract_person(s, text):
    s += sgl.system("Extract person info as JSON.")
    s += sgl.user(text)
    s += sgl.assistant(
        sgl.gen("result", json_schema={
            "type": "object",
            "properties": {
                "name": {"type": "string"},
                "age": {"type": "integer"}
            }
        })
    )

# 运行
state = extract_person.run(text="John is 25 years old")
print(state["result"])  # {"name": "John", "age": 25}

5.1 XGrammar 的技术原理

XGrammar 的 Context-Independent Grammar Splitting: 传统 CFG 约束解码: 每次生成 token 时都要遍历整个 CFG → O(|G|) 每步 XGrammar 优化: 将 CFG 分解为: ┌─────────────────────────────────────────────────┐ │ Context-Independent 部分 (预计算) │ │ → 与当前生成位置无关的规则 │ │ → 编译时一次性计算，运行时直接查表 │ │ → 覆盖 ~80% 的规则 │ ├─────────────────────────────────────────────────┤ │ Context-Dependent 部分 (运行时计算) │ │ → 需要知道当前上下文才能确定的规则 │ │ → 如: 数组元素类型取决于数组定义 │ │ → 只需处理 ~20% 的规则 │ └─────────────────────────────────────────────────┘ 结果: 每步约束计算从 O(|G|) 降到 O(1) → 解码速度提升 10-80x

7. RadixAttention 与 KV Cache 优化

SGLang 的另一个核心创新是 RadixAttention——一种基于 Radix Tree 的 KV Cache 管理策略。与 Compressed FSM/Jump-forward 结合后，产生了双重加速效果。

RadixAttention + Jump-forward 的协同效应: 请求1: {"name": "Alice", "age": 25} 请求2: {"name": "Bob", "age": 30} 普通推理: 请求1: [prefill] → [decode {"name": "Alice", "age": 25}] 请求2: [prefill] → [decode {"name": "Bob", "age": 30}] → 两次都要完整解码 SGLang (RadixAttention + Jump-forward): 请求1: [prefill] → [decode '{'] → [jump '"name":'] → [decode 'Alice'] → [jump '","age":'] → [decode 25] → [jump '}'] KV Cache 缓存: [prefill] + 结构部分 请求2: [prefill] (复用!) → [decode '{'] → [jump '"name":'] → [decode 'Bob'] → [jump '","age":'] → [decode 30] → [jump '}'] KV Cache 复用: [prefill] + 结构部分的 KV 效果: - Prefill 部分: RadixAttention 命中缓存 → 0 开销 - 结构部分: Jump-forward 直接填充 → 0 解码开销 - 只有 "值" 部分需要实际解码 - 批量请求场景下吞吐提升 5-10x

7.1 RadixAttention 的工作原理

RadixAttention 将 KV Cache 组织成一棵 Radix Tree（基数树），其中每个节点代表一段 token 序列的 KV 向量。当新请求到达时，引擎会在 Radix Tree 中查找最长的前缀匹配，复用已有的 KV Cache。对于结构化输出场景，不同请求的 JSON 结构部分（键名、括号等）通常是相同的，因此 KV Cache 命中率很高。

关键洞察：SGLang 的优势不仅在于 Compressed FSM 本身，更在于它与 RadixAttention 的集成——jump-forward 生成的确定性 token 的 KV Cache 可以被缓存和复用，进一步提升了多请求场景的性能。这种"约束解码 + 缓存优化"的组合是 SGLang 独有的优势，Outlines 作为外部库无法获得这种深度集成。

深入探讨：SGLang 的 Compressed FSM 核心思想可以推广到更一般的场景：任何"结构化文本生成"都可以利用确定性路径优化。JSON 只是一个特例——SQL 语句的 SELECT ... FROM ... WHERE 结构、HTML 的标签结构、Markdown 的格式标记，都包含大量确定性部分。理论上，Compressed FSM 可以将这些场景的解码速度提升 2-5 倍。XGrammar 进一步将这一思想扩展到完整的 CFG，支持任意上下文无关文法的约束解码。在 SGLang v0.4+ 中，XGrammar 已成为默认的 grammar 后端，Compressed FSM 作为 JSON Schema 的特化优化继续保留。两者协同工作：XGrammar 处理通用 CFG，Compressed FSM 处理 JSON Schema 的快速路径。这种分层优化策略使得 SGLang 在各种约束解码场景下都能获得最优性能。根据 LMSYS 的官方基准测试，SGLang + XGrammar 的组合在 Llama-3-8B 上可以实现每秒 10000+ token 的结构化输出吞吐量，比无约束解码仅慢 2%。

通俗类比：SGLang的RadixAttention就像图书馆的索引卡片系统——已经查过的书（KV Cache）不需要重新翻阅，直接通过索引（Radix Tree）定位即可，大幅减少了重复工作。
常见误区：不要以为启用了结构化输出就一定能获得高质量结果。约束解码只保证格式正确，不保证内容质量。如果Prompt设计不当，模型可能生成格式完美但内容空洞的JSON。
深入探讨：SGLang的Jump-forward解码在正则表达式确定性路径上可以一次生成多个token，理论加速比取决于正则的确定性比例。对于JSON Schema中的固定字段名（如"type"、"name"），加速比可达5-10x，而变量字段部分则退化为逐token生成。
