🔌 05 · Function Calling 格式

中级 实战

1. 什么是 Function Calling？

Function Calling 让 LLM 输出结构化的工具调用指令，由外部系统执行后将结果返回给 LLM。它是 LLM 从"对话助手"进化为"智能 Agent"的关键能力——没有 Function Calling，LLM 只能生成文本；有了它，LLM 可以操作数据库、调用 API、控制设备。

1.1 Function Calling 的本质

Function Calling 本质上是一种特殊的结构化输出。与普通 JSON Schema 约束不同，Function Calling 的输出包含两个部分：1) 函数名（从预定义的工具列表中选择）；2) 参数（符合该函数的参数 Schema）。这意味着它同时涉及分类（选哪个函数）和结构化生成（生成参数）两个子任务。

Function Calling 的内部机制: 用户消息 + 工具定义列表 │ ▼ ┌─────────────────────────────────────────┐ │ LLM 内部推理过程 │ │ │ │ 1. 理解用户意图 │ │ "北京天气" → 需要查天气 │ │ │ │ 2. 工具匹配 (隐式分类) │ │ 可用工具: [get_weather, search, ...] │ │ 选择: get_weather (概率最高) │ │ │ │ 3. 参数生成 (结构化输出) │ │ 从上下文提取: city="北京" │ │ Schema约東: {city: string} │ │ 输出: {"city": "北京"} │ └─────────────────────────────────────────┘ │ ▼ 特殊token触发工具调用 <|tool_call|> 或 等

1.2 为什么需要 Function Calling？

没有 Function Calling 时，开发者需要用 Prompt Engineering 让模型输出特定格式的文本，然后用正则表达式或 JSON 解析器提取工具调用意图。这种方式有三个致命缺陷：

1) 格式不稳定：模型可能输出 我想调用 get_weather("北京")，也可能是 调用天气查询，城市：北京，正则表达式难以覆盖所有情况。

2) 参数类型不保证：模型可能输出 {"age": "25"}（字符串而非整数），或者 {"city": null}。

3) 无法处理并行调用：一次请求中需要调用多个工具时，Prompt 方式很难可靠地输出多个工具调用。

通俗类比：Function Calling 就像给 AI 一个"工具箱使用手册"——手册里列出了每个工具的名称、参数、用途（Tool 定义），AI 根据用户需求决定用哪个工具、传什么参数。就像你去修车，技师不会自己造工具，而是从工具箱里选合适的扳手（选工具）然后拧螺丝（调用）。关键是手册要写清楚——参数名要自解释，类型要明确，描述要准确。

2. 各平台格式对比

各平台 Function Calling 格式: OpenAI: tool_calls: [{ id: "call_abc123", type: "function", function: { name: "get_weather", arguments: '{"city": "北京"}' } }] Anthropic (Claude): <invoke name="get_weather"> <parameter name="city">北京</parameter> </invoke> Gemini: function_call: { name: "get_weather", args: {city: "北京"} } Mistral: tool_calls: [{ function: { name: "get_weather", arguments: {"city": "北京"} } }]

2.1 工具定义格式对比

各平台的工具定义格式差异不大，都采用 JSON Schema 描述参数，但在细节上有重要区别：

平台定义方式参数格式支持并行调用支持流式
OpenAItools 数组JSON Schema (严格)✅✅
Anthropictools 数组JSON Schema (灵活)✅✅
Geminifunction_declarationsProto Schema✅✅
Mistraltools 数组JSON Schema✅✅
Qwentools 数组JSON Schema✅部分
DeepSeektools 数组JSON Schema✅✅

2.2 tool_choice 控制策略

tool_choice 参数控制模型是否必须调用工具，以及调用哪个工具。这是生产环境中非常重要的控制手段：

tool_choice 值行为使用场景
"auto" (默认)模型自行决定是否调用工具通用对话
"none"禁止调用工具纯文本回复
"required"必须调用至少一个工具Agent 模式
{"function": {"name": "xxx"}}强制调用指定工具精确控制
tool_choice 策略决策: 用户输入 → 需要调用工具吗? │ ├── 不确定 → "auto" (模型自己判断) │ ├── 确定需要 → "required" (强制调用) │ │ │ └── 需要指定具体工具? │ ├── 是 → {"function": {"name": "xxx"}} │ └── 否 → "required" │ └── 确定不需要 → "none" (禁止调用) └── 场景: 用户在闲聊, 无需工具

3. 代码示例

3.1 基础 Function Calling

# OpenAI Function Calling 完整示例
import openai
import json

client = openai.OpenAI()

# 1. 定义工具
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的当前天气信息，包括温度、湿度和天气状况",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名，如'北京'、'上海'"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"],
                         "description": "温度单位，默认摄氏度"}
            },
            "required": ["city"]
        }
    }
}]

# 2. 调用 LLM
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "北京天气"}],
    tools=tools,
    tool_choice="auto"
)

# 3. 解析工具调用
if response.choices[0].message.tool_calls:
    tc = response.choices[0].message.tool_calls[0]
    function_name = tc.function.name  # "get_weather"
    arguments = json.loads(tc.function.arguments)  # {"city": "北京"}

    # 4. 执行工具
    result = get_weather(**arguments)

    # 5. 将结果返回给 LLM
    response2 = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "user", "content": "北京天气"},
            {"role": "assistant", "tool_calls": [tc]},
            {"role": "tool", "tool_call_id": tc.id, "content": json.dumps(result)}
        ]
    )
    print(response2.choices[0].message.content)

3.2 并行 Function Calling

# 并行调用多个工具的完整流程
import asyncio

# 一次请求中模型可能同时调用多个工具
messages = [{"role": "user", "content": "比较北京和上海的天气"}]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=tools,
    tool_choice="auto"
)

# 模型返回多个 tool_calls
# tool_calls: [
#   {id: "call_1", function: {name: "get_weather", arguments: '{"city":"北京"}'}},
#   {id: "call_2", function: {name: "get_weather", arguments: '{"city":"上海"}'}}
# ]

tool_calls = response.choices[0].message.tool_calls

# 并行执行所有工具调用
async def execute_tools(tool_calls):
    tasks = []
    for tc in tool_calls:
        args = json.loads(tc.function.arguments)
        tasks.append(asyncio.to_thread(get_weather, **args))
    return await asyncio.gather(*tasks)

results = asyncio.run(execute_tools(tool_calls))

# 将所有结果返回给 LLM
messages.append({"role": "assistant", "tool_calls": tool_calls})
for tc, result in zip(tool_calls, results):
    messages.append({
        "role": "tool",
        "tool_call_id": tc.id,
        "content": json.dumps(result)
    })

response2 = client.chat.completions.create(model="gpt-4o", messages=messages)
print(response2.choices[0].message.content)

3.3 Anthropic Claude 的 Function Calling

# Anthropic Claude 的工具调用格式 (XML 风格)
import anthropic

client = anthropic.Anthropic()

# Claude 使用不同的格式
tools = [{
    "name": "get_weather",
    "description": "获取指定城市的当前天气信息",
    "input_schema": {  # 注意: Claude 用 input_schema 而非 parameters
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "城市名"}
        },
        "required": ["city"]
    }
}]

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "北京天气怎么样？"}]
)

# Claude 的响应格式
for block in response.content:
    if block.type == "tool_use":
        print(f"工具: {block.name}")       # get_weather
        print(f"参数: {block.input}")      # {"city": "北京"}
        print(f"ID: {block.id}")           # 用于后续回传结果

4. 训练 Function Calling

4.1 训练数据构造

让开源模型学会 Function Calling 需要精心构造训练数据。数据需要覆盖以下场景：

# Function Calling 训练数据构造
training_data = {
    "messages": [
        {"role": "system", "content": "你是一个天气助手。可用工具: get_weather(city, unit)"},
        {"role": "user", "content": "北京天气怎么样？"},
        {"role": "assistant", "content": None, "tool_calls": [
            {"function": {"name": "get_weather", "arguments": '{"city":"北京"}'}}
        ]},
        {"role": "tool", "content": '{"temp":25,"condition":"晴"}'},
        {"role": "assistant", "content": "北京今天天气晴朗，温度25°C。"}
    ]
}

# 负样本: 不需要工具的问题
negative_data = {
    "messages": [
        {"role": "system", "content": "你是一个助手。可用工具: get_weather"},
        {"role": "user", "content": "1+1 等于多少？"},
        {"role": "assistant", "content": "1+1 等于 2。"}
        # 注意: 没有 tool_calls
    ]
}

4.2 训练数据的关键要素

高质量的 Function Calling 训练数据需要包含以下类型：

样本类型占比说明示例
单工具调用40%一次调用一个工具"北京天气" → get_weather
并行工具调用15%一次调用多个工具"北京上海天气" → 2x get_weather
多轮工具调用15%连续多轮调用不同工具"搜索北京酒店" → search → book
无需调用20%不需要工具的普通对话"1+1=?" → 直接回答
错误处理10%参数缺失或工具失败"天气怎么样" → 需要追问城市
常见误区：1) 以为 Function Calling 只能调 API——它本质是"结构化输出"，可以用于任何需要结构化决策的场景（分类、提取、路由）；2) 工具描述太模糊——"搜索信息"不如"在知识库中搜索与查询相关的文档，返回 top-k 结果，每条包含标题和摘要"；3) 忽略 parallel tool call——一次请求可以让 AI 同时调用多个工具，大幅提升效率；4) 训练数据中"不调用"的样本太少——模型会变得"过度工具化"，连简单问题也要调用工具。

5. 高级模式

5.1 强制工具调用

通过 tool_choice 参数，可以强制模型调用特定工具。这在以下场景中非常有用：

# 强制调用特定工具 - 结构化提取场景
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "从用户消息中提取联系人信息"},
        {"role": "user", "content": "我叫张三，电话13800138000，邮箱zhangsan@example.com"}
    ],
    tools=[{
        "type": "function",
        "function": {
            "name": "extract_contact",
            "description": "提取联系人信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "name": {"type": "string"},
                    "phone": {"type": "string", "pattern": "^1[3-9]\\d{9}$"},
                    "email": {"type": "string", "format": "email"}
                },
                "required": ["name"]
            }
        }
    }],
    tool_choice={"type": "function", "function": {"name": "extract_contact"}}
)

# 模型必须调用 extract_contact，输出结构化的联系人信息
# → {"name": "张三", "phone": "13800138000", "email": "zhangsan@example.com"}

5.2 嵌套工具调用链

多轮工具调用链示例: 用户: "帮我订一张明天北京到上海的高铁票" 第1轮: LLM → get_train_schedule(from="北京", to="上海", date="明天") 系统 → [{G101: 07:00, G103: 08:00, G105: 09:00}] 第2轮: LLM → book_ticket(train="G101", from="北京", to="上海", date="明天") 系统 → {order_id: "ORD12345", status: "已预订"} 第3轮: LLM → "已为您预订明天 G101 次列车(07:00)北京到上海的车票， 订单号: ORD12345。" 关键: 每一轮都需要 LLM 根据前一轮结果决定下一步

5.3 流式 Function Calling

流式场景下，工具调用的 JSON 参数是逐 token 生成的。客户端需要在收到完整 JSON 后才能解析工具调用。这带来了一个 UX 挑战：用户看到模型在"打字"，但实际上它在生成工具调用参数。

# 流式 Function Calling 的处理逻辑
for chunk in stream:
    delta = chunk.choices[0].delta

    if delta.tool_calls:
        for tc_delta in delta.tool_calls:
            # 累积工具调用的各个部分
            if tc_delta.function.name:
                current_tool_name = tc_delta.function.name
            if tc_delta.function.arguments:
                # arguments 是逐片段到达的
                # '{"city":' → '"北京"}'
                current_args_buffer += tc_delta.function.arguments

                # 尝试解析完整的 JSON
                try:
                    args = json.loads(current_args_buffer)
                    # JSON 完整了！执行工具调用
                    execute_tool(current_tool_name, args)
                except json.JSONDecodeError:
                    pass  # 还没收到完整的 JSON，继续等待
通俗类比：流式 Function Calling 就像"边听边记"——你在听一个人报电话号码（138-0013-8000），每听到一个数字就记下来，但只有听完全部数字才能拨打电话。如果中途就拨号，号码是不完整的。客户端需要一个"缓冲区"来累积参数片段，直到 JSON 完整。

7. MCP 协议与工具标准化

MCP（Model Context Protocol）是 Anthropic 提出的开放协议，旨在标准化 LLM 与外部工具的交互方式。它定义了工具描述、调用、结果返回的统一格式，使得同一个工具可以被不同模型和框架复用。

7.1 MCP vs 传统 Function Calling

维度传统 Function CallingMCP 协议
工具定义位置每次 API 请求携带MCP Server 集中管理
跨模型兼容每个平台格式不同统一协议
工具发现手动列举自动发现和注册
权限控制应用层实现协议层内置
工具版本管理无内置版本控制
生态系统碎片化标准化，可复用
深入探讨：Function Calling 的技术实现分两层：1) 模型层——通过特殊 token（如 <|tool_call|>）标记工具调用的开始和结束，训练数据中包含工具调用样本。研究表明（Gorilla, 2023），经过 Function Calling 微调的 7B 模型可以达到 GPT-4 级别的工具调用准确率。2) 推理层——约束解码保证参数格式正确。OpenAI 的实现最成熟（支持 parallel tool call、tool choice 控制、strict mode），Anthropic 的 Claude 用 XML 格式（更灵活但解析更复杂），Google 的 Gemini 用 Proto 格式（与 Google Cloud 生态深度集成）。评测基准 BFCL 从 V1 到 V4 逐步增加了多轮对话、并行调用、Agent 场景的评测，当前排行榜前列的模型在基础函数调用准确率上已超过 95%，但在复杂链式调用场景中仍有较大提升空间。

8.2 错误处理策略

# 生产环境中的 Function Calling 错误处理
def safe_tool_call(tool_name, arguments, max_retries=3):
    """带重试和降级的工具调用"""
    for attempt in range(max_retries):
        try:
            # 1. 参数验证
            validate_arguments(tool_name, arguments)

            # 2. 执行工具
            result = execute_tool(tool_name, arguments)

            # 3. 结果验证
            validate_result(tool_name, result)

            return {"status": "success", "data": result}

        except InvalidArgumentError as e:
            # 参数错误: 让 LLM 重新生成参数
            return {"status": "error", "type": "invalid_args",
                    "message": str(e)}

        except ToolTimeoutError:
            if attempt < max_retries - 1:
                continue  # 重试
            return {"status": "error", "type": "timeout",
                    "message": f"工具 {tool_name} 执行超时"}

        except Exception as e:
            return {"status": "error", "type": "unknown",
                    "message": str(e)}
深入探讨（Gorilla论文）：2023年UC Berkeley的Gorilla论文（Patil et al.）系统研究了LLM的工具调用能力。核心发现：1) 经过API调用数据微调的LLaMA-7B在工具调用准确率上超过GPT-4（无微调），证明了微调对Function Calling的巨大价值；2) 工具描述的质量比工具数量更重要——10个描述精良的工具优于100个描述模糊的工具；3) 模型对"不调用"的判断能力（即何时不该调用工具）往往弱于"调用"能力，需要专门的负样本训练。Gorilla提出的AST评测方法后来成为BFCL的基础——通过抽象语法树匹配来判断工具调用是否正确，而非简单的字符串比较，这解决了"同一个调用有多种合法写法"的问题。

通俗类比（工具描述）：写工具描述就像写商品说明书——"功能：搜索"（太模糊）vs "在电商数据库中按关键词搜索商品，返回匹配商品的名称、价格、评分，支持按价格和评分排序"（清晰具体）。好的描述应该让"从未见过这个工具的人"也能准确使用。一个实用技巧：想象你在给一个实习生写工作指引——如果实习生看了描述就能正确调用工具，那这个描述就是合格的。
常见误区（生产环境）：1) 不处理 tool_call_id 不匹配——如果客户端和服务端的 tool_call_id 对不上，LLM 会混淆结果来源；2) 工具返回结果太长——超大结果会消耗大量 token，应截断或摘要化返回；3) 忽略并发安全——并行 tool call 可能访问共享资源，需要做好锁和幂等处理。
