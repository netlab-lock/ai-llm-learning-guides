Plan-and-Execute 模式详解

Wang et al. 2023 提出的 Plan-and-Execute 模式将Agent工作流分为两个明确阶段：先制定全局计划，再逐步执行。相比ReAct的逐步推理，该模式更适合复杂多步骤任务。Plan-and-Execute的核心定理在于：将规划（Planning）和执行（Execution）解耦后，可以使用不同的模型分别处理——用强模型（如GPT-4）进行全局规划，用快模型（如GPT-3.5）进行局部执行，从而在保证质量的同时降低成本和延迟。

Plan-and-Execute的数学模型

从形式化角度看，Plan-and-Execute可以被建模为一个分层决策过程。在高层，规划器π_plan接收任务描述T，输出一个有序步骤序列P = [s_1, s_2, ..., s_n]。在低层，执行器π_exec依次执行每个步骤s_i，生成结果r_i。执行过程中，监控器π_monitor检查结果r_i是否满足预期，如果不满足则触发重规划。这种分层架构的数学优势在于：规划阶段可以在抽象层面搜索最优路径，避免在具体执行细节上浪费计算资源。执行阶段可以并行处理独立步骤，提高整体效率。重规划机制可以在环境变化时动态调整，保证鲁棒性。

Plan-and-Execute 分层架构： ┌─────────────────────────────────────────────────────────────┐ │ Plan-and-Execute 架构 │ ├─────────────────────────────────────────────────────────────┤ │ │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ 高层规划器 (Planner) │ │ │ │ 输入: 任务描述 T │ │ │ │ 输出: 计划 P = [s_1, s_2, s_3, ..., s_n] │ │ │ │ 模型: 强推理模型 (GPT-4, Claude) │ │ │ └──────────────────────┬──────────────────────────────┘ │ │ │ 计划 │ │ ▼ │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ 低层执行器 (Executor) │ │ │ │ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ │ │ 执行 │→│ 执行 │→│ 执行 │→│ 执行 │ │ │ │ │ │ s_1 │ │ s_2 │ │ s_3 │ │ s_n │ │ │ │ │ └──────┘ └──────┘ └──────┘ └──────┘ │ │ │ │ 模型: 快速执行模型 (GPT-3.5) │ │ │ └──────────────────────┬──────────────────────────────┘ │ │ │ 结果 │ │ ▼ │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ 监控器 (Monitor) │ │ │ │ 检查结果 → 是否需要重规划? → 更新计划 │ │ │ └─────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────┘

Plan-and-Execute与ReAct的本质区别

ReAct是"走一步看一步"——每一步都基于当前观察做下一步决策，灵活性高但缺乏全局视野。Plan-and-Execute是"先想好再做"——先制定完整计划再执行，全局最优但灵活性不足。从搜索算法的角度看，ReAct类似于贪心搜索（每步选最优），Plan-and-Execute类似于A*搜索（先规划最优路径再执行）。在任务复杂度较低时，两者差异不大；在任务复杂度较高时，Plan-and-Execute的优势在于能够避免"局部最优陷阱"。

# Plan-and-Execute 伪代码
def plan_and_execute(task):
    # 阶段一：全局规划
    plan = planner.create_plan(task)  # 使用强模型
    # plan = ["步骤1: 收集数据", "步骤2: 分析数据", "步骤3: 生成报告"]
    
    results = []
    for i, step in enumerate(plan.steps):
        # 阶段二：逐步执行
        result = executor.execute(step, context=results)  # 使用快模型
        results.append(result)
        
        # 阶段三：监控与重规划
        if need_replan(result):  # 检查是否需要调整
            remaining = plan.steps[i+1:]
            new_plan = planner.replan(task, results, remaining)
            plan.steps = plan.steps[:i+1] + new_plan.steps
    
    return aggregate(results)
核心优势：模型分工
规划与执行解耦后，可用不同模型分别处理：强模型规划、快模型执行，兼顾质量与成本。实验表明，这种"强规划+快执行"的模式比全程使用强模型降低成本60%以上，质量损失仅5-10%。

任务分解策略与工程实践

任务分解是规划阶段的核心技能。好的任务分解应该满足以下原则：可执行性（每个子步骤都有明确的执行方式）、可验证性（每个子步骤的结果可以被客观评估）、适度粒度（不太粗也不太细）、依赖清晰（步骤间的依赖关系明确）。

策略描述适用场景优势劣势
层级分解将高层目标递归拆解为子任务树项目管理、复杂工程任务结构清晰，易于理解层级过深会导致管理困难
顺序分解按时间或依赖关系线性排列步骤流水线、数据处理执行简单，易于追踪无法利用并行性
并行分解识别可并行执行的独立子任务批量处理、独立查询效率高，速度快需要同步机制
条件分解根据中间结果动态分支决策流程、异常处理灵活适应变化逻辑复杂，难以调试
递归分解将问题分解为同构的子问题分治算法、树形结构通用性强递归深度难以控制

任务分解的工程实践

在实际工程中，任务分解通常需要结合领域知识和经验。以下是几个关键的工程实践建议：使用模板化分解——对于常见任务类型（如数据处理、报告生成、代码开发），预先定义分解模板，减少规划时间。引入人类反馈——在关键节点引入人类审核，确保分解方向正确。支持动态调整——执行过程中发现分解不合理时，能够及时调整。

注意
过度分解会导致步骤数爆炸，每一步都有出错风险。研究表明，3-10步是最佳范围——少于3步可能粒度太粗，多于10步则管理成本急剧上升。建议在规划阶段使用"粗粒度分解"（3-5步），在执行阶段再"细粒度展开"（每步2-3个子操作）。

LLM+P 经典规划集成

Liu et al. 2023 提出将LLM与经典规划器（如PDDL）结合：LLM负责将自然语言描述转化为形式化规划问题（PDDL domain + problem），规划器求解最优计划，最后再由LLM解释执行。这种方法的优势在于：规划器可以保证计划的最优性和完备性（在给定的PDDL模型下），而LLM负责自然语言理解和形式化转换。

# LLM+P 完整流程
1. LLM 将自然语言任务描述转为 PDDL domain + problem
   - domain.pddl: 定义类型、谓词、动作
   - problem.pddl: 定义初始状态和目标状态

2. 经典规划器（Fast Downward / ENHSP）求解
   - 输入: domain.pddl + problem.pddl
   - 输出: 最优动作序列

3. LLM 将 PDDL plan 翻译为自然语言步骤
   - 保持计划的最优性
   - 添加执行细节和上下文

4. Agent 按步骤执行，遇到异常时回到步骤1
适用场景
LLM+P 特别适合有明确前置条件和效果定义的任务，如物流调度、资源分配、日程安排等结构化问题。在这些场景中，经典规划器可以保证最优性，而LLM负责自然语言交互。

PDDL形式化示例

;; LLM将"帮我安排明天的会议"转为PDDL
(define (domain meeting-scheduling)
  (:requirements :strips :typing)
  (:types meeting room time)
  (:predicates
    (scheduled ?m - meeting)
    (available ?r - room ?t - time)
    (conflicts ?m1 - meeting ?m2 - meeting)
    (preferred-time ?m - meeting ?t - time)
  )
  (:action schedule-meeting
    :parameters (?m - meeting ?r - room ?t - time)
    :precondition (and (not (scheduled ?m))
                       (available ?r ?t))
    :effect (and (scheduled ?m)
                 (not (available ?r ?t)))
  )
)

;; LLM将用户需求转为problem
(define (problem schedule-tomorrow)
  (:domain meeting-scheduling)
  (:objects m1 m2 m3 - meeting r1 r2 - room t1 t2 t3 t4 - time)
  (:init (available r1 t1) (available r1 t2) ...)
  (:goal (and (scheduled m1) (scheduled m2) (scheduled m3)))
)

Tree-of-Thought 规划机制

Tree-of-Thought (ToT) 让LLM在规划时探索多个推理分支，通过评估和剪枝找到最优路径。ToT的核心创新在于：将传统的线性推理（CoT）扩展为树形推理，支持回溯和多路径探索。从搜索算法的角度看，ToT对应的是启发式搜索——每个节点的"启发值"由LLM评估，搜索策略可以是BFS、DFS或Beam Search。

ToT的关键步骤

  • 思维生成：在每个节点生成多个可能的下一步（通常3-5个）
  • 状态评估：用LLM评估每个中间状态的前景（打分或分类）
  • 搜索策略：BFS/DFS/Beam Search 选择扩展路径
  • 回溯剪枝：淘汰低分路径，聚焦高质量方案
Tree-of-Thought 搜索过程： ┌───────────┐ │ 初始状态 │ │ (问题) │ └─────┬─────┘ │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ 思路A │ │ 思路B │ │ 思路C │ │ 得分:8 │ │ 得分:5 │ │ 得分:7 │ └────┬────┘ └─────────┘ └────┬────┘ │ │ ┌───────┼───────┐ ┌───────┼───────┐ ▼ ▼ ▼ ▼ ▼ ▼ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ │A1:9│ │A2:6│ │A3:7│ │C1:8│ │C2:5│ │C3:6│ └──┬──┘ └─────┘ └─────┘ └──┬──┘ └─────┘ └─────┘ │ │ │ 剪枝: B分支(5分)被裁剪 │ │ ┌──┴──┐ ┌──┴──┐ │最优 │ │次优 │ │路径 │ │路径 │ └─────┘ └─────┘ 搜索策略对比: BFS: 宽度优先，适合解空间不深的场景 DFS: 深度优先，适合解空间很深的场景 Beam Search: 限制宽度的BFS，平衡质量和成本

实现模式与生产级代码

LangChain 的 PlanAndExecute 链提供了开箱即用的实现。以下是一个生产级的实现，包含了错误处理、日志记录和动态重规划功能。

import asyncio
from typing import List, Dict, Any
from dataclasses import dataclass
from enum import Enum

class StepStatus(Enum):
    PENDING = "pending"
    RUNNING = "running"
    SUCCESS = "success"
    FAILED = "failed"
    SKIPPED = "skipped"

@dataclass
class PlanStep:
    id: int
    description: str
    tool: str
    depends_on: List[int]
    status: StepStatus = StepStatus.PENDING
    result: Any = None
    error: str = None

class ProductionPlanAndExecute:
    """生产级Plan-and-Execute Agent"""
    
    def __init__(self, planner_llm, executor_llm, tools: List, max_replans: int = 3):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = {t.name: t for t in tools}
        self.max_replans = max_replans
        self.replan_count = 0
        self.execution_log = []
    
    async def run(self, task: str) -> str:
        """执行Plan-and-Execute流程"""
        # 阶段一：制定计划
        plan = await self._create_plan(task)
        self._log("plan_created", {"steps": len(plan)})
        
        # 阶段二：逐步执行
        for step in plan:
            if not self._can_execute(step, plan):
                step.status = StepStatus.SKIPPED
                continue
            
            try:
                step.status = StepStatus.RUNNING
                context = self._get_context(step, plan)
                step.result = await self._execute_step(step, context)
                step.status = StepStatus.SUCCESS
                self._log("step_completed", {"step_id": step.id})
            except Exception as e:
                step.status = StepStatus.FAILED
                step.error = str(e)
                self._log("step_failed", {"step_id": step.id, "error": str(e)})
                
                # 动态重规划
                if self.replan_count < self.max_replans:
                    plan = await self._replan(task, plan, step)
                    self.replan_count += 1
                else:
                    return f"执行失败，已达到最大重规划次数({self.max_replans})"
        
        # 阶段三：汇总结果
        return self._aggregate_results(plan)
    
    async def _create_plan(self, task: str) -> List[PlanStep]:
        """使用强模型制定计划"""
        prompt = f"""请将以下任务分解为可执行步骤:
任务: {task}
可用工具: {list(self.tools.keys())}
输出JSON格式的步骤列表。"""
        
        response = await self.planner.generate(prompt)
        steps = self._parse_plan(response.text)
        return [PlanStep(**step) for step in steps]
    
    async def _execute_step(self, step: PlanStep, context: str) -> str:
        """使用快模型执行步骤"""
        prompt = f"""执行以下步骤:
步骤: {step.description}
工具: {step.tool}
上下文: {context}
请调用工具并返回结果。"""
        
        response = await self.executor.generate(prompt)
        return response.text
    
    async def _replan(self, task: str, current_plan: List[PlanStep], failed_step: PlanStep) -> List[PlanStep]:
        """动态重规划"""
        completed = [s for s in current_plan if s.status == StepStatus.SUCCESS]
        prompt = f"""原计划执行失败，需要重新规划:
任务: {task}
已完成步骤: {[s.description for s in completed]}
失败步骤: {failed_step.description}
失败原因: {failed_step.error}
请制定新的计划。"""
        
        response = await self.planner.generate(prompt)
        new_steps = self._parse_plan(response.text)
        return [PlanStep(**step) for step in new_steps]
    
    def _can_execute(self, step: PlanStep, plan: List[PlanStep]) -> bool:
        """检查步骤的前置依赖是否满足"""
        for dep_id in step.depends_on:
            dep = next((s for s in plan if s.id == dep_id), None)
            if dep is None or dep.status != StepStatus.SUCCESS:
                return False
        return True
    
    def _get_context(self, step: PlanStep, plan: List[PlanStep]) -> str:
        """获取步骤执行所需的上下文"""
        context_parts = []
        for dep_id in step.depends_on:
            dep = next((s for s in plan if s.id == dep_id), None)
            if dep and dep.result:
                context_parts.append(f"步骤{dep.id}结果: {dep.result}")
        return "\n".join(context_parts)
    
    def _aggregate_results(self, plan: List[PlanStep]) -> str:
        """汇总所有步骤的结果"""
        results = []
        for step in plan:
            if step.status == StepStatus.SUCCESS:
                results.append(f"[{step.description}]: {step.result}")
        return "\n\n".join(results)
    
    def _parse_plan(self, text: str) -> List[Dict]:
        """解析计划JSON"""
        import json, re
        match = re.search(r'\{.*\}', text, re.DOTALL)
        if match:
            data = json.loads(match.group())
            return data.get("steps", [])
        return []
    
    def _log(self, event: str, data: Dict):
        """记录执行日志"""
        import time
        self.execution_log.append({
            "timestamp": time.time(),
            "event": event,
            "data": data
        })
实现要点
执行器应在每步完成后返回结构化结果，以便规划器判断是否需要调整后续计划。结果的结构化程度直接影响重规划的质量。建议使用JSON Schema定义每步的预期输出格式。

动态重规划与容错机制

当执行过程中环境发生变化或出现意外时，需要动态调整计划。动态重规划是Plan-and-Execute模式与简单流水线的关键区别——它使Agent具备了"应变能力"。

重规划触发条件

  • 执行失败：工具调用返回错误、超时或异常结果
  • 结果偏离预期：输出质量不达标、关键指标未达到阈值
  • 新信息出现：执行过程中获取到新的约束条件或需求变更
  • 资源限制：时间、token或API调用次数接近上限
动态重规划决策流程： 执行步骤 S_i │ ▼ ┌───────────┐ 是 ┌─────────────┐ │ 执行成功? │──────────→│ 记录结果 │ └─────┬─────┘ │ 继续S_{i+1} │ │ 否 └─────────────┘ ▼ ┌───────────┐ 是 ┌─────────────────┐ │ 可重试? │──────────→│ 重试S_i (max=3) │ └─────┬─────┘ └─────────────────┘ │ 否 ▼ ┌───────────┐ 是 ┌─────────────────┐ │ 局部可修? │──────────→│ 调整S_i参数重试 │ └─────┬─────┘ └─────────────────┘ │ 否 ▼ ┌───────────────┐ 是 ┌─────────────────────┐ │ 重规划次数<3? │──────→│ 调用Planner重新规划 │ └───────┬───────┘ │ 保留已完成步骤 │ │ 否 └─────────────────────┘ ▼ ┌─────────────────┐ │ 回退到人类审批 │ │ 报告当前状态 │ └─────────────────┘
重规划策略触发条件重规划范围成本适用场景

全局重规划根本性错误所有未完成步骤高需求变更、环境剧变
重规划陷阱
过于频繁的重规划会导致Agent陷入循环。建议设置最大重规划次数（如3次），超过后回退到人类审批。同时，每次重规划都应该记录原因和变更，便于后续分析和优化。
通俗类比：规划就像项目经理接到一个大项目——先拆解成里程碑（Plan），再分配给团队执行（Execute）。比如"开发一个App"，项目经理不会让你直接写代码，而是先画架构图、排工期、分配任务。Agent的Plan-and-Execute也是这个思路：先想清楚"要做什么"，再一步步做。关键区别是：项目经理可以凭经验跳步，Agent必须严格按计划执行，除非触发重规划。
工程实践：在生产环境中，建议将Plan-and-Execute与人类审批流程结合。对于高风险操作（如删除数据、发送邮件），在执行前自动暂停并请求人类确认。可以设计"审批级别"——低风险操作自动执行，中风险操作异步审批，高风险操作同步审批。这种分级审批机制可以在保证安全的同时不牺牲效率。
性能优化：Plan-and-Execute的主要瓶颈在于规划阶段的LLM调用。优化策略包括：缓存常见任务的规划结果（相同或相似任务直接复用计划）、使用小模型预规划+大模型审核（减少强模型调用次数）、增量规划（只规划下一步，而不是一次性规划所有步骤）。
常见误区：1) 过度规划——把任务拆成50个微步骤，每步都有出错风险，3-10步是最佳范围；2) 规划完就不管了——好的规划系统需要动态重规划，执行中发现不对要及时调整；3) 所有任务都先规划——简单任务直接ReAct就行，规划本身也有token成本；4) 忽略规划质量——垃圾计划导致垃圾执行，必须验证计划的可行性和合理性。
生产环境风险：1) 规划幻觉——LLM可能生成不可执行的计划（引用不存在的工具、设置矛盾的依赖），必须在执行前验证计划的可行性；2) 重规划风暴——当计划持续失败时，Agent可能反复重规划，消耗大量token却没有进展，必须设置重规划上限；3) 状态不一致——重规划后新旧计划的上下文不一致，可能导致重复执行或遗漏步骤。
深入探讨：Tree-of-Thought（ToT）的理论基础ToT是规划的高级形态，让LLM像下棋一样"走一步看三步"。每个思维节点生成多个候选方案，用评估函数打分，BFS/DFS搜索最优路径。关键创新在于：传统CoT是单条链路（走一步看一步），ToT是树形搜索（走一步看多步），代价是推理成本倍增。实践中常用Beam Search限制宽度来平衡质量和成本。ToT的数学原理基于蒙特卡洛树搜索（MCTS）：每个节点的价值V(s) = 胜率 × 探索奖励，LLM充当评估函数和策略函数。最新研究将ToT与强化学习结合，用RL训练节点评估函数，进一步提升搜索效率。
深入探讨：规划的可验证性问题规划阶段的一个核心挑战是"计划的可验证性"——如何在执行前判断一个计划是否可行？经典规划器（如PDDL）通过形式化验证保证计划的正确性，但需要精确的领域模型。LLM生成的计划是自然语言，无法形式化验证。当前的解决方案包括：执行前模拟（用LLM模拟执行每一步，检查是否有矛盾）、约束检查（验证工具可用性、参数合法性）、历史类比（与成功的历史计划对比，评估相似度）。最前沿的研究方向是"计划调试"——像调试程序一样调试计划，逐步执行并检查中间状态。

规划与任务分解的技术原理与数学推导： 从底层算法分析，规划问题的本质是在动作空间中搜索从初始状态到目标状态的路径。 推导：假设每步有b个可选动作、规划深度为d，则朴素搜索空间为b^d。当b=5、d=10时，搜索空间高达约977万种可能。 LLM通过启发式剪枝将搜索空间大幅缩减——本质上是用预训练知识替代穷举搜索。 为什么这很重要？因为规划能力决定了Agent能否处理复杂的多步骤任务——缺乏规划的Agent在面对需要10+步的任务时， 错误率会随步数指数增长。从实现机制看，Plan-and-Execute架构将规划与执行解耦：Planner生成高层计划（子任务列表）， Executor逐个执行子任务。ReWOO方案进一步优化——一次性生成所有子任务计划后批量执行，减少了LLM调用次数。 关键的数学权衡是：规划越详细（d越大），执行越简单但规划本身的错误累积越严重；规划越粗略，执行越灵活但每步的决策难度越高。

    关键要点: ①核心原理的数学推导和物理意义；②关键参数的设计原则和调优方法；③与替代方案的对比分析；④实际部署中的工程考量；⑤性能指标的量化和评估方法。
    实际案例: 以一个生产环境部署场景为例：当请求QPS从100增长到1000时，系统瓶颈从计算受限转变为内存受限。通过Profiling发现Decode阶段HBM带宽利用率达到85%，此时应该采用KV Cache量化（减少带宽需求）或分离架构（将Prefill和Decode部署在不同GPU上）来优化。实测优化后TTFT降低40%，TPS提升2.5x。
  2. 核心原理: 在实际工程应用中，本章介绍的技术需要根据具体场景进行适配。关键技术决策包括：架构选择（一体vs分离）、精度选择（FP16/FP8/INT4）、缓存策略（LRU/LFU/Size-based）等。决策依据是请求特征（prompt长度分布、并发量、SLA要求）和硬件条件（GPU型号、网络带宽、存储类型）。标准化的评估方法是通过Profiling确定瓶颈类型：计算受限→量化/剪枝；内存受限→算子融合/缓存优化；通信受限→拓扑优化/通信压缩。
    关键要点: ①需求分析和场景匹配；②技术参数的调优方法；③性能评估和瓶颈定位；④成本-性能权衡分析；⑤生产环境的最佳实践。

  3. 核心原理: 对比分析需要从多个维度展开：性能（延迟、吞吐）、精度（量化损失）、成本（GPU小时）、复杂度（运维难度）、适用场景（prompt长度、并发模式）。不同技术方案各有优劣：如一体架构简单但GPU利用率低，分离架构高效但引入传输开销。选择依据是业务需求：如果SLA要求TTFT<200ms且QPS>100，应选择分离架构+Prefix Caching；如果QPS<10且prompt短，一体架构即可满足。

    实际案例: 某团队对比了三种方案：A)一体架构+FP16 B)一体架构+FP8+Prefix Caching C)分离架构+FP8+Prefix Caching。方案A月度成本$50K，TTFT=500ms；方案B成本$30K，TTFT=200ms；方案C成本$25K，TTFT=150ms。考虑到运维复杂度和团队能力，选择了方案B作为过渡方案，半年后迁移到方案C。
