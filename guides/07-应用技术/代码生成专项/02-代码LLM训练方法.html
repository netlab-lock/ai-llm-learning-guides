🎓 02 · 代码 LLM 训练方法

中级 核心 实战

代码LLM的训练方法与通用LLM有显著差异。本模块深入讲解代码模型特有的训练技术，包括数据工程、Fill-in-the-Middle、仓库级上下文训练和长上下文扩展等核心技术。掌握这些内容是理解和构建高质量代码AI系统的基础。无论你是想训练自己的代码模型，还是想更好地理解现有模型的能力边界，这些知识都是必不可少的。也是进一步研究代码AI前沿方向的基础。

1. 训练数据工程

数据质量是代码LLM性能的基石。与通用LLM不同，代码数据有独特的质量特征：语法正确性可验证、代码风格可量化、功能可通过测试验证。高质量的代码训练数据需要经过多轮过滤和清洗，确保模型学到的是"正确且优雅"的代码模式，而非低质量的垃圾代码。数据工程通常占据整个训练流程60%以上的工作量。

1.3 数据配比

数据配比决定了模型在不同编程语言上的能力分布。一个好的配比策略需要考虑：各语言的使用频率、语言间的迁移效应（学习Python有助于理解Ruby）、训练数据的可用性。通常Python和JavaScript占据最大比例，因为它们是GitHub上最活跃的语言。

2. Fill-in-the-Middle (FIM) 训练

FIM是代码LLM训练中最重要的技术创新之一。传统的Next-Token Prediction只能从左到右生成代码，但实际的IDE补全场景经常需要"填充中间部分"。FIM通过特殊的训练格式解决了这个问题，让模型同时理解代码的前文和后文，从而生成正确的中间代码。这一技术使得代码LLM在IDE场景中的实用性大幅提升。

2.1 为什么需要 FIM？

标准 Next-Token Prediction 的局限: 标准训练 (左到右): 输入: "def hello():" 目标: " print('hello')" 问题: 代码补全经常需要「填充中间」！ 实际 IDE 场景: 前缀: "def sort(arr):" 后缀: " return arr" 需要填充: 中间的排序逻辑 FIM 训练正是为这种场景设计的！

2.2 FIM 训练格式

# PSM (Prefix-Suffix-Middle) 格式
# 最常用的 FIM 格式

原始代码:
def fibonacci(n):
    if n <= 1:
        return n
    return fibonacci(n-1) + fibonacci(n-2)

# 随机选择切分点，构造 FIM 样本
prefix = "def fibonacci(n):\n    if"
suffix = ":\n        return n\n    return fibonacci(n-1) + fibonacci(n-2)"

# PSM 格式
fim_sample = "<|fim_prefix|>" + prefix + "<|fim_suffix|>" + suffix + "<|fim_middle|>" + " n <= 1" + "<|fim_middle|>"

# SPM (Suffix-Prefix-Middle) 格式
fim_sample = "<|fim_suffix|>" + suffix + "<|fim_prefix|>" + prefix + "<|fim_middle|>" + " n <= 1" + "<|fim_middle|>"

# FIM rate: 通常 0.5-0.9
# 即 50-90% 的训练样本使用 FIM 格式
# 剩余 10-50% 使用标准 Next-Token 格式

关键发现：FIM 训练不会降低标准 Next-Token 的性能，但显著提升中间填充能力。所以几乎所有的现代代码 LLM 都使用 FIM。PSM和SPM两种格式各有优势：PSM更直观（先看前缀再看后缀），SPM在某些场景下效果更好（先理解后缀有助于理解上下文）。最佳实践是同时使用两种格式训练。

PSM vs SPM 格式对比: PSM (Prefix-Suffix-Middle): ┌──────────────────────────────────────────┐ │ <|fim_prefix|> [前文代码] │ │ <|fim_suffix|> [后文代码] │ │ <|fim_middle|> [需要生成的中间代码] │ │ <|fim_middle|> │ └──────────────────────────────────────────┘ 思路: 先看前文 → 再看后文 → 填充中间 SPM (Suffix-Prefix-Middle): ┌──────────────────────────────────────────┐ │ <|fim_suffix|> [后文代码] │ │ <|fim_prefix|> [前文代码] │ │ <|fim_middle|> [需要生成的中间代码] │ │ <|fim_middle|> │ └──────────────────────────────────────────┘ 思路: 先看后文 → 再看前文 → 填充中间 实际效果: 两种格式混合使用效果最佳！

3. Repository-level 训练

仓库级训练是代码LLM区别于通用LLM的关键技术之一。真实的软件开发涉及多个文件的协同修改，如果模型只能看到单个文件，就无法完成跨文件的任务（如添加API端点、修复跨模块Bug）。仓库级训练通过在训练数据中引入跨文件上下文，让模型学会理解项目的整体架构。

3.2 仓库级上下文构建

# 构建仓库级训练样本
def build_repo_context(repo_path, target_file):
    """为 target_file 构建仓库级上下文"""
    context_parts = []

    # 1. 项目结构 (文件树)
    file_tree = get_file_tree(repo_path)
    context_parts.append(f"# Project structure:\n{file_tree}")

    # 2. 相关文件 (通过 import 关系)
    imports = extract_imports(target_file)
    for imp in imports:
        resolved = resolve_import(repo_path, imp)
        if resolved:
            context_parts.append(f"# {resolved}:\n{read_file(resolved)}")

    # 3. 目标文件
    context_parts.append(f"# Target file:\n{read_file(target_file)}")

    return "\n\n".join(context_parts)
仓库级训练的常见误区：1) 盲目将整个代码库塞入上下文——这不仅浪费token，还会引入大量噪声，导致模型注意力分散；2) 忽略上下文排序——相关文件应该靠近目标文件，无关文件应该被过滤掉；3) 训练和推理的上下文构建方式不一致——如果训练时用import链构建上下文，推理时也必须用相同方式，否则会出现分布偏移。

4. 长上下文训练

代码LLM需要处理整个代码仓库，需要128K-1M的上下文窗口。长上下文训练不仅是简单地增加位置编码范围，还需要精心设计的训练策略来确保模型在长距离上保持注意力质量。直接将上下文扩展到128K会导致"注意力稀释"——模型在超长序列中无法有效关注关键信息。

长上下文渐进训练策略: 阶段1 (预训练): 上下文窗口 = 4K ████████░░░░░░░░░░░░░░░░░░░░ 学习基本代码模式 阶段2 (位置扩展): 上下文窗口 = 32K ████████████████░░░░░░░░░░░░ 扩展RoPE基频，学习跨文件关系 阶段3 (长上下文): 上下文窗口 = 128K ████████████████████████████ 学习仓库级上下文理解 关键: 不能一步到位！直接用128K训练会导致注意力质量下降。 渐进式扩展让模型逐步适应更长的上下文。
长上下文训练的技术原理：RoPE（旋转位置编码）的外推是长上下文训练的核心。原始RoPE在训练长度之外的位置编码质量急剧下降。YaRN方法通过以下改进解决这个问题：(1) NTK-aware插值——根据目标长度动态调整RoPE的基频；(2) Attention scaling——对长距离注意力进行温度缩放；(3) 渐进式训练——分阶段扩展上下文长度。数学上，YaRN修改了RoPE的频率基数θ，使得位置编码在更长范围内保持均匀分布。实验表明，这种渐进式策略比直接训练128K的效果好15-20%。

5. 训练策略总结

将上述训练策略综合运用，才能训练出高质量的代码LLM。下表总结了各策略的作用和使用场景：

策略作用使用率
Next-Token Prediction基础语言建模100%
Fill-in-the-Middle代码填充能力50-90% 样本
Repository-level跨文件理解需要时
长上下文训练处理大型代码库渐进式

训练策略的选择建议：对于初学者，建议从Next-Token + FIM开始，这是代码LLM训练的"最低可行配置"。当你需要处理跨文件任务时，再加入Repository-level训练。长上下文训练是进阶技术，需要大量算力支持。记住：数据质量 > 训练技巧 > 模型规模。

6. 训练超参数调优

代码LLM的训练超参数与通用LLM有显著差异。代码数据的特殊性（结构化、语法严格、长距离依赖）要求我们在超参数选择上做出针对性调整。好的超参数配置可以让相同的数据和模型达到更好的效果，差的配置则可能浪费大量算力却得不到理想结果。

超参数通用LLM典型值代码LLM推荐值原因

训练超参数的常见误区：1) 照搬通用LLM的超参数——代码数据的token分布与自然语言不同，直接套用会导致训练不稳定；2) FIM rate设得太高——超过90%会损害标准补全能力，建议保持在50-80%；3) 忽略数据重复次数——代码数据不像自然语言那么多样，过多重复会导致过拟合，建议每个数据源最多重复3轮。

7. 前沿进展

代码LLM训练技术正在快速发展。几个值得关注的方向：(1) 合成数据——用强模型生成训练数据，实现"数据飞轮"；(2) 强化学习——通过代码执行结果的反馈优化模型；(3) 多模态训练——结合代码、注释、图表和屏幕截图进行训练。这些方向正在重塑代码AI的训练范式，使得小模型也能达到大模型的代码能力。

合成数据与数据飞轮：Phi-1模型证明了合成数据的巨大潜力——用GPT-4生成的"教科书级"代码数据训练的小模型（1.3B参数），在HumanEval上超过了多个大10倍的模型。这一发现引发了"数据质量革命"：与其收集更多数据，不如精心构造高质量数据。数据飞轮的运作方式是：用当前最强模型生成数据→训练新模型→新模型生成更好的数据→循环。关键挑战是如何确保合成数据的多样性和正确性——过于同质的合成数据会导致模型"过度拟合"到生成者的风格。

常见误区：1) 以为数据越多越好——The Stack v2有900B tokens但经过严格过滤，未过滤的GitHub数据充满垃圾代码和自动生成文件；2) 忽略FIM训练格式——FIM需要特殊的PSM/SPM格式转换，直接用普通文本训练做不了代码补全；3) 混淆预训练和微调——预训练学"怎么写代码"，微调学"怎么按你的风格写代码"，两者缺一不可。
深入探讨：代码LLM训练的关键技术：1) FIM（Fill-in-the-Middle）——把代码切成prefix+middle+suffix三段训练，让模型学会"填空"而不只是"续写"；2) 去重（MinHash+LSH）——GitHub上大量fork和复制文件，不去重会让模型过拟合到重复代码；3) 质量过滤——去除自动生成代码、测试文件、过短文件；4) 许可证过滤——只用MIT/Apache/BSD等宽松许可证代码，避免法律风险。训练超参方面，代码模型通常需要更长的上下文窗口（16K+）和更多的FIM样本（占比30-50%）。

8. 多语言代码训练

现代代码LLM需要支持数十种编程语言。多语言训练的核心挑战是如何在有限的训练资源下平衡不同语言的能力。过于偏重某种语言会导致其他语言能力退化（"灾难性遗忘"），而完全平均分配又会浪费资源在低频语言上。找到合适的平衡点是多语言代码训练的关键课题。

多语言训练的典型问题: 问题: 训练数据中 Python 占 40%, COBOL 占 0.01% ↓ 如果完全按比例训练: Python ████████████████████████████████████████ 40% Java ████████████████ 16% C++ ████████████ 12% COBOL ░ 0.01% ↓ 结果: Python 能力强，但 COBOL 几乎不会 解决方案: 数据上采样 (Up-sampling) 对低频语言进行适度上采样，保证最低训练量 COBOL: 0.01% → 1% (上采样 100 倍) 但不要过度上采样，否则会过拟合到少量样本
# 多语言数据配比策略
def calculate_language_ratio(lang_stats, min_ratio=0.01, alpha=0.7):
    """
    计算多语言训练的数据配比
    alpha: 平衡因子
        - alpha=1.0: 完全按原始比例
        - alpha=0.0: 完全平均分配
        - alpha=0.7: 偏向原始比例但保证低频语言
    """
    total = sum(lang_stats.values())
    ratios = {}
    for lang, count in lang_stats.items():
        natural_ratio = count / total
        uniform_ratio = 1.0 / len(lang_stats)
        # 混合比例
        adjusted = alpha * natural_ratio + (1 - alpha) * uniform_ratio
        ratios[lang] = max(adjusted, min_ratio)

    # 归一化
    total_ratio = sum(ratios.values())
    return {lang: r / total_ratio for lang, r in ratios.items()}

# 示例: StarCoder2 的多语言配比
# Python: 30%, JS/TS: 18%, Java: 12%, C/C++: 10%
# Go: 5%, Rust: 4%, 其他语言按上采样策略分配
# 关键: 保证每种语言至少有 1% 的训练数据
通俗类比：代码LLM的训练就像培养一个程序员——先让他读大量代码（预训练），然后教他"填空"（FIM训练），再教他理解项目全局（仓库级训练），最后扩展他的"工作记忆"（长上下文训练）。每一步都不可或缺，但最关键的是"读什么代码"（数据质量），而不是"读多少代码"（数据数量）。
代码LLM与通用LLM的本质差异：代码LLM和通用LLM的根本区别在于"输出空间"的结构化程度。自然语言的输出空间是"模糊"的——同一个意思可以用无数种方式表达。但代码的输出空间是"严格"的——一个缩进错误、一个括号缺失都会导致程序崩溃。这意味着代码LLM需要比通用LLM更高的"精确性"。FIM训练和语法检查正是为了培养这种精确性。从信息论角度看，代码的信息熵比自然语言低（因为代码有更多的结构约束），这使得代码LLM在同等参数量下可以达到更高的生成质量。
