1.1 HumanEval 的构建过程

HumanEval 由 OpenAI 于 2021 年发布，包含 164 个 Python 编程问题。每个问题包含：函数签名、文档字符串（docstring）、参考实现和平均 7.7 个测试用例。构建过程中有一个关键设计决策：所有题目都是 OpenAI 内部原创的，而非从互联网收集，以此减少数据泄露风险。

# HumanEval 问题示例 (Problem 0: has_close_elements)
from typing import List

def has_close_elements(numbers: List[float], threshold: float) -> bool:
    """检查列表中是否有两个元素的差值小于 threshold
    
    >>> has_close_elements([1.0, 2.0, 3.0], 0.5)
    False
    >>> has_close_elements([1.0, 2.8, 3.0, 4.0, 5.0, 2.0], 0.3)
    True
    """
    for i in range(len(numbers)):
        for j in range(i + 1, len(numbers)):
            if abs(numbers[i] - numbers[j]) < threshold:
                return True
    return False

# 代码 LLM 需要根据 docstring 和签名生成完整实现
# 核心挑战: 理解 "差值小于 threshold" 的数学含义

MBPP (Mostly Basic Python Problems) 是 Google 于 2021 年发布的基准，与 HumanEval 的主要区别在于：

通俗类比：代码评测就像"编程考试"——HumanEval是"选择题"（简单函数，有标准答案），SWE-bench是"应用题"（真实bug修复，需要理解上下文），CodeContests是"竞赛题"（算法题，考验思维能力）。LiveCodeBench是"模拟高考"（定期更新题目，防止背答案）。只做选择题考得好不代表应用题也行——所以不能只看HumanEval分数就下结论。

2. 评测方法

2.1 pass@k 公式

pass@k = 1 - C(n-c, k) / C(n, k)

其中 n = 总生成数，c = 通过数，k = 选择数。这是代码评测中最核心的指标。它的本质含义是：生成 n 个候选答案，其中有 c 个通过了所有测试，那么随机选 k 个至少有一个通过的概率是多少？

# pass@k 计算
import math

def pass_at_k(n, c, k):
    """
    n: 总生成数
    c: 通过测试的数量
    k: 选择数量
    """
    if n - c < k:
        return 1.0
    return 1.0 - math.prod(1.0 - k / i for i in range(n - c + 1, n + 1))

# 示例: 生成 20 个候选，15 个通过，选 1 个
print(pass_at_k(20, 15, 1))  # pass@1 = 0.75

# 为什么用无偏估计?
# 直觉计算: 15/20 = 0.75 → 这里恰好相等
# 但当 n 小时，有偏估计会高估实际能力
# pass@k 的无偏估计来自超几何分布的期望

2.2 SWE-bench 评测

SWE-bench 评测流程: 1. 准备环境 - 克隆目标仓库 (django, flask, etc.) - 检出到 Issue 提交前的版本 - 安装依赖 2. 应用 Patch - 将模型生成的 Patch 应用到代码 3. 运行测试 - 执行仓库的测试套件 - 检查相关测试是否通过 4. 评分 - 通过 = 1, 失败 = 0 - 最终分数 = 通过的 Issue 数 / 总 Issue 数
# SWE-bench 评测示例
def swebench_eval(model, problem):
    """SWE-bench 评测"""
    # 1. 克隆仓库
    repo = clone_repo(problem["repo"], problem["base_commit"])

    # 2. 生成 Patch
    patch = model.generate(problem["issue_description"])

    # 3. 应用 Patch
    apply_patch(repo, patch)

    # 4. 运行测试
    test_result = run_tests(repo, problem["test_files"])

    # 5. 评分
    return 1 if test_result.passed else 0

2.3 SWE-bench 的数据构建方法

SWE-bench 的数据来源是从 Python 开源项目的 GitHub 历史中挖掘的。具体过程如下：

SWE-bench 数据构建流程: GitHub Commit History │ ↓ ┌──────────────────────┐ │ 筛选条件: │ │ 1. commit 修改了测试 │ │ 2. 测试在 commit 前 │ │ 是失败的 │ │ 3. 测试在 commit 后 │ │ 变为成功 │ └──────────┬───────────┘ ↓ ┌──────────────────────┐ │ 自动生成: │ │ - Issue 描述 (从 PR) │ │ - 基础 commit (父) │ │ - 期望 patch (diff) │ │ - 测试文件 (FAIL→PASS)│ └──────────┬───────────┘ ↓ ┌──────────────────────┐ │ 人工验证: │ │ SWE-bench Verified │ │ 500 题人工复核 │ └──────────────────────┘

SWE-bench Verified 版本解决了原始版本中约 20% 的"脏数据"问题（测试定义模糊、期望行为有争议等），是目前最权威的代码 Agent 评测基准。

2.4 LiveCodeBench：防刷榜设计

LiveCodeBench 的核心创新是"时间切片"——定期从 LeetCode、Codeforces、AtCoder 等竞赛平台收集新题目，确保模型在训练时无法"看到"评测数据。这解决了 HumanEval 等静态基准最大的问题：数据泄露。

# LiveCodeBench 的防刷榜机制
# 静态基准的问题:
training_cutoff = "2024-01"
benchmark_created = "2021-09"  # HumanEval
# → 模型在训练时已经"见过"评测题目 → 分数虚高

# LiveCodeBench 的解决方案:
# 每月收集新竞赛题
# T0 (2024-01): 收集 2024年1月 的竞赛题 → 评测 A
# T1 (2024-06): 收集 2024年6月 的竞赛题 → 评测 B
# T2 (2025-01): 收集 2025年1月 的竞赛题 → 评测 C

# 时间窗口效应:
# 模型在 T1 评测时，只能"作弊"到 T0 之前的数据
# 新收集的 T1 题目是全新的 → 真实能力评估
通俗类比：LiveCodeBench 的防刷榜设计就像"高考出题"——每年高考题都是新出的，你不可能提前背答案。而 HumanEval 就像"公务员考试题库"——题库就那么几百道，背下来就能得高分。LiveCodeBench 通过不断更新题目，确保评测结果反映的是模型的"真实实力"而非"记忆能力"。

3. 评测的局限

刷榜问题：模型可能在基准上过拟合（训练数据包含测试题）。HumanEval 的题目已经在互联网上广泛流传，新一代模型的训练数据几乎必然包含这些题目。

测试覆盖不足：通过测试 ≠ 代码正确（测试可能不完善）。HumanEval 平均只有 7.7 个测试用例，很多边界条件未覆盖。

与实际编程能力的差距：基准分数高不等于生产力高。一个在 HumanEval 上得 90% 的模型，可能在真实项目中连 Git 操作都不会。

语言偏差：大多数基准只测 Python，忽略其他语言。程序员在真实工作中可能需要 JavaScript、Go、Rust 等多种语言。

3.1 评测指标的深层问题

除了上述表面问题，代码评测体系还存在一些更深层的方法论挑战：

3.2 pass@k 的统计学深度分析

pass@k 公式的推导基于超几何分布。理解这个公式对于正确使用评测结果至关重要：

# pass@k 的统计学本质
# 
# 场景: 从 n 个候选中随机选 k 个
# 已知: c 个候选通过了测试
# 
# 问题: 至少选中 1 个通过候选的概率?
# 
# P(至少1个通过) = 1 - P(全部不通过)
# 
# P(全部不通过) = C(n-c, k) / C(n, k)
# 
# 其中 C(n-c, k) 是从 n-c 个不通过候选中选 k 个的方式数
# C(n, k) 是从所有 n 个候选中选 k 个的方式数

# 实际使用中的注意事项:
# 1. n 应该足够大 (通常 n=200) 以获得稳定估计
# 2. pass@1 是最常用的指标 (真实场景只用1个答案)
# 3. pass@10 和 pass@100 反映模型的"潜力"
# 4. 温度参数对 pass@k 影响巨大:
#    - 低温 (T=0): pass@1 高但 pass@100 低 (一致性好但缺乏多样性)
#    - 高温 (T=1): pass@1 低但 pass@100 高 (多样性好但一致性差)

# 不同温度下的 pass@k 对比 (DeepSeek-Coder-33B, HumanEval)
results = {
    "T=0.0": {"pass@1": 56.1, "pass@10": 72.3, "pass@100": 81.5},
    "T=0.2": {"pass@1": 55.8, "pass@10": 78.1, "pass@100": 89.2},
    "T=0.6": {"pass@1": 51.2, "pass@10": 80.5, "pass@100": 93.7},
    "T=1.0": {"pass@1": 43.5, "pass@10": 79.8, "pass@100": 95.1},
}
通俗类比：pass@k 的统计含义就像"抽奖"——你有 n 个球（生成的代码），其中 c 个是"中奖球"（通过测试的代码），随机抽 k 个至少中一个的概率。pass@1 就是"抽1次就中"的概率，pass@10 是"抽10次至少中1次"的概率。所以 pass@10 永远 ≥ pass@1——但实际部署中你只能用 1 个答案，所以 pass@1 才是真正重要的指标。

4.3 自建评测基准指南

如果你需要为特定领域或公司内部代码库建立评测基准，以下是关键步骤：

# 自建代码评测基准的框架
import json
from dataclasses import dataclass
from typing import List

@dataclass
class EvalProblem:
    """评测问题的数据结构"""
    problem_id: str          # 唯一标识
    task_type: str           # "generation" / "repair" / "understanding"
    language: str            # 编程语言
    difficulty: str          # "easy" / "medium" / "hard"
    prompt: str              # 问题描述 (自然语言 + 代码骨架)
    reference_solution: str  # 参考实现
    test_cases: List[dict]   # 测试用例 [{input, expected_output}]
    metadata: dict           # 额外信息 (来源、作者、标签等)

def create_eval_dataset(problems: List[EvalProblem]):
    """创建评测数据集的关键检查"""
    checks = {
        "唯一性": len(set(p.problem_id for p in problems)) == len(problems),
        "多样性": len(set(p.task_type for p in problems)) >= 3,
        "语言覆盖": len(set(p.language for p in problems)) >= 3,
        "测试充分性": all(len(p.test_cases) >= 5 for p in problems),
        "参考正确性": "需要运行验证",  # 所有参考解必须通过所有测试
    }
    
    # 关键: 10% 的测试用例应该对参考实现也暴露边界情况
    # 例如: 空输入、超大输入、特殊字符等
    
    return checks
常见误区：1) 只看pass@1——pass@1只看第一次就对的概率，pass@10（10次中至少对1次）更能反映模型潜力，但实际部署时你只有一次机会；2) 忽略评测数据泄露——HumanEval的题目太出名了，可能在训练数据中，分数虚高，应该同时用 LiveCodeBench 等动态基准交叉验证；3) 用单一基准评判模型——不同基准测不同能力，需要综合多个基准的结果，就像不能只看数学成绩就判断学生能力一样。

深入探讨 — pass@k 的统计本质与方差：pass@k 的无偏估计来源于超几何分布的期望值计算。给定 n 个独立采样中有 c 个成功，从 n 中无放回抽取 k 个时，至少有 1 个成功的概率的无偏估计量为：Ê[pass@k] = 1 - C(n-c, k)/C(n, k)。这个估计量的方差为 Var = (k/(n²(n-1))) × ((n-c)(n-k)(c+k-n) + ...)。实际使用中，当 n=200, c=100, k=1 时，标准差约为 0.035（即 ±3.5% 的置信区间）。这意味着两次独立评测的 pass@1 差异如果 < 3.5%，在统计上是不显著的。另一个关键问题是"评测污染"(Benchmark Contamination)：研究表明，当模型训练数据中包含 HumanEval 题目时，pass@1 会虚高 15-25%。Jain et al. (2024) 提出了"金丝雀测试"——将评测数据中的特定字符串作为标记，如果模型能记住这些标记，则说明存在数据泄露。LiveCodeBench 通过定期更换评测集（每月新增约 50 道题），从根本上解决了这一问题。
深入探讨：代码评测基准的演进反映了代码AI能力的提升：HumanEval（2021，简单函数）→MBPP（2021，基础编程）→CodeContests（2022，竞赛算法）→SWE-bench（2024，真实bug修复）→CrossCodeEval（2024，跨文件理解）。每个新基准都比上一个难10倍。评测方法学的关键问题：1) 数据泄露（基准题目被训练）——HumanEval 发布仅 3 年，GPT-4 的分数已经从 67% 提升到 92%，但部分提升可能来自记忆而非能力提升；2) 评测成本（运行所有测试用例需要大量计算）——评测一个模型在 SWE-bench 上的表现需要 500+ Docker 容器的执行，单次评测成本可达 $500-1000；3) 指标选择（pass@k vs 解决率 vs 代码质量）——业界正在探索"加权解决率"，对更难的问题给予更高权重。
