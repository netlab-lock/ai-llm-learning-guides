🔧 05 · 代码修复 Agent

进阶

1. SWE-bench 的挑战

SWE-bench 是代码修复的标准评测：给定 GitHub Issue 描述，模型需要生成 Patch 修复代码。SWE-bench 已成为衡量代码智能体能力的黄金标准，它模拟了真实软件开发中最常见的场景——根据 Bug 报告修复代码。SWE-bench 的核心难点在于：(1) 需要理解自然语言描述的语义；(2) 在数万行代码仓库中精准定位问题；(3) 生成与项目风格一致的修复代码；(4) 修复不能引入新的回归问题。

SWE-bench 任务流程: 输入: Issue: "在 Django 中，当使用 SQLite 时， DateTimeField 的默认值解析错误" 仓库: django/django 相关文件: django/db/models/fields/__init__.py 输出: Patch: 修改 DateTimeField 的 to_python 方法 --- a/django/db/models/fields/__init__.py +++ b/django/db/models/fields/__init__.py @@ -1234,7 +1234,7 @@ class DateTimeField: - def to_python(self, value): + def to_python(self, value): # 修复默认值解析 评测: 运行仓库的测试套件，检查是否通过 关键约束: - 不能访问互联网 - 只能修改代码,不能修改测试 - 有时间/步数限制 - 必须在原始仓库状态下修复

1.1 SWE-bench 数据集构成

SWE-bench 包含来自 12 个 Python 开源项目的真实 Pull Request，每个样本包含 Issue 描述、相关的文件上下文和需要通过的测试用例。SWE-bench Lite 是一个更小的子集（300 个样本），用于快速评估。每个样本的难度取决于：(1) Bug 涉及的文件数量；(2) 需要理解的代码上下文深度；(3) 修复的复杂度。

1.2 为什么 SWE-bench 如此困难？

SWE-bench 的核心挑战不是"写代码"，而是"理解代码库"。模型需要：(1) 从 Issue 描述中提取关键信息（哪些功能出错、什么条件触发）；(2) 在庞大代码库中找到问题根源——可能涉及多个文件的交互；(3) 理解现有代码的设计意图，确保修复不破坏原有功能；(4) 生成符合项目编码规范的代码。这与简单的代码生成有本质区别，因为修复需要深入理解现有代码的上下文。

SWE-bench 评测陷阱：1) "通过测试"不等于"正确修复"——测试覆盖可能不完整，某些修复可能绕过了测试但没有真正解决问题；2) SWE-bench Lite 的样本偏简单——在 Lite 上 50% 的解决率，在 Full 上可能只有 20%；3) 成本不可忽视——完整评测一次可能花费数百美元的 API 费用，需要在准确率和成本间取舍。

2. Agentless 方法

Agentless 用简单的 3 阶段 Pipeline 修复代码，无需 Agent 循环。它的核心理念是：大多数代码修复可以分解为三个独立的步骤，每个步骤用 LLM 完成，无需复杂的 Agent 框架。这种方法的优雅之处在于它的简洁性——不需要定义工具集、不需要管理 Agent 状态、不需要设计重试逻辑。

2.1 Agentless 的关键创新

  • 无 Agent 循环：不需要 LLM 自主决定下一步行动，避免了 Agent 的不确定性
  • 确定性流程：定位→修复→验证，每步都是确定性的，可复现性强
  • 低成本：SWE-bench Lite ~32%，成本仅 $0.70，远低于 Agent 方法
  • 分层定位：文件级→函数级→行级，逐层缩小范围

2.2 Agentless 定位算法详解

Agentless 的定位阶段采用了"自顶向下"的策略：首先用 Embedding 检索找到最相关的文件（文件级），然后在这些文件中用 LLM 识别最相关的函数（函数级），最后精确定位到需要修改的行。这种分层策略的效率远高于直接让 LLM 分析整个代码库。

2.3 Agentless 修复生成策略

在定位完成后，Agentless 用 LLM 生成 unified diff 格式的 Patch。关键技巧包括：(1) 提供足够的上下文（函数签名、相关 import、类定义）；(2) 使用 structured output 确保生成合法的 diff 格式；(3) 多次采样（k=5-10）生成候选 Patch，增加命中概率。

Agentless 的适用场景：Agentless 方法最适合"局部 Bug"——问题出在 1-2 个文件、修复逻辑清晰、不需要探索性调试。对于这类 Bug，Agentless 的效率和成本远优于 Agent 方法。但如果你面对的是"系统性 Bug"（涉及多个模块的交互、需要运行复杂测试、需要理解整体架构），Agent 方法可能更合适。

3. SWE-agent

SWE-agent 是自主代码修复 Agent，它让 LLM 像开发者一样操作代码仓库。与 Agentless 的确定性流程不同，SWE-agent 给 LLM 提供了一组工具（search_file、view_file、edit_file、run_tests 等），让 LLM 自主决定下一步操作。这种方法的优势在于灵活性——Agent 可以根据中间结果调整策略。

3.1 SWE-agent 的 ACI 设计

SWE-agent 的核心创新是 Agent-Computer Interface (ACI)——为 LLM 设计的代码操作接口。ACI 的设计原则包括：(1) 每个工具的输出必须对 LLM 友好（不能返回 1000 行原始文本）；(2) 工具之间有清晰的组合关系；(3) 错误信息要有建设性（告诉 LLM 下一步该怎么做）。

# SWE-agent ACI 工具设计示例
class SWEAgentACI:
    """Agent-Computer Interface for code repair"""

    def search_file(self, pattern: str, path: str = ".") -> str:
        """搜索文件内容，返回匹配行及上下文"""
        # 关键设计: 返回精简结果, 不是完整 grep 输出
        results = grep(pattern, path, context=3)
        if len(results) > 20:
            return f"找到 {len(results)} 处匹配，显示前 20 处:\n" + results[:20]
        return results

    def view_file(self, path: str, start: int = 1, end: int = 50) -> str:
        """查看文件内容，限制行数避免超出上下文"""
        content = read_file(path, start, end)
        total_lines = count_lines(path)
        return f"[{path}] L{start}-{end}/{total_lines}\n{content}"

    def edit_file(self, path: str, old: str, new: str) -> str:
        """编辑文件，验证修改合法性"""
        if old not in read_file(path):
            return f"错误: 在 {path} 中未找到匹配文本，请检查缩进和空格"
        apply_edit(path, old, new)
        return f"成功修改 {path}"

    def run_tests(self, test_path: str = None) -> str:
        """运行测试，返回结构化结果"""
        result = pytest(test_path, timeout=300)
        # 只返回失败的测试和简要摘要
        return format_test_results(result, max_failures=5)

3.2 SWE-agent 的 Thought-Action-Observation 循环

SWE-agent 使用 ReAct 范式：每一步都包含 Thought（思考）、Action（行动）、Observation（观察）。Agent 根据 Issue 描述制定修复计划，逐步执行，根据观察结果调整策略。这个过程类似于人类开发者的调试流程。

SWE-agent 的原理分析：SWE-agent 的核心原理是"工具增强推理"——LLM 本身不具备搜索代码、编辑文件、运行测试的能力，但通过 ACI 接口获得了这些能力。这与人类使用 IDE 的过程类似：你不需要记住整个代码库，但你可以用搜索功能找到相关代码、用编辑器修改、用测试框架验证。关键的数学直觉是：如果 LLM 每一步操作的正确率为 p，那么经过 n 步后整体正确的概率约为 p^n。对于 p=0.8, n=10，整体正确率只有 0.8^10 ≈ 10.7%。这就是为什么减少步骤（Agentless）或提高每步准确率（更好的 ACI）如此重要。

4.1 Agentless vs Agent：架构差异

Agentless 和 SWE-agent 代表了两种截然不同的设计理念。Agentless 认为"确定性流程足够"——把修复分解为定位、生成、验证三个独立步骤，每步用 LLM 但不需要 LLM 做决策。SWE-agent 认为"需要自主探索"——给 LLM 工具和目标，让它自己决定如何到达。两种理念各有优劣，选择取决于具体场景。

Agentless vs SWE-agent 架构对比: Agentless (确定性 Pipeline): ┌──────┐ ┌──────┐ ┌──────┐ │定位 │───▶│修复 │───▶│验证 │ └──────┘ └──────┘ └──────┘ 特点: 固定流程, 每步独立, 可并行 SWE-agent (自主 Agent): ┌──────────────────────────────────┐ │ Thought ──▶ Action ──▶ Observe │ │ ▲ │ │ │ └────────────────────┘ │ └──────────────────────────────────┘ 特点: 循环探索, 自主决策, 可回溯 选择指南: ┌────────────────┬──────────────────────┐ │ 场景 │ 推荐方案 │ ├────────────────┼──────────────────────┤ │ 局部Bug(1文件) │ Agentless (快速便宜) │ │ 跨文件Bug │ SWE-agent (灵活探索) │ │ 需要测试调试 │ SWE-agent (可运行测试) │ │ 批量修复 │ Agentless (并行高效) │ └────────────────┴──────────────────────┘

4.2 AutoCodeRover 的 AST 增强

AutoCodeRover 的独特之处在于使用 AST（抽象语法树）分析来增强代码定位。它首先解析代码库的 AST，构建类层次结构和方法调用图，然后用这些结构化信息辅助 LLM 定位。这种方法在理解面向对象代码（如 Java、Python 类）时特别有效，因为 AST 能直接揭示继承关系和方法覆写。

# AutoCodeRover 的 AST 定位策略
import ast

def analyze_class_hierarchy(repo_path):
    """分析代码库的类层次结构"""
    hierarchy = {}
    for py_file in find_python_files(repo_path):
        tree = ast.parse(read_file(py_file))
        for node in ast.walk(tree):
            if isinstance(node, ast.ClassDef):
                bases = [b.id if isinstance(b, ast.Name)
                         else ast.dump(b) for b in node.bases]
                hierarchy[node.name] = {
                    'bases': bases,
                    'methods': [n.name for n in node.body
                               if isinstance(n, (ast.FunctionDef, ast.AsyncFunctionDef))],
                    'file': py_file
                }
    return hierarchy

def locate_by_ast(issue_description, hierarchy):
    """根据 Issue 描述和 AST 信息定位问题"""
    # 1. 从 Issue 中提取类名、方法名
    class_names = extract_class_names(issue_description)
    method_names = extract_method_names(issue_description)

    # 2. 在层次结构中查找
    candidates = []
    for cls_name, info in hierarchy.items():
        if cls_name in class_names:
            candidates.append((cls_name, info['file'], info['methods']))

    return candidates
代码修复 Agent 的工程陷阱：1) 不要忽略 Docker 沙箱——SWE-agent 必须在隔离环境中执行，直接在宿主机运行测试可能破坏环境；2) 上下文窗口管理至关重要——发送整个代码库会浪费 token 且降低准确率，只发送相关代码片段效果更好；3) Patch 格式必须严格——unified diff 的空格和缩进敏感，一个字符错误就会导致 patch 应用失败。

5. 文件定位策略

文件定位是代码修复中最关键的步骤，也是区分不同方法的核心差异。定位策略的演进从简单的关键词搜索到复杂的语义理解，每种方法都有其适用场景。

5.1 Embedding 检索定位

Embedding 检索是最常用的文件定位方法：将 Issue 描述和文件摘要分别编码为向量，计算余弦相似度。这种方法的优势是速度快（可以在毫秒级完成），但缺点是可能遗漏"语义相关但词汇不重叠"的文件。

# 文件级定位: Embedding 检索
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("all-MiniLM-L6-v2")

def locate_files_embedding(issue: str, repo_files: list, top_k: int = 5):
    """用 Embedding 检索相关文件"""
    # 编码 Issue 描述
    issue_embedding = model.encode(issue)

    # 编码所有文件的摘要
    file_summaries = []
    for file_path in repo_files:
        # 生成文件摘要: 路径 + 前50行 + 函数签名
        content = read_file(file_path)
        functions = extract_function_signatures(content)
        summary = f"{file_path}: {', '.join(functions[:10])}"
        file_summaries.append((file_path, summary))

    # 批量编码
    summaries = [s[1] for s in file_summaries]
    file_embeddings = model.encode(summaries, batch_size=64)

    # 计算余弦相似度
    similarities = np.dot(file_embeddings, issue_embedding) / (
        np.linalg.norm(file_embeddings, axis=1) * np.linalg.norm(issue_embedding)
    )

    # 返回 Top-K
    top_indices = np.argsort(similarities)[::-1][:top_k]
    return [(file_summaries[i][0], similarities[i]) for i in top_indices]

5.2 LLM 直接定位

另一种方法是直接让 LLM 分析 Issue 描述，输出可能相关的文件列表。这种方法不依赖 Embedding 模型，但成本更高（需要调用 LLM API），且受限于 LLM 的"幻觉"——可能列出不存在的文件。实际应用中，通常将 Embedding 检索和 LLM 定位结合使用：先用 Embedding 候选筛选，再用 LLM 精排。

5.3 结构化定位（AST + 依赖图）

结构化定位利用代码的语法和语义结构：通过 AST 分析找到 Issue 中提到的类和方法的定义位置，通过依赖图（import 关系）找到可能受影响的文件。这种方法不需要 Embedding 模型，但需要代码解析器支持。对于强类型语言（Java、TypeScript），结构化定位特别有效。

定位方法速度准确率成本适用场景
关键词搜索极快低零快速初筛
Embedding 检索快中-高低通用定位
LLM 直接定位慢中高复杂语义理解

混合方法中最高中生产环境

6. Patch 生成与验证

定位完成后，下一步是生成修复 Patch。这个看似简单的步骤实际上有很多细节需要处理：(1) Patch 格式必须严格遵循 unified diff 规范；(2) 上下文行必须与原始代码完全匹配；(3) 缩进和空格必须正确；(4) 多文件修改需要协调。

6.1 Patch 生成策略

最简单的策略是让 LLM 直接输出 unified diff。但这种方法的问题是 LLM 容易产生格式错误（如错误的行号、缺失的上下文行）。更好的方法是让 LLM 输出"修改指令"（如"在第 1234 行后添加以下代码"），然后用程序转换为 diff 格式。

# Patch 生成策略对比

# 策略1: 直接生成 unified diff (简单但易出错)
prompt_v1 = """
生成 unified diff:
--- a/file.py
+++ b/file.py
@@ -10,3 +10,4 @@
 context
+new_line
"""

# 策略2: 生成结构化修改指令 (更可靠)
prompt_v2 = """
输出 JSON 格式的修改指令:
{
  "file": "path/to/file.py",
  "changes": [
    {
      "type": "replace",  # replace / insert / delete
      "line_start": 1234,
      "line_end": 1236,
      "new_code": "..."
    }
  ]
}
"""

# 策略3: 生成完整文件 (最可靠但成本高)
prompt_v3 = """
输出修改后的完整文件内容:
[整个文件内容]
"""

6.2 测试验证的重要性

验证阶段是防止"假修复"的关键。一个 Patch 可能让测试通过但没有真正解决问题（测试覆盖不足），也可能引入新的回归。最佳实践是：(1) 运行与 Issue 直接相关的测试；(2) 运行修改文件的单元测试；(3) 如果可能，运行完整的回归测试套件。

Patch 验证的实际技巧：在生产环境中，建议使用"最小测试集优先"策略——先运行与 Issue 直接相关的测试（通常只需要几秒），通过后再运行更广泛的测试（可能需要几分钟）。这样可以快速排除明显错误的 Patch，节省计算资源。另外，记录每个 Patch 的测试结果有助于后续的 Patch 排序和选择。

7. Claude Code 在 SWE-bench 上的表现

Claude Code（Anthropic 的代码 Agent）在 SWE-bench Verified 上达到了 72.5% 的通过率，这是当前最好的公开结果。Claude Code 的成功归因于几个关键因素：(1) 强大的基础模型（Claude 3.5 Sonnet/Claude 3 Opus）；(2) 精心设计的 ACI；(3) 强大的代码理解能力（包括理解复杂的设计模式和框架约定）。

7.1 Claude Code 的技术特点

Claude Code 的技术特点包括：(1) 长上下文窗口（200K tokens），可以处理大型文件；(2) 强大的代码推理能力，能理解复杂的继承关系和设计模式；(3) 细粒度的文件编辑能力，支持精确的行级修改；(4) 内置的测试运行和错误分析能力。

8. 前沿进展与学习路径

代码修复领域的最新进展集中在几个方向：(1) 多 Agent 协作——一个 Agent 负责定位，另一个负责修复，第三个负责验证；(2) 学习 from 反馈——根据历史修复的成功/失败改进策略；(3) 仓库级上下文压缩——在有限的上下文窗口中塞入最有用的信息；(4) 测试生成——不仅修复 Bug，还生成防止回归的测试用例。

8.1 多 Agent 修复架构

多 Agent 架构将修复任务分解给专门的 Agent：Explorer Agent 负责搜索和理解代码库，Analyzer Agent 负责分析 Bug 的根因，Coder Agent 负责生成修复代码，Reviewer Agent 负责审查 Patch 质量。每个 Agent 专注于自己的任务，通过消息传递协作。

代码修复的理论基础：从理论角度看，代码修复可以形式化为一个搜索问题——在"所有可能的代码修改"空间中，找到一个使测试通过的修改。这个搜索空间的大小是指数级的（每个 token 可能被替换为任意其他 token），因此直接搜索不可行。定位（Localization）的作用是大幅缩小搜索空间：如果能将修改范围从 10000 行缩小到 10 行，搜索空间就从 O(V^10000) 缩小到 O(V^10)，其中 V 是词表大小。这就是为什么定位准确率是代码修复成功的关键预测因素。

8.2 推荐学习路径

如果你对代码修复感兴趣，建议按以下顺序学习：首先了解 SWE-bench 评测标准（模块 04），然后学习 Agentless 方法理解基础 Pipeline，接着学习 SWE-agent 理解 Agent 范式，最后研究 Claude Code 的实现细节。动手实践方面，建议从 SWE-bench Lite 开始，用 Agentless 方法跑一遍评测，感受代码修复的完整流程。继续学习可以关注模块 06「代码LLM前沿方向」中的最新研究。

🔧 05 · 代码修复 Agent的技术原理与深度分析： 核心概念：🔧 05 · 代码修复 Agent是代码生成专项领域的关键技术方向，其理论基础涉及信息论、优化理论和系统工程的交叉。 推导：从信息论角度，任何信息处理系统的吞吐量受限于信道容量——Shannon定理指出C=B×log₂(1+S/N)， 其中B为带宽、S/N为信噪比。在🔧 05 · 代码修复 Agent的语境下，"信道"可以理解为模型的上下文窗口或推理管线， "信噪比"对应有用信息与噪声（无关上下文、冗余计算）的比例。 为什么这很重要？因为理解🔧 05 · 代码修复 Agent的理论限制，才能在工程实践中做出正确的权衡—— 盲目追求指标上限往往导致过度设计，而理解根本约束才能找到最高效的优化路径。 从工程实践看，🔧 05 · 代码修复 Agent的优化通常遵循"先测量、再分析、后优化"的循环： 首先建立完善的评测基准（覆盖质量、延迟、成本三个维度），然后通过profiling定位瓶颈环节， 最后针对性地优化——是计算受限就优化算法，是内存受限就优化数据布局，是通信受限就优化网络拓扑。 这一系统化方法论适用于代码生成专项领域的所有技术问题。

通俗类比：代码修复Agent就像"修车师傅"——你告诉他车坏了（Issue描述），他先检查发动机（定位代码）、找到问题零件（分析bug）、换新零件（编辑代码）、试车验证（运行测试）。Agentless方法就像"按修车手册修"——不靠经验判断，严格按手册步骤来，简单问题能修好但复杂问题搞不定。而Claude Code就像"4S店专家"——工具齐全、经验丰富，什么问题都能修，但费用也最高。
代码修复的实践误区：1) 过度依赖 LLM 生成的 Patch——LLM 可能生成"看起来正确但实际有微妙 Bug"的代码，必须用测试验证；2) 忽略代码风格一致性——修复代码应该与项目现有风格一致（缩进、命名、注释风格），否则不容易被维护者接受；3) 不考虑修复的副作用——一个"正确的"修复可能影响性能、引入安全漏洞或破坏向后兼容性。
