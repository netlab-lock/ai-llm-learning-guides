🤖 01 · Agent 架构总览

入门

1. 什么是 Agent？

AI Agent = LLM + 工具 + 记忆 + 规划。它不仅能生成文本，还能感知环境、制定计划、执行动作。与传统的聊天机器人不同，Agent具有自主行动能力——它可以决定何时调用工具、如何分解复杂任务、以及如何从错误中恢复。

Agent的本质是一个"感知-思考-行动"的循环系统。LLM充当"大脑"，负责理解和推理；工具充当"手脚"，负责执行具体操作；记忆充当"笔记本"，负责存储历史信息；规划充当"策划师"，负责制定行动方案。

通俗类比： Agent就像一个"实习生"——你（用户）给他一个任务，他会自己思考需要哪些步骤、需要用到什么工具（搜索引擎、计算器、代码编辑器等），然后一步步执行。如果中间遇到问题，他会调整策略重新尝试。关键区别在于：普通聊天AI像"客服"，只能回答问题；Agent像"助理"，能帮你做事。
常见误区： 很多人把Agent等同于"更智能的聊天机器人"，这是错误的。核心区别在于：(1) 聊天机器人是"被动响应"，Agent是"主动行动"；(2) 聊天机器人只能生成文本，Agent可以调用外部工具；(3) 聊天机器人没有"状态"，Agent有记忆和规划能力。另一个常见误区是认为Agent一定需要多轮对话——实际上Agent可以是一个"一次性"的任务执行器，接收任务后自主完成所有步骤。
深入探讨： Agent架构的理论基础可以追溯到1980年代的BDI（Belief-Desire-Intention）模型。BDI模型认为智能体有三个核心状态：信念（Belief，对世界的认知）、愿望（Desire，想要达成的目标）、意图（Intention，计划采取的行动）。现代LLM Agent的映射关系是：LLM的上下文窗口=B信念（当前已知信息），用户请求=Desire（目标），规划器输出=Intention（行动计划）。这个映射揭示了Agent设计的一个核心挑战：如何在有限的上下文窗口中维护足够丰富的"信念"，以支持准确的"意图"生成。

2. Agent 的发展

Agent的概念并非新事物，但大模型的出现使其从理论走向了实用。以下梳理了Agent技术的关键里程碑。

Agent 发展时间线: 2022 ─── ReAct ─── 交替推理和行动 │ 论文: "Synergizing Reasoning and Acting in Language Models" │ 意义: 首次证明LLM可以交替进行推理和工具调用 │ 2023 ─── AutoGPT, BabyAGI ─── 自主循环执行 │ 特点: LLM自主决定下一步行动，无需人工干预 │ 局限: 容易陷入循环，缺乏有效的终止条件 │ LangChain Agent ─── 工具调用框架 │ 特点: 标准化的工具接口和Agent执行框架 │ 2024 ─── LangGraph ─── 图编排 │ 特点: 用有向图定义Agent工作流，支持条件分支和循环 │ CrewAI ─── 角色化多Agent │ 特点: 每个Agent有明确的角色和职责 │ Claude Computer Use ─── 桌面操作 │ 特点: Agent可以操作电脑界面 │ 2025 ─── MCP (Anthropic) ─── 工具连接标准 │ 特点: 标准化的工具发现和调用协议 │ A2A (Google) ─── Agent 间通信标准 │ 特点: 不同厂商的Agent可以互相协作 │ OpenHands ─── 开源代码Agent │ Devin ─── 自主编程Agent
通俗类比： Agent的发展就像"自动驾驶"的演进——2022年的ReAct相当于"L2辅助驾驶"，需要人类监督；2023年的AutoGPT相当于"L3自动驾驶"，可以自主行驶但需要随时接管；2025年的MCP/A2A相当于"车联网标准"，让不同品牌的车可以互相通信。我们正处于从L3到L4的关键过渡期。
阶段代表技术自主程度核心挑战成熟度
2022ReAct单步推理+行动需要人工编排⭐⭐⭐⭐
2023AutoGPT自主循环执行容易失控⭐⭐

2025MCP/A2A标准化协作生态建设⭐⭐
常见误区： 关于Agent发展史的常见误解：(1) 认为AutoGPT代表Agent的最高水平——实际上AutoGPT过于简单，缺乏有效的规划和错误恢复机制；(2) 忽视"人机协作"的价值——完全自主的Agent不一定比"人在环中"的Agent更好，很多场景需要人类监督；(3) 混淆"工具调用"和"Agent"——能调用工具的LLM不一定是Agent，Agent还需要规划、记忆和自主决策能力。
深入探讨： 2025年是Agent技术的"标准化元年"。MCP（Model Context Protocol）和A2A（Agent2Agent Protocol）的出现标志着Agent生态从"各自为战"走向"互联互通"。MCP解决的是Agent与工具之间的标准化连接问题——就像USB标准统一了各种外设接口；A2A解决的是Agent之间的通信问题——就像HTTP协议统一了Web通信。这两个标准的组合使得"Agent互联网"成为可能：不同的Agent可以像网站一样互相发现、互相调用，形成一个分布式的智能网络。

3. 执行循环

Agent的核心是一个"观察-思考-行动"（Observe-Think-Act）的循环。这个循环使Agent能够处理复杂的多步骤任务。

Agent 执行循环 (ReAct 模式): 用户: "帮我查一下北京天气，然后发邮件给张三" ┌─────────────────────────────────────────────────────┐ │ Observe: 接收用户请求 │ │ │ 解析意图: 1.查天气 2.发邮件 │ │ ▼ │ │ Think: 需要两步操作 │ │ │ 第一步: 调用天气API查询北京天气 │ │ ▼ │ │ Act: 调用 get_weather("北京") │ │ │ 返回: {temp: 25, condition: "晴"} │ │ ▼ │ │ Observe: 天气查询成功 │ │ │ 更新记忆: 北京今天25°C，晴天 │ │ ▼ │ │ Think: 天气查到了，现在发邮件 │ │ │ 第二步: 调用邮件API │ │ ▼ │ │ Act: 调用 send_email(to="张三", body="...") │ │ │ 返回: 邮件发送成功 │ │ ▼ │ │ Observe: 邮件发送成功 │ │ │ 任务全部完成 │ │ ▼ │ │ 输出: "已查到北京今天25°C晴天，并发送邮件给张三" │ └─────────────────────────────────────────────────────┘
通俗类比： Agent的执行循环就像"做菜"的过程——Observe是"看菜谱"（了解要做什么），Think是"思考步骤"（先切菜还是先热油），Act是"动手做"（实际操作），然后再Observe（看看菜炒得怎么样了），继续Think（还需要加什么调料）...直到菜做好。如果中间盐放多了，Agent会"观察"到问题，"思考"解决办法（加点糖中和），然后"行动"（加糖）。
常见误区： 实现Agent执行循环时的常见错误：(1) 没有终止条件——Agent可能陷入无限循环，需要设置最大步数或明确的停止信号；(2) 错误处理不足——工具调用可能失败，Agent需要能识别错误并尝试恢复；(3) 上下文溢出——长对话可能导致上下文窗口爆满，需要定期压缩或截断历史；(4) 工具选择错误——LLM可能选择了错误的工具，需要有工具选择的校验机制。
深入探讨： ReAct模式的核心洞察是"推理"和"行动"的交替进行。在ReAct之前，有两种主流范式：(1) 纯推理（Chain-of-Thought）——LLM只进行思考，不调用工具，适合需要逻辑推理的任务；(2) 纯行动（Act-only）——LLM直接调用工具，不做中间推理，适合简单的查询任务。ReAct将两者结合：每次行动前先进行推理（Think），行动后观察结果（Observe），然后基于观察结果进行下一轮推理。这种交替模式使Agent能够根据实际结果调整策略，而不是一次性生成所有步骤。

4. Agent 框架分类

当前Agent框架可以分为三大类：单Agent、多Agent和图编排。每种架构适用于不同的场景和复杂度。

架构类型适用场景优势劣势代表框架
单Agent简单任务、工具调用实现简单、调试容易能力有限、难以处理复杂流程LangChain, Function Calling
多Agent复杂协作、角色分工专业化、可扩展协调复杂、通信开销大CrewAI, AutoGen
图编排确定性工作流可控性强、可视化灵活性受限LangGraph, Prefect
通俗类比： 三种Agent架构就像三种"团队协作模式"——单Agent是"独狼"，一个人包办所有事，适合简单任务；多Agent是"项目团队"，每个人负责不同模块，适合复杂项目；图编排是"流水线"，每个步骤有明确的顺序和条件，适合标准化流程。选择哪种架构取决于任务的复杂度和可控性需求。
常见误区： 选择Agent架构时的常见错误：(1) 过度设计——简单任务用多Agent架构，增加了不必要的复杂度；(2) 忽视调试成本——多Agent系统的调试难度远高于单Agent，出了问题很难定位；(3) 混淆"图编排"和"多Agent"——图编排强调的是工作流控制，多Agent强调的是角色分工，两者可以组合使用；(4) 忽视人机协作——不是所有步骤都需要Agent自动完成，关键决策点应该有人类审核。
深入探讨： 多Agent系统的核心挑战是"协调问题"。当多个Agent协作时，需要解决：(1) 任务分配——如何将大任务分解并分配给合适的Agent；(2) 信息共享——Agent之间如何交换中间结果；(3) 冲突解决——当两个Agent给出矛盾的结果时如何裁决；(4) 全局状态管理——如何跟踪整个系统的执行进度。CrewAI通过"角色-任务-流程"三层抽象来解决这些问题：每个Agent有明确的角色定义（Role），任务有清晰的输入输出规范（Task），流程有确定的执行顺序（Process）。这种设计使得多Agent系统更加可控和可预测。

5. 代码示例

以下是一个最简Agent实现，展示了Agent的核心循环逻辑。

5.1 最简 Agent 骨架

# 最简单的 Agent 实现
class SimpleAgent:
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = {t.name: t for t in tools}

    def run(self, query):
        messages = [{"role": "user", "content": query}]

        while True:
            # 1. LLM 推理
            response = self.llm.chat(messages)

            # 2. 检查是否需要调用工具
            if response.tool_calls:
                for tc in response.tool_calls:
                    # 3. 执行工具
                    result = self.tools[tc.name].run(tc.arguments)
                    messages.append({"role": "tool", "content": result})
            else:
                # 4. 返回最终答案
                return response.content

# 使用
agent = SimpleAgent(
    llm=OpenAI("gpt-4o"),
    tools=[WeatherTool(), EmailTool()]
)
result = agent.run("查北京天气，然后发邮件给张三")

5.2 带错误处理的 Agent

# 生产级 Agent 需要完善的异常处理机制
class RobustAgent:
    def __init__(self, llm, tools, max_steps=10, timeout=30):
        self.llm = llm
        self.tools = {t.name: t for t in tools}
        self.max_steps = max_steps
        self.timeout = timeout

    def run(self, query):
        messages = [{"role": "user", "content": query}]
        start_time = time.time()

        for step in range(self.max_steps):
            # 超时检查
            if time.time() - start_time > self.timeout:
                return "执行超时，请简化任务后重试"

            try:
                response = self.llm.chat(messages)
            except RateLimitError:
                time.sleep(2 ** step)  # 指数退避
                continue
            except APIError as e:
                return f"API 调用失败: {e}"

            if response.tool_calls:
                for tc in response.tool_calls:
                    if tc.name not in self.tools:
                        messages.append({
                            "role": "tool",
                            "content": f"错误: 未知工具 {tc.name}"
                        })
                        continue
                    try:
                        result = self.tools[tc.name].run(tc.arguments)
                    except Exception as e:
                        result = f"工具执行错误: {e}"
                    messages.append({"role": "tool", "content": result})
            else:
                return response.content

        return f"达到最大步骤数 {self.max_steps}，任务未完成"
通俗类比： 这段代码就像一个"简易机器人"的工作流程——它有一个"大脑"（LLM）、一双手（tools）和一个"循环"（while True）。每次循环：大脑思考→决定要不要用手做事→如果需要就做→把结果告诉大脑→大脑继续思考...直到大脑说"我做完了"（return）。这就是Agent最核心的逻辑，所有复杂的Agent框架都是在这个基础上添加各种优化。
常见误区： 实现Agent代码时的常见错误：(1) 没有最大循环次数限制——如果LLM一直返回tool_calls，Agent会死循环，应该添加max_iterations参数；(2) 工具异常不处理——工具调用可能抛出异常，需要try-catch并将错误信息返回给LLM；(3) 消息历史无限增长——每次工具调用都会添加消息，应该定期清理或压缩历史；(4) 不验证工具参数——LLM生成的工具参数可能格式错误，需要在执行前校验。

6. 2025 协议标准化

6.1 MCP 与 A2A 概述

2025年是Agent协议标准化的关键年份。Anthropic推出的MCP和Google推出的A2A分别解决了Agent生态的两大核心问题。

6.2 MCP + A2A 互补关系

MCP + A2A 协议: MCP (Model Context Protocol) - Anthropic: Agent ←→ 工具 标准化工具发现和调用 核心能力: ├─ 工具发现: Agent自动发现可用工具 ├─ 能力描述: 工具自描述输入输出格式 ├─ 安全授权: 标准化的权限管理 └─ 传输协议: 支持HTTP/SSE/WebSocket A2A (Agent2Agent Protocol) - Google: Agent ←→ Agent 标准化 Agent 间通信 核心能力: ├─ 任务委派: Agent A请求Agent B执行任务 ├─ 状态同步: 实时同步任务执行状态 ├─ 能力发现: Agent自动发现其他Agent的能力 └─ 安全认证: Agent间的身份验证 两者互补: MCP 解决「Agent 如何连接工具」── 纵向扩展 A2A 解决「Agent 如何协作」── 横向扩展 组合效果: Agent ←─MCP─→ 工具1, 工具2, 工具3 Agent ←─A2A─→ Agent A, Agent B, Agent C
协议发起方解决的问题核心设计生态现状
MCPAnthropicAgent↔工具连接JSON-RPC + Schema描述已广泛集成
A2AGoogleAgent↔Agent通信Task生命周期管理快速推广中
OpenAPIOpenAIAPI描述规范YAML/JSON Schema行业标准
通俗类比： MCP和A2A的关系就像"USB和互联网"——MCP是USB标准，让电脑（Agent）可以即插即用地连接各种外设（工具）；A2A是互联网协议，让不同的电脑（Agent）可以互相通信。有了USB，你可以连接键盘、鼠标、打印机；有了互联网，你可以访问任何网站。同理，有了MCP，Agent可以使用任何工具；有了A2A，Agent可以与任何其他Agent协作。
常见误区： 关于MCP和A2A的常见误解：(1) 认为两者是竞争关系——实际上它们解决的是不同层面的问题，是互补而非竞争；(2) 认为有了标准就不需要框架——标准只是通信协议，Agent的逻辑编排仍需要框架支持；(3) 忽视安全性——标准化也带来了新的安全风险，恶意工具或Agent可能利用标准接口进行攻击；(4) 期望立竿见影——协议标准化需要时间，目前生态仍在建设中。
深入探讨： MCP协议的技术设计值得关注。它采用JSON-RPC 2.0作为传输协议，支持三种传输方式：(1) stdio——通过标准输入输出通信，适合本地工具；(2) HTTP with SSE——通过HTTP POST发送请求，SSE接收响应，适合远程服务；(3) WebSocket——全双工通信，适合需要实时交互的场景。工具描述使用JSON Schema格式，支持复杂的嵌套类型。MCP还定义了"资源"（Resources）和"提示"（Prompts）两个扩展概念——资源是Agent可以读取的数据（如文件、数据库），提示是预定义的对话模板。这使得MCP不仅是工具调用协议，更是一个完整的Agent能力描述框架。

# 什么是 Agent？ 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed

7. Agent 开发学习路径

掌握 Agent 开发需要循序渐进。以下是一条经过验证的学习路径，从基础概念到生产级系统。

推荐学习路径: 阶段一: 基础理解 (1-2周) ├── 理解 LLM 基本原理 ├── 掌握 Prompt Engineering ├── 学习 Function Calling 机制 └── 动手: 实现一个单工具 Agent 阶段二: 核心能力 (2-3周) ├── 深入 Tool Calling 与 MCP 协议 ├── 学习记忆系统 (短期/长期/RAG) ├── 掌握规划策略 (ReAct/Plan-Execute) └── 动手: 实现一个多工具 ReAct Agent 阶段三: 高级编排 (2-3周) ├── 学习 LangGraph 图编排 ├── 掌握多 Agent 编排模式 ├── 了解 CrewAI/AutoGen/MAF └── 动手: 实现一个多 Agent 系统 阶段四: 生产化 (2-4周) ├── Agent 评测方法论 (SWE-bench, tau-bench) ├── 可观测性 (Tracing, Metrics, Logging) ├── 安全与权限控制 └── 动手: 部署一个生产级 Agent
阶段核心内容推荐资源预计时间

核心能力Tool Calling + 记忆 + 规划本模块 02-04 节2-3 周

学习建议： 不要跳过阶段一直接学高级编排。Agent 开发的每一层都依赖下层的理解——不理解 Tool Calling 就无法设计好的多 Agent 系统，不理解记忆系统就无法处理复杂对话。每个阶段都做动手练习，纸上得来终觉浅。
常见误区： 很多初学者一上来就用 CrewAI 或 AutoGen 搭建多 Agent 系统，却对底层的 Tool Calling 和记忆机制一知半解。结果是框架能跑但出了问题完全无法调试。建议先用纯代码实现一个简单的 Agent，理解每一步的原理，再使用框架提高效率。
深入探讨： Agent 架构的设计哲学源于控制论中的"反馈回路"原理。诺伯特·维纳在1948年提出的控制论核心思想就是：任何自主系统都必须具备感知环境、比较目标与现状、采取纠偏行动的能力。这与 Agent 的 Observe-Think-Act 循环完全一致。从数学角度看，Agent 的执行过程可以建模为一个马尔可夫决策过程（MDP）：每一步的状态 s_t 由观察 o_t 和历史记忆 h_t 决定，动作 a_t 由策略函数 π(s_t) 生成，奖励 r_t 由任务完成度评估。LLM 本质上就是一个强大的策略函数 π，它将自然语言描述的状态映射为工具调用动作。这个数学框架为 Agent 的理论分析和性能优化提供了严谨的基础，也解释了为什么 ReAct 模式比纯推理或纯行动更有效——它在每个时间步都进行了一次策略更新。
