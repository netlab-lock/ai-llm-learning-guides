👥 05 · 多 Agent 编排

进阶

1. 多 Agent 的动机

多 Agent 系统的出现源于一个核心观察：复杂任务往往需要多种不同的能力和视角。单个 Agent 即使配备了大量工具，也难以同时精通搜索、分析、写作、审核等多种技能。通过将不同能力分配给不同的专业化 Agent，每个 Agent 可以专注于自己擅长的领域，从而提升整体系统的质量和效率。这种设计思想在软件工程中被称为"关注点分离"（Separation of Concerns），是构建复杂系统的基本原则之一。

  • 专业化分工：每个 Agent 专注一个领域（搜索、分析、写作），可以为每个 Agent 配置专门的 system prompt、工具集和评测标准
  • 并行处理：多个 Agent 同时工作，提升效率——例如同时搜索多个数据源
  • 对抗讨论：多视角辩论，提升决策质量——Debate 模式可以减少单个模型的偏见
  • 复杂任务分解：将大任务拆分为子任务分配给不同 Agent，降低单个 Agent 的认知负担
  • 容错性：一个 Agent 失败时，其他 Agent 可以接管或重试，提升系统鲁棒性

Hierarchical 模式: [顶层 Supervisor] │ ┌────┴────┐ │ │ [Sup1] [Sup2] / \ / \ [A1] [A2] [A3] [A4] 顶层 Supervisor 负责全局调度 子 Supervisor 负责局部协调 适合超大规模任务

2.3 Peer-to-peer 模式

Peer-to-peer 模式: [A1] ←→ [A2] ↕ ↕ [A3] ←→ [A4] Agent 之间直接通信 无中心调度器 适合协作式任务

2.4 Debate 模式

Debate 模式: 问题: "AI 是否会取代程序员？" [Agent A] → "会取代，因为..." [Agent B] → "不会取代，因为..." [Agent C] → "部分取代，因为..." 各自提出观点 → 互相质疑 → 达成共识 优势: 多视角、减少偏见 劣势: Token 消耗大

3. 主流框架

3.1 LangGraph

LangGraph 是 LangChain 团队推出的图编排框架，它将 Agent 工作流建模为有向图。每个节点是一个处理函数（通常是 LLM 调用或工具执行），每条边定义了节点之间的转移条件。LangGraph 的核心创新是引入了"状态持久化"机制——图的全局状态可以在任意节点持久化到数据库，支持长时间运行的任务和故障恢复。

# LangGraph: 基于图的 Agent 编排
from langgraph.graph import StateGraph, END

# 定义状态
class AgentState(TypedDict):
    messages: list
    next_agent: str

# 定义节点 (Agent)
def search_agent(state):
    # 搜索 Agent 逻辑
    return {"messages": [result]}

def analysis_agent(state):
    # 分析 Agent 逻辑
    return {"messages": [result]}

# 构建图
graph = StateGraph(AgentState)
graph.add_node("search", search_agent)
graph.add_node("analysis", analysis_agent)
graph.add_edge("search", "analysis")
graph.add_edge("analysis", END)

# 设置入口
graph.set_entry_point("search")

# 编译运行
app = graph.compile()
result = app.invoke({"messages": [user_input]})

3.2 CrewAI

CrewAI 是一个基于"角色"概念的多 Agent 框架。每个 Agent 被赋予一个明确的角色（如"研究员"、"作家"、"审核员"），角色定义了 Agent 的专业领域、行为约束和工作方式。CrewAI 的核心抽象是"角色-任务-流程"三层结构：角色定义 Agent 是谁，任务定义 Agent 做什么，流程定义任务如何排序和传递。

# CrewAI: 基于角色的多 Agent
from crewai import Agent, Task, Crew

# 定义 Agent (角色)
researcher = Agent(
    role="研究员",
    goal="搜索最新 AI 研究进展",
    backstory="你是一位资深 AI 研究员",
    tools=[search_tool],
    llm="gpt-4o"
)

writer = Agent(
    role="技术作家",
    goal="撰写技术博客",
    backstory="你是一位技术写作专家",
    llm="gpt-4o"
)

# 定义任务
research_task = Task(
    description="搜索 2025 年最新的 AI Agent 研究",
    agent=researcher,
    expected_output="研究摘要"
)

writing_task = Task(
    description="基于研究结果撰写技术博客",
    agent=writer,
    expected_output="完整博客文章"
)

# 创建团队
crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, writing_task],
    verbose=True
)

# 执行
result = crew.kickoff()

4. 挑战与解决方案

挑战一：死循环

Agent A 调用 Agent B，Agent B 又调用 Agent A，无限循环。这是多 Agent 系统最常见的崩溃模式。在分布式系统中，这被称为"请求级联"——一个请求触发另一个请求，形成无限递归链。解决方案包括：(1) 设置最大递归深度（通常 3-5 层）；(2) 使用全局调用栈跟踪，检测循环引用；(3) 为每个 Agent 设置独立的超时时间（建议 30-60 秒）；(4) 使用"已访问"标记，避免重复调用同一个 Agent 处理相同的问题。

挑战二：角色混乱

Agent 不清楚自己的职责，越界执行其他 Agent 的任务。例如，搜索 Agent 不仅搜索了信息，还尝试进行分析和总结，导致分析 Agent 无事可做或重复工作。解决方案：(1) 在 system prompt 中明确角色边界——"你只负责搜索，不要分析结果"；(2) 使用工具限制——每个 Agent 只能访问特定的工具集合；(3) 添加输出格式约束——要求 Agent 只输出原始数据，不做加工。

挑战三：通信开销

多 Agent 通信消耗大量 token。每个 Agent 的上下文都需要包含历史通信记录，随着对话轮数增加，token 消耗呈二次方增长。解决方案：(1) 使用消息压缩——只传递关键信息而非完整对话历史；(2) 使用结构化消息格式（JSON）替代自然语言描述；(3) 实现消息摘要——每 N 轮通信后用 LLM 生成摘要替代原始消息；(4) 按需传递——只传递与当前任务相关的信息。

通俗类比：多 Agent 编排就像一个公司的组织架构：Supervisor 模式像"项目经理制"——项目经理（Supervisor）分配任务给各成员（Agent），简单直接但项目经理容易成为瓶颈；Hierarchical 模式像"事业部制"——总部管大方向，各事业部有自己的经理，适合大型项目；Peer-to-peer 像"合伙人制"——大家平等协商，适合创意讨论；Debate 模式像"辩论赛"——正反双方各抒己见，最后得出更全面的结论。
常见误区：1) 一上来就用多 Agent——很多简单任务单 Agent 就够了，多 Agent 反而增加复杂度和成本；2) Agent 之间的通信没有限制，导致 token 消耗爆炸——应该精简消息，用结构化格式；3) 忘记设置最大递归深度，Agent A 调 B、B 调 A 死循环；4) 没有为每个 Agent 设置明确的角色边界——Agent 越界执行其他 Agent 的职责会导致结果混乱和资源浪费。
深入探讨： 多 Agent 系统的"涌现行为"是一个有趣的现象：当多个简单 Agent 交互时，会自发产生复杂的协作模式，就像蚁群中单个蚂蚁很简单但蚁群整体表现出高度智能。LangGraph 的状态图设计正是利用了这一点——通过定义节点（Agent）和边（通信路径），让复杂的协作模式从简单的规则中"涌现"出来。但这也带来了调试困难：当结果不对时，很难定位是哪个 Agent 出了问题。建议在开发阶段开启 verbose 模式，记录每个 Agent 的输入输出。此外，可以通过"逐步回放"功能重现问题场景——记录每个 Agent 在每一步的输入和输出，然后重放这些步骤来定位问题根因。这种可回放性是多 Agent 系统可观测性的核心要素。
开发调试技巧： 多 Agent 系统的调试比单 Agent 困难 10 倍以上。推荐以下调试策略：(1) 先单独测试每个 Agent——确保每个 Agent 在独立运行时能正确完成自己的任务；(2) 逐步增加 Agent 数量——先测 2 个 Agent 的协作，再逐步增加到 3 个、5 个；(3) 使用 Mock 工具——在测试时用模拟工具替代真实工具，避免外部依赖；(4) 记录完整 Trace——记录每个 Agent 的完整对话历史和工具调用链路，便于事后分析。LangSmith 和 LangFuse 等平台提供了专门的多 Agent 可视化工具，可以将整个系统的执行过程以时间线的方式展示。

5. 多 Agent 通信机制的工作原理

5.1 消息传递协议

多 Agent 系统中的通信方式直接影响系统的可扩展性和可靠性。主流的消息传递协议包括三种：直接消息传递（Agent A 直接向 Agent B 发送消息，简单直接但耦合度高）、共享黑板（所有 Agent 读写同一个共享状态空间，解耦但需要同步机制）、以及事件驱动（Agent 发布事件，订阅者自动接收，最灵活但复杂度最高）。在实际系统中，LangGraph 使用共享状态（黑板模式），CrewAI 使用任务链（直接传递），AutoGen 使用对话（消息传递）。

深入探讨： 多 Agent 通信的理论基础来自分布式系统中的"共识问题"。在分布式计算中，多个节点需要就某个值达成一致（Paxos/Raft 协议）。类似地，多 Agent 系统中的"辩论"模式本质上也是一个共识过程——多个 Agent 提出不同观点，通过多轮讨论达成共识。这个过程的数学模型可以用贝叶斯推理框架描述：每个 Agent 维护一个先验信念 P(θ)，收到其他 Agent 的论据后更新为后验信念 P(θ|evidence)。当所有 Agent 的后验信念收敛到同一分布时，共识达成。研究表明，3-5 个 Agent 的辩论规模是最优的——太少缺乏多样性，太多则收敛困难且 token 消耗过大。

5.2 Agent 间状态同步

在多 Agent 系统中，状态管理是最核心的工程挑战之一。每个 Agent 都有自己的本地状态（当前任务、中间结果、上下文历史），但某些状态需要在 Agent 之间共享。例如，搜索 Agent 找到的结果需要传递给分析 Agent。状态同步的策略包括：全量同步（每次通信传递所有状态，简单但浪费带宽）、增量同步（只传递变化的部分，高效但实现复杂）、以及按需同步（只在需要时请求状态，最灵活但可能增加延迟）。

# 多 Agent 状态管理实现
class SharedState:
    """线程安全的共享状态管理器"""

    def __init__(self):
        self._state = {}
        self._lock = asyncio.Lock()
        self._subscribers = {}

    async def update(self, agent_id, key, value):
        """Agent 更新共享状态"""
        async with self._lock:
            self._state[key] = {
                "value": value,
                "updated_by": agent_id,
                "timestamp": datetime.now()
            }
        # 通知订阅该 key 的 Agent
        await self._notify(key, value)

    async def read(self, key):
        """Agent 读取共享状态"""
        return self._state.get(key, {}).get("value")

    def subscribe(self, agent_id, key, callback):
        """订阅状态变化通知"""
        if key not in self._subscribers:
            self._subscribers[key] = []
        self._subscribers[key].append((agent_id, callback))

    async def _notify(self, key, value):
        """通知所有订阅者"""
        for agent_id, callback in self._subscribers.get(key, []):
            try:
                await callback(key, value)
            except Exception as e:
                print(f"通知 Agent {agent_id} 失败: {e}")

全量同步每次传递完整状态简单可靠token 消耗大状态较小
增量同步只传递变化部分高效实现复杂频繁交互

事件驱动状态变化时广播实时性好消息风暴风险实时协作

6. 多 Agent 系统的性能优化

6.1 Token 成本控制

多 Agent 系统的最大成本来自 token 消耗。当 N 个 Agent 互相通信时，每次通信都需要消耗 token。如果每个 Agent 每轮消耗 1000 token，5 个 Agent 的一轮协作就需要 5000 token。如果协作 10 轮，总消耗达到 50000 token。这在生产环境中是不可忽视的成本。

多 Agent Token 消耗分析: 系统配置: 5 个 Agent, 每轮每 Agent 1000 token Supervisor 模式: ┌─────────────────────────────────────────┐ │ 用户 → Supervisor (500 token) │ │ Supervisor → A1 (500 + 任务描述) │ │ A1 → Supervisor (500 + 结果) │ │ Supervisor → A2 (500 + 上下文) │ │ ... │ │ 总计: ~8000 token / 任务 │ └─────────────────────────────────────────┘ Peer-to-peer 模式: ┌─────────────────────────────────────────┐ │ A1 ↔ A2 (1000 × 2) │ │ A2 ↔ A3 (1000 × 2) │ │ A3 ↔ A4 (1000 × 2) │ │ ... 完全连接: C(5,2) = 10 条通信链路 │ │ 总计: ~20000 token / 轮 │ └─────────────────────────────────────────┘ 优化策略: ├── 精简消息: 用结构化 JSON 替代自然语言 ├── 状态共享: 共享黑板避免重复传递 ├── 按需激活: 只激活相关 Agent └── 结果缓存: 相同查询复用历史结果
优化策略节省比例实现复杂度效果影响
消息精简30-50%低无影响
状态共享黑板20-40%中无影响

结果缓存50-80%低可能过时
模型降级60-90%低质量下降
成本优化建议： 在生产环境中，建议为每个 Agent 配置不同的 LLM 模型。Supervisor 需要最强的理解和决策能力，使用 GPT-4o/Claude Opus；搜索 Agent 和数据处理 Agent 的任务相对简单，可以使用 GPT-4o-mini/Claude Haiku；写作 Agent 需要良好的语言能力，使用中等模型即可。这种"模型分层"策略可以在保证质量的前提下降低 50-70% 的成本。同时建议设置全局 token 预算上限，当预算耗尽时自动降级到更便宜的模型。
多 Agent 开发常见陷阱： (1) Agent 数量过多——3-5 个 Agent 通常是最佳规模，超过 7 个后协调成本呈指数增长；(2) 通信协议不一致——Agent 之间的消息格式应该统一（推荐 JSON Schema），否则解析错误频发；(3) 缺乏超时机制——一个 Agent 卡住会阻塞整个系统，每个 Agent 都应该有独立的超时和重试策略；(4) 日志不完整——多 Agent 调试极为困难，必须记录每个 Agent 的输入输出和决策过程。
状态一致性风险： 在并发执行的多 Agent 系统中，状态一致性是一个容易被忽视的问题。当两个 Agent 同时修改共享状态时，可能产生"竞态条件"——后写入的值覆盖了先写入的值，导致数据丢失。解决方案包括：(1) 使用乐观锁机制，检测冲突后重试；(2) 将共享状态设计为不可变数据结构，只允许追加不允许修改；(3) 使用消息队列串行化状态变更操作。在 LangGraph 中，状态更新是通过 reducer 函数完成的，正确使用 reducer 可以避免大部分状态冲突。

7. 多 Agent 系统的评测指标

深入探讨： 多 Agent 系统的评测比单 Agent 复杂得多。除了单 Agent 的指标（任务完成率、准确率、延迟）外，还需要评估：(1) 协作效率——完成相同任务，多 Agent 比单 Agent 快多少？如果多 Agent 反而更慢，说明协调开销过大；(2) 角色分工合理性——每个 Agent 是否在做自己擅长的事？可以通过分析每个 Agent 的工具调用分布来评估；(3) 通信效率——Agent 之间的通信是否冗余？可以通过计算"有效信息比"（有意义的信息量 / 总通信量）来衡量；(4) 可扩展性——增加 Agent 数量后系统性能是提升还是下降？理想情况下，增加 Agent 应该线性提升吞吐量，但实际上由于协调开销，通常是亚线性增长。推荐使用 SWE-bench 和 tau-bench 的多 Agent 变体进行评测。

相关阅读推荐：(1) LangGraph 官方文档——学习图编排的核心概念和 API；(2) CrewAI 教程——掌握角色化多 Agent 的设计模式；(3) AutoGen/MAF 文档——了解企业级多 Agent 框架的设计理念与实现细节；(4) Multi-Agent Systems 论文综述——深入理解多 Agent 系统的理论基础和前沿研究方向。建议在掌握多 Agent 编排后，继续学习Agent 评测与可观测性模块，以确保你的多 Agent 系统在生产环境中稳定运行。
