📋 04 · 规划与推理

进阶

1. ReAct (Reasoning + Acting)

ReAct 是最基础的 Agent 规划模式：交替进行推理（Thought）和行动（Action）。

ReAct 执行流程: 问题: "2024 年诺贝尔物理学奖得主是谁？" Thought 1: 我需要搜索 2024 年诺贝尔物理学奖的信息 Action 1: search("2024 Nobel Prize Physics") Observation 1: [搜索结果: John Hopfield 和 Geoffrey Hinton...] Thought 2: 我找到了得主信息，现在需要了解他们的具体贡献 Action 2: search("Hopfield Hinton contribution neural networks") Observation 2: [搜索结果: 基于人工神经网络的机器学习...] Thought 3: 我现在有足够的信息来回答问题 Action 3: finish("2024 年诺贝尔物理学奖授予 John Hopfield 和 Geoffrey Hinton，表彰他们在基于人工神经网络的 机器学习方面的基础性发现。")
# ReAct Agent 实现
class ReActAgent:
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = tools

    def run(self, query, max_steps=5):
        messages = [{"role": "user", "content": query}]

        for step in range(max_steps):
            # 1. LLM 推理
            response = self.llm.chat(messages)

            # 2. 解析 Thought 和 Action
            thought = extract_thought(response)
            action = extract_action(response)

            if action and action.name == "finish":
                return action.args["answer"]

            # 3. 执行工具
            observation = self.tools[action.name].run(action.args)

            # 4. 注入结果
            messages.append({"role": "assistant", "content": response})
            messages.append({"role": "tool", "content": observation})

        return "达到最大步骤数"

2. Plan-and-Execute

# Plan-and-Execute Agent 实现
class PlanAndExecuteAgent:
    def __init__(self, planner, executor):
        self.planner = planner
        self.executor = executor

    def run(self, query):
        # 1. 生成计划
        plan = self.planner.generate(query)

        results = []
        for step in plan.steps:
            # 2. 执行每一步
            result = self.executor.run(step, context=results)
            results.append(result)

            # 3. 检查是否需要重规划
            if self.needs_replan(result):
                plan = self.planner.replan(query, results)

        return results[-1]  # 返回最终结果

3. Reflexion

# Reflexion Agent 实现
class ReflexionAgent:
    def __init__(self, llm, tools, memory):
        self.llm = llm
        self.tools = tools
        self.memory = memory  # 存储反思

    def run(self, query, max_attempts=3):
        for attempt in range(max_attempts):
            # 1. 执行任务
            result = self.execute(query)

            # 2. 检查是否成功
            if self.is_success(result):
                return result

            # 3. 反思失败原因
            reflection = self.reflect(query, result)

            # 4. 存储反思
            self.memory.store(reflection)

            # 5. 下次执行时参考反思
            query = f"{query}\n注意: {reflection}"

        return "多次尝试后仍失败"

4. Tree of Thoughts (ToT)

Tree of Thoughts: 问题: "用 1, 3, 4, 6 通过 +,-,×,÷ 得到 24" [初始: 1,3,4,6] / | \ [1+3=4] [1×3=3] [3-1=2] / \ | \ [4×4] [4+4] [3×4] [2×4] =16 =8 =12 =8 ✗ ✗ ✗ ✗ 继续探索其他路径... [6÷(1-3/4)] = 24 ✓ ToT 会评估每个中间状态，剪枝不 promising 的路径

5. 各策略对比

策略规划执行反思搜索适用
ReAct隐式单步❌❌简单任务
Plan-Execute显式多步重规划❌复杂任务
Reflexion隐式多轮✅❌试错任务
ToT树搜索多路径评估✅推理密集
通俗类比：ReAct 就像你边想边做——"我需要查一下天气"（想）→ 打开天气App（做）→ "今天25度"（观察）→ "可以穿短袖"（想）→ 告诉朋友（做）。Plan-and-Execute 则像先列购物清单再去超市——先规划好"要买牛奶、面包、鸡蛋"，然后按清单执行，如果发现牛奶卖完了再调整计划。Reflexion 像考试后的错题本——做完题后反思哪里错了，下次遇到类似题目就避免犯同样的错。
常见误区：1) 以为 ReAct 的 Thought 步骤可以省略——没有推理的 Action 就是盲目的工具调用，容易陷入无效循环；2) Plan-and-Execute 的计划一旦生成就不再调整，遇到意外情况无法应对；3) Reflexion 的反思如果没有存入长期记忆，下一次会话就白反思了。记住：好的规划策略要根据任务复杂度选择——简单任务用 ReAct，复杂任务用 Plan-and-Execute。
深入探讨：Tree of Thoughts (ToT) 的核心思想来自蒙特卡洛树搜索（MCTS）：在每个决策点生成多个候选方案，用"评估器"判断哪些路径更有前途，然后只扩展有希望的节点。这在数学推理和创意写作中特别有效——比如"用1,3,4,6得到24"这类问题，暴力搜索空间很大，但 ToT 可以快速剪枝。最新研究如 MCTS LLM 将 AlphaGo 的搜索思想引入 LLM 推理，在复杂推理任务上提升了 30%+ 的准确率。
向前串联： ReAct (Reasoning + Acting)的知识将在后续的推理框架选型、性能基准测试、生产部署中反复应用。 建议结合实际框架动手验证本章所学内容。

规划策略选型速查

选择规划策略时，先评估任务的三个维度：(1) 复杂度——简单查询（1-3步）用 ReAct，复杂任务（5-10步）用 Plan-and-Execute；(2) 确定性——确定性流程用 Plan-and-Execute，不确定性探索用 Tree of Thoughts；(3) 容错性——允许试错用 Reflexion，不允许试错用 Plan-and-Execute。在生产环境中，推荐组合策略：先用 Plan-and-Execute 生成全局计划，每个子步骤内部用 ReAct 执行，如果某步骤失败则用 Reflexion 反思并重试。每种策略都应设置 token 预算上限（ReAct 约 2000-5000，Plan-and-Execute 约 5000-20000，ToT 约 10000-50000）和步数上限，避免失控。

6. 规划策略的工作机制与数学原理

6.1 规划问题的形式化定义

Agent 规划问题可以形式化为一个搜索问题。给定初始状态 s_0、目标状态 s_g、和动作集合 A，规划器需要找到一个动作序列 (a_1, a_2, ..., a_n)，使得执行这些动作后从 s_0 到达 s_g。在传统 AI 中，这通过 STRIPS 或 PDDL 语言描述。但在 LLM Agent 中，状态和动作都是自然语言描述的，无法使用传统的状态空间搜索。LLM 的解决方案是利用其隐式的世界知识来"想象"动作的后果，这是一种基于语言模型的启发式搜索。

从数学角度看，ReAct 的每一步可以表示为一个条件概率：P(a_t | o_t, h_t)，其中 o_t 是当前观察，h_t = (o_1, a_1, o_2, a_2, ..., o_{t-1}, a_{t-1}) 是历史轨迹。LLM 通过注意力机制隐式地维护了这个历史信息。Plan-and-Execute 则将规划过程分解为两步：先计算 P(plan | query) 生成全局计划，再逐步计算 P(a_t | step_t, context) 执行每一步。两者的权衡在于：ReAct 更灵活但缺乏全局视野，Plan-and-Execute 有全局视野但不够灵活。

深入探讨： Reflexion 的反思机制可以用强化学习的框架来理解。在传统 RL 中，策略改进依赖于奖励信号和梯度更新。Reflexion 用自然语言"反思"替代了梯度更新——Agent 生成一段文字描述"为什么失败了"以及"下次应该怎么做"，然后在下次执行时将这段反思注入上下文。从数学角度看，这等价于在策略函数 π 中增加了一个条件变量 r（反思），即 π(a|s) → π(a|s, r)。实验表明，Reflexion 在编程任务（HumanEval）上的通过率比单次执行提高了 10-15%，在决策任务（Alfworld）上提高了 20%+。其优势在于不需要梯度更新，纯推理即可改进策略，且反思文本对人类可解释。

7. Chain-of-Thought 与推理增强

7.1 CoT 的原理与变体

Chain-of-Thought (CoT) 推理是所有 Agent 规划策略的基础。CoT 的核心思想是让模型在给出最终答案之前，先输出中间推理步骤。这通过在 prompt 中给出"让我们一步步思考"的指令或 few-shot 示例来实现。研究表明，CoT 在数学推理、逻辑推理和常识推理任务上可以将准确率提升 10-40%。

推理技术原理推理深度适用场景
Zero-shot CoT"让我们一步步思考"浅简单推理
Few-shot CoT给出推理示例中通用推理
Self-Consistency多次采样投票中需要高准确率
Tree of Thoughts树搜索+剪枝深复杂推理
MCTS-LLM蒙特卡洛树搜索极深数学证明
CoT 的局限性： Chain-of-Thought 并非万能。研究发现：(1) CoT 对小模型（< 10B 参数）效果有限，因为小模型的推理能力不足，生成的推理步骤可能是错误的——错误的推理步骤反而会误导最终答案；(2) CoT 在简单任务上可能产生"过度推理"，增加 token 消耗但不提升准确率；(3) CoT 的推理步骤不一定可信——模型可能生成看似合理但逻辑错误的推理过程。在 Agent 系统中，建议将 CoT 与工具调用结合（如 ReAct），用工具返回的客观事实来"锚定"推理过程，避免纯推理的幻觉问题。
规划策略常见误区： (1) 以为 ReAct 的 Thought 步骤可以省略——没有推理的 Action 就是盲目的工具调用，容易陷入无效循环；(2) Plan-and-Execute 的计划一旦生成就不再调整，遇到意外情况无法应对；(3) Reflexion 的反思如果没有存入长期记忆，下一次会话就白反思了；(4) ToT 的搜索宽度和深度没有限制，导致 token 消耗爆炸。记住：好的规划策略要根据任务复杂度选择——简单任务用 ReAct，复杂任务用 Plan-and-Execute。
实践建议： 在生产环境中，建议为规划策略设置"预算控制"机制。每种策略都应该有 token 预算上限和步数上限。ReAct Agent 通常 5-10 步内完成，token 消耗约 2000-5000；Plan-and-Execute 的计划生成消耗约 500-1000 token，每个子步骤约 1000-2000 token；ToT 的 token 消耗最高，可能达到 10000-50000 token。建议在用户界面显示当前的 token 消耗和预估成本，让用户决定是否继续执行。

8. 规划系统的工程实现

8.1 计划验证与修复

LLM 生成的计划不一定总是正确的。常见的计划错误包括：步骤顺序不合理（先写报告再搜索数据）、遗漏关键步骤（搜索后忘记分析）、步骤粒度不当（过于粗略或过于详细）、以及步骤之间存在逻辑矛盾。为了提高计划质量，可以在计划生成后增加一个"计划验证"步骤：用另一个 LLM 调用对计划进行审查，检查逻辑一致性、完整性和可行性。

# 计划验证与修复机制
class PlanValidator:
    def __init__(self, llm):
        self.llm = llm

    def validate_and_fix(self, query, plan):
        """验证计划质量并自动修复"""
        validation_prompt = f"""
        请评估以下计划的质量:
        用户目标: {query}
        计划: {plan}

        检查项:
        1. 步骤是否完整覆盖了用户目标?
        2. 步骤顺序是否合理(先搜索后分析)?
        3. 是否有遗漏的关键步骤?
        4. 每个步骤的粒度是否合适?

        如果有问题，请返回修复后的计划。如果没问题，返回原计划。
        """
        result = self.llm.generate(validation_prompt)
        return self._parse_plan(result)

8.2 自适应规划策略

最先进的规划系统能够根据任务特征自动选择规划策略。系统首先对用户输入进行"任务分类"，识别任务的复杂度、领域和约束条件，然后选择最适合的规划策略。例如，简单查询类任务自动使用 ReAct，复杂分析类任务自动使用 Plan-and-Execute，需要反复尝试的编程任务自动使用 Reflexion。这种自适应机制的核心原理是：在任务开始前用一个轻量级 LLM 调用进行任务分类（消耗约 100 token），然后根据分类结果选择策略，从而在效率和质量之间取得最优平衡。

自适应规划策略选择: 用户输入 │ ▼ ┌─ 任务分类器 ─────────────────────────────┐ │ 复杂度: 低/中/高 │ │ 类型: 查询/分析/创作/调试 │ │ 步骤预估: 1-3 / 3-10 / 10+ │ └───────────────┬───────────────────────────┘ ▼ ┌─ 策略路由 ────────────────────────────────┐ │ │ │ 低复杂度 + 查询 → ReAct (快速模式) │ │ 中复杂度 + 分析 → Plan-and-Execute │ │ 高复杂度 + 创作 → Plan + ReAct + Reflexion│ │ 调试/试错类 → Reflexion │ │ 推理密集型 → Tree of Thoughts │ └───────────────────────────────────────────┘
