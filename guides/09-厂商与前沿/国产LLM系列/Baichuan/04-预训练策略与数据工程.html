3.2 去重策略详解

# MinHash + LSH 去重流程
import hashlib
from datasketch import MinHash, MinHashLSH

def minhash_dedup(documents, threshold=0.8):
    """
    MinHash 去重: 将文档转为 n-gram 集合,
    计算 MinHash 签名, 通过 LSH 快速找到近似重复对
    """
    lsh = MinHashLSH(threshold=threshold, num_perm=128)

    for idx, doc in enumerate(documents):
        m = MinHash(num_perm=128)
        for word in doc.split():
            m.update(word.encode('utf8'))
        lsh.insert(str(idx), m)

    # 查询相似文档对并去重
    return lsh

# 百川去重统计 (估算):
# 原始语料: ~10B 网页
# URL去重:  -30% → ~7B
# 文档去重:  -20% → ~5.6B
# 段落去重:  -10% → ~5B (最终保留)

4.2 学习率调度

# 百川学习率调度 (类 cosine schedule with warmup)
import math

def baichuan_lr_schedule(step, total_steps, peak_lr, warmup_steps):
    """
    Warmup + Cosine Decay 调度策略
    - Warmup 阶段: 线性增长
    - 训练阶段: 余弦衰减到 min_lr
    """
    min_lr = peak_lr * 0.1  # 衰减到峰值的 10%

    if step < warmup_steps:
        return peak_lr * (step / warmup_steps)

    progress = (step - warmup_steps) / (total_steps - warmup_steps)
    return min_lr + 0.5 * (peak_lr - min_lr) * (1 + math.cos(math.pi * progress))

# Baichuan-7B 典型超参:
# peak_lr = 3e-4
# warmup_steps = 2000
# total_steps = ~100K (2.6T tokens / batch_size)
# batch_size = 动态增长

  1. 核心原理: 百川预训练数据概览的核心原理是"数据决定模型上限"——通过高质量的中文语料工程，为大语言模型提供丰富的语言知识和世界知识。百川的预训练数据体系包含三大要素：数据规模（2.6T tokens）、数据质量（六层清洗管线）和数据多样性（中文文本、英文文本、代码、数学、学术论文等多源语料）。核心创新在于中文语料的"原生性"——百川不是在英文基座模型上叠加中文数据，而是从一开始就以中文为核心构建训练语料。具体做法包括：①125K超大中文词表覆盖GB2312+GBK+常用罕见字；②中文语料占比35-40%，通过精心配比确保中文能力；③引入约500B tokens的搜索引擎语料增强事实性和时效性；④使用"时效性加权采样"策略，越新的文档获得越高的采样概率。实际应用场景：在构建中文智能客服系统时，百川模型的中文理解能力（成语释义、方言理解、文化背景知识）显著优于在英文基座上叠加中文数据的方案。例如，理解"画蛇添足""塞翁失马"等成语的引申义，以及"打工人""内卷"等网络用语的语境含义，这些都得益于百川原生中文训练数据的高质量。
    关键要点: ①数据规模：2.6T tokens，是7B/13B模型的充足训练数据；②数据质量：六层清洗管线从10B网页筛选到5B高质量文档；③中文原生性：125K词表+35-40%中文占比，而非英文基座叠加中文；④时效性：搜索引擎语料+时间衰减加权采样。
    实际案例: 百川的中文token效率——同样一句"百川智能是中国领先的AI公司"，Baichuan2编码为约8个token，LLaMA编码为约15个token。这意味着在相同上下文窗口下，Baichuan2能处理的中文文本量接近LLaMA的两倍。
    关联知识: 125K词表的构建策略与第3章的注意力机制相关（词表大小影响序列长度），搜索引擎语料的预训练策略与第2章的RAG技术一脉相承，数据清洗管线的工程实现在本章第3节详述。
  2. 核心原理: 百川预训练数据在实际工程中可能遇到的性能瓶颈主要集中在数据处理效率、数据质量控制和训练稳定性三个方面。数据处理效率方面，PB级语料的清洗和去重是计算密集型任务——MinHash去重需要对每个文档计算128个permutation的签名，在10B文档规模下需要数千核小时的计算。数据质量控制方面，自动化清洗管线可能误删高质量文档（假阳性）或保留低质量文档（假阴性），需要人工抽样验证和持续迭代。训练稳定性方面，大规模预训练中的Loss spike（损失突增）是常见问题，可能由数据中的异常样本（如乱码、极长文本）引起。诊断方法：①使用数据质量监控仪表盘跟踪每个清洗阶段的数据分布变化；②使用Perplexity过滤器识别异常低质文本（Perplexity>阈值）；③使用梯度范数监控识别训练中的不稳定信号。解决方案包括：①使用分布式数据处理框架（如Apache Spark）加速清洗管线；②建立多层次的质量评估体系（自动化+人工抽检+模型自评估）；③训练中使用梯度裁剪（max_norm=1.0）和BF16混合精度保障稳定性；④遇到Loss spike时回退到上一个checkpoint重新训练。
    关键要点: ①PB级数据清洗是计算密集型任务，需要分布式框架支持；②自动清洗的假阳性/假阴性问题需要人工抽检补充；③Loss spike是大规模训练的常见问题，通过checkpoint回退解决；④BF16混合精度比FP16更稳定（避免了FP16的溢出问题）。
    实际案例: 百川在Baichuan-13B训练过程中遇到了多次Loss spike，分析发现是由一批包含大量HTML标签残留的低质量网页引起的。解决方案：①回退到spike前的checkpoint；②加强HTML清洗规则；③增加Perplexity过滤的阈值。重新训练后Loss曲线恢复平滑。
    关联知识: BF16 vs FP16的数值精度差异在深度学习系统工程中有详细分析，分布式训练的三维并行策略在本章第4节展开，checkpoint管理是大规模训练的基础设施。
  3. 核心原理: 百川预训练数据的替代方案主要包括：①纯爬虫语料（如Common Crawl的原始数据）——优势是规模大、获取成本低，劣势是质量参差不齐、中文占比低；②合成数据（使用LLM生成训练数据）——优势是可以定向生成特定领域的数据，劣势是存在"模型崩塌"风险（用LLM的输出训练LLM会导致能力退化）；③开源数据集（如RedPajama、The Pile）——优势是标准化、可复现，劣势是中文覆盖不足、时效性差；④商业数据（从出版商、媒体公司购买）——优势是质量高、版权清晰，劣势是成本高、获取渠道有限。百川的方案是"多源混合+精细清洗"——综合使用网页语料（~60%）、书籍语料（~15%）、专业语料（~10%）、搜索引擎语料（~15%），通过六层清洗管线确保质量。权衡要点：如果预算充足且对数据质量要求极高，商业数据+开源数据集的组合是最安全的方案；如果需要大规模中文语料且有工程能力，百川式的多源混合方案性价比最高；如果资源有限，可以在开源数据集基础上补充少量高质量中文语料。
    关键要点: ①纯爬虫语料：大而粗糙，需要大量清洗工程；②合成数据：定向但有"模型崩塌"风险；③开源数据集：标准但中文不足；④百川方案：多源混合+六层清洗是最佳实践。
    实际案例: 某研究团队尝试使用GPT-4合成中文训练数据来微调7B模型——在通用对话任务上效果提升了约3%，但在需要事实准确性的任务上（如知识问答）效果反而下降了约5%，原因是合成数据中的"幻觉"被放大。这验证了百川"真实语料+精细清洗"策略的正确性。
    关联知识: 合成数据的"模型崩塌"问题是当前LLM研究的热点话题，开源数据集的构建方法论在RedPajama和The Pile的论文中有详细描述，搜索引擎语料的时效性加权采样在本章第5节详述。
