Xiaomi

1. 训练数据

1. 训练数据的工程实践涉及数据管线、分布式训练和checkpoint管理三大子系统。Xiaomi的数据管线支持PB级数据的实时处理和质量监控；分布式训练系统采用数据并行+张量并行+流水线并行的三维并行策略；checkpoint管理支持增量保存和快速恢复。这些基础设施的完善程度直接决定了模型训练的效率和稳定性。

训练数据的核心创新点分析

MiMo 48T tokens多模态数据配比的课程学习调度策略

MiMo的48T tokens训练数据并非均匀混合，而是采用课程学习（Curriculum Learning）策略动态调整各模态数据的配比。训练过程分为三个阶段：Phase 1（前30%训练量）以纯文本数据为主（90%文本+10%图文），目标是建立扎实的语言理解和生成能力基础；Phase 2（30%-60%）逐步增加多模态数据比例（60%文本+25%图文+10%音频+5%视频），让模型开始学习跨模态关联；Phase 3（60%-100%）进入全模态训练（45%文本+25%图文+15%音频+15%视频），重点提升多模态推理和生成能力。这种渐进式调度避免了训练初期多模态数据的梯度冲突导致的不稳定性，同时确保模型在每个阶段都能专注于当前最重要的能力学习。在代码数据方面，5T tokens的代码语料贯穿全部三个阶段，占比稳定在约10%，因为代码能力被视为通用推理能力的重要组成部分。

MiMo数据质量评估的多层自动过滤管道实现细节

面对120T+ tokens的原始数据量，MiMo构建了四层自动过滤管道将数据精炼到48T tokens（利用率约40%）。第一层为基础过滤（过滤率约15%）：通过URL哈希去重消除重复爬取、语言检测模型过滤非目标语言、编码规范化修复乱码文本、长度过滤移除过短（<50字符）或过长（>100K字符）的异常文档。第二层为质量评估（过滤率约25%）：使用参考语言模型计算文本困惑度（Perplexity），过滤困惑度异常高的噪声文本和异常低的模板化文本；同时运行质量分类器（基于BERT微调的二分类模型）为每篇文档打分，低质量文档被移除或降采样。第三层为去重与多样性（过滤率约15%）：MinHash+LSH算法检测Jaccard相似度>0.8的近似重复文档，语义去重基于嵌入向量的余弦相似度进一步消除主题重复；多样性采样确保不同领域、语言、风格的数据均衡覆盖。第四层为安全过滤（过滤率约5%）：毒性分类器检测有害内容、NER+正则表达式移除个人信息（PII）、版权检查过滤侵权内容。

1. 训练数据的训练流程通常分为三个阶段：预训练(Pre-training)在大规模无标注语料上学习语言表示，微调(Fine-tuning)在任务特定数据上适配下游任务，对齐(Alignment)通过人类偏好反馈优化输出质量。每个阶段的数据配比、学习率调度和正则化策略都需要精心设计。

MiMo-V2.5在48T tokens上训练，是国产模型中训练数据量最大的之一，也是全球范围内训练数据规模最大的模型之一。如此大规模的训练数据覆盖了网页文本、代码、图文对、音频和视频等多种模态，为MiMo的原生多模态能力奠定了坚实基础。

数据质量控制是大规模预训练的关键挑战。48T tokens的原始数据经过了多轮清洗、去重和质量过滤，最终有效训练数据的利用率约为30-40%。这意味着小米实际处理的原始数据量可能超过120T tokens。

通俗类比： 预训练数据就像培养一个"全能学霸"的教材。48T tokens相当于让这个学生读了数千万本书、看了数百万小时视频、听了数百万小时音频。但关键不在于"读了多少"，而在于"读的是什么"。就像给学生选教材一样——不能只看数量，还要看质量。垃圾教材读再多也培养不出好学生，反而可能"学坏"（模型产生有害输出）。所以数据清洗就像"选教材"——比收集教材本身更重要。
常见误区： 误区一：以为训练数据越多越好。数据质量远比数量重要——低质量数据不仅浪费计算资源，还可能"污染"模型能力。研究表明，在10T高质量tokens上训练可能优于在50T混合质量tokens上训练。误区二：忽略数据去重的重要性。重复数据会导致模型"过拟合"到特定文本模式，降低泛化能力。MiMo在训练前进行了多层次去重（URL级、文档级、段落级）。误区三：认为多模态数据可以简单拼接。不同模态的数据需要不同的预处理流程和质量标准，图文对的质量（图文匹配度）比数量更重要。误区四：忽略数据配比的影响——网页文本、代码、图文数据的混合比例会显著影响模型在不同任务上的表现，需要反复实验找到最优配比。
深入探讨： 48T tokens的训练规模引发了几个深层技术问题。首先是"数据墙"问题——全球高质量文本数据的总量估计在100-200T tokens，48T已经接近可用数据的上限。这意味着未来的模型扩展可能更多依赖合成数据和多模态数据，而非单纯的文本数据量增长。MiMo在这方面已经有所布局——其图文、音频、视频数据的加入不仅扩展了模态能力，也间接缓解了文本数据的瓶颈。 其次是数据配比的优化问题。MiMo的多模态预训练需要精心设计不同模态数据的混合比例。如果网页文本占比过高，模型的语言能力会很强但多模态能力弱；如果图文数据占比过高，可能影响纯文本任务的表现。小米的策略可能是采用"课程学习"（Curriculum Learning）——先以文本数据为主训练基础语言能力，然后逐步增加多模态数据的比例，让模型在不同阶段学习不同层次的能力。 第三是数据质量评估的自动化问题。面对48T tokens的数据量，人工质量评估完全不可行。小米可能采用了多层自动过滤管道：1) 基于规则的过滤（URL黑名单、长度过滤等）；2) 基于分类器的质量评分（训练一个质量评估模型）；3) 基于困惑度的过滤（用参考模型计算文本困惑度，过滤异常值）；4) 基于多样性的采样（确保不同领域、语言、风格的数据都有代表性）。

2. 多模态预训练

Xiaomi在2. 多模态预训练方面的技术积累体现在数据质量控制和训练策略优化两个维度。在数据方面，Xiaomi建立了多层数据过滤管线：基于规则的过滤、基于分类器的质量评分和基于多样性的采样。在训练策略方面，Xiaomi采用了学习率warmup、梯度裁剪和z-loss正则化等技术保障训练稳定性。这些措施确保了模型在大规模训练中的一致性和可复现性。

2. 多模态预训练的性能优化策略

2. 多模态预训练的性能优化涉及多个维度。在计算层面，通过算子融合和混合精度推理减少计算量。在内存层面，通过KV Cache压缩和显存池化减少内存占用。在通信层面，通过计算-通信重叠和梯度压缩减少通信开销。

# DPO训练核心实现
import torch
from transformers import AutoModelForCausalLM

def compute_logprobs(model, input_ids, labels):
    """计算序列的对数概率"""
    logits = model(input_ids).logits
    log_probs = torch.nn.functional.log_softmax(logits, dim=-1)
    # 只计算response部分的log prob
    token_logps = log_probs.gather(2, labels.unsqueeze(-1)).squeeze(-1)
    return token_logps.sum(dim=-1)

def dpo_loss(policy, ref, chosen, rejected, beta=0.1):
    """DPO损失：无需奖励模型的偏好优化"""
    pi_chosen = compute_logprobs(policy, chosen['input_ids'], chosen['labels'])
    pi_rejected = compute_logprobs(policy, rejected['input_ids'], rejected['labels'])
    with torch.no_grad():
        ref_chosen = compute_logprobs(ref, chosen['input_ids'], chosen['labels'])
        ref_rejected = compute_logprobs(ref, rejected['input_ids'], rejected['labels'])

    # 核心公式：隐式奖励差的sigmoid
    log_ratio_chosen = pi_chosen - ref_chosen
    log_ratio_rejected = pi_rejected - ref_rejected
    loss = -torch.nn.functional.logsigmoid(beta * (log_ratio_chosen - log_ratio_rejected))
    return loss.mean()

多模态预训练的核心创新点分析

2. 多模态预训练的训练流程通常分为三个阶段：预训练(Pre-training)在大规模无标注语料上学习语言表示，微调(Fine-tuning)在任务特定数据上适配下游任务，对齐(Alignment)通过人类偏好反馈优化输出质量。每个阶段的数据配比、学习率调度和正则化策略都需要精心设计。

MiMo的独特之处是原生多模态预训练——视觉和音频编码器在预训练阶段就与语言模型一起训练，而非后期拼接。这种"从头一起训练"的策略带来了几个关键优势：模态间的对齐更自然、跨模态理解更深入、多模态推理更流畅。

2.1 原生多模态 vs 后接多模态

大多数多模态模型采用"后接"策略——先训练一个强大的语言模型，然后在其基础上接入视觉编码器（如CLIP ViT），通过微调实现多模态能力。MiMo则采用"原生"策略——视觉编码器、音频编码器和语言模型从预训练的第一天就一起训练。

通俗类比： 原生多模态训练就像"从小双语教育"——孩子从出生就同时接触两种语言，大脑的语言区域会自然地同时处理两种语言，切换自如。而后接多模态就像"成年后学外语"——虽然也能学会，但母语和外语在大脑中是分开处理的，切换时需要额外的"翻译"步骤，效率和自然度都不如从小双语。MiMo选择了"从小双语"的路线，虽然前期投入更大（需要同时准备多模态数据），但最终的跨模态理解能力更自然、更深入。
常见误区： 误区一：认为原生多模态一定比后接多模态好。虽然理论上原生训练的对齐更好，但实际效果还取决于训练数据的质量和规模。如果多模态数据不足，原生训练的效果可能不如在强大语言模型基础上后接视觉编码器。误区二：忽略原生多模态训练的巨大成本。同时训练语言模型和多个编码器需要的计算资源远超纯文本训练，小米在这方面的投入是巨大的。误区三：以为原生多模态意味着所有模态从第一天就等比例训练——实际上通常采用课程学习策略，先以文本为主建立基础语言能力，再逐步增加多模态数据。
深入探讨： 原生多模态预训练的技术挑战远超后接方案。首先是模态间的梯度冲突问题——语言模型和视觉编码器的梯度方向可能不一致，导致训练不稳定。解决方案包括：1) 差异化学习率——为不同模态的组件设置不同的学习率；2) 梯度裁剪——限制各模态梯度的最大范数；3) 交替训练——在不同阶段侧重不同模态的优化。 其次是模态对齐的"鸡生蛋"问题。在训练初期，语言模型还没有学会"理解"视觉特征，视觉编码器也没有学会"用语言描述"视觉内容。两者需要在训练过程中互相引导、共同进步。MiMo可能采用了以下策略：1) 使用预训练的CLIP视觉编码器作为初始化，提供基础的视觉-语言对齐；2) 在训练初期使用大量图文配对数据，强制建立模态间的映射关系；3) 引入跨模态对比学习目标，显式优化模态间的对齐质量。 第三是多模态数据的tokenization问题。文本有成熟的tokenization方案（如BPE），但图像和音频需要不同的tokenization策略。MiMo可能采用了"视觉token"方案——将图像切分为patch，每个patch通过视觉编码器映射为一个token，与文本token在同一个序列中处理。音频则可能采用类似的"音频token"方案。这种统一的token空间使得不同模态的信息可以在同一个Transformer中自由交互。

3. 数据质量控制体系

3. 数据质量控制体系的工程实践涉及数据管线、分布式训练和checkpoint管理三大子系统。Xiaomi的数据管线支持PB级数据的实时处理和质量监控；分布式训练系统采用数据并行+张量并行+流水线并行的三维并行策略；checkpoint管理支持增量保存和快速恢复。这些基础设施的完善程度直接决定了模型训练的效率和稳定性。

3. 数据质量控制体系的系统架构设计

3. 数据质量控制体系在Xiaomi中采用了分层架构设计。底层负责硬件级别的计算调度和内存管理，中间层实现核心算法逻辑包括注意力计算和前馈网络，上层提供标准化的API接口。这种分层设计使得各层可以独立优化，不影响整体系统的稳定性。

3.1 去重策略

数据去重是数据质量控制的第一道防线。重复数据会导致模型过拟合到特定文本模式，降低泛化能力，并浪费宝贵的训练算力。MiMo采用了多层次去重策略：

  • URL级去重: 移除同一URL的重复爬取结果，消除时间维度的重复
  • 文档级去重: 使用MinHash + LSH算法检测近似重复文档，阈值通常设为Jaccard相似度>0.8
  • 段落级去重: 检测跨文档的重复段落和模板化内容
  • N-gram去重: 移除高频重复的n-gram序列，如网站模板、版权声明等

3.2 质量评估

质量评估用于筛选出高质量的训练数据。MiMo可能采用了多种质量信号的组合：

  • 困惑度过滤: 使用参考语言模型计算文本困惑度，过滤异常高或异常低的文本
  • 质量分类器: 训练一个二分类模型区分"高质量"和"低质量"文本
  • 规则过滤: 基于启发式规则过滤（如文本长度、特殊字符比例、语言纯度等）
  • 领域分类: 对文本进行领域分类，确保各领域数据的均衡覆盖
通俗类比： 数据质量控制就像食品加工流水线。去重是"挑出重复的食材"——同一颗土豆不需要出现在每道菜里。质量评估是"检查食材新鲜度"——烂掉的蔬菜要扔掉，不新鲜的肉要淘汰。安全过滤是"检测有害物质"——农药残留超标（有害内容）的食材必须剔除。多样性保障是"确保营养均衡"——不能只吃肉不吃菜（只有网页文本没有代码），也不能只吃甜食（只有娱乐内容没有科学知识）。最终的48T tokens就像精心挑选的"营养均衡的食材组合"。
常见误区： 误区一：过度过滤导致数据多样性下降。质量阈值设得太高会过滤掉大量"边缘"数据，这些数据虽然质量一般但对模型的泛化能力很重要。误区二：忽略去重的副作用。过于激进的去重可能移除合法的重复内容（如常见编程模板、标准文档格式），影响模型在这些模式上的学习。误区三：以为质量评估模型本身是完美的。质量分类器可能有偏见——它倾向于给"像维基百科"的文本打高分，而给口语化、非正式但有价值的文本打低分。误区四：只关注文本质量而忽略多模态数据的图文匹配度——一张模糊的图片配一个错误的描述，比没有这张图片更糟糕。
深入探讨： 数据质量控制的核心难题是"质量"本身的定义。什么算"高质量"数据？对于不同的下游任务，答案可能完全不同。学术论文是高质量的推理训练数据，但可能是糟糕的对话训练数据。代码注释是高质量的代码理解数据，但可能对通用语言能力贡献有限。MiMo的策略可能是"任务感知的数据选择"——根据不同训练阶段的目标动态调整质量标准。 另一个深层问题是合成数据的角色。随着自然文本数据逐渐耗尽，合成数据将成为未来模型训练的重要补充。MiMo可能已经使用了合成数据来增强特定能力（如数学推理、代码生成）。但合成数据存在"模型崩塌"（Model Collapse）的风险——如果用模型A的输出训练模型B，B的输出训练C，如此循环，模型能力会逐代退化。解决方法包括：1) 混合自然数据和合成数据；2) 对合成数据进行严格的质量验证；3) 使用人类反馈来校准合成数据的质量。 第三个关键问题是数据的时效性。48T tokens的训练数据中必然包含大量过时信息（如旧版本的API文档、过期的新闻）。如何在训练中平衡"历史知识"和"最新信息"是一个开放问题。MiMo可能采用了"时间加权"策略——对更新的数据赋予更高的采样权重，同时保留一定比例的历史数据以维持模型的"历史素养"。

4. 训练策略与优化

Xiaomi在4. 训练策略与优化方面的技术积累体现在数据质量控制和训练策略优化两个维度。在数据方面，Xiaomi建立了多层数据过滤管线：基于规则的过滤、基于分类器的质量评分和基于多样性的采样。在训练策略方面，Xiaomi采用了学习率warmup、梯度裁剪和z-loss正则化等技术保障训练稳定性。这些措施确保了模型在大规模训练中的一致性和可复现性。

训练策略与优化的核心创新点分析

4. 训练策略与优化的训练流程通常分为三个阶段：预训练(Pre-training)在大规模无标注语料上学习语言表示，微调(Fine-tuning)在任务特定数据上适配下游任务，对齐(Alignment)通过人类偏好反馈优化输出质量。每个阶段的数据配比、学习率调度和正则化策略都需要精心设计。

除了数据工程，训练策略本身也是MiMo成功的关键因素。48T tokens的训练需要精心设计学习率调度、梯度优化、分布式训练策略等技术细节，以确保训练过程的稳定性和效率。

4.1 学习率调度

大规模预训练通常采用Warmup + Cosine Decay的学习率调度策略。MiMo的48T tokens训练可能需要数百个epoch（如果数据有重复的话），学习率的设置需要在收敛速度和训练稳定性之间取得平衡。

4.2 分布式训练

1T参数的模型训练需要数千张GPU协同工作。MiMo的训练可能采用了3D并行策略：数据并行（DP）、张量并行（TP）和流水线并行（PP）的组合，以最大化训练效率。

通俗类比： 大规模预训练就像建造一座摩天大楼。学习率调度就像"施工节奏"——开始时要慢（warmup），打稳地基；中间可以快一些（高速训练）；最后要精细调整（cosine decay）。分布式训练就像"多个施工队同时作业"——有的负责打地基（底层参数），有的负责建中间层，有的负责封顶。通信就像"施工队之间的协调"——如果协调不好（通信瓶颈），反而会比一个施工队更慢。MiMo的3D并行策略就像"把大楼分成多个区域，每个区域由专门的施工队负责，同时通过电梯（NVLink）和电话（网络）保持协调"。
常见误区： 误区一：以为增加GPU数量就能线性加速训练。通信开销会随着GPU数量增加而增大，存在最优并行度。误区二：忽略训练不稳定性。大规模训练中经常出现loss spike（损失突然飙升），需要及时检测和处理（如回滚到上一个checkpoint）。误区三：认为训练数据可以无限重复。重复训练数据会导致过拟合，MiMo的48T tokens可能刚好覆盖了1-2个epoch，避免了过度重复。误区四：忽略混合精度训练的数值稳定性——BF16训练虽然高效，但在某些操作（如梯度累积、损失计算）中需要保持FP32精度以避免数值溢出。
深入探讨： MiMo的训练面临几个独特的技术挑战。首先是MoE架构的负载均衡训练损失——为了确保不同专家被均匀激活，需要在训练损失中加入辅助的负载均衡损失项。这个损失项的权重需要精心调优：太大会影响主任务的学习，太小会导致专家利用率不均。 其次是多模态训练的梯度冲突。文本、图像、音频三种模态的梯度方向可能不一致，导致训练震荡。解决方案包括：1) 梯度手术（Gradient Surgery）——将冲突梯度投影到正交子空间；2) 不确定性加权——根据各任务的不确定性自动调整损失权重；3) 交替优化——在不同step侧重不同模态。 第三是训练效率优化。1T参数模型在数千张GPU上的训练，每一步的计算和通信开销都是巨大的。关键技术包括：1) 重计算（Activation Checkpointing）——用时间换空间，减少显存占用；2) 通信重叠——将通信操作与计算操作重叠执行；3) 混合精度训练——在FP16/BF16下计算，FP32下更新参数；4) ZeRO优化——将优化器状态分片到不同GPU上，减少每张GPU的显存压力。

小米IoT生态数据飞轮的独特竞争优势与持续学习架构

小米构建的数据飞轮是MiMo区别于其他国产大模型的核心竞争优势之一。小米拥有全球最大的消费级IoT设备生态——截至2026年连接设备超过8亿台，涵盖手机、电视、智能音箱、手表、家电和汽车等品类。这些设备产生的多模态交互数据（语音指令、屏幕截图、设备状态、用户行为序列等）构成了独特的、其他厂商难以复制的训练数据资产。在持续学习架构方面，小米采用"季度全量重训+月度增量更新"的双轨策略：每季度使用完整的更新后数据集进行全量重训以防止能力退化，每月使用新增的高质量数据进行增量训练以快速适配新的应用场景。增量训练采用较低的学习率（约为全量训练的1/10）和较少的训练步数（约全量的5%），配合EWC（弹性权重巩固）正则化防止灾难性遗忘。用户反馈数据经过匿名化和聚合处理后进入训练数据池，形成"模型服务→用户反馈→数据改进→模型提升"的正向循环。

MiMo的48T tokens训练并非终点，而是一个持续演进的过程。小米构建了"数据飞轮"——模型的使用反馈被用来改进下一轮训练的数据质量，形成正向循环。这种持续学习的能力使得MiMo能够不断适应新的应用场景和用户需求。

通俗类比： 数据飞轮就像"滚雪球"——雪球（模型）滚过雪地（数据）时会越滚越大（能力提升），而雪球越大，它能接触到的雪面（新数据）也越多，进而滚得更快。用户使用MiMo的反馈（如点赞/踩、重新生成、任务成功率等）就像"雪的质量指标"——好的反馈意味着这片雪（数据分布）是适合滚雪球的，不好的反馈则提示需要调整方向。

从数学原理来看，大模型推理的性能瓶颈可以用阿姆达尔定律推导：加速比 S = 1/((1-P) + P/N)，其中P为可并行化比例，N为并行度。当P=0.95且N=8时，理论加速比为S=5.9x。这一数学模型解释了为什么即使增加更多GPU，推理速度也不会线性提升——瓶颈在于不可并行的串行部分（如路由决策和结果聚合）。通过优化串行部分的执行效率，可以将P从0.95提升到0.98，从而使加速比从5.9x提升到12.3x。

# 训练数据 性能基准测试
import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
