MiMo-V2.5支持1M tokens原生上下文窗口，是国产模型中最长的之一。这意味着模型可以一次性处理约75万字的中文文本，相当于一整本长篇小说的长度。1M上下文窗口的实现并非简单地将标准Transformer的注意力机制扩展到更长序列，而是需要在架构层面进行系统性创新。

标准Transformer的自注意力机制的计算复杂度为O(n²)，当序列长度从4K扩展到1M时，计算量增加了约62500倍。这在工程上是不可接受的。MiMo采用了一系列技术来实现高效长上下文处理：混合滑动窗口注意力、稀疏注意力模式、KV Cache优化和位置编码外推。

通俗类比： 标准注意力就像在一个会议上，每个人都要和所有其他人交流——人越多，交流次数增长越快（平方级增长）。1M上下文就像一个100万人的超级会议，如果每个人都和其他所有人交流，那是不可能的。MiMo的解决方案就像把这100万人分成小组（滑动窗口），每个人只和身边的人深入交流，但通过"信息接力"（层级化传递）让远处的信息也能逐步传播过来。这样既保持了信息流通，又控制了计算成本。
常见误区： (1)认为上下文越长模型越聪明——上下文长度只是"工作记忆"大小，不等于推理能力。一个1M上下文的模型在简单问答任务上可能还不如一个4K上下文但训练更好的模型。(2)忽略"Lost in the Middle"问题——研究表明，模型对上下文开头和结尾的信息记忆更好，中间部分的信息容易被忽略。MiMo通过特殊的注意力偏置来缓解这一问题。(3)混淆"支持"和"有效利用"——很多模型声称支持长上下文，但在实际使用中，超过一定长度后回答质量会急剧下降。MiMo在1M上下文范围内保持了较好的质量一致性。
深入探讨： MiMo的1M上下文实现包含以下关键技术：(1)混合滑动窗口注意力（Hybrid Sliding Window Attention）——不同层使用不同大小的窗口，浅层使用小窗口捕捉局部信息，深层使用大窗口整合全局信息；(2)RoPE外推——通过YaRN（Yet another RoPE extensioN）技术将位置编码从训练长度外推到更长序列，避免位置编码饱和导致的性能下降；(3)KV Cache压缩——对远距离的Key-Value对进行量化压缩或选择性丢弃，将显存占用控制在合理范围内；(4)分块处理——将超长序列分成多个块，通过块间信息传递机制实现跨块的注意力交互。

# DPO训练核心实现
import torch
from transformers import AutoModelForCausalLM

def compute_logprobs(model, input_ids, labels):
    """计算序列的对数概率"""
    logits = model(input_ids).logits
    log_probs = torch.nn.functional.log_softmax(logits, dim=-1)
    # 只计算response部分的log prob
    token_logps = log_probs.gather(2, labels.unsqueeze(-1)).squeeze(-1)
    return token_logps.sum(dim=-1)

def dpo_loss(policy, ref, chosen, rejected, beta=0.1):
    """DPO损失：无需奖励模型的偏好优化"""
    pi_chosen = compute_logprobs(policy, chosen['input_ids'], chosen['labels'])
    pi_rejected = compute_logprobs(policy, rejected['input_ids'], rejected['labels'])
    with torch.no_grad():
        ref_chosen = compute_logprobs(ref, chosen['input_ids'], chosen['labels'])
        ref_rejected = compute_logprobs(ref, rejected['input_ids'], rejected['labels'])

    # 核心公式：隐式奖励差的sigmoid
    log_ratio_chosen = pi_chosen - ref_chosen
    log_ratio_rejected = pi_rejected - ref_rejected
    loss = -torch.nn.functional.logsigmoid(beta * (log_ratio_chosen - log_ratio_rejected))
    return loss.mean()

  1. 核心原理: 1M上下文的核心原理可以概括为"以空间换时间的容量扩展"——通过将模型的注意力窗口从传统的4K-32K扩展到1M token，让模型能够一次性"看到"和处理极其长的大规模输入，避免了频繁的分段和信息丢失。从技术层面，这需要解决三个核心难题：(1)注意力计算量——标准注意力的O(n²)复杂度在1M长度下计算量暴增62500倍，MiMo通过分层滑动窗口架构将复杂度降至O(n·w*)级别；(2)位置编码饱和——传统的RoPE在超过训练长度后位置信息失效，MiMo通过YaRN技术将Theta参数从10,000提升至640,000来扩展位置感知范围；(3)KV Cache显存——1M token的KV Cache原始大小约16GB，MiMo通过4-bit量化和选择性丢弃压缩到约2.8GB。其实际意义在于：开发者在分析一个有5000个文件的大型代码库时，可以将整个代码库一次性提交给MiMo，让它理解架构全貌、发现跨文件依赖缺陷、甚至重构建议——这是短上下文模型完全无法企及的能力。
    关键要点: ①分层滑动窗口(浅层局部→中层段落→深层全局稀疏)控制计算量；②YaRN外推扩展位置编码感知范围；③KV Cache量化压缩管理显存；④渐进式训练(4K→32K→128K→1M)确保性能平滑过渡
    实际案例: 在金融领域，分析师可以将公司过去5年的全部年报(约3000页文本，合60万token)一次性输入MiMo，让模型提取关键财务趋势、识别风险因素、生成综合投资分析报告——整个过程无需分段也无需担心信息遗漏。
    关联知识: 1M上下文的价值取决于"有效利用长度"而非"支持长度"——MiMo在NIAH测试中1M下的96.2%检索准确率证明了它不是简单地"支持"1M，而是能在整个范围内有效工作。
  2. 核心原理: 1M上下文在实际工程中面临多重性能瓶颈：(1)Prefill瓶颈——处理1M输入的Prefill阶段需要O(n²)注意力计算，单次Prefill可能耗时数十秒，MiMo通过Ring Attention将序列分布到多GPU并行处理来缓解；(2)显存瓶颈——即使经过压缩，1M上下文的KV Cache仍占数GB显存，限制了并发请求数，MiMo通过PagedAttention的16KB分页管理和Prefix Caching跨请求复用(节省约70%重复Prefill开销)来优化；(3)Decode瓶颈——长上下文下每步Decode需要访问大量KV对导致内存带宽受限，MiMo通过KV Cache压缩降低内存访问量；(4)质量瓶颈——研究表明模型对中间位置信息存在"视盲"(Lost in the Middle)，MiMo通过注意力偏置校正和数据增强缓解。
    关键要点: ①Prefill延迟随序列长度近似线性增长；②KV Cache显存占用随序列长度线性增长；③Decode阶段受内存带宽限制而非计算限制；④长上下文检索准确率随长度增加而递减

    关联知识: 性能优化需区分瓶颈类型：计算密集型瓶颈需要更多算力(FPGA/ASIC加速)，内存密集型瓶颈需要更大HBM带宽(如H100的3TB/s)，IO密集型瓶颈需要优化数据搬运(CUDA Graph消除CPU-GPU通信)。
  3. 核心原理: 1M上下文的替代方案各有适用场景：(1)RAG(检索增强生成)——将长文档切分成小块，通过向量检索找到相关片段后输入短上下文模型，优势是成本低、可扩展性强，劣势是检索不准确时会遗漏关键信息；(2)摘要链(Chain of Summarization)——将长文档逐段生成摘要，再将摘要拼接输入模型，优势是处理任意长度，劣势是摘要过程中可能丢失细节；(3)滑动窗口+记忆体——维护一个外部记忆体存储关键信息，每段处理时参考记忆体，优势是灵活可控，劣势是实现复杂；(4)稀疏注意力变体(如Longformer、BigBird)——用稀疏注意力替代全注意力，优势是理论上可处理无限长度，劣势是工程实现复杂。
    关键要点: ①RAG适合"信息分散、只需检索"的场景(如FAQ问答)；②1M上下文适合"需要全局理解"的场景(如代码架构分析)；③摘要链适合"逐段处理"的场景(如书籍总结)；④最优方案通常是混合——短上下文用模型本身、长上下文用RAG+1M混合；⑤选择依据：信息密度(高则用1M)、预算(低则用RAG)、延迟要求(严则用摘要链)
    实际案例: MiMo在实际部署中采用"1M+RAG"混合方案：对于用户提供的长文档(如合同)直接使用1M上下文理解全貌；对于超出1M的海量知识库(如整站文档)RAG检索后将相关片段提交给1M上下文综合分析。
    关联知识: MiMo的1M上下文+RAG组合在RULER基准上达到了88.7%的复杂信息检索准确率，远超纯RAG方案的~70%，体现了原生长上下文的不可替代优势。

import torch
import time

def benchmark(model, input_data, warmup=10, iters=100):
    for _ in range(warmup):
        model(input_data)
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(iters):
        model(input_data)
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / iters
    print(f"Latency: {elapsed*1000:.2f}ms")
    return elapsed
