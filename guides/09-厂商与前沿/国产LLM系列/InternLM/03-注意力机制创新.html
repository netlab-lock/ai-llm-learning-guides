自注意力机制的数学基础与计算特性

InternLM的GQA实现

InternLM2-7B采用8个KV头对应32个Q头（G=4），在几乎不损失模型质量的前提下，将KV Cache显存降低为原来的1/4。这一设计在长序列推理场景中优势尤为明显。

# InternLM GQA核心逻辑
class InternLMGQAAttention:
    def __init__(self, num_heads=32, num_kv_heads=8):
        self.num_heads = num_heads        # Q头数
        self.num_kv_heads = num_kv_heads  # KV头数
        self.num_groups = num_heads // num_kv_heads  # 分组数=4

    def forward(self, x):
        q = self.q_proj(x).reshape(B, S, self.num_heads, D)
        k = self.k_proj(x).reshape(B, S, self.num_kv_heads, D)
        v = self.v_proj(x).reshape(B, S, self.num_kv_heads, D)
        # KV扩展：每个KV头复制num_groups次
        k = repeat_interleave(k, self.num_groups, dim=2)
        v = repeat_interleave(v, self.num_groups, dim=2)
        return scaled_dot_product_attention(q, k, v)
GQA分组策略：InternLM选择G=4（32 Q头 / 8 KV头）是经过大量消融实验确定的最优配置，在推理效率和模型质量之间取得最佳平衡。

# NTK-aware RoPE频率缩放
def ntk_rope_freqs(dim, max_position, base=10000, scale=1.0):
    # 当scale>1时，动态调整base频率

KV Cache压缩技术与推理显存管理

KV Cache是长序列推理的主要显存瓶颈。InternLM采用KV Cache量化（FP16→INT8/INT4）、动态KV淘汰和PagedAttention分页管理三种压缩技术，将长上下文推理的显存需求降低50%-75%，使得百万级token推理在有限硬件资源上成为可能。

工程实现中，3.5 KV Cache优化通常采用分层架构：底层是硬件抽象层(CUDA/Triton kernels)，负责高效的矩阵运算和内存管理；中间层是算法核心层，实现主要的计算逻辑；上层是服务接口层(OpenAI兼容API)，提供标准化的调用方式。这种分层设计使得每一层都可以独立优化——例如底层可以针对不同GPU架构(Ampere/Hopper)编译不同的kernel，而上层接口保持不变。

KV Cache是长序列推理的主要显存瓶颈。InternLM采用多种压缩技术：

  • KV Cache量化：将KV Cache从FP16量化为INT8/INT4，显存降低50%-75%
  • 动态KV淘汰：基于注意力分数动态淘汰不重要的KV对
  • PagedAttention：借鉴虚拟内存的分页机制管理KV Cache
注意：KV Cache压缩可能带来精度损失，需要在具体任务上评估量化后的性能退化程度。InternLM建议在长文本生成场景中使用INT8量化，在对话场景中使用INT4量化。
