Early Fusion 中视觉 Token 的注入与序列化方式

Kimi K2.5 的 Early Fusion 架构将图像通过 Vision Encoder（如 SigLIP 或 InternViT）编码为视觉 token 序列后，直接与文本 token 拼接形成统一的输入序列。每个图像被编码为约 256-1024 个视觉 token（取决于分辨率），这些 token 与文本 token 在 Transformer 的每一层中进行全注意力交互。关键设计决策是视觉 token 的位置编码——K2.5 使用特殊的模态标记（modal separator）来区分视觉和文本区域，使模型能够明确感知模态边界，同时保持跨模态信息的自由流动。

Early Fusion 与零视觉 SFT 的协同效应

Early Fusion 的深层跨模态交互为零视觉 SFT 奠定了基础。当视觉和文本 token 从预训练第一层就开始交互时，模型在学习语言的同时自然建立了视觉理解能力。这使得 SFT 阶段仅使用文本指令数据即可"激活"预训练阶段学到的视觉能力，无需额外的视觉 SFT 数据。这一发现证明了 Early Fusion 的真正价值不仅在于性能提升，更在于训练效率的优化——将多模态对齐的负担从昂贵的后训练阶段转移到了预训练阶段。

Kimi 提出了 Zero-Vision SFT 方法——在微调阶段完全不使用视觉 SFT 数据，仅依赖预训练阶段学到的视觉理解能力。

核心发现：如果预训练阶段图文数据足够丰富且质量高，SFT 阶段无需任何视觉指令数据即可达到优秀视觉理解效果。

Zero-Vision SFT 的意义

  • 降低成本：省去构建大规模视觉指令数据的人工标注成本
  • 减少对齐税：避免视觉 SFT 数据引入的分布偏移
  • 泛化性更强：预训练阶段的视觉知识更具通用性
  • 解耦能力：语言能力与视觉能力独立优化
# Zero-Vision SFT 训练流程
# 阶段1: 多模态预训练 (图文交错数据, 视频帧, 文档等)
model.pretrain(data=mixed_multimodal_corpus)

# 阶段2: 纯文本SFT (仅使用文本指令数据)
model.sft(data=text_only_instruction_data)
# 视觉理解能力自动保留，无需视觉SFT！

SigLIP 与 InternViT 的选型权衡

Kimi-VL 系列在视觉编码器的选择上经历了 SigLIP-SO400M（400M 参数）到 InternViT-6B（6B 参数）的演进。SigLIP 使用 Sigmoid 损失替代 Softmax 对比损失，在小规模数据上也能获得良好的视觉表示。InternViT 则提供了更强大的细粒度识别能力，特别适合文档 OCR、图表解析等需要高分辨率输入的场景。K2.5 最终采用约 1B 参数的专用编码器，支持动态分辨率输入，在性能和效率之间取得了最优平衡。

AnyRes 技术的高分辨率图像处理流程

AnyRes 将高分辨率图像切分为多个 384×384 的子图，每个子图独立通过 Vision Encoder 编码后生成视觉 token 序列。同时，原图经过降采样生成一个全局缩略图，提供整体语义信息。子图 token 和全局 token 拼接后输入 Transformer，使模型既能捕捉细粒度的局部细节（如表格中的数字），又能理解图像的全局结构（如版面布局）。

K2.5 端到端语音理解的架构设计

K2.5 的语音理解采用端到端架构，将音频波形通过 Whisper-style 的音频编码器转换为音频 token 序列，然后与文本 token 一起输入 Transformer 进行联合处理。与传统的 ASR + 文本理解两阶段方案不同，端到端设计避免了语音识别错误的级联传播——即使语音识别不完美，模型也能从音频特征中直接提取语义信息。这一设计使得 K2.5 在中英文混合语音、方言口音和背景噪音等复杂场景下表现出色。

语音对话的低延迟实时交互技术

K2.5 的实时语音对话延迟控制在 500ms 以内，接近人类自然对话的节奏。这一低延迟依赖于流式处理架构——音频编码器以滑动窗口方式处理音频流，每 200ms 生成一批音频 token，模型在收到音频 token 后立即开始生成回复 token，无需等待整个语音输入结束。此外，K2.5 使用了语音活动检测（VAD）技术来识别语音的起止点，避免了不必要的等待时间。

视频帧采样策略与关键帧检测

Kimi 的视频理解首先通过帧采样将连续视频转换为离散的图像序列。对于短视频（< 5 分钟），采用均匀采样策略，每秒抽取 1-2 帧；对于长视频（数小时），采用自适应采样——先通过场景变化检测（基于帧间差异的阈值判断）识别关键帧，然后在关键帧周围加密采样，平坦段稀疏采样。这种自适应策略使得 Kimi 能够在有限的 token 预算内最大化视频信息的覆盖，同时避免冗余帧浪费计算资源。

长视频理解中的时序推理与记忆管理

处理数小时长视频的核心挑战是时序推理——理解事件发生的先后顺序、因果关系和时间跨度。Kimi 通过将视频帧按时间戳标注后输入 Transformer，利用注意力机制学习帧间的时间关系。对于超长视频，K2.5 的 256K 上下文窗口提供了约 1-2 小时视频的处理能力（以每秒 1 帧计）。对于更长的视频，采用分段处理 + 摘要累积的策略：将视频分为多个片段分别处理，每段生成摘要后与下一段一起输入，形成层次化的视频理解。

8.6 K2.5 多模态 SoTA 成就

多模态理解的关键挑战在于模态对齐——如何让视觉特征和文本特征在同一个语义空间中对齐。Kimi通过对比学习和交叉注意力机制，将不同模态的特征映射到共享的表示空间，使得模型能够理解图像中的文字、听懂语音中的含义。

Kimi的多模态架构将视觉、音频等模态与语言模型深度融合。核心技术包括：模态特定编码器（将不同模态转换为统一向量表示）、跨模态注意力（实现模态间的信息交互）、以及统一的序列建模（在同一Transformer中处理所有模态）。这种设计使得模型能够进行跨模态推理。

  2. 核心原理: Early Fusion在实际工程中的主要性能瓶颈包括：(1)视觉token数量膨胀——每张图像编码为256-1024个token，高分辨率图像配合AnyRes可能产生数千token，导致注意力计算量O(n²)急剧增加；(2)KV Cache显存占用——图文混合序列的KV Cache远大于纯文本，256K上下文下可能占用超过70%的GPU显存；(3)跨模态注意力计算开销——每层都需要计算视觉-文本交叉注意力，计算密集度高于单模态。诊断方法：使用profiler分析各层注意力计算时间和显存分布；解决策略包括：视觉token压缩(选择性保留重要token)、KV Cache量化(FP8/INT8)、稀疏注意力(限制视觉token的注意力范围)、分层存储(热数据GPU/冷数据CPU/SSD)。
    关键要点: ①视觉token膨胀是首要瓶颈 ②KV Cache显存管理是关键 ③需要profiler精确定位瓶颈 ④多种优化技术组合使用效果最佳 ⑤AnyRes增加了高分辨率场景的复杂度
    实际案例: 在处理包含50张图片的长文档时，若每张图1024个token，仅视觉token就需51200个，加上文本token可能超过100K。通过自适应采样降低冗余帧数量、对早期token的KV Cache进行INT8量化、使用Prefix Cache共享系统提示词，可将端到端延迟从数秒优化到亚秒级。
    关联知识: 与Mooncake推理系统的分层存储策略紧密关联；与PagedAttention的非连续内存管理配合减少碎片；与vLLM/SGLang框架的连续批处理优化协同提升GPU利用率。
  3. 核心原理: Early Fusion的替代方案主要包括三种：(1)Late Fusion（如LLaVA-1.5、Flamingo）——各模态独立编码后在顶层通过投影层连接，优势是训练简单、模块可替换，劣势是视觉信息被压缩丢失、跨模态交互深度不足；(2)Cross-Attention Fusion（如Flamingo的Gated Cross-Attention）——在语言模型的部分层中插入交叉注意力层连接视觉编码器，优势是计算开销可控，劣势是视觉信息只在特定层注入；(3)混合方案——部分层Early Fusion+部分层独立处理。适用场景权衡：Late Fusion适合快速原型和资源受限场景；Early Fusion适合追求最佳效果且有充足计算资源的场景(如Kimi K2.5)；Cross-Attention适合需要平衡效果和效率的中间场景。
    关键要点: ①Late Fusion训练简单但信息损失大 ②Cross-Attention是折中方案 ③Early Fusion效果最好但计算成本最高 ④Kimi选择Early Fusion是因为其MoE架构(32B激活/1T总参)可以分摊计算成本 ⑤Zero-Vision SFT使得Early Fusion的训练成本优势更加突出
    实际案例: 对比三种方案在文档理解任务上的表现：Late Fusion(如LLaVA-1.5)在DocVQA上约90%，Cross-Attention(如Flamingo)约91%，Early Fusion(K2.5)达95.2%。但在资源受限的边缘设备部署场景，Late Fusion的模块化设计更便于模型裁剪和量化。
    关联知识: 与Qwen3.5-Omni的Thinker-Talker分离架构对比；与GPT-4o的原生多模态方案对比；与Kimi K2的MoE架构设计协同——Early Fusion增加的计算成本可通过稀疏激活来分摊。
