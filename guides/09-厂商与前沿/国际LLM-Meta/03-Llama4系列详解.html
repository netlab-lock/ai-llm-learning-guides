Llama 4 使用 Top-1 路由，即每个 token 仅被发送到 1 个专家处理。路由网络通过可学习的门控机制进行决策：

# Top-1 MoE 路由
scores = Linear(x)           # [batch, seq, num_experts]
probs = Softmax(scores)       # 专家概率分布
gate = TopK(probs, k=1)      # 选择概率最高的专家
expert_output = experts[gate](x)
# 可选：添加 load balancing loss 防止专家坍缩
📊 负载均衡

为避免所有 token 都路由到同一个专家（专家坍缩），训练时加入辅助负载均衡损失（Auxiliary Load Balancing Loss），鼓励专家间的均匀分配。

10M 上下文实现

Scout 实现 10M 上下文的技术方案包括四个关键组件：RoPE 频率调整（大幅扩展 base frequency 支持极长序列）、iRoPE（部分层使用不同 RoPE 配置增强位置编码灵活性）、渐进式训练（从短上下文逐步扩展到 10M 分阶段训练）、以及稀疏注意力（MoE 架构天然降低计算复杂度）。

10M 上下文是开源模型之最，远超 Llama 3.1 的 128K。这意味着 Scout 可以一次性处理整本书、完整的代码仓库或数小时的对话记录，为 RAG、长文档分析等场景提供了前所未有的能力。

通俗类比：10M 上下文实现就像图书馆索引系统——书越多越需要好的索引，1M上下文就像能检索整个图书馆。常见误区：Meta的技术选型通常是混合方案。Meta在实践中会根据具体场景组合使用多种技术——例如在推理优化中同时使用量化和批处理，在训练中同时使用数据并行和张量并行。没有银弹，只有最适合当前场景的组合。

MoE架构的负载均衡机制

MoE架构面临的核心挑战是负载均衡——如果所有token都路由到同一个专家，其他专家就浪费了。解决方案包括：容量因子限制（每个专家处理的token数上限）、辅助损失（惩罚不均匀的路由分布）、专家选择路由（让专家选择token而非token选择专家）。DeepSeek-V4采用的无辅助损失均衡方法通过动态偏置项实现负载均衡，不引入额外的训练信号。

技术选型与工程实践

在实际工程中，需要根据具体场景选择合适的技术方案。关键考量因素包括：数据规模、计算资源、延迟要求和成本预算。建议从最简单的方案开始，逐步优化，避免过度工程化。
