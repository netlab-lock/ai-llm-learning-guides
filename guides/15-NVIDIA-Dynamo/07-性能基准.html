7.3 什么时候用 Dynamo？

决策树: 你的场景是什么？ │ ├── 单机推理，小模型（<70B） │ └── → vLLM 或 SGLang（简单高效） │ ├── 单机推理，大模型（70B+，需多卡） │ └── → TensorRT-LLM（内核优化最好） │ ├── 多机推理，高吞吐需求 │ └── → Dynamo + 推理引擎（编排优势） │ ├── 推理模型（DeepSeek-R1 等，长输入短输出） │ └── → Dynamo 分离式推理（收益最大） │ ├── RAG/Agent（大量共享前缀） │ └── → Dynamo KV 感知路由（2x TTFT 加速） │ └── 生产环境，需要自动扩缩 └── → Dynamo + K8s（SLA 驱动调度）

7.4 性能数据解读注意事项

注意：基准测试数据需要谨慎解读：
  • 硬件绑定：Dynamo 的很多优化（NIXL、NVLink）深度依赖 NVIDIA 硬件，在非 NVIDIA 平台上优势会减弱
  • 模型依赖：分离式推理对长输入/短输出的推理模型收益最大，对短输入/长输出的创意写作模型收益有限
  • 规模依赖：Dynamo 的优势在大规模（多节点、高并发）下才明显，小规模可能不如直接用 vLLM
  • 50x 的来源：GB300 vs Hopper 的 50x 提升包含了硬件换代（Blackwell vs Hopper）的贡献，不完全是 Dynamo 的功劳

7.5 API 与 SDK 使用

Dynamo 提供 OpenAI 兼容的 API 接口，客户端无需修改即可接入：

# 使用 OpenAI SDK 连接 Dynamo
from openai import OpenAI

client = OpenAI(
    base_url="http://dynamo-frontend:8080/v1",
    api_key="not-needed"
)

# 普通文本生成
response = client.chat.completions.create(
    model="meta-llama/Llama-3-70B",
    messages=[{"role": "user", "content": "解释量子计算"}],
    stream=True

for chunk in response:
    print(chunk.choices[0].delta.content, end="")

7.6 Speculative Decoding 与 MTP

Speculative Decoding（投机解码）是提升 Decode 速度的重要技术。Dynamo 本身不实现投机解码，但与后端引擎的投机解码能力深度集成：

通俗理解：投机解码就像"先猜答案再验证"——让一个小模型（Draft Model）快速猜出接下来的 4-8 个 token，然后让大模型一次性验证。如果猜对了（通常 70-97% 的概率），就省了 4-8 次大模型的逐 token 生成。
技术原理加速比Dynamo 集成
Speculative Decoding Draft Model 预测 + Target Model 验证 2-3x 通过 TensorRT-LLM 后端支持
Multi-Token Prediction (MTP) 模型本身训练时预测多个未来 token 1.5-2x DeepSeek-V3/R1 原生支持，Dynamo 直接利用
Self-Speculative 模型用自己的浅层作为 Draft 1.3-1.8x 引擎层实现，Dynamo 透明代理
MTP 与 Dynamo 的协同：DeepSeek-V3/R1 的 MTP 训练使其在推理时可以一次预测多个 token（第一个预测 token 的接受率高达 97%）。Dynamo 的分离式推理架构天然适合 MTP——Prefill 阶段可以生成多个候选 token，Decode 阶段批量验证。这让 DeepSeek-R1 在 GB200 NVL72 上实现了惊人的吞吐提升。

7.7 推理引擎的融合趋势

2026 年的一个重要趋势是推理引擎的功能趋同：

特性vLLMSGLangTensorRT-LLM
分离式推理 已支持（实验） 已支持 原生支持
前缀缓存 已支持 核心特性 已支持
PagedAttention 原创 已支持 已支持
Speculative Decoding 已支持 已支持 已支持
趋势分析：推理引擎正在从"竞争"走向"融合"——每个引擎都在吸收其他引擎的优秀特性。Dynamo 的战略价值在于：当引擎之间的差异越来越小时，编排层成为差异化的关键。Dynamo 不需要比 vLLM 更好的 PagedAttention，它需要的是比单独任何一个引擎更好的多机协调。

⚡ 关键洞察

洞察 1：Dynamo 的最大优势不在单机性能，而在系统级优化——分离式推理、KV 感知路由、多层缓存、自动扩缩的组合效果远大于任何单一优化。

洞察 2：选择推理框架的决策应该基于规模和场景，而非绝对性能。小规模用 vLLM 最简单，大规模用 Dynamo 最强大。

洞察 3：50x 的性能提升数字需要理解其组成——硬件换代（~10x）+ 分离式推理（~2x）+ KV 路由（~1.5x）+ 其他优化（~1.7x）。每个组件都有贡献，但没有一个是"银弹"。

L1 理解：解释为什么 Dynamo 对推理模型（如 DeepSeek-R1）的性能提升比通用模型更大。

L2 分析：如果你是一个 AI 初创公司的架构师，日均请求量 100 万次，模型是 Llama-3-70B，你会选择 Dynamo 还是直接用 vLLM？为什么？

▶显示答案
✅ 参考答案与详解

L1 理解: Dynamo 对推理模型（如 DeepSeek-R1）提升更大的原因在于推理模型的负载特征完美匹配分离式推理的优势。推理模型有三个关键特征：（1）超长输入——推理链（Chain-of-Thought）导致 ISL 高达 32K+ tokens，Prefill 阶段的 O(n²) 计算量巨大，分离后可以分配更多 GPU 专门处理；（2）有限输出——OSL 通常只有 4-8K tokens，Decode 阶段相对较快，与 Prefill 的计算量不对称；（3）MoE 架构——DeepSeek-R1 使用 MoE，分离后可以在 NVL72 上实现 Wide EP（EP64+），Expert 分布更均匀。相比之下，通用模型（如 Llama-3-70B Dense）的 ISL/OSL 通常较短且对称，分离收益有限。Dynamo 的 30x 请求数提升中，约 15x 来自分离式推理消除 Prefill/Decode 干扰，约 2x 来自 KV 路由。

L2 分析: 作为 AI 初创公司架构师，日均 100 万次请求 + Llama-3-70B 的选择分析：
建议选择 Dynamo + vLLM，原因如下：
（1）规模分析：日均 100 万次 ≈ 11.6 QPS（平均），峰值可能 50-100 QPS。假设每次请求 ISL=2K, OSL=1K，需要至少 4-8 张 H100；
（2）Dynamo 的价值：KV 感知路由可将 TTFT 降低 50%（用户体验显著提升），SLA 驱动自动扩缩可在低谷期释放 GPU 节省 5%+ TCO；
（3）但需要评估：如果团队规模小（< 5 人），Dynamo 的运维复杂度可能是负担。此时可先用 vLLM 单机部署（8 卡 H100），等业务增长到日均 500 万次再引入 Dynamo；
（4）如果选 vLLM 单独：部署简单（pip install vllm + 一行命令），但缺乏自动扩缩和 KV 路由，峰值时段可能 SLA 违反。可通过手动增加副本来缓解，但成本更高。

L3 创造: 公平基准测试方案设计：
（1）硬件环境：使用相同的 8 卡 H100 NVL8 节点 × 2（16 卡总计），确保 NVLink 和 IB 配置一致；
（2）模型：Llama-3-70B，FP16/BF16，TP=8（单节点）或 TP=8+PP=2（双节点）；
（3）负载生成：使用 GenAI-Perf 工具，配置 5 种负载模式：① 低并发(10 QPS)，② 中并发(50 QPS)，③ 高并发(200 QPS)，④ 突发流量(0→500 QPS 阶跃)，⑤ 混合长短序列（20% 请求 ISL=32K, 80% ISL=1K）；
（4）对比组：A 纯 vLLM（2 个 8 卡实例，前端用 nginx 轮叫），B Dynamo + vLLM（分离式，Planner SLA 模式，TTFT 目标 500ms）；
（5）核心指标：TTFT P50/P95/P99, ITL P50/P95, 吞吐(tokens/s), GPU 利用率, SLA 违反率；
（6）公平性保证：每组测试预热 10 分钟，正式采集 30 分钟，重复 3 次取中位数。确保两组使用相同的 vLLM 版本和配置参数。
