02 · 核心架构

🎯 学习目标

  • 掌握 Dynamo 的四大核心组件及其职责
  • 理解微服务架构设计与组件间通信
  • 理解请求从进入系统到返回的完整数据流
  • 掌握两种部署模式（Standalone vs Gateway）的区别

2.1 架构全景

通俗理解：Dynamo 的架构就像一个现代化的物流系统——Frontend 是收件窗口，Router 是智能分拣系统，KV Cache Manager 是仓储系统，Planner 是调度中心。每个环节都可以独立扩展。
Dynamo 四大核心组件架构

2.2 四大核心组件详解

组件 1: Frontend（前端网关）

Frontend 是 Dynamo 的入口层，负责：

  • 接收 HTTP/gRPC 请求并转换为内部格式
  • 管理客户端连接和流式响应
  • 支持 OpenAI 兼容 API 格式
  • 请求认证和限流

组件 2: Smart Router（智能路由器）

通俗理解：Smart Router 就像一个经验丰富的快递分拣员——它不仅看包裹的目的地（请求目标），还看哪个仓库（Worker）已经有相关的货物（KV 缓存），能省去重新进货（重算 KV）的时间。

Smart Router 是 Dynamo 最核心的创新之一：

特性实现效果
Radix Tree 追踪 使用带哈希的 Radix Tree 追踪所有 Worker 的 KV 缓存状态 O(1) 查找缓存命中
KV 亲和路由 将请求路由到已有相关 KV 缓存的 Worker TTFT 加速 2x
负载均衡 综合考虑缓存命中率、Worker 负载、GPU 容量 避免热点 Worker
前缀感知 识别共享前缀（如系统提示词），集中路由 减少重复计算

组件 3: Planner（调度规划器）

Planner 是 Dynamo 的"大脑"，负责全局资源调度：

  • SLA 监控：持续追踪 TTFT（首 Token 延迟）和 ITL（Token 间延迟）
  • 动态决策：决定请求是否走分离式推理路径
  • 自动扩缩：根据负载动态增减 Prefill/Decode Worker
  • 资源分配：在 Prefill 和 Decode 池之间动态调配 GPU

组件 4: KV Cache Manager（分布式缓存管理器）

技术深度：KV Cache Manager 实现了层次化缓存（Hierarchical Caching），将 KV 缓存分布在四个存储层：GPU HBM（最快，容量最小）→ CPU DRAM（快，容量中等）→ 本地 SSD（较慢，容量大）→ 网络存储/对象存储（最慢，容量无限）。这类似于 CPU 的 L1/L2/L3/内存层次结构。

2.3 组件间通信架构

Dynamo 使用轻量级的 TCP 通信机制：

场景通信方式说明
组件间通信 TCP 直连 低延迟，无需外部依赖
K8s 部署 CRDs + EndpointSlices 原生 K8s 服务发现，无需 etcd/NATS
本地开发 文件发现 零依赖，适合单机测试
KV 缓存传输 NIXL (RDMA/NVLink/UCX) 零拷贝，GPU 直连

2.4 两种部署模式

模式 1: Standalone（独立模式，默认）

Client → Frontend → Router → Worker

适合中小规模部署，所有组件在同一个集群内运行。

模式 2: Gateway（网关模式，GAIE）

KV 缓存四层层次结构

适合大规模生产环境，集成 K8s Inference Gateway API，支持多模型、多租户。

2.5 请求完整生命周期

1. 客户端发送请求 │ 2. Frontend 接收并解析 │ 3. Router 查询 Radix Tree │ ├─── 缓存命中 → 路由到有缓存的 Decode Worker (快速路径) │ └─── 缓存未命中 → 路由到 Prefill Worker │ 4. Prefill Worker 处理输入 tokens │ · 计算所有输入 token 的注意力 │ · 生成 KV 缓存 │ · 输出第一个 token 的 logits │ 5. NIXL 传输 KV 缓存到 Decode Worker │ · GPU→GPU 零拷贝 (NVLink) │ · 或 GPU→NIC→NIC→GPU (RDMA) │ 6. Decode Worker 逐 token 生成 │ · 使用传输过来的 KV 缓存 │ · 流式输出 token │ 7. Frontend 流式返回给客户端 │ 8. Planner 更新监控指标 · TTFT / ITL / 吞吐量 · 决定是否调整资源

⚡ 关键洞察

洞察 1：Dynamo 的架构设计遵循"关注点分离"原则——每个组件只做一件事，但做到极致。Router 只关心路由，Planner 只关心调度，KV Manager 只关心缓存。

洞察 2：Radix Tree 是 Smart Router 的核心数据结构。它将 KV 缓存的前缀匹配问题转化为树的路径查找问题，复杂度从 O(N) 降到 O(L)（L 为序列长度）。

洞察 3：Dynamo 的服务发现不需要 etcd 或 NATS——在 K8s 中使用原生 CRDs + EndpointSlices，在本地使用文件发现。这大幅降低了部署复杂度。

L2 分析：为什么 Dynamo 选择 Radix Tree 而不是哈希表来追踪 KV 缓存？在什么场景下哈希表会更好？

L3 创造：如果你要设计一个 Dynamo 的替代方案，但只能用一个组件（不能有四个），你会设计什么样的"超级组件"？权衡是什么？
✅ 参考答案与详解

L1 理解: Dynamo 的四大组件数据流如下：客户端请求进入 Frontend（API 网关），Frontend 将请求转给 Smart Router；Router 查询 Radix Tree 找到缓存命中的 Worker 后路由请求；Planner 同时监控所有 Worker 的 SLA 指标（TTFT/ITL），决定是否调整资源；KV Cache Manager 管理四层缓存层次（GPU HBM → CPU DRAM → SSD → 网络存储），在 Prefill Worker 完成后通过 NIXL 将 KV 缓存传输到 Decode Worker。四者协同的核心数据流是：请求路由 → KV 查找 → Worker 调度 → 缓存传输 → 生成输出。

L2 分析: Radix Tree 相比哈希表的核心优势在于支持前缀匹配。在 KV 缓存路由场景中，请求的 token 序列是有序的前缀结构——"系统提示词+用户输入"的前 N 个 token 可能与之前的请求完全相同。Radix Tree 天然支持 O(L) 的最长前缀匹配（L 为前缀长度），而哈希表只能做精确匹配 O(1)，无法发现"部分前缀命中"。例如，如果缓存中有"你是一个翻译助手"的 KV，新请求是"你是一个翻译助手，请翻译Hello"，Radix Tree 能匹配到前半部分，只需计算后半部分，节省大量计算。哈希表在以下场景更优：（1）请求间几乎没有共享前缀（如随机输入）；（2）只需要精确匹配而非前缀匹配；（3）Worker 数量极多且缓存状态变化频繁（哈希表的更新是 O(1)，Radix Tree 插入是 O(L)）。

L3 创造: 设计一个"智能推理代理"（Smart Inference Proxy）超级组件，将 Router、Planner 和 KV Manager 合并为一个有状态的代理进程。核心设计：该代理维护一张全局状态表（Radix Tree + Worker 负载表 + KV 位置表合并为一个统一的跳表结构），在收到请求时一次性完成路由决策、缓存查找和资源调度。同时，它内嵌一个轻量级推理引擎（类似 vLLM 的精简版），可以直接在代理层处理短请求（<128 tokens），避免网络传输开销。权衡：（1）优势——消除了组件间通信延迟，单次决策从多次 RPC 降为一次内存查询，适合低延迟场景；（2）劣势——单点故障风险（所有功能集中在一个进程），水平扩展困难，代码复杂度极高，无法独立替换路由或调度策略，违反了关注点分离原则。
