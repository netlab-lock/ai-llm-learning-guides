03 · 分离式推理（Disaggregated Serving）

🎯 学习目标

  • 深入理解 Prefill 和 Decode 阶段的本质差异
  • 掌握分离式推理的原理和调度策略
  • 理解分离带来的性能收益及其量化分析
  • 了解分离式推理的代价和适用场景

3.1 Prefill vs Decode：为什么必须分离？

通俗理解：想象一个餐厅。Prefill 是"备菜"阶段——厨师要把所有食材洗好切好（处理所有输入 tokens），这个过程需要大量操作台（计算资源）。Decode 是"炒菜"阶段——一道一道上菜（逐 token 生成），这个过程需要快速出菜（带宽）。如果同一个厨师既要备菜又要炒菜，备菜时炒菜要等，炒菜时备菜做不了。分离后，备菜厨师和炒菜厨师各司其职，效率最大化。

计算特性对比

特性Prefill（预填充）Decode（解码）
操作类型 大矩阵乘法（所有输入 tokens 同时处理） 向量-矩阵乘法（逐 token 处理）
瓶颈 计算密集（Compute-bound） 带宽密集（Memory-bandwidth-bound）
GPU 利用率 高（大矩阵充分利用 Tensor Core） 低（小向量无法饱和计算单元）
批处理效率 单请求即可高效（长序列 = 大矩阵） 需要大批量才能提高利用率
理想 GPU 配置 高算力 GPU（FP16/FP8 TFLOPS 高） 高带宽 GPU（HBM 带宽高）
并行策略 低张量并行（减少通信开销） 高张量并行（提高显存带宽利用率）
延迟特征 与输入长度成正比（O(n^2) 注意力） 每 token 延迟相对恒定

数学分析

数学本质：Prefill 阶段的计算量约为 O(n^2 * d)（n 为序列长度，d 为隐藏维度），其中注意力矩阵计算是二次的。Decode 阶段每生成一个 token 的计算量约为 O(n * d)，但主要瓶颈是从 HBM 读取 KV 缓存的 O(n * d) 带宽。

对于一个 70B 参数模型（d=8192），处理 4K tokens 的 Prefill：
- 计算量：~4K^2 * 8192 = ~134 TFLOPs（FP16）
- 在 H100（989 TFLOPS FP16）上需要 ~135ms

Decode 每个 token：
- 带宽需求：~4K * 8192 * 2 bytes * 80 layers = ~5 GB
- 在 H100（3.35 TB/s HBM 带宽）上需要 ~1.5ms
- 但计算量只有 ~0.13 TFLOPs，GPU 计算利用率仅 ~2%

3.3 Dynamo 的三种推理模式

Planner 根据实时负载和 SLA 指标动态选择推理模式：

模式适用场景工作方式
非分离模式 低负载、短序列 请求在同一组 GPU 上完成 Prefill + Decode
分离模式 高负载、长序列 Prefill 和 Decode 在不同 GPU 池完成
混合模式 中等负载、多变序列 Planner 动态决定每个请求走哪条路径
通俗理解：这就像出租车调度——短途（低负载）直接派一辆车全程服务；长途（高负载）在中途换一辆车（分离模式）；中等距离由调度员根据路况动态决定。

3.4 分离式推理的调度策略

策略 1: 静态分离

预先分配固定的 Prefill 和 Decode GPU 比例。简单但不灵活。

策略 2: 动态分离（Dynamo 默认）

Planner 根据实时指标动态调整：

  • 监控 TTFT（首 Token 延迟）和 ITL（Token 间延迟）
  • 当 Prefill 队列积压时，临时将 Decode GPU 转为 Prefill
  • 当 Decode 负载低时，合并空闲 GPU

策略 3: 请求级分离

每个请求独立决定是否走分离路径：

  • 短序列（<512 tokens）：不分离，减少传输开销
  • 长序列（>4K tokens）：分离，充分利用并行性
  • 中等序列：根据当前负载动态决定

3.5 KV 缓存传输：分离的关键挑战

分离式推理的核心挑战是如何高效地将 KV 缓存从 Prefill Worker 传输到 Decode Worker。

Prefill Worker Decode Worker ┌─────────────┐ ┌─────────────┐ │ GPU HBM │ ──── NVLink ────→ │ GPU HBM │ │ KV Cache │ (900 GB/s, <1ms) │ KV Cache │ │ │ │ │ │ │ ──── RDMA ──────→ │ │ │ │ (400 Gb/s, ~ms) │ │ │ │ │ │ │ │ ──── UCX ───────→ │ │ │ │ (通用, 灵活) │ │ └─────────────┘ └─────────────┘

传输延迟分析

传输路径带宽KV 大小 (70B, 4K seq)传输时间
NVLink 5.0（节点内） 1.8 TB/s ~2.5 GB ~1.4 ms
NVLink 4.0（节点内） 900 GB/s ~2.5 GB ~2.8 ms
InfiniBand 400G（跨节点） 50 GB/s ~2.5 GB ~50 ms
RoCE 400G（跨节点） 50 GB/s ~2.5 GB ~50 ms
关键权衡：分离式推理的收益 = GPU 独立优化的收益 - KV 传输的开销。只有当序列足够长、负载足够高时，分离的收益才会超过传输开销。对于短序列（<512 tokens），传输开销可能超过分离收益。

3.6 性能收益量化

NVIDIA 官方公布的基准测试结果：

模型配置指标提升
DeepSeek-R1 GB200 NVL72 请求数 30x
DeepSeek-R1 GB200 NVL72 吞吐量 2x+
MoE 模型 GB200 NVL72 每 GPU 吞吐 7x
MoE 模型 GB300 NVL72 vs Hopper 50x

为什么 DeepSeek-R1 提升最大？

技术原因：DeepSeek-R1 是推理模型（Reasoning Model），其特点是：
  1. 超长输入：推理链（Chain-of-Thought）导致输入序列极长（32K+ tokens）
  2. 计算密集：长序列的 Prefill 计算量是 O(n^2)，对 GPU 压力巨大
  3. MoE 架构：Expert Parallelism 需要跨 GPU 通信，分离后可以独立优化
分离后，Prefill 池可以专注于处理长序列的 O(n^2) 计算，Decode 池可以专注于高效生成，互不干扰。

3.7 分离式推理的代价

代价说明缓解措施
KV 传输延迟 Prefill→Decode 传输 KV 缓存需要时间 NIXL 零拷贝传输，NVLink 高带宽
额外 GPU 内存 KV 缓存在 Prefill 和 Decode 端各存一份 传输完成后释放 Prefill 端缓存
架构复杂度 需要额外的编排层（Dynamo） Dynamo 开源，K8s Operator 简化部署
短序列不划算 短序列的传输开销 > 分离收益 Planner 自动判断是否分离

⚡ 关键洞察

洞察 1：分离式推理的本质是用网络带宽换 GPU 计算效率。当 KV 传输速度足够快（NVLink）且序列足够长时，这笔交易非常划算。

洞察 2：推理模型（如 DeepSeek-R1）是分离式推理的杀手级应用场景——超长输入 + 有限输出的特征完美匹配 Prefill/Decode 分离的优势。

洞察 3：Dynamo 的 Planner 实现了"每个请求独立决策"——不是所有请求都走分离路径，而是根据序列长度和当前负载动态决定。这是比静态分离更聪明的策略。

✅ 参考答案与详解

L1 理解: Decode 阶段 GPU 计算利用率仅约 2% 的根本原因是内存带宽瓶颈。每生成一个 token，需要从 HBM 读取整个 KV 缓存（对于 70B 模型 4K 序列约 5GB），但实际的矩阵计算量只有 ~0.13 TFLOPs。H100 的 HBM 带宽为 3.35TB/s，读取 5GB 需要 ~1.5ms，而 0.13 TFLOPs 的计算在 989 TFLOPS 的算力下仅需 ~0.13μs。GPU 98% 的时间在等待数据从 HBM 搬运到计算单元，这就是所谓的"带宽密集"（Memory-bandwidth-bound）。剩余的算力完全空闲，因为单个 token 的向量-矩阵乘法无法充分利用 Tensor Core 的大规模并行能力。

L2 分析: 2 Prefill + 6 Decode 的配置在以下负载特征下最优：（1）输入序列较短（ISL < 2K tokens），Prefill 计算量不大，2 张卡即可满足；（2）输出序列较长（OSL > 4K tokens），Decode 阶段需要大量带宽和批处理能力，6 张卡可以维持更大的 batch size 提高 GPU 利用率；（3）并发请求数中等（50-200），不足以让 Prefill 积压但 Decode 需要足够容量。对比分析：如果是长输入短输出（ISL=32K, OSL=8K，如 DeepSeek-R1），则应反过来分配 6P+2D，因为 Prefill 的 O(n²) 计算是主要瓶颈；如果是均匀负载（ISL=OSL=2K），则 4P+4D 更均衡。关键公式：Prefill GPU 数 ∝ ISL²/BW_prefill，Decode GPU 数 ∝ 并发数 × ISL × OSL/BW_decode。

L3 创造: 自适应算法设计——"TTFT-ITL 双指标反馈控制器"：
（1）监控：每 10 秒采集 TTFT P95 和 ITL P95，用卡尔曼滤波平滑波动；
（2）决策规则：当 TTFT P95 > 目标值 × 1.2 且 ITL P95 < 目标值 × 0.8 时，说明 Prefill 资源不足而 Decode 有余量，应将 1 张 Decode GPU 转为 Prefill；反之，当 ITL P95 > 目标值 × 1.2 且 TTFT P95 < 目标值 × 0.8 时，应将 1 张 Prefill GPU 转为 Decode；
（3）冷却期：每次调整后设置 60 秒冷却期，避免频繁切换；
（4）安全边界：Prefill GPU 不少于 1 张、不多于 N-1 张；
（5）迁移开销考量：GPU 切换角色需要重新加载模型分片（~30s），因此只在偏差持续超过 2 分钟时才触发切换。增加 Prefill GPU 的条件：TTFT 持续高于目标 + Prefill 队列深度 > 10 + KV 缓存命中率正常（非路由问题）。
