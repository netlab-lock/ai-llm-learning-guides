05 · 智能路由（Smart Router）

🎯 学习目标

  • 理解 Smart Router 的 Radix Tree 数据结构
  • 掌握 KV 感知路由算法的工作原理
  • 理解负载均衡与缓存亲和的权衡
  • 了解路由决策对端到端延迟的影响

5.1 为什么需要智能路由？

通俗理解：普通路由器就像"轮叫调度"——把请求按顺序发给每个 Worker，不管 Worker 的状态。Smart Router 就像一个"智能外卖调度员"——它知道哪个骑手（Worker）手上有同一家店的订单（KV 缓存），可以让骑手顺路多带几单，节省往返时间。

传统路由的三大问题

问题 1: 轮叫路由（Round-Robin） 请求 A (前缀 P1) → Worker 0 ✓ 缓存命中 请求 B (前缀 P1) → Worker 1 ✗ 缓存未命中！重算 P1 请求 C (前缀 P1) → Worker 2 ✗ 缓存未命中！重算 P1 → 同一个前缀被重算了 3 次！ 问题 2: 纯负载均衡 Worker 0: 负载 30%, 有缓存 {P1, P2} Worker 1: 负载 50%, 有缓存 {P3} 请求 (前缀 P1) → Worker 1 ✗ 选了负载低的，但缓存未命中！ 问题 3: 无全局视图 每个 Worker 只知道自己有什么缓存 无法做出全局最优的路由决策

5.2 Radix Tree：路由的核心数据结构

Smart Router 使用带哈希的 Radix Tree（基数树）来追踪所有 Worker 的 KV 缓存状态。

路由查找过程

输入请求: "系统提示:请翻译:Hello World" Step 1: 计算 token 序列的哈希 tokens = ["系统提示:", "请翻译:", "Hello", "World"] Step 2: 在 Radix Tree 中查找最长前缀匹配 [root] → "系统提示:" ✓ (hash: a3f2) → "请翻译:" ✓ (hash: d4e5) → "Hello" ✓ (hash: 1a) → "World" ✗ 未找到 最长匹配: "系统提示:请翻译:Hello" (深度 3) Step 3: 获取有此前缀缓存的 Worker 列表 workers = [Worker 0, Worker 3] Step 4: 在候选 Worker 中选择最优 Worker 0: 负载 40%, 有完整前缀 Worker 3: 负载 60%, 有完整前缀 → 选择 Worker 0（负载较低） Step 5: 路由请求到 Worker 0 Worker 0 只需计算 "World" 的 KV

5.3 路由算法详解

算法 1: KV 亲和路由（默认）

优先选择有最长前缀缓存匹配的 Worker：

  • 在 Radix Tree 中查找最长前缀匹配
  • 从匹配的 Worker 列表中选择负载最低的
  • 目标：最大化缓存命中率

算法 2: 负载感知路由

综合考虑缓存命中和 Worker 负载：

Score(worker) = α × cache_hit_ratio + (1-α) × (1 - load_ratio) 其中: cache_hit_ratio: 该 Worker 对此请求的前缀匹配比例 (0-1) load_ratio: 该 Worker 的当前负载比例 (0-1) α: 权衡因子 (默认 0.7，偏向缓存命中)

算法 3: Token 加权负载均衡

考虑请求的 token 数量，避免长请求集中在单个 Worker：

Effective_Load(worker) = Σ (request_tokens × remaining_ratio) 选择 Effective_Load 最低的 Worker

算法 4: 预测性路由

根据历史请求模式预测未来的缓存需求：

  • 分析请求的前缀分布模式
  • 预取可能被请求的前缀到特定 Worker
  • 适合有明显请求模式的场景（如 RAG）

5.4 路由决策的延迟影响

路由策略TTFT吞吐量适用场景
轮叫路由 基线 基线 无共享前缀
KV 亲和路由 -50% +30% 共享系统提示词
负载感知路由 -30% +40% 混合负载
预测性路由 -60% +20% 可预测的请求模式
NVIDIA 官方数据：KV 感知路由可以将 TTFT 加速 2x。这意味着对于一个 TTFT 为 200ms 的请求，智能路由可以将其降低到 100ms。在高并发场景下，这种加速效果更加显著，因为缓存命中的累积效应会随请求数量增长。

5.5 Radix Tree 的维护

插入操作

当一个请求完成 Prefill 后，Router 更新 Radix Tree：

  1. 计算请求前缀的哈希链
  2. 在 Radix Tree 中插入或更新对应路径
  3. 将 Worker 添加到叶节点的 Worker 列表中

删除操作

当 KV 缓存被驱逐时，Router 更新 Radix Tree：

  1. KV Cache Manager 通知 Router 缓存已驱逐
  2. Router 从对应节点的 Worker 列表中移除
  3. 如果节点没有 Worker 了，修剪该分支

一致性保证

注意：Radix Tree 是 Router 的本地视图，与实际的 KV 缓存状态可能存在短暂不一致。Dynamo 使用最终一致性模型——允许短暂的不一致（路由到没有缓存的 Worker），但通过缓存未命中时的重算机制保证正确性。

5.6 与 Planner 的协同

Planner: "Worker 0 负载过高，将 2 个请求迁移到 Worker 3" │ Router: 更新 Radix Tree │ ├── 将 Worker 0 的前缀信息标记为"迁移中" │ ├── 通过 NIXL 触发 KV 缓存从 Worker 0 → Worker 3 │ └── 迁移完成后，更新 Radix Tree 中的 Worker 映射

⚡ 关键洞察

洞察 1：Smart Router 的核心创新是将缓存局部性（Cache Locality）引入请求路由。这与 CPU 缓存设计中的"亲和性调度"思想一脉相承。

洞察 2：Radix Tree 的哈希设计使得前缀匹配可以在 O(L) 时间内完成（L 为最长前缀长度），而非 O(N)（N 为总请求数）。这是 Dynamo 能在万级 Worker 上保持低延迟路由的关键。

洞察 3：路由算法的 α 参数（缓存亲和 vs 负载均衡的权衡）是一个可调旋钮——在高缓存命中率场景（如 RAG）中增大 α，在均匀负载场景中减小 α。

✅ 参考答案与详解

L1 理解: Radix Tree 在处理 3 个共享前缀的请求时的构建过程：
请求 A："你是一个翻译助手。请翻译 Hello World"
请求 B："你是一个翻译助手。请翻译 Good Morning"
请求 C："你是一个翻译助手。请总结以下内容"

Step 1（处理 A）：root → "你是一个翻译助手。" → "请翻译" → "Hello World"，在叶节点记录 Worker 0。
Step 2（处理 B）：从 root 沿"你是一个翻译助手。"→"请翻译"路径匹配，在"请翻译"节点下新增分支"Good Morning"，记录 Worker 1。此时"请翻译"前缀被 A 和 B 共享。
Step 3（处理 C）：从 root 沿"你是一个翻译助手。"匹配，然后"请总结"是新分支，创建新节点，记录 Worker 2。
最终树结构：root → "你是一个翻译助手。"(共享) → 分支为"请翻译"(Workers[0,1])和"请总结"(Worker[2])。

L2 分析: KV 亲和路由导致严重负载不均衡的场景：
（1）"热点前缀"问题：当 90% 的请求共享同一个系统提示词（如 ChatGPT 的默认提示），KV 亲和路由会将几乎所有请求都路由到第一个缓存了该前缀的 Worker，导致该 Worker 过载而其他 Worker 空闲。检测方法：监控各 Worker 的队列深度方差，当方差 > 阈值时触发告警。
（2）缓解方案：① 引入"缓存复制"——当某个前缀被超过 3 个 Worker 请求时，主动将该前缀的 KV 缓存复制到其他空闲 Worker；② 调整 α 参数——将权重从 0.7（偏向缓存）降低到 0.4，增加负载均衡的权重；③ 设置"负载天花板"——当 Worker 负载 > 80% 时，强制忽略缓存亲和，选择负载最低的 Worker。
（3）量化分析：假设 4 个 Worker，共享前缀命中率 90%，纯 KV 亲和路由下 Worker 0 负载 85%，其他三个各 5%，方差极大。引入缓存复制后，3 个 Worker 都有该前缀缓存，负载均衡为 30%/30%/30%/10%，方差大幅降低。

L3 创造: 分布式 Radix Tree 方案设计——"分片+广播"混合架构：
（1）分片策略：将 Radix Tree 按前缀哈希分片到 N 个 Router 实例。每个 Router 只负责管理一部分前缀路径（如 hash(prefix) mod N）。路由请求时，先计算前缀哈希确定负责的 Router，再查询该 Router 的本地 Radix Tree；
（2）热点广播：对于访问频率 > 100 QPS 的热点前缀，通过 Gossip 协议异步广播到所有 Router 的本地缓存，避免跨 Router 查询；
（3）一致性：使用版本向量（Version Vector）追踪每个分片的更新顺序，Router 间通过心跳同步元数据（每 5 秒一次），允许短暂不一致（最终一致性模型）；
（4）延迟分析：本地命中 < 0.1ms，跨 Router 查询 < 1ms（通过 RDMA），热点广播后全部本地命中。在万级 Worker 规模下，该方案可将路由延迟控制在 P99 < 2ms。
