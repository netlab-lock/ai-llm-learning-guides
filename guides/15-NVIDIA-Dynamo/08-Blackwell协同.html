8.3 Dynamo 对 Blackwell 特性的利用

特性 1: FP4 精度

Blackwell 首次支持 FP4（4 位浮点）精度：

  • FP4 的 TFLOPS 是 FP8 的 2 倍（9,000 vs 4,500）
  • KV 缓存使用 FP4 量化，存储减少 4x
  • Dynamo 的 KV 传输量也随之减少 4x
  • 精度损失通过 SmoothQuant/AWQ 等技术控制

特性 2: NVLink 5.0

NVLink 5.0 将节点内 GPU 间带宽提升到 1.8 TB/s：

  • KV 缓存传输延迟进一步降低（~1.4ms for 2.5GB）
  • 分离式推理的传输开销几乎可以忽略
  • Prefill 和 Decode 可以放在同一个 NVL72 内的不同 GPU 上

特性 3: NVSwitch 全互联

NVL72 配置中的第三代 NVSwitch 实现 72 卡全互联：

  • 任意两个 GPU 之间可以直接通信，无需多跳
  • KV 缓存可以在任意 Prefill GPU 和 Decode GPU 之间直接传输
  • 支持 Wide Expert Parallelism（EP64+），MoE 模型的 Expert 分布在 64+ 张卡上

特性 4: C2C（Chip-to-Chip）

GB200 超级芯片中，Grace CPU 和 Blackwell GPU 通过 C2C 直连：

  • 900 GB/s 双向带宽
  • KV 缓存卸载到 CPU DRAM 的延迟极低
  • Dynamo 的多层缓存架构在 GB200 上效果最佳

性能放大效应：Dynamo 的性能提升在更好的硬件上会被放大。这是因为：更好的硬件 → 更快的 KV 传输 → 分离式推理的传输开销更低 → 分离的净收益更高。在 A100 上，传输开销可能吃掉分离收益的 30%；在 GB200 NVL72 上，传输开销几乎可以忽略。

8.5 Wide Expert Parallelism

Dynamo 在 GB200 NVL72 上实现了 Wide Expert Parallelism（Wide EP），这是 MoE 模型推理的关键优化：

传统 EP (8 卡): ┌────────────────────────────────┐ │ GPU 0: Expert 0-15 │ ← 每卡 16 个 Expert │ GPU 1: Expert 16-31 │ │ ... │ │ GPU 7: Expert 112-127 │ └────────────────────────────────┘ 问题: 激活的 Expert 集中在少数 GPU，负载不均 Wide EP (64 卡, Dynamo): ┌────────────────────────────────────────────────────────┐ │ GPU 0: Expert 0-1 │ ← 每卡仅 2 个 Expert │ │ GPU 1: Expert 2-3 │ │ │ ... │ ← 更好的负载均衡 │ │ GPU 63: Expert 126-127│ ← 更少的 Expert 冲突 │ └────────────────────────────────────────────────────────┘ 优势: 负载更均匀，通信更高效

⚡ 关键洞察

洞察 1：Dynamo 和 Blackwell 是协同设计的结果——Dynamo 的分离式推理需要高带宽 NVLink，Blackwell 的 NVLink 5.0 正好提供了这个能力。这不是巧合，而是 NVIDIA 的系统级优化策略。

洞察 2：NVL72 的 72 卡全互联拓扑使得 Dynamo 可以将整个机柜当作一个"超级 GPU"来调度，这是传统 InfiniBand 互联无法实现的。

洞察 3：FP4 精度不仅提升了计算性能，还间接提升了传输效率——KV 缓存更小，传输更快，分离式推理的净收益更高。

L3 创造：假设你有一个 H100 NVL8 节点和一个 400G InfiniBand 跨节点连接，设计一个 Dynamo 部署方案，最大化 DeepSeek-R1 的推理吞吐。
✅ 参考答案与详解

L1 理解: NVL72 的"全互联"与传统 InfiniBand 互联的本质区别在于拓扑结构和通信跳数。InfiniBand 互联是"交换机层次拓扑"：GPU → 本地交换机 → 核心交换机 → 远端交换机 → GPU，至少 2 跳交换机延迟（~1-2μs/跳），且共享上行带宽。NVL72 的 NVSwitch 全互联是"直连拓扑"：任意 GPU 通过 NVSwitch 直接连接任意 GPU，仅 1 跳，无共享带宽竞争。具体数据：NVL72 中 72 张 GPU 任意两点间带宽均为 1.8TB/s（全双工），总互联带宽 129.6TB/s。而 InfiniBand 400G 单链路仅 50GB/s，且多对 GPU 通信时共享上行带宽。这意味着在 NVL72 内部，KV 缓存传输几乎无延迟开销，而在 IB 互联下，传输延迟和带宽竞争是显著的瓶颈。

L2 分析: A100 集群上部署 Dynamo 分离式推理的瓶颈分析：
（1）A100 NVLink 3.0 带宽：600GB/s。传输 70B 模型 4K 序列的 KV 缓存（2.5GB）需要 ~4.2ms。相比 H100 的 2.8ms，增加了 50%；
（2）分离收益计算：分离式推理的净收益 = GPU 独立优化收益 - KV 传输开销。A100 上，对于 4K 序列，Prefill 时间约 200ms，传输开销 4.2ms（占 2%），净收益显著，分离仍然划算；
（3）临界序列长度：当序列长度 < 512 tokens 时，Prefill 时间约 3ms，传输开销 4.2ms 超过计算本身，分离不划算。临界点约在 512-1024 tokens；
（4）A100 的其他限制：不支持 FP4，KV 缓存无法进一步压缩；HBM 容量仅 80GB，可缓存的序列更短；Tensor Core 性能（312 TFLOPS FP16）约为 H100 的 1/3，Prefill 阶段耗时更长；
（5）结论：A100 上 Dynamo 分离式推理对 ISL > 1K 的请求仍有明显收益，但最佳序列长度阈值比 H100 高约 2 倍。

L3 创造: H100 NVL8 + IB400G 的 DeepSeek-R1 最大化吞吐方案：
（1）集群拓扑：假设 4 个节点，每节点 8 卡 H100 NVL8，节点间 IB400G 互联。总计 32 张 GPU；
（2）GPU 分配：节点 1（8 卡）专职 Prefill，TP=4×2 实例，可同时处理 2 个请求的 Prefill。节点 2-3（16 卡）专职 Decode，TP=8×2 实例。节点 4（8 卡）混合模式，根据负载动态切换；
（3）KV 传输策略：Prefill 在节点 1 完成后，通过 IB400G 传输 KV 到节点 2/3（50ms）。为减少传输开销：① 使用 FP8 量化 KV（DeepSeek-R1 的 MLA 已将 KV 压缩到很小，进一步 FP8 量化后传输量约 0.3GB，传输时间降至 ~6ms）；② 路由器优先利用节点 2/3 上已有的前缀缓存；
（4）Planner 配置：SLA 模式，TTFT 目标 800ms，ITL 目标 30ms。使用 proactive 扩缩策略；
（5）预期效果：DeepSeek-R1 ISL=32K/OSL=8K 场景下，相比非分离方案，吞吐预计提升 10-15 倍。
