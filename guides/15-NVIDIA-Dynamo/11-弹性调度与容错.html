11.1 为什么弹性调度至关重要？

通俗理解：想象一个网约车平台。高峰期（如早晚高峰）需要大量司机上路，低谷期（凌晨）只需要少量司机。如果全天都按高峰期配置司机，成本巨大；如果按低谷期配置，高峰期用户打不到车。Dynamo 的 Planner 就是这个"调度中心"——根据实时路况（负载指标）动态调整司机数量（GPU 资源），确保乘客（请求）在 SLA 时间内上车，同时不浪费司机资源。

LLM 推理的负载特征决定了弹性调度是刚需：

负载特征具体表现对弹性调度的需求
昼夜差异 白天并发 10x 于凌晨 需要自动扩缩 GPU 资源
突发流量 热点事件导致请求暴增 5-50x 需要秒级响应的快速扩容
推理模型长尾 DeepSeek-R1 的推理链长度变化极大（1K-100K tokens） 需要动态调整 Prefill/Decode 比例
GPU 成本高昂 H100 约 $3/h，GB200 NVL72 约 $200K+/机柜 空闲 GPU 是巨大浪费

11.2 Planner：SLA 驱动的自动扩缩器

Dynamo Planner 是一个运行时自动扩缩控制器，它根据流量信号和引擎性能模型动态调整 Prefill 和 Decode 引擎的副本数量。

Planner 四种扩缩模式

四种扩缩模式详解

模式 1: Throughput（吞吐量模式，默认）

通俗理解：就像一个简单的"排队规则"——当排队人数超过 10 人，就多开一个窗口；当窗口空闲超过 5 分钟，就关一个窗口。简单粗暴但有效。
  • 输入：队列深度（Queue Depth）+ KV 缓存利用率
  • 决策：超过静态阈值就扩，低于阈值就缩
  • 优点：无需配置 SLA 目标，开箱即用
  • 适用：对延迟不敏感的离线/批处理场景

模式 2: Load（负载模式）

  • 输入：用户定义的 Prefill 队列 token 阈值 + Decode KV 利用率阈值
  • 决策：基于实际负载指标的反应式扩缩
  • 优点：比 Throughput 更精细的控制
  • 适用：有明确负载特征的中等规模部署

模式 3: SLA（延迟目标模式，最强大）

技术深度：SLA 模式是 Dynamo Planner 最核心的创新。它使用 Rust 引擎性能模型（Performance Model）来预测给定配置下能否满足 TTFT/ITL 目标。性能模型有三种获取方式：
  1. AIC 估计（AIConfigurator）：通过模拟 10K+ 配置快速预测
  2. 在线 FPM 调参：在真实负载下实时校准性能模型
  3. FPM 回归：使用历史数据训练的回归模型
Planner 支持预测式（proactive）和反应式（reactive）两种扩缩策略。预测式根据流量趋势提前扩容，反应式在 SLA 即将违反时紧急扩容。
参数说明示例
ttft 首 Token 延迟目标 500ms
itl Token 间延迟目标 30ms
mode 扩缩策略（proactive/reactive） proactive
kalman_filter 卡尔曼滤波平滑指标波动 启用

模式 4: Hierarchical（分层模式）

  • Global Planner：跨集群的全局调度器
  • 适用：多区域、多集群的大规模部署
  • 能力：跨数据中心的负载均衡和资源调配

Planner 性能数据

官方数据：

  • SLA 违反率降低 80%：相比静态配置，Planner 动态扩缩将 SLA 违反率从 20% 降到 4%
  • TCO 降低 5%：通过精确的 GPU 资源调配，减少 5% 的总拥有成本
  • Kalman 滤波：平滑指标波动，避免频繁扩缩导致的"抖动"问题

11.3 Request Migration：在途请求迁移

Request Migration 是 Dynamo 最独特的容错机制之一——当 Worker 在生成过程中故障时，已生成的 Token 不会丢失，而是无缝迁移到另一个健康的 Worker 上继续生成。

在途请求迁移（Request Migration）容错机制

迁移流程详解

  1. 故障检测：金丝雀健康检查发现 Worker A 故障（连接失败或中断流）
  2. 状态保留：Migration Operator 保留已生成的 Token 序列
  3. 请求重建：在 Worker B 上重建请求，注入已生成的 Token
  4. 无缝续流：Worker B 从断点继续生成，客户端无感知

迁移限制

限制说明
最大迁移次数 可配置（默认 3 次），防止无限迁移循环
n>1 模式 Multiple Choices 模式暂不支持迁移
Guided Decoding 结构化输出（JSON Schema 等）暂不支持迁移

11.4 五层容错体系

Dynamo 容错体系全景
层级机制触发条件恢复时间
L1 金丝雀健康检查 持续探测 秒级发现

11.5 DGDR 零配置部署

DGDR（DynamoGraphDeploymentRequest）是 Dynamo v1.0 引入的零配置部署能力。用户只需一个 YAML 文件，指定模型、硬件和 SLA 目标，Dynamo 自动完成所有配置。

DGDR 零配置部署流程

DGDR 三步走

  1. AIConfigurator 模拟：基于模型特征和硬件拓扑，模拟 10K+ 种配置组合，找到 Pareto 最优解
  2. Planner 优化：确定 Prefill/Decode GPU 分配比例、并行策略、扩缩阈值
  3. 自动部署：Grove Operator 执行拓扑感知的 Gang 调度，确保 NVLink 域内 GPU 不被拆散

11.6 Grove K8s Operator

Grove 是 Dynamo 的 Kubernetes Operator，负责在 K8s 集群上管理推理工作负载的完整生命周期。

核心能力

能力说明为什么重要
Gang 调度 NVL72 的 72 张卡要么全部启动，要么全部不启动 避免部分启动的资源浪费
拓扑感知 调度时考虑 NVLink/NVSwitch/IB 拓扑 确保高带宽通信的 GPU 被安排在同一个 NVLink 域
CRD 管理 通过 DynamoGraphDeployment CRD 声明式管理 GitOps 友好，配置即代码

11.7 Autoscaling 在 K8s 上的实现

Dynamo 在 Kubernetes 上的自动扩缩涉及两个层面：

层面 1: Dynamo 内部扩缩（Planner）

  • 调整 Prefill/Decode Worker 的 Replica 数量
  • 基于 SLA 指标（TTFT/ITL）驱动
  • 与 Prometheus 集成获取实时指标

层面 2: K8s 节点扩缩（Cluster Autoscaler）

  • 当 Dynamo 需要更多 GPU 但节点不够时，自动申请新节点
  • GPU 节点的 Provision 时间比 CPU 节点长（分钟级 vs 秒级）
  • 需要提前配置 GPU Node Pool
GPU 弹性的挑战：GPU 不像 CPU 那样可以秒级弹性。一台 8 卡 H100 节点的启动时间约为 2-5 分钟（包括驱动加载、模型加载）。Dynamo 通过以下方式缓解：
  • ModelExpress：7x 加速模型加载（流式传输权重）
  • KV 缓存预热：新 Worker 启动时从多层缓存加载常用前缀
  • 预测式扩容：Planner 根据流量趋势提前扩容，避免等到 SLA 违反才紧急扩容

11.8 四种扩缩场景实例

场景 1: 日常办公 ChatBot（稳定负载）

模式: throughput
扩缩规则: 队列深度 > 50 时扩容，< 10 时缩容
结果: 白天 8 GPU，夜间 2 GPU，自动切换

场景 2: 推理模型 API 服务（高 SLA 要求）

模式: sla
TTFT 目标: 800ms
ITL 目标: 25ms
策略: proactive（预测式扩容）
结果: SLA 违反率 < 2%，TCO 降低 8%

场景 3: RAG 应用（共享前缀多）

模式: load + KV 感知路由
特殊: 前缀缓存命中率 > 80% 时自动减少 Prefill GPU
结果: 前缀缓存越热，需要的 GPU 越少

场景 4: 多区域部署（全球服务）

模式: hierarchical
Global Planner: 跨 3 个区域的流量调度
Local Planner: 每个区域独立的 SLA 驱动扩缩
结果: 跟随太阳的负载迁移，全球 TTFT < 1s

⚡ 关键洞察

洞察 1：Planner 的 SLA 模式是 Dynamo 最被低估的能力。它不只是简单的"阈值扩缩"，而是一个带性能模型的预测式调度器——它能模拟"如果我加 2 张 Prefill 卡，TTFT 能降到多少"，然后做出最优决策。

洞察 2：Request Migration 是 Dynamo 区别于所有其他推理框架的独特能力。vLLM、SGLang 都没有在途请求迁移——Worker 故障意味着请求丢失。Dynamo 的迁移机制让用户请求"穿过"故障，这对生产环境的 SLA 保证至关重要。

洞察 3：DGDR + AIConfigurator 的组合实现了"声明式推理部署"——用户只声明"我要什么"（模型+SLA），Dynamo 自动决定"怎么做"（配置+拓扑+扩缩）。这是 Kubernetes 声明式哲学在 LLM 推理领域的延伸。

⚠️ 常见误区

误区 1："弹性扩缩 = 秒级扩缩"——GPU 的弹性远不如 CPU。一台 8 卡节点的启动+模型加载需要 2-5 分钟。Dynamo 通过预测式扩容和 ModelExpress 来缓解，但无法做到秒级。
误区 2："Planner 会自动优化一切"——Planner 需要合理的 SLA 目标和足够的监控数据。如果 TTFT 目标设得太低（如 10ms），Planner 会疯狂扩容直到资源耗尽。
误区 3："Request Migration 是万能的"——迁移有成本（状态重建、额外延迟）。频繁迁移说明底层基础设施有问题，应该解决根因而非依赖迁移。

🧠 深度思考题

L1 理解：解释 Planner 的四种扩缩模式的区别。什么时候用 Throughput 模式，什么时候必须用 SLA 模式？

L2 分析：Request Migration 在什么场景下可能反而会降低系统性能？设计一个场景说明迁移的代价。

L3 创造：设计一个"成本感知"的扩缩策略——不仅考虑 SLA 达标，还考虑 GPU 成本。在什么条件下宁可牺牲一点延迟也要缩容节省成本？
✅ 参考答案与详解

L1 理解: Planner 四种扩缩模式的区别：
（1）Throughput 模式：最简单，基于队列深度和 KV 利用率等静态阈值扩缩——超过阈值就加 Worker，低于阈值就减。适合离线/批处理场景，不需要配置 SLA 目标；
（2）Load 模式：基于用户定义的负载指标（Prefill 队列 token 数、Decode KV 利用率）做反应式扩缩。比 Throughput 更精细，适合有明确负载特征的中等规模部署；
（3）SLA 模式：最强大，以 TTFT/ITL 延迟目标为驱动。内部使用性能模型（FPM）预测不同配置下的延迟表现，支持预测式（proactive）和反应式（reactive）两种策略。必须用 SLA 模式的场景：生产环境、有 SLA 承诺、需要精确控制延迟；
（4）Hierarchical 模式：跨多个集群/区域的全局调度，适合多区域全球部署。
选择指南：开发测试用 Throughput，中等规模生产用 Load，大规模有 SLA 要求的用 SLA，全球多区域用 Hierarchical。

L2 分析: Request Migration 反而降低性能的场景分析：
（1）高频迁移场景：假设 Worker 节点的 GPU 存在间歇性硬件问题（如显存 ECC 错误），每 30 秒触发一次故障。每次迁移的代价：① 状态重建延迟 ~100ms（在 Worker B 上重新注入已生成的 token 序列，需要重新计算 KV 缓存）；② 迁移期间客户端流式输出中断 100-200ms；③ 目标 Worker B 需要从当前 batch 中抢占一个槽位，影响该 batch 中其他请求的 ITL；
（2）具体量化：假设一个请求已生成 4K tokens（OSL=8K），迁移需要在 Worker B 上重做 4K tokens 的 Prefill（约 200ms），加上 KV 传输（~50ms），总延迟增加 250ms。如果迁移 3 次（最大限制），累计延迟增加 750ms，加上首次 Prefill 的 200ms，总 TTFT 变为 950ms，远超正常值 200ms；
（3）更优方案：对于这种场景，应该直接放弃该 Worker 的请求（返回错误让客户端重试），而非反复迁移。Dynamo 的迁移机制应该增加"迁移收益评估"——如果预估迁移延迟 > 重新发送请求的延迟，则放弃迁移。

L3 创造: "成本感知扩缩策略"（Cost-Aware Scaling）设计：
（1）核心公式：Cost_Score = α × SLA_Violation_Penalty + β × GPU_Hourly_Cost × Active_GPU_Count。其中 α 和 β 是可调权重，SLA_Violation_Penalty = 违反次数 × 每次违约罚款（如 $0.01/次）；
（2）缩容条件：当满足以下所有条件时，宁可牺牲一点延迟也要缩容：① 当前 TTFT P95 < 目标值 × 0.5（延迟远低于目标，有 2 倍余量）；② GPU 利用率 < 40%（资源大量空闲）；③ 时间处于低谷期（如凌晨 2-6 点）；④ 缩容后预估 TTFT P95 仍 < 目标值 × 0.9；
（3）实施策略：① 低谷期自动将 SLA 目标放宽 20%（如 TTFT 从 500ms 放宽到 600ms），允许 Planner 缩容；② 使用 Spot/抢占式实例作为弹性容量，低谷期释放 Spot 实例节省 60-70% 成本；③ 设置"成本预算上限"——每日 GPU 成本不超过 $X，当接近预算时主动降低 SLA 目标；
（4）具体场景：日间高峰期（9:00-21:00）保持 16 张 H100，SLA 严格（TTFT<500ms）；夜间低谷期（21:00-9:00）缩容到 8 张 H100，SLA 放宽（TTFT<800ms）。每月节省约 37.5% 的 GPU 成本（从 24h×16 卡降到 12h×16+12h×8=288 卡时 vs 原来 384 卡时）。
