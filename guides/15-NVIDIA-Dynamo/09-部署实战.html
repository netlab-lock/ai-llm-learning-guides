9.2 Docker Compose 部署

前置条件

# 硬件要求
- NVIDIA GPU (A100/H100/H200/B200/GB200)
- CUDA 12.x+
- Docker 24.0+
- NVIDIA Container Toolkit

# 检查 GPU
nvidia-smi

# 检查 Docker GPU 支持
docker run --rm --gpus all nvidia/cuda:12.0-base nvidia-smi

快速启动

# 克隆仓库
git clone https://github.com/ai-dynamo/dynamo.git
cd dynamo

# 使用 Docker Compose 启动
cd deploy/docker-compose
docker-compose up -d

# 查看日志
docker-compose logs -f

# 停止
docker-compose down

配置文件示例

# docker-compose.yml
version: "3.8"
services:
  frontend:
    image: nvcr.io/nvidia/dynamo:latest
    command: dynamo-frontend --port 8080
    ports:
      - "8080:8080"
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]

  router:
    image: nvcr.io/nvidia/dynamo:latest
    command: dynamo-router --strategy kv-affinity
    depends_on:
      - frontend

  prefill-worker:
    image: nvcr.io/nvidia/dynamo:latest
    command: >
      dynamo-worker
        --model meta-llama/Llama-3-70B
        --role prefill
        --tp 4
    deploy:
      resources:
        reservations:
          devices:
            - count: 4
              capabilities: [gpu]

  decode-worker:
    image: nvcr.io/nvidia/dynamo:latest
    command: >
      dynamo-worker
        --model meta-llama/Llama-3-70B
        --role decode
        --tp 8
    deploy:
      resources:
        reservations:
          devices:
            - count: 8
              capabilities: [gpu]

9.3 Kubernetes 部署

Grove Operator

Dynamo 提供了 Grove——一个 K8s Operator，用于拓扑感知的 Gang 调度：

# 安装 Grove Operator
helm repo add nvidia https://helm.ngc.nvidia.com/nvidia
helm install dynamo nvidia/dynamo-operator

# 创建 DynamoGraphDeployment (DGD)
kubectl apply -f - <<EOF
apiVersion: dynamo.nvidia.com/v1beta1
kind: DynamoGraphDeployment
metadata:
  name: llama-70b-disaggregated
spec:
  frontend:
    replicas: 1
  router:
    strategy: kv-affinity
  prefill:
    model: meta-llama/Llama-3-70B
    replicas: 2
    tensorParallel: 4
    resources:
      gpu: 4
  decode:
    model: meta-llama/Llama-3-70B
    replicas: 3
    tensorParallel: 8
    resources:
      gpu: 8
EOF

K8s Inference Gateway 集成

Dynamo v1.0+ 支持 K8s Inference Gateway API，可以作为 EPP（Endpoint Picker Plugin）集成：

Client │ ▼ K8s Inference Gateway │ ▼ EPP (Dynamo KV-aware routing) │ ▼ Frontend sidecar │ ├──→ Prefill Workers │ │ │ └── NIXL KV Transfer │ └──→ Decode Workers

9.5 性能调优关键参数

参数说明默认值调优建议
--tp Tensor Parallelism 自动 节点内用 TP，跨节点用 PP
--pp Pipeline Parallelism 1 超大模型跨节点使用
--ep Expert Parallelism 1 MoE 模型使用，NVL72 可用 EP64
--max-batch 最大批次大小 256 增大可提高吞吐，但增加延迟
--kv-cache-size KV 缓存 GPU 显存比例 0.8 根据模型大小调整
--router-strategy 路由策略 kv-affinity RAG 场景用 kv-affinity
--prefill-ratio Prefill/Decode GPU 比例 auto 长输入场景增加 Prefill

9.6 常见故障排查

症状可能原因排查方法
TTFT 异常高 KV 缓存未命中 检查 Radix Tree 命中率
ITL 波动大 Decode Worker 负载不均 检查 Worker 队列深度
KV 传输超时 NVLink/IB 链路故障 nvidia-smi nvlink -e
OOM (显存不足) KV 缓存过大 减小 --kv-cache-size
Worker 无响应 GPU 故障或驱动问题 dmesg | grep NVRM
自动扩缩不生效 Planner 配置错误 检查 SLA 阈值配置

9.7 监控指标

Dynamo 暴露 Prometheus 指标，关键指标包括：

# 核心推理指标
dynamo_ttft_seconds          # 首 Token 延迟
dynamo_itl_seconds           # Token 间延迟
dynamo_request_duration      # 请求总延迟
dynamo_throughput_tokens     # 吞吐量 (tokens/s)

# KV 缓存指标
dynamo_kv_cache_hit_ratio    # 缓存命中率
dynamo_kv_cache_usage        # 缓存使用率
dynamo_kv_transfer_bytes     # KV 传输量
dynamo_kv_transfer_latency   # KV 传输延迟

# Worker 指标
dynamo_worker_queue_depth    # Worker 队列深度
dynamo_worker_gpu_util       # GPU 利用率
dynamo_worker_memory_usage   # 显存使用率

# 路由指标
dynamo_router_decisions      # 路由决策数
dynamo_router_cache_affinity # 缓存亲和路由比例

⚡ 关键洞察

洞察 1：Dynamo 的零配置部署（DGDR）是面向 DGX 用户的杀手级特性——在 DGX 系统上，Dynamo 可以自动检测 GPU 拓扑、NVLink 连接、InfiniBand 网络，无需手动配置。

洞察 2：K8s 部署的关键不是"能不能跑"，而是"能不能自动恢复"。Grove Operator 的 Gang 调度确保 NVL72 的 72 张卡要么全部启动，要么全部不启动，避免部分启动的资源浪费。

洞察 3：监控是生产部署的生命线。KV 缓存命中率是最关键的指标——如果命中率低于 50%，说明路由策略需要调整。

🧠 深度思考题

L1 理解：画出 Dynamo 在 K8s 部署中的组件关系图，包括 CRD、Operator、Worker 和 Gateway。

L2 分析：在一个 8 卡 H100 节点上，如何分配 Prefill 和 Decode 的 GPU 数量？考虑 Llama-3-70B 模型和 50:50 的输入/输出比例。

L3 创造：设计一个监控告警方案，当 KV 缓存命中率下降时自动调整路由策略。
✅ 参考答案与详解

L1 理解: Dynamo 在 K8s 部署中的组件关系图：
用户通过 K8s Ingress/Gateway 进入集群 → DynamoGraphDeployment（DGD CRD）定义整个推理拓扑 → Grove Operator 监听 DGD CRD，创建和管理 Pod → Frontend Pod（API 网关）接收请求 → Router Pod（智能路由）查询 Radix Tree → Prefill Worker Pod 和 Decode Worker Pod 执行推理 → NIXL 在 Worker 间传输 KV 缓存 → KV Cache Manager Pod 管理多层缓存 → Planner Pod 监控 Prometheus 指标并调整 Worker 副本数。关键 CRD：DGD（定义模型、GPU 数、SLA 目标）、InferenceModel（定义模型配置）。Operator 通过 EndpointSlices 发现 Worker，无需 etcd/NATS。

L2 分析: 8 卡 H100 节点部署 Llama-3-70B（50:50 ISL/OSL）的 GPU 分配分析：
（1）模型约束：Llama-3-70B 需要 TP=8 才能放入单节点（70B × 2 bytes = 140GB > 80GB 单卡），因此无论 Prefill 还是 Decode 都至少需要 8 卡；
（2）分离式方案：由于 8 卡全部用于 TP=8，无法在单节点内同时运行独立的 Prefill 和 Decode 实例。方案选择：① 部署 2 个节点（节点 A 做 Prefill，节点 B 做 Decode），通过 IB 传输 KV；② 在单节点内使用"统一 Worker"模式，不做物理分离；
（3）单节点最优配置：使用 Unified Worker（混合模式），Planner 根据实时负载动态决定每个请求是否走分离路径。配置参数：--max-batch=128, --kv-cache-size=0.75（留 25% 显存给激活值），--router-strategy=kv-affinity；
（4）50:50 ISL/OSL 特征：输入输出均衡，Prefill 和 Decode 计算量相当，分离收益不如长输入短输出场景明显。单节点 Unified 模式的吞吐约比分离模式低 10-20%，但部署复杂度大幅降低。

L3 创造: KV 缓存命中率监控告警与自动路由调整方案：
（1）监控层：在 Prometheus 中设置告警规则：当 dynamo_kv_cache_hit_ratio 的 5 分钟滑动平均 < 50% 时触发 Warning，< 30% 时触发 Critical；
（2）诊断层：告警触发后，自动执行诊断脚本：① 检查请求前缀分布（是否有新类型的请求涌入导致前缀多样性增加）；② 检查 Worker 的 KV 缓存使用率（是否因内存压力导致缓存被频繁驱逐）；③ 检查 Router 的 α 参数是否合理；
（3）自动调整层：① 如果命中率下降是因为前缀多样性增加（新业务上线），自动将 α 从 0.7 降到 0.5（增加负载均衡权重），并将前缀缓存的 TTL 延长；② 如果是因为 KV 缓存被驱逐（内存不足），自动触发 Planner 扩容 Decode Worker 副本数，或降低 --max-batch 为每个请求留更多缓存空间；③ 如果是热点前缀变化，触发 Radix Tree 修剪，清除低频前缀释放内存；
（4）恢复确认：调整后持续监控 15 分钟，如果命中率恢复到 > 60%，发送恢复通知并记录此次事件到知识库，用于未来预测性调整。
