# Consolidated Pitfalls — 学习指南制作踩坑全集

所有在实际制作过程中遇到的问题，按类别整理。主SKILL.md只保留关键摘要。

## 一、生成工具选择

### 导航链接插入新文件（2026-06新增）
插入新编号文件到已有序列时（如在06之后插入07-NVIDIA-Dynamo.html），必须：
1. 前一个文件(06)的nav-bottom右链接 → 新文件(07)
2. 新文件(07)的左链接 → 前一个文件(06)，右链接 → 返回首页
3. 检查README.html和00-总览的卡片网格是否需要新增条目
**Pitfall**: patch操作后可能意外创建自引用链接（文件指向自己）。每次patch nav后立即验证。
**Pitfall**: 多文件更新顺序：先创建新文件 → patch已有文件的引用 → patch README/index。反序操作可能产生broken links。

1. **delegate_task 不能生成HTML** — mimo-v2.5-pro上，所有子代理在生成15-20KB HTML时都会超时(600s)。瓶颈是模型生成时间，不是write_file。解决方案：写Python脚本到`/tmp/gen_XX.py`，用`terminal("python3 /tmp/gen_XX.py")`执行。

2. **delegate_task 可以用于搜索** — `toolsets=["terminal"]` + ddgs CLI，3个并行子代理约80秒完成。

3. **execute_code 沙箱隔离** — 终端里pip装的包在execute_code里不可用。需要用pymupdf等库时用terminal。

4. **write_file 批量 vs 脚本** — 5个文件以下直接write_file；10+文件用Python脚本更快(5-10x)。

## 二、Python脚本陷阱

5. **不要用f-strings包含HTML** — CSS的花括号会触发NameError。用字符串拼接或`.format()`。

6. **不要用三引号嵌套** — HTML代码示例中的`"""`会终止Python的三引号字符串。用`&quot;&quot;&quot;`替代或避免三引号。

7. **反斜杠陷阱** — ASCII art中的`\`在Python三引号中是转义字符。用`→`替代`->`，避免regex。

8. **CJK字符陷阱** — 中文标点（。，）在三引号字符串中可能导致SyntaxError。用write_file直接写HTML更安全。

9. **heredoc不可靠** — `cat > /tmp/script.py << 'PYEOF'`在复杂内容时会静默失败。始终用write_file创建脚本。

10. **脚本大小限制** — 每个脚本处理2-3个文件最可靠。5+文件的脚本可能过大。

## 三、HTML/SVG质量检查

20. **SVG标签污染HTML计数** — 检查`<p>`标签平衡时，`<p`正则会匹配SVG内部标签(`<path`,`<polygon`,`<pattern`)。必须先剥离SVG内容再计数。参见`svg-generation-patterns.md`。

21. **SVG box()函数嵌套花括号** — f-string内写`{5 if not label2 else -2}`会被解析为set literal → TypeError。必须先计算变量再用字符串拼接。

22. **三引号内含单引号** — 中文文本如`'超级GPU'`会提前终止`'''...'''`。用`「」`括号或字符串拼接替代。

## 四、深度与质量

11. **首次就要深** — 不要创建10KB占位文件后期加深。目标15-20KB/模块。如果一次做不到15KB，宁可减少模块数。

12. **深度一致性** — 第一批2-3个模块往往比后面深2x+。完成后必须审计：最薄≥最厚的60%。

13. **批量生成会压缩深度** — 40+文件时每文件会压到4-8KB。先生成全部，再加深最薄的。

14. **加深不是追加** — 反复write_file同一文件但大小不变是常见陷阱。必须整体重写，增加新章节，不是改写现有内容。

15. **8维框架是底线** — 背景/名字由来/原理/细节/场景/对比/关联/效果，每个都不能空。

## 四、内容风格

16. **不要泛泛而谈** — "强大能力""显著提升"是禁语。必须有具体数字。

17. **不要高屋建瓴** — 先基础后前沿。用户明确拒绝只有前沿没有基础的指南。

18. **不要只罗列** — 需要作者视角：经验、判断、类比、踩坑。

19. **数据要新** — LLM/AI领域必须搜索验证，不能只靠训练数据。引用来源。

20. **图表不够多** — 用户反复要求"更多图"。每模块≥2个图+1个代码块。

## 五、工作流陷阱

21. **宣布-然后卡住** — 说"我现在写脚本"但不实际调用工具。规则：说写就必须在同一turn写。

22. **一turn一文件** — 25个文件=25个turn=12+分钟。用脚本批量生成。

23. **read_file去重** — execute_code中read_file对已读文件返回`unchanged`。用terminal(cat/sed)替代。

24. **CSS提取** — 从已有文件提取：`file.split("<style>")[1].split("</style>")[0]`。

25. **不要声明完成太早** — 用户push back说明不够深。审计→加深→再审计。

26. **不要过度声明完成** — 声明3次以上用户仍在push，坚定拒绝并说明原因。

## 六、文件与路径

27. **不要放Desktop** — 用户偏好`D:\\学习\\<topic>\\`。

28. **不要用markdown** — 用户明确偏好暗色主题HTML。

29. **必须有导航** — 每个模块需要prev/next链接和footer。

30. **子代理路径漂移** — delegate_task可能写到`/home/atios/`而不是指定路径。完成后必须验证。

31. **PowerShell中文路径** — `D:\\学习\\`在PowerShell中可能乱码。脚本放Desktop或ASCII路径。

## 七、增强与审计

32. **两维审计** — 文件大小(wc -c) + 内容丰富度(图数量+代码块数量)。

33. **优先级** — 核心模块(02-04)优先加深。概述(01)和总结(05-06)可以5-8KB。

34. **搜索再写** — 快速变化领域先ddgs搜索，再写内容。3个并行子代理约80秒。

35. **整体重写** — 增强薄文件时，整体重写比patch可靠。patch在10行以上的HTML块上容易出错。

## 八、SVG图表升级

36. **f-string花括号嵌套** — Python f-string中`{y+h//2+{5 if x else -2}}`会报TypeError（内层`{5...}`被解析为set literal）。必须先算`yoff = -2 if x else 5`再用`{y+h//2+yoff}`。

37. **不要全换SVG** — 只升级每个文件中最复杂的1-2张架构图（总计8-12张SVG）。简单的流程步骤、代码结构保留ASCII。SVG太多会让文件臃肿且加载慢。

38. **SVG需要CSS** — 每个含SVG的文件必须有`.diagram{}`和`.diagram-title{}`CSS类。用`grep -q '.diagram{' file || sed -i`注入。

39. **语义化配色** — SVG组件按类型着色（前端=青色，后端=翠绿，存储=紫罗兰，编排=琥珀，硬件=玫红）。不要随机配色。

## 九、覆盖度审计

40. **用户会抓遗漏** — 多模块指南完成后，用户几乎一定会问"还有无遗漏"。必须在声明完成前主动做覆盖度审计（regex关键词扫描所有文件）。

41. **基础设施指南必查项** — 架构/框架类指南容易遗漏：弹性扩缩(autoscaling)、容错(fault tolerance)、API使用示例、成本分析(TCO)。这四项是高频缺口。

42. **补丁优于新建** — 发现缺口时优先往现有文件中补内容（用patch），而不是新建文件。新建文件会破坏已有的导航链和学习路径。

43. **shell循环审计不可靠** — 复杂的bash for循环+grep在execute_code沙箱中可能被BLOCKED。用Python脚本（re.findall）做覆盖度审计更可靠。

## 十、批量B级文件增强

44. **CSS子串匹配陷阱** — `.callout.tip {` 包含子串 `.tip {`，简单的 `in` 检查会误判。必须用regex负向后瞻: `(?<![a-zA-Z0-9_-])\.tip\s*\{`。适用于所有需要检测独立CSS类的场景。

45. **CSS-only文件检测** — 文件有 `.tip {` CSS定义但没有 `class="tip"` 教学元素时，不能算A级。检测逻辑必须同时检查CSS定义和实际元素使用。

46. **CSS注入必须写入** — 当所有h2节已有教学元素但CSS被修改时（注入了缺失的CSS类），必须写入文件。不能因为"无section插入"就跳过写入。

47. **插入顺序** — 往HTML中多个位置插入内容时，必须从后往前插入（按position降序），避免后续插入点的位置偏移。

48. **index.html无需增强** — 目录索引页通常无h2标签或只有结构性h2，不需要教学元素。脚本应自动跳过无h2文件。

49. **关键词匹配范围** — 教学内容的关键词匹配应同时扫描h2标题文本和文件路径。路径中的目录名（如"推理框架/"、"KV-Cache系统专题/"）提供了重要的上下文信息。

50. **并行目录处理** — 用delegate_task并行处理多个目录时，每批3个子代理是最佳平衡点。脚本本身是CPU-bound（读写文件），单个目录处理很快（秒级），瓶颈在子代理启动开销。
