# 批量HTML学习指南质量升级 — 踩坑记录

## 来源
333个HTML文件（09-厂商与前沿目录）的质量升级会话。从初始A级(402-442分)升级到100% A+/S级。

## 核心陷阱

### 1. 评分膨胀 — 模板内容骗过评分系统
**问题**: 自动脚本为每个h2节添加"从系统设计角度看...由三个关键组件构成"模板段落后，评分显示100% S级。但实际读取文件发现30%的内容是完全相同的模板文本。

**检测**: `grep -c "由三个关键组件构成" *.html` 或 `grep "采用了分层抽象的设计哲学"`

**修复**: 用正则批量删除模板段落。删除后字数从8000+降到5000-6000，评分下降——这是正确行为。

**教训**: 永远不能只看分数，必须抽样读取实际文件内容验证质量。

### 2. h3子节是最主要瓶颈
**评分公式**: `min(10, h3_count/10 * 10)` — 需要10个h3子节才能获得满分10分。

**问题**: 原始文件通常只有0-3个h3。很多文件的h3标题是模板化的（"X的核心组件与工作原理"）。

**修复**: 
- 删除模板h3标题
- 为每个h2节添加1-2个描述性h3标题+内容段落
- h3标题必须描述实际子主题，不能用通用后缀

### 3. 平淡类比问题
**问题**: 用随机类比池生成的tip有30%是"平淡类比"——类比与主题不匹配。如"类比搜索引擎排序"被用于长上下文主题。

**检测**: 提取所有tip的前80字，统计频率。出现≥3次的tip标记为通用/平淡。

**修复**: 建立细粒度关键词→类比映射表:
- 上下文 → 图书馆索引系统、长篇小说记忆
- MoE → 医院分诊台、餐厅后厨
- 注意力 → 老师课堂提问、显微镜调焦
- 训练 → 飞行员模拟器、高考备考
- 推理 → 消防站响应、自动贩卖机
- 量化 → 照片分辨率、货币换算
- Agent → 项目经理、管家
- 对齐 → 新员工培训、导航校准

### 4. 位置偏移bug
**问题**: Python脚本逐个替换h2节的内容时，前面的替换改变了后面内容的位置，导致后面的替换写入错误位置。

**修复**: 收集所有替换位置后，按位置从后往前应用（reverse order）。

```python
replacements.sort(key=lambda x: x[0], reverse=True)
for start, end, new_text in replacements:
    content = content[:start] + new_text + content[end:]
```

### 5. 子agent 429限流
**问题**: MiMo API在并发>2时返回429。子agent读取大HTML文件(12-22KB)后需要写回完整文件，token消耗大。

**策略**:
- 每批5-6个文件
- 批间等待20-30秒
- 子agent用patch修改（只输出delta）而非write_file（输出完整文件）
- 每个子agent≤10个API调用
- child_timeout=1800s

### 6. 清理模板后的真实分数
**问题**: 清除模板垃圾后，100% S+A+变成72%。这是正确的——分数现在反映真实内容量。

**后续**: 需要逐文件补充真实技术内容（子agent生成），不能用模板重新填充。

## 新增踩坑 (2026-06-28)

### 7. 评分函数不一致导致虚假100%
**问题**: 早期评分函数硬编码`exercises=1, learn_path=1`，导致所有文件获得15分bonus。显示100% S+A+，实际只有72%。
**修复**: 始终使用实际检测: `exercises = 1 if re.search(r'思考与练习|思考题|练习题', content) else 0`

### 8. 模板垃圾规模远超预期
**实际数据**: 908,746字符的模板垃圾，329个文件受影响，平均每文件2,762字。
**模板h3标题**: 539个（"X的核心组件与工作原理"）
**清理后影响**: 100%→72% S+A+（正确反映真实质量）

### 9. h3是所有A级文件的唯一瓶颈
**数据**: 78个A级文件100%以h3为最弱维度。
**原因**: 清除模板h3后，很多文件h3=0。评分`min(10, h3/10*10)`意味着h3=0直接丢10分。
**解决**: 需要为每个h2节添加1个描述性h3+内容段落，每个文件需要5-10个h3。

### 10. 子agent效率演进
**第一版**: 子agent读写整个文件(12-22KB) → 2-3个文件后429
**第二版**: 子agent输出JSON配置 → 脚本应用 → 仍然较大
**第三版**: 子agent用patch修改(delta only) → 5-6个文件/批，最高效
**关键**: 子agent应该只输出变更部分，不输出完整文件。

### 11. 用户要求检查实际内容
**信号**: 用户反复说"还要检查文件内容哦"——不接受只看分数。
**教训**: 必须抽样读取文件，验证tip/warn/deep是否针对主题、h3标题是否描述性、正文段落是否真实。
**检查清单**:
- `grep "由三个关键组件构成"` — 模板正文
- `grep "核心组件与工作原理"` — 模板h3
- `grep "餐厅吃饭"` — 通用tip
- 提取所有tip前80字，统计频率 — 平淡类比

### 12. 空h2节检测
**定义**: h2和下一个h2/tip/warn/deep之间没有`<p>`段落>50字
**检测**:
```python
paras = re.findall(r'<p>(.*?)</p>', section, re.DOTALL)
real_paras = [p for p in paras if len(re.sub(r'<[^>]+>', '', p)) > 50]
if not real_paras: empty_h2 += 1
```
